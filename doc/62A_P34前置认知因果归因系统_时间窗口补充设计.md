# 62A 因果归因系统：时间窗口与全场景补充设计

## 1. 这次补充的核心判断

你指出的问题是正确的，而且比“重开对照”更重要。

之前设计里说：

```text
重开后直接吃腐烂红果，可以排除“之前吃太多红果”的解释。
```

这个说法在原型验证里能用，但作为正式系统不够好。

真正合理的设计应该是：

```text
影响有时间窗口。
影响会累积。
影响会衰减。
影响可能延迟发生。
影响可能持续一段时间。
```

所以不是必须重开，而是：

```text
吃红果后，经过足够时间消化，再吃腐烂红果，就不能把之前红果算作同一轮“吃太多”。
```

换句话说：

```text
因果归因 = 观察事实 + 时间窗口 + 累积状态 + 消退机制 + 替代解释排除
```

而不是：

```text
因果归因 = 重开后验证
```

“重开”只是测试工具，不应该成为游戏内因果判断的主要机制。

---

## 2. 为什么必须加入时间

### 2.1 食物场景

例子：

```text
08:00 吃红果
09:00 吃红果
10:00 吃红果
18:00 吃腐烂红果
18:10 不适
```

如果红果已经被消化，那么 18:10 的不适不应该优先归因为“早上红果吃太多”。

但如果是：

```text
08:00 吃红果
08:05 吃红果
08:10 吃红果
08:15 吃腐烂红果
08:20 不适
```

这时“吃太多”和“腐烂”都应该作为候选原因。

### 2.2 毒性延迟场景

例子：

```text
吃毒蘑菇后 3 小时才发作。
```

如果系统只看当前动作，就会错过真正原因。

### 2.3 药物和治疗场景

例子：

```text
喝草药后 30 分钟开始缓解。
效果持续 4 小时。
```

因果归因必须知道：

```text
结果可能不是立即发生。
```

### 2.4 疲劳、饱腹、感染场景

这些都不是瞬间消失的状态。

```text
吃太多 -> 饱腹/积食 -> 一段时间后消退
受伤 -> 感染风险窗口 -> 过几小时发病
疲劳 -> 休息后恢复
寒冷 -> 离开寒冷环境后逐渐恢复
```

所以系统必须记录状态的持续时间。

## 3. 新增核心概念：EffectWindow

建议新增：

```cpp
struct EffectWindow {
    std::string window_key;
    int starts_after_ticks = 0;
    int active_duration_ticks = 0;
    int decay_duration_ticks = 0;
    double decay_rate = 1.0;
    bool stackable = false;
    StackPolicy stack_policy = StackPolicy::None;
};
```

中文解释：

| 字段 | 中文 | 说明 |
|---|---|---|
| starts_after_ticks | 延迟多久开始 | 毒蘑菇延迟发作 |
| active_duration_ticks | 持续多久 | 饱腹、药效、毒性持续 |
| decay_duration_ticks | 多久衰减完 | 消化、恢复、冷却 |
| decay_rate | 衰减速度 | 线性/快速/慢速都可映射 |
| stackable | 是否可叠加 | 多次吃红果是否累计 |
| stack_policy | 叠加规则 | 叠加强度、刷新时间、取最大值 |

---

## 4. 新增枚举：StackPolicy

建议：

```cpp
enum class StackPolicy {
    Unknown,
    None,              // 不叠加
    AddIntensity,      // 强度叠加
    RefreshDuration,   // 刷新持续时间
    MaxIntensity,      // 取最高强度
    AddDose,           // 剂量叠加
    Replace,           // 后者替换前者
    TestOnly
};
```

中文解释：

| 枚举 | 中文 | 例子 |
|---|---|---|
| None | 不叠加 | 一次性开关 |
| AddIntensity | 强度叠加 | 多次中毒更严重 |
| RefreshDuration | 刷新时间 | 持续燃烧延长 |
| MaxIntensity | 取最高 | 防寒衣只取最好的一件 |
| AddDose | 剂量叠加 | 多次吃同类食物 |
| Replace | 替换 | 新状态覆盖旧状态 |

## 5. 新增概念：ExposureWindow

ExposureWindow 用来记录“近期接触”。

例如：

```text
最近 30 分钟吃了 3 个红果。
最近 2 小时接触过污水。
最近 5 回合吸入烟雾。
```

建议结构：

```cpp
struct ExposureWindow {
    std::string exposure_key;
    std::string family_key;
    std::string action_key;
    int window_ticks = 0;
    int count = 0;
    double total_intensity = 0.0;
    Tick first_tick;
    Tick last_tick;
};
```

它不只服务吃东西，也服务：

- 多次接触毒气。
- 多次受寒。
- 多次过度劳累。
- 多次战斗受伤。
- 多次使用某工具导致磨损或熟练。

---

## 6. 新增概念：ActiveEffect

ActiveEffect 表示当前仍在影响角色或对象的效果。

```cpp
struct ActiveEffect {
    std::string effect_key;
    std::string source_subject_id;
    std::string source_action_key;
    std::vector<CauseFeature> cause_features;
    Tick started_tick;
    Tick expected_end_tick;
    double intensity = 0.0;
    double remaining_intensity = 0.0;
    StackPolicy stack_policy;
};
```

例子：

```text
饱腹效果：吃红果后持续 2 小时。
轻微中毒：吃腐烂红果后 10 分钟开始，持续 3 小时。
寒冷影响：处于雪地时持续增加，离开后慢慢消退。
```

## 7. 因果观察必须带时间上下文

原来的 Observation 不够，需要扩展：

```cpp
struct CausalObservationRecord {
    ObservationId observation_id;
    EntityId actor_id;
    std::string subject_id;
    std::string action_key;
    std::string outcome_key;

    Tick action_tick;
    Tick outcome_tick;
    int elapsed_ticks = 0;

    std::vector<std::string> subject_state_keys;
    std::vector<std::string> actor_state_keys;
    std::vector<std::string> environment_state_keys;

    std::vector<ExposureWindow> relevant_exposures;
    std::vector<ActiveEffect> active_effects_at_action;
    std::vector<ActiveEffect> active_effects_at_outcome;
};
```

重点变化：

```text
action_tick 和 outcome_tick 分开。
```

因为很多结果不是立刻发生。

## 8. 因果候选生成规则要考虑时间

当 outcome 发生时，候选原因不只看当前动作，还要看：

```text
1. 当前动作。
2. 近期 exposure。
3. 当前 active effects。
4. 延迟窗口内发生过的动作。
5. 已知可叠加剂量。
6. 环境持续状态。
```

例如：

```text
08:00 吃毒蘑菇
11:00 不适
```

候选原因应包括：

```text
3 小时前吃毒蘑菇。
过去 3 小时吃过的其他食物。
当前身体状态。
当前环境。
```

## 9. 替代解释排除也要考虑时间

### 9.1 吃多了的解释

只有当相关 exposure 仍在窗口内，才应该作为强替代解释。

例如：

```text
红果消化窗口 = 2 小时
```

那么：

```text
2 小时内吃 4 个红果 -> 剂量解释强
8 小时前吃 4 个红果 -> 剂量解释弱或无效
```

### 9.2 腐烂解释

如果在没有剂量暴露的情况下，腐烂红果仍导致不适，那么腐烂解释增强。

不是必须重开，而是：

```text
当前 exposure window 中没有普通红果过量。
```

### 9.3 环境解释

环境解释也有时间窗口。

例如：

```text
在毒雾区域停留后 10 分钟内不适。
离开毒雾 5 小时后不适，就不一定是毒雾。
```

## 10. 全场景覆盖设计

### 10.1 食物剂量

```text
红果：短时间内吃太多可能不适。
```

需要：

- family_key = berry
- action = eat
- exposure window = 2 小时
- dose threshold = 4

### 10.2 腐烂食物

```text
腐烂状态可能导致中毒。
```

需要：

- subject_state = decayed
- action = eat
- effect window = 立即或短延迟
- 替代解释：剂量、actor_state、环境

### 10.3 药物

```text
草药吃下后 30 分钟生效，持续 4 小时。
```

需要：

- starts_after_ticks
- active_duration_ticks
- decay_duration_ticks

### 10.4 毒蘑菇延迟发作

```text
毒蘑菇吃后 3 小时中毒。
```

需要：

- action_tick != outcome_tick
- delayed cause search

### 10.5 火把燃烧

```text
火把点燃后持续燃烧，雨天会熄灭。
```

需要：

- ActiveEffect: burning
- environment_state: rain
- StackPolicy: RefreshDuration 或 Replace

### 10.6 工具磨损

```text
石片使用多次后破损。
```

需要：

- exposure/action count on object
- object active/accumulated state
- DoseWindow 不只对角色，也对物体

### 10.7 熟练度

```text
多次使用石片后更熟练。
```

需要：

- positive exposure accumulation
- actor skill effect window 可能长期存在

### 10.8 感染

```text
伤口 + 污水，过一段时间感染。
```

需要：

- ActiveEffect: wound
- exposure: dirty_water
- delayed outcome
- multi-factor hypothesis

### 10.9 寒冷和疲劳

```text
长时间寒冷 + 疲劳 -> 生病。
```

需要：

- environment exposure duration
- actor_state exposure
- multi-factor threshold

### 10.10 NPC 投喂判断

```text
给 NPC 吃风险食物。
```

NPC 判断应看：

- 知识是否 Confirmed。
- 风险是否 Ambiguous。
- 投喂者是否知道风险。
- NPC 是否知道风险。
- 是否存在替代解释未排除。

不能因为一次原因不明的不适，就判定玩家恶意。

## 12. 状态转换规则补充

### 12.1 Correlation 不等于 Confirmed

重复发生只能增强相关性：

```text
ObservedCorrelation -> WeaklySupported -> StronglySupported
```

但要变成：

```text
Confirmed
```

必须满足：

```text
主要替代解释已排除。
```

### 12.2 时间窗口会降低替代解释强度

如果替代解释已经过期：

```text
alternative_score -= decay
```

例如红果 8 小时前吃的，就不应强烈解释现在中毒。

### 12.3 ActiveEffect 未结束时必须参与归因

如果 actor 当前还有 active effect：

```text
饱腹
中毒中
疲劳
寒冷
受伤
```

这些都必须成为候选原因或背景条件。

## 13. H5 展示补充

H5 不应只显示：

```text
已确认 / 假设
```

还要显示：

```text
影响窗口
可能原因
是否仍在影响中
建议等待多久再测试
```

示例：

```text
腐烂红果 → 吃 → 身体不适
原因不明：可能是腐烂，也可能是短时间内吃了太多浆果。
提示：等消化一段时间后，再单独尝试腐烂红果，可以排除“吃太多”的解释。
```

又如：

```text
红果 → 吃 → 缓解饥饿
已确认：在当前剂量下可靠。
注意：尚未确认大量食用是否安全。
```

## 14. 测试补充

### 14.1 时间消退测试

```text
吃红果 x3
等待超过消化窗口
吃腐烂红果
结果：吃多了的替代解释弱化，腐烂解释增强
```

### 14.2 时间未消退测试

```text
短时间吃红果 x3
立刻吃腐烂红果
结果：腐烂解释和剂量解释并存，不能确认腐烂
```

### 14.3 延迟发作测试

```text
吃毒蘑菇
若干 tick 后中毒
结果：系统能回溯到毒蘑菇，而不是只看当前动作
```

### 14.4 药效持续测试

```text
喝草药
一段时间内恢复更快
超过持续时间后不再归因于草药
```

### 14.5 工具磨损测试

```text
连续使用石片
达到次数后破损
等待不会恢复，因为磨损是物体持久状态，不是时间消退状态
```

### 14.6 环境暴露测试

```text
在寒冷区停留
离开后寒冷影响逐渐消退
太久后生病不能再强归因于寒冷
```

## 15. 对原设计的修正结论

原设计里“重开后验证”的说法应改为：

```text
在相关 exposure window 清空后验证。
```

重开只是最简单的清空方式，不是唯一方式，也不应该是正式玩法要求。

正式规则应该是：

```text
1. 每个影响都有开始时间、持续时间、衰减时间。
2. 每个剂量判断都有统计窗口。
3. 替代解释只在窗口内有效，过期后权重下降。
4. 延迟结果可以回溯窗口内的原因。
5. 持久状态和临时状态要分开：磨损不会自然消退，饱腹会。
```

## 16. 实现建议更新

因果系统必须新增：

```text
EffectWindow
ExposureWindow
ActiveEffect
TemporalAttributionContext
```

并把 CausalObservationRecord 改成带：

```text
action_tick
outcome_tick
elapsed_ticks
relevant_exposures
active_effects_at_action
active_effects_at_outcome
```

CausalAttributionResolver 必须支持：

```text
时间窗口过滤
剂量窗口统计
延迟原因回溯
效果衰减
持久状态和临时状态区分
```

---

## 17. 最终结论

你的补充是必须加入的，否则因果系统仍然会在大量未来场景里失效。

最终系统不能要求玩家靠重开验证，而应该支持：

```text
等待、观察、对照、延迟、消退、持续、叠加
```

这样后续无论是食物、药物、毒物、工具、环境、战斗、NPC 投喂、组合反应，都能用同一套时间因果框架扩展。
