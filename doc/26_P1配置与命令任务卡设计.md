# 26_P1配置与命令任务卡设计

## 1. 设计目标

本章把 P1 拆成可以直接交给 AI 编码执行的小任务卡。

P1 的目标不是实现完整规则引擎，而是在 P0 foundation 之上建立两个最早需要的上层入口：

```text
配置入口：让后续规则有稳定、可校验、可版本追踪的数据来源。
命令入口：让玩家、AI、复盘系统都能用同一种方式提交行动意图。
```

P1 完成后，项目应该具备：

```text
ConfigPackage / ConfigValidationReport 的最小生产类型。
ConfigLoader 的文件读取空壳。
ActionTarget / CommandEnvelope 的最小生产类型。
Config 与 Command 的 dev_notes / context pack。
对应单元测试。
```

## 2. 本阶段不解决的问题

P1 严禁提前实现：

```text
RuleEngine 完整逻辑。
RulePipeline 完整执行。
InteractionRule 结算。
AgentRuntime。
HTTP / WebSocket。
H5 前端。
复杂 JSON schema。
复杂表达式求值。
完整对象库。
完整事件系统。
完整存档复盘。
```

原因：

```text
P1 只搭配置和命令的承载结构。
规则执行必须等 P2/P3 以后。
如果 P1 提前做引擎，会让 AI 上下文变大，边界混乱，后续难审查。
```

## 3. P1 与 P0 的关系

P1 只能依赖 P0 foundation。

允许依赖：

```text
ErrorCode。
ErrorDetail。
Result<T> / Result<void>。
StrongId。
isValidIdString。
Tick。
StateVersion。
HashValue。
RandomSeed 类型定义。
```

禁止依赖尚未实现的模块：

```text
engine。
rules。
pipeline。
state。
event。
agent。
protocol。
frontend。
```

P1 中出现上层概念时，只能以 ID 或轻量枚举表达，不能实现上层系统。

例如：

```text
CommandEnvelope 可以包含 actor_id。
但不能查询 Actor 是否存在。

ConfigPackage 可以记录 files。
但不能解析完整对象规则。

ConfigValidationReport 可以记录错误。
但不能自动修复配置。
```

## 4. P1 设计原则

### 4.1 先承载，再解释

P1 类型只负责承载数据和做基础校验。

```text
ConfigPackage 承载配置包元信息。
CommandEnvelope 承载一次命令请求。
ActionTarget 承载目标引用。
```

不能在 P1 中解释规则含义。

错误示例：

```text
ActionTarget 判断这个对象能不能吃。
CommandEnvelope 直接扣血。
ConfigLoader 计算食物效果。
```

正确示例：

```text
ActionTarget 校验 target_id 非空、target_type 合法。
CommandEnvelope 校验 command_id、source、payload 存在。
ConfigLoader 读取文件文本并返回错误位置。
```

### 4.2 Config 和 Command 可以互相认识，但不能互相执行

P1 允许 Command 引用配置中的 action_id。

```text
ActionCommand.action_id = "eat"
```

但 P1 不校验 `eat` 是否真的存在于完整配置注册表，最多预留校验接口或返回待校验状态。

原因：

```text
完整引用校验依赖 ConfigRegistry。
ConfigRegistry 不是 P1 的核心目标。
过早实现会扩大上下文。
```

### 4.3 所有错误必须进入 Result

P1 不允许用异常作为常规错误通道。

```text
文件不存在 -> Result<T>::fail。
字段非法 -> ConfigValidationIssue。
命令非法 -> Result<void>::fail 或 CommandValidationReport。
```

### 4.4 所有 ID 先用 StrongId 包装

P1 不允许到处裸用 string 代表业务 ID。

允许裸 string 的场景：

```text
原始文件路径。
原始 JSON 文本。
message_key。
调试说明。
```

业务 ID 必须逐步收敛到 StrongId 子类型。

```text

                -> TASK-P1-008 实现 CommandEnvelope
                  -> TASK-P1-009 实现 CommandValidationReport
```

顺序说明：

```text
先配置，再命令。

先目标，再命令主体。
先命令主体，再命令外壳。
最后做聚合边界检查。
```
