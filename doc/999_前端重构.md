# 999_前端重构：手机横屏 H5 可玩界面重构计划

## 1. 阶段定位

`999_前端重构` 是一个额外阶段，不属于主线 P 阶段编号。

它的目标不是继续修补当前 H5 页面，而是允许把现有 `frontend/h5_playable` 的页面结构、样式和交互全部推倒重来，重新做一个“手机横屏优先”的游戏界面。

当前 H5 已经能访问后端：

```text
GET  /index.html
GET  /style.css
GET  /app.js
POST /api/playable/bootstrap
POST /api/playable/turn
```

后端已有：

```text
P22：H5 对话壳。
P32：H5 安全投影。
P33：H5 极简可玩验证壳。
P37：最小剧情闭环。
P38：世界交互结算与可见后果。
P39：知识链式推理与目标规划。
P40：动态目标执行与外界打断系统设计。
```

本重构阶段只处理前端体验和前端协议消费方式，不改规则权威。

一句话定义：

```text
把当前杂乱按钮式 H5，重构成手机横屏下可读、可操作、能展示 P39 推理、并为 P40 执行状态预留区域的游戏界面。
```

## 2. 为什么现在做

当前网页的问题不是“样式不好看”，而是信息架构已经开始阻碍试玩：

```text
按钮太多，排序混乱。
对象、知识、危险、同伴状态混在一起。
玩家不知道当前最重要的事情是什么。
玩家不知道某个动作会影响谁。
P39 后端已经能推理，但网页没有清楚展示“同伴为什么行动”。
P40 后续会引入目标执行、打断、恢复，如果不先重构前端布局，会更乱。
```

如果继续在现有页面上加按钮和文本，会出现：

1. 手机横屏信息挤压。
2. 玩家视线在日志、按钮、对象列表之间来回跳。
3. P39/P40 的关键反馈被淹没在普通文字里。
4. 后续地图、野兽、同伴行动、材料链全部无法读懂。

所以本阶段允许全量重构前端。

## 3. 阶段目标

完成后，H5 应具备以下体验：

1. 默认面向手机横屏，竖屏只显示“请旋转屏幕”的提示。
2. 玩家一眼能看到：当前局势、危险、可操作对象、同伴意图、最近结果。
3. 按钮不再全局乱堆，而是围绕“选中对象”和“当前目标”出现。
4. P39 推理结果能在界面中展示为“同伴正在想 / 下一步 / 原因 / 阻塞”。
5. P40 预留“目标执行栈 / 当前步骤 / 中断事件 / 恢复状态”的 UI 插槽。
6. 仍然使用后端投影结果，前端不推导规则。
7. 支持手机触控，按钮大、分区清楚、误触少。
8. 在 1999/6006 内网穿透访问时，不需要命令行也能完成第一天最小玩法。

## 4. 本阶段不解决的问题

本阶段不做：

```text
不做 3D / Canvas 地图。
不做复杂美术资源。
不做拖拽背包。
不做前端规则推理。
不做前端存档系统。
不做 P40 后端实现。
不做完整新手引导动画。
不做联网多人。
```

原因：当前最重要的是手机横屏可读性和交互闭环，不是视觉包装。

## 5. 核心设计原则

### 5.1 手机横屏优先

主视口按横屏手机设计。

最低目标：

```text
逻辑尺寸：812x375 到 932x430。
可用高度很小，所以禁止大段文本常驻。
主操作区必须在拇指容易点击的位置。
```

竖屏策略：

```text
显示遮罩：请横屏游玩。
允许继续访问，但不承诺布局完整。
```

### 5.2 一屏三焦点

横屏只保留三个主焦点：

```text
左：世界和对象。
中：事件/对话/当前局势。
右：行动与同伴/P39/P40 状态。
```

不要把知识、对象、行动、日志全部铺满。

### 5.3 操作跟随对象

玩家先选对象，再看到对象相关行动。

错误方式：

```text
全局一直显示：吃红果、吃腐烂红果、用斧头砍木头、制作火把、查看知识、教同伴……
```

正确方式：

```text
点红果 -> 显示 吃 / 观察 / 教同伴这条知识。
点斧头 -> 显示 使用到木头 / 查看状态。
点火把 -> 显示 对野兽使用 / 教同伴。
```

### 5.4 后端权威，前端只投影

前端只能显示后端返回的：

```text
object_cards
available actions
action_bar
scene_summary
actor_knowledge
recipient_knowledge
condition_hints
danger_hints
P39 reasoning projection
P40 execution projection
```

禁止前端自己判断：

```text
这个东西能吃。
这个按钮应该可用。
同伴下一步应该做什么。
狼是否会攻击。
材料是否足够。
```

### 5.5 P39 必须可见，但不变成调试器

P39 展示给玩家的是自然语言解释，不是内部枚举和分数。

玩家看到：

```text
同伴正在判断：降低野兽威胁。
下一步：用火把驱赶野兽。
原因：他知道火把能降低野兽威胁，并且野兽正在靠近。
阻塞：没有火把，需要先制作。
```

玩家不看到：

```text
utility_score
raw_state
hidden_truth
candidate_count
expanded_count
```

### 5.6 P40 预留执行状态，不抢 P39 职责

P40 区域展示执行过程，不展示长期推理细节。

预留字段：

```text
当前目标
当前步骤
暂停原因
外界打断
恢复状态
材料缺口
```

### 5.7 日志降级为历史，不做主界面

当前长文本回复不能占据整个屏幕。

主界面只显示最近 1-3 条关键事件。

完整日志放到抽屉：

```text
事件记录
知识记录
世界变化
系统提示
```

### 5.8 所有可点元素必须有状态

按钮状态必须明确：

```text
可执行
不可执行但可解释
需要目标
需要材料
危险操作
已执行冷却
```

不可执行按钮不能悄悄没反应。

## 6. 目标玩家操作循环

### 6.1 一分钟内循环

```text
1. 看中间局势：现在发生什么。
2. 看左侧对象：有哪些东西。
3. 点一个对象。
4. 右侧出现可行动作。
5. 点击行动。
6. 中间显示结果。
7. 右侧显示同伴/P39 是否产生行动。
8. 玩家决定教同伴、等待、继续探索或应对危险。
```

### 6.2 第一夜目标循环

```text
认识材料 -> 砍木头 -> 点火 -> 制作火把 -> 教同伴 -> 等待野兽 -> 同伴用知识协助。
```

这个循环必须不依赖命令行输入。

## 7. 推荐横屏布局

### 7.1 主布局

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶栏：时间 / 阶段 / 危险等级 / 当前主目标                    │
├───────────────┬──────────────────────────────┬───────────────┤
│ 左：世界对象   │ 中：局势与最近事件             │ 右：行动与思考 │
│ Object Grid   │ Situation Feed                │ Action Panel   │
│               │                              │ P39 Companion  │
├───────────────┴──────────────────────────────┴───────────────┤
│ 底栏：常用操作：等待 / 教同伴 / 查看知识 / 重开 / 日志        │
└──────────────────────────────────────────────────────────────┘
```

### 7.2 推荐比例

```text
顶部栏：40px
底部栏：54px
左栏：30% 宽
中栏：40% 宽
右栏：30% 宽
```

对 812x375 横屏：

```text
顶部 36
底部 50
中间可用 289
```

所以每个区域必须紧凑。

### 7.3 左栏：世界对象区

显示对象卡片，但不是大段说明。

每张卡：

```text
名称
数量/状态短标
风险/材料/工具标签
选中态
```

示例：

```text
红果 x2        可尝试吃
木头 x1        材料
斧头 锋利度2   工具
火把 x0        未制作
野兽 接近中    危险
```

点击对象后，中间或右侧显示详情。

### 7.4 中栏：局势与结果区

中栏负责“发生了什么”。

分三块：

```text
当前局势：一句话。
最近结果：最多 3 条。
对象详情：选中对象的用途认知和状态。
```

禁止中栏无限堆日志。

### 7.5 右栏：行动与同伴思考区

右栏负责“我现在能做什么”和“同伴在想什么”。

结构：

```text
选中对象动作
目标动作
同伴/P39 推理卡
P40 执行预留卡
```

动作按钮按优先级显示：

```text
主要动作：吃 / 使用 / 对目标使用
次要动作：观察 / 教同伴 / 查看知识
危险动作：红色或黄色提示
不可执行动作：灰色，但可点查看原因
```

### 7.6 底栏：全局操作

底栏只放不会依赖对象的全局操作：

```text
等待
教同伴
知识
日志
重开
```

不再放所有物品动作。

## 8. P39 接入设计

### 8.1 后端需要提供的投影

前端希望后端在 H5 projection 中提供一个安全结构：

```cpp
struct CompanionReasoningView {
    bool visible;
    std::string actor_label_zh_cn;
    std::string current_goal_zh_cn;
    std::string next_step_zh_cn;
    std::string reason_zh_cn;
    std::optional<std::string> blocked_reason_zh_cn;
    std::vector<std::string> required_object_labels_zh_cn;
    std::vector<std::string> safe_tags;
};
```

如果后端暂时不新增 DTO，前端第一版可以从现有 `reply_text` / `state_summary_lines` 中显示 P39 执行结果，但必须在代码里标记为过渡方案。

### 8.2 UI 展示方式

右栏显示一张“同伴思考”卡。

无计划：

```text
同伴：暂无明确行动。
```

有计划：

```text
同伴目标：降低威胁
下一步：举起火把驱赶野兽
原因：他知道火把能降低野兽威胁。
```

阻塞：

```text
同伴想做：制作火把
卡住：缺少处理好的木头
建议：先砍木头，或教会同伴相关知识。
```

### 8.3 P39 前端禁止事项

```text
禁止显示 utility_score。
禁止显示 candidate_count。
禁止显示 raw_state。
禁止显示 hidden_truth。
禁止前端根据 effect_key 自己组装“原因”。
禁止前端写死 restore_hunger=红果。
```

## 9. P40 预留设计

P40 还未实现，但前端布局必须预留“执行状态卡”。

### 9.1 预留 DTO

```cpp
struct GoalExecutionView {
    bool visible;
    std::string active_goal_zh_cn;
    std::string active_step_zh_cn;
    std::string status_zh_cn;
    std::optional<std::string> interrupt_zh_cn;
    std::optional<std::string> blocked_reason_zh_cn;
    std::vector<std::string> material_shortage_lines_zh_cn;
    std::vector<std::string> trace_safe_tags;
};
```

### 9.2 UI 展示

示例：

```text
当前执行：为建房准备木头
步骤：伐木
状态：进行中
打断：野兽靠近，已暂停
恢复：驱赶野兽后继续
```

### 9.3 前端预留方式

前端代码中建立组件：

```text
renderExecutionCard(projection.execution)
```

如果后端没有字段：

```text
显示“执行系统尚未接入”。
```

这样 P40 实现时只需要填字段，不需要重排界面。

## 10. 前端模块划分

当前允许重写：

```text
frontend/h5_playable/index.html
frontend/h5_playable/style.css
frontend/h5_playable/app.js
```

建议模块化，但不引入构建工具。

### 10.1 `index.html`

职责：

```text
定义横屏主布局容器。
定义 noscript / loading / portrait overlay。
加载 style.css 和 app.js。
```

禁止：

```text
写大量业务逻辑。
写死对象按钮。
```

### 10.2 `style.css`

职责：

```text
横屏布局。
触控按钮尺寸。
卡片状态。
危险色、知识色、材料色。
响应式断点。
```

必须包含：

```text
@media (orientation: portrait)
@media (max-height: 430px)
.safe-area padding
```

### 10.3 `app.js`

职责：

```text
调用 bootstrap / turn。
保存前端选择态 selectedObjectKey。
渲染 layout。
渲染对象、行动、知识、P39、P40 预留卡。
处理按钮点击。
```

禁止：

```text
根据对象 key 自己判断规则结果。
根据 effect_key 自己创建知识。
绕过 /api/playable/turn 修改状态。
```

## 11. 状态模型

前端只允许保存 UI 状态：

```js
const uiState = {
  sessionId: null,
  projection: null,
  selectedObjectKey: null,
  selectedTab: 'situation',
  logDrawerOpen: false,
  isBusy: false,
  lastError: null
};
```

前端不保存规则状态：

```text
不保存物品真实数量。
不保存知识是否成立。
不保存野兽等级。
不保存材料是否足够。
```

这些都来自后端 projection。

## 12. 行动按钮设计

### 12.1 动作来源

动作按钮来源必须是后端 projection：

```text
action_bar
targeted object actions
```

如果当前接口区分不够清楚，第一版可以对现有 action 做分组展示，但不能改变 action key 的语义。

### 12.2 分组

```text
对象动作：跟 selectedObject 相关。
目标动作：例如“用火把驱赶野兽”。
全局动作：等待、教同伴、知识、日志、重开。
```

### 12.3 按钮状态

每个按钮需要支持：

```text
enabled
disabled
warning
danger
primary
cooldown
```

第一版如果后端没有这些字段，前端只按安全默认显示：

```text
所有后端给出的 action 都可点击。
危险词汇只做视觉提醒，不阻止点击。
```

不是美术阶段，但需要统一语义颜色：

```text
背景：深色低饱和，减少夜晚场景刺眼。

可操作：蓝/青。
危险：红/橙。
```

```text

标题 15-17px。

```

按钮最小触控：

```text

```

## 16. 接口兼容策略

### 16.1 不破坏现有 API

继续使用：

```text

POST /api/playable/turn
```

前端重构不要求后端改路由。

### 16.2 兼容旧 projection

如果后端还没有 P39/P40 字段：


后端后续可在 `H5ProjectionSourceBundle` 增加：

```text
companion_reasoning
execution_status
recent_events

```

前端先预留渲染函数。

## 19. 验收标准

1. `frontend/h5_playable` 可以全量重写，但仍能正常 bootstrap 和 turn。
2. 手机横屏是主体验，竖屏有明确提示。
3. 全局按钮不再混乱堆叠。
4. 对象动作必须基于选中对象展示。
5. 最近事件和完整日志分离。
6. 知识列表不再常驻挤压主操作，可放抽屉或折叠区。
7. P39 同伴推理结果有专门展示区域。
8. P40 执行状态有预留展示区域。
9. 前端不写死规则结果，不根据 effect_key 自己推理。
10. H5 playable 回归测试通过。
11. 1999 端口启动后手机可访问并完成第一夜核心路径。

## 20. 风险与对策

### 20.1 风险：重构变成美术重做

对策：

```text
先做信息架构和触控布局。
不引入复杂美术。
不做动画优先。
```

### 20.2 风险：前端为了方便写规则

对策：

```text
所有规则结果来自后端 projection。
normalizeProjection 只整理字段，不创造结果。
代码审核时搜索 effect_key/object_key 特判。
```

### 20.3 风险：P39 展示变成调试器

对策：

```text
只显示自然语言解释。
内部 trace 放日志抽屉或不显示。
禁止 utility_score/raw_state/hidden_truth。
```

### 20.4 风险：P40 预留过度设计

对策：

```text
只预留一张执行卡和 normalizeExecution。
不实现 P40 逻辑。
后端字段为空时显示占位。
```

### 20.5 风险：手机横屏仍然拥挤

对策：

```text
只保留三栏核心。
长文本进抽屉。
对象卡片短标签化。
知识默认折叠。
```

## 21. 禁止清单

```text
禁止继续把所有动作按钮平铺在一个区域。
禁止前端写死“吃红果会缓解饥饿”。
禁止前端写死“火把驱赶野兽”。
禁止前端根据 P39 effect_key 自己生成解释。
禁止 P40 未实现时假装有执行状态。
禁止竖屏下硬挤完整界面。
禁止把完整日志常驻在主屏。
禁止为了好看牺牲可读性和触控尺寸。
```

## 23. 结论

这次前端重构不是“整理按钮”，而是把 H5 从命令行式对话壳升级为真正可手机横屏游玩的信息界面。

核心判断：

```text
左边看世界，中间看发生了什么，右边做决策和看同伴思考。
```

只要这条成立，P39 的同伴推理和 P40 的动态执行才有地方被玩家看懂。
