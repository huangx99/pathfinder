# 60_P32 协议投影扩展任务卡设计

## 1. 阶段定位

P32 是协议投影扩展阶段，属于“后端权威结果 → H5 可读安全数据”的基础设施层。

P32 不是前端 UI 阶段，不是玩法规则阶段，也不是命令裁决阶段。它的职责是把后端已经产生的权威结果、安全摘要和诊断信息，整理成 H5 可以直接显示的只读投影。

P32 位于以下阶段之后：

- P20：已有知识传播系统，知道“谁把什么知识告诉了谁”。
- P21：已有学习闭环，能从行动、反馈、记忆、知识形成学习结果。
- P22：已有 H5 中文对话框最小可玩，能通过中文文本操作。
- P23-P26：已有族群、共同作战、冲突、文明能力等后端状态能力。
- P27：已有受限条件表达式，条件摘要不能再由前端或旧字符串散落判断。
- P28/P28.5：已有对象反应与配置化物品反应流水线。
- P29：已有危险与探索压力，能输出安全危险摘要和 trace。
- P30：已有存档逻辑包。
- P31：已有真实复盘诊断和 golden replay 报告。

P32 为以下阶段打基础：

- P33：H5 正式最小界面扩展。
- P34：端到端 MVP。
- P37：调试工具增强。
- P38：训练环境前置。
- P41-P44：Alpha/Beta/RC/1.0 发布门禁。

P32 的核心定位：

```text
后端规则、学习、知识、危险、族群、复盘都是权威来源。
P32 只把这些权威结果转换成安全、中文友好、H5 可直接渲染的 Projection。
前端不能根据 Projection 自己推导规则结果。
```

## 2. 为什么现在做

现在必须做 P32，原因有三点。

### 2.1 P22 对话框已经能玩，但数据形态仍偏临时

P22 的中文对话框已经能让玩家通过“吃红果、使用石片、教同伴”等文本操作形成最小闭环。

但 P22 的展示 DTO 更接近对话框原型：

- reply_text 是主要输出。
- knowledge_lines 是文本列表。
- quick_actions 是简单快捷操作。
- debug_keys / warning_keys 仍偏调试。

这对最小文本玩法够用，但对 P33 正式 H5 界面不够。P33 需要知道：

- 当前可见对象有哪些。
- 每个对象能尝试哪些行动。
- 行动是否可用，为什么不可用。
- 玩家当前知道什么。
- 同伴知道什么。
- 哪些知识是已确认、候选、假设、共享。
- 危险提示、族群状态、复盘状态如何显示。
- 哪些内容是玩家可见，哪些只能调试可见。

所以必须在 P33 前建立统一协议投影。

### 2.2 后端系统越来越多，前端不能直接拼状态

目前后端已经有：

- 认知。
- 记忆。
- 知识。
- 传播。
- 族群。
- 共同作战。
- 冲突。
- 文明能力。
- 对象反应。
- 危险。
- 存档。
- 复盘诊断。

如果没有 P32，H5 前端很容易出现错误架构：

- 前端自己判断对象能不能吃。
- 前端自己把 knowledge status 翻译成规则结果。
- 前端自己根据 danger key 推断危险后果。
- 前端直接读取 debug key 或 raw state。
- 前端为了展示写第二套规则。

这会破坏整个项目“后端规则引擎权威”的架构。

### 2.3 P31 后需要把复盘状态安全暴露给调试界面

P31 已经能生成 golden replay 诊断报告，但它是研发诊断结构，不适合直接给 H5。

P32 需要把 P31 报告转换成安全投影：

- 只展示“复盘通过/失败/有警告”。
- 只展示安全 issue 摘要。
- 不暴露隐藏真相、真实 HP、死亡、掉落、随机伤害等禁止字段。
- 不让玩家界面依赖复盘诊断来判断游戏规则。

## 3. 阶段目标

P32 完成后，系统应具备以下能力：

1. 定义统一的 H5 只读协议投影结构。
2. 把对象、行动、知识、认知、条件摘要、危险提示、族群摘要、复盘摘要组织成一个 `H5GameProjection`。
3. 明确哪些字段能给玩家 UI，哪些只能给调试 UI，哪些禁止输出。
4. 定义 `ProjectionAudience`，区分 player、developer_debug、training。
5. 定义 `ProjectionMode`，区分 full、delta、debug_snapshot。
6. 定义安全文本、状态标签、行动按钮、知识行、条件摘要的统一格式。
7. 扩展协议 envelope 的 payload 类型设计，让 H5 不直接消费零散 DTO。
8. 为 P33 正式 H5 界面提供稳定数据契约。
9. 为 P37 调试工具和 P38 训练环境保留安全扩展点。
10. 给后续实现提供明确的子阶段、接口、测试和门禁。

P32 设计完成后，研发实现者应能直接新增类似：

```text
pathfinder_h5_projection
H5ProjectionComposer
H5ProjectionSanitizer
H5ProtocolProjectionAdapter
```

而不需要重新讨论“前端能不能自己判断规则”。答案固定：不能。

## 4. 本阶段不解决的问题

P32 明确不做：

1. 不做 H5 正式界面布局。P33 负责 UI。
2. 不做 CSS、美术、动画、移动端适配。P33/P34 负责。
3. 不新增玩法规则。对象反应仍由 P28/P28.5，危险仍由 P29，知识仍由 P18-P21。
4. 不新增命令执行。玩家行动仍必须提交 `CommandEnvelope` 或 P22/P21 已定义的学习入口。
5. 不允许前端通过 projection 直接修改状态。
6. 不做真实网络协议升级，只设计后端 DTO 和 envelope 契约。
7. 不做完整调试报告导出，P37 负责。
8. 不做训练环境 observation/reward/done，P38 负责。
9. 不做多命令 replay 展示，等 replay runner 支持后再扩展。
10. 不做玩家不可见真相展示。

P32 只解决“把已有后端权威结果安全投影给 H5”的问题。

## 5. 核心设计原则

### 5.1 投影只能读，不能裁决

P32 所有 projection 都是只读结构。

它不能：

- 判断对象是否真的能吃。
- 判断行动是否真的成功。
- 修改 `GameState`。
- 修改知识状态。
- 直接触发对象反应。
- 直接触发危险伤害。

它只能读取后端已经给出的安全摘要或权威结果，并把它们转换成 H5 可显示格式。

### 5.2 前端只显示，不推导规则

H5 前端不能根据字段组合推导规则结果。

例如：

- 看到 object tag 为 berry，不能自己判断“可食用”。
- 看到 danger hint，不能自己判断“会中毒”。
- 看到 condition summary，不能自己判断“行动成功”。
- 看到 knowledge status，不能自己修改 quick action 可用性。

quick action 是否 enabled 必须来自后端投影。

### 5.3 中文展示与稳定 key 分离

P32 必须同时提供：

- 稳定 key：给程序、测试、回放、协议使用。
- 中文文本：给 H5 直接展示。

不能只给中文文本，否则后续测试和协议会不稳定。

也不能只给 key，否则界面会继续像调试信息。

### 5.4 安全分层

P32 必须区分不同受众：

- 玩家 UI：只能看到玩家应该知道的内容。
- 开发调试 UI：可以看到更多安全 trace，但仍不能看到隐藏真相。
- 训练环境：可以看到训练需要的安全摘要，但不能读取原始状态。

受众不同，投影内容不同。

### 5.5 条件摘要只来自 P27

所有条件相关投影必须来自 P27 的 condition expression / condition summary。

P32 不允许重新发明：

- condition_key 解析。
- 条件是否满足。
- 条件中文解释。
- 行动失败原因判断。

P32 只接收安全 condition summary 并展示。

### 5.6 知识展示不能变成真相展示

玩家知道“红果吃后缓解饥饿”，是知识。

玩家不知道“腐烂红果真实隐藏 poison value”，就不能展示真相字段。

P32 展示知识时必须基于：

- KnowledgeClaim。
- CognitionClaim。
- Teaching/Propagation 结果。
- 安全 reason key。

不能直接展示对象真实属性。

### 5.7 支持增量，但先保证全量

P32 设计 `ProjectionMode` 支持 full 和 delta。

但第一版实现建议先完成 full snapshot projection。delta projection 只设计接口和兼容字段，不强制本阶段完全实现复杂 diff。

### 5.8 投影必须可测试

每个投影 DTO 都必须可 `validateBasic()`。

每个 builder 都必须测试：

- 正常投影。
- 禁止字段拒绝。
- 玩家模式隐藏 debug。
- 调试模式仍不暴露 hidden truth。
- quick action 不由前端推导。

## 7. 核心枚举设计

### 7.1 `ProjectionAudience`

中文含义：投影受众。

用途：决定哪些字段可以出现在 projection 中。

| 枚举值 | 中文含义 | 使用场景 |
|---|---|---|
| `Unknown` | 未知 | 默认非法值，validate 必须拒绝 |
| `Player` | 玩家 | H5 正式玩家界面，只显示玩家应知道的内容 |
| `DeveloperDebug` | 开发调试 | 开发者调试界面，可显示安全 trace 和 issue 摘要 |
| `Training` | 训练 | 后续 P38 训练环境，显示安全 observation 摘要 |
| `TestOnly` | 测试专用 | 单元测试构造非法或边界路径 |

要求：

- 必须提供 `toString(ProjectionAudience)`。
- 必须提供 `projectionAudienceFromString()`。
- `Player` 是默认产品模式。
- `DeveloperDebug` 也不能暴露 forbidden key。

### 7.2 `ProjectionMode`

中文含义：投影模式。

| 枚举值 | 中文含义 | 使用场景 |
|---|---|---|
| `Unknown` | 未知 | 默认非法值 |
| `Full` | 全量投影 | P33 初版主要使用，返回完整 H5GameProjection |
| `Delta` | 增量投影 | 后续减少前端刷新成本 |
| `DebugSnapshot` | 调试快照 | 给 P37 调试工具显示安全快照 |
| `TrainingObservation` | 训练观察 | 给 P38 训练环境安全观察使用 |
| `TestOnly` | 测试专用 | 测试边界 |

要求：

- P32 设计 full 和 delta，但第一版实现可先完成 full。
- delta 不能省略安全校验。

## 8. 核心 DTO / 数据结构设计

### 8.1 `H5ProjectionRequest`

中文含义：H5 投影请求。

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `request_id` | string | 请求 ID | 是 | 协议层 | Composer | 非空、合法 key | 路径、空格、隐藏字段 |
| `session_id` | string | H5 会话 ID | 是 | H5 session | Composer | 非空 | raw state |
| `audience` | ProjectionAudience | 受众 | 是 | 后端入口 | Sanitizer | 不能 Unknown/TestOnly | 无 |
| `mode` | ProjectionMode | 投影模式 | 是 | 后端入口 | Composer | 不能 Unknown/TestOnly | 无 |
| `requested_sections` | vector<ProjectionSectionKind> | 请求区块 | 否 | 前端/后端 | Composer | 不允许 Unknown | DebugPanel 需权限 |
| `max_items_per_section` | size_t | 每区块最大条目 | 否 | 后端默认 | Builders | 0 表示默认 | 不能无上限 |
| `include_debug_trace` | bool | 是否包含调试 trace | 否 | DeveloperDebug | Sanitizer | Player 必须 false | hidden truth |
| `language` | string | 文本语言 | 否 | 前端 | Text builder | 当前固定 zh_cn | 任意脚本注入 |

说明：

- P32 第一版默认 `language = zh_cn`。
- `requested_sections` 为空表示按 audience 使用默认区块。
- `include_debug_trace` 在 Player 模式下必须被拒绝或强制 false。

### 8.2 `H5ProjectionSourceBundle`

中文含义：H5 投影来源包。

这是 P32 最重要的输入聚合结构。它只能收集安全来源，不能直接暴露 raw state。

## 9. 核心服务接口设计

### 9.1 `ProjectionSanitizer`

中文职责：投影安全清洗器。

建议接口：

```cpp
class ProjectionSanitizer {
public:
    Result<void> validateRequest(const H5ProjectionRequest& request) const;
    Result<void> validateSafeText(const SafeTextProjection& text, ProjectionAudience audience) const;
    Result<void> validateProjection(const H5GameProjection& projection) const;
    Result<H5GameProjection> filterForAudience(H5GameProjection projection, ProjectionAudience audience) const;
};
```

要求：

- Player 模式必须移除 debug panel 和 debug_keys。
- DeveloperDebug 也必须拒绝 forbidden key。
- 不允许“发现 forbidden key 后简单替换成空字符串并继续”，必须记录 `ProjectionIssue`。
- 如果 forbidden key 出现在关键字段，必须阻止该区块输出。

### 9.2 `ObjectCardProjectionBuilder`

中文职责：构建对象卡片。

输入：

- P22 DialogScenarioObject 或后续对象安全摘要。
- Knowledge/Cognition badge 安全摘要。
- Action affordance 安全摘要。

输出：

- `vector<ObjectCardProjection>`。

要求：

- 不读取对象真实效果。
- 不判断吃了会怎么样。
- 不直接读取 raw WorldObject 内部隐藏 trait。
- 如果只有对象 display_name，也可以构建最小卡片。

### 9.3 `ActionAffordanceProjectionBuilder`

中文职责：构建可操作入口。

输入：

- 后端允许的 action candidate。
- P27 condition summary。
- P22 quick action。
- P28/P28.5 reaction 可尝试入口摘要。

输出：

- `vector<ActionProjection>`。

要求：

- `enabled` 必须由后端决定。
- disabled_reason 必须来自 P27 或后端安全 reason。
- 不允许前端拿 `target_object_refs` 自己拼 command 结果。

### 9.4 `KnowledgePanelProjectionBuilder`

中文职责：构建知识面板。

输入：

- `KnowledgeProjection`。
- `CognitionQueryResult` 或 CognitionClaim 安全摘要。
- P20 propagation result 安全摘要。

输出：

- actor knowledge lines。
- recipient knowledge lines。
- cognition lines。

要求：

- 必须把 key 翻译成中文文本。
- 必须保留稳定 key 便于测试。
- 不允许出现 `red_berry + eat -> poison` 这种纯调试格式作为玩家主展示。
- 可以保留 debug key，但 Player 模式不展示。

### 9.5 `ConditionSummaryProjectionBuilder`

中文职责：构建条件提示。

输入：

- P27 `ConditionSummary`。
- 行动可用性 reason。

输出：

- `ConditionSummaryProjection`。

要求：

- 不解析 legacy condition_key。
- 不自己判断条件是否满足。
- 对隐藏条件输出模糊文本，例如“你还缺少必要经验”，而不是隐藏数值。

### 9.6 `DangerHintProjectionBuilder`

中文职责：构建危险提示。

输入：

- P29 hazard resolution safe summary。
- P29 countermeasure summary。
- P29 tribe pressure safe summary。

输出：

- `DangerHintProjection`。

要求：

- 不展示死亡、掉落、真实伤害、真实 HP。
- 不展示“这一定会杀死你”这类真相式文本，除非后端知识系统已让玩家知道。
- 可以展示“看起来有风险”“同伴对此感到不安”等安全文本。

要求：

- Player 模式优先展示模糊等级和中文提示。
- DeveloperDebug 可以展示安全 reason key。
- 不展示内部精确分裂概率，除非该概率被设计为玩家可见指标。

### 9.8 `ReplayStatusProjectionBuilder`

中文职责：构建复盘诊断摘要。

- DeveloperDebug 可以展示 passed/failed、issue summary、condition trace summary。
- 不展示 package raw hash，除非 debug 配置允许且 hash 不含 hidden 信息。
- 不把 replay 结果当玩法结果显示给玩家。

### 9.9 `H5ProjectionComposer`

中文职责：总投影编排器。

建议接口：

```cpp
class H5ProjectionComposer {
public:
    Result<H5GameProjection> compose(
        const H5ProjectionRequest& request,
        const H5ProjectionSourceBundle& sources) const;
};
```

流程：

1. validate request。
2. validate sources。
3. 根据 audience 和 requested_sections 选择区块。
4. 调用各 builder。
5. 合并 issues。
6. 调用 sanitizer。
7. 输出 `H5GameProjection`。

要求：

- Composer 不执行规则。
- Composer 不直接修改来源数据。
- Composer 不吞掉 builder 错误。

### 9.10 `H5ProtocolProjectionAdapter`

中文职责：协议 envelope 适配器。

建议接口：

```cpp
class H5ProtocolProjectionAdapter {
public:
    ProtocolEnvelope buildProjectionEnvelope(
        const H5GameProjection& projection,
        const H5ProjectionProtocolOptions& options) const;
};
```

要求：

- `ProtocolPayloadType` 需要设计新增 `H5GameProjection`。
- `ProtocolDomain` 可继续使用 `ProjectionSync`。
- envelope 只携带 payload 类型和元信息；真实 projection 序列化格式可后续实现。

## 10. 标准流程

### 10.1 全量投影流程

1. H5 或服务层创建 `H5ProjectionRequest`。
2. 后端收集 `H5ProjectionSourceBundle`。
3. `ProjectionSanitizer` 校验 request。
4. `H5ProjectionComposer` 判断默认 sections。
5. `ObjectCardProjectionBuilder` 构建对象卡片。
6. `ActionAffordanceProjectionBuilder` 构建行动栏。
7. `KnowledgePanelProjectionBuilder` 构建知识和认知面板。
8. `ConditionSummaryProjectionBuilder` 构建条件提示。
9. `DangerHintProjectionBuilder` 构建危险提示。
10. `TribeStatusProjectionBuilder` 构建族群摘要。
11. `ReplayStatusProjectionBuilder` 在 Debug audience 下构建复盘摘要。
12. Composer 合并投影并设置 header。
13. Sanitizer 执行最终安全过滤。
14. Adapter 包装 `ProtocolEnvelope`。
15. H5 前端只渲染 projection。

### 10.2 Player 模式过滤流程

Player 模式必须：

- 移除 `DebugPanel`。
- 移除 `debug_keys`。
- 默认不输出 `replay_status`。
- 不输出 forbidden key。
- 不输出 raw state、真实 HP、死亡、掉落、随机伤害。
- 对隐藏条件输出模糊提示。

### 10.3 DeveloperDebug 模式流程

DeveloperDebug 模式可以：

- 展示 projection issues。
- 展示安全 reason keys。
- 展示 P31 replay summary。
- 展示 condition trace summary。

但仍然不能：

- 展示 hidden truth。
- 展示 raw GameState。
- 展示真实内部伤害数值。
- 允许前端裁决规则。

### 10.4 Delta 模式流程

Delta 模式设计保留：

- request 可携带 previous projection id。
- output 可标记 changed section。
- 每个 section 可带 version key。

但 P32 第一版实现可以只返回 full，并在 header 中说明 delta not_supported。

## 11. 典型场景

### 11.1 玩家看到红果和腐烂红果

后端投影应展示：

- 对象卡片：红果、腐烂红果。
- 行动：尝试食用、查看。
- 知识：如果玩家已经学到，则显示“红果：吃后可以缓解饥饿”。
- 腐烂红果知识必须独立显示，不得被红果覆盖。

禁止：

- 前端根据名字判断能不能吃。
- 前端把腐烂红果和红果合并成同一个知识。

### 11.2 玩家学会石片可使用

投影应展示：

- 石片对象卡。
- “尝试使用”行动。
- 知识行：“石片：使用后可能有工具用途”。
- 状态徽章：假设/候选/已确认。

禁止展示：

- `stone_flake + use -> use_hint [active]` 作为玩家主文本。

### 11.3 玩家把知识教给同伴

投影应展示：

- 玩家知识区。
- 同伴知识区。
- teach action 是否 enabled。
- 教导结果后的 shared/hypothesis 状态。

禁止：

- 前端自己把玩家知识复制给同伴。
- 前端自己提升知识状态。

### 11.4 危险提示

如果 P29 给出腐烂红果风险摘要，投影可以显示：

- “你对这个东西有不好的经验。”
- “同伴似乎对这个选择感到不安。”

禁止显示：

- “吃了会扣 20 HP”。
- “这会导致死亡”。
- “会掉落尸体/战利品”。

### 11.5 复盘失败

DeveloperDebug 模式可以显示：

- “复盘失败：事件数量与记录不一致”。
- “建议检查 replay 记录或规则变更”。

Player 模式默认不显示。

禁止：

- 玩家界面根据 replay failed 判断行动失败。
- 前端读取 raw replay log。

## 12. 与现有系统关系

### 12.1 与 P20 知识传播

P32 读取 P20 的传播结果和知识投影。

P32 不负责传播，不改变接收者知识。

### 12.2 与 P21 学习闭环

P32 可以读取学习结果的安全摘要，用于展示“这次经历形成了新知识”。

P32 不负责生成知识。

### 12.3 与 P22 H5 对话框

P22 的 DialogResponseDto 是最小文本原型。

P32 设计后，P33 应逐步改为读取 `H5GameProjection`，而不是从 reply_text 和 knowledge_lines 里硬解析状态。

P22 可以继续作为兼容入口。

### 12.4 与 P27 条件表达式

P32 只读取 P27 的 safe summary。

禁止：

- 在 P32 重新解析 legacy condition_key。
- 在 P32 判断条件是否满足。
- 在前端判断条件是否满足。

### 12.5 与 P28/P28.5 对象反应

P32 可以展示“组合”行动入口，例如火 + 木棍。

P32 不执行反应，不生成火把。

火把是否生成仍由 P28/P28.5 后端规则决定。

### 12.6 与 P29 危险系统

P32 读取危险安全摘要和 countermeasure 提示。

P32 不触发危险，不修改族群压力，不输出真实伤害。

### 12.7 与 P30 存档

P32 可以显示存档相关的安全状态，例如“当前会话可保存”。

但 P32 不读写真实存档，不展示 raw save package。

### 12.8 与 P31 复盘诊断

P32 可以把 P31 report 转成 debug projection。

P32 不能：

- 把 replay 结果当玩法结果。
- 暴露 raw replay log。
- 让玩家 UI 依赖 replay panel。

### 12.9 与 Protocol 系统

P32 需要扩展或设计以下 payload：

- `H5GameProjection`
- `H5ProjectionDelta`
- `H5ProjectionIssueList`

第一版可以只实现 `H5GameProjection`。

### 12.10 与 P33 H5 正式界面

P33 应只根据 P32 projection 渲染界面。

P33 不允许：

- 自己拼知识。
- 自己判断行动 enabled。
- 自己解析 condition key。
- 自己判断危险。

## 13. 安全边界

P32 必须拒绝以下 key 或文本：

- `hidden_truth`
- `true_trait`
- `real_effect`
- `raw_state`
- `gamestate`
- `actual_hp`
- `true_hp`
- `hp_delta`
- `death`
- `kill`
- `corpse`
- `loot`
- `drop`
- `random_damage`
- `frontend_unlock`
- `direct_state`

P32 额外禁止：

- `raw_replay_log`
- `raw_save_package`
- `internal_probability`
- `unmasked_condition`
- `secret_config`

如果发现 forbidden key：

1. Player 模式必须阻止对应字段或区块输出。
2. DeveloperDebug 模式也不能输出原内容。
3. 必须记录 `ProjectionIssueKind::ForbiddenKeyRejected`。
4. 不能静默吞掉。

## 14. 错误实现禁止清单

P32 绝对不允许以下实现：

1. 不允许前端根据 object tag 判断可食用、可使用或危险。
2. 不允许前端根据 knowledge status 修改同伴知识。
3. 不允许前端根据 condition key 判断行动是否可用。
4. 不允许 P32 直接调用规则 resolver 执行玩法结果。
5. 不允许 P32 修改 `GameState`。
6. 不允许 P32 读取并输出 raw WorldObject 隐藏属性。
7. 不允许 P32 输出真实 HP、死亡、掉落、随机伤害。
8. 不允许 P32 把 P31 replay mismatch 展示成玩家行动失败原因。
9. 不允许把 `debug_keys` 暴露到 Player audience。
10. 不允许为了 H5 方便新增第二套轻量规则。
11. 不允许只输出英文调试格式，不给中文解释。
12. 不允许只输出中文文本，不给稳定 key。
13. 不允许让 payload 变成任意 JSON 字符串后绕过 validate。
14. 不允许跳过 sanitizer。
15. 不允许修改 P18-P31 的权威系统边界。

## 18. 风险与控制

### 18.1 风险：P32 变成第二套规则系统

原因：为了让前端方便展示，builder 可能开始判断对象能不能吃、危险会不会触发。

后果：规则权威被破坏，前后端结果不一致。

控制：P32 只能读取后端安全摘要，禁止调用规则 resolver 产生结果。

### 18.2 风险：前端继续解析调试字符串

原因：如果 P32 只输出类似 `red_berry + eat -> poison`，H5 会继续像调试面板。

后果：玩家看不懂，前端也可能硬解析字符串。

控制：每条展示必须同时有稳定 key 和中文 `SafeTextProjection`。

### 18.3 风险：Player 和 Debug 混在一起

原因：为了省事直接把 debug_keys 给前端。

后果：玩家看到内部信息，甚至泄漏隐藏真相。

控制：`ProjectionAudience` 必须参与每次 compose，Player 模式强制过滤。

### 18.4 风险：复盘诊断被误用为玩法反馈

原因：P31 replay failed 看起来像行动失败。

后果：玩家界面显示错误逻辑。

控制：ReplayStatusProjection 默认只给 DeveloperDebug，Player 隐藏。

### 18.5 风险：条件表达式被重复实现

原因：Action disabled reason 需要条件，开发者可能直接解析 condition key。

后果：P27 权威被绕过。

控制：P32 只读取 P27 ConditionSummary，不解析旧 key。

### 18.6 风险：投影 DTO 过大导致后续难维护

原因：一次把所有系统 raw data 塞入 H5GameProjection。

后果：协议臃肿、泄漏、难测试。

控制：按 section 拆分，列表限制，truncated 标记，source bundle 只接安全摘要。

### 18.7 风险：低级 AI 实现时跳过 Sanitizer

原因：为了快速通过测试，直接返回 projection。

后果：安全门禁失效。

控制：测试必须覆盖 sanitizer，Composer 最后必须调用 sanitizer。

1. 不得重构无关模块。

3. 不得新增第二套规则判断。

## 21. 总结

P32 是 P33 正式 H5 界面的前置地基。

它的核心不是“给前端更多数据”，而是“给前端正确、安全、中文友好、后端权威的数据”。

完成 P32 后，H5 应该只负责渲染：

- 展示对象。
- 展示行动按钮。
- 展示知识和认知。
- 展示危险、族群、复盘摘要。
- 提交玩家选择。

H5 不负责：

- 判断规则。
- 修改状态。
- 传播知识。
- 推断危险。
- 解释隐藏条件。
- 裁决 replay。

这样后续 P33/P34 才能做出真正可玩的界面，而不是把规则逻辑散落到前端。
