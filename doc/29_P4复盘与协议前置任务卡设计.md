# 29_P4复盘与协议前置任务卡设计

## 1. 设计目标

P4 的目标是在 P3 已通过的 `unknown_fruit + eat` 真实规则闭环之上，补上两个后续系统必须依赖的工程地基：

```text
复盘地基：同一初始状态 + 同一命令 + 同一随机种子，可以被记录、重放、校验。
协议地基：后端结果可以被包装成稳定的 ProtocolEnvelope，给未来 H5 / Agent / 工具消费。
```

一句话：

```text
P4 不新增玩法，P4 把 P3 变成“可记录、可重放、可对外传输”的工程闭环。
```

P4 之后，后续 AI 不应该再直接拿 `PipelineResult` 当外部输出，也不应该只靠测试临时比较状态，而应该通过：

```text
CommandReplayRecord
RandomReplayLog
ReplayRunner
ProtocolEnvelope
```

来固定复盘与前端前置契约。

## 2. 为什么 P4 先做复盘与协议

P3 证明了第一条规则能跑通，但还缺少两个关键工程能力：

```text
1. 如果玩家吃了未知果实，系统如何证明这次结果以后还能复现？
2. 如果未来 H5 前端要显示结果，它应该读取什么稳定外壳，而不是读取内部 C++ 结构？
```

如果不先做 P4，后续会出现这些风险：

```text
前端直接读取内部 PipelineResult，后续一改结构就崩。
Agent / 强化学习训练无法拿到稳定轨迹。
测试只能断言最终状态，不能解释状态为什么发生。
随机系统一旦加入，无法判断结果变化是 bug 还是随机。
存档系统容易只保存 GameState，而丢失命令和证据。
```

P4 要把 P3 的一次命令执行变成可记录的事实链：

```text
initial_state_hash
command
random_seed
pipeline_result
state_change_ids
event_ids
cognition_update
final_state_hash
protocol_response
```

## 3. 本阶段解决什么问题

P4 解决四个问题：

```text
命令复盘：记录一次命令执行前后发生了什么。
随机复盘：记录 seed 和 draw，未来随机规则可以复现。
重放校验：同样输入再次执行，结果必须稳定。
协议外壳：统一包装后端响应、事件、错误和状态版本。
```

P4 必须证明：

```text
CommandReplayLog 不修改 GameState。
RandomReplayLog 不生成玩法结果，只记录随机过程。
ReplayRunner 通过 RulePipeline 重放，不绕过管线。
ProtocolEnvelope 不暴露 Resolver 内部结构。
前端未来只能消费协议外壳、事件和投影，不直接改状态。
```

## 5. P4 与 P0/P1/P2/P3 的关系

P4 可以依赖：

```text
foundation：StrongId、Result、ErrorDetail、Tick、StateVersion、HashValue、RandomSeed、RandomDrawRecord。
command：CommandEnvelope、ActionCommand、ActionTarget。
state：GameState、GameStateMetadata、StateChangeSet。
event：EventRecord、EventStream。
pipeline：PipelineContext、PipelineResult、RulePipeline。
rules：UnknownFruitFixture，仅用于 P4 集成测试。
cognition：CognitionState 最小结果，仅用于重放校验。
```

P4 不应依赖：

```text
engine。
agent。
frontend。
HTTP / WebSocket。
真实文件系统存档。
复杂配置注册表。
```

P4 的关键边界：

```text
replay 记录和校验事实。
protocol 包装对外消息。
pipeline 仍然是规则执行入口。
rules 仍然只包含 P3 的 eat resolver。
```

## 6. P4 最小闭环定义

### 6.1 输入

```text
GameState：UnknownFruitFixture::createInitialState()。
CommandEnvelope：UnknownFruitFixture::createEatCommand()。
RandomSeed：42。
```

### 6.2 第一次执行

```text
创建 PipelineContext。
调用 RulePipeline::execute。
得到 PipelineResult。
生成 CommandReplayRecord。
追加到 CommandReplayLog。
追加 RandomReplayLog 起始 seed 记录。
把 PipelineResult 包装成 ProtocolEnvelope response。
```

### 6.3 重放执行

```text
重新创建相同 initial GameState。
ReplayRunner 读取 CommandReplayLog 第一条记录。
使用记录中的 command 和 random_seed。
再次调用 RulePipeline::execute。
生成 ReplayCompareReport。
断言状态变化数量、事件类型序列、最终 hunger、consumed、认知 claim 稳定。
```

### 6.4 协议输出

ProtocolEnvelope 至少表达：

```text
protocol_version。
message_id。
message_type = response。
domain = command。
request_id 可选。
state_version。
payload_type = command_result。
errors 为空。
warnings 可为空。
```

P4 不要求 JSON 序列化，但字段必须稳定、可校验、可测试。

## 8. P4 类型中文翻译表

| 类型 | 中文名 | 作用 |
|---|---|---|
| CommandReplayRecord | 命令复盘记录 | 记录一次命令执行的输入、输出摘要和校验信息 |
| CommandReplayLog | 命令复盘日志 | 按顺序保存命令复盘记录 |
| RandomReplayEntry | 随机复盘条目 | 记录一次随机种子或随机抽取 |
| RandomReplayLog | 随机复盘日志 | 保存一次执行中的随机信息 |
| ReplayInput | 重放输入 | 重放所需的初始状态、命令日志、随机日志 |
| ReplayResult | 重放结果 | 重放后的结果和对比报告 |
| ReplayRunner | 重放运行器 | 使用 RulePipeline 重新执行命令并比较结果 |
| ReplayCompareReport | 重放对比报告 | 判断原执行和重放执行是否一致 |
| ProtocolEnvelope | 协议外壳 | 所有未来前后端消息的统一包装 |
| ProtocolMessageType | 协议消息类型 | request / response / event / error 等 |
| ProtocolDomain | 协议域 | command / query / event_stream / replay 等 |
| ProtocolPayload | 协议载荷 | P4 最小字符串载荷，不做复杂 JSON |
| ProtocolError | 协议错误 | 对外错误摘要，不暴露内部对象 |
| ProtocolWarning | 协议警告 | 非致命提醒 |

## 22. P4 不通过条件

```text
复盘绕过 RulePipeline。

P4 实现 H5 页面。
P4 实现 AgentRuntime。
P4 实现完整 SaveManager 或真实文件格式。

P4 破坏 P3 unknown fruit eat 测试。
```

```text
只做复盘记录、随机记录、最小重放和协议外壳；不做新玩法，不做网络，不做前端，不做完整存档。
```

## 24. P4 后续方向

P4 通过后，下一阶段建议进入 P5：

```text
Agent 最小定义与观察。
小型 Agent 可把观察转成 CommandEnvelope。
狼、蜘蛛、人类、外星人都能作为不同规模 Agent 的后续扩展。
```

但 P5 开始前必须确认：

```text
P4 的 replay 可以给 Agent 行为提供训练轨迹。
P4 的 protocol envelope 可以给 H5 和调试工具提供稳定输出。
P4 没有把 Agent 或前端逻辑提前塞进 replay/protocol。
```
