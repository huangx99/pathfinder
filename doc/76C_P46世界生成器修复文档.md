# P46 世界生成器修复文档

## 1. 文档目的

本文档不是实现代码，而是 P46 开发和审核时使用的“修复路线图”。

它把 `doc/76A_P46世界生成器后续扩展风险头脑风暴.md` 中发现的问题，整理成具体修复策略、禁止实现、代码落点和验收方式。

如果 P46 实现后审核发现问题，应优先按本文档判断：

```text
这是必须马上修的 P0，还是可以进入后续阶段的 P1/P2？
应该在哪个模块修？
修复后要补哪条测试？
```

## 2. 修复总原则

P46 的修复不能靠“再写一个 if 特判”。

所有修复必须优先落到以下抽象层：

```text
WorldGenerationConfigValidator：修配置错误。
WorldGenerationRng：修确定性错误。
WorldGenerationPlanner：修分布/保障策略错误。
WorldGenerationApplier：修落地和归属错误。
WorldCommand Handler：修命令输入输出错误。
Projection Builder：修前端可见结果错误。
```

特别门禁：P46/P47 生成或采集物品时，必须通过 P45 的归属接口建立 `OnMap` 或 `InInventory` 状态，不能绕过 runtime 直接改数组。

## 3. P0 修复项

### 3.1 修复：生成物品绕过 P45 归属

触发条件：

```text
P46 直接修改 inventory entries。
P46 直接写 item_locations_。
P46 生成的 ground item 只有 world entity，没有 P45 可识别的 OnMap 归属。
P46/P47 采集产出直接 push 到背包。
```

修复策略：

```text
1. 所有生成结果先形成 GeneratedEntityDraft。
2. WorldGenerationApplier 统一把 draft 应用到 WorldRuntime。
3. 对 ground item，applier 必须调用 P45 可识别的归属接口建立 OnMap。
4. P46 禁止创建玩家背包物品。
5. P47 采集产出必须调用 P45 transfer/spawn 接口建立 InInventory 或 OnMap。
```

建议新增接口或适配层：

```cpp
class IGeneratedItemPlacementPort {
public:
    virtual Result<void> placeGroundItem(const GeneratedEntityDraft& draft) = 0;
    virtual Result<void> registerOnMapOwnership(const std::string& entity_id,
                                                const WorldCellCoord& coord) = 0;
};
```

说明：

```text
接口名可以调整，但职责不能丢：生成器不能自己知道 P45 内部结构，只能通过 port/applier 建立归属。
```

必须补测：

```text
world_generation_ground_items_are_on_map_only
world_generation_does_not_create_player_inventory_items
architecture_world_generation_no_direct_inventory_mutation
```

### 3.2 修复：资源节点和地面物品混淆

触发条件：

```text
树、矿脉、浆果丛可以被 pickup。
resource node 没有 charges/action/output。
地面掉落物被当成 resource node 采集。
```

修复策略：

```text
1. 明确 GeneratedObjectKind。
2. ResourceNodeDraft 和 GroundItemDraft 分开。
3. PickupFromMap 只处理 GroundItem。
4. P47 Gather/Chop/Mine 只处理 ResourceNode。
```

建议枚举：

```cpp
enum class GeneratedObjectKind {
    Unknown,
    TerrainCell,
    ResourceNode,
    GroundItem,
    SpawnPoint,
    DangerZone,
    BiomeMarker
};
```

必须补测：

```text
world_generation_resource_node_not_ground_item
world_generation_resource_node_has_action_and_output_keys
```

### 3.3 修复：重复生成覆盖玩家改动

触发条件：

```text
同一区域第二次 GenerateRegion 后，已经被砍掉的树又出现。
已经捡走的地面物品又出现。
玩家建造的对象被覆盖。
```

修复策略：

```text
1. WorldRuntime 记录 generated_region_manifest。
2. GenerateRegion apply 前检查 region 是否已生成。
3. P46 默认 RegionAlreadyGenerated。
4. 不提供 force_regenerate 给普通玩法命令。
5. 调试 force regenerate 必须走 debug command，并写 trace。
```

必须补测：

```text
world_command_generate_region_repeated_is_idempotent_or_blocked
world_generation_repeated_region_does_not_duplicate_entities
```

### 3.4 修复：确定性不稳定

触发条件：

```text
同 seed 每次生成不同。
不同机器生成结果不同。
测试偶现失败。
```

修复策略：

```text
1. 禁用 random_device/time/rand。
2. 所有随机通过 WorldGenerationRng。
3. makeRegionSeed(world_seed, region_coord, layer_key, generator_version)。
4. 所有候选项排序后再抽样。
5. snapshot hash 使用稳定排序。
```

必须补测：

```text
world_generation_same_seed_same_region_same_result
world_generation_snapshot_hash_stable
```

### 3.5 修复：对象缺少 layer_key

触发条件：

```text
entity/resource/spawn/danger 只有 x/y。
同坐标不同层互相覆盖。
```

修复策略：

```text
1. 所有 coord 类型统一使用 WorldCellCoord。
2. DTO 禁止裸 x/y 作为最终坐标。
3. JSON profile 必须有 default_layer。
4. enabled_layer_keys 为空时用 default_layer，但结果仍必须写入 layer_key。
```

必须补测：

```text
world_generation_creates_cells_with_layer_key
world_generation_same_xy_different_layer_coexist
```

## 4. P1 修复项

### 4.1 修复：出生保障写死具体物品

触发条件：

```text
出生区固定生成 red_berry、wood、stone。
改成蘑菇世界或沙漠世界后最小闭环失效。
```

修复策略：

```text
SpawnSafetyPolicy 只认识 guaranteed_resource_tags。
由 content profile 决定 food_basic 对应红果、蘑菇、鱼、仙人掌果还是营养块。
```

必须补测：

```text
world_generation_spawn_safety_has_basic_food
world_generation_spawn_safety_uses_tags_not_fixed_entity_key
```

### 4.2 修复：内容引用缺少校验

触发条件：

```text
resource rule 引用不存在 action/tool/entity。
生成器静默跳过。
```

修复策略：

```text
WorldGenerationConfigValidator 在生成前一次性验证 profile。
错误返回包含 profile_key、rule_key、missing_key。
```

必须补测：

```text
world_generation_referenced_content_missing_fails
```

### 4.3 修复：投影只返回 ok，不返回变化

触发条件：

```text
GenerateWorld 成功，但前端不知道哪些 cell/entity 变化。
```

修复策略：

```text
CommandResult 必须包含 changed_cells、changed_entities、events、trace。
ProjectionPatch 只读表达生成结果。
```

必须补测：

```text
world_command_generate_region_returns_projection_patch
```

### 4.4 修复：stable id 冲突

触发条件：

```text
不同 region/layer 生成同 id。
同一 resource rule 多个实例 id 重复。
```

修复策略：

```text
stable id = world_id + region_coord + layer_key + object_kind + content_key + local_index/hash。
```

必须补测：

```text
world_generation_many_regions_no_id_collision
```

## 5. P2 修复项

### 5.1 修复：大世界性能不足

短期处理：P46 不做异步，只保证 region_size 有上限。  
后续处理：引入 region cache、异步生成队列、预生成边界。

### 5.2 修复：区域边界割裂

短期处理：P46 可接受单区域体验，但 stable id 和 region seed 必须正确。  
后续处理：加入 neighbor sampling、biome field 连续函数、河流/道路跨区规划。

### 5.3 修复：生态动态不足

短期处理：P46 只生成 danger zone/lair marker。  
后续处理：P52 野兽生态、事件系统、Agent 自主行动改变世界。

## 6. 代码门禁清单

研发实现 P46 时，必须逐条自查：

```text
1. world_generation 模块是否独立？
2. DTO 是否包含 seed/profile/version/region/layer？
3. 所有随机是否来自 deterministic RNG？
4. 生成结果是否先是 draft，再由 applier 落地？
5. GroundItem 是否通过 P45 归属接口建立 OnMap？
6. 是否完全没有给玩家背包塞初始物品？
7. ResourceNode 是否不能 pickup？
8. GenerateRegion 是否防重复 apply？
9. CommandResult 是否返回 changed_cells/changed_entities/events/trace？
10. 前端是否只看 projection，不读 worldgen JSON？
```

## 7. 审核门禁清单

审核 P46 时，必须重点看：

```text
1. 有没有硬编码坐标放红果/木头/石头。
2. 有没有直接改 inventory entries。
3. 有没有直接改 item_locations_。
4. 有没有 random_device/time/rand。
5. 有没有对象缺少 layer_key。
6. 有没有 resource node 被当作普通 item。
7. 有没有重复生成导致实体翻倍。
8. 有没有 command ok 但 projection 没变化。
9. 有没有测试只测数量不测语义。
10. 有没有绕过 P45 建立物品归属。
```

## 8. 推荐修复顺序

如果 P46 审核失败，按以下顺序修：

```text
第一优先级：P45 归属问题。
第二优先级：确定性和 stable id。
第三优先级：ResourceNode/GroundItem 分离。
第四优先级：重复生成与 manifest。
第五优先级：出生安全和配置引用。
第六优先级：projection/trace 完整性。
```

原因：

```text
归属错了，后续所有物品行为都是假的。
确定性错了，复盘和测试都不可信。
节点/物品混淆，P47 采集无法设计干净。
重复生成会破坏玩家改动。
出生安全影响可玩性。
projection/trace 影响前端和调试。
```

## 9. P47 承接修复要求

P47 开始前必须明确：

```text
采集 ResourceNode 后产出物品，不是 P46 生成物品。
产出进玩家背包时，通过 P45 建立 InInventory。
产出掉地上时，通过 P45/WorldRuntime 建立 OnMap。
背包满时，不能丢失物品，必须按 output_location_policy 处理。
采集失败不能修改资源节点 charges 或背包。
```

建议 P47 第一批修复/测试围绕：

```text
world_gather_output_item_created_through_p45_inventory_interface
world_gather_output_item_created_on_map_when_inventory_full
world_gather_does_not_modify_state_on_failure
```

## 10. 结论

P46 可以简化内容规模，但不能简化架构边界。

只要以下规则成立，后续即使扩展到大地图、高级文明、魔法世界、机械危机，也不需要推倒重来：

```text
生成是 draft。
落地走 applier。
物品归属走 P45。
行为改变走 Command。
前端只看 Projection。
复盘依赖 manifest/trace。
```
