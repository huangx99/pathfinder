# P60 正式客户端地图交互与区块保存触发任务卡设计

## 0. 文档状态

阶段编号：P60。

阶段名称：正式客户端地图交互与区块保存触发。

设计目标：在 P53-P56 客户端协议、P57-P58 无限区块生成、P59 区块状态缓存恢复的基础上，建立正式玩家地图交互闭环；玩家可以通过客户端在二维开放世界中查看、移动、选择对象、执行后端授权动作、获得事件与知识反馈，同时玩家移动和视野窗口变化必须触发后端 region 生命周期管理，让离开视野的区块能被安全 seal，回来时优先 restore，而不是由前端假装保存。

本文档遵守：

```text
doc/00_设计文档编写要求.md
doc/研发手册.md
AGENTS.md
```

当前前置结论：

```text
P53-P56 已建立客户端协议、HTTP 网关、真实投影和 available_commands 桥接。
P57 已建立确定性噪声地形与可通行性修复。
P58 已把无限 region ensure 接到移动、投影和客户端命令边界。
P59 已设计区块状态缓存与恢复，但实现复审未通过，存在恢复失败被误判成功等 P0 问题。
```

因此 P60 的实现前置门槛是：

```text
P59 P0 必须先修复并复审通过。
P60 可以先设计，但不能在 P59 恢复安全问题未修复前进入验收实现。
```

一句话：

```text
P60 让玩家真正用客户端玩地图，并让地图交互过程自动驱动后端区块保存/恢复生命周期。
```

---

## 1. 阶段定位

P60 是 V2 开放世界从“后端能力可用”进入“正式玩家交互可用”的阶段。

它不是重新做一个前端 Demo，也不是把规则塞进客户端。P60 的核心是把客户端输入、后端命令、世界投影、事件反馈、知识反馈、区块生命周期连成一个可靠闭环。

P60 之前已经有很多系统：

```text
WorldCommandPipeline：行为入口。
ContentRegistry：内容定义入口。
WorldGridRuntime：世界网格权威。
InventoryRuntime / P45：物品归属权威。
WorldReaction / P48：制作与反应权威。
WorldLearning / P49：经验与知识权威。
Client Protocol / P53-P56：客户端协议与投影桥。
WorldRegionEnsure / P58：按需生成。
WorldRegionState / P59：区块 snapshot/cache/restore。
```

P60 不复制这些系统，只做两件事：

```text
1. 正式地图交互层：让玩家可理解、可选择、可执行、可看到结果。
2. 区块生命周期触发层：让玩家移动和视野变化自动触发 region active/cache/restore/generate。
```

---

## 2. 问题定义

当前系统最大问题不是没有规则，而是玩家还不能稳定地通过正式客户端消费这些规则。

如果 P60 只做界面，会出现：

```text
玩家看见地图但不知道哪些对象能互动。
玩家点击对象但前端自己猜动作，绕过 available_commands。
玩家走远后区块状态没有自动封存，离开再回来可能丢状态或重复生成。
采集、拾取、丢弃、制作、知识反馈散在不同日志里，玩家无法理解“我刚刚学到了什么”。
```

P60 要解决的真正问题是：

```text
玩家每一次地图输入，都能被后端解释为安全、可验证、可回放、可反馈、可保存触发的正式世界行为。
```

---

## 3. 目标

### 3.1 玩家体验目标

P60 完成后，玩家应该能用正式客户端完成以下闭环：

```text
看地图 -> 选择目标 -> 查看对象说明 -> 看到后端授权动作 -> 执行动作 -> 看见世界变化 -> 看见事件反馈 -> 看见经验/知识反馈 -> 移动到新区域 -> 回来仍保留改动。
```

最低可玩体验：

```text
1. 玩家可以点击地块查看地形、资源、物品、NPC/野兽摘要。
2. 玩家可以点击目标地块移动，移动命令必须走 WorldCommandPipeline。
3. 玩家可以点击对象，只看到后端 available_commands 允许的动作。
4. 玩家可以采集资源、拾取地面物品、丢弃物品、对目标使用物品。
5. 客户端能显示动作成功、失败原因、事件、知识形成。
6. 玩家远离区块后，后端能触发区块 seal；回来时优先 restore。
```

### 3.2 工程目标

```text
1. 前端不保存世界状态，不推导规则结果，不绕过 available_commands。
2. 后端提供稳定的 map interaction projection，让客户端不用理解引擎内部结构。
3. 后端提供 region activity window / lifecycle trigger，使移动和刷新时能自动驱动 P59。
4. 所有交互结果都能通过 ClientCommandResponse / patch / event_feed / knowledge feedback 传回客户端。
5. 自动区块保存触发必须可测试，不能只靠人工试玩。
```

---

## 4. 不解决的问题

P60 不解决以下内容，避免范围失控：

```text
1. 不做完整磁盘存档。P60 只接自动 seal/restore 触发；写入 P30 SaveGamePackage / 文件存储后续阶段做。
2. 不重做 P59 的安全修复。P59 P0 必须先修，P60 依赖它。
3. 不新增大型玩法内容包，例如魔法文明、机械危机、高级文明。
4. 不重做战斗系统。
5. 不做复杂 NPC 社会系统。
6. 不把客户端设计成规则引擎。
7. 不让前端直接调用 WorldRuntime、InventoryRuntime、ReactionService、KnowledgeService。
8. 不把地图拖拽、缩放、动画表现作为后端验收核心；表现可以并行，但规则验收看协议和自动化测试。
```

## 5. 头脑风暴：复杂场景会怎样打穿设计

### 5.1 采集后离开再回来

场景：玩家采掉一棵树，走到远处，回来。

要求：树不能重新满血或复活。P60 必须在玩家远离时触发 P59 seal，回来时 ensureAvailable 走 restore。

### 5.2 拾取地上物品后离开

场景：玩家捡走地上的石片，离开区块再回来。

要求：石片不能从旧 snapshot 复制回地图。P60 自动保存触发必须尊重 P45 归属，snapshot 只保存 OnMap。

### 5.3 丢弃物品到另一块区域

场景：玩家在 A 区捡起物品，到 B 区丢下，离开再回来。

要求：A 不复活旧物品，B 保留丢弃物。P60 的 region lifecycle 不能只看玩家当前 region，要考虑物品新归属所在 region。

### 5.4 长距离点击移动

场景：玩家点击远方地块，路径跨越多个 region。

要求：路径规划或逐步移动必须在每步前 ensure 目标邻域；新 region 生成或旧 region 恢复失败时，移动应阻断并给原因。

### 5.5 玩家快速往返边界

场景：玩家在 region 边界来回走。

要求：不能每一步都 seal/detach/restore 抖动。P60 需要 active window 和 hysteresis 策略：离开安全窗口若干 tick 或距离后再 seal。

### 5.6 野兽/NPC 跨区块活动

场景：野兽追玩家跨 region，NPC 在远处采集。

要求：含有活跃 Actor、目标执行中 Agent、危险事件的 region 不能随便 detach。P60 的 lifecycle trigger 必须调用 unload guard。

### 5.7 客户端刷新或断线重连

场景：浏览器刷新，session 仍存在。

要求：客户端 refresh 只能拿后端 projection，不能靠本地缓存恢复世界；region ensure/restore 仍由后端执行。

### 5.8 命令过期

场景：玩家点了对象动作，但这期间 NPC 把对象拿走，或区块被封存。

要求：command 执行前必须重新校验 projection_version / option_id / target state；失败要返回明确原因。

### 5.9 移动到未探索区域

场景：玩家向未知方向走。

要求：后端可以生成新区块，但客户端不能自己造地图；P58 ensure 后投影返回新区域。

### 5.10 多层地图

场景：地表、地下、空中单位同时存在。

要求：P60 不做三维，但必须保留 layer_key / virtual z 概念。客户端地图交互请求和投影都必须带 layer。

### 5.11 后续远程服务器游玩

场景：以后游戏部署到服务器，玩家远程访问。

要求：P60 不能依赖本地内存前端假状态；协议要幂等、可重试、有 request_id、session_id、projection_version。

### 5.12 地图表现很复杂

场景：后续前端要做拖拽、缩放、像素动画、路径粒子。

要求：表现层可以自由做，但它只能消费 projection/patch/event，不得成为规则权威。

---

## 6. 核心方案

P60 采用“三层闭环”：

```text
客户端交互闭环：
输入 -> 后端可用命令 -> 执行命令 -> patch / event / feedback -> 前端展示

地图对象闭环：
projection visible cell/entity/resource -> select target -> available command options -> command materialize -> WorldCommandPipeline

区块生命周期闭环：
actor position / viewport window -> desired active regions -> seal candidates -> unload guard -> P59 seal -> detach -> ensure restore/generate
```

三个闭环必须统一在后端协议里，不允许前端自己拼规则。

## 7. 系统拆解

### 7.1 ClientMapProjectionAdapter

作用：给正式地图客户端提供“可展示但安全”的地图投影。

输入：

```text
session_id
actor_key from session
layer_key
center_coord / viewport_rect
projection_version
```

输出：

```text
visible_cells
visible_entities
visible_resource_nodes
visible_ground_items
actor_summaries
selection_hints
available_command_refs
event_feed_delta
knowledge_feedback_delta
region_lifecycle_summary
```

边界：

```text
不能泄露 hidden truth。
不能返回完整 runtime 私有状态。
不能给前端返回可伪造的规则结果。
```

### 7.2 ClientMapSelectionService

作用：把玩家点击的地图位置或对象 id 转成后端可理解的 selection context。

输入：

```text
cell_coord
entity_id 可选
resource_node_id 可选
inventory_item_id 可选
layer_key
client projection_version
```

输出：

```text
selection_id
selected_cell summary
selected_objects summary
allowed target kinds
available command options
failure reason if selection stale
```

规则：

```text
选中只产生上下文，不产生世界变化。
选中不直接执行动作。
选中后动作仍必须来自 available_commands。
```

### 7.3 MapAvailableCommandAdapter

作用：为当前选中目标生成后端授权动作。

动作类型第一版包括：

```text
MoveToCell / StepMove
InspectCell
InspectEntity
InspectResourceNode
HarvestResource
PickupGroundItem
DropInventoryItem
UseItemOnTarget
CraftFromKnownRecipe
TeachNearbyActor
Wait
```

注意：这些是客户端可选项，不代表一定新增所有 WorldCommandKind。如果已有命令能表达，必须复用已有命令；只有表达不了时才扩展 Command。

输出必须包含：

```text
option_id
command_key
label_zh
short_hint_zh
target_summary
cost_preview 可选
risk_preview 可选
disabled_reason 可选
```

前端只能提交 option_id 或受限 DTO，不能自己生成隐藏 command。

### 7.4 MapCommandGateway

作用：接收正式地图客户端提交的动作，并执行。

输入：

```text
session_id
request_id
projection_version
option_id 或安全 command dto
selection_id 可选
```

流程：

```text
1. 从 session 取 actor_key，不信任客户端 actor。
2. 校验 projection_version。
3. 校验 option_id 来自当前后端快照。
4. materialize WorldCommandDto。
5. 执行前 ensure 目标 region / 邻域。
6. 走 WorldCommandPipeline。
7. 收集 state_delta / event_feed / experience / knowledge_feedback。
8. 调用 RegionActivityWindowService 更新 active window。
9. 生成 projection_patch。
```

输出：

```text
command_status
failure_reason_keys
event_feed
knowledge_feedback
projection_patch
requires_full_refresh
region_lifecycle_events
```

### 7.5 RegionActivityWindowService

作用：根据玩家位置、视野、路径、NPC/危险活动，计算哪些 region 应保持 active，哪些可以 seal/detach。

输入：

```text
actor_coord
viewport_rect
planned_path 可选
vision_radius
active_actor_regions
danger_regions
recently_touched_regions
```

输出：

```text
keep_active_region_keys
prewarm_region_keys
seal_candidate_region_keys
detach_candidate_region_keys
blocked_unload_region_keys
```

第一版策略：

```text
1. 玩家所在 region 必须 active。
2. 玩家视野覆盖 region 必须 active。
3. 玩家周围 preload_margin region 保持 active。
4. 最近 N tick 交互过的 region 延迟 seal，避免边界抖动。
5. 有玩家、同伴、野兽、正在执行目标的 Agent 的 region 不 detach。
6. 有未处理危险事件的 region 不 detach。
7. 超出 active window 且通过 unload guard 的 region 才进入 seal candidate。
```

### 7.6 RegionLifecycleTriggerService

作用：执行自动保存触发，但不自己保存状态。

输入：

```text
seal_candidate_region_keys
detach_candidate_region_keys
WorldRegionSnapshotBuildContext
```

流程：

```text
1. 对 candidate 调用 P59 sealRegion。
2. seal 成功后，如果策略允许 detach，调用 detach port。
3. 记录 lifecycle event。
4. seal/restore/generate 失败必须进入 warning/failure，不允许静默忽略。
```

输出：

```text
sealed_regions
detached_regions
restore_failed_regions
blocked_regions
warning_keys
failure_reason_keys
```

关键边界：

```text
P60 不直接写 snapshot。
P60 不直接复制 region runtime。
P60 不直接清空 runtime。
P60 只触发 P59/P58 的正式服务。
```

### 7.7 MapPathIntentService

作用：处理“点击远方地块移动”的交互意图。

第一版建议采用后端轻量路径规划：

```text
1. 客户端提交 target cell selection。
2. 后端基于当前 projection / ensured region 计算可达路径。
3. 返回 path_preview 和下一步 Move option。
4. 执行时仍以 step command 或 MoveToCell command 进入 WorldCommandPipeline。
5. 每一步后重新校验，因为世界可能被 NPC/野兽改变。
```

不能让前端自己直接连续改坐标。

### 7.8 EventAndKnowledgeFeedbackComposer

作用：把命令结果转成玩家能读懂的反馈。

输出层级：

```text
一级：发生了什么。
二级：为什么成功/失败。
三级：世界有什么变化。
四级：你学到了什么。
五级：同伴/NPC/野兽有什么反应。
```

示例：

```text
你砍伐了年轻树木。
斧头变钝了一点。
你获得了木头。
你记录到：斧头可以处理树木并获得木材。
远处的野兽听到声音，开始靠近。
```

## 8. 区块生命周期详细设计

### 8.1 活动窗口定义

P60 引入后端概念：`RegionActivityWindow`。

字段建议：

```text
world_id
layer_key
center_actor_key
center_coord
viewport_rect
vision_radius
preload_margin_regions
keep_active_regions
prewarm_regions
seal_candidates
detach_candidates
blocked_regions
reason_keys
computed_tick
```

### 8.2 触发时机

必须触发 activity window 更新的情况：

```text
1. bootstrap 后。
2. refresh 请求中 viewport 改变。
3. 玩家移动成功后。
4. path step 成功后。
5. teleport / layer change 后。
6. 采集、拾取、丢弃、制作等改变 OnMap / InInventory / InContainer / Equipped / Consumed / Destroyed / Nowhere 归属状态后。
7. Agent/NPC/野兽跨 region 后。
8. wait/tick 推进后。
```

归属红线：

```text
P60 生成或采集物品时，必须通过 runtime 权威接口与 P45 归属接口建立 OnMap 或 InInventory 等状态。
不得绕过 runtime 直接改 cells_、entities_、resource_nodes_、inventory entries 或临时数组。
如果某个交互缺少权威接口，P60 必须先补通用 runtime / location / inventory port，再实现该交互。
```

### 8.3 seal 策略

第一版不追求极致省内存，优先安全：

```text
active_radius = 玩家视野覆盖 region + 1圈 region margin
seal_delay_ticks = 3 到 10 tick，可配置
detach_delay_ticks = seal 成功后再延迟 1 到 3 tick，可配置
```

区块必须满足以下条件才可 seal：

```text
1. 不在 keep_active_regions。
2. 不包含玩家。
3. 不包含正在执行目标的关键 Agent。
4. 不包含 unresolved danger event。
5. unload_guard 通过。
6. P59 snapshot validation 通过。
```

### 8.4 restore/generate 顺序

进入某 region 前必须统一走：

```text
1. ActiveRuntime -> 直接可用。
2. CachedSnapshot -> restore。
3. NeverGenerated -> P58 generate。
4. RestoreFailed -> BlockedFailure，不允许偷偷 generate 覆盖。
```

这条顺序是 P60 验收红线。

### 8.5 与磁盘存档边界

P60 的自动保存不是磁盘保存。

P60 保存触发含义：

```text
active runtime region -> P59 in-memory snapshot / cache
```

不包含：

```text
写 SaveGamePackage 文件
云存档
跨进程恢复
版本迁移
```

这些后续接 P30 / 正式存档阶段。

## 9. 客户端交互设计

### 9.1 信息层级

正式客户端至少分四层信息：

```text
地图主视图：地形、玩家、NPC、资源、物品、危险。
选中面板：当前选中地块/对象的中文说明和状态。
动作面板：后端 available_commands 返回的动作按钮。
反馈时间线：事件、失败原因、知识形成、区块状态变化。
```

### 9.2 地图点击规则

```text
单击地块：选中并请求 selection projection。
双击地块或点击“前往”：提交 MoveToCell / path intent。
点击对象：选中对象，动作面板刷新为目标动作。
点击空白：清除目标或选中地块。
```

### 9.3 动作按钮规则

动作按钮必须来自后端：

```text
可执行：按钮高亮，显示中文动作。
不可执行但有原因：灰色，显示原因，例如“太远”“缺少工具”“不知道怎么做”。
不该知道的动作：不返回，前端不显示。
```

### 9.4 反馈顺序

每次命令后，客户端展示顺序必须稳定：

```text
1. 输入确认：正在执行什么。
2. 行为结果：成功/失败。
3. 世界变化：资源减少、物品获得、对象移动。
4. 知识变化：学到、修正、强化、教学成功。
5. 区块变化：生成、恢复、封存、卸载，只作为辅助日志，不抢主焦点。
```

### 9.5 手机端原则

P60 不要求完成最终美术，但交互必须适合手机：

```text
1. 主地图占屏幕大部分。
2. 选中信息和动作按钮靠近底部或侧边抽屉。
3. 动作按钮不超过当前上下文的主要动作，更多放展开。
4. 禁止把调试日志当主界面。
5. 失败原因必须中文且靠近按钮或反馈区。
```

## 10. Command 与协议设计

### 10.1 继续复用 P53/P54/P56

P60 不新建平行协议。仍使用：

```text
bootstrap
refresh
command
reset
projection patch
available_commands
```

必要时扩展 DTO，但不能让前端绕过现有 gateway。

### 10.2 推荐新增/扩展 DTO

```text
ClientMapInteractionContext
ClientMapSelectionRequest
ClientMapSelectionResponse
ClientMapActionOptionDto
ClientMapPathPreviewDto
ClientRegionLifecycleHintDto
ClientRegionLifecycleEventDto
ClientKnowledgeFeedbackDto
```

这些 DTO 都是客户端安全视图，不是 runtime 原始状态。

### 10.3 Command materialize 原则

```text
option_id -> 后端 snapshot 查找 -> materialize WorldCommandDto -> WorldCommandPipeline
```

禁止：

```text
前端传 actor_key 决定操作者。
前端传 hidden target state。
前端传 “我采集成功了”。
前端直接传资源产物或知识结果。
```

## 12. 实现任务拆分

### P60-A：P59 安全门禁复查

目标：P60 实现前必须确认 P59 P0 已修复。

验收：

```text
1. P59 复审通过。
2. Restorer 不再把 RuntimeConflict 改成 Restored。
3. applyRestore 写入失败会阻断。
4. harvest_state_preservation 真正 detach 目标 region。
5. p45_pickup_drop_boundary 已注册 CTest。
6. snapshot DTO 校验补齐。
```

### P60-B：RegionActivityWindowService

目标：后端能根据 actor、viewport、路径、危险和 Agent 活动计算 active/prewarm/seal/detach region 集合。

验收：

```text
1. 玩家所在 region 永远 keep active。
2. 视野和 preload margin 内 region keep active。
3. 边界反复移动不会频繁 seal/detach 抖动。
4. 有活跃 Agent / danger 的 region 不 detach。
```

### P60-C：RegionLifecycleTriggerService

目标：把 activity window 的 candidate 接到 P59 seal/restore/detach。

验收：

```text
1. 离开 active window 的 region 会触发 sealRegion。
2. seal 成功后可按策略 detach。
3. 回到 cached region 会 restore，不会 generate。
4. restore failed 会阻断，不会偷偷 generate。
5. lifecycle events 会返回给客户端。
```

### P60-D：ClientMapSelection 与可用动作

目标：点击地块/对象后，后端返回安全 selection 和 available map actions。

验收：

```text
1. 选中地块只返回安全摘要。
2. 动作按钮全部来自后端 option。
3. stale option_id 被拒绝。
4. 不可用动作有中文原因。
```

### P60-E：地图移动与路径意图

目标：正式客户端能点击地块移动，后端负责路径/逐步命令执行边界。

验收：

```text
1. 移动必须走 WorldCommandPipeline。
2. 跨 region 移动前 ensure 目标邻域。
3. restore/generate 失败时移动阻断并返回原因。
4. 每一步移动后 projection_patch 正确。
```

### P60-F：基础对象交互闭环

目标：采集、拾取、丢弃、使用到目标能从地图交互进入正式规则链。

验收：

```text
1. 采集资源后 charge/depleted 正确，事件和知识反馈可见。
2. 拾取后物品进入背包，地图不再显示该物品。
3. 丢弃后物品出现在目标 region，离开回来仍存在。
4. 使用物品到目标必须通过 target command。
```

### P60-G：反馈合成与客户端安全投影

目标：事件、知识、区块生命周期反馈统一返回。

验收：

```text
1. 成功/失败/知识/区块变化都有中文反馈。
2. projection patch 不泄露 hidden truth。
3. 前端无需读后端内部结构就能展示。
```

## 13. 测试设计

### 13.1 单元测试

```text
RegionActivityWindow 计算 active/prewarm/seal/detach。

```

### 13.2 集成测试

```text
bootstrap -> projection -> select cell -> available move -> command -> patch。
select resource -> harvest -> event -> experience -> knowledge feedback。
pickup ground item -> leave region -> seal/detach -> return -> item 不复活。
drop item in another region -> leave/return -> 目标 region 保留物品。
move across generated boundary -> P58 generate -> projection 更新。
move back to cached region -> P59 restore -> 不 generate。
restore failed -> command blocked -> 返回中文失败原因。
stale option_id -> blocked。
stale projection_version -> requires_full_refresh。
```


## 14. 风险与控制

### 风险 1：P59 未修复导致 P60 保存触发放大错误

控制：P60-A 必须作为实现前门禁，P59 不通过不得验收 P60。

### 风险 2：客户端又变成规则层

控制：所有动作来自 available_commands，前端只能提交 option_id。

### 风险 3：自动 seal/detach 太激进导致玩家附近对象消失

控制：第一版 active window 保守，seal_delay/detach_delay 可配置，有 Agent/danger 时不卸载。

### 风险 4：路径移动跨区块导致状态不同步

控制：每一步移动前 ensure，执行后 patch，失败立即停下。

### 风险 5：反馈过多变成调试台

控制：反馈分层，区块生命周期日志默认低优先级，不抢主事件。

### 风险 6：后续远程服务器兼容差

控制：请求使用 request_id/session/projection_version，服务端权威，不依赖本地浏览器状态。

## 18. 最终验收标准

P60 可以验收的标准：

```text
1. P59 复审通过。
2. 客户端地图交互只走 P53/P54/P56 协议和 available_commands。
3. 玩家能通过正式客户端完成移动、查看、采集、拾取、丢弃、使用到目标。
4. 命令结果能返回事件、知识反馈、projection patch。
5. 玩家移动/视野变化能触发 region activity window。
6. 离开区块能自动 seal，回来优先 restore。
7. restore failed 不会偷偷 generate。
8. 自动化测试覆盖采集/拾取/丢弃/跨区块/restore/generate/stale command。
9. 自动化测试必须证明生成/采集/掉落/制作/拾取产物通过 runtime / P45 权威接口建立归属。
10. 架构扫描确认没有前端规则、没有平行管线、没有 runtime 数组直改、没有绕过归属接口。
```

最终结论：P60 不是“做个更好看的地图界面”，而是把开放世界从“后端能跑”推进到“玩家能正式交互，世界能自动记住变化”的阶段。
