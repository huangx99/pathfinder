# P65 安全圈营地与虚拟外界任务系统设计

## 1. 阶段定位

P65 是项目玩法方向重定向阶段，也是后端运行模型从“大世界网格开放探索”切换到“小地图安全圈营地生存”的设计阶段。

本阶段不是简单改 UI，也不是继续扩展无限地图，而是重新定义本项目的最小可玩形态：

```text
一个小营地
-> 玩家和 NPC 在安全圈内生活
-> 圈外事件以虚拟方式刷新
-> 玩家通过学习、教学、命令和资源分配维持营地生存
-> NPC 任务用节点进度条可视化
-> 背包、知识、制作、采集、威胁结果仍由后端真实系统结算
```

P65 接在以下阶段之后：

```text
P45 背包容器与物品归属
P47 资源采集砍伐挖掘
P48 制作与世界反应
P49 世界经验学习
P50 教学与无知 NPC
P53 客户端协议
P56 客户端 runtime bridge
P62 统一内容物品实例化
P63 生命状态与野生 Agent 威胁闭环
P64 运行时玩法逻辑模块化拆分
```

P65 为后续阶段打基础：

```text
P66 删除/旁路大地图运行入口，建立安全圈 runtime host
P67 NPC 任务链与进度条投影实现
P68 圈外事件刷新与威胁压力实现
P69 流浪学者与外来知识实现
P70 第一夜营地 MVP
```

本阶段是玩法系统 + 架构重定向设计，不直接实现代码。

## 2. 为什么现在做

继续做大世界开放生存的风险已经超过收益：

1. 无限地图需要地图生成、区块保存、寻路、迷雾、加载、性能、资源分布、前端镜头和大量美术。
2. 单人开发很难把“大世界”做到既稳定又好玩。
3. 当前项目最有价值的不是地图大小，而是知识、教学、NPC 自主行动和真实背包。
4. 大地图会把精力消耗在移动和显示上，反而掩盖“NPC 因为被教会而行动”的核心卖点。
5. 用户希望转向生存类安全圈：小地图，圈外虚拟刷新怪物和学者，NPC 任务用进度条表达。

所以 P65 必须明确：

```text
地图不再是核心玩法。
安全圈、任务链、知识传播和营地生存才是核心玩法。
```

## 3. 阶段目标

P65 完成后，研发者必须能只看本文档就知道：

- 新游戏不再围绕无限大地图。
- 圈内小地图只展示营地对象和 NPC。
- 圈外资源、怪物、学者、事件可以虚拟化。
- NPC 不需要真实走完整大地图路径，但任务过程必须可视化。
- NPC 背包、玩家背包、营地仓库、知识、制作、采集、战斗/驱赶结果必须真实。
- 后端仍然是权威，前端只展示任务进度、状态、事件和可用命令。
- 大地图相关代码不能一刀切删除；必须先旁路 runtime host，再删前端依赖，再删测试，再删后端模块。
- 旧的 NPC 跟随和狼 tick 原型不适合新玩法，应改为任务系统和圈外威胁系统。

## 4. 本阶段不解决的问题

P65 只做设计，不做以下内容：

- 不直接删除代码。
- 不直接实现安全圈 runtime。
- 不直接重做 Axmol 前端。
- 不实现外星文明。
- 不实现魔法世界。
- 不实现完整多营地。
- 不实现多人在线。
- 不实现复杂建筑系统。
- 不实现 RimWorld 级别任务调度。

这些内容需要后续阶段逐步实现。

## 5. 核心玩法定义

游戏类型：

```text
小地图营地生存 + NPC 知识任务模拟
```

一句话：

```text
在一个安全圈营地里，玩家通过学习、教学和分配任务，让无知的 NPC 学会采集、制作、驱赶怪物，并在不断刷新的圈外事件中活下去。
```

核心循环：

```text
观察营地和圈外事件
-> 学习或获取新知识
-> 教 NPC
-> 给 NPC 下任务
-> NPC 任务进度条推进
-> 真实背包/知识/仓库/威胁状态变化
-> 玩家根据结果调整下一轮策略
```

第一版目标：

```text
活过第一夜。
```

第一夜闭环：

```text
圈外刷新狼
圈外刷新流浪学者
玩家从学者学会火把驱狼
玩家教 NPC
玩家命令 NPC 制作火把
玩家命令 NPC 驱赶狼
NPC 任务进度条推进
狼被驱散
营地安全度上升或夜晚存活
```

## 6. 头脑风暴：为什么这个方向成立

### 6.1 它保留项目独特性

当前项目最独特的是：

```text
知识不是菜单技能，而是来自经验或教学。
NPC 不只是按钮执行者，而是根据知识和背包推进任务。
玩家的核心行为不是亲自跑遍世界，而是让群体变聪明。
```

安全圈玩法正好放大这些优势。

### 6.2 它降低单人开发成本

大世界需要大量工程和美术成本。

安全圈只需要：

```text
一个营地画面
几个 NPC
几个圈外事件图标
几个任务进度条
几个真实背包面板
```

地图变小后，玩家注意力会回到 NPC 行为链。

### 6.3 它适配进度条 UI

NPC 任务链天然适合节点进度条：

```text
接令 -> 检查知识 -> 检查材料 -> 采集/制作 -> 处理危险 -> 返回/交付
```

玩家点击 NPC 时看到任务链，不点击时看头顶短状态。

### 6.4 它适合未来扩展外星文明和魔法世界

安全圈不是原始营地专属。

它可以抽象成：

```text
安全域 SafeZone
边界事件 BoundaryEvent
任务链 TaskPlan
知识源 KnowledgeSource
威胁压力 ThreatPressure
资源机会 ResourceOpportunity
```

未来外星文明可以是：

```text
殖民地安全罩
圈外异星虫群
流浪科学家 / 外星遗迹
能源核心 / 维修机器人 / 生物污染
```

未来魔法世界可以是：

```text
魔法结界
圈外魔物潮
流浪法师
法阵、魔力、诅咒、炼金材料
```

核心模型不变，只换内容包和模块。

## 7. 新世界模型

### 7.1 安全圈 SafeZone

安全圈是游戏主舞台。

数据：

```text
safe_zone_id
name_key
safety_score
firelight_score
defense_score
noise_score
food_days
morale_score
current_phase
active_boundary_events[]
resident_actor_keys[]
storage_inventory_id
```

解释：

- `safety_score`：营地总体安全。
- `firelight_score`：火光/照明对夜晚和怪物的影响。
- `defense_score`：围栏、陷阱、守夜等防御能力。
- `noise_score`：噪音高会提高怪物刷新概率。
- `food_days`：食物储备，用于提示玩家压力。
- `current_phase`：白天/傍晚/夜晚/清晨。

### 7.2 圈内对象

圈内是真实但小规模的对象：

```text
玩家
NPC
火堆
仓库
工作台/制作点
床/休息点
少量样本资源
```

这些对象可以有坐标，但不需要大世界网格。

### 7.3 圈外虚拟世界

圈外不需要完整地图，只需要事件槽位：

```text
north_edge
south_edge
east_edge
west_edge
far_woods
river_path
old_road
ruin_hint
```

事件可以挂在边界槽位上。

示例：

```text
北侧树林：狼群正在靠近
南侧道路：流浪学者停留 2 回合
东侧坡地：发现木材机会
西侧废墟：有未知金属碎片但风险较高
```

## 8. 圈外事件系统

### 8.1 事件类型

第一版事件：

```text
ThreatEvent：狼、野兽、魔物、外星虫群
ScholarEvent：流浪学者、法师、外星研究员
ResourceEvent：木头、石头、食物、草药、矿物
WeatherEvent：雨、寒冷、雾、沙尘
OpportunityEvent：遗迹、陌生人、商队、失物
```

### 8.2 事件状态

事件不是瞬间结算，而是阶段推进：

```text
spawned
noticed
approaching
active
resolved
expired
```

威胁事件可有更具体阶段：

```text
远处出现 -> 靠近观察 -> 试探边界 -> 攻击边缘 -> 突入安全圈
```

### 8.3 事件倒计时

每个事件有倒计时：

```text
remaining_ticks
phase_deadline_ticks
```

玩家可以在事件到期前派 NPC 处理。

### 8.4 事件投影

前端显示：

```text
北侧：狼群试探营地，2 回合后突入
南侧：流浪学者停留，1 回合后离开
东侧：木材机会，风险低，耗时 2
```

前端不计算事件结果，只显示后端投影。

## 9. NPC 虚拟任务系统

### 9.1 任务核心

NPC 任务不再依赖长距离寻路。

任务推进方式：

```text
任务节点 -> 时间推进 -> 节点完成或卡住 -> 调用 Command/权威服务结算 -> 更新背包/知识/事件
```

### 9.2 任务结构

```text
NpcTaskRuntime
- task_id
- actor_key
- task_kind
- title_key
- status
- priority
- current_step_index
- steps[]
- goal_payload
- source_command_id
- created_tick
- updated_tick
```

状态：

```text
pending
active
blocked
interrupted
completed
failed
cancelled
```

### 9.3 任务节点结构

```text
TaskStep
- step_id
- step_kind
- label_key
- status
- headline_key
- required_knowledge[]
- required_items[]
- produced_items[]
- consumed_items[]
- target_event_id
- target_inventory_id
- target_actor_key
- remaining_ticks
- blocking_reason_key
- inserted_by
```

节点状态：

```text
pending
active
done
blocked
failed
skipped
interrupted
```

### 9.4 第一版任务类型

第一版只做：

```text
collect_resource
craft_item
repel_threat
learn_from_scholar
teach_actor
transfer_item
```

暂不做：

```text
复杂建筑
多 NPC 协作
自动职业排班
多任务队列
贸易经济
```

## 10. NPC 进度条 UI 投影

### 10.1 地图头顶状态

未选中 NPC 时只显示短状态：

```text
采集木头
制作火把
驱赶狼
向学者学习
缺少知识
缺少材料
受伤返回
等待指令
```

### 10.2 点击 NPC 展开面板

面板显示：

```text
NPC 名字
当前任务
头顶状态
背包摘要
知识摘要
任务节点进度条
当前节点详情
可用操作按钮
```

示例：

```text
阿木
当前任务：驱赶北侧狼群
状态：正在拿火把

[接令]──[检查知识]──[拿火把]──[前往边界]──[驱赶]──[返回]
                         ↑
                       阿木

当前节点：拿火把
需要：火把 x1
来源：营地仓库
```

### 10.3 卡住时显示原因

示例：

```text
任务：制作火把
当前节点：检查知识
状态：卡住
原因：阿木不知道如何制作火把
解决：教他火把制作知识，或让玩家自己制作
```

### 10.4 动态插入节点

任务过程中可以插入节点：

```text
采集木头 -> 遇到狼 -> 插入处理威胁
制作斧头 -> 缺木头 -> 插入采集木头
砍树 -> 斧头钝了 -> 插入打磨斧头
外出学习 -> NPC 饿了 -> 插入吃东西/返回
```

前端必须支持节点数组变化。

## 11. 真实背包与虚拟行动边界

这是本设计最重要的规则。

虚拟化的是路径，不是结果。

允许虚拟化：

```text
NPC 在圈外具体走了哪几格
采集点具体坐标
学者从哪里来
狼从哪棵树后出现
```

不允许虚拟化：

```text
NPC 背包
玩家背包
营地仓库
物品数量
工具耐久
知识状态
教学结果
制作结果
威胁是否解决
NPC 受伤/饥饿/死亡
```

所以 NPC 采集木头应是：

```text
任务节点完成
-> 后端调用采集/产物结算或专用虚拟采集 Command
-> 木头进入 NPC 背包或营地仓库
-> projection 显示变化
```

不能是：

```text
前端计时结束后直接 +2 木头
```

## 12. Command 管线设计

P65 后续实现必须继续走 Command。

新增命令建议：

```text
AssignNpcTask
AdvanceCampTick
ResolveBoundaryEvent
InteractWithScholar
DepositToCampStorage
WithdrawFromCampStorage
```

其中：

- `AssignNpcTask`：玩家给 NPC 下任务。
- `AdvanceCampTick`：推进营地时间。
- `ResolveBoundaryEvent`：处理圈外事件节点结果。
- `InteractWithScholar`：与学者进行知识交换。
- `DepositToCampStorage` / `WithdrawFromCampStorage`：真实仓库转移。

所有命令仍通过：

```text
WorldCommandPipeline
-> handler
-> runtime/service
-> experience
-> knowledge learning
-> projection
```

## 13. 模块结构设计

### 13.1 新后端目录建议

不再铺一堆顶层小目录。

新增安全圈相关小模块放在：

```text
backend/include/pathfinder/camp/
backend/src/camp/
```

如果你希望更保守，也可以放：

```text
backend/include/pathfinder/world_modules/camp/
backend/src/world_modules/camp/
```

我建议本次用 `camp/` 顶层目录，原因：

```text
安全圈营地是新玩法核心领域，不是 P64 那种小型 runtime module。
它会承载 SafeZone、BoundaryEvent、CampTask、CampStorage、CampProjection 等多个子系统。
作为领域系统拥有顶层目录是合理的。
```

## 14. 大地图代码废弃/删除设计

用户希望“大地图部分从头来”。这是正确方向，但不能一刀切删除，否则会破坏当前 Command/Projection/Inventory/Knowledge 的依赖。

### 14.1 大地图相关模块

明显大地图相关：

```text
world_generation
world_map_interaction
world_region_state
world_region_ensure
MoveTargetRegionGuard
ClientMapSelectionService
ClientMapProjectionAdapter
RegionActivityWindowService
RegionLifecycleTriggerService
Axmol 大地图渲染/寻路/镜头拖拽/地图 tile 绘制
```

### 14.2 暂时不能删的模块

这些虽然名字里有 world，但仍是核心规则，不应删除：

```text
world_runtime：当前 Actor/Entity/Command 仍依赖，后续可缩小为 camp runtime adapter
world_inventory：真实背包必须保留
world_harvest：可以保留作为采集结算服务，也可以后续改 virtual harvest adapter
world_reaction：制作必须保留
world_learning：知识学习必须保留
world_teaching：教学必须保留
world_command：Command 管线必须保留
client_protocol：前后端协议必须保留
client_runtime_host：宿主装配入口保留，但要改成 camp host
```

### 14.3 删除顺序

P66 建议顺序：

```text
1. 新增 CampRuntimeHost 或改 ClientRuntimeHostFactory 为 camp 模式。
2. 前端改为 CampScene，不再引用大地图 tile/pathfinding/joystick。
3. 协议 projection 新增 camp_projection，保留旧 world_projection 兼容测试。
4. NPC 任务改为 camp task，不再依赖 move/pathfinding。
5. 威胁改为 BoundaryEvent，不再依赖 wildlife actor tick。
6. 确认 Axmol 和 HTTP server 都能运行 camp MVP。
7. 删除或隔离 world_map_interaction。
8. 删除或隔离 world_region_state。
9. 删除或隔离 world_generation。
10. 删除旧大地图前端文件。
```

### 14.4 不推荐立刻删除的原因

当前仍有大量测试和 host 构造依赖：

```text
ClientRuntimeHostFactory 还在 initializePlayableWorld 中生成 origin region。
Projection bridge 还读 WorldGridRuntime。
Available command 还从 runtime bridge 生成移动/拾取等命令。
Axmol 当前仍依赖 map projection。
```

所以 P65 只设计，P66 再做“旁路大地图入口”，P67/P68 再逐步删。

## 15. NPC 跟随与狼 tick 的处理

### 15.1 NPC 跟随

旧逻辑：

```text
world_modules/follow
```

新玩法不需要常驻跟随。

处理方案：

```text
废弃 FollowUseCommandHandler。
保留概念但改为任务：escort / return_to_player / move_to_camp_slot。
```

删除时机：P66/P67。

### 15.2 狼 tick

旧逻辑：

```text
world_beast_ecology runtime bridge
wildlife actor chase/attack tick
```

新玩法不需要狼在大地图真实寻路追击。

处理方案：

```text
废弃 wildlife actor tick。
狼改成 BoundaryThreatEvent。
阶段推进：远处 -> 试探 -> 攻击边界 -> 突入。
NPC 驱赶狼是任务节点，不是地图追逐。
```

保留可复用部分：

```text
威胁定义
反制策略
伤害/生命状态
风险评分
```

删除/迁移时机：P68。

## 16. 流浪学者系统

### 16.1 学者定位

学者是圈外知识源，不是普通商店。

数据：

```text
scholar_event_id
scholar_key
available_knowledge_templates[]
trade_requirements[]
remaining_ticks
risk_level
```

### 16.2 学知识流程

```text
学者出现
-> 玩家或 NPC 接触
-> 支付食物/护送/交换知识
-> 获得外来知识
-> 知识状态为 hypothesis / taught / hearsay
-> 通过实践可升级为 active
```

### 16.3 未来扩展

原始世界：流浪学者。

魔法世界：流浪法师、魔女、古代石碑。

外星文明：外星研究员、坠毁 AI、遗迹终端。

## 17. 外星文明与魔法世界适配

P65 的抽象必须支持未来世界观。

### 17.1 不写死原始营地

不要把字段命名成：

```text
wolf_only
fire_only
tribe_only
stone_age_only
```

应使用：

```text
ThreatEvent
Countermeasure
KnowledgeSource
SafeZone
Boundary
TaskStep
ResourceOpportunity
```

### 17.2 外星文明映射

```text
SafeZone = 殖民地护盾
CampFire = 能源核心
Wolf = 异星虫群
Torch = 声波驱散器
Scholar = 外星科学家 / 遗迹终端
```

## 22. 补强设计：采集/制作过程中的危险推理与返家逻辑

### 22.1 为什么必须补这一层

第一版如果只有“派 NPC 外出 -> 时间到 -> 获得资源”，玩法会变成纯计时器，没有生存压力。

真正的营地生存闭环必须包含：

```text
派 NPC 外出/制作
-> 世界状态变化
-> 任务过程中出现危险或资源不足
-> NPC 根据知识、装备和风险做判断
-> 能处理就处理
-> 不能处理或收益不足就立刻返回家园
-> 玩家看到任务进度条中断原因
```

这会让玩家的决策变得有意义：

- 派谁出去？
- 他会不会处理狼？
- 他有没有火把？
- 外界资源还够不够？
- 现在继续采集值不值得？
- 要不要先教他危险知识？

### 22.2 核心规则

P65 后续实现必须遵守：

```text
NPC 在采集、制作、学习、驱赶、外出交换等任务过程中，每个任务 tick 都必须进行风险与资源复查。
```

如果出现以下情况，任务必须插入或切换到 `return_home` 节点：

```text
1. 危险等级超过 NPC 当前可处理阈值。
2. NPC 不知道当前危险的有效反制知识。
3. NPC 知道反制知识但缺少反制物品。
4. NPC 生命/饥饿/疲劳状态不足以继续任务。
5. 目标资源池已经低于本次任务最低可采集量。
6. 任务继续收益低于返家安全收益。
7. 事件即将过期，继续执行已无法完成目标。
8. 玩家召回或营地进入紧急状态。
```

这条规则适用于：

```text
采集木头
采集食物
采集石头
与学者交流
制作物品
驱赶怪物
巡逻边界
运输物品
```

其中“制作物品”也需要检查危险，因为制作可能发生在营地边缘、临时工作点、火堆旁、夜间或怪物试探阶段。

### 22.3 任务 tick 流程

每个 NPC 任务 tick 的流程应固定为：

```text
1. 读取当前任务节点。
2. 读取营地状态 SafeZoneState。
3. 读取圈外事件 BoundaryEventState。
4. 读取相关资源池 ResourcePoolState。
5. 读取 NPC 背包、状态和知识。
6. 运行 TaskSafetyEvaluator。
7. 运行 TaskResourceEvaluator。
8. 如果必须返家：插入 return_home 节点并暂停当前节点。
9. 如果可以处理危险：插入 handle_threat / use_countermeasure 节点。
10. 如果资源不足：插入 return_home，携带 partial_result 或 empty_result。
11. 如果安全且资源足够：推进当前节点。
```

伪代码：

```text
advanceNpcTask(actor, task):
  current = task.current_step
  safety = evaluateTaskSafety(actor, task, current, safe_zone, boundary_events)
  resource = evaluateTaskResources(actor, task, current, resource_pools)

  if safety.must_return_home:
      task.insertNext(return_home(reason=safety.reason))
      task.status = interrupted
      return

  if safety.can_handle_threat:
      task.insertNext(handle_threat(countermeasure=safety.countermeasure))
      return

  if resource.must_return_home:
      task.insertNext(return_home(reason=resource.reason))
      task.status = interrupted
      return

  advanceCurrentStep(current)
```

### 22.4 危险推理输入

`TaskSafetyEvaluator` 输入：

```text
actor_key
actor_health
actor_hunger
actor_fatigue
actor_inventory
actor_knowledge_claims
task_kind
current_step_kind
location_kind
active_boundary_events
safe_zone_state
known_countermeasures
```

输出：

```text
risk_level
can_continue
can_handle_threat
must_return_home
countermeasure_action
reason_key
projected_delay_ticks
```

示例：

```text
NPC 正在采集木头。
北侧狼群从 approaching 进入 active。
NPC 有知识：torch + use -> repel_beast。
NPC 背包没有 torch。
结论：must_return_home = true。
原因：knows_countermeasure_but_missing_item。
```

另一个示例：

```text
NPC 正在采集木头。
北侧狼群 active。
NPC 有火把且知道用火把驱狼。
结论：can_handle_threat = true。
插入节点：使用火把驱赶狼。
影响：采集任务延迟 1-2 ticks。
```

### 22.5 危险对采集速度的影响

危险不一定导致失败，也可以影响速度。

规则：

```text
低风险：采集速度 -10%，不插入节点，只显示“警惕中”。
中风险：采集速度 -30%，可能插入“观察危险”节点。
高风险：必须处理危险或返家。
极高风险：立即返家，不允许继续采集。
```

资源采集节点需要支持：

```text
base_required_ticks
risk_delay_ticks
tool_bonus_ticks
knowledge_bonus_ticks
fatigue_penalty_ticks
final_required_ticks
```

示例：

```text
采集木头基础耗时 3 ticks。
NPC 有斧头：-1 tick。
附近狼群试探：+1 tick。
NPC 疲劳：+1 tick。
最终耗时：4 ticks。
```

前端进度条显示：

```text
采集木头：受狼群干扰，预计变慢
```

### 22.6 资源不足返家规则

圈外资源不能是无限的。

每个资源机会或资源池必须有容量：

```text
resource_pool_id
resource_object_key
current_amount
max_amount
reserved_amount
regen_per_day
regen_delay_ticks
depleted_until_tick
risk_level
```

NPC 接任务时不会立即扣资源，但要做 reservation：

```text
reserved_amount += planned_amount
```

采集完成时：

```text
current_amount -= actual_amount
reserved_amount -= planned_amount
```

如果任务推进过程中发现：

```text
current_amount - reserved_amount < minimum_required_amount
```

则必须：

```text
插入 return_home
原因：resource_depleted
可选：带回 partial_result
```

这可以避免：

- 多个 NPC 同时采同一个资源池，凭空产出无限资源。
- 外界资源耗尽后 NPC 还继续假装采集。
- 玩家没有资源压力。

### 22.7 资源自动生成与上限

每个安全圈周边有资源池，而不是无限资源点。

第一版资源池：

```text
nearby_woods：木头，max=8，每天恢复 2
nearby_bushes：红果，max=6，每天恢复 1
nearby_stones：碎石，max=5，每两天恢复 1
nearby_grass：干草，max=6，每天恢复 2
```

资源池刷新规则：

```text
每天清晨恢复 regen_per_day。
不能超过 max_amount。
被威胁占据时 regen 暂停或减半。
恶劣天气时特定资源恢复变慢。
```

这样玩家会感到压力：

```text
木头今天只剩 1。
狼还没驱走。
要不要派 NPC 冒险采？
还是先保存火把？
```

### 22.8 返回家园节点

`return_home` 是强制任务节点，不是简单结束。

数据：

```text
step_kind = return_home
status = active
reason_key
carried_items[]
partial_result_policy
safe_zone_target
remaining_ticks
```

返回家园完成后：

```text
NPC 回到安全圈。
保留已经采到的物品。
把任务状态改为 interrupted 或 completed_partial。
向前端投影原因。
允许玩家重新分配任务。
```

示例进度条：

```text
[接令]──[检查工具]──[外出采集]──[遭遇危险]──[返回家园]
                                      ↑
                                    阿木
```

头顶状态：

```text
正在返回：狼群靠近
```

### 22.9 制作过程遇险

制作不一定总在绝对安全位置。

制作任务也要检查：

```text
营地是否被攻击
火堆是否熄灭
工作点是否受威胁
材料是否在 NPC 背包或仓库中仍可用
关键材料是否被其他任务占用
```

如果制作中遇到危险：

```text
低风险：制作变慢。
中风险：暂停制作，插入观察/防御节点。
高风险：返家或转入避难节点。
营地突入：取消制作，进入 emergency_defense。
```

如果制作过程中材料不够：

```text
原因：material_reserved_by_other_task
动作：return_home 或 blocked_waiting_material
```

第一版可以简化：

```text
材料不够 -> return_home / blocked。
高风险 -> return_home。
低中风险 -> 增加耗时。
```

### 22.10 与知识系统的关系

危险处理不能写死 NPC 知道什么。

必须通过 KnowledgeRepository 查询：

```text
火把 + 对狼使用 -> 驱赶
陷阱 + 对狼使用 -> 阻挡
草药 + 对自己使用 -> 治疗
```

如果 NPC 没有知识：

```text
不知道如何处理危险 -> return_home
```

如果 NPC 有假设知识：

```text
可尝试处理，但风险更高，可能失败。
```

如果 NPC 有 active/shared 知识：

```text
可稳定处理，风险较低。
```

### 22.11 第一版最低实现要求

P66/P67 第一版至少要实现：

```text
1. 每个圈外资源池有 current/max/regen。
2. NPC 采集任务有资源 reservation。
3. 采集过程中如果资源不足，NPC 立刻 return_home。
4. 采集过程中如果狼威胁达到 high，NPC 立刻 return_home。
5. 如果 NPC 有火把和驱狼知识，可以插入 handle_threat，采集速度变慢但不立刻失败。
6. 制作过程中如果营地威胁 high，制作变为 interrupted，NPC return_home 或 emergency_defense。
7. 前端任务进度条显示 return_home 原因。
```

这 7 点是第一版“玩法全闭环”的底线。

### 22.12 验收用例

必须设计测试：

1. NPC 采木头，资源足够，无危险，正常返回木头。
2. NPC 采木头，中途资源被其他 NPC 预定完，立即返回，任务显示资源不足。
3. NPC 采木头，中途狼进入 high 风险，NPC 没火把，立即返回。
4. NPC 采木头，中途狼进入 high 风险，NPC 有火把但没知识，立即返回。
5. NPC 采木头，中途狼进入 high 风险，NPC 有火把且有知识，插入驱赶节点，采集延迟。
6. NPC 制作火把，中途营地进入 high 威胁，制作中断，返回或进入防御。
7. 资源池每天恢复但不超过上限。
8. 多 NPC 同采同一资源池不会超产。
9. return_home 后 NPC 背包保留已经获得的 partial_result。
10. 前端投影显示中断原因和当前节点。

## 23. 补强后的第一版完整闭环

第一版完整闭环应调整为：

```text
早晨：资源池刷新，但有上限。
白天：玩家派 NPC 采集木头。
采集中：狼群威胁推进，采集速度变慢。
如果 NPC 不会处理狼：返回家园。
玩家向流浪学者学习火把驱狼。
玩家教 NPC。
玩家命令 NPC 制作火把。
制作中：如果威胁过高，制作中断并返回/防御。
NPC 拿到火把后再次外出或去边界。
狼靠近：NPC 用已学知识驱赶。
夜晚：营地存活，资源真实减少，知识真实保留。
```

这才是“安全圈营地生存”的最小可玩闭环。

## 24. 补强设计：可回收知识卡片与知识分配玩法

### 24.1 为什么要把教学改成知识卡片

原来的教学更像一次性把知识复制给 NPC：

```text
玩家知道 -> 教 NPC -> NPC 永久知道
```

这种方式虽然符合“传播知识”，但缺少资源分配压力。安全圈营地玩法需要玩家在有限资源下做选择，所以知识也应该成为一种可管理资源。

P65 补强后，教学应改成“知识卡片分配”：

```text
玩家拥有若干知识卡片
-> 把知识卡片分配给 NPC
-> NPC 持有卡片期间可以使用对应知识
-> 卡片可回收
-> 卡片数量限制同时掌握该知识的 NPC 数量
```

这样会形成新的核心玩法：

```text
我有 1 张砍伐知识卡，只能让 1 个 NPC 会砍伐。
我有 1 张躲避野兽知识卡，只能让 1 个 NPC 会处理野兽风险。
如果我把砍伐卡给阿木，却没给他野兽风险卡，他外出砍伐遇到狼只能返家。
```

### 24.2 知识卡片不是一次性消耗品

知识卡片必须可回收。

规则：

```text
1. 知识卡片分配给 NPC 后，不被销毁。
2. NPC 持有卡片期间获得该知识的使用权。
3. 玩家可以从 NPC 身上收回卡片。
4. 收回后，NPC 失去该卡片授予的知识使用权。
5. 同一张卡同一时间只能由一个持有者使用。
6. 玩家拥有几张同类卡，就能同时让几个 NPC 使用该知识。
```

这让“教学”变成“知识装备/权限分配”，而不是无限复制。

### 24.3 知识卡片与真实知识的关系

需要区分三种状态：

```text
内化知识 InternalizedKnowledge：NPC 自己真正学会的知识。
持有卡片 GrantedKnowledge：NPC 因持有卡片临时可用的知识。
听说知识 HearsayKnowledge：NPC 听过但未验证的弱知识。
```

第一版可以先实现：

```text
KnowledgeCard 授予 usable_by_ai 权限。
NPC 决策时把持有卡片视为可用知识。
收回卡片后，这条可用知识消失。
```

后续可以扩展：

```text
NPC 多次使用卡片成功后，有概率把卡片知识内化为自己的永久知识。
内化后即使卡片被收回，也保留该知识。
```

但第一版不必须做内化，避免复杂度过高。

### 24.4 数据结构建议

新增知识卡片概念：

```text
KnowledgeCard
- card_id
- knowledge_template_key
- subject_object_key
- action_key
- effect_key
- target_object_key
- status
- owner_actor_key
- assigned_actor_key
- transferable
- recoverable
- granted_status
```

状态：

```text
in_player_deck      在玩家可分配卡组中
assigned_to_actor   已分配给 NPC
locked              暂时不可收回，例如 NPC 正在任务关键节点使用
lost                丢失，第一版可不做
```

`granted_status`：

```text
hypothesis
active
shared
```

第一版建议：

```text
玩家亲自验证过的 active 知识卡，分配给 NPC 后按 shared/active 使用。
流浪学者给的知识卡初始为 hypothesis，NPC 可以尝试但风险更高。
```

### 24.5 知识卡片如何进入系统

卡片来源：

```text
玩家亲自实践形成知识 -> 生成 1 张该知识卡片。
流浪学者教学 -> 获得 1 张或多张外来知识卡片。
特殊事件奖励 -> 获得知识卡片。
部落研究/复盘 -> 未来可复制知识卡片。
```

第一版建议：

```text
玩家通过学者获得“火把驱狼”知识卡。
玩家通过实践获得“制作火把”知识卡。
玩家把卡分配给 NPC。
NPC 持卡执行任务。
```

### 24.6 分配与回收操作

玩家点击 NPC 面板时，可以看到：

```text
NPC 当前持有知识卡：
- 砍伐木头
- 制作火把

玩家可分配知识卡：
- 火把驱狼 x1
- 识别毒蘑菇 x1
```

操作：

```text
给卡
收回
替换
查看来源
```

收回限制：

```text
1. NPC 正在安全圈内空闲：可立即收回。
2. NPC 正在外出任务但尚未进入关键节点：可收回，但任务会重新评估。
3. NPC 正在使用该卡处理危险/制作/采集：暂时 locked，节点结束后才能收回。
4. 强制召回 NPC 后可收回。
```

这样避免玩家在 NPC 正在驱狼的一瞬间收回“火把驱狼”卡导致逻辑混乱。

### 24.7 知识卡片与任务推理

NPC 任务推理必须同时读取：

```text
NPC 内化知识
NPC 当前持有的知识卡片
NPC 背包
营地仓库
当前危险
当前资源池
```

决策规则：

```text
如果任务需要某知识，NPC 既没有内化知识，也没有对应卡片 -> 任务 blocked 或 return_home。
如果任务遇到风险，NPC 没有风险处理知识，也没有对应风险处理卡 -> return_home。
如果 NPC 有处理知识但缺物品 -> return_home。
如果 NPC 有处理知识和物品 -> 插入 handle_threat，任务延迟但继续。
```

### 24.8 采集示例：只有砍伐卡，没有避险卡

场景：

```text
玩家给阿木“砍伐木头”知识卡。
玩家没有给阿木“火把驱狼”知识卡。
阿木带斧头去采木头。
采集中狼群风险升高。
```

推理：

```text
阿木会砍树。
阿木不会处理狼。
阿木没有可用风险反制知识。
结论：插入 return_home。
```

进度条：

```text
[接令]──[检查知识]──[外出采集]──[发现狼群]──[返回家园]
                                      ↑
                                    阿木
```

头顶状态：

```text
返回家园：不会处理狼群
```

这就是知识分配玩法的核心。

### 24.9 采集示例：砍伐卡 + 驱狼卡

场景：

```text
玩家给阿木“砍伐木头”知识卡。
玩家也给阿木“火把驱狼”知识卡。
阿木背包有火把。
阿木外出采集时遇到狼。
```

推理：

```text
阿木会砍树。
阿木知道火把可驱狼。
阿木有火把。
结论：插入 handle_threat 节点。
采集速度变慢，但任务可继续。
```

进度条：

```text
[接令]──[检查知识]──[外出采集]──[驱赶狼群]──[继续采集]──[返回]
                                      ↑
                                    阿木
```

### 24.10 制作示例：制作卡被收回

场景：

```text
玩家给阿木“制作火把”知识卡。
阿木开始制作火把。
玩家在制作关键节点前收回卡片。
```

结果：

```text
任务重新评估。
阿木不再拥有制作火把知识。
任务 blocked 或 return_home。
```

如果阿木正在关键制作节点：

```text
卡片 locked。
制作节点结束后才允许收回。
```

### 24.11 与 KnowledgeRepository 的关系

知识卡片不是绕过知识系统。

推荐实现：

```text
KnowledgeRepository 保存内化知识。
KnowledgeCardRuntime 保存卡片分配状态。
NpcTaskReasoner 查询时合并：内化知识 + 当前持卡授予知识。
Projection 区分显示：已学会 / 持卡可用。
```

前端显示：

```text
知识：
- 砍伐木头（持卡）
- 红果可食用（已学会）
- 火把驱狼（未持有）
```

### 24.12 可回收卡片对平衡的影响

卡片可回收会让玩家能调整策略，但也要避免无脑频繁换卡。

第一版限制：

```text
1. 同一 tick 内同一张卡只能转移一次。
2. 外出任务中卡片收回会触发任务复查，可能导致 NPC 返家。
3. 关键节点使用中的卡片暂时 locked。
4. 远离安全圈的 NPC 不允许直接收回卡，必须返回或通过通信能力，第一版默认必须返回。
```

这样卡片是可回收的，但不是瞬间全局魔法菜单。

### 24.13 UI 表现

NPC 面板增加知识卡栏：

```text
阿木
当前任务：采集木头

持有知识卡：
[砍伐木头] [制作火把]

缺失风险知识：
[火把驱狼]

按钮：
[给卡] [收回卡] [查看知识来源]
```

玩家知识卡库：

```text
知识卡库
砍伐木头 x1：已分配给阿木
制作火把 x1：未分配
火把驱狼 x1：未分配
识别毒蘑菇 x1：未分配
```

任务进度条节点详情显示：

```text
当前节点：外出采集
已满足知识：砍伐木头（持卡）
缺失风险知识：火把驱狼
遇到狼群时策略：返回家园
```

### 24.14 与流浪学者的关系

流浪学者给予的不是“永久技能”，而是知识卡片。

示例：

```text
流浪学者提供：
- 火把驱狼知识卡 x1
- 草药治疗知识卡 x1
- 陷阱制作知识卡 x1
```

玩家用食物/护送/交换获得卡片。

卡片状态可以是：

```text
hearsay_card：听说卡，风险较高。
verified_card：验证卡，效果稳定。
```

NPC 使用 hearsay_card 时：

```text
可以尝试，但失败率更高。
成功后可升级卡片可信度，或形成内化知识。
```

第一版可先不做失败率，只显示“外来知识/未验证”。

### 24.15 对第一版闭环的影响

第一版闭环需要更新为：

```text
玩家获得“火把驱狼”知识卡。
玩家获得或制作“制作火把”知识卡。
玩家把“制作火把”卡给 NPC A。
NPC A 制作火把。
玩家把“火把驱狼”卡给 NPC A 或 NPC B。
持卡 NPC 才能执行驱狼任务。
如果只给采集卡不给避险卡，采集遇到狼就 return_home。
```

这会形成最小策略：

```text
知识卡分配 = 谁能做什么 + 谁能应对什么风险。
```

### 24.16 第一版最低实现要求

P66/P67 需要把知识卡作为第一版核心规则，而不是后续扩展。

最低要求：

```text
1. 玩家拥有知识卡库存。
2. 知识卡可分配给 NPC。
3. 知识卡可从 NPC 回收。
4. NPC 决策时把持有卡片视为可用知识。
5. NPC 没有风险处理知识或卡片时，遇到高风险必须 return_home。
6. NPC 有采集知识卡但没有避险知识卡时，可以采集，但遇险返家。
7. NPC 有避险知识卡和对应物品时，可以插入 handle_threat。
8. 前端 NPC 面板显示持卡知识和缺失风险知识。
9. 任务进度条显示“因为缺少某知识卡而返回”。
10. 卡片回收后 NPC 任务会重新评估。
```


### 24.16.1 第一版 UI 与投影补充

本阶段实现知识卡分配时，不能把每张知识卡都展开成一个后端 `available_command`。

原因：

```text
如果后续有 1000 张知识卡，直接生成 1000 个按钮会让 UI、协议和低上下文研发全部失控。
```

第一版必须采用：

```text
1. 后端只暴露“管理某 NPC 知识卡”的入口命令。
2. 进入知识卡库面板后，由后端投影提供卡片列表。
3. 前端只能调用后端提供的 assign/recover 接口，不能自己判断卡片是否生效。
4. 卡片库必须支持分类筛选。
5. 卡片库必须支持搜索。
6. 卡片库必须支持滚动，不允许 1000 张卡直接铺满按钮。
7. 搜索仅影响显示，不改变后端规则结果。
```

NPC 任务进度不能只靠中文状态字符串解析。第一版需要提供结构化任务投影：

```text
actor_tasks[actor_key] = {
  active,
  task_key,
  label,
  nodes,
  node_index,
  progress,
  duration,
  percent,
  outside_safe_zone,
  handling_threat,
  returning_home,
  required_card_key,
  blocked_reason
}
```

前端进度条读取 `actor_tasks`，`actor_work_status` 只作为可读文本和兼容显示。

这条规则是为了避免后续新增魔法、机械、外星文明任务时继续改前端字符串判断。

### 24.17 验收用例补充

新增测试：

1. 玩家有 1 张砍伐卡，只能同时分配给 1 个 NPC。
2. 收回砍伐卡后，原 NPC 不再能开始砍伐任务。
3. NPC 持有砍伐卡但没有驱狼卡，采集中遇狼，return_home。
4. NPC 持有砍伐卡和驱狼卡但没有火把，采集中遇狼，return_home。
5. NPC 持有砍伐卡和驱狼卡且有火把，采集中遇狼，插入 handle_threat。
6. NPC 制作中被收回制作卡，任务 blocked 或 return_home。
7. NPC 正在关键节点使用卡片时，卡片 locked，不能立刻收回。
8. 收回卡片后可分配给另一个 NPC，另一个 NPC 获得可用知识。
9. 前端投影区分“已学会”和“持卡”。
10. 流浪学者奖励知识卡后，玩家卡库数量增加。

## 25. 补强设计：新增玩法规则必须新增模块文件

### 25.1 问题定义

P65 改成安全圈营地玩法后，后续一定会频繁增加新逻辑：

```text
新怪物行为
新营地事件
新 NPC 职业
新外出任务
新风险反制方式
新学者/商人/访客
新知识卡片来源
新资源池刷新规则
新文明或魔法事件
```

如果这些逻辑继续集中写在一个 `CampRuntimeHost`、`TaskEngine`、`client_runtime_host_factory` 或某个巨大的 `switch/if` 文件里，项目会重新回到 P64 之前的问题：能跑，但不可扩展，低上下文 AI 也会继续把新规则塞进旧文件。

所以 P65 后续实现必须把“新增规则”设计为“新增独立规则文件 + 注册入口”，而不是在旧文件里无限追加分支。

### 25.2 核心结论

新增玩法逻辑的默认方式必须是：

```text
新增一个规则/行为/事件/任务文件
-> 在对应 registry/module manifest 中注册
-> 通过通用接口被 CampRuntimeHost / TaskEngine / AgentReasoner 调用
-> 不修改已有规则文件，除非是在扩展通用接口
```

错误方向：

```text
CampTaskEngine.cpp 里继续追加 if task_type == "woodcut"。
CampThreatEngine.cpp 里继续追加 if threat_key == "wolf"。
CampKnowledgeEngine.cpp 里继续追加 if card_key == "torch_repel_wolf"。
CampRuntimeHostFactory.cpp 里注册所有具体怪物、任务、采集、制作、教学规则。
```

正确方向：

```text
camp/tasks/woodcut_task_rule.*
camp/threats/beast_threat_rule.*
camp/knowledge_cards/knowledge_card_assignment_rule.*
camp/visitors/wandering_scholar_event_rule.*
camp/resources/nearby_woods_pool_rule.*
camp/register_camp_core_rules.*
```

### 25.3 分层原则

P65 后续实现建议采用四层：

| 层级 | 职责 | 可以新增文件 | 禁止事项 |
|---|---|---|---|
| Camp Runtime Host | 装配营地运行时、持有通用服务 | 只新增 host/port/registry 基础设施 | 禁止写狼、火把、砍树、学者等具体规则 |
| Camp Rule Registry | 收集任务、风险、资源、知识卡、事件规则 | 新增注册清单或规则 manifest | 禁止直接改 runtime 状态 |
| Camp Rule Module | 单个规则的判断和效果声明 | 每个新规则一个独立文件 | 禁止跨模块偷改背包、知识、资源池 |
| Authority Service | Inventory/Knowledge/Command/Resource/Task 权威状态修改 | 只在需要通用能力时改 | 禁止内容专属硬编码 |

这套分层的目标不是为了复杂，而是为了保证以后加内容时低级 AI 不会把所有东西写进同一个文件。

### 25.4 建议目录结构

第一版可以先用 `backend/include/pathfinder/camp/` 与 `backend/src/camp/`，不要继续塞进 client/http/runtime host：

```text
backend/include/pathfinder/camp/

  camp_rule_registry.h
  camp_task_rule.h
  camp_threat_rule.h
  camp_resource_pool_rule.h
  camp_knowledge_card_rule.h
  camp_boundary_event_rule.h
  camp_projection_adapter.h

backend/src/camp/
  camp_runtime_host.cpp
  camp_rule_registry.cpp
  register_camp_core_rules.cpp

backend/src/camp/tasks/
  gather_resource_task_rule.cpp
  craft_item_task_rule.cpp
  return_home_task_rule.cpp
  handle_threat_task_rule.cpp

backend/src/camp/threats/
  generic_beast_threat_rule.cpp

  finite_resource_pool_rule.cpp

```

如果后续有魔法世界或机械危机，不应该改 `generic_beast_threat_rule.cpp`，而应该新增：

```text
backend/src/camp/threats/magic_corruption_threat_rule.cpp
backend/src/camp/threats/mechanical_swarm_threat_rule.cpp
backend/src/camp/events/alien_probe_event_rule.cpp
```

### 25.5 规则接口要求

每类规则都必须通过统一接口进入系统。

#### 25.5.1 任务规则接口

任务规则不直接执行状态修改，只输出计划节点、需求、阻塞原因和待执行 Command：

```text
CampTaskRule
- rule_key
- canStart(context) -> TaskRuleDecision
- buildPlan(context) -> TaskPlanDraft
- tick(context, task_state) -> TaskTickDecision
- requiredKnowledge(context) -> KnowledgeRequirement[]
- requiredItems(context) -> ItemRequirement[]
- possibleRisks(context) -> RiskRequirement[]
```

任务规则可以决定：

```text
继续当前节点
插入 handle_threat
插入 return_home
等待资源
任务失败
任务完成并提交产物 Command
```

但任务规则不能绕过 Inventory、Knowledge、ResourcePool 或 Command 管线直接改状态。

#### 25.5.2 风险规则接口

风险规则只描述风险如何出现、如何升级、需要什么反制知识和物品：

```text
CampThreatRule
- threat_key
- evaluate(context) -> ThreatState
- requiredCountermeasures(context) -> CountermeasureRequirement[]
- applyUnhandledResult(context) -> ThreatOutcomeDraft
- applyHandledResult(context) -> ThreatOutcomeDraft
```

例如狼、毒雾、机械巡逻机、魔法诅咒都应该是不同规则文件，但都输出同一种 `ThreatState` 和 `CountermeasureRequirement`。

#### 25.5.3 知识卡规则接口

知识卡规则只处理卡片分配、回收、锁定、授予可用知识：

```text
CampKnowledgeCardRule
- canAssign(card, actor, context)
- canRecover(card, actor, context)
- grantUsableKnowledge(card, actor, context)
- revokeUsableKnowledge(card, actor, context)
- lockReason(card, actor, task_state)
```

NPC 推理读取的是统一后的 `UsableKnowledgeView`，不应该关心知识来自“内化”还是“持卡”。

#### 25.5.4 资源池规则接口

资源池规则负责有限资源、预占、恢复和枯竭：

```text
CampResourcePoolRule
- pool_key
- availableAmount(context)
- reserve(request) -> ReservationResult
- consume(reservation) -> ConsumeResult
- release(reservation)
- tickRegen(context)
```

所有采集任务必须通过资源池预占，不能在任务完成时凭空生成材料。

### 25.6 注册方式

注册文件只做汇总，不写规则逻辑：

```text
register_camp_core_rules.cpp
- register GatherResourceTaskRule
- register CraftItemTaskRule
- register ReturnHomeTaskRule
- register HandleThreatTaskRule
- register GenericBeastThreatRule
- register FiniteResourcePoolRule
- register KnowledgeCardAssignmentRule
- register WanderingScholarEventRule
```

如果新增一个规则，允许修改注册清单，但注册清单里只能出现：

```text
registry.registerTaskRule(std::make_unique<NewTaskRule>());
registry.registerThreatRule(std::make_unique<NewThreatRule>());
```

禁止在注册清单里写：

```text
if wolf then ...
if torch then ...
if npcName == ...
```

### 25.7 与 JSON / ContentRegistry 的关系

P65 不推翻已有 JSON / ContentRegistry。

规则文件负责“通用行为类型”，JSON 负责“具体内容参数”：

| 例子 | 应放哪里 | 原因 |
|---|---|---|
红果、毒蘑菇、火把、斧头 | JSON / ContentRegistry | 具体对象内容 |
制作火把配方 | JSON / ContentRegistry | 具体反应内容 |
知识卡显示名、来源、可信度 | JSON / ContentRegistry | 内容数据 |
有限资源池预占算法 | CampResourcePoolRule 代码 | 通用规则 |
任务遇险返家算法 | CampTaskRule 代码 | 通用规则 |
狼需要火把反制 | JSON + GenericBeastThreatRule | 内容参数 + 通用风险规则 |
魔法风暴需要护符反制 | JSON + MagicThreatRule 或 GenericThreatRule | 新风险类型或内容参数 |

判断标准：

```text
只是换名字、数值、产物、标签、配方 -> JSON。
出现新的状态机、结算方式、任务节点类型、风险模型 -> 新增规则文件。
多个内容都会复用的新能力 -> 先补通用接口，再新增规则文件。
```

### 25.8 对当前玩法的头脑风暴压力测试

用后续场景反推当前架构是否够用：

#### 场景 A：新增野猪

```text
野猪会破坏资源池，能被陷阱驱赶。
```

合理落点：

```text
JSON 定义 wild_boar、trap、countermeasure tags。
如果野猪仍是普通生物威胁，复用 GenericBeastThreatRule。
如果野猪会破坏资源池，新增 resource_raider_threat_rule.cpp。
```

不合理落点：在 `CampTaskEngine.cpp` 里写 `if threat == wild_boar`。

#### 场景 B：新增流浪学者

```text
学者进安全圈边界，给玩家知识卡。
```

合理落点：

```text
wandering_scholar_event_rule.cpp
KnowledgeCardService 发放卡片
前端只显示事件和可选 Command
```

#### 场景 C：新增魔法寒潮

```text
寒潮降低任务速度，需要保暖知识卡或火源。
```

合理落点：

```text
magic_cold_wave_event_rule.cpp
或 generic_environment_hazard_rule.cpp
ContentRegistry 定义 countermeasure: warmth_source / shelter_knowledge
```

#### 场景 D：新增机械巡逻机

```text
机械巡逻机会追踪外出 NPC，需要隐蔽知识卡或电磁干扰器。
```

合理落点：

```text
mechanical_patrol_threat_rule.cpp
NPC 没有反制知识 -> return_home
NPC 有反制知识但缺设备 -> return_home
NPC 两者都有 -> handle_threat 并延迟任务
```

#### 场景 E：新增高级文明交易

```text
高级文明提供知识卡复制服务，但消耗稀有资源。
```

合理落点：

```text
advanced_civilization_trade_event_rule.cpp
KnowledgeCardService 增加通用 duplicateCard Command
InventoryService 消耗材料
```

如果复制卡片需要新通用能力，先扩展 KnowledgeCardService，不在事件规则里私自复制卡。

#### 场景 F：NPC 外出中遇到未知风险

```text
NPC 没有任何知识卡能解释风险。
```

合理结果：

```text
TaskEngine 插入 return_home。
任务进度条显示：未知风险，缺少反制知识。
不允许前端猜测风险处理方式。
```

#### 场景 G：资源耗尽

```text
附近树林 current_amount 不足。
```

合理结果：

```text
ResourcePoolRule 拒绝 reserve。
TaskEngine 插入 return_home 或 blocked。
前端显示资源不足，不生成木头。
```

#### 场景 H：多个 NPC 同时采集

```text
两个 NPC 都去采木头。
```

合理结果：

```text
ResourcePoolRule 预占资源。
第二个 NPC 如果 available 不足，直接 blocked/return_home。
不能两个任务各自到完成时都凭空生成木头。
```

### 25.9 文档自检结论

按以上压力测试，P65 当前设计可行，但必须增加以下硬约束，否则后续会再次失控：

```text
1. CampRuntimeHost 只装配，不写具体玩法规则。
2. 每种新任务、新风险、新事件、新资源池规则、新知识卡操作默认新增独立规则文件。
3. 注册文件只注册，不写规则逻辑。
4. JSON / ContentRegistry 负责具体内容参数，不承载复杂状态机。
5. 任务推理统一读取 UsableKnowledgeView，不能到处判断“内化/持卡/听说”。
6. 遇到无法处理的风险或资源不足，统一走 return_home / blocked，而不是静默失败。
7. 前端只显示 projection 与 command options，不实现规则。
```

因此，P66/P67 实现前必须先建立 `CampRuleRegistry` 和上述基础接口。否则即使第一版能跑，也会在新增外星文明、魔法世界、机械危机时重新变成大文件堆 if。

## 26. P65 补充：后端任务图与节点画布投影

### 26.1 为什么需要补充

线性进度条只能表达“任务进行到哪里”，不能表达 NPC 在任务中途遭遇风险、插入推理、处理风险后回到原任务，或处理失败后返家。

P65 第一版必须把任务进度升级为后端任务图投影，前端只渲染节点画布。这样后续无论是采集遇狼、制作时资源不足、魔法污染、机械感染、外星信号干扰，都能表现为任务图上的动态分支，而不是在前端写规则或堆中文字符串。

### 26.2 权威边界

```text
CampRuntime / CampRule 是任务图权威。
CampTaskGraphView 是只读投影。
Axmol 前端只根据 nodes / edges / active_node_id / status / x / y 绘制节点画布。
前端不得根据 NPC 状态自行插入“遭遇野兽”“推理火把”“返回家园”等节点。
前端缩放只改变画布显示比例，不改变任务执行状态。
```

### 26.3 新增投影结构

```text
CampTaskGraphNodeView
- node_id：后端稳定节点 ID。
- label：玩家可见短文本。
- node_type：action / check / event / decision / recovery / fail / merge。
- status：locked / active / completed。
- x / y：后端提供的逻辑布局坐标。

CampTaskGraphEdgeView
- edge_id：后端稳定边 ID。
- from_node_id / to_node_id：连接端点。
- edge_type：main / branch / merge / return。
- status：locked / active / completed。

CampTaskGraphView
- nodes：节点列表。
- edges：边列表。
- active_node_id：当前执行节点。
- graph_mode：linear / branching。
```

### 26.4 当前闭环算法

普通任务尚未遭遇风险时：

```text
接令 -> 检查知识 -> 前往/准备 -> 执行 -> 交付
```

外出任务执行到圈外阶段后，如果威胁达到触发阈值，后端把线性图切换为分支图：

```text
接令 -> 检查知识 -> 圈外执行 -> 风险已处理/返回家园 -> 执行/交付
                       |
                       v
                  遭遇野兽 -> 推理原因 -> 驱赶野兽/返家决策
                                         |
                                         v
                                  合流或返家节点
```

分支不会提前出现在任务图中；只有风险真实发生后才由后端插入。NPC 有反制知识且带有反制物品时进入 recovery 分支，处理完成后回到主线。NPC 缺知识或缺物品时进入 fail/return 分支，并返回安全圈。

### 26.5 前端画布规则

```text
1. NPC 面板展示后端提供的 task_graph。
2. 节点位置来自后端 x/y，前端只做等比例映射。
3. 节点颜色只表达 node_type 与 status，不表达隐藏规则。
4. 边颜色只表达 edge_type 与 status。
5. 鼠标滚轮在任务图区域内只缩放任务图，不缩放世界地图。
6. 画布必须有固定边界、网格底纹、节点居中排版，不能与 NPC 状态、背包、知识区重叠。
7. 不允许在画布上显示“后端投影”等调试词。
```

### 26.6 后续可编辑任务图预留

当前 P65 只做“后端投影 + 前端显示”。后续如果要让玩家编辑任务模板，必须新增 Command：

```text
camp.task_graph.create_template
camp.task_graph.add_node
camp.task_graph.remove_node
camp.task_graph.connect
camp.task_graph.validate
camp.task_graph.assign_to_actor
```

这些 Command 必须由后端验证节点合法性、知识需求、资源需求、风险处理能力和可执行性。前端仍然只是编辑意图的输入层，不能在本地保存一套可执行规则。

### 26.7 验收标准

```text
1. 未遭遇风险时，NPC 任务图是 linear，不显示风险分支。
2. 采集类任务到圈外阶段后，威胁触发才出现 branching 图。
3. 有驱赶知识和火把时，分支显示遭遇 -> 推理 -> 驱赶 -> 回到主线。
4. 缺知识或缺火把时，分支显示遭遇 -> 缺少原因 -> 返家决策。
5. active_node_id 会随 tick 推进，不能一直停在第一个风险节点。
6. 前端滚轮缩放任务图不会触发世界地图缩放。
7. Axmol 构建通过，后端冒烟测试验证成功/失败两条分支。
```

### 26.8 任务图显示补强

```text
1. 主线、分支、合流、返家线必须使用不同折线路由，不能让返家线和继续执行线堆在一起。
2. NPC 任务面板允许拖拽，但拖拽只改变客户端窗口位置，不改变后端状态。
3. NPC 任务图区域默认比普通信息区更大，优先保证节点和连线可读。
4. 任务完成后，最后一张任务图保留在 actor task projection 中，直到该 NPC 接到下一项任务时由新任务覆盖。
5. 完成态任务图只读显示，不能继续推进，也不能被前端当作可执行状态。
```

## 27. P65 补充：营地内容 JSON 化与硬编码清理

### 27.1 为什么需要补充

P65 第一轮为了快速验证安全圈营地玩法，把资源、NPC、初始物品、知识卡、威胁、配方和任务类型集中写进了 `CampRuntime` 与少量规则文件。这样虽然能跑通原型，但会造成三个严重问题：

```text
1. 新增物品或改数值必须改 C++，扩展成本过高。
2. NPC 任务、风险、制作结算会重新变成一堆 wood / axe / torch / wolf if。
3. 低上下文研发者会误以为这些硬编码就是架构，继续把后续内容堆进 runtime。
```

因此本补充把第一版营地内容迁移到 `content/core/camp/basic_camp.json`，并要求 C++ 只保留通用读取、裁决和状态推进逻辑。

### 27.2 权威边界

```text
content/core/camp/basic_camp.json 是当前营地基础内容权威。
CampContentPack 是 JSON 进入运行时后的只读内容定义。
CampRuntime 只保存运行时状态和推进流程，不保存具体物品/配方/任务内容。
CampRule 只能根据 CampContentPack 生成 command option 或执行通用结算。
Axmol 前端只读取 projection / options，不读取 JSON，也不写玩法规则。
```

禁止事项：

```text
禁止在 CampRuntime 里写 red_berry / torch / axe / wolf / herb 等内容专属分支。
禁止在 player_resource_rule.cpp 里为每个资源、配方、威胁追加 if。
禁止在 npc_task_rule.cpp 里为每个任务类型追加 if。
禁止在 scholar_event_rule.cpp 里硬编码奖励知识卡内容。
禁止用内置 fallback 内容替代 JSON；缺少核心 JSON 应直接暴露错误。
```

### 27.3 当前 JSON 字段范围

| 区块 | 作用 | 运行时用途 | 扩展方式 |
|---|---|---|---|
| `items` | 物品定义 | 名称、数值状态、前端显示 | 新增 item key、display_name、numeric_states |
| `initial_inventory` | 营地初始仓库 | reset 时初始化仓库 | 添加或调整 item stack |
| `static_entities` | 安全圈固定实体 | 投影火堆、学者等实体 | 新增 entity_id/entity_key/坐标 |
| `resources` | 圈外资源池 | 采集 option、资源上限、产出、恢复 | 新增资源池和 output_items |
| `actors` | 初始 NPC | NPC 名称、外观 key、坐标、生命值 | 新增 actor 定义 |
| `knowledge_cards` | 可分配知识卡 | 卡库、NPC 任务解锁、风险反制 | 新增 card_id/knowledge_key/category |
| `threats` | 圈外风险 | 威胁投影、遭遇阈值、反制知识/物品、分支文案 | 新增威胁定义，后续状态需从单 threat_level 升级为多威胁表 |
| `recipes` | 营地制作配方 | 玩家制作和 NPC 代工结算 | 新增 inputs/outputs/command_key |
| `tasks` | NPC 可执行任务 | NPC task option、任务时长、资源/配方绑定、节点 | 新增 command_key、resource_key 或 recipe_key |
| `scholar_*` | 学者事件参数 | 学者目标实体、奖励卡片、重复交谈事件 | 修改奖励来源或替换学者实体 |

### 27.4 通用流程

玩家营地操作：

```text
CampRuntime::snapshot()
-> CampResourceCommandRule::buildOptions()
-> 遍历 content.resources / content.recipes / primary threat
-> 生成后端 command option
-> submitOption(option_id)
-> CampResourceCommandRule::execute()
-> 通用扣资源池 / 扣材料 / 加产物 / 降威胁
-> CampRuntime::markChanged()
-> 前端收到新 projection
```

NPC 任务：

```text
知识卡分配给 NPC
-> NpcTaskRule 遍历 content.tasks
-> 只为持有 required_card_key 的 NPC 生成任务 option
-> startNpcTask() 保存 task.definition 的 resource_key / recipe_key / threat_key
-> advanceNpcTask() 按通用任务进度推进
-> 若外出且 threat 触发，读取 threat.counter_knowledge_key / counter_item_key 决策
-> completeNpcTask() 根据 resource_key 或 recipe_key 做通用结算
```

学者事件：

```text
ScholarEventRule 读取 content.scholar_reward_card_id
-> 如果卡库没有该卡，则从 content.knowledge_cards 构造 CampKnowledgeCard
-> 如果已有该卡，则播放 content.scholar_repeat_event
```

### 27.5 当前实现边界

已经清理：

```text
1. 初始资源、NPC、红果、火把、知识卡不再写在 CampRuntime。
2. 物品名称、斧头耐久等数值状态来自 JSON 的 items.numeric_states。
3. 玩家采集、制作、驱赶选项来自 resources / recipes / threats。
4. NPC 采木、采石、做斧头、做火把来自 tasks，不再写死任务列表。
5. 任务完成结算读取 resource.output_items / npc_output_items 或 recipe.inputs / outputs。
6. 学者奖励知识卡来自 scholar_reward_card_id。
7. 任务图风险节点使用 threat 文案和 task 节点，不再靠“已处理”中文字符串判断成功分支。
```

仍然保留为 P65 抽象模型的内容：

```text
1. 当前 CampRuntime 仍是“小地图安全圈”运行时，不是通用开放世界 runtime。
2. 当前威胁运行态仍是单一 threat_level；未来多怪物、多灾害、多区域风险需要升级为 map<threat_key, threat_state>。
3. 当前任务图仍是采集/制作/风险分支的最小任务图引擎；后续玩家可编辑任务图必须单独设计 Command。
4. 当前 terrain cell 是安全圈视觉投影模型，不代表最终地图系统。
```

这些边界不是内容硬编码，但后续扩展魔法灾害、机械危机、外星入侵时必须优先升级为多实例状态，不能把新内容继续塞进单一 `threat_level`。

### 27.6 验收标准

```text
1. 修改 `content/core/camp/basic_camp.json` 的物品显示名或 numeric_states 后，背包投影随之变化。
2. 新增 resources 条目后，玩家营地采集 option 来自 JSON。
3. 新增 recipes 条目后，材料满足时玩家制作 option 来自 JSON。
4. 新增 tasks 条目后，NPC 持有 required_card_key 才能看到对应任务 option。
5. 修改 threats 的 counter_item_key / counter_knowledge_key 后，NPC 遇险处理逻辑随之变化。
6. 删除旧 `camp.camp_repel_beast` 不影响威胁驱赶，因为 command_key 来自 threat.repel_command_key。
7. 后端专项测试必须覆盖内容加载、物品数值投影、玩家采集制作、学者奖励和任务分支。
```
