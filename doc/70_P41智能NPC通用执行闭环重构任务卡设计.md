# P41 智能 Agent 与世界效果通用执行闭环重构任务卡设计

## 1. 阶段定位

P41 是智能 Agent 与世界效果通用执行闭环重构阶段。

它接在 P39 知识链式推理与目标规划、P40 动态目标执行与外界打断系统之后，不是重新设计 Agent，也不是新增一批玩法内容；它要把原始生存、高级文明、魔法世界、机械危机、生态系统、外交信息战等未来内容统一放进同一套底层执行框架。

P41 的核心职责是把当前仍然散落在 C++ 分支里的效果语义、前置条件、世界变化执行、Agent 计划执行，统一收束到一套可配置、可验证、可追踪、可跨时代扩展的世界效果执行闭环里。

当前 P39/P40 已经证明了方向：

- 知识可以变成候选行动。
- 目标可以驱动 Agent 推理。
- 材料数量可以影响前置条件。
- 野兽、同伴都可以作为 Agent 参与世界变化。
- H5 可以投射出执行状态和世界反馈。

但当前仍然存在风险：

- `EffectSemantics` 仍是 C++ 内置表。
- `requiredPreconditionsFor` 仍有硬编码兜底。
- `ReactionPlanningAdapter` 仍把火把、木头、斧头等映射写死。
- `WorldInteractionService` 仍按 `effect_key` 写执行分支。
- `AgentPlanExecutorAdapter` 只对少数 effect 真正执行。
- H5 演示层仍有部分通过文本或固定场景推断世界状态的逻辑。

P41 不是为了增加更多按钮，而是为了让后续增加毒蘑菇、普通蘑菇、斧头耐久、房子御寒、陷阱捕兽、船过河、专家 NPC 教学、族群自动协作时，不再需要反复改核心推理和执行代码。

P41 在总架构中的位置：

- 上游读取：P18 知识系统、P27 条件表达式、P28 对象反应、P28.5 配置化物品与反应流水线、P29 危险与探索压力、P39 Agent 推理、P40 动态执行。
- 本阶段提供：通用效果语义注册、通用前置条件解析、通用世界变化执行、Agent 计划逐步执行桥接。
- 下游支撑：地图、采集、建造、族群协作、野生 NPC 行为、专家 NPC 教学、完整 H5 游戏界面。

## 2. 为什么现在做

现在必须做 P41，原因有三点。

第一，P39/P40 已经把智能 NPC 的“脑”搭出来了，但“手”还不够通用。

推理器可以知道“我需要火把”，但执行层仍然只认识少数写死 effect。如果继续加玩法，会出现推理越来越高级、执行越来越脏的断层。

第二，当前 H5 已经开始暴露系统真实性问题。

玩家能看到同伴是否凭空拥有火把、野兽是否真的会行动、制作火把是否真的消耗材料。如果执行层是硬编码演示逻辑，玩家马上会感到“这不是一个活系统”。

第三，后续内容都会依赖通用执行。

例如：

- 毒蘑菇：吃了中毒，晒干后可入药。
- 普通蘑菇：可食用，但过量也可能不适。
- 斧头：砍木头会变钝，打磨恢复。
- 房子：御寒，需要木材，人口增加后需要更多房子。
- 陷阱：需要木材和藤蔓，可捕兽或防御。
- 狼：可以学习火危险，也可能绕行或伏击。
- 同伴：没有火把时要自己准备火把。

如果这些都继续靠 `if effect_key == ...` 写，项目会快速失控。

所以 P41 要把“知识推导出的行动”真正接到“配置驱动的世界变化执行”。

## 3. 阶段目标

P41 完成后，系统必须具备以下能力。

1. 效果语义可注册

效果语义不再只能写在 `effect_semantics.cpp` 的固定表里。

系统必须支持从内置配置或 fixture 注册：

- effect_key。
- 中文显示名。
- 适配目标类型。
- 对世界、需求、威胁、资源、对象状态的预期影响。
- 风险分。
- 时间成本。
- 最低知识置信度。
- 是否允许 Agent 使用。

2. 前置条件统一来源

Agent 推理前置条件必须来自：

- EffectExecutionSpec。
- ReactionRule。
- P27 条件表达式。

不允许推理器自己知道“火把需要 camp_fire”。

3. 世界变化通用执行

WorldInteractionService 不再为每个 effect 写一段业务分支。

它应该解释执行配置：

- 消耗对象数量。
- 增加对象数量。
- 设置对象状态。
- 增减对象状态数值。
- 增删标签。
- 改变需求。
- 改变威胁。
- 生成事件。
- 输出安全中文反馈。

4. Agent 计划可以逐步执行

AgentPlanExecutorAdapter 不应只真正执行 `repel_beast`。

它应该把每一个 `AgentPlanStep` 转换成通用执行请求，然后交给通用执行器。

如果步骤失败，返回阻塞原因和可重新推理的诊断。

5. H5 仍能玩，但不成为规则权威

H5 只显示后端投影和提交结构化动作。

H5 不允许根据中文文本推导世界状态，也不允许自己判断“火把是否可用”。

6. 保持向后兼容

P41 完成后，现有 H5 最小剧情闭环仍应可玩：

- 吃红果。
- 吃腐烂红果。
- 砍木头。
- 打磨斧头。
- 点燃干草。
- 制作火把。
- 用火把驱兽。
- 教同伴。
- 同伴持有知识但没有火把时不能凭空驱兽。

## 4. 本阶段不解决的问题

P41 不解决以下问题。

1. 不做完整地图寻路

采集、移动、地图格子、路径搜索放到后续地图阶段。

P41 只要求执行器能表示“需要采集动作”或“缺少物品”，不要求真的从地图上找资源。

2. 不做完整族群自动生产

族群库存、分工排程、多人协作由后续族群自动化阶段实现。

P41 只保证单个 Agent 的计划步骤可以通用执行。

3. 不做完整前端重构

前端横屏重构属于 `999_前端重构.md`。

P41 可以补必要的投影字段，但不做大 UI。

4. 不移除所有 Demo 场景

H5 defaultScenario 可以作为 fixture 保留，但其中的对象、反馈、执行效果必须逐步迁移到可注册配置。

5. 不做云端配置热更新

配置先以内置 fixture / C++ builder / JSON 加载接口的形式存在。

后续再决定是否引入热加载。

6. 不要求一次支持所有未来玩法

P41 要设计可扩展架构，并用当前火把、斧头、木头、野兽闭环验证。

毒蘑菇、房子、陷阱、船等作为测试或示例场景，不要求全部进入 H5。

## 5. 核心设计原则

### 5.1 知识产生候选，配置解释效果

Agent 是否知道某个行动，来自 KnowledgeClaim。

行动会产生什么世界变化，来自 EffectExecutionSpec / ReactionRule / EffectSemantics。

禁止把“知道”和“执行效果”混在同一个 if 分支里。

### 5.2 推理器只推理，不执行世界变化

AgentReasoner 负责：

- 目标选择。
- 候选行动选择。
- 前置条件检查。
- 计划链展开。

它不允许直接修改世界。

### 5.3 执行器只解释配置，不理解剧情

WorldEffectExecutor 负责把执行规格解释成 WorldChange。

它不应该知道“火把驱狼很合理”。

它只应该知道：

- 哪个对象数量减少。
- 哪个对象数量增加。
- 哪个威胁被降低或解决。
- 哪条安全反馈文本要输出。

### 5.4 条件判断权威属于 P27

所有复杂前置条件必须通过 P27 条件表达式系统。

禁止在 P41 里重新发明 `if state_key == sharpness` 这种条件判断体系。

简单数量条件可以被转换成表达式，但最终评价必须走统一 evaluator 或等价安全上下文。

### 5.5 世界状态修改必须走 WorldChange

任何行动不能直接改 DialogSessionState。

必须产生 WorldChange，再由 Applier 写回。

这样才能保证：

- 可复盘。
- 可测试。
- 可投影。
- 可诊断。

### 5.6 H5 只读投影，不反推规则

前端不能根据按钮文字、中文描述、知识文本判断规则。

前端只能：

- 显示后端给的可行动作。
- 发送结构化 action_id / object_key / target_key。
- 显示后端返回的世界变化和知识变化。

### 5.7 允许内置 bootstrap，但必须可替换

为了保证项目能跑，P41 可以保留内置配置 builder。

但所有内置内容必须通过同一套 registry 接口注册。

禁止核心系统直接读取硬编码数组作为唯一来源。

### 5.8 失败必须可解释

NPC 计划失败不能只返回 false。

必须说明：

- 缺少什么前置条件。
- 哪个步骤失败。
- 是缺材料、缺知识、条件不满足，还是配置错误。
- 是否可通过继续推理解决。

### 5.9 知识不等于资源权限

Agent 知道“火把能驱赶野兽”，只代表它可以把火把纳入行动计划；这不代表它有权使用玩家制作的火把。

P41 必须把三件事分开：

- `KnowledgeClaim`：Agent 是否知道某个对象能产生某种效果。
- `WorldObjectInstance.actor_quantities`：某个 Actor 实际拥有或持有多少该对象。
- `WorldObjectInstance.permitted_actor_keys`：某个 Actor 是否被明确授权使用该对象。

默认规则：

- 玩家制作出的物品默认属于玩家个人持有数量，不能自动变成营地共享物。
- 同伴、野生 NPC、其他族群 Agent 只能使用自己持有、被分配、被授权或后续明确放入公共库存的资源。
- H5 原型允许世界里显示“火把存在”，但 Agent 自主执行时必须使用 actor-scoped resource view，不能把全局数量当成自己可用数量。

这条规则是未来大世界的基础。后续做地图、营地、族群库存时，只能新增显式动作：

- 给同伴火把。
- 把火把放入营地公共库存。
- 分配工具给某个工作组。
- 通过交易、偷窃、拾取、掉落改变归属。

禁止为了让 NPC 看起来聪明而让它默认取用玩家物品。

## 7. 核心枚举

### 7.1 `EffectExecutionOpKind` 效果执行操作类型

用途：描述一个 effect 执行时会产生什么类型的世界变化。

是否允许 Unknown：允许，用于默认值和非法输入保护。

是否允许 TestOnly：允许，仅测试 fixture 使用。

枚举值：

| 值 | 中文含义 | 使用场景 |
|---|---|---|
| Unknown | 未知 | 默认值，validateBasic 必须拒绝 |
| ConsumeObjectQuantity | 消耗对象数量 | 吃红果、消耗干草、消耗木材 |
| AddObjectQuantity | 增加对象数量 | 生成火把、生成处理木材、生成房子 |
| SetObjectQuantity | 设置对象数量 | 特殊重置或剧情修复，不建议常用 |
| AddObjectStateNumber | 增加对象状态数值 | 增加 processed、恢复 sharpness |
| SetObjectStateNumber | 设置对象状态数值 | 设置斧头锋利度、设置威胁观察火 |
| AddObjectTag | 添加对象标签 | 添加 burning、sharp、resolved |
| RemoveObjectTag | 移除对象标签 | 移除 dull、dormant |
| ChangeActorNeed | 改变 Agent 需求 | 饥饿下降、寒冷下降、健康下降 |
| ChangeThreatLevel | 改变威胁等级 | 野兽靠近或退缩 |
| ResolveThreat | 解决威胁 | 火把驱兽、陷阱阻止攻击 |
| EmitWorldEvent | 发出世界事件 | 生成可复盘事件、外界打断事件 |
| QueueFollowupGoal | 排队后续目标 | 执行后触发继续采集、维修等 |
| TestOnly | 测试专用 | 单元测试覆盖非法或特殊路径 |

字符串转换规则：

- `toString` 使用 snake_case。
- `fromString` 非法值返回结构化错误，不得静默 Unknown。
- 配置必须使用字符串值，不允许使用整数。

禁止事项：

- 不允许配置作者新增未登记枚举。
- 不允许前端传入该枚举作为权威执行来源。

### 7.2 `EffectExecutionTargetKind` 执行目标类型

用途：描述执行操作作用于谁。

| 值 | 中文含义 | 使用场景 |
|---|---|---|
| Unknown | 未知 | 默认非法 |
| ActorSelf | 行动者自身 | 吃东西影响自己 |
| ActorTarget | 目标 Agent | 治疗别人、攻击别人 |
| ObjectSelf | 行动对象 | 使用斧头导致斧头变钝 |
| ObjectTarget | 目标对象 | 斧头作用于木头 |
| Location | 所在地点 | 生火生成火堆 |
| ThreatTarget | 目标威胁 | 驱赶野兽 |
| GroupOrTribe | 群体或族群 | 房子增加族群庇护容量 |
| RuntimeKey | 配置指定运行时 key | 高级配置扩展 |
| TestOnly | 测试专用 | 单元测试 |

规则：

- `ObjectSelf` 默认解析为 request.object_key。
- `ObjectTarget` 默认解析为 request.target_object_key。
- `ThreatTarget` 默认解析为 request.target_threat_key。
- `RuntimeKey` 必须显式配置 stable key。

### 7.3 `ExecutionValueSourceKind` 执行值来源

用途：说明执行操作中的 key 或数值从哪里来。

| 值 | 中文含义 | 使用场景 |
|---|---|---|
| Unknown | 未知 | 默认非法 |
| Constant | 常量 | 数值固定为 1 或 -1 |
| RequestObjectKey | 请求对象 key | 消耗当前对象 |
| RequestTargetKey | 请求目标 key | 作用于目标对象 |
| RequestActorKey | 请求 Agent key | 改变行动者需求 |
| EffectOutputKey | effect 输出 key | 生成 wood_processed / torch |
| RuntimeExpression | 运行时表达式 | 根据状态计算数量 |
| TestOnly | 测试专用 | 测试 |

禁止：

- 不允许使用中文文本作为值来源。
- 不允许从前端文本解析 key。

### 7.4 `ExecutionFailureKind` 执行失败类型

用途：统一描述行动执行失败原因。

| 值 | 中文含义 | 使用场景 |
|---|---|---|
| Unknown | 未知 | 默认非法 |
| SpecNotFound | 找不到执行规格 | effect 没配置 |
| ConditionNotMet | 条件不满足 | 缺前置条件 |
| MissingObject | 缺少对象 | 对象不存在 |
| MissingTarget | 缺少目标 | 需要 target 但没传 |
| InsufficientQuantity | 数量不足 | 没有材料 |
| StateNotMutable | 状态不可修改 | 试图改只读状态 |
| ForbiddenBySafety | 安全禁止 | hidden/raw 字段 |
| InvalidOperation | 操作非法 | op 与 target 不匹配 |
| ExpressionFailed | 表达式计算失败 | P27 返回错误 |
| PartialExecutionRejected | 部分执行被拒绝 | 原子性要求失败 |
| TestOnly | 测试专用 | 测试 |

规则：

- 所有失败必须带 safe_summary_zh_cn。
- 所有失败必须带 trace_keys。
- 失败后默认不修改世界状态。

### 7.5 `AgentStepExecutionMode` Agent 步骤执行模式

用途：决定 Agent 计划步骤如何执行。

| 值 | 中文含义 | 使用场景 |
|---|---|---|
| Unknown | 未知 | 默认非法 |
| DryRun | 只检查 | UI 显示下一步、调试 |
| ExecuteOneStep | 执行一步 | NPC 每回合行动 |
| ExecuteUntilBlocked | 执行到阻塞 | 后台模拟、小规模自动化 |
| ExecuteUntilGoalResolved | 执行到目标完成 | 后续族群自动化，需安全限制 |
| TestOnly | 测试专用 | 测试 |

P41 默认只实现 `ExecuteOneStep` 和 `DryRun`。

## 8. 核心 DTO / 数据结构

### 8.1 `EffectSemanticsDefinition`

中文含义：效果语义定义。

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| effect_key | string | 效果 key | 是 | 配置/fixture | 推理器 | 非空，snake_case，稳定 | 中文、空格 |
| display_zh_cn | string | 中文显示名 | 是 | 配置 | 投影 | 非空，安全文本 | hidden_truth |
| semantic_kind | EffectSemanticKind | 语义类型 | 是 | 配置 | 推理器 | 非 Unknown | 非法枚举 |
| target_scope | string | 目标范围 | 是 | 配置 | 推理器 | 枚举化或受限 key | 任意文本 |
| goal_affinities | vector<AgentGoalKind> | 可满足目标 | 是 | 配置 | 推理器 | 至少一个，除纯风险效果 | Unknown |
| state_deltas | vector<SemanticDelta> | 预期状态变化 | 是 | 配置 | 推理器 | domain/key 合法 | raw_state |
| risk_score | double | 风险分 | 是 | 配置 | 评分器 | 0-100 | 负数 |
| time_cost | uint32_t | 时间成本 | 是 | 配置 | 评分器 | >0 | 0 |
| confidence_floor | KnowledgeStatus | 最低知识状态 | 是 | 配置 | 推理器 | 非 Unknown | 随意降低 |
| required_target_kind | optional<string> | 目标类型要求 | 否 | 配置 | 推理器 | 受限 key | raw target |
| source_config_id | string | 来源配置 ID | 是 | Registry | 审计 | 稳定 | 前端创建 |
| version | string | 版本 | 是 | 配置 | 存档/复盘 | semver 或递增 | 空 |

ID 规则：

## 9. 核心服务接口

### 9.1 `EffectSemanticsRegistry`

职责：注册和查询效果语义。

接口：

```cpp
class EffectSemanticsRegistry {
public:
    Result<void> registerDefinition(const EffectSemanticsDefinition& definition);
    Result<EffectSemanticsDefinition> findByEffectKey(const std::string& effect_key) const;
    Result<std::vector<EffectSemanticsDefinition>> findByGoalKind(AgentGoalKind goal_kind) const;
    Result<void> validateAll() const;
};
```

规则：

- 纯内存注册表，不直接读文件。
- 文件加载由上层 config loader 做。
- 同一个 effect_key 重复注册必须失败，除非显式 allow_override 且来源是测试。
- `validateAll` 必须检查 forbidden key。

### 9.2 `EffectExecutionSpecRegistry`

职责：注册和查询执行规格。

接口：

```cpp
class EffectExecutionSpecRegistry {
public:
    Result<void> registerSpec(const EffectExecutionSpec& spec);
    Result<EffectExecutionSpec> findByEffectKey(const std::string& effect_key) const;
    Result<std::vector<EffectExecutionSpec>> allSpecs() const;
    Result<void> validateAgainst(const EffectSemanticsRegistry& semantics) const;
};
```

规则：

- 一个 effect_key 第一版只允许一个默认 spec。
- 后续可扩展为按环境、工具、物种选择 spec。
- spec 必须引用已登记 semantics。

### 9.3 `ExecutionConditionResolver`

职责：把执行规格和反应规则变成推理可用前置条件。

接口：

```cpp
class ExecutionConditionResolver {
public:
    Result<std::vector<PlanPrecondition>> preconditionsForEffect(
        const std::string& effect_key,
        const EffectExecutionSpecRegistry& execution_specs,
        const std::vector<reaction::ObjectReactionRule>& reaction_rules) const;
};
```

规则：

- 优先读取 EffectExecutionSpec.condition_refs。
- ReactionRule 可补充前置条件。
- 不允许根据 effect_key 写业务分支。
- 如果没有任何前置条件，必须明确返回空列表，而不是猜。

### 9.4 `WorldEffectExecutor`

职责：解释 EffectExecutionSpec，生成 WorldChange。

接口：

```cpp
class WorldEffectExecutor {
public:
    Result<WorldExecutionResult> execute(
        const WorldSnapshot& snapshot,
        const WorldExecutionRequest& request,
        const EffectExecutionSpecRegistry& specs,
        const condition::ConditionExpressionEvaluator& evaluator) const;
};
```

规则：

- 成功时只返回 WorldChange，不直接写 state。
- 失败时 changes 必须为空。
- `atomic=true` 时任一 operation 失败则全部失败。
- dry_run 只检查条件和可执行性，不产生 changes。

### 9.5 `AgentPlanStepExecutor`

职责：执行 AgentPlan 的一个步骤。

接口：

```cpp
class AgentPlanStepExecutor {
public:
    Result<AgentStepExecutionResult> executeStep(
        const WorldSnapshot& snapshot,
        const AgentStepExecutionRequest& request,
        const EffectExecutionSpecRegistry& specs) const;
};
```

规则：

- 不允许根据 `repel_beast`、`make_torch` 等写特判。
- 只能把 step 转成 WorldExecutionRequest。
- 执行结果来自 WorldEffectExecutor。

### 9.6 `AgentExecutionCoordinator`

职责：协调 Agent 推理计划和执行结果。

接口：

```cpp
class AgentExecutionCoordinator {
public:
    Result<AgentAutonomyResult> executePlan(
        const WorldSnapshot& snapshot,
        const AgentPlan& plan,
        AgentStepExecutionMode mode,
        const EffectExecutionSpecRegistry& specs) const;
};
```

规则：

- P41 只要求执行一步。
- 阻塞必须返回可投影原因。
- 不负责重新调用 Reasoner，重推理由上层循环控制。

### 9.7 `ExecutionAuditGate`

职责：扫描代码和配置中不允许出现的新硬编码模式。

接口可先用测试实现，不一定做运行时服务。

检查内容：

- `WorldInteractionService` 不应新增 `effect_key ==` 业务分支。
- `AgentPlanExecutorAdapter` 不应新增具体 effect 特判。
- H5 不应从 `reply_text` 推导 `torch` / `camp_fire`。
- 投影不应出现 forbidden keys。

## 10. 标准流程

### 10.1 玩家行为执行流程

1. H5 提交结构化行动

输入：action_key、object_key、target_key。

执行者：H5ExecutionBridge。

输出：WorldExecutionRequest。

失败：对象不存在或 action 不合法，返回安全错误。

2. 查找知识/反馈

输入：行动请求、当前 session。

执行者：DialogTurnService / LearningAdapter。

输出：可能产生知识经验。

失败：没有反馈模板，返回 condition_not_met。

3. 执行世界变化

输入：WorldSnapshot、WorldExecutionRequest。

执行者：WorldEffectExecutor。

输出：WorldExecutionResult。

失败：返回 ExecutionFailureKind，不修改状态。

4. 应用变化

输入：WorldChange 列表。

执行者：WorldChangeApplier。

输出：新 WorldSnapshot。

失败：返回 world_apply_failed。

5. 写回和投影

输入：新 snapshot。

执行者：WorldSnapshotAdapter、ProjectionMapper。

输出：H5 安全投影。

失败：不暴露 raw state。

### 10.2 Agent 自主行动流程

1. 生成目标

输入：WorldSnapshot、Needs、Threats。

执行者：AgentReasoner。

输出：AgentGoal。

2. 查找可用知识

输入：AgentGoal、Agent known claims。

执行者：AgentReasoner。

输出：ActionCandidate。

3. 解析前置条件

输入：effect_key。

执行者：ExecutionConditionResolver。

输出：PlanPrecondition。

4. 展开计划链

输入：候选行动、前置条件。

执行者：AgentReasoner。

输出：AgentPlan。

5. 执行一步

输入：AgentPlanStep。

执行者：AgentPlanStepExecutor。

输出：AgentStepExecutionResult。

6. 应用变化或返回阻塞

输入：WorldExecutionResult。

执行者：AgentExecutionCoordinator。

输出：AgentAutonomyResult。

失败：should_replan=true 时上层可再次推理。

### 10.3 外界打断流程

1. 当前计划执行中。

2. ThreatEventBridge 产生外界打断。

3. GoalExecutionSystem 暂停当前目标。

4. AgentReasoner 生成更高优先级目标。

5. AgentPlanStepExecutor 执行反制步骤。

6. 成功后 GoalExecutionSystem 恢复原计划。

P41 只保证第 5 步可通用执行。

## 11. 典型玩法和工程场景

### 11.1 火把驱兽闭环

前提：

- 同伴知道火把能驱兽。
- 同伴没有火把。
- 世界没有火把。

期望：

- 同伴不能凭空驱兽。
- 推理器发现缺少火把。
- 如果同伴知道制作火把，计划链包含制作火把。
- 如果缺处理木材，继续找砍木头。
- 如果斧头钝，继续找打磨。

### 11.2 毒蘑菇扩展示例

配置：

- poisonous_mushroom + eat -> poison。
- normal_mushroom + eat -> restore_hunger。
- dried_poison_mushroom + use -> medicine_hint。

期望：

- 新增毒蘑菇不需要改 WorldInteractionService。
- 只通过 EffectExecutionSpec 配置中毒或恢复饥饿。

### 11.3 斧头耐久循环

配置：

- axe + use wood -> cut_wood。
- cut_wood 消耗 wood，增加 wood_processed，降低 axe.sharpness。
- whetstone + use axe -> restore_sharpness。

期望：

- 斧头钝时执行失败。
- Agent 可以推理先打磨再砍木头。
- 不需要 `if effect_key == cut_wood`。

### 11.4 房子御寒链路

配置：

- house -> provide_warmth。
- build_house 需要 wood_processed >= 3。
- cut_wood 产生 wood_processed。

期望：

- 天冷时 Agent 比较生火和建房。
- 紧急寒冷时选择短链路生火。
- 长期目标时可选择建房。

### 11.5 野兽绕行和外界打断

配置：

- fire_is_dangerous 是野兽本能知识。
- camp_fire 可产生 fire_visible。
- beast_shadow 看到 fire_visible 后降低 threat_level。

期望：

- 野兽不是消失，而是状态改变。
- 野兽行为也通过 Agent / EffectSpec 表达。

## 12. 与现有系统关系

### 12.1 与 P18 知识系统

- P41 读取 KnowledgeClaim 作为行动候选来源。
- P41 不修改知识置信度，除非通过学习闭环产生经验。
- 知识是否可给 AI 使用仍看 projection_flags.usable_by_ai。

### 12.2 与 P27 条件表达式

- P41 条件判断必须走 P27 evaluator。
- 不允许新增第二套 condition_key 语法。
- 简单数量条件也要能转换为表达式或统一 safe fact。

### 12.3 与 P28 对象反应系统

- P41 可以读取 ReactionRule。
- ReactionRule 应补充 produces effect_key / execution spec linkage。
- 不允许 ReactionPlanningAdapter 继续靠产品 ID 猜 effect。

### 12.4 与 P28.5 配置流水线

- P41 的 semantics/spec 最终应进入配置包校验。
- 当前可先内置 fixture，但接口必须面向配置。

### 12.5 与 P29 危险系统

- Threat 变化必须通过 WorldChange。
- 危险系统不直接扣隐藏 HP。
- 威胁解决、降低、升级都必须可投影。

### 12.6 与 P39 Agent 推理

- P39 的 candidate 生成保留。
- 前置条件来源改为 ExecutionConditionResolver。
- AgentReasoner 不再直接维护 effect 特定前置条件。

### 12.7 与 P40 动态执行

- P40 的目标暂停/恢复保留。
- P41 提供真实可执行 step。
- P40 不再需要为某个 effect 写特殊执行。

### 12.8 与 H5

- H5 继续作为可玩验证壳。
- H5 不做规则判断。
- H5 可显示缺失材料、阻塞原因、下一步计划。

## 13. 安全边界

P41 禁止输出或配置以下字段：

- hidden_truth
- true_trait
- real_effect
- raw_state
- actual_hp
- true_hp
- hp_delta
- death
- kill
- corpse
- loot
- drop
- random_damage
- frontend_unlock
- direct_state
- internal_memory_raw
- private_reasoning
- secret_goal

额外规则：

- 执行反馈必须是安全中文摘要。
- Trace key 可以暴露工程路径，但不能暴露隐藏真相。
- 前端不能拿到可直接改状态的字段。
- 配置不能声明“直接写 session”。

## 14. 错误实现禁止清单

P41 明确禁止以下实现。

1. 不允许在 WorldInteractionService 新增：

```cpp
if (effect_key == "new_effect") { ... }
```

2. 不允许在 AgentPlanExecutorAdapter 只特判某几个 effect。

3. 不允许在 Reasoner 里写：

```cpp
if (effect == "make_torch") require camp_fire;
```

4. 不允许从中文 `reply_text` 推导世界对象是否存在。

5. 不允许前端根据按钮文字判断行动是否可执行。

6. 不允许新增物品时必须改核心执行器。

7. 不允许执行失败后部分状态已经被修改。

8. 不允许为了测试修改生产逻辑。

9. 不允许把 H5 Demo fixture 当作最终引擎配置格式。

10. 不允许绕过 WorldChangeApplier 直接写 DialogSessionState。

## 15. 子阶段任务拆分

### P41-1：基础枚举与 DTO

目标：定义通用执行相关枚举、DTO、validateBasic。

交付：

- `EffectExecutionOpKind`
- `EffectExecutionTargetKind`
- `ExecutionValueSourceKind`
- `ExecutionFailureKind`
- `AgentStepExecutionMode`
- `EffectExecutionSpec`
- `EffectExecutionOperation`
- `WorldExecutionRequest`
- `WorldExecutionResult`

验收：

- 枚举 roundtrip。
- DTO validateBasic。
- forbidden key 拒绝。

### P41-2：效果语义与执行规格注册表

目标：让 effect semantics 和 execution spec 都通过 registry 查询。

交付：

- `EffectSemanticsRegistry` 改为可注册。
- `EffectExecutionSpecRegistry`。
- 当前火把/斧头/木头/红果效果迁移到内置 fixture 注册。

- 未登记 spec 查询失败。
- 当前 effect 全部可查询。

### P41-3：前置条件统一解析

目标：替换 `requiredPreconditionsFor` 的硬编码兜底。

- ReactionRule linkage 支持。

- make_torch 前置条件来自 spec/reaction。
- repel_beast 前置条件来自 spec。
- Reasoner 不再直接写这些 effect 条件。

### P41-4：通用世界效果执行器

目标：让 WorldEffectExecutor 根据 spec 生成 WorldChange。

交付：

- `WorldEffectExecutor`。
- 操作解释：consume/add/set/state/tag/threat/need/event。
- dry_run。
- atomic rollback。

验收：

- cut_wood 不靠专用分支也能执行。
- make_torch 不靠专用分支也能执行。

## 18. 风险与控制

### 18.1 风险：一次重构范围过大

原因：P41 涉及推理、执行、H5、反应、测试。

后果：AI 上下文压力大，容易改坏旧功能。

控制：按 P41-1 到 P41-7 小步实现，每步单独测试。

### 18.2 风险：配置化后调试困难

原因：错误从 C++ 分支转移到配置。

后果：执行失败时不知道哪里错。

控制：每个 spec、op、condition 都必须带 trace_keys 和 safe_summary。

### 18.3 风险：为了兼容旧测试保留双轨逻辑

原因：旧 WorldInteractionService 分支还在。

后果：新旧两套执行同时存在，结果不一致。

控制：迁移一个 effect 就移除对应专用分支；确实不能同阶段移除时，必须登记审计风险，不能长期双轨。

### 18.4 风险：表达式系统接入不完整

原因：P27 条件表达式可能不支持所有运行时状态。

后果：又写临时 if。

控制：缺表达式能力时先补表达式上下文，不允许绕过。

### 18.5 风险：H5 可玩性短期下降

原因：规则更严格后，玩家需要先准备材料。

后果：测试时感觉“以前能点现在不能点”。

控制：投影必须清楚显示缺什么、为什么失败、下一步建议。

### 18.6 风险：低级 AI 继续复制旧模式

原因：看到旧代码里仍有 `effect_key ==`。

后果：架构继续变脏。

控制：P41 增加审计门禁和错误实现禁止清单。

## 21. 总结

P41 的目标不是“让当前 H5 多几个按钮”，而是把先驱者智能 Agent 与世界效果系统的关键闭环打实：

知识决定 Agent 能想到什么。

目标决定 Agent 为什么行动。

配置决定行动需要什么、会改变什么。

执行器解释配置并产生 WorldChange。

世界反馈再进入记忆、知识、投影和下一次推理。

完成 P41 后，后续新增物品、反应、NPC 行为、族群自动化，才不会继续依赖核心代码硬编码。


## 22. 长期世界观压力测试后的 P41 升级结论

经过高级文明、魔法世界、机械危机、神明回应、瘟疫传播、外交欺骗、时间循环、生态系统、枪械扩散等场景压力测试后，P41 必须从“当前智能 NPC 执行闭环”升级为“智能 Agent 与世界效果通用执行闭环”。

这个升级不推翻现有设计，而是把现有设计扩展到更高抽象层。

原设计关注：

- 火把。
- 斧头。
- 木头。
- 野兽。
- 同伴。
- 短期行动。
- 即时世界变化。

升级后的 P41 必须同时支持：

- 原始时代：采集、砍木头、生火、制作、驱兽。
- 农耕时代：种植、疾病、储粮、村庄建设。
- 工业时代：机械、燃料、工厂、污染、战争。
- 高级文明：星际采矿、巨构工程、文明能源、舰队协作。
- 魔法世界：魔力、法术、禁魔区、范围效果、反噬。
- 机械危机：自我复制、资源吞噬、网络污染、失控风险。
- 神话世界：信仰、祭祀、神明回应、概率奇迹。
- 生态系统：食物链、种群数量、延迟后果。
- 信息战：知识污染、欺骗、伪装、宣传、反宣传。
- 时间/因果扩展：多时间线、因果分支、跨线记忆预留。

因此 P41 的底层抽象不能只围绕“物品 effect”。

它必须围绕更通用的问题：

> 一个 Agent 或世界系统，因为某个目标和知识，触发一个可验证的 effect；该 effect 在某个尺度上、某个时间范围内、以某种确定或概率方式，改变世界状态，并产生可复盘反馈。

## 23. P41 长期设计总原则补强

### 23.1 Agent 多尺度原则

Agent 不只等于 NPC。

P41 之后，Agent 必须可以表示：

- 个体：同伴、狼、蜘蛛、机器人个体、法师。
- 小群体：一队猎人、一群狼、一支巡逻队。
- 组织：村庄、部落、城市、公司、宗教团体。
- 文明：人类文明、机械文明、外星文明。
- 超自然存在：神明、世界意志、魔法网络。
- 系统性 Agent：生态系统、机械蜂群、AI 集群。

不同尺度 Agent 使用同一套闭环：

知识 → 目标 → 推理 → 执行 → 世界变化 → 反馈。

区别只在于：

- 目标粒度不同。
- 行动耗时不同。
- 资源尺度不同。
- 世界变化范围不同。
- 决策频率不同。

禁止：

- 为高级文明另写一套 AI。
- 为野兽另写一套行为系统。
- 为神明另写一套奇迹系统。
- 为机械危机另写一套扩张系统。

它们都应该复用 P41 的 effect / condition / execution / trace 框架。

### 23.2 世界效果比物品效果更高一层

`EffectExecutionSpec` 不应该只描述“物品怎么变”。

它必须能描述：

- 物品变化。
- Agent 状态变化。
- 群体关系变化。
- 族群能力变化。
- 文明资源变化。
- 区域生态变化。
- 天气和环境变化。
- 世界规则变化。
- 知识传播或污染。
- 延迟、持续、周期性变化。

当前火把只是一个最小例子：

```text
火把驱兽 = 对 threat 施加 ResolveThreat 世界效果。
```

未来高级文明例子：

```text
戴森球建设 = 对 star_region / civilization_energy / megastructure_progress 施加长期世界效果。
```

魔法例子：

```text
火球术 = 对 target_agent、area_environment、caster_mana、fire_status 施加复合世界效果。
```

机械危机例子：

```text
自我复制 = 对 robot_population、metal_resource、system_pressure、civilization_threat 施加周期性世界效果。
```

### 23.3 即时、延迟、持续、周期、连锁五类效果统一建模

P41 必须预留五类效果。

1. 即时效果

例如：吃红果马上缓解饥饿。

2. 延迟效果

例如：吃毒蘑菇几小时后中毒；生态崩溃几天后出现。

3. 持续效果

例如：房子持续御寒；瘟疫持续感染；魔法护盾持续防御。

4. 周期效果

例如：农田每天产粮；工厂每 tick 产零件；机械蜂群周期复制。

5. 连锁效果

例如：火球点燃森林，森林火灾吓跑野兽，又导致生态改变。

P41 第一版不需要完整实现全部复杂调度，但 DTO 必须能表达，不得把架构锁死成“一次行动只产生一次变化”。

### 23.4 副作用和风险是一等公民

很多高级场景不是单一收益。

例如：

- 火球术：伤敌，也可能点燃森林。
- 枪械：提升战力，也升级战争烈度。
- 机械复制：提升生产，也提高机械危机风险。
- 祈雨：缓解旱灾，也可能消耗信仰和祭品。
- 外星技术：提升效率，也埋下后门。

所以 EffectExecutionSpec 必须区分：

- 主效果。
- 副作用。
- 风险效果。
- 条件触发效果。
- 失败反噬效果。

禁止只用一个 operations 列表混在一起，导致后续无法解释“这是我想要的结果，还是代价”。

### 23.5 可复盘随机原则

复杂世界需要概率，但随机不能破坏复盘。

任何概率效果都必须满足：

- 使用可复盘随机源。
- 记录 random_trace。
- 记录 seed 或 deterministic roll id。
- 支持 replay 验证。
- 前端只显示安全结果，不显示隐藏真相。

例如：

- 神明是否回应祈雨。
- 疾病是否传染。
- 外交欺骗是否被识破。
- 魔法是否反噬。
- 机械系统是否失控。

禁止：

- 在执行器里直接 `rand()`。
- 用不可复盘随机决定世界关键状态。
- 把真实概率和隐藏判定直接暴露给前端。

## 24. 多尺度 Agent 设计补强

### 24.1 `AgentScale` 扩展要求

现有 AgentScale 思路保留，但 P41 需要在执行层承认不同尺度。

建议扩展或明确以下尺度：

| AgentScale | 中文含义 | 示例 | 执行粒度 |
|---|---|---|---|
| MicroCreature | 微型生物 | 昆虫、微生物、纳米机器人 | 群体聚合执行 |
| SmallAnimal | 小型动物 | 蜘蛛、兔子、鸟 | 简单目标 |
| WildPredator | 野生掠食者 | 狼、熊、怪物 | 威胁/生存目标 |
| IndividualPerson | 个体人 | 同伴、专家 NPC、玩家角色 | 完整知识行动链 |
| Squad | 小队 | 巡逻队、猎人队、机器人队 | 小组战术 |
| Tribe | 族群 | 原始部落、村庄 | 族群知识和库存 |
| CityState | 城邦 | 城市、公司、宗教组织 | 组织资源和外交 |
| Civilization | 文明 | 人类文明、外星文明 | 长期战略和巨构 |
| HiveMind | 蜂群意识 | 机械蜂群、虫族 | 群体复制和扩张 |
| Supernatural | 超自然 Agent | 神明、魔法网络 | 世界规则级 effect |
| WorldSystem | 世界系统 | 生态、气候、瘟疫 | 周期和压力演化 |
| TestOnly | 测试专用 | 单元测试 | 测试 |

P41 不一定实现全部 AgentScale 的行为，但执行请求、目标范围、trace 不能假设 actor 一定是个体 NPC。

### 24.2 多尺度 Agent 的共同接口

无论是狼还是文明，都必须能被抽象为：

```text
AgentRuntimeView
- agent_key
- agent_scale
- knowledge_refs
- goal_slots
- resource_scope
- action_scope
- decision_frequency
- execution_authority
```

字段说明：

| 字段 | 中文含义 | 说明 |
|---|---|---|
| agent_key | Agent key | 稳定运行时引用 |
| agent_scale | Agent 尺度 | 个体、族群、文明等 |
| knowledge_refs | 可用知识 | 可以是个人知识、族群知识、文明知识 |
| goal_slots | 目标槽 | 当前目标和长期目标 |
| resource_scope | 资源范围 | 个人背包、族群库存、文明库存 |
| action_scope | 行动范围 | 当前位置、区域、星区 |
| decision_frequency | 决策频率 | 每回合、每日、每季、事件触发 |
| execution_authority | 执行权威 | 谁能写入世界变化 |

禁止：

- 在执行器里写死 actor_key == companion。
- 在执行器里假设 Agent 一定有个人背包。
- 在执行器里假设 Agent 一定在 forest_edge。

### 24.3 Agent 尺度如何影响推理

Agent 尺度影响：

1. 可见范围

狼只能看到附近火光。
文明能观察星区资源。
神明可能观察信仰网络。

2. 资源范围

同伴使用个人物品。
族群使用族群库存。
文明使用星际资源。

3. 目标时间尺度

狼目标是下一分钟生存。
族群目标是过冬。
文明目标是百年工程。

4. 执行动作粒度

同伴砍一棵树。
族群组织伐木队。
文明建造轨道采矿网。

P41 不需要实现完整策略 AI，但要让 effect execution 不阻碍这些差异。

## 25. 世界目标范围设计补强

### 25.1 `EffectExecutionTargetKind` 扩展

原 P41 设计中 `EffectExecutionTargetKind` 偏向对象和威胁。

升级后必须支持多尺度目标。

建议最终枚举：

| 值 | 中文含义 | 示例 |
|---|---|---|
| Unknown | 未知 | 默认非法 |
| ActorSelf | 行动者自身 | 吃东西影响自己 |
| ActorTarget | 目标 Agent | 治疗同伴、攻击怪物 |
| ObjectSelf | 行动对象 | 斧头变钝 |
| ObjectTarget | 目标对象 | 木头被砍 |
| InventoryScope | 库存范围 | 个人/族群/文明库存 |
| Location | 地点 | 营地生火 |
| Region | 区域 | 森林火灾、瘟疫区域 |
| WorldSystem | 世界系统 | 生态、气候、魔法网络 |
| ThreatTarget | 目标威胁 | 野兽、机械危机 |
| GroupOrTribe | 群体或族群 | 部落关系、族群库存 |
| Faction | 阵营 | 外交、战争、声誉 |
| Civilization | 文明 | 科技、能源、舰队 |
| Megastructure | 巨构 | 戴森球、轨道环 |

### 27.2 应用场景

- Immediate：消耗蘑菇。

3. 瘟疫

- Immediate：感染状态。
- Periodic：每隔一段时间传播。
- Cancel：隔离或治疗后停止。

4. 机械危机

- Periodic：复制机器人。
- Risk：提高系统压力。
- Max trigger：受资源和控制上限限制。

后续不需要重新设计文明 Agent，只要扩展目标生成和资源视图。

### 35.2 对知识系统的归并

原问题：知识传播、冲突、修正已有，但信息战和知识污染不足。

P41 修复：预留 KnowledgeEffectPayload 和知识类 op。

后续外星欺骗、专家教学、谣言传播都能进入同一闭环。

### 35.3 对族群系统的归并

原问题：族群作战和冲突已有，但外交、协议、依赖不足。

P41 修复：预留 RelationshipEffectPayload、Agreement target、Faction scope。

后续贸易、联盟、背叛不需要单独建一套系统。

### 35.4 对危险系统的归并

原问题：危险偏即时威胁。

P41 修复：威胁可以是 immediate、delayed、periodic、system_pressure。

后续瘟疫、机械危机、生态崩溃都能被视为广义危险。

### 35.5 对反应系统的归并

原问题：反应偏物品合成。

P41 修复：反应输出可以连接 EffectExecutionSpec，effect 可以作用于对象、Agent、群体、世界系统。

后续魔法仪式、工业生产、文明工程都可以是广义反应。

### 35.6 对 H5 原型的归并

原问题：H5 是最小可玩，带有演示性固定场景。

P41 修复：H5 继续做验证壳，但规则权威转移到通用执行层。

前端不再承担规则判断。

## 36. P41 升级后的最小实现边界

虽然设计必须长远，但实现仍然要小步。

P41 第一版必须实现：

1. 原始生存闭环完全迁移到通用执行：

- 吃红果。
- 吃腐烂红果。
- 砍木头。
- 打磨斧头。
- 点火。
- 制作火把。
- 驱兽。

2. DTO 支持未来扩展：

- 多尺度 scope。
- 副作用。
- 延迟/持续/周期。
- 随机策略。
- 知识/关系/能力操作。

3. 对未实现复杂功能 fail-safe：

- 不静默成功。
- 不乱改状态。
- 返回 unsupported_features。
- 投影安全提示。

4. 三个未来场景 dry-run 测试：

- 魔法火球范围副作用 dry-run。
- 机械复制资源链前置条件 dry-run。
- 族群能力解锁 dry-run。

这能证明架构没有被当前原始时代锁死。

## 39. 升级后的验收标准补充

P41 除原验收标准外，增加：

1. 文档必须明确 P41 是“智能 Agent 与世界效果通用执行闭环”，不是只服务 NPC。

2. DTO 能表达多尺度 Agent 和世界 scope。

3. DTO 能表达主效果、副作用、风险、失败反噬。

4. DTO 能表达延迟、持续、周期效果。

10. 审核报告必须单独说明：当前实现覆盖了哪些长期扩展点，哪些只是 DTO 预留。

## 40. 升级后的总结

升级后的 P41 是整个项目长期发展的底层拐点。

它要解决的不是“同伴会不会用火把”这一件事。

它要解决的是：

> 任何时代、任何世界观、任何 Agent，只要它拥有知识、目标和可执行条件，就能通过统一 effect 执行框架改变世界。

原始人制作火把、高级文明建造戴森球、法师释放火球、机械蜂群自我复制、神明回应祈祷、瘟疫传播、外交结盟、外星文明污染知识，本质上都应该落到同一个底层闭环：

1. 谁在行动。
2. 它为什么行动。
3. 它依据什么知识。
4. 它需要什么条件。
5. 它影响哪个世界范围。
6. 它产生哪些主效果。
7. 它产生哪些副作用和风险。
8. 它是否延迟、持续、周期或连锁。
9. 它如何被复盘。
10. 它如何反馈成新的认知和知识。

只要 P41 把这层做好，后续扩展世界观就不是推翻架构，而是增加配置、增加目标、增加知识、增加效果规格。

这才符合《先驱者》真正的核心：

不是做一个写死剧情的 H5 文本游戏，而是做一个知识会驱动文明演化的智能世界。

## 41. 升级后实现节奏建议

因为升级后的 P41 是长期底座，不能一次让研发 AI 把所有能力都写完。

建议实现仍然分三层推进。

### 41.1 第一层：必须真实落地

这一层解决当前硬编码风险。

必须完成：

- 当前 H5 原始生存 effect 迁移到通用执行配置。
- 同伴没有火把不能凭空驱兽。
- AgentPlanStep 通过通用执行器执行。
- WorldInteractionService 不再新增 effect 特判。
- 当前 H5 可玩闭环不破坏。

### 41.2 第二层：必须设计和校验，但可以 dry-run

这一层保证长期架构不被锁死。

必须完成 DTO、枚举、validate、dry-run：

- 多尺度 Agent。
- WorldScopeRef。
- 副作用和风险操作。
- 延迟、持续、周期效果。
- 范围目标选择。
- 可复盘随机策略。
- 知识污染和传播 payload。
- 关系和协议 payload。
- 世界规则 payload。

### 41.3 第三层：后续阶段实现

这一层不在 P41 全部实现。

后续阶段再做：

- 完整地图寻路。
- 完整 TemporalEffectRunner。
- 完整范围目标选择器。
- 完整外交系统。
- 完整文明级资源模拟。
- 完整魔法系统。
- 完整机械危机模拟。
- 完整时间线系统。

### 41.4 为什么这样拆

这样拆的好处是：

- 当前 H5 能继续玩。
- 当前硬编码风险能解决。
- 长期大世界扩展点不会缺。
- 研发 AI 不会一次面对过大上下文。
- 后续每个大系统都能复用 P41，而不是推翻 P41。

## 42. P41 与 P40 接线补充要求

本节是 P41 实现后的补充约束，防止后续研发误判“P40 已实现但没有接入”。

### 42.1 同伴自主行动必须走统一链路

同伴、野兽、未来 NPC 的自主行动不能在 H5 层或 WorldInteraction 层直接伪造结果。

标准链路必须是：

1. 知识 Claim 提供可行动经验。
2. P39 Reasoner 生成 AgentPlan。
3. P40 GoalExecutionSystem 负责 Driver、阻塞、材料和外界打断。
4. WorldInteractionService 把 DriverCommand 转为世界交互输入。
5. P41 WorldEffectExecutor 依据 effect spec 产生 WorldChange。
6. WorldChangeApplier 写回世界状态和 actor 资源归属。

### 42.2 资源归属约束

- 玩家制作出的物品默认不属于同伴。
- 同伴只能使用自己持有、被分配、被允许访问或公共无归属的资源。
- 同伴自己制作出的物品必须写入 `actor_quantities[actor_key]`。
- 不允许为了让同伴表现智能而给他隐藏默认火把。

### 42.3 派生材料约束

当前原型允许 `wood.processed` 派生为 `wood_processed` 材料事实。

后续扩展要求：

- 所有派生材料都应进入内容目录或材料别名配置。
- P39、P40、P41 对派生材料的解释必须一致。
- 不允许 P39 认为材料满足，但 P40 执行时又因为 key 口径不同而阻塞。

### 42.4 验收用例

最低验收必须包含：

- 同伴不知道火把知识时不会凭空驱兽。
- 同伴知道火把知识但没有火把时不会使用玩家火把。
- 同伴知道完整链路时，能从缺火把推导到制作火把。
- 制作出的火把归属同伴，而不是自动归属营地或玩家。
- H5 等待回合不会出现无来源的同伴火把。
