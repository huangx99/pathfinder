# 00_设计文档编写要求

## 1. 目的

这份文档是后续所有阶段设计文档的硬性编写要求。

从 P32 开始，任何新的阶段设计、补充设计、重构设计，都必须先按本文档自检。设计文档不是简单说明“要做什么”，而是要让后续研发 AI、低级 AI 或人工程序员在上下文有限的情况下，也能稳定写出符合架构的代码。

本文档的目标是避免再次出现 P31 初版那种问题：方向正确，但文档过薄，缺少字段、接口、流程、风险和实现门禁，导致后续研发容易偏离。

从现在开始，所有阶段设计和研发实现还必须同时遵守：

```text
doc/研发手册.md
```

`doc/研发手册.md` 负责约束实际研发过程：必须走 Command 管线、不得污染引擎、前端不得写规则、精灵和动画必须独立文件维护、审核报告必须进入 `doc/review/`。

## 2. 总原则

### 2.1 文档驱动研发

设计文档必须先于实现，且实现只能围绕设计展开。

文档必须回答：

- 这个阶段为什么存在。
- 它解决什么问题。
- 它明确不解决什么问题。
- 它新增哪些类、枚举、DTO、服务接口。
- 它如何接入已有系统。
- 它不允许破坏哪些已有边界。
- 它的测试和验收标准是什么。

如果文档没有写清楚，不能直接进入开发。

### 2.2 面向低上下文 AI

后续实现者可能不是当前设计者，也可能上下文记忆有限。

所以文档必须做到：

- 不依赖口头约定。
- 不依赖“看之前聊天记录”。
- 不依赖实现者自己猜架构意图。
- 不允许只写概念，不写接口和边界。
- 不允许把关键设计藏在一句话里。

设计文档应该让实现者只看当前文档和被引用的前置文档，就能知道该怎么写。

### 2.3 权威边界优先

所有系统设计必须明确权威来源。

例如：

- 规则结果只能来自规则管线。
- 条件判断只能来自 P27 条件表达式系统。
- 状态修改只能通过安全草稿、状态变更或指定 applier。
- 前端只能显示投影，不能推导规则结果。
- 复盘只能验证和诊断，不能重新实现规则。
- baseline 只能做回归对比，不能当运行时状态源。

每个阶段都必须写清楚“谁是权威，谁只是读取者”。

### 2.4 小步设计，但每一步要完整

小步不是少写。

小步的意思是：

- 本阶段范围要小。
- 但本阶段内部要完整。
- 该阶段涉及的类、枚举、接口、流程、测试必须写全。
- 不做的内容要明确放到后续阶段。

不能用“小步”为理由省略字段、接口、失败情况和测试。

## 3. 文档最低体量要求

体量不是绝对标准，但可以作为质量警报。

| 阶段类型 | 建议行数 | 说明 |
|---|---:|---|
| 新玩法规则系统 | 800-1300 行 | 例如 P28 对象反应、P29 危险与探索压力 |
| 重要基础设施系统 | 350-700 行 | 例如 P30 存档、P31 复盘诊断、P32 协议投影 |
| 配置/适配/桥接系统 | 250-500 行 | 例如 P28.5 配置化反应流水线 |
| 小型补丁或修正规范 | 120-250 行 | 仅限范围非常明确的补充说明 |

如果重要基础设施文档少于 300 行，必须在文档里解释为什么内容确实少，并仍然通过本文档的自检清单。

P31 初版只有 134 行，属于低于预期的情况，后续不应重复。

## 4. 标准文档结构

后续阶段设计文档默认使用以下结构。除非有充分理由，否则不得缺省。

### 4.1 阶段定位

必须说明：

- 当前阶段在总架构中的位置。
- 它接在哪些前置阶段之后。
- 它为哪些后续阶段打基础。
- 它是玩法系统、基础设施、协议投影、调试工具、训练接口，还是内容扩展。

示例要求：

```text
P32 是协议投影扩展阶段，不是前端规则阶段。
它读取后端权威结果，输出 H5 可读的安全投影。
它为 P33 正式 H5 最小界面和 P34 端到端 MVP 打基础。
```

### 4.2 为什么现在做

必须说明这个阶段为什么不能提前、为什么不能继续推迟。

需要回答：

- 前置能力是否已经具备。
- 后续阶段为什么依赖它。
- 如果不做，本项目会出现什么风险。

### 4.3 阶段目标

必须写清楚本阶段完成后系统具备什么能力。

目标必须是可验证的，不允许只写抽象词。

错误写法：

```text
让系统更完善。
```

正确写法：

```text
后端可以把知识、认知、条件摘要、复盘状态转换成 H5 只读投影；前端不需要也不允许自己判断规则结果。
```

### 4.4 本阶段不解决的问题

必须明确边界，防止范围膨胀。

每个“不做”最好说明原因或放到哪个后续阶段。

例如：

- 不做真实云存档，放到存储扩展阶段。
- 不做完整多命令 episode replay，等 replay runner 支持多命令后再做。
- 不做玩家 UI 美术，只做协议投影。

### 4.5 核心设计原则

必须列出 4-8 条原则。

原则必须和当前阶段有关，不能写空话。

每条原则应包括：

- 原则名称。
- 为什么需要。
- 会禁止什么错误实现。

例如：

```text
协议投影只能读，不能裁决：
P32 输出给 H5 的是安全展示结果，不能让前端根据投影反推规则、状态变化或隐藏真相。
```

### 4.6 模块划分

必须列出模块表。

表格至少包括：

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|

模块划分必须和代码 target、目录或服务边界尽量对应。

### 4.7 核心枚举

如果阶段涉及状态、结果、类型、动作、来源、严重度、决策等，必须设计枚举。

每个枚举必须写：

- 英文枚举名。
- 中文含义。
- 每个枚举值。
- 每个枚举值的使用场景。
- 是否允许 `Unknown`。
- 是否允许 `TestOnly`。
- 字符串转换规则。
- 禁止前端或配置随意扩展的说明。

枚举必须避免 magic string 到处飞。

### 4.8 核心 DTO / 数据结构

每个 DTO 必须逐字段说明。

字段说明表至少包括：

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|

如果字段是 ID、key、trace、summary、hash、version、confidence、state draft，必须额外说明：

- ID 来源。
- 是否稳定。
- 是否可序列化。
- 是否可暴露给前端。
- 是否允许为空。
- 是否参与 hash / replay / baseline。

### 4.9 核心服务接口

每个服务或类必须写：

- 类名。
- 中文职责。
- 输入。
- 输出。
- 失败情况。
- 是否修改状态。
- 是否可重复调用。
- 是否依赖随机数。
- 是否需要 trace。
- 与前置系统关系。

接口要写到实现者能知道函数该怎么命名和返回什么。

### 4.10 标准流程

必须把主流程拆成步骤。

每一步写：

- 输入。
- 执行者。
- 输出。
- 失败时产生什么错误或诊断。
- 是否继续后续步骤。

不能只写一条“执行流程”。

### 4.11 典型玩法或工程场景

如果是玩法系统，必须写 3-5 个玩法场景。

如果是基础设施系统，必须写 3-5 个工程场景。

例如 P31 应该写：

- 合法存档复盘通过。
- package hash 被篡改。
- replay event count 不一致。
- condition trace 缺失。
- golden baseline 过期。

### 4.12 与现有系统关系

必须逐个写清楚与已有系统的关系。

格式建议：

```text
### 与 P27 条件表达式
- 本阶段只能读取 condition trace 或调用 evaluator。
- 不允许重新发明 condition_key 判断。
- 输出给前端的只能是安全摘要。
```

至少覆盖：

- 直接依赖的前置阶段。
- 可能冲突的旧系统。
- 后续阶段如何使用。

### 4.13 安全边界

必须写清楚安全规则。

常见禁止内容：

- hidden_truth
- true_trait
- real_effect
- raw_state
- actual_hp
- true_hp
- hp_delta
- death
- kill
- corpse
- loot
- drop
- random_damage
- frontend_unlock
- direct_state

如果阶段有自己的禁止字段，必须追加。

### 4.14 错误实现禁止清单

必须明确“不允许这么写”。

这一节非常重要，必须保留。

示例：

- 不允许前端根据按钮文字推导规则结果。
- 不允许复盘层发现 mismatch 后自动修复存档。
- 不允许危险系统直接扣 HP。
- 不允许对象反应系统绕过 P27 条件表达式。
- 不允许 baseline 当运行时状态源。

### 4.15 子阶段任务拆分

每个阶段必须拆成 4-8 个子任务。

格式：

```text
### Pxx-1：基础类型与枚举
目标：定义枚举、DTO 骨架和基础校验。
交付：头文件、实现文件、单元测试。
验收：枚举 roundtrip、非法值拒绝、禁止字段拒绝。
```

子阶段必须从底层到上层排列：

1. 基础类型与枚举。
2. DTO 与校验。
3. 核心服务。
4. 桥接或投影。
5. 集成测试。
6. 安全门禁和回归。

### 4.16 测试计划

必须写测试计划，不允许只写“补测试”。

测试计划至少包括：

- 枚举 roundtrip。
- DTO validateBasic。
- 正常主流程。
- 关键失败流程。
- forbidden key 拒绝。
- 与前置系统的集成测试。
- 与后续系统的兼容测试。
- 回归测试范围。

复杂系统必须至少列 8-15 个测试点。

### 4.17 验收标准

验收标准必须可执行。

至少包括：

- 编译通过。
- 专项测试通过。
- 相邻系统测试通过。
- 全量测试通过。
- 文档自检通过。
- 没有破坏既有架构边界。

### 4.18 风险与控制

必须列 4-8 个风险。

每个风险写：

- 风险是什么。
- 为什么会发生。
- 会造成什么后果。
- 本阶段如何控制。

### 4.19 给实现阶段的硬性约束

文档最后必须写给研发 AI 的硬性约束。

例如：

- 不得重构无关模块。
- 不得绕过前置系统。
- 不得新增第二套规则判断。
- 不得把安全字段暴露到投影。
- 不得为了测试修改生产逻辑。
- 不得提交代码，除非用户明确要求。

## 5. 类、枚举、接口的详细度要求

### 5.1 枚举要求

每个枚举必须有：

- `Unknown`：用于默认值和非法输入保护。
- 正常业务值。
- 必要时的 `TestOnly`。
- `toString` / `fromString` 设计说明。
- 非法字符串返回结构化错误。

如果某个枚举不适合 `TestOnly`，必须说明原因。

### 5.2 ID 要求

所有 ID 必须说明：

- 是否使用 `StrongId<Tag>`。
- ID 由谁创建。
- 是否允许配置作者填写。
- 是否允许前端填写。
- 是否必须稳定。
- 是否参与 replay / save / baseline。

默认原则：

- 研发不能随便手写业务 ID。
- 配置 ID 必须走配置校验。
- 运行时 ID 必须由对应 builder 或 runtime 创建。
- 前端提交 ID 只能作为引用，不能作为权威创建来源。

### 5.3 DTO 要求

DTO 必须保持简单、可验证、可序列化。

必须包含：

- `validateBasic()` 规则。
- 禁止字段检查。
- 关键字段为空时的错误。
- 与 hash、version、trace 的关系。

### 5.4 服务接口要求

服务必须单一职责。

每个服务要说明：

- 是否纯函数式。
- 是否读状态。
- 是否写状态。
- 是否产生草稿。
- 是否产生事件。
- 是否产生 trace。
- 是否可以失败。
- 失败后状态是否已修改。

默认原则：能只读就只读；能输出草稿就不直接写状态。

## 6. 与 AI 研发配合的要求

### 6.1 文档必须减少 AI 上下文压力

文档要把实现任务拆小，避免一次让 AI 同时处理太多系统。

每个子任务应该满足：

- 写入范围明确。
- 依赖文件明确。
- 测试目标明确。
- 不需要重新理解整个项目。

### 6.2 文档必须避免“东丢一块西丢一块”

如果当前阶段需要接回旧系统，必须在同一文档中列完整：

- 哪些旧模块要接。
- 每个旧模块怎么接。
- 不接会有什么风险。
- 是否需要扫描门禁。

不能只写“建议后续接入”。

如果是本阶段必须的，就写成任务；如果不是本阶段必须的，就明确放到后续阶段。

### 6.3 文档必须给出实现顺序

实现顺序必须从底层到上层：

1. 枚举和 ID。
2. DTO。
3. 校验。
4. 核心服务。
5. 桥接服务。
6. 投影或协议。
7. 测试。
8. 文档和计划更新。

这样 AI 每一步上下文压力最小。

## 8. 后续执行规则

从 P32 开始执行以下规则：

1. 每个阶段设计前，先读取本文档。
2. 每个阶段设计完成后，用本文档做自检。
3. 如果阶段设计文档明显偏薄，必须先补文档，不能直接开发。
4. 如果实现中发现设计缺口，先补设计，再改代码。
5. 如果用户要求快速开发，也不能牺牲架构边界和安全边界。
6. 如果设计者判断某阶段确实内容少，必须在文档中解释原因，并仍然保留测试、验收和错误实现禁止清单。

## 9. P31 处理要求

P31 初版设计文档已经完成并实现通过，但文档详细度不足。

后续应补强 P31 文档，至少补齐：

- 核心枚举详细解释。
- `GoldenReplayBaseline` 字段说明。
- `GoldenReplayRequest` 字段说明。
- `GoldenReplayReport` 字段说明。
- `ReplayDiagnosticIssue` 字段说明。
- `GoldenReplayRunner` 流程细化。
- condition trace diff 策略。
- baseline diff 策略。
- 错误实现禁止清单。
- 子阶段任务拆分。
- 更完整测试计划。

补强 P31 不要求推翻当前代码；除非补文档时发现代码和设计冲突。

## 10. 总结

以后设计文档必须回到 P28/P29 那种方法：

- 先讲阶段定位。
- 再讲目标和边界。
- 再讲原则。
- 再逐个设计枚举、DTO、服务接口。
- 再讲流程、场景、系统关系和安全边界。
- 再拆任务、写测试、写验收、写风险和硬性约束。

设计文档必须能指导实现，而不是只描述想法。
