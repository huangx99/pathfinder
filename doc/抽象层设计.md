# 先驱者：抽象层设计

## 1. 设计目标

本文档定义《先驱者》第一阶段的核心抽象层。当前目标不是实现 Demo，也不是设计完整游戏内容，而是先确定规则引擎需要理解的领域对象、状态边界和类职责。

《先驱者》的核心链路是：

```text
对象 → 行为 → 反馈 → 记忆 → 知识 → 传播 → 文明能力
```

所有抽象都应服务于这条链路。任何暂时不能强化这条链路的系统，例如任务、装备、外交、宗教、市场和大型 AI 规划，第一阶段都不进入核心抽象。

## 2. 总体分层

建议将游戏逻辑拆成以下领域层：

```text
Game State Layer
  ├─ GameState
  ├─ WorldState
  ├─ Region
  ├─ Tile / Location
  └─ WorldObject

Definition Layer
  ├─ ObjectDefinition
  ├─ TraitDefinition
  ├─ ActionDefinition
  ├─ InteractionRule
  └─ CapabilityRule

Cognition Layer
  ├─ CognitionState
  ├─ PerceivedObject
  ├─ TraitKnowledge
  ├─ MemoryEntry
  ├─ KnowledgeEntry
  └─ ExperimentRecord

Actor & Tribe Layer
  ├─ Actor
  ├─ Pioneer
  ├─ Tribe
  ├─ NeedState
  └─ SurvivalState

Civilization Layer
  ├─ CivilizationState
  ├─ CivilizationStage
  ├─ Capability
  └─ UnlockCondition

Rule Engine Layer
  ├─ RuleEngine
  ├─ RuleContext
  ├─ RulePipeline
  ├─ CommandValidator
  ├─ InteractionResolver
  ├─ CognitionResolver
  ├─ PropagationResolver
  ├─ CapabilityResolver
  └─ RuleResult

Event Layer
  ├─ GameEvent
  ├─ FeedbackGeneratedEvent
  ├─ TraitDiscoveredEvent
  ├─ MemoryFormedEvent
  ├─ KnowledgeSharedEvent
  └─ CapabilityUnlockedEvent
```

## 3. 世界与状态类

### 3.1 GameState

`GameState` 是整局游戏的总状态，也是规则引擎执行时最重要的输入与输出对象。

职责：

- 持有当前世界状态。
- 持有先驱者状态。
- 持有族群状态。
- 持有认知状态。
- 持有文明状态。
- 记录当前时间、回合或 Tick。
- 提供状态版本号，支持调试、存档和复盘。

建议字段：

```text
gameId
stateVersion
currentTime
worldState
pioneer
tribe
cognitionState
civilizationState
eventHistory
```

设计原则：

- `GameState` 不负责规则计算，只保存状态。
- 所有规则变化必须通过规则引擎产生。
- 前端不应直接修改 `GameState` 的权威内容。

### 3.2 WorldState

`WorldState` 表示世界当前的客观状态。

职责：

- 管理区域。
- 管理对象实例。
- 管理环境状态。
- 提供对象查询能力。

建议字段：

```text
regions
objects
environmentState
timeOfDay
weather
dangerLevel
```

### 3.3 Region

`Region` 是一个可探索区域。

示例：

```text
营地附近
森林边缘
河岸
洞穴
荒原
遗迹
```

职责：

- 定义区域边界。
- 定义区域危险等级。
- 定义区域对象分布。
- 定义可见环境线索。

### 3.4 Tile / Location

如果游戏采用网格探索，则使用 `Tile` 表示地图格子，使用 `Location` 表示位置。

职责：

- 保存坐标。
- 保存地形。
- 保存可见性。
- 保存对象列表。
- 保存局部危险等级。

第一阶段可以保留该抽象，但不急于实现复杂寻路和地图生成。

### 3.5 WorldObject

`WorldObject` 是世界中的一个具体对象实例。

示例：

```text
这一丛红浆果
这一块石头
这一团火
这一只狼
这一片毒蘑菇
```

职责：

- 记录对象实例 ID。
- 指向对象定义。
- 记录当前位置。
- 记录当前状态。
- 记录是否被破坏、采集或耗尽。

建议字段：

```text
objectId
definitionId
location
currentState
quantity
isDestroyed
isCollected
```

关键原则：

```text
WorldObject 是实例。
ObjectDefinition 是类型。
玩家通常认知的是类型，而不是单个实例。
```

## 4. 定义类

### 4.1 ObjectDefinition

`ObjectDefinition` 定义对象类型的真实规则。

示例：

```text
red_berry
poison_mushroom
stone
branch
fire
wolf
herb
```

职责：

- 定义真实名称。
- 定义真实特性。
- 定义可执行交互。
- 定义基础反馈。
- 定义危险结果。
- 定义组合关系。

建议字段：

```text
definitionId
trueName
category
traits
supportedActions
interactionRules
visualTags
```

设计原则：

- `ObjectDefinition` 是规则引擎知道的真相。
- 玩家是否知道该对象，不影响 `ObjectDefinition` 的真实属性。
- 前端显示名称不直接使用 `trueName`，而是通过认知层决定。

### 4.2 TraitDefinition

`TraitDefinition` 是对象可被认知的最小语义单位。

示例：

```text
Edible
Poisonous
Flammable
Hard
Sharp
Warm
Dangerous
Medicinal
Predator
ToolMaterial
```

职责：

- 定义特性 ID。
- 定义特性名称。
- 定义特性类别。
- 定义该特性可能通过哪些行为被发现。
- 定义该特性对族群或文明的意义。

设计原则：

- 玩家不是一次性识别整个物品，而是逐步发现特性。
- 特性是记忆、知识和文明能力的基础。

### 4.3 ActionDefinition

`ActionDefinition` 定义玩家或实体可执行的行为类型。

第一阶段建议行为：

```text
Observe
Touch
Eat
Attack
Collect
Combine
ShareKnowledge
```

职责：

- 定义行为名称。
- 定义基础风险。
- 定义时间消耗。
- 定义体力消耗。
- 定义是否需要接近目标。
- 定义是否需要工具。
- 定义目标数量限制。

设计原则：

- 行为不是 UI 按钮，而是规则引擎可理解的实验方式。
- 玩家通过行为建立对世界的认知。

### 4.4 InteractionRule

`InteractionRule` 定义行为和对象之间的结算规则。

示例：

```text
Eat + RedBerry → HungerChanged + TraitDiscovered(Edible)
Eat + PoisonMushroom → Poisoned + TraitDiscovered(Poisonous)
Touch + Fire → HealthChanged + TraitDiscovered(Warm, Dangerous)
Combine + Stone + Branch → ToolCreated
```

职责：

- 判断行为作用于目标后会发生什么。
- 生成状态变化。
- 生成反馈。
- 生成可被认知的线索。
- 生成风险或收益。

设计原则：

- `InteractionRule` 不播放动画。
- `InteractionRule` 不直接通知 UI。
- 它只返回规则结果和事件。

### 4.5 CapabilityRule

`CapabilityRule` 定义文明能力的解锁规则。

示例：

```text
识别红浆果 + 确认可食用
→ 解锁安全采集红浆果

理解石头坚硬 + 理解树枝可加工 + 理解尖锐伤害
→ 解锁简易石矛

理解火会燃烧 + 理解火能驱兽 + 理解木材可燃
→ 解锁夜间营地防御
```

职责：

- 定义需要哪些知识。
- 定义需要多高置信度。
- 定义是否需要文明阶段前置。
- 定义解锁后的族群行为变化。

## 5. 认知系统类

### 5.1 CognitionState

`CognitionState` 是认知系统的总状态。

职责：

- 保存个人记忆。
- 保存族群知识。
- 保存对象识别进度。
- 保存特性置信度。
- 保存已尝试行为。
- 保存错误认知记录。

它需要回答：

```text
这个对象现在是否还是马赛克？
玩家知道它有毒吗？
族群是否已经学会避开它？
这个对象能否显示真实名称？
```

### 5.2 PerceivedObject

`PerceivedObject` 是玩家视角下的对象认知，不是真实对象本身。

示例：

```text
真实对象：red_berry

认知阶段：
未知物 A-17
红色植物
疑似可食用植物
红浆果
```

建议字段：

```text
objectDefinitionId
knownName
revealedTraits
confidence
visualRevealLevel
riskHints
```

设计原则：

- 前端显示应主要读取 `PerceivedObject`。
- 真实名称只有在认知达到阈值后才显示。

### 5.3 TraitKnowledge

`TraitKnowledge` 表示玩家或族群对某个对象特性的理解。

示例：

```text
red_berry -> edible -> confidence 90
poison_mushroom -> poisonous -> confidence 75
stone -> hard -> confidence 100
fire -> dangerous -> confidence 80
```

建议字段：

```text
subjectObjectDefinitionId
traitId
confidence
evidenceCount
lastUpdatedTime
sourceMemoryIds
ownerType
```

设计原则：

- 置信度用于表达“尚未完全确定”的认知过程。
- 错误认知可以存在，但需要能被后续证据修正。

### 5.4 MemoryEntry

`MemoryEntry` 是先驱者个人层面的经验记录。

示例：

```text
我吃了未知物 A，饥饿下降，没有受伤。
我触碰未知物 B，生命下降，出现中毒。
我用石头砸树枝，树枝断裂。
```

建议字段：

```text
memoryId
actorId
actionType
targetObjectIds
resultSummary
discoveredTraits
riskLevel
time
location
confidenceDelta
```

设计原则：

- 记忆是个人的，不等于族群知识。
- 记忆可以成为传播知识的证据来源。

### 5.5 KnowledgeEntry

`KnowledgeEntry` 是族群层面的稳定认知。

示例：

```text
红浆果可食用
毒蘑菇有毒
火可以驱赶狼
石头坚硬，可用于加工
```

建议字段：

```text
knowledgeId
subjectObjectDefinitionId
traitId
confidence
spreadLevel
sourceMemoryIds
unlockedBehaviors
```

设计原则：

- 知识是可传播、可复用、可影响族群行为的认知。
- 知识一旦进入族群层，就可以改变自动行为和文明能力。

### 5.6 ExperimentRecord

`ExperimentRecord` 记录一次具体实验。

职责：

- 记录谁做了实验。
- 记录对什么对象做了实验。
- 记录用了什么行为。
- 记录发生了什么反馈。
- 记录是否产生新认知。
- 记录是否造成伤害。
- 记录是否可复现。

用途：

- 调试。
- 回放。
- 族群学习。
- 错误认知修正。
- 未来 AI 分析。

## 6. 行为与交互类

### 6.1 ActionCommand

`ActionCommand` 是玩家本次输入的规则表达。

示例：

```text
观察 red_berry
触碰 fire
食用 poison_mushroom
组合 stone + branch
传播 red_berry_edible
```

建议字段：

```text
commandId
actorId
actionType
targetObjectIds
params
clientSequence
```

设计原则：

- 规则引擎只接受 `ActionCommand`。
- UI 操作必须先转换为 `ActionCommand`。
- 客户端不发送“结果”，只发送“意图”。

### 6.2 InteractionResult

`InteractionResult` 是一次交互的结算结果。

建议字段：

```text
success
stateChanges
feedbacks
events
discoveredTraits
riskConsequences
```

设计原则：

- 它描述规则结果，不描述表现方式。
- 前端根据事件和反馈决定如何展示。

### 6.3 Feedback

`Feedback` 是玩家能感受到的结果，也是认知增长的来源。

示例：

```text
看起来颜色鲜艳
摸起来灼热
吃下后饥饿下降
身体开始中毒
野兽远离火光
石头很硬
```

反馈类型：

```text
生理反馈：饥饿、受伤、中毒、恢复、疲劳
感知反馈：颜色、声音、气味、温度、硬度、运动
环境反馈：燃烧、扩散、吸引野兽、驱赶野兽、污染水源
认知反馈：发现特性、提升置信度、修正错误认知、形成名称
```

设计原则：

- 每次实验都应该给出有用反馈。
- 反馈应服务于玩家判断，而不是只给随机结果。

## 7. 角色与族群类

### 7.1 Actor

`Actor` 是可执行行为的实体基类。

可能包括：

```text
先驱者
族群成员
动物
敌对生物
```

第一阶段可以只实现先驱者，但保留 Actor 抽象有利于后续扩展。

### 7.2 Pioneer

`Pioneer` 是玩家控制的觉醒个体。

职责：

- 探索未知。
- 执行实验行为。
- 承担风险。
- 形成个人记忆。
- 传播知识。

建议字段：

```text
actorId
health
hunger
stamina
location
inventory
availableActions
```

### 7.3 Tribe

`Tribe` 是族群整体状态。

职责：

- 表示种群生存情况。
- 接收知识传播结果。
- 根据知识改变群体行为。
- 支撑文明阶段变化。

建议字段：

```text
population
food
health
safety
morale
knownKnowledgeIds
unlockedCapabilityIds
civilizationStage
```

设计原则：

- 第一阶段不建议细化每个族人。
- 族群先作为总体状态存在，用于反馈知识传播价值。

### 7.4 NeedState

`NeedState` 表示个体或族群的需求压力。

示例：

```text
饥饿
口渴
寒冷
疲劳
疾病
恐惧
```

### 7.5 SurvivalState

`SurvivalState` 表示族群生存指标。

示例：

```text
foodSupply
dangerLevel
diseaseLevel
shelterLevel
defenseLevel
```

知识传播后应该明显改善这些指标，否则传播系统会缺少反馈价值。

## 8. 文明与解锁类

### 8.1 CivilizationState

`CivilizationState` 表示当前文明状态。

职责：

- 保存当前文明阶段。
- 保存已解锁能力。
- 保存传播效率。
- 保存知识保存能力。
- 保存可探索的发展方向。

### 8.2 CivilizationStage

`CivilizationStage` 表示认知传播方式的阶段变化。

建议阶段：

```text
混沌感知
本能生存
工具认知
火与营地
符号文明
系统科学
高级文明
```

设计原则：

- 阶段不是传统等级。
- 阶段代表认知、传播和组织能力的提升。

### 8.3 Capability

`Capability` 是族群因知识积累而获得的能力。

示例：

```text
自动采集红浆果
避开毒蘑菇
夜间火堆防御
制作石矛
草药治疗
记录地图
```

职责：

- 改变族群行为。
- 改善生存状态。
- 解锁新行为或新区域。
- 作为文明阶段推进依据。

### 8.4 UnlockCondition

`UnlockCondition` 是能力或阶段的解锁条件。

它可以检查：

```text
是否掌握某知识
某知识置信度是否足够
是否拥有材料
文明阶段是否达到
族群传播能力是否足够
```

## 9. 规则引擎类

### 9.1 RuleEngine

`RuleEngine` 是规则执行总入口。

职责：

- 接收 `GameState` 和 `ActionCommand`。
- 调用规则管线。
- 修改权威状态。
- 返回 `RuleResult`。

输入输出：

```text
输入：GameState + ActionCommand
输出：RuleResult
```

设计原则：

- 不关心 UI。
- 不关心网络。
- 不关心动画。
- 只负责规则、状态和事件。

### 9.2 RuleContext

`RuleContext` 是一次规则执行的上下文。

建议字段：

```text
currentTime
randomSeed
configRepository
eventCollector
debugInfo
```

设计原则：

- 随机数必须可控，方便复盘。
- 调试信息不应污染核心状态。

### 9.3 RulePipeline

`RulePipeline` 定义一次行为的执行顺序。

建议流程：

```text
1. 校验行为
2. 执行交互
3. 应用状态变化
4. 生成反馈
5. 更新认知
6. 形成记忆
7. 尝试传播知识
8. 检查能力解锁
9. 输出事件
```

设计原则：

- 执行顺序必须稳定。
- 同一个状态和同一个命令应得到可复现结果。

### 9.4 CommandValidator

`CommandValidator` 判断行为是否合法。

校验内容：

```text
目标是否存在
距离是否足够
体力是否足够
对象是否可交互
是否拥有必要工具
当前阶段是否允许该行为
```

### 9.5 InteractionResolver

`InteractionResolver` 结算对象交互。

它回答：

```text
玩家对这个对象做这个动作，会发生什么？
```

### 9.6 CognitionResolver

`CognitionResolver` 处理认知增长。

它回答：

```text
这次反馈让玩家学到了什么？
哪些特性置信度增加？
对象显示是否变清晰？
是否形成新记忆？
```

### 9.7 PropagationResolver

`PropagationResolver` 处理知识传播。

它回答：

```text
这个记忆能否传播？
传播给多少族人？
是否形成族群知识？
是否改变族群行为？
```

### 9.8 CapabilityResolver

`CapabilityResolver` 处理文明能力解锁。

它回答：

```text
当前知识组合是否满足某个能力？
是否解锁新族群行为？
是否进入新文明阶段？
```

### 9.9 RuleResult

`RuleResult` 是规则执行结果。

建议字段：

```text
success
errorCode
events
stateVersion
debugInfo
```

## 10. 事件类

所有重要变化都应以事件表达，而不是静默修改状态。

基础事件：

```text
GameEvent
ActionStartedEvent
ActionRejectedEvent
FeedbackGeneratedEvent
TraitDiscoveredEvent
ConfidenceChangedEvent
MemoryFormedEvent
KnowledgeSharedEvent
ObjectRevealedEvent
CapabilityUnlockedEvent
CivilizationStageChangedEvent
```

示例：玩家食用红浆果后，可能产生以下事件：

```text
ActionStarted
HungerChanged
FeedbackGenerated
TraitDiscovered
ConfidenceChanged
MemoryFormed
ObjectPartiallyRevealed
```

设计原则：

- 事件用于前端表现。
- 事件用于调试和复盘。
- 事件用于测试规则是否正确。
- UI 不应反向推导规则结果。

## 11. 配置类

第一阶段建议使用 JSON、YAML 或表格驱动配置。

核心配置：

```text
ObjectConfig
TraitConfig
ActionConfig
InteractionConfig
CapabilityConfig
RegionConfig
```

示例：对象配置

```text
red_berry:
  traits: plant, edible, soft
```

示例：交互配置

```text
Eat + red_berry:
  hunger: -20
  discover: edible
```

示例：能力配置

```text
safe_berry_gathering:
  requires:
    red_berry.edible confidence >= 80
```

## 12. 第一阶段最小类集合

为了控制复杂度，第一阶段建议只落地这些类：

```text
GameState
WorldState
WorldObject
ObjectDefinition

Pioneer
Tribe

CognitionState
TraitDefinition
TraitKnowledge
MemoryEntry
KnowledgeEntry

ActionCommand
ActionDefinition
InteractionRule
InteractionResult
Feedback

RuleEngine
RuleContext
CommandValidator
InteractionResolver
CognitionResolver
PropagationResolver
CapabilityResolver
RuleResult

GameEvent
Capability
CapabilityRule
```

这些类已经足够支撑最核心体验：

```text
未知对象通过实验变成记忆，记忆通过传播变成知识，知识推动族群能力和文明阶段变化。
```

## 13. 暂缓设计的类

以下系统暂时不要进入第一阶段核心抽象：

```text
Quest
Skill
Equipment
NPCDialogue
TechnologyTree
Building
Profession
Market
Religion
Politics
Diplomacy
LargeAIPlanner
```

原因：

- 它们会稀释“认知世界”的核心玩法。
- 它们容易把项目拉回传统生存或文明模拟框架。
- 当前最需要验证的是认知规则是否成立。

## 14. 抽象关系总览

核心关系可以概括为：

```text
WorldObject --references--> ObjectDefinition
ObjectDefinition --has--> TraitDefinition
ActionCommand --uses--> ActionDefinition
ActionCommand --targets--> WorldObject
InteractionRule --matches--> ActionDefinition + ObjectDefinition
InteractionRule --produces--> Feedback + GameEvent
Feedback --updates--> TraitKnowledge
TraitKnowledge --forms--> MemoryEntry
MemoryEntry --canBecome--> KnowledgeEntry
KnowledgeEntry --satisfies--> CapabilityRule
CapabilityRule --unlocks--> Capability
Capability --changes--> Tribe / CivilizationState
```

## 15. 下一步设计重点

下一份文档建议继续细化以下五个核心抽象：

```text
1. WorldObject 和 ObjectDefinition 的边界
2. TraitDefinition 和 TraitKnowledge 的数据结构
3. MemoryEntry 到 KnowledgeEntry 的转换规则
4. ActionCommand 和 InteractionRule 的匹配规则
5. CapabilityRule 的解锁表达方式
```

其中优先级最高的是：

```text
WorldObject / ObjectDefinition / TraitDefinition
```

它们决定整个认知系统能否稳定扩展。
