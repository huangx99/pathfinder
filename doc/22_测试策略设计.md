# 22_测试策略设计

## 1. 设计目标

测试策略系统负责回答一个核心问题：

```text
以后研发如何证明规则引擎、配置、事件、认知、记忆、知识、传播、族群、文明、Agent、存档、复盘、协议都没有被改坏。
```

先驱者是规则驱动游戏，不是单纯 UI 驱动游戏。

因此测试的目标不是“页面能点”，而是：

```text
同一个输入。
同一个配置。
同一个随机种子。
同一个初始状态。
必须得到同一个规则结果、事件序列、状态哈希、复盘结果。
```

本章要保证：

```text
每个核心系统都有可执行测试入口。
每个规则变化都有回归测试。
每个配置变化能被校验和对比。
每个事件、错误码、状态变更都有断言。
每次存档和复盘都能验证一致性。
每个 Agent 决策都能验证权限、尺度、认知和确定性。
每个前后端协议都能验证幂等、重连、权限和版本兼容。
后续接入强化学习时，可以批量、确定性、可复盘地运行环境。
```

核心原则：

```text
测试只验证规则，不替代规则。
测试不绕过 RuleEngine 修改状态。
测试输入和期望输出必须可版本化。
测试失败必须能定位到配置、命令、管线阶段、事件、状态差异或随机差异。
```

## 2. 不解决的问题

| 不解决的问题 | 原因 | 后续阶段 |
|---|---|---|
| 具体 C++ 测试框架选型 | 本章只定义策略，不绑定 GoogleTest / Catch2 | 实现阶段 |
| CI/CD 平台脚本 | 本章只定义可自动化标准 | 工程阶段 |
| 每个内容规则的完整用例 | 本章定义模板和覆盖要求 | 内容开发阶段 |
| 性能优化实现 | 本章只定义性能测试指标 | 性能优化阶段 |
| 前端视觉截图测试 | 本章只覆盖协议和规则投影 | UI/UX 测试阶段 |
| 强化学习算法评估 | 本章只保证环境可测 | 强化学习阶段 |

禁止做法：

```text
禁止测试直接修改 GameState 伪造通过。
禁止测试绕过 RuleEngine 调内部 Resolver。
禁止测试只看 debug_message，不断言错误码。
禁止用不固定随机源测试核心规则。
禁止只保存最终状态，不保存命令、事件、随机、trace。
禁止把前端本地推导当作规则期望。
禁止让 Agent 测试读取 Hidden / DebugOnly 数据后再声称普通 Agent 可用。
禁止失败后只输出“结果不一致”，必须输出可定位差异。
```

## 3. 测试总原则

### 3.1 确定性优先

所有核心规则测试必须满足：

```text
同输入 + 同配置 + 同 seed = 同结果。
```

必须记录：

```text
config_version_id。
state_version_before。
command_id。
transaction_id。
random_seed。
random_draws。
event_ids。
state_hash_before。
state_hash_after。
```

### 3.2 权威入口统一

测试必须通过正式入口：

```text
RuleEngine.submitCommand。
RuleEngine.query。
ReplayService。
ConfigService。
EventService。
AgentRuntime。
ProtocolService。
```

不能直接调用内部 Resolver 来构造“假成功”。

### 3.3 失败可诊断

任何失败都必须能回答：

```text
哪个测试失败。
哪个命令失败。
哪个管线阶段失败。
哪个断言失败。
实际事件和期望事件差异是什么。
实际状态哈希和期望状态哈希差异是什么。
随机流在哪一步不同。
配置版本是否不同。
Agent 是否使用了过期观察。
协议是否重复提交或越权。
```

### 3.4 测试数据版本化

测试数据必须和配置一样可版本化：

```text
test_case_id。
test_suite_id。
fixture_version。
expected_version。
config_version_id。
engine_version。
protocol_version。
```

### 3.5 小测试验证局部，大测试验证链路

测试分层必须清晰：

```text
单元测试验证小组件。
契约测试验证接口稳定。
组合测试验证系统协作。
管线测试验证事务和阶段顺序。
场景测试验证玩法链路。
复盘测试验证长期确定性。
协议测试验证前后端边界。
压力测试验证规模。
模糊测试验证边界。
性质测试验证永远不能破坏的不变量。
```

## 4. 测试分层

| 层级 | 英文 | 中文 | 目标 | 示例 |
|---|---|---|---|---|
| L0 | StaticCheck | 静态检查 | 文档、配置、枚举、ID、引用完整性 | 枚举值重复、ID 格式错误 |
| L1 | UnitTest | 单元测试 | 单个类或函数行为正确 | ErrorCode 映射、ID 解析 |
| L2 | ContractTest | 契约测试 | 接口输入输出稳定 | RuleEngineResult 字段完整 |
| L3 | ConfigTest | 配置测试 | 配置可加载、可校验、可迁移 | TraitDefinition 缺字段失败 |
| L4 | CombinationTest | 组合测试 | 多系统协作 | 吃果实 -> 反馈 -> 记忆 |
| L5 | PipelineTest | 管线测试 | 阶段顺序、事务、草稿提交 | 阻断后不提交状态 |
| L6 | ScenarioTest | 场景测试 | 小型玩法过程 | 探索、教学、共同作战 |
| L7 | ReplayTest | 复盘测试 | 命令序列可重放 | 状态哈希一致 |
| L8 | ProtocolTest | 协议测试 | H5 / 后端边界 | 幂等、重连、权限 |
| L9 | AgentTest | Agent 测试 | 决策、观察、尺度、权限 | 蜘蛛、狼、人类、外星文明 |
| L10 | PropertyTest | 性质测试 | 永远不变量 | 置信度不越界、生命不负数 |
| L11 | FuzzTest | 模糊测试 | 自动发现边界问题 | 随机合法命令序列 |
| L12 | PerformanceTest | 性能测试 | 规模、耗时、内存 | 大量 Agent 和事件 |

执行顺序建议：

```text
StaticCheck -> UnitTest -> ContractTest -> ConfigTest -> CombinationTest -> PipelineTest -> ScenarioTest -> ReplayTest -> ProtocolTest -> AgentTest -> PropertyTest -> FuzzTest -> PerformanceTest
```

## 5. 测试基础类型

### 5.1 TestId

| 类型 | 中文 | 格式 | 示例 |
|---|---|---|---|
| TestId | 测试 ID | test_{domain}_{name}_{version} | test_agent_spider_flee_v1 |
| TestSuiteId | 测试套件 ID | suite_{domain}_{name} | suite_pipeline_core |
| FixtureId | 夹具 ID | fixture_{domain}_{name}_{version} | fixture_unknown_fruit_v1 |
| ScenarioId | 场景 ID | scenario_{domain}_{name}_{version} | scenario_first_poison_v1 |
| AssertionId | 断言 ID | assert_{domain}_{name} | assert_event_created |
| GoldenId | 金标准 ID | golden_{scenario}_{version} | golden_first_poison_v1 |

约束：

```text
测试 ID 必须稳定。
测试 ID 不能使用自增整数。
测试 ID 变更必须说明原因。
```

### 5.2 TestCase

`TestCase` 是最小可执行测试单元。

| 字段 | 类型 | 中文 | 说明 |
|---|---|---|---|
| test_id | TestId | 测试 ID | 稳定标识 |
| title | string | 标题 | 人类可读 |
| domain | TestDomain | 测试领域 | pipeline / agent / replay 等 |
| level | TestLevel | 测试层级 | L0-L12 |
| fixture_id | FixtureId | 夹具 ID | 初始环境 |
| steps | list<TestStep> | 步骤 | 命令、查询、Agent tick 等 |
| assertions | list<TestAssertion> | 断言 | 期望结果 |
| random_seed | RandomSeed | 随机种子 | 确定性 |
| config_version_id | ConfigVersionId | 配置版本 | 锁定配置 |
| expected_golden_id | optional<GoldenId> | 金标准 | 可选 |
| tags | list<string> | 标签 | smoke / regression / slow |
| owner | string | 负责人 | 维护者 |

### 5.3 TestFixture

`TestFixture` 定义测试初始状态。

| 字段 | 类型 | 中文 | 说明 |
|---|---|---|---|
| fixture_id | FixtureId | 夹具 ID | 稳定标识 |
| config_refs | list<ConfigRef> | 配置引用 | 对象、特性、规则、Agent 等 |
| initial_state | StateSnapshot | 初始状态 | 可最小化 |
| initial_events | list<EventEnvelope> | 初始事件 | 可选 |
| initial_agents | list<AgentState> | 初始 Agent | 可选 |
| initial_memories | list<MemoryRecord> | 初始记忆 | 可选 |
| initial_knowledge | list<KnowledgeClaim> | 初始知识 | 可选 |
| random_seed | RandomSeed | 随机种子 | 默认种子 |
| fixture_hash | Hash | 夹具哈希 | 防篡改 |

要求：

```text
Fixture 必须尽量小。
Fixture 必须可独立加载。
Fixture 不允许依赖本地机器状态。
```

### 5.4 TestStep

| 字段 | 类型 | 中文 | 说明 |
|---|---|---|---|
| step_id | string | 步骤 ID | 测试内唯一 |
| step_type | TestStepType | 步骤类型 | command / query / agent_tick / replay / protocol |
| input | Variant | 输入 | 具体请求 |
| expected_result_type | RuleResultType | 期望结果类型 | Success / Blocked / Error |

| 字段 | 类型 | 中文 | 说明 |
|---|---|---|---|

| tolerance | optional<float> | 容差 | 浮点或性能 |
| message_key | string | 失败信息键 | 本地化 |

### 5.6 TestReport

| 字段 | 类型 | 中文 | 说明 |
|---|---|---|---|

## 7. 单元测试策略

单元测试覆盖最小类型和纯逻辑：

```text
ID 解析和格式。
枚举序列化和反序列化。
错误码映射。
Result<T> 成功和失败包装。
Hash 计算。
StateVersion 比较。
ConfigVersionId 比较。
表达式求值。
条件匹配。
ActionTarget 解析。
```

必须保证：

```text
无随机。
无 IO。
无真实时间。
无全局状态。
```

示例：

| 测试 | 断言 |
|---|---|
| EntityId 格式非法 | 返回 id_invalid_format |
| ErrorCode 缺少中文说明 | 静态检查失败 |
| EventVisibility 序列化 | Hidden -> hidden |
| Confidence Clamp | 置信度不能超过 1.0 |

```text

EventService。
ReplayService。
ConfigService。
SnapshotService。
```

## 10. 规则管线测试策略

管线测试必须验证阶段顺序和事务边界。

测试重点：

```text
ValidateCommand。
ResolveActor。
ResolveTarget。
CheckPreconditions。
ResolveInteraction。
ResolveFeedback。
UpdateCognition。
UpdateMemory。
UpdateKnowledge。
ResolvePropagation。
UpdateTribe。
UpdateCivilization。
GenerateEvents。
CommitTransaction。
```

必须断言：

```text
前置条件失败后不提交状态。
Resolver 只能输出草稿，不能直接改状态。
事件必须在事件闸门后输出。
状态哈希在提交后变化。
StateVersion 只在成功提交后递增。
RuleResultType 与 ErrorCode 匹配。
```

管线失败必须输出：

```text
failed_stage。
blocking_error_code。
state_hash_before。
state_hash_after。
draft_count。
committed_draft_count。
event_count。
trace_id。
```

## 11. 组合测试策略

组合测试用于验证多个系统串起来是否正确。

首批必测组合：

| 组合 | 链路 |
|---|---|
| 未知果实试吃 | Command -> Interaction -> Feedback -> Memory -> Cognition -> KnowledgeCandidate |
| 火加木棍 | Command -> InteractionRule -> ObjectReaction -> NewObject -> Event |
| 教学传播 | Knowledge -> Propagation -> ReceiverCognition -> TribeImpact |
| 共同作战 | Agent -> Command -> TribeState -> BattleEvent -> SplitRisk |
| 文明能力使用 | CivilizationCapability -> CommandAvailable -> RulePipeline -> Event |
| 错误认知修正 | WrongKnowledge -> NewEvidence -> ConflictResolution -> KnowledgeUpdated |
| 延迟反馈 | Command -> DelayedFeedback -> MemoryUpdate -> Event |
| 对象风险 | TraitRisk -> Feedback -> CautionMemory -> AgentDecision |

组合测试必须同时断言：

```text
结果类型。
错误码。
事件类型。
事件顺序。
状态变化。
认知变化。
记忆变化。
知识候选。
复盘记录。
```

## 12. 场景测试策略

场景测试由多个命令组成，模拟一小段可玩过程。

标准场景结构：

```text
LoadFixture。
SubmitCommand / AgentTick 多步。
QueryProjection。
AssertState。
AssertEvents。
SaveGame。
Replay。
AssertHash。
```

首批场景：

| 场景 | 目标 |
|---|---|
| 第一次吃未知果实 | 验证探索、反馈、记忆、认知 |
| 第一次使用火把 | 验证对象反应和工具认知 |
| 教会族人避开毒果 | 验证知识传播和接收者认知 |
| 狼群袭击营地 | 验证 SmallCreature / Pack Agent 和共同作战 |
| 蜘蛛避火 | 验证 Micro Agent 和低认知层限制 |
| 族群分裂风险上升 | 验证士气、资源、忠诚、分裂风险 |
| 文明能力成熟 | 验证知识、族群、条件集合、能力可用 |
| 外星文明探针观察 | 验证 Interstellar Agent 权限和远程影响限制 |

## 13. Agent 测试策略

Agent 测试是本阶段重点之一。

必须覆盖：

```text
AgentDefinition 加载。
AgentScale 默认边界。
AgentCognitionBand 知识使用限制。
AgentEmbodiment 绑定范围。
AgentHierarchyLink 层级影响。
AgentObservation 过滤。
AgentActionSpace 生成。
AgentIntent 到 CommandEnvelope 适配。
AgentPolicy 确定性。
AgentDecisionTrace 可解释性。
AgentReplayLocked 不重新决策。
```

### 13.1 多尺度 Agent 测试矩阵

| Agent | Scale | CognitionBand | 必测 |
|---|---|---|---|
| 蜘蛛 | Micro | Reflex | 不能教学、不能抽象推理、可避火 |
| 狼 | SmallCreature | Instinct | 可追猎、逃跑、召集狼群 |
| 先驱者 | Individual | ToolAware | 可吃、用、观察、教学 |
| 人类族群 | Tribe | SocialLearning | 可传播、共同作战、迁移 |
| 外星文明 | Interstellar | Transcendent | 可远程观察，但不能直接改状态 |

### 13.2 Agent 权限测试

必须断言：

```text
普通 Player Agent 看不到 Hidden。
Creature Agent 看不到文明 Debug 信息。
Training Agent 使用 TrainingMasked。
Debug Agent 才能 OmniscientDebug。
Interstellar Agent 的远程观察仍受权限、科技、距离、资源限制。
```

### 13.3 Agent 确定性测试

同一个 Agent 必须满足：

```text
同 AgentState。
同 Observation。
同 ActionSpace。
同 PolicyConfig。
同 RandomSeed。
= 同 AgentDecision。
```

### 13.4 Agent 层级测试

必须验证：

```text
Pack Agent 只能影响狼个体目标，不能直接改狼位置。
Tribe Agent 只能影响族人目标和可选行动，不能直接改个体生命值。
Civilization Agent 只能发起政策、建设、战争、外交、探针等命令。
顶级外星 Agent 不能直接给低级 Agent 注入知识，必须通过事件、观察、教学、影响或科技渠道。
```

## 14. 认知链路测试策略

认知链路测试验证游戏核心体验。

完整链路：

```text
未知对象 -> 尝试行为 -> 反馈 -> 记忆 -> 认知更新 -> 知识候选 -> 传播 -> 族群认知变化 -> 文明能力候选
```

必须测试：

```text
对象形态不是认知重点。
认知重点是能否食用、能否使用、效果、风险、条件。
错误认知可以存在。
错误认知可以被新证据修正。
记忆可以衰减、强化、压缩、解压。
知识必须有证据和置信度。
传播不能复制标签，必须传递知识声明。
```

断言示例：

| 行为 | 断言 |
|---|---|
| 吃未知果实 | 生成 FeedbackEvidence |
| 反馈中毒 | MemoryRecord 包含负面效果 |
| 多次证据一致 | CognitionConfidence 上升 |
| 教给族人 | ReceiverCognition 生成对应条目 |
| 新证据冲突 | KnowledgeClaim 进入 Disputed 或 Revised |

## 15. 存档与复盘测试策略

存档测试必须验证：

```text
SaveGamePackage 完整。
StateSnapshot 可恢复。
EventLog 连续。
CommandReplayLog 完整。
RandomReplayLog 完整。
TraceLog 可追踪。
AgentState 和 AgentDecisionRecord 已保存。
ConfigVersionId 匹配。
Hash 校验通过。
```

复盘测试必须验证：

```text
同命令序列复盘后 state_hash 一致。
同事件序列复盘后 event_hash 一致。
同随机记录复盘后 random_draws 一致。
ReplayLocked Agent 不重新调用 Policy。
旧版本存档能被识别、迁移或拒绝。
CrashRecoveryJournal 能恢复到一致状态。
```

失败报告必须包含：

```text
first_divergent_command_id。
first_divergent_event_id。
expected_state_hash。
actual_state_hash。
expected_random_draw。
actual_random_draw。
config_version_mismatch。
agent_decision_mismatch。
```

## 16. 前后端协议测试策略

协议测试必须覆盖 HTTP / WebSocket 边界。

命令协议：

```text
SubmitCommandRequest 必须有 request_id、session_id、expected_state_version、idempotency_key。
重复 idempotency_key 返回原结果。
旧 expected_state_version 返回冲突或重试建议。
前端不能提交 StateChangeDraft。
前端不能修改 StateVersion。
```

查询协议：

```text
QueryRequest 只读。
Query 不触发规则结算。
普通前端不能请求完整 StateSnapshot。
Projection 必须按权限过滤。
```

事件流：

```text
server_seq 连续。
断线后能补 missing_events。
断线太久要求 full_sync。
Hidden / DebugOnly 不发给普通 session。
```

版本协议：

```text
protocol_version 不兼容必须拒绝。
config_version_id 不匹配必须提示。
错误码和 message_key 必须稳定。
```

## 18. 性质测试策略

性质测试验证永远不能被破坏的不变量。

首批不变量：

```text
生命值不能低于 0。
置信度必须在 0.0 到 1.0。
StateVersion 不能倒退。
EventSequence 不能倒退。
同一 Actor 同一 tick 只能有一个 primary Agent。
AgentScale 不能直接解锁能力。
Hidden 事件不能发给普通前端。
ReplayLocked Agent 不能重新决策。
知识声明必须有 subject、action、effect、condition、evidence、confidence。
传播结果不能凭空生成无证据知识。
文明能力不能直接解锁，只能由条件和证据推导。
```

## 26. 设计注意事项

研发实现必须遵守：

```text
不要为了测试方便绕过 RuleEngine。
不要让测试依赖真实时间和系统随机。
不要只断言成功，要断言错误码、事件和状态。
不要只测单点，要保留组合链路和场景链路。
不要让 Golden Replay 在没有审查的情况下自动更新。
不要把性能测试和功能测试混在一起判断。
不要让 Debug 权限污染普通 Agent / 前端测试。
不要让 Fuzz 失败只留日志，必须保存最小复现。
```
