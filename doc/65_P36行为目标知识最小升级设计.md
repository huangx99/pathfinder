# P36 行为目标知识最小升级设计

## 目标

把知识从“主体对象 + 行为 + 结果”升级为可以表达“主体对象 + 行为 + 目标对象 + 结果”。

这一步解决斧头、磨石、火、容器、药物、工具等真实交互里最常见的问题：同一个动作只有在作用到正确目标时才有意义。

## 当前问题

P35 之后后端可以表达：

- `axe + use -> cut_wood`
- `whetstone + use -> restore_sharpness`

这能跑通状态变化，但语义不够精确：

- 斧头不是“随便使用就砍开木头”，而是“对木头使用才砍开木头”。
- 磨石不是“随便使用就恢复锋利”，而是“对斧头使用才恢复锋利”。
- 后续 NPC 如果只看到 `axe + use -> cut_wood`，可能误以为拿斧头对任何东西使用都能产木头。

## 设计原则

1. 不重构整个知识系统，复用现有 `KnowledgeSubject.related_subject_ids`。
2. 不把目标写成动作名，例如不要写死 `use_on_wood`。
3. 行为仍是通用行为：`use`、`eat`、`combine`、`ignite`、`repair`。
4. 目标对象进入知识语义键、冲突判断、传播判断、展示文案。
5. 兼容旧知识：没有目标对象的红果、苦叶、石片知识仍然合法。

## 数据设计

### DialogIntent

新增：

- `target_object_key`：玩家输入中的目标对象。

例子：

- `用斧头砍木头`
  - `object_key = axe`
  - `target_object_key = wood`
  - `action = use`

- `用磨石打磨斧头`
  - `object_key = whetstone`
  - `target_object_key = axe`
  - `action = use`

### DialogFeedbackTemplate

新增：

- `target_object_key`：这条反馈规则要求的目标对象。

例子：

- `axe_use_cut_wood.target_object_key = wood`
- `whetstone_restore_axe.target_object_key = axe`

### LearningSafeFeedbackInput

新增：

- `target_subject_id`
- `target_subject_type_key`

学习层不需要理解“斧头”或“木头”的业务，只负责把目标对象传给知识形成。

### KnowledgeFormationInput

新增：

- `related_subject_ids`
- `relation_group_key`

形成知识时写入：

- `claim.subject.subject_id = axe`
- `claim.subject.related_subject_ids = [wood]`
- `claim.subject.relation_group_key = action_target`

## 知识语义

升级后知识应表达为：

- `axe + use -> cut_wood @ wood`
- `whetstone + use -> restore_sharpness @ axe`

中文展示：

- `斧头 对 木头 使用后 可以砍开木头`
- `磨石 对 斧头 使用后 可以恢复工具锋利度`

## 匹配与冲突规则

### 强化匹配

以下字段都相同才认为是同一类知识：

- 主体对象
- 目标对象集合
- 行为
- 关系类型
- 结果
- 条件

### 冲突匹配

以下字段相同，但结果不同，才进入冲突/修订：

- 主体对象
- 目标对象集合
- 行为
- 关系类型
- 条件范围

这样可以避免：

- `axe use wood -> cut_wood`
- `axe use stone -> spark`

被错误认为互相冲突。

### 状态条件必须进入知识范围

行为目标知识不能只记录目标对象，还必须记录触发这条知识时的关键状态条件。

例子：

- `axe{sharpness > 0} + use @ wood -> cut_wood`
- `axe{sharpness <= 0} + use @ wood -> tool_dull`

这两条知识的主体、行为、目标对象都相同，但条件范围不同，所以不能互相覆盖，也不能在传播给同伴时互相修订。

设计要求：

1. `DialogFeedbackTemplate.state_conditions` 要转换为学习层 `condition_keys`。
2. 知识去重、冲突判断、教学传播都必须比较条件范围。
3. 没有状态条件的知识仍然合法，例如红果、苦叶、石片。
4. 后续新增“斧头耐久”“药效持续时间”“夜晚可见度”“载具油量”等状态知识，都走同一套条件范围机制。

### 传播 ID 唯一性

同一次教学可能传播多条同主体知识。

例子：

- 同时教会同伴“锋利斧头能砍木头”。
- 同时教会同伴“钝斧头砍不动”。

如果接收者知识 ID 只由“接收者 + 主体 + 关系 + 时间”组成，两条知识会发生 ID 碰撞，后写入的知识会覆盖先写入的知识。

设计要求：

1. 同一次传播内每条新知识必须拥有独立序号。
2. 后续更完整版本可把行为、目标、效果、条件摘要也纳入 ID 工厂。
3. 应用传播结果时，不能因为 ID 碰撞丢失不同效果或不同条件的知识。

## 后续真实扩展场景

这套设计要能覆盖以下“奇葩但真实”的情况：

1. 斧头砍木头：工具作用到材料。
2. 磨石打磨斧头：维护工具作用到另一个工具。
3. 火点燃木棍：能量/状态源作用到材料。
4. 水浇火：液体作用到燃烧对象。
5. 药草敷伤口：物品作用到生物部位/状态。
6. 钥匙开门：工具作用到机关。
7. 容器装水：容器作用到资源点。
8. 毒涂箭头：材料作用到武器。
9. 两个物品合成：木棍 + 火 -> 火把。
10. NPC 教玩家开车：专家主体把技能作用到玩家能力。

P36 只做单目标对象；多目标合成以后可升级为 `related_subject_ids` 多值，不破坏结构。

## 最小验收标准

1. `用斧头砍木头` 形成 `axe + use -> cut_wood @ wood`。
2. `用磨石打磨斧头` 形成 `whetstone + use -> restore_sharpness @ axe`。
3. 斧头变钝和打磨恢复仍然正常。
4. 目标对象进入去重和教学传播，重复教学不产生重复知识。
5. 红果、腐烂红果、苦叶、石片旧流程不被破坏。
