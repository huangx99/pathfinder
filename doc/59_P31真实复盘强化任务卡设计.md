# 59_P31 真实复盘强化任务卡设计

## 1. 阶段目标

P31 的目标是把 P30 的 `SaveGamePackage` 变成可验证、可定位、可回归的真实复盘入口。

P30 已经解决“哪些数据能组成一个存档逻辑包”；P31 进一步解决：

- 这个存档包能不能被真实规则管线重新跑一遍。
- 重跑结果和当时记录是否一致。
- 如果不一致，问题落在完整性、恢复计划、命令重放、随机流、事件数量、状态变更数量、状态版本、条件表达式 trace、配置版本还是 golden baseline。
- 后续 AI 或低级研发只看报告，也能知道问题大概在哪个系统，而不是只能看到一串失败日志。

P31 是“复盘诊断层”，不是新的规则系统。

## 2. 本阶段不做什么

P31 明确不做：

- 不做真实文件存档读取。
- 不做云端回放。
- 不做多命令完整 episode replay，P4 当前仍只支持单命令 replay，P31 必须尊重这个限制。
- 不在复盘层修改 `GameState`。
- 不在复盘层重新实现规则、条件、知识、学习或危险判定。
- 不把隐藏真相、真实 HP、死亡、掉落、随机伤害等禁止字段写入诊断 trace。

## 3. 核心设计

P31 新增 `pathfinder_replay_diagnostics` 模块，核心对象为：

| 类/结构 | 中文含义 | 作用 |
|---|---|---|
| `GoldenReplayBaseline` | 黄金复盘基线 | 保存某次被认可的复盘期望结果，用于后续回归比较 |
| `GoldenReplayBaselineBuilder` | 黄金基线构建器 | 从合法 `SaveGamePackage` 中生成基线 |
| `GoldenReplayRequest` | 复盘请求 | 携带存档包、目标版本、可选基线和期望条件 trace |
| `GoldenReplayReport` | 复盘报告 | 汇总 replay 结果、差异、条件 trace 和问题列表 |
| `ReplayDiagnosticIssue` | 复盘诊断问题 | 用结构化方式说明失败原因、位置、期望和实际值 |
| `GoldenReplayRunner` | 黄金复盘执行器 | 校验存档、制定恢复计划、调用 P4 replay、生成差异报告 |

## 4. 与现有系统关系

P31 必须复用现有模块：

| 系统 | 使用方式 |
|---|---|
| P30 SavePackage | 作为复盘输入，只读取，不修改 |
| P30 SavePackageValidator | 复盘前必须先校验完整性 |
| P30 SaveRestorePlanner | 选择目标版本对应 snapshot，并生成恢复计划 |
| P4 ReplayRunner | 执行真实规则管线重放 |
| P27 ConditionExpression | 只保存安全的条件 trace key，用于报告差异 |
| Foundation Error/Result | 所有失败必须结构化，不抛异常 |

## 5. Golden Replay 流程

P31 的标准流程：

1. 校验 `GoldenReplayRequest`。
2. 校验 `SaveGamePackage` 完整性。
3. 用 `SaveRestorePlanner` 生成恢复计划。
4. 根据恢复计划选择 snapshot。
5. 组装 `ReplayInput`。
6. 调用 `ReplayRunner::replayOne`。
7. 比较 replay report：事件数量、状态变更数量、pipeline 状态、state version。
8. 收集 `PipelineTraceSection.condition_trace_keys`。
9. 对比请求或 baseline 中的 expected condition trace。
10. 计算 deterministic signature。
11. 如提供 baseline，对比 package hash、config version、config content hash、target version、记录数量和 signature。
12. 输出 `GoldenReplayReport`。

## 6. 诊断问题分类

`ReplayDiagnosticIssueKind` 必须覆盖：

| 枚举 | 中文含义 | 触发条件 |
|---|---|---|
| `PackageIntegrityInvalid` | 存档完整性错误 | P30 validator 失败 |
| `RestorePlanInvalid` | 恢复计划错误 | 无法为目标版本生成恢复计划 |
| `SnapshotMissing` | 快照缺失 | 恢复计划选中的 snapshot 找不到 |
| `ReplayExecutionFailed` | 复盘执行失败 | P4 replay 返回错误 |
| `ReplayMismatch` | 复盘结果不一致 | P4 compare status 为 mismatch |
| `StateChangeCountMismatch` | 状态变更数量不一致 | expected/actual state change count 不同 |
| `EventCountMismatch` | 事件数量不一致 | expected/actual event count 不同 |
| `PipelineStatusMismatch` | 管线状态不一致 | expected/actual pipeline status 不同 |
| `StateVersionMismatch` | 状态版本不一致 | expected/actual state version 不同 |
| `ConditionTraceMissing` | 条件 trace 缺失 | 期望 trace 未出现 |
| `ConditionTraceUnexpected` | 条件 trace 多出 | 禁止额外 trace 时出现未声明 trace |
| `BaselineMismatch` | 黄金基线不一致 | 与 golden baseline 的关键字段或 signature 不同 |
| `ForbiddenKeyDetected` | 禁止字段出现 | trace 或诊断字段含隐藏真相类 key |

## 7. 工程边界

P31 必须保持以下边界：

- `GoldenReplayRunner` 只能读 `SaveGamePackage`，不能修复 package。
- `GoldenReplayRunner` 不能绕过 `SavePackageValidator`。
- `GoldenReplayRunner` 不能自己解释规则结果，只能复用 P4 `ReplayRunner` 的 compare report。
- `GoldenReplayBaseline` 是回归基线，不是存档数据源。
- `condition_trace_keys` 只能是安全摘要，例如 `condition:test:eq:*`、`expression_id:*`、`canonical_condition_key:*`。
- P31 的报告面向研发和 AI 审核，不面向玩家 UI。

## 9. 验收标准

P31 必须通过以下测试：

1. 合法 P30 package 可以生成 golden baseline。
2. 合法 P30 package 可以执行 golden replay，报告为 passed。
3. package hash 被篡改时，报告 `PackageIntegrityInvalid`。
4. replay 记录的 event count 与真实 replay 不一致时，报告 `ReplayMismatch` 和 `EventCountMismatch`。
5. 期望 condition trace 缺失时，报告 `ConditionTraceMissing`。
6. 禁止额外 condition trace 时，报告 `ConditionTraceUnexpected`。
7. baseline 与当前 package/signature 不一致时，报告 `BaselineMismatch`。
8. target state version 超出 package 范围时，报告 `RestorePlanInvalid`。
9. P31 专项测试、P30/P31 相邻测试和全量测试通过。

## 10. 后续扩展

- 与 P37 调试工具增强合并，输出更完整的诊断报告。
- 与 P38 训练环境结合，生成稳定的训练回归样本。
