# P28 对象反应系统任务卡设计

## 1. 阶段定位

P28 是《先驱者》从“单对象尝试”走向“对象之间会互相影响”的关键阶段。

P3 已经完成未知对象首条规则闭环，P18-P22 已经让行动反馈能进入认知、记忆、知识和 H5 展示，P27 已经把条件判断统一到受限条件表达式系统。P28 必须在这些基础上补上对象反应系统，让玩家可以通过组合、使用、接触等行为发现世界规律。

一句话：

```text
P28 让 use/combine/touch 这类行为可以触发对象之间的可解释反应，并输出安全的对象变化、反馈、事件和知识候选。
```

典型目标：

```text
火 + 干木棍 -> 火把。
火 + 生肉 -> 熟肉。
水 + 火 -> 熄灭。
火 + 潮湿木棍 -> 失败或低效反馈。
```

---

## 2. 为什么 P28 现在做

P28 不能早于 P27。

原因：对象反应天然依赖条件：

```text
火必须 burning。
木棍必须 dry。
肉必须 raw。
水必须 available。
对象数量必须 > 0。
行动者必须能接触目标。
```

如果 P28 自己再写一套条件判断，会立刻破坏 P27 的架构回正。

因此 P28 的硬前提是：

```text
ObjectReactionRule.condition_ref -> ConditionExpressionRef
ObjectReactionResolver -> ConditionExpressionEvaluator
```

禁止在 P28 中恢复：

```cpp
if (state == "dry") { ... }
if (object_key == "fire" && target_key == "branch") { ... }
if (condition_key == "burning") { ... }
```

P28 只能把对象快照转成 `ConditionEvaluationContext`，再交给 P27 evaluator。

---

## 3. 本阶段目标

P28 必须完成一个可测试、可回放、可扩展的最小对象反应系统。

必须做到：

```text
1. 新增 pathfinder_reaction 模块。
2. 定义 ObjectReactionRule、ReactionInputSet、ReactionObjectRef、ReactionOutput、ReactionTrace。
3. 反应规则必须使用 ConditionExpressionRef，不允许自建 condition_key 判断。
4. 实现 ObjectReactionResolver，支持匹配、条件求值、优先级、输出草稿。
5. 支持 Transform / Consume / AddState / RemoveState / ResourceDelta / ProduceObject 这些最小反应输出类型。
6. 火 + 干木棍 -> 火把 成为 P28 核心验收场景。
7. 火 + 湿木棍 条件失败必须有安全 trace 和反馈原因。
8. 反应结果不能直接修改 WorldObject，只能输出 draft / change request。
9. 反应可以生成 KnowledgeCondition / KnowledgeCandidate 所需的安全 key。
10. H5 或学习入口能读取对象反应输出，形成玩家可读反馈。
11. 补充 gate scan，防止对象反应绕过 P27 condition 模块。
```

---

## 4. 不解决的问题

P28 不做完整化学系统，也不做真实物理模拟。

本阶段明确不做：

```text
不做连续环境自动反应，例如火自动蔓延。
不做完整温度、湿度、质量、体积模拟。
不做随机掉落。
不做真实伤害、死亡、尸体、loot。
不做危险探索压力，那是 P29。
不做存档迁移，那是 P30。
不做完整前端正式 UI，那是 P33。
不重构整个 InteractionRule 引擎，只做对象反应最小闭环。
```

P28 允许为后续 InteractionRule 留接口，但不能为了“一步到位”吞掉 P29-P34 范围。

## 5. 核心设计原则

### 5.1 反应规则是权威，不是对象硬编码

对象系统只提供：

```text
object_id
object_definition_id
category
runtime state / tags / resource values
quantity
```

对象系统不允许知道：

```text
火 + 木棍一定等于火把。
```

这个关系必须写在 `ObjectReactionRule` 中。

### 5.2 反应只输出草稿，不直接改状态

P28 resolver 输出：

```text
ReactionOutput
ReactionObjectChangeDraft
ReactionEventDraft
ReactionFeedbackDraft
ReactionKnowledgeDraft
ReactionTrace
```

不直接调用 `ObjectStore::markConsumed()`，不直接改 `WorldObject`。

最终状态应用继续交给后续 state / pipeline / applier。

### 5.3 条件权威只来自 P27

所有反应条件统一使用：

```text
ConditionExpressionRef
ConditionEvaluationContext
ConditionExpressionEvaluator
ConditionEvaluationResult
```

P28 可以提供 `ReactionConditionContextBuilder`，但 builder 只负责把对象快照转换为安全 context，不负责判断规则成立。

### 5.4 规则结果必须可解释

每次反应必须能回答：

```text
匹配了哪条规则？
哪些对象以什么角色参与？
哪些条件通过？
哪些条件失败？
为什么选择这条规则而不是另一条？
输出了哪些对象变化？
产生了哪些反馈和知识？
```

### 5.5 最小闭环优先

P28 第一版只证明：

```text
玩家能用两个对象触发反应。
系统能判断条件。
系统能输出对象变化草稿。
玩家能看到结果反馈。
系统能形成知识候选。
测试能复现整个链路。
```

不追求内容量。

## 7. 核心类型设计

### 7.1 ReactionActionKind

用于把当前行为归类为对象反应意图。

```cpp
enum class ReactionActionKind {
    Unknown,
    Use,
    Combine,
    Touch,
    Apply,
    Cook,
    Extinguish,
    TestOnly
};
```

注意：

```text
ReactionActionKind 是 P28 内部归类，不替代 ActionCommand。
ActionCommand 仍是输入来源。
```

### 7.2 ReactionObjectRole

对象在反应中的角色。

```cpp
enum class ReactionObjectRole {
    Unknown,
    Source,
    Target,
    Material,
    Tool,
    Catalyst,
    Container,
    Product,
    TestOnly
};
```

示例：

```text
火 + 干木棍 -> 火把
Source = fire_source
Material = dry_branch
Product = torch
```

### 7.3 ReactionObjectRef

反应输入对象快照。

```cpp
struct ReactionObjectRef {
    ReactionObjectRole role;
    ObjectId object_id;
    ObjectDefinitionId definition_id;
    std::string object_key;
    ObjectCategory category;
    int quantity;
    std::vector<std::string> state_keys;
    std::vector<std::string> tag_keys;
    std::map<std::string, double> resource_values;
};
```

设计要求：

```text
必须是快照，不允许 resolver 持有 WorldObject*。
state_keys / tag_keys 必须是安全 key。
resource_values 必须是有限数值。
```

### 7.4 ReactionInputSet

resolver 输入。

```cpp
struct ReactionInputSet {
    std::string input_key;
    ReactionActionKind action_kind;
    EntityId actor_id;
    std::vector<ReactionObjectRef> objects;
    Tick tick;
    std::vector<std::string> safe_context_keys;
};
```

校验要求：

```text
action_kind 不能 Unknown / TestOnly。
objects 数量至少 2，P28 暂定最多 3。
同一个 role 不应重复，除非规则显式允许。
所有 key 不能包含 forbidden token。
```

### 7.5 ObjectReactionRule

反应规则定义。

```cpp
struct ObjectReactionRule {
    std::string rule_key;
    ReactionActionKind action_kind;
    std::vector<ReactionObjectPattern> object_patterns;
    std::vector<ConditionExpressionRef> condition_refs;
    std::vector<ReactionOutputTemplate> output_templates;
    int priority;
    ReactionConflictPolicy conflict_policy;
    std::string feedback_key;
    std::string feedback_text;
    std::string knowledge_effect_key;
    std::vector<std::string> safe_tags;
};
```

强约束：

```text
condition_refs 只能是 P27 ConditionExpressionRef。
object_patterns 只能做基础匹配，不做复杂状态判断。
复杂状态判断必须进入 condition_refs。
output_templates 只能描述草稿，不直接改对象。
feedback_text 是规则数据的一部分，扩展新反应时优先由规则提供中文安全反馈，避免为每条内容修改 resolver。
```

### 7.6 ReactionObjectPattern

基础对象匹配。

```cpp
struct ReactionObjectPattern {
    ReactionObjectRole role;
    std::optional<ObjectDefinitionId> definition_id;
    std::optional<ObjectCategory> category;
    std::vector<std::string> required_tag_keys;
    std::vector<std::string> forbidden_tag_keys;
    bool quantity_required;
};
```

注意：

```text
required_tag_keys 只用于对象类别粗匹配，例如 fire_source、combustible、container_like。
required_tag_keys 不得承载会改变规则结果的状态语义。
dry / wet / burning / poisoned / cooked / raw 这类状态判断必须进入 P27 condition_ref。
如果某个 tag 实际代表动态状态或危险结果，必须改成 ConditionExpressionRef，不允许放在粗匹配里偷判定。
```

### 7.7 ReactionOutputKind

```cpp
enum class ReactionOutputKind {
    Unknown,
    TransformObject,
    ConsumeObject,
    AddState,
    RemoveState,
    ResourceDelta,
    ProduceObject,
    FeedbackOnly,
    KnowledgeOnly,
    TestOnly
};
```

### 7.8 ReactionOutputTemplate

```cpp
struct ReactionOutputTemplate {
    ReactionOutputKind output_kind;
    ReactionObjectRole target_role;
    ObjectDefinitionId product_definition_id;
    std::string state_key;
    std::string resource_key;
    double resource_delta;
    int quantity_delta;
    std::string feedback_key;
    std::string knowledge_key;
};
```

### 7.9 ReactionObjectChangeDraft

resolver 输出的对象变化草稿。

```cpp
struct ReactionObjectChangeDraft {
    ReactionOutputKind change_kind;
    ObjectId object_id;
    ReactionObjectRole role;
    ObjectDefinitionId from_definition_id;
    ObjectDefinitionId to_definition_id;
    std::string state_key;
    std::string resource_key;
    double resource_delta;
    int quantity_delta;
    std::vector<std::string> reason_keys;
};
```

示例：

```text
TransformObject: dry_branch -> torch
RemoveState: fire_source remove burning
ResourceDelta: fire_source fuel -5
```

### 7.10 ReactionKnowledgeDraft

P28 不直接创建 KnowledgeClaim，但要输出足够信息给 learning/knowledge。

```cpp
struct ReactionKnowledgeDraft {
    std::string knowledge_key;
    std::string subject_id;
    std::string action_key;
    std::string effect_key;
    std::vector<KnowledgeCondition> conditions;
    std::vector<std::string> reason_keys;
};
```

条件必须使用 P27 规范化结果：

```text
KnowledgeCondition.condition_ref
KnowledgeCondition.canonical_condition_key
```

### 7.11 ReactionResult

```cpp
struct ReactionResult {
    ReactionDecision decision;
    std::string selected_rule_key;
    std::vector<ReactionObjectChangeDraft> object_changes;
    std::vector<ReactionFeedbackDraft> feedbacks;
    std::vector<ReactionKnowledgeDraft> knowledge_drafts;
    std::vector<ReactionEventDraft> events;
    ReactionTrace trace;
};
```

### 7.12 ReactionDecision

```cpp
enum class ReactionDecision {
    Unknown,
    Reacted,
    NoMatchingRule,
    ConditionBlocked,
    NoEffect,
    Rejected,
    Failed,
    TestOnly
};
```

## 8. ConditionEvaluationContext 构建

P28 必须新增 `ReactionConditionContextBuilder`。

职责：

```text
把 ReactionInputSet 转成 ConditionEvaluationContext。
把 Source / Target / Material 映射到 P27 的 source / target / safe_context_keys。
把对象 state_keys / tag_keys 放入 ObjectConditionView。
把 actor 可用能力放入 ActorConditionView。
不得在 builder 里判断条件是否成立。
```

建议映射：

```text
Source -> ConditionEvaluationContext.source
Material 或 Target -> ConditionEvaluationContext.target
actor -> ConditionEvaluationContext.actor
safe_context_keys -> context.safe_context_keys
```

火 + 木棍例子：

```text
source.state_keys = [burning]
target.state_keys = [dry]
target.tag_keys = [combustible]
```

对应 condition_refs：

```text
condition:object_state:eq:dry
condition:source_state:eq:burning   // P28 可能需要扩展 P27 evaluator 支持 source_state
```

注意：如果 P27 当前 evaluator 只支持 `object_state` 默认 target，P28A 必须先扩展 P27 condition 模块支持：

```text
condition:source_state:eq:<state>
condition:target_state:eq:<state>
condition:source_tag:has:<tag>
condition:target_tag:has:<tag>
```

不能在 P28 resolver 里临时写 source/target 状态分支绕过 evaluator。

## 9. 火 + 干木棍 最小规则

P28 核心规则：

```text
rule_key = fire_dry_branch_to_torch
reaction_action = Combine
objects:
  Source: fire_source 或 category Hazard/Tool + tag fire_source
  Material: dry_branch 或 category Material + tag combustible
conditions:
  source has state burning
  material has state dry
outputs:
  TransformObject material -> torch
  ResourceDelta source fuel -5
  Feedback: reaction.fire_branch_torch
  Knowledge: dry_branch + combine_with_fire -> transforms_into_torch under burning + dry
priority: 100
conflict_policy: Exclusive
```

验收表现：

```text
输入：combine fire_source + dry_branch
输出：Reacted
object_changes 包含 dry_branch -> torch
feedbacks 包含 火把形成的安全反馈 key
knowledge_drafts 包含 transforms_into_torch
trace 包含两个 condition evaluation result
```

## 10. 火 + 湿木棍 失败规则

P28 需要有失败样例，不然玩家无法理解条件。

两种实现方式选其一：

### 10.1 条件失败产生 ConditionBlocked

同一条 `fire_dry_branch_to_torch` 规则匹配到对象，但 `material dry` 不成立。

输出：

```text
decision = ConditionBlocked
feedback = reaction.branch_too_wet
trace = material missing dry / has wet
```

### 10.2 专门湿木棍 NoEffect 规则

新增规则：

```text
rule_key = fire_wet_branch_smoke_no_effect
condition: material wet
output: FeedbackOnly smoke / branch_too_wet
```

P28 推荐：先做 10.1，若 H5 体验需要更明确，再补 10.2。

## 12. Resolver 流程

`ObjectReactionResolver::resolve(input, rules)` 流程：

```text
1. validate input / rules。
2. 按 action_kind 初筛规则。
3. 用 object_patterns 匹配对象角色。
4. 为每条候选规则构建 ConditionEvaluationContext。
5. 对 condition_refs 调用 ConditionExpressionEvaluator。
6. 记录每个条件的 ConditionEvaluationTrace。
7. 过滤必需条件失败的规则。
8. 按 priority / specificity 选择规则。
9. 根据 output_templates 生成 ReactionObjectChangeDraft。
10. 生成 feedback / knowledge / event draft。
11. 返回 ReactionResult。
```

失败路径必须区分：

```text
NoMatchingRule：没有规则匹配动作和对象。
ConditionBlocked：规则匹配但条件失败。
Rejected：输入非法或安全校验失败。
Failed：系统错误或规则配置错误。
```

## 13. 安全边界

P28 必须继承 P26/P27 的安全策略。

禁止 key 包含：

```text
hidden_truth
true_trait
real_effect
raw_state
gamestate
game_state
savegame
random
system_clock
mutation
direct_state
hp_delta
death
kill
loot
corpse
```

禁止输出：

```text
真实隐藏属性。
直接 HP/死亡/掉落语义。
前端自行解锁。
未经 condition evaluator 的规则判断。
直接修改 WorldObject。
```

所有 DTO 必须实现 `validateBasic()`。

## 14. 与已有系统关系

### 14.1 与 Object 系统

P28 使用 ObjectDefinition / WorldObject 的快照信息，但不直接修改 ObjectStore。

如果现有 ObjectRuntimeFlag 不够表达 burning/dry/wet/raw/cooked，P28 可先在 `ReactionObjectRef.state_keys` 中表达，不强行扩大 WorldObject 实体模型。

### 14.2 与 Condition 系统

P28 是 P27 的第一个正式业务用户。

任何复杂状态判断都必须落到：

```text
ConditionExpressionRef -> ConditionExpressionEvaluator
```

### 14.3 与 Knowledge / Learning

P28 不直接写知识库。

P28 输出 `ReactionKnowledgeDraft`，后续由 LearningLoop 或 KnowledgeFormationService 消化。

### 14.4 与 H5

P28 可以提供最小 H5 适配：

```text
玩家输入“用火点木棍”或“组合火和木棍”。
后端构造 ReactionInputSet。
resolver 输出 feedback。
H5 显示“木棍被点燃，变成了可携带的火把”。
```

但 P28 不做完整正式 UI。

### 14.5 与 P29

P29 危险探索将复用 P28 输出：

```text
火把可以降低夜间危险。
水可以熄灭火导致危险上升。
错误反应可能生成危险反馈。
```

P28 不实现危险压力本身，只提供可被 P29 读取的对象反应结果。

## 18. 风险与控制

### 18.1 风险：P28 变成完整物理/化学系统

控制：

```text
只做规则驱动反应，不做模拟。
只做 2-3 条核心规则。
```

### 18.2 风险：绕过 P27 条件系统

控制：

```text
所有状态条件必须进入 ConditionExpressionRef。
新增 gate scan。
```

### 18.3 风险：直接修改对象状态

控制：

```text
resolver 只输出 ReactionObjectChangeDraft。
状态应用不在 P28 resolver 内完成。
```

### 18.4 风险：H5 为展示又写一套规则

控制：

```text
H5 只消费 ReactionResult / feedback，不推导反应结果。
```

### 18.5 风险：知识链路重复造 DTO

控制：

```text
ReactionKnowledgeDraft 使用 KnowledgeCondition，并填充 condition_ref / canonical_condition_key。
```

## 19. 给实现阶段的硬性约束

实现 P28 时必须遵守：

```text
不要修改 P27 condition 模块的安全边界，除非是为了支持 source/target object state/tag 的泛化求值。
不要在 reaction resolver 里写 dry/wet/burning 分支判断。
不要让 ObjectDefinition 知道具体反应结果。
不要让 H5 直接判断 fire + branch。
不要提前引入 death/loot/hp_delta。
不要把随机概率作为 P28 第一版核心。
```

最小可行实现优先级：

```text
P0：反应类型、规则、resolver、condition evaluator 接入。
P0：火 + 干木棍成功，火 + 湿木棍失败。
P1：水 + 火熄灭。
P1：学习/H5 最小接入。
P2：更多对象内容、环境自动反应、概率反应，后续再做。
```

结论：P28 应该做成“规则驱动对象反应最小闭环”，不是物理模拟系统，也不是完整交互规则重构。完成后，P29 危险探索才有可用对象结果，例如火把、水源、熄灭火、失败尝试等。
