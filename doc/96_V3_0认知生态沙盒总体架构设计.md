# V3.0 认知生态沙盒总体架构设计

## 1. 阶段定位

V3.0 是项目玩法方向的第二次重大重心调整。

V1 的核心是“未知物品尝试 -> 经验 -> 知识 -> 教同伴”。
V2 的核心是“开放世界网格 + Command + 物品归属 + NPC 任务”。
P65 的核心是“小地图安全圈营地 + NPC 派工 + 进度条任务图”。

V3.0 的核心改为：

```text
玩家作为上帝视角的世界塑造者，绘制固定地图、投放对象和 NPC；
无知 NPC 在地图中自主探索、尝试、学习、教学、繁衍、形成族群；
玩家通过解锁更多可投放内容，观察并影响一个认知生态系统的演化。
```

V3.0 不是简单做一个新前端，也不是把 P65 营地玩法换皮。
它是把项目重新拉回最初最有生命力的方向：

```text
红果、腐烂红果、苦叶、石片。
NPC 不知道它们有什么用，只能通过尝试形成认知。
同伴不知道你学到了什么，除非你教他。
```

V3.0 要把这套认知闭环放到一个可绘制、可投放、可观察、可演化的固定地图沙盒里。

---

## 2. 为什么现在做

### 2.1 当前 P65 的问题

P65 安全圈营地已经把很多后端能力串起来：

```text
物品。
营地仓库。
知识卡。
NPC。
任务进度。
风险分支。
JSON 内容。
Axmol 客户端。
```

但它的核心循环仍然偏向：

```text
玩家点 NPC -> 分配任务 -> 等进度条 -> 看结果。
```

这个循环的问题是：

1. 玩家像在操作管理工具，而不是在玩一个活的世界。
2. NPC 的“白痴学习”被派工系统掩盖了。
3. 大部分乐趣变成等待任务完成，而不是观察未知行为涌现。
4. 后续要靠大量任务、数值、美术才能撑住玩法，个人开发压力很大。
5. 进度条任务图虽然结构正确，但作为主玩法容易变成调试面板。

### 2.2 V3.0 更适合本项目

本项目真正独特的部分不是开放世界，也不是营地经营，而是：

```text
无知个体如何通过尝试形成知识；
知识如何改变行为；
知识如何传播给同伴；
群体如何因为知识、资源和环境差异走向不同演化路线。
```

粉末游戏式固定地图非常适合承载这个核心：

```text
玩家投入一个变量，世界自己发生变化。
玩家看到 NPC 从无知到理解。
玩家不需要控制每一步，只需要塑造条件、观察结果、适度干预。
```

### 2.3 现在具备的前置能力

V3.0 不是从零开始，已有系统可以复用：

| 已有能力 | V3.0 用法 |
|---|---|
| Command 管线 | 玩家绘制、投放、擦除、检查、暂停、加速都走 Command |
| Reaction 系统 | 对象 + 行为 -> 反馈 -> 经验 |
| Knowledge 系统 | NPC 尝试后形成、修正、传播知识 |
| Teaching 系统 | NPC 之间教学、族群知识传播 |
| Content JSON | 物品、地形、行为、反应、解锁内容配置化 |
| Agent 基础 | NPC 自主探索、选择目标、尝试行为 |
| Axmol 经验 | 用固定地图替代大世界，用程序化表现支撑原型 |

### 2.4 如果不做的风险

如果继续沿 P65 营地派工做下去，会出现：

```text
为了好玩不断堆任务类型。
为了表现不断堆 UI 面板。
为了内容不断写特殊规则。
NPC 的认知系统变成后台说明，而不是核心玩法。
最终游戏像“营地管理调试器”，而不是一个有生命力的认知沙盒。
```

因此 V3.0 是必要重构方向。

## 3. 阶段目标

V3.0 总目标：

```text
做一个固定地图认知生态沙盒。
玩家能绘制地形、投放对象、投放无知 NPC。
NPC 能自主探索世界、尝试对象、形成知识，并让知识反过来改变行为。
后续能自然扩展到教学、族群、繁衍、分裂、文明、魔法、机械、外星内容。
```

第一轮 V3.0 最小可玩目标：

```text
1. 玩家可以在固定地图上绘制地形。
2. 玩家可以投放已解锁对象。
3. 玩家可以投放一个或多个无知 NPC。
4. NPC 会自主移动、观察、尝试吃/使用/拾取/组合/采集。
5. NPC 的尝试会产生反馈，并形成知识。
6. NPC 后续行为会被已形成知识影响。
7. 玩家可以查看 NPC 的知识、记忆、当前需求、最近尝试。
8. 玩家可以通过 NPC 学会的内容逐步解锁更多可投放对象。
```

不是第一轮目标，但必须预留：

```text
1. NPC 教 NPC。
2. 族群知识库。
3. 繁衍和后代学习。
4. 族群分裂。
5. 族群冲突。
6. 高级文明。
7. 魔法世界。
8. 机械危机。
9. 外星遗物。
10. 玩家编辑或导入固定地图剧本。
```

## 4. 本阶段不解决的问题

V3.0 总体设计阶段只解决方向、架构和阶段拆分，不直接进入大规模实现。

本阶段不做：

```text
1. 不立刻删除 P65 代码。
2. 不立刻重做全部前端。
3. 不立刻做完整族群繁衍。
4. 不立刻做战争、文明、魔法、机械、外星。
5. 不立刻做无限地图。
6. 不立刻做复杂美术资源。
7. 不让前端承载规则。
8. 不绕过 Command 和 Reaction/Knowledge 管线。
```

原因：

```text
V3.0 第一轮必须先证明“白痴 NPC 在玩家绘制地图中自主学习”这个核心循环成立。
如果这个循环不好玩，后续族群、文明、美术都没有意义。
```

## 5. 核心设计原则

### 5.1 玩家是世界塑造者，不是角色本人

玩家不直接控制某个小人。
玩家的核心动作是：

```text
画地形。
投放对象。
投放 NPC。
移除对象。
观察知识。
调整时间。
解锁新内容。
```

禁止：

```text
把玩家写成一个必须移动、拾取、背包管理的角色。
把主循环重新做成玩家控制小人砍树采矿。
```

### 5.2 NPC 必须从无知开始

NPC 初始不知道红果能吃，不知道腐烂果有毒，不知道火危险，不知道石片能当工具。

NPC 的行为来自：

```text
需求。
感知。
好奇心。
已有知识。
最近记忆。
风险评估。
随机尝试。
模仿或教学。
```

禁止：

```text
给 NPC 写“饿了就吃红果”的内容专属 if。
给 NPC 写“遇到狼就用火把”的内容专属 if。
```

正确方式：

```text
NPC 饿了 -> 推理需要恢复饥饿 -> 查找已知 restore_hunger 知识 -> 找到 red_berry/eat -> 产生行动目标。
如果不知道 restore_hunger，则随机尝试附近可吃对象。
```

补充边界：NPC 不是上帝视角。

```text
1. NPC 只能读取感知半径内的格子、对象、同伴和危险。
2. NPC 不能扫描全图查找红果、狼、工具或最佳路线。
3. NPC 不能读取对象隐藏真相，只能读取安全外观标签、已尝试反馈和已形成知识。
4. NPC 的目标可以来自知识反推，但执行时仍只能对当前可感知、可到达、可验证的对象行动。
5. 第一版 V3 最小闭环中，NPC 感知半径默认为 1，看到附近对象才会尝试；远处对象不会被自动发现。
```

### 5.3 世界对象是统一实体

地图上的红果、背包里的红果、反应系统里的红果、知识系统里的红果必须是同一个 content object key。

禁止：

```text
地图上有一套 red_berry。
反应系统有另一套 red_berry。
前端显示又有一套红果。
```

统一对象必须经过：

```text
ContentRegistry -> WorldObjectInstance -> Command -> Reaction -> Feedback -> Experience -> Knowledge。
```

### 5.4 玩家绘制和投放也必须走 Command

前端不能直接改地图数组。

所有玩家工具都提交 Command：

```text
world.paint_terrain
world.place_object
world.place_agent
world.erase_cell
world.inspect_cell
world.set_time_scale
unlock.select_palette
```

后端返回安全投影。

禁止：

```text
前端本地创建红果并告诉后端“已经创建好了”。
前端直接判断某格能不能放火。
前端直接给 NPC 添加知识。
```

### 5.5 知识改变行为，而不是行为假装有知识

NPC 不能因为规则写死而看起来聪明。

如果 NPC 会吃红果，必须能追溯到：

```text
它自己吃过红果。
它看过别人吃红果。
它被教过红果知识。
它继承了族群知识。
它通过推理链得出红果可解决饥饿。
```

投影必须能显示原因：

```text
当前行动：寻找红果。
原因：饥饿 + 已知 red_berry/eat -> restore_hunger。
知识来源：亲身经验 / 教学 / 族群传统 / 观察假设。
```

### 5.6 解锁是玩家目标，不是纯菜单

玩家可投放内容不是一开始全开。

解锁来自世界认知进展：

```text
NPC 第一次发现红果 -> 解锁红果投放。
NPC 学会石片使用 -> 解锁更多工具材料。
NPC 学会火 -> 解锁火种、干草、夜间危险。
NPC 形成稳定族群 -> 解锁多 NPC 投放。
NPC 掌握工具链 -> 解锁矿石、金属、机械残片。
```

禁止：

```text
所有物品一开始都能放，导致玩家没有推进目标。
纯靠等级数字解锁，与世界认知无关。
```

### 5.7 固定地图优先，不做无限地图

V3.0 第一轮采用固定地图。

原因：

```text
玩家要像红警/粉末游戏一样观察一个局部生态。
固定地图更利于布局、实验、教学和复盘。
无限地图会再次把精力拖回生成、寻路、迷雾和性能问题。
```

后续可以做地图包和关卡，但不回到无边大世界作为主玩法。

### 5.8 前端是观察和编辑工具，不是规则系统

前端可以做得好看、直观、像游戏。
但前端不能判断规则结果。

前端负责：

```text
地图绘制交互。
对象投放操作。
NPC 状态展示。
知识面板展示。
解锁面板展示。
动画、粒子、音效。
```

后端负责：

```text
对象是否能放。
NPC 做什么。
动作是否成功。
知识是否形成。
族群是否分裂。
解锁是否发生。
```

## 6. 玩法核心循环

### 6.1 一分钟循环

玩家每分钟重复做：

```text
观察 NPC 当前状态。
观察地图资源和危险。
投放一个对象或修改一小块地图。
等待 NPC 自主探索和尝试。
查看 NPC 学到了什么。
根据结果继续投放或调整环境。
```

### 6.2 五分钟循环

玩家五分钟内应该体验到：

```text
NPC 从无知到形成第一条知识。
NPC 因错误尝试受伤或中毒。
NPC 后续行为因为知识改变。
玩家解锁一个新投放物。
地图局势出现一个新问题。
```

### 6.3 三十分钟循环

玩家三十分钟内应该看到：

```text
多个 NPC 之间知识差异。
知识传播或教学。
资源消耗导致迁移或争夺。
危险导致死亡、逃跑或新知识形成。
玩家解锁一条新的技术链。
```

### 6.4 长期循环

长期玩法是：

```text
从自然物 -> 工具 -> 火 -> 营地 -> 族群 -> 农业/狩猎 -> 金属 -> 文明分支 -> 魔法/机械/外星路线。
```

但每一层都必须建立在：

```text
对象可投放。
NPC 可感知。
NPC 可尝试。
反馈可学习。
知识可传播。
行为会改变。
```

## 7. 玩家体验目标

### 7.1 观察涌现

玩家应该经常产生这种感觉：

```text
我只是放了一个东西，NPC 居然自己学会了。
我以为他会吃红果，结果他先吃了腐烂果。
我给一个族群火种，他们开始压制另一个族群。
我在水边种了树，他们慢慢迁过去了。
```

### 7.2 可理解的因果

NPC 不能乱动到玩家看不懂。

每个关键行为都要能解释：

```text
为什么他吃这个？
为什么他逃跑？
为什么他教别人？
为什么他不再碰腐烂果？
为什么这个族群分裂？
```

### 7.3 低操作负担

玩家不应该反复下达“采木、做斧头、送货”命令。

玩家主要做：

```text
塑造条件。
观察结果。
进行少量干预。
```

### 7.4 强反馈

每次 NPC 学会重要知识时，要有明显反馈：

```text
头顶小气泡。
地图小光效。
知识面板新增卡片。
事件日志简短提示。
可投放物解锁提示。
```

但不能把界面做成调试台。

## 8. 总体模块划分

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| SandboxMapRuntime | 沙盒地图运行时 | 固定地图、格子、地形、对象实例 | Command / Tick | 地图状态草稿 / 投影 | 不写对象反应规则 |
| SandboxToolCommand | 玩家工具命令 | 绘制、投放、擦除、检查 | 前端工具意图 | WorldCommand / Result | 不直接改前端状态 |
| WorldObjectRuntime | 世界对象实例 | 统一对象实例、位置、归属、状态 | Content object key | ObjectInstance | 不复制物品定义 |
| AgentLifeRuntime | NPC 生命运行时 | 饥饿、生命、疲劳、繁殖基础 | Tick / Feedback | Need state / Life events | 不写具体吃什么 |
| AgentPerception | NPC 感知 | 附近格子、对象、危险、同伴 | Map snapshot | Perception frame | 不读取 hidden truth |
| AgentCuriosityPolicy | 好奇心策略 | 无知时选择尝试对象和动作 | Need / Perception / Memory | Candidate action | 不写内容专属 if |
| AgentKnowledgePolicy | 知识行为策略 | 已知时根据知识选择行动 | Need / Knowledge | Goal / Action | 不绕过知识系统 |
| SandboxActionSystem | 沙盒动作系统 | 观察、吃、使用、拾取、放下、组合、采集 | Agent action command | Action result | 不产生知识 |
| ReactionBridge | 反应桥 | 对象+动作接入 Reaction | Action result | Feedback | 不写新反应管线 |
| ExperienceBridge | 经验桥 | Feedback 转经验 | Feedback / Context | Experience entry | 不直接写稳定知识 |
| KnowledgeLearningBridge | 知识学习桥 | 经验形成知识、修正冲突 | Experience | Knowledge claim | 不跳过学习门槛 |
| TeachingRuntime | 教学运行时 | NPC 教学、模仿、知识传播 | Proximity / Social action | Hypothesis knowledge | 不无限复制全知 |
| TribeRuntime | 族群运行时 | 族群、关系、共享知识、分裂 | Agent social state | Tribe state | 不替代个体知识 |
| UnlockRuntime | 解锁运行时 | 玩家可投放内容解锁 | Knowledge / Milestone | Palette unlock | 不用纯等级代替认知 |
| SandboxProjection | 沙盒投影 | 给前端安全展示 | Runtime state | Client projection | 不暴露 hidden truth |
| SandboxReplay | 复盘诊断 | 重放沙盒关键事件 | Command / Event | Debug report | 不作为运行时权威 |

## 9. 核心数据结构草案

### 9.1 SandboxMapDefinition

地图定义来自 JSON 或编辑器保存文件。

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `map_key` | string | 地图唯一 key | 是 | content/map loader | runtime | 不为空 | 不能用中文名当 key |
| `width` | int | 宽度 | 是 | map loader | runtime/frontend | 8-512 | 第一版不做无限 |
| `height` | int | 高度 | 是 | map loader | runtime/frontend | 8-512 | 第一版不做无限 |
| `default_terrain_key` | string | 默认地形 | 是 | content | runtime | 必须存在 | 不写死 grass |
| `initial_cells` | list | 初始地形覆盖 | 否 | map file | runtime | 坐标在范围内 | 不存规则结果 |
| `initial_objects` | list | 初始对象 | 否 | map file | runtime | object key 存在 | 不写知识 |
| `initial_agents` | list | 初始 NPC | 否 | map file | runtime | agent archetype 存在 | 不写全知 NPC |

### 9.2 SandboxCellState

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `x` | int | X 坐标 | 是 | runtime | all | 地图范围内 | 无 |
| `y` | int | Y 坐标 | 是 | runtime | all | 地图范围内 | 无 |
| `terrain_key` | string | 地形 key | 是 | runtime | projection/action | 必须在 content | 不写中文显示名 |
| `object_ids` | list | 该格对象实例 | 是 | runtime | action/perception | ID 必须存在 | 前端不能改 |
| `flags` | map | 临时状态 | 否 | runtime | rule | key 受控 | 不塞 UI 文案 |

### 9.3 WorldObjectInstance

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `object_id` | string | 实例 ID | 是 | runtime | all | 全局唯一 | 前端不能自造最终 ID |
| `object_key` | string | 内容 key | 是 | content/runtime | all | ContentRegistry 存在 | 不分地图物品/反应物品两套 |
| `x/y` | int | 位置 | 视归属 | runtime | projection/perception | OnMap 时必填 | 背包物品不能有地图坐标 |
| `owner_agent_id` | string | 持有者 | 视归属 | inventory/runtime | action | InInventory 时必填 | 不能和 x/y 同时权威 |
| `quantity` | int | 数量 | 是 | runtime | projection | >0 | 不用负数表示消耗 |
| `state_values` | map | 实例状态 | 否 | runtime/reaction | rules/projection | 受 content schema 约束 | 不写 UI-only 字段 |
| `discovered_by` | set | 被哪些主体识别 | 否 | perception/knowledge | projection | 可为空 | 不暴露隐藏真相 |

### 9.4 AgentState

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `agent_id` | string | NPC ID | 是 | runtime | all | 唯一 | 不用显示名当 ID |
| `archetype_key` | string | 原型 | 是 | content/runtime | projection | Content 存在 | 不写具体角色逻辑 |
| `x/y` | int | 位置 | 是 | runtime | perception/projection | 地图范围内 | 前端不能直接改 |
| `needs` | map | 饥饿、疲劳、恐惧等 | 是 | life runtime | policy | 范围 0-100 | 不写“想吃红果” |
| `health` | int | 生命 | 是 | life runtime | projection | 0-max | 不用事件代替状态 |
| `inventory_id` | string | 背包容器 | 是 | inventory | action | 存在 | 不直接数组拼物品 |
| `knowledge_profile_id` | string | 个体知识库 | 是 | knowledge | policy/projection | 存在 | 不复制族群知识 |
| `memory_stream_id` | string | 记忆流 | 是 | memory | learning/policy | 存在 | 不只存最后事件 |
| `tribe_id` | string | 所属族群 | 否 | tribe runtime | social policy | 可为空 | 不强制一开始有族群 |
| `current_intent` | object | 当前意图 | 否 | agent policy | projection/action | 可解释 | 不塞前端按钮状态 |

### 9.5 SandboxActionIntent

NPC 自主行动和玩家工具最终都要编译为后端可裁决的行动意图。

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `actor_id` | string | 行动者 | 是 | AgentPolicy / ClientGateway | Command | 必须可行动 | 前端不能伪造 NPC 自主行动 |
| `action_key` | string | 行为 | 是 | policy/client | ActionSystem | Content action 存在 | 不写 C++ enum 专属内容 |
| `target_object_id` | string | 目标对象 | 否 | policy/client | ActionSystem | 可见/可达 | 不允许 hidden target |
| `target_cell` | coord | 目标格 | 否 | policy/client | ActionSystem | 地图内 | 无 |
| `tool_object_id` | string | 使用工具 | 否 | policy | ActionSystem | 在背包或可用 | 不凭空生成工具 |
| `reason_trace` | object | 原因链 | 否 | policy | projection/replay | 可序列化 | 不影响裁决结果 |

### 9.6 KnowledgeUnlockRule

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `unlock_key` | string | 解锁 ID | 是 | content | UnlockRuntime | 唯一 | 不用显示名 |
| `requires_knowledge` | list | 需要的知识 | 否 | content | UnlockRuntime | knowledge key 存在 | 不写 C++ 判断 |
| `requires_event` | list | 需要事件 | 否 | content | UnlockRuntime | event key 存在 | 不用中文日志匹配 |

## 11. NPC 自主行为算法

### 11.1 Tick 流程

每个沙盒 tick：

```text
1. 更新世界自然状态。
2. 更新对象状态，例如火焰扩散、腐烂、植物生长。
3. 更新 NPC needs，例如饥饿、疲劳、恐惧。
4. 为每个 NPC 构建感知帧。
5. AgentPolicy 根据 needs + knowledge + memory + perception 选择意图。
6. ActionSystem 裁决意图。
7. ReactionBridge 产生反馈。
8. ExperienceBridge 写经验。
9. KnowledgeLearningBridge 形成或修正知识。
10. Teaching/Tribe/UnlockRuntime 处理传播和解锁。
11. 输出事件与投影 patch。
```

### 11.2 无知探索策略

NPC 没有相关知识时，不能停着不动。

候选行为：

```text
观察附近新对象。
靠近未识别对象。
尝试可食用外观对象。
尝试可使用外观对象。
拾取小型对象。
把两个已拾取对象组合。
远离造成伤害的对象。
```

权重示例：

| 条件 | 行为倾向 |
|---|---|
| 饥饿高 | 提高吃未知植物/食物外观对象权重 |
| 生命低 | 降低危险尝试，提高远离威胁权重 |
| 好奇心高 | 提高观察/使用未知对象权重 |
| 最近受伤 | 降低相同对象相同行为权重 |
| 看到同伴成功 | 提高模仿权重 |

### 11.3 有知识策略

NPC 有知识时，行为应从知识反推目标：

```text
need: hunger high
-> desired_effect: restore_hunger
-> known claim: red_berry + eat -> restore_hunger
-> find object: nearest red_berry
-> action: move_to + eat
```

```text
threat nearby
-> desired_effect: repel_threat / reduce_fear / survive
-> known claim: torch + use(target: beast) -> repel_threat
-> required item: torch
-> if has torch: use torch
-> if knows craft torch and material available: craft torch
-> if cannot solve: flee
```

第一版不做完整链式制作，只做一层反推：

```text
需求 -> 直接已知行为 -> 执行。
```

后续再接回 P39/P40 的链式目标执行。

### 11.4 行为失败处理

失败不能静默。

失败会产生：

```text
短期记忆。
风险记忆。
可能的候选知识修正。
行为权重调整。
```

例如：

```text
吃腐烂红果 -> 中毒 -> 形成 negative feedback -> 下次降低 eat rotten_berry 权重。
使用石片没有明显效果 -> no_visible_effect -> 不直接删除好奇心，但降低重复尝试。
```

## 12. 认知与知识闭环

### 12.1 必须保留的最小闭环

```text
Object + Action -> Feedback -> Experience -> KnowledgeClaim -> BehaviorPolicy。
```

例子：

```text
red_berry + eat -> restore_hunger
rotten_berry + eat -> poison
stone_flake + use(tree) -> cut_tree / produce wood
wood + use(fire) -> burn
fire + near -> danger_heat
```

### 12.2 知识状态

继续沿用：

```text
candidate
hypothesis
active
shared
false/rejected
```

但 V3.0 要更强调来源：

```text
self_experience
observed_other
taught_by_agent
tribe_tradition
inherited_childhood
player_seeded_for_test 仅测试可用，正式玩法禁用
```

### 12.3 知识对行为的影响

| 知识状态 | 行为影响 |
|---|---|
| candidate | 轻微提高或降低尝试权重 |
| hypothesis | 中等影响，仍可能验证 |
| active | 强影响，可主动用于需求解决 |
| shared | 可作为族群默认教学内容 |
| rejected | 降低重复尝试，保留特殊条件再验证 |

### 12.4 教学

教学不是第一版核心，但接口必须预留。

触发条件：

```text
两个 NPC 距离近。
一个 NPC 有 active/shared 知识。
另一个 NPC 有相关需求或好奇。
关系允许交流。
语言/族群条件满足。
```

教学结果：

```text
接收者获得 hypothesis，而不是 active。
接收者后续亲身验证后才能转 active。
```

### 12.5 观察学习

NPC 看到别人吃红果恢复饥饿，应形成观察假设。

```text
observed: agent_A eat red_berry
observed feedback: agent_A hunger reduced
agent_B claim: red_berry + eat -> restore_hunger [hypothesis, source=observed_other]
```

这会让群体演化更自然。

## 14. 投放与解锁系统

### 14.1 投放对象分类

| 分类 | 示例 | 解锁来源 |
|---|---|---|
| 基础地形 | 草地、水、土、沙 | 初始或地图包 |
| 基础植物 | 红果、苦叶、树 | 发现/学习采集 |
| 危险植物 | 腐烂果、毒蘑菇 | 学会基础食物后 |
| 材料 | 石片、木头、干草 | 学会使用/采集 |
| 火系 | 火种、火堆、火把 | 学会燃烧/火危险 |
| 生物 | 狼、兔、牛 | 学会食物链/族群 |
| 工具 | 斧头、磨石、容器 | 学会组合/制作 |
| 文明 | 房屋、农田、围栏 | 族群稳定后 |
| 奇幻 | 魔法水晶、符文火 | 魔法路线 |
| 机械 | 齿轮、电池、机器人残骸 | 机械路线 |
| 外星 | 信标、异星种子、遗物 | 外星路线 |

### 14.2 解锁来源

```text
发现对象。
形成知识。
形成族群知识。
完成环境里程碑。
观察到新反应。
NPC 存活若干天。
NPC 繁衍成功。
族群分裂。
```

### 14.3 解锁不是奖励弹窗，而是玩法推进

例如：

```text
NPC 学会吃红果 -> 玩家解锁“投放红果”。
NPC 学会石片使用 -> 玩家解锁“投放树枝/木头”。
NPC 学会火危险 -> 玩家解锁“投放狼”，因为 NPC 有基本自保知识。
NPC 学会火把驱赶 -> 玩家解锁“夜晚事件”。
```

这样玩家会有明确目标：

```text
我要引导他们学会火。
我要让他们能对抗狼。
我要解锁繁衍。
我要看两个族群怎么分裂。
```

## 15. 族群与繁衍预留

### 15.1 族群形成

族群不是一开始强制存在。

可能触发：

```text
多个 NPC 长时间在同一区域活动。
共享食物来源。
互相教学成功。
共同躲避危险。
繁衍或照顾幼体。
```

### 15.2 族群知识

族群知识不是替代个体知识。

结构：

```text
TribeKnowledge = 多个个体 active/shared 知识的聚合。
个体可以从 TribeKnowledge 获得 hypothesis。
稳定仪式/教育后，后代可以继承部分 hypothesis 或 active 基础知识。
```

### 15.3 繁衍条件

第一版不做。
后续条件：

```text
食物稳定。
安全水平足够。
空间足够。
成年 NPC 数量足够。
族群关系稳定。
```

繁衍结果：

```text
生成幼体 NPC。
幼体有弱 needs，需要照顾。
幼体逐步学习，不是出生全知。
```

### 15.4 族群分裂条件

```text
资源不足。
居住距离变远。
知识差异过大。
领导者死亡。
危险切断路径。
玩家刻意投放不同资源诱导分化。
```

分裂结果：

```text
新 tribe_id。
复制一部分共享知识。
保留个体知识差异。
形成关系：友好/竞争/敌对/陌生。
```

```text
形成指定知识后投放面板新增对象。
```

### P71：多 NPC 与教学观察学习

目标：

```text
多个 NPC。
观察别人行为形成 hypothesis。
近距离教学。
```

验收：

```text
一个 NPC 亲身学会红果后，另一个 NPC 可通过教学获得假设。
假设需要验证才能 active。
```

### P72：族群最小状态

目标：

```text
多个 NPC 在同一区域形成 tribe。
族群知识聚合。
基础共享。
```

验收：

```text
族群有 ID、成员、共享知识摘要。
个体仍保留自己的知识差异。
```

### P73：繁衍与后代学习

目标：

```text
食物、安全、空间满足时产生后代。
后代继承部分族群假设知识。
```

验收：

```text
后代不是全知。
可以通过观察和教学成长。
```

### P74：族群分裂与关系

目标：

```text
资源不足或距离导致族群分裂。
分裂后知识和关系不同。
```

验收：

```text
同一地图上存在两个 tribe。
两者知识库不同。
```

## 20. 前端总体方向

### 20.1 画面结构

```text
左侧/主体：固定地图。
底部：粉末游戏式工具栏。
右侧：选中对象/NPC 面板。
顶部：时间、解锁提示、暂停/速度。

```

### 20.2 地图交互

```text
左键拖拽绘制。
右键擦除或检查。
滚轮缩放。

```



## 21. 测试策略

### 21.1 后端单元测试

```text
paint terrain command 不绕过 runtime。
place object command 检查 unlock。
NPC 感知不暴露隐藏对象。
NPC 无知识时会产生尝试意图。
Reaction feedback 能进入 experience。
Experience 能形成 knowledge。
Knowledge 能改变下一次行为权重。
```

### 21.2 集成测试

```text
玩家投放红果和 NPC。
NPC 随机探索并吃红果。
NPC 形成 red_berry/eat -> restore_hunger。
再次饥饿时 NPC 主动寻找红果。
玩家投放腐烂果。
NPC 中毒后形成危险知识并降低重复尝试。
```

### 21.3 架构扫描

```text
前端不得出现 restore_hunger / poison 等规则判断。
AgentPolicy 不得出现 red_berry / torch / wolf 等内容 key 专属 if。
Runtime 不得直接改 content registry。
地图对象必须使用统一 object_key。
```

### 21.4 可玩性测试

```text
3 分钟内玩家能看到 NPC 第一次尝试。
5 分钟内玩家能看到第一条知识形成。
10 分钟内玩家能通过投放引导 NPC 行为改变。
```

## 22. 风险与应对

### 22.1 NPC 太随机，不像智能

风险：NPC 到处乱走，玩家看不懂。

应对：

```text
每个 NPC 显示当前意图和原因。
好奇心随机只在无知识时占主导。
有需求和知识后，行为必须明显变得有目标。
```

### 22.2 NPC 太聪明，失去学习感

风险：NPC 一开始就能正确吃红果、躲危险。

应对：

```text
初始知识为空。
只有感知标签和对象外观倾向。
正确行为必须能追溯到经验、观察或教学。
```

### 22.3 内容配置变复杂

风险：JSON 字段越来越多，难扩展。

应对：

```text
对象定义、行为定义、反应定义、解锁定义分文件。
每个 content package 做校验。
新增内容必须有最小回归测试。
```

### 22.4 性能问题

风险：很多 NPC 每 tick 都扫描全图。

应对：

```text
固定地图空间索引。
NPC 只感知半径内对象。
Tick 分层：高频移动、低频思考、低频学习压缩。
```

### 22.5 前端变成调试台

风险：信息太多，玩家感觉像看日志。

应对：

```text
地图第一，状态气泡第二，详细面板第三。
默认只显示关键状态。
知识和记忆折叠到 NPC 面板。
```

### 22.6 又变成大重构做不完

风险：想一次做族群、繁衍、文明。

应对：

```text
P66-P69 只证明一个 NPC 学会吃红果和避开危险。
族群和繁衍必须等最小认知沙盒好玩后再做。
```

## 24. 总结

V3.0 的一句话：

```text
把项目从“玩家派工营地管理”改回“玩家塑造世界，白痴 NPC 自主学习并演化”的认知生态沙盒。
```

第一版不要追求内容多，而要证明：

```text
玩家投放对象。
NPC 自主尝试。
反馈形成知识。
知识改变行为。
玩家因此解锁新对象。
```

只要这个闭环成立，后续族群、繁衍、文明、魔法、机械、外星都能自然长出来。
如果这个闭环不成立，继续堆任务、地图、美术都不会让游戏真正好玩。
