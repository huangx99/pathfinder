# 25_P0后端基础类型任务卡设计

## 1. 设计目标

本章把 `24_后端工程骨架设计.md` 中的 P0 后端基础类型继续拆成最底层、最小上下文、可直接交给 AI 编码的任务卡。

核心目标：

```text
从最底层开始搭。
一次只实现一个小能力。
每个任务都有明确输入、输出、允许文件、禁止文件、验收命令。
让 AI 不需要理解整个项目，也能稳定写出正确代码。
```

本章不是写代码，而是给后续 AI 编码执行者的任务清单。

## 2. P0 拆分总原则

### 2.1 从最底层到上层

P0 foundation 的实现顺序必须是：

```text
目录和 CMake 骨架
-> 编译验证空测试
-> Error 基础枚举
-> Result 基础包装
-> StrongId 基础包装
-> Version / Tick
-> Hash
-> RandomSeed
-> Serialization 约定
-> foundation 聚合 README 和 dev_notes
```

不能反过来。

错误顺序：

```text
先实现 CommandEnvelope。
先实现 ConfigLoader。
先实现 RuleEngine。
先实现 AgentRuntime。
```

正确顺序：

```text
先让 backend 能 cmake。
再让一个空测试能跑。
再实现最小 Error。
再实现 Result。
再实现 ID。
再实现版本、hash、tick、随机种子。
```

### 2.2 每个任务必须很小

单任务限制：

```text
最多修改 1-4 个文件。
最多新增 50-200 行代码。
最多读取 1-3 个文档片段。
必须有测试或明确说明为什么当前任务只建骨架。
必须更新 dev_notes。
```

### 2.3 每个任务都要能独立验收

任务验收不依赖未来大系统。

```text
Error 测试不依赖 Result。
Result 测试可以依赖 Error。
StrongId 测试不依赖 Config。
Hash 测试不依赖 StateSnapshot。
RandomSeed 测试不依赖 ReplayRunner。
```

### 2.4 每个任务都要给下一任务留下记忆

每个任务完成后更新：

```text
backend/dev_notes/foundation.md
```

记录：

```text
任务 ID。
完成内容。
新增文件。
测试命令。
下一任务建议。
不能破坏的约束。
```

## 3. P0 任务依赖图

```text
TASK-P0-000 读取任务卡和创建 context pack
  -> TASK-P0-001 创建 backend CMake 骨架
    -> TASK-P0-002 创建最小测试入口
      -> TASK-P0-003 实现 ErrorCode / ErrorDetail
        -> TASK-P0-004 实现 Result<void>
          -> TASK-P0-005 实现 Result<T>
            -> TASK-P0-006 实现 StrongId 基础
              -> TASK-P0-007 实现 StrongId 哈希和格式校验
                -> TASK-P0-008 实现 StateVersion / Tick
                  -> TASK-P0-009 实现 HashValue
                    -> TASK-P0-010 实现 RandomSeed / RandomDrawRecord
                      -> TASK-P0-011 实现 serialization.h 约定
                        -> TASK-P0-012 foundation 聚合检查
```

说明：

```text
Result 拆成 Result<void> 和 Result<T>，降低模板复杂度。
StrongId 拆成基础包装和格式/哈希，降低一次任务难度。
Version/Tick 先于 Hash/Random，方便后续 replay。
最后做聚合检查，不在中途扩大范围。
```
