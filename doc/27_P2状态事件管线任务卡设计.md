# 27_P2状态事件管线任务卡设计

## 1. 设计目标

本章把 P2 拆成可以直接交给 AI 编码执行的小任务卡。

P2 的目标不是做完整规则结算，而是在 P0 foundation、P1 config + command 之上建立规则执行前必须存在的三层空壳：

```text
状态层：GameState 最小快照 + StateChange 最小结构。
事件层：EventRecord 最小结构 + EventStream / EventStore 最小追加能力。
管线层：RulePipeline 空壳 + PipelineContext / PipelineResult。
```

P2 完成后，项目应该具备：

```text
一个命令可以进入 Pipeline 空壳。
Pipeline 可以读取 GameState 元信息。
Pipeline 可以返回 PipelineResult。
PipelineResult 可以包含空 StateChangeSet 和空 EventStream。
EventRecord 可以被构造、校验、追加到内存 EventStore。
所有结构都可测试，但不执行真实游戏规则。
```

一句话：

```text
P2 建桥，不跑车。
```

## 2. 本阶段解决什么问题

P2 要解决的是“后续规则结算放在哪里”的工程问题。

没有 P2，会出现这些风险：

```text
P3 unknown fruit 规则不知道该把状态写到哪里。
反馈事件不知道该如何绑定 StateVersion。
测试无法断言一次命令产生了哪些状态变更和事件。
后续 RuleEngine 容易直接调用 resolver，绕过统一管线。
AI 实现时可能把规则逻辑塞进 Command 或 Config。
```

P2 必须建立这些边界：

```text
Command 只表达意图。
Pipeline 只编排步骤。
StateChange 描述状态如何变化。
GameState 是状态权威快照。
EventRecord 记录已经发生的事实。
EventStore 只追加事件，不修改状态。
PipelineResult 是一次执行的统一输出。
```

## 3. 本阶段不解决的问题

P2 严禁提前实现：

```text
完整 RuleEngine 门面。
完整 RuleResolver。
完整 InteractionRule 结算。
unknown fruit 吃浆果闭环。
认知、记忆、知识真实更新。
传播、族群、文明真实规则。
AgentRuntime。
HTTP / WebSocket。
H5 前端。
完整存档复盘。
复杂对象库。
复杂配置注册表。
```

P2 允许出现这些名字，但只能作为空壳或 ID 引用：

```text
RulePipeline。
PipelineContext。
PipelineResult。
StateChange。
EventRecord。
GameState。
```

不能出现这些行为：

```text
根据 action_id == eat 扣饥饿值。
根据 target_id 查对象是否可食用。
根据配置生成真实效果。
根据事件反向修改 GameState。
让 Pipeline 直接知道红浆果、火把、狼、蜘蛛等具体内容。
```

## 5. P2 设计原则

### 5.1 状态先于事件提交

P2 必须延续 `17_规则管线设计.md` 的原则：

```text
StateChange 先校验。
StateChange 再提交。
EventRecord 绑定提交后的 StateVersion。
EventStore 最后追加。
```

P2 只做空壳，因此可以只验证顺序和数据结构，不需要做真实对象字段变更。

### 5.2 Event 是事实，不是命令

事件只能描述已经发生的结果。

错误：

```text
EventRecord 表示“准备吃浆果”。
EventBus 触发扣血。
EventStore 负责修改 GameState。
```

正确：

```text
CommandEnvelope 表示“想吃浆果”。
StateChange 表示“饥饿值变化草稿”。
EventRecord 表示“饥饿值已变化”。
```

P2 还没有真实饥饿值，所以只建立结构，不写具体数值规则。

### 5.3 Pipeline 是编排，不是规则大杂烩

P2 的 RulePipeline 空壳只负责：

```text
接收 CommandEnvelope。
接收 GameState。
创建 PipelineContext。
按固定 Stage 顺序输出空结果。
返回 PipelineResult。
```

P2 的 RulePipeline 不负责：

```text
判断 eat/use/combine。
判断对象是否存在。
判断对象能否食用。
修改角色属性。
生成具体认知知识。
```

### 5.4 所有输出都必须可测试

P2 即使是空壳，也必须可以测试：

```text
空 GameState 有合法版本。
StateChange 缺 ID 会失败。
EventRecord 缺 event_id 会失败。
EventStore 追加后数量增加。
Pipeline 空执行返回成功状态。
Pipeline 空执行不修改输入 GameState。
Pipeline 空执行输出空 StateChangeSet 和空 EventStream。
```

### 5.5 不为未来过度抽象

P2 只建立必要骨架。

暂不做：

```text
复杂 EventPayload variant。
复杂 StatePath 查询语言。
复杂事务日志。
复杂 Pipeline 插件注册器。
复杂异步 EventBus。
复杂并发锁。
```

后续扩展点要留，但实现必须小。
