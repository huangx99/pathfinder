# P48 制作与世界反应接入任务卡设计

更新时间：2026-05-21
阶段：P48
依赖：P42、P43、P44、P45、P46、P47、P28/P28.5 旧反应设计
状态：待实现

## 1. 问题定义

P47 已经能从世界资源节点中采集材料，但材料还不能通过统一世界 Command 变成新物品。

P48 要解决的是：

```text
玩家或 NPC 持有 / 周围存在材料
-> 提交 Craft / Reaction Command
-> 根据 JSON reaction 定义校验材料、状态、数量、催化物
-> 通过 P45 权威归属接口消耗材料
-> 通过 P45 / WorldRuntime 权威接口生成成品
-> 返回 events / deltas / projection patch / experience
-> 后续 P49 可从 experience 形成知识
```

P48 不是旧 H5 的“制作火把按钮”，也不是写死 `wood + fire = torch`。

P48 是 V2 开放世界里“材料如何变成成品”的统一后端规则层。

## 2. 玩家体验目标

玩家体验上，P48 要让下面闭环成立：

```text
砍树得到木头
拾取木头进入背包
靠近火源或持有火源
执行制作 / 反应命令
木头减少
火源不一定消耗
火把成为真实物品
火把进入背包或落在地图上
前端收到明确结果和变化
```

后续 NPC 也能走同一条路：

```text
NPC 知道“火把能驱赶野兽”
NPC 发现没有火把
NPC 根据 P39/P40 规划触发制作火把
P48 消耗材料并生成火把
NPC 再执行使用火把
```

## 3. 本阶段不解决的问题

P48 只做制作 / 反应落地，不扩散范围。

不做：

```text
不形成稳定知识：P49 做。
不做 NPC 自主规划：P50/P51 做。
不做野兽行为：P52 做。
不做 H5 新界面：P53 做。
不做复杂配方搜索和自动补材料：P51/P52 后续做。
不做制作台、建筑施工、多阶段工程：后续扩展做。
不做概率产物、品质、耐久继承：后续扩展做。
不做真实事务系统：P48 先写清原子性边界。
```

## 4. 核心结论

P48 必须新增一个 V2 世界反应桥接模块，建议命名：

```text
backend/include/pathfinder/world_reaction/
backend/src/world_reaction/
backend/tests/unit/world_reaction/
backend/tests/integration/world_reaction/
```

P48 的权威规则来源：

```text
ContentRegistry::findReaction(reaction_key)
ReactionDefinitionContent.inputs
ReactionDefinitionContent.consume
ReactionDefinitionContent.outputs
ReactionDefinitionContent.effect_key
ReactionDefinitionContent.knowledge_templates
```

P48 的运行态权威来源：

```text
IWorldRuntime / IWorldEntityLocationPort：地图、Actor、OnMap 对象。
IInventoryRuntime：Actor 背包、物品查询、转移。
```

P48 的 Command 入口：

```text
WorldCommandKind::Craft
```

P48 的最小输入：

```text
command.target.recipe_key
或 command.parameters["reaction_key"]
```

P48 不允许前端传“木头 + 火 = 火把”的结果。
前端只能传 command，结果必须由后端依据 JSON reaction 和 runtime 状态算出。

## 5. 为什么 P48 必须补强 P45 Inventory

当前 P45 `InventoryTransferKind` 已经预留：

```text
ConsumeToNowhere
SpawnToInventory
SpawnToMap
```

但当前 `WorldInventoryRuntime::transfer()` 只支持：

```text
PickupFromMap
DropToMap
```

如果 P48 直接改 `InventoryRuntimeState.entries`，就会绕过 P45 权威归属系统，后续大世界一定会出问题。

所以 P48 必须补上至少两个 P45 Transfer 能力：

```text
ConsumeToNowhere：从 Actor inventory 消耗指定 entity_key / entity_id / quantity。
SpawnToInventory：把新产物生成到 Actor inventory，并同步 WorldEntityInstance 为 InInventory。
```

可选但建议同时补：

```text
SpawnToMap：通过 P45 transfer 语义生成 OnMap 产物；内部可复用 spawnEntityOnMap。
```

如果时间控制严格，P48 MVP 至少实现：

```text
ConsumeToNowhere
SpawnToInventory
```

OnMap 产物可以先由 P48 通过 `IWorldEntityLocationPort::spawnEntityOnMap` 实现，但必须写清楚它仍是 P45 location port 权威入口，不是直接改数组。

## 6. 模块拆解

| 模块 | 中文名 | 职责 | 输入 | 输出 | 边界 |
|---|---|---|---|---|---|
| WorldReactionTypes | 世界反应类型 | 定义请求、证明、草稿、结果、失败码 | command / reaction | DTO | 不读写 runtime |
| WorldReactionRecipeResolver | 反应配方解析器 | 从 ContentRegistry 找 reaction 并验证结构 | reaction_key | ReactionDefinitionContent | 不看背包 |
| WorldReactionInputResolver | 输入材料解析器 | 在 actor 背包和附近地图寻找材料 / 催化物 | inputs / actor | input proofs | 不修改状态 |
| WorldReactionDraftBuilder | 反应草稿构建器 | 把 reaction + proofs 变成消耗草稿和产物草稿 | proofs | draft | validate-only |
| WorldReactionApplier | 反应应用器 | 消耗材料、生成产物、生成事件 | draft | result | 必须走 P45 / location port |
| WorldReactionService | 编排服务 | validate + apply + execute | request | result | 不注册 command |
| CraftCommandHandler | Craft 命令处理器 | 接入 WorldCommandKind::Craft | WorldCommandDto | WorldCommandExecutionResult | 不解析具体配方逻辑 |
| WorldReactionProjectionBridge | 投影桥 | 生成 changed inventories / entities / cells patch | result | WorldProjectionPatchDto | 不读 JSON |

## 7. 核心枚举设计

### 7.1 WorldReactionActionKind

P48 MVP 不新增新的 WorldCommandKind，只把 `Craft` 映射到内容 reaction 的 action。

```cpp
enum class WorldReactionActionKind {
    Unknown,
    Craft,
    Use,
    Combine
};
```

中文说明：

```text
Craft：制作，强调把材料做成成品。
Use：使用，强调主物体作用于目标物体。
Combine：组合，强调多个物体共同反应。
```

P48 Command 只实现 `Craft` handler，但内部允许匹配 JSON reaction 的 `action_key`：

```text
craft
use
combine
```

匹配策略：

```text
如果 command.target.recipe_key 指定 reaction_key，则直接使用该 reaction。
如果 command.parameters["reaction_key"] 指定 reaction_key，则直接使用该 reaction。
P48 不做自动配方搜索，避免多反应歧义。
```

### 7.2 WorldReactionInputSourceKind

```cpp
enum class WorldReactionInputSourceKind {
    Unknown,
    ActorInventory,
    SameCellMap,
    AdjacentMap
};
```

中文说明：

```text
ActorInventory：Actor 背包中的材料。
SameCellMap：Actor 当前格子的地图物品或对象。
AdjacentMap：Actor 相邻格子的地图物品或对象。
```

P48 MVP 搜索顺序必须确定：

```text
1. ActorInventory
2. SameCellMap
3. AdjacentMap，按 cell_id / entity_id 稳定排序
```

这样 replay 和测试才稳定。

### 7.3 WorldReactionOutputLocationPolicy

```cpp
enum class WorldReactionOutputLocationPolicy {
    Unknown,
    ActorInventory,
    DropOnMap
};
```

中文说明：

```text
ActorInventory：成品进入操作者背包，适合火把、工具、食物。
DropOnMap：成品生成在地图格子，适合营火、建筑雏形、陷阱。
```

P48 MVP 默认：

```text
ActorInventory
```

参数解析规则：

```text
缺省 -> ActorInventory
actor_inventory -> ActorInventory
drop_on_map -> DropOnMap
其他字符串 -> 失败，不能静默修正
```

## 8. 核心 DTO 设计

### 8.1 WorldReactionRequest

```cpp
struct WorldReactionRequest {
    std::string reaction_command_id;
    std::string actor_key;
    std::string reaction_key;
    WorldReactionActionKind action_kind = WorldReactionActionKind::Craft;
    WorldReactionOutputLocationPolicy output_location_policy = WorldReactionOutputLocationPolicy::ActorInventory;
    std::map<std::string, std::string> parameters;
};
```

校验：

```text
reaction_command_id 非空。
actor_key 非空。
reaction_key 非空。
action_kind 不能 Unknown。
output_location_policy 只能 ActorInventory / DropOnMap。
```

### 8.2 WorldReactionInputProof

```cpp
struct WorldReactionInputProof {
    std::string object_key;
    std::string required_state;
    int required_quantity = 1;
    int available_quantity = 0;
    WorldReactionInputSourceKind source_kind = WorldReactionInputSourceKind::Unknown;
    std::string entity_id;
    std::string inventory_id;
    std::optional<world_runtime::WorldCellCoord> coord;
    std::vector<std::string> state_keys;
};
```

它证明：

```text
某个输入材料在运行态中真实存在。
数量足够。
状态匹配。
来源可追踪。
后续 apply 可以按 proof 消耗或保留。
```

### 8.3 WorldReactionConsumeDraft

```cpp
struct WorldReactionConsumeDraft {
    std::string object_key;
    std::string entity_id;
    int quantity = 1;
    WorldReactionInputSourceKind source_kind = WorldReactionInputSourceKind::Unknown;
    std::string inventory_id;
    std::optional<world_runtime::WorldCellCoord> coord;
};
```

注意：

```text
ReactionDefinitionContent.inputs 只是参与条件。
ReactionDefinitionContent.consume 才决定是否消耗。
例如 camp_fire 可以作为 input / catalyst，但不在 consume 中，因此不被消耗。
```

### 8.4 WorldReactionOutputDraft

```cpp
struct WorldReactionOutputDraft {
    std::string entity_id;
    std::string entity_key;
    std::string display_name_key;
    int quantity = 1;
    bool stackable = true;
    std::string stack_key;
    std::vector<std::string> state_keys;
    std::map<std::string, double> numeric_states;
    std::vector<std::string> tag_keys;
    WorldReactionOutputLocationPolicy location_policy = WorldReactionOutputLocationPolicy::ActorInventory;
    std::optional<world_runtime::WorldCellCoord> coord;
};
```

产物 id 必须确定性：

```text
reaction_command_id + "_" + output_object_key + "_" + output_index
```

不能使用真实时间、随机数、state_version。

### 8.5 WorldReactionDraft

```cpp
struct WorldReactionDraft {
    std::string draft_id;
    WorldReactionRequest request;
    content::ReactionDefinitionContent reaction;
    std::vector<WorldReactionInputProof> input_proofs;
    std::vector<WorldReactionConsumeDraft> consume_drafts;
    std::vector<WorldReactionOutputDraft> output_drafts;
    std::string effect_key;
    std::vector<std::string> knowledge_template_keys;
};
```

### 8.6 WorldReactionResult

```cpp
struct WorldReactionResult {
    bool ok = false;
    WorldReactionFailureKind failure_kind = WorldReactionFailureKind::None;
    std::vector<std::string> changed_inventory_ids;
    std::vector<std::string> changed_entity_ids;
    std::vector<std::string> changed_cell_ids;
    std::vector<world_command::WorldStateDeltaDto> state_deltas;
    std::vector<world_command::WorldEventDto> events;
    std::vector<world_command::WorldExperienceDto> experiences;
};
```

## 9. 失败码设计

```cpp
enum class WorldReactionFailureKind {
    None,
    InvalidRequest,
    ActorMissing,
    ReactionMissing,
    ReactionInvalid,
    ActionMismatch,
    InventoryMissing,
    InputMissing,
    InputStateMismatch,
    QuantityInsufficient,
    OutputInvalid,
    OutputPolicyInvalid,
    ConsumeFailed,
    SpawnFailed,
    RuntimeConflict
};
```

中文说明：

```text
InvalidRequest：命令字段缺失或格式错误。
ActorMissing：操作者不存在。
ReactionMissing：JSON 中找不到 reaction_key。
ReactionInvalid：reaction 定义结构不满足 P48 运行要求。
ActionMismatch：command action 与 reaction action 不匹配。
InventoryMissing：actor 背包不存在或不可访问。
InputMissing：找不到某个输入材料。
InputStateMismatch：找到对象但状态不符合要求。
QuantityInsufficient：数量不足。
OutputInvalid：产物为空、数量非法、对象定义不存在。
OutputPolicyInvalid：产物归属策略非法。
ConsumeFailed：消耗材料失败。
SpawnFailed：生成产物失败。
RuntimeConflict：validate 后 apply 前状态变化。
```

## 10. Reaction 匹配规则

P48 不做复杂自动推理，只执行指定 reaction。

Reaction 选择顺序：

```text
1. command.target.recipe_key
2. command.parameters["reaction_key"]
3. 如果都没有，InvalidRequest
```

不能根据 outputs 自动猜 recipe。

原因：

```text
后续一个物品可能有多个制作路线。
例如火把可以用木棍+布+油，也可以用树脂木棍+火。
自动猜会破坏确定性。
```

## 11. 输入材料解析规则

对 `reaction.inputs` 中每个 input：

```text
object_key 必须存在于 ContentRegistry objects。
min / quantity 必须大于 0，优先使用 min；如果 min 为 0 可用 quantity。
state 为空表示不要求状态。
```

搜索顺序：

```text
1. Actor inventory 中 entity_key == object_key 的 item。
2. Actor 当前格子 OnMap entity_key == object_key。
3. Actor 相邻格子且 visible 的 OnMap entity_key == object_key。
```

状态匹配：

```text
如果 input.state 非空，则对应 entity / entry 的 state_keys 必须包含该 state。
```

数量匹配：

```text
available_quantity >= required_quantity。
```

多个输入引用同一种 object_key 时：

```text
P48 MVP 可以先拒绝重复 input object_key，返回 ReactionInvalid。
```

原因：

```text
否则需要做跨 stack 分配和 proof 合并，容易在 P48 范围内做坏。
复杂多 stack 材料分配可留到后续 P55+。
```

## 12. 消耗规则

只消耗 `reaction.consume` 中声明的材料。

consume 规则：

```text
consume.object_key 必须能在 input_proofs 中找到。
consume.delta 必须为负数。
abs(delta) 是消耗数量。
如果 consume.state 非空，必须与 proof 状态匹配。
```

消耗来源：

```text
如果 proof 来源 ActorInventory：调用 P45 InventoryTransferKind::ConsumeToNowhere。
如果 proof 来源 SameCellMap / AdjacentMap：通过 IWorldEntityLocationPort 的 setEntityStackData / destroyEntity 权威接口更新。
```

注意：

```text
不能直接修改 InventoryRuntimeState.entries。
不能直接修改 WorldRuntimeSnapshot.entities。
不能直接从 vector erase。
```

## 13. 产物生成规则

对每个 output：

```text
output.object_key 必须存在于 ContentRegistry objects。
quantity_delta 必须 > 0。
entity_id 必须确定性。
stack_key = object_key + ":crafted"。
state_keys 至少包含 "crafted"。
tag_keys 至少包含 "reaction_output"。
```

生成位置：

```text
ActorInventory：调用 P45 InventoryTransferKind::SpawnToInventory。
DropOnMap：调用 IWorldEntityLocationPort::spawnEntityOnMap，coord = actor.coord。
```

为什么 DropOnMap 可以直接用 location port：

```text
P45 已把 IWorldEntityLocationPort 定义为地图归属权威入口。
只要不直接改 runtime 数组，就不算绕过架构。
```

## 14. validate-before-apply

P48 必须拆成：

```text
validate(request) -> WorldReactionDraft
apply(draft) -> WorldReactionResult
execute(request) -> validate + apply
```

validate 只能读：

```text
ContentRegistry
IWorldRuntime
IInventoryRuntime
IWorldEntityLocationPort
```

validate 不得写状态。

apply 前必须复验关键状态：

```text
Actor 仍存在。
需要消耗的 input 仍在原位置。
数量仍足够。
状态仍匹配。
```

如果复验失败：

```text
RuntimeConflict / QuantityInsufficient / InputMissing
```

失败必须不生成产物。

## 15. 原子性边界

P48 的理想原子性：

```text
所有材料消耗成功 + 所有产物生成成功，才算成功。
任何一步失败，状态不应部分改变。
```

但当前 runtime 还不是事务系统。

P48 MVP 要求：

```text
先完整 validate 所有输入和产物。
apply 时先复验。
尽量先消耗后生成，避免复制产物。
代码中必须写明：当前缺少事务，未来持久化 / 多 runtime 需要 Transaction / Compensation。
```

P48 必须避免明显的坏顺序：

```text
不能先生成产物，再发现材料不够。
不能产物已生成但材料未扣。
```

对于 apply 中途失败的补偿：

```text
P48 可以先不实现完整补偿，但必须测试常见失败路径在 validate 阶段被拦截。
```

## 16. Command Handler 设计

新增：

```cpp
std::shared_ptr<IWorldCommandHandler> createCraftCommandHandler(WorldReactionService& service);
```

handler 解析：

```text
command.command_kind 必须 Craft。
request.reaction_command_id = command.command_id。
request.actor_key = command.actor_key。
request.reaction_key 来自 target.recipe_key 或 parameters["reaction_key"]。
output_location_policy 来自 parameters["output_location_policy"]。
```

成功返回：

```text
result_kind = Succeeded
state_deltas = reaction_result.state_deltas
events = reaction_result.events，tick 写 context.currentTick()
experiences = reaction_result.experiences，tick 写 context.currentTick()
changed_inventory_ids / changed_entity_ids / changed_cell_ids
projection_patch_override
```

失败返回：

```text
result_kind = Failed
failure_reason_keys 包含 failure kind 字符串
事件 CraftFailed
不返回成功 deltas
```

## 17. Projection Patch

P48 patch 至少包含：

```text
changed_inventories：材料减少、产物增加的 inventory。
changed_entities：被消耗或生成的 entity。
changed_cells：OnMap 材料减少或 DropOnMap 产物出现的 cell。
```

不能要求前端全量刷新。

## 18. Experience 输出

P48 不形成知识，但必须给 P49 留入口。

成功 Craft 输出：

```text
WorldExperienceDto
experience_id = command_id + "_experience"
actor_key = actor_key
command_key = "craft"
subject_entity_key = 第一个 output.object_key
目标 / target_entity_key = reaction_key 或 catalyst object_key
effect_key = reaction.effect_key
tick = command context tick
reason_keys 包含 reaction_key 和 knowledge_template_keys
```

P49 会基于 experience + knowledge_templates 决定是否形成知识。

## 19. 与 P42 JSON 的关系

P48 必须使用当前 `content/core/reactions/basic_reactions.json`：

```json
wood_plus_fire_make_torch
```

该 reaction 表示：

```text
输入：processed wood + camp_fire
动作：use
效果：make_torch
产物：torch +1
消耗：processed wood -1
知识模板：knowledge_make_torch
```

P48 实现后，新增一个物品和反应的流程应接近：

```text
1. 在 objects JSON 添加新 object。
2. 在 reactions JSON 添加新 reaction。
3. 在 knowledge JSON 添加可选知识模板。
4. 不改 CraftCommandHandler。
5. 不改 WorldReactionService 主流程。
```

## 20. 与 P47 的关系

P47 产物默认 OnMap。

玩家需要先：

```text
P47 采集 / 砍伐 -> OnMap 产物
P45 Pickup -> ActorInventory
P48 Craft -> 消耗 ActorInventory 材料并生成成品
```

P48 不应该直接从 ResourceNode 取材料。

如果要从资源节点直接制作，是后续自动任务系统的链式计划，不是 P48。

## 21. 与 P49 的关系

P48 只输出 experience，不直接写 KnowledgeRepository。

原因：

```text
制作成功是一段经验。
这段经验是否成为知识、是否可教学、是否可信，需要 P49 认知学习链处理。
```

## 22. 与 P50/P51 的关系

P48 要让 NPC 后续可以调用同一服务。

因此：

```text
actor_key 不能写死 player。
InventoryOwner 必须基于 actor_key。
Command source 可以是 PlayerInput / AgentDecision。
事件和 experience 必须记录 actor_key。
```

## 23. 最小验收用例

P48 最少要让下面流程成立：

```text
1. 生成 actor。
2. 给 actor 背包 processed wood。
3. 在 actor 当前格或相邻格生成 camp_fire。
4. 执行 Craft command，reaction_key = wood_plus_fire_make_torch。
5. processed wood 数量 -1。
6. camp_fire 不消耗。
7. torch +1 进入 actor inventory。
8. result events / deltas / patch / experience 完整。
```

## 24. 代码门禁

Kimi 实现 P48 时必须遵守：

```text
不写死 red_berry / torch / wood 的主流程。
测试中可以使用 wood_plus_fire_make_torch 作为样例，但服务必须通用。
不绕过 P45 inventory / location port。
不使用真实时间或随机数生成 id。
不让前端决定 reaction 输出。
不让 CraftCommandHandler 自己写材料扣减逻辑。
不把 knowledge 直接写入知识库。
不新增 H5 UI。
```
