# 先驱者：世界对象设计

## 1. 本文档目标

本文档用于设计《先驱者》规则引擎中的世界对象系统。

世界对象是游戏中可被发现、尝试、交互、认知、传播知识和记录事件的核心实体。

它要把配置系统中的对象定义，变成运行时世界里真实存在的对象实例。

一句话：

```text
ObjectDefinition 定义“这类东西是什么”。
WorldObject 表示“世界中这一具体东西现在怎么样”。
KnowledgeClaim 表达“玩家或族群知道这个东西在某种行为下有什么效果”。
```

世界对象系统必须兼容：

```text
01 基础类型与 ID。
02 通用枚举。
03 错误码与结果类型。
04 事件系统。
05 配置系统与受限表达式。
后续行为、交互、反馈、认知、知识、传播、存档和强化学习。
```

## 2. 设计边界

### 2.1 本文档要解决的问题

```text
定义 WorldObject 的运行时结构。
定义 ObjectDefinition 与 WorldObject 的关系。
定义对象运行时状态、资源、位置、关系、标签。
定义对象如何被查询、创建、快照、状态变更。
定义对象如何连接配置、事件、认知、表达式和存档。
为后续交互规则、反馈系统、知识系统、强化学习提供稳定对象模型。
```

### 2.2 本文档暂不解决的问题

```text
不设计完整地图寻路。
不设计完整背包和装备系统。
不设计完整生物 AI。
不设计完整交互结算。
不设计完整认知系统。
不设计完整存档格式。
不设计渲染表现层。
```

## 3. 核心原则

### 3.1 定义与实例分离

```text
ObjectDefinition 是配置定义，静态、可复用、由 ConfigRegistry 提供。
WorldObject 是运行时实例，动态、可变化、进入存档和事件。
```

禁止在 `WorldObject` 中复制完整 `ObjectDefinition`。

`WorldObject` 只保存 `ObjectDefinitionId`，需要静态信息时通过 `ConfigRegistry` 查询。

### 3.2 对象不保存认知

世界对象不直接保存“玩家知道什么”。

```text
WorldObject 提供被认知的主体。
CognitionState / KnowledgeClaim 保存玩家或族群知道什么。
```

这样同一个对象可以对不同主体有不同认知状态。

### 3.3 状态变化必须可追踪

对象状态变化不能偷偷发生。

```text
对象变化通过 ObjectStateChange / StateChange 表达。
对象变化产生 EventRecord。
对象变化可进入存档和复盘。
```

### 3.4 对象模型必须支持表达式上下文

受限表达式需要读取对象状态。

`WorldObject` 必须能提供：

```text
target.object_state
target.object_category
target.trait_ids
target.tags
target.quantity
target.durability
target.runtime_flags
```

### 3.5 第一阶段不要做万能物品系统

第一阶段目标是认知规则引擎，不是传统 RPG 物品系统。

```text
不做装备品质。
不做复杂背包格子。
不做交易经济。
不做所有权系统。
不做建筑系统。
```

## 4. 世界对象系统总架构

第一阶段世界对象系统包含：

| 模块 | 中文名 | 作用 |
|---|---|---|
| ObjectDefinition | 对象定义 | 配置层对象类型定义 |
| WorldObject | 世界对象 | 运行时对象实例 |
| ObjectInstanceData | 对象实例数据 | 对象实例的基础数据 |
| ObjectRuntimeState | 对象运行时状态 | 当前状态、数量、耐久、标记等 |
| ObjectRelation | 对象关系 | 对象与对象、区域、实体之间的关系 |
| RegionObjectIndex | 区域对象索引 | 按区域管理对象 |
| ObjectQuery | 对象查询 | 查询附近、可见、可交互对象 |
| ObjectFactory | 对象工厂 | 根据配置创建对象实例 |
| ObjectSnapshot | 对象快照 | 存档、复盘、测试用快照 |
| ObjectStateChange | 对象状态变化 | 对象变化的状态变更描述 |
| ObjectContextProvider | 对象上下文提供器 | 为表达式和规则提供对象上下文 |

## 5. ObjectDefinition 与 WorldObject 的关系

### 5.1 ObjectDefinition

`ObjectDefinition` 来自 `05_配置系统设计.md`。

它描述对象类型：

```text
red_berry 是植物。
red_berry 默认可见。
red_berry 有 trait_edible_fresh。
red_berry 允许 eat / collect / teach。
```

它不描述运行时变化。

### 5.2 WorldObject

`WorldObject` 是运行时对象实例。

它描述具体对象：

```text
obj_000001 是一丛 red_berry。
它位于 region_000001。
当前数量为 5。
当前状态为 normal。
它可以被采集。
```

### 5.3 使用规则

```text
规则判断静态能力时查 ObjectDefinition。
规则判断当前可用性时查 WorldObject.runtime_state。
表达式需要对象上下文时由 ObjectContextProvider 汇总 Definition + RuntimeState。
```

## 6. WorldObject 字段设计

`WorldObject` 是规则层持有的运行时对象。

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| object_id | ObjectId | 对象实例 ID | 是 | 运行时唯一对象 ID |
| definition_id | ObjectDefinitionId | 对象定义 ID | 是 | 对应配置定义 |
| region_id | RegionId | 所在区域 | 是 | 当前对象所在区域 |
| instance_data | ObjectInstanceData | 实例数据 | 是 | 创建时间、位置、数量等 |
| runtime_state | ObjectRuntimeState | 运行时状态 | 是 | 当前状态、资源、标记等 |
| relations | list<ObjectRelation> | 对象关系 | 否 | 与对象、区域、实体的关系 |
| version | StateVersion | 对象版本 | 是 | 对象自身状态版本 |

### 6.1 设计注意事项

```text
WorldObject 不直接保存 ObjectDefinition 完整内容。
WorldObject 不直接保存 ObjectKnowledge。
WorldObject 不直接执行规则。
WorldObject 可以进入存档。
WorldObject 的变化必须产生 StateChange。
```

## 7. ObjectInstanceData（对象实例数据）

`ObjectInstanceData` 描述对象实例不频繁变化的基础数据。

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| spawned_tick | Tick | 创建帧 | 是 | 对象创建时间 |
| spawned_game_time | GameTime | 创建游戏时间 | 是 | 游戏内创建时间 |
| position | ObjectPosition | 位置 | 是 | 对象位置 |
| stack_count | Count | 堆叠数量 | 否 | 同类对象数量 |
| initial_seed | RandomSeed | 初始随机种子 | 否 | 对象生成随机来源 |
| origin | ObjectOrigin | 来源 | 是 | 初始、生成、转化、产出等 |

### 7.1 ObjectPosition（对象位置）

第一阶段位置保持抽象，不绑定具体地图算法。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| region_id | RegionId | 区域 | 对象在哪个区域 |
| slot_key | string | 位置槽 | 区域内抽象位置，例如 north_edge、near_water |
| local_index | Count | 局部序号 | 同一位置多个对象的排序 |

说明：

```text
第一阶段不做复杂坐标系统。
后续如果需要二维、六边形、网格或导航图，可以扩展 ObjectPosition，而不影响 WorldObject 主结构。
```

### 7.2 ObjectOrigin（对象来源）

建议作为小型枚举或字符串分类：

```text
Initial：初始世界生成。
Spawned：规则生成。
Produced：行为产出。
Transformed：由其他对象转化。
Imported：存档或调试导入。
```

## 8. ObjectRuntimeState（对象运行时状态）

`ObjectRuntimeState` 描述对象当前运行时状态。

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| object_state | ObjectState | 对象状态 | 是 | normal、consumed、decayed 等 |
| visibility_state | ObjectVisibilityState | 可见状态 | 是 | visible、remembered 等 |
| quantity | Count | 数量 | 否 | 可采集数量、剩余次数等 |
| durability | BoundedValue | 耐久 | 否 | 可用耐久 |
| resource_values | map<string, BoundedValue> | 资源值 | 否 | 燃料、水量、成熟度等 |
| runtime_flags | list<string> | 运行时标记 | 否 | wet、burning、rotten 等 |
| disabled_reasons | list<string> | 禁用原因 | 否 | 当前不可交互原因 |
| last_changed_tick | Tick | 上次变化帧 | 是 | 最近状态变化时间 |

### 8.1 quantity 与 durability 的边界

```text
quantity 表示数量或剩余可采集份数。
durability 表示单个对象的耐久或可用程度。
resource_values 表示多种资源槽，例如 fuel、water、heat。
```

示例：

```text
一丛红浆果：quantity = 5
一根石器：durability = 30/100
火堆：resource_values.fuel = 40/100, resource_values.heat = 70/100
```

### 8.2 runtime_flags 的边界

`runtime_flags` 用于临时状态标签。

示例：

```text
wet
burning
rotten
fresh
blocked
recently_collected
```

注意：

```text
长期稳定特性应放 TraitDefinition。
临时运行时状态放 runtime_flags。
可枚举且通用的状态优先放 ObjectState。
```

## 9. ObjectRelation（对象关系）

`ObjectRelation` 描述对象与对象、区域、实体之间的关系。

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| relation_type | ObjectRelationType | 关系类型 | 是 | near、inside、burns、produces 等 |
| source_id | string | 来源 ID | 是 | 关系来源 |
| target_id | string | 目标 ID | 是 | 关系目标 |
| strength | Ratio | 强度 | 否 | 关系强弱 |
| tags | list<string> | 标签 | 否 | 关系标记 |

### 9.1 使用示例

```text
red_berry grows_on bush
fire burns dry_branch
water_source inside cave_region
wolf near fire_source
```

### 9.2 设计注意事项

```text
ObjectRelation 不直接表示知识。
玩家是否知道这个关系，由认知系统表达。
ObjectRelation 是世界真实关系。
KnowledgeClaim 是主体对关系或效果的认知。
```

## 10. RegionObjectIndex（区域对象索引）

`RegionObjectIndex` 管理区域内对象。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| region_id | RegionId | 区域 ID | 关联区域 |
| object_ids | list<ObjectId> | 对象列表 | 区域内对象 |
| by_definition | map<ObjectDefinitionId, list<ObjectId>> | 按定义索引 | 同类对象索引 |
| by_tag | map<string, list<ObjectId>> | 按标签索引 | 标签对象索引 |
| by_visibility | map<ObjectVisibilityState, list<ObjectId>> | 按可见性索引 | 可见对象索引 |

### 10.1 使用规则

```text
RegionObjectIndex 是查询优化结构。
权威对象数据仍在 WorldObjectStore。
索引可以重建。
索引错误不应污染对象状态。
```

## 11. ObjectQuery（对象查询）

`ObjectQuery` 用于规则和前端查询对象。

### 11.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| region_id | RegionId | 区域 | 限定区域 |
| definition_ids | list<ObjectDefinitionId> | 定义过滤 | 指定对象类型 |
| categories | list<ObjectCategory> | 类别过滤 | 植物、材料等 |
| required_tags | list<string> | 必要标签 | 必须包含标签 |
| allowed_states | list<ObjectState> | 允许状态 | 状态过滤 |
| visibility_states | list<ObjectVisibilityState> | 可见性过滤 | 可见、记忆中等 |
| relation_filter | object | 关系过滤 | near、inside 等 |
| limit | Count | 数量限制 | 最多返回多少 |

### 11.2 查询示例

```text
查找当前区域可见对象。
查找附近可采集植物。
查找所有燃烧中的对象。
查找能作为燃料的对象。
查找和火有关系的对象。
```

## 12. ObjectFactory（对象工厂）

`ObjectFactory` 根据配置创建 `WorldObject`。

### 12.1 输入

```text
ObjectDefinitionId
RegionId
ObjectPosition
ObjectOrigin
RandomSeed
初始数量 / 初始状态覆盖项
```

### 12.2 输出

```text
Result<WorldObject>
```

### 12.3 创建流程

```text
1. 校验 ObjectDefinitionId 是否存在。
2. 从 ConfigRegistry 读取 ObjectDefinition。
3. 创建 ObjectId。
4. 初始化 ObjectInstanceData。
5. 根据 ObjectDefinition.default_state 初始化 ObjectRuntimeState。
6. 应用初始覆盖项。
7. 生成 ObjectCreated / ObjectDiscovered 相关事件草稿。
8. 返回 WorldObject。
```

### 12.4 错误处理

```text
定义不存在：id_definition_not_found。
枚举非法：validation_enum_unknown。
初始状态非法：validation_config_invalid。
随机流缺失：rule_random_stream_missing。
```

## 13. WorldObjectStore（世界对象存储）

`WorldObjectStore` 是运行时对象集合。

### 13.1 职责

```text
按 ObjectId 查询 WorldObject。
按 RegionId 查询对象集合。
应用 ObjectStateChange。
维护 RegionObjectIndex。
提供 ObjectSnapshot。
```

### 13.2 接口建议

```text
get_object(ObjectId) -> Result<WorldObject>
find_objects(ObjectQuery) -> Result<list<ObjectId>>
add_object(WorldObject) -> Result<void>
apply_change(ObjectStateChange) -> Result<void>
snapshot_object(ObjectId) -> Result<ObjectSnapshot>
```

### 13.3 约束

```text
WorldObjectStore 不直接读取配置文件。
WorldObjectStore 可以使用 ConfigRegistry 验证 definition_id。
所有修改必须通过 ObjectStateChange。
```

## 14. ObjectSnapshot（对象快照）

`ObjectSnapshot` 用于存档、复盘、测试和调试。

### 14.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| object_id | ObjectId | 对象 ID | 实例 ID |
| definition_id | ObjectDefinitionId | 定义 ID | 配置引用 |
| region_id | RegionId | 区域 ID | 所在区域 |
| instance_data | ObjectInstanceData | 实例数据 | 基础数据 |
| runtime_state | ObjectRuntimeState | 运行时状态 | 当前状态 |
| relations | list<ObjectRelation> | 关系 | 对象关系 |
| version | StateVersion | 版本 | 对象版本 |

### 14.2 注意事项

```text
ObjectSnapshot 不保存完整 ObjectDefinition。
加载快照时必须检查 definition_id 在当前 ConfigRegistry 中存在。
如果配置版本不兼容，应进入存档迁移流程。
```

## 15. ObjectStateChange（对象状态变化）

`ObjectStateChange` 描述对象状态如何变化。

它是 StateChange 的对象领域子类型。

### 15.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| state_change_id | StateChangeId | 状态变更 ID | 是 | 变更唯一标识 |
| object_id | ObjectId | 对象 ID | 是 | 被修改对象 |
| change_type | StateChangeType | 变更类型 | 是 | update、consume、transform 等 |
| field_path | string | 字段路径 | 是 | 修改字段 |
| before | any | 变化前 | 否 | 旧值 |
| after | any | 变化后 | 否 | 新值 |
| reason | string | 原因 | 是 | 行为、规则或事件原因 |
| caused_by_event_id | EventId | 事件 ID | 否 | 关联事件 |
| caused_by_command_id | CommandId | 命令 ID | 否 | 关联命令 |

### 15.2 示例

```text
采集红浆果：quantity 5 -> 3
火堆燃料减少：resource_values.fuel 40 -> 35
树枝被点燃：runtime_flags add burning
生肉变成熟肉：definition_id raw_meat -> cooked_meat，change_type = Transform
```

### 15.3 约束

```text
ObjectStateChange 必须可序列化。
ObjectStateChange 必须可测试断言。
ObjectStateChange 应关联 CommandId 或 EventId。
禁止直接改 WorldObject 而不生成 ObjectStateChange。
```

## 16. ObjectContextProvider（对象上下文提供器）

`ObjectContextProvider` 为规则和表达式提供对象上下文。

### 16.1 输入

```text
ObjectId
ConfigRegistry
WorldObjectStore
```

### 16.2 输出 EvaluationContext 片段

```text
target.object_id
target.definition_id
target.object_state
target.object_category
target.trait_ids
target.tags
target.quantity
target.durability
target.resource_values
target.runtime_flags
target.region_id
```

### 16.3 设计注意事项

```text
表达式读取的是上下文快照，不是直接持有 WorldObject 引用。
表达式不能通过上下文修改对象。
上下文必须来自同一个 StateVersion。
```

## 17. 对象与配置系统关系

```text
ObjectDefinition 来自配置。
WorldObject 保存 definition_id。
ConfigRegistry 提供 ObjectDefinition、TraitDefinition、ActionDefinition。
ObjectFactory 通过 ConfigRegistry 创建对象。
ObjectContextProvider 通过 ConfigRegistry 补全对象静态信息。
```

关键约束：

```text
对象实例不复制完整配置。
配置变更不直接修改运行中对象。
单局开始后 config_version 锁定。
```

## 18. 对象与事件系统关系

对象系统产生或关联以下事件：

```text
ObjectDiscovered
ObjectStateChanged
ResourceChanged
ActionResolved
FeedbackGenerated
KnowledgeCreated
RuleErrorOccurred
```

示例：

```text
玩家第一次看到红浆果：ObjectDiscovered
采集红浆果导致数量减少：ObjectStateChanged / ResourceChanged
吃红浆果获得反馈：FeedbackGenerated
形成红浆果可食用知识：KnowledgeCreated
```

事件记录事实，ObjectStateChange 记录状态变化。

## 19. 对象与认知系统关系

对象是认知主体，但不保存认知内容。

```text
WorldObject：世界真实对象。
ObjectDefinitionId：知识通常绑定的对象类型。
KnowledgeClaim：对象 + 行为 + 效果 + 证据 / 置信度 / 条件。
CognitionState：某主体知道什么。
```

示例：

```text
WorldObject = obj_000001
Definition = red_berry
KnowledgeClaim = red_berry + eat + restore_hunger
```

### 19.1 为什么知识优先绑定定义 ID

```text
玩家通常学习的是“红浆果这种东西可以吃”，不是“obj_000001 这一丛可以吃”。
```

但事件和复盘仍应保留具体 ObjectId。

## 20. 对象与行为系统关系

行为命令目标通常引用 ObjectId。

```text
ActionCommand.target_id = obj_000001
```

规则执行时：

```text
1. 通过 WorldObjectStore 查 WorldObject。
2. 通过 definition_id 查 ObjectDefinition。
3. 校验 action 是否允许。
4. 校验 runtime_state 是否可交互。
5. 构建 EvaluationContext。
6. 执行交互规则。
7. 生成 ObjectStateChange 和 Event。
```


## 21. 对象与对象反应关系

对象之间可以发生反应。

例如：

```text
火 + 木棍 = 火把
火 + 生肉 = 熟肉
水 + 火 = 熄灭
石头 + 树枝 = 简易工具
毒蘑菇 + 烹饪 = 毒性降低
```

但第六阶段只设计对象模型如何支持反应，不在这里完成反应结算规则。

真正的对象反应匹配、条件判断、效果生成和状态变化，会在后续 `09_交互规则设计.md` 中细化。

### 21.1 为什么对象系统要支持反应

《先驱者》的核心玩法不是单个对象孤立尝试，而是探索世界规律。

很多知识来自对象组合：

```text
什么东西会燃烧？
什么东西能被火加工？
什么东西组合后有新用途？
什么东西会相互抵消？
什么东西会产生危险？
```

因此世界对象必须能表达对象之间的关系和反应结果。

### 21.2 对象系统负责什么

对象系统负责提供反应所需的对象基础能力：

```text
对象实例 ID。
对象定义 ID。
对象当前状态。
对象特性。
对象标签。
对象资源值。
对象关系。
对象状态变化能力。
对象快照。
表达式上下文。
```

对象系统不负责决定“火 + 木棍 一定变火把”。

它只负责让交互规则能够安全地查询和修改对象。

### 21.3 交互规则负责什么

对象反应应由 `InteractionRuleConfig` 和后续 `InteractionRule` 负责。

职责包括：

```text
匹配输入对象组合。
检查动作类型，例如 use / combine / touch。
检查对象状态，例如火是否 burning，木棍是否 dry。
检查对象特性，例如 combustible。
调用表达式计算概率、成本、风险和结果。
生成 Effect。
生成 ObjectStateChange。
生成 Event。
生成 KnowledgeClaim。
```

### 21.4 对象反应数据流

建议数据流：

```text
ActionCommand(use/combine)
        ↓
WorldObjectStore 查询输入对象
        ↓
ConfigRegistry 查询 ObjectDefinition / TraitDefinition
        ↓
ObjectContextProvider 构建 EvaluationContext
        ↓
InteractionRule 匹配反应规则
        ↓
ExpressionEvaluator 计算条件和概率
        ↓
生成 ObjectStateChange / Effect / Event / KnowledgeClaim
```

### 21.5 火 + 木棍 = 火把 示例

示例不是正式配置，只表达设计方向：

```text
输入对象：fire_source + dry_branch
行为：use / combine
条件：fire_source.runtime_flags contains burning
条件：dry_branch.trait_ids contains combustible
结果：dry_branch Transform -> torch
事件：ObjectStateChanged / ActionResolved / KnowledgeCreated
知识：dry_branch + use_with_fire + transforms_into_torch
```

这里应形成的知识不是“木棍长得像火把”，而是：

```text
木棍在接触火时可以变成可携带火源。
```

也就是 `对象 + 行为 + 效果 + 条件`。

### 21.6 对象反应与 ObjectRelation 的关系

`ObjectRelation` 可以表达对象之间真实存在的关系。

例如：

```text
fire burns dry_branch
dry_branch transforms_into torch
water extinguishes fire
```

但关系不等于反应结算。

```text
ObjectRelation 表达世界关系。
InteractionRule 表达规则如何结算。
KnowledgeClaim 表达主体知道什么。
```

### 21.7 对象反应与状态变化

对象反应最终必须落到状态变化。

可能的结果包括：

```text
Transform：对象转化，例如 dry_branch -> torch。
Consume：对象被消耗，例如 dry_branch 数量减少。
Update：对象状态改变，例如 fire fuel 增加。
AddFlag：添加 burning、cooked、wet 等运行时标记。
RemoveFlag：移除 raw、wet 等运行时标记。
Produce：产生新对象，例如 ash、torch、cooked_meat。
```

### 21.8 设计注意事项

```text
对象系统只提供反应基础，不硬编码具体反应。
具体反应必须配置化，并由交互规则解释。
对象反应产生的知识必须仍然是 KnowledgeClaim。
对象反应必须产生可追踪 Event 和 StateChange。
对象反应不能绕过 ConfigRegistry、ExpressionEvaluator 和 RuleResult。
```

## 22. 对象与存档 / 复盘关系

```text
存档保存 ObjectSnapshot。
复盘读取 EventRecord 和 ObjectStateChange。
加载存档时通过 definition_id 重新关联配置。
配置版本不兼容时进入迁移流程。
```

## 23. 对象与强化学习关系

世界对象为强化学习提供 observation 基础。

但强化学习不直接读取完整 WorldObject。

建议流程：

```text
WorldObject + ObjectDefinition + CognitionState
        ↓
RLObservationProjection
        ↓
ObservationSnapshot
        ↓
RLTrajectory
```

可提供给 AI 的对象观察包括：

```text
对象是否可见。
对象类别。
对象已知知识状态。
可尝试行为。
已知风险。
位置关系。
资源状态摘要。
```

注意：

```text
AI 观察不等于世界真实全量对象状态。
AI 观察应受认知系统约束。
```

测试结论：

```text
当前世界对象设计可以支撑配置创建、区域查询、行为目标、对象状态变化、对象反应基础、事件记录、认知主体、表达式上下文、存档快照和强化学习观察。
```

## 26. 研发实现注意事项

```text
禁止 WorldObject 复制完整 ObjectDefinition。
禁止 WorldObject 保存玩家或族群认知。
禁止绕过 ObjectStateChange 直接修改对象状态。
禁止表达式直接持有 WorldObject 可变引用。
禁止 RegionObjectIndex 成为权威状态。
禁止在 WorldObject 中硬编码“火 + 木棍 = 火把”这类具体反应，具体反应必须进入交互规则。
新增对象字段必须考虑存档、事件、表达式和测试。
```
