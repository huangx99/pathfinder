# P39 知识链式推理与目标规划任务卡设计

## 1. 阶段定位

P39 是“知识链式推理与目标规划”阶段。

它接在 P38 世界交互结算与可见后果之后，不是继续给同伴、野兽、族人手写更多 `if/else` 行为规则，也不是把 H5 做得更像游戏。P38 已经让动作能改变世界，物品能消耗和生成，野兽能推进、避让、攻击，同伴能在明确规则下举火把协助。但是试玩和讨论暴露出一个更深层问题：

```text
同伴知道“红果能缓解饥饿”，但如果同伴饿了，系统还需要专门写一条“饿了吃红果”的规则。
族人知道“房子能御寒”，但天气冷、孩子增多、房子不够时，系统还不能自己推出“要造房子”。
系统知道“造房子需要木头”，但还不能链式推出“先砍木头”。
系统知道“砍木头需要斧头锋利”，但还不能链式推出“先打磨斧头”。
如果当前可以直接生火御寒，系统还不能比较“生火链短、造房链长”，从而先选生火。
```

所以 P39 要解决的不是“再加一个同伴行为”，而是补上一个统一的推理层：

```text
需求/目标 -> 可解释效果 -> 候选行动 -> 前置条件链 -> 代价/风险评分 -> 生成可执行计划。
```

P39 的一句话定义：

```text
让 Agent 能根据自己掌握的知识和当前世界状态，推导出能改善目标的行动链，而不是依赖为每一种情况手写行为规则。
```

### 1.1 当前已有基础

P39 必须复用以下前置系统：

```text
P18：KnowledgeClaim，提供 Agent 主观知识。
P19：知识冲突与修正，提供知识状态和可信度。
P20：知识传播，决定同伴/族群是否真的知道某条知识。
P21：学习闭环，动作反馈进入知识。
P27：受限条件表达式，统一条件判断。
P28/P28.5：对象反应与配置化物品/反应流水线。
P29：危险与探索压力，提供 threat 目标来源。
P32：H5 安全投影，前端只显示后端结果。
P34：因果归因，避免错误知识被过早确认。
P35：状态化对象，支持斧头变钝、打磨恢复。
P36：行为目标知识，保留 action target。
P38：WorldSnapshot、WorldChange、WorldChangeApplier、AgentAutonomyService、世界可见后果。
```

P39 不推翻这些系统，而是在它们上方增加一个“目标规划与知识效果语义”层。

### 1.2 为什么 P39 必须现在做

P38 后继续扩内容会立即遇到规则爆炸。

如果没有 P39，后续每增加一个需求都要写规则：

```text
饿了吃红果。
冷了生火。
冷了造房。
房子不够造房。
斧头钝了打磨。
野兽来了用火把。
孩子饿了大人采食物。
雨天火堆熄了找干草。
```

这会带来四个风险：

1. **行为规则无限膨胀**：每个内容都要写专门逻辑。
2. **知识系统被架空**：Agent 表面上“知道”，实际决策不靠知识。
3. **链式推导缺失**：能执行单步，但不能自动准备前置条件。
4. **选择缺少策略**：短链行动和长链行动无法比较，导致不符合常识。

P39 正是为了避免后续变成“看起来很智能，实际全是硬编码”。

## 2. 问题定义

### 2.1 当前问题

当前 `AgentAutonomyService` 的主要模式是：

```text
如果威胁等级 >= 50
并且同伴有火把
并且同伴知道火把相关知识
则同伴举火把。
```

这在 P38 是合理的最小闭环，但它不是通用推理。

更通用的逻辑应该是：

```text
Agent 当前有一个目标：降低威胁。
Agent 知道某个行为效果：火把对野兽使用 -> 降低威胁。
Agent 检查火把是否存在。
如果火把不存在，继续找“生成火把”的知识。
生成火把需要处理木材和火源。
处理木材需要斧头砍木头。
如果斧头钝了，需要打磨斧头。
最终形成计划链。
```

也就是说，P39 要把“行为规则触发”升级成“目标驱动的知识规划”。

### 2.2 用户期望的典型案例

#### 案例 A：同伴饿了自己吃红果

```text
已知知识：红果 + 吃 -> 缓解饥饿。
当前状态：同伴 hunger 高。
推理：缓解饥饿能改善 hunger 目标。
计划：吃红果。
执行：红果数量减少，同伴 hunger 下降。
```

系统不应该写死“饿了吃红果”，而应该从 `restore_hunger` 效果语义推出。

#### 案例 B：寒冷时优先生火

```text
已知知识：火堆 -> 御寒。
已知知识：房子 -> 御寒。
当前状态：天气冷，族群 warmth 不足。
候选计划 1：点燃干草 -> 火堆 -> 御寒。
候选计划 2：造房子 -> 需要木头 -> 砍木头 -> 需要斧头锋利 -> 打磨斧头 -> 造房。
评分：生火链短、代价低、即时收益高。
选择：先生火。
```

#### 案例 C：孩子增加，房子不够，族人造房

```text
已知知识：房子容量可以提供御寒和居住容量。
当前状态：child_count 增加，shelter_capacity < population，天气变冷。
目标：提升 shelter_capacity / warmth。
候选计划：造房子。
前置链：需要木头 -> 砍木头；斧头钝 -> 打磨斧头。
执行：按计划逐步准备材料和工具。
```

#### 案例 D：野兽迂回，同伴用火把

```text
已知知识：火把对野兽使用 -> 降低威胁。
当前状态：野兽绕过火堆继续靠近。
目标：降低 threat.beast。
候选计划：用火把驱赶野兽。
条件：火把存在，同伴知道该知识，威胁达到阈值。
执行：同伴举火把协助。
```

## 3. 阶段目标

P39 完成后，系统必须具备以下能力：

1. 能把知识效果 key 转换为机器可理解的语义效果。
2. 能从 Agent 当前需求、危险、族群状态、天气状态生成目标。
3. 能从 Agent 已掌握知识中生成候选行动。
4. 能判断候选行动是否直接可执行。
5. 能在不可执行时寻找前置知识链。
6. 能构建有限深度的行动计划链。
7. 能对多个计划按收益、风险、时间、资源、可信度、紧急度评分。
8. 能选择最合适的计划，而不是总选第一个。
9. 能输出可解释的计划理由，方便 H5 和审计报告展示。
10. 能接入现有 `AgentAutonomyService`，替代单点写死策略。
11. 能阻止无限递归、循环依赖和不可信知识导致的乱行动。
12. 能用测试验证“饿了吃已知食物”“冷了优先生火”“房子不够链式准备材料”“火把驱赶野兽”等场景。

## 4. 本阶段不解决的问题

### 4.1 不做完整 GOAP/HTN 引擎

P39 做“受限链式目标规划”，不做完整通用 AI 规划器。

不做：

```text
任意谓词演算
复杂 HTN 域语言
无限搜索
概率规划
多智能体联合拍卖
复杂路径规划
完整建筑施工模拟
```

原因：当前项目需要的是可控、可测试、可解释的最小规划层，而不是不可控黑盒 AI。

### 4.2 不做大模型推理

P39 不调用外部 AI API，也不让大模型在运行时决定行动。

所有运行时决策必须来自：

```text
KnowledgeClaim
EffectSemantics
WorldSnapshot
AgentNeedState
ConditionExpression
UtilityScorer
```

大模型可以辅助开发文档和内容配置，但不能成为规则权威。

### 4.3 不做完整建筑系统

本阶段可以设计“房子提供御寒/容量”的语义和测试 fixture，但不实现完整建筑 UI、地图占地、施工时间、耐久和维护。

房子在 P39 中只作为计划链样例和最小世界对象：

```text
house -> provides_warmth / provides_shelter_capacity
build_house -> generates house
```

完整建筑放到后续内容/基地系统阶段。

### 4.4 不做正式天气系统

P39 可以定义 `WeatherPressure` / `cold_exposure` 作为目标来源，但不做完整季节、温度曲线、雨雪模拟。

正式天气可以后续基于目标生成器扩展。

### 4.5 不做多 Agent 任务调度

P39 先支持单个 Agent 规划。

多 Agent 分工，例如：

```text
A 去砍木头，B 去生火，C 照顾孩子。
```

可以作为后续族群任务调度阶段。

P39 只要求：一个 Agent 能从自己的知识和可见世界中推导一条行动链。

## 5. 核心设计原则

### 5.1 知识驱动，不写死对象名

禁止实现：

```cpp
if (agent.hunger > 70) eat("red_berry");
```

必须实现为：

```text
目标：降低 hunger。
查找知识：哪些行动效果能降低 hunger。
找到：red_berry + eat -> restore_hunger。
检查对象数量和风险。
生成候选行动。
```

红果只是内容，不是架构规则。

### 5.2 效果语义是推理桥梁

`restore_hunger`、`repel_beast`、`make_torch`、`provide_warmth` 不能只是中文展示 key。

它们必须映射到机器可理解的语义：

```text
restore_hunger -> need.hunger -30
repel_beast -> threat.beast -100
make_torch -> object.torch +1
provide_warmth -> need.cold -40 / group.warmth +40
build_house -> object.house +1 / shelter.capacity +N
```

没有语义的知识不能参与自动推理，只能用于展示。

### 5.3 推理只使用主观知识

Agent 只能用自己知道的知识规划。

例如：

```text
玩家知道红果能吃，但同伴没被教过 -> 同伴不能用该知识自动吃红果。
族群知识中有“房子御寒” -> 普通族人可以根据族群知识规划造房。
野兽观察到火危险 -> 野兽可以避火或绕火。
```

禁止直接读取 ObjectDefinition 的隐藏真相来决策。

### 5.4 规划不直接改世界

P39 的规划器只输出计划，不直接修改状态。

状态修改仍然必须通过：

```text
WorldInteractionService / ReactionResolver / RulePipeline
-> WorldChange
-> WorldChangeApplier
```

禁止规划器自己扣资源、生成物品、修改 hunger。

### 5.5 选择必须可解释

每个计划必须能解释：

```text
为什么选它。
它解决哪个目标。
用了哪条知识。
需要哪些前置步骤。
为什么没选其他方案。
风险是什么。
```

H5 可以显示简化版，例如：

```text
同伴选择生火，因为它能最快缓解寒冷；造房也能御寒，但需要先准备木头。
```

### 5.6 搜索必须受限

为了防止规划爆炸，必须限制：

```text
max_depth
max_candidates_per_goal
max_plan_steps
max_total_expansions
禁止重复状态
禁止重复 effect_key 循环
```

例如：

```text
造房需要木头 -> 木头需要斧头 -> 斧头需要房子保护
```

这种循环必须被检测并拒绝。

### 5.7 风险和可信度必须参与评分

知识不是绝对真理。

规划必须考虑：

```text
active > hypothesis > candidate
负面效果降低评分
冲突知识降低评分
近期因果不明确降低评分
危险条件提高紧急度
```

例如：

```text
腐烂红果也可能缓解饥饿，但已知会中毒 -> 不选。
苦叶无明显效果 -> 低收益。
红果已确认缓解饥饿 -> 高评分。
```

### 5.8 近期最小链优先，但不是永远最短

当目标紧急时，短链优先。

例如寒冷马上造成伤害：

```text
生火 1-2 步 -> 先做。
造房 5 步 -> 放入后续计划。
```

但如果短链资源即将耗尽、风险高、效果短暂，系统可以选择长链。

例如：

```text
火堆只能撑一晚，寒潮持续多天，孩子增加 -> 造房优先级上升。
```

## 6. 模块划分

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| `pathfinder_agent_reasoning` | Agent 推理模块 | 承载目标生成、候选行动、计划搜索和评分 | WorldSnapshot、KnowledgeClaim、AgentNeedState | AgentPlan / ReasoningTrace | 不直接改世界 |
| `EffectSemanticsRegistry` | 效果语义注册表 | 把 effect_key 映射为机器可理解效果 | effect_key、配置、内置表 | EffectSemantics | 不读取隐藏真相 |
| `AgentGoalGenerator` | 目标生成器 | 从需求、威胁、族群状态生成目标 | Agent 状态、世界、危险 | AgentGoal 列表 | 不创建行动 |
| `KnowledgeActionCandidateBuilder` | 知识候选构建器 | 从主观知识生成候选行动 | Agent 知识、世界对象 | ActionCandidate 列表 | 不使用未掌握知识 |
| `PlanPreconditionResolver` | 前置条件解析器 | 为不可执行行动寻找准备链 | 候选行动、世界状态、知识 | PlanStep 树 | 不无限递归 |
| `PlanUtilityScorer` | 计划评分器 | 按收益、代价、风险、可信度排序 | AgentGoal、AgentPlan | 分数和解释 | 不随机裁决 |
| `AgentReasoner` | 总编排器 | 生成并选择计划 | ReasoningRequest | ReasoningResult | 不绕过规则管线 |
| `AgentPlanExecutorAdapter` | 计划执行适配器 | 把计划第一步转为现有交互输入 | AgentPlan、WorldSnapshot | WorldInteractionInput 或 AgentAutonomyResult | 不一次执行整条长链 |
| `ReasoningProjectionMapper` | 推理投影 | 输出安全中文解释 | ReasoningTrace | H5 安全摘要 | 不泄露 raw state |

## 7. 核心枚举

### 7.1 `AgentGoalKind` / Agent 目标类型

中文：Agent 当前想改善的目标类型。

枚举值：

| 枚举值 | 中文 | 使用场景 |
|---|---|---|
| `Unknown` | 未知 | 反序列化兜底，不允许运行时作为有效值 |
| `ReduceHunger` | 降低饥饿 | hunger 超阈值 |
| `RestoreHealth` | 恢复健康 | health 低或中毒 |
| `ReduceCold` | 降低寒冷 | cold_exposure 高、天气冷 |
| `IncreaseWarmth` | 提升保暖 | 族群 warmth 不足 |
| `IncreaseShelterCapacity` | 提升庇护容量 | 人口/孩子数量超过房屋容量 |
| `ReduceThreat` | 降低威胁 | 野兽、敌对族群、环境危险逼近 |
| `AcquireObject` | 获得对象 | 需要火把、房子、工具、材料 |
| `RestoreToolState` | 恢复工具状态 | 斧头变钝、火把将灭 |
| `MaintainFire` | 维持火源 | 火堆快灭、天气寒冷 |
| `ProtectDependent` | 保护依赖者 | 小孩、伤员、老人需要照顾 |
| `TestOnly` | 测试专用 | 单元测试 fixture |

规则：

```text
Unknown 不允许进入评分。
TestOnly 只能测试使用。
字符串转换必须稳定 snake_case。
前端不得自定义 AgentGoalKind。
```

### 7.2 `EffectSemanticKind` / 效果语义类型

中文：知识效果对世界或目标的机器含义。

枚举值：

| 枚举值 | 中文 | 示例 |
|---|---|---|
| `NeedDelta` | 需求数值变化 | hunger -30、cold -20 |
| `ConditionDelta` | 状态变化 | poison +1、wounded -1 |
| `ThreatDelta` | 威胁变化 | beast_threat -100 |
| `ObjectQuantityDelta` | 对象数量变化 | torch +1、wood -1 |
| `ObjectStateDelta` | 对象状态变化 | axe.sharpness +3 |
| `CapabilityDelta` | 能力变化 | shelter_capacity +4 |
| `KnowledgeDelta` | 知识变化 | teach_back / warning |
| `LocationSafetyDelta` | 地点安全变化 | camp_safety +20 |
| `TimeCost` | 时间消耗 | cost_ticks +8 |
| `RiskDelta` | 风险变化 | poison_risk +50 |
| `TestOnly` | 测试专用 | fixture |

### 7.3 `PlanStepKind` / 计划步骤类型

| 枚举值 | 中文 | 说明 |
|---|---|---|
| `DirectAction` | 直接行动 | 当前可立即执行 |
| `PrepareObject` | 准备对象 | 例如制作火把、砍木头 |
| `RestoreTool` | 恢复工具 | 例如打磨斧头 |
| `BuildStructure` | 建造结构 | 例如造房子 |
| `TeachKnowledge` | 传播知识 | 先教同伴再让其执行 |
| `WaitForCondition` | 等待条件 | 等火燃烧、等待天气变化，不得无限等 |
| `Abort` | 放弃计划 | 风险或条件不满足 |
| `TestOnly` | 测试专用 | 测试 |

### 7.4 `PlanSelectionReason` / 计划选择原因

| 枚举值 | 中文 |
|---|---|
| `HighestUtility` | 综合收益最高 |
| `ShortestChain` | 链路最短 |
| `LowestRisk` | 风险最低 |
| `OnlyExecutable` | 唯一可执行 |
| `EmergencyOverride` | 紧急状态强制选择 |
| `InsufficientKnowledge` | 知识不足，不能形成计划 |
| `BlockedByResource` | 资源不足 |
| `BlockedByCondition` | 条件不足 |
| `CycleDetected` | 检测到循环依赖 |
| `SearchLimitReached` | 达到搜索上限 |

### 7.5 `KnowledgeUsability` / 知识可用性

| 枚举值 | 中文 | 说明 |
|---|---|---|
| `Usable` | 可用于规划 | active 或足够可信 hypothesis |
| `Tentative` | 可试探 | candidate/hypothesis，低风险时可用 |
| `Conflicted` | 冲突 | 存在相反知识，降低评分或禁止 |
| `Dangerous` | 危险 | 已知负面效果，默认避开 |
| `UnknownEffect` | 效果无语义 | 只能展示，不能规划 |
| `NotOwnedByAgent` | Agent 不知道 | 不允许使用 |

## 8. 核心 DTO / 数据结构

### 8.1 `EffectSemantics`

中文：效果语义。把知识效果 key 转成机器可规划信息。

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `effect_key` | string | 效果 key | 是 | 内置表/配置 | 推理器 | 非空，稳定 snake_case | 中文句子 |
| `display_zh_cn` | string | 中文解释 | 是 | 配置 | 投影 | 安全文本 | raw_state |

## 9. 核心服务接口

### 9.1 `EffectSemanticsRegistry`

职责：注册和查询 effect_key 语义。

接口：

```cpp
class EffectSemanticsRegistry {
public:
    Result<EffectSemantics> findByEffectKey(const std::string& effect_key) const;
    Result<std::vector<EffectSemantics>> findByGoalKind(AgentGoalKind goal_kind) const;
    Result<void> validateAll() const;
};
```

规则：

```text
内置最小表必须覆盖 restore_hunger、poison、no_visible_effect、cut_wood、restore_sharpness、ignite_fire、make_torch、repel_beast、provide_warmth、build_house。
找不到语义时，知识仍可展示，但不能进入规划。
```

### 9.2 `AgentGoalGenerator`

职责：从状态生成目标。

接口：

```cpp
class AgentGoalGenerator {
public:
    Result<std::vector<AgentGoal>> generateGoals(const GoalGenerationInput& input) const;
};
```

输入包括：

```text
AgentNeedState
WorldSnapshot
ThreatRuntimeState
TribeStatusProjection / TribeState 摘要
WeatherPressure 摘要
```

最小目标规则：

```text
hunger >= 60 -> ReduceHunger
cold >= 50 -> ReduceCold / IncreaseWarmth
threat.level >= 50 -> ReduceThreat
shelter_capacity < population && cold >= 30 -> IncreaseShelterCapacity
tool.sharpness <= 0 && 目标链需要砍木头 -> RestoreToolState
```

### 9.3 `KnowledgeActionCandidateBuilder`

职责：从主观知识生成候选行动。

接口：

```cpp
class KnowledgeActionCandidateBuilder {
public:
    Result<std::vector<ActionCandidate>> buildCandidates(const CandidateBuildInput& input) const;
};
```

规则：

```text
只读取 actor_knowledge 或 tribe_knowledge 中该 Agent 可用的知识。
必须保留 target_object_key。
必须检查 KnowledgeUsability。
必须从 EffectSemanticsRegistry 找语义。
不得根据 ObjectDefinition 隐藏属性创建候选。
```

### 9.4 `PlanPreconditionResolver`

职责：为候选行动补齐前置链。

接口：

```cpp
class PlanPreconditionResolver {
public:
    Result<AgentPlan> resolvePreconditions(const PreconditionResolveInput& input) const;
};
```

例子：

```text
目标：AcquireObject(torch)
候选：make_torch
前置：wood.processed > 0、camp_fire.quantity > 0
wood.processed 缺失 -> 找 cut_wood
cut_wood 前置：axe.sharpness > 0、wood.quantity > 0
axe.sharpness 缺失 -> 找 restore_sharpness
```

循环检测：

```text
visited_effect_keys
visited_missing_keys
visited_object_action_pairs
```

### 9.5 `PlanUtilityScorer`

职责：给计划评分。

接口：

```cpp
class PlanUtilityScorer {
public:
    Result<PlanScore> score(const PlanScoreInput& input) const;
};
```

建议公式：

```text
utility = goal_urgency * expected_benefit
          - total_time_cost * time_weight
          - total_risk_score * risk_weight
          - resource_cost * resource_weight
          + confidence_bonus
          - chain_length_penalty
```

紧急目标调整：

```text
如果 urgency >= 80，time_weight 提高。
如果 health <= 30，risk_weight 提高。
如果 food 数量很少，resource_weight 提高。
```

### 9.6 `AgentReasoner`

职责：总推理入口。

接口：

```cpp
class AgentReasoner {
public:
    Result<ReasoningResult> reason(const ReasoningRequest& request) const;
};
```

流程：

```text
生成目标。
按紧急度排序。
为每个目标生成候选行动。
为候选行动解析前置链。
评分。
选择计划。
输出解释和 trace。
```

### 9.7 `AgentPlanExecutorAdapter`

职责：把计划第一步交给现有系统执行。

接口：

```cpp
class AgentPlanExecutorAdapter {
public:
    Result<AgentAutonomyResult> toAutonomyResult(const AgentPlan& plan, const WorldSnapshot& snapshot) const;
    Result<WorldInteractionInput> toWorldInteractionInput(const AgentPlanStep& step, const WorldSnapshot& snapshot) const;
};
```

规则：

```text
每回合只执行第一步。
执行成功后下一回合重新推理，不盲目继续旧计划。
执行失败必须进入 trace，不得静默吞掉。
```

### 9.8 `ReasoningProjectionMapper`

职责：输出玩家可理解的推理说明。

示例输出：

```text
同伴选择生火，因为它能最快缓解寒冷。
同伴暂时没有造房，因为造房还需要先准备木头。
同伴没有吃腐烂红果，因为已知它可能导致中毒。
```

禁止输出：

```text
raw_state
hidden_truth
utility_score 原始调试值
未投影的内部阈值
```

## 10. 标准流程

### 10.1 单回合 Agent 推理流程

```text
1. DialogTurnService / 世界调度器完成玩家行动结算。
2. WorldSnapshotAdapter 生成当前 WorldSnapshot。
3. AgentNeedStateProvider 读取 Agent 需求状态。
4. KnowledgeProvider 读取该 Agent 可用知识。
5. AgentReasoner 生成目标。
6. KnowledgeActionCandidateBuilder 生成候选行动。
7. PlanPreconditionResolver 尝试补齐链式前置。
8. PlanUtilityScorer 评分。
9. AgentReasoner 选择一个计划。
10. AgentPlanExecutorAdapter 只执行计划第一步。
11. 执行仍走 WorldInteractionService / WorldChangeApplier。
12. ReasoningProjectionMapper 输出安全解释。
13. H5 只显示投影，不参与推理。
```

### 10.2 饿了吃红果流程

```text
输入：同伴 hunger = 80。
目标：ReduceHunger，urgency = 80。
知识：red_berry + eat -> restore_hunger。
语义：restore_hunger -> hunger -30。
候选：吃红果。
前置：red_berry.quantity > 0。
评分：收益高、风险低、链长 1。
执行：吃红果。
后果：红果 -1，同伴 hunger 降低。
```

### 10.3 冷了优先生火流程

```text
输入：cold = 70，天气冷。
目标：ReduceCold / IncreaseWarmth。
候选 A：ignite_fire -> provide_warmth，链长 1-2。
候选 B：build_house -> provide_warmth + shelter_capacity，链长 4-6。
评分：A 时间短，紧急目标优先。
执行：先点火。
保留：B 作为后续长期计划。
```

### 10.4 造房链式推导流程

```text
输入：population = 4，shelter_capacity = 2，cold = 50。
目标：IncreaseShelterCapacity。
候选：build_house。
前置：wood_processed >= 3。
缺 wood_processed -> 找 cut_wood。
cut_wood 前置：wood.quantity > 0、axe.sharpness > 0。
axe.sharpness = 0 -> 找 restore_sharpness。
计划链：restore_sharpness -> cut_wood -> cut_wood -> cut_wood -> build_house。
评分：长链，但解决长期容量问题。
```

### 10.5 野兽迂回与同伴火把流程

```text
输入：beast_threat >= 50，camp_fire 存在但野兽已观察并迂回。
目标：ReduceThreat。
候选：torch + use @ beast -> repel_beast。
前置：torch.quantity > 0。
同伴知识：repel_beast 已传播。
评分：直接解决威胁，紧急。
执行：同伴举火把。
```

## 12. 计划评分细节

### 12.1 基础评分维度

| 维度 | 含义 | 倾向 |
|---|---|---|
| `expected_benefit` | 对目标改善幅度 | 越高越好 |
| `urgency` | 目标紧急程度 | 越高越优先 |
| `time_cost` | 总耗时 | 越低越好 |
| `resource_cost` | 消耗资源价值 | 越低越好 |
| `risk_score` | 负面风险 | 越低越好 |
| `knowledge_confidence` | 知识可信度 | 越高越好 |
| `chain_length` | 计划步骤数 | 越短越好 |
| `durability` | 效果持续性 | 越高越好 |

### 12.2 生火 vs 造房示例评分

寒冷紧急时：

```text
生火：收益 40，耗时 1，风险 10，链长 1，持续性中。
造房：收益 50，耗时 6，风险 15，链长 5，持续性高。
紧急度 80 -> time_weight 高 -> 生火胜出。
```

长期寒冷时：

```text
cold = 40，天气持续，人口增长。
生火：短期收益，资源持续消耗。
造房：长链但持续性高，容量收益高。
紧急度较低 -> durability_weight 提高 -> 造房进入优先计划。
```

## 13. 头脑风暴：后续真实场景

### 13.3 工具链

```text
斧头钝了先打磨。
没有斧头先制作斧头。
制作斧头需要石片和木柄。

野兽怕火但会迂回。

房子御寒。

狼学会火危险后绕开火堆。

### 13.7 社会与信任

同伴不信任玩家时，不执行高风险计划。
族人知道毒蘑菇有毒，玩家还让 NPC 吃，会判断玩家可能在害他。
专家 NPC 会根据学生目标选择教学内容。
族群领袖会选择长期收益计划，而孩子会选择即时需求计划。
```

### 13.8 资源竞争

```text
最后一个红果是给玩家吃、同伴吃还是孩子吃。
木头用于火堆、火把、房子还是陷阱。
短期生火和长期造房冲突。
打磨斧头花时间，野兽已接近，是否冒险直接用钝斧。
```

### 13.9 知识冲突

```text
一个族人认为红果安全，另一个族人认为红果有毒。
经验来自腐烂红果，不能错误推广到新鲜红果。
天气变化导致旧知识部分失效：湿木头不能点火。
新工具出现后，旧计划不再最优。
```

### 13.10 强化学习接入

P39 的计划和评分可以为后续训练提供：

```text
observation：目标、候选计划、计划评分、阻塞原因。
action：选择计划或计划第一步。
reward：目标改善、风险降低、资源节省。
done：目标达成、失败、超时。
explanation：可解释策略标签。
```

注意：RL 可以学习调整权重，但不能绕过规则权威。

## 18. 风险与对策

### 18.1 风险：规划器变成新规则大杂烩

对策：

```text
所有效果必须先进入 EffectSemantics。
所有候选必须来自 KnowledgeClaim 或配置反应。
规划器不得写 object_key 专属行为。
```

### 18.2 风险：链式搜索爆炸

对策：

```text
max_depth <= 4。
max_total_expansions <= 64。
每目标最多 8 个候选。
循环检测必须测试。
```

### 18.3 风险：Agent 用了自己不知道的知识

对策：

```text
CandidateBuildInput 必须传 owner knowledge。
测试中玩家知道但同伴不知道时，同伴不得行动。
```

### 18.4 风险：短期行动永远压过长期建设

对策：

```text
评分引入 durability 和长期压力。
紧急度低时长期收益权重上升。
```

### 18.5 风险：前端显示像调试器

对策：

```text
ReasoningProjectionMapper 输出中文自然解释。
内部分数和搜索树只进入审计，不进 Player 投影。
```

## 22. 结论

P39 是从“规则触发型 Agent”升级到“知识推理型 Agent”的关键阶段。

完成 P39 后，系统应该开始具备以下能力：

```text
不是因为程序员写了“饿了吃红果”，同伴才吃红果；
而是因为同伴知道“红果能缓解饥饿”，又真的饿了，所以推理出吃红果。

不是因为程序员写了“冷了造房”，族人才造房；
而是因为族群知道“房子能御寒/提供容量”，又发现孩子增加、房子不够、天气变冷，所以推理出造房，并进一步推导出要砍木头、要打磨斧头。

不是因为程序员写了“火把驱赶野兽”，同伴才举火把；
而是因为同伴知道“火把能降低野兽威胁”，世界里有火把，野兽正在逼近，所以选择该计划。
```

这套能力是后续族群自治、野生 NPC 学习、专家 NPC 教学、文明建设和强化学习接入的基础。
