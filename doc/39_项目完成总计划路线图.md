# 《先驱者》项目完成总计划路线图

## 1. 总结论

本路线图不是重新设计，而是在现有 `01-24` 总设计文档与 `P0-P13` 阶段任务卡基础上，继续向“可玩、可审计、可扩展、可交给 AI 稳定实现”的完整游戏推进。

核心原则：

```text
先地基，再玩法。
先可观察，再智能。
先安全输入，再真实数据。
先单体闭环，再族群文明。
先本地工具，再 H5 表现。
手机可玩优先时，允许插入最小 H5 文本入口验证体验。
先确定性复盘，再训练/AI增强。
```

当前已完成/已设计到：

```text
P0-P4：后端基础、配置命令、状态事件管线、未知对象首条规则、复盘协议前置。
P5-P14：Agent 定义、观察动作空间、运行时、复盘锁定、训练样本草稿、历史投影、本地调试、真实安全输入适配。
P15-P21：认知、记忆、知识、传播和学习闭环主线。
P22-P28.5：H5 中文对话、族群状态、共同作战、族群冲突、文明能力、条件表达式、对象反应、配置化物品与反应流水线。
P29-P36：危险探索、存档前置、真实复盘、协议投影、H5 验证壳、因果归因、状态化对象、行为目标知识。
```

后续总路线：

```text
P37-P42：最小剧情闭环、世界交互结算与可见后果、H5 玩家界面去调试化、知识链式推理、动态目标执行、基础内容包、继续游戏体验。
P43-P44：训练环境前置、压测与性能。
P45-P48：Alpha、Beta、发布候选、1.0 发布。
```

## 2. 完成标准

项目“完成”不是指所有想法都实现，而是达到下面标准：

```text
玩家可以从无知状态开始探索未知对象。
尝试吃/用/组合会产生反馈。
反馈会形成认知。
认知会进入记忆。
记忆会沉淀为知识。
知识可以传播给个体/族群。
族群行为会影响协作、冲突、分裂、文明能力。
世界对象可以组合反应。
所有关键状态变化有事件。
所有关键命令可复盘。
前端只展示后端投影，不复制规则。
调试工具能解释 Agent 为什么这样行动。
训练/AI 接入不会绕过规则权威。
```

## 4. P13-P14：调试输入链路收口

### P13：Agent 本地调试 CLI 前置

目标：

```text
提供最小 CLI，用内置 fixture 调用 P11/P12，生成本地 txt/md 调试文件。
```

交付：

```text
AgentDebugCliOptions。
AgentDebugCliParser。
AgentDebugFixtureFactory。
AgentDebugCliRunner。
pathfinder_agent_debug_cli。
```

验收：

```text
--help 成功。
unknown_fruit markdown 导出成功。
no_decision dry-run 不写文件。
--json / --load-save / 非法 base_name 失败。
P13/P12/P11/P10 回归通过。
```

### P14：Agent 调试输入适配前置

目标：

```text
定义 CLI 未来如何接入真实但安全的数据来源，不直接读 GameState 或真实存档内部结构。
```

交付：

```text
AgentDebugInputSource。
AgentDebugInputManifest。
AgentDebugInputAdapter。
AgentDebugInputValidationReport。
只读输入包格式草案。
```

边界：

```text
不做 JSON。
不做 SaveManager。
不读完整存档。
不读隐藏真相。
只允许输入 P10/P11/P12 安全投影或测试包。
```

验收：

```text
安全输入包可转换成 AgentDebugReport。
非法输入包失败。
隐藏真相字段失败。
CLI 可选择 input source，但仍只走安全 DTO。
```

## 5. P15-P21：认知、记忆、知识、传播学习主线

### P15：认知系统正式化

目标：

```text
把 P3/P6 的最小 CognitionClaim 扩展为正式认知层。
```

解决问题：

```text
一个生物不关心物体颜色形状，而关心它能否食用、能否使用、使用后效果、风险、证据来源和置信度。
```

交付：

```text
CognitionSubject。
CognitionTarget。
CognitionAspect。
CognitionClaimV2。
CognitionEvidenceRecord。
CognitionConfidenceModel。
CognitionQueryService。
```

验收：

```text
吃一口会生成 edible/risk/effect 相关认知。
用一下会生成 usable/effect 相关认知。
认知不读取 ObjectDefinition hidden truth。
认知只由反馈和事件更新。
```

### P16：记忆系统

目标：

```text
实现短期、中期、长期记忆的基础结构。
```

交付：

```text
MemoryRecord。
MemoryScope：ShortTerm / MidTerm / LongTerm。
MemoryStrength。
MemoryDecayPolicy。
MemoryStore。
MemoryWriteService。
```

验收：

```text
一次尝试行为可以形成短期记忆。
重复证据可以增强记忆。
低强度短期记忆会衰减。
关键事件可晋升中期/长期记忆。
```

### P17：记忆压缩与检索

目标：

```text
让大量经验可以压缩成可用摘要，并能被 Agent 查询。
```

交付：

```text
MemoryCompressionPolicy。
MemorySummary。
MemoryRecallQuery。
MemoryRecallResult。
MemoryDecompressionView。
```

验收：

```text
多次吃同类对象可压缩成“该类对象大概率可食用/有害”的摘要。
Agent Observation 可读取记忆摘要，但不能读取隐藏真相。
记忆压缩可复盘。
```

### P18：知识系统

目标：

```text
从稳定记忆中形成可传播、可引用的知识声明。
```

交付：

```text
KnowledgeClaim。
KnowledgeSource。
KnowledgeScope。
KnowledgeConfidence。
KnowledgeRepository。
KnowledgeFormationService。
```

验收：

```text
长期记忆可以形成知识。
知识保留来源和证据链。
知识可以被 Agent/族群查询。
知识不直接等同真实 ObjectDefinition。
```

### P19：知识冲突与修正

目标：

```text
允许错误知识存在、传播、被证伪、被修正。
```

交付：

```text
KnowledgeConflict。
KnowledgeRevision。
ContradictionDetector。
KnowledgeMergePolicy。
KnowledgeTrustModel。
```

验收：

```text
“可食用”和“有害”知识可共存但形成冲突。
新证据可降低旧知识置信度。
修正过程有事件和复盘记录。
```

### P20：传播系统

目标：

```text
让一个个体把知识传给其他个体或族群。
```

交付：

```text
PropagationChannel。
TeachingAction。
PropagationAttempt。
PropagationResult。
KnowledgeTransferPolicy。
```

验收：

```text
个体可 Teach/Share/Signal。
被传授者获得知识投影而非真实隐藏属性。
传播受信任、距离、语言/文明能力影响。
错误知识也可以传播。
```

### P21：学习闭环

目标：

```text
打通“尝试 -> 反馈 -> 认知 -> 记忆 -> 知识 -> 传播 -> 他人行为变化”。
```

交付：

```text
LearningLoopService。
LearningEventChain。
LearningDebugReport。
LearningReplayTest。
```

验收：

```text
先驱者吃未知果实后形成认知和记忆。
重复尝试后形成知识。
传授给同族后，同族观察中知道该对象可食用/危险。
全链路可导出调试报告。
```

## 6. P22：H5 中文对话框最小可玩文本游戏

路线调整说明：

```text
原 P22 是族群状态。
因为当前最需要验证“手机上能不能玩”，P22 调整为 H5 中文对话框最小可玩。
原 P22-P27 的族群、共同作战、冲突、文明、对象反应、危险阶段整体顺延为 P23-P28。
这不是重做架构，而是在 P21 学习闭环之后插入一个最小玩家入口。
```

### P22：H5 中文对话框最小可玩

任务卡：`doc/48_P22H5中文对话框最小可玩任务卡设计.md`。
上下文包：`context_packs/h5_dialog_p22.md`。

目标：

```text
手机通过内网穿透访问 1999 端口。
页面只有中文对话框、中文输入、中文反馈和少量快捷按钮。
玩家通过中文行动完成“观察 -> 尝试 -> 学习 -> 形成知识 -> 教给同伴 -> 同伴认知变化”的第一轮可玩体验。
```

交付：

```text
H5DialogServer：本地 1999 服务，同时提供静态页面和最小 dialog 接口。
H5DialogPage：中文对话框网页，适配手机竖屏。
DialogSessionState：最小会话状态，记录当前故事、角色、已执行行动和展示摘要。
DialogIntentParser：中文指令映射，不做自由 NLP。
DialogScenarioCatalog：可配置故事/对象目录，避免把知识写死在前端。
DialogResponseDto：中文回复、状态摘要、快捷行动、调试 key。
LearningLoopAdapter：把对话意图转成 P21 LearningLoopInput，并读取 P21 输出。
```

第一版指令映射：

```text
“观察”“看看周围” -> observe。
“吃红果”“吃一口红果” -> eat red_berry。
“再吃一口”“继续吃红果” -> repeat eat red_berry。
“告诉同伴红果能吃”“教学”“教同伴” -> teach red_berry。
“查看知识”“我知道什么” -> inspect actor knowledge。
“查看同伴”“同伴知道什么” -> inspect recipient knowledge。
“吃腐烂红果”“尝试腐烂红果” -> eat decayed_red_berry。
```

防止写死知识的要求：

```text
H5 前端不得写死“红果可食用”。
DialogIntentParser 只识别玩家意图，不产生知识结论。
DialogScenarioCatalog 可以配置多个对象和安全反馈，但知识必须由 P21/P15-P20 形成。
DialogResponseDto 的中文结论必须来自后端 LearningLoopResult、LearningEventChain、LearningDebugReport 的摘要。
同一个流程至少覆盖 3 类对象/情况，不能只有红果硬编码：
1. red_berry：吃后产生 restore_hunger 反馈，用于形成正向食用知识。
2. decayed_red_berry：吃后产生 poison/risk 反馈，用于验证旧知识修正。
3. bitter_leaf 或 stone_flake：产生 no_visible_effect/use_hint/risk 等不同反馈，用于证明对话层不是只服务红果。
```

界面要求：

```text
全中文。
手机竖屏可读。
一个主对话区。
一个输入框。
一个发送按钮。
少量快捷按钮：观察、吃一口、教学、查看知识、查看同伴。
一个状态摘要区：当前主体、同伴、当前可见对象、我的知识、同伴知识。
```

接口要求：

```text
第一版只需要 POST /api/dialog。
请求包含 session_id 和中文 input_text。
响应包含 reply_text、state_summary、quick_actions、debug_keys。
不做 WebSocket。
不做账号。
不做云存档。
不接入大语言模型自由聊天。
不让 H5 推导规则结果。
```

验收：

```text
手机能通过 1999 页面打开游戏。
页面、按钮、回复全部是中文。
玩家能完成红果学习故事。
玩家能完成红果教学传播故事。
玩家能触发腐烂红果纠错故事。
至少 3 类对象/情况通过同一套对话与学习闭环运行。
知识结论来自 P21 学习闭环，不来自前端硬编码。
前端只展示后端投影和中文摘要。
```

验收：

```text
负面反馈会形成候选原因，而不是直接覆盖所有旧知识。
重复证据、时间间隔和对象差异会影响归因置信度。
归因结果可以进入知识更新和 H5 中文解释。
```

### P35：状态化对象规则后端最小实现

目标：

```text
让对象拥有可演化状态，例如斧头锋利、变钝、磨石恢复，让“工具会损耗和维护”成为规则层能力。
```

验收：

```text
斧头砍木头会消耗锋利度。
斧头变钝后砍木头反馈不同。
磨石可以恢复斧头状态。
状态变化有事件、trace 和知识草稿。
```

### P36：行为目标知识最小升级

目标：

```text
让知识支持“用什么，对什么，满足什么条件，产生什么效果”，例如用斧头砍木头、用磨石打磨斧头。
```

验收：

```text
知识保留 subject/action/target/condition/effect。
教学传播不得丢失 target 或 condition。
H5 能显示中文知识注释，而不是只显示调试 key。
```

## 9. P37-P42：从验证器走向最小可玩游戏

### P37：最小剧情闭环

目标：

```text
把已有规则串成 5-10 分钟“第一天生存”流程：探索、尝试、学习、修正、教同伴、使用工具/反应解决一个危险。
```

原则：

```text
不新增大地图。
不做复杂战斗。
不做自由 AI 文本。
先证明玩家能玩懂、能做决策、能看到知识产生价值。
```

最低内容：

```text
红果/腐烂红果/苦叶/石片/斧头/木头/磨石/火把或简易营地。
至少一个必须用知识解决的危险。
至少一次“错误或不确定知识被修正”的体验。
至少一次“教同伴后同伴行为或状态改变”的体验。
```

### P38：世界交互结算与可见后果

目标：

```text
把 P37 的“按钮形成知识”升级为“动作改变世界”：物品会消耗或生成，工具会损耗或恢复，火源和火把可见，野兽威胁会推进与结算，同伴学会知识后能在危险中自动协助。
```

验收：

```text
玩家能看到红果减少、木头被处理、干草被点燃、火源出现、火把出现、野兽退去、同伴协助。
所有 H5 按钮点击后不能出现 feedback_not_found。
前端仍只消费后端 projection，不裁决世界变化。
```

### P38.5：H5 玩家界面去调试化

目标：

```text
在 P38 世界变化成立后，把 H5 从系统验证器改成玩家界面，保留调试入口但默认隐藏 key、状态码和工程词。
```

验收：

```text
玩家看到中文叙事、可行动作、结果反馈、我的知识、同伴知识、当前目标。
知识解释用自然中文，同时内部保留稳定 key 方便复盘。
前端仍不推导规则。
```

### P39：知识链式推理与目标规划

目标：

```text
让 Agent 根据自己掌握的知识、当前需求和世界状态推导行动链，而不是为每一种情况手写行为规则。
例如：饿了根据 restore_hunger 选择食物；冷了比较生火和造房；房子不够时链式推出砍木头、打磨斧头。
```

验收：

```text
effect_key 有机器可理解语义。
Agent 能生成目标、候选行动、前置链和评分。
寒冷紧急时优先生火，长期容量不足时能规划造房。
同伴/族人只使用自己知道或族群传播的知识。
H5 显示安全中文推理解释，不暴露 raw_state、hidden_truth 或内部分数。
```

### P40：动态目标执行与外界打断系统

目标：

```text
把 P39 选出的计划变成可持续执行、可中断、可恢复、可重规划的目标执行状态机。
支持建房缺木头触发伐木、伐木斧头钝触发打磨、伐木遇狼触发外界打断，处理后恢复原目标。
```

验收：

```text
目标栈可 push/pause/resume/cancel/complete。
ActionDriver 只处理动作 mechanics，外界打断统一由 InterruptSystem 处理。
内部阻塞能生成子目标，外界打断能插入紧急目标。
DriverCommand 必须接入 P38 世界结算。
H5 只显示安全中文执行摘要。
```

### P41：基础内容包扩展

目标：

```text
用配置化流程扩展第一批内容，不靠前端或临时代码写死。
```

最低内容量：

```text
10+ 世界对象。
20+ 交互反馈。
5+ 对象反应。
3+ 危险类型。
2+ 族群/个体互动变化。
1 个文明能力解锁闭环。
```

### P42：存档读档与继续游戏体验

目标：

```text
把 P30 存档前置接到 H5 玩家流程，支持继续游戏、重开确认和版本兼容提示。
```

验收：

```text
刷新页面后能继续。
重开需要确认。
存档版本不兼容时给中文提示。
存档/读档仍不绕过规则层。
```

## 11. 每阶段固定门禁

每个 P 阶段必须满足：

```text
1. 有任务卡。
2. 有 context_pack。
3. 有明确允许/禁止文件范围。
4. 有单元测试。
5. 有集成测试。
6. 有边界扫描。
7. 有审核报告。
8. 邮件发送审核结论。
9. 不修无关问题。
10. 不绕过 RulePipeline / CommandEnvelope / Event / Replay 权威链。
```

## 12. 最关键的架构红线

```text
Agent 是决策层，不是规则层。
H5 是表现层，不是规则层。
学习算法不是规则层。
存档不能绕过规则层。
复盘不能重新调用策略。
知识不是隐藏真相。
记忆不是 ObjectDefinition。
传播传的是主观知识，不是真实属性。
所有状态变化必须有事件。
所有关键行为必须可复盘。
```

## 13. 为什么这个计划适合 AI 实现

```text
阶段从底层到上层，不跳跃。
每阶段只改少量文件。
每阶段都有 context_pack 防止上下文爆炸。
每阶段都有禁止范围，避免 AI 乱改。
每阶段都有测试和审计，能及时纠偏。
先做工具链，再做复杂学习，方便后续定位问题。
```

## 14. 当前下一步

当前建议：

```text
P37：最小剧情闭环设计已完成。
下一步进入 P37 实现与自测。
P37 实现重点不是继续堆底层系统，而是把 P15-P36 已有能力串成玩家能完成的一轮“第一天生存”。
P37 必须验证：玩家能看懂自己在做什么、为什么学会某个知识、同伴为什么获得知识、知识如何帮助解决危险。
P37 通过后，先进入 P38 做世界交互结算与可见后果；世界会变之后，再进入 P38.5 把 H5 从系统验证器改成玩家界面。
```

当前阶段调整原因：

```text

继续加底层系统会降低验证效率。
现在最关键风险是“系统很强，但玩家玩不懂/感受不到目标”。
所以先做最小剧情闭环，再做世界交互结算，然后做界面去调试化和内容扩展。
```
