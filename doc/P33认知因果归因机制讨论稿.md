# 先驱者：认知因果归因机制讨论稿

## 1. 这次问题的判断

你指出的问题是正确的。

我刚才做的“吃几次就确认”仍然过于简单，因为它只解决了两个问题：

```text
1. 同一知识需要多次经验才能确认。
2. 腐烂红果不应该直接降低普通红果知识。
```

但它没有解决真正关键的问题：

```text
玩家/生物如何判断一个结果到底由什么原因造成？
```

比如：

```text
吃红果 3 次 -> 没事 / 缓解饥饿
吃腐烂红果 1 次 -> 不适
```

这时不能马上确认：

```text
腐烂红果会中毒
```

因为仍然存在其他可能解释：

```text
可能是腐烂导致的。
可能是红果吃多了导致的。
可能是刚好这次身体状态不好。
可能是吃红果后累计到某个量才出问题。
可能是环境或随机因素导致的。
```

所以正确设计不是简单地按“物体 + 行动 + 效果 + 条件”计数，而应该加入一层：

```text
因果归因系统 / Causal Attribution System
```

---

## 2. 当前旧逻辑为什么不够

旧逻辑本质是：

```text
观察到 A 后发生 B -> A 会导致 B
重复几次 -> 确认
```

这个逻辑适合非常简单的原型，但不适合《先驱者》的核心玩法。

《先驱者》的核心是认知探索。认知探索的乐趣不只是“试一下得到答案”，而是：

```text
我看到了结果，但我还不知道真正原因。
我需要继续设计尝试，排除错误解释。
最后我才敢把知识传给族群。
```

如果系统太快给出正确结论，游戏会变成按钮测试器。  
如果系统能保留不确定性，玩家才会有真正的探索感。

---

## 3. 应该新增的核心概念

### 3.1 观察事实 Observation

每次行动先不要直接变成知识，而是先形成观察事实。

例如：

```text
第 1 回合：吃红果 -> 饥饿缓解
第 2 回合：吃红果 -> 饥饿缓解
第 3 回合：吃红果 -> 饥饿缓解
第 4 回合：吃腐烂红果 -> 身体不适
```

这些只是事实，不等于原因。

观察事实应该记录：

| 字段 | 含义 |
|---|---|
| object_key | 物体，例如红果 |
| object_state | 物体状态，例如腐烂 |
| action | 行动，例如吃 |
| outcome | 结果，例如缓解饥饿 / 中毒 |
| recent_count | 最近吃过几次同类物体 |
| actor_state | 当时角色状态，例如饥饿、饱腹、不适 |
| time_window | 发生时间窗口 |
| context | 环境上下文 |

重点：

```text
观察事实只说“发生了什么”，不直接说“为什么”。
```

---

### 3.2 因果候选 Cause Candidate

当出现结果时，系统生成多个可能原因。

例如吃腐烂红果后不适，系统不应只生成：

```text
腐烂红果导致中毒
```

而应生成多个候选：

```text
候选 1：腐烂状态导致中毒
候选 2：红果吃多了导致中毒
候选 3：连续吃红果后身体不适
候选 4：未知原因导致不适
```

这些候选都不是最终知识，只是解释模型。

---

### 3.3 因果归因状态 AttributionStatus

建议新增枚举：

```cpp
enum class CausalAttributionStatus {
    Unknown,              // 未知
    ObservedCorrelation,  // 只观察到相关性
    Ambiguous,            // 存在多个可能原因
    WeaklySupported,      // 有弱支持
    StronglySupported,    // 有强支持
    Confirmed,            // 已确认
    Refuted,              // 已排除
    TestOnly
};
```

中文含义：

| 枚举 | 中文 | 含义 |
|---|---|---|
| Unknown | 未知 | 没有归因 |
| ObservedCorrelation | 观察到相关 | 只知道 A 后发生了 B |
| Ambiguous | 原因不明 | 可能原因不止一个 |
| WeaklySupported | 弱支持 | 有一些证据支持 |
| StronglySupported | 强支持 | 多次证据支持，但仍未最终确认 |
| Confirmed | 已确认 | 排除了主要替代解释 |
| Refuted | 已排除 | 后续证据证明该原因不成立 |
| TestOnly | 测试专用 | 只能测试使用 |

---

### 3.4 替代解释 Alternative Explanation

这是这次问题的关键。

当系统看到：

```text
吃红果多次后，又吃腐烂红果，然后中毒
```

它必须同时保留这些解释：

```text
可能是腐烂导致。
可能是吃多了导致。
可能是红果累计摄入导致。
可能是腐烂 + 累计摄入共同导致。
```

这些替代解释没有被排除前，不能进入“已确认”。

## 4. 知识应该分成三层

我建议把知识分成三层，而不是现在直接显示一条结论。

### 4.1 事实层 Fact Knowledge

只记录发生了什么。

例如：

```text
我吃过红果，饥饿缓解了。
我吃过腐烂红果，身体不适了。
```

这层可以很快形成。

### 4.2 相关层 Correlation Knowledge

记录 A 和 B 有关联，但原因不确定。

例如：

```text
吃腐烂红果后出现不适。
连续吃红果后出现不适。
```

这层显示给玩家应该是：

```text
观察到相关：吃腐烂红果后身体不适，但原因尚未确认。
```

### 4.3 因果层 Causal Knowledge

只有排除主要替代解释后，才能变成因果知识。

例如：

```text
腐烂红果会导致中毒。
红果吃多了会导致不适。
```

这层才可以作为强知识传给族群、指导 NPC 行为。

## 5. 针对“红果 / 腐烂红果”的正确推理流程

### 5.1 玩家吃 3 次红果

观察：

```text
红果 -> 吃 -> 缓解饥饿
红果 -> 吃 -> 缓解饥饿
红果 -> 吃 -> 缓解饥饿
```

可以形成：

```text
红果吃后通常可以缓解饥饿：已确认
```

但注意，这只说明：

```text
在目前剂量和状态下，普通红果安全且有益。
```

不代表：

```text
红果无限吃都安全。
腐烂红果也安全。
```

### 5.2 玩家吃 1 次腐烂红果

观察：

```text
腐烂红果 -> 吃 -> 身体不适
```

这时不能直接确认腐烂红果有毒。

应该生成：

```text
观察到相关：吃腐烂红果后身体不适。
原因候选：腐烂状态 / 红果累计摄入 / 未知身体状态。
```

界面可以显示：

```text
腐烂红果 → 吃 → 身体不适
原因不明：可能与腐烂有关，也可能与连续进食有关。
```

### 5.3 如果玩家重开或隔一段时间后直接吃腐烂红果也不适

这会支持：

```text
腐烂状态导致中毒
```

因为排除了“前面吃太多普通红果”的解释。

### 5.4 如果玩家连续吃很多普通红果也不适

这会支持：

```text
红果吃多了会导致不适
```

也就是剂量知识。

### 5.5 如果两种测试都成立

系统可以形成两个知识：

```text
腐烂红果会中毒。
红果吃多了也可能不适。
```

这两条并不冲突。

## 6. 应该如何确认一个因果知识

我建议不要只按次数确认，而按“证据质量”确认。

### 6.1 确认需要满足三个条件

一个因果知识从“相关”变成“确认”，至少需要：

```text
1. 支持证据足够。
2. 主要替代解释被削弱或排除。
3. 反例没有明显推翻它。
```

### 6.2 红果例子

要确认：

```text
腐烂红果导致中毒
```

需要类似证据：

```text
支持：多次吃腐烂红果后不适。
排除：没有先吃很多普通红果时，吃腐烂红果也不适。
反例：没有出现“吃腐烂红果后完全没事”的稳定反例。
```

### 6.3 吃多了导致不适

要确认：

```text
红果吃多了会不适
```

需要类似证据：

```text
支持：连续吃普通红果达到某个数量后不适。
排除：只吃少量普通红果时不适不发生。
反例：连续大量吃普通红果仍然没事，会削弱该结论。
```

## 7. 需要新增的系统模块

### 7.1 CausalObservationRecord

记录每次观察事实。

建议字段：

```cpp
struct CausalObservationRecord {
    ObservationId observation_id;
    EntityId actor_id;
    std::string subject_id;
    std::string action_key;
    std::string outcome_key;
    std::vector<std::string> object_state_keys;
    std::vector<std::string> actor_state_keys;
    std::vector<std::string> recent_exposure_keys;
    int recent_same_subject_count;
    int recent_same_action_count;
    Tick observed_tick;
};
```

中文解释：

| 字段 | 中文 |
|---|---|
| subject_id | 观察对象 |
| action_key | 行动 |
| outcome_key | 结果 |
| object_state_keys | 物体状态，例如腐烂 |
| actor_state_keys | 行动者状态，例如饥饿、饱腹 |
| recent_exposure_keys | 最近接触过什么 |
| recent_same_subject_count | 最近同类对象次数 |
| observed_tick | 观察时间 |

---

### 7.2 CausalHypothesis

记录一个可能原因。

建议字段：

```cpp
struct CausalHypothesis {
    HypothesisId hypothesis_id;
    std::string cause_key;
    std::string subject_id;
    std::string action_key;
    std::string outcome_key;
    std::vector<std::string> required_condition_keys;
    std::vector<std::string> alternative_cause_keys;
    CausalAttributionStatus status;
    double support_score;
    double contradiction_score;
    double ambiguity_score;
    int supporting_observation_count;
    int counterexample_count;
};
```

例子：

```text
假设 A：腐烂状态 + 吃红果 -> 中毒
假设 B：连续吃红果次数过多 -> 中毒
假设 C：未知身体状态 -> 中毒
```

---

### 7.3 CausalAttributionResolver

职责：根据观察事实更新因果假设。

输入：

```text
历史观察事实
当前观察事实
已有因果假设
```

输出：

```text
新增假设
被增强的假设
被削弱的假设
被排除的假设
可以升级成知识的假设
```

---

### 7.4 KnowledgeCausalBridge

职责：把“足够强的因果假设”转成正式知识。

规则：

```text
ObservedCorrelation / Ambiguous：不能转正式因果知识，只能展示为观察相关。
WeaklySupported：可以展示为经验假设。
StronglySupported：可以作为行动参考，但提示仍有不确定性。
Confirmed：可以进入正式知识系统和族群传播。
Refuted：标记为已排除。
```

## 8. 前端应该怎么显示

当前 UI 不应该直接显示：

```text
腐烂红果 → 吃 → 会中毒：已确认
```

在证据不足时应该显示：

```text
腐烂红果 → 吃 → 身体不适
原因不明：可能与腐烂有关，也可能与连续进食有关。
```

或者：

```text
观察到相关：吃腐烂红果后身体不适。
待验证：是否由腐烂状态导致。
```

当系统逐步排除其他解释后，再显示：

```text
腐烂红果 → 吃 → 会导致中毒
已确认：多次观察支持，且已排除“吃多了普通红果”的主要解释。
```

## 9. 对 NPC 和族群传播的影响

这个设计非常重要，因为它会影响 NPC 行为。

### 9.1 不确定知识不能让 NPC 过度反应

如果知识只是：

```text
吃腐烂红果后不适，原因不明
```

NPC 不应该马上认为：

```text
你给我腐烂红果就是在害我
```

它最多应该认为：

```text
这东西有风险，我需要谨慎。
```

### 9.2 已确认因果知识才改变道德判断

如果族群已经确认：

```text
腐烂红果会中毒
```

这时你给 NPC 吃腐烂红果，NPC 才可以判断：

```text
你可能在害我。
```

### 9.3 这能让族群知识更真实

族群传播的不只是结论，还可以传播不确定性：

```text
我们观察到腐烂红果有风险，但还没完全确认原因。
```

这比简单传播“有毒/没毒”更符合游戏主题。

## 10. 推荐的阶段实现方案

我建议不要直接在当前 P33 里硬改完整因果系统。应该拆成一个正式阶段。

### 阶段 A：先修当前 P33 的临时策略

目标：避免当前表现明显错误。

临时规则：

```text
同一知识第一次出现：候选
同一知识第二次出现：假设
同一知识第三次出现：确认
如果存在替代解释，则最高只能到“原因不明/强相关”，不能确认因果。
```

这一步适合作为当前 H5 可玩壳的补丁。

### 阶段 B：新增因果观察与假设系统

目标：正式支持原因不明、多候选解释、替代解释。

新增模块：

```text
causal_observation
causal_hypothesis
causal_attribution_resolver
knowledge_causal_bridge
```

### 阶段 C：接入 H5 投影

目标：让玩家看见“为什么还不能确认”。

新增展示：

```text
观察事实
可能原因
待验证问题
已排除解释
确认原因
```

### 阶段 D：接入 NPC 和族群

目标：让 NPC 根据知识确定度做行为判断。

例如：

```text
原因不明：NPC 谨慎。
强相关：NPC 拒绝尝试。
已确认有毒：NPC 认为投喂者有恶意。
```

## 11. 我建议的最终规则

最终应该是：

```text
知识确认 = 重复次数 + 条件一致性 + 替代解释排除 + 反例检查
```

而不是：

```text
知识确认 = 重复次数
```

更具体：

```text
1. 同一现象重复出现，只能增强相关性。
2. 如果存在多个可能原因，状态保持“原因不明”。
3. 只有通过对照尝试排除主要替代解释，才能进入因果确认。
4. 不同分支独立累计证据，但可以互相提供替代解释。
5. 族群传播必须传播确定度，不只是传播结论。
```

```text

已确认
```

因为这时还没有排除“吃多了”的解释。

## 13. 结论

你的判断是对的。当前问题不能靠简单计数彻底解决。

正确方向应该是：

```text
从“经验计数系统”升级为“因果归因系统”。
```

这套系统会让《先驱者》的认知玩法更高级：

- 玩家不是直接获得标准答案。
- 玩家会看到不确定性。
- 玩家需要通过对照实验排除错误原因。
- NPC 和族群会根据知识确定度做不同反应。
- 后续强化学习也能从观察、假设、验证、确认中获得更清晰的训练信号。

我建议下一步先不要继续在 P33 上堆补丁，而是把这个作为 P34 或 P35 前置的“因果归因小阶段”来正式设计。
