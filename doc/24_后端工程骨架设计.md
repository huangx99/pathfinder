# 24_后端工程骨架设计

## 1. 设计目标

后端工程骨架设计负责把第 23 阶段的工程路线图进一步收敛成 C++ 后端第一批可落地文件、目录、命名空间、CMake 目标、测试入口和 AI 任务卡。

本章不是实现代码，而是定义：

```text
AI 第一次开始写后端代码时，应该创建哪些目录。
哪些文件属于 P0。
哪些文件暂时不能创建。
CMake 目标如何拆。
命名空间如何定。
测试工程如何挂载。
每个 AI 任务允许改哪些文件。
每个任务如何验收。
任务完成后如何留下上下文记忆。
```

本章目标：

```text
让 AI 可以在小上下文下开始实现 backend。
让第一批代码可以编译、测试、扩展。
让后续 config、command、pipeline、engine、agent、save、protocol 不需要推翻目录结构。
让每个模块边界从第一天就清楚。
```

核心原则：

```text
先建骨架，不实现复杂规则。
先建 foundation，不建 RuleEngine。
先建测试入口，不等后面补测试。
先建 README / dev_notes，不让 AI 接力失忆。
```

## 2. 不解决的问题

| 不解决的问题 | 原因 | 后续阶段 |
|---|---|---|
| 完整规则实现 | 本章只定义骨架 | P3 规则闭环阶段 |
| HTTP / WebSocket 实现 | protocol 依赖 engine，不能提前 | 协议阶段 |
| H5 前端实现 | 前端必须等 protocol | 前端阶段 |
| AgentRuntime 完整实现 | 依赖 command / engine | Agent 阶段 |
| 存档文件格式完整实现 | 依赖 state / command / event | save 阶段 |
| 第三方库最终选择 | 本章优先标准库和可替换接口 | 技术选型阶段 |

禁止做法：

```text
禁止在 P0 创建完整 rules 规则实现。
禁止在 P0 创建网络服务。
禁止在 P0 写 H5。
禁止在 P0 让 foundation 依赖 config / engine / protocol。
禁止为了演示而绕过未来 RuleEngine 设计。
禁止一次性生成大量空文件导致 AI 后续迷失。
```

## 4. CMake 目标设计

### 4.1 顶层目标

建议后端第一阶段 CMake 目标：

| 目标 | 类型 | 作用 |
|---|---|---|
| pathfinder_foundation | static library | 基础类型库 |
| pathfinder_tests_foundation | executable | foundation 单元测试 |
| pathfinder_all_tests | custom target | 聚合测试入口 |

未来扩展目标预留：

| 目标 | 阶段 | 作用 |
|---|---|---|
| pathfinder_config | P1 | 配置模块 |
| pathfinder_command | P1 | 命令模块 |
| pathfinder_state | P2 | 状态模块 |
| pathfinder_event | P2 | 事件模块 |
| pathfinder_pipeline | P2 | 管线模块 |
| pathfinder_engine | P3 | 引擎门面 |
| pathfinder_agent | P5 | Agent 模块 |
| pathfinder_save | P4 | 存档复盘 |
| pathfinder_protocol | P4/P6 | 协议映射 |

### 4.2 CMake 规则

P0 CMake 必须满足：

```text
C++ 标准至少 C++20。
不要求第三方库。
测试可通过 ctest 运行。
include 目录对外公开。
src 目录只作为实现。
每个模块单独 target。
```

禁止：

```text
在 foundation target 链接 engine。
在 foundation target 链接 protocol。
在 foundation target 引入网络库。
在 P0 引入复杂包管理器。
```

## 6. foundation P0 文件设计

### 6.1 result.h

职责：

```text
定义 Result<T>。
定义 Result<void>。
表达成功或失败。
携带 ErrorDetail。
禁止抛异常作为业务错误通道。
```

建议公开类型：

```cpp
template <typename T>
class Result;

template <>
class Result<void>;
```

最低能力：

```text
ok(value)。
fail(error)。
is_ok()。
is_error()。
value()。
error()。
```

禁止依赖：

```text
config。
state。
event。
command。
engine。
protocol。
```

### 6.2 error.h / error.cpp

职责：

```text
定义 ErrorCode。
定义 ErrorSeverity。
定义 ErrorCategory。
定义 ErrorDetail。
提供 errorCodeToString。
提供 errorSeverityToString。
提供 errorCategoryToString。
```

P0 最小错误码：

```text
ok。
unknown_error。
id_invalid_format。
id_empty。
config_invalid。
command_invalid。
precondition_failed。
internal_error。
not_implemented。
```

注意：

```text
debug_message 只能调试，不参与业务逻辑。
message_key 必须稳定。
```

### 6.3 id.h / id.cpp

职责：

```text
定义 StrongId<Tag>。
提供字符串包装。
提供 empty 判断。
提供相等比较。
提供哈希。
提供格式校验函数。
```

P0 类型别名：

```cpp
using EntityId = StrongId<EntityIdTag>;
using ObjectId = StrongId<ObjectIdTag>;
using ObjectDefinitionId = StrongId<ObjectDefinitionIdTag>;
using CommandId = StrongId<CommandIdTag>;
using EventId = StrongId<EventIdTag>;
using TransactionId = StrongId<TransactionIdTag>;
using AgentId = StrongId<AgentIdTag>;
using ConfigVersionId = StrongId<ConfigVersionIdTag>;
```

P0 禁止：

```text
不要实现自增 ID 生成器。
不要访问数据库。
不要依赖全局随机。
不要把定义 ID 和实例 ID 混用。
```

### 6.4 version.h / version.cpp

职责：

```text
定义 StateVersion。
定义 ProtocolVersion。
定义 EngineVersion。
支持比较、递增、字符串输出。
```

P0 要求：

```text
StateVersion 不能倒退。
StateVersion 递增函数不直接改 GameState。
ConfigVersionId 仍走 StrongId。
```

### 6.5 hash.h / hash.cpp

职责：

```text
定义 HashValue。
提供稳定字符串 hash 包装。
提供 combineHash。
提供 empty hash。
```

P0 不要求密码学安全。

要求：

```text
同输入同 hash。
不能依赖对象内存地址。
不能依赖 unordered_map 非稳定遍历顺序。
```

### 6.6 time.h / time.cpp

职责：

```text
定义 Tick。
定义 DurationTicks。
定义 TimestampMs。
```

规则：

```text
核心规则使用 Tick，不使用真实系统时间。
真实系统时间只能用于日志或测试报告。
```

### 6.7 random.h / random.cpp

职责：

```text
定义 RandomSeed。
定义 RandomDrawRecord 最小结构。
预留 DeterministicRandom 接口。
```

P0 只做最小接口，不实现复杂随机系统。

要求：

```text
随机必须可记录。
同 seed 同调用顺序同结果。
不能在规则层直接调用 std::random_device。
```

### 6.8 serialization.h

职责：

```text
定义 ToString 约定。
定义后续 JSON 序列化边界。
P0 不绑定具体 JSON 库。
```

## 7. 测试骨架设计

### 7.1 测试原则

P0 测试必须：

```text
不依赖配置。
不依赖 RuleEngine。
不依赖网络。
不依赖真实时间。
可以用 ctest 执行。
```

### 7.2 测试文件

| 文件 | 测试内容 |
|---|---|
| result_test.cpp | Result<T> 成功、失败、Result<void> |
| error_test.cpp | ErrorCode 字符串、ErrorDetail message_key |
| id_test.cpp | StrongId 构造、空 ID、格式校验、比较、哈希 |
| version_hash_test.cpp | StateVersion 比较、HashValue 稳定性、Tick 基础行为 |

### 7.3 测试命名

测试函数命名：

```text
模块_场景_期望
```

示例：

```text
result_ok_contains_value。
result_fail_contains_error。
id_empty_is_invalid。
state_version_cannot_go_backward。
hash_same_input_same_output。
```

