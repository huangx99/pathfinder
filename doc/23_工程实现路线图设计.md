# 23_工程实现路线图设计

## 1. 设计目标

工程实现路线图负责把 `01` 到 `22` 的规则引擎设计转成可执行研发计划。

前面的文档解决了：

```text
系统应该是什么。
类和接口应该怎么抽象。
规则如何确定性执行。
知识、认知、记忆、传播、族群、文明、Agent 如何组合。
存档、复盘、协议、测试如何保证长期可维护。
```

本章解决：

```text
研发第一天从哪里开始写代码。
C++ 后端工程如何分层。
H5 前端如何接入。
配置、测试、存档、工具如何落目录。
哪些模块先实现，哪些模块后实现。
如何一步一步做出最小可运行闭环。
如何保证后续扩展不推翻整体架构。
如何让 AI 在有限上下文和有限记忆下稳定写出好代码。
如何把大系统拆成 AI 可执行、可验证、可恢复的小任务。
```

核心目标：

```text
先落规则核心，再落内容规模。
先落可测试闭环，再落复杂表现。
先落权威后端，再落前端体验。
```

最终要做到：

```text
任何行为都能通过 CommandEnvelope 进入 RuleEngine。
任何状态修改都经过 RulePipeline。
任何结果都能产出 EventEnvelope。
任何关键命令都能存档和复盘。
任何 Agent 都通过统一观察、意图和命令适配行动。
任何前端显示都来自协议投影和事件，不自己推导规则。
任何新功能都有对应测试入口。
```

## 2. 不解决的问题

| 不解决的问题 | 原因 | 后续阶段 |
|---|---|---|
| 具体 C++ 代码实现 | 本章是路线图，不写源码 | 实现阶段 |
| 最终网络框架选型 | 本章只定义适配边界 | 技术选型阶段 |
| H5 美术和完整 UI | 本章只定义前端工程结构 | UI/UX 阶段 |
| 完整 CI/CD 脚本 | 本章只定义运行要求 | DevOps 阶段 |
| 数据库选型 | 首版可先文件存储 | 服务端工程阶段 |
| 完整内容生产工具 | 本章只预留工具目录 | 内容工具阶段 |
| 强化学习算法 | 本章只预留训练环境接口 | RL 阶段 |

禁止做法：

```text
禁止先做一个绕过 RuleEngine 的 demo。
禁止前端直接改状态。
禁止 Agent 直接改状态。
禁止测试直接改状态伪造通过。
禁止配置绕过 ConfigService 加载。
禁止存档只保存最终 StateSnapshot 不保存命令和随机。
禁止为了快速显示 UI 在前端复制一套规则。
禁止一开始就做大而全内容库，必须先做可验证闭环。
禁止把 01-22 全部文档一次性塞给 AI 要它实现。
禁止让 AI 在一个任务里同时改后端、前端、配置、测试、文档。
禁止没有任务卡、上下文清单、验收命令就开始编码。
```

## 3. 工程总原则

### 3.1 分层稳定

工程分层必须稳定：

```text
foundation：基础类型、ID、Result、ErrorCode、Hash、Version。
config：配置加载、校验、版本、表达式。
state：权威状态、快照、状态草稿。
event：事件定义、事件总线、事件存储、事件投影。
command：CommandEnvelope、ActionCommand、目标、参数。
rules：对象、特性、交互、反馈、认知、记忆、知识、传播、族群、文明规则。
pipeline：规则管线、阶段契约、事务提交。
engine：RuleEngine 门面和服务接口。
agent：Agent 观察、动作空间、策略、运行时。
save：存档、复盘、随机日志、差异报告。
protocol：HTTP / WebSocket 映射。
tests：测试夹具、场景、golden replay、fuzz、报告。
frontend：H5 表现层。
tools：配置校验、文档检查、回放工具、调试工具。
```

### 3.2 依赖单向

依赖方向必须单向：

```text
foundation <- config
foundation <- state
foundation <- event
foundation <- command
config + state + event + command <- rules
rules <- pipeline
pipeline <- engine
engine <- protocol
engine <- save
engine <- agent
protocol <- frontend
engine + save + agent <- tests
```

禁止反向依赖：

```text
foundation 不能依赖 engine。
rules 不能依赖 protocol。
state 不能依赖 frontend。
pipeline 不能依赖 H5。
RuleEngine 不能依赖具体 UI。
AgentPolicy 不能直接依赖内部 Resolver。
```

### 3.3 测试先行

每个阶段都必须有测试：

```text
基础类型有单元测试。
配置有校验测试。
规则有组合测试。
管线有事务测试。
Agent 有确定性测试。
存档有复盘测试。
协议有幂等和重连测试。
前端有协议契约测试。
```

### 3.4 最小闭环优先

第一阶段最小闭环不是完整游戏，而是：

```text
加载配置。
创建世界对象。
提交 eat/use/observe 命令。
规则管线结算。
产出反馈事件。
更新记忆和认知。
保存命令和随机。
复盘得到同样结果。
H5 显示事件和认知变化。
```

### 3.5 AI 编码友好优先

这份文档最终交给 AI 实现代码，因此工程路线必须按 AI 的上下文和记忆限制设计。

AI 编码的核心风险：

```text
一次读太多文档，抓不住当前任务重点。
一次改太多模块，破坏边界。
上下文过长后忘记早期约束。
为了让测试通过临时绕过架构。
不知道该读哪些文档，反复做重复设计。
不知道验收命令，写完无法验证。
```

所以每个研发任务必须满足：

```text
小上下文。
小改动。
单一目标。
明确输入文档。
明确禁止事项。
明确文件范围。
明确验收命令。
明确回滚边界。
```

AI 不应该一次实现一个“系统”，而应该一次实现一个“可验证切片”。

```text
错误示例：实现规则引擎。
正确示例：实现 foundation::Result<T>，并通过 result_unit_test。

错误示例：实现 Agent 系统。
正确示例：实现 AgentDefinition 数据结构和序列化，不实现 AgentRuntime。

错误示例：实现存档和复盘。
正确示例：实现 CommandReplayRecord 写入内存日志，并通过 replay_record_test。
```

### 3.6 文档读取最小化原则

AI 每次编码任务最多读取：

```text
1 个当前任务卡。
1 个直接设计文档。
最多 2 个依赖设计文档。
1 个相关测试策略片段。
现有代码中 3-8 个相关文件。
```

如果需要读取更多，说明任务切得太大，必须拆分。

### 3.7 任务完成后必须留下局部记忆

每个任务完成后必须更新或生成局部说明：

```text
做了什么。
改了哪些文件。
依赖哪些设计。
哪些约束不能破坏。
跑了哪些测试。
下一步应该接什么。
```

建议落在：

```text
doc/dev_notes/
```

或者对应模块的：

```text
backend/src/<module>/README.md
backend/include/pathfinder/<module>/README.md
```

这样下一次 AI 不需要重新理解全项目。

## 9. 实现阶段 A-J

### Phase A：工程骨架

目标：建立可编译、可测试、可扩展的工程骨架。

交付：

```text
CMake 工程。
backend/include 和 backend/src。
foundation 模块。
最小测试入口。
基础 README。
```

验收：

```text
能编译。
能运行 foundation 单元测试。
能输出测试报告。
```

### Phase B：配置与基础数据

目标：让游戏内容进入配置系统。

交付：

```text
ConfigLoader。
ConfigValidator。
最小对象、特性、行为、交互配置。
fixture_unknown_fruit_v1。
```

验收：

```text
配置加载成功。
错误配置能失败。
配置版本可读取。
```

### Phase C：规则管线最小闭环

目标：跑通第一条真实规则。

交付：

```text
CommandEnvelope。
RulePipelineExecutor。
GameState。
StateChangeDraft。
EventDraft。
吃未知果实规则。
```

验收：

```text
submit eat_unknown_fruit 返回 Success 或明确错误码。
产出 FeedbackEvent。
状态通过事务提交。
失败不修改状态。
```

### Phase D：认知、记忆、知识链路

目标：让尝试行为产生认知成长。

交付：

```text
FeedbackEvidence。
MemoryRecord。
CognitionEntry。
KnowledgeCandidate。
```

验收：

```text
吃未知果实后形成记忆。
多次证据提升认知置信度。
满足条件后生成知识候选。
```

### Phase E：对象反应和组合规则

目标：支持对象之间反应。

交付：

```text
InteractionRule 组合匹配。
ObjectReaction。
火 + 木棍 -> 火把。
```

验收：

```text
use fire with wood_stick 生成 torch。
事件顺序稳定。
复盘一致。
```

### Phase F：传播、族群、文明

目标：从个人认知扩展到群体进步。

交付：

```text
PropagationAttempt。
ReceiverCognitionUpdate。
TribeStateUpdate。
CivilizationCapabilityCandidate。
```

验收：

```text
teach poisonous_fruit 影响族人认知。
族群状态随知识变化。
文明能力只产生候选，不直接解锁。
```

### Phase G：Agent 系统

目标：让非玩家实体也能统一决策。

交付：

```text
AgentRuntime。
AgentObservationBuilder。
AgentActionSpaceBuilder。
AgentCommandAdapter。
Player Agent。
Spider Agent。
Wolf Agent。
Tribe Agent。
```

验收：

```text
蜘蛛能避火。
狼能召集狼群。
Player Agent 走前端命令。
ReplayLocked Agent 不重新决策。
所有 Agent 都通过 CommandEnvelope。
```

### Phase H：存档、复盘、Golden Replay

目标：保证规则可长期回归。

交付：

```text
SaveGamePackage。
CommandReplayLog。
RandomReplayLog。

```

验收：


### Phase I：协议和 H5 最小前端

## 14. 风险和防偏措施

| 风险 | 表现 | 防偏措施 |
|---|---|---|
| 先做 UI | 页面能点但规则不可信 | 先完成 RuleEngine smoke |
| 规则散落 | Resolver 直接改状态 | 所有修改走 StateChangeDraft |
| 测试滞后 | 改动后不敢重构 | 每阶段必须有 smoke |
| 配置失控 | 内容 ID 和引用混乱 | ConfigValidator 先行 |
| 前端越权 | H5 自己推导规则 | 前端只消费 Projection |
| Agent 作弊 | AI 直接读真实状态 | Observation 过滤和权限测试 |
| 复盘不一致 | 随机无法重现 | RandomReplayLog 必须先做 |
| 过度设计 | 迟迟不可运行 | 最小闭环限制范围 |
| 内容过早膨胀 | 大量对象难维护 | 每个内容必须有 fixture 和测试 |
