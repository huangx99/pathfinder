# P53 客户端协议风险头脑风暴

## 1. 风险定位

P53 最大风险是把“协议测试”又写成“界面原型”，或者让客户端为了通过演示而重新实现规则。

本阶段必须抵抗两个诱惑：

```text
1. 先做个网页看看。
2. 前端先自己生成几个按钮。
```

这两个都会把项目带回旧临时网页的问题。

## 2. 主要风险表

| 风险 | 后果 | P53 对策 |
|---|---|---|
| P53 写 UI | 又变调试台或假游戏界面 | P53 禁止 UI / CSS / DOM 验收 |
| 协议命名绑定 H5 | 后续 Godot/Unity/App 难复用 | 统一使用 Client / client_protocol |
| 客户端生成按钮 | 规则双轨 | 只提交后端 available_commands 的 option_id |
| 客户端读取 JSON | 内容权威分裂 | 扫描禁止 content/core / recipes / reactions |
| 客户端本地改状态 | 状态漂移 | 只接受 full projection / patch |
| 忽略 projection_version | 多命令后错乱 | 所有 command 必带 known_projection_version |
| event 当规则源 | 日志变规则 | event_feed 只用于反馈 |
| hint 当规则源 | 表现层反推规则 | frontend_hints 只用于表现 |
| 传 KnowledgeClaim | 前端注入知识 | 扫描禁止 knowledge_json / recipient_claim_json |
| 直接改 runtime | 绕过 P43-P52 | client_protocol 禁止 runtime arrays mutation |
| fake client 太薄 | 协议没被真实验证 | 必须模拟 bootstrap-command-patch-refresh |
| option_id 不可追踪 | 客户端无法复盘 | option_id 必须可由后端映射 command |
| 失败命令无响应 | 客户端只能猜 | Failed/Blocked 也必须返回 safe response |
| Reset 混入规则 | 重开污染经验 | Reset 独立于普通 WorldCommand experience |
| DTO 带 UI 字段 | P53 偷偷设计界面 | DTO 只表达协议，不表达布局 |

## 3. 场景压力测试

### 3.1 网页客户端

网页只是 P54 之后的一种实现。P53 不能因为第一个客户端可能是网页，就把协议命名为 H5。

### 3.2 Godot 客户端

Godot 应该能用相同的 bootstrap / command / refresh 协议，不需要理解网页 DOM、CSS 或 HTML。

### 3.3 移动 App 客户端

App 应该能复用 Client DTO，只把 projection 转成原生 UI。

### 3.4 自动化机器人客户端

测试机器人可以不显示界面，只提交 option_id 并验证 patch。这正是 P53 应该优先支持的客户端。

### 3.5 断线恢复

客户端版本落后时，不能强行应用 patch，必须 full refresh。

### 3.6 无可用命令

如果 `available_commands` 为空，客户端只能显示“暂无可执行操作”的协议状态，不能自己补按钮。

### 3.7 过期 option_id

过期 option_id 必须由后端拒绝，不能执行成未知命令。

### 3.8 恶意知识注入

客户端提交 knowledge_json 或 recipient_claim_json 必须被拒绝或扫描阻断。

### 3.9 高级文明客户端

未来高级文明界面可以更复杂，但依旧只通过 Client Protocol 与规则层交互。

### 3.10 多客户端观察同一世界

P53 可以不实现多人同步，但 protocol_version / projection_version 不能阻碍后续多客户端扩展。

## 4. 红线

```text
如果 P53 出现 UI 布局设计，重写。
如果 P53 出现 H5V2 / h5_v2 命名，重写。
如果 P53 让客户端读 JSON，重写。
如果 P53 让客户端根据对象 key 生成按钮，重写。
如果 P53 没有 fake client 代码测试，不能进入 P54。
```
