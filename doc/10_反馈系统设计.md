# 先驱者：反馈系统设计

## 1. 设计目标

反馈系统负责把交互规则输出的 `FeedbackCandidate` 转换为稳定、可记录、可展示、可认知吸收的反馈结果。

在《先驱者》中，反馈不是 UI 文案，也不是简单日志。反馈是玩家理解世界的证据来源。

核心链路：

```text
InteractionResult
  ↓
FeedbackCandidate
  ↓
FeedbackResolver
  ↓
FeedbackResult / FeedbackPayload / FeedbackEvidence
  ↓
FeedbackGenerated Event
  ↓
认知状态 / 记忆 / 知识 / 前端表现 / 训练投影
```

反馈系统要解决以下问题：

```text
行为发生后，玩家到底感受到了什么？
这个感受有多强？
这个感受是否可靠？
它来自哪里？
它能否成为认知证据？
它应该如何写入事件流？
它是否对 AI 训练有奖励提示价值？
```

本系统必须保证：

- 反馈可复盘。
- 反馈可配置。
- 反馈可聚合。
- 反馈可延迟。
- 反馈可作为认知证据。
- 反馈可进入事件系统。
- 前端不自行推导反馈。
- 强化学习后续可从反馈事件中提取训练线索。

## 2. 不解决的问题

反馈系统不解决以下内容：

| 问题 | 原因 | 归属文档 |
|---|---|---|
| 修改生命、饥饿、对象状态 | 反馈只描述感知结果，不直接改状态 | 后续效果系统 / 规则管线 |
| 创建最终记忆 | 反馈只提供证据，不直接生成记忆 | `12_记忆系统设计.md` |
| 创建最终知识 | 反馈只提供证据，不直接合并知识 | `13_知识系统设计.md` |
| 传播知识 | 反馈可描述传播结果，但不执行传播 | `14_传播系统设计.md` |
| 前端动画细节 | 反馈给前端语义，不规定动画实现 | `20_前后端协议设计.md` |
| 最终 RL 奖励函数 | 反馈可给 reward_hint，但不定义最终奖励 | 后续 AI / 训练设计 |
| 自由脚本文案逻辑 | 文案使用 message_key，不允许脚本生成规则结果 | 配置系统 |

关键边界：

```text
StateChange 是机器状态变化。
Feedback 是玩家、系统或观察者可感知的结果。
Event 是事实记录。
Memory 是后续对反馈证据的主观沉淀。
Knowledge 是后续对多个证据的稳定断言。
```

示例：

```text
吃红浆果后饥饿 -10

StateChange：hunger 减少 10
Feedback：你感到没那么饥饿
Event：FeedbackGenerated，记录反馈事实
Memory：我吃过红浆果，似乎能缓解饥饿
Knowledge：红浆果在新鲜状态下可食用并缓解饥饿
```

## 3. 核心概念

| 概念 | 中文名 | 含义 |
|---|---|---|
| FeedbackCandidate | 反馈候选 | 交互规则产生的原始反馈请求 |
| FeedbackRule | 反馈规则 | 配置中定义如何把候选转换为反馈结果 |
| FeedbackResolver | 反馈解析器 | 反馈系统核心服务，负责筛选、聚合、补全和输出 |
| FeedbackResult | 反馈结果 | 一次反馈解析的完整结果 |
| FeedbackPayload | 反馈载荷 | 可写入事件、可发给前端的反馈数据 |
| FeedbackEvidence | 反馈证据 | 可供认知、记忆、知识系统使用的证据 |
| FeedbackAggregation | 反馈聚合 | 多个候选合并、去重、排序、压缩的过程 |
| DelayedFeedback | 延迟反馈 | 行为后不是立即发生，而在未来条件满足时出现的反馈 |
| FeedbackTrace | 反馈解释链 | 说明反馈由哪些候选、规则、表达式和状态产生 |
| FeedbackPresentationHint | 表现提示 | 给前端的弱提示，不是规则结果 |
| FeedbackRewardHint | 奖励提示 | 给后续强化学习投影的弱提示，不是最终 reward |

反馈系统的核心定位：

```text
反馈系统是交互规则和认知系统之间的证据桥梁。
```

## 4. 枚举设计

本阶段不新增通用枚举，优先使用 `02_通用枚举设计.md` 中已经定义的枚举。

### 4.1 使用的既有枚举

| 枚举 | 中文名 | 来源 | 本系统用途 |
|---|---|---|---|
| FeedbackType | 反馈类型 | 02 | 描述反馈属于生理、风险、知识等类型 |
| FeedbackIntensity | 反馈强度 | 02 | 描述反馈强弱 |
| FeedbackReliability | 反馈可靠性 | 02 | 描述反馈证据可靠程度 |
| FeedbackSource | 反馈来源 | 02 | 描述反馈来自直接经验、观察、对象反应等 |
| EventType | 事件类型 | 02 | 输出 `FeedbackGenerated` |
| EventVisibility | 事件可见性 | 02 | 决定反馈事件是否前端可见 |
| EventImportance | 事件重要性 | 02 | 决定反馈事件展示优先级 |
| ErrorCode | 错误码 | 03 | 表达反馈解析失败原因 |
| ErrorSeverity | 错误严重级别 | 02/03 | 表达工程错误影响 |
| RuleStepType | 规则步骤类型 | 02 | 标记当前处于 `ResolveFeedback` |
| ObjectStateType | 对象状态类型 | 02 | 判断反馈来源对象状态 |
| TraitCategory | 特性类别 | 02 | 推导反馈类型和证据方向 |
| ActionType | 行为类型 | 02 | 绑定反馈和行为 |
| EvidenceType | 证据类型 | 02 | 转换为认知证据时使用 |

### 4.2 枚举使用约定

```text
FeedbackType 可以多选。
FeedbackSource 可以多选。
FeedbackIntensity 必须单选。
FeedbackReliability 必须单选。
FeedbackGenerated Event 必须带 FeedbackEventPayload。
```

### 4.3 FeedbackType 使用规则

| 类型 | 典型来源 | 典型后续用途 |
|---|---|---|
| Physiological | 吃、受伤、治疗、中毒 | 个人认知、风险判断 |
| Risk | 受伤、中毒、威胁、失败 | 警告记忆、危险知识 |
| Knowledge | 新效果、新用途、反证 | 知识候选、记忆候选 |
| ObjectState | 燃烧、熄灭、变质、转化 | 对象状态记忆、交互规则调试 |
| Resource | 获得、消耗、损坏、产出 | 资源认知、工具链推断 |
| Failure | 无效果、条件不足、目标不匹配 | 失败记忆、排除错误假设 |
| Social | 族群反应、传授反馈 | 传播系统输入 |
| Civilization | 能力解锁、阶段推进 | 文明能力事件辅助 |

### 4.4 FeedbackIntensity 使用规则

```text
Intensity 不等于数值大小。
Intensity 表达玩家可感知程度和认知权重。
```

示例：

| 场景 | 数值变化 | Intensity | 原因 |
|---|---:|---|---|
| 饥饿 -1 | 很小 | Trace | 几乎不可判断 |
| 饥饿 -10 | 中等 | Moderate | 可明显感到缓解 |
| 中毒 5 回合 | 中等 | Major | 决策意义强 |
| 死亡 | 极大 | Critical | 关键事件 |
| 火把点亮黑暗 | 不一定有数值 | Major | 改变探索能力 |

### 4.5 FeedbackReliability 使用规则

可靠性由以下因素共同决定：

```text
反馈来源
是否亲自发生
是否多次发生
条件是否稳定
是否被其他证据反驳
是否来自推断或转述
```

基本建议：

| 来源 | 初始可靠性建议 |
|---|---|
| DirectExperience | Anecdotal / Observed |
| Observation | Observed |
| Repeated DirectExperience | Repeated |
| Controlled Test | Controlled |
| TribeMember | Reported |
| Inference | Anecdotal |
| RuleSystem Civilization | Established |
| 与旧证据冲突 | Contradicted |

### 4.6 FeedbackSource 使用规则

`FeedbackSource` 用来区分证据来源，不表达可信度本身。

```text
DirectExperience 通常更可靠，但可能因条件误判而错误。
Reported 通常较弱，但可能来自族群长期稳定知识。
Inference 初始不能直接等同于事实。
ObjectReaction 对对象反应很关键，例如火 + 木棍 = 火把。
```

## 5. 数据结构

### 5.1 FeedbackCandidate（反馈候选）

`FeedbackCandidate` 已在 `09_交互规则设计.md` 中定义，本章补充其进入反馈系统后的使用规则。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| feedback_type | FeedbackType | 反馈类型 | 可多选 |
| intensity | FeedbackIntensity | 强度 | 候选建议强度 |
| reliability | FeedbackReliability | 可靠性 | 候选建议可靠性 |
| source | FeedbackSource | 来源 | 可多选 |
| message_key | string | 文案键 | 候选建议文案 |
| related_effect_id | EffectId | 相关效果 | 关联效果定义 |
| target_ids | list<string> | 目标 | 对象、实体、区域等目标 |

补充约定：

```text
FeedbackCandidate 可以不完整。
FeedbackResolver 负责根据规则、配置和上下文补全。
FeedbackCandidate 不允许直接发送给前端。
FeedbackCandidate 不允许直接写入 EventStore。
```

### 5.2 FeedbackRule（反馈规则）

`FeedbackRule` 定义如何把候选转为最终反馈。

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| id | string | 反馈规则 ID | 是 | 配置内唯一 |
| name | string | 名称 | 是 | 研发和策划可读名称 |
| enabled | bool | 是否启用 | 是 | 可灰度关闭 |
| match_feedback_types | list<FeedbackType> | 匹配反馈类型 | 否 | 为空表示不限 |
| match_sources | list<FeedbackSource> | 匹配来源 | 否 | 为空表示不限 |
| match_action_types | list<ActionType> | 匹配行为 | 否 | 为空表示不限 |
| match_effect_ids | list<EffectId> | 匹配效果 | 否 | 为空表示不限 |
| condition_ids | list<ExpressionId> | 条件表达式 | 否 | 使用受限表达式 |
| intensity_override | FeedbackIntensity | 强度覆盖 | 否 | 覆盖候选强度 |
| reliability_override | FeedbackReliability | 可靠性覆盖 | 否 | 覆盖候选可靠性 |
| message_key_override | string | 文案键覆盖 | 否 | 覆盖候选文案 |
| presentation_hint | FeedbackPresentationHint | 表现提示 | 否 | 给前端的展示建议 |
| evidence_policy | FeedbackEvidencePolicy | 证据策略 | 是 | 如何生成认知证据 |
| aggregation_key_template | string | 聚合键模板 | 否 | 用于聚合相似反馈 |
| priority | int | 优先级 | 是 | 多规则匹配时排序 |

注意：

```text
FeedbackRule 是配置，不是 C++ 继承类。
FeedbackRule 不能修改状态。
FeedbackRule 不能产生随机数。
FeedbackRule 不能直接创建事件。
FeedbackRule 只能影响反馈结果如何形成。
```

### 5.3 FeedbackEvidencePolicy（反馈证据策略）

`FeedbackEvidencePolicy` 定义是否以及如何从反馈生成证据。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| create_evidence | bool | 是否生成证据 | false 表示只展示不进入认知 |
| evidence_type | EvidenceType | 证据类型 | 直接实验、观察、转述、推断等 |
| confidence_delta | Delta | 置信度建议变化 | 给认知系统使用 |
| contradiction_possible | bool | 是否可能反证 | 是否允许和旧知识冲突 |
| subject_from | string | 主体来源 | target / source_object / actor / effect |
| action_from_context | bool | 行为来自上下文 | 通常为 true |
| condition_summary_template | string | 条件摘要模板 | 供后续知识系统使用 |
| effect_summary_template | string | 效果摘要模板 | 供后续知识系统使用 |

### 5.4 FeedbackResult（反馈结果）

`FeedbackResult` 是 `FeedbackResolver` 的输出。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| success | bool | 是否成功 | 是否解析成功 |
| payloads | list<FeedbackPayload> | 反馈载荷 | 可写事件、可发前端 |
| evidences | list<FeedbackEvidence> | 反馈证据 | 给认知、记忆、知识系统 |
| delayed_feedbacks | list<DelayedFeedback> | 延迟反馈 | 等待后续触发 |
| event_drafts | list<EventDraft> | 事件草稿 | 给事件系统构造 `FeedbackGenerated` |
| trace | FeedbackTrace | 解释链 | 调试和测试用 |
| errors | list<ErrorDetail> | 错误 | 解析失败或配置问题 |

约定：

```text
success=false 时，不能输出不可信的 payload。
success=true 但 payloads 为空是合法的，例如无明显反馈。
```

### 5.5 FeedbackPayload（反馈载荷）

`FeedbackPayload` 是反馈系统内的标准载荷，可转换为 `FeedbackEventPayload`。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| feedback_id | string | 反馈实例 ID | 本次反馈的临时唯一标识 |
| feedback_type | list<FeedbackType> | 反馈类型 | 可多选 |
| intensity | FeedbackIntensity | 强度 | 最终强度 |
| reliability | FeedbackReliability | 可靠性 | 最终可靠性 |
| source | list<FeedbackSource> | 来源 | 可多选 |
| message_key | string | 文案键 | 前端本地化键 |
| message_params | map<string,string> | 文案参数 | 对象名、数值摘要等 |
| related_effects | list<EffectId> | 相关效果 | 关联效果定义 |
| related_objects | list<ObjectId> | 相关对象 | 关联对象实例 |
| related_definitions | list<ObjectDefinitionId> | 相关定义 | 关联对象定义 |
| related_traits | list<TraitId> | 相关特性 | 关联特性定义 |
| related_action_type | ActionType | 相关行为 | 行为类型 |
| visibility | EventVisibility | 可见性 | 玩家可见或仅调试 |
| importance | EventImportance | 重要性 | 事件重要程度 |
| presentation_hint | FeedbackPresentationHint | 表现提示 | 展示建议 |
| reward_hint | FeedbackRewardHint | 奖励提示 | 强化学习预留 |

注意：

```text
FeedbackPayload 不直接包含大对象快照。
related_* 只放 ID，不放完整对象。
message_params 只能放展示参数，不能放规则判断逻辑。
```

### 5.6 FeedbackEvidence（反馈证据）

`FeedbackEvidence` 是反馈系统给认知、记忆、知识系统的输入。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| evidence_id | string | 证据 ID | 本次证据临时 ID |
| source_feedback_id | string | 来源反馈 | 来自哪个反馈 |
| subject_id | string | 主体 ID | 对象定义、对象实例、区域、行为等 |
| subject_kind | string | 主体类型 | object_definition / object_instance / region / action |
| action_type | ActionType | 行为类型 | 吃、用、组合、观察等 |
| effect_summary | string | 效果摘要 | restore_hunger、poison、transform 等 |
| condition_summary | string | 条件摘要 | fresh、burning、with_fire 等 |
| evidence_type | EvidenceType | 证据类型 | 直接测试、观察、族群报告等 |
| reliability | FeedbackReliability | 可靠性 | 证据可靠性 |
| intensity | FeedbackIntensity | 强度 | 证据强度 |
| confidence_delta | Delta | 置信度建议变化 | 交给认知系统解释 |
| contradiction_hint | bool | 反证提示 | 是否可能反驳旧知识 |
| related_event_hint | EventType | 事件提示 | 通常为 FeedbackGenerated |
| trace_id | string | 解释链 ID | 方便追踪来源 |

设计重点：

```text
FeedbackEvidence 不是 KnowledgeClaim。
FeedbackEvidence 不是 Memory。
FeedbackEvidence 只表达“这次反馈可以作为某种证据”。
```

### 5.7 FeedbackAggregation（反馈聚合）

`FeedbackAggregation` 描述候选合并规则。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| aggregation_key | string | 聚合键 | 相同键可合并 |
| candidates | list<FeedbackCandidate> | 候选列表 | 被合并的候选 |
| merged_type | list<FeedbackType> | 合并类型 | 合并后的反馈类型 |
| merged_source | list<FeedbackSource> | 合并来源 | 合并后的来源 |
| final_intensity | FeedbackIntensity | 最终强度 | 通常取最高或规则计算 |
| final_reliability | FeedbackReliability | 最终可靠性 | 根据来源和冲突计算 |
| selected_message_key | string | 选中文案 | 聚合后文案 |
| suppressed_candidate_ids | list<string> | 被压制候选 | 不单独展示的候选 |

聚合原则：

```text
相同对象、相同行为、相同效果的反馈优先聚合。
严重风险反馈不能被普通反馈压制。
失败反馈可以和知识反馈合并。
Critical 反馈必须保留单独事件或高优先级事件。
```

### 5.8 DelayedFeedback（延迟反馈）

`DelayedFeedback` 表示未来才出现的反馈。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| delayed_feedback_id | string | 延迟反馈 ID | 唯一标识 |
| source_action_event_id | EventId | 来源行动事件 | 哪次行为导致 |
| source_candidate | FeedbackCandidate | 来源候选 | 原始候选摘要 |
| trigger_time | GameTime | 触发时间 | 到时间触发 |
| trigger_condition_ids | list<ExpressionId> | 触发条件 | 条件满足才触发 |
| expire_time | GameTime | 过期时间 | 超过后不再触发 |
| target_ids | list<string> | 目标 | 反馈目标 |
| expected_payload_template | FeedbackPayloadTemplate | 载荷模板 | 触发后生成反馈 |
| status | string | 状态 | pending / triggered / expired / cancelled |

示例：

```text
吃毒蘑菇后，不是立刻中毒，而是 3 回合后出现腹痛反馈。
```

第一阶段可只设计结构，不必须实现完整调度器。

### 5.9 FeedbackPresentationHint（反馈表现提示）

`FeedbackPresentationHint` 给前端提供展示建议。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| channel | string | 展示通道 | toast / log / panel / alert / silent |
| emphasis | string | 强调程度 | normal / important / danger / discovery |
| icon_key | string | 图标键 | 前端资源键 |
| sound_key | string | 音效键 | 前端资源键 |
| camera_hint | string | 镜头提示 | none / focus_actor / focus_target |
| duration_ms | int | 展示时长 | 建议值 |
| allow_merge_in_ui | bool | 允许 UI 合并 | 前端可合并展示 |

注意：

```text
PresentationHint 不是规则结果。
前端可以根据设备和 UI 方案调整表现，但不能改变反馈事实。
```

### 5.10 FeedbackRewardHint（反馈奖励提示）

`FeedbackRewardHint` 给后续强化学习投影提供弱信号。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| category | string | 奖励类别 | survival / discovery / risk / waste / social / civilization |
| sign | int | 方向 | -1 / 0 / 1 |
| magnitude_hint | float | 强度提示 | 0 到 1 的建议值 |
| reason_key | string | 原因键 | 解释奖励原因 |
| sparse_candidate | bool | 稀疏奖励候选 | 是否可能成为长期奖励 |

边界：

```text
reward_hint 不是最终 reward。
最终 reward 必须由后续训练系统结合目标函数计算。
```

### 5.11 FeedbackTrace（反馈解释链）

`FeedbackTrace` 用于调试和测试。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| trace_id | string | 解释链 ID | 唯一标识 |
| input_candidate_count | Count | 输入候选数 | 输入数量 |
| matched_rule_ids | list<string> | 命中规则 | 哪些反馈规则命中 |
| skipped_rule_ids | list<string> | 跳过规则 | 条件不满足的规则 |
| aggregation_records | list<FeedbackAggregation> | 聚合记录 | 如何合并 |
| generated_payload_ids | list<string> | 输出载荷 | 生成了哪些反馈 |
| generated_evidence_ids | list<string> | 输出证据 | 生成了哪些证据 |
| evaluation_traces | list<EvaluationTrace> | 表达式解释 | 条件表达式求值 |
| warnings | list<string> | 警告 | 非致命问题 |

## 6. 接口设计

### 6.1 FeedbackResolver

`FeedbackResolver` 是反馈系统核心接口。

```text
resolve_feedback(context, candidates) -> Result<FeedbackResult>
```

| 项目 | 说明 |
|---|---|
| 输入 | FeedbackContext、list<FeedbackCandidate> |
| 输出 | Result<FeedbackResult> |
| 是否修改 GameState | 否 |
| 是否产生事件 | 不直接写事件，只输出 event_drafts |
| 是否读取配置 | 是，读取 FeedbackRule 和文案键配置 |
| 是否使用随机数 | 否 |
| 是否允许失败 | 是 |
| 失败返回 | ErrorDetail，通常为配置错误、候选非法、表达式错误 |
| 调用顺序 | 在 InteractionResolver 之后，EventBuilder 之前 |

### 6.2 FeedbackContext

```text
build_feedback_context(rule_context, interaction_result) -> FeedbackContext
```

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| game_time | GameTime | 游戏时间 | 当前结算时间 |
| actor_id | EntityId | 行动者 | 谁触发反馈 |
| action_type | ActionType | 行为 | 当前行为类型 |
| command_id | CommandId | 命令 ID | 来源命令 |
| interaction_trace_id | string | 交互解释链 | 来源交互 trace |
| object_context | ObjectContext | 对象上下文 | 目标对象信息 |
| trait_context | TraitContext | 特性上下文 | 已解析特性信息 |
| evaluation_context | EvaluationContext | 表达式上下文 | 受限表达式求值 |
| episode_id | EpisodeId | 训练局 ID | 可选，RL 预留 |
| decision_id | DecisionId | 决策 ID | 可选，AI 决策追踪 |
| trace_enabled | bool | 是否启用 trace | 调试 / 测试 |

### 6.3 FeedbackRuleMatcher

```text
match_rules(context, candidate) -> Result<list<FeedbackRule>>
```

职责：

```text
根据反馈类型、来源、行为、效果、表达式条件匹配反馈规则。
```

副作用：无。

失败情况：

| 失败 | 处理 |
|---|---|
| 规则引用不存在的 ExpressionId | DataError |
| 表达式访问非法字段 | DataError |
| 候选缺少必要字段 | RuleError |

### 6.4 FeedbackAggregator

```text
aggregate(context, candidate_results) -> Result<list<FeedbackPayload>>
```

职责：

```text
合并重复反馈，保留高优先级反馈，避免前端信息爆炸。
```

副作用：无。

### 6.5 FeedbackEvidenceBuilder

```text
build_evidence(context, payload, policy) -> Result<optional<FeedbackEvidence>>
```

职责：

```text
把反馈载荷转换为认知证据。
```

副作用：无。

注意：

```text
EvidenceBuilder 不判断知识是否成立。
只负责把“这次反馈可作为证据”表达出来。
```

### 6.6 FeedbackEventDraftBuilder

```text
build_feedback_event_drafts(context, payloads) -> Result<list<EventDraft>>
```

职责：

```text
把 FeedbackPayload 转为 FeedbackGenerated 事件草稿。
```

副作用：无。

输出必须满足 `04_事件系统设计.md`：

```text
EventType = FeedbackGenerated
Payload = FeedbackEventPayload
payload 只放最小事实
不包含大对象快照
```

### 6.7 DelayedFeedbackScheduler 接口预留

```text
schedule_delayed_feedback(context, delayed_feedbacks) -> Result<ScheduleResult>
```

第一阶段只设计，不强制实现。

职责：

```text
把延迟反馈注册到后续规则管线或时间系统。
```

副作用：可以修改后续待处理队列，但不能直接改对象状态。

## 7. 状态流转

### 7.1 即时反馈流程

```text
1. ActionCommand 进入交互规则。
2. InteractionResolver 输出 InteractionResult。
3. InteractionResult 携带 FeedbackCandidate 列表。
4. FeedbackResolver 读取 FeedbackRule。
5. FeedbackRuleMatcher 匹配规则。
6. ExpressionEvaluator 校验反馈条件。
7. FeedbackAggregator 聚合重复反馈。
8. FeedbackResolver 输出 FeedbackPayload。
9. FeedbackEvidenceBuilder 输出 FeedbackEvidence。
10. FeedbackEventDraftBuilder 输出 FeedbackGenerated EventDraft。
11. 事件系统创建 EventRecord。
12. 后续认知系统读取 FeedbackEvidence。
13. 前端读取 FeedbackGenerated Event 展示结果。
```

### 7.2 延迟反馈流程

```text
1. 当前行为产生 DelayedFeedback。
2. 规则管线保存待触发反馈。
3. 时间推进或条件变化时检查触发条件。
4. 条件满足则重新进入 FeedbackResolver。
5. 生成新的 FeedbackGenerated Event。
6. 认知系统把延迟反馈作为新证据。
```

### 7.3 无明显反馈流程

```text
1. 行为合法。
2. 交互规则输出 Failure / NoEffect 类型候选。
3. 反馈系统生成 Minor 或 Trace 级别反馈。
4. 事件系统记录 FeedbackGenerated。
5. 认知系统可形成“此行为当前条件下无明显效果”的证据。
```

无反馈不是没有信息。

```text
石头砸水面没有实用结果，也是一条可学习的排除性证据。
```

## 8. 事件输出

反馈系统只输出事件草稿，不直接写事件存储。

### 8.1 FeedbackGenerated

每个最终 `FeedbackPayload` 原则上对应一个 `FeedbackGenerated` 事件。

事件字段建议：

| 字段 | 值 |
|---|---|
| event_type | FeedbackGenerated |
| payload | FeedbackEventPayload |
| visibility | 来自 FeedbackPayload.visibility |
| importance | 来自 FeedbackPayload.importance |
| actor_id | 来自 FeedbackContext.actor_id |
| command_id | 来自 FeedbackContext.command_id |
| episode_id | 可选，来自 FeedbackContext.episode_id |
| decision_id | 可选，来自 FeedbackContext.decision_id |
| reward_hint | 可选，来自 FeedbackPayload.reward_hint |

### 8.2 FeedbackEventPayload 映射

| FeedbackPayload 字段 | FeedbackEventPayload 字段 |
|---|---|
| feedback_type | feedback_type |
| intensity | intensity |
| reliability | reliability |
| source | source |
| message_key | message_key |
| related_effects | related_effects |

### 8.3 事件可见性规则

| 场景 | EventVisibility | EventImportance |
|---|---|---|
| 玩家直接感受到饥饿降低 | PlayerVisible | Normal |
| 中毒、严重受伤、死亡 | PlayerVisible | Important / Critical |
| 调试用微弱表达式反馈 | DebugOnly | Low |
| AI 训练内部反馈 | Hidden | Normal |
| 文明能力解锁反馈 | PlayerVisible | Critical |

### 8.4 与 ActionResolved 的关系

```text
ActionResolved 说明行为结算完成。
FeedbackGenerated 说明行为产生了哪些可感知结果。
```

同一次行为可以产生：

```text
CommandReceived
ActionStarted
ActionResolved
ObjectStateChanged
EntityStateChanged
FeedbackGenerated
MemoryCreated
KnowledgeUpdated
```

前端必须以事件顺序展示，不允许自己根据 ActionResolved 推导反馈。

## 9. 配置格式

反馈配置属于 `ConfigPack` 的一部分，由 `ConfigRegistry` 统一访问。

### 9.1 JSON 示例

```json
{
  "feedback_rules": [
    {
      "id": "feedback_eat_restore_hunger",
      "name": "食用后缓解饥饿反馈",
      "enabled": true,
      "match_feedback_types": ["physiological", "knowledge"],
      "match_sources": ["direct_experience"],
      "match_action_types": ["eat"],
      "match_effect_ids": ["restore_hunger_small"],
      "condition_ids": ["expr_actor_hunger_decreased"],
      "intensity_override": "moderate",
      "reliability_override": "anecdotal",
      "message_key_override": "feedback.eat.restore_hunger",
      "presentation_hint": {
        "channel": "log",
        "emphasis": "discovery",
        "icon_key": "hunger_down",
        "sound_key": "soft_success",
        "camera_hint": "focus_actor",
        "duration_ms": 1800,
        "allow_merge_in_ui": true
      },
      "evidence_policy": {
        "create_evidence": true,
        "evidence_type": "direct_test",
        "confidence_delta": 0.12,
        "contradiction_possible": true,
        "subject_from": "target_definition",
        "action_from_context": true,
        "condition_summary_template": "state:{target_state}",
        "effect_summary_template": "restore_hunger"
      },
      "aggregation_key_template": "{actor}:{action}:{target_definition}:{effect}",
      "priority": 100
    }
  ]
}
```

### 9.2 配置约束

```text
feedback_rules.id 必须唯一。
message_key 必须存在于文案表或允许前端兜底。
match_effect_ids 必须引用存在的 EffectId。
condition_ids 必须引用存在的 ExpressionId。
evidence_policy.create_evidence=true 时，必须填写 evidence_type。
confidence_delta 必须在允许范围内。
priority 不允许缺省。
```

### 9.3 表达式使用限制

反馈规则中的表达式只能用于：

```text
判断反馈是否成立。
判断反馈强度是否覆盖。
判断是否生成证据。
生成 trace。
```

禁止：

```text
修改 GameState。
创建事件。
调用随机数。
访问非白名单字段。
执行自由脚本。
```

## 11. 校验规则

### 11.1 候选校验

| 规则 | 错误级别 |
|---|---|
| feedback_type 为空 | RuleError |
| intensity 为空 | RuleError |
| reliability 为空 | RuleError |
| source 为空 | RuleError |
| related_effect_id 不存在但声明了效果 | DataError |
| target_ids 中存在非法 ID | RuleError |
| message_key 为空且无规则覆盖 | Warning / DataError，取决于配置策略 |

### 11.2 配置校验

| 规则 | 错误级别 |
|---|---|
| FeedbackRule.id 重复 | DataError |
| priority 缺失 | DataError |
| ExpressionId 不存在 | DataError |
| EffectId 不存在 | DataError |
| evidence_policy 不完整 | DataError |
| aggregation_key_template 引用未知字段 | DataError |
| reward_hint.magnitude_hint 超出 0 到 1 | DataError |

### 11.3 运行时校验

| 规则 | 处理 |
|---|---|
| 候选没有匹配任何规则 | 使用默认反馈规则或生成 NoEffect 反馈 |
| 多条规则同优先级冲突 | 返回 RuleError，并写 RuleErrorOccurred 草稿 |
| 表达式求值失败 | 返回 DataError 或 RuleError |
| 聚合后没有 payload | 合法，但 trace 必须说明原因 |
| Critical 反馈被压制 | 非法，返回 RuleError |

### 11.4 兼容性校验

反馈系统必须和前置系统满足以下关系：

```text
09 的 FeedbackCandidate 可以被 10 读取。
10 的 FeedbackPayload 可以映射到 04 的 FeedbackEventPayload。
10 的 FeedbackEvidence 可以被 11/12/13 使用。
10 的 expression 条件遵守 05 的受限表达式规则。
10 的错误返回遵守 03 的 Result / ErrorDetail 规则。
10 的 ID 字段遵守 01 的强类型 ID 约定。
```

### 12.1 单元测试

| 测试 | 输入 | 期望 |
|---|---|---|
| 解析单个生理反馈 | eat + restore_hunger | 输出 Physiological payload |
| 解析风险反馈 | eat + poison | 输出 Risk payload，importance 高 |
| 解析对象反应反馈 | fire + branch | 输出 ObjectState / Resource payload |
| 解析无效果反馈 | use stone on water | 输出 Failure / Knowledge payload |
| 反馈规则条件不满足 | condition=false | 不生成对应 payload |
| 反馈规则覆盖强度 | candidate=Minor，rule=Major | 输出 Major |
| 反馈规则覆盖文案 | message_key_override | 使用覆盖文案 |
| 生成证据 | create_evidence=true | 输出 FeedbackEvidence |
| 不生成证据 | create_evidence=false | 只输出 payload |
| 生成事件草稿 | payloads 非空 | 输出 FeedbackGenerated EventDraft |

### 12.2 聚合测试

| 场景 | 期望 |

### 12.3 认知证据测试

| 场景 | 期望 |
|---|---|

## 13. 待定问题

| 问题 | 当前决策 |
|---|---|
| FeedbackId 是否需要进入 01 基础 ID | 第一阶段先用字符串临时 ID，等反馈实现时再决定是否强类型化 |
| DelayedFeedback 是否由反馈系统调度 | 第一阶段只设计接口，实际调度归规则管线或时间系统 |
| message_params 是否允许数值 | 允许展示数值摘要，但不允许前端据此反推规则 |
| 可靠性升级是否在反馈系统完成 | 不在反馈系统完成，交给认知/知识系统 |
| reward_hint 是否参与核心规则 | 不参与，只作为事件和训练投影的弱提示 |

## 14. 设计注意事项

```text
反馈不是状态变化。
反馈不是事件本身。
反馈不是记忆。
反馈不是知识。
反馈不是 UI 文案。
```

研发实现时必须遵守：

```text
不要在前端根据 hp/hunger 自己推导反馈。
不要让 FeedbackResolver 修改 GameState。
不要让 FeedbackRule 执行自由脚本。
不要把 FeedbackCandidate 直接写入 EventStore。
不要把 FeedbackEvidence 当成 KnowledgeClaim。
不要把 reward_hint 当成强化学习最终 reward。
不要压制 Critical 反馈。
不要丢弃无效果反馈，因为失败也是认知证据。
```

## 16. 阶段结论

反馈系统把“行为结算结果”转换成“玩家能理解、系统能记录、认知能吸收、AI 能投影”的统一反馈层。

它是以下系统之间的桥梁：

```text
交互规则
  ↓
反馈系统
  ↓
事件系统 / 认知系统 / 记忆系统 / 前端表现 / 强化学习投影
```

当前设计可以支持：

- 直接反馈。
- 延迟反馈。
- 无效果反馈。
- 风险反馈。
- 对象反应反馈。
- 文明反馈。
- 聚合与去重。
- 认知证据输出。
- 事件流记录。
- 后续强化学习接入。

下一阶段进入：

```text
11_认知状态设计.md
```

认知状态系统将负责记录“某个主体当前知道什么、不确定什么、误解了什么、对什么有多大置信度”。
