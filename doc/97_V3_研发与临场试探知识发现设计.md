# 97_V3_研发与临场试探知识发现设计

## 1. 阶段定位

本阶段是 V3 认知生态沙盒在“地图绘制 / 原料投放 / 无知 NPC 探索”之后的知识发现核心阶段。

它解决两个问题：

```text
1. NPC 如何通过研发逐步发现合成知识，而不是瞬间知道 JSON 配方。
2. NPC 如何在危险场景中通过临场试探发现目标知识，而不是上帝视角知道火把能驱赶狼。
```

本阶段属于玩法规则系统和 Agent 知识闭环阶段，不是前端表现阶段，不是临时 demo 规则。

它接入以下已有系统：

```text
ContentRegistry
WorldCommandPipeline
Reaction / Feedback
Experience Extractor
Knowledge Learning
Agent Runtime / Agent Policy
Client Projection
```

本阶段为后续能力打基础：

```text
知识反推行动
教学与观察学习
知识卡分配
族群知识传播
文明研发树
魔法、机械、外星内容扩展
```

## 2. 为什么现在做

V3 的核心乐趣不是玩家直接操控小人做对事，而是玩家塑造环境后观察无知 NPC 自己探索、试错、学习、传播。

如果没有本阶段：

```text
NPC 只能靠写死规则行动。
新增物品和反应无法自然进入知识系统。
火把驱狼、石头砸狼、木头挡狼等行为会变成 if 分支。
NPC 会显得像上帝视角，不像生活在世界中的生命。
玩家投放原料后没有长期观察价值。
```

因此本阶段必须在大规模扩展内容前完成。

## 3. 阶段目标

本阶段完成后，系统必须具备：

```text
1. NPC 可以启动“研发任务”，通过 tick 消耗时间研究物品组合。
2. 研发复杂度随组合数量、材料陌生度、失败历史变化。
3. 每天只有固定研发时间或研发点数，不能无限研发。
4. 已发现配方不再重复研发。
5. 研发失败会记录失败知识，避免无限重复无效组合。
6. NPC 遇到危险时不会知道正确答案，只会从当前可用行为中临场试探。
7. 火把驱狼、石头砸狼、木头挡狼等都必须由 Reaction 裁决。
8. 成功和失败都进入 Experience，再进入 Knowledge。
9. 前端只显示研发进度、试探行为和知识结果，不推导规则。
```

## 4. 本阶段不解决的问题

本阶段不做：

```text
完整科技树 UI。
玩家手写任务图编辑器。
复杂族群制度。
后代继承学习。
大型文明阶段。
魔法和机械的完整内容包。
NPC 语言交流系统。
```

这些放到后续阶段。

本阶段只做“发现机制”的底座。

## 5. 核心设计原则

### 5.1 NPC 不读隐藏配方

NPC 不能直接读取 Reaction 的真实产物作为行动依据。

允许读取：

```text
safe_tags
safe_trait_keys
当前可见对象
自己背包
已知知识
失败记录
当前需求
```

禁止读取：

```text
reaction.outputs
隐藏 feedback 真相
威胁弱点答案
未学习知识
```

### 5.2 研发是长期任务，不是按钮瞬间完成

研发必须是运行态任务：

```text
ResearchTask
progress_tick
required_tick
daily_budget
candidate_combination
```

Command 只能启动或推进研发，不能瞬间把配方塞进知识库。

### 5.3 临场试探不是研发

研发用于稳定环境下研究组合。

临场试探用于危险、饥饿、寒冷、受伤等即时需求。

两者都能产生知识，但节奏不同：

```text
研发：慢、可计划、组合多、成本可控。
临场试探：快、压力大、候选少、风险高。
```

### 5.4 失败也是知识

无效果、低效、有害、材料浪费都必须形成经验。

否则 NPC 会无限重复同一个错误。

### 5.5 候选动作来自可感知属性，不来自答案

危险时 NPC 不能因为“狼怕火”而优先火把。

它只能因为：

```text
火把：可挥舞、有火、有光、有热。
石头：可投掷、坚硬。
木头：可挥舞、坚硬。
红果：可投掷但不像武器。
```

生成候选动作。

Reaction 决定哪个有效。

### 5.6 同一规则只有一个权威来源

合成是否成功：Reaction 权威。

知识是否形成：Knowledge Learning 权威。

NPC 是否选择尝试：Agent Policy 权威。

前端不能写这些规则。

## 6. 模块划分

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| `ResearchCandidateProvider` | 研发候选生成器 | 根据 NPC 可见物品、背包、线索生成可研发组合 | NPC 状态、物品线索、失败记录、已知知识 | 候选组合列表 | 读取隐藏配方产物 |
| `ResearchTaskRuntime` | 研发任务运行态 | 保存当前研发进度、材料、消耗 tick、每日预算 | 研发命令、tick | 研发进度 | 直接写知识 |
| `ResearchCommandHandler` | 研发命令处理器 | 启动、推进、取消研发 | Command | Runtime 变更、事件 | 绕过管线 |
| `ResearchResolver` | 研发结算器 | 研发完成时提交组合到 Reaction 系统裁决 | 研发组合 | 成功/失败反馈 | 自己实现配方判断 |
| `ExperimentMemoryIndex` | 试错索引 | 记录已试过组合和失败次数 | Experience / Knowledge | 去重判断 | 只存在前端 |
| `SituationalExperimentPlanner` | 临场试探规划器 | 在危险/需求下生成候选行动 | 需求、可用物品、线索、知识 | 候选行动 | 根据隐藏弱点排序 |
| `ThreatTrialCommandProvider` | 危险试探命令提供者 | 生成对威胁使用物品、投掷、逃跑等 Command | 可见威胁、可用物品 | Command option / Agent command | 写 wolf/torch if |
| `ExperienceToKnowledgeBridge` | 经验转知识桥 | 把研发和临场结果写入经验并触发知识形成 | Command result / feedback | Knowledge claim | 直接写稳定知识 |
| `ResearchProjectionAdapter` | 研发投影 | 给前端展示研发进度、预算、候选摘要 | Runtime | Projection | 暴露隐藏配方 |

## 7. 研发系统设计

### 7.1 研发对象

研发对象不是配方本身，而是“组合假设”。

```cpp
struct ResearchCandidate {
    std::string candidate_id;
    std::string actor_key;
    std::vector<std::string> object_keys;
    std::vector<std::string> perceived_trait_keys;
    int combination_size;
    int novelty_score;
    int previous_failure_count;
    int estimated_required_ticks;
};
```

示例：

```text
候选 A：木头 + 火源
候选 B：碎石 + 木头
候选 C：碎石 + 木头 + 藤蔓
候选 D：蘑菇 + 火源
```

NPC 不知道这些会产出什么。

### 7.2 研发任务

```cpp
struct ResearchTask {
    std::string task_id;
    std::string actor_key;
    std::vector<std::string> object_keys;
    int required_ticks;
    int progressed_ticks;
    int daily_tick_budget_used;
    std::string status; // running / paused / completed / failed / cancelled
    std::vector<std::string> consumed_sample_entity_ids;
};
```

研发不是瞬间完成。

每个世界 tick 或每天研发 tick 推进：

```text
如果当前时间在研发时段内 -> progress + actor.research_speed
否则暂停
```

### 7.3 研发时间公式

第一版使用简单公式：

```text
required_ticks = base + combination_size_factor + novelty_factor + failure_factor
```

建议：

```text
base = 20
combination_size_factor = (组合数量 - 1) * 20
novelty_factor = 未知材料数量 * 10
failure_factor = 相似失败次数 * 5
```

示例：

```text
木头 + 火源：40 tick
碎石 + 木头：40 tick
碎石 + 木头 + 藤蔓：60 tick
未知蘑菇 + 火源 + 黏土：80+ tick
```

### 7.4 每日研发时间

NPC 每天只有固定研发预算。

```cpp
struct ActorResearchBudget {
    std::string actor_key;
    int day_index;
    int max_research_ticks_per_day;
    int used_research_ticks;
};
```

默认：

```text
每天 60 tick 可用于研发。
```

如果 NPC 饥饿、受伤、害怕，研发效率下降或停止。

### 7.5 已发现不重复研发

开始研发前必须查：

```text
是否已经知道同一组合可以产生某结果。
是否已经知道同一组合无明显效果。
是否已经达到失败次数上限。
```

如果已经知道，就不生成候选。

注意：

```text
木头 + 火源 -> 火把
火源 + 木头 -> 火把
```

需要归一化为同一个组合 key。

```text
research_combo_key = sorted(object_keys).join("+")
```

### 7.6 研发成功

研发完成时调用 Reaction：

```text
ResearchResolver -> ReactionService.evaluateCombination(candidate)
```

如果成功：

```text
生成产物或效果
写入 Experience
Knowledge Learning 形成合成知识
投影显示“发现了新做法”
```

示例知识：

```text
木头 + 火源 -> 制作火把
碎石 + 木头 + 藤蔓 -> 制作斧头
```

### 7.7 研发失败

如果 Reaction 无匹配或条件不足：

```text
写入失败经验
形成低优先级失败知识
增加组合失败次数
短时间内不再尝试
```

示例：

```text
红果 + 石头 -> 无明显合成结果
木头 + 蘑菇 -> 无明显合成结果
```

失败知识不一定永久封死，可以有过期或条件变化重试：

```text
新增第三材料后可以再次尝试。
NPC 学到新工具知识后可以重试。
世界阶段变化后可以重试。
```

## 8. 临场试探系统设计

### 8.1 临场试探触发条件

触发源：

```text
饥饿
寒冷
受伤
野兽靠近
被攻击
火灾
中毒
迷路
族群冲突
```

本阶段第一版只要求：

```text
野兽靠近 / 被攻击
```

### 8.2 先查已有知识

NPC 遇到危险时先查知识：

```text
有没有“某对象 对该威胁 使用 -> 有效反制”的知识？
有没有“逃跑有效”的知识？
有没有“呼救有效”的知识？
```

如果有，优先使用已知解。

如果没有，才进入试探。

### 8.3 候选动作生成

候选动作来自可感知属性。

示例：

| 物品 | 可感知属性 | 可生成候选 |
|---|---|---|
| 石头 | 坚硬、可投掷 | 投掷石头、砸击 |
| 木头 | 坚硬、长条、可挥舞 | 挥舞木头、阻挡 |
| 火把 | 火、光、热、可挥舞 | 挥舞火把、靠近威胁 |
| 红果 | 小、可投掷、食物 | 丢出红果 |
| 藤蔓 | 可捆绑 | 尝试缠绕 |

这里没有任何“狼怕火”的知识。

### 8.4 候选排序

排序不能用隐藏答案。

允许使用：

```text
动作风险
动作距离
物品是否在手边
物品是否可投掷
物品是否像武器
物品是否已经失败过
NPC 性格：胆小/莽撞/好奇
当前生命值
```

示例：

```text
胆小 NPC：逃跑 > 呼救 > 投掷 > 近身挥舞
莽撞 NPC：投掷 > 挥舞 > 逃跑
好奇 NPC：未尝试物品权重更高
```

### 8.5 Reaction 裁决

临场试探执行 Command：

```text
UseOn(torch, beast)
Throw(stone, beast)
UseOn(wood, beast)
Throw(red_berry, beast)
Flee()
```

Reaction / Threat 系统裁决：

```text
火把可能驱赶狼。
石头可能造成轻微伤害或短暂击退。
木头可能阻挡一次攻击。
红果可能无效或吸引动物。
逃跑可能成功也可能失败。
```

NPC 不提前知道结果。

### 8.6 知识形成

成功示例：

```text
火把 -> 对野兽使用 -> 驱赶
石头 -> 投掷野兽 -> 短暂击退
木头 -> 对野兽使用 -> 降低伤害
```

失败示例：

```text
红果 -> 投掷野兽 -> 无明显效果
徒手靠近野兽 -> 受伤
```

这些都进入 Knowledge。

### 8.7 临场试探限制

为了避免 NPC 像机器人刷试验：

```text
每次危险事件最多试探 N 次。
生命过低直接逃跑。
同一危险中失败过的动作不重复。
缺少物品不能尝试。
不可见物品不能尝试。
```

第一版建议：

```text
每次危险最多试探 2 次。
失败后优先逃跑。
```

## 9. 研发与临场试探的区别

| 项 | 研发 | 临场试探 |
|---|---|---|
| 触发 | 空闲/计划 | 需求/危险 |
| 时间 | 多 tick | 立即或短 tick |
| 候选 | 多材料组合 | 手头行为 |
| 成本 | 材料样本、时间 | 生命风险、物品损耗 |
| 知识 | 合成知识 | 目标关系知识 |
| 示例 | 木头+火=火把 | 火把对狼=驱赶 |

## 10. Command 设计

新增命令建议：

```cpp
enum class WorldCommandKind {
    StartResearch,
    AdvanceResearch,
    CancelResearch,
    TryUseOnThreat,
    ThrowObject
};
```

也可以第一版复用：

```text
Use / UseOn / Craft / SystemTick
```

但必须保证语义清晰。

建议第一版新增：

```text
StartResearch
AdvanceResearch
TryObjectOnTarget
```

### 10.1 StartResearch

输入：

```text
actor_key
candidate_id
object_entity_ids
```

校验：

```text
actor 存在
物品可见或在背包
组合未发现
组合未在失败冷却中
当天还有研发预算
```

输出：

```text
ResearchTaskCreated
```

### 10.2 AdvanceResearch

由 tick 或等待推进。

输入：

```text
task_id
tick_delta
```

校验：

```text
任务存在
当前处于研发时间
NPC 状态允许研发
材料仍可用
```

输出：

```text
ResearchProgressed
ResearchCompleted
ResearchFailed
```

### 10.3 TryObjectOnTarget

用于临场试探。

输入：

```text
actor_key
source_object_entity_id
target_entity_id
action_key
```

校验：

```text
actor 能感知 target
actor 能接触或投掷 source
source 未被隐藏
同一危险事件未重复尝试
```

输出：

```text
ThreatTrialResolved
ExperienceWritten
```

## 12. Projection 设计

前端需要看到：

```text
NPC 正在研发什么类型的组合。
研发进度条。
当天剩余研发时间。
研发成功/失败事件。
危险中 NPC 尝试了什么。
尝试结果形成了什么知识。
```

不能看到：

```text
隐藏配方产物。
未完成研发的真实答案。
狼的隐藏弱点。
Reaction 内部命中规则 ID。
```

投影示例：

```json
{
  "research_tasks": [
    {
      "task_id": "research_001",

    {
      "actor_key": "npc_amu",

      "last_trial": "投掷碎石",
      "result": "短暂击退"
    }
  ]
}
```

## 13. 前端表现要求

前端可以显示：

```text
NPC 头顶：研发中 / 试探中 / 逃跑中
NPC 面板：研发进度条
事件面板：阿木尝试用碎石砸向野兽
知识面板：碎石投掷野兽可短暂击退
```

前端禁止：

```text
根据物品标签自行决定研发结果。

绕过 Command 给 NPC 加知识。
```

## 14. 测试设计

### 14.1 研发不是瞬间完成

测试：

```text
启动 木头+火源 研发
确认立即没有火把知识
推进不足 required_ticks
确认仍未形成知识
推进到完成
确认 Reaction 产出经验和知识
```

### 14.2 每日研发预算限制

测试：

```text
一天内启动多个研发
预算耗尽后无法继续推进
第二天预算刷新
```

### 14.3 已发现配方不重复研发

测试：

```text
NPC 已知 木头+火源->火把
候选生成器不再生成同组合
```

### 14.4 失败组合进入冷却

测试：

```text
红果+石头研发失败
同一天不再生成同候选
失败记录进入 ExperimentMemoryIndex
```

### 14.5 危险试探不是上帝视角

测试：

```text
NPC 有石头、木头、火把，但没有驱狼知识
候选中包含石头、木头、火把、逃跑
不能只生成火把
```

### 14.6 火把知识由试探形成

测试：

```text
NPC 不知道火把驱狼
对狼使用火把成功
形成 火把->对野兽使用->驱赶 知识
```

### 14.7 石头也能形成知识

测试：

```text
NPC 投掷石头命中野兽
形成 石头->投掷野兽->短暂击退 知识
```

### 14.8 无效行为降低优先级

测试：

```text
NPC 对狼丢红果无效
下一次同危险中不重复丢红果
知识里记录无明显效果
```

### 14.9 前端不能伪造

测试：

```text
前端提交不存在的研发 candidate_id
后端 Blocked
前端提交隐藏配方结果
后端 Blocked
```

## 15. 风险与解决方案

### 15.1 组合爆炸

风险：物品多后组合数量爆炸。

解决：

```text
只生成 NPC 可见/可持有物品组合。
限制组合大小第一版 <= 3。
根据 safe_tags 过滤完全无关组合。
每日预算限制。
失败冷却。
```

### 15.2 NPC 仍然像上帝视角

风险：候选排序不小心使用了隐藏结果。

解决：

```text
候选层只能读取 safe tags、已知知识、失败记录。
测试禁止读取 reaction.outputs 做排序。
架构扫描禁止 ResearchCandidateProvider include ReactionDefinition outputs。
```

### 15.3 NPC 太蠢不好看

风险：完全随机导致玩家看不懂。

解决：

```text
候选不是纯随机，而是基于可感知属性。
例如石头更像投掷物，火把更像挥舞物，红果优先级低。
但每个行为是否有效仍由 Reaction 决定。
```

### 15.4 失败知识过早封死

风险：NPC 因一次失败永久不尝试。

解决：

```text
失败记录有冷却，不一定永久封死。
新增材料、新工具、新知识后可以重新生成扩展组合。
```

### 15.5 研发和制作混淆

风险：NPC 已知道配方后仍走研发。

解决：

```text
未知组合走 Research。
已知配方走 Craft / Goal Execution。
```

## 16. 最小可行版本

第一版只做以下闭环：

```text
1. NPC 能研发 2-3 个材料组合。
2. 木头+火源 研发成功形成火把知识。
3. 红果+石头 研发失败形成失败记录。
4. 狼靠近时 NPC 在石头/木头/火把/逃跑中选择临场试探。
5. 火把成功驱狼后形成目标知识。
6. 石头也可以形成短暂击退知识。
7. 前端显示研发进度和新知识。
```

不做完整文明、不做多代传播、不做复杂 UI。

## 17. 实现门禁

实现前必须确认：

```text
ResearchTask 是否通过 Command 创建。
研发推进是否由 tick / SystemTick / Wait 驱动。
Reaction 是否仍是产物权威。
Knowledge Learning 是否仍是知识权威。
Agent 是否没有读取隐藏配方结果。
前端是否只读取 Projection。
```

如果任意一项不满足，本阶段不能验收。

## 19. 阶段结论

本阶段的核心不是做一个“配方解锁按钮”，而是建立 NPC 的真实知识发现过程。

最终链路必须是：

```text
研发：
可见材料 -> 组合假设 -> 多 tick 研发 -> Reaction 裁决 -> Experience -> Knowledge

临场试探：
需求/危险 -> 可用物品候选 -> 非上帝视角尝试 -> Reaction 裁决 -> Experience -> Knowledge
```

这样后续新增内容时，只要补充 JSON 物品、反应、反馈和安全线索，NPC 就能在同一套发现机制里逐步学会新的玩法。
