# 先驱者：交互规则设计

## 1. 本文档目标

本文档用于设计《先驱者》规则引擎中的交互规则系统。

交互规则系统负责在行为命令进入规则引擎后，结算对象、行为、特性、表达式、效果、反馈和知识候选之间的关系。

一句话：

```text
ActionCommand 表达“想做什么”。
InteractionRule 判断“这个行为与这些对象会发生什么”。
InteractionResult 输出“产生了哪些效果、反馈、状态变化和知识候选”。
```

交互规则系统必须兼容：

```text
01 基础类型与 ID。
02 通用枚举。
03 错误码与结果类型。
04 事件系统。
05 配置系统与表达式。
06 世界对象系统。
07 特性系统。
08 行为命令系统。
后续反馈、认知、记忆、知识、传播、文明和规则管线。
```

## 2. 设计边界

### 2.1 本文档要解决的问题

```text
定义 InteractionRule。
定义规则如何匹配 ActionCommand。
定义交互上下文。
定义交互条件、优先级、冲突策略。
定义 InteractionResolver 如何生成结果。
定义 EffectResolutionRequest、FeedbackCandidate、KnowledgeCandidate。
定义 InteractionTrace。
定义对象反应规则如何表达。
```

### 2.2 本文档暂不解决的问题

```text
不设计 Feedback 最终聚合展示。
不设计 Memory 创建细节。
不设计 KnowledgeClaim 最终入库。
不设计传播执行。
不设计文明能力解锁。
不设计完整规则管线编排。
不直接应用最终 StateChange。
```

交互规则只负责核心交互结算，最终状态应用由规则管线负责。

## 3. 核心原则

### 3.1 交互规则不直接偷偷改状态

交互规则输出结果，不直接修改世界。

```text
输出 EffectResolutionRequest。
输出 ObjectStateChange 草稿。
输出 FeedbackCandidate。
输出 KnowledgeCandidate。
输出 Event 草稿。
```

最终是否应用，由规则管线统一处理。

### 3.2 规则优先基于特性，而不是具体对象

优先写泛化规则：

```text
任何 edible 对象 + eat -> 可能恢复饥饿。
任何 combustible 对象 + fire -> 可能燃烧。
任何 repels_predator 对象 + predator -> 降低危险。
```

必要时才写具体对象规则。

### 3.3 交互规则必须可解释

每次交互结算必须能回答：

```text
匹配了哪些规则？
为什么这些规则生效？
为什么某些规则没生效？
用了哪些表达式？
产生了哪些效果？
生成了哪些知识候选？
```

### 3.4 交互结果是认知来源，不是认知本身

交互规则可以输出 `KnowledgeCandidate`。

但 Memory 和 KnowledgeClaim 的正式创建由后续认知、记忆、知识系统处理。

### 3.5 对象反应必须走规则

禁止在 `WorldObject` 或具体函数中硬编码：

```text
火 + 木棍 = 火把
```

对象反应必须通过交互规则匹配、条件、表达式、效果和状态变化产生。

## 4. 交互规则系统总架构

第一阶段交互规则系统包含：

| 模块 | 中文名 | 作用 |
|---|---|---|
| InteractionRule | 交互规则 | 一条运行时可执行的交互规则 |
| InteractionRuleConfig | 交互规则配置 | 配置层规则定义 |
| InteractionContext | 交互上下文 | 当前命令、对象、特性、表达式上下文 |
| InteractionMatcher | 规则匹配器 | 判断命令是否匹配规则 |
| InteractionCondition | 交互条件 | 规则触发条件 |
| InteractionPriority | 优先级策略 | 规则排序和冲突处理 |
| InteractionResolver | 交互解析器 | 执行匹配、条件、结果生成 |
| InteractionResult | 交互结果 | 交互结算输出 |
| EffectResolutionRequest | 效果结算请求 | 请求后续效果系统处理 |
| FeedbackCandidate | 反馈候选 | 请求后续反馈系统处理 |
| KnowledgeCandidate | 知识候选 | 请求后续认知系统处理 |
| InteractionTrace | 交互解释链 | 调试、测试、前端解释、AI 分析 |

## 5. InteractionRuleConfig 与 InteractionRule

### 5.1 InteractionRuleConfig

`InteractionRuleConfig` 来自配置系统。

它声明规则的静态内容：

```text
匹配什么行为。
匹配什么目标。
需要哪些特性。
需要哪些条件。
产生哪些效果。
生成什么知识候选模板。
优先级是多少。
```

### 5.2 InteractionRule

`InteractionRule` 是运行时规则对象。

它由 `InteractionRuleConfig` 加载、校验、编译或缓存后得到。

### 5.3 为什么分开

```text
Config 是外部数据。
Rule 是规则引擎内部可执行结构。
后续可以加入编译缓存、索引、热加载和版本管理，而不改变配置格式。
```

## 6. InteractionRule（交互规则）

### 6.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| rule_id | string | 规则 ID | 是 | 规则唯一标识 |
| action_types | list<ActionType> | 行为类型 | 是 | 可匹配行为 |
| target_pattern | InteractionTargetPattern | 目标模式 | 是 | 目标数量、角色、类型 |
| required_object_definitions | list<ObjectDefinitionId> | 必要对象定义 | 否 | 具体对象匹配 |
| required_traits | list<TraitId> | 必要特性 | 否 | 特性匹配 |
| forbidden_traits | list<TraitId> | 禁止特性 | 否 | 排除条件 |
| conditions | list<InteractionCondition> | 条件 | 否 | 表达式条件 |
| effect_requests | list<EffectResolutionRequestTemplate> | 效果模板 | 否 | 效果请求模板 |
| feedback_templates | list<FeedbackCandidateTemplate> | 反馈模板 | 否 | 反馈候选模板 |
| knowledge_templates | list<KnowledgeCandidateTemplate> | 知识模板 | 否 | 知识候选模板 |
| priority | Count | 优先级 | 是 | 匹配排序 |
| conflict_policy | InteractionConflictPolicy | 冲突策略 | 是 | 叠加、互斥、阻断等 |
| tags | list<string> | 标签 | 否 | 分类和调试 |

### 6.2 设计注意事项

```text
InteractionRule 不直接保存 WorldObject。
InteractionRule 不直接修改状态。
InteractionRule 可以引用 EffectId、ExpressionId、TraitId、ObjectDefinitionId。
InteractionRule 必须可校验、可测试、可解释。
```

## 7. InteractionTargetPattern（交互目标模式）

`InteractionTargetPattern` 描述规则需要什么目标。

### 7.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| required_roles | list<string> | 必要角色 | primary、source、material 等 |
| allowed_target_types | list<ActionTargetType> | 允许目标类型 | 单对象、对象对、区域等 |
| min_targets | Count | 最少目标 | 目标数量下限 |
| max_targets | Count | 最多目标 | 目标数量上限 |

### 7.2 示例

吃对象：

```text
required_roles = [primary]
allowed_target_types = [single_object]
min_targets = 1
max_targets = 1
```

火加木棍：

```text
required_roles = [source, material]
allowed_target_types = [single_object]
min_targets = 2
max_targets = 2
```

## 8. InteractionContext（交互上下文）

`InteractionContext` 是交互结算的只读上下文。

### 8.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| command | CommandEnvelope | 命令 | 原始命令 |
| action_command | ActionCommand | 行为命令 | 行为主体 |
| actor_id | EntityId | 行动者 | 执行动作者 |
| targets | list<ResolvedCommandTarget> | 已解析目标 | 目标对象、定义、快照 |
| trait_results | map<ObjectId, TraitEvaluationResult> | 特性结果 | 每个对象的特性解析 |
| expression_context | EvaluationContext | 表达式上下文 | 求值上下文 |
| config_version | ConfigVersionId | 配置版本 | 当前配置版本 |
| state_version | StateVersion | 状态版本 | 当前状态版本 |
| random_context | object | 随机上下文 | 随机流引用，不直接随机 |

### 8.2 ResolvedCommandTarget

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| role | string | 目标角色 | primary、source、material 等 |
| object_id | ObjectId | 对象 ID | 目标对象 |
| definition_id | ObjectDefinitionId | 定义 ID | 对象定义 |
| object_snapshot | ObjectSnapshot | 对象快照 | 当前对象快照 |
| active_traits | list<TraitId> | 生效特性 | TraitResolver 输出 |
| runtime_flags | list<string> | 运行标记 | wet、burning 等 |

### 8.3 约束

```text
InteractionContext 是只读的。
InteractionContext 必须来自同一个 StateVersion。
表达式不能通过 Context 修改对象。
```

## 9. InteractionMatcher（规则匹配器）

`InteractionMatcher` 判断某条规则是否匹配当前上下文。

### 9.1 匹配维度

```text
action_type 是否匹配。
目标数量是否匹配。
目标 role 是否匹配。
目标类型是否匹配。
对象定义是否匹配。
必要特性是否存在。
禁止特性是否不存在。
对象状态是否符合。
基础关系是否符合。
```

### 9.2 输出

```text
MatchResult
```

字段建议：

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| matched | bool | 是否匹配 | 是否通过基础匹配 |
| score | Score | 匹配分 | 具体规则可高于泛化规则 |
| failed_reasons | list<string> | 失败原因 | 调试和 trace |
| matched_targets | list<ObjectId> | 匹配目标 | 命中的目标 |

### 9.3 匹配注意事项

```text
具体对象规则优先级可高于泛化特性规则。
匹配阶段不求值复杂效果。
匹配阶段不修改状态。
```

## 10. InteractionCondition（交互条件）

`InteractionCondition` 描述规则触发前必须满足的条件。

### 10.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| condition_id | string | 条件 ID | 本规则内条件 ID |
| expression_id | ExpressionId | 表达式 ID | 条件表达式 |
| required | bool | 是否必需 | false 时作为修正或提示 |
| fail_result | RuleResultType | 失败结果 | Blocked、NoEffect、Failed 等 |
| fail_reason_key | string | 失败原因 | 翻译和调试键 |

### 10.2 示例

```text
火必须 burning。
木棍不能 wet。
对象必须 quantity > 0。
行动者必须距离足够近。
族群必须已掌握某能力。
```

## 11. InteractionPriority 与冲突策略

多个规则可能同时匹配。

### 11.1 InteractionConflictPolicy

建议第一阶段作为字符串或小枚举，后续稳定后回填通用枚举。

| 值 | 中文名 | 含义 |
|---|---|---|
| stack | 叠加 | 多个规则结果都保留 |
| exclusive | 互斥 | 只保留优先级最高规则 |
| block_lower | 阻断低优先级 | 当前规则命中后阻断低优先级规则 |
| require_all | 必须全部满足 | 多规则必须都满足才成功 |
| first_match | 首个命中 | 按排序取第一个 |

### 11.2 排序建议

```text
高 priority 优先。
具体对象规则优先于泛化特性规则。
危险或阻断规则优先于收益规则。
Correction / Contradiction 规则优先于普通知识规则。
```

### 11.3 示例

腐烂红浆果：

```text
poisonous / decayed 规则优先于 edible 规则。
最终可能产生中毒反馈，并修正“红浆果总是可食用”的旧知识。
```

## 12. InteractionResolver（交互解析器）

`InteractionResolver` 执行交互规则结算。

### 12.1 输入

```text
CommandEnvelope
WorldObjectStore
ConfigRegistry
TraitResolver
ExpressionEvaluator
RandomStream
```

### 12.2 输出

```text
Result<InteractionResult>
```

### 12.3 标准流程

```text
1. 读取 ActionCommand。
2. 解析 CommandTarget。
3. 查询 WorldObject 和 ObjectDefinition。
4. 调用 TraitResolver 获取 TraitEvaluationResult。
5. 构建 InteractionContext。
6. 使用 InteractionMatcher 找到候选规则。
7. 按优先级和冲突策略排序。
8. 求值 InteractionCondition。
9. 生成 EffectResolutionRequest。
10. 生成 FeedbackCandidate。
11. 生成 KnowledgeCandidate。
12. 生成 ObjectStateChange 草稿。
13. 生成 Event 草稿。
14. 生成 InteractionTrace。
15. 返回 InteractionResult。
```

### 12.4 约束

```text
Resolver 不直接写入 WorldObjectStore。
Resolver 不直接写入 EventStore。
Resolver 不直接创建最终 KnowledgeClaim。
Resolver 只输出候选和请求。
```

## 13. InteractionResult（交互结果）

`InteractionResult` 是交互结算输出。

### 13.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| result_type | RuleResultType | 结果类型 | Success、PartialSuccess、NoEffect 等 |
| matched_rules | list<string> | 命中规则 | 匹配成功的规则 ID |
| blocked_rules | list<string> | 被阻断规则 | 被冲突策略阻断的规则 |
| effect_requests | list<EffectResolutionRequest> | 效果请求 | 后续效果系统处理 |
| object_state_changes | list<ObjectStateChange> | 对象变化草稿 | 后续状态系统应用 |
| feedback_candidates | list<FeedbackCandidate> | 反馈候选 | 后续反馈系统处理 |
| knowledge_candidates | list<KnowledgeCandidate> | 知识候选 | 后续认知系统处理 |
| event_drafts | list<EventRecord> | 事件草稿 | 后续事件系统处理 |
| trace | InteractionTrace | 解释链 | 交互过程解释 |
| errors | ErrorList | 错误 | 错误列表 |
| warnings | ErrorList | 警告 | 警告列表 |

### 13.2 结果类型说明

```text
Success：规则成功结算。
PartialSuccess：部分效果成功，部分失败或有副作用。
NoEffect：行为合法，但没有实际效果。
Blocked：条件不满足，未进入核心结算。
Failed：玩法失败，但系统状态可信。
CriticalFailure：严重玩法失败，例如中毒、受伤。
SystemError：规则或配置错误。
```

## 14. EffectResolutionRequest（效果结算请求）

`EffectResolutionRequest` 请求后续效果系统处理具体效果。

### 14.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| effect_id | EffectId | 效果 ID | 配置效果定义 |
| source_rule_id | string | 来源规则 | 哪条规则产生 |
| source_object_id | ObjectId | 来源对象 | 效果来源 |
| target_ids | list<string> | 目标 ID | 效果目标 |
| computed_value | number | 计算值 | 表达式计算结果 |
| probability | Ratio | 概率 | 触发概率 |
| evaluation_trace | EvaluationTrace | 表达式 trace | 数值来源解释 |

### 14.2 边界

```text
EffectResolutionRequest 不是 StateChange。
它只是请求“后续效果系统把这个效果结算成状态变化”。
```

## 15. FeedbackCandidate（反馈候选）

`FeedbackCandidate` 描述可能生成的反馈。

### 15.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| feedback_type | FeedbackType | 反馈类型 | 生理、知识、风险等 |
| intensity | FeedbackIntensity | 强度 | 反馈强度 |
| reliability | FeedbackReliability | 可靠性 | 反馈可靠性 |
| source | FeedbackSource | 来源 | 反馈来源 |
| message_key | string | 文案键 | 前端翻译键 |
| related_effect_id | EffectId | 相关效果 | 哪个效果导致 |
| target_ids | list<string> | 目标 | 反馈目标 |

### 15.2 边界

```text
FeedbackCandidate 不直接展示给前端。
后续反馈系统负责聚合、筛选、转换成 FeedbackGenerated Event。
```

## 16. KnowledgeCandidate（知识候选）

`KnowledgeCandidate` 是交互规则输出给认知系统的知识候选。

### 16.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| subject_definition_id | ObjectDefinitionId | 知识主体 | 通常是对象定义 ID |
| action_type | ActionType | 行为 | 吃、用、组合等 |
| effect_summary | string | 效果摘要 | restore_hunger、poison、transform 等 |
| condition_summary | string | 条件摘要 | fresh、decayed、with_fire 等 |
| evidence_type | EvidenceType | 证据类型 | 直接实验、观察、事故等 |
| confidence_delta | Delta | 置信度变化 | 对知识置信度影响 |
| source_event_hint | EventType | 事件提示 | 可能对应的事件类型 |
| object_knowledge_flag | ObjectKnowledgeFlag | 知识标记 | 可选快速标记 |

### 16.2 设计重点

知识候选必须表达：

```text
对象 + 行为 + 效果 + 条件
```

示例：

```text
red_berry + eat + restore_hunger + fresh
poison_mushroom + eat + poison + raw
dry_branch + use_with_fire + transforms_into_torch + dry
fire + use_near_wolf + repel_danger + burning
```

### 16.3 边界

```text
KnowledgeCandidate 不是最终 KnowledgeClaim。
认知系统决定是否创建 Memory、KnowledgeClaim、置信度变化和传播资格。
```

## 17. InteractionTrace（交互解释链）

`InteractionTrace` 解释一次交互如何得出结果。

### 17.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| command_id | CommandId | 命令 ID | 来源命令 |
| candidate_rules | list<string> | 候选规则 | 初步匹配规则 |
| matched_rules | list<string> | 命中规则 | 最终生效规则 |
| failed_conditions | list<string> | 失败条件 | 条件失败原因 |
| trait_traces | list<TraitExplainTrace> | 特性解释 | 特性生效过程 |
| evaluation_traces | list<EvaluationTrace> | 表达式解释 | 表达式求值过程 |
| conflict_resolution | string | 冲突处理 | 如何处理多规则 |
| generated_effects | list<EffectId> | 生成效果 | 生成哪些效果 |
| generated_knowledge | list<string> | 知识摘要 | 生成哪些知识候选 |

### 17.2 价值

```text
研发调试：为什么规则没匹配。
前端解释：为什么这次没有效果。
测试断言：哪些规则命中。
强化学习分析：状态、特性、表达式如何影响结果。
```

## 18. 交互与命令系统关系

```text
ActionCommand 是交互输入。
CommandTarget 提供目标 ID。
CommandRouter 将命令路由到 InteractionResolver。
InteractionResult 返回给规则管线。
```

交互规则不负责命令标准化和基础格式校验。

## 19. 交互与对象 / 特性系统关系

```text
WorldObjectStore 提供对象实例。
ConfigRegistry 提供 ObjectDefinition。
TraitResolver 提供 active_traits 和 TraitExplainTrace。
InteractionMatcher 基于对象、特性、状态和关系匹配规则。
```

关键边界：

```text
对象系统不决定规则结果。
特性系统不产生状态变化。
交互规则把对象和特性转化为效果请求和知识候选。
```

## 20. 交互与配置 / 表达式系统关系

```text
InteractionRuleConfig 来自配置。
EffectDefinition 来自配置。
ExpressionDefinition 来自配置。
ExpressionEvaluator 计算条件、概率和数值。
```

配置校验必须保证：

```text
规则引用的 EffectId 存在。
规则引用的 ExpressionId 存在。
规则引用的 TraitId 存在。
规则目标模式合法。
规则优先级和冲突策略合法。
```

## 21. 交互与事件 / 错误系统关系

交互结果可能产生事件草稿：

```text
ActionResolved
ActionBlocked
ObjectStateChanged
FeedbackGenerated
KnowledgeCreated
KnowledgeContradicted
RuleErrorOccurred
RandomRolled
```

错误处理：

```text
没有匹配规则：NoEffect 或 Failed，视行为定义决定。
条件不满足：Blocked。
配置引用缺失：SystemError + id_definition_not_found。
表达式非法：SystemError + validation_config_invalid。
规则输出非法：SystemError + rule_output_invalid。
```

## 22. 交互与反馈 / 认知系统关系

```text
InteractionResult 输出 FeedbackCandidate。
反馈系统将 FeedbackCandidate 转为反馈事件和玩家可见信息。
InteractionResult 输出 KnowledgeCandidate。
认知系统将 KnowledgeCandidate 转为 Memory / KnowledgeClaim。
```

关键原则：

```text
交互规则提供证据来源。
认知系统决定知识是否形成、置信度如何变化、是否可传播。
```

## 23. 对象反应规则设计

对象反应是交互规则的重要用法。

### 23.1 火 + 木棍 = 火把

规则方向：

```text
action_type = combine / use
target source = fire_source
target material = dry_branch
required_traits source = fire_source
required_traits material = combustible / fuel
condition source.runtime_flags contains burning
condition material.runtime_flags not contains wet
effect = transform material -> torch
knowledge = dry_branch + use_with_fire + transforms_into_torch + dry
```

### 23.2 生肉 + 火 = 熟肉

```text
action_type = use / combine
required_traits target = cookable
required_traits source = fire_source
effect = transform raw_meat -> cooked_meat
knowledge = raw_meat + use_with_fire + transforms_into_cooked_food
```

### 23.3 水 + 火 = 熄灭

```text
action_type = use / combine
required_traits source = water
required_traits target = fire_source
effect = remove burning / reduce heat / reduce fuel efficiency
knowledge = water + use_on_fire + extinguish_fire
```

### 23.4 设计注意事项

```text
对象反应不是 ObjectRelation 自动触发。
对象反应必须由 ActionCommand 触发，或由后续环境规则触发。
对象反应结果必须进入 ObjectStateChange、Event 和 KnowledgeCandidate。
```

| 场景 | 输入 | 命中规则 | 输出 |
|---|---|---|---|
| 吃新鲜红浆果 | eat + red_berry fresh | edible_eat_rule | restore_hunger、FeedbackCandidate、KnowledgeCandidate |
| 吃腐烂红浆果 | eat + red_berry decayed | decayed_poison_rule 优先 | poison feedback、KnowledgeContradiction candidate |

| 规则引用不存在 EffectId | 配置错误 | rule validation | SystemError / id_definition_not_found |
| 表达式返回类型错误 | expression invalid | condition eval failed | SystemError / validation_config_invalid |

测试结论：

```text
当前交互规则设计可以支撑命令到对象、特性、表达式、效果、反馈、事件和知识候选的完整链路。
交互规则不直接写状态，而是输出请求和草稿，交给后续规则管线统一应用。
对象反应可以通过配置化规则表达，不需要硬编码具体对象函数。
InteractionTrace 可以解释规则匹配、条件失败、表达式求值和知识候选来源。
```

## 26. 研发实现注意事项

```text
禁止在具体对象函数里硬编码交互结果。
禁止 InteractionResolver 直接修改 WorldObjectStore。
禁止 InteractionResolver 直接写 EventStore。
禁止 InteractionResolver 直接创建最终 KnowledgeClaim。
禁止绕过 TraitResolver 手写特性判断。
禁止表达式直接调用随机或修改状态。
新增 InteractionRuleConfig 必须补充匹配测试、条件测试、冲突测试和 trace 测试。
```
