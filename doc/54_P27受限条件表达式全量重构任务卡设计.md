# P27 受限条件表达式全量重构任务卡设计

## 1. 阶段定位

P27 是《先驱者》规则系统的一次必要架构回正。

此前总设计已经在 `05_配置系统设计.md` 中设计了受限表达式系统，并在知识、传播、反馈、文明能力、Agent、交互等设计里大量引用了：

```text
ExpressionId
ExpressionRef
EvaluationContext
ExpressionEvaluator
EvaluationTrace
condition_ids
condition_keys
KnowledgeCondition
```

但 P0-P26 的实现为了先完成最小框架，大量系统使用了临时方式：

```text
condition_key 字符串。
condition_keys 数组。
固定映射。
局部比较函数。
H5 展示层条件解释。
个别系统内部自行判断条件。
```

这些做法在早期可以接受，但从 P28 对象反应、P29 危险探索、后续 NPC 教学/谋害识别/训练环境开始，条件判断会快速变复杂。

如果继续让各系统自行判断条件，后续会出现：

```text
知识系统一套条件。
传播系统一套条件。
文明能力一套条件。
对象反应一套条件。
H5 中文显示一套条件。
训练环境又一套条件。
```

这会破坏规则一致性，也会让 AI 后续实现时不断重复造轮子。

P27 的目标不是“补一个小工具”，而是：

```text
把全项目条件判断权威入口统一回来。
```

一句话：

```text
P27 结束后，业务系统可以保留旧 condition_key 作为兼容输入，但不能再把 condition_key 字符串判断作为规则权威。
所有条件判断、条件规范化、条件中文摘要、条件 trace，都必须经过统一受限条件表达式系统。
```

---

## 2. 本阶段目标

P27 必须完成全量重构，而不是只给未来系统预留接口。

必须做到：

1. 建立独立条件表达式模块，建议命名 `pathfinder_condition`。
2. 实现受限表达式核心类型、注册表、求值器、规范化器、trace、中文摘要 key。
3. 将旧 `condition_key / condition_keys / KnowledgeCondition.expression_ids` 纳入统一规范化流程。
4. 重构知识形成、知识修正、学习闭环、传播、反馈、文明能力与 H5 条件展示入口。
5. 禁止业务系统直接用 `condition_key == "xxx"` 决定规则结果。
6. 保证旧数据和旧测试仍可运行，但旧条件必须通过 adapter 进入新系统。
7. 为 P28 对象反应提供稳定 `ConditionExpressionRef` / `ConditionEvaluationContext` / `ConditionEvaluationResult` 接口。
8. 为 P29 危险探索、后续 NPC、训练环境提供同一个条件判断入口。
9. 补充扫描/门禁测试，防止后续再次出现第二套条件系统。
10. 全量回归 P18-P26 测试，证明重构没有破坏红果/腐烂红果、知识传播、文明能力、族群冲突等已有逻辑。

---

## 3. 最重要结论

P27 不是对象反应。

P27 是对象反应之前必须补上的条件表达式基础。

调整后阶段顺序：

```text
P26：文明能力。
P27：受限条件表达式全量重构。
P28：对象反应系统。
P29：危险与探索压力。
P30：存档前置实现。
P31：真实复盘强化。
P32：协议投影扩展。
P33：H5 正式最小界面扩展。
P34：端到端 MVP。
```

P27 完成后，P28 可以直接写：

```text
ObjectReactionRule.condition_ref -> ConditionExpressionRef
ObjectReactionResolver -> ConditionExpressionEvaluator
```

而不需要重新设计条件系统。

## 4. 设计总原则

### 4.1 统一权威入口

全项目条件判断只有一个权威入口：

```cpp
ConditionExpressionEvaluator
```

业务系统只能调用它或调用它上层的专用 facade。

禁止：

```cpp
if (condition_key == "state_decayed") { ... }
if (condition_keys.contains("decayed")) { ... }
```

允许：

```cpp
auto normalized = normalizer.normalize(legacy_condition);
auto result = evaluator.evaluate(normalized.expression_ref, context);
```

### 4.2 表达式只读无副作用

表达式只能回答：

```text
条件是否成立。
数值是多少。
枚举是否匹配。
集合是否包含。
中文摘要是什么。
为什么成立或不成立。
```

表达式禁止：

```text
修改世界状态。
创建事件。
创建知识。
删除对象。
修改生命值。
修改族群关系。
写存档。
访问网络/文件/系统 API。
调用随机数。
循环。
递归。
调用任意 C++ 函数。
访问 hidden truth。
```

### 4.3 旧条件必须迁移，不是丢弃

旧结构可以保留：

```text
condition_key
condition_keys
KnowledgeCondition
expression_ids
object_state_keys
actor_requirement_keys
```

但它们必须变成：

```text
LegacyConditionInput
  -> ConditionNormalizer
  -> ConditionExpressionRef
  -> canonical_condition_key
  -> localized_summary_key
```

旧字段存在的目的：

```text
兼容旧测试。
兼容旧存档/旧 H5 对话数据。
兼容 P18-P26 已有输入。
降低一次重构风险。
```

旧字段不能再作为规则判断权威。

### 4.4 不做完整脚本语言

P27 只做受限条件表达式，不做通用脚本语言。

P27 支持：

```text
固定字段读取。
布尔组合。
枚举/字符串比较。
数值比较。
集合包含。
标签判断。
状态判断。
能力判断。
知识声明判断。
安全 trace。
中文摘要 key。
```

P27 不支持：

```text
自由字符串公式。
变量赋值。
函数定义。
循环。
递归。
随机数。
状态修改。
热更新复杂表达式。
AI 策略表达式。
对象反应效果执行。
```

### 4.5 表达式不是规则执行器

表达式只判断条件，不执行效果。

规则执行仍然属于对应系统：

```text
知识系统负责知识形成/修正。
传播系统负责传播结果。
反馈系统负责反馈选择。
文明能力系统负责能力草稿。
对象反应系统负责反应效果草稿。
危险探索系统负责危险草稿。
```

表达式系统只输出：

```text
ConditionEvaluationResult
ConditionEvaluationTrace
ConditionSummary
```

## 6. 核心类型设计

### 6.1 ConditionExpressionId

中文：条件表达式 ID。

建议复用 `foundation::ExpressionId` 或新增强类型 `ConditionExpressionId`。

如果已有 `ExpressionId` 实现稳定，可以：

```cpp
using ConditionExpressionId = pathfinder::foundation::ExpressionId;
```

如果需要区分配置表达式和条件表达式，可以新增：

```cpp
struct ConditionExpressionId : StrongId<ConditionExpressionId> {};
```

推荐 P27 初版：

```text
复用 ExpressionId，减少 ID 体系膨胀。
在 condition 模块中使用 ConditionExpressionRef 区分语义。
```

### 6.2 ConditionExpressionRef

中文：条件表达式引用。

```cpp
struct ConditionExpressionRef {
    pathfinder::foundation::ExpressionId expression_id;
    std::string inline_canonical_key;
    std::string summary_key;
    bool inline_legacy_condition = false;

    pathfinder::foundation::Result<void> validateBasic() const;
};
```

规则：

```text
expression_id 和 inline_canonical_key 至少一个有效。
summary_key 不能包含 hidden truth。
inline_legacy_condition 只能由 adapter 设置。
业务系统不得手写 inline_legacy_condition=true。
```

### 6.3 ConditionExpressionDefinition

中文：条件表达式定义。

```cpp
struct ConditionExpressionDefinition {
    pathfinder::foundation::ExpressionId expression_id;
    std::string context_type;
    ConditionExpressionNode root;
    std::string canonical_key;
    std::string summary_key;
    bool trace_enabled = true;

    pathfinder::foundation::Result<void> validateBasic() const;
};
```

字段说明：

| 字段 | 中文 | 含义 |
|---|---|---|
| expression_id | 表达式 ID | 配置或内联表达式 ID |
| context_type | 上下文类型 | knowledge、feedback、propagation、capability、reaction 等 |
| root | 根节点 | 条件表达式 AST 的受限树 |
| canonical_key | 规范条件键 | 用于比较、去重、知识合并 |
| summary_key | 摘要翻译键 | 用于 H5、trace、审核报告 |
| trace_enabled | 是否追踪 | 是否输出求值解释 |

### 6.4 ConditionExpressionNode

中文：条件表达式节点。

```cpp
enum class ConditionExpressionNodeKind {
    Unknown,
    LiteralBool,
    LiteralNumber,
    LiteralString,
    FieldRef,
    Compare,
    Contains,
    HasTag,
    HasState,
    HasCapability,
    KnowledgeHasClaim,
    And,
    Or,
    Not,
    TestOnly
};

struct ConditionExpressionNode {
    ConditionExpressionNodeKind kind = ConditionExpressionNodeKind::Unknown;
    std::string field_path;
    ConditionExpressionOperator op = ConditionExpressionOperator::Unknown;
    ConditionExpressionValue value;
    std::vector<ConditionExpressionNode> children;
    std::string summary_key;

    pathfinder::foundation::Result<void> validateBasic() const;
};
```

### 6.5 ConditionExpressionOperator

中文：条件表达式操作符。

```cpp
enum class ConditionExpressionOperator {
    Unknown,
    Eq,
    NotEq,
    Gt,
    Gte,
    Lt,
    Lte,
    In,
    Contains,
    HasTag,
    HasState,
    HasCapability,
    TestOnly
};
```

### 6.6 ConditionExpressionValue

中文：条件表达式值。

```cpp
struct ConditionExpressionValue {
    std::variant<std::monostate, bool, int64_t, double, std::string, std::vector<std::string>> value;

    bool isEmpty() const;
    std::string stableDebugString() const;
};
```

规则：

```text
禁止存储对象指针。
禁止存储可变状态引用。
禁止存储函数。
禁止存储任意 JSON blob 作为规则执行来源。
```

## 8. 求值结果设计

### 8.1 ConditionEvaluationResult

```cpp
struct ConditionEvaluationResult {
    bool ok = false;
    bool matched = false;
    ConditionExpressionValue value;
    std::string canonical_condition_key;
    std::string summary_key;
    ConditionEvaluationTrace trace;
    std::vector<pathfinder::foundation::ErrorDetail> errors;
};
```

规则：

```text
ok=false 表示表达式无法安全求值。
matched=false 表示表达式安全求值但条件不成立。
ok=false 不能被当作 matched=false 静默吞掉。
```

### 8.2 ConditionEvaluationTrace

```cpp
struct ConditionEvaluationTrace {
    pathfinder::foundation::ExpressionId expression_id;
    std::string canonical_condition_key;

    std::vector<std::string> failed_summary_keys;
    std::vector<std::string> blocked_reason_keys;

```

Trace 用于：

```text
测试。
审核。

强化学习 observation 解释。
回放差异定位。
```

```text
hidden truth。
raw state dump。
```

## 9. Registry 与 Evaluator 接口

### 9.1 ConditionExpressionRegistry

```cpp
class ConditionExpressionRegistry {
public:
    pathfinder::foundation::Result<void> registerDefinition(ConditionExpressionDefinition definition);
    pathfinder::foundation::Result<const ConditionExpressionDefinition*> find(pathfinder::foundation::ExpressionId id) const;
    pathfinder::foundation::Result<void> validateAll() const;
    bool contains(pathfinder::foundation::ExpressionId id) const;
};
```

规则：

```text
注册后只读。
重复 ID 必须失败。
无效 context_type 必须失败。
引用未知字段必须失败。
summary_key 缺失必须失败或降级为安全摘要 key。
```

### 9.2 ConditionExpressionEvaluator

```cpp
class ConditionExpressionEvaluator {
public:
    explicit ConditionExpressionEvaluator(const ConditionExpressionRegistry& registry);

    pathfinder::foundation::Result<ConditionEvaluationResult> evaluate(
        const ConditionExpressionRef& ref,
        const ConditionEvaluationContext& context) const;

    pathfinder::foundation::Result<ConditionEvaluationResult> evaluateDefinition(
        const ConditionExpressionDefinition& definition,
        const ConditionEvaluationContext& context) const;
};
```

规则：

```text
Evaluator 不持有可变世界状态。
Evaluator 不写事件。
Evaluator 不改知识。
Evaluator 不访问配置文件。
Evaluator 不访问随机数。
```

## 10. 旧条件映射规则

### 10.1 canonical_condition_key

中文：规范条件键。

它用于：

```text
知识去重。
知识冲突判断。
传播合并。
H5 展示选择。
测试断言。
```

格式建议：

```text
condition:<domain>:<field>:<op>:<value>
```

例子：

```text
condition:object_state:eq:fresh
condition:object_state:eq:decayed
condition:actor_requirement:has:tool_use
condition:region_tag:has:rainy
condition:civilization_capability:usable:SafeForaging
```

### 10.2 旧 key 映射表

P27 必须内置第一批旧 key 映射。

| 旧 key | canonical_condition_key | 中文摘要 |
|---|---|---|
| fresh | condition:object_state:eq:fresh | 新鲜状态 |
| decayed | condition:object_state:eq:decayed | 腐烂状态 |
| state_fresh | condition:object_state:eq:fresh | 新鲜状态 |
| state_decayed | condition:object_state:eq:decayed | 腐烂状态 |
| cooked | condition:object_state:eq:cooked | 熟制状态 |
| raw | condition:object_state:eq:raw | 生的状态 |
| wet | condition:object_state:eq:wet | 潮湿状态 |
| dry | condition:object_state:eq:dry | 干燥状态 |
| has_tool | condition:actor_requirement:has:tool | 拥有工具 |
| taught | condition:knowledge_source:eq:taught | 由他人传授 |
| observed | condition:knowledge_source:eq:observed | 亲自观察 |

规则：

```text
未知旧 key 不允许静默通过。
如果是测试专用 key，必须有 test_only 标记。
如果 key 可能泄露 hidden truth，必须失败。
```

### 10.3 KnowledgeCondition 规范化

旧：

```cpp
KnowledgeCondition {
  condition_key = "state_decayed";
  object_state_keys = {"decayed"};
}
```

新规范化后：

```text
canonical_condition_key = condition:object_state:eq:decayed
summary_key = condition.summary.object_state.decayed
expression_ref.inline_canonical_key = condition:object_state:eq:decayed
```

## 11. 系统回接细则

### 11.1 知识形成回接

知识形成阶段必须将候选条件规范化。

输入：

```text
feedback.condition_keys
source_claim.conditions
candidate_conditions
```

处理：

```text
normalize -> canonical_condition_key -> stable sort -> de-duplicate -> attach to KnowledgeClaim
```

要求：

```text
同一 subject/action/effect/condition 的知识强化。
同一 subject/action 不同 condition 的知识共存。
同一 subject/action/condition 不同 effect 才进入冲突/修正。
```

### 11.2 知识修正回接

KnowledgeRevision 判断冲突时，不再比较原始字符串。

必须比较：

```text
subject canonical key。
action canonical key。
condition canonical key set。
effect key。
```

规则：

```text
条件不同，不直接互相覆盖。
条件不同且 effect 不同，生成条件化知识。
条件相同且 effect 不同，才是同条件冲突。
```

### 11.3 学习闭环回接

LearningLoop 从反馈生成知识时：

```text
LearningSafeFeedbackInput.condition_keys
  -> ConditionNormalizer
  -> KnowledgeCondition.normalized fields
```

如果条件规范化失败：

```text
本次学习失败。
返回 validation_failed 或 expression_invalid。
不能生成无条件知识。
```

### 11.4 传播系统回接

传播系统传播的是条件化知识。

要求：

```text
source_claim.conditions 必须先规范化。
recipient_claim.conditions 使用规范化结果。
传播上下文 condition_keys 只能作为输入，不能直接用于判断。
误解产生的新条件也必须通过 normalizer。
```

### 11.5 反馈系统回接

FeedbackResolver 条件匹配必须走 Evaluator。

要求：

```text
FeedbackRule.condition_ids -> ConditionExpressionRef。
FeedbackRule legacy condition_keys -> Normalizer -> ConditionExpressionRef。
反馈选择 trace 必须包含 ConditionEvaluationTrace 引用。
```

### 11.6 文明能力回接

CivilizationCapability 条件必须走 Evaluator。

要求：

```text
CapabilityUnlockCondition 只保存 ConditionExpressionRef 或 normalized condition set。
CapabilityAvailabilityCondition 走 Evaluator。
CapabilityDegradationCondition 走 Evaluator。
CivilizationTrace 记录 condition trace summary。
```

禁止：

```text
if capability_condition_key == "safe_foraging_ready"。
if stage == Foraging then unlock。
if P25 outcome == Win then unlock。
```

### 11.7 H5 中文摘要回接

H5 不负责理解条件。

H5 只能接收：

```text
localized_summary_key
safe_summary_text
condition_display_label
```

H5 禁止：

```text
自己把 decayed 翻译成腐烂并参与规则判断。
自己用 condition_key 拼接知识唯一键。
显示裸 condition_key 给玩家。
```

## 18. 与 P28 的接口承诺

P28 对象反应系统不得重新设计条件系统。

P28 必须直接使用：

```text
ConditionExpressionRef
ConditionEvaluationContext
ConditionExpressionEvaluator
ConditionEvaluationResult

```

P28 的 `ObjectReactionRule` 只能引用：

```text
condition_ref: ConditionExpressionRef
```

P28 不能出现：

```text

reaction.forbidden_states 自己判断。
reaction.condition_key 自己比较。
```

如果为了配置易读需要 `required_tags` 等简写，也必须在加载/校验阶段转换为 `ConditionExpressionRef`。
