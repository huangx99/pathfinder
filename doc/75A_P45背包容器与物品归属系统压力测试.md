# P45 背包、容器、掉落物与物品归属系统压力测试

测试日期：2026-05-20  
对象文档：`doc/75_P45背包容器与物品归属系统设计.md` 初版

## 1. 压力测试结论

初版方向正确，但需要补强 8 类设计点，否则后续会在 P48 制作、P51 Agent、P53 H5 上出现边界不清。

必须回填到设计文档的修改：

```text
1. 增加 Transaction / Draft 事务模型，保证物品转移原子性。
2. 明确 SourceInventoryProof，防止“默认共享营地库存”。
3. 明确 ItemStack 与 ItemInstance 的权威关系，避免堆叠和唯一物品混乱。
4. 明确地面掉落物 GroundStack 策略，避免大量同 key 地面实体爆炸。
5. 明确可见性与已发现限制，防止拾取未知格子物品。
6. 明确 Reservation / Lock 预留机制，给多 Agent 抢同一物品留接口。
7. 明确 InventoryPatch entries 结构化方案，不要长期使用字符串协议。
8. 明确 P45 不做装备效果，但 Equipped 归属必须能表达。
```

最终判断：

```text
P45 可以继续作为独立阶段推进，但必须把以上补强写回主设计文档后再交给研发。
```

## 2. 压力测试方法

从以下维度冲击初版设计：

```text
真实开放世界物品流转。
玩家/NPC/野兽多主体。
采集、制作、掉落、容器、营地库存。
并发/复盘/前端投影。
未来魔法、高级文明、机械危机等扩展。
```

评判标准：

```text
是否会产生凭空物品。
是否会产生一物多处。
是否会让前端推规则。
是否会让 NPC 绕过知识和库存。
是否能复盘。
是否能给后续 P48/P51/P53 直接使用。
```

## 3. 压力场景

### 3.1 玩家拾取地面木头

流程：

```text
木头 entity 在 surface:1:0。
玩家在 surface:0:0。
玩家执行 Pickup。
```

期望：

```text
距离允许。
木头从 origin/target cell entity_ids 移除。
木头 location_kind = InInventory。
木头 coord 清空。
玩家 inventory 增加木头。
changed_cells / changed_entities / changed_inventories 同时返回。
```

初版覆盖：基本覆盖。

补强：必须说明 Pickup 可以同格或相邻，且可见/已发现限制由策略控制。

### 3.2 玩家尝试拾取远处未知格子物品

流程：

```text
玩家知道 entity_id，但物品在未探索格子。
前端或恶意客户端发 Pickup。
```

风险：

```text
如果只按 entity_id 查找，会远程拾取未知物。
```

结论：初版不足。

需要回填：

```text
Pickup 必须校验 actor 当前坐标、目标实体 coord、可见性/交互范围。
不可见或未发现时返回 target_not_visible 或 target_too_far。
```

### 3.3 玩家背包木头和营地库存木头不能默认共享

流程：

```text
玩家背包 1 木头。
营地库存 10 木头。
玩家制作火把。
```

风险：

```text
如果 P48 查询所有库存，会默认偷用营地库存。
```

结论：初版提到 InventoryScope，但不够硬。

需要回填：

```text
SourceInventoryProof：每次材料消耗必须记录来源 inventory_id、owner、quantity。
默认 scope = actor_self。
camp/tribe/container 需要显式授权。
```

### 3.4 同伴没有火把，不能凭空使用火把

流程：

```text
同伴知道火把驱狼。
同伴背包没有火把。
玩家背包有火把。
营地没有火把。
狼来了。
```

期望：

```text
同伴不能直接 use torch。
同伴可以产生目标：获取火把。
P51/P52 再决定向玩家请求、制作或寻找。
```

结论：初版方向正确，但需要回填“使用动作必须校验 inventory ownership”。

### 3.5 NPC 自己制作火把需要链式查询

流程：

```text
NPC 有知识：火把需要木头 + 火源。
NPC 无火把。
NPC 背包无木头。
附近地上有木头。
营地有干草。
```

P45 只需支持：

```text
query owner inventory。
query nearby OnMap item。
query authorized camp inventory。
```

结论：初版有 queryItems，但缺少 nearby OnMap 查询。

需要回填：

```text
InventoryQueryService 增加 queryNearbyMapItems(actor_key, entity_key, radius)。
```

### 3.6 地上 100 个红果是否创建 100 个实体

流程：

```text
红果灌木掉落 100 个红果。
```

风险：

```text
如果每个红果都是 WorldEntityInstance，地图实体爆炸。
如果只用 stack，又无法表达腐烂程度不同。
```

结论：初版不足。

需要回填：

```text
GroundStack 策略：同 key、同状态、stackable=true 的地面物品可以合并为一个 stack entity。
状态不同或 stackable=false 必须独立 instance。
```

### 3.7 腐烂红果与普通红果不能合并

流程：

```text
玩家有 2 个普通红果。
地上有 1 个腐烂红果。
拾取后不能变成 3 个普通红果。
```

结论：初版提到状态不同不能合并，但要更明确 stack_key。

需要回填：

```text
StackKey = entity_key + normalized_state_signature。
```

### 3.8 斧头耐久不同不能合并

流程：

```text
玩家有耐久 80 的斧头。
地上有耐久 20 的斧头。
```

期望：

```text
两个独立 InventoryItemEntry。
stackable=false。
quantity=1。
```

初版覆盖：基本覆盖。

补强：测试必须覆盖 stackable=false 不合并。

### 3.9 玩家丢弃部分数量

流程：

```text
玩家背包 5 红果。
丢弃 2 红果。
```

期望：

```text
背包剩 3。
地上出现 2。
如果地上已有同 stack_key 红果，则合并。
```

初版覆盖：不足。

需要回填：

```text
Drop 支持 partial stack split。
TransferResult 必须记录 source_entry_update 和 created/destination_entry_update。
```

### 3.10 容器关闭或上锁

流程：

```text
箱子在玩家旁边。
箱子是 closed/locked。
玩家 Pickup from container。
```

P45 不做锁机制，但必须预留：

```text
ContainerClosed
ForbiddenByOwner
```

初版已有 failure kind，但要说明 P45 最小策略。

### 3.11 地上容器本身被捡起

流程：

```text
地上有箱子，箱子里有石头。
玩家捡起箱子。
```

复杂点：

```text
容器实体进入背包时，容器 inventory 是否跟随？
容器里的物品是否仍在 container inventory？
```

P45 可以不实现嵌套容器，但必须禁止或明确。

需要回填：

```text
P45 禁止拾取非空容器，返回 container_not_empty。
后续阶段再支持嵌套容器。
```

### 3.12 物品被消耗成 Nowhere

流程：

```text
吃红果。
红果从背包消失。
```

P45 不做 Eat，但 P48/P后续会用。

需要回填：

```text
consumeToNowhere 是内部 transfer kind，但 P45 不暴露 command。
必须更新 item_locations_by_entity_id。
```

### 3.13 Agent 并发抢同一物品

流程：

```text
两个 NPC 都计划拾取同一木头。
NPC A 先执行成功。
NPC B 后执行。
```

期望：

```text
NPC B 返回 item_not_available，不崩溃，不复制物品。
```

初版提到原子化，但缺少 Reservation。

需要回填：

```text
P45 最小实现串行 command 原子化。
预留 ReservationToken，P51 再用。
```

### 3.14 前端重复点击拾取

流程：

```text
手机端网络卡顿，玩家连续点击拾取。
```

期望：

```text
第一次成功。
第二次 item_not_on_map 或 item_not_available。
不能重复加背包。
```

需要回填：

```text
Transfer 必须先校验 source location 与 request.from 一致。
```

### 3.15 掉落到阻挡格

流程：

```text
玩家站在水边，指定把石头丢到水里。
```

P45 策略必须明确：

```text
默认只能 drop 到 actor 当前格或可放置相邻格。
cell 是否允许 drop 由 can_place_item 或 blocks_item_drop 控制。
P45 可先允许当前格，不允许远程指定。
```

### 3.16 物品在装备栏

流程：

```text
玩家装备斧头。
制作或丢弃时是否能消耗装备斧头？
```

P45 不做装备效果，但必须表达 Equipped。

需要回填：

```text
Equipped 是 location_kind。
P45 不实现 EquipCommand，但 TransferKind 预留。
默认材料搜索不包含 Equipped，除非显式 scope。
```

### 3.17 死亡掉落

流程：

```text
NPC 死亡，背包物品掉落到所在格。
```

P45 不做死亡，但 transfer API 应支持批量 drop。

需要回填：

```text
transferBatch / dropAllToMap 预留到 P52/P后续。
P45 单次 transfer 结构不要阻止批量扩展。
```

### 3.18 魔法世界：空间袋

流程：

```text
魔法袋容量比普通背包大，可能跨空间。
```

需要：

```text
capacity_slots 可以被 content/config 覆盖。
InventoryOwnerKind::Container 足以表达空间袋。
```

初版可覆盖。

### 3.19 高级文明：载具货舱

流程：

```text
飞船货舱有燃料和机器人零件。
```

需要：

```text
InventoryOwnerKind::Vehicle。
Inventory 不依赖二维坐标，但 owner entity 可以在地图上。
```

初版可覆盖。

### 3.20 机械危机：机器人部件既是装备又是身体

流程：

```text
机器人安装电池，电池是 Equipped 还是 InInventory？
```

需要：

```text
Equipped 预留即可。
P45 不处理装备效果。
```

初版可覆盖。

## 4. 压力测试后必须修改主设计的内容

### 4.1 新增 Transaction/Draft 模型

原因：防止物品转移中途失败导致一物多处或物品消失。

主设计需新增：

```text
InventoryTransferDraft
validate -> reserve -> apply -> emit patch
```

### 4.2 新增 SourceInventoryProof

原因：防止制作和 Agent 使用物品时默认共享所有库存。

主设计需新增：

```text
SourceInventoryProof { inventory_id, owner_kind, owner_key, entity_id/entity_key, quantity }
```

### 4.3 新增 StackKey

原因：普通红果、腐烂红果、不同耐久斧头不能错误合并。

主设计需新增：

```text
StackKey = entity_key + normalized_state_signature
```

### 4.4 新增 GroundStack 策略

原因：地面大量同类物品需要合并，但状态不同不能合并。

### 4.5 新增可见性/交互范围硬门禁

原因：防止恶意 command 远程拾取未知物品。

### 4.6 新增 ReservationToken 预留

原因：后续多 Agent 同时规划同一物品时需要占用/失效机制。

P45 最小只需要串行二次校验，不实现长期 reservation。

### 4.7 改造 InventoryPatch entries

原因：字符串 entries 不利于前端和 AI 稳定解析。

建议 P45 设计 DTO：

```text
InventoryEntryPatchDto
```

如果暂时不能改 P43 DTO，就在 fields 中使用 entries_json，但设计文档必须要求 JSON 结构字段稳定。

### 4.8 明确嵌套容器策略

P45 不支持拾取非空容器，避免箱中箱无限复杂。

## 5. 压力测试结论

初版设计方向正确，但必须回填以上修改后才适合交给研发。

回填后，P45 可以作为长期物品归属基础，而不是只服务拾取/丢弃 demo。
