# 先驱者：特性系统设计

## 1. 本文档目标

本文档用于设计《先驱者》规则引擎中的特性系统。

特性系统用于描述对象为什么有用、为什么危险、为什么能反应、为什么会产生某些反馈和知识。

一句话：

```text
对象是被探索的主体。
特性是对象产生用途、风险、反应和知识的原因。
效果是规则结算后真正发生的变化。
知识是主体对对象、行为、效果和条件的认知。
```

特性系统必须兼容：

```text
01 基础类型与 ID。
02 通用枚举。
05 配置系统与受限表达式。
06 世界对象系统。
后续交互规则、反馈系统、认知状态、记忆系统、知识系统、传播系统。
```

## 2. 设计边界

### 2.1 本文档要解决的问题

```text
定义 TraitDefinition。
定义特性如何和效果绑定。
定义特性何时生效。
定义特性如何被发现。
定义特性的风险信息。
定义 TraitResolver 如何根据对象和上下文解析有效特性。
定义特性与对象、配置、表达式、交互、认知、事件和反馈的关系。
```

### 2.2 本文档暂不解决的问题

```text
不设计完整知识图谱。
不设计完整认知状态。
不设计完整交互规则结算。
不设计完整状态效果系统。
不设计完整数值平衡。
不设计复杂 TraitInstance 系统。
```

这些内容会在后续文档中展开。

## 3. 核心原则

### 3.1 特性不是效果

```text
edible 是特性。
restore_hunger 是效果。
poisonous 是特性。
apply_poison 是效果。
combustible 是特性。
burn / transform / increase_fire_fuel 是效果。
```

特性描述“对象具备什么性质”。

效果描述“规则结算后状态如何变化”。

### 3.2 真实特性不等于已知特性

对象可以真实拥有某个特性，但玩家或族群不知道。

```text
poison_mushroom 真实拥有 poisonous。
玩家第一次见到它时不知道 poisonous。
玩家吃了以后中毒，才可能形成 KnowledgeClaim。
```

特性系统只负责真实规则层面的特性解析。

玩家知道什么由认知系统、记忆系统和知识系统表达。

### 3.3 第一阶段不做复杂 TraitInstance

第一阶段不单独设计复杂 `TraitInstance`。

原因：

```text
TraitDefinition 已能表达静态特性。
WorldObject.runtime_state 已能表达运行时状态。
ExpressionDefinition 已能表达条件化生效。
```

因此第一阶段采用：

```text
TraitDefinition + WorldObject.runtime_state + EvaluationContext + ExpressionEvaluator
```

共同决定某个特性是否在当前上下文中生效。

后续如果出现“特性本身需要独立成长、衰减、堆叠、存档”的需求，再引入 TraitInstance，不推翻当前架构。

### 3.4 特性必须可组合

对象可以拥有多个特性。

例如火把：

```text
combustible
fire_source
light_source
repels_predator
consumes_fuel
portable
```

规则不应死绑具体对象，而应尽量基于特性组合匹配。

### 3.5 特性必须可解释

特性是否生效、为什么生效、为什么不生效，必须能输出解释。

这对调试、前端提示、测试和强化学习都重要。

## 4. 特性系统总架构

第一阶段特性系统包含：

| 模块 | 中文名 | 作用 |
|---|---|---|
| TraitDefinition | 特性定义 | 配置层定义某种特性 |
| TraitEffectBinding | 特性效果绑定 | 描述特性可能触发哪些效果 |
| TraitActivationRule | 特性生效规则 | 描述特性在什么条件下生效 |
| TraitDiscoveryRule | 特性发现规则 | 描述特性如何被发现 |
| TraitRiskProfile | 特性风险信息 | 描述特性带来的风险 |
| TraitContext | 特性上下文 | 求值时需要的对象、行为、环境、知识信息 |
| TraitResolver | 特性解析器 | 判断哪些特性在当前上下文中生效 |
| TraitEvaluationResult | 特性求值结果 | 输出生效特性、未生效原因和 trace |
| TraitExplainTrace | 特性解释链 | 解释特性判断过程 |

## 5. TraitDefinition（特性定义）

`TraitDefinition` 来自配置系统。

它描述一种特性本身。

### 5.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| trait_id | TraitId | 特性 ID | 是 | 特性唯一标识 |
| category | TraitCategory | 特性类别 | 是 | 食物、危险、材料等 |
| discovery_methods | list<TraitDiscoveryMethod> | 发现方式 | 是 | 吃、触碰、观察等 |
| effect_domains | list<TraitEffectDomain> | 影响领域 | 是 | 饥饿、健康、温度等 |
| default_certainty | TraitCertaintyLevel | 默认确信度 | 是 | 初始确信水平 |
| activation_rule_ids | list<string> | 生效规则 | 否 | TraitActivationRule 本地 ID |
| effect_bindings | list<TraitEffectBinding> | 效果绑定 | 否 | 可能触发的效果 |
| discovery_rules | list<TraitDiscoveryRule> | 发现规则 | 否 | 如何被认知发现 |
| risk_profile | TraitRiskProfile | 风险信息 | 否 | 风险描述 |
| tags | list<string> | 标签 | 否 | 查询和组合用标签 |
| display_name_key | string | 显示名键 | 是 | 翻译键 |

### 5.2 示例

```json
{
  "trait_id": "trait_combustible",
  "category": "material",
  "discovery_methods": ["use", "accident", "observe"],
  "effect_domains": ["fire", "temperature"],
  "default_certainty": "unknown",
  "activation_rule_ids": ["active_when_not_wet"],
  "effect_bindings": [
    {
      "action_type": "use",
      "effect_id": "effect_burn_or_transform",
      "condition_expression_id": "expr_can_burn_target"
    }
  ],
  "tags": ["fire_reaction", "material"],
  "display_name_key": "trait.combustible.name"
}
```

### 5.3 设计注意事项

```text
TraitDefinition 不保存玩家是否知道该特性。
TraitDefinition 不直接修改状态。
TraitDefinition 可以引用 EffectId 和 ExpressionId。
TraitDefinition 可以被多个 ObjectDefinition 复用。
```

## 6. TraitEffectBinding（特性效果绑定）

`TraitEffectBinding` 描述某个特性在某种行为或上下文下可能触发什么效果。

### 6.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| action_type | ActionType | 行为类型 | 否 | 吃、用、触碰、组合等 |
| effect_id | EffectId | 效果 ID | 是 | 关联效果定义 |
| condition_expression_id | ExpressionId | 条件表达式 | 否 | 效果触发条件 |
| probability_expression_id | ExpressionId | 概率表达式 | 否 | 效果概率 |
| priority | Count | 优先级 | 否 | 多效果排序 |
| tags | list<string> | 标签 | 否 | 分类和调试 |

### 6.2 使用示例

```text
edible + eat -> restore_hunger
poisonous + eat -> apply_poison
combustible + use_with_fire -> burn_or_transform
healing + eat -> restore_health
repels_predator + use_near_predator -> reduce_danger
```

### 6.3 边界

```text
TraitEffectBinding 只声明“可能触发什么效果”。
真正是否触发，由交互规则和表达式求值决定。
真正状态变化，由 Effect / StateChange 执行。
```

## 7. TraitActivationRule（特性生效规则）

`TraitActivationRule` 描述特性在什么条件下生效。

### 7.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| rule_id | string | 规则 ID | 是 | 特性内本地规则 ID |
| expression_id | ExpressionId | 表达式 ID | 是 | 生效条件表达式 |
| fail_reason_key | string | 失败原因键 | 否 | 不生效原因翻译键 |
| required | bool | 是否必需 | 是 | false 时可作为修正项 |

### 7.2 示例

```json
{
  "rule_id": "active_when_not_wet",
  "expression_id": "expr_target_not_wet",
  "fail_reason_key": "trait.combustible.inactive.wet",
  "required": true
}
```

### 7.3 使用示例

```text
fresh 时 edible 生效。
decayed 时 poisonous 生效。
wet 时 combustible 不生效或效率降低。
cooked 后 poisonous 强度降低。
night 时 luminous 生效。
```

## 8. TraitDiscoveryRule（特性发现规则）

`TraitDiscoveryRule` 描述特性如何被发现。

它不直接创建知识，但为后续认知系统提供规则依据。

### 8.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| method | TraitDiscoveryMethod | 发现方式 | 是 | 吃、触碰、观察、事故等 |
| action_type | ActionType | 行为类型 | 否 | 对应行为 |
| evidence_type | EvidenceType | 证据类型 | 是 | 直接实验、观察、事故等 |
| confidence_delta | Delta | 置信度变化 | 否 | 对知识置信度影响 |
| knowledge_flag | ObjectKnowledgeFlag | 知识标记 | 否 | 可添加的对象知识标记 |
| creates_warning | bool | 是否创建警告 | 否 | 是否形成风险记忆 |

### 8.2 示例

```text
吃红浆果恢复饥饿：Eat / DirectTest / KnowsNutritionEffect
吃毒蘑菇中毒：Eat / Accident / KnowsPoisonousWhenEaten
触碰火焰受伤：Touch / DirectTest / KnowsContactRisk
观察狼避开火：Observe / Observation / KnowsRepelsDanger
```

### 8.3 边界

```text
TraitDiscoveryRule 不保存 Memory。
TraitDiscoveryRule 不保存 KnowledgeClaim。
它只说明“这种结果应该如何成为证据”。
```

## 9. TraitRiskProfile（特性风险信息）

`TraitRiskProfile` 描述特性带来的风险。

### 9.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| base_risk_level | ActionRiskLevel | 基础风险 | 是 | 风险等级 |
| risk_domains | list<TraitEffectDomain> | 风险领域 | 是 | 健康、中毒、伤害等 |
| risk_feedback_type | FeedbackType | 风险反馈 | 否 | 反馈类型 |
| warning_flag | ObjectKnowledgeFlag | 警告标记 | 否 | 形成的知识标记 |
| mitigation_expression_ids | list<ExpressionId> | 缓解表达式 | 否 | 烹饪、工具、能力降低风险 |

### 9.2 示例

```text
poisonous：High / Poison / KnowsPoisonousWhenEaten
combustible：Medium / Fire / KnowsContactRisk
sharp：Low / Injury / KnowsUseRisk
repelled_by_fire：Low / Risk / KnowsRepelsDanger
```

## 10. TraitContext（特性上下文）

`TraitContext` 是 TraitResolver 判断特性时使用的只读上下文。

### 10.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| object_id | ObjectId | 对象实例 | 当前对象 |
| definition_id | ObjectDefinitionId | 对象定义 | 对象类型 |
| action_type | ActionType | 行为类型 | 当前行为 |
| actor_id | EntityId | 行动者 | 执行动作者 |
| region_id | RegionId | 区域 | 当前区域 |
| object_state | ObjectState | 对象状态 | 当前状态 |
| runtime_flags | list<string> | 运行时标记 | wet、burning、rotten 等 |
| trait_ids | list<TraitId> | 配置特性 | 对象定义上的特性 |
| expression_context | EvaluationContext | 表达式上下文 | 供表达式读取 |

### 10.2 设计注意事项

```text
TraitContext 是只读快照。
TraitContext 不允许修改 WorldObject。
TraitContext 必须来自同一个 StateVersion。
```

## 11. TraitResolver（特性解析器）

`TraitResolver` 判断某个对象在某个上下文中哪些特性生效。

### 11.1 输入

```text
WorldObject
ObjectDefinition
ActionType
TraitContext
ConfigRegistry
ExpressionEvaluator
```

### 11.2 输出

```text
Result<TraitEvaluationResult>
```

### 11.3 解析流程

```text
1. 从 WorldObject.definition_id 获取 ObjectDefinition。
2. 读取 ObjectDefinition.trait_ids。
3. 从 ConfigRegistry 获取 TraitDefinition。
4. 构建 TraitContext。
5. 对每个 TraitActivationRule 求值。
6. 过滤未生效特性。
7. 收集 TraitEffectBinding。
8. 收集 TraitRiskProfile。
9. 生成 TraitExplainTrace。
10. 返回 TraitEvaluationResult。
```

### 11.4 错误处理

```text
trait_id 不存在：id_definition_not_found。
表达式不存在：id_definition_not_found。
表达式结果类型错误：validation_config_invalid。
表达式求值失败：rule_output_invalid 或 validation_failed。
```

## 12. TraitEvaluationResult（特性求值结果）

`TraitEvaluationResult` 是特性解析输出。

### 12.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| active_traits | list<TraitId> | 生效特性 | 当前上下文生效的特性 |
| inactive_traits | list<TraitId> | 未生效特性 | 当前上下文未生效的特性 |
| effect_bindings | list<TraitEffectBinding> | 效果绑定 | 可供交互规则使用的效果绑定 |
| risk_profiles | list<TraitRiskProfile> | 风险信息 | 当前上下文相关风险 |
| discovery_rules | list<TraitDiscoveryRule> | 发现规则 | 后续认知可使用的发现依据 |
| traces | list<TraitExplainTrace> | 解释链 | 特性解析过程 |
| warnings | ErrorList | 警告 | 非阻断问题 |

### 12.2 使用示例

```text
红浆果 fresh + eat：active_traits = [edible]
红浆果 decayed + eat：active_traits = [poisonous], inactive_traits = [edible]
湿树枝 + fire：active_traits = [combustible], 但燃烧效率 modifier 降低
```

## 13. TraitExplainTrace（特性解释链）

`TraitExplainTrace` 用于解释特性为什么生效或不生效。

### 13.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| trait_id | TraitId | 特性 ID | 被判断的特性 |
| checked_rules | list<string> | 检查规则 | 检查过哪些生效规则 |
| matched_rules | list<string> | 命中规则 | 哪些规则成立 |
| failed_rules | list<string> | 失败规则 | 哪些规则失败 |
| evaluation_traces | list<EvaluationTrace> | 表达式解释 | 表达式求值 trace |
| final_active | bool | 最终是否生效 | 结果 |
| reason_key | string | 原因键 | 翻译和调试用 |

### 13.2 价值

```text
研发可调试为什么某特性没生效。
前端可解释为什么湿木棍点不着。
测试可断言表达式命中链。
强化学习可分析环境条件对结果的影响。
```

## 14. 特性与对象系统关系

```text
ObjectDefinition.trait_ids 声明对象具备哪些潜在特性。
WorldObject.runtime_state 影响特性是否生效。
ObjectContextProvider 提供表达式上下文。
TraitResolver 结合配置和运行时状态得出 active_traits。
```

关键边界：

```text
对象系统不解释特性。
特性系统不修改对象。
交互规则使用特性系统结果进行结算。
```

## 15. 特性与配置系统关系

```text
TraitDefinition 来自配置。
TraitEffectBinding 可以引用 EffectId。
TraitActivationRule 可以引用 ExpressionId。
TraitDiscoveryRule 可以引用枚举和知识标记。
TraitRiskProfile 可以引用风险和反馈枚举。
```

配置校验必须检查：

```text
TraitId 是否重复。
EffectId 是否存在。
ExpressionId 是否存在。
枚举字符串是否合法。
表达式 result_type 是否匹配使用位置。
```

## 16. 特性与表达式系统关系

特性通过表达式实现条件化。

表达式可用于：

```text
判断特性是否生效。
修正效果强度。
修正风险概率。
判断发现条件。
判断缓解条件。
```

示例：

```text
expr_target_not_wet：湿树枝不易燃。
expr_target_is_decayed：腐烂浆果触发毒性。
expr_actor_has_cooking：烹饪能力降低毒性。
expr_region_is_rainy：雨季火焰效率下降。
```

约束：

```text
表达式只读上下文。
表达式不修改状态。
表达式不创建事件。
表达式不调用随机数。
```

## 17. 特性与交互规则关系

交互规则可以基于特性匹配对象。

例如：

```text
required_trait_ids = [combustible]
action_type = use_with_fire
result = burn_or_transform
```

比直接写死对象更可扩展：

```text
dry_branch 可燃。
dry_grass 可燃。
cloth 可燃。
torch 可燃且可携带火源。
```

交互规则使用 TraitEvaluationResult：

```text
active_traits 决定能否匹配。
effect_bindings 提供可能效果。
risk_profiles 提供风险信息。
discovery_rules 提供后续认知依据。
```

## 18. 特性与认知 / 知识关系

真实特性不等于已知特性。

```text
TraitDefinition：对象真实具备什么潜在特性。
TraitEvaluationResult：当前上下文哪些特性生效。
KnowledgeClaim：玩家或族群知道什么对象 + 行为 + 效果 + 条件。
```

示例：

```text
真实特性：red_berry has edible。
行为结果：eat red_berry -> restore_hunger。
知识断言：red_berry + eat + restore_hunger。
```

特性可以帮助生成知识，但知识不是直接复制 TraitId。

原因：

```text
玩家理解的是“吃红浆果会恢复饥饿”，不是“红浆果有 trait_edible_fresh 这个内部标签”。
```

## 19. 特性与事件 / 反馈关系

特性影响行为结果，行为结果产生反馈和事件。

可能事件：

```text
TraitDiscovered
FeedbackGenerated
KnowledgeCreated
KnowledgeUpdated
KnowledgeContradicted
RuleErrorOccurred
```

示例：

```text
poisonous 生效 -> FeedbackGenerated(中毒) -> KnowledgeCreated(poison_mushroom + eat + poison)
combustible 生效 -> ObjectStateChanged(燃烧) -> KnowledgeCreated(dry_branch + use_with_fire + burns)
```

## 20. 特性与强化学习关系

特性系统可以为强化学习提供更稳定的抽象。

AI 不应只记住对象 ID，而应逐渐学习特性规律。

例如：

```text
不是只学 red_berry 可以吃。
而是学 fresh + edible + low_risk 的对象适合尝试食用。
```

可用于 RLObservationProjection 的信息：

```text
已知特性摘要。
潜在风险摘要。
可尝试行为。
特性生效 trace。
对象状态对特性的影响。
```

注意：

```text
AI 观察仍应受认知系统约束。
AI 不应直接看到玩家未知的真实 TraitDefinition，除非处于调试或全知训练模式。
```

## 22. 组合测试样例

| 场景 | 输入 | 特性判断 | 预期结果 |
|---|---|---|---|
| 吃新鲜红浆果 | red_berry + eat + fresh | trait_edible_fresh 生效 | 产生 restore_hunger 效果和 KnowledgeClaim |
| 吃腐烂红浆果 | red_berry + eat + decayed | edible 不生效，poisonous 生效 | 产生风险反馈和知识修正 |
| 吃毒蘑菇 | poison_mushroom + eat | trait_poisonous 生效 | 产生中毒反馈和警告记忆 |
| 湿树枝遇火 | dry_branch + fire + wet | combustible 生效但效率降低或被阻断 | 输出 trace 说明 wet 影响 |
| 干树枝遇火 | dry_branch + fire + dry | combustible / fuel 生效 | 火 fuel 增加或树枝燃烧 |
| 火加木棍做火把 | fire_source + dry_branch + combine | combustible + fuel + 条件满足 | dry_branch transform -> torch |
| 狼遇火 | wolf + near fire_source | repels_predator 生效 | 狼远离或危险降低 |
| 烹饪毒蘑菇 | poison_mushroom + cook | cookable 生效，poisonous 风险降低 | EvaluationTrace 记录 cooking mitigation |
| 观察发光蘑菇 | mushroom + night + observe | luminous 夜间生效 | 产生观察证据 |
| 洞穴作为庇护 | cave + rest | shelter 生效 | 环境风险降低 |

测试结论：

```text
当前特性系统可以支持对象用途、风险、条件化生效、对象反应、反馈生成、知识形成和表达式解释。
TraitDefinition 与 WorldObject 分离，真实特性与已知知识分离。
TraitResolver 可以作为对象和交互规则之间的稳定中间层。
后续新增对象时优先通过组合 TraitDefinition 扩展，而不是硬编码具体对象规则。
```

## 23. 研发实现注意事项

```text
禁止把 TraitId 直接当作玩家可见知识。
禁止 TraitDefinition 直接修改 WorldObject。
禁止 TraitResolver 产生 StateChange。
禁止在交互规则中绕过 TraitResolver 重复实现特性判断。
禁止表达式读取非白名单字段。
新增 TraitDefinition 必须补充配置校验、翻译和测试。
```
