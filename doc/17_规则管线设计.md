# 17_规则管线设计

## 1. 设计目标

规则管线系统负责把一次命令从“玩家意图”稳定、可复盘、可测试地推进到“状态提交”和“事件输出”。

前面已经设计了命令、交互、反馈、认知、记忆、知识、传播、族群状态、文明能力等系统。如果没有统一管线，各系统会出现以下问题：

```text
执行顺序不稳定。
模块直接改状态。
事件重复或遗漏。
错误发生后状态半提交。
随机结果无法复盘。
前端、测试、AI 看到的结果不一致。
后续新增系统需要改动大量旧代码。
```

本章目标：

```text
定义规则执行顺序。
定义每个阶段的输入输出契约。
定义所有草稿如何收集、排序、合并、校验和提交。
定义状态变更闸门。
定义事件提交闸门。
定义确定性执行规范。
定义错误处理和事务边界。
定义后续扩展方式，保证新增阶段不推翻整体架构。
```

核心结论：

```text
规则管线不是函数调用列表。
规则管线是确定性事务流程。
```

标准理念：

```text
各系统只解析、只输出草稿、只解释原因。
管线统一收集草稿、统一校验、统一应用、统一发事件、统一持久化。
```

## 2. 不解决的问题

本章不解决：

| 不解决的问题 | 原因 | 对应系统 |
|---|---|---|
| 具体行为规则 | 已由 08、09、16 等系统负责 | 行为 / 交互 / 文明 |
| 具体战斗伤害 | 管线只负责阶段和提交 | 后续战斗系统 |
| 前端协议字段细节 | 管线只输出结果结构 | 20_前后端协议设计.md |
| 存档文件格式 | 管线只定义提交点 | 19_存档与复盘设计.md |
| 性能优化细节 | 第一阶段先保证正确性 | 后续实现阶段 |
| AI 策略算法 | 管线只提供可观察 trace | 强化学习系统 |
| 配置编辑器 UI | 管线只消费配置 | 工具链 |

禁止做法：

```text
禁止阶段直接写 GameState。
禁止阶段直接写 EventStore。
禁止阶段自己提交事务。
禁止阶段自己生成最终 StateVersion。
禁止阶段使用真实系统时间。
禁止阶段使用未记录随机。
禁止依赖 map 无序遍历结果。
禁止前端根据局部草稿自行推导最终状态。
禁止新增系统时修改大量旧阶段代码。
```

## 3. 核心概念

### 3.1 管线是事务，不是流水账

一次命令执行要么完整合法提交，要么失败后保持状态一致。

正确模型：

```text
PipelineTransaction
  input_snapshot
  stage_outputs
  draft_sets
  validation_report
  commit_plan
  final_result
```

错误模型：

```text
InteractionResolver 直接改对象。
KnowledgeResolver 直接写知识库。
CivilizationResolver 直接发 CapabilityUnlocked。
```

### 3.2 管线阶段必须有契约

每个阶段必须声明：

```text
读什么输入。
依赖哪些前置阶段。
输出哪些草稿。
允许哪些错误。
是否可跳过。
是否可重复执行。
是否允许产生玩家可见事件草稿。
```

这叫 `StageContract`。没有契约的阶段不能进入管线。

### 3.3 草稿总线是模块之间的隔离层

各模块不直接互相修改状态，只把输出放入草稿集合。

草稿集合包括：

```text
StateChangeSet
EventDraftSet
EffectDraftSet
ProjectionDeltaSet
CandidateSet
TraceSet
```

后续阶段可以读取前面阶段的草稿摘要，但不能越权提交。

### 3.4 状态变更闸门是唯一写状态入口

所有状态变更必须经过 `StateChangeGate`：

```text
收集 → 规范化 → 排序 → 冲突检测 → 合并 → 校验 → 生成 CommitPlan → 应用。
```

如果两个阶段同时修改同一字段，必须由闸门决定：

```text
可合并。
按优先级覆盖。
拒绝提交。
转为冲突错误。
```

### 3.5 事件提交闸门是唯一写事件入口

所有事件必须经过 `EventCommitGate`：

```text
EventDraft → EventValidation → EventEnvelope → EventStore → EventBus。
```

模块只输出 `EventDraft`，不能直接输出最终事件。

### 3.6 RuleStepType 与 PipelineStageKey 分层

`02_通用枚举设计.md` 已定义 `RuleStepType`，它适合对外、日志、错误定位。

但工程实现需要更细的阶段，因此本章新增字符串键：

```text
PipelineStageKey
```

关系：

```text
PipelineStageKey 是内部可扩展阶段键。
RuleStepType 是外部稳定大类。
一个 RuleStepType 可以包含多个 PipelineStageKey。
```

例子：

```text
RuleStepType.ResolveCognition
  - resolve_cognition_state
  - resolve_memory
  - resolve_knowledge
```

这样后续新增 `resolve_emotion`、`resolve_belief` 不需要改通用枚举，只要注册新的 StageKey。

### 3.7 扩展靠注册，不靠改主流程

新增系统时，不修改 `RulePipeline` 主逻辑。

新增内容：

```text
StageAdapter
StageContract
StageDefinition
StageOrderPolicy
DraftMapper
ValidationRule
```

主流程只认识统一接口：

```text
IRulePipelineStage::execute(input, context) -> Result<RuleStageOutput>
```

## 4. 管线阶段设计

### 4.1 标准阶段顺序

第一阶段推荐管线：

| 顺序 | PipelineStageKey | RuleStepType | 中文名 | 主要职责 |
|---|---|---|---|---|
| 10 | receive_command | ReceiveCommand | 接收命令 | 接收外部命令并绑定请求 ID |
| 20 | parse_command | ParseCommand | 解析命令 | 转成内部 CommandEnvelope |
| 30 | validate_command | ValidateCommand | 校验命令 | 校验 ID、参数、目标、版本 |
| 40 | build_context | BuildContext | 构建上下文 | 构建快照、配置、随机流、trace |
| 50 | check_preconditions | CheckPreconditions | 检查前置 | 检查行动者、目标、能力、环境 |
| 60 | calculate_cost | CalculateCost | 计算成本 | 计算体力、时间、资源、风险 |
| 70 | reserve_cost | CalculateCost | 预留成本 | 输出成本消耗草稿，不立即提交 |
| 80 | resolve_action | ResolveAction | 结算行为 | 行为核心解析 |
| 90 | resolve_interaction | ResolveInteraction | 结算交互 | 对象、特性、环境反应 |
| 100 | resolve_feedback | ResolveFeedback | 结算反馈 | 生成反馈和证据 |
| 110 | resolve_cognition_state | ResolveCognition | 认知更新 | 更新认知状态草稿 |
| 120 | resolve_memory | ResolveCognition | 记忆更新 | 形成、强化、压缩、衰减记忆 |
| 130 | resolve_knowledge | ResolveCognition | 知识更新 | 形成、修正、冲突、泛化知识 |
| 140 | resolve_propagation | ResolvePropagation | 传播解析 | 处理知识传播和接收结果 |
| 150 | resolve_tribe_state | ResolvePropagation | 族群状态 | 解析族群行为、风险、资源、凝聚 |
| 160 | resolve_civilization | ResolveCivilization | 文明能力 | 解析文明能力和阶段 |
| 170 | merge_effects | ResolveCivilization | 合并效果 | 合并能力、行为、规则效果草稿 |
| 180 | validate_state_changes | ApplyStateChanges | 校验变更 | 校验所有状态变更 |
| 190 | build_commit_plan | ApplyStateChanges | 构建提交计划 | 排序、合并、冲突处理 |
| 200 | apply_state_changes | ApplyStateChanges | 应用变更 | 写入 GameState，生成新 StateVersion |
| 210 | build_events | EmitEvents | 构建事件 | 从草稿构建事件信封 |
| 220 | validate_events | EmitEvents | 校验事件 | 校验 payload、可见性、重要性 |
| 230 | emit_events | EmitEvents | 提交事件 | 写入 EventStore，发布 EventBus |
| 240 | update_projections | Finish | 更新投影 | 更新前端、AI、调试投影 |
| 250 | persist_state | PersistState | 持久化 | 保存状态、事件、随机记录 |
| 260 | finish | Finish | 完成 | 输出 PipelineResult |

### 4.2 顺序原则

```text
Command → Interaction → Feedback → Cognition → Memory → Knowledge → Propagation → Tribe → Civilization → Commit → Event → Projection
```

关键依赖：

```text
反馈是认知、记忆、知识的证据。
知识是传播的内容。
传播影响族群覆盖和误解。
族群状态输出文明候选。
文明能力消费族群候选。
状态必须先提交，事件再绑定新 StateVersion。
```

### 4.3 扩展阶段插槽

| 插槽 | 推荐位置 | 用途 |
|---|---|---|
| pre_action_extension | check_preconditions 后 | 情绪、信念、任务、AI 策略前置 |
| post_interaction_extension | resolve_interaction 后 | 特殊环境反应、连锁反应 |
| post_feedback_extension | resolve_feedback 后 | 感官增强、异常反馈 |
| post_knowledge_extension | resolve_knowledge 后 | 信仰、文化、语言系统 |
| post_tribe_extension | resolve_tribe_state 后 | 派系、外交、组织制度 |
| post_civilization_extension | resolve_civilization 后 | 科技、文化路线、胜利条件 |
| pre_commit_extension | validate_state_changes 前 | 统一效果转换、冲突补偿 |
| post_commit_extension | apply_state_changes 后 | 投影、成就、统计 |

扩展阶段必须注册契约，不允许直接插入裸函数。

### 5.4 PipelineStageKey（管线阶段键）

`PipelineStageKey` 是稳定字符串，不是 C++ enum。

命名规则：

```text
小写 snake_case。
必须全局唯一。
必须有中文翻译和说明。
必须映射到一个 RuleStepType。
新增阶段不改 RuleStepType，除非对外协议也需要新大类。
```

### 5.5 StageContract（阶段契约）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| input_requirements | StageInputRequirements | 输入要求 | 需要哪些快照、草稿、候选 |
| output_permissions | StageOutputPermissions | 输出权限 | 允许输出哪些草稿 |
| read_permissions | list<StateDomainKey> | 读权限 | 可读取状态域 |
| write_domains | list<StateDomainKey> | 写域声明 | 允许产生哪些状态域草稿 |
| event_permissions | list<EventType> | 事件权限 | 允许产生哪些事件草稿 |
| dependencies | list<PipelineStageKey> | 依赖阶段 | 必须在这些阶段后执行 |
| invariants | list<PipelineInvariantRef> | 不变量 | 阶段必须满足的规则 |
| idempotent | bool | 幂等 | 同输入重复执行是否同输出 |
| can_skip | bool | 可跳过 | 条件不足是否跳过 |

## 7. 执行流程

### 7.1 完整流程

```text
1. 接收 PipelineRequest。
2. 加载 PipelineDefinition。
3. 校验 PipelineDefinition。
4. 构建 PipelineSnapshot。
5. 创建 PipelineTransaction。
6. 按 order 执行 Stage。
7. DraftCollector 收集每阶段输出。
8. 遇到错误时按 StageErrorPolicy 处理。
9. 所有阶段完成后进入 StateChangeGate。
10. StateChangeGate 生成 CommitPlan。
11. 提交状态变更，得到新 StateVersion。
12. EventCommitGate 构建并校验事件。
13. 提交事件并发布 EventBus。
14. ProjectionUpdateGate 更新投影。
15. PersistState 保存状态、事件、随机记录。
16. 输出 PipelineResult。
```

### 7.2 状态提交顺序

状态提交必须在事件提交之前。

原因：

```text
事件必须绑定最终 StateVersion。
如果事件先发，状态提交失败会造成前端看到不存在的结果。
```

顺序：

```text
StateChangeDraft → CommitPlan → GameStateStore → StateVersion → EventEnvelope → EventStore/EventBus
```

### 7.3 事件提交顺序

事件提交必须在状态提交成功后。

事件排序规则：

```text
stage order
event importance
source draft deterministic_key
event type stable order
```

禁止依赖容器遍历顺序。

### 7.4 失败路径

失败分三类：

| 类型 | 状态是否提交 | 事件是否提交 | 结果 |
|---|---|---|---|
| 用户阻断 | 否 | 可提交 ActionBlocked | Blocked |
| 规则错误 | 否 | 可提交 RuleErrorOccurred | SystemError |
| 提交失败 | 尝试回滚 | 不发布业务事件 | Fatal 或 Critical |

用户阻断不是系统错误，可以输出玩家可见事件。规则错误必须阻止提交。

## 8. 状态变更闸门

### 8.1 校验步骤

`StateChangeGate` 必须执行：

```text
1. draft_id 唯一性校验。
2. source_stage 权限校验。
3. domain 权限校验。
4. target_id 格式校验。
5. path 合法性校验。
6. operation 与字段类型匹配校验。
7. precondition 校验。
8. StateVersion 校验。
9. 冲突检测。
10. 合并策略校验。
11. 不变量校验。
12. CommitPlan 生成。
```

### 8.2 冲突处理

冲突键：

```text
domain + target_id + path
```

| 情况 | 处理 |
|---|---|
| 两个 Add 到同一数值 | SumNumeric 可合并 |
| 两个 Set 到不同值 | 默认 RejectOnConflict |
| Append 同一 ID | AppendUnique 去重 |
| 高优先级规则覆盖低优先级 | 必须显式 HighestPriorityWins |
| 自定义合并 | 必须注册 CustomResolver |

### 8.3 不变量校验

第一阶段必须检查：

```text
Ratio 在 0-1。
Count 不小于 0。
StateVersion 匹配。
ID 类型前缀合法。
KnowledgeClaim 不缺主体、行为、效果。
CapabilityUnlocked 不缺 unlock_conditions。
Lost 能力不能有 active_effect。
事件引用的状态变更必须存在。
```

### 8.4 提交原子性

推荐实现：

```text
内存副本应用 → 校验新状态 → 生成 StateVersion → 原子替换。
```

如果持久化存储支持事务，应使用存储事务。

如果不支持，必须有：

```text
RollbackPlan
StateVersion CAS
提交日志
崩溃恢复检查
```

## 9. 事件提交闸门

### 9.1 EventDraft 到 EventEnvelope

事件闸门负责补齐：

```text
event_id

tick
game_time
command_id
source_stage
rule_step

```

模块不允许自己补最终 `event_id`。

### 9.2 事件校验

兼容 `04_事件系统设计.md`：

```text
event_id 必填。
event_type 必填。
tick / game_time 必填。
state_version 必填。
payload 必填。
visibility 必填。
importance 必填。

ActionResolved 必须携带 action_type、actor_id 和 result_summary。
```

### 9.3 事件去重

事件去重键：

```text
transaction_id + source_stage + event_type + deterministic_key
```

例子：

```text
Stable 能力重复维持，不重复输出 CapabilityUnlocked。
同一 MemoryRecord 被强化一次，只输出一次 MemoryUpdated。
```

## 10. 确定性执行

### 10.1 必须确定的内容

```text
阶段顺序。
草稿排序。
事件排序。
ID 生成。
随机流。
时间来源。
配置版本。
浮点处理。
冲突合并。
```

### 10.2 排序规则

所有集合进入规则计算前必须排序。

推荐排序键：

```text
priority desc
stage_order asc
domain asc
target_id asc
path asc
draft_id asc
```

### 10.3 随机规则

允许随机，但必须可复盘：

```text
使用命名 RandomStream。
记录 seed、stream、purpose、roll_index、result。
输出 RandomRolled 事件或随机记录。
回放时按记录重放，不重新抽。
```

### 10.4 时间规则

规则使用：

```text
GameTime
Tick
```

禁止使用：

```text
std::chrono::system_clock 直接影响规则结果。
本地时区影响规则结果。
真实时间延迟影响规则结果。
```

### 10.5 浮点规则

建议：

```text
Ratio 使用定点或统一精度。
比较使用明确 epsilon。
配置中阈值保留固定小数位。
跨平台测试必须覆盖关键阈值。
```

## 11. 错误处理策略

### 11.1 StageErrorPolicy

| 策略 | 外部字符串 | 中文名 | 含义 |

| ConvertToBlocked | convert_to_blocked | 转为阻断 | 用户可理解的阻断 |
| RequireRollback | require_rollback | 要求回滚 | 已提交过程失败时使用 |
| FatalStop | fatal_stop | 致命停止 | 状态不可信，停止服务或存档 |

### 11.2 错误分级处理

| ErrorSeverity | 管线处理 |
|---|---|
| Info | 继续，记录 trace |
| Warning | 按阶段策略继续或阻断 |
| UserError | 阻断命令，不提交业务状态 |
| Recoverable | 可阻断或降级继续 |
| RuleError | 中断，不提交状态 |
| Critical | 中断，必要时回滚 |
| Fatal | 停止，标记状态不可信 |

### 11.3 state_mutated 规则

阶段执行期间不应真正修改状态，因此多数错误必须满足：

```text
state_mutated = false
```

只有进入 `apply_state_changes` 后出现存储异常，才可能需要回滚。

## 14. 校验规则

### 14.1 管线定义校验

| 规则 | 错误级别 |
|---|---|
| stage_key 重复 | ConfigError |
| order 重复且无 tie_breaker | ConfigError |
| 依赖阶段不存在 | ConfigError |
| 阶段依赖成环 | ConfigError |
| stage 未注册 adapter | ConfigError |
| StageContract 缺少 RuleStepType | ConfigError |
| 扩展阶段插槽不存在 | ConfigError |

### 14.2 阶段输出校验

| 规则 | 错误级别 |
|---|---|
| 阶段输出未授权状态域 | RuleError |

| event_draft payload 类型不匹配 | RuleError |
| trace_required 但无 trace | RuleError |

### 14.3 状态提交校验

| 规则 | 错误级别 |
|---|---|
| StateVersion 不匹配 | Critical |

| Ratio 越界 | RuleError |
| Count 小于 0 | RuleError |

### 14.4 事件提交校验

| 规则 | 错误级别 |
|---|---|

| 规则 | 错误级别 |
|---|---|

## 17. 设计注意事项

研发实现必须遵守：

```text
不要让 Resolver 直接写 Store。
不要让 Resolver 直接发布 EventBus。
不要把 PipelineStageKey 写死在业务 if/else 大串里。
不要用真实时间影响规则结果。
不要用未记录随机。
不要依赖无序遍历。
不要让前端消费未提交草稿当最终结果。
不要在扩展新系统时修改 RulePipelineExecutor 主流程。
不要把用户玩法失败当系统错误。
不要把系统 RuleError 当普通玩法失败。
```

推荐实现顺序：

```text
1. 实现 StageRegistry 和 PipelineDefinitionValidator。
2. 实现 RuleStageInput / Output 和 DraftCollector。
3. 实现 StateChangeGate 的校验、排序、冲突检测。
4. 实现 EventCommitGate。
5. 接入 08-10：命令、交互、反馈。
6. 接入 11-13：认知、记忆、知识。
7. 接入 14-16：传播、族群、文明。
8. 实现回放测试和 output_hash。
9. 再做性能优化和批处理。
```
