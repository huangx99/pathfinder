# P37 最小剧情闭环任务卡设计

## 1. 阶段定位

P37 是“最小剧情闭环”阶段。

它接在 P34 前置认知因果归因、P35 状态化对象规则、P36 行为目标知识之后，不是重新做规则引擎，也不是正式大地图阶段。

P37 的核心任务是把目前已经完成的底层系统，组织成玩家能在 H5 上完成的一轮 5-10 分钟“第一天生存”流程。

当前系统已经具备：

```text
P15-P21：认知、记忆、知识、传播、学习闭环。
P22/P33：H5 中文对话和极简可玩验证壳。
P23-P26：族群状态、共同作战、冲突、文明能力。
P27：受限条件表达式。
P28/P28.5：对象反应与配置化物品反应流水线。
P29：危险与探索压力。
P30/P31：存档前置与真实复盘。
P32：H5 安全协议投影。
P34：时间窗口和候选原因驱动的因果归因。
P35：对象状态、工具消耗、工具维护。
P36：行为目标知识和目标对象传播。
```

P37 要做的是“内容与问题闭环”，不是继续堆抽象能力。

本阶段必须先准备一个可以自洽的小世界：

```text
环境：林地边缘，即将入夜。
核心问题：夜晚寒冷与野兽靠近。
核心材料：红果、腐烂红果、苦叶、石片、斧头、木头、磨石、干草、火种、火把。
核心行动：观察、吃、使用、对目标使用、组合、等待、教同伴、查看知识。
核心结果：恢复饥饿、中毒风险、砍木头、斧头变钝、打磨斧头、生火、制作火把、驱赶野兽、撑过夜晚。
```

P37 成功后，游戏不再只是“点按钮看知识列表”，而是出现最小玩家目标：

```text
在不知道物品用途的情况下，探索材料，形成知识，教会同伴，制作或使用关键物品，解决夜晚危险。
```

## 2. 为什么现在做

### 2.1 前置系统已经足够

如果没有 P34，红果与腐烂红果会出现错误归因：一次负面反馈可能错误推翻旧知识。

如果没有 P35，工具不能损耗和维护，斧头、磨石、木头无法形成真实工具闭环。

如果没有 P36，系统只能表达“斧头使用”，不能表达“斧头对木头使用”，后续火把、火源、野兽反制都会变得含糊。

现在 P34-P36 已完成，所以 P37 可以开始做“材料与问题闭环”。

### 2.2 不能继续只做底层

继续增加抽象系统会带来风险：

- 系统越来越完整，但玩家仍然不知道目标是什么。
- H5 仍然像调试工具，不像游戏。
- 知识列表越来越多，但没有“知识帮助我活下来”的体验。
- 研发 AI 会继续添加孤立能力，缺少玩家流程约束。

P37 必须把已有能力拉回到玩家体验。

### 2.3 不能直接做完整地图或正式 UI

当前还不应该做大地图、背包、战斗动画、完整任务系统。

原因：

- 核心认知玩法还需要最小剧情验证。
- 大地图会引入移动、寻路、视野、碰撞等新复杂度。
- 正式 UI 会掩盖底层流程是否成立的问题。

P37 先做地点式小场景，P38 再做玩家界面去调试化。

## 3. 阶段目标

P37 完成后，系统必须具备以下能力：

1. 后端存在一个“第一天生存”最小场景定义。
2. 场景内有足够材料支撑完整问题闭环。
3. 玩家可以通过 H5 完成观察、尝试、学习、教学、制作/使用、等待。
4. 夜晚危险会随时间推进，不再只是静态提示。
5. 火、火把或简易营地至少有一个可以作为危险反制手段。
6. 野兽或寒冷至少有一个作为明确外部问题。
7. 每个关键物品都通过规则、反馈和知识形成接入，不允许前端写死用途。
8. 玩家可以形成至少 6 条不同类型知识。
9. 同伴可以通过教学获得关键知识。
10. 关键知识能影响危险处理结果。
11. 工具状态会影响流程，例如斧头变钝后需要磨石恢复。
12. 对象反应或组合至少参与一次关键目标，例如木头 + 火种 -> 火，木棍 + 火 -> 火把。
13. H5 能展示当前目标、时间压力、危险前兆和结果反馈。
14. 所有新增内容必须能被测试验证，不依赖手动点击。
15. 本阶段只做最小闭环，不做完整内容包。

## 4. 本阶段不解决的问题

### 4.1 不做大地图

P37 不做网格地图、寻路、碰撞、视野、战争迷雾。

允许使用“地点式场景”：

```text
林地边缘
溪边
灌木丛
临时营地
```

但地点只是场景标签，不引入坐标系统。

### 4.2 不做复杂战斗

野兽在 P37 是危险事件，不是完整战斗单位。

不做：

```text
野兽 HP
攻击动画
技能
掉落
尸体
战利品
```

野兽只通过 P29 危险压力表达：靠近、威慑、被火把/火光驱赶、造成失败后果摘要。

### 4.3 不做正式背包

P37 可以记录场景物品是否可用、是否已生成、是否被消耗。

但不做完整背包系统、重量、堆叠、装备栏。

物品状态仍由后端场景状态或现有对象状态管理，不由前端保存。

### 4.4 不做自由文本 NLP

玩家中文输入仍走已有限定解析，优先使用按钮。

允许新增固定意图：

```text
制作火把
点燃火把
用火把驱赶野兽
等待一会
收集干草
查看目标
```

不做大模型理解自由文本。

### 4.5 不做完整任务系统

P37 可以有 `StoryObjectiveProjection` 这类最小目标投影。

但不引入完整任务链、支线、奖励表、任务脚本语言。

### 4.6 不做训练环境

P37 产生的流程可以为 P42 训练环境提供样本基础，但本阶段不定义 Observation/Reward/Done。

## 5. 核心设计原则

### 5.1 问题先于物品

新增物品不是为了堆内容，而是为了解决明确问题。

P37 的问题是：

```text
入夜后温度下降，野兽靠近，玩家必须利用学习到的知识准备火或火把。
```

所以火、火把、木头、干草、火种、斧头、磨石都必须服务于这个问题。

禁止添加没有闭环用途的物品。

### 5.2 每个关键物品必须能形成知识

P37 新增或启用的物品必须至少属于一种知识路径：

- 食用知识。
- 使用知识。
- 目标行为知识。
- 组合/反应知识。
- 危险反制知识。
- 状态维护知识。

例如：

```text
火把 + use @ beast -> repel_beast
火种 + use @ dry_grass -> ignite_fire
axe + use @ wood -> cut_wood
whetstone + use @ axe -> restore_sharpness
```

### 5.3 危险必须逼迫玩家使用知识

夜晚危险不能只是装饰文本。

它必须有：

- 前兆。
- 时间推进。
- 可反制条件。
- 成功结果。
- 失败结果。
- 可形成的知识。

### 5.4 前端仍然不是规则层

H5 可以展示目标、危险、材料和行动按钮。

但它不能判断：

- 火是否能驱赶野兽。
- 火把是否已点燃。
- 斧头是否锋利。
- 玩家是否撑过夜晚。
- 同伴是否真的学会。

这些都必须来自后端投影。

### 5.5 场景编排只能调用已有规则能力

P37 可以新增场景编排服务，但它只是编排者，不是第二套规则引擎。

它可以：

- 读取安全投影。
- 推进时间。
- 选择当前阶段危险。
- 生成可用行动建议。
- 调用已有学习/反馈/反应/危险流程。

它不可以：

- 直接修改知识状态。
- 直接判定真实物品效果。
- 直接绕过 P27 条件表达式。
- 直接绕过 P28/P28.5 反应流水线。
- 直接绕过 P29 危险系统。

### 5.6 最小剧情不是写死剧情

P37 可以固定“第一天生存”流程，但不能把结果写死成剧情文本。

正确做法：

```text
配置/场景定义提供初始物品、危险阶段和目标提示。
玩家行动进入现有规则系统。
规则输出反馈、知识、状态、事件。
场景服务根据后端状态和投影判断下一阶段提示。
```

错误做法：

```text
玩家点击“制作火把”后，前端直接显示“你成功驱赶了野兽”。
```

### 5.7 时间必须是轻量但真实的

P37 需要时间压力，但不做完整日夜系统。

最小时间模型：

```text
Morning / Afternoon / Dusk / Night / Survived
```

每个有效行动推进一个阶段或消耗一个时间点。

危险在 Dusk 出现前兆，在 Night 结算。

### 5.8 教学必须影响闭环

教同伴不能只是复制知识列表。

P37 至少要验证：

- 玩家教会同伴“火把可以驱赶野兽”。
- 或玩家教会同伴“斧头对木头使用可以获得木材”。
- 同伴知识投影发生变化。
- 场景目标提示能显示“同伴已经理解关键知识”。

本阶段不要求同伴自主行动，但必须为后续 P38/P39 留接口。

## 6. 模块划分

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| `pathfinder_story` | 最小剧情模块 | 定义第一天生存场景、阶段、目标和编排 | 场景配置、行动结果、投影摘要 | 场景状态、目标投影、阶段事件 | 不得裁决物品真实效果 |
| `StoryScenarioDefinition` | 剧情场景定义 | 声明初始地点、物品、危险、目标、成功条件 | 静态定义或配置草案 | 可校验场景 | 不得包含 hidden truth |
| `StoryRuntimeState` | 剧情运行状态 | 记录时间阶段、目标进度、危险阶段、生成物状态引用 | 行动结果、事件摘要 | 可持久化状态草稿 | 不得保存前端权威状态 |
| `StoryProgressionService` | 剧情推进服务 | 根据行动结果和时间推进目标/危险阶段 | 当前状态、行动摘要、安全投影 | 新状态草稿、目标变化、提示 | 不得直接写知识或对象状态 |
| `StoryObjectiveEvaluator` | 目标评估器 | 判断最小目标是否完成 | 安全投影、知识摘要、危险摘要、对象状态摘要 | 目标状态、缺失材料提示 | 不得读取 raw_state/hidden_truth |
| `StoryActionAdvisor` | 行动建议器 | 生成当前阶段可提示的行动按钮 | 目标状态、可见物体、已有 projection actions | 建议行动投影 | 不得替代 ActionProjection 权威 |
| `StoryMaterialCatalog` | 材料闭环目录 | 记录 P37 所需物品、反应、反馈和危险用途 | 场景定义 | 材料校验报告 | 不得只登记无用途物品 |
| `StoryProjectionAdapter` | H5 投影桥接 | 把剧情阶段、目标、危险前兆转换为 H5 安全展示 | StoryRuntimeState、安全摘要 | `StoryProjection` | 不得暴露真实伤害/死亡/掉落 |

## 7. 核心枚举

### 7.1 `StoryScenarioKind`：剧情场景类型

| 枚举值 | 中文含义 | 使用场景 | 是否持久化 | 是否给前端 | 备注 |
|---|---|---|---|---|---|
| `Unknown` | 未知 | 默认值、非法输入保护 | 是 | 否 | 解析失败返回结构化错误 |
| `FirstDaySurvival` | 第一天生存 | P37 主场景 | 是 | 可转中文 | 本阶段唯一正式场景 |
| `TestOnly` | 测试场景 | 单元测试构造最小场景 | 否 | 否 | 不能进入正式配置 |

设计要求：

- `toString(FirstDaySurvival) = "first_day_survival"`。
- `fromString` 遇到非法值必须返回错误码，不得默默返回 `FirstDaySurvival`。
- 前端不能提交该枚举作为创建权威，只能由后端 session bootstrap 选择。

### 7.2 `StoryTimePhase`：剧情时间阶段

| 枚举值 | 中文含义 | 使用场景 | 是否持久化 | 是否给前端 | 备注 |
|---|---|---|---|---|---|
| `Unknown` | 未知 | 默认保护 | 是 | 否 | 非法状态 |
| `Morning` | 早晨 | 初始探索，安全学习 | 是 | 是 | 可提示观察和尝试 |
| `Afternoon` | 午后 | 材料准备，工具使用 | 是 | 是 | 允许砍木、打磨、组合 |
| `Dusk` | 黄昏 | 危险前兆出现 | 是 | 是 | 野兽/寒冷提示增强 |
| `Night` | 夜晚 | 危险结算阶段 | 是 | 是 | 必须有反制手段 |
| `Survived` | 已撑过夜晚 | 成功状态 | 是 | 是 | 本阶段胜利 |
| `Failed` | 未撑过夜晚 | 失败状态 | 是 | 是 | 只输出安全失败摘要 |
| `TestOnly` | 测试阶段 | 测试推进 | 否 | 否 | 不进正式配置 |

推进原则：

```text
有效探索/尝试行动会推进 time_budget。
达到阈值后 Morning -> Afternoon -> Dusk -> Night。
Night 结算后进入 Survived 或 Failed。
```

### 7.3 `StoryObjectiveStatus`：剧情目标状态

| 枚举值 | 中文含义 | 使用场景 | 是否持久化 | 是否给前端 |
|---|---|---|---|---|
| `Unknown` | 未知 | 默认保护 | 是 | 否 |
| `Hidden` | 暂未显示 | 目标未解锁 | 是 | 否 |
| `Active` | 进行中 | 当前可见目标 | 是 | 是 |
| `Blocked` | 被阻塞 | 缺材料/缺知识/条件不满足 | 是 | 是 |
| `Completed` | 已完成 | 目标达成 | 是 | 是 |
| `Failed` | 已失败 | 目标失败 | 是 | 是 |
| `TestOnly` | 测试值 | 单测 | 否 | 否 |

目标必须有稳定 key 和中文说明，例如：

```text
survive_first_night：撑过第一个夜晚。
prepare_fire_source：准备能驱赶野兽或保暖的火源。
teach_companion_key_knowledge：把关键知识告诉同伴。
```

### 7.4 `StoryThreatKind`：剧情危险类型

| 枚举值 | 中文含义 | 使用场景 | 是否持久化 | 是否给前端 |
|---|---|---|---|---|
| `Unknown` | 未知 | 默认保护 | 是 | 否 |
| `ColdNight` | 夜晚寒冷 | 温度危险 | 是 | 是 |
| `ApproachingBeast` | 野兽靠近 | 野兽威胁 | 是 | 是 |
| `HungerPressure` | 饥饿压力 | 食物压力 | 是 | 是 |
| `TestOnly` | 测试危险 | 单测 | 否 | 否 |

P37 最小实现建议至少启用：

```text
ApproachingBeast
```

`ColdNight` 可以同时作为说明和后续扩展，但不要求复杂数值。

### 7.5 `StoryMaterialRole`：材料在闭环中的角色

| 枚举值 | 中文含义 | 使用场景 | 是否持久化 | 是否给前端 |
|---|---|---|---|---|
| `Unknown` | 未知 | 默认保护 | 是 | 否 |
| `FoodCandidate` | 食物候选 | 红果/腐烂红果/苦叶 | 是 | 是 |
| `Tool` | 工具 | 斧头、石片、磨石 | 是 | 是 |
| `Fuel` | 燃料 | 木头、干草 | 是 | 是 |
| `IgnitionSource` | 点火源 | 火种、火石 | 是 | 是 |
| `LightSource` | 光源/火源 | 火、火把 | 是 | 是 |
| `ThreatCounter` | 危险反制物 | 火把、火堆 | 是 | 是 |
| `CraftResult` | 制作产物 | 火把、简易营地 | 是 | 是 |
| `TestOnly` | 测试材料 | 单测 | 否 | 否 |

同一物品可以有多个 role，例如火把既是 `LightSource`，也是 `ThreatCounter`。

### 7.6 `StoryActionRole`：剧情行动角色

| 枚举值 | 中文含义 | 使用场景 | 是否持久化 | 是否给前端 |
|---|---|---|---|---|
| `Unknown` | 未知 | 默认保护 | 是 | 否 |
| `Explore` | 探索 | 观察地点、发现材料 | 是 | 是 |
| `Experiment` | 尝试 | 吃/用未知物 | 是 | 是 |
| `TargetUse` | 对目标使用 | 斧头砍木头、火把对野兽 | 是 | 是 |
| `Combine` | 组合/反应 | 制作火、火把 | 是 | 是 |
| `Teach` | 教学 | 教同伴关键知识 | 是 | 是 |
| `Wait` | 等待 | 推进时间/触发危险 | 是 | 是 |
| `Inspect` | 查看 | 查看知识、同伴、目标 | 是 | 是 |
| `TestOnly` | 测试行动 | 单测 | 否 | 否 |

## 8. ID 与命名规则

### 8.1 场景 ID

建议类型：

```cpp
using StoryScenarioId = StrongId<StoryScenarioTag>;
```

规则：

- 正式场景 ID 由后端场景注册器创建。
- 配置作者只能填写稳定 key，例如 `first_day_survival`。
- 前端不能创建场景 ID。
- 场景 ID 参与 save/replay/baseline。

### 8.2 目标 ID

建议类型：

```cpp
using StoryObjectiveId = StrongId<StoryObjectiveTag>;
```

规则：

- 目标 ID 由 `StoryScenarioDefinition` 构建。
- 目标 key 必须稳定，例如 `prepare_fire_source`。
- 目标状态可以持久化。
- 前端只能展示目标 ID/key，不能提交目标完成。

### 8.3 危险阶段 ID

建议类型：

```cpp
using StoryThreatStageId = StrongId<StoryThreatStageTag>;
```

规则：

- 由后端构建。
- 用于复盘危险推进。
- 不允许前端伪造危险阶段。

### 8.4 材料 key

P37 材料 key 必须复用已有对象 key 风格：

```text
red_berry
decayed_red_berry
bitter_leaf
stone_flake
axe
wood
whetstone
dry_grass
fire_seed
camp_fire
torch
beast_shadow
```

注意：

- `beast_shadow` 是危险对象/投影引用，不是完整战斗实体。
- `camp_fire` 和 `torch` 可以是反应产物。
- 新 key 必须进入配置/场景校验，不能只写在前端按钮里。

## 9. DTO 与数据结构

### 9.1 `StoryScenarioDefinition`

中文含义：剧情场景定义。

字段：

| 字段 | 类型 | 含义 | 校验 |
|---|---|---|---|
| `scenario_id` | `StoryScenarioId` | 场景 ID | 必填 |
| `scenario_key` | `std::string` | 稳定 key | 非空，只允许小写下划线 |
| `kind` | `StoryScenarioKind` | 场景类型 | 不能为 Unknown/TestOnly 正式运行 |
| `title_zh_cn` | `std::string` | 中文标题 | 非空，不含禁止字段 |
| `initial_time_phase` | `StoryTimePhase` | 初始时间 | 必须是 Morning 或测试值 |
| `initial_location_key` | `std::string` | 初始地点 key | 非空 |
| `visible_object_keys` | `std::vector<std::string>` | 初始可见物品 | 至少 6 个 |
| `material_entries` | `std::vector<StoryMaterialEntry>` | 材料闭环定义 | 每个 key 唯一 |
| `objectives` | `std::vector<StoryObjectiveDefinition>` | 目标定义 | 至少 3 个 |
| `threats` | `std::vector<StoryThreatDefinition>` | 危险定义 | 至少 1 个 |
| `time_budget` | `StoryTimeBudget` | 时间推进规则 | 阈值合法 |
| `success_objective_key` | `std::string` | 成功目标 | 必须存在 |

`validateBasic()`：

- 拒绝空 key。
- 拒绝重复物品 key。
- 拒绝没有用途 role 的材料。
- 拒绝没有反制条件的危险。
- 拒绝标题或说明包含 forbidden key。
- 拒绝正式运行使用 `TestOnly`。

### 9.2 `StoryMaterialEntry`

中文含义：剧情材料条目。

字段：

| 字段 | 类型 | 含义 | 校验 |
|---|---|---|---|
| `object_key` | `std::string` | 对象 key | 非空、稳定 |
| `display_name_zh_cn` | `std::string` | 中文名 | 非空 |
| `roles` | `std::vector<StoryMaterialRole>` | 材料角色 | 至少 1 个非 Unknown |
| `required_for_objectives` | `std::vector<std::string>` | 关联目标 key | 引用必须存在 |
| `expected_actions` | `std::vector<StoryActionRole>` | 预期行动类型 | 至少 1 个 |
| `knowledge_paths` | `std::vector<std::string>` | 预期知识路径 key | 至少 1 个关键材料必须有 |
| `reaction_rule_keys` | `std::vector<std::string>` | 相关反应规则 | 可空，但反应材料必须非空 |
| `danger_counter_keys` | `std::vector<std::string>` | 可反制危险 | 反制物必须非空 |

示例：

```text
torch:
  roles = LightSource, ThreatCounter, CraftResult
  expected_actions = TargetUse, Combine
  knowledge_paths = torch_use_beast_repel
  danger_counter_keys = approaching_beast
```

中文含义：剧情目标定义。

字段：

| 字段 | 类型 | 含义 | 校验 |
|---|---|---|---|
| `objective_id` | `StoryObjectiveId` | 目标 ID | 必填 |
| `objective_key` | `std::string` | 稳定 key | 非空唯一 |
| `title_zh_cn` | `std::string` | 中文标题 | 非空 |
| `hint_zh_cn` | `std::string` | 中文提示 | 非空 |
| `initial_status` | `StoryObjectiveStatus` | 初始状态 | 不能 Unknown |
| `required_knowledge_keys` | `std::vector<std::string>` | 需要的知识 | 可空 |
| `required_object_keys` | `std::vector<std::string>` | 需要的对象/产物 | 可空 |
| `required_threat_states` | `std::vector<std::string>` | 需要的危险状态 | 可空 |
| `unlocks_objective_keys` | `std::vector<std::string>` | 解锁目标 | 引用必须存在 |

目标不直接写规则结果，只写检查条件引用。

### 9.4 `StoryThreatDefinition`

中文含义：剧情危险定义。

字段：

| 字段 | 类型 | 含义 | 校验 |

| `success_feedback_key` | `std::string` | 成功反馈 | 非空 |
| `failure_feedback_key` | `std::string` | 失败反馈 | 非空 |

禁止字段：

```text

true_damage
death
kill
corpse
loot
drop
```

### 9.5 `StoryRuntimeState`

中文含义：剧情运行状态。

字段：

| 字段 | 类型 | 含义 | 校验 |

`StoryRuntimeState` 可以进入存档，但不能保存真实隐藏效果。

要求：

- 如果 `accepted=false`，不得部分修改 `StoryRuntimeState`。

## 10. 服务接口设计

### 10.1 `StoryScenarioRegistry`

职责：注册和读取场景定义。

接口：

```cpp
Result<StoryScenarioDefinition> getScenario(StoryScenarioId id) const;
Result<StoryScenarioDefinition> getScenarioByKey(std::string_view key) const;
ValidationReport validateScenario(const StoryScenarioDefinition& scenario) const;
```

副作用：无。

禁止：

- 不得根据前端输入动态创建正式场景。
- 不得在 registry 内执行规则结果。

### 10.2 `StoryRuntimeFactory`

职责：从场景定义创建初始运行状态。

接口：

```cpp
Result<StoryRuntimeState> createInitialState(const StoryScenarioDefinition& scenario,
                                             const StoryRuntimeCreateOptions& options) const;
```

副作用：无或只创建草稿。

要求：

- 初始状态必须可复盘。
- 初始可见物品必须来自场景定义。
- 初始时间必须是 `Morning`。

### 10.3 `StoryProgressionService`

职责：推进剧情时间、目标和危险。

接口：

```cpp
Result<StoryTurnResult> applyTurn(const StoryTurnInput& input,
                                  const StoryRuntimeState& current_state,
                                  const H5GameProjection& safe_projection,
                                  const LearningLoopResult& learning_result,
                                  const DangerResolutionSummary& danger_summary) const;
```

副作用：

- 不直接写全局状态。
- 输出新 `StoryRuntimeState` 草稿。
- 输出 `StoryEvent`。

失败规则：

- 输入非法时返回错误，不修改状态。
- 下游 projection 缺关键字段时返回 `MissingProjectionData`。
- 目标引用不存在时返回 `InvalidStoryScenario`。

### 10.4 `StoryObjectiveEvaluator`

职责：根据安全摘要判断目标状态。

接口：

```cpp
Result<std::vector<StoryObjectiveChange>> evaluateObjectives(
    const StoryScenarioDefinition& scenario,
    const StoryRuntimeState& state,
    const StoryEvaluationContext& context) const;
```

`StoryEvaluationContext` 只能包含：

- 安全知识摘要。
- 安全对象状态摘要。
- 安全危险摘要。
- 已生成物品摘要。
- 最近行动摘要。

禁止包含：

```text
hidden_truth
raw_state
true_effect
actual_hp
random_damage
```

### 10.5 `StoryThreatBridge`

职责：把 P37 剧情危险接入 P29 危险系统。

接口：

```cpp
Result<DangerResolutionRequest> buildDangerRequest(
    const StoryThreatDefinition& threat,
    const StoryRuntimeState& state,
    const StoryEvaluationContext& context) const;

Result<StoryThreatChange> mapDangerResult(
    const DangerResolutionResult& result,
    const StoryThreatDefinition& threat) const;
```

禁止：

- 不得自己实现伤害、死亡、掉落。
- 不得绕过 P29 的危险结果。

### 10.6 `StoryReactionBridge`

职责：把剧情材料组合接入 P28/P28.5 对象反应。

接口：

```cpp
Result<ReactionResolveRequest> buildReactionRequest(const StoryTurnInput& input,
                                                    const StoryRuntimeState& state) const;

Result<StoryMaterialChange> mapReactionResult(const ReactionResolveResult& result) const;
```

禁止：

- 不得前端点击“制作火把”就直接生成 `torch`。
- 不得绕过 ReactionResolver。

### 10.7 `StoryProjectionAdapter`

职责：生成 H5 剧情补充投影。

接口：

```cpp
Result<StoryProjection> composeProjection(const StoryScenarioDefinition& scenario,
                                          const StoryRuntimeState& state,
                                          const StoryProjectionContext& context,
                                          ProjectionAudience audience) const;
```

要求：

- Player audience 不输出 debug keys。
- 中文文本必须来自安全字段或静态文案。
- 建议行动必须引用后端 ActionProjection，不得自行启用不可用行动。

## 11. 主流程设计

### 11.1 Bootstrap 流程

```text
1. H5 请求 bootstrap。
2. 后端选择 first_day_survival 场景。
3. StoryScenarioRegistry 返回场景定义。
4. StoryRuntimeFactory 创建 Morning 初始状态。
5. H5PlayableTurnService 生成基础 H5GameProjection。
6. StoryProjectionAdapter 追加剧情目标、时间、危险前兆。
7. 前端显示：林地边缘、当前目标、可见材料、可行动作。
```

### 11.2 普通尝试流程

```text
1. 玩家点击“吃红果”或“使用石片”。
2. H5 提交 ActionCommand。
3. 现有对话/学习适配层处理反馈。
4. P21 学习闭环形成认知、记忆、知识。
5. P34 因果归因参与知识修正。
6. StoryProgressionService 根据 accepted action 推进时间点。
7. StoryObjectiveEvaluator 检查目标是否变化。
8. StoryProjectionAdapter 输出新目标和提示。
```

### 11.3 工具使用流程

```text
1. 玩家点击“用斧头砍木头”。
2. P36 行为目标知识保留 target=wood。
3. P35 状态化对象检查 axe.sharpness。
4. 如果锋利，产生 cut_wood，降低 sharpness，可能生成 wood_piece。
5. 如果钝，产生 tool_dull，不继续砍。
6. 学习链路形成“斧头对木头使用”的条件知识。
7. 剧情目标检查是否已获得木材或是否需要打磨。
```

### 11.4 反应/制作流程

```text
1. 玩家点击“组合干草和火种”或“制作火把”。
2. StoryReactionBridge 构造 ReactionResolveRequest。
3. P28/P28.5 ReactionResolver 判断材料和条件。
4. 成功生成 camp_fire 或 torch。
5. ReactionKnowledgePlanner 产生反应知识草稿。
6. StoryRuntimeState 记录 generated_object_keys。
7. H5 投影出现新物品和目标变化。
```

### 11.5 教同伴流程

```text
1. 玩家点击“教同伴”。
2. P20 传播系统处理知识传播。
3. P36 保留 target 和 condition。
4. 同伴知识投影更新。
5. StoryObjectiveEvaluator 检查 teach_companion_key_knowledge 是否完成。
6. H5 显示同伴已经理解的中文摘要。
```

### 11.6 危险前兆与结算流程

```text
1. 时间推进到 Dusk。
2. StoryThreatDefinition 激活 approaching_beast。
3. H5 显示“树林里传来低吼”。
4. 玩家可以继续准备火/火把，或等待。
5. 时间推进到 Night。
6. StoryThreatBridge 调用 P29 危险系统。
7. 如果玩家拥有火把/火源并形成相关反制知识，危险被缓解。
8. 如果没有反制，输出安全失败摘要。
9. StoryTimePhase 进入 Survived 或 Failed。
```

## 12. 典型玩法场景

### 12.1 成功撑过夜晚

```text
观察 -> 吃红果 -> 用斧头砍木头 -> 斧头变钝 -> 用磨石打磨斧头 -> 收集干草 -> 点火 -> 制作火把 -> 教同伴 -> 夜晚用火把驱赶野兽 -> Survived
```

验证点：

- 食物知识、工具知识、状态知识、反应知识、危险反制知识均参与。
- 同伴获得至少一条关键知识。
- 最终成功来自后端危险系统，不来自前端文本。

### 12.2 材料不足导致阻塞

```text
玩家一直吃红果和等待，没有准备火源。
Dusk 出现野兽前兆。
Night 结算时缺少火把/火源。
系统进入 Failed 或输出强失败反馈。
```

验证点：

- 危险不是装饰。
- 失败可读但不暴露死亡/伤害真实值。
- 目标卡片显示缺少反制手段。

### 12.3 工具变钝后必须维护

```text
玩家连续用斧头砍木头。
axe.sharpness 降到 0。
再次砍木头得到 tool_dull。
玩家用磨石打磨斧头。
再次砍木头成功。
```

验证点：

- P35 状态化对象生效。
- P36 条件知识不互相覆盖。
- H5 中文解释不显示裸 key。

### 12.4 错误归因被控制

```text
玩家吃红果获得正向反馈。
过一段时间吃腐烂红果获得负面反馈。
系统不直接把“红果可食用”推翻。
负面反馈归因到腐烂红果或形成候选原因。
```

验证点：

- P34 因果归因生效。
- P37 时间推进不会破坏已有归因逻辑。

### 12.5 教学影响同伴理解

```text
玩家学会火把驱赶野兽。
玩家教同伴。
同伴知识显示“火把对野兽使用可以驱赶”。
目标 teach_companion_key_knowledge 完成。
```

验证点：

- target/condition 在传播后保留。
- 同伴知识不是玩家知识的裸复制文本。

## 13. 与现有系统关系

### 13.1 与 P21 学习闭环

- P37 行动必须进入 P21 学习链路。
- P37 不直接创建知识。
- P37 只能读取学习结果摘要和知识投影。

### 13.2 与 P27 条件表达式

- P37 条件判断必须复用 P27。
- 不允许重新发明 condition_key 字符串判断。
- 目标完成条件如果需要表达式，必须进入受限表达式。

### 13.3 与 P28/P28.5 对象反应

- 火、火把、营地等产物必须通过反应流水线生成。
- P37 可以声明“需要这些材料”，但不能直接生成产物。
- 反应失败必须返回安全中文原因。

### 13.4 与 P29 危险系统

- 野兽和寒冷属于危险，不是完整战斗系统。
- 危险结算必须走 P29 的 resolver/bridge。
- P37 只负责把剧情阶段映射为危险请求。

### 13.5 与 P32/P33 H5 投影

- P37 新增 `StoryProjection` 必须作为 H5 安全投影补充。
- 前端只渲染投影。
- 目标和危险提示不能泄露 hidden truth。

### 13.6 与 P34 因果归因

- P37 时间推进必须与 P34 时间窗口兼容。
- 吃多了、腐烂、延迟效果等归因不能被剧情目标简单覆盖。
- P37 新增反馈必须提供足够 evidence metadata。

### 13.7 与 P35 状态化对象

- 斧头、火把、火源等运行状态必须通过对象状态表达。
- 工具损耗和维护必须保留事件和 trace。
- P37 不直接修改 `sharpness`。

### 13.8 与 P36 行为目标知识

- 斧头砍木头、磨石打磨斧头、火把驱赶野兽都必须保留 target。
- 教学传播不能丢失 target/condition。
- 中文知识解释必须包含目标对象。

### 13.9 与 P30/P31 存档复盘

- `StoryRuntimeState` 必须可序列化和可复盘。
- 时间推进、目标变化、危险变化必须形成事件或 trace。
- 同一命令日志和 seed 应复现相同故事阶段。

## 15. 错误实现禁止清单

## 19. 风险与控制

### 19.1 风险：P37 变成写死剧情脚本

原因：为了快速可玩，容易直接在 H5 或 story service 里写“点击 A 得到 B”。

后果：破坏规则权威，后续内容无法扩展。

控制：所有产物必须走 P28/P28.5，所有危险必须走 P29，所有知识必须走 P21/P36。

### 19.2 风险：新增物品没有用途

原因：想让世界丰富，随手加材料。

后果：玩家困惑，测试无法证明玩法闭环。

控制：`StoryMaterialEntry` 必须声明 role、目标、行动或知识路径。

### 19.3 风险：危险压力过强或过弱

原因：P37 首次加入时间压力，节奏可能不合适。

后果：玩家要么无压力，要么来不及理解。

控制：只做 Morning/Afternoon/Dusk/Night 轻量阶段，测试成功与失败两条路径。

### 19.4 风险：H5 信息仍像调试器

原因：知识 key 和对象 key 直接展示。

后果：玩家看不懂。

控制：P37 投影必须提供中文目标、危险、材料提示；P38 再进一步去调试化。

### 19.5 风险：目标判断绕过现有系统

原因：目标系统需要判断“是否有火把/知识”，容易读取 raw state。

后果：破坏安全边界。

控制：`StoryObjectiveEvaluator` 只能读取安全投影和摘要。

### 19.6 风险：同伴教学没有实际意义

原因：P37 不做同伴自主行动，教学可能仍只是列表变化。

后果：传播系统体验弱。

控制：至少设置一个目标要求“同伴理解关键知识”，并在投影中显示同伴理解变化。

### 19.7 风险：引入火和野兽后误入战斗系统

原因：野兽天然像敌人。

后果：范围膨胀到战斗、HP、掉落。

控制：P37 明确野兽是危险事件，不是战斗实体；复杂战斗后续再做。
