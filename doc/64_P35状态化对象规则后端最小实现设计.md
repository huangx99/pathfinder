# P35 状态化对象规则后端最小实现设计

## 目标

让后端可以支持“工具使用会改变对象状态，状态再影响后续规则”的最小闭环。

本阶段不做前端 UI，不做完整 JSON 配置加载，只在现有 H5 后端规则流里加入通用状态能力，用斧头、木头、磨石验证。

## 要解决的问题

蘑菇、红果属于“对象状态 + 行为 -> 结果”。

斧头属于更进一步的规则：

1. 对象有运行时状态。
2. 行为会改变对象状态。
3. 改变后的状态会影响下一次行为。
4. 另一个行为可以修复或维护状态。

## 最小模型

### DialogObjectRuntimeState

会话级对象状态，不写死到具体物品。

字段：

- `object_key`：对象 id。
- `numeric_states`：数值状态，例如 `sharpness = 3`。
- `tag_states`：标签状态，例如 `sharp`、`dull`。

### DialogStateCondition

反馈规则的运行时条件。

字段：

- `object_key`：检查哪个对象。
- `state_key`：检查哪个状态。
- `op`：比较方式，最小支持 `gt`、`gte`、`lte`、`eq`、`has_tag`、`missing_tag`。
- `number_value` / `tag_value`：比较值。

### DialogStateMutation

反馈成功后应用的状态变更。

字段：

- `object_key`：修改哪个对象。
- `op`：最小支持 `add_number`、`set_number`、`add_tag`、`remove_tag`。
- `state_key`：状态字段。
- `number_value` / `tag_value`：变更值。

## 斧头验证规则

初始状态：

- `axe.sharpness = 3`
- `axe` 拥有 `sharp` 标签

规则：

1. 使用斧头砍木头：
   - 条件：`axe.sharpness > 0`
   - 结果：`cut_wood`
   - 变更：`axe.sharpness -= 1`

2. 使用钝斧头砍木头：
   - 条件：`axe.sharpness <= 0`
   - 结果：`tool_dull`
   - 不再继续降低锋利度

3. 打磨斧头：
   - 对象：磨石
   - 行为：使用
   - 结果：`restore_sharpness`
   - 变更：`axe.sharpness = 3`，添加 `sharp`，移除 `dull`

状态变更后，系统会自动同步标签：

- `sharpness <= 0`：移除 `sharp`，添加 `dull`
- `sharpness > 0`：添加 `sharp`，移除 `dull`

## 知识配合

每次规则仍然走现有学习链路：

- 砍木头形成“斧头/使用/可以砍木头”的经验。
- 斧头变钝后形成“斧头/使用/工具变钝”的经验。
- 打磨形成“磨石/使用/恢复锋利”的经验。

后续如果要更精确，可以把“使用目标”升级为双对象知识：`axe + use_on wood -> cut_wood`。本阶段先保持现有单对象知识模型，避免大改学习系统。

## 验收标准

1. 连续使用斧头三次后，`axe.sharpness` 降到 0。
2. 第四次使用斧头返回 `tool_dull`。
3. 使用磨石后，`axe.sharpness` 恢复到 3。
4. 再次使用斧头可以重新 `cut_wood`。
5. 状态变更不影响红果、腐烂红果现有因果归因测试。
