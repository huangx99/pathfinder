# P24 共同作战任务卡设计

## 1. 阶段定位

P24 是《先驱者》从“族群状态”进入“群体协作行为”的第一阶段。

P23 已经完成：

```text
TribeState。
TribeMember。
TribeMemberRole。
TribeMorale。
TribeTrust。
TribeSplitRisk。
TribeProjection。
TribeStateInput。
TribeStateResolver。
```

P24 要解决的是：

```text
当族群遇到威胁或危险行动时，多个成员能不能以可验证、可解释、可复盘的方式产生“协作、援护、防御、集火、共同撤退”的群体意图。
```

一句话：

```text
P24 让“族群成员不是各做各的”，而是能基于角色、士气、信任和安全威胁形成共同作战协调结果。
```

P24 是共同作战协调，不是完整战斗结算，也不是族群战争。

---

## 2. 本阶段目标

P24 必须在现有 `tribe` 模块内新增共同作战协调能力。

它必须做到：

1. 定义共同作战最小类型：`GroupIntent`、`AssistAction`、`DefendAction`、`PackTactic`、`CombatCoordinationResult`。
2. 支持从 `TribeState` 和安全威胁摘要构造 `CombatCoordinationInput`。
3. 根据成员角色、士气、信任、分裂风险、威胁压力，确定性生成协作意图。
4. 输出可投影、可复盘的 `CombatCoordinationProjection` 和 `CombatCoordinationTrace`。
5. 输出对 P23 族群状态的安全影响草稿，例如士气变化、信任变化、安全压力变化，但不能绕过 `TribeStateResolver` 直接改状态。
6. 禁止随机命中、随机伤害、随机死亡、随机分裂。
7. 禁止读取 raw `GameState`、`SaveGame`、hidden truth 或真实怪物属性。
8. 必须明确遵守 P23 的重复成员事件约束：同一个 `TribeStateInput` 内不能出现重复 `member_id`。

核心原则：

```text
P24 只做共同作战协调，不做伤害结算。
P24 只处理同一族群内部协作，不做族群对族群战争。
P24 只消费安全摘要，不读世界真相。
P24 输出结果和状态影响草稿，不直接修改 TribeState。
P24 的所有结果必须确定性、可解释、可测试。
```

## 3. 玩家体验映射

P24 让玩家第一次感受到：

```text
我不是一个人在面对危险。
我教会同伴工具和危险知识后，他们能配合我。
守护者会挡在前面。
学习者会被保护。
探索者会预警。
教师/长者能稳定群体。
士气低或信任低时，协作会变差。
```

### 3.1 玩家未来感受

示例：

```text
你发现前方有狼。
守护者站到你和同伴前面。
探索者提醒大家保持距离。
采集者停止采集并退到后方。
族群形成了简单的防御阵势。
你们没有击杀狼，但降低了被突袭的风险。
```

这里的重点不是“打掉多少血”，而是：

```text
玩家看到族群成员会互相配合。
玩家理解知识、分工、士气、信任会影响协作质量。
玩家能从反馈中知道为什么协作成功或失败。
```

### 3.2 P24 需要证明的体验链

P24 的测试必须证明：

```text
成员角色会影响共同作战分工。
士气和信任会影响协作评分。
分裂风险高会削弱协作。
威胁压力高会触发防御或共同撤退。
输出结果只说明协作和风险变化，不提前结算伤害或死亡。
```

## 5. 本阶段不做什么

P24 明确禁止：

```text
不做具体伤害结算。
不做命中率。
不做血量扣减。
不做死亡、击杀、掉落。
不做武器数值系统。
不做完整战斗技能系统。
不做族群对族群战争，这是 P25。
不做仇恨、外交、宣战、停战，这是 P25。
不做文明能力解锁，这是 P26。
不做对象反应，这是 P27。
不做危险探索压力，这是 P28。
不做 H5 新界面。
不做 HTTP / WebSocket / JSON API。
不做专家 NPC 教学。
不做 NPC 识别玩家谋害。
不读 raw GameState。
不读 hidden truth。
不使用 AgentRuntime 或 Policy 作为规则权威。
不让前端推导共同作战结果。
```

P24 可以输出给后续阶段使用的安全摘要，但不能提前实现后续玩法。

## 6. P23 约束必须写死

P23 复审已经确认：

```text
同一个 TribeStateInput 内重复 member_id 会被拒绝。
```

所以 P24/P25 必须遵守：

```text
如果同一 tick 内一个成员受到多个协作影响，必须先在 P24 内聚合成一个 MemberCoordinationSummary。
如果必须拆成多个成员状态事件，就拆成多个 input tick。
P24 输出给 P23 的 TribeStateInputDraft 不允许出现重复 member_id。
P25 设计时也必须继续遵守这条规则。
```

错误示例：

```text
同一个 TribeStateInput：
- companion morale changed
- companion trust changed
- companion role changed
```

正确示例 A：

```text
P24 先聚合：
MemberCoordinationSummary(companion): morale_delta=-0.1, trust_delta=+0.05, reason_keys=[...]
然后只生成一个 member event，或只生成 tribe-level delta。
```

正确示例 B：

```text
tick 100：companion morale changed。
tick 101：companion trust changed。
```

P24 第一版建议：

```text
默认只输出 tribe-level morale_delta / trust_delta / safety_pressure / casualty_pressure。
暂不输出逐成员 TribeMemberEvent。
这样可以天然避开 P23 重复 member_id 问题。
```

如果实现者要输出逐成员事件，必须有测试证明：

```text
CombatCoordinationResult 生成的 TribeStateInputDraft 中 member_events 不含重复 member_id。
```

## 8. 枚举设计

### 8.1 GroupIntentKind

群体意图类型。

```cpp
enum class GroupIntentKind {
    Unknown,
    HoldTogether,
    AssistMember,
    DefendGroup,
    FocusThreat,
    ScreenRetreat,
    EscortVulnerable,
    AvoidEngagement,
    TestOnly
};
```

中文说明：

| 枚举 | 中文 | 含义 |
|---|---|---|
| Unknown | 未知 | 非法默认值 |
| HoldTogether | 保持队形 | 族群先不散开，保持最基础协作 |
| AssistMember | 援护成员 | 一个或多个成员帮助指定成员 |
| DefendGroup | 防御族群 | 优先保护整体安全 |
| FocusThreat | 集中压制威胁 | 多成员对同一威胁形成压力，但不结算伤害 |
| ScreenRetreat | 掩护撤退 | 部分成员掩护，整体降低风险后撤 |
| EscortVulnerable | 护送脆弱成员 | 保护儿童、学习者、受伤者、关键成员 |
| AvoidEngagement | 避免交战 | 威胁过高或协作过低时不硬拼 |
| TestOnly | 测试专用 | 正式 DTO 禁止 |

### 8.2 AssistActionKind

援护动作类型。

```cpp
enum class AssistActionKind {
    Unknown,
    Warn,
    GuidePosition,
    ShareTool,
    PullBack,
    CoverApproach,
    StabilizeMorale,
    TestOnly
};
```

中文说明：

| 枚举 | 中文 | 含义 |
|---|---|---|
| Warn | 预警 | 告诉成员危险方向 |
| GuidePosition | 引导站位 | 指引成员靠近安全位置 |
| ShareTool | 共享工具 | 临时把工具或知识用于协作，不转移所有权 |
| PullBack | 拉回 | 帮助成员脱离危险区域摘要 |
| CoverApproach | 掩护接近 | 让关键成员能靠近目标或撤离 |
| StabilizeMorale | 稳定士气 | 通过信任或领导力降低混乱 |

### 8.3 DefendActionKind

防御动作类型。

```cpp
enum class DefendActionKind {
    Unknown,
    GuardMember,
    GuardResource,
    BlockThreat,
    FormPerimeter,
    HoldLine,
    ScreenEscape,
    TestOnly
};
```

中文说明：

| 枚举 | 中文 | 含义 |
|---|---|---|
| GuardMember | 保护成员 | 保护指定成员 |
| GuardResource | 保护资源 | 保护食物、火源、工具等摘要目标 |
| BlockThreat | 阻挡威胁 | 阻挡或延缓威胁接近 |
| FormPerimeter | 形成外围 | 形成简单防御圈 |
| HoldLine | 稳住防线 | 不推进，只降低突破风险 |
| ScreenEscape | 掩护逃离 | 为撤退创造安全窗口 |

### 8.4 PackTacticKind

群体战术类型。

```cpp
enum class PackTacticKind {
    Unknown,
    None,
    PairSupport,
    GuardedGather,
    FocusPressure,
    RotatingDefense,
    EncircleThreat,
    RetreatScreen,
    TestOnly
};
```

中文说明：

| 枚举 | 中文 | 含义 |
|---|---|---|
| None | 无战术 | 协作不足，无法形成战术 |
| PairSupport | 双人互助 | 两人互相支援，是最小共同作战单元 |
| GuardedGather | 掩护采集 | 守护者保护采集者行动 |
| FocusPressure | 集中压力 | 多成员对同一威胁施压，不结算伤害 |
| RotatingDefense | 轮换防御 | 成员轮换承担压力，降低单点风险 |
| EncircleThreat | 包围威胁 | 形成包围倾向，但不做击杀结算 |
| RetreatScreen | 掩护撤退 | 一部分成员延缓威胁，整体撤退 |

### 8.5 CoordinationQuality

协作质量。

```cpp
enum class CoordinationQuality {
    Unknown,
    Failed,
    Weak,
    Stable,
    Strong,
    TestOnly
};
```

阈值建议：

```text
score < 0.20 -> Failed
score < 0.45 -> Weak
score < 0.70 -> Stable
score >= 0.70 -> Strong
```

## 9. 数据结构设计

### 9.1 ThreatPressureSummary

安全威胁摘要。

字段建议：

```text
threat_id: EntityId
threat_key: string
threat_pressure: double 0..1
proximity_pressure: double 0..1
target_pressure: double 0..1
known_counter_knowledge_ids: list<KnowledgeId>
reason_keys: list<string>
```

说明：

```text
threat_key 只能是安全摘要，例如 wolf_like_threat、spider_like_threat、unknown_predator。
不能写 real_damage、hidden_trait、true_hp、actual_poison 等真实属性。
known_counter_knowledge_ids 表示族群公开知道的应对知识，不表示世界真相。
```

### 9.2 CombatParticipant

共同作战参与者。

字段建议：

```text
member_id: EntityId
role: TribeMemberRole
available: bool
trust: double 0..1
morale: double 0..1
fatigue_pressure: double 0..1
injury_pressure: double 0..1
known_knowledge_ids: list<KnowledgeId>
reason_keys: list<string>
```

说明：

```text
CombatParticipant 是从 TribeState 中提取或由安全摘要提供的参与者视图。
不能包含真实战斗数值、隐藏属性或未被认知的对象真相。
```

校验：

```text
member_id 不为空。
role 不能 Unknown/TestOnly。
trust/morale/fatigue/injury 必须 0..1。
known_knowledge_ids 不重复。
reason_keys 不含 forbidden key。
```

### 9.3 GroupIntent

群体意图。

字段建议：

```text
kind: GroupIntentKind
priority: double 0..1
target_threat_id: optional<EntityId>
target_member_id: optional<EntityId>
participant_ids: list<EntityId>
summary_key: string
reason_keys: list<string>
```

校验：

```text
kind 不能 Unknown/TestOnly。
priority 必须 0..1。
participant_ids 不允许重复。
AssistMember / EscortVulnerable 必须有 target_member_id。
DefendGroup / FocusThreat / ScreenRetreat 建议有 target_threat_id。
summary_key 不为空且不能包含 forbidden key。
```

### 9.4 AssistAction

援护动作。

字段建议：

```text
kind: AssistActionKind
actor_member_id: EntityId
target_member_id: EntityId
support_score: double 0..1
risk_reduction: double 0..1
trust_delta_hint: double -1..1
morale_delta_hint: double -1..1
summary_key: string
reason_keys: list<string>
```

说明：

```text
AssistAction 表示“谁帮谁、帮到什么程度”。
它不是治疗、不是加血、不是技能释放。
```

### 9.5 DefendAction

防御动作。

字段建议：

```text
kind: DefendActionKind
actor_member_id: EntityId
protected_member_id: optional<EntityId>
protected_key: optional<string>
threat_id: optional<EntityId>
defense_score: double 0..1
risk_reduction: double 0..1
summary_key: string
reason_keys: list<string>
```

说明：

```text
DefendAction 表示防御意图和安全收益摘要。
它不结算格挡、不结算伤害、不决定死亡。
```

### 9.6 PackTactic

群体战术。

字段建议：

```text
kind: PackTacticKind
quality: CoordinationQuality
participant_ids: list<EntityId>
focus_threat_id: optional<EntityId>
coordination_score: double 0..1
risk_reduction: double 0..1
morale_delta_hint: double -1..1
trust_delta_hint: double -1..1
summary_key: string
reason_keys: list<string>
```

说明：

```text
PackTactic 是 P24 的核心产物。
它告诉后续系统“族群形成了什么协作形态”。
它不告诉后续系统“谁被打死了、怪物掉了什么”。
```

### 9.7 MemberCoordinationSummary

成员协作汇总。

字段建议：

```text
member_id: EntityId
participated: bool
assist_count: int
defend_count: int
received_assist_count: int
coordination_score: double 0..1
morale_delta_hint: double -1..1
trust_delta_hint: double -1..1
reason_keys: list<string>
```

重要：

```text
这是 P24 为遵守 P23 重复 member_id 约束而必须保留的聚合层。
同一成员在本 tick 内所有援护、防御、被援护结果都先合并到这里。
如果未来要生成 P23 member event，只能从这里为每个 member_id 生成最多一个事件。
```

### 9.8 CombatCoordinationInput

共同作战输入。

字段建议：

```text
tribe_id: TribeId
input_tick: Tick
tribe_state: TribeState
threats: list<ThreatPressureSummary>
requested_intent: optional<GroupIntentKind>
resource_pressure: double 0..1
safety_pressure: double 0..1
reason_keys: list<string>
```

说明：

```text
第一版直接传入 TribeState，避免 P24 自己重建族群状态。
threats 是安全威胁摘要，不是 raw GameState。
requested_intent 是上游希望尝试的协作方向；resolver 可以根据状态降级或拒绝。
```

校验：

```text
tribe_id 必须和 tribe_state.tribe_id 一致。
tribe_state 必须 validateBasic 通过。
threats 的 threat_id 不重复。
ratio 字段必须 0..1。
reason_keys 不含 forbidden key。
```

### 9.9 CombatCoordinationOptions

共同作战选项。

字段建议：

```text
morale_weight: double 0..1
trust_weight: double 0..1
role_weight: double 0..1
threat_weight: double 0..1
split_risk_penalty_weight: double 0..1
```

默认建议：

```text
morale_weight = 0.25
trust_weight = 0.25
role_weight = 0.25
threat_weight = 0.15
split_risk_penalty_weight = 0.10
```

说明：

```text
options 只能影响确定性评分权重。
不能加入 random_seed。
不能加入 difficulty_hidden_truth。
```

### 9.10 CombatCoordinationStateDraft

给 P23 的族群状态影响草稿。

字段建议：

```text
tribe_id: TribeId
input_tick: Tick
morale_delta: double -1..1
trust_delta: double -1..1
safety_pressure: double 0..1
casualty_pressure: double 0..1
knowledge_conflict_pressure: double 0..1
member_summaries: list<MemberCoordinationSummary>
reason_keys: list<string>
```

说明：

```text
这不是直接状态修改。
它用于构造或提示后续的 TribeStateInput。
第一版可以不生成 member_events，只提供 tribe-level delta。
如果后续生成 member_events，必须从 member_summaries 聚合后生成，且 member_id 唯一。
```

### 9.11 CombatCoordinationProjection

共同作战投影。

字段建议：

```text
tribe_id: TribeId
intent_summary: string
tactic_summary: string
participant_summary: string
quality_summary: string
risk_summary: string
warning_keys: list<string>
```

说明：

```text
Projection 给 H5、Agent、训练环境看摘要。
不要塞完整战斗计算过程。
不要暴露隐藏真相。
```

### 9.12 CombatCoordinationTrace

共同作战解释链。

字段建议：

```text
trace_key: string
input_tick: Tick
steps: list<string>
matched_rule_keys: list<string>
reason_keys: list<string>
```

用于解释：

```text
为什么选择防御而不是集火。
为什么协作质量高或低。
为什么士气/信任产生变化草稿。
为什么出现撤退建议。
```

### 9.13 CombatCoordinationResult

共同作战结果。

字段建议：

```text
decision: string
intent: GroupIntent
assist_actions: list<AssistAction>
defend_actions: list<DefendAction>
pack_tactic: PackTactic
member_summaries: list<MemberCoordinationSummary>
state_draft: CombatCoordinationStateDraft
projection: CombatCoordinationProjection
trace: CombatCoordinationTrace
warning_keys: list<string>
reason_keys: list<string>
```

`decision` 允许值：

```text
coordinated
weak_coordination
avoid_engagement
no_available_member
invalid
```

## 10. 服务与接口

### 10.1 CombatCoordinationResolver

核心入口：

```cpp
class CombatCoordinationResolver {
public:
    Result<CombatCoordinationResult> resolve(
        const CombatCoordinationInput& input,
        const CombatCoordinationOptions& options = {}) const;
};
```

职责：

```text
校验输入。
```

## 11. 核心规则建议

### 11.1 成员可用性

成员参与共同作战必须满足：

```text
active member。
available=true。
role 合法。
morale > 0。
trust > 0。
```

第一版不做体力消耗，只用 `fatigue_pressure` 和 `injury_pressure` 降低评分。

### 11.2 角色贡献

建议基础贡献：

| Role | 中文 | 贡献倾向 |
|---|---|---|
| Pioneer | 先驱者 | 综合协调、引导意图 |
| Explorer | 探索者 | 预警、引导站位、掩护撤退 |
| Gatherer | 采集者 | 守护采集、资源保护 |
| Guardian | 守护者 | 防御、阻挡威胁、保护成员 |
| Teacher | 教师 | 稳定士气、传达知识 |
| Learner | 学习者 | 低贡献，但需要保护，可参与简单互助 |
| Healer | 治疗者 | 第一版不治疗，只降低伤亡压力摘要 |
| Crafter | 工匠 | 工具协作、共享工具 |
| Elder | 长者 | 稳定信任和士气 |
| Child | 儿童 | 不能作为主动战斗成员，优先被护送 |

注意：

```text
Guardian 不等于一定能打赢。
Healer 不等于本阶段有治疗系统。
Crafter 不等于本阶段有装备系统。
Child 不能被安排为主动防御者。
```

### 11.3 协作评分

建议公式：

```text
base_member_score = clamp(
    role_score * role_weight
  + member.morale * morale_weight
  + member.trust * trust_weight
  + (1 - fatigue_pressure) * 0.10
  + (1 - injury_pressure) * 0.10
)
```

群体评分：

```text
coordination_score = clamp(
    average(member_score)
  + role_coverage_bonus
  - tribe.split_risk.risk * split_risk_penalty_weight
  - input.safety_pressure * 0.10
)
```

角色覆盖加成建议：

```text
有 Pioneer 或 Elder：+0.05
有 Guardian：+0.10
有 Explorer：+0.05
有 Teacher：+0.05
成员数 >= 3：+0.05
```

### 11.4 意图选择

建议确定性规则：

```text
没有可用成员 -> no_available_member。
威胁压力高且协作评分低 -> AvoidEngagement / ScreenRetreat。
有 Child/Learner 且威胁压力中高 -> EscortVulnerable。
有 Guardian 且威胁压力中高 -> DefendGroup。
有多个可用成员且协作评分高 -> FocusThreat。
默认 -> HoldTogether。
```

### 11.5 状态影响草稿

建议：

```text
Strong coordination：morale_delta +0.05，trust_delta +0.05，safety_pressure 降低。
Stable coordination：morale_delta +0.02，trust_delta +0.02。
Weak coordination：morale_delta -0.02，trust_delta -0.01。
Failed coordination：morale_delta -0.05，trust_delta -0.05，safety_pressure 升高。
Avoid engagement：不算失败；如果威胁过高，合理撤退可以小幅提高 trust。
```

重要：

```text
casualty_pressure 只能作为摘要压力，不表示具体成员受伤或死亡。
如果未来 P25/P28 产生真实伤亡摘要，也必须由对应阶段输出安全事件，再进入 P23。
```

```text

true_trait
object_truth
raw_state
GameState
SaveGame
AgentRuntime
Policy

damage_roll
critical_hit
true_hp
actual_damage
kill_result
loot_drop
war_result
```

说明：

```text

P24 可以说“协作降低风险”。
P24 不能说“随机命中后造成 7 点伤害”。
```

```text

ratio < 0 或 > 1。
delta < -1 或 > 1。
Unknown/TestOnly 枚举进入正式 DTO。

P24 结果中出现 damage / death / kill / loot 语义。
```
