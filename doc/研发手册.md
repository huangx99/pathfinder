# 先驱者：研发手册

本文档是后续所有研发、修复、扩展、前端改造、AI 接力开发的硬性手册。

它的优先级：

```text
用户明确要求 > 当前阶段设计文档 > 本研发手册 > 既有代码风格 > 临时实现方便
```

如果本手册和某个阶段设计文档冲突，必须先暂停研发，写清冲突原因，等待审核或补充设计，不能自行选择更方便的做法。

---

## 1. 手册目标

本手册解决三个问题：

```text
1. 后续研发不能污染引擎核心。
2. 所有玩法必须走既有管线和权威接口。
3. 前端只能作为客户端表现层，不能偷偷写规则、知识、状态或资源变化。
```

项目当前已经进入 V2 开放世界阶段，系统越来越大，后续研发者可能是低上下文 AI，也可能只看当前阶段文档。因此必须把“怎么写才不破坏架构”固定下来。

本手册不是玩法设计文档，也不是某个阶段任务卡。它是所有阶段共用的研发纪律。

---

## 2. 一句话总原则

后续所有功能都必须遵守：

```text
玩家/前端提交意图；
Command 管线裁决行为；
规则、配置、条件、知识、Agent 决策由后端权威系统处理；
Runtime / Inventory / Region / Knowledge 等专属模块修改状态；
Projection / Patch / Event 把结果给前端展示；
前端永远不自己生成规则结果。
```

任何不符合这条链路的实现，都默认不允许合入。

---

## 3. 研发前必须确认

每次开始一个新阶段或修复前，研发者必须先回答：

```text
1. 本阶段对应哪个设计文档？
2. 本阶段新增的是玩法、基础设施、协议、前端、测试，还是内容？
3. 本阶段要接入哪些已有系统？
4. 本阶段不能破坏哪些边界？
5. 本阶段是否需要新增 Command、DTO、枚举、接口、配置或测试？
6. 本阶段有没有可能被前端误写成规则？
7. 本阶段有没有可能绕过 Inventory / Runtime / Knowledge / Region 等权威模块？
```

如果答不清楚，不能写代码。

## 4. 目录边界

### 4.1 后端引擎目录

后端引擎在：

```text
backend/include/pathfinder/
backend/src/
backend/tests/
content/
```

后端引擎负责：

```text
规则裁决。
世界状态。
物品归属。
资源节点。
制作反应。
知识、认知、记忆、教学。
Agent 推理和行动。
Command 管线。
客户端安全投影。
存档、复盘、诊断。
```

禁止：

```text
为了某个前端按钮在引擎里写临时分支。
为了某个 demo 在引擎里写内容硬编码。
为了通过测试直接改 runtime map/vector。
为了表现效果把 UI 字段塞进核心规则 DTO。
```

### 4.2 前端客户端目录

当前客户端在：

```text
app/frontend/client/
```

未来如果新增其他客户端，必须单独开目录，例如：

```text
app/frontend/mobile_client/
app/frontend/desktop_client/
app/frontend/editor_client/
```

前端负责：

```text
输入采集。
地图显示。
角色和物品表现。
动画、音效、反馈。
命令选项展示。
事件日志展示。
协议请求和响应处理。
```

前端禁止：

```text
自行判断一个行为是否成功。
自行增减背包、地图实体、资源数量。
自行生成知识、记忆、NPC 决策。
自行把红果、狼、火把等内容写成玩法规则。
绕过 available_commands 构造非法 command。
绕过 projection 显示 hidden truth。
```

### 4.3 扩展层目录

后续新增的“可替换玩法内容、内容包、后端扩展适配”必须优先放在：

```text
ext/
```

`ext/` 的目标是让项目形成清晰分层：

```text
backend/ = 引擎核心和通用系统。
content/ = 当前核心内容包或被引擎测试依赖的基础内容。
app/frontend/ = 客户端壳和通用渲染/交互代码。
ext/ = 后续可删除、可重做、可替换的后端内容扩展和扩展适配。
```

删除 `ext/` 后，必须满足：

```text
引擎核心仍能编译。
核心测试仍能运行。
基础客户端不受 ext/ 删除影响；前端资源是否删除由前端目录自己决定。
不会丢失 backend/ 的通用规则能力。
不会让 content/core 的基础内容引用断裂。
```

适合放入 `ext/`：

```text
新物品内容包。
新资源节点内容包。
新地形/生态/怪物内容包。
新剧情、任务、区域事件内容包。
非核心玩法 mod 的后端内容和注册适配。
某个扩展玩法的 adapter、注册表或桥接代码。
```

不适合放入 `ext/`：

```text
WorldCommandPipeline 本体。
WorldRuntime / Inventory / Knowledge / Region 等权威核心。
客户端协议核心 DTO。
基础错误码、Result、StrongId。
必须被所有内容复用的通用引擎能力。
```

如果一个扩展发现必须改引擎，正确流程是：

```text
1. 先写补充设计，说明为什么 ext/content 配置不够。
2. 在 backend/ 增加小而通用的 engine port / service / DTO。
3. 在 ext/ 里使用这个通用能力实现具体内容。
4. 补测试证明删除 ext/ 后 engine 仍干净。
```

禁止：

```text
为了一个 ext 玩法把专属 if 写进 backend 核心。
让 backend 核心 include ext/ 下的具体内容头文件。
让核心测试依赖 ext/ 里的非基础内容。
删除 ext/ 后 backend 编译失败。
```

### 4.4 文档目录

设计文档在：

```text
doc/
```

审核报告在：

```text
doc/review/
```

审核报告命名使用中文，例如：

```text
doc/review/P59世界区块状态缓存与恢复实施审核报告.md
```

除非用户明确要求英文附件名，否则项目内报告默认中文文件名。

## 5. 后端总管线铁律

### 5.1 玩家行为必须走 Command

所有玩家、NPC、野兽、系统行为，只要会改变世界状态，都必须进入 Command 或系统等价入口。

标准链路：

```text
ClientRequest
-> ClientCommandGateway
-> WorldCommandDto / option_id materialize
-> WorldCommandPipeline
-> 校验 actor/session/authority
-> 对应规则服务执行
-> Runtime/Inventory/Knowledge/Region 状态变化
-> StateDelta / Event / Experience
-> ProjectionPatch
-> ClientCommandResponse
```

禁止：

```text
前端直接调用某个 runtime 函数改状态。
HTTP Gateway 直接修改地图、背包或知识。
Client adapter 自己创造规则结果。
测试为了方便绕过 Command 后还声称验证了玩法链路。
```

### 5.2 Command 不是万能垃圾桶

Command 负责表达“意图”，不是把所有系统细节都塞进去。

正确做法：

```text
Move：表达移动意图。
Harvest：表达采集意图。
Craft：表达制作意图。
Teach：表达教学意图。
Use：表达对目标使用某物的意图。
```

错误做法：

```text
Command 里直接传最终产物。
Command 里直接传知识结论。
Command 里直接传 NPC 决策结果。
Command 里直接传资源节点最终剩余次数。
Command 里直接传前端猜出来的规则命中结果。
```

Command 只能带玩家/系统可表达的输入，不能带隐藏真相和裁决结果。

## 6. 状态权威边界

### 6.1 WorldRuntime 权威

`WorldRuntime` 是当前激活世界状态的权威。

它负责：

```text
cell。
entity 当前地图位置。
actor 当前状态。
resource node runtime。
exploration runtime。
region active 标记。
```

禁止：

```text
其他模块直接持有一份 parallel world state。
前端 projection 被当成 runtime snapshot。
为了恢复状态直接清空 runtime 再盲目 apply。
```

### 6.2 P45 物品归属权威

物品在哪里，必须由 P45 归属系统裁决。

合法归属包括：

```text
OnMap。
InInventory。
InContainer。
Equipped。
Consumed / Destroyed / Nowhere。
```

禁止：

```text
region snapshot 复制 InInventory 物品。
制作系统直接往数组 push 产物。
采集系统绕过 location port 生成地面物。
生成、采集、掉落、制作、拾取产物时绕过 runtime / P45 归属接口。
直接修改 cells_、entities_、resource_nodes_、inventory entries 等 runtime 内部数组或 map 来伪造 OnMap / InInventory。
前端自己把地上物移入背包。
```

硬性要求：

```text
任何新物品、采集产物、掉落物、制作产物、NPC 生成物，都必须通过 runtime 暴露的权威接口和 P45 归属接口建立 OnMap、InInventory、InContainer、Equipped、Consumed、Destroyed 或 Nowhere 状态。
测试代码也不得为了方便直接改 runtime 内部容器；如果缺少权威接口，必须先补通用接口和测试，再接玩法。
```

### 6.3 P47 资源节点权威

资源节点状态由资源采集系统和 runtime 节点状态负责。

必须保留：

```text
remaining_charges。
max_charges。
node_state_str。
required_action_key。
required_tool_key。
output_entity_keys。
```

禁止：

```text
区块重新生成导致采过的树/矿/植物回满。
前端根据图标判断资源能不能采。
采集结果绕过经验和知识链。
```

### 6.4 P48 制作与反应权威

制作和对象反应必须由后端内容、规则和反应系统裁决。

正确链路：

```text
Craft/React Command
-> ContentRegistry reaction/crafting definition
-> 条件/材料/工具校验
-> P45 消耗材料
-> P45 生成产物归属
-> Experience/Event/Patch
```

禁止：

```text
前端看到木头和火种就自己显示“获得火把”。
后端为了火把写专属 if，不走反应定义。
产物不经过归属系统。
```

### 6.5 P49-P50 知识教学权威

知识形成和传播必须走认知、记忆、知识、教学系统。

正确链路：

```text
WorldExperience
-> LearningLoop / Knowledge service
-> actor 私有 KnowledgeClaim
-> Teach Command
-> recipient 自己形成/接收知识
```

禁止：

```text
前端显示知识时顺便把 NPC 教会。
把玩家知识全局共享给所有 NPC。
复制 claim 但不改 owner。
NPC 行为直接读玩家知识。
```

### 6.6 P51-P52 Agent 权威

NPC、野兽、同伴都属于 Agent 或可行动生物，不是按钮脚本。

Agent 行动必须来自：

```text
观察。
目标。
知识。
需求。
风险。
可用 action space。
Command 编译。
WorldCommandPipeline 执行。
```

禁止：

```text
前端定时让同伴使用火把。
遇到狼就写死 companion_do_torch。
野兽不是 Agent，只是全局事件按钮。
NPC 行动绕过 CommandPipeline。
```

### 6.7 P59 区块状态权威

P59 负责缓存和恢复区块状态，但不能替代世界规则。

正确链路：

```text
Command 改 WorldRuntime
-> RegionLifecycle seal active region
-> SnapshotBuilder 读取 runtime slice
-> IWorldRegionStateStore 保存 snapshot
-> ensureAvailable 优先 active，再 restore，再 generate
```

禁止：

```text
P59 重新生成区块来冒充恢复。
P59 复制 Inventory 物品回地图。
P59 用旧 snapshot 覆盖 active runtime。
P59 绕开 P58 形成第二套生成入口。
```

## 7. 内容与配置开发规范

### 7.1 内容优先走 JSON / ContentRegistry

新增物品、资源、地形、反应、知识模板，默认应优先考虑内容配置。

适合放配置：

```text
普通物品定义。
资源节点定义。
制作/反应配方。
知识模板。
显示名 key。
基础数值。
地形标签。
```

适合写代码：

```text
新的通用规则类型。
新的条件函数。
新的执行器。
新的 runtime 状态结构。
新的 projection adapter。
```

如果只是新增“毒蘑菇”“普通蘑菇”“铁矿”“木门”，不应该改引擎核心。

### 7.2 不允许内容硬编码

禁止在核心服务里写：

```cpp
if (entity_key == "red_berry")
if (target_key == "wolf")
if (item_key == "torch")
```

除非这是测试用例或明确的临时兼容层，并且必须有注释说明何时移除。

正确做法：

```text
通过 tag、trait、content definition、condition expression、reaction definition 判断。
```

### 7.3 新增内容的最小流程

新增一个可玩法对象时，必须考虑：

```text
1. content 定义。
2. 是否会生成在地图上。
3. 是否能被采集/拾取/使用/制作。
4. 是否会形成经验。
5. 是否有知识模板。
6. 是否会被 NPC 理解和使用。
7. 是否需要前端精灵。
8. 是否需要测试。
```

不能只加一个图标或一个 if。

### 7.4 扩展内容默认进 ext

从本手册生效后，新增玩法内容默认按下面顺序判断：

```text
能只靠 JSON / 配置表达 -> 放 ext/content/。
需要扩展注册或 adapter -> 放 ext/backend/，但只能依赖 backend 暴露的通用 port。
前端表现不受 ext/ 约束，按前端章节和实际客户端目录管理。
需要改引擎通用能力 -> 先设计，再改 backend/，具体内容仍放 ext/。
```

举例：

```text
毒蘑菇、普通蘑菇、铁矿、野猪、草药 -> ext/content/。
毒蘑菇像素图、野猪行走动画 -> 不受 ext/ 约束，按前端客户端目录规范放置。
魔法世界内容包、机械危机内容包、高级文明内容包 -> ext/content/packs/。
魔法区域效果如果只是内容参数 -> ext/content/。
魔法区域效果如果需要通用 AreaEffect 引擎 -> backend/ 先加通用 AreaEffect，再由 ext/content/ 配置具体魔法。
```

删除 `ext/` 应该表示“删除后端扩展内容和扩展适配”，不应该表示“删除引擎能力”；前端资源是否删除不由 ext/ 规则约束。

必须明确：从本手册生效后，后端内容扩展默认都写在 `ext/`，但 `ext/` 不是第二套引擎。它只能承载内容、内容包、扩展注册适配和扩展测试；所有状态变更、命令执行、反应结算、知识学习、Agent 决策仍然必须走 `backend/` 的通用架构。

当前边界：`ext/content/` 目前是目录规范和职责边界，不代表 JSON 自动加载已经实现。未实现 ext content package loader 前，可以先通过 ContentDraftRegistry / ContentRegistry 把扩展内容接入测试或开发流程；后续如果要做到“把 JSON 放进 ext/content/ 就自动生效”，必须单独设计加载器、校验器、冲突检测、版本兼容和回归测试。

### 7.5 新增物品与反应必须能进入知识闭环

新增一个物品、反应规则或内容包时，不能只验证“产物生成了”，还必须验证它是否进入完整玩法闭环：

```text
ContentRegistry -> Command -> Reaction/Runtime -> Experience -> Knowledge Learning -> Teaching/Agent 可用
```

最低验收要求：

```text
1. 新物品必须通过 content definition 进入 ContentRegistry。
2. 新反应必须通过 reaction definition 表达输入、消耗、输出、效果和知识模板。
3. 玩家或 Agent 的操作必须通过 WorldCommandPipeline 执行，不能直接调用反应服务跳过 Command。
4. 反应成功后必须产生 WorldExperienceDto，不能只改背包或地图状态。
5. 经验必须能被 WorldKnowledgeLearningService 学习成知识。
6. 如果知识声明为 teachable / usable_by_ai，后续教学和 Agent 使用不能额外写专属 if。
7. 测试必须覆盖至少一条“新增内容 -> Command -> 反应 -> 经验 -> 知识”的链路。
```

当前已有验证样例：

```text
backend/tests/integration/world_learning/world_learning_integration_test.cpp
测试名：world_learning_ext_item_reaction_command_to_knowledge
```

该测试证明：只要扩展内容进入 ContentRegistry，就能严格走 Command、反应、经验和知识学习链路。

验收时不得把“测试里手动构造 ContentRegistry”误判为“ext/content/ JSON 已自动加载”。两者是不同能力，必须分开验收。

## 8. 引擎扩展规则

### 8.1 什么时候允许改引擎

只有下面情况允许扩展引擎：

```text
现有 Command 无法表达新意图。
现有 Runtime 没有必要状态字段。
现有 Inventory/Region/Knowledge port 无法安全表达状态变化。
现有 ContentRegistry 无法表达新通用规则。
现有 Projection 无法安全展示必要信息。
现有测试无法覆盖新增通用能力。
```

### 8.2 改引擎必须先做设计

改引擎前必须有设计说明，至少写清：

```text
为什么不能只改配置。
新增模块属于哪个系统。
新增 DTO/枚举/接口。
接入哪条管线。
状态权威在哪里。
如何测试不破坏旧系统。
```

如果没有设计，不能直接改核心代码。

### 8.3 引擎扩展必须小而通用

正确：

```text
新增 ToolDurability 系统，所有工具可复用。
新增 AreaEffect 系统，火焰、毒雾、魔法区域可复用。
新增 Container port，箱子、背包、仓库可复用。
```

错误：

```text
新增 AxeGetsDullSpecialCase。
新增 WolfAfraidOfTorchSpecialCase。
新增 RedBerryPoisonCounterSpecialCase。
```

## 9. 客户端协议与前端开发规范

### 9.1 前端只能走协议

前端必须通过客户端协议与后端交互。

合法输入：

```text
bootstrap。
refresh。
command option_id。
command DTO（仅在后端允许的安全范围内）。
reset / debug request（仅调试环境）。
```

合法输出来源：

```text
ClientWorldProjection。
ProjectionPatch。
WorldEventDto。
WorldCommandOptionDto。
CommandResponse。
```

前端不允许直接读：

```text
content/core JSON 来生成规则按钮。
backend runtime 内部结构。
KnowledgeClaim 原始数据。
hidden truth。
世界生成器内部 seed 结果。
```

### 9.2 available_commands 是行动按钮权威

玩家能做什么，前端必须以 `available_commands` 为准。

前端可以做：

```text
把 available_commands 分组。
把 option 显示成中文按钮。
把移动类 option 显示成点击地图、路径或快捷按钮。
把失败 response 展示给玩家。
```

前端不能做：

```text
看到相邻格就自己构造 Move。
看到物品就自己构造 Pickup。
看到材料就自己构造 Craft。
没有 option 时自己猜一个 command。
```

### 9.3 前端状态只能是表现状态

前端可以保存：

```text
镜头位置。
缩放比例。
选中格子。
hover 对象。
路径预览。
动画进度。
粒子效果。
本地日志滚动位置。
```

前端不能保存为权威：

```text
玩家真实坐标。
NPC 真实坐标。
背包数量。
资源剩余次数。
知识状态。
区块是否生成。
区块是否缓存。
```

这些必须以后端 projection 为准。

## 10. 前端目录与文件规范

### 10.1 前端必须和引擎隔离

前端代码只能放在：

```text
app/frontend/client/
```

或未来新增的独立客户端目录。

禁止：

```text
为了 UI 方便改 backend 核心 DTO 字段。
为了动画效果改世界规则。
为了显示一个按钮在后端写前端专用规则。
```

如果前端确实缺信息，正确流程是：

```text
前端提出缺少的安全展示字段
-> 设计 projection/patch 扩展
-> 后端 adapter 输出只读字段
-> 前端展示
```

### 10.2 精灵必须单独文件

新增一个精灵，必须新增或修改对应单独文件：

```text
app/frontend/client/sprites/<sprite_key>.js
```

例如：

```text
app/frontend/client/sprites/player.js
app/frontend/client/sprites/wolf.js
app/frontend/client/sprites/loose_stone.js
```

禁止：

```text
把大量精灵画法写进 app.js。
把多个无关精灵塞进一个巨大 sprites.js。
把动画帧散落在不同文件。
```

### 10.3 精灵动画写在同一个精灵文件

一个精灵的静态图、行走动画、方向动画、受击动画、使用动画，必须尽量写在同一个文件里。

推荐结构：

```javascript
(() => {
  const size = 48;

  function make(draw) { ... }
  function block(ctx, fill, x, y, w, h) { ... }

  function drawIdle() { ... }
  function drawWalkRight(step) { ... }
  function drawWalkDown(step) { ... }
  function drawUse() { ... }

  const walk = {
    down: [drawWalkDown(0), drawWalkDown(-2), drawWalkDown(2)],
    up: [...],
    left: [...],
    right: [...],
  };

  window.SPRITES = window.SPRITES || {};
  window.SPRITES.example = drawIdle();
  window.SPRITES.exampleWalk = walk;
})();
```

禁止：

```text
动画帧在 app.js 里临时拼。
某个方向缺帧导致移动时角色看不出方向。
同一精灵的 idle/walk/use 分散在多个文件，后续难维护。
```

### 10.4 精灵注册规则

每个精灵文件必须只负责注册自己的 key。

推荐：

```javascript
window.SPRITES = window.SPRITES || {};
window.SPRITES.wolf = idleCanvas;
window.SPRITES.wolfWalk = walkFrames;
```

命名约定：

```text
基础图：window.SPRITES.<sprite_key>
行走帧：window.SPRITES.<sprite_key>Walk
使用帧：window.SPRITES.<sprite_key>Use
受击帧：window.SPRITES.<sprite_key>Hit
死亡帧：window.SPRITES.<sprite_key>Dead
```

### 10.5 index.html 引入规则

新增精灵文件后，需要在客户端入口引入：

```text
app/frontend/client/index.html
```

但引入顺序必须清晰：

```text
地形精灵。
物品/资源精灵。
角色/NPC/野兽精灵。
特效精灵。
主 app.js。
```

禁止把精灵文件放在 `app.js` 后面，避免运行时注册晚于使用。

### 10.6 前端映射不能变成规则

前端可以做显示映射：

```text
entity_key -> sprite_key。
terrain_key -> terrain sprite。
display_name_key -> i18n 文本。
```

但这种映射只能决定“怎么显示”，不能决定“能不能做”。

错误：

```javascript
if (entity_key.includes('berry')) canEat = true;
if (entity_key.includes('wolf')) showAttackButton();
```

正确：

```text
后端 available_commands 给 Eat option，前端显示“吃”。
后端 available_commands 给 Attack option，前端显示“攻击”。
```

## 11. UI / UX 研发规范

### 11.1 UI 不是调试台

正式玩家界面必须先回答：

```text
玩家现在在哪里？
周围有什么重要对象？
玩家现在能做什么？
为什么这个动作失败？
刚刚发生了什么？
哪个 NPC/野兽正在行动？
```

禁止把 backend DTO 原样摊给玩家。

### 11.2 反馈顺序必须清楚

一次操作后的反馈顺序建议：

```text
1. 玩家输入反馈。
2. 命令执行状态。
3. 世界变化。
4. 角色/NPC/资源变化。
5. 知识/经验变化。
6. 可继续行动提示。
```

不能让玩家只看到一堆日志，不知道世界发生了什么。

### 11.3 客户端可以优化交互，但不能改规则

允许：

```text
双击格子寻路。
地图拖拽。
缩放。
镜头跟随开关。
路径预览。
选中对象详情。
像素粒子。
行走补间动画。
```

禁止：

```text
前端寻路成功就直接移动角色坐标。
前端路径越过后端不可通行格。
前端为了手感跳过 command response。
```

前端路径可以是“计划”，实际移动必须逐步走后端命令。

## 12. 测试规范

### 12.1 每个阶段必须有专项测试

新增模块必须配套测试：

```text
DTO 校验测试。
枚举往返测试。
核心服务单元测试。
与前置系统集成测试。
失败路径测试。
架构扫描测试。
```

只测 happy path 不够。

### 12.2 测试不能假通过

禁止：

```text
测试遇到没有资源就 skipped。
测试封存 A 区块却 detach B 区块。
测试只断言函数返回 ok，不断言状态真的变了。
测试绕过正式管线，却声称验证了正式玩法。
```

必须断言：

```text
状态变化前是什么。
执行后是什么。
离开/恢复后是否保真。
失败时是否没有污染状态。
事件/patch/experience 是否符合预期。
```

### 12.3 必须有反作弊式测试

每个阶段至少考虑：

```text
非法 actor。
非法 owner。
非法 option_id。
旧 projection_version。
缺材料。
目标不存在。
跨 region。
重复执行。
失败后重试。
```

越是底层系统，越要测失败路径。

## 13. 审核与复审规范

### 13.1 审核报告位置

所有审核报告写入：

```text
doc/review/
```

文件名用中文：

```text
Pxx阶段名实施审核报告.md
Pxx阶段名复审报告.md
```

### 13.2 审核不能只看测试通过

审核必须同时看：

```text
设计文档要求。
代码实现路径。
测试是否真的覆盖。
是否绕过管线。
是否污染引擎。
是否硬编码内容。
是否破坏前后端边界。
```

测试通过但测试本身误测，必须判不通过。

### 13.3 审核结论必须明确

只能使用：

```text
通过。
不通过，需修复后复审。
有条件通过，遗留项进入后续阶段。
```

不能写模糊结论。

## 14. 常见错误实现清单

后续研发如果出现以下情况，默认驳回：

```text
1. 在前端写规则判断。
2. 在引擎核心写内容硬编码。
3. 绕过 CommandPipeline 修改世界。
4. 绕过 P45 修改物品归属。
5. 绕过 P47 修改资源节点。
6. 绕过 P49/P50 直接给 NPC 知识。
7. 用 projection 当 runtime state。
8. 用旧 snapshot 覆盖 active runtime。
9. 为一个 demo 写不可扩展分支。
10. 测试只断言 ok，不断言状态。
11. 把多个精灵和动画塞进 app.js。
12. 为了 UI 手感跳过后端响应。
13. 因为低级 AI 上下文小，就省略设计边界。
```

## 15. 新功能开发检查单

每次新增功能前，复制这份清单自检：

```text
[ ] 有对应设计文档或补充设计。
[ ] 确认这是配置扩展还是引擎扩展。
[ ] 新增可替换内容默认放入 `ext/`，不污染 backend/content core。
[ ] 如果改引擎，说明为什么配置不够。
[ ] 如果改引擎，确认具体玩法内容仍留在 `ext/` 或内容包中。
[ ] 所有状态变化都走对应权威模块。
[ ] 玩家/NPC/系统行为进入 Command 或等价系统入口。
[ ] 前端只消费 projection/events/options，不裁决规则。
[ ] 新增内容没有写死到核心服务。
[ ] 新增精灵在独立 sprite 文件，动画同文件维护。
[ ] 新增协议字段是安全投影，不泄露 hidden truth。
[ ] 有专项测试和失败路径测试。
[ ] 有审核报告或自检报告。
```

## 16. 给低上下文 AI 的硬性提示

如果你是接手本项目的研发 AI，必须先读：

```text
doc/研发手册.md
doc/00_设计文档编写要求.md
doc/程序设计计划.md
当前阶段任务卡设计文档
相关 review 报告
```

然后才能写代码。

你不能说“为了简单先写死”。本项目允许小步，但不允许错误方向的小步。

正确的小步是：

```text
少做功能，但边界正确。
```

错误的小步是：

```text
功能看起来能跑，但绕过管线、污染引擎、前端写规则。
```

## 17. 最终铁律

后续所有研发都以这句话收口：

```text
引擎只做通用规则和权威状态；
后端可替换扩展默认放在 ext/，删除 ext/ 不应污染引擎；
内容通过配置和注册扩展；
玩法通过 Command 管线进入；
知识通过学习链形成；
Agent 通过目标和知识行动；
前端只做输入、展示和表现；
测试必须证明真实链路，而不是证明 demo 能跑。
```
