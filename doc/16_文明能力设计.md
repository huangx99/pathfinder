# 16_文明能力设计

## 1. 设计目标

文明能力系统负责把长期稳定的知识、族群行为、资源条件、组织能力和可复现结果，转化为族群真正拥有的稳定能力。

本章解决的问题：

```text
族群是否真的掌握某种能力。
能力为什么可以获得。
能力处于什么成熟度。
能力给规则系统带来什么影响。
能力是否会退化、暂停或冲突。
族群整体文明阶段如何由能力组合推导。
```

核心原则：

```text
文明能力不是科技树按钮。
文明能力不是 tribe_level。
文明能力不是单次行为成功。
文明能力不是配置直接解锁。
文明能力必须能追溯到知识、行为、资源、传播和族群状态证据。
```

文明能力的定义：

```text
文明能力 = 知识证据 + 族群行为 + 资源条件 + 稳定传播 + 长期可复现结果。
```

## 2. 不解决的问题

本章不解决：

| 不解决的问题 | 原因 | 所属后续系统 |
|---|---|---|
| 完整科技树 UI | 本章只设计规则层和数据层 | 前端协议 / UI 系统 |
| 具体数值平衡 | 只定义结构和约束 | 配置和调参阶段 |
| 现代科技、国家政治、经济制度 | 第一阶段只覆盖原始文明前期 | 后续文明扩展 |
| 战斗伤害、命中、阵型 | 文明能力只提供能力影响 | 战斗系统 |
| 资源生产链细节 | 这里只输出能力效果 | 资源系统 |
| AI 训练算法 | 只提供可观察状态和奖励信号候选 | 强化学习系统 |
| 前端自行判断能力 | 规则权威必须在后端 | 前后端协议 |

禁止做法：

```text
禁止 civilization_level = 3。
禁止只靠 CivilizationStage 解锁能力。
禁止配置直接写 unlocked = true。
禁止事件直接修改 CivilizationCapabilityState。
禁止前端根据进度条自行判断能力获得。
禁止把单次成功当作能力掌握。
禁止把个人知识当作族群文明能力。
禁止能力效果直接改状态，必须输出 StateChangeDraft 或 EventDraft。
```

## 3. 核心概念

### 3.1 文明能力不是等级

错误设计：

```text
civilization_level = 4
```

正确设计：

```text
CivilizationState
  stage = FireKeeping
  capabilities = [IdentifyEdible, IdentifyDanger, FireKeeping]
  readiness_candidates = [...]
  stage_evidence = [...]
```

文明阶段只是摘要，不是权威能力来源。

规则判断时优先看：

```text
CapabilityType + CapabilityMaturityState + CapabilityUsabilityState + evidence。
```

### 3.2 文明能力不是单次知识

例如：

```text
KnowledgeClaim: 树枝可燃
```

不等于：

```text
CapabilityType.FireKeeping
```

要形成守火能力，至少需要：

```text
知道哪些材料可燃。
知道火会熄灭。
族群有人会补充燃料。
族群能稳定获得燃料。
族群能传播守火方法。
多次守火成功。
失败风险可控。
```

### 3.3 文明能力有生命周期

能力不是一次解锁永久有效。

可能状态：

```text
Candidate → Emerging → Stable → Institutionalized
                    ↓
                 Suspended / Degraded / Lost
```

例子：

```text
族群学会守火，但连续暴雨、燃料断供、守火者死亡。
FireKeeping 不应永久可用，而应进入 Suspended 或 Degraded。
```

### 3.4 文明能力要输出效果，不直接乱改状态

能力生效后可以影响：

```text
可用行为候选。
交互规则前置条件。
风险计算。
资源消耗和产出。
传播效率。
族群凝聚。
文明阶段推导。
AI 观察和奖励信号。
```

但能力模块不直接修改世界对象、资源、实体血量或前端显示。

正确输出：

```text
CapabilityEffectDraft
CivilizationStateChangeDraft
EventDraft
CivilizationProjectionDelta
```

### 3.5 文明能力支持多路线

同一个族群可以同时发展多个领域：

```text
生存：识别食物、识别危险、安全采集。
火：生火、守火、火焰防御、烹饪。
工具：基础工具使用、工具制作。
社会：教学实践、委托教学、任务分配。
空间：路径标记、庇护使用、庇护建造。
记忆：符号记录、长期记忆、资源规划。
```

领域不是互斥职业，而是能力归类和推导工具。

## 4. 枚举设计

本章优先复用 `02_通用枚举设计.md` 已有枚举：

| 枚举 | 中文名 | 用途 |
|---|---|---|
| CivilizationStage | 文明阶段 | 族群整体文明摘要 |
| CapabilityType | 能力类型 | 族群稳定能力类型 |
| UnlockConditionType | 解锁条件类型 | 解锁原因 |
| KnowledgeStatus | 知识状态 | 判断知识是否确立或可执行 |
| KnowledgeScope | 知识范围 | 判断知识是否达到 Tribe / Civilization |
| PropagationResultType | 传播结果 | 判断传播是否有效 |
| EventType | 事件类型 | 输出文明事件 |
| RuleStepType | 规则步骤 | ResolveCivilization 阶段 |

本章新增模块内枚举。模块内枚举优先服务文明能力系统，不放入通用枚举，除非后续多个系统都需要。

### 4.1 CapabilityMaturityState（能力成熟状态）

描述能力掌握程度。

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Unknown | unknown | 未知 | 没有形成候选 |
| Candidate | candidate | 候选 | 有证据提示，但条件不足 |
| Emerging | emerging | 萌芽 | 多个条件成立，但尚不稳定 |
| Stable | stable | 稳定 | 能被族群稳定执行 |
| Institutionalized | institutionalized | 制度化 | 已成为稳定分工、传承或长期规程 |
| Suspended | suspended | 暂停 | 临时不可用，条件恢复后可恢复 |
| Degraded | degraded | 退化 | 能力仍在但可靠性下降 |
| Lost | lost | 丢失 | 缺少记忆、人员、资源或传承，能力失效 |

注意：

```text
Stable 才能作为多数行为前置。
Institutionalized 可以影响文明阶段推进。
Suspended 不能被当作 Lost。
Lost 后重新获得必须重新走证据链，不能直接恢复。
```

### 4.2 CapabilityUsabilityState（能力可用状态）

描述当前环境下能力能否被使用。

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Usable | usable | 可用 | 当前满足使用条件 |
| ConditionallyUsable | conditionally_usable | 条件可用 | 需要特定对象、地点、时间或资源 |
| BlockedByResource | blocked_by_resource | 资源阻断 | 缺少材料、工具、食物或燃料 |
| BlockedByPopulation | blocked_by_population | 人口阻断 | 缺少执行者、教师或协作者 |
| BlockedByEnvironment | blocked_by_environment | 环境阻断 | 天气、季节、昼夜或区域不满足 |
| BlockedByConflict | blocked_by_conflict | 冲突阻断 | 战争、袭击、分裂导致不可用 |
| BlockedByKnowledge | blocked_by_knowledge | 知识阻断 | 知识遗忘、冲突、误解或证据不足 |
| DisabledByRule | disabled_by_rule | 规则禁用 | 配置、剧情或调试显式禁用 |

使用方式：

```text
FireKeeping 成熟度可以是 Stable。
但暴雨且无干燥燃料时，可用状态是 BlockedByEnvironment / BlockedByResource。
```

### 4.3 CapabilityEffectTarget（能力效果目标）

描述能力效果作用到哪个系统。

| 枚举 | 外部字符串 | 中文名 | 含义 |

| InteractionRule | interaction_rule | 交互规则 | 影响规则匹配或前置条件 |
| Risk | risk | 风险 | 降低或提高环境、对象、战斗风险 |
| Resource | resource | 资源 | 影响资源消耗、获得、保存或规划 |
| Knowledge | knowledge | 知识 | 影响知识形成、稳定或冲突处理 |
| Propagation | propagation | 传播 | 影响教学、演示、标记、误解率 |
| TribeBehavior | tribe_behavior | 族群行为 | 影响族群行为采纳或制度化 |
| TribeCohesion | tribe_cohesion | 族群凝聚 | 影响凝聚、分裂风险或协作 |
| Combat | combat | 战斗 | 影响族群战斗准备或防御倾向 |
| CivilizationStage | civilization_stage | 文明阶段 | 影响阶段推导 |
| AiSignal | ai_signal | AI 信号 | 输出可观察特征或奖励候选 |

## 5. 数据结构

### 5.1 CivilizationState（文明状态）

族群文明系统的聚合根。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| civilization_state_id | string | 文明状态 ID | 可用 tribe_id 派生 |
| tribe_id | TribeId | 族群 ID | 所属族群 |
| stage | CivilizationStage | 当前阶段 | 推导出来的文明阶段 |
| stage_confidence | Ratio | 阶段置信度 | 阶段判断可靠度 |
| stage_evidence_ids | list<EventId> | 阶段证据 | 支撑阶段的事件 |
| capabilities | map<CapabilityType, CivilizationCapabilityState> | 能力状态 | 各能力状态 |
| capability_order | list<CapabilityType> | 能力顺序 | 用于展示和复盘，不表示科技树强制顺序 |
| active_effects | list<CapabilityEffectInstance> | 生效效果 | 当前有效能力效果 |
| blocked_capabilities | list<BlockedCapabilityRecord> | 阻断记录 | 当前不可用能力 |
| conflicts | list<CapabilityConflictRecord> | 能力冲突 | 冲突和解释 |
| last_resolved_at | GameTime | 最近解析时间 | 最近计算时间 |
| source_state_version | StateVersion | 来源状态版本 | 输入族群状态版本 |
| version | StateVersion | 文明状态版本 | 当前版本 |

约束：

```text
CivilizationState 只属于一个 Tribe。
stage 必须由 capability 和证据推导。
capabilities 不允许只有 stage 而没有具体能力。
active_effects 必须来自 Stable 或 Institutionalized 能力，除非配置允许 Emerging 产生弱效果。
```

### 5.2 CivilizationCapabilityDefinition（文明能力定义）

配置层定义一种能力是什么。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| capability_type | CapabilityType | 能力类型 | 通用能力枚举 |
| display_name | LocalizedText | 显示名 | 中文、英文等 |
| description | LocalizedText | 描述 | 人类可读说明 |
| domain | CivilizationDomainKey | 领域 | survival、fire、tool、social 等 |
| tags | list<string> | 标签 | 调试、筛选、AI 特征 |
| requirement_set | CivilizationRequirementSet | 前置条件 | 获得或维持能力条件 |
| maturity_policy | CapabilityMaturityPolicy | 成熟策略 | 候选、稳定、制度化阈值 |
| usability_policy | CapabilityUsabilityPolicy | 可用策略 | 当前是否可用 |
| effect_set | CivilizationEffectSet | 效果集合 | 能力效果 |
| conflict_policy | CapabilityConflictPolicy | 冲突策略 | 与其他能力或状态的冲突 |
| decay_policy | CapabilityDecayPolicy | 衰退策略 | 退化、暂停、丢失规则 |
| stage_hints | list<CivilizationStage> | 阶段提示 | 可能推动的阶段 |
| enabled | bool | 是否启用 | 配置开关 |
| priority | int | 优先级 | 冲突和展示排序 |

注意：

```text
definition 只描述规则，不保存运行时状态。
不能在 definition 中写 unlocked = true。
```

### 5.3 CivilizationCapabilityState（文明能力状态）

运行时记录某个族群对某个能力的掌握情况。

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| capability_type | CapabilityType | 能力类型 | 对应能力 |
| maturity | CapabilityMaturityState | 成熟状态 | 掌握程度 |
| usability | CapabilityUsabilityState | 可用状态 | 当前可用性 |
| progress | Ratio | 进度 | 候选到下一状态的进度 |
| confidence | Ratio | 置信度 | 判断可靠性 |
| stability | Ratio | 稳定性 | 长期稳定程度 |
| coverage | Ratio | 覆盖率 | 族群中理解或参与比例 |
| practice_count | Count | 实践次数 | 相关成功实践次数 |
| failure_count | Count | 失败次数 | 相关失败次数 |
| first_seen_at | GameTime | 首次出现 | 首次成为候选时间 |
| unlocked_at | optional<GameTime> | 解锁时间 | 首次达到 Stable 或以上 |
| last_used_at | optional<GameTime> | 最近使用 | 最近生效或执行时间 |
| last_changed_at | GameTime | 最近变化 | 状态变化时间 |
| requirement_snapshot | CapabilityRequirementSnapshot | 条件快照 | 最近一次条件评估 |
| evidence | CapabilityEvidenceBundle | 证据束 | 支撑证据 |
| active_effect_ids | list<string> | 生效效果 ID | 当前效果实例 |
| blocked_reasons | list<CapabilityBlockReasonRecord> | 阻断原因 | 为什么不可用 |
| change_history_ids | list<string> | 变化历史 | 可选，关联事件或 trace |

约束：

```text
maturity 为 Unknown 时 progress 必须为 0。
maturity 为 Lost 时 active_effect_ids 必须为空。
usability 非 Usable 时必须有 blocked_reasons。
Stable 以下默认不能作为硬前置能力。
```

| 字段 | 类型 | 中文名 | 含义 |

| required_behaviors | list<TribeBehaviorRequirement> | 行为条件 | 必须被族群采纳的行为 |
| required_resources | list<ResourceRequirement> | 资源条件 | 必须拥有或稳定获取的资源 |
| required_population | optional<PopulationRequirement> | 人口条件 | 人口、角色、教师数量 |
| required_environment | list<EnvironmentRequirement> | 环境条件 | 季节、昼夜、天气、区域 |
| required_propagation | optional<PropagationRequirement> | 传播条件 | 覆盖率、成功率、误解率 |
| required_memory | optional<MemoryRequirement> | 记忆条件 | 长期记忆、符号、跨代保持 |
| required_cohesion | optional<CohesionRequirement> | 凝聚条件 | 协作、分裂风险 |
| required_combat | optional<CombatRequirement> | 战斗条件 | 共同作战或防御准备 |
| expression | optional<ExpressionRef> | 复合表达式 | 用配置表达复杂条件 |

```text

```

### 5.29 CivilizationDomainKey（文明领域键）

`CivilizationDomainKey` 是字符串键，不是通用枚举。

原因：文明领域后续会频繁扩展，如果做成 C++ 枚举，每次增加农业、航海、仪式、制度等领域都要改代码。

第一阶段建议内置领域：

| key | 中文名 | 说明 |
|---|---|---|
| survival | 生存 | 食物、危险、采集、生理安全 |
| fire | 火 | 生火、守火、防御、烹饪 |
| tool | 工具 | 使用、制造、维修、组合 |
| shelter | 庇护 | 利用地点、建造、维护、安全区域 |
| social | 社会 | 教学、分工、协作、制度行为 |
| memory | 记忆 | 长期记忆、符号、记录、传承 |
| combat | 战斗 | 共同防御、威慑、风险规程 |
| mobility | 迁徙 | 路径、标记、路线规划、迁移 |
| resource | 资源 | 储备、分配、消耗规划 |

约束：

```text
领域只用于归类、展示、阶段推导和 AI 特征。
领域不能替代 CapabilityType。
配置新增领域必须有本地化名称和说明。
```

## 6. 接口设计

### 6.1 CivilizationResolver

文明能力系统主入口。

```cpp

public:
    virtual Result<CivilizationResolveResult> resolve(
        const CivilizationResolveInput& input,
        const CivilizationRuleContext& context
```

职责：


## 7. 文明能力生命周期

### 7.1 总流程

```text
1. 输入 TribeState、KnowledgeProjection、传播候选、文明候选。
2. CandidateBuilder 生成 CivilizationCapabilityCandidate。
3. RequirementEvaluator 评估每个能力定义。
4. CapabilityStateResolver 更新成熟状态。
5. UsabilityResolver 计算当前可用性。
6. ConflictResolver 检测冲突。
7. EffectResolver 生成能力效果草稿。
8. StageResolver 推导文明阶段。
9. EventBuilder 生成事件草稿。
10. ProjectionBuilder 输出前端 / AI 投影。
```

### 7.2 状态晋升规则

默认晋升路径：

```text
Unknown → Candidate → Emerging → Stable → Institutionalized
```

晋升要求：

| 目标状态 | 最低要求 |
|---|---|
| Candidate | 有候选或知识提示 |
| Emerging | 多项前置满足，存在成功实践 |
| Stable | 条件满足、成功率稳定、传播覆盖达标 |
| Institutionalized | 长期使用、分工稳定、可传承、风险可控 |

### 7.3 暂停、退化和丢失

```text
Stable + 临时缺资源 → Suspended
Stable + 多次失败 + 传播断裂 → Degraded
Degraded + 证据丢失 + 无教师 + 无记忆 → Lost
Suspended + 条件恢复 → 恢复到原成熟状态
Lost + 新证据出现 → Candidate，而不是直接 Stable
```

### 7.4 能力可用性和成熟度分离

例子：

```text
族群已稳定掌握 FireKeeping。
当前暴雨且燃料湿透。

maturity = Stable
usability = BlockedByEnvironment
```

这避免把“暂时不能用”误判成“族群忘了”。

### 7.5 文明阶段推导

阶段推导输入：

```text
稳定能力集合。
制度化能力集合。
长期记忆能力。
任务分工和教学能力。
资源规划能力。
风险规程能力。
族群凝聚和分裂风险。
```

阶段推导输出：

```text
stage
stage_confidence
stage_evidence_ids
CivilizationStageChanged 事件草稿
```

阶段变化示例：

```text
IdentifyEdible + IdentifyDanger + SafeForaging 稳定 → Foraging。
FireKeeping 稳定，FireDefense 萌芽 → FireKeeping。
BasicToolUse + ToolCrafting 稳定 → ToolUsing。
SymbolRecording + TeachingPractice + LongTermMemory 稳定 → Symbolic 或 OrganizedTribe 候选。
```

## 8. 事件输出

### 8.1 CapabilityUnlocked

当能力首次达到 `Stable` 或 `Institutionalized` 时输出。

Payload 使用 `CivilizationEventPayload`：

| 字段 | 要求 |
|---|---|
| capability | 必填 |
| unlock_conditions | 必填，来自 UnlockConditionType |
| tribe_id | 必填 |
| stage_before | 可填 |
| stage_after | 可填 |

注意：

```text
Emerging 不输出 CapabilityUnlocked，可输出普通 trace 或投影变化。
Stable 第一次达成才输出 CapabilityUnlocked。
Stable 后再次维持不重复输出，除非 Lost 后重新获得。
```

### 8.2 CivilizationStageChanged

当文明阶段变化时输出。

要求：

```text
必须有 stage_before 和 stage_after。
必须有证据。
必须说明变化原因。
不能因为单个事件立即跳多个阶段，除非配置允许并有完整证据。
```

### 8.3 能力退化事件

通用枚举中暂未单独定义 `CapabilityDegraded`。

第一阶段处理方式：

```text
能力退化只写 CivilizationStateChangeDraft 和 CivilizationProjectionDelta。
如需要玩家可见，再使用 CivilizationStageChanged 或 TribeBehaviorChanged 的解释。
后续如事件量变多，可以扩展 EventType。
```

### 8.4 事件可见性

| 场景 | visibility | importance |
|---|---|---|
| 能力首次稳定获得 | TribeVisible | Critical |
| 文明阶段变化 | TribeVisible | Critical |
| 能力候选出现 | Debug / TribeVisible | Normal / Important |
| 能力临时阻断 | Debug / PlayerVisible | Normal |
| 能力丢失 | TribeVisible | Important |

## 11. 校验规则

### 11.1 能力状态校验

| 规则 | 错误级别 |
|---|---|
| Stable 但没有证据束 | RuleError |
| Institutionalized 但没有长期使用或传播证据 | RuleError |
| Lost 但仍有 active_effects | RuleError |
| Unknown 但 progress 大于 0 | Warning |
| Usability 非 Usable 但无阻断原因 | RuleError |
| CapabilityUnlocked 缺少 unlock_conditions | RuleError |

### 11.2 条件校验

| 规则 | 错误级别 |
|---|---|
| RequiredCapability 指向不存在能力 | RuleError |
| RequiredStage 作为唯一条件 | RuleError |
| KnowledgeRequirement 无任何 subject / id / effect 限定 | Warning |
| ResourceRequirement min_amount 小于 0 | ConfigError |
| PropagationRequirement min_coverage 超出 0-1 | ConfigError |
| Expression 引用不存在字段 | ConfigError |

### 11.3 阶段校验

| 规则 | 错误级别 |
|---|---|
| stage 变化没有证据 | RuleError |
| 阶段满足但没有任何 stable capability | RuleError |
| 高阶段 priority 低于低阶段且无说明 | Warning |
| 一次跳过多个阶段且无配置允许 | RuleError |

### 11.4 效果校验

| 规则 | 错误级别 |
|---|---|
| Lost 能力产生效果 | RuleError |
| Blocked 能力产生普通效果 | RuleError |
| effect target 不存在 | ConfigError |
| 同一 target 有互斥 effect 且无冲突策略 | RuleError |
| effect 直接修改状态 | RuleError |

### 11.5 兼容性校验

```text
必须消费 15 的 CivilizationReadinessCandidate。
必须兼容 13 的 CivilizationKnowledgeCandidate。
必须兼容 14 的 TribeInfluenceCandidate。
CapabilityUnlocked 必须符合 04 的 CivilizationEventPayload。
CivilizationStageChanged 必须符合 04 的 CivilizationEventPayload。
ResolveCivilization 必须在规则管线中位于 TribeState 之后。
```

## 12. 测试点

### 12.1 单元测试

| 测试 | 预期 |
|---|---|
| 条件不足 | Candidate 不晋升 Stable |
| 知识达标但传播不足 | Emerging 或 Candidate |
| 传播达标但行为未采纳 | 不解锁能力 |
| Stable 能力首次获得 | 输出 CapabilityUnlocked |
| Stable 重复维持 | 不重复输出 CapabilityUnlocked |
| 资源临时缺失 | maturity 不变，usability blocked |
| 证据长期断裂 | Degraded 或 Lost |
| RequiredStage 单独解锁 | RuleError |

### 12.2 组合测试

| 场景 | 输入 | 预期 |
|---|---|---|
| 红浆果安全采集 | 知识 + 传播 + 行为稳定 | SafeForaging Stable |
| 守火 | 可燃知识 + 燃料 + 守火行为 | FireKeeping Stable |
| 火焰防御 | FireKeeping + 夜间防御知识 + 协作 | FireDefense Emerging / Stable |
| 烹饪 | FireKeeping + 食物加工知识 + 资源 | Cooking Candidate / Stable |
| 路径标记 | 标记行为 + 区域记忆 + 传播 | PathMarking Stable |
| 符号记录 | PathMarking + LongTermMemory + 外部记录 | SymbolRecording Emerging / Stable |
| 任务分配 | 人口 + 分工行为 + 凝聚 | TaskAssignment Stable |

### 12.3 事件测试

| 场景 | 预期事件 |
|---|---|
| 能力首次 Stable | CapabilityUnlocked |
| 阶段从 Awakening 到 Foraging | CivilizationStageChanged |
| 能力阻断 | 默认无关键事件，只输出投影变化 |
| 能力 Lost 后重新 Stable | 可再次输出 CapabilityUnlocked，但 evidence 新 |
| 配置错误 | RuleErrorOccurred 或 ConfigError |

### 12.4 回放测试

同一输入必须得到：

```text
相同能力状态。
相同阶段。
相同事件草稿。
相同 deterministic_key。
相同 trace 输入哈希。
```

禁止：

```text
不记录随机流的随机解锁。
依赖 map 遍历无序结果。
依赖当前真实系统时间。
```

### 12.5 强化学习兼容测试

文明系统应输出可用于 RL 的稳定特征：

```text
当前 stage。
stable capability bitset。
emerging capability progress。
blocked capability reasons。
active effect summaries。
next candidate readiness。
stage confidence。
能力变化事件。
```

奖励信号候选：

```text
首次获得稳定能力。
能力制度化。
文明阶段提升。
关键能力恢复可用。
危险能力误用减少。
传播误解率下降。
```

注意：

```text
RL 不应直接篡改 CivilizationState。
RL 只能选择行动或策略，文明状态仍由规则系统解析。
```

## 13. 与前面系统的关系

### 13.1 与知识系统

```text
KnowledgeClaim 提供能力前置知识。
CivilizationKnowledgeCandidate 提供能力候选。
KnowledgeStatus / KnowledgeScope 判断知识是否足以支撑能力。
知识冲突会阻断能力晋升或降低置信度。
```

### 13.2 与传播系统

```text
PropagationResult 决定知识是否进入族群。
TribeInfluenceCandidate 影响覆盖率、误解率和教学能力候选。
传播失败可能导致能力退化或误用风险。
```

### 13.3 与族群状态系统

```text
TribeState 提供人口、资源、行为、风险、凝聚、战斗状态。
CivilizationReadinessCandidate 是 16 的主要输入候选。
TribeBehaviorProfile 决定知识是否变成族群行为。
TribeCohesionProfile 决定制度化能力是否稳定。
TribeCombatProfile 支撑 FireDefense、RiskProtocol、共同作战类能力。
```

### 13.4 与事件系统

```text
文明系统只生成 EventDraft。
事件系统负责事件 ID、顺序、持久化、可见性和复盘。
CapabilityUnlocked / CivilizationStageChanged 使用 CivilizationEventPayload。
```

### 13.5 与配置系统

```text
能力定义、阶段定义、阈值和效果来自配置。
配置必须通过强校验。
热更新不能直接改变已存在能力状态，只能影响下一轮解析。
```

## 14. 设计注意事项

研发实现时必须遵守：

```text
不要把文明能力写成 bool unlocked。
不要把 CivilizationStage 当作等级条。
不要只靠 RequiredStage 解锁能力。
不要让配置直接解锁能力。
不要让事件直接修改文明状态。
不要让前端自行推导能力获得。
不要把个人知识等同于文明能力。
不要把能力暂时不可用等同于能力丢失。
不要让能力效果绕过规则管线直接改状态。
不要在文明系统中实现具体战斗、生产链或 UI。
```

推荐实现顺序：

```text
先实现 CivilizationCapabilityDefinition 加载和校验。
再实现 CandidateBuilder。
再实现 RequirementEvaluator。
再实现 CapabilityStateResolver。
再实现 EffectResolver。
最后实现 StageResolver 和 ProjectionBuilder。
```
