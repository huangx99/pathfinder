# 63 P34 条件化因果归因最小实现设计

## 1. 本次实现目标

本次不做完整真实世界模拟，而是把当前 H5 学习闭环从“重复次数确认知识”升级为“带条件的最小因果归因”。

要解决的问题：

```text
吃三次红果确认后，再吃一次腐烂红果，不能马上确认腐烂红果一定有毒。
因为可能是短时间内浆果吃太多，也可能是腐烂状态导致。
```

本次实现必须支持：

```text
1. 新鲜红果和腐烂红果形成不同条件知识。
2. 知识确认受近期暴露窗口影响。
3. 短时间内存在替代解释时，危险知识不能直接升级到已确认。
4. 玩家可以通过“等待一会”让临时暴露窗口消退。
5. H5 展示仍然是中文，不暴露调试式 key。
```

---

## 2. 不做的内容

P34 最小实现不做以下完整系统：

```text
1. 完整概率统计模型。
2. 完整个体体质系统。
3. 完整药理和慢性病模型。
4. 完整 NPC 恶意判断。
5. 完整 JSON 配置化因果规则。
```

但本次代码要预留这些方向，不能写死成红果特判的不可扩展逻辑。

---

## 3. 核心设计

### 3.1 暴露记录

新增会话内最小暴露记录：

```cpp
struct DialogCausalExposureRecord {
    std::string object_key;
    std::string family_key;
    std::string action_key;
    std::string effect_key;
    uint64_t observed_tick;
    double dose_amount;
    std::vector<std::string> state_keys;
    std::vector<std::string> reason_keys;
};
```

中文解释：

| 字段 | 中文 | 例子 |
|---|---|---|
| object_key | 具体对象 | red_berry / decayed_red_berry |
| family_key | 同类对象 | berry |
| action_key | 行为方式 | eat / use |
| effect_key | 观察结果 | restore_hunger / poison |
| observed_tick | 发生时间 | 当前回合 tick |
| dose_amount | 剂量 | 当前先用 1.0 |
| state_keys | 对象状态 | fresh / decayed |
| reason_keys | 归因原因 | causal_route_eat |

### 3.2 时间窗口

本次定义一个最小窗口：

```text
浆果消化窗口 = 6 tick
等待一会 = 推进 8 tick
```

含义：

```text
如果 6 tick 内吃了多个浆果，剂量解释仍有效。
如果等待一会后再吃，旧的浆果暴露不再作为强替代解释。
```

后续可替换成配置：

```json
{
  "family": "berry",
  "exposure_window_ticks": 6,
  "safe_wait_ticks": 8
}
```

### 3.3 条件化知识

红果知识不能再是：

```text
red_berry + eat -> restore_hunger
```

而应是：

```text
red_berry{新鲜状态} + eat -> restore_hunger
red_berry{腐烂状态} + eat -> poison
```

H5 中文显示：

```text
红果：吃后可以缓解饥饿；已确认的有效知识
腐烂红果：吃后会让身体不适或中毒；经验假设，需要继续验证
```

### 3.4 替代解释压制确认

当满足以下条件时，危险知识不能升到 Active：

```text
1. 当前行为是吃腐烂红果。
2. 结果是身体不适 / 中毒。
3. 近期消化窗口内存在多个浆果暴露。
4. 近期暴露里存在新鲜红果或其他同族食物。
```

此时知识最多是：

```text
Hypothesis / 经验假设
```

原因：

```text
现在还不能排除“短时间吃太多浆果”的替代解释。
```

### 3.5 等待行为

新增中文指令：

```text
等待
等待一会
休息
过一会
```

效果：

```text
推进时间，让短期暴露窗口自然消退。
```

等待不会清空知识，也不会清空长期记忆，只影响短期归因窗口。

## 4. 和现有系统的关系

### 4.1 学习系统

`LearningLoopService` 仍负责：

```text
认知 -> 记忆 -> 知识形成 -> 知识修正 -> 传播
```

本次只增加一条规则：

```text
如果 feedback.reason_keys 包含 causal_ambiguous_dose_window，则知识状态最高只能到 Hypothesis。
```

这样未来其他系统也可以通过 reason key 触发同类压制，不和红果绑定。

### 4.2 H5 对话层

`DialogTurnService` 负责：

```text
根据会话内暴露记录判断是否存在近期替代解释。
```

原因：H5 原型的时间、对象、行动都在会话层最完整。

### 4.3 投影层

`H5PlayableProjectionMapper` 负责：

```text
把条件知识投影成玩家能读懂的中文。
```

不能让玩家看到：

```text
condition:object_state:eq:decayed
```

## 5. 验收标准

### 5.1 新鲜红果稳定学习

```text
吃红果 x3
结果：红果少量食用缓解饥饿成为已确认知识。
```

### 5.2 短期混杂不能确认腐烂

```text
吃红果 x3
立刻吃腐烂红果 x3
结果：腐烂红果有害最多是经验假设，不能成为已确认。
```

### 5.3 等待后可排除短期剂量解释

```text
吃红果 x3
等待一会
吃腐烂红果 x3
结果：腐烂红果有害可以逐步成为已确认。
```

### 5.4 新鲜知识不被腐烂分支降级

```text
红果确认后，再学习腐烂红果。
结果：新鲜红果知识仍保持已确认。
```

### 5.5 传播仍能工作

```text
形成知识后教同伴。
结果：同伴获得对应条件知识，且不会把腐烂红果误显示成普通红果。
```

## 6. 自查重点

实现后必须自查：

```text
1. 是否还有 red_berry 和 decayed_red_berry 显示混淆。
2. 是否还有一次危险体验直接确认的情况。
3. 等待是否只推进时间，不清空知识。
4. 条件是否进入 KnowledgeClaim，而不是只写在 UI。
5. 教学传播是否保留条件。
6. JSON/前端是否仍不泄露调试 key。
```
