# P62 统一内容物品实例化与地图生成接入任务卡设计

## 1. 阶段定位

P62 解决 V2 开放世界当前最危险的内容分裂问题：地图生成、资源节点、地面物品、采集产物仍有一部分由 C++ 写死，而知识、反应、反馈、背包系统使用 `ContentRegistry` 中的 `object_key`。

本阶段不是新增一套“地图物品系统”，而是把地图上的所有物品、资源产物和后续掉落产物统一回现有内容体系。

最终目标：

```text
content/core/objects/*.json 中的 object_key
-> 世界生成规则引用 object_key
-> 地面实体 / 资源节点产物实例化同一个 object_key
-> Pickup / Inventory 保留同一个 object_key
-> Reaction / Feedback / Experience / Knowledge / Agent 使用同一个 object_key
```

如果地图上生成的物品不能参与背包、反应、知识和 Agent 决策，它就是假物品；P62 必须消除这个风险。

## 2. 背景与现状

现有内容体系已经有：

```text
ContentRegistry
ObjectDefinitionContent
ReactionDefinitionContent
FeedbackDefinitionContent
KnowledgeTemplateContent
```

但 V2 地图生成仍存在写死内容：

```text
backend/src/world_generation/world_generation_service.cpp
- berry_bush
- young_tree
- loose_stone_node
- red_berry
- wood
- stone_flake
- loose_stone

backend/src/world_runtime/world_grid_runtime.cpp
- unknown_bush
- loose_stone
```

这些字符串虽然和 content 里的 key 名字相同，但来源不是同一个权威定义。后续会导致：

```text
1. JSON 里删掉物品，地图仍可能生成幽灵物品。
2. 地图生成物品缺 display / tags / initial_state。
3. 资源采集产物无法保证能进入 Reaction / Knowledge。
4. 新物品扩展必须改 C++，破坏内容包边界。
5. 后续掉落物会继续复制这套错误模式。
```

## 3. 目标

P62 必须达成：

```text
1. 地图地面物品只引用 ContentRegistry 中存在的 object_key。
2. 资源节点采集产物只引用 ContentRegistry 中存在的 object_key。
3. 生成实体的 display_key、quantity、stack_key、tags、numeric/state 默认值来自 ObjectDefinitionContent，可由生成规则有限覆盖。
4. 内容 JSON 支持配置 worldgen_profiles / resource_rules / ground_item_rules。
5. C++ 世界生成核心不再写死红果、树、碎石、石片等具体内容。
6. 旧的 inspect-only 稀疏实体生成被删除或明确废弃，不能继续生成绕过 ContentRegistry 的物品。
7. 如果配置引用不存在的 object_key，严格模式必须报错，不允许静默生成。
```

## 4. 不做什么

本阶段不做：

```text
1. 不重做 Reaction 引擎。
2. 不重做 Inventory 归属系统。
3. 不做完整掉落表战斗系统。
4. 不做内容包热更新。
5. 不做前端表现重构。
6. 不做 NPC 自主采集策略。
```

但 P62 必须预留掉落表数据结构，避免后续又写死 `wolf -> meat` 这类规则。

## 5. 设计原则

### 5.1 一个 object_key 一个身份

同一个物品在地图、背包、反应、知识中必须使用同一个 `object_key`。

禁止：

```text
地图里叫 loose_stone，背包里叫 stone_item，反应里叫 stone_flake_like。
```

### 5.2 地图规则只引用内容，不定义物品身份

`ground_item_rules` 和 `resource_rules.outputs` 只能引用 `object_key`，不能重新定义物品。

允许有限覆盖：

```text
quantity
stack_key
state_keys
numeric_states
tag_keys
```

禁止覆盖：

```text
object identity
kind
核心 display/description 来源
```

### 5.3 C++ 只保留算法，不保留具体内容

C++ 可以知道如何按密度、地形、距离放置资源，但不能知道什么是红果、碎石、树、木头。

禁止：

```cpp
if (entity_key == "loose_stone") { ... }
profile.resource_rules = {{ "berry_bush", ... }};
```

### 5.4 缺内容必须失败

如果 worldgen rule 引用不存在的 object_key，不能生成占位实体，也不能 fallback 到 key + `_name`。

### 5.5 采集产物必须进入同一实例化路径

资源节点采集产物不能只是字符串奖励。必须通过 `ObjectDefinitionContent` 生成 `ResourceHarvestOutputDraft`，再由 `IWorldEntityLocationPort::spawnEntityOnMap` 进入 P45 归属系统。

### 5.6 前端不参与规则解释

前端只能展示 projection。物品是否可拾取、可采集、可反应，由后端 Command/available_commands 决定。

## 6. 模块划分

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| `content_types` | 内容类型 | 增加 worldgen profile / resource / ground item / drop table DTO 与 Content | JSON 字段 | ContentDraftRegistry / ContentRegistry | 不依赖 world runtime |
| `json_content_loader` | JSON 加载 | 解析 `worldgen/*.json` | 内容文件 | Draft worldgen profiles | 不写默认玩法内容 |
| `content_validation` | 内容校验 | 校验 object_key 引用存在 | Draft registry | validation report | 不静默忽略缺失引用 |
| `world_generation_service` | 世界生成服务 | 使用 ContentRegistry 提供的 profile | WorldGenerationRequest | WorldGenerationResult | 不写死具体内容 |
| `resource_distribution_generator` | 分布算法 | 按规则放置 resource node / ground item | Profile + cells | Drafts | 不查询前端，不改 runtime |
| `resource_harvest_service` | 采集服务 | 根据 ObjectDefinition 生成采集产物 draft | Resource node | Output draft | 不用 `entity_key + _name` 伪造 display |
| `world_generation_applier` | 生成落地 | 通过 Runtime/LocationPort 写入 | Drafts | Runtime state | 不直接改数组 |

## 7. 核心数据结构

### 7.1 `WorldgenProfileDefinitionDto`

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `profile_key` | string | 生成配置 key | 是 | JSON | world_generation | 合法 id | 不能为空 |
| `region_size` | int | 区块尺寸 | 是 | JSON | world_generation | >0 | 不能用来锁死无限地图 |
| `default_layer` | string | 默认层 | 是 | JSON | world_generation | 非空 | 不能写前端层 |
| `terrain` | object | 地形噪声配置 | 是 | JSON | terrain_generator | 结构合法 | 不写内容物品 |
| `resource_rules` | array | 资源节点规则 | 否 | JSON | resource_distribution | 引用 object_key 存在 | 不写实体实例 |
| `ground_item_rules` | array | 地面物品规则 | 否 | JSON | resource_distribution | 引用 object_key 存在 | 不绕过 object 定义 |
| `spawn_safety` | object | 出生安全配置 | 否 | JSON | spawn_safety | key 合法 | 不直接生成物品实例 |

### 7.2 `WorldgenResourceRuleDto`

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `resource_key` | string | 资源节点 key | 是 | JSON | world_generation | 合法 id | 不能为空 |
| `allowed_terrain_tags` | string[] | 允许地形/标签 | 是 | JSON | resource_distribution | 可为空表示不限 | - |
| `density` | double | 密度 | 是 | JSON | resource_distribution | >=0 | 不能无限生成 |
| `node_kind` | string | 节点类型 | 是 | JSON | world_generation | enum 字符串 | Unknown 不允许 |
| `required_action_key` | string | 采集动作 | 是 | JSON | harvest | gather/chop/mine/dig | 不能和 Command 脱节 |
| `required_tool_key` | string | 所需工具 object_key | 否 | JSON | harvest | 非空时必须存在 | 不能写显示名 |
| `output_object_keys` | string[] | 产物 object_key | 是 | JSON | harvest | 每个必须存在于 objects | 不能是地图专用 key |
| `charges` | int | 可采次数 | 是 | JSON | harvest | >0 | 不能为负 |

### 7.3 `WorldgenGroundItemRuleDto`

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `object_key` | string | 地面物品 object_key | 是 | JSON | world_generation | 必须存在 | 不能写不存在的临时物品 |
| `allowed_terrain_tags` | string[] | 允许地形/标签 | 是 | JSON | resource_distribution | 可为空 | - |
| `density` | double | 密度 | 是 | JSON | resource_distribution | >=0 | 不能无限生成 |
| `quantity` | int | 生成数量 | 否 | JSON | entity draft | >0，缺省用 object default | - |
| `stack_key` | string | 堆叠 key | 否 | JSON | inventory | 空则 object_key:default | 不改变 object 身份 |
| `state_keys` | string[] | 初始状态标签 | 否 | JSON | runtime | 合法字符串 | - |
| `numeric_states` | map | 初始数值状态 | 否 | JSON | runtime | 数字 | - |
| `tag_keys` | string[] | 额外标签 | 否 | JSON | projection/query | 合法字符串 | 不替代 object safe_tags |

### 7.4 `WorldgenDropRuleDto`（预留）

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `drop_rule_key` | string | 掉落规则 key | 是 | JSON | future drop service | 合法 id | 不能为空 |
| `source_object_key` | string | 来源对象 | 否 | JSON | future drop service | 非空必须存在 | 不写 C++ if |
| `output_object_keys` | string[] | 掉落物 | 是 | JSON | future drop service | 必须存在 | 不生成幽灵物品 |

P62 只解析和校验 drop rule，可暂不接入战斗死亡流程。

## 8. JSON 文件组织

新增：

```text
content/core/worldgen/first_world.json
```

Manifest 新增：

```json
{ "path": "worldgen/first_world.json", "category": "worldgen", "required": true }
```

示例：

```json
{
  "worldgen_profiles": [
    {
      "profile_key": "first_world",
      "region_size": 16,
      "default_layer": "surface",
      "resource_rules": [
        {
          "resource_key": "berry_bush",
          "allowed_terrain_tags": ["plain", "forest"],
          "density": 0.08,
          "node_kind": "Plant",
          "required_action_key": "gather",
          "output_object_keys": ["red_berry"],
          "charges": 3
        }
      ],
      "ground_item_rules": [
        {
          "object_key": "loose_stone",
          "allowed_terrain_tags": ["stone_field", "plain"],
          "density": 0.02,
          "quantity": 1
        }
      ]
    }
  ]
}
```

## 9. 旧硬编码删除要求

P62 完成后必须删除或废弃：

```text
world_generation_service.cpp 中 buildFirstWorldProfile 的具体内容资源规则。
world_grid_runtime.cpp 中 unknown_bush / loose_stone inspect-only 稀疏实体生成。
harvest_service 中 display_name_key = entity_key + "_name" 的伪造逻辑。
```

允许保留：

```text
first_world 作为 profile_key。
地形噪声默认算法。
通用 enum 转换。
通用密度放置算法。
测试中手动构造最小 registry 的 fixture。
```

## 10. 玩家体验映射

P62 不直接做新玩法，但它决定后续玩法是否是真的。

玩家看到碎石：

```text
地图显示 loose_stone
-> 双击/点击可拾取
-> 进入背包仍是 loose_stone
-> 后续如果 JSON 定义 loose_stone + use/craft/teach，系统能识别
-> NPC/Agent 也能围绕同一个 object_key 学习和使用
```

如果没有 P62，玩家会遇到：

```text
地图上的东西捡起来不能用。
采集出来的东西不能合成。
NPC 学不会地图物品用途。
新增内容必须改代码。
```

## 11. 验收标准

必须满足：

```text
1. content loader 支持 worldgen category。
2. first_world 的资源节点和地面物品来自 JSON。
3. loose_stone 在 objects JSON 中有唯一物品定义。
4. 地面 loose_stone 由 object_key 实例化，不再由 C++ 写死 display。
5. 采集产物 red_berry / wood / stone_flake 必须引用 objects JSON。
6. 引用不存在 object_key 的 worldgen JSON 在 strict 模式报错。
7. 旧 inspect-only loose_stone / unknown_bush 生成逻辑删除或不再执行。
8. 采集产物 display_name_key 来自 ObjectDefinitionContent。
9. 单元测试覆盖 JSON 解析和引用校验。
10. 集成测试覆盖生成 -> 地图实体 -> 拾取/采集产物使用同一个 object_key。
```

## 12. 风险与处理

| 风险 | 后果 | 处理 |
|---|---|---|
| 直接让 world_generation 依赖 content 导致循环依赖 | 编译或架构混乱 | 只让 world_generation 读取已转换的 WorldgenProfile，Content 转换在 loader 内完成 |
| 旧测试依赖内置 profile | 测试失败 | 保留无内容时的 test fallback 或更新测试显式注册 profile |
| 引用校验过严导致现有 content 失败 | 阻塞启动 | 先补齐 loose_stone object，再 strict 校验 |
| 前端显示缺翻译 | 玩家看到 key | 同步补 locale |
| 采集产物默认 display 改变 | 测试断言变化 | 改测试断言为 object display_key 来源 |

## 13. 实施顺序

```text
1. 扩展 ContentFileCategory: Worldgen。
2. 增加 worldgen DTO / Content / DraftRegistry / Registry 查询。
3. JsonContentParser 增加 parseWorldgenProfiles。
4. ContentReferenceValidator 校验 resource outputs / ground items / tool refs。
5. content/core 增加 worldgen/first_world.json，manifest 引用。
6. WorldGenerationService 支持从 ContentRegistry 注册 worldgen profiles。
7. ClientServerRuntimeFactory 启动时加载 core content 并注入 world_generation。
8. ResourceHarvestService 注入 ObjectDefinition 查询器，产物按 object 定义实例化。
9. 删除 world_grid_runtime 旧稀疏实体生成。
10. 补测试并构建验证。
```
