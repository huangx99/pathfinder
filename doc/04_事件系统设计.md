# 先驱者：事件系统设计

## 1. 本文档目标

本文档用于设计《先驱者》规则引擎中的事件系统。

事件系统是规则引擎的事实记录层。

它不是单纯日志，也不是单纯前端提示，而是把规则结算后发生过的事实，以稳定、可查询、可回放、可测试的方式记录下来。

事件系统要回答：

```text
发生了什么？
由哪个命令导致？
发生在哪个规则步骤？
影响了哪些对象、实体、知识或族群？
对应哪些状态变化？
前端应该看到什么？
日志和复盘如何还原过程？
测试如何断言规则确实发生？
```

## 2. 设计边界

### 2.1 本文档要解决的问题

```text
定义事件系统总架构。
定义 EventRecord 基础结构。
定义 EventPayload 分类。
定义 EventBuilder / EventFactory。
定义 EventStream、EventStore、EventBus、EventSubscriber。
定义 EventProjection 与 EventReplay。
定义 EventVisibility 与 EventImportance。
定义事件校验、生成流程、消费规则和测试样例。
```

### 2.2 本文档暂不解决的问题

```text
不实现具体数据库表结构。
不实现网络消息队列。
不设计 UI 动画协议。
不设计剧情脚本事件系统。
不设计完整成就、任务、通知系统。
不设计分布式事件一致性算法。
```

这些内容后续可以在前后端协议、存档与复盘、测试策略中继续细化。

## 3. 事件系统总架构

第一阶段事件系统采用完整架构设计、分层实现策略。

完整架构包含：

| 模块 | 中文名 | 作用 |
|---|---|---|
| EventRecord | 事件记录 | 表示一条已经发生的事实 |
| EventPayload | 事件载荷 | 表示事件具体数据 |
| EventBuilder / EventFactory | 事件构造器 | 统一创建事件，避免字段遗漏 |
| EventStream | 事件流 | 聚合一次命令产生的有序事件 |
| EventStore | 事件存储 | 追加式保存事件 |
| EventBus | 事件总线 | 分发事件给订阅者 |
| EventSubscriber | 事件订阅者 | 消费事件的系统接口 |
| EventProjection | 事件投影 | 将事件转成前端、日志、复盘等读模型 |
| EventReplay | 事件复盘 | 按事件还原发生过程 |
| EventQuery | 事件查询 | 按条件查询事件 |
| EventValidation | 事件校验 | 保证事件结构合法 |

### 3.1 分层落地优先级

| 优先级 | 模块 | 说明 |
|---|---|---|
| P0 | EventRecord | 必须实现，事件系统最小单位 |
| P0 | EventPayload | 必须实现，事件具体内容 |
| P0 | EventBuilder / EventFactory | 必须实现，避免事件字段混乱 |
| P0 | EventStream | 必须实现，一次命令的事件集合 |
| P0 | EventStore | 必须实现，事件追加保存 |
| P0 | EventValidation | 必须实现，事件写入前校验 |
| P1 | InProcessEventBus | 强烈建议实现，进程内同步分发 |
| P1 | EventSubscriber | 强烈建议实现，订阅扩展点 |
| P1 | EventProjection | 强烈建议实现，前端和日志读模型 |
| P2 | AsyncEventBus | 后续增强，异步事件分发 |
| P2 | RemoteEventPublisher | 后续增强，远程推送 |
| P2 | DistributedEventStore | 后续增强，分布式事件存储 |

### 3.2 核心原则

```text
事件记录事实，不直接修改世界状态。
核心世界状态只能由规则管线和 StateChange 修改。
EventBus 不负责核心状态写入。
前端读事件或投影，不自己推导规则结果。
事件一旦写入 EventStore，默认不可修改。
需要修正历史事件时，追加 CorrectionEvent，不覆盖旧事件。
```

## 4. EventRecord（事件记录）

`EventRecord` 是事件系统的最小单位。

### 4.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| event_id | EventId | 事件 ID | 是 | 事件唯一标识 |
| event_type | EventType | 事件类型 | 是 | 发生了什么 |
| command_id | CommandId | 命令 ID | 条件必填 | 由哪个命令产生，系统事件可为空 |
| causation_id | string | 直接原因 ID | 否 | 直接导致该事件的命令、事件或规则 ID |
| correlation_id | string | 关联链路 ID | 否 | 一批相关事件的链路 ID |
| tick | Tick | 逻辑帧 | 是 | 事件发生的逻辑顺序 |
| game_time | GameTime | 游戏时间 | 是 | 游戏内时间 |
| rule_step | RuleStepType | 规则步骤 | 否 | 由哪个规则步骤产生 |
| result_type | RuleResultType | 规则结果 | 否 | 关联的规则结果 |
| state_version | StateVersion | 状态版本 | 是 | 事件对应的状态版本 |
| state_change_ids | list<string> | 状态变更 ID | 否 | 事件对应的状态变化 |
| source_id | string | 来源 ID | 否 | 行动者、规则、系统或对象 |
| target_ids | list<string> | 目标 ID | 否 | 被影响的对象、实体、知识等 |
| payload | EventPayload | 事件载荷 | 是 | 事件具体内容 |
| visibility | EventVisibility | 可见性 | 是 | 谁能看见事件 |
| importance | EventImportance | 重要程度 | 是 | 前端和日志处理优先级 |
| tags | list<string> | 标签 | 否 | 查询、调试、分类用标签 |

### 4.2 字段说明

```text
causation_id 表示“这个事件直接由谁导致”。
correlation_id 表示“这一批事件属于同一条链路”。
state_change_ids 用于连接 Event 与 StateChange。
payload 必须与 event_type 匹配。
visibility 决定事件对玩家、族群、调试、复盘是否可见。
importance 决定前端提示强度和日志过滤优先级。
```

### 4.3 设计注意事项

```text
EventRecord 不允许包含裸指针。
所有 ID 对外使用字符串。
EventRecord 不允许直接包含大对象快照。
payload 中只放事件需要表达的最小事实。
```

## 5. EventPayload（事件载荷）

`EventPayload` 表示事件具体数据。

EventType 是事件大类，payload 是事件细节。

### 5.1 Payload 分类

| Payload | 中文名 | 对应事件 |
|---|---|---|
| ActionEventPayload | 行动事件载荷 | ActionStarted、ActionResolved、ActionBlocked |
| ObjectEventPayload | 对象事件载荷 | ObjectDiscovered、ObjectStateChanged |
| FeedbackEventPayload | 反馈事件载荷 | FeedbackGenerated |
| KnowledgeEventPayload | 知识事件载荷 | KnowledgeCreated、KnowledgeUpdated、KnowledgeContradicted |
| PropagationEventPayload | 传播事件载荷 | KnowledgeShared、PropagationFailed |
| CivilizationEventPayload | 文明事件载荷 | CapabilityUnlocked、CivilizationStageChanged |
| ErrorEventPayload | 错误事件载荷 | RuleErrorOccurred |
| StateEventPayload | 状态事件载荷 | StatePersisted、ResourceChanged、EntityStateChanged |
| RandomEventPayload | 随机事件载荷 | RandomRolled |

### 5.2 ActionEventPayload（行动事件载荷）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| action_type | ActionType | 行为类型 | 吃、用、采集、传播等 |
| actor_id | EntityId | 行动者 | 谁执行行动 |
| target_ids | list<string> | 目标 | 行动目标 |
| cost_summary | object | 成本摘要 | 时间、体力、资源等成本 |
| result_summary | string | 结果摘要 | 行动结果简述 |

### 5.3 FeedbackEventPayload（反馈事件载荷）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| feedback_type | FeedbackType | 反馈类型 | 生理、知识、风险等 |
| intensity | FeedbackIntensity | 强度 | 反馈强度 |
| reliability | FeedbackReliability | 可靠性 | 反馈可靠性 |
| source | FeedbackSource | 来源 | 反馈来源 |
| message_key | string | 文案键 | 前端翻译键 |
| related_effects | list<string> | 相关效果 | 饥饿下降、中毒、治疗等 |

### 5.4 KnowledgeEventPayload（知识事件载荷）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| knowledge_id | KnowledgeId | 知识 ID | 被创建或更新的知识 |
| subject_id | string | 知识主体 | 对象、区域、行为等主体 |
| action_type | ActionType | 相关行为 | 吃、用、采集、组合等 |
| effect_summary | string | 效果摘要 | 知识描述的效果 |
| confidence_delta | Delta | 置信度变化 | 本次事件导致的置信度变化 |
| status_after | KnowledgeStatus | 后续状态 | 事件后的知识状态 |

### 5.5 PropagationEventPayload（传播事件载荷）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| knowledge_id | KnowledgeId | 知识 ID | 被传播的知识 |
| channel | PropagationChannelType | 传播渠道 | 教学、演示、标记等 |
| result | PropagationResultType | 传播结果 | 接受、误解、失败等 |
| block_reason | PropagationBlockReason | 阻断原因 | 失败原因 |
| sender_id | string | 传播者 | 谁传播 |
| receiver_ids | list<string> | 接收者 | 谁接收 |

### 5.6 CivilizationEventPayload（文明事件载荷）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| stage_before | CivilizationStage | 原阶段 | 变化前阶段 |
| stage_after | CivilizationStage | 新阶段 | 变化后阶段 |
| capability | CapabilityType | 能力 | 解锁或变化的能力 |
| unlock_conditions | list<UnlockConditionType> | 解锁条件 | 为什么解锁 |
| tribe_id | TribeId | 族群 ID | 相关族群 |

### 5.7 ErrorEventPayload（错误事件载荷）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| error | ErrorDetail | 错误详情 | 错误码、严重级别、步骤等 |
| recoverable | bool | 是否可恢复 | 是否可继续流程 |
| state_mutated | bool | 是否已改状态 | 错误前是否发生状态变化 |

### 5.8 RandomEventPayload（随机事件载荷）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| stream | RandomStream | 随机流 | 使用的随机流 |
| seed | RandomSeed | 种子 | 随机种子 |
| purpose | string | 用途 | 为什么随机 |
| roll_index | Count | 序号 | 第几次随机 |
| result_summary | string | 结果摘要 | 随机结果摘要 |

## 6. EventBuilder / EventFactory（事件构造器）

研发不应该到处手写 `EventRecord`。

事件必须通过构造器创建。

### 6.1 推荐构造方法

```text
make_command_received(...)
make_action_started(...)
make_action_resolved(...)
make_action_blocked(...)
make_feedback_generated(...)
make_knowledge_created(...)
make_knowledge_updated(...)
make_knowledge_shared(...)
make_propagation_failed(...)
make_capability_unlocked(...)
make_rule_error_occurred(...)
make_random_rolled(...)
```

### 6.2 构造器责任

```text
自动填充 EventId。
自动绑定 CommandId、Tick、GameTime。
自动绑定 RuleStepType。
自动设置默认 visibility 和 importance。
校验 payload 与 EventType 是否匹配。
禁止创建字段不完整的事件。
```

## 7. EventStream（事件流）

`EventStream` 是一次命令或一次规则执行产生的有序事件集合。

### 7.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| command_id | CommandId | 命令 ID | 本事件流对应的命令 |
| correlation_id | string | 关联链路 | 本次执行的链路 ID |
| events | list<EventRecord> | 事件列表 | 有序事件 |
| highest_importance | EventImportance | 最高重要度 | 本流中最高重要事件 |
| has_error | bool | 是否有错误 | 是否包含错误事件 |

### 7.2 使用规则

```text
EventStream 内事件顺序必须稳定。
同一 CommandId 下事件必须可完整查询。
EventStream 返回给前端前必须通过 EventValidation。
EventStream 写入 EventStore 后不允许修改。
```

## 8. EventStore（事件存储）

`EventStore` 负责持久化事件。

### 8.1 存储规则

```text
事件只追加，不覆盖。
历史事件不直接修改。
修正历史时追加 CorrectionEvent。
每条事件绑定 StateVersion。
EventStore 写入失败属于 storage_write_failed。
```

### 8.2 查询能力

第一阶段 EventStore 至少支持：

```text
按 EventId 查询。
按 CommandId 查询。
按 Tick / GameTime 范围查询。
按 EventType 查询。
按 source_id 查询。
按 target_id 查询。
按 importance 查询。
按 tags 查询。
```

## 9. EventBus（事件总线）

`EventBus` 负责把事件分发给订阅者。

第一阶段采用进程内同步事件总线：`InProcessEventBus`。

### 9.1 接口概念

```text
publish(events) -> EventDeliveryResult
subscribe(event_type, subscriber)
unsubscribe(event_type, subscriber)
dispatch(event) -> EventDeliveryResult
```

### 9.2 重要约束

```text
EventBus 不修改核心世界状态。
核心世界状态只能由规则管线和 StateChange 修改。
EventBus 应在 StateChange 成功应用、EventStore 成功写入后再发布。
订阅者失败默认不回滚世界状态。
关键订阅者失败需要记录 EventDeliveryError。
```

## 10. EventSubscriber（事件订阅者）

`EventSubscriber` 是事件消费者接口。

### 10.1 第一阶段订阅者

| 订阅者 | 中文名 | 作用 |
|---|---|---|
| FrontendProjectionSubscriber | 前端投影订阅者 | 生成前端可读消息和视图变化 |
| LogSubscriber | 日志订阅者 | 写入工程日志和调试日志 |
| ReplaySubscriber | 复盘订阅者 | 记录复盘需要的事件轨迹 |
| DebugSubscriber | 调试订阅者 | 开发期观察事件 |
| TestProbeSubscriber | 测试探针订阅者 | 测试中捕获事件断言 |

### 10.2 后续可扩展订阅者

```text
AchievementSubscriber：成就系统。
QuestSubscriber：任务系统。
NotificationSubscriber：通知系统。
CivilizationObserver：文明观察器。
AITrainingSubscriber：AI 学习或分析系统。
```

### 10.3 订阅者失败规则

```text
订阅者失败不应默认导致规则失败。
订阅者失败必须记录错误事件或交付结果。
如果订阅者被标记为 critical_subscriber，失败可升级为 SystemError。
```

## 11. EventProjection（事件投影）

`EventProjection` 将事件转换成读模型。

### 11.1 常见投影

```text
前端消息列表。
玩家时间线。
族群知识变化摘要。
调试面板。
复盘轨迹。
错误报告。
重要事件提醒。
```

### 11.2 投影原则

```text
投影可以重建。
投影不是核心世界状态。
前端读投影，不直接推导规则结果。
投影失败不应破坏核心状态。
```

## 12. EventReplay（事件复盘）

`EventReplay` 用于解释历史过程。

### 12.1 复盘与存档恢复的区别

```text
StateSnapshot 用于快速恢复世界状态。
EventReplay 用于解释发生过程。
```

### 12.2 复盘要回答的问题

```text
玩家为什么中毒？
某个知识是怎么形成的？
谁把知识传给了谁？
族群为什么解锁守火能力？
哪个命令导致状态变化？
随机结果来自哪个随机流？
```

## 13. EventVisibility（事件可见性）

`EventVisibility` 描述谁能看见事件。

### 13.1 枚举建议

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Hidden | hidden | 隐藏 | 默认不对玩家显示，只供系统使用 |
| PlayerVisible | player_visible | 玩家可见 | 玩家前端可见 |
| TribeVisible | tribe_visible | 族群可见 | 族群层面可见 |
| DebugOnly | debug_only | 仅调试 | 研发调试可见 |
| ReplayOnly | replay_only | 仅复盘 | 复盘时可见 |

### 13.2 使用示例

```text
RandomRolled：DebugOnly
吃红浆果恢复饥饿：PlayerVisible
族群学会采集红浆果：TribeVisible
状态保存成功：Hidden
复盘解释事件：ReplayOnly
```

## 14. EventImportance（事件重要程度）

`EventImportance` 描述事件重要程度。

### 14.1 枚举建议

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Trace | trace | 细节 | 细粒度过程，默认不强提示 |
| Normal | normal | 普通 | 常规玩家可见事件 |
| Important | important | 重要 | 需要明显提示 |
| Critical | critical | 关键 | 重大变化或严重问题 |

### 14.2 使用示例

```text
火堆燃料减少 1：Trace
吃红浆果恢复饥饿：Normal
发现毒蘑菇有毒：Important
解锁守火能力：Critical
存档损坏：Critical
```

## 15. EventValidation（事件校验）

每条事件写入前必须校验。

### 15.1 通用校验规则

```text
event_id 必填。
event_type 必填。
tick / game_time 必填。
state_version 必填。
payload 必填。
visibility 必填。
importance 必填。
payload 类型必须匹配 event_type。
系统事件可没有 command_id，但必须说明 source_id。
非系统事件必须有 command_id。
```

### 15.2 特殊校验规则

```text
RuleErrorOccurred 必须携带 ErrorDetail。
RandomRolled 必须携带 RandomStream、RandomSeed、purpose、roll_index。
KnowledgeCreated 必须携带 KnowledgeId 或 KnowledgeClaim 摘要。
KnowledgeShared 必须携带 sender_id、receiver_ids、channel、result。
CapabilityUnlocked 必须携带 capability 和 unlock_conditions。
ActionResolved 必须携带 action_type、actor_id 和 result_summary。
```

### 15.3 校验失败处理

```text
EventValidation 失败属于 RuleError。
不允许写入 EventStore。
不允许发布到 EventBus。
返回 rule_output_invalid 或 validation_failed。
```

## 16. 事件生成流程

标准流程：

```text
1. 接收 Command。
2. 校验 Command。
3. 执行规则。
4. 生成 StateChange 草稿。
5. 生成 Event 草稿。
6. 校验 StateChange。
7. 应用 StateChange。
8. 绑定 Event 的 StateVersion / StateChangeId。
9. 校验 Event。
10. 写入 EventStream。
11. 写入 EventStore。
12. EventBus 发布事件。
13. EventProjection 更新读模型。
14. 返回 CommandResult 给前端。
```

关键约束：

```text
EventBus 发布必须在状态成功应用、事件成功存储之后。
前端拿到的事件必须对应已经成功写入的状态版本。
```

## 17. 事件与其他系统的关系

### 17.1 Event 与 Command

```text
Command 表示玩家或系统想做什么。
Event 表示命令结算后发生了什么。
一个 Command 可以产生多个 Event。
```

### 17.2 Event 与 StateChange

```text
StateChange 记录状态怎么变。
Event 记录发生了什么事实。
StateChange 给机器恢复世界。
Event 给人、前端、日志、复盘和测试理解过程。
```

示例：

```text
StateChange：hunger 80 -> 60
Event：FeedbackGenerated，玩家感觉饥饿降低
```

### 17.3 Event 与 Feedback

```text
Feedback 不单独做事件流。
Feedback 作为 FeedbackGenerated 事件的 payload。
前端只订阅 Event，不需要同时订阅 Feedback 和 Event 两套系统。
```

### 17.4 Event 与 Error

```text
ErrorDetail 描述错误。
RuleErrorOccurred 是错误事件。
不是所有玩法负面结果都是 Error。
吃毒蘑菇中毒是 CriticalFailure 玩法事件，不是 SystemError。
```

## 18. 前端消费规则

```text
前端通过 Event / Projection 展示结果。
前端不根据 ErrorCode 自己修改状态。
前端根据 EventVisibility 决定是否展示。
前端根据 EventImportance 决定展示强度。
前端根据 message_key 做本地化显示。
前端需要详情时按 CommandId 拉取事件流。
```

## 19. 强化学习预留说明

事件系统可以为后续强化学习提供数据基础，但事件系统本身不直接承担强化学习训练职责。

### 19.1 设计原则

```text
事件系统记录事实。
投影系统生成观察。
奖励模型计算奖励。
轨迹系统导出训练样本。
强化学习系统训练策略。
```

因此当前阶段只做预留，不把强化学习细节塞进事件系统。

### 19.2 建议预留字段

未来如果接入强化学习，`EventRecord` 可扩展以下可选字段：

| 字段 | 中文名 | 用途 |
|---|---|---|
| episode_id | 训练局 ID | 标记事件属于哪一局或哪一段训练 |
| decision_id | 决策 ID | 标记事件来自哪一次 AI 决策 |
| actor_control_type | 行动者控制类型 | 区分 Human、AI、Script、System |

这些字段第一阶段可以不实现，但命名和语义先保留。

### 19.3 建议预留扩展点

| 扩展点 | 中文名 | 用途 |
|---|---|---|
| RLObservationProjection | 强化学习观察投影 | 将状态和事件转换成 AI 可读观察 |
| TrajectoryReplay | 训练轨迹复盘 | 按训练步导出样本 |
| reward_hint | 奖励提示 | 事件可携带奖励提示，但不作为最终奖励 |

### 19.4 与强化学习样本的关系

强化学习通常需要以下样本结构：

```text
observation
action
reward
next_observation
done
info
```

事件系统不能直接等同于这份样本。

建议未来流程：

```text
EventRecord / StateSnapshot
        ↓
EventProjection
        ↓
RLTrajectoryBuilder
        ↓
Transition 样本
        ↓
强化学习训练
```

### 19.5 注意事项

```text
事件不是奖励。
KnowledgeCreated 只是说明产生了知识，不代表一定是正奖励。
吃毒蘑菇中毒可能同时产生知识收益和生命损失。
最终奖励应由 RewardModel 根据事件、状态变化和训练目标计算。
不要让强化学习系统直接依赖原始事件内部结构，应通过投影层读取。
```

## 20. 日志与复盘规则

### 20.1 日志规则

```text
所有 RuleErrorOccurred 必须进入工程日志。
Critical 事件必须进入高优先级日志。
RandomRolled 必须记录随机流和用途。
事件日志必须能追踪 CommandId、EventId、StateVersion。
```

### 20.2 复盘规则

```text
复盘按 Tick / EventId 顺序读取事件。
复盘不直接修改世界状态。
复盘用于解释过程，不替代 StateSnapshot。
复盘必须能展示知识形成、传播和文明解锁链路。
```

## 21. 组合测试样例

以下样例用于检查事件系统是否能支撑核心玩法。

| 场景 | 关键事件 | Payload | Visibility | Importance | 关键断言 |
|---|---|---|---|---|---|
| 接收吃红浆果命令 | CommandReceived | ActionEventPayload | DebugOnly | Trace | 绑定 CommandId |
| 吃红浆果成功 | ActionResolved | ActionEventPayload | PlayerVisible | Normal | 有 actor_id、target_id |
| 饥饿降低反馈 | FeedbackGenerated | FeedbackEventPayload | PlayerVisible | Normal | feedback_type=Physiological |
| 形成红浆果可食用知识 | KnowledgeCreated | KnowledgeEventPayload | PlayerVisible | Important | subject + action + effect 完整 |
| 吃毒蘑菇中毒 | ActionResolved / FeedbackGenerated | Action + Feedback | PlayerVisible | Important | 是 CriticalFailure，不是 SystemError |
| 目标太远无法采集 | ActionBlocked | ActionEventPayload | PlayerVisible | Normal | 不产生 StateChange |
| 传播红浆果知识成功 | KnowledgeShared | PropagationEventPayload | TribeVisible | Important | 有 sender、receiver、channel |
| 传播知识失败 | PropagationFailed | PropagationEventPayload | PlayerVisible | Normal | 有 block_reason |
| 族群解锁守火能力 | CapabilityUnlocked | CivilizationEventPayload | TribeVisible | Critical | 有 capability 和 unlock_conditions |
| 随机判定狼是否靠近 | RandomRolled | RandomEventPayload | DebugOnly | Trace | 有 stream、seed、purpose |
| 规则输出非法事件 | RuleErrorOccurred | ErrorEventPayload | DebugOnly | Critical | EventValidation 失败不写入 EventStore |
| 状态保存成功 | StatePersisted | StateEventPayload | Hidden | Trace | 绑定 StateVersion |

测试结论：

```text
当前事件系统可以表达行动、反馈、知识、传播、文明、随机、错误和状态持久化事实。
事件与 StateChange 边界清晰。
Feedback 已统一作为 Event payload。
EventStore 负责保存事实，EventBus 负责分发事实，EventProjection 负责生成读模型。
前端、日志、复盘和测试都可以基于事件工作，而不侵入规则代码。
```

## 22. 研发实现注意事项

```text
禁止在业务代码中手写不完整 EventRecord。
必须通过 EventBuilder / EventFactory 创建事件。
EventStore 只追加，不覆盖。
EventBus 不允许修改核心世界状态。
EventValidation 失败必须阻止写入。
前端展示依赖事件，不依赖规则内部对象。
订阅者失败必须记录，但默认不回滚规则结算。
新增 EventType 必须同步补充 payload、校验、翻译和测试。
```

## 23. 第一阶段验收标准

```text
每个 Command 都能查询到对应 EventStream。
每条 EventRecord 都能通过 EventValidation。
每条玩家可见事件都有 visibility 和 importance。
每个 RuleErrorOccurred 都携带 ErrorDetail。
每个 RandomRolled 都携带随机流信息。
每个 KnowledgeCreated 都能说明对象、行为、效果。
每个 CapabilityUnlocked 都能说明解锁条件。
前端只依赖 Event / Projection 即可展示规则结果。
测试可以通过事件断言规则过程。
```
