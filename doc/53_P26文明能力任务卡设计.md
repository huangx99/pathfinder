# P26 文明能力任务卡设计

## 1. 阶段定位

P26 是《先驱者》从“知识、族群、共同作战、族群冲突”进入“文明能力”的第一阶段。

P18-P20 已经完成：

```text
知识形成。
知识冲突与修正。
知识传播。
```

P23-P25 已经完成：

```text
族群状态。
共同作战。
族群冲突。
```

P26 要解决的是：

```text
族群如何把长期稳定的知识、传播覆盖、重复实践、组织协作、冲突经验和安全资源条件，转化为可解释、可复盘、可投影的文明能力。
```

一句话：

```text
P26 让“知道某件事”进一步变成“族群稳定掌握某种能力”，但不能把一次成功、单个知识或前端判断直接当作文明能力。
```

P26 是文明能力规则层，不是完整科技树 UI，不是生产链，不是对象反应，不是危险探索，不是战斗伤害系统。

---

## 2. 本阶段目标

P26 必须新增文明能力模块，建议新建 `civilization` 模块，而不是继续塞进 `tribe`。

它必须做到：

1. 定义文明能力最小运行时类型：`CivilizationCapability`、`CivilizationCapabilityState`、`CivilizationProgress`、`CapabilityUnlockCondition`、`CapabilityEffectProjection`。
2. 定义文明解析输入输出：`CivilizationResolveInput`、`CivilizationResolveResult`、`CivilizationProjection`、`CivilizationTrace`。
3. 支持能力成熟度：`Candidate`、`Emerging`、`Stable`、`Institutionalized`，并区分 `Suspended`、`Degraded`、`Lost`。
4. 支持能力可用性：能力可以“已掌握但当前不可用”，不能把临时阻断当作遗忘。
5. 只读取知识、传播、族群、共同作战、族群冲突的安全摘要和投影。
6. 不直接读取 raw `GameState`、`SaveGame`、hidden truth。
7. 不绕过 P25 的 `ConflictResolutionResult`、`ConflictProjection`、`ConflictTrace` 去推导冲突胜负。
8. 输出能力效果草稿、文明状态草稿、事件草稿、投影和 trace。
9. 第一版至少支持 4 类能力闭环：`IdentifyEdible`、`IdentifyDanger`、`SafeForaging`、`ConflictDeescalation`。
10. 必须有测试证明：能力不是 bool unlocked、不是单次成功、不是前端推导、不是阶段等级条。

核心原则：

```text
文明能力 = 稳定知识 + 传播覆盖 + 重复实践 + 族群状态 + 安全证据。
文明阶段只是摘要，不是能力来源。
能力效果只输出草稿，不直接改世界状态。
能力解锁必须确定性、可解释、可复盘。
```

## 3. 玩家体验映射

P26 让玩家第一次感受到：

```text
我教会族群的知识，不只是被记住，还会改变族群整体能力。
族群反复安全采集后，会形成“安全采集”能力。
族群多次把危险传播清楚后，会形成“识别危险”能力。
族群多次低损撤退或停战后，会形成“冲突降级”候选能力。
能力需要维持，资源断供、分裂风险、误解传播会让能力暂停或退化。
```

### 3.1 玩家未来感受

示例一：安全采集。

```text
你多次尝试并确认红果可食用。
你把知识教给同伴。
族群成员反复采集已知食物，没有再把腐烂红果当成普通红果。
系统提示：族群开始形成“安全采集”能力。
一段时间后，该能力稳定，采集误食风险下降。
```

示例二：冲突降级。

```text
你们与陌生族群多次争夺资源。
族群学会在士气低时撤退，在资源压力下降时提出停战。
多次低损冲突后，系统提示：族群形成“冲突降级”能力候选。
它不是战斗胜利能力，而是降低损失、保留文明积累的社会能力。
```

这里的重点不是“科技树亮了一个图标”，而是：

```text
玩家能理解能力为什么出现。
玩家能看到缺什么条件。
玩家能知道能力为什么暂停或退化。
玩家能从投影中看到下一步该做什么。
```

## 5. 本阶段不做什么

P26 明确禁止：

```text
不做科技树 UI。
不做 H5 新界面。
不做 HTTP / WebSocket / JSON API。
不做完整生产链。
不做对象反应，这是 P27。
不做危险探索压力，这是 P28。
不做存档前置，这是 P29。
不做复盘强化，这是 P30。
不做协议扩展，这是 P31。
不做战斗伤害、命中、HP、死亡、击杀、掉落。
不做专家 NPC 教学。
不做 NPC 谋害识别。
不做大型政治、外交、贸易、联盟。
不读 raw GameState。
不读 SaveGame。
不读 hidden truth。
不让前端推导能力解锁。
不让配置直接写 unlocked=true。
不让事件直接修改 CivilizationState。
不使用 AgentRuntime 或 Policy 作为文明规则权威。
```

## 6. 必须写死的注意点

### 6.1 能力不是 bool unlocked

禁止：

```cpp
bool unlocked;
int civilization_level;
```

必须使用：

```text
CapabilityMaturityState
CapabilityUsabilityState
CivilizationProgress
CapabilityEvidenceBundle
CapabilityRequirementSnapshot
```

中文解释：

```text
能力不是开关。
能力有候选、萌芽、稳定、制度化、暂停、退化、丢失。
能力成熟度和当前可用性必须分开。
```

### 6.2 文明阶段不是等级条

禁止：

```text
stage = level_3，所以自动解锁 FireKeeping。
```

正确：

```text
稳定能力集合 + 证据 + 族群凝聚 + 长期记忆 -> 推导 stage。
stage 只是结果摘要，不是能力来源。
```

### 6.3 单次成功不能解锁能力

禁止：

```text
吃一次红果成功 -> SafeForaging Stable。
一次停战成功 -> ConflictDeescalation Stable。
```

正确：

```text
单次成功只能产生 candidate 或 evidence。
Stable 必须满足重复成功、传播覆盖、行为采纳和风险可控。
```

### 6.4 P26 不能绕过 P25

P26 可以读取：

```text
ConflictResolutionResult。
ConflictProjection。
ConflictTrace。
ConflictStateDraft。
TribeRelationDraft。
```

P26 禁止：

```text
重新计算冲突胜负。
直接读取敌方真实状态。
用 hidden truth 判断谁赢。
把 P25 outcome 当作无条件能力解锁。
```

P25 输出只能作为证据，例如：

```text
Intimidated + low loss -> group_intimidation_evidence。
ForcedRetreat + low loss -> organized_retreat_evidence。
TruceOffered + repeated success -> conflict_deescalation_candidate。
Standoff + controlled safety pressure -> resource_defense_candidate。
```

### 6.5 P26 不能破坏 P23/P24/P25 状态草稿边界

P26 不直接修改：

```text
TribeState。
CombatCoordinationResult。
ConflictResolutionResult。
KnowledgeGraph。
```

P26 只能输出：

```text
CivilizationStateChangeDraft。
CapabilityEffectDraft。
EventDraft。
CivilizationProjection。
CivilizationTrace。
```

## 8. 枚举设计

如果 `02_通用枚举设计.md` 中已有同名枚举，必须复用现有枚举；如果代码中尚未实现，则在 P26 模块落地最小集合。

### 8.1 CivilizationStage

文明阶段摘要。

```cpp
enum class CivilizationStage {
    Unknown,
    Awakening,
    Foraging,
    FireKeeping,
    ToolUsing,
    OrganizedTribe,
    SymbolicCulture,
    ProtoCivilization,
    TestOnly
};
```

中文说明：

| 枚举 | 中文 | 含义 |
|---|---|---|
| Unknown | 未知 | 非法默认值 |
| Awakening | 觉醒期 | 初始阶段，开始形成认知与传播 |
| Foraging | 采集期 | 具备稳定识别食物/危险和安全采集能力 |
| FireKeeping | 守火期 | 能稳定保存、使用或围绕火形成行为 |
| ToolUsing | 用具期 | 能稳定使用或制作基础工具 |
| OrganizedTribe | 组织部落期 | 出现稳定分工、教学、任务分配 |
| SymbolicCulture | 符号文化期 | 出现标记、记录、长期传承 |
| ProtoCivilization | 原文明期 | 多领域稳定能力组合形成复杂社会基础 |
| TestOnly | 测试专用 | 正式 DTO 禁止 |

注意：

```text
stage 不能作为能力前置的唯一条件。
stage 不能直接解锁能力。
```

### 8.2 CapabilityType

文明能力类型。

```cpp
enum class CapabilityType {
    Unknown,
    IdentifyEdible,
    IdentifyDanger,
    SafeForaging,
    TeachingPractice,
    BasicToolUse,
    FireKeeping,
    FireDefense,
    OrganizedRetreat,
    ConflictDeescalation,
    ResourceDefense,
    TaskAssignment,
    LongTermMemory,
    SymbolRecording,
    TestOnly
};
```

第一版 P26 必须至少实现：

```text
IdentifyEdible。
IdentifyDanger。
SafeForaging。
ConflictDeescalation。
```

其余可作为枚举占位，但不要实现复杂玩法。

### 8.3 CapabilityMaturityState

能力成熟度。

```cpp
enum class CapabilityMaturityState {
    Unknown,
    Candidate,
    Emerging,
    Stable,
    Institutionalized,
    Degraded,
    Lost,
    TestOnly
};
```

中文说明：

| 枚举 | 中文 | 含义 |
|---|---|---|
| Candidate | 候选 | 有证据苗头，但未形成稳定能力 |
| Emerging | 萌芽 | 多项条件满足，有重复实践但不稳定 |
| Stable | 稳定 | 已可稳定使用，可产生正式效果 |
| Institutionalized | 制度化 | 长期使用、传播、分工稳定 |
| Degraded | 退化 | 能力仍存在但可靠性下降 |
| Lost | 丢失 | 证据断裂或族群不再掌握 |
| TestOnly | 测试专用 | 正式 DTO 禁止 |

### 8.4 CapabilityUsabilityState

能力可用性。

```cpp
enum class CapabilityUsabilityState {
    Unknown,
    Usable,
    BlockedByResource,
    BlockedByEnvironment,
    BlockedByCohesion,
    BlockedByConflict,
    Suspended,
    TestOnly
};
```

中文说明：

```text
maturity 表示会不会。
usability 表示现在能不能用。
```

能力变化原因。

```cpp
enum class CapabilityChangeReason {
    Unknown,
    NewEvidence,
    RequirementMet,
    RequirementMissing,

    PropagationFailed,
    ResourceChanged,
    PopulationChanged,
    ConflictChanged,
    MemoryDecayed,
    TestOnly
};
```

### 8.6 CapabilityEffectTarget

能力效果目标。

```cpp
enum class CapabilityEffectTarget {
    Unknown,
    ActionAvailability,
    Risk,
    Knowledge,
    Propagation,
    TribeCohesion,
```

效果操作类型。


## 9. 数据结构设计

### 9.1 CivilizationCapability

能力定义的运行时只读视图。

字段建议：

```text
capability_type: CapabilityType
display_key: string
domain_key: string
priority: int
required_conditions: list<CapabilityUnlockCondition>
effect_definitions: list<CapabilityEffectDefinition>
reason_keys: list<string>
```

校验：

```text
capability_type 不能 Unknown/TestOnly。
display_key/domain_key 不能为空，不能包含 unsafe key。
required_conditions 不能只包含 RequiredStage。
effect_definitions 不能直接修改状态。
```

### 9.2 CivilizationCapabilityState

族群掌握某个能力的状态。

字段建议：

```text
tribe_id: TribeId
capability_type: CapabilityType
maturity: CapabilityMaturityState
usability: CapabilityUsabilityState
progress: CivilizationProgress
stability: double 0..1
coverage: double 0..1
evidence: CapabilityEvidenceBundle
active_effect_keys: list<string>
blocked_reasons: list<CapabilityChangeReason>
last_changed_tick: Tick
reason_keys: list<string>
```

校验：

```text
Stable/Institutionalized 必须有 evidence。
Lost 不能有 active_effect_keys。
Usability 非 Usable 时必须有 blocked_reasons。
progress 必须 0..1。
coverage/stability 必须 0..1。
```

### 9.3 CivilizationProgress

能力进度。

字段建议：

```text
candidate_score: double 0..1
emerging_score: double 0..1
stable_score: double 0..1
institutionalized_score: double 0..1
success_count: int
failure_count: int
last_success_tick: optional<Tick>
last_failure_tick: optional<Tick>
reason_keys: list<string>
```

说明：

```text
Progress 是能力成熟推导依据，不是前端经验条权威。
前端只能展示 projection，不能用 progress 自己判断解锁。
```

### 9.4 CapabilityUnlockCondition

能力解锁条件。

字段建议：

```text
condition_key: string
condition_type: UnlockConditionType
required_score: double 0..1
current_score: double 0..1
met: bool
source_evidence_ids: list<EventId>
reason_keys: list<string>
```

说明：

```text
condition_type 必须来自通用枚举。
第一版可用 string fallback，但必须集中封装，后续替换成强枚举。
```

### 9.5 CapabilityEvidenceBundle

能力证据束。

字段建议：

```text
knowledge_ids: list<KnowledgeId>
propagation_event_ids: list<EventId>
practice_event_ids: list<EventId>
conflict_event_ids: list<EventId>
tribe_state_version: StateVersion
confidence: double 0..1
reason_keys: list<string>
```

安全要求：

```text
不能包含 hidden truth。
不能包含 raw GameState。
不能包含 HP、死亡、击杀、掉落。
不能把 P25 trace 原文全部塞进 evidence，只能引用安全 event_id / summary key。
```

### 9.6 CapabilityEffectDefinition

能力效果定义。

字段建议：

```text
effect_key: string
target: CapabilityEffectTarget
operation: EffectOperationType
value_key: string
value_number: optional<double>
priority: int
reason_keys: list<string>
```

禁止：

```text
effect_key = kill_enemy。
target = raw_state。
operation = direct_mutate_state。
```

### 9.7 CapabilityEffectDraft

能力效果草稿。

字段建议：

```text
draft_id: EventId
tribe_id: TribeId
capability_type: CapabilityType
effect_key: string
target: CapabilityEffectTarget
operation: EffectOperationType
value_key: string
value_number: optional<double>
strength: double 0..1
source_evidence_ids: list<EventId>
reason_keys: list<string>
```

说明：

```text
EffectDraft 给规则管线后续消费。
P26 不直接应用效果。
```

### 9.8 CivilizationState

文明状态聚合根。

字段建议：

```text
tribe_id: TribeId
stage: CivilizationStage
stage_confidence: double 0..1
capabilities: list<CivilizationCapabilityState>
active_effects: list<CapabilityEffectDraft>
blocked_capabilities: list<CapabilityType>
last_resolved_tick: Tick
version: StateVersion
reason_keys: list<string>
```

约束：

```text
同一个 capability_type 不能重复。
stage 必须能由 stable/institutionalized capabilities 推导。
active_effects 只能来自 Stable/Institutionalized 且 Usable 的能力。
```

### 9.9 CivilizationCapabilityCandidate

能力候选。

字段建议：

```text
candidate_id: EventId
tribe_id: TribeId
capability_type: CapabilityType
readiness: double 0..1
requirement_snapshot: CapabilityRequirementSnapshot
evidence: CapabilityEvidenceBundle
recommended_next_steps: list<string>
can_promote: bool
reason_keys: list<string>
```

### 9.10 CapabilityRequirementSnapshot

条件评估快照。

字段建议：

```text
evaluated_tick: Tick
total_score: double 0..1
met_condition_keys: list<string>
missing_condition_keys: list<string>
failed_requirements: list<string>
warning_requirements: list<string>
source_candidate_ids: list<EventId>
reason_keys: list<string>
```

### 9.11 CivilizationStateChangeDraft

文明状态变化草稿。

字段建议：

```text
change_id: EventId

stage_after: optional<CivilizationStage>

maturity_after: optional<CapabilityMaturityState>
```

### 9.12 CivilizationProjection

文明安全投影。

字段建议：

```text
tribe_id: TribeId
stage: CivilizationStage
stage_label_key: string
stable_capabilities: list<CapabilityType>
emerging_capabilities: list<CapabilityType>
blocked_capabilities: list<CapabilityType>
```

安全要求：

```text

Projection 不能包含内部 trace 全量数据。
```

### 9.13 CivilizationTrace

文明解析 trace。

字段建议：

```text
trace_id: EventId
input_summary_keys: list<string>
candidate_steps: list<string>
requirement_steps: list<string>
state_transition_steps: list<string>
effect_steps: list<string>
stage_steps: list<string>

```


### 11.1 CivilizationCandidateBuilder

```cpp
class CivilizationCandidateBuilder {
```

职责：

职责：

职责：

## 12. 第一版能力规则

### 12.1 IdentifyEdible

候选来源：

```text
known_edible_count >= 1。
teachable_knowledge_count >= 1。
```

Stable 建议条件：

```text
known_edible_count >= 2。
coverage >= 0.50。
misunderstanding_rate <= 0.25。
stable_tribe_knowledge_count >= 1。
```

效果草稿：

```text
target = Risk
operation = Multiply
value_number = 0.90
value_key = reduce_unknown_food_risk
```

### 12.2 IdentifyDanger

Stable 建议条件：

```text
known_danger_count >= 1。
coverage >= 0.45。
misunderstanding_rate <= 0.25。
conflicted_knowledge_count == 0 或风险可解释。
```

效果草稿：

```text
target = Risk
operation = Multiply
value_number = 0.88
value_key = reduce_known_danger_misuse
```

### 12.3 SafeForaging

前置：

```text
IdentifyEdible >= Emerging。
IdentifyDanger >= Emerging。
传播覆盖达标。
重复实践成功。
```

Stable 建议条件：

```text
known_edible_count >= 2。
known_danger_count >= 1。
coverage >= 0.55。
success_rate >= 0.70。
misunderstanding_rate <= 0.20。
```

效果草稿：

```text
target = ActionAvailability
operation = UnlockCandidate
value_key = safe_foraging_actions
```

以及：

```text
target = Risk
operation = Multiply
value_number = 0.85
value_key = reduce_foraging_risk
```

### 12.4 ConflictDeescalation

候选来源：

```text
P25 truce_success_count > 0。
P25 forced_retreat_low_loss_count > 0。
P25 standoff_controlled_count > 0。
```

Stable 建议条件：

```text
truce_success_count >= 2 或 forced_retreat_low_loss_count >= 2。
open_conflict_pressure <= 0.50。
loss_pressure <= 0.40。
coordination_score >= 0.45。
```

效果草稿：

```text
target = ConflictResolution
operation = Multiply
value_number = 0.90
value_key = reduce_conflict_loss_pressure
```

注意：

```text
ConflictDeescalation 不是“战斗胜利”。
它是降低冲突损失、提高撤退/停战可控性的社会能力。
```

## 13. 状态流转

### 13.1 成熟度晋升

```text
无状态 + 有候选 -> Candidate。
Candidate + 多条件满足 -> Emerging。
Emerging + 重复成功 + 传播覆盖 -> Stable。
Stable + 长期稳定 + 角色/行为制度化 -> Institutionalized。
```

第一版可先实现到 Stable，Institutionalized 只保留枚举和校验。

### 13.2 阻断

```text
资源不足 -> Usability = BlockedByResource。
分裂风险高 -> BlockedByCohesion。
开放冲突压力高 -> BlockedByConflict。
环境不匹配 -> BlockedByEnvironment。
```

第一版重点实现：

```text
BlockedByCohesion。
BlockedByConflict。
```

因为 P23/P25 已有相关安全摘要。

### 13.3 退化和丢失

第一版只实现最小退化：

```text
Stable + repeated_failure 或 evidence_confidence 太低 -> Degraded。
Degraded + 无证据 -> Lost。
```

不要写复杂时间系统。
