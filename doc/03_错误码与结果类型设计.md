# 先驱者：错误码与结果类型设计

## 1. 本文档目标

本文档用于设计《先驱者》规则引擎中的错误码与结果类型。

错误码与结果类型的目标不是“给程序报错用一下”，而是保证规则引擎在任何情况下都能明确回答：

```text
这次请求是否成功？
如果失败，失败在哪里？
失败是玩家操作问题、规则问题、数据问题，还是系统问题？
失败后世界状态是否仍然可信？
前端应该如何展示？
日志、复盘、测试应该如何定位？
```

第一阶段设计规则引擎需要的通用结果结构、错误分类、核心错误码、使用规范和测试场景。

## 2. 设计边界

### 2.1 本文档要解决的问题

```text

定义 ErrorCode 的命名和分类方式。

定义错误与 RuleStepType、RuleResultType、ErrorSeverity 的关系。
定义哪些错误可以恢复，哪些错误必须停止流程。
定义前端、日志、测试如何消费错误信息。
定义第一阶段核心错误码清单。
定义错误码组合测试样例。
```

## 3. 核心原则

### 3.1 规则结果必须显式表达

规则函数不能只返回 `bool`。

原因：

```text
bool 只能表达成功或失败。
但规则引擎需要知道失败原因、失败位置、严重程度、是否产生事件、是否修改状态。
```

因此第一阶段统一使用结果类型表达规则执行结果。

### 3.2 玩法失败不等于系统错误

必须区分：

```text
玩家行动失败：玩法结果，世界状态仍然可信。
系统执行失败：程序、配置、存档或规则错误，可能影响状态可信度。
```

示例：

```text
目标太远，无法采集：玩法阻断。
吃毒蘑菇中毒：玩法严重失败。
前端传入不存在的 ObjectId：用户输入错误。
对象配置缺少必要特性：数据错误。
规则试图传播裸 ObjectId：规则错误。
存档版本冲突且无法回滚：严重系统错误。
```

### 3.3 错误必须能定位到规则步骤

任何错误都应该尽量记录发生在哪个规则步骤。

依赖枚举：

```text
RuleStepType
RuleResultType
ErrorSeverity
```

示例：

```text
ValidateCommand 阶段发现 ObjectId 不存在。
CheckPreconditions 阶段发现目标距离过远。
ResolvePropagation 阶段发现 KnowledgeClaim 不完整。
ApplyStateChanges 阶段发现 StateVersion 冲突。
PersistState 阶段发现存档写入失败。
```

### 3.4 错误信息必须给机器读，不只给人看

错误信息至少分两层：

```text
机器层：稳定 ErrorCode，用于协议、测试、日志、分支判断。
人类层：message / 中文翻译，用于调试和前端展示。
```

规则判断不能依赖中文文案。

### 3.5 错误不能偷偷修改状态

如果规则结果是非法命令、数据错误、规则错误或致命错误，默认不得修改世界状态。

如果部分成功已经产生状态变化，必须通过事件和 StateChange 明确记录。

示例：

```text
吃未知蘑菇：可以产生饥饿下降、中毒、记忆创建。
传入不存在对象：不能修改世界状态，只能返回错误。
传播裸 ObjectId：不能创建知识，必须返回规则错误。
```

## 4. 核心类型总览

第一阶段确认以下结果与错误类型：

| 类型 | 中文名 | 作用 |
|---|---|---|
| Result<T> | 通用结果 | 表达一个函数是否成功，并携带值或错误 |
| ErrorCode | 错误码 | 稳定机器可读错误标识 |
| ErrorDetail | 错误详情 | 描述单个错误的完整上下文 |
| ErrorList | 错误列表 | 收集多个错误或警告 |
| RuleExecutionResult | 规则执行结果 | 规则管线最终返回结果 |
| CommandResult | 命令结果 | 玩家或系统命令执行后的外部结果 |
| ValidationResult | 校验结果 | 参数、配置、存档、状态校验结果 |
| StateApplyResult | 状态应用结果 | 状态变更写入结果 |

边界：

```text
Result<T> 是最底层通用结构。
ErrorDetail 是错误信息载体。
RuleExecutionResult 是规则管线内部结果。
CommandResult 是对前端或调用方返回的结果。
ValidationResult 专门用于校验，不直接修改状态。
StateApplyResult 专门用于状态变更写入。
```

## 5. ErrorCode（错误码）

`ErrorCode` 是稳定机器可读错误标识。

### 5.1 命名规则

错误码使用稳定英文字符串，对外格式为 `lower_snake_case`。

建议内部 C++ 使用 `enum class ErrorCode`，外部 JSON、日志、存档和测试使用字符串。

命名结构：

```text
领域 + 具体原因
```

示例：

```text
command_invalid_format
id_not_found
knowledge_claim_incomplete
state_version_conflict
storage_write_failed
```

### 5.2 错误码分类

第一阶段错误码分为十类：

| 分类 | 前缀 | 中文名 | 用途 |
|---|---|---|---|
| Common | common_ | 通用错误 | 通用兜底、未知错误、未实现 |
| Command | command_ | 命令错误 | 命令格式、参数、目标错误 |
| Id | id_ | ID 错误 | ID 缺失、不存在、类型不匹配 |
| Validation | validation_ | 校验错误 | 配置、参数、状态校验失败 |
| Preconditions | precondition_ | 前置条件错误 | 距离、成本、状态、能力不满足 |
| Rule | rule_ | 规则错误 | 规则内部不一致或禁止行为 |
| Knowledge | knowledge_ | 知识错误 | 认知、记忆、知识声明错误 |
| Propagation | propagation_ | 传播错误 | 知识传播失败或被阻断 |
| State | state_ | 状态错误 | 状态版本、状态应用、状态一致性错误 |
| Storage | storage_ | 存储错误 | 存档、加载、持久化错误 |

### 5.3 第一阶段核心错误码清单

#### Common 通用错误

| ErrorCode | 中文名 | 默认严重级别 | 默认结果 | 含义 |
|---|---|---|---|---|
| common_unknown | 未知错误 | Critical | SystemError | 未分类错误，只能作为兜底 |
| common_not_implemented | 未实现 | RuleError | SystemError | 功能尚未实现 |
| common_unsupported | 不支持 | Recoverable | Failed | 当前版本不支持该操作 |
| common_internal_invariant_broken | 内部不变量破坏 | Fatal | SystemError | 系统内部假设被破坏，状态可能不可信 |

#### Command 命令错误

| ErrorCode | 中文名 | 默认严重级别 | 默认结果 | 含义 |
|---|---|---|---|---|
| command_invalid_format | 命令格式非法 | UserError | InvalidCommand | 命令 JSON 或结构格式错误 |
| command_unknown_type | 未知命令类型 | UserError | InvalidCommand | 命令类型不存在或未注册 |
| command_missing_required_field | 缺少必填字段 | UserError | InvalidCommand | 命令缺少必要字段 |
| command_invalid_argument | 命令参数非法 | UserError | InvalidCommand | 参数值不合法 |
| command_not_allowed_now | 当前不允许执行 | Recoverable | Blocked | 当前阶段或状态不允许该命令 |
| command_duplicate | 重复命令 | Warning | NoEffect | 命令重复提交或已处理 |

#### Id ID 错误

| ErrorCode | 中文名 | 默认严重级别 | 默认结果 | 含义 |
|---|---|---|---|---|
| id_missing | ID 缺失 | UserError | InvalidCommand | 必需 ID 未提供 |
| id_invalid_format | ID 格式非法 | UserError | InvalidCommand | ID 字符串不符合规范 |
| id_not_found | ID 不存在 | UserError | InvalidCommand | 引用的实例或定义不存在 |
| id_type_mismatch | ID 类型不匹配 | UserError | InvalidCommand | 例如把 ObjectId 当成 TribeId 使用 |
| id_definition_not_found | 定义 ID 不存在 | DataError | SystemError | 配置引用了不存在的定义 |

#### Validation 校验错误

| ErrorCode | 中文名 | 默认严重级别 | 默认结果 | 含义 |
|---|---|---|---|---|
| validation_failed | 校验失败 | Recoverable | Failed | 通用校验失败 |
| validation_enum_unknown | 未知枚举值 | UserError | InvalidCommand | 外部传入未知枚举字符串 |
| validation_value_out_of_range | 数值越界 | UserError | InvalidCommand | 数值不在允许范围内 |
| validation_config_invalid | 配置非法 | DataError | SystemError | 配置文件结构或内容非法 |
| validation_save_invalid | 存档非法 | DataError | SystemError | 存档内容校验失败 |

#### Preconditions 前置条件错误

| ErrorCode | 中文名 | 默认严重级别 | 默认结果 | 含义 |
|---|---|---|---|---|
| precondition_target_too_far | 目标太远 | UserError | Blocked | 目标超出行动范围 |
| precondition_target_unavailable | 目标不可用 | Recoverable | Blocked | 目标不存在于可交互状态 |
| precondition_actor_unavailable | 行动者不可用 | Recoverable | Blocked | 行动者昏迷、死亡、忙碌或不可行动 |
| precondition_insufficient_cost | 成本不足 | UserError | Blocked | 时间、体力、资源等成本不足 |
| precondition_missing_capability | 缺少能力 | UserError | Blocked | 族群或行动者缺少必要能力 |
| precondition_missing_tool | 缺少工具 | UserError | Blocked | 行动需要工具但未满足 |
| precondition_environment_mismatch | 环境不匹配 | Recoverable | Blocked | 季节、天气、昼夜、地点不满足 |
| precondition_risk_too_high | 风险过高 | Warning | Blocked | 规则或策略阻止过高风险行动 |

#### Rule 规则错误

| ErrorCode | 中文名 | 默认严重级别 | 默认结果 | 含义 |
|---|---|---|---|---|
| rule_invariant_failed | 规则不变量失败 | RuleError | SystemError | 规则内部状态不一致 |
| rule_invalid_state_transition | 非法状态转移 | RuleError | SystemError | 状态不能从 A 变到 B |
| rule_missing_resolver | 缺少规则解析器 | RuleError | SystemError | 找不到处理该行为的规则模块 |
| rule_conflict | 规则冲突 | RuleError | Conflict | 多条规则产生互斥结果 |
| rule_random_stream_missing | 缺少随机流 | RuleError | SystemError | 需要随机但没有合法随机流 |
| rule_output_invalid | 规则输出非法 | RuleError | SystemError | 规则输出缺少必要事件或状态变更 |

| knowledge_claim_conflict | 知识声明冲突 | Warning | Conflict | 新知识与已有知识冲突 |
| knowledge_not_found | 知识不存在 | UserError | Failed | 请求引用的知识不存在 |
| knowledge_not_teachable | 知识不可传授 | UserError | Blocked | 知识状态尚未达到可传授 |
| knowledge_insufficient_confidence | 知识置信度不足 | Warning | Blocked | 证据不足，不允许传播或应用 |
| memory_not_found | 记忆不存在 | UserError | Failed | 请求引用的记忆不存在 |

#### Propagation 传播错误

| propagation_receiver_unavailable | 接收者不可用 | Recoverable | Blocked | 接收者无法参与传播 |
| propagation_channel_unavailable | 传播渠道不可用 | UserError | Blocked | 当前无法使用指定传播方式 |

### 5.4 错误码使用注意事项

```text
禁止用 common_unknown 掩盖已知错误。
禁止业务逻辑依赖中文错误文案。
新增错误码必须同步更新翻译、协议、测试和文档。
ErrorCode 一旦对外发布，不允许随意改名。
如果确实废弃错误码，应保留兼容映射并标记 deprecated。
```

## 6. ErrorDetail（错误详情）

`ErrorDetail` 描述单个错误的完整上下文。

### 6.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| code | ErrorCode | 错误码 | 是 | 稳定机器可读错误标识 |
| severity | ErrorSeverity | 严重级别 | 是 | 错误处理等级 |
| result_type | RuleResultType | 结果类型 | 是 | 对规则结果的影响 |
| step | RuleStepType | 规则步骤 | 否 | 错误发生在哪个规则步骤 |
| message_key | string | 文案键 | 是 | 用于翻译的稳定键 |
| debug_message | string | 调试信息 | 否 | 给研发看的补充信息 |
| related_ids | list<string> | 相关 ID | 否 | 相关对象、实体、命令、事件等 ID |
| field_path | string | 字段路径 | 否 | 发生错误的字段路径 |
| expected | string | 期望值 | 否 | 期望的值或类型 |
| actual | string | 实际值 | 否 | 实际收到的值或类型 |
| recoverable | bool | 是否可恢复 | 是 | 当前错误是否允许继续流程 |
| state_mutated | bool | 是否已修改状态 | 是 | 错误发生前是否已经修改状态 |

### 6.2 字段约束

```text
code、severity、result_type、message_key、recoverable、state_mutated 必填。
step 能填必须填。
debug_message 不作为协议稳定字段，不允许前端依赖。
field_path 用于命令、配置、存档校验。
related_ids 只放外部稳定字符串，不放裸指针或临时地址。
state_mutated = true 时，必须能通过事件和 StateChange 追踪已发生变化。
```

### 6.3 示例

目标太远：

```text
code = precondition_target_too_far
severity = UserError
result_type = Blocked
step = CheckPreconditions
message_key = error.precondition.target_too_far
related_ids = [actor_000001, obj_000042]
field_path = command.target_id
recoverable = true
state_mutated = false
```

规则试图传播裸 ObjectId：

```text
code = knowledge_claim_incomplete
severity = RuleError
result_type = SystemError
step = ResolvePropagation
message_key = error.knowledge.claim_incomplete
related_ids = [obj_000042]
field_path = propagation.knowledge_claim
recoverable = false
state_mutated = false
```

吃毒蘑菇导致中毒：

```text
code = common_unsupported 不适用。
这不是系统错误，应由 RuleResultType.CriticalFailure + Feedback/Event/StateChange 表达。
如果需要提示风险，使用玩法事件，不使用 ErrorCode。
```

## 7. Result<T>（通用结果）

`Result<T>` 是函数级返回结构。

### 7.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| ok | bool | 是否成功 | 表示是否获得可用 value |
| value | T | 成功值 | ok=true 时存在 |
| errors | ErrorList | 错误列表 | ok=false 或有警告时存在 |
| warnings | ErrorList | 警告列表 | 不阻断流程的问题 |

### 7.2 使用规则

```text
ok=true 表示 value 可用。
ok=false 表示 value 不可用，必须检查 errors。
warnings 不影响 ok。
Result<T> 不直接表达规则事件，规则事件属于 RuleExecutionResult。
```

### 7.3 禁止事项

```text
禁止 ok=false 但 errors 为空。
禁止 ok=true 但 value 缺失。
禁止把玩法负面结果全部塞进 errors。
禁止通过 debug_message 做程序分支判断。
```

## 8. ValidationResult（校验结果）

`ValidationResult` 用于命令、配置、存档、状态校验。

### 8.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| valid | bool | 是否通过 | 校验是否通过 |
| errors | ErrorList | 错误列表 | 阻断校验的问题 |
| warnings | ErrorList | 警告列表 | 不阻断但需要记录的问题 |

### 8.2 使用规则

```text
ValidationResult 不修改世界状态。
命令校验失败返回 UserError。
配置校验失败返回 DataError。
规则校验失败返回 RuleError。
存档校验失败通常是 DataError、Critical 或 Fatal。
```

## 9. RuleExecutionResult（规则执行结果）

`RuleExecutionResult` 是规则管线内部最终结果。

### 9.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| result_type | RuleResultType | 规则结果类型 | 成功、失败、阻断、系统错误等 |
| severity | ErrorSeverity | 最高严重级别 | 本次执行最高错误等级 |
| errors | ErrorList | 错误列表 | 阻断或失败原因 |
| warnings | ErrorList | 警告列表 | 非阻断问题 |
| events | list<Event> | 事件列表 | 本次执行产生的事件 |
| state_changes | list<StateChange> | 状态变更列表 | 本次执行产生的状态变化 |
| state_mutated | bool | 是否修改状态 | 是否已经应用状态变化 |
| final_state_version | StateVersion | 最终状态版本 | 执行后状态版本 |

### 9.2 使用规则

```text
Success 可以有 warnings。
Blocked 通常不应有 state_changes。
InvalidCommand 不应修改状态。
SystemError 默认不应修改状态；如果已经修改，必须 state_mutated=true 并记录补偿策略。
PartialSuccess 必须有 events 说明哪些成功、哪些失败。
CriticalFailure 是玩法严重失败，不等于 SystemError。
```

## 10. CommandResult（命令结果）

`CommandResult` 是返回给前端或外部调用方的结果。

### 10.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| command_id | CommandId | 命令 ID | 本次命令唯一标识 |
| accepted | bool | 是否被接受 | 命令是否进入规则处理 |
| result_type | RuleResultType | 结果类型 | 命令最终结果 |
| errors | ErrorList | 错误列表 | 给调用方处理的错误 |
| warnings | ErrorList | 警告列表 | 给调用方处理的警告 |
| events | list<Event> | 事件列表 | 前端展示依据 |
| state_version | StateVersion | 状态版本 | 返回后的状态版本 |

### 10.2 使用规则

```text
前端应优先读取 events 展示结果。
前端不应根据 ErrorCode 自己推导状态变化。
accepted=false 表示命令未进入核心规则结算。
accepted=true 但 result_type=Blocked 表示命令合法但当前条件不满足。
```

## 11. StateApplyResult（状态应用结果）

`StateApplyResult` 用于状态变更写入。

### 11.1 字段设计

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| applied | bool | 是否已应用 | StateChange 是否写入世界状态 |
| errors | ErrorList | 错误列表 | 写入失败原因 |
| applied_changes | list<StateChangeId> | 已应用变更 | 已成功应用的变更 ID |
| final_state_version | StateVersion | 最终状态版本 | 写入后的状态版本 |

### 11.2 使用规则

```text
如果 applied=false，不能假设任何状态已修改。
如果部分应用，必须返回 applied_changes，并由规则管线决定回滚或补偿。
状态版本冲突使用 state_version_conflict。
非法状态变化使用 state_change_invalid。
```

## 12. 错误与事件的关系

错误本身不等于事件，但关键错误需要产生事件。

建议规则：

| 情况 | 是否产生 Event | 推荐 EventType |
|---|---|---|
| 命令格式非法 | 是 | RuleErrorOccurred |
| 前置条件阻断 | 是 | ActionBlocked |
| 玩法失败 | 是 | ActionResolved / FeedbackGenerated |
| 知识冲突 | 是 | KnowledgeContradicted |
| 传播失败 | 是 | PropagationFailed |
| 能力解锁失败 | 视情况 | RuleErrorOccurred / FeedbackGenerated |
| 系统错误 | 是 | RuleErrorOccurred |
| 存档失败 | 是 | RuleErrorOccurred |

注意：

```text
前端展示应以 Event 为主。
ErrorDetail 用于解释失败原因。
错误不应绕过事件系统直接改 UI。
```

## 13. 错误与状态修改的关系

### 13.1 默认状态修改规则

| RuleResultType | 默认是否允许修改状态 | 说明 |
|---|---|---|
| Success | 允许 | 正常成功 |
| PartialSuccess | 允许 | 必须明确哪些变化已发生 |
| NoEffect | 不允许 | 没有效果就不应修改状态，除非追加日志事件 |
| Failed | 允许 | 玩法失败可以产生后果 |
| Blocked | 不允许 | 前置条件不满足，不进入核心结算 |
| InvalidCommand | 不允许 | 非法命令不能修改状态 |
| Conflict | 通常不允许 | 除非记录知识冲突事件或记忆 |
| Interrupted | 允许 | 中断前可能已消耗成本或产生风险 |
| CriticalFailure | 允许 | 玩法严重失败，状态仍可信 |
| SystemError | 不允许 | 系统错误默认不得修改状态 |

### 13.2 关键区别

```text
CriticalFailure 是玩法结果：例如中毒、受伤、引来危险。
SystemError 是系统错误：例如配置缺失、状态冲突、规则输出非法。
两者绝不能混用。
```

## 14. 前端、日志、测试消费规则

### 14.1 前端消费规则

```text
前端显示结果：看 events。
前端显示失败原因：看 errors.message_key。
前端禁止根据 ErrorCode 修改状态。
前端可以根据 ErrorSeverity 决定提示强度。
```

### 14.2 日志消费规则

```text
日志必须记录 ErrorCode、RuleStepType、ErrorSeverity、CommandId、EventId、StateVersion。
DataError、RuleError、Critical、Fatal 必须进入工程日志。
UserError 可以进入操作日志，不一定报警。
```

### 14.3 测试消费规则

```text
测试断言必须优先断言 ErrorCode、RuleResultType、RuleStepType。
测试不应依赖 debug_message。
测试需要覆盖状态是否被修改。
测试需要覆盖事件是否生成。
```

## 15. 组合测试样例

以下样例用于检查错误码、结果类型、严重级别、规则步骤、事件和状态修改是否能正确组合。

| 场景 | ErrorCode | RuleStepType | RuleResultType | ErrorSeverity | EventType | 是否改状态 |
|---|---|---|---|---|---|---|
| 前端传入未知命令类型 | command_unknown_type | ParseCommand | InvalidCommand | UserError | RuleErrorOccurred | 否 |
| 命令缺少 target_id | command_missing_required_field | ValidateCommand | InvalidCommand | UserError | RuleErrorOccurred | 否 |
| 传入不存在的 ObjectId | id_not_found | ValidateCommand | InvalidCommand | UserError | RuleErrorOccurred | 否 |
| 目标太远无法采集 | precondition_target_too_far | CheckPreconditions | Blocked | UserError | ActionBlocked | 否 |
| 行动者昏迷无法行动 | precondition_actor_unavailable | CheckPreconditions | Blocked | Recoverable | ActionBlocked | 否 |
| 缺少工具无法采集硬壳果 | precondition_missing_tool | CheckPreconditions | Blocked | UserError | ActionBlocked | 否 |
| 吃红浆果成功恢复饥饿 | 无错误 | ResolveAction / ResolveCognition | Success | Info | ActionResolved / KnowledgeCreated | 是 |
| 吃毒蘑菇中毒 | 无系统错误 | ResolveAction / ResolveFeedback | CriticalFailure | Warning | ActionResolved / FeedbackGenerated | 是 |
| 使用石头砸水面无效果 | 无错误 | ResolveInteraction | NoEffect | Info | ActionResolved | 否 |
| 族人已有相反知识导致传播冲突 | propagation_contradictory_knowledge | ResolvePropagation | Conflict | Warning | KnowledgeContradicted / PropagationFailed | 通常否 |
| 传播 KnowledgeClaim 缺少效果 | knowledge_claim_incomplete | ResolvePropagation | SystemError | RuleError | RuleErrorOccurred | 否 |
| 规则找不到行为解析器 | rule_missing_resolver | ResolveAction | SystemError | RuleError | RuleErrorOccurred | 否 |
| 随机判定缺少随机流 | rule_random_stream_missing | ResolveInteraction | SystemError | RuleError | RuleErrorOccurred | 否 |
| 状态版本冲突 | state_version_conflict | ApplyStateChanges | SystemError | Critical | RuleErrorOccurred | 否 |
| 存档校验和不匹配 | storage_checksum_mismatch | PersistState | SystemError | Fatal | RuleErrorOccurred | 否 |

测试结论：

```text
当前错误码与结果类型可以区分玩家输入错误、玩法阻断、玩法失败、知识冲突、规则错误、状态错误和存储错误。
CriticalFailure 与 SystemError 已明确分离。
InvalidCommand、Blocked、SystemError 默认不修改世界状态。
Success、PartialSuccess、Failed、Interrupted、CriticalFailure 可以在规则允许时产生状态变化。
前端展示依赖 Event，测试断言依赖 ErrorCode / RuleResultType / RuleStepType。
```

## 16. 研发实现注意事项

```text
Result<T> 不应使用异常作为正常业务分支。
ErrorCode 必须有字符串转换器。
ErrorCode 必须有中文翻译器。
未知错误码字符串必须报错，不能静默转成 unknown。
新增 ErrorCode 必须补充测试。
RuleError、DataError、Critical、Fatal 必须打工程日志。
Fatal 不允许继续写入存档。
所有 SystemError 都必须能追踪到 RuleStepType。
```

## 17. 第一阶段验收标准

```text
所有规则函数都能返回明确 Result 或 RuleExecutionResult。
所有失败都有 ErrorCode。
所有 ErrorCode 都有中文翻译。
所有错误都能关联 RuleStepType 或明确说明无法关联。
非法命令不会修改世界状态。
系统错误不会静默继续执行。
玩法严重失败不会被误判成系统错误。
测试可以通过 ErrorCode、RuleResultType、RuleStepType 断言结果。
```
