# P33 H5 极简可玩验证壳任务卡设计

## 1. 阶段定位

P33 是 H5 极简可玩验证壳阶段。

它不是正式 UI 阶段，也不是地图阶段，更不是重新设计玩法规则阶段。

P33 接在 P32 协议投影扩展之后，目标是把 P32 输出的 `H5GameProjection` 变成玩家能在手机浏览器里看懂、点击、输入、验证的最小操作壳。

P33 在总架构中的位置：

```text
P21 学习闭环
P22 H5 中文对话框最小可玩
P23-P26 族群、共同作战、冲突、文明能力
P27 条件表达式
P28/P28.5 对象反应与配置化物品反应
P29 危险与探索压力
P30 存档前置
P31 真实复盘
P32 H5 安全协议投影
P33 H5 极简可玩验证壳
P34 端到端 MVP
```

P33 的核心价值不是“界面好看”，而是验证：

- P32 projection 是否足够支撑玩家理解。
- 玩家是否能从界面上完成一次认知闭环。
- 前端是否真的不需要解析调试字符串。
- 前端是否真的不会变成第二套规则系统。

## 2. 为什么现在做

### 2.1 不能更早做

在 P32 之前做 H5 界面，前端只能直接消费 P22 对话返回或调试字符串。

这会导致：

- 前端展示 `red_berry + eat -> poison` 这种调试信息。
- 前端为了让按钮可用，不得不猜对象和动作关系。
- 前端容易直接读取知识 key、危险 key、条件 key 并自行解释。
- 后续 UI 会和规则层耦合，难以维护。

P32 已经提供安全投影，所以现在才适合让 H5 只负责渲染。

### 2.2 不能继续推迟到全部系统完成后

如果等所有系统全部写完再做任何界面，风险更大：

- 后端逻辑可能正确，但玩家完全看不懂。
- 知识、认知、危险、族群投影可能字段齐全，但反馈顺序不适合游玩。
- 后续 P34 MVP 才发现 P32 projection 缺字段，会返工协议层。
- 研发 AI 可能继续堆后端系统，却没有玩家体验反馈。

所以 P33 不做正式界面，只做验证壳。

### 2.3 P33 的定位结论

P33 是“低成本体验验收工具”。

它让玩家能完成：

```text
看见当前地点
看见当前物体
选择一个物体
点击尝试吃/使用/观察/组合/教学
看到中文反馈
看到我的知识变化
看到同伴知识变化
看到危险或条件提示
继续下一步尝试
```

这足以验证核心玩法，不需要地图、美术、背包、复杂布局。

## 3. 阶段目标

P33 完成后，系统必须具备以下能力：

1. H5 页面可以在手机浏览器访问。
2. 默认服务仍支持 `1999` 端口。
3. 前端只通过 P32 projection 渲染主要界面。
4. 前端不直接解析规则 key、知识 key、危险 key 的含义。
5. 前端不直接读取 raw state。
6. 前端不自行判断行动是否成功。
7. 玩家可以用按钮或中文输入触发行动。
8. 玩家可以看到中文行动反馈。
9. 玩家可以看到“我的知识”和“同伴知识”。
10. 玩家可以看到对象卡片和可行动作。
11. 玩家可以看到危险/条件/警告提示。
12. 玩家可以重开会话。
13. 界面在手机竖屏下可用。
14. 测试能证明 P33 没有绕开 P32。

## 4. 本阶段不解决的问题

### 4.1 不做正式地图

P33 不做网格地图、坐标、寻路、视野、迷雾、碰撞。

原因：当前核心玩法是认知、尝试、知识传播，不是地图移动。

P33 只做“地点式探索展示”。

允许展示：

```text
当前地点：林地边缘
可见物体：红果、腐烂红果、苦叶、石片
可行动作：吃、使用、观察、教同伴
```

不允许实现：

```text
x/y 坐标
地图格子
角色移动动画
碰撞检测
寻路
战争迷雾
```

### 4.2 不做正式美术

P33 不做角色立绘、地图贴图、复杂动效、复杂音效。

允许轻量样式：

- 卡片。
- 按钮。
- 标签。
- 颜色提示。
- 手机端布局。

### 4.3 不做新规则

P33 不能新增吃、使用、组合、教学的真实规则。

所有行动结果必须来自现有后端流程。

### 4.4 不做前端状态权威

前端可以保存 UI 临时状态，例如当前选中的对象卡片。

前端不能保存玩法权威状态，例如：

- 玩家是否中毒。
- 某个物体是否真实可食用。
- 某个知识是否 active。
- 同伴是否真的学会。
- 某个危险是否已解除。

### 4.5 不做完整内容扩展

P33 可以使用已有红果、腐烂红果、苦叶、石片等内容。

不要求新增大量物品、地点、敌人、文明能力。

内容扩展放到 P35。

### 4.6 不做训练环境

P33 不输出 RL observation/action/reward/done。

训练接口放到 P38。

## 5. 核心设计原则

### 5.1 前端只渲染，不裁决

H5 前端只能根据 P32 projection 渲染。

禁止：

- 根据对象名判断能不能吃。
- 根据标签判断会不会中毒。
- 根据知识状态修改行动结果。
- 根据危险提示自动阻止行动。
- 根据 condition key 自己算条件是否满足。

行动按钮是否可用必须来自 `ActionProjection.enabled`。

### 5.2 中文体验优先，但保留稳定 key

H5 展示给玩家的文字必须优先使用 `SafeTextProjection.zh_cn`。

稳定 key 只用于：

- DOM 标识。
- 自动化测试。
- 调试面板。
- 后续埋点。

玩家界面默认不展示调试 key。

### 5.3 P33 是验证壳，不是正式界面

设计必须压住范围。

本阶段成功标准是“能验证玩法闭环”，不是“像正式商业游戏”。

### 5.4 所有信息必须有来源

界面出现的信息必须来自：

- P32 `H5GameProjection`。
- 后端行动反馈。
- 静态 UI 文案。

不能出现前端自己拼出的规则解释。

### 5.5 输入与按钮走同一条后端路径

玩家点击“吃红果”和输入“吃红果”，最终都必须走后端统一行动处理。

不能按钮一套逻辑，输入框一套逻辑。

### 5.6 先验证地点式探索，不做地图系统

P33 只需要让玩家知道当前在哪里、能看见什么、能尝试什么。

真实地图系统等核心闭环稳定后再做。

### 5.7 失败也要可读

行动失败、条件不满足、投影被截断、安全字段被隐藏，都要能在界面看到中文说明。

不能只在控制台报错。

### 5.8 手机端优先

用户主要通过手机浏览器访问内网穿透端口。

P33 布局必须优先适配手机竖屏。

## 6. 模块划分

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| `h5_playable_shell` | H5 可玩验证壳前端 | 展示 P32 projection 并收集玩家输入 | `H5PlayableResponse` JSON | DOM 界面 | 不得推导规则结果 |
| `ProjectionClient` | 投影接口客户端 | 请求后端回合接口，接收 projection | session、input、client_turn | response JSON | 不得缓存玩法权威状态 |
| `ProjectionStore` | 前端投影快照 | 保存最近一次 projection 方便渲染 | response.projection | 当前 UI 快照 | 不得修改 projection 语义 |
| `SceneRenderer` | 场景渲染器 | 渲染地点、场景描述、警告 | scene projection | 场景 DOM | 不得生成隐藏真相 |
| `ObjectCardRenderer` | 对象卡片渲染器 | 渲染物体列表、标签、对象内动作 | object_cards | 对象卡片 DOM | 不得根据 object_ref 判断用途 |
| `ActionBarRenderer` | 行动栏渲染器 | 渲染全局动作按钮 | action_bar | 按钮 DOM | 不得改 enabled |
| `KnowledgeRenderer` | 知识渲染器 | 渲染玩家与同伴知识 | knowledge lines | 知识 DOM | 不得显示调试串 |
| `HintRenderer` | 提示渲染器 | 渲染条件、危险、issue | condition/danger/issues | 提示 DOM | 不得输出 raw key 给玩家 |
| `CommandController` | 命令控制器 | 统一处理按钮和输入框 | click/input | command request | 不得绕过后端 |
| `H5PlayableTurnService` | 后端可玩回合服务 | 调 P22 回合，再构建 P32 projection | request | playable response | 不得执行第二套规则 |
| `H5ProjectionResponseCodec` | 投影 JSON 编码器 | 把 P32 DTO 写成前端 JSON | H5GameProjection | JSON string | 不得输出 forbidden 字段 |

## 7. 文件与目录设计

### 7.1 前端目录

建议新增目录：

```text
frontend/h5_playable/
```

建议文件：

```text
frontend/h5_playable/index.html
frontend/h5_playable/app.js
frontend/h5_playable/style.css
```

理由：

- 不破坏 P22 `frontend/h5_dialog`。
- P33 可以独立验证 P32 projection。
- 未来 P34 可以选择继续演进 `h5_playable`。

### 7.2 后端目录

建议新增：

```text
backend/include/pathfinder/h5_playable/
backend/src/h5_playable/
backend/tests/unit/h5_playable/
backend/tests/integration/p33/
```

建议 target：

```text
pathfinder_h5_playable
pathfinder_tests_h5_playable
pathfinder_tests_p33_playable_flow
```

### 7.3 服务入口

P33 不要求新增一个全新二进制。

优先复用现有：

```text
pathfinder_h5_dialog_server_exe
```

但需要支持 P33 静态根目录：

```bash
./pathfinder_h5_dialog_server_exe --port 1999 --host 0.0.0.0 --static-root frontend/h5_playable
```

如果实现者发现复用 P22 server 会让接口耦合过重，可以新增 `pathfinder_h5_playable_server_exe`，但必须保持：

- 默认端口 `1999`。
- 静态文件可访问。
- API 返回 P32 projection。

## 8. 后端核心 DTO 设计

### 8.1 `H5PlayableRequest`

中文含义：H5 可玩验证壳回合请求。

字段：

| 字段 | 类型 | 中文含义 | 必填 | 说明 |
|---|---|---|---|---|
| `session_id` | `std::string` | 会话 ID | 是 | 由前端生成或后端返回 |
| `client_turn` | `uint64_t` | 客户端回合号 | 是 | 用于基本幂等和调试 |
| `input_text` | `std::string` | 中文输入文本 | 否 | 例如“吃红果” |
| `action_key` | `std::optional<std::string>` | 投影动作 key | 否 | 点击按钮时传入 |
| `target_object_ref` | `std::optional<std::string>` | 目标对象引用 | 否 | 点击对象动作时传入 |
| `request_projection` | `bool` | 是否返回 projection | 是 | P33 默认 true |

校验规则：

- `session_id` 必须非空。
- `client_turn` 必须大于 0，bootstrap 可以用 0。
- `input_text`、`action_key` 至少一个存在。
- `input_text` 最大长度 128。
- `action_key`、`target_object_ref` 必须通过安全 key 校验。
- 不允许出现 `raw_state`、`hidden_truth`、`hp_delta`、`direct_state` 等 forbidden key。

### 8.2 `H5PlayableResponse`

中文含义：H5 可玩验证壳回合响应。

字段：

| 字段 | 类型 | 中文含义 | 必填 | 说明 |
|---|---|---|---|---|
| `ok` | `bool` | 请求是否成功 | 是 | 网络和业务层结果 |
| `session_id` | `std::string` | 会话 ID | 是 | 返回当前会话 |
| `server_turn` | `uint64_t` | 服务端回合号 | 是 | 以后可用于同步 |
| `reply_text` | `SafeTextProjection` 或安全字符串 | 行动反馈 | 是 | 中文反馈 |
| `projection` | `H5GameProjection` | H5 安全投影 | 是 | P32 输出 |
| `issues` | `std::vector<ProjectionIssue>` | 响应级问题 | 否 | 安全隐藏、截断、输入错误 |
| `debug_keys` | `std::vector<std::string>` | 调试 key | 否 | Player 默认不返回 |

P33 前端必须以 `projection` 为主渲染，不得只依赖 `reply_text`。

### 8.3 `H5PlayableBootstrapRequest`

中文含义：进入页面时的初始化请求。

字段：

| 字段 | 类型 | 中文含义 | 必填 | 说明 |
|---|---|---|---|---|
| `session_id` | `std::string` | 会话 ID | 是 | 前端 localStorage 中的会话 |
| `reset` | `bool` | 是否重开 | 否 | true 时清空会话 |
| `language` | `std::string` | 语言 | 是 | P33 只支持 `zh_cn` |

行为：

- 页面首次打开时请求 bootstrap。
- 如果没有会话，则创建新会话。
- 如果点击“重开”，则 `reset=true`。
- 返回完整 projection。

## 9. 后端核心服务设计

### 9.1 `H5PlayableTurnService`

中文含义：H5 可玩回合服务。

职责：

1. 接收 `H5PlayableRequest`。
2. 统一处理按钮和输入框命令。
3. 调用已有 P22 `DialogTurnService` 或等价后端行动入口。
4. 从行动结果和会话状态构建 `H5ProjectionSourceBundle`。
5. 调用 P32 `H5ProjectionComposer`。
6. 返回 `H5PlayableResponse`。

建议接口：

```cpp
class H5PlayableTurnService {
public:
    pathfinder::foundation::Result<H5PlayableResponse> bootstrap(
        const H5PlayableBootstrapRequest& request);

    pathfinder::foundation::Result<H5PlayableResponse> handleTurn(
        const H5PlayableRequest& request);
};
```

禁止事项：

- 不得自己判断吃/使用/组合结果。
- 不得自己修改知识仓库。
- 不得自己修改族群状态。
- 不得绕开 P21/P22/P28/P29 已有流程。
- 不得返回 raw state。

### 9.2 `H5PlayableProjectionMapper`

中文含义：可玩响应到 P32 source 的映射器。

职责：把现有 P22/P21 的安全响应整理成 P32 `H5ProjectionSourceBundle`。

建议接口：

```cpp
class H5PlayableProjectionMapper {
public:
    pathfinder::foundation::Result<pathfinder::h5_projection::H5ProjectionSourceBundle> buildSourceBundle(
        const DialogTurnResponse& turn_response,
        const H5PlayableSessionView& session_view) const;
};
```

映射规则：

- `reply_text` 不直接替代 projection，只作为 scene summary 或反馈区内容。
- `state_summary_lines` 映射为 `scene_summary`。
- `quick_actions` 映射为 `action_bar` 或对象内 `actions`。
- `actor_knowledge_lines` 必须映射成中文 `KnowledgeLineProjection`。
- `recipient_knowledge_lines` 必须映射成中文 `KnowledgeLineProjection`。
- P22 中仍然是调试格式的知识行时，P33 不得直接展示，应优先从 P21/P32 可用结构映射；如果暂时只能拿到字符串，必须只放入开发调试，不放入 Player 知识面板。

### 9.3 `H5ProjectionResponseCodec`

中文含义：P32 projection JSON 编码器。

职责：把 `H5GameProjection` 安全写成 JSON 给前端。

建议接口：

```cpp
class H5ProjectionResponseCodec {
public:
    pathfinder::foundation::Result<std::string> encodeResponse(
        const H5PlayableResponse& response) const;
};
```

编码要求：

- 必须正确转义中文、引号、反斜杠、换行。
- 不允许输出 `DebugSummary` 到 Player 响应。
- 不允许输出 forbidden key。
- 空数组输出 `[]`。
- 空 optional 输出 `null` 或省略，需统一。
- 字段命名使用 snake_case，与后端 DTO 保持一致。

## 10. 前端核心对象设计

### 10.1 `ProjectionClient`

职责：和后端通信。

建议 JS 接口：

```js
class ProjectionClient {
  async bootstrap({ sessionId, reset }) {}
  async sendTurn({ sessionId, clientTurn, inputText, actionKey, targetObjectRef }) {}
}
```

禁止：

- 不得在 client 内解释规则。
- 不得把 `object_ref_key` 翻译成用途。
- 不得吞掉后端 issue。

### 10.2 `ProjectionStore`

职责：保存最近一次 projection。

建议字段：

```js
{
  sessionId,
  clientTurn,
  lastResponse,
  projection,
  selectedObjectRef,
  pending
}
```

说明：

- `selectedObjectRef` 是 UI 状态，不是玩法状态。
- `pending` 用于禁用重复提交。
- projection 必须每次以后端返回为准。

### 10.3 `RendererRegistry`

职责：统一调度各区域渲染。

建议接口：

```js
function renderProjection(projection, response) {
  renderScene(projection);
  renderObjects(projection.object_cards);
  renderActions(projection.action_bar);
  renderKnowledge(projection.actor_knowledge, projection.recipient_knowledge);
  renderHints(projection.condition_hints, projection.danger_hints, projection.issues);
  renderFeedback(response.reply_text);
}
```

要求：

- 渲染前清空旧 DOM。
- 所有玩家文本使用 `zh_cn`。
- key 只写入 `data-key`，默认不展示。
- 没有数据时展示友好空态。

## 11. 前端页面布局设计

P33 页面必须面向手机竖屏。

建议结构：

```text
顶部栏
- 游戏名：先驱者
- 当前回合
- 重开按钮

场景区
- 当前地点/场景标题
- 场景描述
- 重要提示/警告

对象区
- 当前可见物体卡片
- 每个物体展示名称、描述、知识徽章、对象动作按钮

反馈区
- 最近一次行动反馈
- 最近一次学习/知识变化摘要

知识区
- 我的知识
- 同伴知识

提示区
- 条件提示
- 危险提示
- projection issue

底部输入区
- 快捷动作按钮
- 中文输入框
- 发送按钮
```

### 11.1 场景区

显示内容来自：

- `projection.scene_title.zh_cn`
- `projection.scene_summary[].zh_cn`
- `projection.warning_keys` 对应的安全提示，如果有中文映射则展示中文。

禁止：

- 不得显示 raw key 作为玩家正文。
- 不得根据 scene title 推断地点规则。

### 11.2 对象区

每张对象卡片显示：

- 名称：`display_name.zh_cn`
- 描述：`description.zh_cn`
- 状态徽章：`knowledge_badges[].label.zh_cn`
- 可行动作：`actions[]`

动作按钮：

- 文案使用 `action.label.zh_cn`。
- 是否可点使用 `action.enabled`。
- 点击后发送 `action_key`、`target_object_ref`、`input_text`。
- 如果 `disabled_reason` 存在，显示中文原因。

禁止：

- 看到 `TryEat` 就自己拼“吃xxx”的规则结果。
- 看到 `object_ref_key=red_berry` 就自己说“可食用”。

### 11.3 反馈区

反馈区展示最近一次后端响应。

优先级：

1. 行动输入确认。
2. 行动结果中文反馈。
3. 新知识/认知变化。
4. 危险/条件提示。
5. issue。

反馈区不应该长期堆无限历史。

P33 可以保留最近 20 条简短日志，避免手机端过长。

### 11.4 知识区

知识区分为：

- 我的知识。
- 同伴知识。

每条知识展示：

```text
对象：行为 → 结果（状态/信心）
```

但必须使用中文字段拼接：

- `subject_label.zh_cn`
- `predicate_label.zh_cn`
- `effect_summary.zh_cn`
- `confidence_label.zh_cn` 如果存在。

禁止展示：

```text
red_berry + eat -> poison [active]
```

### 11.5 提示区

提示区展示：

- 条件提示：为什么不能做某事。
- 危险提示：当前风险或应对方式。
- 投影 issue：内容被截断、调试信息被隐藏等。

提示区必须区分：

- 普通提示。
- 警告。
- 危险。
- 系统安全提示。

## 13. 与 P32 的关系

P33 必须把 P32 当作 UI 数据权威。

P33 可以做：

- 把 P32 DTO 序列化为 JSON。
- 把 P32 projection 渲染成 DOM。
- 根据 projection 的 action 生成按钮。
- 根据 projection 的 issue 显示安全提示。

P33 不可以做：

- 重新构造知识状态。
- 重新解释危险状态。
- 重新判断行动可用性。
- 绕过 `ProjectionSanitizer`。
- 输出被 P32 拒绝的调试摘要。

## 14. 与 P22 的关系

P22 是最小中文对话框，已有：

- 中文输入。
- 对话响应。
- 快捷动作。
- 简单知识展示。

P33 不应推翻 P22。

P33 应复用：

- `DialogTurnService` 的输入处理能力。
- P22 的会话基础能力。
- P22 的中文命令解析基础。

P33 应替换或升级：

- 不再以前端直接展示 `actor_knowledge_lines` 字符串为主。
- 不再以前端 debug panel 为主。
- 不再让 H5 页面从 P22 response 自己拼状态区。

P33 的页面主数据来源必须是 P32 projection。

## 15. 与 P28/P28.5 的关系

P33 可以展示对象反应和组合动作，但不实现反应逻辑。

例如：

```text
火 + 干木棍 = 火把
水 + 火 = 火熄灭
```

P33 前端只展示：

- 可组合对象。
- 组合按钮。
- 组合结果反馈。
- 生成物品的新对象卡片。
- 由学习系统形成的知识。

真实判断必须来自 P28/P28.5 后端流程。

## 16. 与 P29 的关系

P33 可以展示危险提示。

例如：

```text
这里有野兽活动的迹象。
你知道火把可能有助于驱赶危险。
```

但 P33 不能自己判断：

- 玩家是否受伤。
- 危险是否触发。
- 危险是否被工具抵消。
- NPC 是否因为危险死亡。

危险相关展示必须来自 P32 `danger_hints`。

## 17. 与 P31 的关系

P33 默认不展示复盘调试面板。

P31 复盘结果只允许在开发调试模式下显示。

Player 页面：

- 不显示 raw replay log。
- 不显示 baseline mismatch 细节。
- 不显示内部 deterministic signature。

DeveloperDebug 页面后续可以扩展，但不属于 P33 必做。

## 18. 核心枚举设计

P33 前端和后端可以新增少量 UI 枚举。

### 18.1 `PlayablePanelKind`

中文含义：可玩验证壳面板类型。

枚举值：

| 枚举值 | 字符串 | 中文含义 | 使用场景 |
|---|---|---|---|
| `Unknown` | `unknown` | 未知 | 非法输入 |
| `Scene` | `scene` | 场景面板 | 当前地点和描述 |
| `Objects` | `objects` | 对象面板 | 可见物体卡片 |
| `Feedback` | `feedback` | 反馈面板 | 行动结果 |
| `Knowledge` | `knowledge` | 知识面板 | 玩家和同伴知识 |
| `Hints` | `hints` | 提示面板 | 条件、危险、issue |
| `Command` | `command` | 命令面板 | 输入框和快捷动作 |
| `TestOnly` | `test_only` | 测试专用 | 单元测试 |

规则：

- 允许 `Unknown` 作为解析失败值。
- 业务运行不允许使用 `Unknown`。
- 生产数据不允许使用 `TestOnly`。

### 18.2 `PlayableInputKind`

中文含义：玩家输入来源。

枚举值：

| 枚举值 | 字符串 | 中文含义 | 使用场景 |
|---|---|---|---|
| `Unknown` | `unknown` | 未知 | 非法输入 |
| `FreeText` | `free_text` | 自由文本 | 玩家输入中文命令 |
| `ProjectionAction` | `projection_action` | 投影动作 | 点击 P32 action 按钮 |
| `Reset` | `reset` | 重开 | 点击重开 |
| `Bootstrap` | `bootstrap` | 初始化 | 页面加载 |
| `TestOnly` | `test_only` | 测试专用 | 单元测试 |

### 18.3 `PlayableFeedbackTone`

中文含义：反馈显示语气。

枚举值：

| 枚举值 | 字符串 | 中文含义 | 使用场景 |
|---|---|---|---|
| `Unknown` | `unknown` | 未知 | 非法输入 |
| `Neutral` | `neutral` | 普通 | 一般反馈 |
| `Positive` | `positive` | 正向 | 学到知识、缓解饥饿 |
| `Warning` | `warning` | 警告 | 条件不满足、内容截断 |
| `Danger` | `danger` | 危险 | 中毒、不适、危险靠近 |
| `System` | `system` | 系统 | 网络错误、会话重置 |
| `TestOnly` | `test_only` | 测试专用 | 单元测试 |

说明：

P33 允许根据 P32 issue severity 或后端 response tone 显示颜色，但不能根据 tone 改变规则结果。

## 19. 最小可玩流程

### 19.1 首次进入

```text
玩家打开网页
前端生成 session_id
调用 /api/playable/bootstrap
后端返回 P32 projection
前端渲染当前地点、物体、动作、知识空态
```

验收：

- 手机浏览器能打开。
- 页面显示中文。
- 不显示调试知识串。

### 19.2 尝试吃红果

```text
玩家点击红果卡片的“尝试吃”
前端发送 action_key 和 target_object_ref
后端走统一回合服务
后端形成反馈、记忆、知识
后端构建 P32 projection
前端刷新知识区
```

验收：

- 玩家看到中文反馈。
- 我的知识出现中文知识。
- 同伴知识不自动增加。

### 19.3 教同伴

```text
玩家点击“教同伴”
后端走知识传播
P32 projection 返回同伴知识
前端刷新同伴知识区
```

验收：

- 同伴知识增加。
- 不出现重复调试行。
- 状态中文可读。

### 19.4 尝试危险或负面对象

```text
玩家吃腐烂红果
后端返回不适反馈
知识发生修正或新增
危险/警告提示更新
前端显示中文危险反馈
```

验收：

- 不显示 HP 数值。
- 不显示 raw damage。
- 不显示 hidden_truth。

### 19.5 重开

```text
玩家点击重开
前端调用 bootstrap reset=true
后端清空或重建会话
前端清空旧知识和反馈
```

验收：

- 页面回到初始状态。
- localStorage session 处理稳定。

## 21. 安全门禁

P33 必须新增门禁测试，避免前端倒退成调试界面。

### 21.1 禁止展示调试知识串

测试扫描前端代码和响应样例，禁止玩家可见区域出现：

```text
" + "
" -> "
"[active]"
"[hypothesis]"
"red_berry + eat"
```

说明：这些可以在测试数据 key 中出现，但不能作为玩家正文输出。

### 21.2 禁止 raw state

测试响应 JSON 不得包含：

```text
raw_state
hidden_truth
hp_delta
actual_hp
true_hp
direct_state
raw_replay_log
raw_save_package
```

### 21.3 禁止前端规则判断

扫描前端代码，禁止出现明显规则硬编码：

```js
if (objectRef === 'red_berry')
if (action === 'eat' && object === ...)
poison
restore_hunger
```

例外：测试文件可以包含这些字符串，但必须限制在测试样例断言中。

### 21.4 禁止绕过 P32

P33 后端 response 必须包含 `projection`。

前端主界面必须从 `projection` 渲染，而不是从 P22 legacy fields 渲染。

## 25. 风险与控制

### 25.1 风险：P33 变成正式 UI 大工程

控制：只做验证壳，不做地图、美术、复杂布局。

### 25.2 风险：前端重新写规则

控制：扫描前端硬编码，按钮 enabled 必须来自 P32。

### 25.3 风险：继续展示调试知识串

控制：知识区只展示 P32 `KnowledgeLineProjection` 中文字段。

### 25.4 风险：P22 legacy response 继续主导界面

控制：P33 主渲染入口只能是 `projection`。

### 25.5 风险：P32 JSON 编码泄露 forbidden key

控制：编码前调用 P32 validate/sanitizer，编码后测试扫描 forbidden key。

### 25.6 风险：手机端不好用

控制：布局优先竖屏，底部输入固定，卡片宽度自适应。

## 26. 研发注意事项

1. 不要为了 P33 重构所有 H5 服务器。
2. 不要删除 P22 页面。
3. 不要把 P33 写成复杂 SPA 框架，原生 HTML/CSS/JS 足够。
4. 不要引入大型前端依赖。
5. 不要把 UI 状态写回后端当玩法状态。
6. 不要让前端通过 object key 判断物品用途。
7. 不要让前端通过 knowledge status 判断行动结果。
8. 不要在玩家界面展示 debug panel。
9. 不要输出 raw replay、raw save、HP、掉落、死亡细节。
10. 实现完成后必须人工自审，而不是只跑测试。

## 27. 对 P34 的交付承诺

P33 完成后，P34 可以基于它做端到端 MVP。

P34 不需要重新解决：

- H5 如何访问后端。
- projection 如何转 JSON。
- 前端如何渲染对象、动作、知识、提示。
- 玩家如何通过手机操作。
- 如何避免前端绕过 P32。

P34 需要继续解决：

- 10 分钟可玩内容链路。
- 更多地点式探索内容。
- 更完整的行动反馈节奏。
- 更完整的存档/复盘/MVP 验收。

## 28. 最终结论

P33 应该做，但必须做小。

P33 的正确目标是：

```text
用最小 H5 验证壳证明 P32 projection 能支撑玩家理解和操作。
```

P33 的错误目标是：

```text
现在就做正式地图、正式 UI、正式游戏客户端。
```

只要 P33 坚持“前端只渲染、安全投影为权威、地点式探索、不做地图”，它就能在不拖慢后端系统建设的情况下，提前验证核心玩法是否真的可读、可玩、可继续扩展。
