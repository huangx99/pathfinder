# P47 资源采集、砍伐、挖掘风险头脑风暴

## 1. 目的

本文档用于提前识别 P47 后续扩展风险，避免把采集做成新的硬编码 demo。

P47 不是“点一下给物品”，而是未来 NPC、制作、知识、生态都要复用的材料获取入口。

## 2. P0 风险

### 2.1 产物绕过 P45

风险：采集成功后直接 push inventory entries 或直接改实体数组。

后果：物品归属混乱，NPC/玩家/地面状态不可信。

门禁：产物必须调用 P45 location port 建立 OnMap，后续进背包必须走 Pickup。

### 2.2 失败仍修改状态

风险：工具缺失或距离太远时，先扣 charge 后返回失败。

门禁：validate 阶段只读；apply 前复验；失败不能生成产物、不能扣 charge。

### 2.3 ResourceNode 与 GroundItem 混淆

风险：树被当作普通物品 pickup，或采集产物被当作 ResourceNode。

门禁：Gather/Chop/Mine/Dig 只处理 ResourceNode；Pickup 只处理 GroundItem。

### 2.4 action 不匹配

风险：对树执行 gather 也能出木头，对浆果丛 chop 也能出果子。

门禁：required_action_key 必须匹配 command kind。

### 2.5 产物 ID 不确定

风险：使用时间戳/随机数导致复盘无法重现。

门禁：output id 基于 command_id、node_id、output_index。

### 2.6 节点耗尽未表达

风险：remaining_charges 到 0 后仍可采。

门禁：0 charge 必须 Depleted，后续命令失败。

## 3. P1 风险

### 3.1 工具要求半实现

P47 只检查工具存在，不处理耐久。必须明确这是边界，不能假装已消耗工具。

### 3.2 产物默认 OnMap 影响手感

MVP 产物掉地上，玩家还要 pickup。可接受，因为 P45 归属正确优先；后续再做 PreferInventory。

### 3.3 节点坐标不可见

如果可见性不校验，玩家能隔空采未知资源。必须接 P44 可见性。

### 3.4 资源输出配置过薄

P47 MVP 每个 output quantity=1。后续 P48/P42 可配置概率、数量、状态。

### 3.5 多 actor 争抢

单线程 MVP 可复验 node charges。服务器并发留到后续。

## 4. P2 扩展场景

### 4.1 NPC 自主采集

P51 需要把 ResourceHarvestService 当成 Agent 行动执行器，不应重写采集逻辑。

### 4.2 采集形成知识

P49 需要把 ResourceHarvested 事件转成 WorldExperienceRecord。

### 4.3 工具耐久

后续工具耐久/锋利度应由工具状态系统处理，P47 只预留 tool key。

### 4.4 资源再生

生态/季节/时间系统可以让 Depleted -> Regenerating -> Available。

### 4.5 危险打断

P51/P52 中，采集过程中遇到狼属于外界打断，不应塞进 P47 MVP。

## 5. 复杂场景压力

```text
1. 玩家采浆果，得到红果掉地上，再 pickup 入背包。
2. 玩家砍树，没有斧头，失败且树不掉 charge。
3. 玩家有斧头砍树，木头掉地上，树 charge -1。
4. 树 charge 归零后变 depleted，不能继续砍。
5. NPC 后续学会 chop 后也提交同一个 Chop command。
6. 采矿需要矿镐，缺工具失败。
7. 地下层矿脉后续支持，但 P47 不改 P46 layer 策略。
8. 背包满时，因为 P47 MVP 只 OnMap，材料不会丢失。
9. 前端只看 CommandResult，不读 resource JSON。
10. 复盘通过 command_id 和 node_id 重建产物 id。
```

## 6. 结论

P47 的成功标准不是“能得到木头”，而是：

```text
材料获取是可复盘、可验证、可归属、可被 NPC 复用的正式世界行为。
```
