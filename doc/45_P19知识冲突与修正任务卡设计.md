# P19 知识冲突与修正任务卡设计

## 1. 阶段定位

P19 是《先驱者》认知、记忆、知识、传播学习主线的第五个核心玩法阶段。

前置阶段已经完成：

```text
P15：认知系统正式化，回答“主体当前相信什么”。
P16：记忆系统基础，回答“一次经历是否会被记住、强化、衰减”。
P17：记忆压缩与检索，回答“大量记忆如何安全摘要、可追溯召回”。
P18：知识系统，回答“稳定记忆如何形成可查询知识声明”。
```

P19 解决：

```text
当新证据和旧知识一致时，如何强化旧知识。
当新证据和旧知识冲突时，如何降低置信度。
当冲突来自条件差异时，如何拆成更具体的条件化知识。
当旧知识被更准确知识替代时，如何废弃但不删除。
当旧知识被强反证证明错误时，如何证伪但保留历史。
```

一句话：

```text
P19 让“红果能吃”在遇到“腐烂红果有毒”后，不是简单翻车，而是修正为“新鲜红果可食用，腐烂红果危险”。
```

P19 在主链路中的位置：

```text
探索 → 反馈 → 认知 → 记忆 → 压缩/召回 → 知识形成 → 知识修正 → 传播 → 文明
```

P19 只做到“个人/主体知识的冲突识别、修正、废弃、证伪”，不做传播、不做族群融合、不做文明能力解锁。

## 2. 本阶段目标

P19 的目标是让 P18 形成的 `KnowledgeClaim` 不再是一次性静态结果，而是可以被新证据安全更新的主观知识。

必须做到：

1. 接收 P18 已有 `KnowledgeClaim` 与新的 P17 `MemorySummary` / P16 `MemoryRecord` 证据。
2. 判断新证据是支持、削弱、冲突，还是需要条件化修正。
3. 对一致证据强化旧知识，提高 confidence 或状态。
4. 对冲突证据降低 confidence，增加 conflict_count，记录反证证据。
5. 对条件差异优先创建 specialized claim，而不是粗暴推翻旧知识。
6. 支持将旧知识标记为 `Deprecated`，并通过 `superseded_by_knowledge_id` 指向新知识。
7. 支持将旧知识标记为 `Disproven`，但必须保留证据链和历史。
8. 不物理删除 `MemoryRecord`、`MemorySummary`、`KnowledgeClaim`。
9. 所有 `Result::ok(dto)` 返回值必须能通过 `validateBasic()`。
10. 继续保持 hidden truth 防线：不能读取 ObjectDefinition hidden truth，不能读取 raw GameState。

核心原则：

```text
知识修正优先条件化，不优先推翻。
旧知识可以失效，但不能消失。
冲突证据必须可追溯。
P19 只处理主观知识变化，不改变世界真实规则。
```

## 3. 玩家体验映射

P19 对玩家体验的意义：

```text
玩家最初多次吃红果，形成“红果可食用”。
后来吃到腐烂红果中毒。
系统不应该直接把“红果可食用”删除。
系统应该形成：
- 新鲜红果可食用。
- 腐烂红果有中毒风险。
- 原来的“红果可食用”变成过宽旧知识，被更精确知识替代。
```

玩家感受到的是：

```text
世界不是贴标签，而是会被经验不断修正。
知识不是神谕，而是可犯错、可反证、可细化的经验体系。
```

P19 不是 UI 阶段，但它必须为后续 UI 提供明确状态：

```text
这个知识被强化了。
这个知识出现冲突。
这个知识被条件化修正。
这个旧知识已经废弃。
这个旧知识已被证伪。
```

## 5. 本阶段不做什么

P19 明确禁止：

```text
不做知识传播。
不做族群共享知识合并。
不做文明能力解锁。
不做 H5/API/HTTP/WebSocket/JSON。
不做 SaveManager。
不做 RL reward/done。
不让 AgentRuntime 或 Policy 直接写 KnowledgeRepository。
不从 ObjectDefinition hidden truth 推导冲突。
不从 GameState raw state 推导冲突。
不删除 MemoryRecord。
不删除 MemorySummary。
不物理删除 KnowledgeClaim。
不做跨主体知识融合。
不做自由文本 AI 自动生成知识。
```

P19 可以为 P20 传播系统保留字段和事件，但不能执行传播。

## 7. 枚举设计

### 7.1 KnowledgeConflictType（知识冲突类型）

中文：知识冲突类型。

| 枚举 | 字符串 | 中文 | 用途 |
|---|---|---|---|
| Unknown | unknown | 未知 | 非法默认值 |
| SameClaimSupport | same_claim_support | 同知识支持 | 新证据支持旧知识 |
| OppositeEffect | opposite_effect | 相反效果 | 同对象同行为出现相反效果 |
| RiskDiscovered | risk_discovered | 发现风险 | 旧知识遗漏风险，例如可食用但有毒副作用 |
| ConditionMismatch | condition_mismatch | 条件不匹配 | 新证据条件与旧知识不同 |
| Overgeneralized | overgeneralized | 过度泛化 | 旧知识范围太宽，需要特化 |
| DuplicateClaim | duplicate_claim | 重复知识 | 新知识与旧知识等价或近似重复 |
| WeakContradiction | weak_contradiction | 弱反证 | 反证存在但证据不足 |
| StrongContradiction | strong_contradiction | 强反证 | 反证足以显著削弱或证伪 |
| SourceInvalid | source_invalid | 来源非法 | 输入证据或旧知识非法 |
| TestOnly | test_only | 测试专用 | 默认禁止 |

规则：

```text
Unknown/TestOnly 默认禁止进入正式 DTO。
SourceInvalid 只用于错误结果或调试 reason，不应作为 ok revision 的正常冲突类型。
ConditionMismatch 优先走 Specialized，不直接 Disproven。
```

### 7.2 KnowledgeRevisionDecision（知识修正决策）

中文：知识修正决策。

| 枚举 | 字符串 | 中文 | 用途 |
|---|---|---|---|
| Unknown | unknown | 未知 | 非法默认值 |
| Skipped | skipped | 跳过 | 证据不足或没有匹配知识 |
| NoChange | no_change | 无变化 | 新证据不足以改变旧知识 |
| ReinforcedClaim | reinforced_claim | 强化知识 | 支持证据增强旧知识 |
| WeakenedClaim | weakened_claim | 削弱知识 | 弱冲突降低置信度 |
| CreatedSpecializedClaim | created_specialized_claim | 创建特化知识 | 条件化拆分新知识 |
| DeprecatedClaim | deprecated_claim | 废弃旧知识 | 旧知识被更准确知识替代 |
| DisprovenClaim | disproven_claim | 证伪旧知识 | 旧知识被强反证推翻 |
| Rejected | rejected | 拒绝 | 输入非法或安全边界失败 |
| TestOnly | test_only | 测试专用 | 默认禁止 |

规则：

```text
Skipped/NoChange 可以作为 Result::ok，但返回 DTO 必须 validateBasic 通过。
Rejected 不应作为 Result::ok 的成功结果，非法输入必须 Result::fail。
DisprovenClaim 只能由 P19 创建，P18 仍不得主动创建 Disproven。
```

### 7.3 KnowledgeResolutionStrategy（知识解决策略）

中文：知识解决策略。

| 枚举 | 字符串 | 中文 | 用途 |
|---|---|---|---|
| Unknown | unknown | 未知 | 非法默认值 |
| Reinforce | reinforce | 强化 | 追加支持证据并提升置信 |
| Weaken | weaken | 削弱 | 追加反对证据并降低置信 |
| Specialize | specialize | 特化 | 创建条件更具体的新知识 |
| Deprecate | deprecate | 废弃 | 旧知识被新知识替代 |
| Disprove | disprove | 证伪 | 旧知识被强反证推翻 |
| MergeDuplicate | merge_duplicate | 合并重复 | 后续预留，P19 第一版可不实现 |
| KeepBoth | keep_both | 并存 | 条件不同，旧新知识并存 |
| TestOnly | test_only | 测试专用 | 默认禁止 |

规则：

```text
P19 第一版必须实现 Reinforce / Weaken / Specialize / Deprecate / Disprove / KeepBoth。
MergeDuplicate 可预留枚举，但第一版不实现合并写入。
```

## 8. P18 兼容性调整

P19 必须调整 P18 的部分约束，否则无法表达“废弃”和“证伪”。

### 8.1 KnowledgeClaim::validateBasic 状态规则调整

P18 当前禁止 `Deprecated` / `Disproven` 进入正式 `KnowledgeClaim`。P19 后需要改为：

```text
KnowledgeClaim::validateBasic() 允许 Deprecated / Disproven，但必须满足更严格条件。
```

建议规则：

```text
status == Deprecated 时：
- superseded_by_knowledge_id 必须非空；或
- reason_keys 必须包含 deprecated_by_specialization / deprecated_by_revision 等安全 reason。
- evidence_refs 不得为空。

status == Disproven 时：
- confidence.conflict_count > 0。
- evidence_refs 中至少存在 supports_claim == false 的反证证据。
- reason_keys 必须包含 disproven_by_contradiction / disproven_by_revision 等安全 reason。
```

仍然禁止：

```text
Unknown
TestOnly
```

注意：

```text
P18 FormationPlan / FormationService 仍不得主动创建 Deprecated / Disproven。
禁止规则从 KnowledgeClaim 下沉到 P18 formation 校验，而不是继续放在通用 KnowledgeClaim 校验里。
```

### 8.2 KnowledgeProjectionBuilder 调整

普通知识投影仍不展示 `Deprecated` / `Disproven`。

P19 后建议：

```text
ProjectionBuilder 先 validate claim。
如果 claim.status 是 Deprecated / Disproven，则默认跳过，不生成 KnowledgeProjectionItem。
除非未来 DebugProjection 显式允许。
```

原因：

```text
KnowledgeProjectionItem 是 Agent/普通视图，不应该让失效知识进入行动输入。
历史/审计视图以后单独做。
```

### 8.3 KnowledgeQuery 调整

`KnowledgeQuery` 建议新增：

```cpp
bool include_deprecated = false;
bool include_disproven = false;
```

默认行为：

```text
query 不返回 Deprecated / Disproven。
只有显式 include_deprecated / include_disproven 才返回历史失效知识。
```

`min_status` 仍不建议使用 Deprecated / Disproven 作为阈值，因为它们不是强度等级，而是失效状态。

## 9. DTO 设计

所有 DTO 必须实现：

```cpp
pathfinder::foundation::Result<void> validateBasic() const;
```

任何 `Result::ok(dto)` 返回的 DTO 必须能通过 `validateBasic()`。

### 9.1 KnowledgeRevisionOptions

中文：知识修正选项。

```cpp
struct KnowledgeRevisionOptions {
    double reinforce_weight = 0.12;
    double weak_conflict_penalty = 0.15;
    double strong_conflict_penalty = 0.35;
    double disprove_confidence_max = 0.20;
    size_t strong_conflict_min_count = 2;
    bool prefer_specialization = true;
    bool allow_deprecated = true;
    bool allow_disproven = true;
    bool allow_create_specialized_claim = true;
    bool keep_old_claim_history = true;
    size_t max_revision_evidence_refs = 50;

    pathfinder::foundation::Result<void> validateBasic() const;
};
```

规则：

```text
所有 double 必须在 0..1。
strong_conflict_min_count 不能为 0。
max_revision_evidence_refs 不能为 0。
如果 allow_disproven=false，则强反证只能 Weaken 或 Deprecate，不能 Disproven。
如果 prefer_specialization=true，ConditionMismatch / Overgeneralized 优先 Specialize。
```

### 9.2 KnowledgeConflictSignal

中文：知识冲突信号。

```cpp
struct KnowledgeConflictSignal {
    KnowledgeConflictType conflict_type = KnowledgeConflictType::Unknown;
    pathfinder::foundation::KnowledgeId existing_knowledge_id;
    KnowledgeSubject subject;
    KnowledgePredicate existing_predicate;
    KnowledgePredicate incoming_predicate;
    std::vector<KnowledgeCondition> existing_conditions;
    std::vector<KnowledgeCondition> incoming_conditions;
    std::vector<KnowledgeEvidence> evidence_refs;
    double severity = 0.0;
    bool condition_explains_conflict = false;
    bool should_specialize = false;
    std::vector<std::string> reason_keys;

    pathfinder::foundation::Result<void> validateBasic() const;
};
```

规则：

```text
conflict_type 不能是 Unknown/TestOnly。
existing_knowledge_id 必须非空。
subject/predicate/evidence 必须 validateBasic。
severity 必须在 0..1。
reason_keys 不允许 hidden truth。
如果 should_specialize=true，则 incoming_conditions 或 existing_conditions 至少一边非空。
```

### 9.3 KnowledgeRevisionInput

中文：知识修正输入。

```cpp
struct KnowledgeRevisionInput {
    KnowledgeFormationInput formation_input;
    std::vector<KnowledgeClaim> existing_claims;
    pathfinder::foundation::Tick revision_tick;
    std::vector<std::string> reason_keys;

    pathfinder::foundation::Result<void> validateBasic() const;
};
```

规则：

```text
formation_input 必须 validateBasic。
existing_claims 可以为空；为空时通常 Skipped 或交给 P18 formation 创建新知识。
existing_claims 中每条 claim 必须 validateBasic。
existing_claims.owner 必须与 formation_input.owner 一致。
existing_claims 不允许 TestOnly、hidden truth、非法状态。
revision_tick 不强制大于 claim.updated_tick，但更新结果的 updated_tick 必须使用 revision_tick 或更合理的 summary tick。
reason_keys 不允许 hidden truth。
```

### 9.4 KnowledgeRevisionAssessment

中文：知识修正评估。

```cpp
struct KnowledgeRevisionAssessment {
    std::vector<pathfinder::foundation::KnowledgeId> matched_claim_ids;
    std::vector<KnowledgeConflictSignal> conflict_signals;
    KnowledgeResolutionStrategy recommended_strategy = KnowledgeResolutionStrategy::Unknown;
    double confidence_delta = 0.0;
    bool has_support = false;
    bool has_conflict = false;
    bool should_create_specialized_claim = false;
    bool should_deprecate_existing = false;
    bool should_disprove_existing = false;
    std::vector<std::string> reason_keys;

    pathfinder::foundation::Result<void> validateBasic() const;
};
```

规则：

```text
recommended_strategy 不能是 Unknown/TestOnly。
confidence_delta 必须在 -1..1。
如果 has_conflict=true，则 conflict_signals 不能为空。
如果 should_disprove_existing=true，则 has_conflict 必须为 true。
如果 should_create_specialized_claim=true，则 recommended_strategy 必须是 Specialize 或 KeepBoth。
```

### 9.5 KnowledgeClaimPatch

中文：知识声明补丁。

```cpp
struct KnowledgeClaimPatch {
    pathfinder::foundation::KnowledgeId knowledge_id;
    KnowledgeStatus status_before = KnowledgeStatus::Unknown;
    KnowledgeStatus status_after = KnowledgeStatus::Unknown;
    KnowledgeConfidence confidence_before;
    KnowledgeConfidence confidence_after;
    std::vector<KnowledgeEvidence> added_evidence_refs;
    pathfinder::foundation::KnowledgeId superseded_by_knowledge_id;
    std::vector<pathfinder::foundation::KnowledgeId> supersedes_knowledge_ids;
    std::vector<std::string> reason_keys;

    pathfinder::foundation::Result<void> validateBasic() const;
};
```

规则：

```text
knowledge_id 必须非空。
status_before/status_after 不能 Unknown/TestOnly。
P19 允许 status_after 是 Deprecated / Disproven。
added_evidence_refs 可以为空，但如果 status_after 是 Disproven，必须包含至少一条 supports_claim=false。
reason_keys 不允许 hidden truth。
```

### 9.6 KnowledgeRevisionPlan

中文：知识修正计划。

```cpp
struct KnowledgeRevisionPlan {
    std::string plan_key;
    KnowledgeRevisionInput input;
    KnowledgeRevisionAssessment assessment;
    std::vector<KnowledgeClaimPatch> patches;
    std::vector<KnowledgeClaim> candidate_new_claims;
    std::vector<std::string> reason_keys;
    std::vector<std::string> warning_keys;

    pathfinder::foundation::Result<void> validateBasic() const;
};
```

规则：

```text
plan_key 非空且不含 hidden truth。
input/assessment/patches/candidate_new_claims 必须 validateBasic。
candidate_new_claims 如果包含新特化知识，必须有 evidence_refs 和 supersedes_knowledge_ids。
Result::ok(plan) 必须 validateBasic 通过。
```

### 9.7 KnowledgeRevisionResult

中文：知识修正结果。

```cpp
struct KnowledgeRevisionResult {
    KnowledgeRevisionDecision decision = KnowledgeRevisionDecision::Unknown;
    std::optional<KnowledgeRevisionPlan> plan;
    std::vector<KnowledgeClaim> updated_claims;
    std::vector<KnowledgeClaim> created_claims;
    std::vector<KnowledgeClaimPatch> applied_patches;
    std::vector<KnowledgeEventDraft> event_drafts;
    std::vector<KnowledgeStateChangeDraft> state_changes;
    std::vector<std::string> reason_keys;
    std::vector<std::string> warning_keys;

    bool ok() const;
    pathfinder::foundation::Result<void> validateBasic() const;
};
```

规则：

```text
decision 不能 Unknown/TestOnly。
Rejected 不作为 ok 成功结果。
Skipped/NoChange 可以没有 plan、claim、event。
ReinforcedClaim/WeakenedClaim/DeprecatedClaim/DisprovenClaim 必须有 updated_claims 和 applied_patches。
CreatedSpecializedClaim 必须有 created_claims，通常也有 updated_claims。
所有 updated_claims/created_claims/event/state_change 必须 validateBasic。
Result::ok(result) 必须 validateBasic 通过。
```

## 10. 服务设计

### 10.1 KnowledgeRevisionPlanner

中文：知识修正计划器。

```cpp
class KnowledgeRevisionPlanner {
public:
    pathfinder::foundation::Result<KnowledgeRevisionPlan> planRevision(
        const KnowledgeRevisionInput& input,
        const KnowledgeRevisionOptions& options = {}) const;
};
```

职责：

```text
验证输入安全。
从 formation_input 生成 incoming predicate/subject/evidence。
匹配 existing_claims 中相关知识。
分类支持/冲突/条件差异。
计算 recommended_strategy。
生成 KnowledgeClaimPatch 和 candidate_new_claims。
证据不足返回 fail 或 Skipped 所需错误，不返回 invalid plan。
```

不做：

```text
不写 Repository。
不发事件。
不改 GameState。
```

### 10.2 KnowledgeRevisionService

中文：知识修正服务。

```cpp
class KnowledgeRevisionService {
public:
    pathfinder::foundation::Result<KnowledgeRevisionResult> revise(
        const KnowledgeRevisionInput& input,
        const KnowledgeRevisionOptions& options = {}) const;
};
```

职责：

```text
调用 Planner。
把 plan 转成 updated_claims / created_claims。
生成 KnowledgeEventDraft / KnowledgeStateChangeDraft。
对证据不足或没有匹配知识返回有效 Skipped/NoChange。
对 validation/security 错误返回 Result::fail。
不直接写 Repository。
```

### 10.3 KnowledgeRevisionApplier（可选）

P19 第一版建议不做复杂 applier。如果实现，必须是纯内存辅助，不直接持久化。

```cpp
class KnowledgeRevisionApplier {
public:
    pathfinder::foundation::Result<std::vector<KnowledgeClaim>> applyToSnapshot(
        const std::vector<KnowledgeClaim>& current_claims,
        const KnowledgeRevisionResult& result) const;
};
```

规则：

```text
只返回新 snapshot。
不修改传入参数。
不写 Repository。
所有输出 claim 必须 validateBasic。
```

## 11. 冲突判断算法

### 11.1 匹配候选旧知识

候选匹配条件：

```text
owner 相同。
subject 相同或可解释为同一对象/同一对象类型。
predicate.relation_type 相同或同一风险/效果族。
action_key/action_id 相同或安全等价。
```

P19 第一版只做保守匹配：

```text
subject 完全相同。
relation_type 完全相同或 HasEffect/HasRisk 的安全组合。
action_key 完全相同。
```

不做模糊语义匹配。

### 11.2 支持证据

判定为支持：

```text
新证据与旧 claim 的 subject、action、relation、effect_key 兼容。
条件相同或新证据条件包含旧条件。
新 evidence supports_claim=true。
```

处理：

```text
追加支持证据。
confidence 上升。
support_count 上升。
last_verified_tick 更新。
状态可能 Hypothesis → Active → Teachable。
```

### 11.3 弱冲突

判定为弱冲突：

```text
新证据与旧知识效果不同，但样本不足。
或 evidence reliability 较低。
或新证据条件不完整。
```

处理：

```text
追加 supports_claim=false 的反证证据。
confidence 下降。
conflict_count 上升。
status 通常不直接 Disproven。
```

### 11.4 强冲突

判定为强冲突：

```text
同 owner、同 subject、同行为、同条件下，多条可靠证据产生相反效果。
冲突证据数量 >= strong_conflict_min_count。
扣减后 confidence <= disprove_confidence_max。
```

处理：

```text
如果 allow_disproven=true，则旧 claim.status = Disproven。
否则旧 claim.status 降级为 Hypothesis 或 Candidate。
必须保留反证 evidence_refs。
```

### 11.5 条件化修正

判定为条件化修正：

```text
新证据和旧知识冲突，但条件不同。
例如 old 无条件，incoming 有 state:decayed。
或 old state:fresh，incoming state:decayed。
```

处理优先级：

```text
1. 创建新 specialized claim。
2. 新 claim 的 conditions 使用 incoming_conditions。
3. 新 claim supersedes 或 related old claim。
4. 如果旧 claim 过宽且 options.allow_deprecated=true，则旧 claim Deprecated。
5. 如果旧 claim 仍有适用条件，则 KeepBoth，不废弃。
```

### 11.6 废弃

废弃条件：

```text
旧知识不是完全错误，而是被更准确、更条件化的新知识替代。
```

必须满足：

```text
旧 claim.status_after = Deprecated。
旧 claim.superseded_by_knowledge_id 指向新 claim。
新 claim.supersedes_knowledge_ids 包含旧 claim。
不删除旧 claim。
```

### 11.7 证伪

证伪条件：

```text
旧知识在相同关键条件下被强反证推翻。
不是因为条件不同。
不是因为样本不足。
```

必须满足：

```text
旧 claim.status_after = Disproven。
旧 claim.evidence_refs 中至少有一条 supports_claim=false。
旧 claim.confidence.conflict_count > 0。
旧 claim.reason_keys 包含 disproven_by_revision。
不删除旧 claim。
```

## 12. 置信度更新规则

### 12.1 基础公式

P19 第一版使用简单、确定、可测试公式：

```text
support_delta = summary.aggregate_strength * options.reinforce_weight
weak_conflict_delta = -summary.aggregate_strength * options.weak_conflict_penalty
strong_conflict_delta = -summary.aggregate_strength * options.strong_conflict_penalty
new_confidence = clamp(old_confidence + delta, 0.0, 1.0)
```

### 12.2 band 更新

更新后调用：

```cpp
confidenceToBand(new_confidence)
```

### 12.3 状态更新

建议状态规则：

| 置信度 | 状态 |
|---|---|
| < 0.10 | Candidate |
| 0.10 - 0.34 | Hypothesis |
| 0.35 - 0.74 | Active |
| >= 0.75 | Teachable |

例外：

```text
Deprecated / Disproven 由修正策略决定，不只由 confidence 决定。
Shared / Operational 仍归后续阶段，不由 P19 主动创建。
```

## 13. 事件与状态变更草稿

P19 只生成事件草稿，不直接写 EventStore。

建议 event_key：

| 事件 key | 触发 |
|---|---|
| knowledge.claim_reinforced | 旧知识被支持证据强化 |
| knowledge.claim_weakened | 旧知识被弱反证削弱 |
| knowledge.claim_contradicted | 旧知识出现冲突证据 |
| knowledge.claim_specialized | 创建条件化特化知识 |
| knowledge.claim_deprecated | 旧知识被更准确知识废弃 |
| knowledge.claim_disproven | 旧知识被强反证证伪 |

规则：

```text
event_drafts 必须 validateBasic。
Skipped/NoChange 默认不生成 event_drafts，避免空 knowledge_id 或非法状态。
state_changes 必须带 valid knowledge_id。
reason_keys 不允许 hidden truth。
```

## 14. Repository / Projection 规则

### 14.1 Repository

P19 后 Repository 必须允许保存 `Deprecated` / `Disproven` 的 `KnowledgeClaim`，前提是 claim 自身 validateBasic 通过。

默认 query：

```text
不返回 Deprecated。
不返回 Disproven。
```

显式查询：

```text
include_deprecated=true 时可返回 Deprecated。
include_disproven=true 时可返回 Disproven。
```

### 14.2 Projection

普通 Projection：

```text
永远不输出 Deprecated / Disproven 的 ProjectionItem。
```

原因：

```text
Projection 是 Agent/普通系统行动输入，不是历史审计视图。
失效知识不能进入行动候选。
```

后续如果需要历史审计，单独设计 DebugKnowledgeProjection。

## 15. 安全边界

P19 必须保持：

```text
不读取 ObjectDefinition hidden truth。
不读取 GameState raw state。
不读取 AgentTickRecord raw data。
不读取 SaveGame。
不根据配置真值直接判定谁对谁错。
```

P19 只能依据：

```text
KnowledgeClaim 中已有主观知识。
MemorySummary 中的安全摘要字段。
MemoryRecord / MemoryEvidenceRef 的安全引用。
KnowledgeFormationInput 的安全 action_key/effect_key/conditions。
```

所有新增 string key 字段必须调用 `containsKnowledgeForbiddenKey`。

## 17. 验收标准

P19 完成必须满足：

1. 能对支持证据强化旧知识。
2. 能对弱冲突降低旧知识置信度。

## 19. AI 实现注意事项

1. 不要为了让 `Deprecated/Disproven` 通过，直接删除所有状态校验。

4. 不要在 RevisionService 内直接写 Repository。
5. 不要用 `remove()` 表示废弃或证伪。
6. 不要吞掉 validation/security 错误变成 Skipped/NoChange。
7. 不要读取 hidden truth 判断谁对谁错。
8. 不要让 Result::ok 返回 validateBasic 失败的 DTO。
9. 不要把条件冲突直接 Disproven，优先 Specialize。
10. 不要让 Shared/Operational 在 P19 被主动创建。
