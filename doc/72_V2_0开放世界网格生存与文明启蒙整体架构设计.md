# V2.0 开放世界网格生存与文明启蒙整体架构设计

## 1. 阶段定位

V2.0 是《先驱者》从“最小文本可玩原型”升级为“开放世界网格生存与文明启蒙游戏”的总体架构阶段。

它不是重新做一套游戏，不是推翻 P0-P42，也不是单纯重构 H5 前端。

V2.0 的目标是把现有已经形成的核心能力：

```text
基础类型
配置系统
事件系统
规则管线
对象反应
条件表达式
认知
记忆
知识
知识传播
Agent
目标推理
动态执行
外界打断
通用世界效果
JSON 内容包
H5 投影
```

接入一个真正的世界网格中，让玩家可以像开放世界生存游戏一样移动、探索、采集、砍伐、挖掘、制作、遭遇野兽、发现 NPC，并通过亲身尝试形成知识，再把知识教给无知 NPC，最终带领种群走向文明。

一句话定位：

```text
V2.0 是“世界网格成为游戏主场景”的版本，不是“文本按钮继续堆功能”的版本。
```

## 2. 与现有架构关系

V2.0 必须融入现有架构。

现有系统不是废弃，而是分层接入：

| 现有系统 | V2.0 中的位置 | 是否保留 | 接入方式 |
|---|---|---|---|
| P0-P3 基础类型、状态、事件 | 底层通用能力 | 保留 | 世界网格状态变化继续产出事件 |
| P4 复盘与协议前置 | 调试与验收能力 | 保留 | 世界操作必须可复盘 |
| P5-P14 Agent 调试链 | Agent 可观察与调试 | 保留 | NPC / 野兽都是 Agent 实例 |
| P15-P21 认知记忆知识传播 | 游戏核心 | 强化 | 玩家探索产生知识，NPC 通过教学获得知识 |
| P22-P33 H5 与投影 | 前端协议基础 | 保留但升级 | 旧文本 UI 被 V2 地图投影替代 |
| P27 条件表达式 | 条件权威 | 必须保留 | 采集、制作、移动、危险触发都用条件表达式 |
| P28/P28.5 对象反应流水线 | 反应与合成基础 | 必须保留 | 砍树、挖石、制作火把等接入反应系统 |
| P29 危险与探索压力 | 危险基础 | 保留并地图化 | 野兽在网格上移动与遭遇 |
| P30-P31 存档与复盘 | 世界持久化基础 | 扩展 | 世界网格、背包、NPC 位置必须可存档 |
| P32/P33 协议投影/H5 壳 | 前端只读原则 | 强化 | 前端只消费 WorldProjection |
| P34-P36 因果与目标知识 | 知识精度 | 保留 | 采集、食用、制作、避险继续形成条件化知识 |
| P37-P38 世界交互后果 | 世界变化基础 | 扩展 | 由单场景交互升级为网格世界交互 |
| P39 知识链式推理 | NPC 智能基础 | 强化 | NPC 用知识链决定采集、吃、制作、避险 |
| P40 动态目标执行 | 行动执行基础 | 强化 | NPC 执行链可以移动、采集、制作、逃跑 |
| P41 通用效果执行 | 世界效果权威 | 强化 | 所有世界变化通过通用效果执行 |
| P42 JSON 内容包 | 内容来源 | 必须修复后接入 | 地形、资源、物品、动作、NPC、野兽由内容包定义 |

V2.0 的核心要求：

```text
不要绕开现有知识 / Agent / 规则 / 事件系统。
不要在大地图里重新写一套临时逻辑。
不要让前端直接理解规则。
```

## 3. 为什么现在做 V2.0 总体架构

当前项目已经做到了“最小文本可玩”：

```text
玩家能尝试物体。
玩家能形成知识。
玩家能教同伴。
同伴可以依据知识行动。
野兽危险、火把、材料、制作、目标推理已有雏形。
```

但是继续在旧 H5 文本界面上修补会出现 5 个问题：

1. 世界没有空间，采集、移动、野兽、NPC 都像浮在按钮上。
2. 玩家没有背包，物品归属容易再次混乱。
3. 资源没有实例，红果、树、石头无法自然消耗和再生。
4. NPC 不能真正“去某地做某事”，只能像脚本一样响应。
5. 前端无法承载开放世界体验。

所以 V2.0 必须先设计整体架构，再拆开发阶段。

## 4. V2.0 核心体验目标

V2.0 完成后，玩家体验应从：

```text
在一个文本框里点按钮，验证知识传播。
```

升级为：

```text
在世界网格中出生。
探索未知地形。
发现红果、树、石头、野兽、NPC。
通过采集、食用、砍伐、挖掘、制作形成知识。
用知识解决生存问题。
找到无知 NPC，把知识教给他们。
NPC 从不会行动，逐渐学会采集、吃、避险、制作、协作。
野兽在地图上真实移动、靠近、攻击、绕行或逃离。
种群从无知状态进入文明萌芽。
```

核心爽点不是装备数值，而是：

```text
我理解了世界。
我改变了世界。
我教会了族群。
族群因为我的知识开始自我运转。
```

## 5. V2.0 不解决的问题

V2.0 第一轮不做以下内容：

```text
不做 3D。
不做真实 Minecraft 方块破坏动画。
不做复杂寻路算法。
不做真实 3D 高度、3D 碰撞、3D 摄像机。
不做完整地下世界、空中生态、水下生态、室内建筑层的全部玩法。
不做完整建筑系统。
不做完整农业生态。
不做大型战争。
不做复杂经济系统。
不做多人联网。
不做 Mod 市场。
不做真实 AI 大模型接入。
```

这些不是不重要，而是不能抢在世界网格、背包、交互、知识闭环之前。

V2.0 第一轮只做：

```text
二维世界网格。
虚拟 Z 层坐标预留。
玩家移动与探索。
资源采集。
背包归属。
物品实例。
基础制作。
知识发现。
教学。
NPC 基于知识行动。
野兽地图行动。
H5 地图可玩。
```

## 6. 核心设计原则

### 6.0 二维世界与虚拟 Z 层

《先驱者》V2.0 是二维游戏，不是 3D 游戏。

世界主平面是：

```text
X / Y 二维网格
```

但系统必须从一开始支持虚拟 Z 层：

```text
layer_key
```

虚拟 Z 层不是 3D 坐标，也不代表真实高度模拟。它只表示逻辑空间层级：

```text
surface       地表
underground   地下 / 矿洞
water         水中
air           空中
interior      室内
canopy        树冠 / 高处
```

第一版只实现：

```text
surface
```

但所有坐标、实体位置、移动 command、投影 DTO、存档字段都必须带 `layer_key`。

这样后续可以支持：

```text
地下矿洞有矿石和洞穴怪物。
空中有鸟、飞虫、飞行敌人。
水中有鱼和水草。
屋内有 NPC 和设施。
树冠层有果实、蜂巢、鸟巢。
```

禁止把 V2.0 设计成 3D：

```text
禁止 3D 坐标系。
禁止 3D 碰撞。
禁止 3D 摄像机。
禁止把 layer_key 当成连续高度。
```

正确理解是：

```text
二维地图 + 多个逻辑空间层。
每个 layer 都是一张二维网格。
实体同一时刻只处于一个 layer。
跨 layer 需要入口、动作、能力或规则。
```

### 6.1 世界网格是空间权威

所有玩家、NPC、野兽、资源、掉落物都必须存在于世界网格或明确的持有容器中。

禁止：

```text
火把凭空属于所有人。
狼只是一个全局危险值。
红果只是一个按钮，不存在于格子或背包中。
NPC 没有位置却能采集。
```

### 6.2 背包与库存是物品归属权威

V2.0 必须区分：

```text
玩家背包
NPC 背包
营地公共库存
地面掉落物
资源节点库存
制作设施库存
```

任何物品移动都必须经过归属变更。

### 6.3 内容包是定义来源，运行时是状态来源

JSON 内容包定义：

```text
地形类型
资源节点类型
物品定义
动作定义
效果定义
反应规则
NPC 模板
野兽模板
知识模板
世界生成配置
```

运行时状态保存：

```text
具体某个树实例还剩多少木头
某个红果在谁背包里
某只狼在什么坐标
某个 NPC 学会了哪些知识
某个格子是否被探索
```

### 6.4 前端只投影，不推规则

H5 前端只做：

```text
显示地图投影
显示对象和行动
发送玩家意图
播放结果反馈
```

禁止前端判断：

```text
这个树能不能砍。
这个 NPC 会不会吃红果。
这个狼是否害怕火。
这个材料是否足够。
这个知识能不能教。
```

### 6.5 玩家是先驱者，NPC 初始无知

NPC 初始不是工具人，也不是默认会生存的 AI。

NPC 初始只具备：

```text
基础移动本能
基础恐惧本能
基础跟随 / 靠近群体本能
基础受伤逃离本能
基础生命需求
```

NPC 不具备：

```text
红果能吃
树能砍
石头能挖
火能驱赶野兽
木头能制作火把
房子能御寒
```

这些必须通过玩家教学、亲身经验或族群传播获得。

### 6.6 知识必须能反向驱动行动

知识不是收藏品。

知识必须能参与：

```text
目标推理
行动选择
路径寻找
材料缺口补全
危险规避
教学选择
族群生产
```

例如：

```text
NPC 饿了。
NPC 知道红果能缓解饥饿。
NPC 知道红果来自红果灌木。
NPC 记得东边格子有红果灌木。
NPC 移动过去采集红果。
NPC 吃红果。
```

### 6.7 所有变化必须可复盘

V2.0 会显著增加系统复杂度，必须保留复盘能力。

每个世界回合至少要记录：

```text
玩家意图
NPC 决策摘要
执行动作
状态变化
资源变化
知识变化
事件变化
失败原因
```

## 8. V2.0 模块划分

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| WorldGrid | 世界网格 | 管理格子、地形、探索、可见性 | 世界生成结果、状态变更 | 格子状态、空间查询 | 不处理背包规则 |
| WorldPosition | 世界位置 | 表达玩家/NPC/物品所在坐标 | actor/object id | x/y/layer_key、朝向 | 不保存知识 |
| WorldLayer | 虚拟 Z 层 | 区分地表、地下、水中、空中、室内等逻辑层 | layer_key、入口规则 | layer 查询、跨层校验 | 不实现 3D 高度 |
| WorldGenerator | 世界生成器 | 根据 seed 生成地形、资源、出生区 | seed、生成配置 | 初始 WorldGrid | 不写死演示地图 |
| WorldCommandPipeline | 统一世界命令管线 | 接收玩家、NPC、野兽、系统、生成器、区域事件发出的命令并统一校验、执行、记录、投影 | WorldCommandDto | WorldCommandResponseDto | 不允许外部绕过管线直接改世界 |
| Inventory | 背包库存 | 管理持有物和归属 | item change | item stack state | 不推导制作规则 |
| ObjectInstance | 物体实例 | 区分定义和具体实例 | object definition | instance state | 不把 definition 当 runtime |
| ResourceNode | 资源节点 | 管理可采集资源 | 采集动作 | 资源减少/产物 | 不直接写知识 |
| WorldInteraction | 世界交互 | 处理移动、采集、砍伐、挖掘、拾取、丢弃 | WorldCommandDto | WorldCommandResultDto | 不绕过 Command 管线 |
| CraftRuntime | 制作运行时 | 用背包/附近设施/内容规则制作物品 | craft command | 消耗/产物 | 不凭空产物 |
| DiscoveryBridge | 发现桥接 | 把世界交互结果转换为经验/知识输入 | action result | learning request | 不直接伪造最终知识 |
| TeachingBridge | 教学桥接 | 把知识传播接入 NPC | teach command | recipient knowledge | 不绕过传播系统 |
| AgentWorldBridge | Agent 世界桥 | 把 Agent 目标执行映射为 WorldCommand | goal step | world command | 不写死 NPC 行为 |
| BeastRuntime | 野兽运行时 | 野兽感知、移动、攻击、恐惧 | world tick | beast action | 不做全局危险按钮 |
| WorldProjection | 世界投影 | 输出前端安全视图 | world state | projection DTO | 不暴露隐藏真相 |
| WorldSave | 世界存档 | 保存/加载网格、背包、实例、知识 | runtime state | save data | 不保存前端临时 UI |

## 9. 核心运行循环

V2.0 的一个回合不是单纯“玩家点一下按钮”。

标准回合：

```text
1. 前端、Agent、野兽、系统 tick 或世界生成器提交 WorldCommand。
2. WorldCommandPipeline 标准化 command source、actor、target、layer_key、request_id。
3. 管线校验距离、权限、知识门禁、材料、区域、虚拟 Z 层和冷却。
4. 管线把命令分派给世界交互、背包、制作、教学、区域事件或生成器适配器。
5. 规则管线 / 效果执行结算状态变化，并产出状态变更草稿。
6. DiscoveryBridge 生成经验与知识候选。
7. 事件系统记录变化，复盘系统记录 command trace。
8. Agent 系统推进 NPC / 野兽行动，必要时继续提交内部 WorldCommand。
9. 外界打断系统处理遭遇、攻击、危险，并把打断也记录为 command 派生结果。
10. 更新世界网格、背包、实例、知识、记忆。
11. WorldProjectionPatch 输出前端可增量应用的安全变化。
12. WorldCommandResponse 返回 result、patch、event_feed、frontend_hints、available_commands。
13. 前端只渲染响应，不根据本地逻辑推导按钮或规则结果。
```

```cpp
enum class WorldCellVisibility {
```

### 10.1A WorldLayerKind

```cpp
enum class WorldLayerKind {
```

### 10.7 WorldCommandResultKind

```cpp
enum class WorldCommandResultKind {
    Accepted,          // 命令被接收并执行
    Rejected,          // 命令格式、权限、目标非法
    Failed,            // 命令合法但执行失败
    Blocked,           // 被地形、距离、材料、知识、区域限制阻挡
    Interrupted,       // 执行中被外界打断
    ProducedKnowledge, // 产生知识或知识候选
    ProducedMemory,    // 产生记忆但未形成知识
    TriggeredEncounter,// 触发遭遇或危险
    Pending,           // 长动作未完成，等待后续 tick
    NoChange           // 合法执行但没有状态变化
};
```

## 11. 核心 DTO 草案

### 11.1 WorldCoordinateDto

```cpp
struct WorldCoordinateDto {
    int x = 0;
```

### 13.1 P42 复审后必须纳入 V2 的事项

本节只记录仍会影响 V2 架构的事项。

#### 13.1.1 V2 严格内容加载

V2 世界启动和世界生成必须明确内容加载模式：

```text
StrictContentRequired：正式运行、自动化测试、存档恢复默认使用。
DevelopmentFallback：只允许本地开发工具显式开启。
```

约束：

```text
StrictContentRequired 下，manifest、结构校验、引用校验、schema 版本不通过时，V2 bootstrap 必须返回错误。
不能生成半可用世界。
不能用内置演示内容静默替代失败内容包。
不能让前端误以为世界已经正常启动。
```

#### 13.1.2 content actions 必须接入 CommandOption

P42 已有 `ActionDefinitionDto` 和 `ContentRegistry::findAction()`，但 `content/core` 第一版尚未形成完整 actions 目录。

V2 必须补齐：

```text
content/core/actions/*.json
ActionDefinitionDto -> WorldCommandKind 映射
ActionDefinitionDto -> WorldCommandOptionDto 显示文案
ActionDefinitionDto -> targeting / safety / time_cost / allowed_actor_scales
available_commands 生成时读取 action 定义
```

约束：

```text
前端按钮只能来自 available_commands。
WorldCommandOption 不能写死中文按钮列表。
新增采集、砍伐、挖掘、制作、教学等动作时，优先扩展 content actions 和 Command Adapter。
如果某个动作尚未有 content action 定义，必须在对应阶段文档中写明临时原因和迁移门禁。
```

#### 13.1.3 内容包版本元数据必须进入存档和复盘

P42 当前已经能加载 core 内容包，但 V2 需要更强的内容版本权威。

P54 前必须补齐：

```text
loaded_package_keys
content_version
schema_version
load_order
content_hash 或 manifest_hash
save_required_content_versions
replay_required_content_versions
```

约束：

```text
存档恢复必须知道当时使用了哪个内容包版本。
复盘重放必须知道当时使用了哪个内容包版本。
内容包版本不兼容时必须返回可解释错误或迁移计划。
不能只靠当前磁盘上的 content/core 猜测旧存档规则。
```

## 15. V2.0 最小可玩闭环

```text
1. 玩家出生在网格世界。
2. 玩家探索并发现红果灌木、树、石块、野兽踪迹、无知 NPC。
3. 玩家采集红果并吃红果，形成知识。
4. 玩家砍树获得木头，形成知识。
5. 玩家制作火把，形成制作知识。
6. 玩家遇到狼，用火把驱赶，形成危险知识。
7. 玩家找到 NPC，教会红果来源、红果食用、火把驱狼。
8. NPC 饿了后能自己去采集红果并吃。
9. 狼靠近时，NPC 如果有火把和知识，能自主处理；否则逃跑或求助。
10. 前端能完整显示地图、背包、知识、NPC 行动和事件。
```

## 18. 最大风险与防线

### 18.1 风险：V2.0 变成重写项目

防线：所有阶段必须写明接入现有模块，不允许新建平行知识系统、平行 Agent 系统、平行规则系统。

### 18.2 风险：大地图前端先行导致假地图

防线：前端阶段排到 P53，必须等 Command 管线、网格、背包、交互、知识、NPC 行动有投影后再正式重构。

### 18.3 风险：NPC 又变成写死脚本

防线：NPC 行动必须通过知识门禁和目标执行链。测试必须验证 NPC 不知道时不会做，教会后才会做。

### 18.4 风险：物品归属再次混乱

防线：P45 必须优先完成背包和归属系统。所有制作、采集、使用都必须明确输入容器和输出容器。

### 18.5 风险：内容包接入半成品

防线：P42 必须先修复运行时接入、effect 接入、初始状态接入和错误阻断。V2.0 不应建立在半接入 JSON 上。P43-P48 必须补齐 content actions 与 WorldCommandOption 的映射，P54 必须补齐内容包版本元数据。

### 18.6 风险：知识系统与地图脱节

防线：P49 必须把每种世界交互产生的经验明确转成知识输入，不能只更新背包不学习。

### 18.7 风险：Command 按钮和内容动作再次分裂

防线：`available_commands` 必须由后端根据 `WorldCommandOptionDto` 生成，并逐步读取 content action 定义。前端不能维护自己的按钮规则，Command Adapter 不能长期硬编码动作文案。

### 18.8 风险：存档复盘无法还原内容版本

防线：P54 必须保存和校验内容包 key、content_version、schema_version、load_order 和 manifest hash。复盘和存档恢复不能只依赖当前磁盘 content/core。

## 19. P42 到 V2.0 的前置修复要求

进入 V2.0 开发前，P42 至少要完成：

```text
V2 runtime 能优先使用 content/core 内容包。
EffectExecutionSpec / EffectSemantics 能从 JSON 生成或合并。
Scenario 初始数量和 numeric 能进入运行时初始化。
配置 Error 时不能构建可用 registry。
default_agent_knowledge 能校验和适配。
重复 key 不能静默覆盖。
StrictContentRequired 下内容加载失败不能静默生成世界。
content actions 必须在 P43-P48 接入 WorldCommandOption / available_commands。
内容包版本元数据必须在 P54 接入存档和复盘。
```

如果 P42 没修完，P43 可以先做纯 C++ Command 管线最小结构，但必须预留内容包入口，不能把命令种类、地形、资源、动作写死成长期方案。

## 21. V2.0 验收标准

V2.0 不是“所有未来功能完成”。

V2.0 验收标准是：

```text
玩家能在 H5 网格世界中探索。
世界资源存在于格子上。
玩家有背包。
采集、砍伐、挖掘有真实产物和消耗。
制作消耗真实材料并产生真实归属物品。
玩家行为能形成知识。
玩家能把知识教给 NPC。
NPC 初始无知，教会后能用知识自主行动。
野兽在地图上移动并能造成外界打断。
前端只显示后端投影。
核心流程可保存、可复盘、可测试。
```

## 22. 结论

V2.0 的正确路线不是先画大地图，也不是继续修旧 H5 按钮。

正确路线是：

```text
先把世界网格、背包、实例、生成、交互、知识、Agent、野兽全部接入同一条后端权威链。
再让前端消费投影，做真正可玩的开放世界界面。
```

V2.0 的核心不是地图大小，而是：

```text
世界真实存在。
物品真实归属。
知识真实产生。
NPC 真实从无知变聪明。
文明真实从玩家教学开始扩散。
```
