# P40 动态目标执行与外界打断系统任务卡设计

## 1. 阶段定位

P40 是“动态目标执行与外界打断系统”阶段。

它接在 P39 知识链式推理与目标规划之后，不是继续扩充内容包，也不是继续给 Agent 写更多固定规则。P39 解决的是：

```text
Agent 为什么要做这件事？
有哪些可行方案？
哪条方案收益更高、风险更低、链路更短？
```

但 P39 只解决“想明白”，不解决“做下去”。

P40 要解决的是：

```text
Agent 已经有目标和计划之后，如何一步一步执行？
执行中缺材料、工具坏了、地点不对怎么办？
执行中遇到狼、下雨、敌袭、小孩受伤这种外界事件怎么办？
紧急事件处理完后，如何恢复原目标？
执行失败后，如何重新推理而不是卡死？
```

P40 的一句话定义：

```text
把 P39 选出的计划变成可持续执行、可中断、可恢复、可重规划的目标执行状态机。
```

### 1.1 和 P39 的边界

P39：目标规划。

```text
输入：知识、需求、世界状态。
输出：AgentPlan。
回答：为什么选这个计划。
```

P40：目标执行。

```text
输入：AgentPlan、世界事件、执行状态。
输出：下一步 DriverCommand / WorldInteractionInput / InterruptDecision。
回答：现在具体做哪一步；被打断后怎么处理；处理完怎么恢复。
```

举例：

```text
P39：房子能御寒，当前房子不够，所以计划造房。
P40：造房需要木头；木头不够，先伐木；伐木遇到狼，暂停伐木，插入处理狼；狼退走后恢复伐木；木头够后恢复造房。
```

### 1.2 为什么现在做

如果只有 P39，没有 P40，系统会出现：

```text
能生成计划，但不能稳定执行计划。
能知道要造房，但缺木头时不知道如何持续获取。
能知道要伐木，但斧头钝了就失败。
能知道狼危险，但伐木时遇狼无法中断当前目标。
能处理紧急事件，但处理完不知道回到原目标。
```

这会导致玩家看到“同伴好像会思考”，但实际行动断断续续。

P40 是后续地图、采集、伐木、建造、战斗、照顾小孩、族群分工、野生 NPC 生态的执行地基。

## 2. 头脑风暴：必须覆盖的动态场景

P40 设计前先列出可能遗漏的真实情况，确保抽象不窄。

### 2.1 内部阻塞类

内部阻塞是当前行为执行时发现自身前置条件不足。

```text
吃东西时发现食物没了。
采集时发现目标资源已被别人采完。
伐木时发现斧头钝了。
伐木时发现斧头坏了。
建房时发现木头不够。
建房时发现地点不适合。
生火时发现干草湿了。
做火把时发现火源灭了。
搬运时发现负重过高。
治疗时发现草药不够。
教学时发现对方不在身边。
战斗时发现武器耐久不足。
```

内部阻塞特点：

```text
由当前 ActionDriver 检查出来。
通常和当前步骤直接相关。
可以转化为子目标。
解决后通常恢复原步骤。
```

### 2.2 外界打断类

外界打断不是当前行为自己发现的阻塞，而是世界事件进入。

```text
伐木时狼出现。
采集时蜘蛛靠近。
建房时下暴雨。
睡觉时敌对族群袭击。
做饭时小孩受伤。
火堆旁火势蔓延。
搬运时山路塌方。
探索时发现陌生族群。
守夜时外星飞行器出现。
冬夜寒潮突然加重。
同伴突然中毒。
资源堆被野兽破坏。
```

外界打断特点：

```text
来自 WorldEvent / ThreatEvent / WeatherEvent / SocialEvent。
可能与当前任务无关。
可能优先级高于当前目标。
需要统一打断系统判断是否抢占。
```

### 2.3 并发和竞争类

```text
两个族人都想拿最后一个红果。
一个族人在建房，一个族人在烧木头取暖，争同一批木头。
玩家命令同伴做 A，但同伴当前紧急目标是保护小孩。
狼追逐时，多个 Agent 同时逃向同一安全点。
```

P40 最小实现不做完整多 Agent 调度，但必须预留：

```text
资源锁。
目标所有权。
执行声明。
冲突诊断。
```

### 2.4 地图与位置类

```text
红果在东边，需要移动。
木头在树林，需要寻路。
狼在伐木地点，绕路更安全。
桥断了，路径阻塞。
火堆在营地，寒冷中的孩子必须靠近。
```

P40 不实现完整地图，但 Driver 接口必须预留：

```text
required_location_key
current_location_key
path_request
movement_driver
path_blocked_interrupt
```

### 2.5 长期任务类

```text
建房需要多个回合。
驯服野兽需要多次接触。
治疗伤员需要持续护理。
种植需要等待生长。
守夜持续一整晚。
```

P40 必须支持：

```text
DriverExecutionState
progress
remaining_ticks
pause/resume
cancel
timeout
```

### 2.6 执行失败类

```text
采集失败，没有找到红果。
伐木失败，木头太硬。
打磨失败，磨石耗尽。
建房失败，结构倒塌。
驱赶狼失败，狼没有退。
逃跑失败，被拦住。
```

执行失败后不能卡死，必须产生：

```text
FailureSignal
ReasoningRequest 重新推理
可见反馈
trace
```

### 2.7 学习反馈类

执行过程也会产生知识。

```text
第一次采集红果，学会红果来自灌木。
伐木时发现钝斧效率低。
狼被火把逼退，确认火把能驱狼。
雨天火堆熄灭，学习雨水会影响火。
```

P40 不直接写知识，但必须把执行反馈交给 P21/P34 学习链路。

### 2.8 中断后恢复类

```text
伐木遇狼 -> 处理狼 -> 继续伐木。
建房下雨 -> 暂停建房 -> 避雨/保护材料 -> 天晴后继续。
采集遇小孩受伤 -> 回去救小孩 -> 采集目标可能过期，重新评估。
```

恢复必须考虑：

```text
原目标是否仍然有效。
资源是否还在。
地点是否还安全。
计划是否过期。
是否需要重新推理。
```

## 3. 阶段目标

P40 完成后，系统必须具备以下能力：

1. 能保存 Agent 当前目标栈。
2. 能把 P39 `AgentPlan` 转成可执行状态。
3. 能逐 tick 执行当前计划步骤。
4. 能通过 ActionDriver 执行采集、吃、使用、伐木、打磨、建造等动作的最小抽象。
5. 能区分内部阻塞和外界打断。
6. 能把内部阻塞转成子目标。
7. 能把外界打断转成紧急目标。
8. 能暂停当前执行上下文。
9. 能处理完紧急目标后恢复原目标。
10. 能在恢复前验证原目标是否仍有效。
11. 能在失败或过期时请求 P39 重新推理。
12. 能输出执行 trace 和安全中文投影。
13. 能让“伐木遇狼”这种场景不写在 ChopWoodDriver 里，而由 InterruptSystem 统一处理。
14. 能为后续地图寻路、长期建造、多 Agent 调度预留接口。

## 4. 本阶段不解决的问题

### 4.1 不做完整地图寻路

P40 只定义 `MoveToDriver` 和 `PathRequest` 抽象，不实现完整网格寻路。

后续地图系统可以接入：

```text
current_location_key
target_location_key
path_blocked
estimated_ticks
```

### 4.2 不做完整多 Agent 调度

P40 支持单 Agent 目标栈和执行上下文。

不做：

```text
多人任务分配
资源拍卖
队列调度
群体施工
多人协同路径规划
```

但必须预留 resource reservation。

### 4.3 不做复杂动画/前端表现

H5 只显示文字反馈和当前目标，不做动画队列。

### 4.4 不做完整战斗系统

狼、蜘蛛、野兽在 P40 中是外界打断和威胁目标，不是完整战斗单位。

### 4.5 不做完整生产系统

建房、采集、伐木只做目标执行抽象和最小测试 fixture，不做正式经济平衡。

## 5. 核心设计原则

### 5.1 推理和执行分离

P39 负责选计划，P40 负责执行计划。

禁止：

```text
ActionDriver 自己决定长期目标。
GoalExecutionSystem 自己发明知识。
InterruptSystem 自己修改世界。
```

### 5.2 Driver 只处理自己的动作

`ChopWoodDriver` 只处理伐木，不处理狼。

狼出现必须走：

```text
WorldEvent -> InterruptSystem -> EmergencyGoal -> P39 Reasoner -> Threat Driver
```

### 5.3 内部阻塞和外界打断分开

```text
斧头钝了：InternalBlocker。
狼出现：ExternalInterrupt。
```

这两者必须用不同 DTO 和处理流程，不能混在一个 `failed` 里。

### 5.4 每回合只执行有限步骤

P40 不允许一口气完成整条长链。

每个 tick：

```text
先处理外界打断。
再执行当前 driver 一个 tick。
必要时生成 WorldChange。
然后保存执行状态。
```

### 5.5 可恢复、可取消、可重规划

每个执行上下文必须支持：

```text
pause
resume
cancel
expire
replan
```

### 5.6 状态变化仍由 P38 权威结算

P40 不能直接改对象数量、工具状态、威胁状态。

它只能输出：

```text
WorldInteractionInput
ReactionInputSet
DriverCommand
WorldChange 请求
```

实际变更仍通过现有规则/世界结算链。

### 5.7 外界打断必须有优先级

不是所有事件都打断当前目标。

例如：

```text
发现一块石头，不一定打断建房。
狼近身，必须打断伐木。
小孩受伤，高优先级。
远处下雨，如果当前在山洞里，不一定打断。
```

### 5.8 玩家可理解

H5 需要显示：

```text
同伴正在做什么。
为什么暂停。
发生了什么外界打断。
处理完后是否恢复原任务。
```

不能只显示工程词。

## 6. 模块划分

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| `pathfinder_goal_execution` | 目标执行模块 | 目标栈、执行上下文、暂停恢复 | AgentPlan、WorldEvent、DriverResult | ExecutionResult | 不推理长期目标 |
| `GoalExecutionSystem` | 目标执行系统 | 编排目标栈和当前步骤 | ExecutionRequest | ExecutionResult | 不直接改世界 |
| `GoalStackManager` | 目标栈管理器 | push/pop/pause/resume/cancel | GoalFrame | GoalStackState | 不评分计划 |
| `ActionDriverRegistry` | 行为驱动注册表 | action_key 到 Driver 映射 | DriverRequest | ActionDriver | 不执行规则结算 |
| `ActionDriver` | 行为驱动接口 | 执行具体动作一个 tick | DriverTickInput | DriverTickResult | 不处理外界打断 |
| `InternalBlockerResolver` | 内部阻塞解析 | 阻塞转子目标 | InternalBlocker | SubGoalRequest | 不处理狼/天气 |
| `InterruptSystem` | 外界打断系统 | 判断世界事件是否抢占 | WorldEventSignal、ExecutionState | InterruptDecision | 不写 Driver 逻辑 |
| `InterruptPriorityEvaluator` | 打断优先级评估 | 比较事件与当前任务优先级 | InterruptSignal、GoalFrame | priority | 不生成 WorldChange |
| `ExecutionResumeValidator` | 恢复验证 | 判断原任务能否恢复 | PausedContext、WorldSnapshot | ResumeDecision | 不盲目恢复 |
| `DriverCommandAdapter` | 驱动命令适配 | DriverCommand 转现有结算输入 | DriverCommand | WorldInteractionInput/ReactionInputSet | 不绕过 P38 |
| `ExecutionProjectionMapper` | 执行投影 | 安全中文解释 | ExecutionTrace | H5 摘要 | 不泄露 raw_state |

## 7. 核心枚举

### 7.1 `ExecutionFrameStatus`

| 枚举值 | 中文 | 说明 |
|---|---|---|
| `Unknown` | 未知 | 不允许运行时有效 |
| `Pending` | 等待执行 | 已入栈未开始 |
| `Running` | 执行中 | 当前活跃 |
| `Paused` | 暂停 | 被打断或等待条件 |
| `Blocked` | 内部阻塞 | 缺材料/工具/条件 |
| `Interrupted` | 被外界打断 | 狼、雨、敌袭等 |
| `Completed` | 完成 | 目标达成 |
| `Cancelled` | 取消 | 主动放弃 |
| `Expired` | 过期 | 目标不再有效 |
| `Failed` | 失败 | 执行失败 |
| `TestOnly` | 测试专用 | 测试 |

| 枚举值 | 中文 | 说明 |

| `MoveTo` | 移动到地点 | 地图接入前置 |
| `Teach` | 教学/传播 | 知识传播 |
| `Flee` | 逃离 | 威胁处理 |
| `CounterThreat` | 反制威胁 | 火把驱狼、陷阱 |
| `CareDependent` | 照顾依赖者 | 小孩、伤员 |
| `Wait` | 等待 | 等天气/条件 |
| `TestOnly` | 测试专用 | 测试 |

| 枚举值 | 中文 | 示例 |

## 8. 核心 DTO

### 8.1 `GoalFrame`

中文：目标栈中的一帧。

| 字段 | 类型 | 中文含义 | 说明 |
|---|---|---|---|
| `frame_id` | string | 帧 ID | 稳定 trace |
| `actor_key` | string | Agent | companion |
| `goal_id` | string | 对应 P39 目标 | 必填 |
| `plan_id` | string | 对应 P39 计划 | 必填 |
| `parent_frame_id` | optional<string> | 父目标 | 子目标时必填 |
| `status` | ExecutionFrameStatus | 状态 | Running/Paused 等 |
| `priority` | double | 优先级 | 紧急目标更高 |
| `created_tick` | uint64 | 创建 tick | 必填 |
| `expires_tick` | optional<uint64> | 过期 tick | 可空 |
| `resume_policy` | ResumeDecisionKind | 恢复策略 | 默认 Resume |
| `public_reason_zh_cn` | string | 中文原因 | H5 可显示 |
| `trace_keys` | vector<string> | trace | 可复盘 |

### 8.2 `ExecutionContext`

中文：Agent 当前执行上下文。

| 字段 | 类型 | 中文含义 |
|---|---|---|
| `context_id` | string | 执行上下文 ID |
| `actor_key` | string | Agent |
| `goal_stack` | vector<GoalFrame> | 目标栈 |
| `active_frame_id` | optional<string> | 当前帧 |
| `active_driver_state` | optional<DriverExecutionState> | 当前驱动状态 |
| `paused_contexts` | vector<PausedExecutionContext> | 暂停上下文 |
| `reserved_resources` | vector<ResourceReservation> | 资源预定 |
| `last_update_tick` | uint64 | 更新时间 |
| `trace_keys` | vector<string> | trace |

### 8.3 `DriverExecutionState`

中文：行为驱动运行状态。

| 字段 | 类型 | 中文含义 |
|---|---|---|
| `driver_state_id` | string | 状态 ID |
| `driver_kind` | ActionDriverKind | 驱动类型 |
| `action_key` | string | 动作 key |
| `object_key` | optional<string> | 对象 |
| `target_key` | optional<string> | 目标 |
| `current_location_key` | optional<string> | 当前地点 |
| `required_location_key` | optional<string> | 需要地点 |
| `progress` | double | 进度 0-1 |
| `remaining_ticks` | uint32 | 剩余 tick |
| `attempt_count` | uint32 | 尝试次数 |
| `local_memory_keys` | vector<string> | 局部执行记忆 |

字段：

```text

```

### 8.5 `DriverTickResult`

字段：

## 9. 核心服务接口

### 9.1 `GoalExecutionSystem`

职责：单 tick 执行入口。

```cpp
class GoalExecutionSystem {
public:
    Result<ExecutionResult> tick(const ExecutionTickRequest& request) const;
};
```

执行顺序：

```text
读取 ExecutionContext。
处理外界打断。
如果被抢占，暂停当前帧并插入紧急目标。
否则调用当前 ActionDriver tick。
处理内部阻塞。
适配 DriverCommand 到 P38 结算。

输出投影和 trace。
```

### 9.2 `GoalStackManager`

接口：

```cpp
pushGoal(frame)
```

规则：

```text

```

```cpp
class ActionDriver {
```

### 9.7 `InterruptPriorityEvaluator`

建议评分：

```text
interrupt_priority = severity * 0.45
                   + urgency * 0.35
                   + actor_vulnerability * 0.10
                   + goal_relevance * 0.10
```

打断阈值：

```text
>= 70：抢占当前任务。
40-69：记录并提示，可能触发重规划。
< 40：忽略或只观察。
```

### 9.8 `ExecutionResumeValidator`

职责：紧急目标结束后判断能否恢复。

检查：

```text

工具是否仍可用。
危险是否解除。
Agent 需求是否改变。
```

输出：

```text
Resume / Replan / Cancel / Wait / Fail。
```

职责：把 DriverCommand 交给现有系统。


输出示例：

```text
同伴正在为建房准备木头。

狼退走后，同伴回到树林继续伐木。
斧头变钝，同伴决定先打磨斧头。
```

### 9.11 `MaterialRequirementEvaluator`

职责：统一评估材料数量、状态、质量、预定和替代材料。

接口：

```cpp
class MaterialRequirementEvaluator {
public:
    Result<MaterialEvaluationResult> evaluate(const MaterialEvaluationInput& input) const;
    Result<std::vector<InternalBlocker>> toBlockers(const MaterialEvaluationResult& result) const;
};
```

输入：

```text
WorldSnapshot

MaterialRequirementSet

```

输出：

```text
每个材料的 MaterialAvailability。
是否满足全部需求。
缺口数量。
是否可用替代材料。
是否可通过 P28/P28.5 反应或 P39 计划获得。
```

硬性规则：

```text
所有 Driver 的材料判断必须走 MaterialRequirementEvaluator。
BuildStructureDriver 不允许自己写 wood >= 5。
MakeTorch/Build/Fire/Fuel 等材料组合必须优先复用 P28/P28.5 反应定义。
计划阶段只能 reservation，不能真正消耗材料。
执行结算阶段必须交给 P38 WorldChangeApplier 或 P28.5 ReactionChangeApplier 扣减。
```

### 9.12 `MaterialReservationManager`

职责：管理材料预定，防止多个目标同时认为自己拥有同一批材料。

接口：

```cpp
class MaterialReservationManager {
public:
    Result<ResourceReservation> reserve(const MaterialReserveRequest& request) const;
    Result<void> release(const std::string& reservation_id) const;
    Result<void> releaseByGoal(const std::string& goal_id) const;
    Result<int> reservedQuantity(const std::string& object_key) const;
};
```

规则：

```text
预定不是消耗。
目标完成、取消、失败、过期时必须释放预定。
执行真正消耗后必须同步清理预定。
恢复旧任务前必须重新验证 reservation 是否仍有效。
```

### 9.13 `ReactionMaterialResolver`

职责：当材料缺失时，连接 P28/P28.5 反应系统，查找可产出材料的反应链。

接口：

```cpp
class ReactionMaterialResolver {
public:
    Result<std::vector<ReactionMaterialLink>> findProducers(const MaterialRequirement& requirement) const;
};
```

示例：

```text
缺 torch -> 找 make_torch。
make_torch 需要 processed_wood + camp_fire。
缺 processed_wood -> 找 cut_wood。
缺 camp_fire -> 找 ignite_fire。
```

边界：

```text
ReactionMaterialResolver 只返回可选产出链，不直接选择方案。
选择仍交给 P39 评分。
执行仍交给 P40 Driver + P38/P28.5 结算。
```

## 10. 标准流程

### 10.1 正常执行流程

```text
1. P39 生成 AgentPlan：建房。
2. GoalExecutionSystem 创建 GoalFrame。
3. BuildStructureDriver 检查前置。
4. 发现木头不足 -> InternalBlocker。
5. InternalBlockerResolver 生成子目标 AcquireObject(wood)。
6. P39 为子目标生成计划：伐木。
7. GoalStackManager push 子目标。
8. ChopWoodDriver 执行。
9. 生成木头 WorldChange。
10. 子目标完成。
11. 恢复 BuildStructureDriver。
12. 建房完成。
```

### 10.2 内部阻塞流程：伐木时斧头钝

```text
当前：ChopWoodDriver running。
Driver 检查 axe.sharpness <= 0。
返回 InternalBlocker(ToolStateInsufficient)。
InternalBlockerResolver 生成 RestoreToolState(axe)。
P39 选择 SharpenToolDriver。
打磨完成。
ResumeValidator 检查伐木仍有效。
恢复伐木。
```

### 10.3 外界打断流程：伐木遇狼

```text
当前：ChopWoodDriver running。
WorldEvent：wolf_appeared at forest。
InterruptSystem 判断该 Agent 在 forest，事件可感知。
InterruptPriorityEvaluator 评分 >= 70。
Decision：PauseAndInsertEmergencyGoal。
GoalStackManager 暂停伐木帧。
插入 ReduceThreat(wolf) 或 FleeThreat(wolf)。
P39 根据知识选择火把/逃跑/求助。
CounterThreatDriver 执行。
威胁解除。
ResumeValidator 检查 forest 是否安全、axe 是否还在、wood 是否还需要。
恢复伐木或重规划。
```

### 10.4 外界打断流程：下雨影响建房

```text
当前：BuildStructureDriver running。
WorldEvent：rain_started。
InterruptSystem 判断雨影响建房材料。
如果 severity 中等：PauseAndInsertEmergencyGoal(protect_materials)。
如果 severity 低：ObserveOnly，建房继续但进度降低。
如果材料被毁：Cancel/Replan。
```

### 10.5 玩家命令覆盖流程

```text
玩家命令：停止建房，回营地。
ExternalInterruptKind::CommandOverride。
InterruptDecision：CancelCurrentGoal 或 Pause。
记录原因：player_override。
不得把玩家命令当作隐藏规则。
```

## 12. 外界打断系统设计

### 12.1 WorldEvent 到 InterruptSignal

P40 需要一个适配层：

```text
ThreatEvent -> ThreatAppeared / ThreatEscalated
WeatherEvent -> WeatherChanged
TribeEvent -> DependentInDanger / SocialConflict
MapEvent -> PathBlocked / DiscoveryFound
ObjectEvent -> ResourceDestroyed / ToolLostOrBroken
```

### 12.2 感知过滤

不是所有世界事件 Agent 都知道。

过滤条件：

```text
同地点。
可见范围内。
由同伴告知。
族群广播。
声音/气味/痕迹可感知。
```

P40 最小实现可以用：

```text
same location -> perceivable
explicit target_actor_keys -> perceivable
```

### 12.3 优先级分类

| 类别 | 优先级 | 示例 |
|---|---:|---|
| 生命威胁 | 90-100 | 狼近身、敌袭、小孩濒危 |
| 严重损失 | 70-89 | 房子着火、食物被毁 |
| 任务阻碍 | 50-69 | 下雨影响建房、路径阻塞 |
| 机会发现 | 30-49 | 发现新资源 |
| 背景变化 | 0-29 | 远处响动 |

### 12.4 打断策略

```text
生命威胁：暂停当前任务，插入紧急目标。
严重损失：暂停或重规划。
任务阻碍：内部化为 blocker 或重规划。
机会发现：只记录或低优先级目标。
背景变化：只投影或记忆。
```

## 13. 恢复与重规划

### 13.1 恢复检查

紧急目标完成后必须检查：

```text
原目标还需要吗？
原目标是否过期？
原地点是否安全？
原材料是否还在？
原工具是否还可用？
Agent 需求是否发生重大变化？
```

### 13.2 恢复结果

```text
Resume：继续原 driver。
Replan：保留主目标，重新调用 P39。
Cancel：目标不再有效。
Wait：等待条件恢复。
Fail：标记失败并反馈。
```

### 13.3 示例

```text
狼被赶走 -> Resume 伐木。
狼没走但 Agent 逃回营地 -> Replan 建房材料来源。
木头被烧毁 -> Replan。
孩子受伤后健康仍危险 -> Cancel 建房，继续照顾孩子。
```

## 14. 和 P39 的接入方式

P40 不生成长期计划，但会在三种情况下请求 P39：

```text
新主目标进入。
内部阻塞需要子目标。
外界打断需要紧急目标。
恢复检查要求重规划。
```

请求必须带：

```text
当前 GoalStack。
被阻塞原因。
外界打断原因。
禁止重复的失败计划。
当前资源预定。
```

P39 返回新计划后，P40 负责入栈执行。

### 14.1 P39 再次审核后的必须承接项

P39 已验收，但再次审核报告明确留下三类必须由 P40/P41 承接的边界。P40 研发时不能把这些当作“以后再说”的口头事项，必须在执行层设计中显式预留接口和测试。

#### 14.1.1 `known_claims` 必须成为 Agent 自主行为的知识来源

P39 已经把 `WorldActorRuntimeState` 从只保存 `known_effect_keys` 推进到保存真实 `known_claims`。

P40 必须继续遵守：

```text
Agent 自主执行所需知识，必须来自 actor.known_claims / KnowledgeClaim。
P40 不得重新退回 effect_key -> object_key 的合成映射。
P40 调用 P39 重规划时，必须传入当前执行者真实可用的 KnowledgeClaim。
```

禁止实现：

```text
if effect_key == restore_hunger -> red_berry
if effect_key == repel_beast -> torch
if effect_key == make_torch -> wood_processed
```

正确实现方向：

```text
DialogSession / KnowledgeRepository 产生 KnowledgeClaim。
WorldSnapshotAdapter 把 KnowledgeClaim 投影到 WorldActorRuntimeState.known_claims。
P40 ExecutionContext 只缓存 claim 引用或安全副本。
P40 请求 P39 时原样传递这些 claim。
P39 根据 claim.subject / claim.predicate / claim.conditions / effect_semantics 规划。
```

#### 14.1.2 `ReactionPlanningAdapter` 是过渡桥，P40 不能绕过它

P39 已新增 `ReactionPlanningAdapter`，把 P28 reaction fixture 的输入模式转换为 `PlanPrecondition`。

P40 必须复用这个方向，并在执行层继续扩展为通用材料/反应桥：

```text
ReactionRule -> MaterialRequirement
ReactionRule -> DriverCommand
ReactionOutputTemplate -> 可产出对象/状态
ConditionExpressionRef -> ObjectStateInvalid / ToolStateInvalid / MissingCondition
```

P40 不允许在 Driver 内部重新写：

```text
make_torch = wood_processed + camp_fire
cut_wood = wood + axe + sharpness
restore_sharpness = whetstone + axe
```

这些关系必须来自：

```text
P28/P28.5 ObjectReactionRule
ReactionPlanningAdapter / ReactionMaterialResolver
MaterialRequirementEvaluator
```

#### 14.1.3 `build_house` 仍是 P39 fallback，P40 必须补建筑反应/任务配置入口

P39 当前 `build_house` 仍允许 fallback：

```text
wood_processed >= 3
```

这只能作为 P39 测试语义，不允许成为最终建筑系统。

P40 最少要做到：

```text
BuildStructureDriver 不直接写死 wood_processed 数量。
BuildStructureDriver 从 MaterialRequirementSet 或 building recipe 读取需求。
ReactionMaterialResolver 能把缺少的 building material 转成获取链。
后续 P41/P42 可以把 building recipe 放到配置，不需要推翻 P40 Driver 接口。
```

如果本阶段无法完整实现建筑 recipe，也必须留下明确接口和测试 stub，证明 `build_house` 不再只能依赖 P39 fallback。

## 15. 和 P38 的接入方式

P40 输出的 DriverCommand 必须转换为 P38 可结算输入。

```text
WorldInteractionInput
WorldChange
WorldChangeApplier
WorldProjectionPatch
```

P40 不直接扣资源。

例如：


## 16. 材料数量、预定与反应系统接入

P40 必须把材料处理作为执行层硬能力，否则 P28/P28.5 反应系统会变成“能配置但执行层不会用”的空壳。

### 16.1 材料处理的权威边界

```text
P39：决定“需要什么材料”和“哪条获取方案更合适”。
P40：检查材料是否足够、预定材料、发现缺口、触发子目标。
P28/P28.5：提供材料如何反应、如何产出、如何组合的规则来源。
P38：实际扣减、生成、状态变化的世界结算权威。
H5：只显示材料缺口和执行说明。
```

禁止：

```text
P40 直接生成 torch。
P40 直接扣 wood。
P40 在 Driver 里写死 make_torch = wood + fire。
P39 根据隐藏定义直接知道某材料产出方式。
```

### 16.2 标准材料检查流程

```text
1. Driver 声明 MaterialRequirementSet。
2. MaterialRequirementEvaluator 读取 WorldSnapshot。
3. MaterialRequirementEvaluator 扣除 ExecutionContext.reserved_resources。
4. 检查数量、状态、质量、替代材料。
5. 如果满足，MaterialReservationManager 预定材料。
6. 如果不满足，生成 InternalBlocker(InsufficientQuantity/ObjectStateInvalid/ResourceReserved)。
7. InternalBlockerResolver 把缺口转成 AgentGoalRequest。
8. P39 基于知识和 ReactionMaterialResolver 查询产出链并选择方案。
9. P40 执行获取材料的 Driver。
10. 真实消耗/生成通过 P38/P28.5 结算。
```

### 16.3 建房材料例子

```text
目标：build_house。
需求：processed_wood 5，binding_material 2。
当前 processed_wood 3。
其中 1 被 fire_maintenance 目标预定。
可用 processed_wood 2。
缺口 3。
MaterialRequirementEvaluator 生成 blocker：InsufficientQuantity(processed_wood, missing=3)。
ReactionMaterialResolver 查找 processed_wood 生产方式：cut_wood。
P39 选择伐木。
P40 执行 ChopWoodDriver。
ChopWoodDriver 又通过 MaterialRequirementEvaluator 检查 axe.sharpness 和 wood.quantity。
```

### 16.4 火把材料例子

```text
目标：AcquireObject(torch) 或 ReduceThreat。
需求：torch.quantity > 0。
当前 torch = 0。
ReactionMaterialResolver 找到 make_torch。
make_torch 需求：processed_wood >= 1，camp_fire >= 1。
processed_wood 不足 -> cut_wood。
camp_fire 不足 -> ignite_fire。
P39 比较链路成本。
P40 逐步执行。
```

### 16.5 替代材料

材料需求必须支持替代组：

```text
building_wood：processed_wood / straight_branch / plank。
fuel：dry_grass / dry_branch / charcoal。
binding_material：vine / rope / leather_strip。
```

选择替代材料仍由 P39 评分：

```text
processed_wood 最适合建房。
dry_branch 可以临时修补但耐久低。
湿木头需要先烘干。
```

### 16.6 材料状态

材料不是只有数量。

必须支持：

```text
wet / dry
fresh / rotten
processed / raw
sharp / dull
stable / fragile
burning / extinguished
```

状态不满足时，不应该简单说数量不足，而要生成：

```text
InternalBlocker(ObjectStateInvalid)
```

例如：

```text
有木头 5，但都是 wet。
建房可以用，生火不能用。
火把制作需要 dry 或 processed。
```

### 16.7 预定和消耗

计划阶段只做预定：

```text
Reserve wood 5 for build_house。
```

执行结算才消耗：

```text
WorldChange(ObjectConsumed wood -5)。
```

取消时释放：

```text
build_house cancelled -> release wood reservation。
```

中断时保持或释放取决于恢复策略：

```text
狼来了，暂停建房 -> reservation 保留一段时间。
暴雨毁坏材料 -> reservation 失效，恢复前 Replan。
```

### 16.8 和 P28/P28.5 的硬连接

P40 必须把反应系统当作材料来源查询入口。

```text
缺目标对象 -> 查 ReactionMaterialResolver。
ReactionMaterialResolver 读取 P28/P28.5 的 ReactionContentPack / ObjectReactionRule。
返回能产出目标对象的 ReactionMaterialLink。
P39 对这些反应链评分。
P40 通过 Driver 执行反应输入。
P28/P28.5 负责反应结果合法性。
```

验收要求：

```text
测试中必须证明：缺 torch 时不是写死 make_torch，而是通过 ReactionMaterialResolver 找到 make_torch 链。
测试中必须证明：缺 processed_wood 时能从反应/行为产出链找到 cut_wood。
测试中必须证明：湿木头不能用于需要 dry fuel 的反应，且返回 ObjectStateInvalid。
```

## 21. 风险与对策

### 21.1 风险：P40 变成另一个推理器

```text
Driver 只处理动作 mechanics。
缺什么返回 InternalBlocker。
```

## 24. 结论

P40 的价值在于让 Agent 不只是“想出计划”，而是能在动态世界中持续完成目标。

完成 P40 后，系统应该能表达：

```text
同伴想建房。
建房缺木头，所以去伐木。
伐木时斧头钝了，所以先打磨。
伐木时狼来了，所以暂停伐木，插入处理狼。
狼退走后，检查目标仍有效，于是继续伐木。
木头够了，回到建房。
```

这不是写死剧情，而是：

```text
P39 负责推理目标和方案。
P40 负责执行、阻塞、打断、恢复、重规划。
P38 负责世界结算。
P21/P34 负责学习和因果归因。
H5 负责安全展示。
```

这套层次打通后，后续地图、狼、雨、孩子、房子、族群分工和文明建设都可以自然接入。
