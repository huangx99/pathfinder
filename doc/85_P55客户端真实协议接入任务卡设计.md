# P55 客户端真实协议接入任务卡设计

## 1. 阶段定位

P55 是 V2.0 开放世界路线中“客户端真实接入后端协议”的阶段。

它接在 P53 客户端协议和 P54 本地 Client HTTP 网关之后，不是正式 UI 阶段，也不是新增玩法阶段。

一句话定位：

```text
P55 让 app/frontend 下的客户端从 mock / 调试状态，切换为只通过 P54 HTTP Gateway 消费 P53 Client Protocol 的真实客户端。
```

P55 完成后的链路必须是：

```text
玩家操作客户端
  ↓
ClientApi.fetch('/api/client/bootstrap|command|refresh|reset')
  ↓
P54 Client HTTP Gateway
  ↓
P53 Client Protocol
  ↓
ClientSessionGateway / ClientCommandGateway
  ↓
WorldCommandPipeline
  ↓
世界 Runtime / 背包 / 采集 / 制作 / 学习 / 教学 / Agent
  ↓
ClientWorldProjection / projection_patch / available_commands / event_feed
  ↓
客户端只读展示与按钮渲染
```

P55 的核心不是“界面好看”，而是证明：

```text
前端不写规则，也能真实驱动后端世界。
```

## 2. 为什么现在做

P54 已经提供了本地 HTTP Gateway，并通过复审：

```text
1. 可以服务 app/frontend 静态文件。
2. 可以接收 /api/client/bootstrap。
3. 可以接收 /api/client/command。
4. 可以接收 /api/client/refresh。
5. 可以接收 /api/client/reset。
6. 真实 socket HTTP smoke test 已通过。
7. P53 与 P43-P52 回归已通过。
```

如果 P54 之后直接做正式 UI，会出现以下风险：

```text
1. UI 为了“看起来能玩”继续写 mock 逻辑。
2. 前端自行拼红果、狼、火把、采集、制作等玩法结果。
3. 玩家看到的是前端假世界，不是后端真实世界。
4. 后续 UI 越做越复杂，越难切回真实协议。
5. command / projection / patch 的真实边界没有被验证。
```

所以 P55 必须先做“真实协议接入层”。

本阶段完成后，后续 P56 才能安心做正式中文横屏游戏 UI，因为 P56 只需要消费 P55 适配好的真实状态，不需要理解引擎内部。

## 3. 阶段目标

P55 完成后，客户端必须具备以下能力：

```text
1. 页面启动后自动调用 POST /api/client/bootstrap。
2. 客户端能保存 client_id、session_id、projection_version。
3. 客户端能显示当前连接状态：真实后端 / 连接失败 / 等待响应。
4. 客户端能读取后端 full_projection 并转换成前端只读 view model。
5. 客户端能读取后端 available_commands 并生成按钮。
6. 玩家点击按钮时，只提交后端给出的 option_id。
7. 命令响应后，客户端能处理 projection_patch。
8. 如果 requires_full_refresh=true，客户端能自动调用 refresh。
9. reset 能清理本地 session 并重建后端会话。
10. mock 只能作为开发备用，不允许作为 P55 验收路径。
```

P55 最重要的验收句：

```text
前端没有任何玩法结果权威，只有协议适配、状态展示和用户输入提交。
```

## 4. 本阶段不解决的问题

P55 不解决以下问题：

```text
1. 不设计正式游戏 UI。
2. 不做最终中文横屏界面。
3. 不做美术、动画、音效。
4. 不新增世界玩法规则。
5. 不新增红果、狼、火把、房屋等内容。
6. 不重构 P53 Client Protocol。
7. 不修改 P54 HTTP 路径语义。
8. 不做 WebSocket。
9. 不做多人在线。
10. 不做公网安全、账号、权限系统。
11. 不做完整存档系统。
12. 不做离线缓存和断线重放。
```

这些内容放到后续阶段：

```text
P56：正式中文横屏客户端 UI。
P57：真实地图投影增强与交互可读性。
P58：本地存档 / 读档 HTTP API。
P59：可选 WebSocket 或事件推送。
P60：桌面打包启动器。
```

## 4.4 旧前端与旧后端清理说明

在 P55 开发前，旧 `app/frontend/h5_playable` 与旧 H5 后端原型已经清理，避免低上下文 AI 继续沿用过时架构。

P55 当前唯一允许接入的客户端目录是：

```text
app/frontend/client/
```

该目录当前只是干净占位，不是完整游戏 UI。P55 研发必须在这个干净目录上接入真实协议，而不是恢复旧 H5 原型。

已经废弃的旧概念：

```text
backend/include/pathfinder/h5_dialog/
backend/src/h5_dialog/
backend/include/pathfinder/h5_playable/
backend/src/h5_playable/
backend/include/pathfinder/h5_projection/
backend/src/h5_projection/
pathfinder_h5_dialog_server
pathfinder_h5_playable_server
scripts/restart_h5_1999.sh
```

禁止在 P55 中重新引入这些旧目录或旧二进制。

## 4.5 现有前端继承约束

P55 必须基于当前干净客户端占位继续接入，不允许恢复旧 H5 前端，也不允许另起一套无关前端。

当前已有前端文件：

```text
app/frontend/client/index.html
app/frontend/client/app.js
app/frontend/client/style.css
```

P55 的研发策略是：

```text
保留当前 app/frontend/client 作为唯一客户端入口。
在当前占位页面上接入真实协议状态、命令、投影和错误展示。
可以扩展 DOM 区域，但不能恢复旧 H5 原型。
不能新增 mock 作为默认数据来源。
新增模块必须服务于真实协议接入，而不是重新做一套 UI。
```

允许做的改动：

```text
1. 在现有 app.js 内部分区重构，增加 ClientApi / Store / Controller。
2. 或在 app/frontend/client/ 下新增协议适配文件，再由现有 app.js 引入。
3. 修改 index.html 以增加真实连接状态、错误提示、调试折叠区。
4. 修改 style.css 以支持真实模式状态展示。
5. 如果新增开发 mock，必须显式隔离，确保默认不走 mock。
```

禁止做的改动：

```text
1. 禁止新建另一套与现有页面无关的客户端壳来绕过旧代码。
2. 禁止把现有 app/frontend/client 整体废弃。
3. 禁止为了接入协议重写正式游戏 UI。
4. 禁止把 mock 逻辑换个名字继续作为真实模式。
5. 禁止在不说明迁移关系的情况下删除现有交互入口。
```

如果研发认为当前占位前端某段代码必须删除，必须在 P55 自检报告中写清楚：

```text
删除了什么。
为什么不能复用。
删除后由哪个真实协议模块替代。
是否影响后续 P56 正式 UI。
```

验收时必须检查：

```text
P55 是“当前干净客户端接真实后端”，不是“恢复旧 H5”或“重新写一个无关 demo”。
```

## 5. 核心设计原则

### 5.1 后端是唯一规则权威

客户端不得判断：

```text
吃什么会怎样。
砍树会得到什么。
制作需要什么材料。
狼会不会攻击。
NPC 会不会学习。
命令是否成功。
知识是否形成。
```

客户端只能展示后端返回的结果。

错误实现：

```text
if (itemId === 'red_berry') hunger += 10
if (command === 'craft_torch') addItem('torch')
```

正确实现：

```text
点击后端给出的 option_id。
等待后端返回 command response。
根据 projection_patch / full_projection 更新展示。
```

### 5.2 available_commands 是按钮唯一来源

客户端按钮必须来自后端 `available_commands`。

禁止：

```text
前端扫描对象类型后自己生成“采集”“吃”“砍”“制作”按钮。
前端根据 itemId 自己拼 WorldCommandDto。
前端硬编码按钮对应的命令参数。
```

允许：

```text
前端按 command_kind / display_text / option_id 渲染按钮。
前端可以做排序、分组、禁用态展示。
前端提交时只传 option_id 和必要 selection_context。
```

### 5.3 Projection 是只读展示，不是规则输入

客户端可以把 `ClientWorldProjection` 转成 view model，但不得反推隐藏真相。

例如：

```text
可显示 visible_cells。
可显示 visible_entities。
可显示 inventories。
可显示 knowledge。
可显示 area_effects。
```

但不得：

```text
根据 projection 里的名字推断配置 JSON。
根据 visible entity key 猜物品真实效果。
根据前端本地表决定命令结果。
```

### 5.4 Patch 优先，Refresh 兜底

命令响应后：

```text
如果 requires_full_refresh=false，优先应用 projection_patch。
如果 requires_full_refresh=true，必须调用 refresh 获取 full_projection。
如果 patch 应用失败，也必须调用 refresh。
```

客户端不得因为 patch 失败而自己修补状态。

### 5.5 Session 是协议状态，不是玩法状态

客户端可以保存：

```text
client_id
session_id
projection_version
last_available_commands
connection_mode
```

客户端不得保存：

```text
真实物品效果
NPC 隐藏知识
世界未探索区域
规则配置 JSON
后端 runtime 内部状态
```

### 5.6 Mock 只能开发，不能验收

P55 可以保留 mock 作为前端开发备用，但必须满足：

```text
1. 默认优先真实后端。
2. mock 模式必须有明显标识。
3. P55 验收必须禁用 mock 或证明未走 mock。
4. mock 文件不能被当作规则来源。
```

### 5.7 客户端目录可删除，不影响后端

客户端代码必须保持在：

```text
app/frontend/...
```

不得写入：

```text
backend/src/world_*
backend/src/client_protocol
backend/src/client_http
content/core
```

除非确实发现 P53/P54 协议缺陷，并单独形成后端修复任务。

## 6. 模块划分

P55 主要是前端适配层，建议在当前客户端目录下新增或重组模块。

当前目录：

```text
app/frontend/client/
```

P55 可以继续使用该目录做临时真实联调，但代码命名必须使用通用 client 概念，不要新增 H5V2 / h5_v2 后端命名。

建议前端模块：

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| `client_api.js` | 客户端 API | 封装 bootstrap/command/refresh/reset | session、option_id、request dto | 后端 response dto | 禁止写玩法规则 |
| `client_session_store.js` | 会话存储 | 管理 client_id/session_id/version | bootstrap/command/refresh response | 当前 session 状态 | 禁止保存隐藏世界真相 |
| `client_projection_store.js` | 投影存储 | 保存 full_projection 并应用 patch | full_projection/projection_patch | 当前只读 projection | 禁止自行推导规则结果 |
| `client_command_store.js` | 命令存储 | 保存 available_commands 并提供按钮 view model | 后端 available_commands | command button list | 禁止自己生成玩法按钮 |
| `client_connection_state.js` | 连接状态 | 管理真实后端/失败/请求中状态 | fetch 生命周期 | UI 状态 | 禁止吞掉错误 |
| `client_protocol_adapter.js` | 协议适配器 | 把 DTO 转成前端 view model | P53 DTO | view model | 禁止修改 DTO 语义 |
| `client_event_log.js` | 事件日志 | 展示后端事件和本地请求结果 | event_feed/response | 中文日志列表 | 禁止伪造后端事件 |
| `main.js` 或现有 `app.js` | 页面入口 | 初始化、绑定 UI、调度模块 | DOM/load/click | 页面状态 | 禁止内嵌规则判断 |

P55 不要求必须拆成多个文件；但如果继续单文件实现，也必须在代码结构上有清晰分区，并通过架构扫描限制玩法硬编码。

## 7. 核心枚举

P55 前端应定义少量客户端状态枚举，避免 magic string 到处飞。

### 7.1 ClientConnectionMode

中文：客户端连接模式。

用途：表示当前客户端连接的权威来源。

枚举值：

| 值 | 中文 | 使用场景 |
|---|---|---|
| `unknown` | 未知 | 页面刚加载，还没尝试连接 |
| `real_backend` | 真实后端 | bootstrap 成功，正在使用 P54 HTTP Gateway |
| `mock_dev` | 开发模拟 | 仅开发备用，不可作为验收 |
| `offline_error` | 离线错误 | 无法连接后端 |

规则：

```text
默认 unknown。
bootstrap 成功后进入 real_backend。
fetch 连接失败进入 offline_error。
mock_dev 必须人工显式开启，并在 UI 显示明显警告。
```

### 7.2 ClientRequestState

中文：请求状态。

用途：控制按钮是否 pending、防止重复点击。

枚举值：

| 值 | 中文 | 使用场景 |
|---|---|---|
| `idle` | 空闲 | 可以提交命令 |
| `bootstrapping` | 初始化中 | 正在 bootstrap |
| `submitting_command` | 命令提交中 | 正在 command |
| `refreshing` | 刷新中 | 正在 refresh |
| `resetting` | 重开中 | 正在 reset |
| `failed` | 请求失败 | 最近一次请求失败 |

规则：

```text
同一时间只允许一个主要请求状态。
submitting_command 时应禁用命令按钮，避免重复提交。
失败后必须允许玩家重试 bootstrap/refresh。
```

### 7.3 ClientSyncState

中文：投影同步状态。

用途：表示本地 projection 是否可信。

枚举值：

| 值 | 中文 | 使用场景 |
|---|---|---|
| `empty` | 无投影 | 尚未 bootstrap |
| `synced` | 已同步 | projection_version 与后端响应一致 |
| `patching` | 应用补丁中 | 正在处理 projection_patch |
| `needs_full_refresh` | 需要全量刷新 | 后端要求或本地 patch 失败 |
| `stale` | 已过期 | 本地版本与请求版本不一致 |

规则：

```text
bootstrap 成功后进入 synced。
command 返回 patch 时短暂进入 patching。
requires_full_refresh=true 时进入 needs_full_refresh 并自动 refresh。
refresh 成功后回到 synced。
```

### 7.4 ClientErrorKind

中文：客户端错误类型。

用途：将错误转成中文提示。

枚举值：

| 值 | 中文 | 使用场景 |
|---|---|---|
| `none` | 无错误 | 正常 |
| `network_failed` | 网络失败 | fetch 抛错或 HTTP 不可达 |
| `bad_response` | 响应格式错误 | JSON 解析失败或缺字段 |
| `api_error` | API 错误 | 后端返回 ok=false |
| `command_blocked` | 命令被阻止 | 后端业务结果 blocked |
| `version_stale` | 版本过期 | requires_full_refresh=true |
| `unknown` | 未知错误 | 兜底 |

规则：

```text
错误提示必须中文化。
不能因为错误而切回 mock，除非开发者显式操作。
```

## 8. 核心 DTO / 数据结构

P55 不重新定义 P53 DTO，只在客户端定义本地状态结构和 view model。

### 8.1 ClientRuntimeState

中文：客户端运行时状态。

职责：聚合客户端当前协议状态。

字段表：

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `clientId` | string | 客户端 ID | 是 | session store | API 层 | 不为空，localStorage 可持久化 | 不得包含玩法含义 |
| `sessionId` | string | 会话 ID | 是 | session store | API 层 | 不为空，bootstrap 可创建 | 不得当世界存档用 |
| `projectionVersion` | number | 投影版本 | 是 | projection store | command/refresh | 必须来自后端响应 | 不得前端自增 |
| `connectionMode` | ClientConnectionMode | 连接模式 | 是 | connection state | UI | 必须是枚举值 | 不得默认 mock |
| `requestState` | ClientRequestState | 请求状态 | 是 | API 层 | UI | 同时只能一个 | 不得忽略 pending |
| `syncState` | ClientSyncState | 同步状态 | 是 | projection store | UI/API | patch/refresh 驱动 | 不得本地伪造 synced |
| `lastError` | ClientErrorViewModel/null | 最近错误 | 否 | API 层 | UI | 错误必须中文可读 | 不得吞掉后端错误 |
| `projection` | ClientWorldProjection/null | 后端投影 | 否 | projection store | adapter/UI | 只来自 bootstrap/refresh/patch | 不得写隐藏真相 |
| `availableCommands` | WorldCommandOptionDto[] | 可用命令 | 是 | command store | UI/API | 只来自后端 | 不得前端新增玩法按钮 |
| `eventLog` | ClientEventViewModel[] | 事件日志 | 是 | event log | UI | 最多保留 N 条 | 不得伪造后端事件 |

### 8.2 ClientCommandButtonViewModel

中文：命令按钮展示模型。

职责：把后端 `WorldCommandOptionDto` 转成按钮。

字段表：

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `optionId` | string | 后端选项 ID | 是 | command adapter | click handler | 必须来自后端 available_commands | 不得前端生成 |
| `label` | string | 按钮中文标题 | 是 | command adapter | UI | 优先用后端 display text，否则安全 fallback | 不得写死玩法结果 |
| `description` | string | 按钮说明 | 否 | command adapter | UI | 可为空 | 不得泄露隐藏真相 |
| `group` | string | 分组 | 否 | command adapter | UI | 可按 command_kind 显示 | 不得影响执行 |
| `disabled` | boolean | 是否禁用 | 是 | UI state | UI | pending 时 true | 不得当业务校验 |
| `rawCommandKind` | string | 原始命令类型 | 否 | command adapter | debug UI | 只用于调试/分组 | 不得驱动规则结果 |

### 8.3 ClientProjectionViewModel

中文：客户端投影展示模型。

职责：给临时联调页面展示后端世界状态。

字段表：

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `summary` | string | 当前世界摘要 | 是 | projection adapter | UI | 可根据可见字段生成 | 不得加入隐藏规则 |
| `visibleCells` | array | 可见格子 | 是 | projection adapter | UI | 来自 projection.visible_cells | 不得补未知地图 |
| `visibleEntities` | array | 可见实体 | 是 | projection adapter | UI | 来自 projection.entities | 不得补配置真相 |
| `inventories` | array | 可见背包 | 是 | projection adapter | UI | 来自 projection.inventories | 不得本地增删物品 |
| `knowledge` | array | 可见知识 | 是 | projection adapter | UI | 来自 projection.knowledge | 不得本地生成知识 |
| `areaEffects` | array | 区域效果 | 是 | projection adapter | UI | 来自 projection.area_effects | 不得本地推断区域魔法 |
| `projectionVersion` | number | 投影版本 | 是 | projection store | UI/API | 来自后端 | 不得自增 |

### 8.4 ClientErrorViewModel

中文：客户端错误展示模型。

字段表：

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| `kind` | ClientErrorKind | 错误类型 | 是 | API 层 | UI | 必须是枚举 | 不得为空 |
| `title` | string | 中文标题 | 是 | error mapper | UI | 中文可读 | 不得只显示英文 key |
| `message` | string | 中文详情 | 是 | error mapper | UI | 尽量带后端 message | 不得泄露隐藏状态 |
| `retryAction` | string/null | 建议重试动作 | 否 | error mapper | UI | bootstrap/refresh/reset | 不得自动乱重置 |
| `raw` | object/string | 原始错误 | 否 | API 层 | debug | 仅开发显示 | 不得常规 UI 暴露敏感 body |

## 9. 核心服务接口

以下是建议接口，研发可按当前前端技术栈做等价实现。

### 9.1 ClientApi

中文职责：统一 HTTP 请求入口。

建议接口：

```js
class ClientApi {
  constructor({ baseUrl, fetchImpl }) {}

  async bootstrap(request) {}
  async submitCommand(request) {}
  async refresh(request) {}
  async reset(request) {}
}
```

输入输出：

| 方法 | 输入 | 输出 | 失败情况 |
|---|---|---|---|
| `bootstrap` | ClientBootstrapRequest | ClientBootstrapResponse | 网络失败、400、500、JSON 格式错 |
| `submitCommand` | ClientCommandRequest | ClientCommandResponse | 网络失败、API 错误、业务 blocked |
| `refresh` | ClientRefreshRequest | ClientRefreshResponse | session 不存在、版本异常 |
| `reset` | ClientResetRequest | ClientResetResponse | 未确认、网络失败 |

要求：

```text
1. baseUrl 默认同源，即 ''。
2. fetch 路径固定为 /api/client/*。
3. 不允许 ClientApi 内出现 itemId、wolf、torch、berry 等玩法分支。
4. 所有 response 必须先校验基本结构，再交给 store。
5. HTTP 200 也可能是业务 blocked，不可当成一定成功。
```

### 9.2 ClientSessionStore

中文职责：管理协议会话状态。

建议接口：

```js
class ClientSessionStore {
  load() {}
  save() {}
  clear() {}
  ensureClientId() {}
  applyBootstrapResponse(response) {}
  applyResetResponse(response) {}
  getCommandContext() {}
}
```

规则：

```text
client_id 可以 localStorage 持久化。
session_id 可以 localStorage 持久化，但后端不认时必须重新 bootstrap。
projection_version 只能来自后端响应，不允许前端自增。
```

### 9.3 ClientProjectionStore

中文职责：保存与更新后端投影。

建议接口：

```js
class ClientProjectionStore {
  applyFullProjection(fullProjection, version) {}
  applyPatch(patch, nextVersion) {}
  markNeedsFullRefresh(reason) {}
  getProjection() {}
  toViewModel() {}
}
```

Patch 规则：

```text
P55 可以先实现保守策略：
1. 如果 patch 结构完整且可安全 merge，则 merge。
2. 如果 patch 缺字段或 merge 失败，立即 markNeedsFullRefresh。
3. 不允许前端猜测缺失字段。
```

如果 P53 patch 语义还不够完整，P55 可以选择：

```text
命令后 always refresh。
```

但必须在文档和代码注释中写清楚这是临时保守策略，不是前端规则推导。

### 9.4 ClientCommandStore

中文职责：保存后端可用命令并生成按钮。

建议接口：

```js
class ClientCommandStore {
  applyAvailableCommands(commands) {}
  getByOptionId(optionId) {}
  toButtonViewModels(requestState) {}
  clear() {}
}
```

规则：

```text
commands 只来自 bootstrap/command/refresh 响应。
点击按钮前必须确认 option_id 仍在当前 commands 中。
如果 option_id 不存在，必须 refresh，不得提交旧 option。
```

### 9.5 ClientRuntimeController

中文职责：页面动作调度器。

建议接口：

```js
class ClientRuntimeController {
  async start() {}
  async bootstrap() {}
  async submitOption(optionId) {}
  async refresh() {}
  async reset() {}
  render() {}
}
```

职责：

```text
1. 页面加载时 start。
2. start 调 bootstrap。
3. bootstrap 成功后渲染 projection 和 commands。
4. submitOption 只提交 option_id。
5. command 后处理 patch 或 refresh。
6. 所有状态变化后 render。
```

禁止：

```text
在 controller 里写 if optionId includes 'gather' then add item。
在 controller 里写 if entity is wolf then damage player。
在 controller 里写 if command_kind craft then create result。
```

## 10. 请求流程设计

### 10.1 启动流程

```text
页面加载
  -> ClientRuntimeController.start
  -> ClientSessionStore.load
  -> ensureClientId
  -> ClientApi.bootstrap
  -> ClientSessionStore.applyBootstrapResponse
  -> ClientProjectionStore.applyFullProjection
  -> ClientCommandStore.applyAvailableCommands
  -> render
```

失败处理：

```text
网络失败：显示“无法连接后端，请确认 1999 服务已启动”。
JSON 解析失败：显示“后端响应格式异常”。
API 错误：显示后端 error_key 和中文解释。
不得自动切 mock。
```

### 10.2 命令提交流程

```text
玩家点击按钮
  -> UI 取 optionId
  -> ClientCommandStore.getByOptionId(optionId)
  -> 如果不存在：refresh
  -> 构造 ClientCommandRequest
  -> ClientApi.submitCommand
  -> 如果 command blocked：显示中文原因，处理 requires_full_refresh
  -> 如果 succeeded：处理 projection_patch / available_commands / event_feed
  -> render
```

请求中必须携带：

```text
session_id
client_id
client_sequence
known_projection_version
submit_mode = option_id
option_id
selection_context
```

P55 的 selection_context 可以先最小化：

```json
{
  "target": { "target_kind": "none" },
  "selected_actor_key": "",
  "selected_entity_id": "",
  "selected_inventory_id": "",
  "selected_area_id": ""
}
```

如果后端命令需要 target，P55 只允许从后端 projection 可见对象中选择，不允许前端凭空构造隐藏目标。

### 10.3 Patch / Refresh 流程

```text
收到 command response
  -> 如果 requires_full_refresh=true：ClientApi.refresh
  -> 否则尝试 applyPatch
  -> applyPatch 成功：更新 projection_version
  -> applyPatch 失败：ClientApi.refresh
  -> 更新 available_commands
  -> render
```

保守策略允许：

```text
P55 初版命令后 always refresh。
```

但要求：

```text
1. 仍要读取 requires_full_refresh。
2. 不能因此前端自增 version。
3. 不能因此绕过 command response。
4. 后续必须能替换成 patch merge，不改 API 层。
```

### 10.4 Reset 流程

```text
玩家点击重开
  -> 二次确认
  -> ClientApi.reset
  -> ClientSessionStore.applyResetResponse
  -> ClientProjectionStore.applyFullProjection
  -> ClientCommandStore.applyAvailableCommands
  -> render
```

Reset 禁止：

```text
只清 localStorage 不通知后端。
前端自行清空世界后显示新世界。
```

## 11. 前端展示要求

P55 不是正式 UI，但必须能看懂真实协议状态。

最小页面区域：

```text
1. 连接状态区：真实后端 / 请求中 / 失败。
2. 世界摘要区：projection 的基本信息。
3. 可见对象区：visible entities / cells / inventories 的原始安全摘要。
4. 可用行动区：available_commands 生成的按钮。
5. 事件日志区：后端 event_feed 和命令结果。
6. 调试信息区：projection_version、session_id、client_sequence，可折叠。
```

中文要求：

```text
玩家可见状态必须中文。
原始 enum/key 可以放在调试区，但不能作为主要 UI 文案。
```

P55 不要求好看，但要求：

```text
1. 玩家知道现在是否连上真实后端。
2. 玩家知道当前有哪些后端允许的行动。
3. 玩家点击后能看到后端返回的变化。
4. 失败时知道为什么失败，以及能否重试。
```

## 12. 架构扫描要求

P55 必须增加前端架构扫描，防止重新写死玩法。

建议扫描范围：

```text
app/frontend/client/
```

建议扫描项：

```text
1. 禁止明显玩法结果硬编码：restore_hunger、poison、craft_result、spawn_wolf 等。
2. 禁止读取 content/core。
3. 禁止 fetch content/core JSON。
4. 禁止在前端源码出现 KnowledgeClaim 构造。
5. 禁止在前端源码出现直接 inventory add/remove 玩法函数。
6. 禁止默认 mock_dev 验收路径。
7. 禁止手写 /api 之外的后端内部路径。
```

注意：扫描不能太粗暴。

例如前端可以显示后端返回的字符串，所以不能简单禁止所有 `torch`、`wolf`、`berry` 字样。真正要禁的是：

```text
前端根据这些 key 写规则结果。
```

建议扫描更多针对函数名和危险模式：

```text
if .*red_berry
if .*wolf
if .*torch
addItem(
removeItem(
applyDamage(
restoreHunger(
createKnowledge(
forceLearn(
fetch\(['"]content/core
```

如果旧前端已有 mock 数据，P55 可以先不删除，但必须：

```text
1. mock 文件与真实模式隔离。
2. 默认不进入 mock。
3. 架构扫描允许 mock 文件但验收命令必须跑真实模式测试。
```

## 14. 验收标准

P55 必须满足：

```text
1. 客户端默认尝试连接真实后端。
2. bootstrap 成功后进入 real_backend 模式。
3. 页面明确显示当前是真实后端模式。
4. 前端保存 client_id/session_id/projection_version。
5. projection_version 只来自后端响应，不由前端自增。
6. available_commands 只来自后端响应。
7. 行动按钮只由 available_commands 生成。
8. 点击行动只提交 option_id，不自己拼玩法规则。
9. command 响应后能更新 projection 或触发 refresh。
10. requires_full_refresh=true 时能自动 refresh。
11. reset 通过 /api/client/reset 调后端，而不是只清本地。
12. 网络失败时显示中文错误，不自动进入 mock。
13. mock_dev 不是默认验收路径。
14. 前端源码不读取 content/core。
15. 前端源码不硬编码玩法结果。
16. P54 client_http 测试仍通过。
17. P53 client_protocol 测试仍通过。
18. P43-P52 关键世界回归仍通过。
19. P55 必须复用 app/frontend/client 作为唯一客户端入口，不允许恢复旧 H5 或另起一套前端替代现有页面。
```

## 15. 研发实现顺序

推荐顺序：

```text
1. 先阅读并保留 app/frontend/client 当前 index.html / app.js / style.css，把它作为唯一真实协议接入口。
2. 新增 ClientApi，封装四个 HTTP endpoint。
3. 新增 ClientSessionStore，管理 client_id/session_id/version。
4. 新增 ClientCommandStore，保存后端 available_commands。
5. 新增 ClientProjectionStore，保存 full_projection，先保守处理 patch。
6. 新增 ClientRuntimeController，串起 bootstrap/command/refresh/reset。
7. 改页面启动流程，默认 start -> bootstrap。
8. 改按钮渲染，只显示后端 available_commands。
9. 改点击流程，只提交 option_id。
10. 改 reset 流程，真实调用后端 reset。
11. 增加中文错误提示。
12. 增加真实后端模式标识。
13. 增加前端架构扫描或最小 Node 检查脚本。
14. 手动打开浏览器验证真实链路。
15. 跑 P54/P53/P43-P52 回归。
16. 写 P55 自检报告。
```

## 16. 研发注意事项

```text
1. 不要为了让页面显示丰富而写假世界。
2. 不要把 mock 数据复制到真实模式。
3. 不要在前端写物品效果、合成结果、知识形成。
4. 不要绕过 available_commands 直接拼 WorldCommandDto。
5. 不要让前端自增 projection_version。
6. 不要在 command 失败时本地强行修改 projection。
7. 不要吞掉后端错误。
8. 不要默认进入 mock。
9. 不要改 P54 HTTP Gateway 路径语义。
10. 不要改 P53 DTO 字段名来迁就前端。
11. 如果字段不匹配，先写适配层，不直接改引擎。
12. 如果发现后端 projection 信息不够，记录为 P56/P57 需求，不在前端补真相。
```

## 17. 风险与处理

### 17.1 后端 projection 当前信息少

风险：

```text
页面看起来很空，研发可能想写假数据填充。
```

处理：

```text
P55 接受页面简陋，但必须真实。
缺少展示字段，记录为后续 projection 增强。
```

### 17.2 available_commands 不够游戏化

风险：

```text
按钮文字像调试信息，不像玩家操作。
```

处理：

```text
P55 只做协议按钮。
P56 再做中文按钮文案和交互布局。
```

### 17.3 Patch merge 复杂

风险：

```text
前端 patch merge 写错导致状态脏。
```

处理：

```text
P55 可先命令后 refresh。
只要接口层预留 applyPatch，后续可无痛优化。
```

### 17.4 Mock 掩盖真实失败

风险：

```text
后端挂了，页面自动用 mock，误以为验收通过。
```

处理：

```text
P55 验收必须显示 real_backend。
mock_dev 必须明显警告。
```

### 17.5 前端继续写死旧 H5 玩法

风险：

```text
旧代码里可能已有 eat berry / use torch 等临时逻辑。
```

处理：

```text
真实模式代码必须隔离旧逻辑。
能删就删；不能删就保证真实模式不引用。
增加架构扫描或人工审查。
```

## 18. 与 P56 的关系

P55 是协议接入层，P56 是正式客户端体验层。

P55 交付给 P56 的能力：

```text
1. 真实后端连接状态。
2. 稳定 session。
3. 当前 projection view model。
4. 当前 command button view model。
5. 中文错误 view model。
6. 统一 command 提交流程。
7. reset/refresh 能力。
```

P56 不应该再关心：

```text
fetch 怎么写。
session 怎么保存。
option_id 怎么提交。
requires_full_refresh 怎么处理。
后端错误怎么转换。
```

P56 只应该关心：

```text
怎么把 P55 的 view model 做成好玩的横屏游戏界面。
```

## 20. 最终说明

P55 的成功标准不是“页面像正式游戏”。

P55 的成功标准是：

```text
前端像一个干净的客户端，后端像唯一的游戏规则服务器。
```

只要这条边界建立起来，后续做横屏 UI、地图交互、中文剧情、NPC 行动反馈，都会更稳。
