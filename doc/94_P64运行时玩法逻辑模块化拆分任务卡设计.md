# P64 运行时玩法逻辑模块化拆分任务卡设计

## 1. 阶段定位

P64 是运行时玩法逻辑的架构拆分阶段，不是新增玩法阶段，也不是前端表现阶段。

它承接 P53-P63 已经形成的能力：

```text
Client Protocol
-> WorldCommandPipeline
-> WorldRuntime / InventoryRuntime / Harvest / Reaction / Teaching / Knowledge
-> Projection / Event Feed
```

P64 要解决的问题是：随着正式客户端、NPC 代工、跟随、野兽生态、生命状态、自动 tick 逐步接入，`client_runtime_host/client_runtime_host_factory.cpp` 开始承担过多职责。它原本只应该创建服务和网关，却逐渐混入了：

```text
NPC 跟随状态
NPC 代工任务状态机
野兽生态 tick 桥接
知识查看 handler
Use 命令分发规则
默认场景实体投放
一批 runtime-aware command handler 注册
```

如果继续这样写，后续新增内容时低上下文研发 AI 很容易把新规则继续塞进 runtime host factory，最终导致：

```text
传输层或宿主层污染玩法规则
前端/协议/玩法边界不清
新增一个生物或任务要改中心大文件
删除某个内容包会影响核心编译
世界规则无法按模块替换
```

P64 的目标是把这些运行时玩法逻辑从 runtime host factory 中拆出去，形成默认世界模块注册入口，让 runtime host factory 只负责依赖装配。

## 2. 为什么现在做

现在必须做，不能继续推迟，原因如下：

1. P63 已经把野生 Agent、生命状态、NPC 代工、火把反制等可玩闭环接进正式客户端。
2. 这些逻辑已经开始跨越多个系统：Command、Runtime、Inventory、Reaction、Knowledge、Projection、Client Gateway。
3. 用户后续明确希望“新加一个东西就是加新文件/模块”，而不是继续改一个超大 factory。
4. 如果不先拆职责，后续做世界规则、区域事件、NPC 职业、任务系统、生态系统时都会复用错误结构。

P64 不追求把所有历史系统重构到最完美，而是先切断最危险的增长点：

```text
client_runtime_host/client_runtime_host_factory.cpp 不再承载具体玩法规则。
```

## 3. 阶段目标

P64 完成后必须具备以下能力：

- runtime host factory 只创建 runtime、service、gateway、adapter，并调用默认世界模块注册入口。
- NPC 跟随逻辑独立在 `world_follow` 模块中。
- NPC 代工/采集/制作/交付任务独立在 `world_npc_work` 模块中。
- 野兽生态运行时桥接独立在 `world_beast_ecology` 模块中。
- 默认 Use 命令用 router 分发给多个模块，避免多个 handler 互相覆盖。
- 默认场景实体投放移出 runtime host factory，放入默认世界 bootstrap 模块。
- 教学、知识查看、采集、制作、移动、攻击等 handler 的注册统一通过 `registerDefaultWorldModules()` 完成。
- post-command hook 由注册表统一管理，不能在 factory 中手写每个玩法 tick。
- 拆分后关键回归必须通过：NPC 做斧头、NPC 做火把反制威胁、狼不可拾取、野兽追击攻击。

## 4. 本阶段不解决的问题

P64 不解决以下问题：

- 不设计新的 UI。
- 不新增新的 NPC 职业。
- 不新增新的物品或反应。
- 不实现完整内容包自动加载。
- 不把所有历史模块都改成插件系统。
- 不重构 WorldCommandPipeline 本身。
- 不重做地图生成、背包、知识、反应的权威逻辑。
- 不把默认场景的 companion / camp_fire / beast_shadow 彻底配置化；本阶段只把它从 runtime host factory 移到可替换 bootstrap 模块。

这些问题可以放到后续内容包、世界规则注册、AI 任务系统阶段继续推进。

## 5. 头脑风暴：后续场景影响

### 5.1 新增物品

例如新增：

```text
毒蘑菇、普通蘑菇、铁矿、燧石、药草、魔法水晶、机械零件
```

P64 后不应该改 runtime host factory。

正确路径：

```text
ContentRegistry 定义 object_key
-> Harvest / Reaction / Runtime 模块引用 object_key
-> Command handler 产生经验
-> Knowledge Learning 形成知识
-> Agent 后续通过知识使用
```

如果物品只需要现有采集/制作/拾取/使用能力，不需要新增 C++ 模块。

只有当物品引入新通用行为类型时，才新增模块，例如：

```text
world_equipment
world_farming
world_magic
world_machine
```

### 5.2 新增行为

例如新增：

```text
钓鱼、耕种、烹饪、熔炼、驯服、治疗、交易、修理、布置陷阱
```

P64 后新增行为应进入独立模块：

```text
backend/include/pathfinder/world_xxx/
backend/src/world_xxx/
```

模块提供：

```text
createXxxCommandHandler()
createXxxUseCommandHandler() 可选
runXxxTicks() 可选
registerXxxModule(context) 可选
```

然后由默认模块入口注册。

禁止做法：

```text
在 client_runtime_host_factory.cpp 里加 if object_key == "xxx"
在前端根据名字判断按钮
绕过 WorldCommandPipeline 直接改 runtime
```

### 5.3 新增 NPC 类型

例如新增：

```text
猎人、农夫、矿工、商人、医生、驾校教练、敌对盗贼、野生动物
```

P64 后 NPC 不应该由 runtime host factory 判断身份。

正确方向：

```text
ContentRegistry / AgentTemplate 定义 actor tags、能力、初始状态
world_agent_runtime 或专属 world_xxx 模块注册通用行为
Knowledge / Task / Command 决定 NPC 能做什么
Projection 只展示结果
```

如果是“职业差异”，优先是数据和知识差异；如果是“新行动体系”，再加模块。

### 5.4 区域事件

例如：

```text
夜晚狼群靠近
毒雾扩散
区域魔法阵持续掉血
暴雨熄灭火堆
机械危机刷新巡逻机器人
```

P64 后区域事件不应该写进 factory 的 tick。

正确方向：

```text
world_region_event 模块
-> post-command hook 或 world tick hook
-> 读取 Runtime / Content / Knowledge
-> 通过 CommandPipeline 或权威 service 产生状态变化
-> 输出 event_feed / projection_patch
```

### 5.5 高级文明

例如：

```text
NPC 建村庄、生产工具、分配职业、生育、建房御寒、储备食物
```

P64 的影响：

- 不直接解决文明 AI。
- 但它避免文明逻辑堆入 runtime host factory。
- 后续应拆成 `world_settlement`、`world_job`、`world_goal_execution` 等模块。

### 5.6 魔法世界

例如：

```text
火球术、护盾、召唤、传送门、区域结界、魔力消耗
```

P64 的影响：

- 魔法可以作为 `world_magic` 模块。
- spell 仍走 Command。
- 魔法造成的状态变化由 runtime/service 权威执行。
- 前端只表现投影，不计算伤害或效果。

### 5.7 机械危机

例如：

```text
机器人巡逻、自动炮塔、电力网络、维修、黑客入侵
```

P64 的影响：

- 机械行为可放 `world_machine_ecology`。
- 电力网络可放 `world_power_grid`。
- 机器人仍是 Agent/Actor，不是前端脚本。
- 维修/制造仍走 Reaction/Command/Knowledge。

### 5.8 任务与代工

例如：

```text
命令 NPC 给我做斧头
命令 NPC 去采药
命令 NPC 建房子
命令 NPC 护送玩家
```

P64 当前把原型代工拆入 `world_npc_work`，但后续更完整的目标执行系统应继续扩展，不应该让 `world_npc_work` 无限膨胀。

边界建议：

```text
world_npc_work：短期代工任务原型
world_goal_execution：长期链式目标执行
world_job：群体职业与生产分配
world_follow：跟随关系
```

### 5.9 多客户端或服务器部署

P64 的影响：

- HTTP gateway 仍只是可选传输网关，不包含玩法规则；本地 Axmol 也通过同一个 runtime host 使用协议层。
- 以后换成本地 WASM、Godot、Axmol、远程服务器，都复用 WorldCommandPipeline。
- 客户端不需要知道 NPC 工作、狼生态、知识学习如何实现。

### 5.10 删除或替换内容包

P64 的影响：

- factory 不再引用具体内容 object_key。
- 默认 bootstrap 仍引用默认内容，但它是可替换模块，不是 运行时宿主核心。
- 后续做内容包 loader 后，可以把默认 bootstrap 改成读取 scenario config。

## 6. 核心设计原则

### 6.1 Factory 只装配，不写规则

`client_runtime_host_factory.cpp` 只能：

```text
创建 runtime/service/adapter/gateway
连接依赖
调用 registerDefaultWorldModules(context)
调用 resetWorld()
```

不能：

```text
判断 NPC 行为
判断狼行为
判断物品使用效果
手写某个 object_key 的规则
直接注册一堆玩法 handler
```

### 6.2 模块拥有自己的状态和 tick

如果一个系统需要运行时状态，例如：

```text
following actors
active npc work tasks
wildlife tick bridge
```

这些状态必须放在对应模块中，不能作为 factory 静态变量或匿名 namespace 大块逻辑。

### 6.3 Use 命令必须可组合

多个模块都可能响应 `WorldCommandKind::Use`：

```text
跟随
命令 NPC 干活
使用工具
使用物品
交互设施
```

不能每个模块都直接注册一个互相覆盖的 Use handler。

P64 引入 `WorldUseCommandRouter`，一个 Use handler 负责按模块顺序分发。

### 6.4 Hook 生命周期必须安全

post-command hook 不能捕获临时 `WorldModuleContext&`。

正确做法：

```text
注册时捕获长期存在的 runtime/service 指针或引用包装
```

否则会出现测试随机崩溃、bad_alloc、segfault。

### 6.5 所有状态变化仍走权威模块

拆分不等于允许模块乱改 runtime。

模块内部仍必须遵守：

```text
可玩行为走 WorldCommandPipeline 或对应 Command handler
物品归属走 P45 Inventory/Ownership
采集走 ResourceHarvestService
制作走 WorldReactionService
知识走 KnowledgeRepository / WorldKnowledgeLearningService
野兽移动/攻击走 Move / Attack command
```

### 6.6 内容专属逻辑不能回流核心

默认 bootstrap 中仍存在默认场景 key，这是可接受的阶段性种子逻辑。

但新增正式玩法不能把 `object_key == xxx` 写进 runtime host factory 或核心 runtime。

## 7. 模块划分

| 模块 | 中文含义 | 职责 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|---|
| `world_modules` | 默认世界模块入口 | 注册默认 command handlers、use router、post hooks、通用工具 | `WorldModuleContext` | handler/hook 注册 | 写具体复杂玩法状态机 |
| `world_follow` | 跟随模块 | 管理 NPC 跟随关系与 tick 推进 | runtime、actor key、response | move command / events | 写代工、战斗、采集 |
| `world_npc_work` | NPC 代工模块 | 接收 NPC 工作命令，多 tick 采集/制作/交付 | runtime、inventory、harvest、reaction、knowledge | events、inventory/runtime 变化 | 写所有未来职业系统 |
| `world_beast_ecology` | 野兽生态模块 | 运行野生 Agent 感知、决策、命令编译与 tick | runtime、content、knowledge、pipeline | move/attack events | 前端判断狼攻击 |
| `default_playable_world_bootstrap` | 默认场景种子 | 初始化默认 companion、campfire、beast 等原型实体 | runtime、content、config | runtime entities | 写宿主或传输层规则 |
| `world_use_command_router` | Use 命令路由 | 聚合多个 Use 子 handler | Use command | 第一个可处理结果 | handler 互相覆盖 |

## 8. 接入流程

### 8.1 服务装配

```text
ClientRuntimeHostFactory
-> 创建 WorldGridRuntime
-> 创建 Inventory / Harvest / Reaction / Knowledge
-> 创建 WorldGeneration / Ensure / Lifecycle
-> 创建 Command Pipeline
-> 构造 WorldModuleContext
-> registerDefaultWorldModules(context)
```

### 8.2 命令执行

```text
ClientCommandGateway
-> WorldCommandPipeline
-> WorldCommandHandlerRegistry
-> 模块注册的 handler
-> Runtime/Service 权威修改
-> Experience/Knowledge Learning
-> post_command_hooks.runHooks()
-> Projection/Patch/Event 返回客户端
```

### 8.3 自动 tick

```text
Wait command 成功
-> post_command_hooks
-> world_npc_work tick
-> world_follow tick
-> world_beast_ecology tick
```

后续如果有区域事件、天气、农作物成长，也应该注册 hook，而不是改 factory。

## 9. 新增模块模板

后续新增 `world_xxx` 推荐结构：

```text
backend/include/pathfinder/world_xxx/world_xxx_module.h
backend/src/world_xxx/world_xxx_module.cpp
```

最小接口：

```cpp
void registerWorldXxxModule(pathfinder::world_modules::WorldModuleContext& context);
```

如果只需要 Use 子行为：

```cpp
std::shared_ptr<pathfinder::world_modules::IWorldUseCommandHandler> createXxxUseCommandHandler(...);
```

如果需要 tick：

```cpp
void runXxxTicks(...);
void clearXxxRuntimeState();
```

默认入口只新增一行：

```cpp
registerWorldXxxModule(context);
```

## 10. 测试设计

P64 至少验证：

1. 编译通过，证明模块依赖没有破坏。
2. NPC 代工做斧头测试通过。
3. NPC 学会火把后能处理威胁测试通过。
4. 野兽不可拾取测试通过。
5. 野兽会追击/攻击测试通过。
6. runtime host factory 不再包含 NPC/野兽/代工/火把/采集/制作/教学 handler 的具体注册逻辑。
7. post-command hook 不捕获临时 context 引用。
8. 默认场景 object_key 不留在 runtime host factory。

## 11. 风险与后续处理

### 11.1 world_npc_work 可能继续膨胀

当前 `world_npc_work` 承担代工原型，后续如果加入建房、职业、生产链，它可能再次变大。

后续处理：

```text
把长期链式目标迁移到 world_goal_execution / world_job。
world_npc_work 保留短期命令代工。
```

### 11.2 默认 bootstrap 仍是代码种子

它比写在 runtime host factory 中好，但还不是最终内容包方案。

后续处理：

```text
Content package loader / scenario loader
-> 读取初始 actors/entities/regions
-> bootstrap 模块只负责调用 loader
```

### 11.3 模块注册入口可能继续变大

`default_world_modules.cpp` 未来如果注册越来越多，也会变成列表大文件。

后续处理：

```text
每个模块提供 registerXxxModule(context)
默认入口只保留模块顺序，不写内部细节
```

### 11.4 Hook 顺序需要治理

当前 hook 顺序是 NPC work、follow、beast ecology。

后续如果增加天气、火势、区域魔法、疾病，需要明确 tick phase：

```text
pre_actor
actor_task
follow
ecology
area_effect
cleanup
```

这可作为后续 P65/P66 的调度阶段。

## 12. 验收标准

P64 验收必须满足：

- `client_runtime_host_factory.cpp` 中不再出现 NPC 代工、跟随、野兽生态具体实现。
- `client_runtime_host_factory.cpp` 中不再直接注册采集、制作、教学、Use、野兽等玩法 handler。
- `client_runtime_host_factory.cpp` 中不再硬编码 `companion`、`beast_shadow`、`camp_fire`。
- 新模块编译进 `pathfinder_world_modules`。
- 关键回归通过。
- 自检报告写入 `doc/review/`。


## 13. 命名迁移说明

P64 后，运行时装配入口正式命名为：

```text
backend/include/pathfinder/client_runtime_host/client_runtime_host_factory.h
backend/src/client_runtime_host/client_runtime_host_factory.cpp
pathfinder::client_runtime_host::ClientRuntimeHostFactory
pathfinder_client_runtime_host
```

旧阶段文档中出现的 `ClientServerRuntimeFactory` 或 `client_http/client_server_runtime_factory.*` 是历史名称，后续研发不得继续使用旧路径。

`client_http/` 目录和 `pathfinder_client_http` target 只保留 HTTP 传输、路由、静态文件和 gateway 相关代码；本地 Axmol、未来 WASM、未来远程 HTTP 都应通过 `client_runtime_host` 装配同一套协议、Command、Runtime、Knowledge 和 world modules。

## 14. 目录收敛补充

P64 不允许把每个小玩法模块都铺成 `pathfinder/` 下的新顶层目录。

运行时玩法模块统一收敛到：

```text
backend/include/pathfinder/world_modules/
backend/src/world_modules/
```

当前分组：

```text
world_modules/core/        默认模块注册、hook registry、Use router、共享工具
world_modules/bootstrap/   默认 playable 场景种子
world_modules/knowledge/   运行时知识查看等通用模块
world_modules/follow/      跟随关系与跟随 tick
world_modules/npc_work/    NPC 短期代工任务
```

保留 `world_beast_ecology/` 作为已经存在的领域系统目录；P64 只把它的 runtime bridge 注册进 `world_modules` target，不再新增 `world_follow/`、`world_npc_work/` 这类顶层目录。

后续新增小型运行时玩法模块时，优先放入 `world_modules/<feature>/`；只有成长为独立领域系统、拥有大量类型/服务/测试时，才升级为新的顶层 `world_xxx/` 目录。
