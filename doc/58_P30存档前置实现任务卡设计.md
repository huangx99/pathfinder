# 58_P30 存档前置实现任务卡设计

## 1. 阶段目标

P30 的目标不是做完整文件存档 UI，也不是做云存档，而是把后端规则引擎产生的权威结果整理成一套稳定的 `SaveGamePackage` 逻辑结构。

它要解决的问题：

```text
规则结果如何被保存。
状态快照、事件、命令复盘、随机记录、配置版本如何对齐。
如何知道一个存档包是否完整、是否被篡改、是否能恢复。
后续 H5 最小可玩版本如何保存玩家探索进度。
后续强化学习和回放如何拿到可验证的历史数据。
```

P30 完成后，系统应具备：

```text
能把 GameState 快照、EventLog、CommandReplayLog、RandomReplayLog、PipelineTrace、ConfigVersionManifest 打包。
能生成 SaveIndex 和 IntegrityManifest。
能校验存档包的一致性。
能生成 RestorePlan，说明恢复时该读什么、按什么顺序读。
能检测 StateVersion 不一致、缺失快照、缺失配置版本、hash 被篡改等问题。
```

## 2. 本阶段不做什么

P30 明确不做：

```text
不做真实文件读写。
不做数据库。
不做云存档。
不做前端存档 UI。
不做压缩算法。
不做加密反作弊。
不做跨版本迁移工具。
不从事件日志反推权威 GameState。
不允许存档系统绕过规则管线修改状态。
```

P30 只做“存档包的数据契约、构建、校验、恢复计划”。

## 3. 与现有系统关系

P30 必须复用已有系统：

| 系统 | P30 使用方式 |
|---|---|
| P0 foundation | ID、Tick、StateVersion、HashValue、Result |
| P1 command | 保存 CommandEnvelope |
| P2 state/event/pipeline | 保存 GameState 快照、EventRecord、Pipeline 状态 |
| P4 replay | 保存 CommandReplayLog、RandomReplayLog |
| P27 condition | trace 中保存 expression_id / canonical_condition_key，不保存隐藏条件真相 |
| P28/P28.5 reaction | 保存其产生的事件、命令、trace 引用，不重新解释反应结果 |
| P29 danger | 保存危险系统产生的事件、trace、知识草稿引用，不直接重算危险 |

关键原则：

```text
StateSnapshot 负责恢复。
EventLog 负责解释。
CommandReplayLog 负责重放。
RandomReplayLog 负责确定性。
ConfigVersionManifest 负责绑定配置。
IntegrityManifest 负责发现损坏和篡改。
RestorePlan 只说明恢复顺序，不直接修改运行时。
```

## 4. 核心类与职责

### 4.1 SaveHeader

存档头。

字段：

```text
save_id：存档包 ID。
save_format_version：存档格式版本，第一版固定 save_package.v1。
game_build_version：游戏构建版本。
game_id：游戏 ID。
player_id：玩家 ID。
config_version：主配置版本。
created_tick：创建 tick。
base_state_version：起点 StateVersion。
latest_state_version：包内最新 StateVersion。
package_hash：整个包的 hash。
```

规则：

```text
save_id、game_id、player_id、config_version 必须合法。
latest_state_version 不能小于 base_state_version。
package_hash 由 builder 计算，不能手填后跳过校验。
```

### 4.2 SaveIndex / SaveIndexEntry

存档索引。

作用：

```text
快速知道存档包里有哪些段。
每段有多少记录。
每段覆盖哪个 StateVersion 范围。
每段对应哪个 section_hash。
```

段类型：

```text
header。
state_snapshot。
event_log。
command_log。
random_log。
pipeline_trace。
config_manifest。
integrity_manifest。
crash_recovery_journal。
```

### 4.3 StateSnapshotSection

状态快照段。

作用：

```text
保存一个或多个 GameState 快照。
恢复时优先加载目标版本之前最近的快照。
```

规则：

```text
至少一个 snapshot。
snapshot_id 必须合法。
snapshot.snapshot.validateBasic() 必须通过。
snapshot_hash 不能为空。
快照 StateVersion 必须有序且不可重复。
```

### 4.4 EventLogSection

事件日志段。

作用：

```text
保存已经提交的 EventRecord。
用于解释为什么状态发生变化。
```

规则：

```text
EventRecord 必须 validateBasic 通过。
事件 StateVersion 不能小于存档 base_state_version。
事件不能替代 StateSnapshot 成为权威恢复来源。
```

### 4.5 CommandLogSection

命令复盘段。

作用：

```text
保存 CommandReplayLog。
用于重放玩家或 Agent 行为。
```

规则：

```text
必须复用 P4 CommandReplayRecord。
command_id 必须与 CommandEnvelope.command_id 一致。
state_version_before / after 必须与包版本范围兼容。
```

### 4.6 RandomLogSection

随机复盘段。

作用：

```text
保存 RandomReplayLog。
保证回放确定性。
```

规则：

```text
每条随机记录必须 validateBasic 通过。
与 CommandReplayLog 中的 command_id 能对齐。
不能在回放时重新抽随机替代记录。
```

### 4.7 PipelineTraceSection

管线 trace 段。

作用：

```text
保存规则解释链索引。
保存 pipeline stage key、event_id、condition_trace_key、reason_key。
```

规则：

```text
允许保存 expression_id 或 canonical_condition_key。
禁止保存 hidden_truth、true_hp、raw_state、death、loot 等隐藏/战斗实现字段。
trace 只解释规则过程，不作为恢复权威状态。
```

### 4.8 ConfigVersionManifest

配置版本清单。

作用：

```text
说明这个存档依赖哪些配置版本。
```

字段：

```text
active_config_version。
required_config_versions。
config_content_hash。
compatibility_key。
```

规则：

```text
active_config_version 必须合法。
required_config_versions 不能重复。
config_content_hash 不能为空。
```

### 4.9 IntegrityManifest

完整性清单。

作用：

```text
保存 header、snapshot、event、command、random、trace、config、journal、package 的 hash。
```

规则：

```text
builder 生成。
validator 重新计算并比较。
sealed=true 时，任意 hash 不匹配都必须失败。
```

### 4.10 CrashRecoveryJournal

崩溃恢复日志。

作用：

```text
记录尚未提交完成的事务。
用于后续恢复工具判断回滚、重试或丢弃。
```

P30 只定义结构与校验，不做真实恢复。

### 4.11 SavePackageBuilder

作用：

```text
接收构建输入。
校验所有段。
生成 SaveIndex。
生成 IntegrityManifest。
生成 package_hash。
输出 SaveGamePackage。
```

它不做：

```text
不写文件。
不压缩。
不加密。
不修改 GameState。
```

### 4.12 SavePackageValidator

作用：

```text
校验 SaveGamePackage 是否可被信任。
校验基础字段。
校验 StateVersion 范围。
校验 index 与 sections 是否一致。
重新计算 IntegrityManifest。
检查 ConfigVersionManifest。
```

### 4.13 SaveRestorePlanner

作用：

```text
根据 SaveGamePackage 和目标 StateVersion 生成 RestorePlan。
```

RestorePlan 只描述：

```text
校验完整性。
加载哪个 snapshot。
应用哪些 event log。
挂载哪些 replay log。
加载 config manifest。
进入 ready 状态。
```

它不直接恢复运行时，也不修改 GameState。

## 5. 验收标准

P30 必须通过以下测试：

```text
1. SaveSectionKind 枚举 roundtrip。
2. forbidden key 被拒绝。
3. SavePackageBuilder 能构建合法包。
4. SavePackageValidator 能通过合法包。
5. 篡改 hash 后 validator 必须失败。
6. 缺失 snapshot 必须失败。
7. StateVersion 范围不一致必须失败。
8. ConfigVersionManifest 缺失或 hash 为空必须失败。
9. RestorePlanner 能选择目标版本之前最近的 snapshot。
10. RestorePlanner 不直接修改 GameState。
11. P4 replay 记录能进入 SaveGamePackage。
12. 用存档包里的 snapshot + command/random log 能继续 replay。
```

## 6. 后续扩展

P30 后续可扩展：

```text
真实文件序列化。
增量存档。
压缩与分片。
加密签名。
跨版本迁移。
自动保存策略。
H5 存档/读档 UI。
云同步。
训练数据批量导出。
```

这些都不需要推翻 P30 的逻辑结构。
