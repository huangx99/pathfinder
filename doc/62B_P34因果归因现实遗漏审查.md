# 62B P34 因果归因系统现实遗漏审查

## 1. 审查结论

进一步结合真实世界后，我认为 62A 的时间窗口设计是正确方向，但还不是完整闭环。

62A 解决了：

```text
同一影响会持续、消退、延迟、叠加。
```

但真实世界里，因果误判还会来自：

```text
个体差异、剂量差异、物品状态、制备方式、混合反应、背景状态、观察误差、概率结果、长期累积、知识适用范围。
```

所以最终规则应修正为：

```text
知识确认 != 重复次数
知识确认 != 单纯时间窗口
知识确认 = 时间窗口 + 剂量 + 对象状态 + 个体体质 + 环境背景 + 替代解释排除 + 观察质量 + 适用范围
```

这不是要把 P34 做到无限复杂，而是要在架构上把这些因素预留出来，避免后面重构。

---

## 2. 遗漏一：个体差异

### 2.1 问题

同一个东西，对不同生物不一定有同样效果。

例子：

```text
人吃某种果子没事。
狼吃可能中毒。
幼体吃可能不适。
体弱者吃可能更容易生病。
某个 NPC 可能过敏。
高级文明生物可能完全免疫。
```

如果知识只写成：

```text
red_berry + eat -> safe
```

以后会误导所有 agent。

### 2.2 设计补充

新增概念：`AgentPhysiologyProfile`。

```cpp
struct AgentPhysiologyProfile {
    std::string species_key;
    std::string life_stage_key;
    std::vector<std::string> trait_keys;
    std::vector<std::string> immunity_keys;
    std::vector<std::string> vulnerability_keys;
};
```

中文解释：

| 字段 | 中文 | 作用 |
|---|---|---|
| species_key | 物种 | 人、狼、蜘蛛、外星人 |
| life_stage_key | 生命阶段 | 幼体、成年、老年 |
| trait_keys | 体质标签 | 强壮、虚弱、耐寒、过敏 |
| immunity_keys | 免疫标签 | 抗毒、抗寒、抗腐败 |
| vulnerability_keys | 弱点标签 | 怕火、怕毒、怕寒 |

### 2.3 知识表达修正

知识不能只说：

```text
红果可以吃。
```

而应表达为：

```text
对当前物种，在正常剂量下，新鲜红果可缓解饥饿。
```

H5 中文可显示：

```text
红果：对先驱者而言，少量食用通常能缓解饥饿。
```

而不是绝对说：

```text
红果：所有生物吃了都安全。
```

## 3. 遗漏二：剂量不等于次数

### 3.1 问题

现实里不是“吃 3 次”有毒，而是摄入量、浓度、大小、有效成分不同。

例子：

```text
吃 3 小口红果 ≠ 吃 3 整颗红果。
吃 1 个大毒蘑菇可能比吃 3 个小毒蘑菇更危险。
腐烂程度高的果子，一口也可能中毒。
```

### 3.2 设计补充

新增：`ExposureDose`。

```cpp
struct ExposureDose {
    double amount = 0.0;
    std::string unit_key;
    double potency = 1.0;
    double absorption_rate = 1.0;
};
```

中文解释：

| 字段 | 中文 | 例子 |
|---|---|---|
| amount | 数量 | 半个、一个、三口 |
| unit_key | 单位 | bite、piece、gram |
| potency | 效力 | 腐烂程度、毒性浓度 |
| absorption_rate | 吸收率 | 吃下去比触摸吸收高 |

### 3.3 规则结论

P34 不应该写死：

```text
三次红果确认。
```

而应该写成：

```text
在一段时间内，某类暴露的有效剂量达到阈值。
```

---

## 4. 遗漏三：接触方式不同

### 4.1 问题

同一个物品，不同行为导致不同风险。

例子：

```text
吃毒蘑菇会中毒。
摸毒蘑菇可能没事。
闻毒气会中毒。
把草药外敷可能止血，吃下去可能没效果。
石片拿来切割有用，吃石片不应该变成“石片吃了可使用”。
```

### 4.2 设计补充

新增：`ExposureRoute`。

```cpp
enum class ExposureRoute {
    Unknown,
    Eat,
    Drink,
    Touch,
    Smell,
    Inhale,
    Inject,
    ApplyToWound,
    UseAsTool,
    Wear,
    TestOnly
};
```

知识 key 必须包含 route/action。

---

## 5. 遗漏四：物品状态历史

### 5.1 问题

物品不是静态的。

例子：

```text
新鲜红果可以吃。
放久后腐烂。
洗过后降低污染。
煮熟后解除毒性。
烧焦后不能吃。
受污染的水即使看起来是水也会致病。
```

### 5.2 设计补充

新增：`ObjectStateSnapshot` 和 `ObjectStateHistory`。

```cpp
struct ObjectStateSnapshot {
    std::string object_id;
    std::string base_kind_key;
    std::vector<std::string> state_keys;
    int created_tick = 0;
    int last_changed_tick = 0;
};
```

知识形成时必须记录观察到的状态：

```text
不是 red_berry，而是 red_berry + fresh。
不是 water，而是 water + contaminated。
```

### 5.3 对当前红果问题的影响

`红果` 和 `腐烂红果` 不应该共享同一个主体 key。

正确方式：

```text
base_kind = berry
variant/state = fresh / rotten
```

知识投影显示：

```text
新鲜红果：少量食用可缓解饥饿。
腐烂红果：食用后可能导致不适。
```

而不是都显示成：

```text
红果。
```

---

## 6. 遗漏五：制备方式和加工过程

### 6.1 问题

同一个原料，加工后性质可能完全不同。

例子：

```text
生肉会生病，烤肉可以吃。
苦叶直接吃没用，煮成药汤有用。
毒果晒干后毒性降低。
木棍 + 火 = 火把，火把不是木棍的简单状态，而是新物品。
```

### 6.2 设计补充

对象反应系统生成新物品时，必须带来源记录。

```cpp
struct ProvenanceRecord {
    std::vector<std::string> input_object_ids;
    std::string reaction_rule_id;
    int produced_tick = 0;
};
```

这样系统能知道：

```text
这个药汤来自苦叶 + 水 + 加热。
```

以后 NPC 学会的不是“苦叶有用”，而是：

```text
苦叶煮成汤后可缓解某种状态。
```

---

## 7. 遗漏六：混合反应与相互作用

### 7.1 问题

A 单独安全，B 单独安全，但 A+B 有毒。

例子：

```text
红果安全。
苦叶安全。
红果 + 苦叶一起吃导致腹痛。
草药能治疗轻毒，但和酒一起喝会失效。
火 + 油会燃烧。
水 + 火会灭火。
```

### 7.2 设计补充

新增：`InteractionCandidate`。

```cpp
struct InteractionCandidate {
    std::vector<std::string> factor_keys;
    std::string outcome_key;
    int interaction_window_ticks = 0;
    double support_score = 0.0;
};
```

因果候选不能只有单原因，还要支持组合原因：

```text
原因候选 A：腐烂红果
原因候选 B：短时间吃太多浆果
原因候选 C：红果 + 苦叶组合
原因候选 D：身体虚弱 + 腐烂红果
```

---

## 8. 遗漏七：背景状态会改变结果

### 8.1 问题

同一行为，在不同身体状态下结果不同。

例子：

```text
饥饿时吃红果缓解饥饿。
吃饱时再吃红果可能撑。
受伤时喝药有效，健康时喝药没明显效果。
疲劳时战斗更容易受伤。
寒冷状态下吃某些食物可能更有用。
```

### 8.2 设计补充

Observation 必须记录 actor state。

```cpp
struct ActorStateSnapshot {
    std::vector<std::string> need_state_keys;
    std::vector<std::string> health_state_keys;
    std::vector<std::string> temporary_effect_keys;
};
```

知识要带条件：

```text
当饥饿时，吃新鲜红果可以缓解饥饿。
```

而不是：

```text
吃新鲜红果永远有正面效果。
```

---

## 9. 遗漏八：观察误差与“没效果”误判

### 9.1 问题

“没看到效果”不等于“没有效果”。

例子：

```text
毒性还没发作。
药效很弱，玩家没感觉到。
症状被别的效果抵消了。
角色观察能力太低，看不出细微变化。
```

### 9.2 设计补充

新增：`ObservationQuality`。

```cpp
struct ObservationQuality {
    double clarity = 1.0;
    double sensitivity = 1.0;
    double reliability = 1.0;
    std::vector<std::string> missing_signal_keys;
};
```

知识状态中应区分：

```text
未观察到效果
确认无效果
观察条件不足
```

H5 中文显示：

```text
暂时没有明显效果，但还不能确认它完全无用。
```

---

## 10. 遗漏九：概率结果

### 10.1 问题

不是所有规则都是 100% 触发。

例子：

```text
吃腐烂食物有 60% 概率不适。
被蜘蛛咬有 30% 概率中毒。
草药有 70% 概率缓解轻毒。
潮湿环境下木头有概率发霉。
```

如果系统要求每次都一致，知识会反复冲突。

### 10.2 设计补充

新增：`OutcomeDistribution`。

```cpp
struct OutcomeDistribution {
    std::string outcome_key;
    double observed_frequency = 0.0;
    int sample_count = 0;
    double confidence = 0.0;
};
```

知识中文显示可以是：

```text
腐烂红果：多次观察后，食用有较高概率导致不适。
```

而不是：

```text
腐烂红果：一定中毒。
```

---

## 11. 遗漏十：长期累积和慢性影响

### 11.1 问题

有些影响不是几小时窗口，而是长期累积。

例子：

```text
长期喝污水后生病。
长期吃某种植物导致虚弱。
长期训练提升技能。
长期使用工具提升熟练。
长期接触污染降低健康。
```

### 11.2 设计补充

时间窗口要分层：

```cpp
enum class AccumulationHorizon {
    Immediate,
    ShortTerm,
    LongTerm,
    Permanent,
    TestOnly
};
```

中文：

| 枚举 | 中文 | 例子 |
|---|---|---|
| Immediate | 立即 | 吃下立刻恢复饥饿 |
| ShortTerm | 短期 | 几小时内吃太多 |
| LongTerm | 长期 | 长期污染累积 |
| Permanent | 永久 | 工具磨损、伤疤、技能学习 |

## 12. 遗漏十一：治疗不等于推翻原因

### 12.1 问题

如果吃腐烂红果后中毒，再喝草药好了，不能说“腐烂红果无毒”。

同理：

```text
药治好了症状，只说明药可能有效，不说明原原因不存在。
```

### 12.2 设计补充

ActiveEffect 应区分：

```text
原因来源
当前症状
治疗干预
恢复过程
```

新增：`InterventionRecord`。

```cpp
struct InterventionRecord {
    std::string intervention_key;
    int applied_tick = 0;
    std::vector<std::string> target_effect_keys;
};
```

---

## 13. 遗漏十二：知识适用范围

### 13.1 问题

当前知识系统如果只写：

```text
red_berry + eat -> restore_hunger
```

会丢失大量限定条件。

### 13.2 设计补充

新增：`KnowledgeApplicabilityCondition`。

```cpp
struct KnowledgeApplicabilityCondition {
    std::vector<std::string> required_subject_states;
    std::vector<std::string> excluded_subject_states;
    std::vector<std::string> required_actor_states;
    std::vector<std::string> required_environment_states;
    std::string species_scope_key;
    double max_safe_dose = 0.0;
    int dose_window_ticks = 0;
};
```

中文显示：

```text
新鲜红果：对先驱者，少量食用可缓解饥饿；尚未确认大量食用是否安全。
```

## 14. 遗漏十三：知识来源和传播可信度

### 14.1 问题

自己亲身试过、同伴听说、族群传说、专家教学，可信度不应该一样。

例子：

```text
自己吃过红果。
同伴告诉你红果能吃。
族群里流传毒蘑菇危险。
专家 NPC 教你如何辨别药草。
```

### 14.2 设计补充

新增：`KnowledgeProvenance`。

```cpp
struct KnowledgeProvenance {
    std::string source_agent_id;
    std::string source_type_key;
    int observed_sample_count = 0;
    double trust_score = 0.0;
    bool firsthand = false;
};
```

传播后知识应降级或标注来源：

```text
同伴认为：腐烂红果可能有害。来源：你传授。
```

而不是直接等同自己亲测。

## 15. 遗漏十四：实验安全与伦理

### 15.1 问题

如果系统鼓励玩家/NPC 反复吃毒物确认，会很奇怪。

高级 agent 应该会：

```text
先小剂量测试。
先让低风险对象测试。
先观察动物反应。
知道危险后拒绝继续试吃。
怀疑别人故意投毒。
```

### 15.2 设计补充

新增：`ExperimentRiskGate`。

```cpp
struct ExperimentRiskGate {
    double known_risk_score = 0.0;
    double uncertainty_score = 0.0;
    double max_allowed_risk = 0.0;
    bool requires_consent = false;
};
```

NPC 投喂判断以后可以基于：

```text
投喂者是否知道风险
被投喂者是否知道风险
风险知识是否已确认
是否存在替代解释
是否获得同意
```

## 16. 遗漏十五：反例不是简单抵消

### 16.1 问题

一次没中毒，不代表腐烂红果安全。

例子：

```text
这次吃腐烂红果没事，可能是剂量小。
可能这个个体抗毒。
可能腐烂程度轻。
可能毒性是概率触发。
```

### 16.2 设计补充

反例必须检查上下文是否等价。

```text
同物品状态
同剂量
同接触方式
同物种
同环境
同时间窗口
同观察质量
```

只有上下文等价，反例才强。

## 17. 遗漏十六：记忆压缩不能丢条件

### 17.1 问题

后续记忆系统会压缩知识。

如果压缩成：

```text
红果能吃。
```

就丢掉了：

```text
新鲜、少量、先驱者、饥饿状态、短期安全、非腐烂。
```

### 17.2 设计补充

记忆压缩必须保留：

```text
主体状态
动作方式
剂量范围
物种范围
时间窗口
确认程度
样本数量
例外条件
```

---

## 18. 遗漏十七：测试必须包含“现实误判”场景

P34 测试不能只测红果。

至少要覆盖：

```text
1. 短时间多次吃新鲜红果后，再吃腐烂红果，不能确认腐烂是唯一原因。
2. 等待消化窗口后吃腐烂红果，腐烂解释增强。
3. 少量腐烂红果无事，大量腐烂红果不适，形成剂量知识。
4. 新鲜红果对先驱者安全，对狼不安全，形成物种限定知识。
5. 生苦叶没效果，煮成药汤有效，形成制备知识。
6. 红果和苦叶单独没事，一起吃不适，形成组合知识。
7. 毒蘑菇延迟发作，系统能回溯原因。
8. 草药治疗中毒，不会推翻毒蘑菇导致中毒。
9. 腐烂红果一次没事，不会直接反证腐烂有害。
10. 同伴传授的知识可信度低于亲身验证。
11. 族群知识能影响 NPC 是否拒绝危险投喂。
12. 长期喝污水后生病，形成长期暴露知识。
```

## 19. 对 P34 实现范围的建议

这些缺口不代表 P34 要一次实现完整真实世界。

P34 必须实现的最小架构预留：

```text
1. Observation 带 actor、subject、environment、time、dose、route、state snapshot。
2. Knowledge 带 applicability condition，不再生成无条件绝对知识。
3. Attribution 支持多个候选原因，而不是单原因覆盖。
4. Resolver 支持窗口内替代解释，窗口外权重衰减。
5. H5 显示“可能原因/适用条件/尚未确认”，不要只显示调试 key。
```

P34 可以暂不完整实现，但必须留接口：

```text
1. 个体差异。
2. 概率结果。
3. 长期累积。
4. 制备来源。
5. 混合反应。
6. 观察误差。
7. 实验风险门控。
```

---

## 20. 最终修正后的核心公式

最终因果归因公式应为：

```text
一次观察 = 谁 + 在什么状态下 + 对什么状态的对象 + 用什么方式 + 用了多少 + 在什么环境 + 什么时候 + 过了多久 + 观察到什么 + 观察是否可靠
```

知识形成公式应为：

```text
知识 = 原因候选 + 适用条件 + 时间窗口 + 剂量范围 + 支持证据 + 反例证据 + 替代解释 + 可信度 + 来源
```

玩家可读表达应为：

```text
不是“红果能吃”，而是“新鲜红果在少量食用时，通常能让先驱者缓解饥饿；大量食用是否安全还不确定”。
```

---

## 21. 结论

是的，62A 之后仍然有遗漏。

最重要的遗漏不是某个具体枚举，而是：

```text
知识必须从绝对结论，升级为带条件的结论。
```

只要 P34 把 Observation、Attribution、Knowledge 三层都改成“带条件、带时间、带剂量、带来源”，后续无论加新水果、新药物、新物种、新 NPC、新合成物、新环境，都不需要推翻架构。
