# P39 知识链式推理与目标规划再次审核报告

审核时间：2026-05-19

## 1. 审核结论

结论：P39 再次审核通过，可以验收。

本轮代码已经针对上次复审保留的两个主要架构边界做了实质性修正：

1. `known_effect_keys -> synthetic KnowledgeClaim` 的过渡问题已明显改善。
   - `WorldActorRuntimeState` 新增 `known_claims`。
   - `WorldSnapshotAdapter` 直接把 `state.actor_claims` / `state.recipient_claims` 写入 actor 的 `known_claims`。
   - `AgentAutonomyService` 现在从 actor 的真实 `known_claims` 构造推理输入，不再用 effect_key 固定合成 red_berry、torch、dry_grass 等对象知识。

2. 反应系统接入已从散落在 reasoner 内的最小函数，抽成独立 `ReactionPlanningAdapter`。
   - P39 新增 `reaction_planning_adapter.h/.cpp`。
   - `requiredPreconditionsFor` 优先调用 `ReactionPlanningAdapter::preconditionsForEffect`。
   - P28 core reaction fixture 新增 `cutWoodWithAxeRule` 与 `sharpenAxeRule`。
   - 测试新增 `reaction_adapter_reads_crafting_rules`，覆盖 `make_torch`、`cut_wood`、`restore_sharpness` 三类反应前置。

本轮实现已经比上次更接近“知识 -> 目标推理 -> 反应规则前置 -> 世界执行”的正确架构，不再只是为了 P39 测试临时拼一套逻辑。

## 3. 仍需注意的边界

这些不是 P39 阻塞项，但要带到 P40/P41 后续设计里。

### 3.1 ReactionPlanningAdapter 仍有内容映射，但位置正确

当前 adapter 仍然会识别：

```text
def_fire_source -> camp_fire
fire_source -> camp_fire
def_raw_wood / wood_material -> wood
def_whetstone / sharpening_tool -> whetstone
def_axe / cutting_tool -> axe
sharp condition -> axe.sharpness
```

这说明它还不是“完全任意配置零代码接入”。但这次问题已经从“推理器里散落硬编码”下降为“专门 adapter 的翻译层”。这是可接受的阶段性设计。

后续 P40/P41 应继续把这层升级为：

```text
ObjectDefinitionId / tag -> runtime object key 的通用索引
ReactionOutputTemplate -> producer candidate 的通用生成
ConditionExpressionRef -> PlanPrecondition 的通用转换
数量、消耗、替代材料、工具耐久统一进入材料/行动执行层
```

### 3.2 `build_house` 还未接入建筑反应规则

`build_house` 当前仍走 fallback：

```text
wood_processed >= 3
```

这不阻塞 P39，因为 P39 任务卡允许建房作为测试语义。但后续真正做建筑/庇护容量时，需要新增建筑反应或建筑任务配置，不应继续留在 fallback 中。

### 3.3 执行闭环仍属于 P40

P39 负责计划，不直接改世界。当前 `AgentPlanExecutorAdapter` 真正执行得最完整的是 `repel_beast`；`cut_wood`、`make_torch`、`restore_sharpness`、`build_house` 的完整逐步执行仍应由 P40 动态目标执行管线承接。

这符合阶段边界，不作为 P39 问题。
