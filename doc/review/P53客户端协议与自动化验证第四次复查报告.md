## 4. 新增阻塞问题

### 4.1 WorldCommandDto target 值可伪造

严重性：**阻塞**。

位置：

```text
backend/src/client_protocol/client_command_gateway.cpp:72
backend/src/client_protocol/client_command_gateway.cpp:76
backend/src/client_protocol/client_command_gateway.cpp:78
```

问题代码逻辑：

```cpp
if (opt.command_kind == command.command_kind &&
    opt.command_key == command.command_key &&
    opt.target.target_kind == command.target.target_kind) {
    matched = true;
}
```

问题说明：

```text
这只验证了目标类型，没有验证目标身份。
如果后续 available_commands 签发 “inspect entity A”，客户端可以提交 “inspect entity B”。
如果签发 “attack wolf_1”，客户端可以提交 “attack npc_king”。
如果签发 “use item torch”，客户端可以提交 “use item rare_artifact”。
```

为什么阻塞：

```text
P53 的核心协议是 available_commands 是操作权威来源。
WorldCommandDto 如果允许目标值替换，就仍然是绕过 option authority 的后门。
P54 客户端可能会面对大量 entity / inventory / actor / area 目标，这个问题不能留到 UI 阶段。
```

必须修复：

```text
1. WorldCommandDto 匹配 snapshot option 时必须比较完整 WorldCommandTargetDto。
2. 至少比较：
   target_kind
   target_entity_id
   target_actor_key
   target_item_key
   target_inventory_id
   target_area_id
   knowledge_key
   recipe_key
   target_coord 是否同为空 / 同值，包括 x、y、layer_key
3. 如 command.parameters 未来可由 option 携带，也必须绑定参数 allowlist 或与 option materialization 结果一致。
4. 新增测试：已签发 Entity target=issued_entity，但 WorldCommandDto 提交 forged_entity 必须 Blocked。
5. 新增测试：Coordinate target 的 x/y/layer 任一不一致必须 Blocked。
```
