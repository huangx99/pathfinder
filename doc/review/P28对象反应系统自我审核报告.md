# P28 对象反应系统自我审核报告

## 1. 审核结论

结论：P28 核心对象反应系统已完成，并通过自我审核。

本次不是“测试通过就算审核”。我已人工检查模块边界、条件权威、draft 输出、可扩展性、规则数据驱动能力和安全门禁，确认 P28 没有把对象反应写成硬编码物理/化学模拟，也没有绕过 P27 条件表达式系统。

P28 当前达到的核心能力：

```text
对象之间可以按规则发生反应。
反应条件统一交给 P27 ConditionExpressionEvaluator。
反应结果只输出 draft，不直接修改 WorldObject。
火 + 干木棍 -> 火把 可跑通。
火 + 湿木棍 -> 条件阻塞，可解释失败原因。
水 + 火 -> 熄灭，可输出 RemoveState / ConsumeObject。
规则可扩展：新增规则、条件、输出模板、中文反馈，不需要改 resolver 主流程。
```

### 2.2 P27 条件扩展

为了让对象反应不在 resolver 里手写状态判断，P27 条件表达式补充了 source/target 域：

```text
condition:source_state:eq:<state>
condition:target_state:eq:<state>
condition:source_tag:has:<tag>
condition:target_tag:has:<tag>
```

这些条件由 `ConditionExpressionEvaluator` 统一求值，P28 resolver 不直接判断 `dry/wet/burning`。

## 3. 可扩展性审核

### 3.1 新增一个反应不需要改 resolver

新增例如：

```text
火 + 生肉 -> 熟肉
酸液 + 金属 -> 腐蚀
药草 + 水 -> 草药汤
石片 + 木棍 -> 简易矛
```

应通过新增 `ObjectReactionRule` 完成：

```text
object_patterns：只做 Source / Material / Tool / Container 的粗匹配。
condition_refs：写具体动态条件，例如 source_state、target_state、target_tag。
output_templates：声明 TransformObject / AddState / RemoveState / ConsumeObject / ProduceObject / ResourceDelta。
feedback_text：写中文安全反馈。
knowledge_effect_key：给学习/知识系统消费。
```

resolver 主流程不需要增加 `if 火 + 木棍` 这种分支。

### 3.2 动态状态没有塞进 tag 粗匹配

人工检查结果：

```text
required_tag_keys 只承担 combustible、fire_source、water_source 这类粗匹配。
dry / wet / burning 这类动态状态由 ConditionExpressionRef 判断。
ReactionObjectPattern::validateBasic() 会拒绝 dynamic-state-like tag。
```

这保证后续加“湿木棍”“半干木棍”“燃烧中”“冷却中”等状态时，不会把规则系统变成混乱的 tag 判断。

### 3.3 输出是 draft，不直接改对象

人工检查结果：

```text
ObjectReactionResolver 输出 ReactionObjectChangeDraft / FeedbackDraft / KnowledgeDraft / EventDraft。
没有持有 WorldObject*。
没有调用 ObjectStore / markConsumed。
没有直接修改 WorldObject。
```

这保证 P28 后续可以接入 pipeline、replay、save、H5 投影，而不会产生隐藏副作用。

### 3.4 规则反馈改为数据驱动

自审中发现如果中文反馈全部写在 resolver 的 `feedback_key` 映射里，后续每加一条内容都要改代码，不够可扩展。

已修正：

```text
ObjectReactionRule 新增 feedback_text。
resolver 优先使用 rule.feedback_text。
没有配置 feedback_text 时才走 fallback feedbackText()。
新增 reaction_resolver_data_driven_feedback 测试。
```

这意味着后续内容配置可以直接提供中文反馈，避免把内容写死进 resolver。

### 3.5 ProduceObject 扩展能力已补齐

自审中发现 `ReactionOutputKind::ProduceObject` 原本存在，但 `ReactionObjectRole::Product` 被 change draft 校验拒绝，会导致“反应产生新对象”不可用。

已修正：

```text
ProduceObject 必须使用 Product role。
Product role 只能用于 ProduceObject。
ProduceObject draft 可以没有现有 object_id，但必须有 to_definition_id。
新增 reaction_resolver_produce_object_draft 测试。
```

这让后续“火星生成灰烬”“材料组合生成新工具”“容器倒出液体”等扩展不需要重构。

## 7. 当前限制

这些不是本次阻断问题，但需要后续阶段继续做：

```text
1. P28 当前提供 ReactionKnowledgeDraft，但没有把 P22 H5 对话框命令正式改造成“选择两个对象并触发反应”的 UI 闭环。
2. P28 当前是手写 fixture 规则，后续 P35 内容配置扩展时应把 ObjectReactionRule 从配置加载。
3. P28 当前只做一次性主动反应，不做环境自动反应，例如火蔓延、雨水熄火。
4. P28 当前不做危险压力，火把对夜间危险的影响应进入 P29。
5. P28 当前不做存档迁移，ReactionTrace / ReactionResult 的持久化应进入 P30 或后续协议阶段。
```

## 8. 对 P29 的建议

P29 危险与探索压力应该直接复用 P28：

```text
火把来自 P28 的 TransformObject 或 ProduceObject 输出。
危险降低来自 P29 的 HazardRule / ThreatResolver。
条件仍由 P27 ConditionExpressionEvaluator 判断。
危险后果仍输出 draft / event / trace，不直接写死 HP / death。
H5 只显示安全反馈与投影，不推导隐藏真实危险。
```

P29 不应重新设计对象反应，也不应绕过 P28 结果去写一套“火把是否存在”的硬编码判断。

## 9. 最终结论

P28 可以进入用户复审或提交前检查。

我认为当前 P28 的架构方向是正确的：它把“对象之间能反应”做成了规则驱动、条件统一、结果草稿化、反馈数据化的最小系统。后续扩展内容时主要增加规则和配置，不需要推翻 resolver 主流程。
