# P20 知识传播系统审核报告

审核日期：2026-05-17  
审核范围：P20 knowledge propagation 模块、P20 枚举与 DTO、传播 Planner/Service/Applier、P20 单元/集成测试、阶段边界扫描。  
审核依据：`doc/46_P20知识传播系统任务卡设计.md`、`context_packs/knowledge_p20.md`、`doc/45_P19知识冲突与修正任务卡设计.md`、`context_packs/knowledge_p19.md`。

## 1. 审核结论

**结论：暂不通过，需修复后复审。**

P20 当前实现已经完成了基础文件、枚举、DTO、Planner、Service、Applier 与测试接入；构建、P20 定向测试、knowledge 定向测试和全量回归均通过。但核心传播语义存在阻塞问题：

1. 接收者已有相同知识时，`ReinforcedRecipientClaim` 没有更新接收者已有 claim，而是生成了一个新 claim，Applier 会把它追加成重复知识。
2. 接收者已有同 subject/action/relation 但不同 `effect_key` 的知识时，系统没有识别冲突，反而创建新的接收者 claim，绕过 P19 修正路径。
3. `allow_create_recipient_claim=false` / `allow_update_recipient_claim=false` 选项没有被执行，调用方禁止创建时仍会创建知识。
4. `KnowledgePropagationResult::ok()` 不包含 `Failed`，但 Service 会返回 `Result::ok(Failed)`，与 P20 任务卡“Failed 可以作为 ok DTO”不一致。
5. 当前测试只断言 decision，缺少“更新的是已有 claim id”“冲突不同 effect 不得静默创建”“Applier 不得产生重复”等关键验收。

这些问题会直接影响 P20 最核心的玩家体验：

```text
A 教 B 红果可食用。
如果 B 已经知道红果可食用，应该强化 B 的旧知识。
如果 B 认为红果有毒，应该产生知识冲突/修正，而不是悄悄并排生成两条互相矛盾的行动知识。
```

当前实现还不能稳定支撑 P21 的“传播后他人行为变化”学习闭环。

## 4. 阻塞问题

### BLOCKER-1：ReinforcedRecipientClaim 没有更新接收者已有知识，而是生成重复新知识

位置：

- `backend/src/knowledge/knowledge_propagation.cpp:578`
- `backend/src/knowledge/knowledge_propagation.cpp:599`
- `backend/src/knowledge/knowledge_propagation.cpp:600`
- `backend/src/knowledge/knowledge_propagation.cpp:612`
- `backend/src/knowledge/knowledge_propagation.cpp:620`
- `backend/src/knowledge/knowledge_propagation.cpp:622`
- `backend/src/knowledge/knowledge_propagation.cpp:760`
- `backend/src/knowledge/knowledge_propagation.cpp:842`

问题：

当 target 已有同 subject/action/relation/effect 的 claim 时，Planner 正确给出：

```text
recommended_decision = ReinforcedRecipientClaim
should_update_existing = true
```

但随后生成的 `KnowledgeRecipientClaimDraft` 不是基于 target existing claim，而是重新创建了一个新的 `KnowledgeClaim`：

```text
makeKnowledgeId(attempt.target.owner, claim.subject, claim.predicate, propagation_tick, 0)
recipient_claim.knowledge_id = 新 id
```

Service 又把这个新 claim 放入 `updated_claims`。Applier 根据 `knowledge_id` 替换旧知识，但由于 id 不同，找不到旧 claim，于是追加成第二条 claim。

探针确认：

```text
decision=reinforced_recipient_claim
updated_id=know_agent_b_berry_red_has_effect_100_0
existing_id=know_b_restore_existing
snapshot_size=2
```

影响：

- “强化接收者旧知识”变成了“新增一条重复知识”。
- 接收者知识库会出现重复同义 claim。
- 后续 P21 行为选择会面对多条重复知识，难以解释“B 为什么这样行动”。
- 传播链路无法稳定表达“教学使已有知识更可信”。

建议修复：

- 如果 `should_update_existing=true`，必须基于 matched target claim 生成更新结果。
- `updated_claims[0].knowledge_id` 必须等于 target existing claim 的 `knowledge_id`。
- 更新应追加 Teaching evidence，提高 confidence/support_count/source_summary_count/updated_tick。
- Applier 对 ReinforcedRecipientClaim 后 snapshot 大小应保持 1。

必须补测试：

```text
reinforced_existing_claim_updates_same_knowledge_id
reinforced_existing_claim_applier_does_not_duplicate
```

断言：

```text
result.updated_claims[0].knowledge_id == existing_target.knowledge_id
snapshot.size() == 1
snapshot[0].confidence.confidence > existing_target.confidence.confidence
```

### BLOCKER-2：接收者已有不同 effect 知识时没有进入冲突/修正路径，而是静默创建新 claim

位置：

- `backend/src/knowledge/knowledge_propagation.cpp:60`
- `backend/src/knowledge/knowledge_propagation.cpp:578`
- `backend/src/knowledge/knowledge_propagation.cpp:581`
- `backend/src/knowledge/knowledge_propagation.cpp:582`
- `backend/src/knowledge/knowledge_propagation.cpp:588`
- `backend/src/knowledge/knowledge_propagation.cpp:606`
- `backend/src/knowledge/knowledge_propagation.cpp:607`

问题：

`claimsMatchForReinforcement()` 要求 `effect_key` 完全相同。Planner 只用这个函数判断是否 matched：

```text
if (claimsMatchForReinforcement(claim, tc)) matched = true
```

因此当接收者已有：

```text
B：berry_red + eat -> restore_hunger
```

发送者传播：

```text
A：berry_red + eat -> poison
```

由于 effect 不同，系统认为“没有匹配知识”，然后创建新 claim，而不是进入 `WeakenedRecipientClaim` / `should_route_to_revision`。

探针确认：

```text
conflict_ok=1 decision=created_recipient_claim created=1 updated=0 patches=0 valid=1 created_effect=poison
```

影响：

- 接收者可以同时得到“红果恢复饥饿”和“红果有毒”两条行动知识，但系统没有标记冲突。
- P20 绕过了 P19 知识冲突与修正，破坏 P19 已经修好的核心语义。
- 后续 P21 中，B 的行为变化无法解释：是相信可食用，还是相信有毒？

设计依据：

`doc/46_P20知识传播系统任务卡设计.md` 要求：

```text
不同 effect / 条件冲突：WeakenedRecipientClaim 或 route_to_revision。
不同 effect 的接收者旧知识不会被 P20 静默覆盖。
```

建议修复：

- 拆分两个匹配函数：
  - `isSameKnowledgeCandidate()`：subject/relation/action 相同，用于发现相关旧知识。
  - `isSameEffect()`：effect/conditions 兼容，用于 Reinforce。
- 如果 same subject/relation/action 但 effect 不同：
  - `recommended_decision = WeakenedRecipientClaim` 或 `should_route_to_revision=true`。
  - 不得创建普通 recipient claim。
  - 如果生成 patch，patch 必须 validateBasic 通过，或明确只生成 route signal 不生成 invalid patch。

必须补测试：

```text
propagation_different_effect_routes_to_revision
propagation_different_effect_does_not_create_plain_claim
```

断言：

```text
result.decision != CreatedRecipientClaim
result.created_claims.empty()
plan.assessments[0].should_route_to_revision == true
```

### BLOCKER-3：`allow_create_recipient_claim` / `allow_update_recipient_claim` 选项未执行

位置：

- `backend/include/pathfinder/knowledge/knowledge_propagation.h:33`
- `backend/include/pathfinder/knowledge/knowledge_propagation.h:34`
- `backend/src/knowledge/knowledge_propagation.cpp:606`
- `backend/src/knowledge/knowledge_propagation.cpp:612`
- `backend/src/knowledge/knowledge_propagation.cpp:756`

问题：

`KnowledgePropagationOptions` 定义了：

```text
allow_create_recipient_claim
allow_update_recipient_claim
```

但 Planner/Service 没有使用这两个选项。

探针确认：

```text
allow_create_recipient_claim=false
结果：decision=created_recipient_claim created=1
```

影响：

- 调用方无法在只评估/只计划模式下禁止创建接收者知识。
- P21 编排时如果想先 dry-run 或只输出冲突信号，P20 会违反调用方限制。
- 选项存在但不生效，会误导后续 AI 实现。

建议修复：

- 创建前检查 `options.allow_create_recipient_claim`。
- 更新前检查 `options.allow_update_recipient_claim`。
- 被禁止时应返回 valid `Failed` 或 `Skipped` assessment，并写入明确 reason/warning。

必须补测试：

```text
allow_create_false_does_not_create_claim
allow_update_false_does_not_update_existing_claim
```

### BLOCKER-4：`Failed` 作为 ok DTO 返回，但 `KnowledgePropagationResult::ok()` 返回 false

位置：

- `backend/src/knowledge/knowledge_propagation.cpp:355`
- `backend/src/knowledge/knowledge_propagation.cpp:361`
- `backend/src/knowledge/knowledge_propagation.cpp:750`
- `backend/src/knowledge/knowledge_propagation.cpp:818`

问题：

任务卡允许：

```text
Skipped/Failed 可以作为 Result::ok，但 DTO 必须 validateBasic 通过。
```

当前 Service 会返回：

```text
Result::ok(result.decision = Failed)
```

但：

```text
KnowledgePropagationResult::ok() == false
```

探针确认：

```text
failed_ok=1 decision=failed result_ok=0 valid=1
```

影响：

- 调用方无法统一使用 `result.ok()` 判断业务层有效结果。
- “传播失败但流程合法”会被误判为失败 DTO。
- 和 P18/P19 阶段的 DTO 契约风格不一致。

建议修复：

- `KnowledgePropagationResult::ok()` 应包含 `KnowledgePropagationDecision::Failed`。
- 或者修改任务卡明确 Failed 不属于 ok 业务结果，但当前 Service 和测试已按 ok 返回，因此建议修 `ok()`。

必须补测试：

```text
failed_result_is_business_ok_and_validate_ok
```

## 5. 非阻塞问题与改进建议

### P1-1：`target_revision_patches` 当前生成 invalid patch，且冲突分支实际不可达

位置：

- `backend/src/knowledge/knowledge_propagation.cpp:662`
- `backend/src/knowledge/knowledge_propagation.cpp:665`
- `backend/src/knowledge/knowledge_propagation.cpp:667`
- `backend/src/knowledge/knowledge_propagation.cpp:668`

问题：

如果 `should_route_to_revision=true`，代码会生成：

```text
patch.status_before = Unknown
patch.status_after = Unknown
```

但 `KnowledgeClaimPatch::validateBasic()` 不允许 Unknown/TestOnly。也就是说，一旦冲突路由分支真正触发，`plan.validateBasic()` 会失败。

当前之所以没有暴露，是因为 BLOCKER-2 导致不同 effect 根本没有进入 route_to_revision。

建议：

- 如果 P20 只负责 route signal，不要生成 `KnowledgeClaimPatch`。
- 如果要生成 patch，必须填入 target existing claim 的真实 status/confidence/evidence。
- 更推荐第一版只在 assessment 中设置 `should_route_to_revision=true`，由 P21 或调用方显式调用 P19。

### P1-2：`ExternalRecord` 第一版语义不清

位置：

- `backend/tests/integration/p20/knowledge_propagation_security_test.cpp:194`
- `backend/tests/integration/p20/knowledge_propagation_security_test.cpp:208`

问题：

任务卡说 `ExternalRecord` 第一版只保留枚举，不接入自由外部文本。当前测试断言 `ExternalRecord` channel 可以正常传播。

这不一定越界，因为没有接入自由文本；但建议在实现和测试注释中明确：

```text
ExternalRecord 只是枚举通道，不读取外部文本、不生成自由知识。
```

避免后续 AI 误以为 P20 可以接入外部文本知识。

### P1-3：`max_claims_per_attempt` 与 `max_shared_confidence` 未实际使用

位置：

- `backend/include/pathfinder/knowledge/knowledge_propagation.h:25`
- `backend/include/pathfinder/knowledge/knowledge_propagation.h:35`
- `backend/src/knowledge/knowledge_propagation.cpp:492`
- `backend/src/knowledge/knowledge_propagation.cpp:553`

问题：

选项存在但未执行：

max_claims_per_attempt

## 6. 测试缺口

当前测试通过，但缺少 P20 最关键的负向/状态断言：

1. ReinforcedRecipientClaim 必须更新 existing target claim id。
2. ReinforcedRecipientClaim 经 Applier 后 snapshot 不得变成 2 条重复知识。
3. same subject/action/relation but different effect 必须进入 conflict/route_to_revision，不得 CreatedRecipientClaim。
4. `allow_create_recipient_claim=false` 时不得创建 claim。
5. `allow_update_recipient_claim=false` 时不得更新 claim。
6. `Failed` result 的 `result.ok()` 应符合任务卡定义。
7. route_to_revision 分支不得生成 invalid patch。

这些测试补齐前，P20 不能标记完成。

## 7. 必须修复清单

P20 复审前至少完成：

1. 修复 ReinforcedRecipientClaim，确保更新 target existing claim，而不是生成新 claim。
2. 修复 Applier 行为，强化已有知识后 snapshot 不增加重复知识。
3. 修复不同 effect 的接收者旧知识匹配，进入 conflict/route_to_revision，不得静默创建新 claim。
4. 修复或移除 invalid `target_revision_patches` 生成逻辑。
5. 执行 `allow_create_recipient_claim` 和 `allow_update_recipient_claim`。
6. 修复 `KnowledgePropagationResult::ok()` 对 `Failed` 的语义。
7. 补齐上述核心测试。
8. 重新运行 P20 定向测试、knowledge 定向测试和全量回归。

## 8. 最终判断

P20 当前不能标记完成。

它已经具备“从 A 创建一条 B 的教学知识”的基础能力，但还没有正确处理“B 已经知道什么”的关键传播语义。

P20 的核心不是单纯复制知识，而是：

```text
把 A 的主观知识安全地影响 B 的主观知识状态。
```

当前实现对已有知识的强化与冲突处理都不正确，会直接影响 P21 学习闭环。因此建议修复后复审。
