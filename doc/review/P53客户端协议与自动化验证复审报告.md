## 1. 复审结论

结论：**暂不完全通过，需要再修一个协议一致性阻塞点后复审**。

研发本轮已经修复了首次审核中的大部分关键问题：

```text
1. 客户端伪造 actor：已修复。
2. projection_version 过期仍执行：已修复。
3. option_id 改为基于 session snapshot 校验：主体已修复，但 command 响应后的 snapshot 更新仍缺失。
4. projection / patch codec 丢字段：已修复。
5. WorldCommandDto 权限校验：已修复。
```

当前剩余的核心问题是：

```text
ClientCommandGateway 在执行命令后返回了新的 available_commands，
但没有把这批 response.available_commands 写回 ClientSessionGateway 的 last_available_commands。
```

这会导致后续真实玩法接入后，客户端看到的新按钮可能不是服务端下一次 materialize 使用的权威快照。

因此 P53 现在已经接近可验收，但还需要补这一处协议一致性问题。

## 3.3 option_id 无会话权威

结论：**部分修复，仍有一个阻塞问题**。

已修复部分：

```text
SessionState 增加 last_available_commands。
bootstrap / refresh 会保存 available_commands 快照。
ClientCommandGateway materialize option 前会调用 isOptionValid()。
ClientAvailableCommandAdapter 增加 snapshot overload。
伪造 opt_wait_wolf 会被 blocked。
```

相关代码：

```text
backend/src/client_protocol/client_session_gateway.cpp:53
backend/src/client_protocol/client_session_gateway.cpp:93
backend/src/client_protocol/client_command_gateway.cpp:50
backend/src/client_protocol/client_available_command_adapter.cpp:58
```

仍需修复的问题：

```text
命令执行成功后，ClientCommandGateway 会构造 response.available_commands，
但没有把 response.available_commands 写回 ClientSessionGateway.last_available_commands。
```

风险：

```text
1. 当前 Wait / Inspect 因为是静态 stub，看不出问题。
2. 一旦后续 available_commands 接入真实地图、背包、采集、制作、教学，命令执行后的可用动作会随世界状态变化。
3. 客户端收到 response.available_commands 后会显示新动作。
4. 但下一次提交 option_id 时，服务端仍拿旧 snapshot 校验。
5. 结果可能出现“前端显示可点，但后端判定 option_id 无效”的协议断裂。
```

必须修复：

```text
1. ClientSessionGateway 增加 updateAvailableCommands(session_id, options) 接口。
2. ClientCommandGateway 在确定 response.available_commands 后，必须写回 session snapshot。
3. 如果 response.available_commands 为空但 fallback buildOptions 成功，也必须写回。
4. 增加测试：command response 返回的新 option_id，下一次 command 可以被 snapshot materialize。
5. 增加测试：不在最新 response.available_commands 中的旧 option_id 被 blocked。
```

建议同时优化：

```text
option_id 最好带 projection_version 或 option_snapshot_id，避免静态 opt_wait_player 在复杂状态下无法区分新旧快照。
```

## 4. 新发现问题

## 4.1 命令后 available_commands 未写回 session snapshot

严重性：**阻塞**。

位置：

```text
backend/src/client_protocol/client_command_gateway.cpp:200
```

问题说明：

```text
handleCommand() 会优先使用 pipeline_response.available_commands，
否则 fallback 到 available_command_adapter_.buildOptions(actor_key, "surface")。
但这两种路径都只写 response.available_commands，未更新 session.last_available_commands。
```

为什么这是 P53 必须修的问题：

```text
P53 的核心协议是：available_commands 是客户端下一步操作的唯一权威。
如果 response.available_commands 与 session snapshot 不一致，客户端看到的权威动作和后端 materialize 使用的权威动作就不是同一个东西。
这会直接影响 P54 客户端，不应该留到界面阶段修。
```

## 4.2 client_id 尚未参与 session 校验

严重性：**非阻塞，但建议 P53 顺手补**。

说明：

```text
ClientSessionGateway 已经新增 getSessionClientId()，但 ClientCommandGateway / refresh 目前没有使用它校验 request.client_id。
```

风险：

```text
只要知道 session_id，就可能用不同 client_id 提交 command / refresh。
```

建议：

```text
1. command / refresh / reset 校验 request.client_id == session.client_id。
2. 增加 client_id_mismatch_blocked 测试。
```

## 4.3 public legacy materializeOption() 仍可前缀伪造

严重性：**非阻塞，但建议限制使用范围**。

位置：

```text
backend/src/client_protocol/client_available_command_adapter.cpp:90
```

说明：

```text
ClientCommandGateway 已经使用 snapshot overload，所以正式路径安全性比之前好。
但无 snapshot 的 materializeOption(option_id, actor_key) 仍然是 public，并且仍按 opt_wait_ / opt_inspect_ 前缀识别。
```

建议：

```text
1. 如果没有外部必要，删除无 snapshot overload。
2. 如果为了历史单元测试保留，至少标注 TestOnly，并增加架构扫描禁止 ClientCommandGateway 使用该 overload。
```
