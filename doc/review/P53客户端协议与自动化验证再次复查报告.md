# P53 客户端协议与自动化验证再次复查报告

## 1. 复查结论

结论：**仍不通过，但上轮 3 个阻塞点已修复；本轮新增发现 1 个 WorldCommandDto 权限阻塞点**。

本轮确认研发已修复上次最终复审中的 3 个阻塞问题：

```text
1. Blocked / Failed 响应现在可 validateBasic() 且可 codec roundtrip。
2. 多 session 共享 WorldCommandPipeline 时，执行前会把 pipeline projection_version 同步到当前 session version。
3. WorldCommandDto 会拒绝 allow_hidden_debug 和部分 unsafe parameters。
```

新增发现：

```text
WorldCommandDto 模式仍缺少 command_kind / command_key 权限策略。
客户端可以绕过 available_commands，直接提交 GenerateWorld 等未签发动作，并执行成功。
```

因此 P53 还不能作为 P54 客户端协议层验收。

## 4. 新增阻塞问题

### 4.1 WorldCommandDto 可以绕过 available_commands 执行未签发命令

严重性：**阻塞**。

位置：

```text
backend/src/client_protocol/client_command_gateway.cpp:23
backend/src/client_protocol/client_command_gateway.cpp:102
backend/tests/integration/client_protocol/client_protocol_integration_test.cpp:39
backend/tests/integration/client_protocol/client_protocol_integration_test.cpp:43
```

问题说明：

```text
validateClientCommandDto() 当前校验：
1. actor_key 必须等于 session actor。
2. source 必须是 PlayerInput。
3. allow_hidden_debug 不允许。
4. 部分 unsafe parameters 不允许。

但它没有校验 command_kind / command_key 是否属于该 session 最新 available_commands，
也没有 client-safe command allowlist。
```

因此客户端可以：

```text
1. bootstrap 拿到后端签发的 Wait / Inspect。
2. 不提交 option_id。
3. 改为提交 WorldCommandDto(command_kind=GenerateWorld)。
4. 只要 handler registry 中注册了 GenerateWorld handler，就会执行成功。
```

本次复现使用的 harness registry 本身就注册了：

```text
createGenerateWorldCommandHandler()
```

而 `available_commands` 并没有给客户端签发 generate_world option。

为什么阻塞：

```text
P53 文档把 available_commands 定为操作权威来源。
WorldCommandDto 虽然允许“测试或高级客户端”使用，但文档同样要求“仍由后端校验”。
当前后端校验不足，WorldCommandDto 成为绕过 option authority 的后门。
后续一旦 registry 接入 move / gather / craft / teach / worldgen / spawn 类 handler，客户端可直接构造未授权动作。
```

必须修复，建议二选一：

```text
方案 A：P53 正式客户端路径禁用 WorldCommandDto，只允许 TestOnly / harness 使用；生产 ClientCommandRequest 只接受 OptionId。
方案 B：WorldCommandDto 也必须绑定 latest available_commands：command_kind、command_key、target 必须匹配某个已签发 option，否则 Blocked。
```

如果保留高级客户端能力，还应补充：

```text
1. client-safe command allowlist，至少禁止 GenerateWorld、GenerateRegion、SpawnEntity、DespawnEntity、SystemTick、ApplyAreaEffectTick 等系统命令。
2. 测试：WorldCommandDto GenerateWorld 被 Blocked。
3. 测试：WorldCommandDto command_key 不在 available_commands 中被 Blocked。
4. 测试：WorldCommandDto target 与 available_commands 中 option target 不一致被 Blocked。
```

## 5. 非阻塞问题

### 5.1 bootstrap client_id mismatch 处理方式不安全

位置：

```text
backend/src/client_protocol/client_session_gateway.cpp:16
backend/src/client_protocol/client_session_gateway.cpp:20
```

当前实现：

```text
如果 session_id 已存在但 client_id 不同，直接 erase 旧 session 并创建新 session。
```

风险：

```text
知道 session_id 的其他 client 可以通过 bootstrap 抢占 / 重置 session。
虽然不再泄露旧 projection，但会造成会话劫持或 DoS。
```

建议：

```text
bootstrap 遇到已有 session_id 且 client_id mismatch 时应返回 error / blocked，不能自动 erase。
```

### 5.2 stale blocked response 顶层版本与 patch 版本不完全一致

位置：

```text
backend/src/client_protocol/client_command_gateway.cpp:193
backend/src/client_protocol/client_command_gateway.cpp:197
```

说明：

```text
stale response 的 response.base_projection_version / new_projection_version 是 current/current，
但 projection_patch.base_projection_version / new_projection_version 是 known/current。
```

建议：

```text
为了减少客户端实现歧义，建议让顶层版本字段与 projection_patch 版本语义保持一致，或明确文档化差异。
```
