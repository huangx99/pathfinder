复查依据：`doc/review/P20知识传播系统审核报告.md`、`doc/46_P20知识传播系统任务卡设计.md`、`context_packs/knowledge_p20.md`。

## 1. 复查结论

**结论：通过，可以进入下一阶段。**

上次审核发现的 4 个阻塞项已经修复：

1. `ReinforcedRecipientClaim` 已更新接收者已有 claim，不再生成重复新 claim。
2. 接收者已有同 subject/action/relation 但不同 `effect_key` 的知识时，已进入冲突/route-to-revision 路径，不再静默创建普通 claim。
3. `allow_create_recipient_claim` / `allow_update_recipient_claim` 已生效。
4. `Failed` 作为合法业务结果时，`KnowledgePropagationResult::ok()` 已返回 true。

从玩法语义看，P20 现在能正确表达：

```text
A 教 B 红果可食用。
如果 B 已经知道红果可食用，则强化 B 的旧知识。
如果 B 已经有不同效果知识，则产生冲突路由，不直接覆盖或并排创建行动知识。
```

这满足 P20 “传播影响接收者主观知识状态，而不是复制真相”的核心要求。

## 5. 非阻塞建议

### P1：WeakenedRecipientClaim 当前是 route signal，不直接改知识

当前不同 effect 冲突时，P20 返回：

```text
decision = WeakenedRecipientClaim
assessment.should_route_to_revision = true
created_claims / updated_claims / applied_patches 为空
```

这符合“P20 不重写 P19 修正逻辑”的设计方向，但后续 P21 编排必须明确消费这个 route signal，并调用 P19 revision 或等效修正流程。

建议 P21 任务卡中写入：

```text
如果 P20 result.decision == WeakenedRecipientClaim 且 should_route_to_revision=true，P21 必须进入知识修正编排，而不是把传播当作完成。
```

### P1：失败原因可进一步细分

当前 `allow_create=false` / `allow_update=false` 使用 `ClaimNotTeachable` 作为 failure_reason。语义上可以工作，但后续可考虑新增更精确 reason：

```text
CreateNotAllowed
UpdateNotAllowed
```

这不是 P20 阻塞项，只是可诊断性增强。

## 6. 最终判断

P20 修复后达到阶段目标：

- Agent A 可以把有效主观知识传播给 Agent B。
- Agent B 获得自己的 Teaching-derived `KnowledgeClaim`。
- 接收者已有同知识时会强化旧 claim，不会重复创建。
- 接收者已有不同 effect 知识时会进入冲突路由，不会静默创建并列行动知识。
- 创建/更新开关生效。
- Failed 业务结果语义一致。
- Service 不直接写 Repository。
- 未发现 hidden truth/raw GameState/Save/API/RL 越界依赖。

**复查结论：P20 通过，可以进入 P21 学习闭环设计。**
