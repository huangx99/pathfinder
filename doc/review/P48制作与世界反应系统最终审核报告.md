# P48 制作与世界反应系统最终审核报告

## 1. 审核结论

结论：通过，可以提交推送。

P48 已完成“制作与世界反应接入”的阶段目标：

- `WorldCommandKind::Craft` 已接入世界反应服务。
- 制作反应从 `ContentRegistry::findReaction(reaction_key)` 读取配置，不在反应服务主逻辑写死具体物品。
- 材料消耗统一经过 P45 背包/地图归属接口，不直接绕过 runtime 改数组。
- 产物可以进入 Actor 背包，也可以落到地图 `OnMap`。
- 命令结果会返回 `events`、`state_deltas`、`projection_patch`、`experiences`。
- P48 只产出经历，不直接形成知识，后续由 P49 接入认知/学习闭环。

## 2. 本次复查重点

### 2.1 配置驱动

已检查 `backend/src/world_reaction`：

- 未发现 `wood_plus_fire_make_torch`、`torch`、`camp_fire`、`wood` 等具体内容写死在反应服务主逻辑中。
- 反应通过 `reaction_key` 查询 `ContentRegistry`。
- 输出对象通过 `content_registry_.findObject(output.object_key)` 校验。

接受的设计偏差：

- `Craft` 命令允许驱动 `craft`、`use`、`combine` 三类 `action_key`。
- 这是 P48 的合理取舍，因为前端和 NPC 只需要一个统一“制作/组合”入口，底层仍由配置控制具体反应。

### 2.2 P45 归属接口

已检查：

- 背包产物通过 `InventoryTransferKind::SpawnToInventory` 进入背包。
- 地图产物通过 `IWorldEntityLocationPort::spawnEntityOnMap()` 生成 `OnMap` 实体。
- 新增 `IWorldEntityLocationPort::spawnEntityInInventory()`，确保进入背包的实体也同步存在于 `WorldRuntime`，状态为 `InInventory`，`owner_ref` 指向 actor。
- 地图材料消耗通过 `setEntityStackData()` 或 `destroyEntity()` 处理，没有直接改 cell/entity 数组。

### 2.3 确定性与可回放

已检查：

- P48 反应产物 ID 使用 `reaction_command_id + object_key + index` 生成。
- 不再依赖 `state_version_`、时间、随机数作为制作产物 ID。
- 已补充 `world_reaction_output_id_is_command_deterministic` 测试。

### 2.4 错误分类

已检查：

- 材料不存在返回 `InputMissing`。
- 材料数量不足返回 `QuantityInsufficient`。
- 材料状态不匹配返回 `InputStateMismatch`。
- 之前“状态不匹配被折叠成缺材料”的阻塞问题已修复。

### 2.5 Command 输出给前端/NPC

已检查：

- `CraftCommandHandler` 会调用 `WorldReactionService`。
- 成功后返回 `events`、`state_deltas`、`experiences`。
- `WorldReactionProjectionBridge` 会生成 `projection_patch`。
- `projection_patch.new_projection_version = base_projection_version + 1`，版本推进正确。

## 3. 已修复的阻塞问题

### 3.1 背包产物没有同步到 WorldRuntime

原问题：

- `SpawnToInventory` 只写背包，未保证 `WorldEntityInstance` 存在。
- 后续世界网格、NPC、知识系统会找不到这个物品实体。

当前状态：已修复。

验证：

- `world_inventory_spawn_to_inventory_creates_world_entity`
- `world_reaction_craft_output_entity_exists_in_world_runtime`

### 3.2 产物 ID 依赖运行时版本

原问题：

- 制作产物 ID 依赖 `state_version_`，不利于回放、测试和同步。

当前状态：已修复。

验证：

- `world_inventory_spawn_to_inventory_uses_request_entity_id`
- `world_reaction_output_id_is_command_deterministic`

### 3.3 状态不匹配错误分类不准确

原问题：

- 需要 `processed` 状态时，如果只有未处理材料，错误会表现成缺材料。

当前状态：已修复。

验证：

- `world_reaction_validate_state_mismatch`

### 3.4 地图材料和地图产物覆盖不足

原问题：

- 缺少同格地图材料、相邻地图材料、落地输出的测试。

- `world_reaction_drop_on_map_output_spawns_entity_on_actor_cell`

### 3.5 投影版本没有推进

原问题：

## 5. 非阻塞遗留风险

### 5.1 反应事务回滚仍是后续增强

当前 P48 是“先验证再消耗再生成”的顺序执行：

- validate 阶段会提前检查材料和输出配置。
- apply 阶段如果产物生成失败，理论上存在“材料已消耗但产物失败”的事务补偿风险。

本阶段为什么不阻塞：

- 当前测试覆盖了失败不污染状态、缺材料、状态不匹配、落地输出等关键路径。
- P48 的目标是完成制作/反应接入，不是完整事务引擎。
- 事务补偿应在后续“通用世界命令事务层”统一解决，不能只在 P48 局部打补丁。

建议后续阶段处理：

- 增加 `WorldMutationTransaction` 或 `CommandMutationBatch`。
- 所有 consume/spawn/move/update 先进入 batch。
- 全部校验通过后统一 commit。
- commit 失败时提供 rollback 或确保 mutation 是原子应用。

### 5.2 重复 input object_key 暂时拒绝

当前 P48 对同一 `object_key` 的重复输入直接拒绝。

本阶段为什么不阻塞：

- 可以避免“同一种材料多个不同状态输入”导致消耗歧义。
- 后续如果需要复杂配方，例如两个不同状态的同类材料，需要引入 `input_slot_id`，不能简单放开。

## 6. 最终判断

P48 当前实现满足阶段验收条件：

- 制作入口接入世界命令。
- 反应读取配置。
- 材料消耗和产物归属走 P45 权威接口。
- 背包产物和地图产物均能被世界运行时追踪。
- 结果能返回给前端、NPC 和后续认知学习系统。
- 关键失败路径和回归测试通过。

最终结论：P48 通过，可以提交推送。
