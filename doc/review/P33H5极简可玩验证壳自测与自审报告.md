# P33 H5 极简可玩验证壳自测与自审报告

## 1. 审查结论

P33 已完成设计要求对应的最小可玩验证壳实现，并通过专项、相邻、全量测试和 HTTP 冒烟测试。

本阶段实现结论：可以作为“手机浏览器访问的中文文本可玩壳”使用，用来验证玩家通过 H5 对话框进行探索、尝试、形成知识、查看同伴知识、传授知识和重开的最小闭环。

P33 仍然不是正式地图、不是正式 UI、不是新规则系统；它只把既有 P22 对话回合结果和会话状态映射为 P32 安全投影，再由 H5 前端渲染。

## 2. 实现范围

### 2.1 后端模块

新增 `pathfinder_h5_playable` 模块：

- `H5PlayableTurnService`：承接 H5 输入，调用 P22 `DialogTurnService::handleDetailed()`，再生成 P32 `H5GameProjection`。
- `H5PlayableProjectionMapper`：把 P22 的对话响应、玩家知识、同伴知识、条件、危险提示映射为安全投影。
- `H5PlayableWireCodec`：解析 form-urlencoded 请求并输出 JSON。
- `H5PlayableServer`：提供极简 HTTP 服务和静态文件服务。
- `pathfinder_h5_playable_server`：可执行程序，默认用于启动 H5 验证壳。

### 2.2 P22 兼容扩展

扩展 `DialogTurnService`，新增：

- `handleDetailed()`：保留原 response，同时返回会话状态。
- `loadOrCreateSessionSnapshot()`：供 P33 bootstrap 使用。
- `resetSession()`：供 H5 重开使用。

原 `handle()` 仍保持兼容，内部委托给 `handleDetailed()`。

### 2.3 前端页面

新增 `frontend/h5_playable`：

- `index.html`：手机竖屏优先的极简中文文本界面。
- `app.js`：只调用 bootstrap/turn API，只渲染 projection，不裁决规则。
- `style.css`：极简可读样式。

前端展示内容包括：

- 场景标题和场景说明。
- 可尝试对象卡片。
- 对象行动按钮。
- 全局行动按钮。
- 最近反馈。
- 我的知识。
- 同伴知识。
- 条件提示。
- 危险提示。
- 重开按钮。

## 3. 架构边界自审

### 3.1 P33 没有新增玩法规则

人工检查结论：通过。

P33 不直接修改世界对象、不直接执行对象反应、不直接更新知识、不直接裁决吃/用/教结果。

实际行动裁决仍由 P22/P21 既有流程处理：

```text
H5 输入 -> H5PlayableTurnService -> DialogTurnService::handleDetailed()
        -> 既有对话/知识流程 -> DialogResponseDto + DialogSessionState
        -> H5PlayableProjectionMapper -> H5ProjectionComposer -> H5GameProjection
        -> JSON -> 前端渲染
```

### 3.2 前端没有硬编码知识规则

人工检查结论：通过。

已检查 `frontend/h5_playable`：

- 没有 `red_berry`。
- 没有 `restore_hunger`。
- 没有 `poison`。
- 没有 ` -> ` 调试式知识字符串。
- 前端知识展示只读取 `projection.actor_knowledge` 和 `projection.recipient_knowledge`。

这保证后续加新物品、新效果、新知识时，前端不需要跟着写规则判断。

### 3.3 安全输出边界

人工检查结论：通过。

P33 response 校验拒绝玩家可见 `DebugSummary` 文本；P32 composer 继续负责安全投影合成和输出门禁。

HTTP JSON 冒烟测试确认响应中没有：

- `raw_state`
- `hidden_truth`
- `hp_delta`

### 3.4 模块职责

人工检查结论：通过。

- `H5PlayableTurnService`：只编排回合，不写规则。
- `H5PlayableProjectionMapper`：只做安全展示映射，不改会话状态。
- `H5PlayableWireCodec`：只做请求解析和 JSON 编码，不判断玩法结果。
- `H5PlayableServer`：只处理 HTTP 路由、静态文件和 API 调用。
- `frontend/h5_playable/app.js`：只渲染 projection，不解析后端调试字符串，不读 raw state。

### 4.5 HTTP 冒烟测试

由于当前机器 `1999` 端口已被旧进程占用：

```text
pathfinder_h5_d pid=935317 正在监听 0.0.0.0:1999
```

为避免误杀用户正在运行的服务，本次用临时端口 `2999` 验证 P33 新服务本身：

- `GET /`：通过，能返回 H5 页面。
- `POST /api/playable/bootstrap`：通过，能返回安全 projection。
- `POST /api/playable/turn`，输入“吃红果”：通过，能返回回合反馈和更新后的 projection。
- JSON 中未发现 `raw_state`、`hidden_truth`、`hp_delta`。

结论：二进制和 API 可用；如要使用 `1999`，需要先停止当前占用 `1999` 的旧服务。

## 5. 发现的问题与处理

### 5.1 构建错误码不匹配

问题：初次构建时发现 `ErrorCode::runtime_error` 不存在。

处理：改用当前工程已有的 `common_internal_invariant_broken`。

结果：构建通过。

### 5.2 前端门禁测试路径问题

问题：前端门禁测试在不同执行目录下可能找不到 `frontend/h5_playable/app.js`。

处理：测试中补充候选路径查找。

结果：门禁测试通过。

### 5.3 1999 端口被旧服务占用

问题：P33 服务尝试绑定 1999 时失败。

处理：未强杀旧进程；改用 2999 做服务本身冒烟测试。

结果：P33 服务本身验证通过；实际访问 1999 前需要停止旧服务。

## 6. 当前限制

P33 是验证壳，不是最终正式 H5：

- 没有正式地图，只是地点式探索。
- 没有复杂 UI 动效，只保证手机上能读、能点、能操作。
- 不新增规则内容，仍围绕 P22 默认场景验证最小闭环。
- JSON 编码器是 P33 局部手写实现，足够当前验证；后续 P34/P35 可考虑抽成通用 serializer。
- `H5PlayableProjectionMapper` 仍围绕默认场景做安全映射，后续内容配置扩展后应逐步配置化。

这些限制不影响 P33 目标，因为 P33 的目标是先验证“浏览器可玩壳 + 后端规则闭环 + 安全投影链路”是否成立。

## 7. 启动方式

如 `1999` 未被占用，可启动：

```bash
./build/backend/pathfinder_h5_playable_server --host 0.0.0.0 --port 1999 --static-root frontend/h5_playable
```

然后通过浏览器访问：

```text
http://<内网穿透地址>:1999/
```

如果提示启动失败，先检查端口占用：

```bash
ss -ltnp '( sport = :1999 )'
```

## 8. 自审结论

P33 实现符合设计文档边界：

- 可以在手机浏览器上形成中文文本可玩闭环。
- 前端没有写死物品知识和规则结果。
- 后端仍是唯一规则裁决来源。
- 输出经过 P32 安全投影，不暴露 raw state/hidden truth。
- 已通过构建、专项、相邻、全量测试和 HTTP 冒烟测试。

建议下一步进入 P34：端到端 MVP。P34 应在 P33 可玩壳基础上补齐 10 分钟基础探索循环、端到端验收脚本、调试导出和复盘入口。
