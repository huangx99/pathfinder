# P24 共同作战审核报告

## 审核结论

P24 已经完成共同作战协调模块的基础实现，整体方向符合 `doc/51_P24共同作战任务卡设计.md`：

- 新增了 `tribe_coordination` 类型、resolver、projection 和 trace。
- 能基于 `TribeState`、成员角色、士气、信任、分裂风险和威胁摘要生成确定性协作结果。
- 覆盖了守护者保护学习者、低士气撤退、高信任集中压制、分裂风险削弱协作、确定性结果等主要流程。
- 没有发现直接扣血、死亡、掉落、族群外交或文明解锁实现。
- `pathfinder_tribe` 仍只依赖 `pathfinder_foundation` 和 `pathfinder_knowledge`。
- P24 专项测试、P23+P24 回归、全量测试均通过。

但本次审核发现 **3 个必须修复问题** 和 **3 个建议补强点**。建议修复 P1 后再进入 P25。

## P1 问题：`DefendAction.protected_key` 没有做 forbidden key 校验

位置：

- `backend/include/pathfinder/tribe/tribe_coordination.h`
- `backend/src/tribe/tribe_coordination.cpp`

`DefendAction` 有字段：

```cpp
std::optional<std::string> protected_key;
```

但 `DefendAction::validateBasic()` 只校验了：

```text
kind
actor_member_id
protected_member_id
threat_id
defense_score
risk_reduction
summary_key
reason_keys
```

没有校验 `protected_key`。

风险：

```text
protected_key 可以携带 hidden_truth、true_hp、actual_damage、kill_result 等禁止语义并通过 validateBasic。
这违反 P24 “Projection/DTO 不能暴露隐藏真相、不能包含伤害死亡语义”的安全边界。
```

修复建议：

```text
如果 protected_key.has_value()，必须检查非空并执行 containsCoordinationForbiddenKey(protected_key.value())。
增加测试：DefendAction.protected_key = "hidden_truth_cache" 应拒绝。
```

## P1 问题：伤害/死亡/击杀/掉落 forbidden key 覆盖不完整

位置：

- `backend/src/tribe/tribe_coordination.cpp`
- `backend/tests/integration/p24/tribe_coordination_security_test.cpp`

P24 任务卡要求拒绝：

```text
P24 结果中出现 damage / death / kill / loot 语义。
```

当前 forbidden token 包含：

```text
random_damage
actual_damage
damage_roll
kill_result
loot_drop
true_hp
```

但没有覆盖通用语义：

```text
damage
death
kill
loot
hp
```

风险：

```text
summary_key = "death_summary"、"kill_enemy"、"damage_summary"、"loot_hint" 可能绕过校验。
当前 p24_security_rejects_damage_or_death 测试名虽然包含 death，但实际上没有测试 death/kill/loot 通用词。
```

修复建议：

```text
补齐 forbidden token：damage、death、kill、loot、hp。
如果担心误伤正常词，至少补齐 death、kill、loot，并对 damage/hp 使用明确命名规范测试。
增加 summary_key/reason_keys/warning_keys/projection 字段中的 damage/death/kill/loot 拒绝测试。
```

## P1 问题：`CombatCoordinationInput` 允许空 `tribe_state` 通过

位置：

- `backend/src/tribe/tribe_coordination.cpp`

当前逻辑：

```cpp
if (!tribe_state.tribe_id.empty()) {
    auto ts_result = tribe_state.validateBasic();
}
```

这意味着：

```text
input.tribe_id 有值，但 input.tribe_state 为空时，validateBasic 仍可能通过。
resolver 会把它当作 no_available_member，而不是非法输入。
```

这与 P24 任务卡不一致：

```text
tribe_state 必须 validateBasic 通过。
P24 从 TribeState 和安全威胁摘要构造 CombatCoordinationInput。
```

风险：

```text
上游忘记传 TribeState 时，系统不会报错，而是生成 no_available_member，导致问题被隐藏。
```

修复建议：

```text
CombatCoordinationInput::validateBasic() 必须强制 tribe_state.validateBasic()。
并要求 tribe_state.tribe_id == input.tribe_id。
增加测试：只有 input.tribe_id、空 tribe_state 时应拒绝。
```

## P2 建议：`CombatCoordinationOptions::threat_weight` 当前未实际参与评分

位置：

- `backend/include/pathfinder/tribe/tribe_coordination.h`
- `backend/src/tribe/tribe_coordination.cpp`

`CombatCoordinationOptions` 定义了：

```cpp
double threat_weight{0.15};
```

但 resolver 中协作评分没有使用 `options.threat_weight`。

当前威胁压力影响：

```text
intent 选择。
risk_reduction 计算。
```

但 `threat_weight` 这个 option 本身没有效果。

风险：

```text
后续调参时，以为 threat_weight 会改变协作评分，实际不会。
```

修复建议二选一：

```text
方案 A：把 threat_weight 纳入 coordination_score 或 risk_reduction 的确定性公式。
方案 B：删除 threat_weight，避免假配置。
```

## P2 建议：KnowledgeId 格式校验不完整

位置：

- `backend/src/tribe/tribe_coordination.cpp`

以下字段只检查了空和重复，没有检查 `isValidIdString`：

```text
ThreatPressureSummary.known_counter_knowledge_ids
CombatParticipant.known_knowledge_ids
```

P23 对 knowledge id 做了格式校验，P24 最好保持一致。

修复建议：

```text
对 KnowledgeId 调用 foundation::isValidIdString(id.value())。
增加非法 knowledge id 格式拒绝测试。
```

## 架构符合项

本次审核确认以下点符合 P24：

- P24 新增实现集中在 `tribe` 模块内。
- 没有新增 H5、HTTP、WebSocket、SaveManager 或 AgentRuntime 依赖。
- `CombatCoordinationResolver` 是确定性 resolver，没有随机数。
- `MemberCoordinationSummary` 已作为聚合层，结果和 state draft 中 member id 唯一。
- `CombatCoordinationStateDraft` 不直接修改 `TribeState`，只是输出士气、信任、安全压力等草稿。
- 没有实现 P25 的族群关系、外交、战争 resolver。
- 没有实现 P26 文明能力解锁。

## 审核结论建议

```text
P24 基础实现方向正确，测试也全部通过，但安全校验还有缺口。
建议先修复 3 个 P1 问题，再进入 P25。
```

推荐处理顺序：

1. 给 `DefendAction.protected_key` 增加 forbidden key 校验和测试。
2. 补齐 damage/death/kill/loot/hp 语义拒绝测试和 forbidden token。
3. 强制 `CombatCoordinationInput.tribe_state` 必须 validateBasic 通过。
4. 补 `p24_explorer_warns_group_flow`。
5. 决定 `threat_weight` 是接入公式还是删除。
6. 补 KnowledgeId 格式校验。
