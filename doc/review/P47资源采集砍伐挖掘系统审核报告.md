# P47 资源采集砍伐挖掘系统审核报告

更新时间：2026-05-21
审核对象：Kimi 实现的 P47 资源采集、砍伐、挖掘、采矿系统
审核结论：暂不通过，需要修复后复审

## 1. 已通过项

### 1.1 编译与测试

已运行：

```bash
cmake -S backend -B build/backend
cmake --build build/backend --target pathfinder_tests_world_harvest pathfinder_tests_world_harvest_integration -j 2
ctest --test-dir build/backend -R '^(world_harvest_|world_command_(gather|chop|mine|dig|harvest)|world_generation_)' --output-on-failure
```

结果：73/73 通过。

### 1.2 架构方向

- 新增 `world_harvest` 独立模块，没有把采集逻辑塞回基础 `world_command`。
- 产物生成调用了 P45 `IWorldEntityLocationPort::spawnEntityOnMap`，没有直接塞背包。
- `Gather/Chop/Mine/Dig` 已接入 `WorldCommand` handler。
- 采集会消费 `WorldResourceNodeRuntime.remaining_charges`。
- 缺工具、距离过远、不可见、节点耗尽等基础失败路径已覆盖。

## 2. 阻塞问题

### 2.1 `PreferInventory` 被半实现，违反 P47 文档边界

设计文档明确写了：

- P47 MVP 只实现 `DropOnMap`。
- `PreferInventory` 预留，不能半实现。
- P47 不允许直接把产物塞进背包。

当前代码问题：

- `harvest_command_handler.cpp` 会从 command 参数读取 `output_location_policy=prefer_inventory`。
- `ResourceHarvestRequest` 会接受 `HarvestOutputLocationPolicy::PreferInventory`。
- `ResourceHarvestService::apply()` 实际仍然全部 `spawnEntityOnMap`。

这会造成“命令说优先背包，但系统实际掉落地图”的语义错位，属于半实现。

修复要求：

- P47 阶段只允许 `DropOnMap`。
- 如果 command 参数传入 `prefer_inventory`，必须失败，failure kind 使用 `InvalidRequest` 或更合适的 P47 错误码。
- 增加测试：`prefer_inventory` 在 P47 被拒绝，且不扣 charge、不生成产物。

### 2.2 空产物节点会成功采集并扣 charge

当前代码问题：

- `validate()` 直接遍历 `node.output_entity_keys` 创建 output draft。
- 如果 `output_entity_keys` 为空，会生成空 `output_drafts`。
- `apply()` 会成功扣 charge，并返回 ok，但没有任何产物。

这会让资源节点“被采集掉却没有产物”，不符合 P47 的资源采集闭环，也容易掩盖 P46 配置错误。

修复要求：

- `node.output_entity_keys` 为空时必须失败，建议 failure kind 使用 `OutputInvalid`。
- 任一 output entity key 为空也必须失败。
- 失败必须不扣 charge、不生成产物。
- 增加测试：空 output、空 output key 两种情况。

### 2.3 事件 tick 固定为 0，未接入 CommandContext 当前 tick

当前代码问题：

- `ResourceHarvestService::apply()` 里 `ResourceHarvested` 和 `NodeDepleted` 事件的 `tick` 都写死为 0。
- `HarvestCommandHandler` 转发事件时没有把 `context.currentTick()` 写回事件。

这会影响后续回合、区域事件、追踪回放、前端时间线表现。

修复要求：

- P47 可以保持 service 不依赖 command context。
- 但 command handler 必须在转发事件前，把事件 tick 统一设置为 `context.currentTick()`。
- 增加测试：command context tick 为非 0 时，采集事件和耗尽事件 tick 都等于当前 tick。

## 3. 非阻塞风险

### 3.1 apply 原子性还不完整

当前 apply 顺序是：先生成产物，再更新资源节点。

在当前 `WorldGridRuntime::updateResourceNodeRuntime()` 几乎不会失败，所以实际风险较低。但从接口语义看，如果未来 runtime update 可能失败，就会出现“产物已生成但节点未扣 charge”的部分提交。

本阶段不强制重构事务系统，但需要在代码中加注释，说明 P47 当前依赖内存 runtime 更新必定成功；未来 P48/P50 如果引入持久化或多 runtime，需要引入事务或补偿机制。

## 6. 复审追加阻塞问题

### 6.1 未知 output_location_policy 被静默当成 DropOnMap

复审发现：

- `ResourceHarvestService::validate()` 只拒绝 `PreferInventory`，没有拒绝 `HarvestOutputLocationPolicy::Unknown`。
- `HarvestCommandHandler` 对 command 参数中除了 `prefer_inventory` 以外的任何字符串，都会静默当成 `DropOnMap`。

这会导致前端或脚本传入错误配置时，系统没有报错，反而执行成功。后续 JSON 配置和 V2 command 协议会因此隐藏错误。

修复要求：

1. `ResourceHarvestService::validate()` 必须只接受 `HarvestOutputLocationPolicy::DropOnMap`。
2. `HarvestOutputLocationPolicy::Unknown`、`PreferInventory` 都必须失败。
3. `HarvestCommandHandler` 参数解析必须明确：
   - 缺省：`DropOnMap`。
   - `drop_on_map`：`DropOnMap`。
   - `prefer_inventory`：设置为 `PreferInventory`，让 service 拒绝。
   - 其他任意字符串：设置为 `Unknown` 或直接失败，但不能静默当作 `DropOnMap`。
4. 增加测试：传入 `output_location_policy=bad_policy` 必须失败，不扣 charge，不生成产物。
5. 增加测试：直接构造 `ResourceHarvestRequest.output_location_policy=Unknown` 必须失败。
