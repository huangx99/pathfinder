# P40 动态目标执行与外界打断系统审核报告

## 审核结论

结论：**不建议验收，需要修复后复审**。

本次实现已经建立了 `pathfinder_goal_execution` 模块，并且新增测试全部通过；但对照 `doc/69_P40动态目标执行与外界打断系统任务卡设计.md`，仍存在几个关键落地缺口。最大问题是：**H5 只新增了 `execution_status` 编码和前端消费能力，但实际 H5 响应没有填充 P40 执行状态**。这会导致玩家界面仍显示“执行系统尚未接入”，不满足 P40 的可见执行要求。

## 已通过项

1. 新增 `pathfinder_goal_execution` target。
2. 新增目标执行类型：`GoalFrame`、`ExecutionContext`、`PausedExecutionContext`、`ResourceReservation`。
3. 新增 Driver 抽象和默认 Driver：`Eat/Gather/ChopWood/SharpenTool/BuildStructure/UseObject/MoveTo/CounterThreat`。
4. 新增内部阻塞：`InternalBlocker`、`InternalBlockerResolver`。
5. 新增外界打断：`ExternalInterruptSignal`、`InterruptSystem`、`InterruptPriorityEvaluator`。
6. 新增材料入口：`MaterialRequirementEvaluator`、`MaterialReservationManager`、`ReactionMaterialResolver`。
7. 新增 P38 适配入口：`DriverCommandAdapter` 输出 `WorldInteractionInput`，没有直接改 `WorldSnapshot`。
8. `ChopWoodDriver` 没有直接处理狼，狼由 `InterruptSystem` 在 tick 开头抢占。
9. `BuildStructureDriver` 在没有 `MaterialRequirementSet` 时返回 `MissingCondition`，没有直接写死建房材料。
10. 新增测试覆盖 tick 顺序、玩家命令延迟、能力过滤、reservation 释放、known_claims、反应链等。

## 必须修复问题

### P0-1：H5 execution_status 没有实际接入 GoalExecutionSystem

现象：

- `H5PlayableResponse` 增加了 `execution_status` 字段。
- `H5PlayableWireCodec` 会编码该字段。
- 前端 `normalizeExecution` 会读取该字段。
- 但 `H5PlayableTurnService::buildResponse` 没有调用 `GoalExecutionSystem`，也没有给 `response.execution_status` 赋值。

位置：

- `backend/src/h5_playable/playable_turn_service.cpp:180`
- `backend/src/h5_playable/playable_turn_service.cpp:230`
- `backend/src/h5_playable/playable_wire_codec.cpp:377`
- `frontend/h5_playable/app.js:329`

影响：

- H5 页面仍然只能显示“执行系统尚未接入”。
- P40 文档要求的“当前目标 / 当前步骤 / 外界打断 / 恢复提示”无法出现在玩家界面。
- 当前 H5 测试只检查 JSON 字段存在，没有检查字段非空或来自后端执行投影。

修复要求：

1. 在 H5 turn 流程中构造最小 `ExecutionTickRequest`。
2. 调用 `GoalExecutionSystem::tick` 或引入独立 adapter。
3. 将 `ExecutionResult.projection` 映射到 `H5PlayableExecutionStatus`。
4. 至少在等待/同伴驱赶野兽/伐木遇狼测试中证明 `execution_status.visible == true`。
5. 更新 H5 测试，禁止只检查字段名存在。

### P0-2：内部阻塞生成的子目标没有进入目标栈

现象：

`GoalExecutionSystem::tick` 中调用了：

```cpp
auto goals = blocker_resolver_.resolve({result.updated_context, result.internal_blockers});
(void)goals;
```

但结果没有 push 到 `GoalStackManager`，也没有放进 `ExecutionResult` 的可执行输出。

位置：

- `backend/src/goal_execution/goal_execution_system.cpp:815`

影响：

- 文档要求“内部阻塞能生成子目标”，但当前只是生成后丢弃。
- 斧头钝 -> 打磨斧头、缺木头 -> 获取木头这类动态链路还不能真正进入执行栈。

修复要求：

1. `ExecutionResult` 增加 `generated_subgoals` 或直接将子目标转换为 `GoalFrame` push 入栈。
2. 测试必须断言目标栈新增了子目标，而不是只断言 `requires_reasoning == true`。

### P0-3：暂停恢复没有恢复 active_driver_state

现象：

`pauseFrame` 会把 `active_driver_state` 保存进 `PausedExecutionContext`，但 `resumeFrame` 只改 `GoalFrame.status` 和 `active_frame_id`，没有把 `PausedExecutionContext.driver_state` 恢复到 `active_driver_state`。

位置：

- `backend/src/goal_execution/goal_execution_system.cpp:374`
- `backend/src/goal_execution/goal_execution_system.cpp:394`

影响：

- 外界打断处理完成后，原任务可能恢复了 frame，但没有恢复具体 Driver 状态。
- “狼退走后继续伐木”在真实执行层可能只能恢复目标名，不能恢复伐木步骤。

修复要求：

1. `resumeFrame` 必须从对应 `PausedExecutionContext` 恢复 `driver_state`。
2. 恢复后应移除或标记 paused context，避免重复恢复。
3. 测试必须断言 `active_driver_state` 被恢复，且 `driver_kind == ChopWood`。

### P1-1：MaterialRequirementEvaluator 对 state tag 的检查不可信

现象：

`MaterialRequirementEvaluator` 当前从 `WorldObjectInstance::tag_keys` 检查状态标签，但现有世界对象状态大量存在于 `numeric_states` / `runtime tag_states` / dialog runtime 中。P40 单测使用的是人工构造对象，不能证明真实 H5 世界里的 wet/dry、sharp/dull、processed/raw 都能正确进入材料判断。

位置：

- `backend/src/goal_execution/goal_execution_system.cpp:482`

影响：

- “湿木头不能当干燃料”“钝斧需要打磨”等在真实玩法中可能出现双轨判断。

修复建议：

1. 明确 `WorldSnapshotAdapter` 如何把 dialog runtime 状态投到 P40 所需 `tag_keys`。
2. 增加从真实 H5/world snapshot 生成材料状态的集成测试。

### P1-2：ReactionMaterialResolver 仍有硬映射风险

现象：

`productKey` / `patternObjectKey` 里有大量硬编码：`def_torch -> torch`、`def_wood_processed -> wood_processed`、tag 到对象 key 的映射等。

位置：

- `backend/src/goal_execution/goal_execution_system.cpp:58`
- `backend/src/goal_execution/goal_execution_system.cpp:67`

影响：

- 这违背“后续加物品要像加喝水一样简单”的目标。
- 当前可作为 P40 最小桥接，但必须标注为临时 adapter，后续要读取配置/对象定义注册表。

修复建议：

1. 当前阶段至少补注释和测试边界，说明这是 P28 fixture bridge，不是最终物品映射。
2. 后续阶段迁移到配置化 object definition / reaction content pack 查询。

### P1-3：自审报告测试路径写错

现象：

自审报告写的是：

```text
ctest --test-dir backend/build
```

实际本仓库使用的是：

```text
ctest --test-dir build/backend
```

位置：

- `doc/review/P40动态目标执行与外界打断系统自审报告.md`

影响：

- 后续研发照报告复现会失败。

修复要求：

- 改成实际路径 `build/backend`。

## 建议新增测试

1. `h5_execution_status_visible_when_goal_execution_runs`：H5 response 里 `execution_status.visible == true`，并包含 current_goal / active_step。
2. `internal_blocker_pushes_subgoal_frame`：钝斧阻塞后目标栈新增打磨子目标。
3. `resume_restores_driver_state`：狼打断伐木后恢复，active driver 仍是 ChopWood。
4. `material_tags_from_dialog_runtime_snapshot`：从真实 dialog/world adapter 生成的状态能被材料系统识别。
5. `reaction_material_resolver_no_unknown_material_for_core_rules`：核心规则不应产出 `unknown_material` 输入链。

## 最终判断

P40 的“模块骨架、类型、单测”已经打下来了，方向是对的；但目前更像**后端执行模块最小雏形**，还没有完全达到“玩家可见、可持续执行、可恢复”的 P40 验收要求。

建议修复 P0 问题后再复审。
