# P27 受限条件表达式重构审核报告

## 1. 审核结论

结论：通过。

P27 已将旧 `condition_key / condition_keys / KnowledgeCondition` 条件入口接入统一受限条件表达式系统。当前实现满足 P28 对象反应继续使用 `ConditionExpressionRef`、`ConditionEvaluationContext`、`ConditionExpressionEvaluator` 的前置要求。

本次复审确认：

```text
1. 已新增独立 pathfinder_condition 模块。
2. 旧条件 key 通过 LegacyConditionAdapter / ConditionNormalizer 规范化为 canonical condition key。
3. 条件判断权威入口统一为 ConditionExpressionEvaluator。
4. KnowledgeCondition 保留旧 condition_key 兼容输入，同时新增 condition_ref / canonical_condition_key。
5. 知识形成、知识修正、学习反馈、知识传播、文明能力、H5 展示均已接入 P27 条件系统。
6. H5 条件展示不再输出 raw debug key，改用安全中文摘要。
7. P26 文明能力条件通过 ConditionNormalizer + ConditionExpressionEvaluator 计算。
8. 已补 P27 单元测试、迁移流测试和 gate scan 防回退测试。
9. 全量 CTest 通过：786/786。
```

## 6. 剩余建议

非阻断建议：

```text
1. P28 对象反应规则直接使用 ConditionExpressionRef，不再新增对象反应专用条件系统。
2. 后续如果需要复杂表达式，应扩展 ConditionExpressionNode / registry，而不是在业务系统恢复 condition_key 字符串分支。
3. 可在 P28/P29 增加更多 evaluator trace 展示测试，方便调试对象反应和危险探索。
4. 当前 P27 已保留旧 condition_key 兼容字段；后续存档迁移可逐步补 canonical_condition_key。
```

结论：P27 可进入 P28。
