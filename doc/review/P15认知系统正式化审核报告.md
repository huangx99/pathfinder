# P15 认知系统正式化审核报告

审核日期：2026-05-17  
审核对象：P15 认知系统正式化实现  
审核结论：暂不通过，需修复后复审

---

## 1. 总结论

P15 当前实现方向正确，模块边界整体符合设计：

- 已新增 `CognitionClaimV2`、`CognitionEvidenceRecord`、`CognitionStateV2`。
- 已新增 `CognitionConfidenceModel`、`CognitionUpdaterV2`、`CognitionEvidenceBuilder`、`CognitionQueryService`。
- 已通过 P3 legacy evidence 桥接到 P15 V2 cognition。
- 未发现记忆、知识、传播、H5、HTTP、SaveManager、强化学习 reward/done 等越界实现。
- 构建、P15 定向测试、相关回归、全量测试均通过。

但本次审核发现一个 P15 核心数据一致性缺口：`CognitionClaimV2::validateBasic()` 没有落实设计文档要求的证据计数一致性校验，也没有覆盖对应失败测试。

P15 是后续 P16 记忆系统的输入基础。如果 claim 的证据计数可以处于非法状态，后续记忆强化、知识晋升、可传授候选判断都会建立在不可信数据上。

因此结论为：暂不通过，修复后复审。

## 5. 阻塞问题

### BLOCKER-1：`CognitionClaimV2::validateBasic()` 未校验证据计数一致性

设计文档明确要求：

```text
evidence_count = supporting + refuting 或大于等于二者之和。
```

设计位置：`doc/41_P15认知系统正式化任务卡设计.md:789`

当前实现只校验了：

- key 是否合法。
- confidence 是否在 0 到 1。
- utility_score 是否在 -1 到 1。
- conflicted=true 时 polarity 是否为 Mixed。
- reason_keys / warning_keys 是否包含 hidden truth。

实现位置：`backend/src/cognition/cognition_v2_types.cpp:500`

缺失校验：

```cpp
if (evidence_count < supporting_evidence_count + refuting_evidence_count) {
    return fail(...);
}
```

当前风险：

```text
supporting_evidence_count = 3
refuting_evidence_count = 2
evidence_count = 1
```

这种明显非法的 claim 目前可以通过 `validateBasic()`。

为什么阻塞：

1. P15 是 P16 记忆系统的输入基础。
2. 证据计数会影响置信度、冲突判断、可传授候选和未来知识晋升。
3. 如果非法 claim 能进入 `CognitionStateV2`，后续系统会把错误认知当作可靠事实处理。
4. 这是设计文档明确写出的校验规则，不是增强建议。

必须修复：

- 在 `CognitionClaimV2::validateBasic()` 中增加 evidence count 一致性校验。
- 在 `backend/tests/unit/cognition/cognition_v2_types_test.cpp` 中增加失败用例。

建议测试：

```text
claim.evidence_count = 1
claim.supporting_evidence_count = 1
claim.refuting_evidence_count = 1
claim.validateBasic() 必须失败
```

## 6. 建议修复项

以下不作为本次阻塞，但建议一起修，成本很低。

### SUGGESTION-1：`CognitionClaimV2` 建议校验 `claim_id` 非空

当前 `CognitionClaimV2::validateBasic()` 没有检查 `claim_id`。

位置：`backend/src/cognition/cognition_v2_types.cpp:500`

原因：

- P15 设计已经定义 `CognitionClaimV2Id`。
- `CognitionStateV2::findById()` 依赖 claim id。
- 后续 P16 记忆引用、P18 知识证据引用会需要稳定 claim id。

建议：

```text
claim_id.empty() -> validateBasic 失败。
```

### SUGGESTION-2：`CognitionEvidenceRecord` 建议校验 `evidence_id` 与 `source_event_id` 非空

当前 `CognitionEvidenceRecord::validateBasic()` 没有检查：

- `evidence_id` 非空。
- `source_event_id` 非空。
- `outcome_signal != Unknown`。

位置：`backend/src/cognition/cognition_v2_types.cpp:468`

原因：

- P15 强调认知只来自反馈和事件。
- P16 记忆系统需要可追踪来源。
- 没有 evidence id / source event id，会削弱复盘和审计能力。

建议至少补：

```text
evidence_id.empty() -> 失败。
source_event_id.empty() -> 失败。
outcome_signal == Unknown -> 失败。
```

### SUGGESTION-3：`CognitionStateV2::upsertClaim()` 建议先校验 claim

当前 `upsertClaim()` 直接插入或替换，不校验 claim。

位置：`backend/src/cognition/cognition_v2_state.cpp:20`

原因：

- 虽然 `CognitionUpdaterV2` 正常路径会生成合法 claim，但公开方法本身仍可能被测试、工具或后续代码直接调用。
- P15 是基础模块，状态入口应尽量自保护。

建议：

```text
将 upsertClaim 改为返回 Result<void>，或新增 checkedUpsertClaim。
```

如果不想改接口，至少在任务说明中明确：`upsertClaim()` 仅限内部可信路径使用。

## 8. 最终判定

P15 暂不通过。

原因：核心 DTO `CognitionClaimV2` 的证据计数一致性校验未按设计实现。

修复 BLOCKER-1 后，可以进入复审。若同时补上三个建议项，P15 会更适合作为 P16 记忆系统的稳定输入基础。
