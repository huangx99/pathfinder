# P53 客户端协议与自动化验证最终复审报告

## 1. 复审结论

结论：**仍不通过，需要继续修复后再验收**。

本轮已经确认上一次复审指出的 `available_commands` command response 后未写回 session snapshot 问题已修复：

```text
ClientSessionGateway 已新增 updateAvailableCommands(session_id, options)。
ClientCommandGateway 在构造 response.available_commands 后会写回 session snapshot。
command_updates_snapshot_authority / client_id_mismatch_blocked 手动执行通过。
```

但是本次复审新增发现 3 个阻塞问题：

```text
1. Blocked / Failed 安全响应不是合法 wire response，无法通过 ClientProtocolCodec 编码。
2. 多 session 共用 WorldCommandPipeline 时 projection_version 会串线，第二个 session 会执行后才被标记 full refresh。
3. WorldCommandDto 模式仍可携带 allow_hidden_debug / recipient_claim_json / force_teach 等不安全字段并成功执行。
```

因此 P53 还不能作为 P54 真客户端的稳定协议层。

## 4. 新增阻塞问题

### 4.1 Blocked 响应无法编码成合法 wire response

严重性：**阻塞**。

位置：

```text
backend/src/client_protocol/client_command_gateway.cpp:99
backend/src/client_protocol/client_command_gateway.cpp:106
backend/src/client_protocol/client_command_gateway.cpp:112
backend/src/world_command/world_command_types.cpp:358
```

问题说明：

```text
ClientCommandGateway::buildBlockedResponse() 只设置 result.result_kind 和 failure_reason_keys。
它没有设置 response.result.command_id，也没有设置 result.command_key / actor_key。
WorldCommandResultDto::validateBasic() 要求 command_id 非空。
ClientProtocolCodec::encodeCommandResponse() 编码前会调用 response.validateBasic()。
```

实际结果：

```text
unknown option_id / client_id mismatch / stale version 等 Blocked 响应在内存测试里能返回 ok，
但一旦走真实 JSON wire codec，就会因为 WorldCommandResultDto.command_id 为空而编码失败。
```

复现摘要：

```text
blocked ok=1 result_cmd_id='' validate=0
encode=0 err=WorldCommandResultDto.command_id cannot be empty
```

为什么阻塞：

```text
P53 要求 Failed / Blocked 也必须返回 safe response。
当前失败路径不能稳定发给客户端，P54 接口会在最常见错误场景下断线或返回 500。
```

必须修复：

```text
1. buildBlockedResponse() 必须生成合法 command_id，例如 blocked_<client_sequence> 或复用 request.command.command_id。
2. OptionId 模式至少填充 command_key="blocked" 或 reason-specific key。
3. 所有 Blocked 路径必须 response.validateBasic().is_ok()。
4. 增加 blocked response codec roundtrip 测试：expired option、stale projection、client_id mismatch、session_not_found。
```

### 4.2 多 session projection_version 串线并在执行后才 full refresh

严重性：**阻塞**。

位置：

```text
backend/src/client_protocol/client_session_gateway.cpp:22
backend/src/client_protocol/client_command_gateway.cpp:160
backend/src/client_protocol/client_command_gateway.cpp:184
backend/src/world_command/world_command_pipeline.cpp:160
backend/src/world_command/world_command_pipeline.cpp:182
```

问题说明：

```text
ClientSessionGateway 为每个新 session 初始化 projection_version=1。
WorldCommandPipeline 内部只有一个全局 projection_version_。
ClientCommandGateway 只检查 request.known_projection_version == session.projection_version。
执行前没有把 pipeline_.projection_version_ 同步到该 session 的 current_version。
```

实际结果：

```text
同一个 pipeline 下：
session_1 bootstrap=1，执行命令后 pipeline 变为 2。
session_2 bootstrap 仍为 1。
session_2 提交 known_projection_version=1 会通过 gateway 前置检查。
pipeline 实际产出 patch base=2/new=3。
gateway 发现 patch base 与 known 不一致后标记 requires_full_refresh=true，但命令已经执行成功。
```

复现摘要：

```text
s1 boot=1 cmd base=1 new=2 refresh=0
s2 boot=1 cmd base=2 new=3 refresh=1 result=Succeeded
```

为什么阻塞：

```text
这重新引入了“projection_version 不匹配仍执行命令”的问题，只是在多 session / 多客户端时触发。
P53 的版本契约必须保证版本不匹配时不执行，而不是执行后让客户端 full refresh。
```

必须修复，二选一：

```text
方案 A：P53 暂时声明 single authoritative world version，bootstrap 新 session 时使用 pipeline.currentProjectionVersion()，所有 session 共享当前版本。
方案 B：每个 session / world 绑定独立 pipeline version，执行前 pipeline_.setCurrentProjectionVersion(session_version)，并保证 response patch base == request.known_projection_version。
```

无论选哪种，都必须新增测试：

```text
同一个 HarnessFixture 中 bootstrap 两个 session。
session_1 执行后推进版本。
session_2 再提交命令时不得出现“result=Succeeded 且 requires_full_refresh=true”。
```

### 4.3 WorldCommandDto 模式仍允许不安全字段注入

严重性：**阻塞**。

位置：

```text
backend/src/client_protocol/client_command_gateway.cpp:23
backend/src/client_protocol/client_command_gateway.cpp:74
backend/src/client_protocol/client_protocol_codec.cpp:75
backend/src/client_protocol/client_protocol_codec.cpp:95
backend/src/client_protocol/client_protocol_codec.cpp:108
```

问题说明：

```text
validateClientCommandDto() 当前只校验 actor_key 与 source。
客户端提交 WorldCommandDto 时，context.allow_hidden_debug 会被 codec 接收。
parameters 是任意 map，recipient_claim_json / knowledge_json / force_teach / force_learn 等 key 不会被拒绝。
```

实际结果：

```text
WorldCommandDto(command_key=wait, allow_hidden_debug=true,
parameters[recipient_claim_json]={}, parameters[force_teach]=true)
仍然通过 gateway 并执行成功。
```

复现摘要：

```text
ok=1 result=Succeeded failures=0
```

为什么阻塞：

```text
P53 文档明确要求 client_request_rejects_recipient_claim_json / force_teach / hidden debug fields。
当前架构扫描只检查 client_protocol 源码是否包含这些字面量，不能阻止客户端运行时通过 JSON parameters 注入。
```

必须修复：

```text
1. ClientCommandGateway::validateClientCommandDto() 拒绝 context.allow_hidden_debug=true。
2. 对 command.parameters 做 denylist，至少拒绝 recipient_claim_json、knowledge_json、force_learn、force_teach、learned、raw_state、hidden_truth、content_key_rule_hint。
3. 如 WorldCommandDto 仅用于测试或高级客户端，建议引入更严格的 allowlist：只允许后端标记为 client_safe 的 command_kind / parameter key。
4. 增加 codec/gateway 集成测试，确保这些字段走 JSON decode 后仍被 Blocked，而不是只做源码 grep。
```

## 7. 最终判断

P53 当前已经比上一轮更接近目标，但仍存在真实客户端会触发的协议断裂：

```text
失败响应发不出去。
多客户端 / 多 session 版本会串线。
高级 WorldCommandDto 入口仍可注入 debug / knowledge / force 参数。
```

因此本次最终复审结论为：**不通过，需修复以上阻塞点后再次复审**。
