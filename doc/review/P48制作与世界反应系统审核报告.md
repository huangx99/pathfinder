## 2. 阻塞问题

### 2.1 `SpawnToInventory` 没有同步创建 `WorldEntityInstance`

当前问题：

- `WorldInventoryRuntime::applySpawnToInventory()` 只向 inventory entries 和 `item_locations_` 写入数据。
- 它没有在 `WorldGridRuntime` / `IWorldEntityLocationPort` 中创建对应的 `WorldEntityInstance`。
- 因此 `changed_entity_ids` 里出现的新物品 id，调用 `world_runtime.findEntity(id)` 会失败。
- `WorldReactionProjectionBridge` 会拿这些 id 构建 entity patch，但 runtime 里没有实体，前端拿不到完整实体投影。

这违反 P48 设计：

```text
SpawnToInventory：把新产物生成到 Actor inventory，并同步 WorldEntityInstance 为 InInventory。
```

修复要求：

1. 给 P45 location port 增加一个明确接口，例如：

```cpp
spawnEntityInInventory(entity_id, entity_key, display_name_key, owner_ref, container_ref, quantity, stack_key, stackable, state_keys, numeric_states, tag_keys)
```

或同等语义的接口。

2. `WorldGridRuntime` 必须创建 / 更新对应 `WorldEntityInstance`：

```text
location_kind = InInventory
coord = nullopt
owner_ref = actor_key 或 inventory owner ref
quantity / stack_key / stackable / state_keys / numeric_states / tag_keys 同步
```

3. `SpawnToInventory` 新建 stack 时必须调用该接口。
4. 合并已有 stack 时必须同步更新已有 WorldEntityInstance 的 quantity / stack data。
5. 增加测试：

```text
world_inventory_spawn_to_inventory_creates_world_entity
world_reaction_craft_output_entity_exists_in_world_runtime
```

断言：

```text
world_runtime.findEntity(output_entity_id).is_ok()
entity.location_kind == InInventory
entity.entity_key == torch
entity.quantity == 1
entity.state_keys contains crafted
```

### 2.2 产物 id 不符合 P48 确定性要求

当前问题：

- `WorldReactionService::buildOutputDrafts()` 先生成了确定性 id：`command_id + "_" + output_key + "_" + index`。
- 但 `applySpawns()` 调用 `SpawnToInventory` 时没有把 `od.entity_id` 传给 `InventoryTransferRequest`。
- `WorldInventoryRuntime::applySpawnToInventory()` 新建 entry 时使用 `request.entity_key + "_spawn_" + state_version_`。
- 然后 `WorldReactionService` 又把 `od.entity_id` 改成 `changed_entity_ids[0]`。

这会让 P48 产物 id 依赖 inventory state_version，而不是 command id。

修复要求：

1. `WorldReactionService::applySpawns()` 调用 `SpawnToInventory` 时必须设置：

```cpp
req.entity_id = od.entity_id;
```

2. `WorldInventoryRuntime::applySpawnToInventory()` 必须优先使用 `request.entity_id`。
3. 如果 `request.entity_id` 为空，fallback 也不能使用 `state_version_`，应使用 transfer_id / entity_key / stack_key 组成的确定性 id。
4. 不能在 P48 中把 output id 改成 state_version 生成的 id。
5. 增加测试：

```text
world_reaction_output_id_is_command_deterministic
world_inventory_spawn_to_inventory_uses_request_entity_id
```

断言：

```text
Craft command_id = craft1 时，torch entity id == craft1_torch_0。
重复同样 draft 生成的 id 稳定。
```

### 2.3 `InputStateMismatch` 被错误折叠成 `InputMissing`

当前测试写了：

```cpp
// Should be InputMissing because wood without processed state is not a match
assert(result.value().failure_kind == WorldReactionFailureKind::InputMissing);
```

这和 P48 设计文档冲突。

P48 失败码设计明确区分：

```text
InputMissing：找不到某个输入材料。
InputStateMismatch：找到对象但状态不符合要求。
```

当前情况是：找到了 `wood`，但缺少 `processed` state，因此应返回 `InputStateMismatch`。

修复要求：

1. `resolveInputs()` 增加 state mismatch 标记。
2. 如果找到相同 object_key，但 state_keys 不包含 required_state，应返回 `input_state_mismatch`。
3. 修改测试期望：

```text
world_reaction_validate_state_mismatch -> InputStateMismatch
```

4. 确保状态不匹配时不消耗材料、不生成产物。

```text
world_reaction_drop_on_map_output_spawns_entity_on_actor_cell

```

断言：

```text

AdjacentMap 材料 quantity 减少或实体 destroyed。

projection patch changed_cells / changed_entities 非空。
```

### 2.5 Projection patch 版本没有推进

当前代码：

```cpp
patch.base_projection_version = combined_version;
patch.new_projection_version = combined_version;
```

P46/P47 的 projection bridge 都是：

```text
new_projection_version = base_projection_version + 1
```

P48 如果 base 和 new 相同，前端可能认为没有新 patch。

修复要求：

1. P48 bridge 改为：

```cpp
patch.new_projection_version = patch.base_projection_version + 1;
```

或至少保证：

```text
new_projection_version > base_projection_version
```

2. 增加测试：

```text
world_command_craft_projection_version_advances
```
