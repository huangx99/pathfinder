## 6. 本次用户反馈修复：动作误识别、跨物体污染、知识重复刷屏

### 6.1 问题现象

用户反馈流程中出现三类问题：

```text
1. 输入“吃石片”，系统却当成“使用石片”。
2. 输入“吃苦叶”，知识错误写成 stone_flake + eat -> no_visible_effect。
3. 连续多次“吃腐烂红果”，red_berry + eat -> poison 被不断追加，查看知识时刷出大量重复行。
```

### 6.2 根因

```text
1. H5 意图解析里存在“石片”这种泛关键词，导致只要文本包含石片就优先解析为 Use。
2. 学习闭环在找不到当前物体记忆摘要时，会回退到上一件物体的摘要，造成苦叶借用了石片摘要。
3. 重复经历会生成多个语义相同但 ID 不同的 KnowledgeClaim，H5 会话层没有做语义去重。
4. 轻微风险被当成 contradiction_count，导致苦叶这种“无明显效果”的经历置信度被压低，无法形成自己的知识。
```

### 6.3 修复方案

文件：

```text
backend/src/h5_dialog/dialog_intent.cpp
backend/src/h5_dialog/dialog_learning_adapter.cpp
backend/src/learning/learning_loop.cpp
backend/tests/unit/h5_dialog/dialog_intent_test.cpp
backend/tests/integration/p22/h5_dialog_learning_story_test.cpp
```

修复点：

```text
1. 意图解析改为“先识别动作词，再识别对象”：吃石片 = Eat + stone_flake，然后由场景规则拒绝。
2. 查看同伴知识优先于普通“知识”解析，避免“同伴知识”被当成“我的知识”。
3. 学习闭环不再回退到上一件物体摘要；没有当前物体摘要时，用当前反馈生成当前物体摘要。
4. 只有高风险反馈 risk_delta >= 0.7 才标记 contradiction_count，轻微风险不再阻止知识形成。
5. H5 会话应用学习结果时，按 owner + subject + action + effect + condition 做 KnowledgeClaim 语义去重。
6. 重复命题保留更强状态：active 优先于 hypothesis，hypothesis 优先于 candidate。
```

### 6.4 修复后行为

真实接口验证：

```text
使用石片：
形成 stone_flake + use -> use_hint [active]。

吃石片：
返回 rejected：你不能对那个东西做这个动作。
不会新增 stone_flake 知识。

吃苦叶：
形成 bitter_leaf + eat -> no_visible_effect [active]。
不会再生成 stone_flake + eat。

连续多次吃腐烂红果：

状态会升级为 active，但不会重复刷屏。

教同伴：
同伴只获得一条 red_berry + eat -> poison [hypothesis]。
```

## 9. 本次用户反馈修复：腐烂条件知识显示为腐烂红果

### 9.1 问题现象

用户反馈知识列表里出现：

```text
red_berry + eat -> restore_hunger [active]（红果：吃后可以缓解饥饿；已确认的有效知识）
red_berry + eat -> poison [active]（红果：吃后会让身体不适或中毒；已确认的有效知识）
```

这会让玩家误以为“同一个普通红果既能缓解饥饿又会中毒”。

### 9.2 根因

系统设计上，腐烂红果不是一个完全独立的新物种，而是“红果 + 腐烂条件”：

```text
decayed_red_berry -> red_berry + condition: decayed
```

这样设计是为了让认知系统可以把“普通红果经验”和“腐烂红果经验”放在同一个红果概念下做修正、冲突和细化。

问题在展示层：之前知识行只展示 subject/action/effect，没有展示 condition，所以 `red_berry + decayed + poison` 被玩家看成了普通 `red_berry + poison`。

### 9.3 修复方案

文件：

```text
backend/src/h5_dialog/dialog_presenter.cpp
backend/tests/unit/h5_dialog/dialog_presenter_test.cpp
backend/tests/integration/p22/h5_dialog_learning_story_test.cpp
```

修复点：

```text
1. 知识行原始 key 增加条件后缀：red_berry{decayed}。
2. 中文说明根据 condition: decayed 显示为“腐烂红果”。
3. 普通红果继续显示为 red_berry 和“红果”。
4. 同伴学习后的知识列表也保留条件显示，避免传授后再次歧义。
5. 测试覆盖展示层、P22 学习流程、重复腐烂红果去重流程。
```

### 9.4 修复后展示

```text

red_berry{decayed} + eat -> poison [hypothesis]（腐烂红果：吃后会让身体不适或中毒；经验假设，需要继续验证）
```

## 10. 本次用户反馈修复：普通红果再次学习不应丢失腐烂红果知识

### 10.1 问题现象

用户按下面顺序操作：

```text
吃红果
吃腐烂红果
吃红果
```

第三次吃普通红果后，原本已经学到的腐烂红果知识丢失：

```text
red_berry{decayed} + eat -> poison
```

不再出现在知识列表里。

### 10.2 根因

腐烂红果在系统中是：

```text
red_berry + condition: decayed
```

第三次“吃红果”形成的是无条件普通红果经验：

```text
red_berry + eat -> restore_hunger
```

旧逻辑在判断知识冲突时，只看 subject/action/effect，没有正确考虑 condition scope，导致“无条件普通红果经验”被当成了对“腐烂红果条件知识”的反驳，进而把腐烂红果中毒知识修掉。

### 10.3 修复方案

文件：

```text
backend/src/learning/learning_loop.cpp
backend/tests/integration/p22/h5_dialog_learning_story_test.cpp
```

修复点：

```text
1. 新增 condition scope 判断。
2. 无条件新经验可以修正无条件旧知识。
3. 带条件新经验可以修正更泛化的无条件旧知识。
4. 无条件新经验不能覆盖带条件旧知识。
5. P22 集成测试补充“吃红果 -> 吃腐烂红果 -> 吃红果”回归用例。
```

### 10.4 修复后结果

真实 1999 HTTP 验证：

```text
吃红果后：
- red_berry + eat -> restore_hunger [active]（红果：吃后可以缓解饥饿；已确认的有效知识）

吃腐烂红果后：
- red_berry + eat -> restore_hunger [candidate]（红果：吃后可以缓解饥饿；候选判断，还不稳定）
- red_berry{decayed} + eat -> poison [hypothesis]（腐烂红果：吃后会让身体不适或中毒；经验假设，需要继续验证）

再次吃红果后：
- red_berry + eat -> restore_hunger [active]（红果：吃后可以缓解饥饿；已确认的有效知识）
- red_berry{decayed} + eat -> poison [hypothesis]（腐烂红果：吃后会让身体不适或中毒；经验假设，需要继续验证）
```
