# P19 知识冲突与修正审核报告

审核日期：2026-05-17  
审核范围：P19 knowledge revision 模块、P18 兼容修改、P19 单元/集成测试、知识仓库与投影兼容、阶段边界扫描。  
审核依据：`doc/45_P19知识冲突与修正任务卡设计.md`、`context_packs/knowledge_p19.md`、`doc/44_P18知识系统任务卡设计.md`、`doc/13_知识系统设计.md`。

## 1. 审核结论

**结论：暂不通过，需修复后复审。**

P19 当前实现已经完成了模块骨架、DTO、Planner、Service、Applier、Repository 兼容、Projection 过滤和测试接入；全量测试也通过。但核心“知识冲突与修正”语义存在 P0 级问题：

1. 同对象、同行为、同关系但 `effect_key` 不同的新证据，会被错误当作支持证据强化旧知识。
2. 条件化冲突创建 specialized claim 时，新知识复制了旧 claim 的 predicate/effect，而不是使用 incoming evidence 的新 effect。
3. 当前测试没有覆盖“新证据效果不同”的核心场景，因此测试通过不能证明 P19 玩法逻辑正确。
4. `KnowledgeRevisionService` 最终返回 `Result::ok(result)` 前缺少统一 `validateBasic()` 防线，后续改动容易再次产生 ok + invalid DTO。

这些问题会直接破坏《先驱者》的核心认知体验：玩家/Agent 从“红果能吃”遇到“腐烂红果有毒”后，系统应形成“腐烂红果有毒”的新知识；当前实现会创建“腐烂红果仍然恢复饥饿”的错误特化知识，或者直接把“红果恢复饥饿”强化。

- DTO 自身 validateBasic 通过，但玩法语义错误。

## 4. 阻塞问题

### BLOCKER-1：`effect_key` 不同的新证据被错误当作支持证据

位置：

- `backend/src/knowledge/knowledge_revision.cpp:29`
- `backend/src/knowledge/knowledge_revision.cpp:35`
- `backend/src/knowledge/knowledge_revision.cpp:38`
- `backend/src/knowledge/knowledge_revision.cpp:398`
- `backend/src/knowledge/knowledge_revision.cpp:400`
- `backend/src/knowledge/knowledge_revision.cpp:402`

问题：

`isMatchingClaim()` 只匹配 owner、subject、action_key、relation_type，没有纳入 `effect_key` 的兼容性判断。随后 Planner 只用 `summary.contradiction_count == 0` 判断是否支持旧知识。

结果是：

```text
旧知识：红果 + 吃 -> 恢复饥饿
新证据：红果 + 吃 -> 中毒
```

只要 summary 没有 contradiction_count，系统就把它当作 `SameClaimSupport`，并执行 `ReinforcedClaim`。

影响：

- “有毒”和“可食用”这种核心认知差异会被吞掉。
- 玩家/Agent 会越来越相信错误知识。
- P19 不能正确支撑后续知识传播；错误知识一旦传播，会污染族群知识。

设计依据：

`doc/45_P19知识冲突与修正任务卡设计.md` 明确要求支持证据必须满足：

```text
新证据与旧 claim 的 subject、action、relation、effect_key 兼容。
```

建议修复：

- 从 `KnowledgeFormationInput` 构造真实 incoming predicate：`target_relation + action_key + effect_key`。
- `isMatchingClaim()` 只负责找候选，不直接判定支持。
- 支持/冲突分类必须显式比较 `existing.predicate.effect_key` 与 `incoming_predicate.effect_key`。
- 如果 relation/action 相同但 effect 不兼容，至少判定为 `WeakContradiction` / `OppositeEffect` / `RiskDiscovered`，不能 Reinforce。
- 补测试：同 subject/action/relation、不同 effect、contradiction_count=0 时不得 Reinforce。

### BLOCKER-2：Specialized claim 复制旧 predicate，导致新知识内容错误

位置：

- `backend/src/knowledge/knowledge_revision.cpp:391`
- `backend/src/knowledge/knowledge_revision.cpp:392`
- `backend/src/knowledge/knowledge_revision.cpp:523`
- `backend/src/knowledge/knowledge_revision.cpp:526`
- `backend/src/knowledge/knowledge_revision.cpp:529`
- `backend/src/knowledge/knowledge_revision.cpp:540`

问题：

Planner 构建冲突信号时：

```text
signal.incoming_predicate = claim.predicate
```

生成 specialized claim 时：

```text
new_claim.predicate = claim.predicate
```

也就是说，新知识不是根据 incoming evidence 创建，而是复制旧知识。

探针已经确认：

```text
旧知识：berry_red + eat -> restore_hunger
新证据：decayed berry_red + eat -> poison
实际新 specialized claim：decayed berry_red + eat -> restore_hunger
```

影响：

- 条件化修正功能表面存在，但生成的知识语义是错的。
- “腐烂红果有毒”会被实现成“腐烂红果恢复饥饿”。
- 旧知识还会被 `Deprecated`，导致正确方向完全反过来：旧知识失效，新错误知识变成有效知识。

设计依据：

`doc/45_P19知识冲突与修正任务卡设计.md` 要求：

```text
创建新 specialized claim。
新 claim 的 conditions 使用 incoming_conditions。
旧 claim.superseded_by_knowledge_id 指向新 claim。
新 claim.supersedes_knowledge_ids 包含旧 claim。
```

这里“新 claim”必须表达新证据的 predicate/effect，否则 specialized 没有意义。

建议修复：

- 增加 helper：从 `KnowledgeFormationInput` 构造 `KnowledgePredicate incomingPredicate`。
- `signal.incoming_predicate` 必须使用 incoming predicate。
- `new_claim.predicate` 必须使用 incoming predicate。
- `makeKnowledgeId()` 生成新知识 ID 时也应使用 incoming predicate，避免不同 effect 生成同一个语义 ID。
- 补测试：条件化冲突中 incoming `effect_key=poison` 时，`created_claims[0].predicate.effect_key == "poison"`。

### BLOCKER-3：冲突分类过度依赖 `summary.contradiction_count`

位置：

- `backend/src/knowledge/knowledge_revision.cpp:398`
- `backend/src/knowledge/knowledge_revision.cpp:400`
- `backend/src/knowledge/knowledge_revision.cpp:407`
- `backend/src/knowledge/knowledge_revision.cpp:418`
- `backend/src/knowledge/knowledge_revision.cpp:429`

问题：

当前分类逻辑基本是：

```text
contradiction_count == 0 -> support
contradiction_count > 0 -> conflict
```

这不符合知识修正的语义。`contradiction_count` 是 summary 层的统计信号，不应替代 predicate/effect 的语义比较。

典型错误：

```text
新证据第一次观察到 poison，历史 summary 里 contradiction_count 可能还是 0。
但 poison 与 restore_hunger 已经是不同 effect，不能当作支持。
```

影响：

- 首次发现风险时容易被错判为支持旧知识。
- P19 的 `OppositeEffect` / `RiskDiscovered` 枚举无法发挥作用。
- 知识修正变成“只相信 summary 计数”，不是“比较新旧知识语义”。

建议修复：

- 分类顺序调整为：先比较 subject/action/relation/effect/conditions，再参考 contradiction_count 和 aggregate_strength。
- `effect_key` 不兼容时，无论 contradiction_count 是否为 0，都不能判定为 `SameClaimSupport`。
- `contradiction_count` 只作为弱/强冲突强度依据，不作为是否冲突的唯一依据。
- 补测试：`effect_key` 不同但 contradiction_count=0 时进入冲突路径。

### BLOCKER-4：测试缺少 P19 最核心的“效果不同”断言

位置：

- `backend/tests/unit/knowledge/knowledge_revision_test.cpp:29`
- `backend/tests/unit/knowledge/knowledge_revision_test.cpp:117`
- `backend/tests/integration/p19/knowledge_revision_flow_test.cpp:193`
- `backend/tests/integration/p19/knowledge_revision_flow_test.cpp:202`
- `backend/tests/integration/p19/knowledge_revision_flow_test.cpp:211`

问题：

现有测试 helper 默认一直使用：

```text
action_key = eat
effect_key = restore_hunger
```

条件化测试只增加了 `state_decayed` condition，但没有把 incoming effect 改成 `poison` 或其他不同效果，因此无法发现 specialized claim 复制旧 predicate 的问题。

影响：

- 当前 31/31 与 548/548 全部通过，但没有覆盖 P19 任务卡最关键的例子。
- 后续研发会误以为 P19 已经能处理“腐烂红果有毒”。

必须补测试：

1. `revision_different_effect_without_contradiction_must_not_reinforce`
2. `revision_condition_mismatch_created_claim_uses_incoming_predicate`
3. `revision_incoming_predicate_recorded_in_conflict_signal`
4. `revision_opposite_effect_or_risk_discovered_classification`
5. `revision_created_specialized_claim_id_uses_incoming_predicate`

验收断言至少包括：

```text
created_claims[0].predicate.effect_key == incoming_effect_key
updated_claims[0].status == Deprecated 或 KeepBoth，不能错误强化
conflict_signals[0].incoming_predicate.effect_key == incoming_effect_key
```

## 5. 非阻塞问题与改进建议

### P1-1：Service 缺少最终统一 validateBasic 防线

位置：

- `backend/src/knowledge/knowledge_revision.cpp:810`

问题：

`KnowledgeRevisionService::revise()` 最终直接：

```text
return Result::ok(result)
```

虽然当前主路径多数结果可以 validateBasic 通过，但任务卡明确要求所有 `Result::ok(dto)` 必须能通过 `validateBasic()`。建议 Service 所有 ok 返回前统一执行：

```text
auto valid = result.validateBasic();
if (valid.is_error()) return Result::fail(valid.errors());
return Result::ok(result);
```

影响：

- 属于工程防线问题，不是当前最主要的玩法语义错误。
- 一旦后续新增事件草稿、状态变更、KeepBoth、MergeDuplicate，容易再次出现 ok + invalid DTO。

### P1-2：no-match plan 用 Reinforce 作为占位语义不清

位置：

- `backend/src/knowledge/knowledge_revision.cpp:355`
- `backend/src/knowledge/knowledge_revision.cpp:357`
- `backend/src/knowledge/knowledge_revision.cpp:628`

问题：

没有匹配旧知识时，Planner 为了让 assessment validate 通过，把 `recommended_strategy` 设为 `Reinforce`，Service 再转成 `Skipped`。

影响：

- 当前结果能通过 validate，但语义不够干净。
- 日后如果 UI/日志/训练样本读取 plan，会误读为“推荐强化”。

建议：

- 给 `KnowledgeResolutionStrategy` 增加正式 `Skip` / `NoRevision`，或允许 no-match assessment 使用特定安全状态。
- 如果暂不改枚举，至少加入 reason/warning 明确 `Reinforce` 是 no-match 占位，不得被执行。

### P1-3：`incoming_evidence.supports_claim` 使用全局 has_conflict，容易污染多 claim 场景

位置：

- `backend/src/knowledge/knowledge_revision.cpp:381`
- `backend/src/knowledge/knowledge_revision.cpp:382`
- `backend/src/knowledge/knowledge_revision.cpp:387`
- `backend/src/knowledge/knowledge_revision.cpp:485`

问题：

当前 `has_conflict` 是整个 matched_claims 级别的布尔值。如果多个旧知识同时匹配，某个 claim 的冲突可能影响其他 claim 的 evidence supports 标记。

建议：

- 每个 claim 独立计算 `claim_has_conflict` / `claim_incoming_supports`。
- patch 的 evidence supports 应来自当前 claim 的分类结果，而不是全局变量。

## 6. 必须修复清单

P19 复审前至少完成：

1. 从 `KnowledgeFormationInput` 正确构造 incoming predicate。
2. 支持/冲突分类必须比较 `effect_key` 兼容性。
3. `effect_key` 不同不能 Reinforce。
4. `KnowledgeConflictSignal::incoming_predicate` 必须记录 incoming predicate。
5. specialized claim 必须使用 incoming predicate/effect。
6. specialized claim 的 ID 生成必须基于 incoming predicate。
7. 为不同 effect、条件化不同 effect、incoming predicate 记录补齐测试。
8. Service ok 返回前增加最终 `validateBasic()` 防线。
9. 重新运行 P19 定向测试和全量回归测试。

## 7. 复审验收标准

修复后需要满足：

```text
旧：berry_red + eat -> restore_hunger
新：berry_red + eat -> poison
结果：不得 ReinforcedClaim。
```

```text
旧：berry_red + eat -> restore_hunger
新：state_decayed + berry_red + eat -> poison
结果：CreatedSpecializedClaim，created claim 的 effect_key 必须是 poison。
```

```text
旧知识 Deprecated 时：
- superseded_by_knowledge_id 指向新 specialized claim。
- 新 specialized claim 的 supersedes_knowledge_ids 包含旧知识。
```

```text
Result::ok(result) 时：
- result.validateBasic().is_ok() 必须为 true。
```

```text
Projection 默认仍不得输出 Deprecated / Disproven。
Repository 默认查询仍不得输出 Deprecated / Disproven。
P18 Formation 仍不得主动创建 Deprecated / Disproven。
```

## 8. 最终判断

P19 当前不能标记完成。

原因不是工程骨架缺失，而是核心修正规则方向错误：它还不能可靠表达“新证据改变了我对物体用途/风险的认知”。在《先驱者》的玩法里，知识修正是从个人经验走向族群传播的关键链路；如果这一层把 `poison` 误写成 `restore_hunger`，后续传播、文明、AI 学习都会建立在错误知识之上。

建议按本报告的 P0 修复项完成后再复审。
