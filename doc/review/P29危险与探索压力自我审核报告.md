# P29 危险与探索压力自我审核报告

## 1. 审核结论

P29 已完成实现、自测和人工自我审核，可以作为后续 P30 存档前置和 P33/P34 最小可玩探索闭环的规则基础。

本阶段没有做成战斗系统，也没有引入 HP、死亡、掉落或隐藏真相；P29 只负责把“探索遇到危险”抽象成可配置、可评估、可学习、可传播、可影响族群状态的安全规则结果。

## 3. 核心架构审查

### 3.1 ThreatProfileBuilder

`ThreatProfileBuilder` 负责把运行时对象或生物投影转成危险系统可理解的 `ThreatProfile`。

当前支持：

- 运行时对象：例如燃烧物、腐烂食物、锋利工具、未知物体。
- 生物投影：例如狼、蜘蛛、外星生物等小型/大型/顶级 Agent 的危险来源。
- 输出只包含安全标签、状态、公开名称、压力、可预测性和严重度，不暴露隐藏真相。

审查结论：

- 该层没有直接读取或修改世界状态。
- 该层没有生成 HP、死亡、掉落、真实伤害等字段。
- 后续增加新物品或新生物时，可通过对象标签、状态和配置规则扩展，不需要重构核心解析器。

### 3.2 HazardResolver

`HazardResolver` 是 P29 的规则权威层，输入 `HazardExposureInput` 和 `HazardRule`，输出 `HazardResolutionResult`。

当前支持：

- 触发类型：探索、靠近、触摸、吃、使用、合成、休息、教学、冲突等枚举。
- 来源类型：环境、对象反应、生物、族群冲突、资源压力、知识冲突等枚举。
- 条件判断：统一通过 P27 `ConditionExpressionEvaluator`，不重新发明 condition_key 判断。
- 防护措施：知识、工具、群体支援、距离、逃生路线、文明能力。
- 输出草稿：伤害压力草稿、恐惧信号、前端安全反馈、危险知识草稿、族群压力草稿、事件草稿、trace。

审查结论：

- 危险规则不直接写角色状态、不直接扣血、不直接杀死对象。
- 所有结果都是 draft / event / trace，方便后续由状态系统或表现层消费。
- 条件表达式是权威来源，避免未来配置、知识、反应、危险各自维护一套条件逻辑。

### 3.3 CountermeasureEvaluator

`CountermeasureEvaluator` 负责判断危险是否被知识、工具、逃生路线、群体协作等降低。

当前支持：

- 工具匹配：根据运行时物品的 object_id、definition_id、object_key、tag、state、resource key 命中。
- 知识匹配：根据知识 id、subject、action、effect、规则效果和威胁定义命中。
- 条件匹配：可使用 P27 条件表达式。
- 逃生路线：通过安全 context key 命中。
- 群体支援：可读取共同作战投影的风险降低值。

审查结论：

- 防护不是写死在某个 UI 或单个 if 里，而是规则化输入。
- 后续“火把吓退狼”“毒蘑菇知识避免误食”“驾校教练知识降低驾驶风险”都可走同一套机制。

### 3.4 DangerKnowledgePlanner

`DangerKnowledgePlanner` 把危险经历转成可进入 P28.5 反应学习链路的 `ReactionKnowledgeDeliveryDraft`。

当前支持：

- 玩家直接经历：`DirectExperience`，置信度 1.0。
- NPC 观察学习：`Observation`，置信度 0.6。
- 教学传播：`Taught`，置信度 0.75。
- 保留 shareable、teachable、npc_learnable 标记。

审查结论：

- P29 没有直接写入 KnowledgeRepository，而是生成学习交付草稿。
- NPC 能否学会、能否被教会由草稿和后续知识系统继续裁决。
- 这和用户要求的“同伴不知道，除非你教他；NPC 也能学习危险知识”保持一致。

### 3.5 TribeDangerBridge

`TribeDangerBridge` 把危险压力草稿聚合成 P23 `TribeStateInput`。

当前支持：

- morale_delta 聚合。
- trust_delta 聚合。
- safety_pressure 聚合。
- resource_pressure 聚合。
- knowledge_conflict_pressure 聚合。
- casualty_pressure 聚合。
- split_risk_hint 以压力形式影响族群风险。

审查结论：

- 该桥没有重复 member_id，因此遵守 P23 的同 tick 输入约束。
- P29 不直接修改 TribeState，而是交给 `TribeStateResolver` 统一处理。
- 后续族群分裂、恐慌、共同作战失败，都能通过族群状态系统继续扩展。

## 4. 与既有系统兼容性

### 4.1 复用 P27 条件表达式

P29 条件全部通过：

- `ConditionExpressionRef`
- `ConditionEvaluationContext`
- `ConditionExpressionEvaluator`

当前测试覆盖了：

- `condition:target_state:eq:rotten`
- `condition:test:eq:*` 间接能力保留
- 条件不满足时返回 `BlockedByCondition`

### 4.2 复用 P28/P28.5 对象反应和配置化物品

P29 不替代 P28/P28.5。

对象是否存在、物品是否燃烧、火把是否可用，仍由对象/反应系统表达；P29 只读取 `ReactionRuntimeObject` 的安全字段，判断它是否能作为防护工具或危险来源。

当前测试覆盖了：

- 腐烂红果作为对象危险来源。
- 火把作为工具防护来源。
- 生成物品后续可作为危险防护对象继续使用。

### 4.3 复用 P18-P21 知识学习传播

P29 只生成危险知识草稿，不直接写知识仓库。

当前测试覆盖了：

- 危险经历生成知识草稿。
- 玩家直接经验形成交付。
- NPC 观察形成交付。
- 教学形成交付。
- 已有知识可以作为危险防护措施。

### 4.4 复用 P23-P25 族群链路

P29 只生成 `TribeDangerPressureDraft`，再由 `TribeDangerBridge` 转成 `TribeStateInput`。

当前测试覆盖了：

- 危险导致 morale 下降。
- 危险导致 safety pressure 上升。
- 危险导致 split risk 上升。
- 不在 P29 内直接修改族群状态。

## 6. 发现并修复的问题

### 6.1 浮点精度断言问题

发现：

- `tribe_danger_bridge_pressure` 初始测试使用 `== -0.3` 比较浮点数。

修复：

- 改为 `std::fabs(value - expected) < 0.000001`。

结论：

- 这是测试断言问题，不是业务逻辑问题。

### 6.2 头文件依赖问题

发现：

- `danger_types.cpp` 使用 `std::tolower`，需要显式包含 `<cctype>`。
- 使用 `std::pair` 场景补充 `<utility>` 更稳妥。

修复：

- 补充标准库 include。

结论：

- 避免不同编译器/平台下隐式 include 导致不稳定。

## 7. 安全边界自审

已人工检查以下边界：

- P29 不输出 HP、死亡、掉落、真实伤害、隐藏真相。
- `containsDangerForbiddenKey` 复用 P27 条件系统禁词，并额外拦截危险系统禁词。
- 前端可见 `DangerFeedbackDraft.safe_text` 和 key 会被校验。
- `HazardTrace` 只保留规则 key、条件 trace key、防护 key 和原因 key。
- 危险系统不直接修改对象、角色、知识仓库、族群状态。
- 条件判断不绕过 P27。
- 族群桥不制造重复 member_id。

## 8. 当前限制

P29 当前刻意不做以下内容：

- 不做完整战斗 AI。
- 不做 HP、死亡、掉落和尸体。
- 不做物品耐久和真实伤害数值。
- 不直接写 KnowledgeRepository。
- 不直接写 TribeState。
- 不直接消费 H5 命令。
- 不做完整剧情演出。

这些限制是为了保证 P29 是规则引擎抽象层，而不是提前膨胀成战斗系统或表现系统。

## 9. 后续建议

P29 完成后，下一阶段可以进入 P30 存档前置。

原因：

- 当前已经有对象反应、学习、知识传播、族群压力、危险探索结果。
- 下一步需要保证这些规则结果可以稳定保存、复盘和回归。
- 存档前置完成后，P33/P34 做最小可玩 H5 会更稳，因为每次探索、学习、教学、危险结果都能持久化和复盘。
