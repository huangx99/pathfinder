## 4. P28 设计复审

结论：**P28 设计通过。**

通过原因：

```text
1. P28 明确不是完整物理/化学模拟，而是规则驱动对象反应系统。
2. P28 明确依赖 P27：ObjectReactionRule.condition_refs 使用 ConditionExpressionRef。
3. P28 明确 ObjectReactionResolver 调用 ConditionExpressionEvaluator。
4. P28 明确 ReactionConditionContextBuilder 只构造安全上下文，不判断条件。
5. P28 明确 resolver 只输出 ReactionObjectChangeDraft / FeedbackDraft / KnowledgeDraft / EventDraft，不直接改 WorldObject。
6. P28 明确火 + 干木棍 -> 火把、火 + 湿木棍 -> 条件失败、水 + 火 -> 熄灭为最小验收场景。
7. P28 明确 ReactionKnowledgeDraft 使用 KnowledgeCondition，并填充 condition_ref / canonical_condition_key。
8. P28 明确 H5 只消费 ReactionResult / feedback，不推导反应结果。
9. P28 明确需要 gate scan 防止 reaction 模块绕过 P27 condition 模块。
```

P28 实现时必须继续遵守：

```text
required_tag_keys 只能用于对象类别粗筛，例如 fire_source、combustible、container_like。
dry / wet / burning / poisoned / cooked / raw 这类会改变规则结果的状态语义，必须进入 ConditionExpressionRef。
source_state / target_state / source_tag / target_tag canonical domain 必须在 P28A 作为 P0 扩展到 P27 evaluator。
不能在 reaction resolver 里临时写 dry/wet/burning 分支判断。
```
