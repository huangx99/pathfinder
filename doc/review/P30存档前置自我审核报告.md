## 3. 核心实现审查

### 3.1 SaveGamePackage

`SaveGamePackage` 是 P30 的核心逻辑包，包含：

- `SaveHeader`：存档元信息、版本范围、配置版本和 package hash。
- `SaveIndex`：每个 section 的快速索引、记录数量、版本范围和 section hash。
- `StateSnapshotSection`：权威 GameState 快照。
- `EventLogSection`：解释状态变化原因的事件日志。
- `CommandLogSection`：P4 `CommandReplayLog`。
- `RandomLogSection`：P4 `RandomReplayLog`。
- `PipelineTraceSection`：pipeline stage、event_id、condition trace、安全 reason key。
- `ConfigVersionManifest`：配置版本依赖。
- `IntegrityManifest`：各段 hash 和 package hash。
- `CrashRecoveryJournal`：崩溃恢复预留结构。

审查结论：

- 包结构与 `doc/19_存档与复盘设计.md` 一致。
- 快照、事件、命令、随机、trace、配置、完整性各自独立，不混用职责。
- 没有通过事件日志反推权威状态，符合“Snapshot 负责恢复，EventLog 负责解释”的原则。

### 3.2 SavePackageBuilder

`SavePackageBuilder` 负责从输入段构建完整存档包。

已实现：

- 校验 header、snapshot、event、command、random、trace、config、journal。
- 自动生成 `SaveIndex`。
- 自动生成 `IntegrityManifest`。
- 自动写入 `header.package_hash`。
- 校验 command replay 与 random replay 的 command_id 对齐。
- 校验 StateVersion 范围不能超出 header。

审查结论：

- builder 不修改 GameState，不写文件，不绕过规则管线。
- package_hash 不依赖调用方手工填写，避免研发手填错误。
- command/random 对齐检查可以支撑后续确定性 replay。

### 3.3 SavePackageValidator

`SavePackageValidator` 负责判断存档包是否可信。

已实现：

- 基础结构校验。
- 必要 section 缺失检查。
- `IntegrityManifest` 重新计算和比较。
- `header.package_hash` 与实际 package hash 比较。
- `SaveIndex` 中 section_hash 与实际 section hash 比较。
- sealed=false 拒绝。
- snapshot 缺失、config hash 缺失、版本范围错误、hash 被篡改均会失败。

审查结论：

- validator 不是只信任包内保存的 hash，而是重新计算。
- index 也被纳入完整性检查，避免索引和真实 section 不一致。
- 这是后续文件存档和云存档之前必须有的安全门。

### 3.4 SaveRestorePlanner

`SaveRestorePlanner` 负责生成恢复计划，不直接恢复运行时。

已实现步骤：

1. `VerifyIntegrity`
2. `LoadConfigManifest`
3. `LoadSnapshot`
4. `ApplyEventLog`
5. `AttachReplayLogs`
6. `AttachTraceLog`
7. `Ready`

审查结论：

- planner 只输出计划，不修改 GameState。
- planner 会选择目标 StateVersion 之前最近的 snapshot。
- target version 超出 package 范围会失败。
- 如果存在 recovery journal，会输出 warning key，便于后续恢复工具处理。

## 4. 与现有系统兼容性

### 4.1 P4 replay 兼容

P30 直接复用：

- `CommandReplayLog`
- `CommandReplayRecord`
- `RandomReplayLog`
- `RandomReplayEntry`
- `ReplayRunner`

集成测试验证：

### 4.2 P2 state/event/pipeline 兼容

P30 保存：

- `GameState` 快照。
- `EventRecord` 事件日志。
- pipeline stage key 和 event_id trace。

P30 不做：

- 不直接执行 pipeline。
- 不应用 state change。
- 不修改对象、角色、认知、知识或族群状态。

### 4.3 P27 condition trace 兼容

P30 的 `PipelineTraceRecord.condition_trace_keys` 可以保存：

- `expression_id`
- `canonical_condition_key`
- 安全 condition trace key

同时禁止：

- `hidden_truth`
- `true_hp`
- `raw_state`
- `death`
- `loot`
- `direct_state`

### 4.4 P29 danger 兼容

P30 不重算危险，不直接解释危险结算，只保存 P29 后续产生的事件、trace、命令、随机、快照和配置引用。

这保证危险系统仍是规则权威，存档系统只负责保存与校验。

## 6. 人工自审发现与处理

### 6.1 command/random 对齐

风险：

- 只有 command log，没有对应 random log，会导致后续 replay 不确定。

处理：

- `SavePackageBuilderInput::validateBasic()` 已检查每条 command replay record 必须有匹配 random log entry。
- random log entry 引用未知 command_id 会失败。

### 6.2 index hash 与真实 section hash

风险：

- index 中的 section_hash 与真实 section 不一致，会误导后续快速加载。

处理：

- `SavePackageValidator` 已重新计算每个 section hash，并与 index entry 比较。

### 6.3 package_hash 手填风险

风险：

- 研发误填 package_hash 或直接复制旧 hash。

处理：

- builder 自动计算 `IntegrityManifest` 和 `header.package_hash`。
- validator 重新计算，不信任手填值。

## 7. 当前限制

P30 当前刻意不做：

- 不做真实文件写入。
- 不做 JSON / 二进制序列化。
- 不做云同步。
- 不做压缩。
- 不做加密签名。
- 不做跨版本迁移。
- 不做崩溃恢复执行器。
- 不做 H5 存档 UI。

这些限制符合“前置实现”的定位，避免过早绑定具体存储方案。
