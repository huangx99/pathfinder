# P45 背包、容器、掉落物与物品归属系统审核报告

审核对象：P45 实现与设计文档

- 主设计：`doc/75_P45背包容器与物品归属系统设计.md`
- 压力测试：`doc/75A_P45背包容器与物品归属系统压力测试.md`
- 核心实现：`backend/include/pathfinder/world_inventory/`、`backend/src/world_inventory/`
- 测试：`backend/tests/unit/world_inventory/`、`backend/tests/integration/world_inventory/`

## 1. 审核结论

结论：不通过，存在设计级阻塞项。

虽然当前 P45 编译通过，且现有 25 个 P45 相关测试全部通过，但实现没有完整满足 P45 设计文档中最关键的“归属权威、可见性硬门禁、状态堆叠、掉落物一致性、材料来源证明基础”要求。

这不是简单测试小问题，而是会影响后续 P46/P47/P48/P51 的基础架构问题。建议先修复后再验收 P45。

## 3. 阻塞问题

### P0-1：Pickup 没有真正校验可见性

设计要求：

- `doc/75_P45背包容器与物品归属系统设计.md` 明确要求 Pickup 不能只按 `entity_id` 成功。
- 目标格必须对 actor 可见，P45 推荐最小策略是目标格必须 `Visible`。
- 压力测试也要求未知格子或不可见格子返回 `target_not_visible` 或 `target_too_far`。

当前实现：

- `backend/src/world_inventory/world_inventory_runtime.cpp` 的 `validatePickupFromMap` 只调用 `findCell` 确认格子存在。
- 只检查同格或相邻。
- 没有调用 `WorldGridRuntime::isCellVisibleToActor`。
- `IWorldEntityLocationPort` 也没有提供可见性查询接口，导致 P45 runtime 从接口层就无法严格实现可见性门禁。

风险：

- 恶意客户端只要知道相邻或同格 entity_id，就可能绕过探索可见性。
- 后续迷雾、潜行、地下、隐身、地图未探索区域都会被这个漏洞影响。

修复建议：

- 在 `IWorldEntityLocationPort` 增加 `isCellVisibleToActor(actor_key, coord)` 或 `getCellVisibilityForActor`。
- `validatePickupFromMap` 必须先校验目标格可见。
- 增加真实测试 `world_inventory_pickup_requires_visible_target`，不是只测 too_far。

### P0-2：堆叠系统没有实现设计中的 StackKey 状态隔离

设计要求：

- `stack_key = entity_key + normalized_state_signature`。
- 普通红果和腐烂红果不能合并。
- 耐久不同工具不能合并。
- `stackable=false` 时 `quantity` 必须为 1。
- 状态不同的物品不得自动合并。

当前实现：

- `applyPickupFromMap` 固定调用 `makeStackKey(entity->entity_key, {}, {})`。
- `InventoryItemEntry::state_keys` 和 `numeric_states` 没有从物品实体、配置、状态系统中读取。
- `entry.stackable = true` 被硬编码为默认值。
- 当前 `WorldEntityInstance` 本身也没有可承载物品状态、耐久、腐烂程度的字段。

风险：

- 腐烂红果和普通红果只要 `entity_key` 相同就会合并。
- 不同耐久斧头、不同铭文工具、不同归属标记物品可能错误合并。
- 这是后续“吃多了有毒、腐烂状态、工具磨损、魔法附魔、机械零件损坏”的基础风险。

现有测试问题：

- `world_inventory_stack_key_separates_states` 只验证 `stack_key` 非空，没有制造两件同 key 不同状态的物品。
- `world_inventory_non_stackable_not_merged` 是手动往 inventory vector 塞数据，没有走 runtime transfer，所以没有验证真实逻辑。

修复建议：

- P45 至少需要定义 `ItemStackTraits` 或从内容配置/实体 tags 生成 `state_keys`、`numeric_states`、`stackable`。
- `applyPickupFromMap` 不能固定 `stackable=true`。
- 测试必须走真实 pickup 流程覆盖：普通红果/腐烂红果、耐久 80/耐久 20 斧头、stackable=false。

### P0-3：Drop partial stack 会产生错误或不完整的地图实体

设计要求：

- 丢弃部分堆叠后，背包剩余数量正确。
- 地上出现被丢弃数量。
- 如果地上已有同 `stack_key` 物品，应合并。
- TransferResult 必须记录 source entry update 和 created/destination update。

当前实现：

- `applyDropToMap` 部分丢弃时生成 `dropped_entity_id = request.entity_id + "_drop_" + state_version_`。
- `WorldGridRuntime::placeEntityOnMap` 如果实体不存在，会根据 entity_id 第一个下划线前的字符串推断 `entity_key`。
- 对 `red_berry_001_drop_1` 这类 id，会错误推断 `entity_key = red`，不是 `red_berry`。
- 地图实体没有记录 `quantity`、`stack_key`、`state_keys`、`numeric_states`。
- 没有实现地面已有 stack 的合并。

风险：

- 背包里是红果，丢到地上可能变成 key 错误的实体。
- 部分掉落的数量只存在于 inventory 逻辑里，地图实体本身没有权威数量。
- 后续采集、制作、NPC 拾取、地图显示都会出现“地上到底有几个”的不一致。

现有测试问题：

- `world_inventory_drop_partial_stack` 只检查背包剩余数量，没有检查地上新实体的 `entity_key`、数量、位置、stack_key。

修复建议：

- 不允许 `placeEntityOnMap` 从字符串猜 `entity_key` 作为权威逻辑。
- drop 需要显式传入 `entity_key`、`stack_key`、状态与数量，或者新增创建地面 stack 的专用 port。
- 增加测试验证地面实体 key、stack、quantity 与 changed ids。

### P0-4：拾取合并多个 map entity 后会产生“无 entry 的 InInventory 实体”

当前实现流程：

1. 第一个红果 pickup，生成 entry，entry.entity_id = 第一个实体 id。
2. 第二个同 key 红果 pickup，因为 stack_key 相同，数量合并到第一个 entry。
3. 第二个实体被 `removeEntityFromMap`，并 `setEntityLocationKind(InInventory)`。
4. `item_locations_[第二个实体id] = InInventory`。
5. 但 inventory entries 中没有第二个实体 id 对应的 entry。

风险：

- item_locations 显示第二个实体在背包，但 inventory entries 不包含它。
- 后续 Drop 全栈只会处理第一个 entry/entity，第二个实体可能永久处于 InInventory/无 entry 状态。
- 这破坏了 P45 “物品归属权威”和“地图/背包不能双重存在”的基础一致性。

修复建议：

- 选择一种权威模型：
  - 要么 stack entry 代表一组数量，不再保留每个实体 instance；
  - 要么每个实体都有独立 entry，不做数量合并；
  - 要么引入 ground/inventory stack entity，并明确 member entity 生命周期。
- P45 设计偏向 StackKey + GroundStack，因此建议让 stack entry 成为数量权威，entity instance 作为 stack carrier，而不是让多个 entity 残留在 InInventory。

### P1-1：validate 阶段会修改 runtime 状态

设计要求：

- P45 要体现 validate/apply 两阶段。
- validate 应该尽量只验证并生成 draft，apply 才改变世界状态。

当前实现：

- `validatePickupFromMap` 会调用 `ensureActorInventory`，可能创建 actor inventory 并增加 state_version。
- `validatePickupFromMap` 会调用 `ensureContainerInventory(request.entity_id)`，会给任意被拾取物创建 container inventory。
- 即使后续校验失败或 transfer 没有 apply，也可能污染 inventory runtime。

风险：

- 失败命令也可能改变状态版本或创建空库存。
- 任意物品都会被创建一个 container inventory，导致容器语义污染。
- 后续存档、同步、回放、事务撤销会很难保证一致。

修复建议：

- validate 中不要调用会创建状态的 `ensure*`。
- 拆分为 `findInventory` / `buildCreateInventoryDraft` / `applyCreateInventory`。
- container 判断必须基于实体配置/tag，而不是通过 ensureContainerInventory 反推。

### P1-2：容器语义没有可靠实现

设计要求：

- P45 不支持拾取非空容器。
- 如果 target entity 是 container，且 container inventory 非空，Pickup 返回 `container_not_empty`。

当前实现：

- 没有判断目标 entity 是否真的是 container。
- `ensureContainerInventory(request.entity_id)` 会为任意实体创建一个 container inventory。
- 空容器默认是否可拾取没有接配置策略。

风险：

- 普通红果、木头也会被当成“有一个空容器库存”的实体。
- 后续真实箱子、木堆、建筑库存、尸体掉落容器会难以区分。

修复建议：

- WorldEntityInstance 或内容配置需要提供 container trait/tag。
- 只有 container trait 的实体才检查 container inventory。
- 空容器策略要由配置控制。

### P1-3：InventoryProjection entries_json 是手拼 JSON，没有转义

当前实现：

- `inventory_projection_adapter.cpp` 使用 `ostringstream` 拼 JSON。
- 字符串字段没有 escape。

风险：

- 如果后续 `entity_id`、`entity_key`、`stack_key` 来自 JSON 配置或 mod，包含引号、反斜杠、换行时会破坏 JSON。

修复建议：

- 增加最小 JSON escape helper。
- 或引入项目已有 JSON 序列化方式。
- 增加包含特殊字符的 entries_json 测试。

### P1-4：projection version 只使用 world_runtime.stateVersion，未纳入 inventory stateVersion

当前实现：

- `buildFullProjectionPatch` 设置 `patch.new_projection_version = world_runtime.stateVersion()`。
- 但 P45 背包变化会改变 inventory runtime，不一定只靠 world runtime 版本表达。

风险：

- 客户端如果用 projection version 判断是否应用更新，inventory-only 变化可能版本语义不完整。

修复建议：

- 明确 projection version 来源：世界版本 + 背包版本，或由 command pipeline 统一生成。
- 至少在设计文档里写清楚 P45 patch version 规则。

### P1-5：测试覆盖不足，当前通过不能代表 P45 通过

缺失或不足测试：

- `world_inventory_pickup_requires_visible_target` 没有真实实现。
- `world_inventory_container_not_empty_blocked` 未加入 CTest。
- `world_command_drop_projection_entries_json` 未加入 CTest。
- `world_inventory_stack_key_separates_states` 没有验证状态隔离。
- `world_inventory_non_stackable_not_merged` 没有走 runtime。
- `world_inventory_drop_partial_stack` 没有验证地面实体。
- 没有验证 validate 失败不改变 stateVersion。
- 没有验证重复 pickup 合并后 item_locations 与 entries 一致。

## 4. 非阻塞但建议修复

### P2-1：错误映射不完整

`validateDropToMap` 返回 `drop_cell_invalid`，但 `transfer` 的 failure_kind 映射表没有映射它，最终会落到 `RuntimeMismatch`。

建议新增 `InventoryFailureKind::DropCellInvalid` 或映射为已有明确错误。

### P2-2：GroundStack helper 未真正使用

`findGroundStackEntity` 已存在，但没有被 drop 逻辑用于合并地面 stack。

建议要么实现，要么标注 TODO，避免误导后续研发以为已经接入。

### P2-3：SourceInventoryProof 只是结构体，没有形成可用接口

P45 设计允许先不实现制作，但后续 P48 需要材料消耗证明。当前 `queryItems` 返回 entries，能辅助构造证明，但没有封装 `buildSourceProof` 或 `querySourceInventory`。

建议 P45 修复时至少补一个只读 helper 或文档 TODO。

## 5. 建议修复顺序

1. 先修复 item ownership 权威模型：明确 stack entry 与 entity instance 的关系。
2. 修复 partial drop：不再从 entity_id 猜 key，地面 stack 必须有数量/状态权威。
3. 修复 pickup 可见性：扩展 `IWorldEntityLocationPort`，接入 `isCellVisibleToActor`。
4. 修复 validate 不得写状态：拆分 ensure/find/apply create。
5. 修复 StackKey：接入 item trait/state 来源，禁止默认全部 stackable=true。
6. 补强测试：让测试通过真实 command/runtime 流程，而不是手动改 vector。
7. 修复 entries_json escaping 和 projection version 语义。

## 6. 验收建议

P45 目前不能验收。

可以保留已搭好的模块拆分和 command 接入方式，因为方向是对的：

- `world_inventory` 独立模块是对的。
- `world_inventory_command` 独立接入 runtime-aware handler 是对的。
- `pathfinder_world_command` 不依赖 inventory 是对的。
- Pickup/Drop 走 WorldCommandPipeline 是对的。
- `changed_inventories` 已有基础投影是对的。

但 P45 的核心目标是“物品归属权威”，当前实现还没有达到。建议修复上述 P0/P1 后再进入 P46。
