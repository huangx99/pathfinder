## 2. 本轮发现并修复的问题

### 2.1 隐藏问题：模板 key 可能生成非法 KnowledgeId

上一轮修复中，默认知识模板会用：

- `knowledge_default_` + `template_key` + `_` + `session_id`

生成 `KnowledgeId`。

但默认威胁模板曾使用类似：

- `threat.beast.counter.fire`

这种带点号的 key。

根据当前基础 ID 规则，合法 ID 只能包含：

- 小写字母。
- 数字。
- 下划线。
- 中划线。

点号不是合法字符。

虽然当前默认场景没有启用 `default_knowledge_templates`，不会立刻触发，但后续一旦配置默认知识，就可能生成非法 `KnowledgeId`，导致存储、校验或知识系统失败。

### 2.2 修复方案

已修复：

- `DialogDefaultKnowledgeTemplate::validateBasic()` 增加 ID 格式校验。
- `DialogScenarioActionTemplate::validateBasic()` 增加 action/effect/target key 格式校验。
- `DialogScenarioThreatKnowledgeTemplate::validateBasic()` 增加 template/threat/effect/fallback key 格式校验。
- 默认威胁模板 key 从 `threat.beast.counter.fire` 改为 `threat_beast_counter_fire`。
- `DialogScenarioCatalog::defaultScenario()` 返回前执行整体场景校验。
- 测试新增非法点号 template key 校验。

相关文件：

- `backend/src/h5_dialog/dialog_types.cpp`
- `backend/src/h5_dialog/dialog_scenario.cpp`
- `backend/tests/unit/h5_dialog/dialog_scenario_test.cpp`

## 3. 再次确认的架构问题

### 3.1 Story 不再依赖 H5 Dialog

上一轮自审已经发现并修正 `Story -> H5 Dialog` 的反向依赖。本轮再次扫描确认：

- `backend/src/story` 不再 include `pathfinder/h5_dialog/dialog_scenario.h`。
- `pathfinder_story` 不再链接 `pathfinder_h5_dialog`。
- `pathfinder_h5_playable` 同时依赖 `h5_dialog` 与 `story`，这是允许的组合层依赖。

结论：分层方向正确。

### 3.2 H5 Dialog 的场景目录引用属于本层合理依赖

残留扫描中仍能看到 `DialogScenarioCatalog` 被 H5 相关代码引用，例如：

- `dialog_session.cpp`
- `dialog_presenter.cpp`
- `playable_turn_service.cpp`
- `playable_projection_mapper.cpp`

这属于 H5 原型层和 H5 可玩层读取 H5 场景配置，当前合理。

真正要避免的是底层 Story / GoalExecution / WorldEffectExecutor 反向依赖 H5 内容目录。本轮未发现这类问题。
