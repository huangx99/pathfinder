# P45 背包、容器、掉落物与物品归属系统复审报告

复审对象：P45 首轮审核后修复版本

- 首轮审核：`doc/review/P45背包容器与物品归属系统审核报告.md`
- 主设计：`doc/75_P45背包容器与物品归属系统设计.md`
- 压力测试：`doc/75A_P45背包容器与物品归属系统压力测试.md`
- 核心实现：`backend/include/pathfinder/world_inventory/`、`backend/src/world_inventory/`
- 关联世界运行时：`backend/include/pathfinder/world_runtime/`、`backend/src/world_runtime/`

## 1. 复审结论

结论：暂不通过。

本次修复方向正确，首轮审核中的大部分 P0/P1 架构问题已经被处理：可见性门禁、实体状态堆叠、容器识别、地面 stack、entries_json 转义、validate 不创建背包等都有明显补强。

但当前仍存在一个会影响物品丢弃稳定性的 P0 问题：`applyDropToMap` 在 full drop 路径中 erase 背包 entry 后继续使用已失效指针。这是 C++ 未定义行为，必须修复后才能验收 P45。

此外还剩几个 P1/P2 级问题，建议一并处理，避免后续 P46/P48/P51 接入时返工。

## 3. 首轮阻塞项关闭情况

### 3.1 Pickup 可见性门禁：已基本关闭

修复点：

- `IWorldEntityLocationPort` 增加 `isCellVisibleToActor` 与 `getCellVisibilityForActor`。
- `WorldGridRuntime` 实现可见性查询。
- `validatePickupFromMap` 在距离判断前校验目标格必须对 actor 可见。
- 新增 `world_inventory_pickup_requires_visible_target` 测试。

复审评价：方向正确，核心门禁已经补上。

注意：当前测试里有条件分支，如果没有找到 origin item 或移动后仍可见，测试会直接通过。建议后续改成强断言式测试，避免假阳性。

### 3.2 StackKey 状态隔离：已基本关闭

修复点：

- `WorldEntityInstance` 增加 `quantity`、`stackable`、`stack_key`、`state_keys`、`numeric_states`。
- `applyPickupFromMap` 使用实体的 `state_keys` / `numeric_states` 生成 stack_key。
- pickup 合并时同时要求 `entry.stackable && entity->stackable`。
- 新增 runtime 路径测试 `world_inventory_non_stackable_runtime`。

复审评价：基础能力已经具备，能支撑腐烂、耐久、附魔等状态隔离。

保留建议：`world_inventory_stack_key_separates_states` 仍然偏弱，建议后续补一个“同 entity_key、不同 state_keys、stackable=true 也不能合并”的真实 runtime 测试。

### 3.3 地面 stack 与 partial drop：大部分已关闭，但 full drop 有 P0 问题

修复点：

- `IWorldEntityLocationPort` 增加 `spawnEntityOnMap`，不再靠 entity_id 字符串猜 entity_key。
- `WorldEntityInstance` 承载地面数量、stack_key 与状态。
- drop 会尝试合并已有地面 stack。
- 新增 `world_inventory_drop_partial_stack_ground` 测试，验证地面实体 key、quantity、stack_key。

复审评价：partial drop 的方向正确，但 full drop 代码存在未定义行为，见第 4.1。

### 3.4 拾取合并导致幽灵实体：已基本关闭

修复点：

- pickup 合并到已有 inventory entry 后，调用 `destroyEntity` 删除被合并的 map entity。
- 合并实体被删除后，清理 `item_locations_`。
- 新增 `world_inventory_pickup_merge_consistency` 测试。

复审评价：首轮提到的“无 entry 的 InInventory 实体”问题已经处理。

### 3.5 validate 阶段写状态：已基本关闭

修复点：

- `validatePickupFromMap` 不再调用 `ensureActorInventory` 创建背包。
- 容器检查不再对任意实体调用 `ensureContainerInventory`。
- 新增 `world_inventory_validate_no_state_change` 测试。

复审评价：事务分层明显变好。

保留建议：capacity 校验还需要修复，见第 4.2。

### 3.6 容器语义：已基本关闭

修复点：

- 新增 `isContainerEntity`，通过 `container` tag 判断容器。
- 只有容器实体才检查 container inventory 是否非空。
- 新增 `world_inventory_container_not_empty_blocked` 测试。

复审评价：P45 最小容器策略已经可用。

### 3.7 entries_json 转义：已关闭

修复点：

- `InventoryProjectionAdapter` 增加 `jsonEscape`。
- `entries_json` 输出稳定 JSON 字符串。

复审评价：满足 P45 结构化投影要求。

### 3.8 projection version：部分关闭

修复点：

- `buildFullProjectionPatch` 改为 `max(world_runtime.stateVersion(), inventory_runtime.stateVersion())`。

复审评价：比首版更好，但语义仍不完美。当前 pickup/drop 都会改 world runtime，所以短期可接受。后续如果出现 inventory-only command，`max` 可能无法保证投影版本单调递增，需要统一 projection version 生成器。

## 4. 剩余问题

### P0-1：Full drop 路径 erase 后继续使用失效指针

位置：`backend/src/world_inventory/world_inventory_runtime.cpp`

问题代码逻辑：

```cpp
InventoryItemEntry* target_entry = &inv->entries[i];
...
if (request.quantity < target_entry->quantity) {
    target_entry->quantity -= request.quantity;
    dropped_entity_id = ...;
} else {
    inv->entries.erase(inv->entries.begin() + target_index);
    inv->used_slots -= 1;
}
...
world_port_.spawnEntityOnMap(
    dropped_entity_id,
    target_entry->entity_key,
    ...,
    target_entry->stack_key,
    target_entry->stackable,
    target_entry->state_keys,
    target_entry->numeric_states,
    {}
);
```

当 full drop 时，`erase` 会让 `target_entry` 指针失效，后面继续读取 `target_entry->entity_key`、`stack_key` 等字段属于 C++ 未定义行为。

风险：

- 当前测试通过只是因为内存还没被覆盖，不代表逻辑安全。
- 后续 vector 扩容、不同编译器、优化级别、Sanitizer、更多 entry 数量下可能崩溃或生成错误物品。
- 这会直接破坏 P45 的物品归属权威。

修复建议：

- 在修改/erase 前复制一份 `InventoryItemEntry dropped_entry = *target_entry;`。
- 后续 spawn/merge 全部使用 `dropped_entry`。
- full drop 后如果使用原 entity_id，应保证 world entity 从 InInventory 转回 OnMap 时数据来自 `dropped_entry`。
- 增加一个 full drop 明确测试：背包有多个 entry，丢弃中间 entry，验证地面 entity_key/stack_key/state_keys/numeric_states 正确。

### P1-1：容量校验仍然硬编码 20，没有使用 inventory.capacity_slots

位置：`backend/src/world_inventory/world_inventory_runtime.cpp`

当前逻辑：

```cpp
int used_slots = 0;
auto inv_ptr_res = findInventory(actor_inventory_id);
if (inv_ptr_res.is_ok()) {
    used_slots = inv_ptr_res.value()->used_slots;
}
if (used_slots >= 20) {
    return capacity_exceeded;
}
```

问题：

- 如果背包 `capacity_slots = 1`，`used_slots = 1`，这里不会拦截。
- 现有 `world_inventory_capacity_exceeded` 测试可能因为数据不足或分支条件没触发而没有暴露问题。

修复建议：

- 校验逻辑改为读取真实 capacity：`if (inv && inv->used_slots >= inv->capacity_slots)`。
- 如果 inventory 不存在，则使用默认容量 20。
- 增加强断言测试：手动 spawn 两个同格、不同 entity_key 的物品，capacity=1，第二次 pickup 必须 `CapacityExceeded`。

### P1-2：drop_cell_invalid 被映射为 TargetCellMissing，语义错误

位置：`backend/src/world_inventory/world_inventory_runtime.cpp`

当前逻辑：

```cpp
else if (mk == "drop_cell_invalid") result.failure_kind = InventoryFailureKind::TargetCellMissing;
```

问题：

- 目标格存在但不允许丢到非当前格，不是 `TargetCellMissing`。
- 前端和 AI 决策层会收到错误原因，后续可能误以为地图格不存在。

修复建议：

- 新增 `InventoryFailureKind::DropCellInvalid`，或至少映射为更接近的 `ForbiddenByOwner` / `RuntimeMismatch` 不建议。
- 同步更新 `toString` 和测试。

### P1-3：changed_entity_ids 对新生成 drop entity 不完整

位置：`backend/src/world_inventory/world_inventory_runtime.cpp`

当前 `validateDropToMap` 只把 `request.entity_id` 放入 `changed_entity_ids`，但 partial drop 会生成新的 `dropped_entity_id`。

风险：

- command projection 的 `changed_entities` 可能不包含新掉落实体。
- 目前 changed_cells 能让前端刷新格子，但如果前端依赖 entity patch 精确更新，会漏实体。

修复建议：

- apply 阶段生成新 entity id 后，把它补入 draft/result 的 `changed_entity_ids`。
- 或在 validate 阶段就确定 dropped_entity_id，保证 draft 可预测。

### P2-1：可见性测试存在条件跳过风险

位置：`backend/tests/unit/world_inventory/world_inventory_test.cpp`

`world_inventory_pickup_requires_visible_target` 中，如果 `origin_item.empty()` 或目标仍可见，核心断言不会执行，测试仍输出 passed。

修复建议：

- 使用 `spawnEntityOnMap` 在确定坐标创建测试物品。
- 移动 actor 到确定不可见坐标，或设置 vision_radius 辅助测试。
- 不满足前置条件时直接 `assert(false)`。

### P2-2：SourceInventoryProof 仍主要停留在结构体层

当前 `SourceInventoryProof` 已有结构定义，`queryItems` 可辅助后续构造来源证明。P45 最小阶段可以接受。

建议 P48 制作/消耗接入前，补一个只读 helper：

```cpp
querySourceInventory(actor_key, entity_key, scope)
buildSourceInventoryProof(...)
```

避免制作系统自己遍历背包造成重复逻辑。

## 5. 复审验收建议

当前不建议验收 P45。

必须先修：

1. `applyDropToMap` full drop 的失效指针问题。
2. capacity 校验读取真实 `capacity_slots`。
3. `drop_cell_invalid` 错误码语义。
4. partial drop 新实体补入 `changed_entity_ids`。

修完后建议新增/调整测试：

- full drop 多 entry 场景验证 entity_key / stack_key / state_keys / numeric_states。
- capacity=1 的强断言测试。
- drop 非当前格返回明确 `drop_cell_invalid`。
- partial drop projection 包含新掉落实体。
- 可见性测试去掉条件跳过。

## 6. 总体评价

本次修复比首版明显进步，架构方向已经接近 P45 目标：

- world_command 依赖边界保持正确。
- inventory 独立 runtime 与 command handler 分层正确。
- 可见性、状态、地面 stack、容器 tag 都已经接入。
- P45 测试数量从 25 增加到 31。

但 P45 是后续采集、制作、NPC 自主行动、世界网格玩法的物品归属底座，不能带着 C++ 未定义行为进入下一阶段。建议修复第 4 节问题后再次复审。
