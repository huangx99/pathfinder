## 3. 仍需保留的边界风险

### 3.1 `claimsFromActorKnowledge` 仍是 synthetic claim 过渡层

级别：P1 后续补强，不阻塞 P39 通过。

确认位置：`backend/src/world_interaction/world_services.cpp`

现状仍然是：

```text
known_effect_keys -> synthetic KnowledgeClaim
```

并且存在固定映射：

```text
repel_beast -> torch + use
make_torch -> wood_processed + use
cut_wood -> wood + use
restore_sharpness -> whetstone + use
restore_hunger -> red_berry + eat
ignite_fire -> dry_grass + use
build_house -> wood_processed + use
```

判断：

- 核心 `AgentReasoner` 没有被这个问题污染，因为它仍然接受完整 `KnowledgeClaim`。
- 但 P38 world adapter 当前只保存 `known_effect_keys`，所以不得不临时合成 subject/action/target。
- 后续如果新增普通蘑菇、毒蘑菇、鱼、浆果、陷阱、斧头升级，这层必须改成真实 KnowledgeClaim 来源，否则会重新变成固定映射堆积。

建议后续阶段明确任务：

```text
WorldActorRuntimeState 增加 known_claim_refs 或 known_claims 投影。
AgentAutonomyService 从真实知识仓库/会话知识里抽取该 Agent 可用 KnowledgeClaim。
claimsFromActorKnowledge 只做筛选，不做 effect_key 到对象的创造。
```

### 3.2 reaction 接回还是最小映射，不是完整配置化计划生成

级别：P1 后续补强，不阻塞 P39 通过。

当前 `reactionRulePreconditionsFor` 只处理了少数 tag/definition：

```text
fire_source
combustible
branch_like
water_source
def_fire_source
def_dry_branch
```

这对火把足够，但还不是完整的“任意 reaction config -> plan precondition -> producer search”。

后续需要把 P40/P41 任务写清楚：

```text
从 ReactionRule 的输入 pattern 自动生成 PlanPrecondition。
从 ReactionRule 的 output/effect 自动生成 producer candidate。
支持材料数量、消耗、替代材料、工具状态、目标对象状态。
支持配置新增物品后无需改 C++ switch。
```

### 3.3 执行器仍只真正执行 `repel_beast`

级别：P1 后续执行层任务。

`AgentPlanExecutorAdapter` 对 `repel_beast` 能产生真实 `WorldChange`，但对：

```text
cut_wood
make_torch
restore_sharpness
build_house
```

目前主要是输出行动意图或摘要，完整执行还要交给后续 P40 动态目标执行、材料系统和世界交互管线承接。

这符合 P39 阶段定位，不作为本阶段阻塞项。

## 6. 最终意见

P39 可以验收。

但验收时必须带着两个后续约束进入 P40/P41：

1. 把 `known_effect_keys -> synthetic KnowledgeClaim` 替换为真实知识来源，否则 Agent 自主行为扩内容时会回到硬编码。
2. 把 `reactionRulePreconditionsFor` 从火把最小适配扩展成通用 ReactionRule adapter，否则材料、合成、生成物品还不能做到“加配置就能推理”。

当前代码没有必要退回重做。它已经满足 P39 的阶段目标，并且比上一版更接近“知识 + 反应 + 世界执行”统一架构。
