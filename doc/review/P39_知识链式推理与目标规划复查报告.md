# P39 知识链式推理与目标规划复查报告

复查时间：2026-05-19

## 1. 复查结论

结论：P39 核心推理模块可以通过复查，但需要明确一个后续风险边界。

P39 已经实现了任务卡要求的核心闭环：

```text
需求/目标 -> effect_key 语义 -> Agent 已知知识候选 -> 前置链递归 -> 风险/时间/收益评分 -> 输出计划/安全解释。
```

这不是简单的 `if hunger then eat red_berry` 主逻辑。核心推理层确实是通过 `KnowledgeClaim.predicate.effect_key` 查 `EffectSemanticsRegistry`，再匹配目标和语义生成候选。

但是，`world_interaction` 接入层目前为了把 P38 的 `known_effect_keys` 转成 P39 可用的 `KnowledgeClaim`，存在一层临时合成知识适配。该适配对 `restore_hunger -> red_berry`、`ignite_fire -> dry_grass` 等内容有固定映射。它不是 P39 推理器的主逻辑，但后续如果要让“任意新物品/新反应/新食物”自然接入 Agent 自主行为，必须在 P40/P41 继续把这层替换为真实知识仓库/反应配置读取，否则会限制扩展。

因此本次复查建议：

```text
P39 后端核心：通过。
P39 作为最终可扩展自治接入：有边界，需在后续阶段继续补强。
不需要退回重做 P39，但不能把当前 world adapter 当成最终架构。
```

## 3. 通过项

### 3.1 独立模块与构建目标

通过。

`backend/CMakeLists.txt` 已新增 `pathfinder_agent_reasoning` 静态库，包含 reasoning types、effect semantics、agent reasoner，并被 `pathfinder_world_interaction` 依赖。

复查点：

- `backend/CMakeLists.txt` 中存在 `pathfinder_agent_reasoning` target。
- `backend/tests/CMakeLists.txt` 中存在 `pathfinder_tests_agent_reasoning` 和 13 个 P39 测试注册。

### 3.2 EffectSemantics 最小语义表

通过。

`EffectSemanticsRegistry` 内置表覆盖 P39 要求的最小效果：

- `restore_hunger`
- `poison`
- `no_visible_effect`
- `cut_wood`
- `restore_sharpness`
- `ignite_fire`
- `make_torch`
- `repel_beast`
- `provide_warmth`
- `build_house`

这些 effect_key 已经映射到机器可理解语义，例如：

```text
restore_hunger -> need.hunger -30
repel_beast -> threat.beast -100
make_torch -> object.torch +1
build_house -> object.house +1 / shelter_capacity +4 / warmth +50
```

对应代码：`backend/src/agent_reasoning/effect_semantics.cpp`。

### 3.3 目标生成

基本通过。

已支持：

- `hunger >= 60` 生成 `ReduceHunger`。
- `cold >= 50` 生成 `ReduceCold` / `IncreaseWarmth`。
- `fear >= 60` 或 active threat 生成 `ReduceThreat`。
- `population > shelter_capacity && cold >= 30` 生成 `IncreaseShelterCapacity`。

注意：`RestoreToolState` 没有作为独立环境目标生成，但在 `cut_wood` 链式前置中能推出 `restore_sharpness`。这符合当前 P39 最小闭环，但如果后续要做“工具主动维护”，需要单独扩展目标来源。

对应代码：`backend/src/agent_reasoning/agent_reasoner.cpp` 的 `AgentGoalGenerator::generateGoals`。

### 3.4 候选行动来自 Agent 已知知识

通过。

候选构建流程会检查：

- owner 是否匹配当前 actor 或 tribe。
- `projection_flags.usable_by_ai` 是否允许 AI 使用。
- relation 是否属于 `HasEffect` / `Produces` / `IsUsableFor`。
- effect_key 是否能在 `EffectSemanticsRegistry` 中找到语义。
- 知识状态、可信度、冲突、危险是否可用。

这证明核心推理层不是直接从对象定义隐藏真相生成行动。

对应代码：`backend/src/agent_reasoning/agent_reasoner.cpp` 的 `buildCandidatesCore`。

### 3.5 危险知识与冲突知识过滤

通过。

`poison` 或高风险语义会被标记为 `Dangerous`，不会作为普通可用候选进入自动规划。`Deprecated`、`Disproven`、有冲突计数的知识会被标记为 `Conflicted`。

这满足“腐烂红果已知有毒，不应自动吃”的方向。

### 3.6 链式前置解析

通过。

已实现最小链式准备：

```text
build_house -> wood_processed >= 3
wood_processed 缺失 -> cut_wood
cut_wood -> wood >= 1 + axe.sharpness >= 1
axe.sharpness 缺失 -> restore_sharpness
make_torch -> wood_processed + camp_fire
repel_beast -> torch
```

同时支持：

- 多次重复生成材料，例如造房需要 3 个 `wood_processed`。
- `visited_effects` 防止 effect 循环。
- `visited_missing` 防止重复缺失条件循环。
- `max_depth`、`max_total_expansions`、`max_plan_steps` 上限。

对应代码：`backend/src/agent_reasoning/agent_reasoner.cpp` 的 `requiredPreconditionsFor`、`appendResolvedSteps`、`findProducer`。

### 3.7 计划评分

通过当前阶段要求。

评分考虑：

- 目标紧急度。
- 预期收益。
- 总耗时。
- 总风险。
- 链长惩罚。
- 紧急寒冷时短链优先。
- 非紧急时给 `build_house` 长期收益加成。

这能覆盖 P39 任务卡中的“寒冷紧急时优先生火，而不是长链造房”。

对应代码：`backend/src/agent_reasoning/agent_reasoner.cpp` 的 `PlanUtilityScorer::score`。

### 3.8 H5 安全投影

通过。

`ReasoningProjectionMapper` 输出中文目标、下一步、阻塞原因，不输出：

- `utility_score`
- `hidden_truth`
- `raw_state`

对应测试：`agent_reasoning_h5_projection_safe`。

### 3.9 P38 AgentAutonomyService 接入

部分通过。

同伴威胁分支已经调用 `AgentReasoner`，不再直接在同伴分支里新增一套“有野兽就举火把”的主逻辑。执行结果仍然转成 `AgentAutonomyResult` / `WorldChange`，继续走 P38 结算管线。

对应代码：`backend/src/world_interaction/world_services.cpp` 的 `AgentAutonomyService::tryAct`。

## 4. 发现的问题与风险

### 4.1 非阻塞但必须记录：world adapter 仍有固定 effect 到对象映射

级别：P1 后续补强，不阻塞 P39 核心验收。

位置：`backend/src/world_interaction/world_services.cpp` 的 `claimsFromActorKnowledge`。

现状：

```text
known_effect_keys 只有 effect_key，没有 subject/object/action/target 的完整知识结构。
因此适配层把 effect_key 合成为 KnowledgeClaim。
```

这导致当前世界接入层有固定映射：

```text
restore_hunger -> red_berry + eat
ignite_fire -> dry_grass + use
make_torch -> wood_processed + use
default repel_beast -> torch + use
```

判断：

- 这不是 P39 核心推理器的问题，核心推理器可以接受任意 `KnowledgeClaim`。
- 但如果后续新增“蘑菇缓解饥饿”“鱼缓解饥饿”“毒蘑菇中毒”“湿草不能点火”，仅靠 `known_effect_keys` 会丢失对象维度。
- 后续必须把 world adapter 改成从真实 KnowledgeClaim / KnowledgeRepository / reaction config 中取 subject、action、target，而不是只保存 effect_key。

建议后续阶段处理方式：

```text
WorldActorRuntimeState 不再只保存 known_effect_keys。
新增 actor_known_claim_refs 或直接注入 KnowledgeClaim 列表。
claimsFromActorKnowledge 只做筛选，不再做 effect_key -> 对象映射。
```

### 4.2 非阻塞边界：前置条件仍是 P39 fixture，不是完整 P28 反应系统读取

级别：P1 后续补强，不阻塞 P39 核心验收。

位置：`backend/src/agent_reasoning/agent_reasoner.cpp` 的 `requiredPreconditionsFor` 和 `applyExpectedFacts`。

现状：

P39 已按任务卡的最小实现支持 torch、wood_processed、axe.sharpness、house 等链式 fixture。但还没有真正遍历 P28/P28.5 的反应规则，把“反应输入/输出/条件”自动转成计划前置和产物。

判断：

- 对 P39 最小链式推理是够的。
- 对后续“加一个物品、一个反应，像喝水一样简单”还不够。
- 这正好应该在 P40 动态行动执行/材料系统阶段继续接上。

### 4.3 非阻塞边界：AgentReasoner 目前规划第一步，执行适配有限

级别：P1 后续补强，不阻塞 P39。

`AgentPlanExecutorAdapter` 对 `repel_beast` 可以生成执行变化；对 `cut_wood`、`make_torch`、`restore_sharpness` 等准备步骤目前返回“可作为行动意图”，但很多步骤还没有完全进入世界执行闭环。

判断：

- 符合 P39 “规划不直接改世界，只执行第一步交给现有系统”的边界。
- 真正让同伴连续砍木头、打磨斧头、造房，需要 P40 的“目标执行器 / 动态行动管线 / 外界打断”继续承接。

## 7. 最终建议

建议验收 P39，但在计划文档或后续 P40/P41 中明确写入两条不能遗忘的架构债：

1. `known_effect_keys -> synthetic KnowledgeClaim` 只是过渡适配，不能作为最终知识来源。
2. `requiredPreconditionsFor` / `applyExpectedFacts` 只是 P39 fixture 级链式前置，后续必须接回 P28/P28.5 反应配置和材料系统。

如果不处理这两点，系统会在新增毒蘑菇、普通蘑菇、斧头维护、房屋建造、陷阱、采集、野生 Agent 学习时重新出现“固定映射越来越多”的风险。

当前 P39 不需要退回重做；它已经为 P40 的动态目标执行、材料数量判断、行动中重新推理打好了可用基础。
