# P22 H5 中文对话框代码审核报告

## 1. 审核结论

**结论：不通过，暂时不能进入 P23。**

P22 当前代码可以编译，P22 定向测试和全量测试也能通过，并且服务在正确启动方式下可以监听 `1999` 并返回 H5 页面。但是当前实现存在核心玩法语义错误：

```text
1. 吃红果时已经自动把知识传播给同伴，导致“教同伴”这个玩家动作失去意义。
2. 教同伴会再次跑知识形成，造成 actor 重复生成相同知识，而不是单纯传播已有知识。
3. 腐烂红果流程没有真正触发旧知识修正，只是新增了一个 decayed_red_berry 的独立知识。
4. 集成测试过弱，没有断言知识快照、传播来源、修正 debug key 和重复知识问题。
5. HTTP 服务停止逻辑有阻塞风险，容易残留进程占用端口。
```

因此当前 P22 只能算“页面 + dialog API + P21 调用骨架初版”，还不是合格的“手机 H5 最小可玩学习入口”。

## 3. 阻塞问题

### BLOCKER-1：普通尝试动作自动传播给同伴，教学动作失去意义

位置：

```text
backend/src/h5_dialog/dialog_learning_adapter.cpp:85
backend/src/h5_dialog/dialog_learning_adapter.cpp:88
backend/src/h5_dialog/dialog_turn_service.cpp:181
```

问题：

```text
DialogLearningAdapter::buildLearningInput() 无论玩家是 TryEat/TryUse 还是 TeachRecipient，都会设置 input.recipient = state.recipient。
LearningLoopOptions 默认 enable_knowledge_propagation=true。
因此玩家只要“吃红果”，P21 就已经把知识传播给同伴。
```

实际手动结果：

```text
输入：吃红果
actor=['red_berry + eat -> restore_hunger [active]']
recipient=['red_berry + eat -> restore_hunger [hypothesis]']
debug=['knowledge_formed','knowledge_propagated','learning_loop_completed']
```

这违反 P22 目标：

```text
玩家应该先通过尝试形成自己的知识，再通过“教同伴”让同伴知识变化。
```

修复要求：

```text
TryEat/TryUse 阶段不要设置 recipient，或关闭 enable_knowledge_propagation。
TeachRecipient 阶段才设置 recipient 并开启 propagation。
```

### BLOCKER-2：TeachRecipient 不是传播已有知识，而是在伪造反馈后重新形成知识

位置：

```text
backend/src/h5_dialog/dialog_turn_service.cpp:142
backend/src/h5_dialog/dialog_turn_service.cpp:146
backend/src/h5_dialog/dialog_turn_service.cpp:147
backend/src/h5_dialog/dialog_turn_service.cpp:149
backend/src/h5_dialog/dialog_turn_service.cpp:159
backend/src/h5_dialog/dialog_learning_adapter.cpp:102
```

问题：

```text
TeachRecipient 会构造一个新的 DialogFeedbackTemplate。
如果找不到可教知识，还默认 object=red_berry、effect=restore_hunger。
然后继续构造 LearningLoopInput，让 P21 再跑一次知识形成。
```

实际手动结果：

```text
输入：吃红果 -> 教同伴
actor=['red_berry + eat -> restore_hunger [active]', 'red_berry + eat -> restore_hunger [active]']
```

这说明：

```text
教学导致 actor 重复生成知识。
教学不是“把已有知识传播给同伴”。
```

修复要求：

```text
1. actor_claims 为空时，TeachRecipient 必须直接 Rejected，并回复“你还没有可教给同伴的知识”。
2. TeachRecipient 不应构造新的感知反馈来形成 actor 知识。
3. TeachRecipient 应只把已有 actor_claims 作为传播源交给 P21/P20。
4. 教学成功后不应新增重复 actor claim。
```

### BLOCKER-3：腐烂红果没有真正触发“旧知识修正”

位置：

```text
backend/src/h5_dialog/dialog_scenario.cpp:73
backend/src/h5_dialog/dialog_scenario.cpp:74
backend/src/h5_dialog/dialog_scenario.cpp:75
backend/src/h5_dialog/dialog_learning_adapter.cpp:69
backend/src/h5_dialog/dialog_presenter.cpp:140
backend/src/h5_dialog/dialog_presenter.cpp:163
```

问题：

```text
decayed_red_berry 被当成一个独立 subject。
吃腐烂红果后，P21 形成的是 decayed_red_berry + eat -> poison。
它没有和 red_berry + eat -> restore_hunger 形成冲突。
因此不会真正修正旧知识。
```

实际手动结果：

```text
输入：吃红果 -> 吃腐烂红果
actor=['red_berry + eat -> restore_hunger [active]', 'decayed_red_berry + eat -> poison [active]']
debug=['knowledge_formed','knowledge_propagated','learning_loop_completed']
未出现 knowledge_revised 或 knowledge_revised_routed。
```

这违反 P22 验收：

```text
腐烂红果纠错故事可完成。
旧知识出现反例/被修正。
```

修复要求：

```text
方案 A：把腐烂红果反馈映射到同一个 knowledge_subject=red_berry，并附加 condition_key=decayed。
方案 B：保留 decayed_red_berry 可见对象，但 learning adapter 转换知识主体时使用 red_berry + condition decayed。
方案 C：显式构造已有 red_berry 知识与新 decayed 条件知识的 P19 revision 输入。
```

### BLOCKER-4：失败或异常结果仍可能覆盖 session 快照

位置：

```text
backend/src/h5_dialog/dialog_turn_service.cpp:186
backend/src/h5_dialog/dialog_turn_service.cpp:199
backend/src/h5_dialog/dialog_learning_adapter.cpp:119
backend/src/h5_dialog/dialog_learning_adapter.cpp:120
```

问题：

```text
LearningLoopService::run 返回 Result::ok(result) 不代表 result.decision 一定成功。
当前代码只要 result_r.is_ok()，就调用 applyLearningResult()。
applyLearningResult() 无条件用 result.actor_claim_snapshot_after / recipient_claim_snapshot_after 覆盖 session。
```

风险：

```text
如果 P21 返回 Failed / Rejected / Skipped，session 可能被空快照覆盖，导致玩家已有知识丢失。
```

修复要求：

```text
只有 Completed / PartialCompleted / RoutedToRevision 且关键阶段符合当前 intent 预期时，才能 apply。
Failed / Rejected 时必须保留旧 session，并返回中文错误。
```

位置：

## 4. 重要问题

### MAJOR-1：服务停止逻辑会阻塞，容易残留进程占用端口

位置：

```text
backend/src/h5_dialog/dialog_http_server.cpp:95
backend/src/h5_dialog/dialog_http_server.cpp:97
backend/src/h5_dialog/dialog_http_server.cpp:101
backend/src/h5_dialog/dialog_http_server.cpp:114
backend/apps/h5_dialog_server_main.cpp:10
backend/apps/h5_dialog_server_main.cpp:13
```

现象：

```text
Ctrl+C 后打印 Shutting down...，但进程没有退出。
需要 kill -9 才能清理。
```

原因：

```text
stop() 调用 close(server_fd_) 后 join accept 线程，但 accept() 不一定被可靠唤醒。
signal handler 中调用 cout、stop、join 也不是安全的信号处理方式。
```

影响：

```text
旧进程可能占用 1999，导致研发以为新代码启动了，实际访问的是旧进程或根本无法绑定。
```

修复要求：

```text
1. signal handler 只设置 atomic stop flag。
2. 主循环发现 flag 后调用 server.stop()。
3. stop() 使用 shutdown(server_fd_, SHUT_RDWR) 后 close，确保 accept 退出。
4. 或用非阻塞 socket / select 超时轮询 running_。
```

## 6. 修复优先级

### P0 必须修

```text
1. TryEat/TryUse 不得自动传播给同伴。
2. TeachRecipient 必须只传播已有知识，不得重复形成 actor 知识。
```
