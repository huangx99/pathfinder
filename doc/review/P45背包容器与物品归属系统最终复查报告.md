## 1. 最终结论

结论：通过复查，P45 当前可以验收。

第四次复审指出的最后一个 P0 阻塞项已经关闭：

- 部分拾取地面 stack 时，地面实体继续留在地图上并减少数量。
- 背包中拿走的那部分物品不再复用地面实体 id，而是生成独立 inventory carrier id。
- 新 inventory carrier 会登记到 `item_locations_`，归属为 `InInventory`。
- 地面实体不会被错误标记为 `InInventory`。

这说明 P45 的核心归属规则已经成立：一个物品实例在同一时刻不能同时属于地图和背包；部分分堆时必须形成独立载体。

## 2. 本轮重点复查

### 2.1 partial pickup 不再复用地面 entity id

位置：`backend/src/world_inventory/world_inventory_runtime.cpp`

当前实现要点：

```cpp
if (partial_pickup) {
    int new_ground_quantity = entity->quantity - request.quantity;
    world_port_.setEntityStackData(request.entity_id, new_ground_quantity, stack_key, entity->stackable);
    draft.changed_entity_ids.push_back(request.entity_id);
}
...
if (partial_pickup) {
    entry.entity_id = request.entity_id + "_pickup_" + std::to_string(state_version_);
    item_locations_[entry.entity_id] = InInventory;
    draft.changed_entity_ids.push_back(entry.entity_id);
}
```

复查判断：通过。

原因：

- 地面原实体只代表剩余 stack。
- 背包 entry 使用新的 `entry.entity_id`。
- `item_locations_` 只给新的 inventory carrier 建立背包归属。
- 原地面 id 仍保留地图语义，没有被覆盖成背包归属。

### 2.2 partial pickup 数量守恒仍然成立

测试覆盖：`world_inventory_pickup_partial_ground_stack_keeps_remainder`

断言覆盖：

- 地面 5 个红果，拾取 2 个后，背包得到 2 个。
- 地面实体仍存在，quantity 为 3。
- 地面实体仍是 `OnMap`。
- 背包 entry 的 `entity_id` 不等于地面 entity id。
- 背包新 carrier 的 location 是 `InInventory`。
- 地面 entity id 不会被登记成 `InInventory`。

复查判断：通过。

### 2.3 新增 distinct inventory entity 测试有效

测试覆盖：`world_inventory_pickup_partial_ground_stack_has_distinct_inventory_entity`

该测试直接验证第四次复审指出的问题：

- 背包 entry 必须使用不同于 `berry_ground_001` 的实体 id。
- 新实体 id 必须能在 `findItemLocation` 中查到 `InInventory`。
- 原 `berry_ground_001` 必须仍然是地面实体。

复查判断：通过。

### 2.4 full drop merge 与 stack_key 隔离仍然保持

本轮没有发现回退：

- `world_inventory_full_drop_merge_ground_cleans_source_entity` 覆盖 full drop 合并后清理源实体。
- `world_inventory_drop_merge_ground_changed_entity_ids` 覆盖合并地面 stack 后投影变更通知。
- `world_inventory_stack_key_blocks_merge` 覆盖相同 `entity_key`、不同状态的 stack 不会错误合并。

复查判断：通过。

## 4. 对 P45 设计目标的验收判断

### 4.1 背包归属

通过。

P45 已经能表达：

- Actor 背包。
- 地图物品。
- 背包物品。
- 地图到背包的拾取。
- 背包到地图的丢弃。
- 分堆、合并、容量、可见性、距离、归属校验。

### 4.2 物品唯一归属

通过。

当前关键路径已经避免：

- 同一实体同时在地图和背包。
- 地面 stack 被部分拾取后整堆消失。
- full drop 合并地面 stack 后留下幽灵源实体。
- 不同状态物品因为同 `entity_key` 被错误合并。

### 4.3 Command 集成

通过。

P45 的 pickup/drop 已经通过 world command 集成测试，并且 `world_command` 没有直接包含 inventory/runtime 头文件。当前依赖方向符合 P43/P44/P45 的组合设计。

### 4.4 为 P46/P47 的承接能力

通过，但有边界。

P45 当前足以支撑：

- P46 世界生成把资源生成到地图上。
- P47 世界交互通过 command 拾取、采集后入背包。
- 后续 NPC/玩家共享同一套 inventory transfer 规则。

需要注意：P46/P47 生成或采集物品时，必须通过 P45 的归属接口建立 `OnMap` 或 `InInventory` 状态，不能绕过 runtime 直接改数组。

## 5. 非阻塞后续建议

这些不是 P45 验收阻塞，但建议后续阶段继续收敛：

1. 后续 inventory-only 变化增多后，建议引入统一 projection version 生成器，不再只用 `max(world_runtime.stateVersion(), inventory_runtime.stateVersion())`。
2. P48 之前建议补一个 `SourceInventoryProof` 辅助构造器，避免 NPC 行动链路手写归属证明。
3. 后续接容器、装备、消耗时，需要继续遵守当前 P45 规则：所有转移都必须先 validate，再 apply，不允许 validate 写入运行时状态。
4. 后续如果需要更强的 inventory-only carrier id，应把当前字符串拼接替换为统一 id allocator，但这不是当前验收阻塞。

## 6. 最终验收建议

建议验收 P45。

理由：

- 第四次复审指出的 P0 阻塞项已经修复。
- P45 专项测试 40/40 通过。
- P43/P44/P45 回归测试 52/52 通过。
- 架构依赖扫描通过。
- 当前实现没有发现新的 P0/P1 阻塞问题。

P45 可以作为后续 P46 世界生成器、P47 世界交互、NPC 自主行动、采集/制作/掉落闭环的物品归属基础。
