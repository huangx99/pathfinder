# P45 背包、容器、掉落物与物品归属系统再次复审报告

复审对象：P45 第二轮复审后修复版本

- 上次复审：`doc/review/P45背包容器与物品归属系统复审报告.md`
- 主设计：`doc/75_P45背包容器与物品归属系统设计.md`
- 核心实现：`backend/include/pathfinder/world_inventory/`、`backend/src/world_inventory/`
- 测试：`backend/tests/unit/world_inventory/`、`backend/tests/integration/world_inventory/`

## 1. 复审结论

结论：仍不通过。

上次复审提出的 4 个明确遗留项已经大部分修复：

1. full drop 失效指针：已用 `dropped_entry` 复制修复。
2. capacity 校验硬编码 20：已改为读取 `capacity_slots`。
3. `drop_cell_invalid` 语义：已新增 `InventoryFailureKind::DropCellInvalid`。
4. partial drop 新实体 changed ids：已在 apply 阶段补入 `changed_entity_ids`。

但本次继续按“物品归属权威”和“堆叠数量守恒”复核后，发现仍有新的 P0 级一致性漏洞：

- 从地面堆叠物品中部分拾取会丢失剩余数量。
- 丢弃到已有地面 stack 时，full drop 会留下背包源实体幽灵状态。
- 丢弃到已有地面 stack 时，changed_entity_ids 没有包含被更新的地面实体。

这些问题都会直接影响后续采集、制作、NPC 拾取、地图显示与背包同步。因此 P45 仍不建议验收。

## 4. 新发现阻塞问题

### P0-1：从地面堆叠物品部分拾取会丢失剩余数量

位置：`backend/src/world_inventory/world_inventory_runtime.cpp` 的 `applyPickupFromMap`

当前流程：

```cpp
// entity->quantity 可能是 5，request.quantity 可能是 2
world_port_.removeEntityFromMap(request.entity_id);
world_port_.setEntityLocationKind(request.entity_id, InInventory);
entry.quantity = request.quantity;
```

问题：

如果地面 entity 是一个 stack，例如地上 5 个红果，玩家只拾取 2 个：

- 当前实现会把整个地面 entity 从地图移除。
- 背包 entry 只增加 2 个。
- 剩余 3 个不会留在地图上。
- WorldEntityInstance 仍可能保留原 quantity=5，但位置变成 InInventory，inventory entry 却是 2，实体与背包数量不一致。

风险：

- 物品数量不守恒。
- 地图资源会凭空消失。
- NPC/玩家采集、拾取地面堆叠物、多人抢同一 stack 都会出错。
- 后续做“采集红果堆”“木材堆”“矿石堆”时会直接影响玩法。

修复建议：

- `applyPickupFromMap` 必须区分 full pickup 与 partial pickup。
- 如果 `request.quantity < entity->quantity`：
  - 地面 entity 数量减少：`entity.quantity -= request.quantity`。
  - 地面 entity 仍保持 OnMap。
  - 背包增加对应数量 entry。
  - `changed_entity_ids` 包含地面 entity。
  - `item_locations_` 不应把原地面 entity 改成 InInventory。
- 如果 full pickup，才允许将原 entity 从 OnMap 转入 inventory 或销毁合并。

必须新增测试：

```text
world_inventory_pickup_partial_ground_stack_keeps_remainder
```

测试要验证：地上 5 个红果，拾取 2 个后，背包有 2，地上同 entity 或新 stack 剩 3，位置仍 OnMap。

### P0-2：Full drop 合并到已有地面 stack 后会留下背包源实体幽灵状态

位置：`backend/src/world_inventory/world_inventory_runtime.cpp` 的 `applyDropToMap` merged_to_ground 分支

当前逻辑：

```cpp
if (merged_to_ground) {
    if (request.quantity < target_entry->quantity) {
        target_entry->quantity -= request.quantity;
    } else {
        inv->entries.erase(inv->entries.begin() + target_index);
        inv->used_slots -= 1;
    }
}
```

问题：

如果背包里 `berry_001` 全量丢弃，并且地上已有同 stack_key 的 `berry_ground_001`：

- 地面 stack 数量增加了。
- 背包 entry 被删除了。
- 但 `berry_001` 这个源实体如果曾经通过 pickup 进入背包，它在 world runtime / item_locations 中可能仍是 InInventory。
- 当前 merged_to_ground 分支没有 destroy 源实体，也没有清理 `item_locations_[request.entity_id]`。

风险：

- 背包已经没有 entry，但 world/runtime location 仍可能认为该实体在背包。
- 后续 Drop / Pickup / Projection / SaveLoad 会出现幽灵物品。
- 这是与首轮“无 entry 的 InInventory 实体”同类的一致性问题，只是发生在 drop merge 场景。

修复建议：

- 在 merged_to_ground 分支中，如果 full drop：
  - 删除 inventory entry 后，应 `destroyEntity(request.entity_id)` 或清理该 source stack carrier。
  - `item_locations_.erase(request.entity_id)`。
  - `draft.changed_entity_ids` 加入 `request.entity_id` 和 `ground_entity_id`。
- partial drop merge 时源 entry 仍留在背包，不应清理 source。

必须新增测试：

```text
world_inventory_full_drop_merge_ground_cleans_source_entity
```

测试要模拟：背包有 berry_001，地上已有 berry_ground，full drop 后，地面 quantity 增加，背包 entry 删除，berry_001 不再有 InInventory location。

### P0-3：合并到已有地面 stack 时 projection changed_entity_ids 漏掉被更新实体

位置：`backend/src/world_inventory/world_inventory_runtime.cpp` 的 `applyDropToMap` merged_to_ground 分支

当前情况：

- 非合并新建实体路径已补 `dropped_entity_id` 到 `changed_entity_ids`。
- 但合并已有地面 stack 时，只更新了 `ground_entity_id` 的 quantity，没有把 `ground_entity_id` 加入 `draft.changed_entity_ids`。

风险：

- 前端可能收不到地面 stack 数量变化的 entity patch。
- AI 或同步层如果依赖 patch 精确更新，会认为地面物品数量没变。

修复建议：

- 在 `setEntityStackData(ground_entity_id, ...)` 成功后：

```cpp
draft.changed_entity_ids.push_back(ground_entity_id);
```

- 如果 full drop 同时清理 source，也要把 source id 加入 changed ids。

必须新增测试：

```text
world_inventory_drop_merge_ground_changed_entity_ids
```

## 5. 非阻塞建议

### P2-1：capacity_real_slots 测试仍有条件跳过风险

`world_inventory_capacity_real_slots` 中，如果没有找到第二个可拾取物，核心 CapacityExceeded 断言不会执行。

建议改为显式 `spawnEntityOnMap` 两个不同 entity_key 的同格物品，确保第二次 pickup 必定执行。

### P2-2：同 entity_key 不同 state_keys 的 stackable=true 场景仍建议补测

当前已有 non-stackable runtime 测试，但还建议补：

```text
同 entity_key = red_berry
state_keys = fresh / rotten
stackable = true
拾取后必须两个 entry，不能合并
```

这能更直接保护“普通红果/腐烂红果不能合并”的核心设计。

## 6. 验收建议

当前仍不建议验收 P45。

虽然上次遗留项已经认真修了，而且测试数量从 31 增加到 35，但 P45 的核心是“物品归属与数量守恒”。只要 partial pickup 地面 stack 会丢数量、full drop merge 会留下幽灵源实体，就不能作为后续采集/制作/NPC 行动的稳定底座。

建议下一轮最小修复范围：

1. 修复 pickup partial ground stack split。
2. 修复 full drop merge ground 清理 source entity / item location。
3. 修复 drop merge ground changed_entity_ids。
4. 补上述三个强测试，避免再次只覆盖新建地面 entity 路径。

修完后再复审。如果这三个问题关闭，P45 就基本可以进入验收状态。
