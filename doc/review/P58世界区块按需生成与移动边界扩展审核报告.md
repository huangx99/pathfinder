# P58 世界区块按需生成与移动边界扩展审核报告

## 1. 审核结论

结论：**不建议验收，当前 P58 仍然 Blocked。**

用户在网页中实测“没有无限地图”是合理的。代码确实没有把“移动后生成下一圈区块并返回新的移动 option”这条链路接完整。

当前实现已经完成了一些正确基础：

```text
1. 新增了 WorldRegionMath，负坐标 floorDiv 思路正确。
2. 新增了 WorldRegionEnsureService，可以调用 P57 WorldGenerationService + Applier 生成区块。
3. 新增了 ClientWorldRegionEnsureAdapter。
4. 新增了 MoveTargetRegionGuard。
5. 启动 factory 不再预生成 11x11，而是只生成 origin，方向正确。
6. 相关测试编译和运行通过。
```

但是 P58 的核心目标是：

```text
玩家向外走时，后端自动生成相邻区块，并让前端拿到新的可用移动命令。
```

当前实现只完成了“能生成区块的服务”，没有把它完整接入网页实际使用的 command response 链路。因此玩家走到已生成窗口边缘后，前端仍然拿不到继续向外的 Move option。

验收状态：**Blocked**。

## 3. 阻塞问题

### 阻塞 1：移动命令执行后返回 available_commands 时没有先 ensure 新边界区块

问题位置：

```text
backend/src/client_protocol/client_command_gateway.cpp:299
backend/src/client_protocol/client_command_gateway.cpp:301
backend/src/client_protocol/client_command_gateway.cpp:318
```

当前 command response 流程是：

```text
1. 前端提交 option_id。
2. 后端执行 Move。
3. ClientCommandGateway 重新 buildOptions。
4. updateAvailableCommands 保存新快照。
5. 返回前端。
```

但是第 3 步直接调用：

```cpp
available_command_adapter_.buildOptions(actor_key, "surface")
```

没有在 buildOptions 前调用：

```text
ensureForAvailableCommands(actor, layer)
```

这会导致：

```text
1. 玩家在已生成区块内部移动时正常。
2. 玩家走到已生成窗口边缘时，下一步目标格所属 region 还没生成。
3. MovementCommandOptionProvider 查询 runtime.findCell(target) 失败。
4. 对应方向的 Move option 变成 disabled 或前端无法 resolve 到 enabled option。
5. 网页表现就是“没有无限地图 / 不能继续往外走”。
```

这正是用户实测的问题。

必须修复：

```text
ClientCommandGateway 在构建 command response 的 available_commands 前，也必须执行 available-command-window ensure。
```

建议方案：

```text
1. 给 ClientCommandGateway 注入 ClientWorldRegionEnsureAdapter，或让 ClientSessionGateway 提供 ensureAndBuildAvailableCommands 封装。
2. handleCommand 执行成功后、buildOptions 前，调用 ensureForAvailableCommands(session actor, session layer)。
3. ensure 成功后再 buildOptions，并更新 session snapshot。
4. 如果 ensure 失败，返回 warning 或 blocked，不允许静默返回旧的边界命令。
```

### 阻塞 2：Refresh 路径也没有在 buildOptions 前 ensure 邻近移动区块

问题位置：

```text
backend/src/client_protocol/client_session_gateway.cpp:121
backend/src/client_protocol/client_session_gateway.cpp:123
backend/src/client_protocol/client_session_gateway.cpp:139
```

当前 refresh 只做了：

```cpp
ensureForRefresh(session.actor_key, session.layer_key, 0)
```

然后直接：

```cpp
available_command_adapter_.buildOptions(session.actor_key, session.layer_key)
```

这里有两个问题：

```text
1. vision_radius 被硬编码成 0，根本不是玩家真实视野。
2. refresh 构建 available_commands 前没有调用 ensureForAvailableCommands。
```

结果是：

```text
即使玩家手动刷新，后端也不一定会生成边界外的相邻 region，前端仍然拿不到继续移动的 option。
```

必须修复：

```text
1. refresh 应使用 actor.vision_radius 或 request viewport/vision 参数，而不是 0。
2. refresh 在 buildOptions 前也必须 ensureForAvailableCommands。
```

注意：如果把 vision_radius 改成当前客户端默认 12，`max_regions=4` 可能不够，因为在 region_size=16 且中心附近时，视野窗口可能覆盖 3x3 共 9 个 region。需要同步调整策略，否则 refresh 会触发 LimitExceeded。

### 阻塞 3：MoveTargetRegionGuard 只保护“执行已有命令”，不能解决前端没有 option 的问题

问题位置：

```text
backend/src/world_runtime/world_runtime_command_handlers.cpp:49
backend/src/world_runtime/world_runtime_command_handlers.cpp:59
backend/src/world_runtime/world_runtime_command_handlers.cpp:95
```

当前 Move handler 确实在执行前调用了 `MoveTargetRegionGuard`。

这部分方向是对的，但不足以让网页无限移动。

原因：

```text
前端不会自己构造 WorldCommandDto。
前端只能提交后端给的 option_id。
如果边界外的 Move option 没有被生成，前端根本没有命令可提交。
MoveTargetRegionGuard 永远不会被触发。
```

所以 P58 不能只依赖 Move 执行 guard。

必须同时保证：

```text
1. available commands 生成前 ensure 邻近目标 region。
2. Move 执行前二次 ensure 目标 region。
```

当前只做了第 2 条，没有完整做第 1 条。

### 阻塞 4：边界移动测试是无效测试，没有覆盖真实 region 边界

问题位置：

```text
backend/tests/integration/world_generation/world_region_ensure_integration_test.cpp:109
backend/tests/integration/world_generation/world_region_ensure_integration_test.cpp:112
backend/tests/integration/world_generation/world_region_ensure_integration_test.cpp:129
backend/tests/integration/world_generation/world_region_ensure_integration_test.cpp:131
```

当前测试写的是：

```text
Player is at (0,0), region (0,0) is generated.
Ensure neighbor moves should generate region (1,0) if needed.
```

但玩家在 `(0,0)` 时，上下左右目标是：

```text
(0,-1), (0,1), (-1,0), (1,0)
```

这些坐标全部仍在 region `(0,0)` 内，因为 region_size=16 时 region `(0,0)` 覆盖：

```text
x = -8..7
y = -8..7
```

所以这条测试根本不会触发 region `(1,0)` 的按需生成。

更严重的是，测试里明明计算了 `has_r10`，但没有断言：

```text
We don't strictly assert has_r10
```

最后只验证了：

```cpp
findCell({1,0,"surface"})
```

而 `(1,0)` 本来就在 origin region 中，当然存在。

这条测试会让“没有真实无限地图”仍然通过测试。

必须修复测试：

```text
1. 把玩家放到 x=7,y=0，也就是 origin region 右边界。
2. 调用 ensureForAvailableCommands。
3. 必须断言 region (1,0) GeneratedAndApplied 或 AlreadyAvailable。
4. 必须断言 cell (8,0) 存在。
5. 必须通过 ClientCommandGateway 或 available_command_adapter 验证向右 Move option enabled。
```

### 阻塞 5：P58 没有端到端测试“网页真实链路”

当前缺少最关键测试：

```text
bootstrap -> 移动到 region 边界 -> command response 返回新的 available_commands -> 继续向外移动
```

应该新增集成测试模拟前端真实行为：

```text
1. 创建 ClientServerRuntimeFactory 或等价真实 runtime fixture。
2. bootstrap。
3. 连续提交向右 option_id，直到玩家到 x=7。
4. 检查 command response 中存在 target x=8 的 enabled Move option。
5. 提交该 option。
6. 检查玩家移动到 x=8。
7. 检查 runtime 生成了 region (1,0)。
8. 再走到 x=23，继续检查 region (2,0) 能生成。
```

这条测试才是真正证明“网页可以无限向外走”的测试。

## 4. 非阻塞但必须关注的问题

### 4.1 projection_version 与 runtime stateVersion 可能不同步

MoveTargetRegionGuard 在 Move handler 内部执行 ensure 时，WorldGenerationApplier 会大量修改 runtime，导致 runtime stateVersion 增长。

但 WorldCommandPipeline 最终仍用：

```text
new_version = projection_version_ + 1
```

这可能造成：

```text
1. runtime 实际版本增长很多。
2. command response 返回给前端的 projection_version 只 +1。
3. 后续 refresh 又会跳到 runtime stateVersion。
```

当前不一定马上阻塞网页移动，但后续可能造成版本跳变、stale 判断混乱。

建议后续把“命令中 ensure 造成 runtime 修改”纳入 projection version 策略，不能只让 pipeline 自增 1。

### 4.2 refresh 的 max_regions 策略需要重新审视

当前：

```text
ensureForRefresh(..., vision_radius, max_regions=4)
```

如果视野半径是 12，region_size 是 16，那么中心附近可能覆盖 9 个 region。

如果后续把 refresh 的 radius 从 0 改成真实值，可能马上 LimitExceeded。

建议：

```text
1. Bootstrap/Refresh 视野窗口可以 max=9。
2. AvailableCommandWindow 维持 max=4。
3. MoveTarget 维持 max=1。
```

## 7. 最终判断

P58 当前实现不能验收。

核心原因不是“生成器不会生成”，而是：

```text
生成器能力没有接到网页真实移动循环的 available_commands 重新生成阶段。
```

这会让网页表现为：


这与用户实测完全一致。

## 8. 本次复审与修复记录

复审结论：P58 后端按需生成链路已通过，但网页仍让用户感觉“不是无限地图”，原因不再是后端不能跨区块，而是客户端显示和刷新链路存在两个问题。

### 8.1 已确认的真实链路

本次使用真实 HTTP 接口验证：

```text
bootstrap -> 连续提交 move_east option_id -> 跨过多个 region -> command response 持续返回新的 available_commands -> refresh 后仍在玩家当前位置
```

验证结果：

```text
1. 1999 当前接口可连续向东移动到 x=39。
2. 新构建的 2999 临时服务可连续向东移动 48 步，从 x=34 到 x=82。
3. 移动过程中 projection_patch 持续返回 changed_cells / changed_entities。
4. command response 持续返回可用移动 option。
5. refresh 后 full_projection 中玩家仍位于 x=82。
```

因此后端 P58 的 region ensure、MoveTarget guard、available_commands 重新生成链路已经成立。

### 8.2 修复 1：客户端刷新中心不再固定为原点

问题：

```text
app/frontend/client/app.js 的 apiRefresh() 固定发送 viewport_center_x=0, viewport_center_y=0。
```

风险：

```text
玩家跨区块后触发 refresh 时，客户端请求语义仍像在原点刷新，后续接入真正 viewport 投影时会导致视野被拉回原点。
```

修复：

```text
apiRefresh() 现在读取 getActorCoord()，把玩家当前坐标作为 viewport_center_x / viewport_center_y 发给后端。
```

### 8.3 修复 2：未知区域显示为探索雾，而不是黑色断边

问题：

```text
客户端只绘制后端已经投影过的 visible_cells，投影窗口外直接露出黑色背景。
```

玩家体验：

```text
这会让玩家误以为地图到边界了，像固定地图，而不是开放世界。
```

修复：

```text
renderMap() 先按当前视窗铺一层 unknown / fog tile。
后端已返回的 visible_cells 再覆盖真实地形。
缺失格子不再表现为世界尽头，而表现为尚未探索区域。
```

### 8.4 修复 3：静态文件禁止浏览器缓存

问题：

```text
手机端容易继续加载旧 app.js / style.css，造成用户刷新后仍看到旧行为。
```

修复：

```text
ClientStaticFileService 对静态文件响应增加 Cache-Control: no-store, max-age=0。
```

这不会影响游戏规则，只影响开发阶段浏览器拿新文件的稳定性。

### 8.5 修复 4：CTest 注册真实跨区块测试

问题：

```text
client_runtime_bridge_cross_region_move / client_runtime_bridge_negative_cross_region_move 函数存在，但没有注册进 CTest。
```

修复：

```text
backend/tests/CMakeLists.txt 已注册：

- client_runtime_bridge_negative_cross_region_move
```

以后跑 CTest 会自动覆盖正负方向跨 region 移动。

## 9. 首屏预取与探索同步补充修复

用户反馈：虽然 P58 已经支持无限移动，但网页需要走到边缘附近才看到新地图，体验像“边走边刷”，非常难受。

本次判断：这不是前端应该用迷雾或假地形掩盖的问题，而是后端 Bootstrap/Refresh 已经 ensure 了周边 region 后，没有及时刷新玩家探索可见性，导致 full_projection 只返回当前已探索窗口内的格子。

### 9.1 修复内容

```text
1. IWorldRuntime 增加 refreshActorExploration(actor_key)。
2. WorldGridRuntime 实现 refreshActorExploration，复用已有 updateExplorationForActor。
3. ClientWorldRegionEnsureAdapter 在 ensureRegions 成功后主动刷新 actor exploration。
4. Bootstrap/Refresh 的 region ensure 上限从 9 提高到 25，允许更大的首屏/视野预取窗口。
5. client_server 默认玩家 vision_radius 从 12 提高到 24。
6. 前端不再绘制伪迷雾，只显示后端真实投影。
```

### 9.2 为什么这样改

```text
前端不能自己制造地图。
无限地图应该由后端生成、Runtime 落地、探索系统标记可见、Projection 返回给客户端。
这次修复让“提前生成”和“投影可见”进入同一条后端权威链路。
```
