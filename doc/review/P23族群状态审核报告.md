## P0 问题：P23 必须覆盖的安全测试不完整

位置：

- `doc/50_P23族群状态任务卡设计.md`
- `backend/tests/unit/tribe/tribe_state_test.cpp`
- `backend/tests/integration/p23/tribe_state_security_test.cpp`

设计文档第 12 节明确要求必须测试拒绝：

```text
tribe_id 为空。
member_id 为空。
ratio < 0 或 > 1。
active_population > population_total。
Unknown/TestOnly 枚举进入正式 DTO。
重复 member_id。
重复 shared_knowledge_ids。
reason_keys 包含 forbidden key。
输入中出现随机分裂语义。
```

当前测试只覆盖了部分 forbidden key、random_split、部分 population mismatch、部分 Unknown/ratio 场景。

缺少的关键覆盖包括：

- `TribeStateInput.tribe_id` 为空应拒绝。
- `TribeMemberEvent.member_id` 为空应拒绝。
- `TribeStateInput` 的 ratio 字段越界应拒绝。
- `active_population > population_total` 应拒绝。
- `TribeState` 重复 `member_id` 应拒绝。
- `TribeState.shared_knowledge_ids` 重复应拒绝。
- `TribeState`、`TribeMemberEvent`、`TribeStateChangeDraft` 中 `TestOnly` 应拒绝。

影响：

```text
虽然当前实现大部分校验已经存在，但验收测试没有覆盖完整安全边界。
后续 AI 继续开发时，容易无意破坏这些约束而测试不报警。
```

修复建议：

```text
补齐 P23 security/unit 测试，不一定需要改业务逻辑。
```

## P1 问题：成员离开语义没有被测试且可能不符合“族群人数”直觉

位置：

- `backend/src/tribe/tribe_state.cpp`
- `backend/tests/unit/tribe/tribe_state_test.cpp`
- `backend/tests/integration/p23/tribe_state_flow_test.cpp`

当前实现中：

```cpp
MemberRemoved -> member.active = false
population_total = members.size()
active_population = active member count
```

也就是说，一个成员“离开”后：

```text
population_total 不下降。
active_population 下降。
```

这在工程上可以解释为“历史成员仍保留在 members 中，只是不活跃”，但从游戏语义上，P23 任务卡写的是“成员加入、离开”和“族群有多少人”，玩家/后续系统可能会把 `population_total` 理解为当前族群总人数。

风险：

```text
P24 共同作战、P25 族群冲突、P26 文明能力如果直接读取 population_total，可能把已离开成员也算进族群规模。
```

建议二选一明确：

1. 如果 `population_total` 表示当前族群总人数：成员离开时应从当前成员统计中移除，或 `population_total` 只统计 active 成员。
2. 如果 `population_total` 表示历史成员总数：需要新增/明确 `current_population` 或在文档中写明 `population_total=历史成员数，active_population=当前可用成员数`。

同时必须新增测试：

```text
加入 2 人 -> 移除 1 人 -> 校验 population_total / active_population / projection 的预期语义。
```

## P1 问题：成员事件的必填字段校验不完整

位置：

- `backend/src/tribe/tribe_state.cpp`

当前 `TribeMemberEvent::validateBasic()` 对 `MemberAdded` 和 `MemberRoleChanged` 要求 `role`，但没有要求：

- `TrustChanged` 必须带 `trust`。
- `MoraleChanged` 必须带 `morale`。

结果是下面这种输入会通过基础校验，但 resolver 会静默忽略：

```text
kind = TrustChanged，但 trust 为空。
kind = MoraleChanged，但 morale 为空。
```

风险：

```text
后续管线或 AI 生成错误事件时，系统不会报错，只会 no_change，排查困难。
```

修复建议：

```text
在 TribeMemberEvent::validateBasic() 中补充 kind-specific 必填字段校验，并增加对应拒绝测试。
```

## P2 建议：TribeStateChangeDraft 校验可以更严格

位置：

- `backend/src/tribe/tribe_state.cpp`

当前 `TribeStateChangeDraft::validateBasic()` 只做通用校验，没有按变更类型要求字段组合。

建议后续补强：

```text
MemberAdded 必须有 member_id/new_role。
MemberRemoved 必须有 member_id/old_role。
MemberRoleChanged 必须有 member_id/old_role/new_role。
TrustChanged/MoraleChanged/SplitRiskChanged 必须有合理 old_value/new_value。
KnowledgeLinked 第一版至少保持 summary_key 和 new_value 有效。
```

这样能防止后续 AI 实现误生成“看起来 ok，但语义不完整”的 state change。

```text
如果未来测试确实需要 TestOnly，则要把 allow_test_only 接入校验路径，但默认仍必须 false。
```

## P2 建议：输入事件去重策略可以更明确

位置：

- `backend/src/tribe/tribe_state.cpp`

当前 resolver 对重复 `knowledge_events` 会自然去重，因为写入 `shared_knowledge_ids` 前会检查是否已存在。

但 `TribeStateInput::validateBasic()` 本身没有拒绝重复输入事件。

建议文档和代码二选一：

```text
方案 A：输入允许重复，resolver 幂等去重。
方案 B：输入不允许重复，validateBasic 直接拒绝。
```

从工程可调试性看，建议 P23 采用方案 B：输入重复就拒绝，避免上游管线重复发事件时被悄悄吞掉。

## 架构符合项

本次审核确认以下点符合 P23：

- `pathfinder_tribe` 只依赖 `pathfinder_foundation` 和 `pathfinder_knowledge`。
- `TribeProjection` 只输出摘要，没有塞完整 `KnowledgeClaim`。
- `TribeSplitRisk` 由资源压力、信任压力、知识冲突压力、伤亡压力确定性计算。
- `cohesion_state` 会根据 risk 派生并校验。
- `Result::ok(dto)` 主要出口已调用 `validateBasic()`。
- forbidden key 能拒绝 `hidden_truth`、`raw_state`、`GameState`、`SaveGame`、`random_split`、`probability_split` 等语义。
