# P53 客户端协议与自动化验证审核报告

## 1. 审核结论

结论：**不通过，需修复后复审**。

P53 当前实现已经建立了独立的 `client_protocol` 模块，并且自带测试与 P43-P52 回归测试均通过；但代码层面存在几个会影响后续正式客户端和开放世界扩展的阻塞问题：

```text
1. 客户端可以伪造 actor 执行命令。
2. projection_version 过期时仍然执行命令，只是返回 requires_full_refresh。
3. option_id 不是后端会话内签发的权威令牌，存在伪造 / 过期无法严格识别的问题。
4. JSON Codec 没有完整序列化 projection / patch 内容，未来客户端会丢世界状态。
5. 测试覆盖了“能跑通”，但没有覆盖上述安全与协议完整性边界。
```

因此本阶段不能验收为“客户端协议稳定层”。

## 4. 已完成且符合设计的部分

### 4.1 模块边界基本正确

新增模块位于：

```text
backend/include/pathfinder/client_protocol/
backend/src/client_protocol/
```

命名使用 `client_protocol`，没有继续使用 `H5V2` / `h5_v2`，符合“客户端协议不绑定 H5”的方向。

### 4.2 没有新增前端 UI

P53 没有新增 HTML / CSS / DOM 交互，也没有设计按钮布局，符合“本阶段只做协议，不做界面”的要求。

### 4.3 没有直接读取 content JSON

`client_protocol` 当前没有读取 `content/core`、`recipes`、`ContentRegistry` 等内容配置，符合“客户端协议层不读玩法 JSON”的要求。

### 4.4 命令入口接入 WorldCommandPipeline

`ClientCommandGateway::handleCommand()` 最终调用：

```cpp
pipeline_.execute(request.session_id, command)
```

方向上符合“客户端动作必须走 WorldCommandPipeline”。

### 4.5 自动化测试已经建立

P53 已加入：

```text
client_protocol 单元测试
client_protocol 集成测试
architecture_client_protocol_* 架构扫描
```

这是正确的阶段成果。

## 5. 阻塞问题

## 5.1 阻塞问题一：客户端可以伪造 actor 执行命令

位置：

```text
backend/src/client_protocol/client_command_gateway.cpp
```

问题代码：

```cpp
std::string actor_key = request.selection_context.selected_actor_key;
if (actor_key.empty()) {
    actor_key = "player";
}
```

问题：

```text
ClientCommandGateway 没有从 session 中读取绑定 actor。
而是信任客户端传入的 selection_context.selected_actor_key。
```

这会导致客户端可以伪造：

```text
selected_actor_key = wolf
selected_actor_key = npc_x
selected_actor_key = companion
```

然后通过 option_id 或 WorldCommandDto 让非本人 actor 执行动作。

这违反 P53 设计中的核心原则：

```text
客户端只能提交意图，不能决定谁执行。
后端 session / actor ownership 才是权威。
```

严重性：**阻塞**。

必须修复：

```text
1. ClientSessionGateway 必须提供只读 session actor 查询接口。
2. ClientCommandGateway 必须从 session 中取 actor_key。
3. selection_context.selected_actor_key 只能作为“选择对象上下文”，不能作为执行 actor。
4. WorldCommandDto 模式下，如果 command.actor_key 与 session actor 不一致，必须拒绝或重写为 session actor。
5. 增加测试：客户端伪造 selected_actor_key 不得改变执行 actor。
6. 增加测试：客户端提交 WorldCommandDto.actor_key = other_actor 必须被拒绝或规范化。
```

## 5.2 阻塞问题二：projection_version 过期时仍执行命令

位置：

```text
backend/src/client_protocol/client_command_gateway.cpp
```

问题代码逻辑：

```cpp
bool version_mismatch = (request.known_projection_version != current_version);

auto pipeline_result = pipeline_.execute(request.session_id, command);

if (version_mismatch || patch_contract_.requiresFullRefresh(...)) {
    response.requires_full_refresh = true;
}
```

问题：

```text
客户端 projection_version 已经过期时，后端仍然执行了命令。
只是执行后告诉客户端 requires_full_refresh。
```

这会导致：

```text
1. 玩家看到旧世界状态，却能提交动作改变真实世界。
2. 多客户端 / 断线重连 / 延迟包场景下，旧命令可能污染世界状态。
3. 后续开放世界多人或 NPC 并发时，状态一致性风险扩大。
```

P53 设计要求是：

```text
版本不一致时，客户端必须进入 full refresh，不允许强行合并 patch。
```

对于服务端而言，更安全的做法应该是：

```text
版本不一致时不执行命令，直接返回 Blocked / RequiresFullRefresh。
```

严重性：**阻塞**。

必须修复：

```text
1. 在 resolveCommand / pipeline.execute 之前检查 known_projection_version。
2. 如果版本不一致，直接返回 safe response。
3. response.requires_full_refresh = true。
4. projection_patch.requires_full_refresh = true。
5. new_projection_version 不应因该请求推进。
6. 增加测试：stale command 不得执行，不得推进 projection version。
```

## 5.3 阻塞问题三：option_id 不是会话内权威令牌

位置：

```text
backend/src/client_protocol/client_available_command_adapter.cpp
```

问题代码：

```cpp
if (option_id.find("opt_wait_") == 0) { ... }
if (option_id.find("opt_inspect_") == 0) { ... }
```

问题：

```text
当前只要 option_id 以 opt_wait_ 或 opt_inspect_ 开头，就会被接受。
没有校验该 option_id 是否来自当前 session 的 latest available_commands。
没有校验 option_id 是否属于当前 actor。
没有校验 option_id 是否属于当前 projection_version。
没有校验 option_id 是否已经过期。
```

这会导致：

```text
opt_wait_anything 可以被伪造成合法命令。
旧 projection 的 option_id 可能继续生效。
其他 actor 的 option_id 可能被复用。
```

严重性：**阻塞**。

必须修复：

```text
1. available_commands 必须由后端按 session/projection_version 生成并登记。
2. materializeOption 必须校验 option_id 是否存在于当前 session 的可用命令快照。
3. option_id 必须绑定 session_id、actor_key、projection_version 或 command option registry。
4. 过期 option_id 必须返回 Blocked，不得执行。
5. 增加测试：伪造 opt_wait_xxx 不得执行。
6. 增加测试：上一版本 option_id 在版本推进后不得执行。
7. 增加测试：其他 actor 的 option_id 不得执行。
```

## 5.4 阻塞问题四：Codec 丢失 projection / patch 内容

位置：

```text
backend/src/client_protocol/client_protocol_codec.cpp
```

问题代码：

```cpp
json encodeWorldProjection(const ClientWorldProjection& v) {
    return json{
        {"projection_version", v.projection_version},
        {"actor_key", v.actor_key},
        {"active_layer_key", v.active_layer_key},
        {"safe_summary_keys", v.safe_summary_keys}
    };
}

json encodePatch(const WorldProjectionPatchDto& v) {
    return json{
        {"base_projection_version", v.base_projection_version},
        {"new_projection_version", v.new_projection_version},
        {"requires_full_refresh", v.requires_full_refresh}
    };
}
```

问题：

```text
ClientWorldProjection 中的 visible_cells / visible_entities / inventories / knowledge / area_effects 没有进入 JSON。
WorldProjectionPatchDto 中的 changed_cells / changed_entities / changed_inventories / changed_knowledge / changed_area_effects 没有进入 JSON。
```

这会导致：

```text
后续 P54 客户端拿不到地图格子。
拿不到物品实例。
拿不到背包。
拿不到知识变化。
拿不到区域效果。
patch roundtrip 后状态丢失。
```

当前测试没发现，是因为测试没有构造非空 projection / patch 做 codec roundtrip。

严重性：**阻塞**。

必须修复：

```text
1. 补全 ClientWorldProjection 全字段 JSON 编解码。
2. 补全 WorldProjectionPatchDto 全字段 JSON 编解码。
3. 补全 WorldCellPatchDto / WorldEntityPatchDto / InventoryPatchDto / KnowledgePatchDto / AreaEffectPatchDto 的嵌套编解码。
4. 增加测试：非空 projection roundtrip 后字段不丢失。
5. 增加测试：非空 patch roundtrip 后 changed_* 不丢失。
6. 增加测试：mergePatch 后再 encode/decode，客户端仍能得到相同安全投影。
```

## 5.5 阻塞问题五：WorldCommandDto 模式缺少服务端权限校验

位置：

```text
backend/src/client_protocol/client_command_gateway.cpp
```

问题：

```text
ClientSubmitMode::WorldCommandDto 模式下，客户端提交的 command 基本被直接拿去执行。
只补了 client_sequence，没有强制校验 actor/source/target 是否允许。
```

当前 `WorldCommandPipeline` 的部分校验仍是 stub，不能把安全责任完全下放给 pipeline。

风险：

```text
客户端可以构造未来命令。
客户端可以构造非本人 actor。
客户端可以设置不该由 PlayerInput 发起的 source。
客户端可以绕过 available_commands。
```

严重性：**阻塞**。

必须修复：

```text
1. WorldCommandDto 模式必须只作为“高级客户端 / 调试客户端”路径。
2. 默认正式客户端建议只允许 OptionId。
3. 如果保留 WorldCommandDto，必须经过 ClientCommandPolicy 校验。
4. 至少校验 actor_key、source、command_kind、target 是否符合当前 session 和 available_commands。
5. 增加测试：非法 source / 非本人 actor / 未授权 command_kind 必须 blocked。
```

## 6. 非阻塞但必须进入后续优化的问题

## 6.1 available_commands 当前仍是协议测试 stub

位置：

```text
backend/src/client_protocol/client_available_command_adapter.cpp
```

当前只返回：

```text
Wait
Inspect
```

这可以作为 P53 最小协议测试，但不能成为 P54 正式客户端的数据来源。

后续要求：

```text
1. P54 前必须确认 available_commands 能聚合 P43-P52 的真实命令来源。
2. 不能让前端为了能玩自己生成采集、移动、制作按钮。
3. 如果 P53 暂不接真实玩法，也必须在文档中标记为 P54 前置阻塞。
```

## 6.2 SessionState 缺少客户端序列防重放

当前 session 记录：

```text
session_id
client_id
actor_key
layer_key
projection_version
```

但没有记录：

```text
last_client_sequence
processed_command_ids
option_snapshot_version
```

后续风险：

```text
同一个 client_sequence 可以重复提交。
网络重试可能导致命令重复执行。
旧 option_id 难以判定是否过期。
```

建议 P53 修复或 P54 前补齐。

## 6.3 Refresh 没有处理 known_projection_version 语义

`ClientRefreshRequest` 带了 `known_projection_version`，但 `refresh()` 当前直接返回当前 projection。

这不一定是错误，但协议应该明确：

```text
refresh 永远返回 full projection；
或 refresh 根据 known version 返回 no-op / full projection。
```

建议文档和测试补清楚。

## 7. 测试覆盖不足

当前测试虽然全部通过，但主要验证了“happy path”和浅层架构扫描。缺少以下关键测试：

```text
1. 客户端伪造 selected_actor_key 不得改变执行 actor。
2. 客户端提交 WorldCommandDto.actor_key != session actor 必须被拒绝或规范化。
3. stale projection command 不得执行 pipeline，不得推进 projection version。
4. 伪造 opt_wait_fake_actor 不得通过 materialize。
5. 旧 projection_version 的 option_id 不得继续使用。
6. command response 的 projection_patch.changed_* codec roundtrip 不丢字段。
7. bootstrap / refresh 的 full_projection 非空字段 codec roundtrip 不丢字段。
8. client_sequence 重复提交必须 blocked 或幂等。
9. client_id 与 session 创建者不匹配时必须拒绝。
10. WorldCommandDto 模式下非法 source 必须 blocked。
```

## 8. 复审前必须完成的修复清单

```text
[ ] ClientSessionGateway 增加 session actor / client / layer 只读查询接口。
[ ] ClientCommandGateway 不再信任 request.selection_context.selected_actor_key。
[ ] WorldCommandDto.actor_key 必须与 session actor 一致，否则拒绝或规范化。
[ ] known_projection_version 不一致时，必须在 pipeline.execute 前返回 requires_full_refresh，不执行命令。
[ ] available option 必须绑定 session/projection_version/actor，materialize 只能从当前有效快照中取。
[ ] Codec 必须完整支持 projection / patch 的 changed_* 与 visible_* 字段。
[ ] 增加安全边界测试和非空 projection/patch roundtrip 测试。
[ ] 更新 P53 测试文档，明确这些测试是验收项。
```

## 9. 审核结论说明

P53 当前实现方向是对的：模块独立、没有前端 UI、没有 H5 命名污染、测试框架也已经建立。

但它现在更像“协议雏形 + happy path harness”，还不是可以支撑 P54 正式客户端的稳定协议层。

尤其是：

```text
actor 权限
版本过期不执行
option_id 权威化
projection/patch 完整传输
```

这四项是客户端协议的根基，必须修复后才能进入 P54。

最终结论：**P53 暂不验收，需修复后复审。**
