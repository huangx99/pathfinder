## P0 问题：P24 状态草稿没有被完整合并

状态：必须修复。

位置：

```text
backend/src/tribe/tribe_conflict.cpp:787
backend/src/tribe/tribe_conflict.cpp:795
backend/src/tribe/tribe_conflict.cpp:847
backend/src/tribe/tribe_conflict.cpp:851
```

问题：

`ConflictStateDraftBuilder::buildForTribe()` 声称合并 P24 `CombatCoordinationStateDraft`，但当前只使用了：

```text
csd.morale_delta
csd.trust_delta
```

P24 的以下字段没有进入 P25 状态草稿：

```text
csd.safety_pressure
csd.casualty_pressure
csd.knowledge_conflict_pressure
```

更严重的是，函数末尾强制执行：

```cpp
draft.trust_delta = clampDelta(csd.trust_delta);
```

这会覆盖前面根据 P25 冲突结果计算出的 `trust_delta`，导致：

```text
Standoff / LossPressure / ForcedRetreat 等冲突结果对 trust_delta 的影响被抹掉。
```

为什么这是 P0：

```text
P25 设计要求“聚合 P24 coordination_state_draft 和 P25 conflict draft”。
当前实现只部分读取 P24，且覆盖 P25 自己的冲突影响。
这会让共同作战和族群冲突的状态影响链不可信。
```

修复要求：

```text
1. 明确 P24 -> P25 字段映射：
   - csd.morale_delta -> ConflictStateDraft.morale_delta baseline
   - csd.trust_delta -> ConflictStateDraft.trust_delta baseline
   - csd.safety_pressure -> ConflictStateDraft.safety_pressure_delta baseline 或合并项
   - csd.casualty_pressure -> ConflictStateDraft.loss_pressure_delta / casualty pressure 等价摘要
   - csd.knowledge_conflict_pressure -> ConflictStateDraft.split_risk_delta 或 reason key 摘要
2. P25 outcome 的 delta 应该在 baseline 上叠加，而不是最后被覆盖。
3. 所有 delta 必须 clamp 到 -1..1。
4. 增加测试：P24 safety/casualty/knowledge_conflict pressure 输入非 0 时，P25 draft 能反映出来。
5. 增加测试：Standoff/LossPressure 产生的 trust_delta 不会被 P24 trust_delta 覆盖为 0。
```

## P0 问题：TruceOffered 几乎不可达，停战规则与设计不符

状态：必须修复。

位置：

```text
backend/src/tribe/tribe_conflict.cpp:321
backend/src/tribe/tribe_conflict.cpp:419
backend/src/tribe/tribe_conflict.cpp:948
```

当前逻辑：

```cpp
if (participant.loss_pressure_summary >= 0.6 &&
    pressure.resource_pressure < 0.4 &&
    hostility <= HostilityState::Hostile) {
    return ConflictIntentKind::NegotiateTruce;
}

if (source_intent == ConflictIntentKind::NegotiateTruce &&
    target_strength < options.truce_pressure_threshold) {
    return ConflictOutcomeKind::TruceOffered;
}
```

问题：

默认 `truce_pressure_threshold = 0.25`，但测试里能停战是因为把目标强度压得很低。
实际设计要求是：

```text
双方 loss_pressure 高，resource_pressure 降低，一方提出停战，另一方压力低或继续冲突收益低，可以 TruceOffered。
```

当前代码把停战结果绑定到“对方 strength < 0.25”，这太窄，会导致：

```text
两个都受损严重但仍有基本协作能力的族群，反而无法提出停战。
```

为什么这是 P0：

```text
P25 任务卡明确把 TruceOffered 作为核心冲突结果之一。
当前实现的触发条件偏离“损失压力 + 资源压力下降”的设计语义。
```

修复要求：

```text
1. resolveOutcome 需要传入或读取 loss_pressure / resource_pressure / hostility 等停战上下文，不能只看 strength。
2. 推荐条件：
   - 任一方 intent = NegotiateTruce；
   - 双方或至少一方 loss_pressure_summary 较高；
   - resource_pressure 低于阈值；
   - 对方不是 Pursue/Raid 的高压继续冲突姿态；
   - 或双方 safety/loss pressure 已高到继续冲突收益低。
3. 增加测试：双方 loss_pressure 高、resource_pressure 低、双方 strength 中等时仍能 TruceOffered。
```

## P1 问题：投影压力等级忽略 target 族群压力

状态：需要修复。

位置：

```text
backend/src/tribe/tribe_conflict.cpp:754
backend/src/tribe/tribe_conflict.cpp:767
```

问题：

`ConflictProjectionBuilder::build()` 只读取：

```text
result.source_state_draft.loss_pressure_delta
result.source_state_draft.safety_pressure_delta
result.source_state_draft.retreat_pressure_delta
```

没有读取：

```text
result.target_state_draft.*
```

这会导致：

```text
如果 source 是强势方、target 是被迫撤退方，投影可能显示压力很低。
玩家看不到“对方被逼退/局势很紧张”的摘要强度。
```

修复要求：

```text
1. visible_pressure_level 应该综合 source 和 target 两边的 loss/safety/retreat pressure。
2. state_pressure_changed 也应该综合两边。
3. 增加测试：target 有 retreat/loss pressure，source 没有时，projection 仍显示压力变化。
```

## P1 问题：Forbidden key 校验存在误伤风险

状态：需要调整或明确策略。

位置：

```text
backend/src/tribe/tribe_conflict.cpp:246
```

问题：

当前 forbidden token 包含非常短的：

```text
hp
```

使用 `lower.find(token)` 子串匹配，会误伤任意包含 `hp` 的 key。

虽然这不是当前测试失败点，但长期会让安全 key 命名很脆弱。

修复建议：

```text
1. 对短 token 使用分隔符匹配，例如 _hp_、hp_delta、true_hp、actual_hp。
2. 或只保留明确危险词：true_hp、actual_hp、hp_delta、hp_remaining。
3. 保持 death/kill/loot/damage 等危险词继续严格拒绝。
4. 增加测试：普通安全 key 不应被 hp 子串误伤。
```

## P2 问题：事件覆盖不完整

状态：建议修复。

位置：

```text
backend/src/tribe/tribe_conflict.cpp:1076
```

问题：

每次只输出一个 `ConflictEvent`。

设计中至少有以下事件类型：

```text
RelationChanged
HostilityChanged
IntimidationApplied
RetreatTriggered
ResourceContested
ConflictPressureChanged
TruceProposed
```

当前实现会用 outcome 覆盖 event_kind，因此如果同一次冲突既有关系变化又有资源争夺，只保留一个事件。

第一版不是必须一次输出多个事件，但建议至少：

```text
1. 如果 hostility 变化，单独输出 HostilityChanged。
2. 如果 pressure 变化，输出 ConflictPressureChanged。
3. outcome 行为事件另行输出。
```

如果暂不实现，也要在代码注释或文档中明确“第一版每次只输出主事件”。
