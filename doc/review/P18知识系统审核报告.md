# P18 知识系统审核报告

审核日期：2026-05-17  
审核范围：P18 knowledge 模块、知识形成服务、知识仓库、知识投影、P18 单元/集成测试、阶段边界扫描。  
审核依据：`doc/44_P18知识系统任务卡设计.md`、`context_packs/knowledge_p18.md`、`doc/13_知识系统设计.md`、P15/P16/P17 已完成设计与实现。

## 1. 审核结论

**结论：暂不通过，需修复后复审。**

P18 当前实现已经建立了 knowledge 模块骨架，编译、P18 定向测试和全量测试均通过；但存在多处设计契约级问题，集中在：

1. `KnowledgeFormationService` 的失败分支会返回 `Result::ok(Skipped)`，但结果自身 `validateBasic()` 失败。
2. `KnowledgeFormationService` 把所有 planner 错误都吞成 `Skipped`，包括 hidden truth / 输入非法等应失败或 `Rejected` 的情况。
3. `KnowledgeFormationInput` 没有校验 owner 与 `MemorySummary.key.owner` 一致，导致 A Agent 可以从 B Agent 的记忆形成自己的知识。
4. TestOnly 默认未真正拒绝，多个 DTO 可以合法通过 TestOnly。
5. Hidden truth guard 覆盖不完整，例如 `KnowledgePredicate.action_key = hidden_truth` 可以通过。
6. `KnowledgeFormationPlanner` 允许 `HasEffect` 知识缺少 `effect_key`，会形成没有效果语义的知识声明。

这些问题不是编译问题，而是 P18 最核心的“知识可信边界、主观归属、证据安全、DTO 契约”问题。建议修复后再标记 P18 完成。

## 4. 阻塞问题

### BLOCKER-1：Skipped 结果仍然是 ok + invalid DTO

位置：

- `backend/src/knowledge/knowledge_formation.cpp:215`
- `backend/src/knowledge/knowledge_formation.cpp:224`
- `backend/src/knowledge/knowledge_formation.cpp:226`
- `backend/src/knowledge/knowledge_claim.cpp:401`
- `backend/src/knowledge/knowledge_claim.cpp:418`
- `backend/src/knowledge/knowledge_claim.cpp:361`

问题：

`KnowledgeFormationService::formFromMemorySummary()` 在 planner 失败时返回：

```text
Result::ok(KnowledgeFormationResult{ decision = Skipped, event_drafts = [invalid draft] })
```

但该 skipped event draft 的 `knowledge_id` 是空 ID，`status` 也是 `Unknown`。

`KnowledgeFormationResult::validateBasic()` 会遍历 `event_drafts` 并调用 `KnowledgeEventDraft::validateBasic()`，而 `KnowledgeEventDraft::validateBasic()` 要求 `knowledge_id` 非空。

额外探针确认：

```text
skipped_ok=1 decision=skipped validate_ok=0 event_count=1
```

影响：

- 违反 P18 任务卡“所有 Result::ok 返回 DTO 必须 validateBasic”。
- 和 P17 曾经出现过的 `Skipped + invalid plan` 是同类问题。
- 后续 AI/调用方无法信任 Service 的成功返回值。

建议修复：

优先方案：

```text
Skipped 结果不要生成 KnowledgeEventDraft。
只保留 decision=Skipped、reason_keys、warning_keys。
```

可选方案：

```text
如果必须保留 knowledge.formation_skipped event，则 KnowledgeEventDraft 需要明确支持 optional knowledge_id，且 validateBasic 根据 decision=Skipped 放宽 knowledge_id。
```

但 P18 第一版建议不要复杂化，直接不生成 skipped event draft。

必须补测试：

```text
formation_service_insufficient_evidence_returns_valid_skipped_result
```

断言：

```text
result.is_ok()
result.value().decision == Skipped
result.value().validateBasic().is_ok()
```

### BLOCKER-2：Service 把所有 Planner 错误都吞成 Skipped，包括非法输入和 hidden truth

位置：

- `backend/src/knowledge/knowledge_formation.cpp:215`
- `backend/src/knowledge/knowledge_formation.cpp:216`
- `backend/src/knowledge/knowledge_formation.cpp:217`

问题：

当前代码只要 `planner.planFromMemorySummary()` 返回 error，就无条件转换为：

```text
decision = Skipped
Result::ok(result)
```

这会把以下错误也吞掉：

- hidden truth 输入。
- owner 非法。
- summary 非法。
- predicate 非法。
- TestOnly 非法。
- 其他 validation_failed。

额外探针确认：

```text
hidden_service_is_ok=1 decision=skipped validate_ok=0
```

影响：

- 安全错误被包装成业务可接受的 skipped。
- 调用方无法区分“证据不足”与“输入越界/含 hidden truth”。
- P18 的 hidden truth 防线在 service 层不够硬。

建议修复：

```text
只把 ErrorCode::knowledge_insufficient_confidence 或明确 reason=insufficient_evidence 的错误转换成 Skipped。
validation_failed、hidden truth、TestOnly、非法 owner/subject/predicate 必须 Result::fail，或返回 decision=Rejected 且 KnowledgeFormationResult::ok() 不把 Rejected 当 ok。
```

必须补测试：

```text
formation_service_hidden_truth_input_returns_error_or_rejected
formation_service_invalid_summary_returns_error_or_rejected
formation_service_missing_safe_predicate_is_not_invalid_ok_result
```

### BLOCKER-3：KnowledgeFormationInput 未校验 owner 与 MemorySummary.key.owner 一致

位置：

- `backend/src/knowledge/knowledge_claim.cpp:297`
- `backend/src/knowledge/knowledge_formation.cpp:35`
- `backend/src/knowledge/knowledge_formation.cpp:244`

问题：

P18 任务卡要求：

```text
owner 必须与 summary.key.owner 可转换匹配。
```

当前 `KnowledgeFormationInput::validateBasic()` 只分别校验 `owner.validateBasic()` 和 `summary.validateBasic()`，没有校验二者是否一致。

额外探针确认：

```text
owner_mismatch_plan_ok=1
```

也就是说：

```text
agent_999 可以拿 agent_001 的 MemorySummary 形成自己的 KnowledgeClaim。
```

影响：

- 破坏“知识是主体主观经验沉淀”的核心设定。
- 允许跨主体偷取记忆形成知识。
- 后续 P20 传播会失去“谁真正知道什么”的边界。

建议修复：

新增 owner 转换/匹配函数，例如：

```text
KnowledgeOwnerKind::Agent + entity_id == MemoryOwnerKind::Agent + entity_id
KnowledgeOwnerKind::Actor + entity_id == MemoryOwnerKind::Actor + entity_id
KnowledgeOwnerKind::Tribe + tribe_id == MemoryOwnerKind::Tribe + tribe_id
```

在 `KnowledgeFormationInput::validateBasic()` 或 Planner 入口强制校验。

必须补测试：

```text
formation_rejects_owner_summary_owner_mismatch
```

### BLOCKER-4：TestOnly 默认未真正拒绝

位置：

- `backend/src/knowledge/knowledge_claim.cpp:17`
- `backend/src/knowledge/knowledge_claim.cpp:48`
- `backend/src/knowledge/knowledge_claim.cpp:76`
- `backend/src/knowledge/knowledge_claim.cpp:125`
- `backend/src/knowledge/knowledge_claim.cpp:209`
- `backend/src/knowledge/knowledge_types.cpp:38`
- `backend/src/knowledge/knowledge_types.cpp:73`
- `backend/src/knowledge/knowledge_types.cpp:118`
- `backend/src/knowledge/knowledge_types.cpp:180`
- `backend/src/knowledge/knowledge_types.cpp:213`

问题：

P18 任务卡要求 TestOnly 默认不允许进入正式 DTO。

当前：

- `KnowledgeOwnerKind::TestOnly` 只要求 `external_key` 非空即可通过。
- `KnowledgeSubjectKind::TestOnly` 没有被拒绝。
- `KnowledgeRelationType::TestOnly` 没有被拒绝。
- `KnowledgeEvidenceKind::TestOnly` 没有被拒绝。
- `KnowledgeStatus::TestOnly` 没有被拒绝。
- `KnowledgeFormationOptions.allow_test_only` 存在但没有真正贯穿到 DTO 校验。

额外探针确认：

```text
test_owner_valid=1
```

现有安全测试只把 `TestOnly owner.external_key` 置空，所以失败原因是 missing external_key，而不是 TestOnly gate 生效。

影响：

- 测试专用状态可能进入正式知识仓库。
- 后续 AI 可能误以为 TestOnly 是可用业务值。

建议修复：

```text
所有正式 DTO validateBasic 默认拒绝 TestOnly。
如果确实需要 TestOnly，应只在测试专用 builder 或显式 allow_test_only 路径中使用，不能默认通过。
```

必须补测试：

```text
knowledge_owner_test_only_rejected_even_with_external_key
knowledge_subject_test_only_rejected
knowledge_relation_test_only_rejected
knowledge_evidence_kind_test_only_rejected
knowledge_status_test_only_rejected
```

### BLOCKER-5：Hidden Truth Guard 覆盖不完整

位置：

- `backend/src/knowledge/knowledge_claim.cpp:76`
- `backend/src/knowledge/knowledge_claim.cpp:83`
- `backend/src/knowledge/knowledge_claim.cpp:86`
- `backend/src/knowledge/knowledge_claim.cpp:17`
- `backend/src/knowledge/knowledge_claim.cpp:48`

问题：

P18 任务卡要求扫描：

```text
KnowledgeOwner.external_key
KnowledgeSubject.related_subject_ids / relation_group_key
KnowledgePredicate.action_key / effect_key / value_summary_key / polarity_key / safe_tags
```

当前实现只扫描了部分字段：

- `KnowledgePredicate` 扫描了 `effect_key`、`value_summary_key`、`safe_tags`。
- 没有扫描 `action_key`。
- 没有扫描 `polarity_key`。
- `KnowledgeOwner` 没有扫描 `external_key`。
- `KnowledgeSubject` 没有扫描 `related_subject_ids`、`relation_group_key`。

额外探针确认：

```text
predicate_hidden_action_valid=1
```

影响：

- hidden truth 可以从未扫描字段绕进 KnowledgeClaim。
- P18 的“知识不是 hidden truth”防线不完整。

建议修复：

补全所有字符串字段扫描，至少包括：

```text
KnowledgeOwner.external_key
KnowledgeSubject.subject_id
KnowledgeSubject.subject_type_key
KnowledgeSubject.related_subject_ids
KnowledgeSubject.relation_group_key
KnowledgeSubject.safe_tags
KnowledgePredicate.action_key
KnowledgePredicate.effect_key
KnowledgePredicate.polarity_key
KnowledgePredicate.value_summary_key
KnowledgePredicate.safe_tags
KnowledgeFormationPlan.plan_key
KnowledgeEventDraft.event_key
KnowledgeStateChangeDraft.change_key
```

必须补测试：

```text
predicate_action_key_hidden_truth_rejected
predicate_polarity_key_hidden_truth_rejected
owner_external_key_hidden_truth_rejected
subject_related_ids_hidden_truth_rejected
subject_relation_group_key_hidden_truth_rejected
```

### BLOCKER-6：HasEffect 知识可以缺少 effect_key

位置：

- `backend/src/knowledge/knowledge_formation.cpp:73`
- `backend/src/knowledge/knowledge_formation.cpp:78`
- `backend/src/knowledge/knowledge_formation.cpp:91`
- `backend/src/knowledge/knowledge_claim.cpp:76`

问题：

P18 的核心表达是：

```text
主体 + 行为 + 效果 + 条件 + 证据 + 置信度
```

任务卡也要求：

```text
target_relation、action_key、effect_key 必须由安全认知/记忆摘要传入。
```

当前 Planner 只要求：

```text
target_relation != Unknown
action_key 非空
```

不要求 `effect_key` 非空。

额外探针确认：

```text
missing_effect_plan_ok=1
```

影响：

- 可以形成“红果 + Eat + 无效果”的知识。
- 这种 KnowledgeClaim 对 Agent 行动、传播、文明前置都缺少可解释语义。
- 后续 P19/P20 会难以判断知识内容。

建议修复：

对于以下关系必须要求 `effect_key` 或等价安全字段：

```text
HasEffect
HasRisk
IsEdibleUnder
IsDangerousUnder
IsUsableFor
ReactsWith
TransformsInto
Produces
Prevents
FailsUnder
```

至少 P18 第一版要对 `HasEffect` 强制 `effect_key` 非空。

必须补测试：

```text
planner_rejects_has_effect_without_effect_key
service_missing_effect_key_returns_error_or_valid_skipped_not_invalid_ok
```

## 5. 重要非阻塞问题

### ISSUE-1：KnowledgeEvidence 未校验 memory_evidence_refs 内部合法性

位置：

- `backend/src/knowledge/knowledge_claim.cpp:125`

`KnowledgeEvidence::validateBasic()` 检查了 source trace 是否存在，但没有逐个调用 `MemoryEvidenceRef::validateBasic()`。

建议补充：

```text
for memory_evidence_refs: ref.validateBasic()
```

否则无效或含 forbidden reason 的 P16 evidence ref 可能被包进知识证据。

### ISSUE-2：FormationService 没有把 plan 写入 CreatedClaim result

位置：

- `backend/src/knowledge/knowledge_formation.cpp:304`

`KnowledgeFormationResult` 设计有 `plan` 字段。当前成功形成 claim 后只设置 `claim`，未设置 `plan`。

这不一定破坏功能，但会降低审计能力。建议 CreatedClaim 时保留 `result.plan = plan`，前提是 plan.validateBasic() 通过。

### ISSUE-3：KnowledgeEventDraft 校验字段不足

位置：

- `backend/src/knowledge/knowledge_claim.cpp:357`

`KnowledgeEventDraft::validateBasic()` 未校验：

```text
relation_type != Unknown
status != Unknown
formation decision != Unknown
```

建议补充，或明确 Skipped 事件另建结构。

### ISSUE-4：测试覆盖偏乐观

当前 526/526 通过，但未覆盖：

- Service skipped result validateBasic。
- Service hidden truth 输入不能被吞成 skipped。
- owner 与 summary owner 不匹配。
- TestOnly 带合法 external_key 也必须被拒绝。
- predicate.action_key hidden truth。
- HasEffect 缺 effect_key。

建议修复时同步补齐这些测试。

## 6. 阶段边界结论

P18 当前没有明显越界到：

- P19 KnowledgeConflictResolver / KnowledgeRevision。
- P20 Propagation。
- H5/API/HTTP/WebSocket/JSON。
- SaveManager。
- RL reward/done。
- AgentRuntime / Policy 直接写知识。

但 P18 内部安全边界和 DTO 契约尚未达标，因此不能通过。

## 7. 建议修复清单

P0 必修：

1. 修复 `KnowledgeFormationService` skipped 结果 validateBasic 失败问题。
2. Service 只将证据不足转 Skipped，validation/security 错误必须 fail 或 Rejected。
3. 校验 `KnowledgeFormationInput.owner` 与 `summary.key.owner` 一致。
4. 默认拒绝所有 TestOnly DTO 值。
5. 补全 hidden truth guard 覆盖字段，尤其 `KnowledgePredicate.action_key`。
6. `HasEffect` 至少必须要求 `effect_key` 非空。
7. 补齐上述失败路径测试。

P1 建议：

1. `KnowledgeEvidence::validateBasic()` 校验每个 `MemoryEvidenceRef`。
2. CreatedClaim 结果保留 valid plan，增强审计能力。
3. `KnowledgeEventDraft::validateBasic()` 补齐 relation/status/decision 校验。
