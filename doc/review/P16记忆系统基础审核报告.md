# P16 记忆系统基础审核报告

审核日期：2026-05-17  
审核对象：P16 记忆系统基础实现  
审核结论：暂不通过，需修复后复审

---

## 1. 总结论

P16 当前实现已经建立了记忆系统的基本骨架：

- 已新增 `pathfinder::memory` 命名空间与 `pathfinder_memory` 静态库。
- 已实现记忆枚举、DTO、候选生成、写入、强化、晋升、衰减、遗忘、基础查询、事件草稿和状态变更草稿。
- 已接入 P15 cognition v2，能够从 `CognitionEvidenceRecord` 生成 `MemoryCandidate`。
- 未发现 P17 记忆压缩/复杂召回、P18 知识、传播、H5/API、SaveManager、HTTP/WebSocket/JSON、RL reward/done 等实质越界实现。
- 构建、P16 定向测试、相关回归、全量测试均通过。

但是，本次审核重点检查了“实现逻辑是否正确”。结论是：测试通过不代表逻辑可交付。当前实现存在多个 P16 核心契约缺口，尤其是输入安全、证据追踪、高风险保护、隐藏真值防线。P16 是后续 P17 记忆压缩、P18 知识系统、P20 传播系统的基础层，如果这些缺口不修，后续 AI 按文档实现时会继承错误边界。

因此结论为：暂不通过，需修复后复审。

## 5. 阻塞问题

### BLOCKER-1：`MemoryCandidateFactory` 没有拒绝非法 P15 输入

设计要求：`memory_boundary_security_test` 必须验证 `MemoryCandidateFactory 不接受 Unknown/TestOnly 默认输入`。

设计位置：`doc/42_P16记忆系统基础任务卡设计.md:1525`

当前实现位置：`backend/src/memory/memory_writer.cpp:54`

问题：

- `fromCognitionEvidence()` 不调用 `evidence.validateBasic()`。
- 它直接读取 `evidence.key.subject.kind`、`evidence.key.target.kind`、`evidence.source_event_id` 等字段。
- Unknown/TestOnly/default 输入仍会返回 `Result::ok(MemoryCandidate)`。
- 当前安全测试名称写的是“rejects”，但断言却是 `result.is_ok()`，再依赖 `candidate.validateBasic()` 失败。

当前测试位置：`backend/tests/integration/p16/memory_boundary_security_test.cpp:78`

这不是“工厂拒绝”，而是“工厂生产了一个非法候选，后续校验才发现”。这会误导后续研发，让非法输入进入记忆写入链上更深的位置。

必须修复：

- `MemoryCandidateFactory::fromCognitionEvidence()` 首行校验 `evidence.validateBasic()`。
- 显式拒绝 `CognitionSubjectKind::Unknown/TestOnly`、`CognitionTargetKind::Unknown/TestOnly`，除非后续有测试专用 options；P16 当前没有这个 options。
- 工厂返回前调用 `candidate.validateBasic()`，失败则返回 `Result::fail`。
- 修改安全测试：非法 evidence 应断言 `result.is_error()`，不能断言 `result.is_ok()`。

### BLOCKER-2：P15 enum 到 P16 enum 使用裸 `static_cast`，映射不安全

当前实现位置：

- `backend/src/memory/memory_writer.cpp:61`
- `backend/src/memory/memory_writer.cpp:65`

问题：

```cpp
candidate.owner.kind = static_cast<MemoryOwnerKind>(evidence.key.subject.kind);
candidate.subject.kind = static_cast<MemorySubjectKind>(evidence.key.target.kind);
```

这依赖 P15/P16 枚举声明顺序永远一致。当前看起来“碰巧能用”，但后续任一 enum 增删成员，记忆拥有者或记忆对象就可能被错误映射，且编译器不会报错。

必须修复：

- 增加显式映射函数，例如 `mapCognitionSubjectKindToMemoryOwnerKind()`、`mapCognitionTargetKindToMemorySubjectKind()`。
- 所有 Unknown/TestOnly 或不支持值必须返回 `Result::fail`。
- 添加测试覆盖每个允许枚举和拒绝枚举。

### BLOCKER-3：`MemoryRecord::validateBasic()` 允许无证据记忆进入 Store

设计验收要求：所有记忆记录都保留 `source_event_id` 或 cognition evidence 引用。

设计位置：`doc/42_P16记忆系统基础任务卡设计.md:1753`

当前实现位置：

- `backend/src/memory/memory_record.cpp:116`
- `backend/src/memory/memory_record.cpp:154`
- `backend/src/memory/memory_store.cpp:8`

问题：

- `MemoryRecord::validateBasic()` 只在 `evidence_refs` 存在时循环校验。
- 它没有要求 `evidence_refs` 非空。
- `MemoryStore::put()` 只依赖 `record.validateBasic()`，因此手写的无证据记录可以进入 store。
- 当前 `memory_record_test.cpp` 甚至构造了无 `evidence_refs` 的 record 并断言通过。

当前测试位置：`backend/tests/unit/memory/memory_record_test.cpp:67`

这会破坏 P16 的可审计性和复盘链路。后续 P17/P18 无法判断一条记忆来自哪次事件或哪条认知证据。

必须修复：

- `MemoryRecord::validateBasic()` 要求 `evidence_refs` 非空。
- 至少一条 ref 必须包含 `source_event_id` 或 `cognition_evidence_id` / `cognition_claim_id`。
- `MemoryEvidenceRef::validateBasic()` 对 `CognitionEvidence`/`CognitionUpdate` 必须要求 `cognition_evidence_id` 或 `source_event_id` 非空。
- 修改所有测试构造器，补齐合法 evidence ref。

### BLOCKER-4：隐藏真值防线大小写敏感，可被绕过

设计要求隐藏真值和原始状态字段被拒绝。

设计位置：`doc/42_P16记忆系统基础任务卡设计.md:1301`、`doc/42_P16记忆系统基础任务卡设计.md:1754`

当前实现位置：`backend/src/memory/types.cpp:36`

问题：

```cpp
if (text.find(key) != std::string::npos) {
    return true;
}
```

该实现大小写敏感。它能挡住 `GameState`，但挡不住 `gamestate`、`GAMESTATE`、`Game_State` 等常见变体。P15 的 `containsHiddenTruthKey()` 已经做了 lowercase 标准化，P16 反而退化了。

必须修复：

- P16 hidden truth guard 应与 P15/P14 安全策略一致，先 lowercase，再匹配 lowercase forbidden key。
- 安全测试必须新增大小写变体：`gamestate`、`GAMESTATE`、`Health_Delta`、`HIDDEN_TRUTH`、`raw_state`。

### BLOCKER-5：高风险/关键记忆保护逻辑不完整

设计目标要求保护高风险、首次发现、强冲突、可教学候选等关键记忆。

设计位置：`doc/42_P16记忆系统基础任务卡设计.md:56`、`doc/42_P16记忆系统基础任务卡设计.md:1749`

当前实现位置：

- `backend/src/memory/memory_writer.cpp:78`
- `backend/src/memory/memory_writer.cpp:299`
- `backend/src/memory/memory_writer.cpp:312`
- `backend/src/memory/memory_writer.cpp:361`

问题：

- 工厂只对 `Critical` risk 设置 `protect_from_decay/protect_from_forgetting`，`High` risk 没有设置。
- 新建 high-risk 记忆会因为 `importance == Critical` 被直接设为 LongTerm/protected，这是偶然由 writer 的创建分支补上的。
- 但已有记忆被 high-risk 候选强化时，candidate 的保护标记仍是 false；如果 outcome 是 Warning 而不是 Accident，它可能只晋升 MidTerm，不会 protected/LongTerm。
- 这会造成同样的高风险经历：第一次发生会保护，后续作为强化发生反而可能不保护，逻辑不一致。

必须修复：

- High/Critical risk 应在工厂层显式映射为关键保护候选。
- writer 强化分支遇到 `candidate.importance == Critical` 或 `Warning/Accident/Discovery/Contradiction/TeachingRelated` 时，应按设计统一保护或晋升。
- 增加测试：已有普通记忆遇到 high-risk warning evidence 后必须 protected 或 LongTerm。

### BLOCKER-6：设计中的 Discovery / TeachingRelated / Contradiction 晋升语义没有完整落地

设计位置：

- `doc/42_P16记忆系统基础任务卡设计.md:709`
- `doc/42_P16记忆系统基础任务卡设计.md:711`
- `doc/42_P16记忆系统基础任务卡设计.md:713`
- `doc/42_P16记忆系统基础任务卡设计.md:1199`

当前实现位置：`backend/src/memory/memory_writer.cpp:94`

问题：

- `fromCognitionUpdate()` 读取了 `update_result.decision`，但没有在 Created 时加入 `Discovery`。
- teachable band 只设置 `teaching_candidate = true`，没有加入 `TeachingRelated`，也没有体现 MidTerm 候选语义。
- `Contradiction` 只在相似 memory 已匹配时计数；如果候选只有 Contradiction 且与旧 memory 没有 kind 交集，会新建一条孤立 contradiction memory，而不是强化/标记原记忆。
- `shouldPromoteShortToMid()` 只看 `Accident/Warning/Discovery`，不看 `Contradiction/TeachingRelated`。

必须修复：

- `fromCognitionUpdate()` 应按 `CognitionUpdateDecision::Created/Reinforced/Conflicted` 映射 `Discovery/Success/Failure/Contradiction`。
- teachable candidate 应加入 `TeachingRelated` 或有明确 reason key。
- contradiction 匹配不应完全依赖 `memory_kinds` 交集，至少同 owner+subject 的 contradiction 应能影响已有记录。
- 增加 fromCognitionUpdate 单元测试和冲突晋升测试。

## 8. 修复要求

建议按下面顺序修复，避免越修越乱：

### P0：安全入口修复

- 工厂入口先校验 P15 evidence/update。
- 去掉裸 `static_cast`，改为显式 enum mapping。
- 工厂返回前校验 candidate。
- 修正安全测试断言。

### P0：证据追踪修复

- `MemoryEvidenceRef` 必须要求可追踪来源。
- `MemoryRecord` 必须至少有一条 evidence ref。
- `MemoryStore::put()` 继续只接受 validate 通过的 record。

### P0：隐藏真值防线修复

- P16 forbidden key 检测改为 case-insensitive。
- 增加大小写变体安全测试。

### P1：关键记忆语义修复

- High/Critical risk 统一保护策略。
- Discovery/TeachingRelated/Contradiction 映射补齐。
- 高风险强化已有记忆的晋升/保护行为补测试。

### P1：校验和测试补强

- Event/state draft 校验补强。
- options 阈值顺序校验补强。
