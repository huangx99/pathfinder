# P60 正式客户端地图交互与区块保存触发实施审核报告

## 1. 审核结论

结论：不通过，P60 当前实现不能验收。

原因不是单个小问题，而是存在运行时崩溃、生命周期触发未真正接入正式客户端闭环、地图选择与可用动作权威性不足、对象交互链路没有覆盖 P60 设计要求等阻塞问题。

本次审核依据：

```text
doc/研发手册.md
doc/00_设计文档编写要求.md
doc/90_P60正式客户端地图交互与区块保存触发任务卡设计.md
AGENTS.md
```

P60 设计目标要求：正式客户端输入必须走后端 `available_commands` 与 `WorldCommandPipeline`，地图交互必须触发 region activity window 与 P59 seal/restore，采集、拾取、丢弃、制作等产物必须通过 runtime / P45 归属接口建立 `OnMap` 或 `InInventory` 状态，不能绕过 runtime 直接拼状态。

当前实现没有达到这些验收标准。

## 3. 阻塞问题

### 阻塞 1：正式客户端运行时集成出现段错误

位置：

```text
backend/include/pathfinder/client_http/client_server_runtime_factory.h
backend/src/client_http/client_server_runtime_factory.cpp
```

问题：`ClientServerRuntimeFactory` 的成员声明顺序和构造初始化顺序存在明显风险。

在头文件中，`option_bridge` 声明在 `registry`、`harvest_service`、`inventory_runtime` 之前；但构造函数初始化列表里：

```cpp
option_bridge(std::make_shared<ClientRuntimeCommandOptionBridge>(*world_runtime, registry, harvest_service.get(), inventory_runtime.get(), ...))
```

此时：

```text
registry 尚未构造完成。
harvest_service / inventory_runtime 尚未构造完成。
```

这属于 C++ 成员初始化顺序风险，可能导致未定义行为。当前 `client_runtime_bridge_*` 集成测试集中段错误，首要嫌疑就是这里。

影响：

```text
1. bootstrap 可能直接崩溃。
2. projection / available_commands 不能稳定生成。
3. 正式客户端不能作为 P60 验收入口。
```

修复要求：

```text
1. 调整 ClientServerRuntimeFactory 成员声明顺序，确保 registry、harvest_service、inventory_runtime 先于 option_bridge 构造。
2. 或者把 option_bridge 延迟到构造函数体内创建，并在所有依赖对象完成初始化后再注入 adapter。
3. 修复后必须重跑全部 client_runtime_bridge 集成测试。
```

### 阻塞 2：Region lifecycle 触发层没有真正接入正式客户端闭环

位置：

```text
backend/src/world_map_interaction/region_activity_window_service.cpp
backend/src/world_map_interaction/region_lifecycle_trigger_service.cpp
backend/src/client_http/client_http_gateway.cpp
backend/src/client_http/client_server_runtime_factory.cpp
```

P60 设计要求 bootstrap、refresh、移动成功、path step、采集、拾取、丢弃、制作、wait/tick、Agent 跨区等都必须触发 activity window 更新，并驱动 P59 seal/restore。

当前代码中可见：

```text
RegionActivityWindowService 被创建。
RegionLifecycleTriggerService 被创建。
但正式 ClientHttpGateway / ClientCommandGateway 路径没有调用 computeWindow + processWindow。
```

搜索结果显示 `processWindow()` 基本只在测试中调用，正式 bootstrap / refresh / command 执行链路没有接入。

影响：

```text
1. 玩家移动不会自动触发 region activity window。
2. 离开区块不会通过 P60 自动触发 seal。
3. 回到 cached region 的 restore 流程没有形成客户端闭环证明。
4. P60 “世界能自动记住变化”的核心目标没有完成。
```

修复要求：

```text
1. 在 command 成功后统一触发 activity window + lifecycle trigger。
2. bootstrap / refresh / wait/tick 也应按设计触发或至少返回 lifecycle hint。
3. lifecycle events 必须进入客户端响应或 patch/feed，不能只在服务内部产生。
```

### 阻塞 3：RegionActivityWindow 策略不符合 P60 设计

位置：

```text
backend/src/world_map_interaction/region_activity_window_service.cpp
```

具体问题：

```text
1. computePrewarmRegions() 直接返回空，未实现 prewarm ring。
2. 未 sealed 的 active region 一离开 keep_active 就直接进入 seal_candidate，seal_delay_ticks 没有真正作为“离开后等待 N tick 再 seal”的滞后策略。
3. region_sealed_at_ 需要 markSealed()，但 lifecycle trigger 没有在 seal 成功后回写 markSealed()，detach 延迟状态无法可靠推进。
4. collectBlockedRegionIds() 只按 actor 阻断，未实现活跃 Agent、danger event、ongoing command 的阻断。
```

这违反设计中：

```text
有活跃 Agent / danger 的 region 不 detach。
边界反复移动不能频繁 seal/detach 抖动。
prewarm regions 用于平滑客户端体验和后续跨区交互。
```

影响：

```text
1. 玩家在边界移动可能造成 seal/detach 抖动。
2. 有 NPC、野兽或危险事件的区块可能被错误卸载。
3. detach candidate 状态机不完整，自动保存触发不可靠。
```

修复要求：

```text
1. 增加 left_active_since 或 equivalent 状态，离开 keep_active 后先等待 seal_delay_ticks。
2. seal 成功后记录 sealed_at，再等待 detach_delay_ticks。
3. 实现 prewarm ring。
4. 接入 Agent / danger / ongoing command unload guard，暂时没有接口也要先补通用接口，不能 TODO 掉验收项。
```

### 阻塞 4：RegionLifecycleTriggerService 没有完整执行 seal/detach 状态机

位置：

```text
backend/src/world_map_interaction/region_lifecycle_trigger_service.cpp
```

具体问题：

```text
1. seal 成功后只产生 sealed event，没有通知 activity window 已 sealed。
2. detach_candidate 如果已经是 CachedSnapshot，只返回 detached event，但没有执行真实 detach，也没有验证 runtime 是否已卸载。
3. ActiveRuntime 的 detach 分支再次调用 sealRegion，但没有区分 seal 和 detach 两步，也没有把 detach 成功/失败状态返回给 activity service。
4. prewarm ensure 失败只放 warning，没有形成可验收的 restore_failed / generated / restored 生命周期反馈。
```

影响：

```text
1. P60 自动保存触发只像“事件日志”，不是完整生命周期控制。
2. region 是否真的从 runtime 卸载、是否从 snapshot 恢复，无法由 P60 流程证明。
3. 客户端 lifecycle feedback 不可信。
```

修复要求：

```text
1. 明确 seal、detach、restore/generate 三类事件来源和状态变化。
2. detach 必须调用 P59 / runtime 正式能力，不能只发 event。
3. restore failed 必须阻断，不允许 fallback generate。
4. 补充 fake client 集成测试证明离开、seal、detach、回来 restore 的完整链路。
```

### 阻塞 5：ClientMapSelectionService 没有完整遵守 projection / option 权威边界

位置：

```text
backend/src/world_map_interaction/client_map_selection_service.cpp
backend/src/client_http/client_http_gateway.cpp
```

具体问题：

```text
1. ClientMapSelectionRequest 有 known_projection_version，但 select() 没有校验版本是否 stale。
2. select() 直接 buildOptions(actor_key, layer_key)，没有基于当前 session 快照和 projection_version 做一致性校验。
3. HTTP select 会把 selection-specific options 写回 session available command 快照，但没有和 selection_id / version 绑定，可能覆盖普通命令快照。
4. ResourceNode 选择只按 command kind 过滤 Gather/Chop/Mine/Dig，没有匹配 request.resource_node_id，可能点击 A 资源返回 B 资源动作。
5. Cell 选择返回 Move/Inspect，但没有验证 option 的 target_coord 是否就是被选中的 cell。
6. summarizeCellObjects() 使用 snapshotForDebug() 查 resource_nodes，生产 selection 摘要依赖 debug snapshot，不符合正式投影边界。
```

影响：

```text
1. 过期选择可能生成可执行动作。
2. 目标动作可能错绑到其他对象。
3. 前端看到的按钮不一定对应当前点击对象。
4. 会破坏 P53/P54 中 option_id 快照权威策略。
```

修复要求：

```text
1. selection 必须校验 projection_version。
2. selection options 必须绑定 session、selection_id、projection_version 和目标对象。
3. target 过滤必须精确匹配 coordinate / entity_id / resource_node_id / inventory_item_id。
4. 用正式 projection/query port 替代 snapshotForDebug()。
```

### 阻塞 6：采集、拾取、丢弃 available command 存在实际执行风险

位置：

```text
backend/src/client_runtime_bridge/client_runtime_command_option_bridge.cpp
backend/src/world_inventory/world_inventory_command_handlers.cpp
```

具体问题：

```text
1. InventoryCommandOptionProvider::buildDropOptions() 调用 ensureActorInventory()。available option 查询不应产生状态写入或隐式创建背包。
2. Drop option 把 entry.entity_id 写入 target_item_key，但 DropCommandHandler 读取的是 target_entity_id 或 parameters["entity_id"]，因此从 option materialize 后的 Drop 命令可能缺少 item ref。
3. Harvest option 通过 node_kind_str == Plant/Tree/Stone/Ore 映射 command，但 ResourceHarvestService 实际更应该以 required_action_key 判断，当前逻辑对内容扩展不稳。
4. HarvestCommandOptionProvider 注入了 ResourceHarvestService&，但 buildOptions() 没有使用它做可采集性/工具/距离/剩余次数校验。
```

影响：

```text
1. 查询 available_commands 可能改变世界状态。
2. 丢弃按钮可能显示出来但执行失败。
3. 内容扩展后资源节点动作可能错配。
4. 采集动作没有真正由后端资源服务确认授权。
```

修复要求：

```text
1. available option 查询必须只读。
2. Drop option materialize 后必须能被 DropCommandHandler 正确消费。
3. Harvest option 应通过 ResourceHarvestService 或正式 query port 判断 required_action_key、charge、工具、距离等。
4. 补充 pickup/drop/harvest 从 option_id 到 command 到 runtime/P45 归属的集成测试。
```

3. 每个验收点必须有结果断言，不能只断言不崩溃。

## 4. 架构红线核查

### 4.1 WorldCommandPipeline

当前移动、基础 command 仍有 pipeline，但 P60 新增 selection/lifecycle 没有完整接回 command 响应链路。

结论：部分符合，但未完成。

### 4.2 前端 / 客户端规则边界

HTTP select 由后端提供，但 selection option 权威性不足，目标绑定和版本校验缺失。

结论：存在风险。

### 4.3 runtime / P45 归属红线

Drop/Pickup/Harvest 的 option 生成与执行链路没有被集成测试证明；Drop option 字段与 handler 读取字段不一致；available 查询有隐式创建 inventory 的风险。

结论：不满足 P60 验收。

### 4.4 region restore/generate 顺序

P59 可能已修，但 P60 没有把 restore/generate 顺序接入正式客户端 command/refresh/bootstrap 流程，也没有用 fake client 验证。

结论：P60 未证明。

## 5. 必须补充的测试

P60 修复后，至少需要新增或改造以下测试：

```text
1. client_runtime_bridge bootstrap 不崩溃，projection 非空，move options 正常。
2. bootstrap -> select cell -> available move -> command -> patch。
3. select resource node 精确匹配目标 node_id，只返回该节点动作。
4. harvest resource -> charge/depleted 变化 -> event -> experience/knowledge feedback。
5. pickup ground item -> InInventory -> 地图不再显示。
6. drop inventory item -> OnMap -> 目标 region 显示。
7. harvest/pickup/drop 后离开 region -> P59 seal/detach -> 回来 restore -> 状态保持。
8. stale projection_version 的 select/command 被拒绝。
9. stale selection option_id 被拒绝。
10. restore failed 阻断，不 fallback generate。
11. 有 active agent / danger 的 region 不 detach。
12. available command 查询不创建 inventory、不改 runtime 状态版本。
```

## 6. 修复顺序建议

建议按以下顺序修：

```text
1. 先修 ClientServerRuntimeFactory 初始化顺序，消除 client_runtime_bridge 段错误。
2. 修 Drop/Harvest/Inventory available option 的只读性和字段映射。
3. 修 ClientMapSelectionService 的 projection_version、selection_id、target 精确绑定。
4. 把 activity window + lifecycle trigger 接入 bootstrap / refresh / command 成功后流程。
5. 修 RegionActivityWindow 的 seal_delay、detach_delay、prewarm、Agent/danger block。
6. 重写 P60 测试，让测试走 fake client / protocol harness，而不是直接 DebugOnly 操作 runtime。
```

## 7. 最终结论

P60 当前实现不能验收。

最严重的问题是：

```text
1. 相关客户端运行时集成测试直接段错误。
2. P60 的自动 region seal/restore 触发没有接入正式客户端命令闭环。
3. 地图选择和 available_commands 的目标/版本权威性不足。
4. 采集、拾取、丢弃与 P45 归属闭环没有被证明，且 Drop option 存在字段错误。
5. 测试绕过正式入口，无法证明 P60 设计目标完成。
```

因此本阶段结论为：

```text
审核不通过。
需要按阻塞项修复后再复审。
```
