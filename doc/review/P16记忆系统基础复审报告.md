# P16 记忆系统基础复审报告

审核日期：2026-05-17  
审核对象：P16 记忆系统基础修复后实现  
复审结论：仍暂不通过，需补修 1 个核心逻辑问题后再复审

## 1. 总结论

本次复审确认：上次审核报告中的大部分阻塞项已经修复，整体质量明显提升。

已确认修复：

- `MemoryCandidateFactory` 已先校验 P15 `CognitionEvidenceRecord`，非法输入会返回 `Result::fail`。
- P15 enum 到 P16 enum 已从裸 `static_cast` 改为显式映射。
- `MemoryRecord` 已要求至少保留一条 `evidence_ref`。
- `MemoryEvidenceRef` 已要求 cognition source 至少具备 `cognition_evidence_id` 或 `source_event_id`。
- hidden truth guard 已改为大小写不敏感匹配。
- High/Critical risk reinforcement 已能保护已有记忆。
- `Discovery`、`TeachingRelated` 的 `fromCognitionUpdate()` 映射已补上。
- Options 阈值顺序、event/state draft 校验也做了补强。
- 构建、P16 定向测试、相关回归、全量测试均通过。

但仍有一个核心逻辑没有修完整：`Contradiction` 候选与已有相反记忆的匹配逻辑仍然不正确。当前实现只在“已有记忆本身也包含 Contradiction”时放宽匹配，不能覆盖最关键的场景：旧记忆是 `Success`，新证据是 `Failure + Contradiction`；或者旧记忆是 `Failure`，新证据是 `Success + Contradiction`。

这会导致冲突经验不更新旧记忆，而是新建一条孤立的矛盾记忆，且新建记录的 `contradiction_count` 仍为 0。对于 P16 的“记忆强化/冲突/晋升”主线，这是逻辑错误。

因此复审结论：仍暂不通过。只需要补修这个 contradiction 匹配问题并加测试，修完后可以再次复审。

## 4. 剩余阻塞问题

### BLOCKER-RECHECK-1：Contradiction 只能匹配已有 Contradiction，不能匹配被冲突的旧记忆

当前实现位置：`backend/src/memory/memory_writer.cpp:253`

当前逻辑：

```cpp
if (candidate_has_contradiction && record_has_contradiction) {
    return true;
}
```

之后继续要求 `candidate.memory_kinds` 与 `record.memory_kinds` 有交集。

这会漏掉最关键的冲突场景：

```text
旧记忆：owner=A, subject=berry_red, memory_kinds={Success}
新候选：owner=A, subject=berry_red, memory_kinds={Failure, Contradiction}
```

当前结果：

- owner/subject 相同。
- candidate 有 `Contradiction`，record 没有 `Contradiction`，不触发放宽匹配。
- `{Failure, Contradiction}` 与 `{Success}` 没有交集。
- `isSimilarMemory()` 返回 false。
- writer 走 create 分支，创建一条新记忆。
- 旧 `Success` 记忆没有追加 `Contradiction`。
- `contradiction_count` 没有增加。

这不符合 P16 的冲突记忆语义。冲突不是“只有和已有冲突记忆相似才算冲突”，而是“新证据与同 owner+subject 的旧经验冲突”。

必须修复：

1. 当 candidate 包含 `Contradiction` 时，同 owner + subject 的非 Forgotten record 应允许作为匹配对象，即使 memory kind 没有交集。
2. 如果存在多条同 subject 记录，应选择最新或最强的那条更新，保持当前 deterministic tie-break。
3. 更新后必须：
   - `contradiction_count += 1`
   - 追加 `MemoryKind::Contradiction`
   - 增加 `memory.contradiction_observed`
   - 根据现有晋升规则至少进入 `MidTerm`，必要时可进入 `LongTerm`
4. 增加单元测试：旧 `Success` + 新 `Failure/Contradiction` 必须更新旧记录，而不是创建新记录。
5. 增加 fromCognitionUpdate 流测试：`CognitionUpdateDecision::Conflicted` 生成的 candidate 写入已有相反记忆时，结果应为 `Reinforced` 或 `Promoted`，且 `updated_records[0].contradiction_count == 1`。

## 5. 非阻塞建议

### ISSUE-1：`MemoryIdFactory` 仍未校验 evidence ref

`MemoryIdFactory::makeMemoryId()` 已改成 `Result<MemoryId>`，并校验 owner/subject，这是进步。  
但它仍没有调用 `evidence_ref.validateBasic()`。

对应位置：`backend/src/memory/memory_writer.cpp:217`

因为 `MemoryWriteService` 在调用前已经校验 candidate，所以主路径暂时安全。但 `MemoryIdFactory` 是 public service，后续 AI 研发可能直接调用。建议补上 evidence ref 校验，避免公共接口留下半安全状态。

### ISSUE-2：边界测试中的 hidden truth 大小写场景可以更纯粹

`memory_boundary_security_test` 的大小写 hidden truth case 未设置 `candidate.evidence_ref.source_event_id`，导致候选即使没有 hidden truth 也会失败。

对应位置：`backend/tests/integration/p16/memory_boundary_security_test.cpp:107`

当前已有 `memory_types_test` 直接证明大小写匹配有效，所以不阻塞。但建议补齐 `source_event_id`，让该集成测试只验证 hidden truth，而不是混入 evidence ref 失败。

---

## 6. 复审结论

本轮修复质量整体较好，P16 已经接近通过。

但 `Contradiction` 逻辑仍是核心玩法链路的一部分：探索中出现相反经验时，系统必须能把“旧认知被冲突”记录到同一条记忆上，而不是创建孤立记录。这个问题会影响后续 P17 记忆检索、P18 知识冲突、P19 知识修正。

最终结论：仍暂不通过。

修复 `BLOCKER-RECHECK-1` 并补测试后，再次复审即可。
