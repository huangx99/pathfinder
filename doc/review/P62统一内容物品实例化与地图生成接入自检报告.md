# P62 统一内容物品实例化与地图生成接入自检报告

## 1. 自检结论

P62 本次实现通过自检：地图生成、地面物品、资源节点、采集产物已经接到 `ContentRegistry` 的统一 `object_key` 体系，旧的运行时 inspect-only 地图假物品生成逻辑已移除。

本阶段不是新增玩法内容，而是修复内容架构根基。修复后，地图上生成的物品不再是“地图专用假物品”，而是能继续进入背包、采集、反应、经验和知识系统的同一个对象身份。

## 2. 已完成内容

### 2.1 ContentRegistry 扩展

- 新增 `worldgen` 内容文件分类。
- 新增 `WorldgenProfileDefinitionContent` 及 terrain/resource/ground/drop/spawn safety DTO。
- `JsonContentLoader` 支持解析 `worldgen_profiles`。
- `ContentDraftRegistry` 和 `ContentRegistry` 支持保存和查询 worldgen profile。
- `ContentReferenceValidator` 会校验 worldgen 中引用的 `required_tool_key`、采集产物、地面物品、掉落来源和掉落产物是否存在于 objects。

### 2.2 WorldGeneration 接入内容定义

- `WorldGenerationService::registerContentProfiles()` 会把 `ContentRegistry` 中的 worldgen profile 转为运行时 `WorldgenProfile`。
- 地面物品生成时使用 `ObjectDefinitionContent.display_key/default_quantity/default_numeric/safe_tags`。
- 资源节点的采集产物只保存统一 `output_object_keys`，不再自造独立地图物品 key。
- 内置 `first_world` 保留地形/噪声兼容能力，但资源和地面物品由 JSON profile 覆盖。

### 2.3 Harvest 接入内容定义

- `ResourceHarvestService` 增加 `setContentRegistry()`。
- 采集生成产物时，如果已注入 registry，会强制要求产物 `entity_key` 能在 objects 中找到。
- 采集产物的显示 key、默认数值、标签来自 `ObjectDefinitionContent`。
- 保留无 registry 的测试兼容路径，但真实客户端运行时已注入 registry。

### 2.4 旧硬编码清理

- 删除 `WorldGridRuntime::generateInitialWorld()` 中旧的稀疏 inspect-only `unknown_bush` / `loose_stone` 实体生成逻辑。
- `loose_stone` 改为 `content/core/objects/basic_survival.json` 中的正式 object。
- `content/core/worldgen/first_world.json` 负责生成 `berry_bush`、`young_tree`、`loose_stone_node` 和地面 `loose_stone`。

## 3. 架构边界复查

### 3.1 是否仍有地图专用物品系统

未发现新的地图专用物品系统。

当前路径为：

```text
ContentRegistry objects
-> worldgen profile 引用 object_key
-> WorldGenerationService 转换 profile
-> ResourceDistributionGenerator 生成 entity/resource node draft
-> WorldGenerationApplier 写入 runtime OnMap/resource node
-> Harvest/Inventory/Reaction/Knowledge 继续使用同一 key
```

### 3.2 是否绕过 Command/Runtime 权威

本阶段没有新增绕过 Command 的前端玩法入口。地图生成和采集仍通过既有 runtime/applier/harvest 服务，状态落地仍由 runtime 权威模块处理。

### 3.3 是否把具体内容写死在核心服务

核心服务不再写死 `loose_stone` 这类地面实体。测试中为了构造最小 registry 有测试专用内容对象，这是测试夹具，不是运行时内容包。

当前仍保留内置 terrain/noise profile 作为兼容 fallback；资源、地面物品和可玩内容来源已迁移到 JSON worldgen。

### 3.4 客户端边界

`ClientServerRuntimeFactory` 作为后端装配入口会加载 `ContentRegistry` 并注入 worldgen/harvest 服务；加载路径会从当前工作目录向上查找 `content/core/manifest.json`；如果 strict content 加载或 worldgen profile 注册失败，启动期直接报错，不允许静默退回空内容地图。HTTP 网关、静态文件服务和前端仍不能读取 content JSON、不能推导玩法规则、不能绕过 available_commands。

对应架构扫描已调整为禁止 HTTP/前端传输层读内容规则，同时允许后端 runtime factory 做服务装配。

## 4. 测试结果

已执行：

```text
cmake --build build/backend -j2
ctest --test-dir build/backend -R '^world_generation_' --output-on-failure
ctest --test-dir build/backend -R 'content_|world_harvest|world_reaction|client_http|client_runtime_bridge|p60' --output-on-failure
ctest --test-dir build/backend -R 'content_loader|world_generation_|world_harvest|client_runtime_bridge_bootstrap_non_empty|architecture_client_http_no_content_json_read' --output-on-failure
ctest --test-dir build/backend -R 'world_generation_then_gather|world_generation_then_chop|content_loader' --output-on-failure
```

结果：上述构建和定向测试全部通过。随后执行全量测试 `ctest --test-dir build/backend --output-on-failure`，结果为 1137/1140 通过，剩余 3 个失败均为既有 P21 revision/low confidence teaching trace 失败，不属于 P62 改动范围。

重点覆盖：

- `content_loader`：worldgen JSON 可解析；strict 模式会拒绝不存在的 object 引用。
- `world_generation_*`：内容注册后资源节点、地面物品、spawn safety、runtime apply 全部正常。
- `world_generation_then_gather_berry_bush`：真实 core content worldgen 生成资源后可采集。
- `world_generation_then_chop_tree_with_tool`：真实 core content worldgen 生成木材资源后可采集。
- `world_generation_then_gather_loose_stone`：真实 core content worldgen 生成石类资源后可采集。
- 采集产物测试额外验证输出实体的 display key 来自对应 object 定义。
- 客户端桥接测试确认 bootstrap/projection/移动链路未破坏。
- `world_inventory_*` 全部通过；旧测试夹具已改为显式生成测试物品，不再依赖 runtime 默认硬编码地面物品。

## 5. 剩余风险

### 5.1 Drop rule 尚未接入死亡/生态流程

P62 已提供 drop rule DTO、解析和引用校验，但没有把掉落规则接入战斗死亡、野兽尸体、分解或生态系统。

后续接入时必须复用：

```text
source_object_key / output_object_keys -> ContentRegistry objects -> Runtime OnMap/InInventory
```

不能写 `wolf -> meat` 这种专属 if。

### 5.2 Content loader 仍是 core package 手动加载

当前 `ClientServerRuntimeFactory` 加载 core content 是运行时装配逻辑。后续如果要多内容包、mod、版本兼容、冲突检测和热加载，需要单独设计 content package loader 阶段。

### 5.3 Resource node 本身还不是完整 object

本阶段统一的是资源节点产物和地面物品。`berry_bush` / `young_tree` / `loose_stone_node` 当前仍是 resource node key，不等同于 object 实例。

这在当前采集玩法中可接受；如果后续需要“整棵树可搬走”“灌木可燃烧”“矿脉可被魔法改变”，需要再设计 ResourceNode 与 Object/Entity 的统一或桥接规则。

## 6. 验收判断

P62 可以进入验收。它已经修复“地图物品与知识/反应物品不是同一个体系”的核心风险，并为后续通过 JSON 增加地图资源、采集产物、地面物品和掉落预留了统一入口。
