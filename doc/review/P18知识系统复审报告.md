## 4. 新增阻塞问题

### BLOCKER-7：仍有 `TestOnly` 枚举可以通过正式 DTO 校验

严重级别：P0 阻塞  
性质：DTO 契约漏洞 / 测试专用枚举泄漏  
影响范围：知识查询、知识形成计划、知识事件草稿、知识投影项

任务卡约束：

- `doc/44_P18知识系统任务卡设计.md:309`：测试专用 enum 值默认不允许进入正式 DTO。
- `doc/44_P18知识系统任务卡设计.md:383`：`KnowledgeStatus::TestOnly` 默认禁止。
- `doc/44_P18知识系统任务卡设计.md:433`：`KnowledgeQueryMode::TestOnly` 默认禁止。
- `doc/44_P18知识系统任务卡设计.md:445`：任何 `Result::ok(dto)` 返回的 DTO 必须能通过 `validateBasic()`。

#### 4.1 `KnowledgeQuery::validateBasic()` 未拒绝 `KnowledgeQueryMode::TestOnly`

位置：

- `backend/src/knowledge/knowledge_repository.cpp:14`
- `backend/src/knowledge/knowledge_repository.cpp:17`

当前只拒绝 `Unknown`，没有拒绝 `TestOnly`。

额外探针：

```text
query_testonly_valid=1
```

影响：

- 测试专用查询模式可以进入正式 repository/projection 查询入口。
- 后续如果 AI 实现者根据“所有 DTO validateBasic 都可信”扩展查询逻辑，会留下不稳定分支。

#### 4.2 `KnowledgeQuery` 的过滤枚举未拒绝 `TestOnly`

位置：

- `backend/src/knowledge/knowledge_repository.cpp:20`
- `backend/src/knowledge/knowledge_repository.cpp:24`

当前 `KnowledgeQuery` 对 `relation_type`、`min_status`、`min_confidence_band` 只做存在性/limit 校验，没有拒绝非法枚举值。

额外探针：

```text
query_relation_testonly_valid=1
query_status_testonly_valid=1
```

影响：

- `KnowledgeRelationType::TestOnly` 可作为正式查询过滤条件。
- `KnowledgeStatus::TestOnly` 可作为正式查询过滤条件。
- `KnowledgeConfidenceBand::Unknown` 也建议拒绝，避免无意义的最小置信过滤。

#### 4.3 `KnowledgeFormationPlan::validateBasic()` 未拒绝 `projected_status = TestOnly`

位置：

- `backend/src/knowledge/knowledge_claim.cpp:370`
- `backend/src/knowledge/knowledge_claim.cpp:392`

当前只拒绝 `Unknown`，没有拒绝 `TestOnly`、`Deprecated`、`Disproven` 等 P18 不应主动创建的状态。

额外探针：

```text
plan_status_testonly_valid=1
```

影响：

- 手工构造或未来扩展路径可以产生一个“计划有效但结果状态为 TestOnly”的 DTO。
- 这会削弱 `KnowledgeFormationResult.plan.validateBasic()` 的可信度。

#### 4.4 `KnowledgeProjectionItem::validateBasic()` 未拒绝 `status = TestOnly`

位置：

- `backend/src/knowledge/knowledge_projection.cpp:14`
- `backend/src/knowledge/knowledge_projection.cpp:24`

当前只拒绝 `Unknown`，没有拒绝 `TestOnly`。

额外探针：

```text
projection_item_status_testonly_valid=1
```

影响：

- Projection 是后续 Agent/AI/前端读取知识的主观视图入口，不能允许测试状态进入正式投影。
- 虽然 repository `put()` 会拒绝非法 `KnowledgeClaim`，但 `KnowledgeProjectionBuilder` 接收的是 `std::vector<KnowledgeClaim>`，并不强制只能来自 repository，因此 DTO 自校验必须严格。

#### 4.5 `KnowledgeEventDraft::validateBasic()` 未拒绝 TestOnly relation/status

位置：

- `backend/src/knowledge/knowledge_claim.cpp:408`
- `backend/src/knowledge/knowledge_claim.cpp:418`
- `backend/src/knowledge/knowledge_claim.cpp:421`

当前只拒绝 `Unknown`，没有拒绝：

- `KnowledgeRelationType::TestOnly`
- `KnowledgeStatus::TestOnly`

额外探针：

```text
event_testonly_valid=1
```

影响：

- 知识事件草稿可能被后续事件系统、日志系统或调试系统消费。
- 如果事件草稿允许测试枚举进入，会污染事件流，后续排查困难。

## 5. 建议修复方案

### 5.1 统一补齐 DTO 枚举校验

建议在以下位置补充 `TestOnly` 和阶段禁止状态校验：

1. `KnowledgeQuery::validateBasic()`：
   - `mode == Unknown/TestOnly` 均失败。
   - `relation_type == Unknown/TestOnly` 时失败。
   - `min_status == Unknown/TestOnly/Deprecated/Disproven` 时失败；如 P18 查询允许历史状态，至少必须拒绝 `Unknown/TestOnly`。
   - `min_confidence_band == Unknown` 时失败。

2. `KnowledgeFormationPlan::validateBasic()`：
   - `projected_status == Unknown/TestOnly/Deprecated/Disproven` 均失败。
   - P18 第一版建议仅允许 `Candidate/Hypothesis/Active/Teachable`。

3. `KnowledgeProjectionItem::validateBasic()`：
   - `status == Unknown/TestOnly` 均失败。
   - 是否拒绝 `Deprecated/Disproven` 可按 P19 设计再开放；P18 当前建议拒绝。

4. `KnowledgeEventDraft::validateBasic()`：
   - `relation_type == Unknown/TestOnly` 均失败。
   - `status == Unknown/TestOnly` 均失败。

### 5.2 补测试

必须补充以下测试：

```text
knowledge_query_rejects_test_only_mode
knowledge_query_rejects_test_only_relation_filter
knowledge_query_rejects_test_only_status_filter
knowledge_formation_plan_rejects_test_only_projected_status
knowledge_projection_item_rejects_test_only_status
knowledge_event_draft_rejects_test_only_relation_and_status
```

建议同时补充：

```text
knowledge_query_rejects_unknown_optional_filters
knowledge_projection_item_rejects_p18_forbidden_statuses
knowledge_event_draft_rejects_p18_forbidden_statuses
```

## 6. 非阻塞观察

### 6.1 `KnowledgeOwnerKind::Civilization` 目前没有专属 ID 字段

位置：

- `backend/include/pathfinder/knowledge/knowledge_claim.h:16`
- `backend/src/knowledge/knowledge_claim.cpp:17`

当前 `KnowledgeOwner` 有 `entity_id`、`tribe_id`、`region_id`、`external_key`，但没有 `civilization_id`。`KnowledgeOwnerKind::Civilization` 在 `validateBasic()` 中也没有专属必填校验。

本项不阻塞 P18，因为文明级知识是后续阶段预留；但建议在 P20/P21 前明确：

- 增加 `CivilizationId`；或
- 规定文明级 owner 使用 `external_key = "civilization:<id>"`；或
- P18 先显式禁止 `Civilization` 进入正式 DTO。

### 6.2 `allow_test_only` 选项当前未形成明确语义

位置：

- `backend/include/pathfinder/knowledge/knowledge_claim.h:150`

`KnowledgeFormationOptions` 有 `allow_test_only = false`，但当前修复方向是 DTO 层默认拒绝 TestOnly。建议后续要么删除该选项，要么明确只有测试辅助工厂能使用，不能绕过正式 `validateBasic()`。

## 7. 最终判断

P18 的核心链路已经明显好转：从 P17 `MemorySummary` 形成知识、证据追溯、hidden truth guard、owner 归属、`Skipped` 结果有效性都已修好。

但 `TestOnly` 漏口属于基础 DTO 契约问题。P18 是后续 P19 知识更新、P20 知识传播、P21 文明知识、Agent 主观知识查询的底座，如果这类枚举边界不统一收紧，后续 AI 实现代码时会误以为 `validateBasic()` 已经代表“正式可用”。

**结论：本次复审暂不通过。建议先修复 BLOCKER-7，并补齐对应测试；修复后再做一次短复审即可。**
