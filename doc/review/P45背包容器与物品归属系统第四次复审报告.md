# P45 背包、容器、掉落物与物品归属系统第四次复审报告

复审对象：P45 第三轮复审后修复版本

- 上次复审：`doc/review/P45背包容器与物品归属系统再次复审报告.md`
- 主设计：`doc/75_P45背包容器与物品归属系统设计.md`
- 核心实现：`backend/src/world_inventory/world_inventory_runtime.cpp`
- 测试：`backend/tests/unit/world_inventory/world_inventory_test.cpp`

## 1. 复审结论

结论：仍不通过。

上次复审指出的三个 P0 问题，本轮都有对应修复和测试：

1. 地面 stack 部分拾取不再直接移除整个地面实体。
2. full drop 合并到已有地面 stack 时，会清理源实体和 `item_locations_`。
3. drop merge ground 会把被更新的地面实体加入 `changed_entity_ids`。

但是继续按“物品归属权威”和“地图实体不能同时属于背包”复核后，发现当前 partial pickup 仍有一个新的 P0 一致性问题：

- 部分拾取地面 stack 时，地面 entity 继续 OnMap，但背包 entry 使用同一个 `entity_id`。

这意味着同一个 entity id 同时代表地面剩余 stack，又代表背包中拿走的那部分物品。虽然数量没有丢了，但归属权威仍然冲突。

## 4. 新发现阻塞问题

### P0-1：partial pickup 后同一个 entity_id 同时属于地图和背包

位置：`backend/src/world_inventory/world_inventory_runtime.cpp` 的 `applyPickupFromMap`

当前流程：

```cpp
if (partial_pickup) {
    // 地面 entity 仍然 OnMap，只减少 quantity
    world_port_.setEntityStackData(request.entity_id, new_ground_quantity, stack_key, entity->stackable);
}
...
if (!merged) {
    InventoryItemEntry entry;
    entry.entity_id = request.entity_id;
    entry.quantity = request.quantity;
    inv->entries.push_back(entry);
}
```

问题：

如果地面 `berry_ground_001` 有 5 个红果，玩家拾取 2 个：

- 地面 `berry_ground_001` 仍然 OnMap，quantity 变成 3。
- 背包新增 entry，但 `entry.entity_id` 仍是 `berry_ground_001`，quantity 是 2。

这样同一个 entity id 同时表达两个不同归属：

```text
berry_ground_001 在地图上：3 个红果
berry_ground_001 在背包里：2 个红果
```

这违反 P45 的基础规则：

- OnMap：有 coord，不在 inventory。
- InInventory：在 inventory，coord 清空。
- 一个实体不能同时存在于地图和背包。

风险：

- 后续 drop 背包中的 `berry_ground_001` 时，会误操作地面剩余 stack。
- 前端投影可能显示同一个 id 同时在地图和背包。
- NPC 计划/SourceInventoryProof 会把地面物品误认为自己背包物品。
- 存档恢复时无法判断 `berry_ground_001` 到底归谁。

当前测试为什么没抓住：

`world_inventory_pickup_partial_ground_stack_keeps_remainder` 只验证了：

- 背包有 `red_berry` quantity=2。
- 地面 entity quantity=3 且 OnMap。

它没有验证：

- 背包 entry 的 `entity_id` 不能等于仍在地图上的 entity id。
- `findItemLocation("berry_ground_001")` 不能显示 InInventory。
- 背包 entry 应该有独立 stack carrier id。

修复建议：

部分拾取必须做 stack split：

1. 原地面 entity 保留 OnMap，quantity 减少。
2. 背包里新增一个独立 entry，不能复用原地面 entity id。
3. 新 entry 的 entity id 可以是：

```text
<source_entity_id>_pickup_<state_version>
```

或使用统一 id 生成器。

4. 对这个新 inventory stack carrier 建立 `item_locations_[new_entry_entity_id] = InInventory`。
5. `changed_entity_ids` 同时包含地面 entity 和新 inventory carrier。
6. 如果未来不希望 inventory stack 都有 world entity instance，也必须明确：inventory entry 的 `entity_id` 是 inventory-only stack id，不能指向 OnMap entity。

必须新增测试：

```text
world_inventory_pickup_partial_ground_stack_has_distinct_inventory_entity
```

测试断言：

- 地面 `berry_ground_001` 仍 OnMap quantity=3。
- 背包 entry quantity=2。
- 背包 entry.entity_id != `berry_ground_001`。
- `findItemLocation(entry.entity_id)` 是 InInventory。
- `findItemLocation("berry_ground_001")` 不应是 InInventory。

## 5. 非阻塞建议

### P2-1：capacity_real_slots 仍建议改成强造数据

当前测试已通过，但更稳妥的做法是显式 spawn 两个可拾取物，避免依赖世界生成随机物。

### P2-2：projection version 未来仍建议统一

当前 `max(world_runtime.stateVersion(), inventory_runtime.stateVersion())` 对 P45 pickup/drop 暂时可用，但后续 inventory-only 命令增多后，建议用统一 projection version 生成器，避免版本语义不够严格。

## 6. 验收建议

当前仍不建议验收 P45。

这轮修复已经很接近验收，但 partial pickup 的 entity id 归属冲突是 P45 底层规则问题，必须修。否则后续世界网格、采集、制作、NPC 自主拾取都会在“同一实体到底在地图还是背包”上出错。

建议只修这一项后再次复审。如果该问题关闭，并补上对应强测试，P45 基本可以进入可验收状态。
