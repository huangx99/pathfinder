## 问题闭环检查

### 1. 安全测试覆盖

状态：已通过。

当前已补充以下测试：

- `p23_security_tribe_id_empty_rejected`
- `p23_security_member_id_empty_rejected`
- `p23_security_ratio_out_of_range_rejected`
- `p23_security_active_population_exceeds_total_rejected`
- `p23_security_duplicate_member_id_rejected`
- `p23_security_duplicate_knowledge_id_rejected`
- `p23_security_test_only_enum_rejected`
- `p23_security_trust_changed_missing_trust_rejected`
- `p23_security_morale_changed_missing_morale_rejected`
- `p23_security_duplicate_member_events_rejected`
- `p23_security_duplicate_knowledge_events_rejected`
- `p23_security_draft_validation_rejected`

这些测试覆盖了 P23 设计文档第 12 节要求的主要拒绝边界。

### 2. 成员离开语义

状态：已通过。

当前实现中，`MemberRemoved` 不再只是把成员标记为 inactive，而是从 `members` 当前列表中移除：

```text
加入 2 人 -> 移除 1 人 -> population_total=1，active_population=1，members.size()=1。
```

这与“族群当前人数”的语义一致，也能避免 P24/P25/P26 后续系统把已离开成员计入当前族群规模。

### 3. 成员事件必填字段

状态：已通过。

当前 `TribeMemberEvent::validateBasic()` 已明确拒绝：

```text
TrustChanged 缺少 trust。
MoraleChanged 缺少 morale。
```

这能避免错误事件被 resolver 静默吞掉。

### 4. StateChangeDraft 字段组合校验

状态：已通过。

当前 `TribeStateChangeDraft::validateBasic()` 已按变更类型检查关键字段：

```text
MemberAdded 需要 member_id + new_role。
MemberRemoved 需要 member_id + old_role。
MemberRoleChanged 需要 member_id + old_role + new_role。
TrustChanged / MoraleChanged / SplitRiskChanged / CohesionChanged 需要 0..1 的 old_value/new_value。
KnowledgeLinked 需要非负 new_value。
```

这能保证 trace/change draft 不只是“格式合法”，而是“语义基本完整”。

### 5. allow_test_only 字段

状态：已通过。

`TribeStateOptions` 中未使用的 `allow_test_only` 已移除，避免后续研发误以为正式 DTO 可以通过 options 放行 `TestOnly`。

### 6. 输入事件重复策略

状态：已通过。

当前 `TribeStateInput::validateBasic()` 已拒绝同一输入中的重复成员事件和重复知识事件。

这符合上次审核建议中的“方案 B”：

```text
输入重复直接拒绝，避免上游管线重复发事件时被悄悄吞掉。
```

## 复审补充观察

当前没有发现阻塞 P23 通过的问题。

有一个后续注意点：

```text
由于 P23 已选择“同一个 TribeStateInput 内重复 member_id 直接拒绝”，后续 P24/P25 如果需要同一 tick 内对同一成员连续执行多个族群事件，需要先在上游聚合为一个规范事件，或明确拆成多个 input tick。
```

这不是当前缺陷，只是后续管线设计需要遵守的约束。
