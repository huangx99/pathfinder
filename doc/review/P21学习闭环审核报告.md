## 1. 审核结论

**结论：不通过，不能进入 P22。**

P21 当前代码可以编译，现有测试也全部通过，但核心玩法语义没有达成任务卡要求。

最关键问题是：

```text
P21 标记为 Completed，但红果完整学习故事实际没有形成 KnowledgeClaim，也没有传播给接收者。
P20 should_route_to_revision 已出现时，P21 进入的 P19 recipient revision 输入不完整，实际修正失败。
现有 P21 测试覆盖不足，测试通过不能证明学习闭环可玩。
```

因此当前 P21 只能算“DTO + 编排骨架初版”，还不是合格的“后端最小可玩学习闭环”。

## 4. 阻塞问题

### BLOCKER-1：完整红果学习故事没有形成知识，也没有传播，但结果返回 Completed

位置：

```text
backend/src/learning/learning_loop.cpp:1046
backend/src/learning/learning_loop.cpp:1091
backend/src/learning/learning_loop.cpp:1109
backend/src/learning/learning_loop.cpp:1202
backend/src/learning/learning_loop.cpp:1396
backend/tests/integration/p21/learning_loop_discovery_story_test.cpp:49
backend/tests/integration/p21/learning_loop_discovery_story_test.cpp:102
```

我用 P21 任务卡要求的最小故事做了临时探针：

```text
A 吃 red_berry。
反馈 restore_hunger。
开启 cognition/memory/compression/recall/formation/propagation。
recipient 为 B。
```

实际输出：

```text
decision=completed ok=1 actor_claims=0 recip_claims=0
feedback_captured:executed:none
cognition_updated:executed:none
memory_written:executed:none
memory_compressed:executed:none
memory_recalled:executed:none
knowledge_formed:skipped_no_input:none
knowledge_revised:skipped_no_input:none
knowledge_propagated:skipped_no_input:none
recipient_influenced:skipped_no_input:none
report_built:executed:none
```

这说明：

```text
没有 KnowledgeClaim。
没有 recipient Teaching claim。
没有接收者投影变化。
但最终 decision 是 Completed。
```

这直接违反 P21 的完成定义：

```text
尝试 -> 反馈 -> 认知 -> 记忆 -> 压缩/召回 -> 知识形成 -> 知识传播 -> 接收者受影响
```

现有测试没有发现问题，因为 discovery 测试没有强制断言 `KnowledgeFormed`、actor claim、`KnowledgePropagated`、recipient Teaching claim 必须存在。

必须修复：

```text
1. P21 不能在 KnowledgeFormed / KnowledgePropagated skipped_no_input 时仍对 FullLearningLoop 返回 Completed。
2. 红果完整故事必须实际形成 actor KnowledgeClaim。
3. actor KnowledgeClaim 必须通过 P20 传播给 B。
4. B 必须得到 Teaching 来源的主观 KnowledgeClaim。
5. discovery integration test 必须强断言这些结果，而不是允许“may be skipped”。
```

### BLOCKER-2：P20 should_route_to_revision 被检测到后，P21 实际没有成功进入有效 P19 修正

位置：

```text
backend/src/learning/learning_loop.cpp:1257
backend/src/learning/learning_loop.cpp:1294
backend/src/learning/learning_loop.cpp:1298
backend/src/learning/learning_loop.cpp:1303
backend/tests/integration/p21/learning_loop_revision_story_test.cpp:80
backend/tests/integration/p21/learning_loop_revision_story_test.cpp:101
backend/tests/integration/p21/learning_loop_revision_story_test.cpp:113
```

我构造了一个直接触发 P20 路由的探针：

```text
A 有 red_berry + eat -> poison。
B 已有 red_berry + eat -> restore_hunger。
A 向 B 传播 poison。
P20 assessment.should_route_to_revision=true。
```

实际输出：

```text
decision=partial_completed claims=1 prop=weakened_recipient_claim
route=1 rec_dec=weakened_recipient_claim
knowledge_propagated:executed:none
knowledge_revised:failed:revision_failed
recip know_b_restore restore_hunger status=active
```

问题：

```text
P20 已经给出 route_to_revision=true。
P21 构造的 recipient KnowledgeRevisionInput 只有 owner/existing_claims/revision_tick/reason_keys。
formation_input.summary、target_relation、action_key、effect_key、代表证据等核心输入缺失。
P19 修正失败。
最终没有形成有效 recipient revision。
```

这违反 P21 任务卡关键要求：

```text
如果 P20 result.decision == WeakenedRecipientClaim 且 assessment.should_route_to_revision=true，P21 必须进入知识修正编排，而不是把传播当作完成。
```

必须修复：

```text
1. P21 需要从 P20 plan/source claim/incoming claim 构造完整 recipient KnowledgeRevisionInput。
2. 至少要填充 formation_input.owner、summary、target_relation、action_key、effect_key、candidate_conditions、representative_records 或等价证据输入。
3. 如果第一版无法实际调用 P19 成功，也必须返回明确 RoutedToRevision 业务结果，而不是 failed trace 后 PartialCompleted。
4. revision integration test 必须断言：P20 should_route_to_revision=true、P21 有 KnowledgeRevised/Routed 或 Executed、不能出现 KnowledgeRevised/Failed。
```


具体问题：


当前测试通过只能说明：

```text
DTO 能 validate。

部分预置 claim 可以走传播。
```

不能证明：

```text
后端最小可玩学习闭环成立。
P20 冲突路由被正确消费。
安全边界覆盖完整。
```

必须修复测试：

```text
1. discovery story 强断言 KnowledgeFormed executed。
2. discovery story 强断言 actor_claim_snapshot_after 包含 red_berry/eat/restore_hunger。
3. discovery story 强断言 KnowledgePropagated executed。
4. discovery story 强断言 recipient_claim_snapshot_after 包含 B owner + Teaching evidence。
5. revision story 不得关闭 knowledge_formation。
6. revision story 强断言 P20 should_route_to_revision=true。
7. revision story 强断言 P21 KnowledgeRevised trace decision 是 Routed 或 Executed，不是 Skipped/Failed。
8. owner mismatch security test 必须 assert Rejected/Failed，并说明由哪个边界拒绝。
```

## 5. 重要问题

### MAJOR-1：Trace id 写入了错误步骤

位置：

```text
backend/src/learning/learning_loop.cpp:1107
backend/src/learning/learning_loop.cpp:1176
```

问题：

```text
KnowledgeFormed 成功后，在 push KnowledgeFormed trace 之前调用 result.traces.back().knowledge_ids.push_back。
此时 traces.back() 是上一阶段 trace，通常是 MemoryRecalled。
KnowledgeRevised 应用 updated_claims 时，也在 push KnowledgeRevised trace 之前写 result.traces.back().updated_knowledge_ids。
```

影响：

```text
LearningEventChain 可能有 id，但 StepTrace 追溯链错位。
后续 H5/debug 报告会显示“记忆召回产生知识 id”或“上一阶段更新知识 id”。
```

建议：

```text
先构造本阶段 trace，把 ids 填进去，再 push_back。
不要用 result.traces.back() 写当前阶段产物。
```

### MAJOR-2：输入快照 owner mismatch 没有在 P21 validateBasic 中明确拒绝

位置：

```text
backend/src/learning/learning_loop.cpp:481
backend/src/learning/learning_loop.cpp:485
backend/tests/integration/p21/learning_loop_security_test.cpp:118
```

问题：

```text
LearningLoopInput::validateBasic 只检查 KnowledgeClaim 自身合法。
没有检查 actor_existing_claims 的 owner 必须等于 input.actor.knowledge_owner。
没有检查 recipient_existing_claims 的 owner 必须等于 input.recipient.knowledge_owner。
```

影响：

```text
错误 owner 的 claim 可能进入 learning 编排。
如果后续没有触发 propagation attempt，就不会被 P20 target owner mismatch 捕获。
当前 security 测试也没有断言拒绝结果，因此没有保护这个边界。
```

建议：

```text
P21 validateBasic 直接拒绝 owner mismatch。
actor_existing_memories / summaries 也应检查 owner 与 actor.memory_owner 一致。
```

### MAJOR-3：LearningLoopPlanner 的计划与真实执行不一致

位置：

```text
backend/src/learning/learning_loop.cpp:701
backend/src/learning/learning_loop.cpp:723
backend/src/learning/learning_loop.cpp:730
backend/src/learning/learning_loop.cpp:737
```

问题：

```text
Planner 根据 input.actor_existing_memories / summaries 预判步骤。
但真实执行中 MemoryWritten 会产生新的 memory，MemoryCompressed 才可能有输入。
Planner 可能把 MemoryCompressed 计划为 SkippedNoInput，但 Service 实际执行。
Planner 也无条件把 KnowledgeFormed/KnowledgeRevised 标为 Executed，只要选项开启。
```

影响：

```text
Planner 不能作为可靠执行预期。
后续 H5/debug 如果用 Planner 展示即将发生的步骤，会误导玩家和研发。
```

建议：

```text
Planner 只规划“可能执行/需要执行”的阶段，不提前标 Executed。
或者明确区分 PlannedStepDecision 与 RuntimeStepDecision。
```

## 6. 非阻塞观察

### 6.1 Rejected / Failed 的 ok 语义需要统一

`LearningLoopResult::ok()` 当前不包含 Rejected / Failed。

这可以接受，但需要在文档和测试中明确：

```text
Result<LearningLoopResult>::ok 表示 C++ 调用成功。
LearningLoopResult::ok 表示业务成功/部分成功/路由成功。
```

安全拒绝测试只检查 `result_r.is_ok()` 是合理的，但最好也检查 `decision == Rejected`，避免 Failed 和 Rejected 混用。

### 6.2 validateBasic 对 Unknown failure_reason 没有真正拒绝

位置：

```text
backend/src/learning/learning_loop.cpp:527
backend/src/learning/learning_loop.cpp:553
```

代码进入 if 后没有 return，实际允许 Unknown/TestOnly failure_reason。

如果这是故意允许，需要删掉空 if 或写清楚；如果不是故意，应该拒绝 Unknown/TestOnly，失败 trace 必须有明确 failure_reason。
