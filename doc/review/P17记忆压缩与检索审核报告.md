# P17 记忆压缩与检索审核报告

审核日期：2026-05-17  
审核范围：P17 记忆摘要、压缩计划、压缩服务、召回服务、P17 单元/集成测试、阶段边界扫描。  
审核依据：`doc/43_P17记忆压缩与检索任务卡设计.md`、`context_packs/memory_p17.md`、P15/P16 已有认知与记忆设计。

## 1. 审核结论

**结论：暂不通过，需修复后复审。**

P17 当前实现已经建立了正确的模块骨架，基础编译、P17 专项测试和全量测试均通过；但存在 3 个会影响阶段契约的关键问题：

1. 压缩记录不足时返回 `ok + Skipped`，但携带的 `MemoryCompressionPlan` 不能通过自身 `validateBasic()`。
2. `MemoryRecallSort::DeterministicKeyAsc` 实际按 ID 倒序返回，违反确定性升序排序契约。
3. 压缩证据去重只按 `source_event_id`，会丢失只有 `cognition_evidence_id` 的多条证据引用，削弱 P17 到 P18 的证据候选追溯链。

这些问题不是编译问题，而是“公共 DTO 可信度、召回确定性、证据可追溯性”的设计契约问题。建议修复后再标记 P17 完成。

## 4. 阻塞问题

### BLOCKER-1：Skipped 压缩结果携带不可校验的 plan

位置：

- `backend/src/memory/memory_compression.cpp:115`
- `backend/src/memory/memory_compression.cpp:168`
- `backend/src/memory/memory_summary.cpp:292`
- `backend/src/memory/memory_summary.cpp:324`

问题：

当符合条件的记录数少于 `min_records_to_summarize` 时，`MemoryCompressionPlanner::planForOwnerSubject()` 返回 `ok(plan)`，但该 plan 的 `source_memory_ids` 与 `representative_memory_ids` 为空。

随后 `MemoryCompressionService::compress()` 把这个 plan 放入 `MemoryCompressionResult`，并返回：

```text
Result::ok(MemoryCompressionResult{ decision = Skipped, plan = invalid_plan })
```

但 `MemoryCompressionResult::validateBasic()` 会校验 plan，而 `MemoryCompressionPlan::validateBasic()` 要求：

- `source_memory_ids` 非空
- `representative_memory_ids` 非空

因此公共 API 出现矛盾：

```text
compress() 返回 ok
result.ok() 为 true
但 result.validateBasic() 失败
```

影响：

- 调用方无法信任 `Result::ok` 返回的 DTO。
- P17 的“计划可审计”契约被破坏。
- 后续 P18/P20 如果复用压缩结果，会遇到“业务成功但 DTO 非法”的隐性错误。

修复建议：

优先方案：

- `MemoryCompressionPlan` 只表示“可执行压缩计划”。
- 记录不足时不要返回 plan，`MemoryCompressionResult` 直接返回 `decision = Skipped`、`plan = std::nullopt`、`reason_keys = {"skipped", "insufficient_records", "count:n"}`。

可选方案：

- 允许 `MemoryCompressionPlan` 增加 `plan_status` 或 `is_executable`，并让 `validateBasic()` 根据状态区分 skipped plan 与 executable plan。
- 但第一版不建议增加复杂度，直接不携带无效 plan 更清晰。

必须补测试：

- `compress_insufficient_records_returns_valid_skipped_result`
- 断言：`result.is_ok()`、`decision == Skipped`、`result.value().validateBasic().is_ok()`。

### BLOCKER-2：DeterministicKeyAsc 实际倒序

位置：

- `backend/src/memory/memory_recall.cpp:392`
- `backend/src/memory/memory_recall.cpp:404`

问题：

`MemoryRecallSort::DeterministicKeyAsc` 语义是确定性 key 升序，但当前 comparator 对所有主排序都使用：

```cpp
if (a_less != b_less) return b_less;
```

这对 `StrengthDesc` 等降序排序可以工作，但对 `DeterministicKeyAsc` 会反向。

额外探针确认：

```text
输入 ID：mem_b, mem_a, mem_c
DeterministicKeyAsc 输出：mem_c, mem_b, mem_a
```

影响：

- 违反 P17 “确定性排序 + ID tie-break”的契约。
- 后续 replay、训练样本、知识证据候选如果依赖召回顺序，会得到反向稳定序。
- 当前测试未覆盖这个排序分支，因此全量测试无法发现。

修复建议：

把排序 comparator 改成更直接、不可误读的结构：

```cpp
switch (sort_key) {
  case StrengthDesc:
    if (a_score != b_score) return a_score > b_score;
    break;
  case LastTouchedDesc:
    if (a_tick != b_tick) return a_tick > b_tick;
    break;
  case DeterministicKeyAsc:
    if (a_id != b_id) return a_id < b_id;
    break;
}
// common tie-breakers...
```

不要用 `a_less/b_less` 同时表达升序与降序，容易再次出错。

必须补测试：

- `recall_sort_deterministic_key_asc_orders_ids_ascending`
- `recall_sort_last_touched_desc_orders_ticks_descending`
- `recall_sort_importance_desc_orders_importance_descending`
- `recall_sort_evidence_count_desc_orders_count_descending`
- 每个排序都验证最后 ID 升序 tie-break。

### BLOCKER-3：证据去重只看 source_event_id，会丢 cognition evidence 证据链

位置：

- `backend/src/memory/memory_compression.cpp:272`
- `backend/src/memory/memory_compression.cpp:276`
- `backend/src/memory/memory_compression.cpp:282`

问题：

压缩摘要收集所有源记录的 `evidence_refs` 后，只用：

```cpp
ref.source_event_id.value()
```

作为去重 key。

但 P16 `MemoryEvidenceRef` 支持 cognition evidence：

- `cognition_evidence_id`
- `cognition_claim_id`
- `source_event_id`

如果多条证据只有不同的 `cognition_evidence_id`，但 `source_event_id` 为空，当前实现会把它们全部当作同一个空 key，只保留第一条。

额外探针确认：

```text
3 条不同 cognition_evidence_id 输入
压缩后 evidence_refs.size() == 1
```

影响：

- P17 给 P18 留下的“证据候选”会丢失原始认知证据。
- 摘要虽然仍有 `source_memory_ids`，但 `evidence_refs` 的直接证据链不完整。
- 这会降低后续知识系统做置信度、冲突、可解释审计的质量。

修复建议：

建立统一证据 key：

```text
source_kind + cognition_evidence_id + cognition_claim_id + source_event_id + observed_tick
```

至少按优先级选择：

```text
source_event_id 非空 → event:<id>
cognition_evidence_id 非空 → cognition_evidence:<id>
cognition_claim_id 非空 → cognition_claim:<id>
否则 → source_kind + observed_tick + reason_keys hash
```

同时删除或改造 fallback：

```cpp
fallback.source_event_id = EventId("compression_fallback")
```

P17 不应凭空制造假的追溯源。如果没有任何有效 evidence ref，应返回失败或 skipped，并给出 `missing_evidence_refs` 原因。

必须补测试：

- `compression_preserves_distinct_cognition_evidence_refs`
- `compression_rejects_or_skips_when_all_evidence_refs_invalid`

## 5. 重要非阻塞问题

### ISSUE-1：MemoryCompressionPlan 没校验 representative 属于 source

位置：

- `backend/src/memory/memory_summary.cpp:286`

`MemorySummary::validateBasic()` 已经校验 representative 必须来自 source，但 `MemoryCompressionPlan::validateBasic()` 没有同样校验。

影响：

- 错误 plan 可能在进入 summary 前才暴露。
- “计划可审计”不够自洽。

建议：

在 `MemoryCompressionPlan::validateBasic()` 中补充：

```text
representative_memory_ids 每一项都必须存在于 source_memory_ids
```

并补测试。

### ISSUE-2：MemoryCompressionOptions 缺少代表数与源数关系校验

位置：

- `backend/src/memory/memory_summary.cpp:263`

当前只校验 `max_source_records > 0` 和 `max_representative_records > 0`，没有校验：

```text
max_representative_records <= max_source_records
```

建议补充，避免配置自相矛盾。

### ISSUE-3：MemorySummary::validateBasic() 未校验 summary_id 格式

位置：

- `backend/src/memory/memory_summary.cpp:207`
- `backend/src/memory/memory_summary.cpp:459`

`MemorySummaryIdFactory` 会校验生成 ID 是否满足 `foundation::isValidIdString()`，但 `MemorySummary::validateBasic()` 只检查非空。

建议：

`MemorySummary::validateBasic()` 也校验 `summary_id.value()` 是否满足 `foundation::isValidIdString()`，防止绕过 factory 手写非法 ID 后被 `MemorySummaryStore::put()` 接受。

### ISSUE-4：测试覆盖偏乐观，缺少负向与排序分支覆盖

当前 517/517 通过，但 P17 关键分支没有覆盖到：

- `Skipped` 结果自身 `validateBasic()`。
- `DeterministicKeyAsc`。
- 其他排序模式的精确顺序。
- cognition-only evidence 去重。
- `MemoryCompressionPlan` representative/source 不一致。
- `MemoryCompressionOptions` 参数关系不一致。

建议 P17 修复时同步补齐测试，否则后续容易回归。

## 6. 阶段边界结论

P17 没有明显越过以下边界：

- 没有实现知识系统。
- 没有实现传播系统。
- 没有接 H5/API。
- 没有接存档。
- 没有接强化学习 reward/done。
- 没有让 AgentRuntime/Policy 写 summary store。

但必须修复压缩结果、召回排序、证据链问题后，才能认为 P17 作为 P18 知识系统前置层是可靠的。

## 7. 建议修复清单

P0 必修：

1. 修复 `Skipped` 压缩结果不可校验问题。
2. 修复 `DeterministicKeyAsc` 倒序问题。
3. 修复 evidence dedupe key，保留不同 cognition evidence。
4. 移除或改为失败处理 `compression_fallback` 伪证据。
5. 补齐上述 3 个阻塞问题的测试。

P1 建议：

1. `MemoryCompressionPlan::validateBasic()` 校验 representative/source 关系。
2. `MemoryCompressionOptions::validateBasic()` 校验代表数不超过源记录数。
3. `MemorySummary::validateBasic()` 校验 summary id 格式。
4. 给所有 recall sort 分支补精确顺序测试。
