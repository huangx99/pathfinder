## 1. 审核结论

**结论：暂不通过，需修复后再复审。**

本次审核确认：上次指出的 `TestOnly` 漏口已经基本补齐，`KnowledgeQuery`、`KnowledgeFormationPlan`、`KnowledgeProjectionItem`、`KnowledgeEventDraft` 的直接 `validateBasic()` 探针均已能拒绝非法枚举。

但本轮发现一个新的 P0 阻塞项：`KnowledgeProjectionBuilder::buildProjection()` 在接收非法 `KnowledgeClaim` 输入时，会返回 `Result::ok(KnowledgeProjection)`，但返回的 `KnowledgeProjection` 自身无法通过 `validateBasic()`。这违反 P18 任务卡“任何 `Result::ok(dto)` 返回的 DTO 必须能通过 `validateBasic()`”的核心契约。

因此，本次不是否定 TestOnly 修复，而是指出：**底层 DTO 校验已经收紧，但投影构建器没有在输出前做 claim 输入校验或 projection 输出校验，导致 `Result::ok` 仍可能携带 invalid DTO。**

## 4. 新增阻塞问题

### BLOCKER-8：ProjectionBuilder 可返回 `ok + invalid KnowledgeProjection`

严重级别：P0 阻塞  
性质：Result/DTO 契约破坏  
影响范围：知识投影、Agent 读取知识视图、后续 UI/AI/RL 输入稳定性

位置：

- `backend/src/knowledge/knowledge_projection.cpp:64`
- `backend/src/knowledge/knowledge_projection.cpp:78`
- `backend/src/knowledge/knowledge_projection.cpp:124`
- `backend/src/knowledge/knowledge_projection.cpp:164`
- `backend/src/knowledge/knowledge_projection.cpp:165`

问题：

`KnowledgeProjectionBuilder::buildProjection()` 只校验了 `query.validateBasic()`，没有校验传入的 `claims`，也没有在返回前校验 `projection.validateBasic()`。

当前逻辑会把 claim 字段直接复制到 `KnowledgeProjectionItem`：

```text
claim.status -> item.status
claim.subject -> item.subject
claim.predicate -> item.predicate
claim.confidence -> item.confidence
```

如果传入的 claim 本身非法，例如 `status = Deprecated`，builder 会照样生成 item，并返回 `Result::ok(projection)`。但 `KnowledgeProjectionItem::validateBasic()` 已经拒绝 `Deprecated`，因此最终返回的 projection 自身是 invalid。

额外探针输出：

```text
projection_builder_with_invalid_claim_ok=1 projection_valid=0 items=1
```

这说明：

- `buildProjection()` 返回了 `Result::ok`。
- 但 `result.value().validateBasic()` 失败。
- 这直接违反 P18 任务卡 `doc/44_P18知识系统任务卡设计.md:445` 的约束。

影响：

1. 后续 Agent/AI/前端如果信任 `Result::ok(KnowledgeProjection)`，会读到非法知识视图。
2. `ProjectionBuilder` 的入参是 `std::vector<KnowledgeClaim>`，不是只能由 `KnowledgeRepository` 生成；所以不能依赖 repository `put()` 已经过滤非法 claim。
3. P18 作为 P19/P20/P21 的底座，投影层是知识对外读取接口，必须保证 `ok` 结果自洽。

建议修复：

优先方案：

```text
KnowledgeProjectionBuilder::buildProjection() 在处理每个匹配 claim 前调用 claim.validateBasic()。
如果 claim invalid，直接 Result<KnowledgeProjection>::fail(...)。
```

并在返回前加最后一道保护：

```text
auto projection_result = projection.validateBasic();
if (projection_result.is_error()) return Result<KnowledgeProjection>::fail(projection_result.errors());
return Result<KnowledgeProjection>::ok(std::move(projection));
```

这样可以同时保证：

- 输入 claim 不可信时不会生成非法投影。
- 即使未来 builder 增加新字段，也不会破坏 `Result::ok(dto)` 契约。

必须补测试：

```text
knowledge_projection_builder_rejects_invalid_claim
knowledge_projection_builder_never_returns_invalid_projection
```

建议测试构造：

```text
claim.status = Deprecated
builder.buildProjection({claim}, valid_query)
assert(result.is_error())
```

或如果产品希望“跳过非法 claim”，也必须确保：

```text
result.is_ok()
result.value().validateBasic().is_ok()
```

但 P18 第一版建议直接 fail，避免静默吞掉数据污染。

## 5. 非阻塞观察

### 5.1 Service 错误分类建议更严格

位置：

- `backend/src/knowledge/knowledge_formation.cpp:244`
- `backend/src/knowledge/knowledge_formation.cpp:247`

当前 `KnowledgeFormationService` 只要错误列表里存在 `knowledge_insufficient_confidence`，就会走 `Skipped`。目前 planner 基本是单错误早返回，所以实际风险不高。

建议后续改成：

```text
只有全部错误都是 knowledge_insufficient_confidence，才允许转 Skipped。
只要混入 validation_failed/security 错误，就必须 Result::fail。
```

本项不阻塞 P18 当前修复，但能增强后续扩展安全性。

### 5.2 `KnowledgeOwnerKind::Civilization` 仍是预留形态

当前 `KnowledgeOwnerKind::Civilization` 没有专属 ID 字段，也没有必填校验。P18 暂不需要文明级知识，因此不阻塞；建议 P20/P21 前明确文明 owner ID 表达方式。

## 6. 最终判断

本次修复把上次报告里的 TestOnly 漏口基本补齐，方向正确，测试也通过。

但 `ProjectionBuilder` 仍可能返回 `ok + invalid DTO`，这是 P18 底层契约级问题。知识投影是后续 Agent、UI、AI/RL 读取主观知识的接口，必须保证输出可信。

**结论：本次审核暂不通过。请先修复 BLOCKER-8，并补充对应测试；修复后再复审。**
