# P28.5 配置化物品与反应流水线自我审核报告

## 1. 审核结论

P28.5 已完成设计、实现和自我审核。

本阶段不是临时把“火把”“喝水”写死在代码里，而是补齐了一条可扩展流水线：

```text
物品定义 -> 运行态对象 -> 配置化反应规则 -> 反应裁决 -> 世界对象应用 -> 生成物继续可用 -> 反应知识分发给玩家/NPC
```

结论：

```text
通过。
```

它满足当前目标：

```text
1. 新物品可以通过 ItemDefinitionContent 进入 ItemCatalog。
2. 新反应可以通过 ObjectReactionRule 进入 ReactionContentPack。
3. 反应生成物可以被 ReactionChangeApplier 写回 ReactionRuntimeWorld。
4. 写回后的生成物可以再次由 ReactionInputBuilder 构造成 ReactionObjectRef，继续参与后续反应。
5. 反应产生的知识可以通过 ReactionKnowledgePlanner 分发给玩家、观察 NPC、被教学对象。
6. 单对象规则已经支持，后续“喝水”“吃果子”“点燃火把”“使用工具”不需要伪造第二个对象。
```

## 2. 本阶段解决的问题

用户提出的核心问题是：

```text
生成出来的物品能不能用？
能不能从原料、合成到成品形成一套可扩展流程？
后续程序员加一个物品、加一个反应，能不能像加喝水一样简单？
这些反应知识能不能和现有知识系统配合，让 NPC 也能学会？
```

P28.5 的回答是：

```text
能。
```

但它现在完成的是“规则层和运行态最小流水线”，不是完整内容编辑器。

当前支持：

```text
配置式物品定义。
配置式反应规则。
配置式知识模板。
运行态对象实例化。
反应输入自动构造。
反应结果应用到运行态世界。
生成物再次进入反应输入。
玩家直接经验学习草稿。
NPC 观察学习草稿。
教学传播学习草稿。
```

暂不支持：

```text
真实 JSON 文件解析。
完整背包/地图系统。
完整 NPC 行为决策 AI。
H5 页面展示全部运行态物品。
反应后知识直接写入 KnowledgeRepository。
```

这些不是遗漏，而是为了保持阶段边界：P28.5 先把 C++ DTO、校验、应用、知识分发接口打牢，后续 P35 再接 JSON 内容包。

## 5. 核心架构审核

### 5.1 物品定义层

`ItemDefinitionContent` 表示“一个物品是什么”。

它包含：

```text
definition_id
object_key
display_name
category
tag_keys
default_state_keys
default_resource_values
safe_description
```

审核结论：

```text
通过。
```

原因：

```text
1. definition_id 是稳定配置身份。
2. object_key 是规则和展示可用的安全短名。
3. display_name / safe_description 给 H5 和中文界面预留展示字段。
4. tag_keys 只表达物品长期类别，比如 tool、combustible、drinkable。
5. default_state_keys 表达运行态状态，比如 dry、burning、clean、available。
6. default_resource_values 表达燃料、水量、耐久等可变化资源。
```

重要约束：

```text
tag_keys 不允许写 dry / wet / burning 这类动态状态。
动态状态必须写入 default_state_keys 或运行态 state_keys。
```

这样避免后续出现：

```text
一个物体 tag 里说 wet，state 里又说 dry。
```

### 5.2 运行态对象层

`ReactionRuntimeObject` 表示“世界里某个具体物品实例”。

它包含：

```text
object_id
definition_id
object_key
category
quantity
state_keys
tag_keys
resource_values
location_key
```

审核结论：

```text
通过。
```

原因：

```text
1. 它不替代未来完整 WorldObject / Inventory / Map。
2. 它只负责 P28/P28.5 反应系统需要的最小运行态对象。
3. 生成物写回这里后，可以继续进入 ReactionInputBuilder。
```

这正面解决了“生成出来的物品能用吗”的问题。

### 5.3 内容包层

`ReactionContentPack` 统一承载：

```text
ItemCatalog
ObjectReactionRule 列表
ReactionKnowledgeTemplate 列表
```

审核结论：

```text
通过。
```

原因：

```text
1. 物品、反应、知识模板放在同一个内容包里，便于整体校验。
2. 规则仍复用 P28 ObjectReactionRule，没有另起炉灶。
3. 后续 JSON 只需要解析到这些 DTO，不需要重写反应系统。
```

## 6. 流水线审核

完整流程为：

```text
ItemCatalog
  -> ReactionRuntimeWorld
  -> ReactionInputBuilder
  -> ObjectReactionResolver
  -> ReactionResult
  -> ReactionChangeApplier
  -> 新 ReactionRuntimeWorld
  -> ReactionKnowledgePlanner
  -> 玩家/NPC 学习草稿
```

审核结论：

```text
通过。
```

原因：

```text
1. Resolver 仍然只负责裁决，不直接改世界。
2. Applier 单独负责把 ReactionResult 应用到运行态世界。
3. KnowledgePlanner 单独负责把反应知识变成不同来源的学习草稿。
4. UI 和 H5 不需要自己猜测规则结果。
5. NPC 学习不是写死行为，而是通过 Observation / Taught / DirectExperience 三种来源进入后续学习链路。
```

这是工程上正确的拆法：

```text
裁决、应用、学习分离。
```

## 7. 生成物可用性审核

已验证流程：

```text
火源 + 干树枝 -> 火把
```

实际运行链路：

```text
1. ItemCatalog 定义 fire_source、dry_branch、torch。
2. ReactionRuntimeWorld 实例化 obj_fire、obj_branch。
3. ReactionInputBuilder 构造 Combine 输入。
4. ObjectReactionResolver 命中 fire_dry_branch_to_torch。
5. ReactionChangeApplier 把 obj_branch TransformObject 为 def_torch。
6. 再次调用 ReactionInputBuilder，把变成火把的 obj_branch 构造成 ReactionObjectRef。
```

审核结论：

```text
通过。
```

这说明生成物不是一段文本，也不是不可交互草稿，而是回到了运行态对象列表里。

## 8. “喝水一样简单”审核

已验证单对象规则：

```text
clean_water + use -> consume clean_water
```

过去 P28 只支持 2..3 个对象，会导致“喝水”这种行为必须伪造第二个对象。

P28.5 已调整为：

```text
ReactionInputSet：1..3 个对象。
ObjectReactionRule：1..3 个对象模式。
ReactionConditionContextBuilder：单对象自动作为 target 上下文。
```

审核结论：

```text
通过。
```

这对后续内容非常关键，因为游戏里大量行为都是单对象行为：

```text
吃果子。
喝水。
使用石片。
点燃火把。
阅读石碑。
穿戴装备。
打开容器。
```

如果不支持单对象，后续配置会大量扭曲。

## 10. 安全与边界审核

已检查：

```text
1. 物品 definition_id 必须是合法 ID。
2. object_key / display_name / safe_description 禁止危险隐藏字段。
3. tag_keys 禁止 dynamic state 语义。
4. runtime tag_keys 也禁止 dynamic state 语义。
5. resource value 必须是有限数字。
6. ReactionContentValidator 会拒绝重复 rule_key。
7. ReactionContentValidator 会拒绝缺失的 product_definition_id。
8. ReactionContentValidator 会拒绝 knowledge template 引用不存在的 rule。
9. ReactionChangeApplier 遇到缺失对象或缺失定义会失败，不静默吞错。
```

审核结论：

```text
通过。
```

自我审核时额外加严了一点：

```text
ReactionRuntimeObject::tag_keys 也不能包含 dry / wet / burning 这类动态状态。
```

原因是：运行态对象如果允许把动态状态塞进 tag，后续规则会再次出现“双轨判断”。

新增测试用例：


### 15.1 覆盖流程

测试覆盖：

```text
1. 玩家反复吃红果，形成并强化“红果可缓解饥饿”知识。
2. 玩家再吃腐烂红果，形成“腐烂红果会中毒”知识。
3. 玩家把两条不同知识教给同伴。
4. 同伴自己再吃腐烂红果，强化已有中毒知识。
5. 同伴再把知识教给第三个 NPC。
6. 第三个 NPC 再教回玩家。
7. 玩家已有知识被强化，不应该重复，也不应该因为别人教回来的弱知识而降低置信度。
```

还覆盖：

```text
同一次教学里，一条知识是新建，另一条知识是已有知识强化。
```

这个场景之前没有被旧测试打到。

### 15.2 发现并修复的问题

复杂测试发现两个真实问题：

```text
问题一：新建传播知识没有设置 support_count，导致后续直接经验强化后 support_count 不符合预期。
问题二：一次教学里同时出现“新建知识”和“强化已有知识”时，KnowledgePropagationResult 只按总 decision 分类，可能把强化草稿误放进 created_claims，造成重复知识。
问题三：别人教回来的低置信知识强化已有高置信知识时，旧公式会被 max_created_confidence 错误压低，导致强知识被弱教学降级。
```

修复：

```text
1. 新建 recipient_claim 时设置 confidence.support_count = 1。
2. KnowledgePropagationService 按每条 draft 是否匹配 target_existing_claims 分类为 created_claims 或 updated_claims，而不是按总 decision 分类。
3. 强化已有知识时保证置信度单调不下降，上限为 1.0，不再用 max_created_confidence 压低已有强知识。
4. 创建事件和强化事件也改成按 created_claims / updated_claims 分别生成，不再依赖总 decision。
```
