## 6. 后续影响评估

### 6.1 新玩法扩展

后续新增钓鱼、耕种、治疗、驯服、交易、魔法、机械、电力等行为时，不应该改 runtime host factory。

应新增 `world_xxx` 模块并通过默认模块入口注册。

### 6.2 新内容扩展

如果只是新增物品、采集产物或制作反应，优先走 JSON / ContentRegistry / Reaction / Harvest，不新增 C++ 规则。

只有新增通用行为类型才需要 C++ 模块。

### 6.3 AI 扩展

NPC 职业、群体生产、建房、寒冷处理、链式目标执行，不应该继续塞进 `world_npc_work`。

建议后续拆：

```text
world_goal_execution
world_job
world_settlement
world_region_event
```

### 6.4 默认场景配置化

默认场景种子已移出 factory，但仍是代码模块。

后续应由 scenario/content package loader 替代。

## 7. 遗留风险

1. `default_world_modules.cpp` 当前仍是默认模块顺序入口，未来模块多了可能需要分层 registry。
2. `world_npc_work` 当前是代工原型，后续不能无限扩成完整 AI 大脑。
3. post-command hook 目前只有简单顺序，后续天气、区域魔法、火势、疾病等需要 tick phase。
4. 默认 bootstrap 仍有默认内容 key，后续应配置化。

## 9. 命名迁移复查

用户指出 `client_http` 文件夹名会误导，因为当前运行时宿主并不只服务 HTTP。

已修复：

```text
client_http/client_server_runtime_factory.*
-> client_runtime_host/client_runtime_host_factory.*
ClientServerRuntimeFactory
-> ClientRuntimeHostFactory
pathfinder_client_http 中移除 runtime host factory
新增 pathfinder_client_runtime_host target
```

复查结论：

- `client_http/` 现在只保留 HTTP transport / router / gateway / static file。
- `client_runtime_host/` 负责本地 Axmol 与 HTTP server 共享的 runtime 装配。
- 代码中不再引用旧 `client_server_runtime_factory` 或 `ClientServerRuntimeFactory`。
