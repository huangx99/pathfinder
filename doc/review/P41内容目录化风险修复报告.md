# P41 内容目录化风险修复报告

## 1. 修复结论

结论：已修复本轮再次审核中能立即处理的 P2 风险。

本次修复目标不是重写 H5 / Story，而是把上一轮指出的两个最容易继续扩散的问题先收敛：

- 同伴默认火把知识不再由会话层无条件写死注入。
- H5 / Story 的威胁知识、建议动作、可见执行计划开始从场景配置读取。

当前结果：

- 正式默认场景中，同伴开局不再凭空拥有火把驱兽知识。
- 如果以后需要教学 / 测试默认知识，可以通过 `DialogScenario.default_knowledge_templates` 配置，而不是改 `dialog_session.cpp`。
- Story 建议动作从 `DialogScenario.suggested_action_templates` 读取。
- H5 同伴自动处理威胁所需知识从 `DialogScenario.threat_knowledge_templates` 读取。
- H5 可见执行计划的 `effect_key` 从场景反馈 / 威胁配置读取，不再直接写死 `repel_beast` / `cut_wood`。

## 2. 修改内容

### 2.1 新增场景配置 DTO

文件：`backend/include/pathfinder/h5_dialog/dialog_types.h`

新增：

- `DialogDefaultKnowledgeTemplate`
- `DialogScenarioActionTemplate`
- `DialogScenarioThreatKnowledgeTemplate`

并在 `DialogScenario` 中新增：

- `default_knowledge_templates`
- `suggested_action_templates`
- `threat_knowledge_templates`

文件：`backend/src/h5_dialog/dialog_types.cpp`

新增对应 `validateBasic()`，确保配置字段不为空、不含禁止 key、默认知识置信度范围合法。

### 2.2 默认同伴知识改为配置驱动

文件：`backend/src/h5_dialog/dialog_session.cpp`

已删除/替换原来的无条件逻辑：

- 不再有 `defaultCompanionTorchKnowledge(...)`。
- 不再有 `playtest_helper` 默认火把知识注入。
- 不再在 `loadOrCreate()` 时强行给旧会话补同伴驱兽知识。

新的逻辑：

- 只读取 `DialogScenario.default_knowledge_templates`。
- 默认场景该列表为空，所以同伴开局没有凭空知识。
- 后续如果要教学模式，可在场景配置里声明默认知识。

### 2.3 威胁知识从场景配置读取

文件：`backend/src/h5_dialog/dialog_scenario.cpp`

默认场景新增：

- `threat.beast.counter.fire`
- `threat_object_key = beast_shadow`
- `required_effect_key = repel_beast`
- `fallback_effect_keys = ignite_fire, make_torch`

文件：`backend/src/h5_dialog/dialog_turn_service.cpp`

同伴自动处理威胁时，不再直接写：

- `required_knowledge_effect_key = "repel_beast"`

改为：

- `requiredThreatKnowledgeEffect(scenario, target_threat_key)`

### 2.4 Story 建议动作从场景配置读取

文件：`backend/src/h5_dialog/dialog_scenario.cpp`

默认场景新增建议动作模板：

- 制作火把。
- 点燃火源。
- 用火把驱赶野兽。
- 查看目标。

文件：`backend/src/story/story_system.cpp`

Story 投影不再手写：

- `storyAction("make_torch", ...)`
- `storyAction("repel_beast", ...)`

改为读取：

- `DialogScenario.suggested_action_templates`

同时，第一天生存威胁的 counter knowledge 也从 `DialogScenario.threat_knowledge_templates` 读取。

### 2.5 H5 可见执行计划从场景读取 effect_key

文件：`backend/src/h5_playable/playable_turn_service.cpp`

可见执行计划中：

- 野兽相关步骤的 effect_key 从 `threat_knowledge_templates` 读取。
- 砍木头相关步骤的 effect_key 从 `DialogFeedbackTemplate` 读取。

这让 H5 状态展示不再自己知道“野兽=repel_beast、斧头砍木头=cut_wood”。

## 4. 仍然保留的边界

本次没有把所有 H5 / Story 内容完全迁移到外部 JSON。

仍然保留在 `dialog_scenario.cpp` / `story_system.cpp` 中的内容 key，当前定义为“内置场景内容配置”，不是核心逻辑硬编码。后续如果要做到真正内容包化，需要继续做：

- `DialogScenarioCatalog` 从 JSON / 配置包加载。
- `StoryScenarioRegistry` 从内容目录加载。
- `DialogIntentParser` 的中文词表从对象配置读取。
- `DialogPresenter` / H5 展示文案从对象、效果、知识配置读取。

## 5. 最终判断

本次修复已经把上一轮审核中能立即改掉的风险收敛掉：

- 同伴不再凭空获得关键生存知识。
- 威胁知识、建议动作、可见计划 effect_key 不再分散写死在会话、Story 投影和 H5 执行展示里。
- 后续要加教学默认知识、换一个威胁、换一套建议动作，可以先改场景配置结构，而不是继续修改多个业务流程函数。

下一步建议进入“内容包 / JSON 加载”阶段，把当前内置场景配置迁移到外部配置文件。
