# 先驱者：通用枚举设计

## 1. 设计目标

本文档用于逐步确认《先驱者》规则引擎中的通用枚举。

枚举设计必须遵守 `01_基础类型与通用约定.md` 中的约定：

```text
C++ 内部使用 enum class
配置、JSON、日志、存档使用稳定英文字符串
枚举字符串使用 lower_snake_case
枚举需要中文翻译能力
未知枚举字符串必须报错
新增枚举值必须同步更新转换器、翻译、校验、协议和测试
```

## 2. 已确认枚举范围

### 2.1 对象相关枚举清单

对象相关枚举用于支撑“未知对象如何被感知、交互、认知和关联”。

第一阶段确认包含以下六个对象相关枚举：

| 枚举 | 中文名 | 作用 |
|---|---|---|
| ObjectCategory | 对象类别 | 描述对象在真实世界中的大类 |
| ObjectState | 对象状态 | 描述对象实例当前在世界中的运行时状态 |
| ObjectVisibilityState | 对象可见状态 | 描述玩家是否能在当前环境中感知到对象 |
| ObjectKnowledgeState | 对象知识状态 | 描述玩家或族群对该对象整体知识成熟度 |
| ObjectKnowledgeFlag | 对象知识标记 | 描述玩家或族群具体知道了哪些行为、用途、风险和效果 |
| ObjectRelationType | 对象关系类型 | 描述对象之间的关系 |

这六个枚举的边界不同，不能混用：

```text
ObjectCategory 是真实类型分类。
ObjectState 是对象实例当前状态。
ObjectVisibilityState 是感知可见性。
ObjectKnowledgeState 是整体知识成熟度。
ObjectKnowledgeFlag 是具体行为、用途、风险和效果知识标记。
ObjectRelationType 是对象之间的关系。
```

示例：一丛红浆果。

```text
ObjectCategory = Plant（植物）
ObjectState = Normal（正常）
ObjectVisibilityState = Visible（可见）
ObjectKnowledgeState = Unknown（未知）
```

说明：

```text
玩家能看见它，但还不知道它能做什么。
所以“可见”不等于“已掌握用途、风险和行为结果”。
```

### 2.2 第一阶段暂不加入的对象枚举

第一阶段暂不加入以下对象相关枚举：

| 枚举 | 中文名 | 暂缓原因 |
|---|---|---|
| ObjectRarity | 对象稀有度 | 暂时不是认知规则核心 |
| ObjectFaction | 对象阵营 | 阵营偏战斗或政治系统，第一阶段不做 |
| ObjectOwnership | 对象归属 | 所有权会把设计拉向建造、经济或库存系统 |
| ObjectQuality | 对象品质 | 容易变成装备或物品系统 |

新增对象相关枚举必须先讨论、邮件确认，再写入本文档。



## 3. 对象相关枚举

### 3.1 ObjectCategory（对象类别）

`ObjectCategory` 用于描述对象在真实世界中的大类。

它属于 `ObjectDefinition` 的基础分类，是规则引擎知道的真实分类，不代表玩家已经认知。

玩家看到的内容应由认知系统、对象知识状态、对象知识标记和认知层名称决定。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 用途 |
|---|---|---|---|
| Unknown | unknown | 未知 / 未分类 | 配置错误兜底或临时占位，正式配置中不建议使用 |
| Plant | plant | 植物 | 浆果、蘑菇、草药、树、藤蔓等 |
| Animal | animal | 动物 | 狼、鹿、鸟、虫类、可狩猎生物、危险生物等 |
| Mineral | mineral | 矿物 | 石头、矿石、金属矿、晶体等 |
| Terrain | terrain | 地形 | 洞穴、河岸、泥地、峭壁、沼泽等不可简单拾取的环境对象 |
| WaterSource | water_source | 水源 | 河流、水潭、泉眼、雨水积聚点等 |
| Fire | fire | 火 | 火堆、自然火焰、燃烧源等 |
| Material | material | 材料 | 树枝、骨头、皮革、纤维、木材等可被加工或组合的材料 |
| Tool | tool | 工具 | 石矛、容器、火把等已经具有明确用途的工具对象 |
| Structure | structure | 结构物 | 营地、庇护所、围栏、简易建筑等 |
| Relic | relic | 遗迹 | 遗迹、未知造物、古老符号、文明残留等 |
| Phenomenon | phenomenon | 现象 | 风暴、雾、异常光、地震、神秘波动等非实体但可感知或影响规则的对象 |

#### 使用示例

```text
红浆果：ObjectCategory = Plant
毒蘑菇：ObjectCategory = Plant
狼：ObjectCategory = Animal
石头：ObjectCategory = Mineral
河流：ObjectCategory = WaterSource
火堆：ObjectCategory = Fire
树枝：ObjectCategory = Material
石矛：ObjectCategory = Tool
营地围栏：ObjectCategory = Structure
古老石碑：ObjectCategory = Relic
浓雾：ObjectCategory = Phenomenon
```

#### 设计注意事项

`ObjectCategory` 不等于玩家认知。

例如：

```text
真实分类：Plant（植物）
玩家当前看到：未知马赛克
```

这表示规则引擎知道它是植物，但玩家还没有认知到这一点。

`Phenomenon（现象）` 必须保留，因为《先驱者》的认知对象不应只限于物品。玩家也可以认知自然现象和异常现象。

#### 第一阶段暂不加入的类别

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| Food | 食物 | 可食用应该是 Trait（特性），不是 Category（类别） |
| Poison | 毒物 | 有毒应该是 Trait（特性），不是 Category（类别） |
| Weapon | 武器 | 武器更像 Tool 的用途或 Capability 结果，第一阶段先用 Tool 覆盖 |
| Building | 建筑 | 用 Structure 覆盖，避免过早进入建造系统 |
| Resource | 资源 | 资源是玩家或族群视角，不是真实世界类别 |



### 3.2 ObjectState（对象状态）

`ObjectState` 用于描述 `WorldObject（世界对象实例）` 当前在世界中的运行时状态。

它是对象实例状态，不是对象类别，也不是玩家认知。

示例：

```text
一棵树的 ObjectCategory 是 Plant（植物）。
这棵树当前的 ObjectState 可以是 Normal、Burning、Destroyed。
```

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 用途 |
|---|---|---|---|
| Normal | normal | 正常 | 对象处于默认可交互状态 |
| Depleted | depleted | 已耗尽 | 对象仍存在，但资源被采尽或暂时不可产出 |
| Collected | collected | 已采集 | 对象已被玩家或族群采走，通常不再留在原地或只留下痕迹 |
| Destroyed | destroyed | 已破坏 | 对象结构被破坏，无法按原用途继续交互 |
| Dead | dead | 死亡 | 用于动物或生命体死亡状态 |
| Burning | burning | 燃烧中 | 对象正在燃烧，会影响周围环境或产生持续效果 |
| Extinguished | extinguished | 已熄灭 | 曾经燃烧或发光的对象已经熄灭 |
| Growing | growing | 生长中 | 植物或可成长对象处于未成熟阶段 |
| Decayed | decayed | 腐败 / 枯萎 | 对象因时间、环境或规则效果变质、腐败、枯萎 |
| Dormant | dormant | 休眠 / 未激活 | 对象存在但当前未激活 |
| Active | active | 激活中 | 对象处于激活状态，正在产生效果 |

#### 使用示例

```text
红浆果丛：Normal -> Depleted
树枝：Normal -> Collected
狼：Normal -> Dead
火堆：Burning -> Extinguished
草药：Growing -> Normal -> Decayed
遗迹装置：Dormant -> Active
```

#### 设计注意事项

`ObjectState` 用于覆盖第一阶段最核心的世界变化：

```text
采集
破坏
死亡
燃烧
熄灭
生长
腐败
激活
```

#### 第一阶段暂不加入的状态

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| Poisoned | 中毒 | 中毒通常是 Actor 状态或 StatusEffect，不是通用对象状态 |
| Frozen | 冻结 | 环境和元素系统暂未设计，后续再加 |
| Hidden | 隐藏 | 隐藏 / 可见性应由 ObjectVisibilityState 表达 |
| Revealed | 已揭示 | 已删除揭示层设计，改由 ObjectKnowledgeState / ObjectKnowledgeFlag 表达用途和效果知识 |
| Owned | 已拥有 | 归属系统第一阶段暂不设计 |
| Locked | 锁定 | 锁定偏交互限制，可后续由 InteractionRule 或状态效果表达 |



### 3.3 ObjectVisibilityState（对象可见状态）

`ObjectVisibilityState` 用于描述玩家当前是否能在环境中感知到某个对象。

可见状态不等于认知揭示程度。玩家可能看得见一个对象，但仍然不知道它是什么。

示例：

```text
ObjectVisibilityState = Visible（可见）
ObjectKnowledgeState = Unknown（未知）
```

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 用途 |
|---|---|---|---|
| NotVisible | not_visible | 不可见 | 对象当前不在玩家感知范围内，前端不应展示 |
| Detected | detected | 已察觉 | 玩家察觉到附近存在某物，但不能确定位置或形态 |
| Obscured | obscured | 被遮挡 | 对象存在于可感知范围内，但被雾、黑暗、遮挡物等影响 |
| Visible | visible | 可见 | 对象处于清楚可见状态，但不代表玩家已经认识它 |
| OutOfRange | out_of_range | 超出范围 | 对象已被系统记录，但当前距离太远，无法直接感知 |
| Remembered | remembered | 记忆中 | 对象当前不可见，但玩家或族群记得它曾经在那里 |

#### 使用示例

```text
听到草丛动静：Detected
雾中的轮廓：Obscured
眼前的未知浆果：Visible
地图记忆中的远处水源：OutOfRange
曾经发现但当前不在视野内的遗迹：Remembered
```

#### 设计注意事项

`Remembered（记忆中）` 必须保留，因为《先驱者》的核心是认知和记忆。对象不在眼前时，仍可能通过记忆影响地图、导航和决策。

#### 第一阶段暂不加入的可见状态

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| Hidden | 隐藏 | 更像对象或生物主动隐藏的状态，第一阶段先不用 |
| Invisible | 隐形 | 偏魔法或特殊机制，第一阶段不做 |
| Revealed | 已揭示 | 已删除揭示层设计，改由 ObjectKnowledgeState / ObjectKnowledgeFlag 表达用途和效果知识 |
| Known | 已知 | 是否已知属于认知系统，不属于可见性 |



### 3.4 对象知识与传授方向

已确认删除 `ObjectRevealLevel（对象揭示程度）`。

原因：

```text
游戏核心不是“看清对象形态”，而是“通过尝试理解对象的用途、风险和行为结果”。
```

对象出现时，玩家可以直接进行尝试，例如：

```text
Eat（食用）
Use（使用）
Touch（触碰）
Collect（采集）
Attack（攻击）
Combine（组合）
```

玩家真正要学习的是：

```text
能不能吃
吃了会怎样
能不能用
用了会怎样
能不能采集
采集得到什么
能不能组合
组合产生什么
是否危险
是否能传授给族群
```

因此对象知识拆成两个枚举方向：

```text
ObjectKnowledgeState（对象知识状态）：整体成熟度。
ObjectKnowledgeFlag（对象知识标记）：具体知道了哪些行为、用途、风险和效果。
```

传授知识时，传授的不是对象本身，而是知识断言：

```text
对象 + 行为 + 效果
```

示例：

```text
red_berry + eat + restore_hunger
poison_mushroom + eat + poison
```

族群学到后，会基于该知识改变行为：

```text
red_berry + eat + restore_hunger -> 族群把红浆果当作食物来源
poison_mushroom + eat + poison -> 族群避免食用毒蘑菇
```

后续知识系统文档必须单独设计：

```text
KnowledgeClaim（知识断言）
```



### 3.5 ObjectKnowledgeState（对象知识状态）

`ObjectKnowledgeState` 表示玩家或族群对某个对象“用途、风险、行为结果”的整体掌握程度。

它不负责记录具体知道了什么。具体知道了什么由 `ObjectKnowledgeFlag（对象知识标记）` 和后续 `KnowledgeClaim（知识断言）` 表达。

示例：

```text
ObjectKnowledgeState = PartiallyKnown
```

只能说明“已经知道一些东西”，不能说明“知道它可食用”。

“知道它可食用”应由以下结构表达：

```text
ObjectKnowledgeFlag = KnowsEdible
KnowledgeClaim = red_berry + eat + restore_hunger
```

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Unknown | unknown | 未知 | 玩家遇到过该对象，但没有可靠用途或风险知识 |
| Tried | tried | 已尝试 | 玩家至少对该对象做过一次尝试，但结果还不足以形成稳定判断 |
| PartiallyKnown | partially_known | 部分已知 | 玩家知道至少一个稳定行为结果 |
| WellKnown | well_known | 较清楚 | 玩家知道多个行为结果，包含用途或风险 |
| Teachable | teachable | 可传授 | 玩家已经能把至少一条知识清楚传授给族群 |
| Operational | operational | 可稳定利用 | 族群已经能基于该知识稳定行动 |

#### 关键边界

```text
Unknown 不等于没有见过。
Unknown 表示没有可靠用途或风险知识。
对象可以是可见的，但知识状态仍然是 Unknown。
```

```text
Tried 不等于知道。
Tried 只表示做过尝试，但还没形成稳定知识。
```

```text
Teachable 不等于族群已掌握。
Teachable 表示玩家可以教。
Operational 表示族群已经能稳定用。
```

#### 状态推进示例

红浆果：

```text
Unknown -> Tried -> PartiallyKnown -> Teachable -> Operational
```

过程：

```text
见到红浆果，但不知道用途。
吃了一次，发现饥饿下降。
多次验证后确认可食用。
可以传授给族群。
族群开始稳定采集。
```

毒蘑菇：

```text
Unknown -> Tried -> PartiallyKnown -> Teachable -> Operational
```

过程：

```text
见到毒蘑菇，不知道用途。
吃了一次，中毒。
确认食用有风险。
传授族群避免食用。
族群稳定避开。
```

注意：

```text
Operational 不一定表示“利用它获得收益”。
也可以表示“稳定避开风险”。
```

#### 第一阶段暂不加入的状态

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| Identified | 已识别 | 识别名称不是核心；核心是行为和效果 |
| Mastered | 已掌握 | 容易太宽泛，Operational 更强调可转化为稳定行动 |
| Named | 已命名 | 命名属于文本或知识表达，不是整体知识状态 |
| Shared | 已共享 | 共享是传播结果，应由传播系统或 KnowledgeClaim owner 表达 |



### 3.6 ObjectKnowledgeFlag（对象知识标记）

`ObjectKnowledgeFlag` 用于表示玩家或族群具体知道了哪些行为、用途、风险和效果。

它不代替 `KnowledgeClaim（知识断言）`。

边界：

```text
ObjectKnowledgeFlag：快速判断“我大概知道哪些类型的事情”。
KnowledgeClaim：精确表达“某对象 + 某行为 + 某效果 + 置信度”。
```

示例：

```text
ObjectKnowledgeFlag = KnowsEdible
```

只能表示“知道可食用”。

更精确的知识需要由 `KnowledgeClaim` 表达：

```text
red_berry + eat + restore_hunger + confidence 85
```

#### 第一阶段分组

`ObjectKnowledgeFlag` 第一阶段按以下六组设计：

| 分组 | 中文名 | 回答的问题 | 示例方向 |
|---|---|---|---|
| Eating | 食用相关 | 能不能吃？吃了会怎样？ | 尝试过食用、知道可食用、知道有毒、知道营养效果 |
| Using | 使用相关 | 能不能用？用了会怎样？ | 尝试过使用、知道使用效果、知道工具用途、知道燃料用途 |
| Collecting | 采集相关 | 能不能采？采了得到什么？ | 尝试过采集、知道采集产出、知道可再生 |
| Combining | 组合相关 | 能不能和别的东西组合？组合后产生什么？ | 尝试过组合、知道组合结果 |
| Risk | 风险相关 | 会不会伤害我？如何安全处理？ | 知道危险、知道安全处理方式 |
| TeachingTribe | 传播与族群相关 | 我能不能教别人？族群能不能用？ | 可传授给族群、族群已知、族群可利用 |

后续将按分组逐个细化具体枚举值。


#### Eating（食用相关）

Eating 分组用于表达玩家或族群对“这个对象能不能吃、吃了会怎样”的知识。

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| HasTriedEating | has_tried_eating | 尝试过食用 | 玩家曾经对该对象执行过 Eat（食用）行为 |
| KnowsEdible | knows_edible | 知道可食用 | 玩家知道该对象可以作为食物 |
| KnowsInedible | knows_inedible | 知道不可食用 | 玩家知道该对象不能作为正常食物 |
| KnowsPoisonousWhenEaten | knows_poisonous_when_eaten | 知道食用有毒 | 玩家知道吃它会导致中毒或毒性风险 |
| KnowsNutritionEffect | knows_nutrition_effect | 知道营养效果 | 玩家知道吃它会恢复饥饿、体力或其他生理资源 |
| KnowsHealingEffect | knows_healing_effect | 知道治疗效果 | 玩家知道吃它或服用它会带来治疗效果 |
| KnowsHarmfulEffect | knows_harmful_effect | 知道有害效果 | 玩家知道吃它会造成伤害、虚弱、呕吐、幻觉等非毒性负面效果 |

注意：

```text
HasTriedEating 只表示尝试过，不代表知道它安全或有用。
KnowsPoisonousWhenEaten 表示毒性。
KnowsHarmfulEffect 表示更宽泛的不良效果。
```

第一阶段暂不加入：

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| TastesGood | 味道好 | 口味不是第一阶段规则核心 |
| FavoriteFood | 偏好食物 | 偏族群 AI 或个体偏好系统，后续再设计 |
| Cookable | 可烹饪 | 烹饪系统第一阶段未展开，可后续由组合、使用或能力系统表达 |
| RawEdible | 可生食 | 太细，后续食物系统再加 |


#### Using（使用相关）

Using 分组用于表达玩家或族群对“这个对象能不能用、用了会怎样”的知识。

`Use（使用）` 是通用行为，不等于 `Eat（食用）`、`Collect（采集）`、`Combine（组合）`。它表达把对象作为工具、燃料、防御物、触发物或功能物来使用。

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| HasTriedUsing | has_tried_using | 尝试过使用 | 玩家曾经对该对象执行过 Use（使用）行为 |
| KnowsUseEffect | knows_use_effect | 知道使用效果 | 玩家知道使用该对象会产生某种效果 |
| KnowsToolUse | knows_tool_use | 知道工具用途 | 玩家知道该对象可以作为工具使用 |
| KnowsFuelUse | knows_fuel_use | 知道燃料用途 | 玩家知道该对象可以作为燃料或参与燃烧 |
| KnowsLightUse | knows_light_use | 知道照明用途 | 玩家知道该对象可以提供光亮或改善视野 |
| KnowsDefenseUse | knows_defense_use | 知道防御用途 | 玩家知道该对象能防御、驱赶、保护或降低危险 |
| KnowsShelterUse | knows_shelter_use | 知道庇护用途 | 玩家知道该对象可以遮蔽、保护、保温或提供营地安全 |
| KnowsTriggerUse | knows_trigger_use | 知道触发用途 | 玩家知道使用该对象会触发某种机制或反应 |

第一阶段暂不加入：

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| KnowsWeaponUse | 知道武器用途 | 第一阶段先用 ToolUse 或 DefenseUse 覆盖，避免提前进入武器系统 |
| KnowsMagicUse | 知道魔法用途 | 魔法系统未设计 |
| KnowsCraftUse | 知道制作用途 | 制作更接近 Combine / Capability，先不放在 Use |


#### Collecting（采集相关）

Collecting 分组用于表达玩家或族群对“这个对象能不能采集、采集后得到什么、是否会再生”的知识。

`Collect（采集）` 关注从对象中取得资源或样本，不同于 `Use（使用）` 和 `Combine（组合）`。

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| HasTriedCollecting | has_tried_collecting | 尝试过采集 | 玩家曾经对该对象执行过 Collect（采集）行为 |
| KnowsCollectible | knows_collectible | 知道可采集 | 玩家知道该对象可以被采集 |
| KnowsNotCollectible | knows_not_collectible | 知道不可采集 | 玩家知道该对象不能被正常采集 |
| KnowsCollectYield | knows_collect_yield | 知道采集产出 | 玩家知道采集它会得到什么 |
| KnowsCollectRisk | knows_collect_risk | 知道采集风险 | 玩家知道采集它可能带来风险 |
| KnowsRenewable | knows_renewable | 知道可再生 | 玩家知道该对象会重新生长、恢复或再次产出 |
| KnowsDepletable | knows_depletable | 知道会耗尽 | 玩家知道该对象采集后会耗尽或减少 |

第一阶段暂不加入：

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| KnowsRareYield | 知道稀有产出 | 稀有掉落不是第一阶段核心 |
| KnowsPerfectHarvest | 知道完美采集 | 采集技巧系统暂未设计 |
| KnowsToolRequired | 知道需要工具 | 工具需求更适合由 InteractionRule 或 Capability 表达，后续再决定是否变成知识标记 |


#### Combining（组合相关）

Combining 分组用于表达玩家或族群对“这个对象能不能和其他对象组合、组合后产生什么”的知识。

`Combine（组合）` 是认知玩法的关键行为，用于发现对象之间的关系。

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| HasTriedCombining | has_tried_combining | 尝试过组合 | 玩家曾经尝试把该对象和其他对象组合 |
| KnowsCombinable | knows_combinable | 知道可组合 | 玩家知道该对象可以和某些对象产生组合反应或组合用途 |
| KnowsNotCombinable | knows_not_combinable | 知道不可组合 | 玩家知道该对象在当前尝试范围内无法和目标对象产生有效组合 |
| KnowsCombinationResult | knows_combination_result | 知道组合结果 | 玩家知道某个组合会产生什么结果 |
| KnowsCraftingInput | knows_crafting_input | 知道可作为制作输入 | 玩家知道该对象可以作为某种制作或构造的材料 |
| KnowsReactionWithFire | knows_reaction_with_fire | 知道与火反应 | 玩家知道该对象与火组合或接触会产生反应 |
| KnowsReactionWithWater | knows_reaction_with_water | 知道与水反应 | 玩家知道该对象与水组合或接触会产生反应 |

示例：

```text
石头 + 树枝 -> 简易工具
火 + 木材 -> 持续燃烧
草药 + 水 -> 药液
```

第一阶段暂不加入：

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| KnowsRecipe | 知道配方 | 配方系统后续会更复杂，第一阶段先用 CombinationResult 表达 |
| KnowsAdvancedCraft | 知道高级制作 | 高级制作属于文明能力或科技阶段 |
| KnowsAlchemy | 知道炼金 | 炼金 / 化学系统暂未设计 |


#### Risk（风险相关）

Risk 分组用于表达玩家或族群对“这个对象是否危险、危险来自哪里、如何安全处理”的知识。

Risk 是知识标记，不是具体伤害系统。具体伤害、毒性、灼烧等数值效果应由 `InteractionRule`、`Effect` 或 `KnowledgeClaim` 表达。

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| KnowsDangerous | knows_dangerous | 知道危险 | 玩家知道该对象会造成伤害或危险 |
| KnowsSafeHandling | knows_safe_handling | 知道安全处理方式 | 玩家知道如何安全接触、使用或避开该对象的风险 |
| KnowsContactRisk | knows_contact_risk | 知道接触风险 | 玩家知道触碰该对象可能造成伤害或负面效果 |
| KnowsEatingRisk | knows_eating_risk | 知道食用风险 | 玩家知道吃该对象有风险 |
| KnowsUseRisk | knows_use_risk | 知道使用风险 | 玩家知道使用该对象可能带来风险 |
| KnowsAttractsDanger | knows_attracts_danger | 知道会吸引危险 | 玩家知道该对象或行为会吸引危险来源 |
| KnowsRepelsDanger | knows_repels_danger | 知道能驱避危险 | 玩家知道该对象可以驱赶、阻止或降低危险 |

注意：

```text
采集风险已由 Collecting.KnowsCollectRisk 表达，Risk 分组不重复加入 KnowsCollectRiskDetailed。
```

第一阶段暂不加入：

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| KnowsDeathRisk | 知道致死风险 | 致死是危险程度或效果强度，不单独作为第一阶段 Flag |
| KnowsFearEffect | 知道恐惧效果 | 情绪系统暂未设计 |
| KnowsDiseaseRisk | 知道疾病风险 | 疾病系统后续再展开 |


#### TeachingTribe（传播与族群相关）

TeachingTribe 分组用于表达玩家能否把某条对象知识传授给族群，以及族群是否已经能基于该知识行动。

传播的不是对象本身，而是 `KnowledgeClaim（知识断言）`：

```text
对象 + 行为 + 效果
```

示例：

```text
red_berry + eat + restore_hunger
poison_mushroom + eat + poison
```

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| CanTeachTribe | can_teach_tribe | 可传授给族群 | 玩家已经能把某条对象知识清楚传授给族群 |
| HasTaughtTribe | has_taught_tribe | 已传授给族群 | 玩家已经执行过传授行为 |
| TribeKnows | tribe_knows | 族群已知 | 族群已经掌握该对象的某条知识 |
| TribeCanUse | tribe_can_use | 族群可利用 | 族群已经能把该知识转化成稳定行动 |
| TribeAvoidsRisk | tribe_avoids_risk | 族群会避险 | 族群已经能基于风险知识避免危险行为 |
| TribeCanTeachOthers | tribe_can_teach_others | 族群可继续传播 | 族群中的其他成员也能传播该知识 |

边界：

```text
CanTeachTribe：玩家能教。
HasTaughtTribe：玩家教过。
TribeKnows：族群知道。
TribeCanUse：族群能用。
TribeAvoidsRisk：族群会避险。
TribeCanTeachOthers：族群能继续传播。
```

该分组支撑文明进程：

```text
个体经验 -> 传授 -> 群体知识 -> 群体行动 -> 群体再传播
```

第一阶段暂不加入：

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| TribeWorships | 族群崇拜 | 宗教系统暂不设计 |
| TribeTrades | 族群交易 | 经济系统暂不设计 |
| TribeResearches | 族群研究 | 系统科学阶段后续再设计 |




### 3.7 ObjectRelationType（对象关系类型）

`ObjectRelationType` 用于描述对象与对象之间的关系。

它不是对象自己的属性，而是两个对象之间的关系。

示例：

```text
浆果生长在灌木上。
火会燃烧木材。
狼会捕食鹿。
草药可以和水组合。
```

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Near | near | 靠近 | 两个对象在空间上接近 |
| On | on | 在其上 | 一个对象位于另一个对象表面 |
| Inside | inside | 在其内 | 一个对象位于另一个对象内部 |
| GrowsOn | grows_on | 生长在 | 植物或生物依附某对象生长 |
| Produces | produces | 产生 | 一个对象会产生另一个对象或资源 |
| Consumes | consumes | 消耗 | 一个对象或过程会消耗另一个对象 |
| Burns | burns | 燃烧 | 一个对象会燃烧另一个对象或使其进入燃烧状态 |
| Repels | repels | 驱避 | 一个对象会驱赶、阻止或降低另一个对象的威胁 |
| Attracts | attracts | 吸引 | 一个对象会吸引另一个对象或危险来源 |
| Predates | predates | 捕食 | 一个动物捕食另一个动物 |
| CombinesWith | combines_with | 可组合 | 两个对象可以组合产生新效果或新对象 |
| Requires | requires | 需要 | 一个对象、行为或效果需要另一个对象作为条件 |
| TransformsInto | transforms_into | 转化为 | 一个对象在规则作用下转化成另一个对象 |

#### 使用示例

```text
石头 Near 河岸
```

#### 第一阶段暂不加入的关系

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| Owns | 拥有 | 所有权系统第一阶段不设计 |
| TradesWith | 交易 | 经济系统暂不设计 |

| Commands | 指挥 | 组织 / 军队系统暂不设计 |



## 4. 特性相关枚举

特性相关枚举用于支撑 `Trait（特性）` 的分类、发现、确信和规则影响。

Trait 是《先驱者》认知系统的核心单位。玩家不是单纯知道对象名称，而是逐步知道对象具有哪些特性。

示例：

```text
可食用
有毒
可燃
坚硬
可治疗
能驱兽
可作为材料
```

### 4.1 特性相关枚举清单

第一阶段确认包含以下四个特性相关枚举：

| 枚举 | 中文名 | 作用 |
|---|---|---|
| TraitCategory | 特性类别 | 描述特性本身属于哪个大类 |
| TraitDiscoveryMethod | 特性发现方式 | 描述这个特性通常通过什么行为被发现 |
| TraitCertaintyLevel | 特性确信程度 | 描述玩家或族群对某个特性的确信程度 |
| TraitEffectDomain | 特性影响领域 | 描述该特性主要影响哪个规则领域 |

四者边界：

```text
TraitCategory 回答“它是什么类型的特性”。
TraitDiscoveryMethod 回答“它怎么被发现”。
TraitCertaintyLevel 回答“我们有多确定”。
TraitEffectDomain 回答“它影响什么系统”。
```

示例：红浆果的“可食用”特性。

```text
TraitCategory = Food（食用类）
TraitDiscoveryMethod = Eat（通过食用发现）
TraitCertaintyLevel = Confirmed（已确认）
TraitEffectDomain = Hunger（影响饥饿）
```

示例：火的“会灼伤”特性。

```text
TraitCategory = Risk（风险类）
TraitDiscoveryMethod = Touch（通过触碰发现）
TraitCertaintyLevel = Confirmed（已确认）
TraitEffectDomain = Health（影响生命）
```

### 4.2 第一阶段暂不加入的特性枚举

| 枚举 | 中文名 | 暂缓原因 |
|---|---|---|
| TraitRarity | 特性稀有度 | 稀有度不是认知规则核心 |
| TraitOwner | 特性归属 | 特性归属应由对象或知识系统表达 |
| TraitPowerLevel | 特性强度等级 | 强度更适合作为数值字段，不作为枚举 |

新增特性相关枚举必须先讨论、邮件确认，再写入本文档。



### 4.3 TraitCategory（特性类别）

`TraitCategory` 用于描述特性本身属于哪个大类。

它回答：

```text
这个特性是什么类型的认知？
```

`TraitCategory` 不描述特性如何发现，也不描述特性影响哪个数值系统。发现方式由 `TraitDiscoveryMethod` 表达，影响领域由 `TraitEffectDomain` 表达。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 用途 |
|---|---|---|---|
| Food | food | 食用 | 表示对象与能否食用、营养、饱腹相关 |
| Medicine | medicine | 医疗 | 表示对象与治疗、解毒、恢复、缓解负面状态相关 |
| Poison | poison | 毒性 | 表示对象与中毒、毒性风险、毒性用途相关 |
| Material | material | 材料 | 表示对象可作为加工、制作、搭建、组合的材料 |
| Tool | tool | 工具 | 表示对象可作为工具执行某种功能 |
| Fuel | fuel | 燃料 | 表示对象可燃烧、供能、维持火焰 |
| Fire | fire | 火焰 | 表示对象与点火、燃烧、灼烧、火源相关 |
| Water | water | 水 | 表示对象与饮水、清洗、溶解、浸泡、灭火相关 |
| Shelter | shelter | 庇护 | 表示对象与遮蔽、保温、营地安全、休息相关 |
| Defense | defense | 防御 | 表示对象与防御、驱赶、阻挡、降低危险相关 |
| Danger | danger | 危险 | 表示对象会造成伤害、吸引危险或引发灾害 |
| Environment | environment | 环境 | 表示对象影响地形、天气、光照、温度、湿度等环境条件 |
| Signal | signal | 信号 | 表示对象可传递信息、标记位置、发出声光烟等信号 |
| Knowledge | knowledge | 知识 | 表示对象承载符号、记录、线索、可学习信息 |
| Social | social | 社会 | 表示对象影响族群协作、传播、仪式、组织关系 |
| Civilization | civilization | 文明 | 表示对象与文明能力、技术解锁、长期发展相关 |
| Phenomenon | phenomenon | 现象 | 表示自然现象或异常现象的可认知特性 |

#### 组合示例

一个对象可以拥有多个特性类别。

```text
红浆果：Food + Material
毒蘑菇：Poison + Medicine
树枝：Material + Fuel + Tool
火：Fire + Defense + Danger + Environment
水源：Water + Environment + Medicine
洞穴：Shelter + Danger + Knowledge
遗迹石碑：Knowledge + Civilization + Phenomenon
烟：Signal + Fire + Danger
草药：Medicine + Poison + Material
狼：Danger + Food + Material
```

#### 设计注意事项

```text
Food 不代表对象类别是食物，而是说明它具有食用相关特性。
Poison 不代表对象类别是毒物，而是说明它具有毒性相关特性。
Material 不代表对象一定是材料类对象，而是说明它可以被当作材料使用。
```

对象类别由 `ObjectCategory` 表达，特性类别由 `TraitCategory` 表达。

示例：

```text
毒蘑菇：ObjectCategory = Plant，TraitCategory = Poison + Medicine
火堆：ObjectCategory = Fire，TraitCategory = Fire + Defense + Danger
树枝：ObjectCategory = Material，TraitCategory = Material + Fuel + Tool
```

### 4.4 TraitDiscoveryMethod（特性发现方式）

`TraitDiscoveryMethod` 用于描述某个特性通常通过什么行为、反馈或传播方式被发现。

它回答：

```text
这个特性是怎么被玩家或族群知道的？
```

一个特性可以对应多个发现方式。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 用途 |
|---|---|---|---|
| Observe | observe | 观察发现 | 通过观察对象行为、环境影响、变化趋势发现 |
| Touch | touch | 触碰发现 | 通过触碰、接触、靠近发现温度、硬度、刺痛等 |
| Eat | eat | 食用发现 | 通过食用发现营养、毒性、治疗、伤害等 |
| Use | use | 使用发现 | 通过使用对象发现工具、照明、防御、触发等效果 |
| Collect | collect | 采集发现 | 通过采集发现产出、再生、耗尽、采集风险等 |
| Combine | combine | 组合发现 | 通过与其他对象组合发现反应、产物、配方、材料关系 |
| Attack | attack | 攻击发现 | 通过攻击、破坏、狩猎发现硬度、反击、掉落、危险性 |
| Wait | wait | 等待发现 | 通过时间流逝发现生长、腐败、熄灭、迁徙、再生等 |
| EnvironmentalChange | environmental_change | 环境变化发现 | 通过昼夜、天气、温度、水火等环境变化发现 |
| Accident | accident | 意外发现 | 通过非主动实验、事故、误食、误触、灾害发现 |
| Teach | teach | 被传授发现 | 通过先驱者、族群或记录传授获得 |
| Infer | infer | 推断发现 | 通过已有知识组合推断出新特性 |
| RepeatedTest | repeated_test | 重复验证发现 | 通过多次实验提高确信度 |

#### 组合示例

```text
可食用：Eat + RepeatedTest
有毒：Eat + Touch + Accident
可燃：Use + Combine + EnvironmentalChange
坚硬：Touch + Attack + Use
会腐败：Wait + Observe
可再生：Collect + Wait + RepeatedTest
能驱兽：Use + Observe + RepeatedTest
可作为材料：Collect + Combine + Use
遗迹符号可学习：Observe + Infer + Teach
```

#### 设计注意事项

```text
Observe 不等于看外观，而是观察行为和结果。
Eat 不只发现食物，也可能发现毒性或治疗效果。
Accident 允许世界事件产生知识，但仍必须进入事件和记忆系统。
Teach 表示知识来源是传播，不代表玩家亲自实验过。
Infer 表示基于已有知识推断，但初始确信度应低于直接验证。
```

### 4.5 TraitCertaintyLevel（特性确信程度）

`TraitCertaintyLevel` 用于描述玩家或族群对某个特性的确信程度。

它回答：

```text
我们有多确定这个特性是真的？
```

确信程度不等于数值置信度。数值置信度仍然用 `Percent（百分比）` 表达。

`TraitCertaintyLevel` 是对置信度区间的语义分层。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 推荐置信度范围 | 含义 |
|---|---|---|---|---|
| Unknown | unknown | 未知 | 0 | 没有有效证据 |
| Suspected | suspected | 猜测 | 1 - 29 | 有弱线索，但不足以指导行动 |
| Hypothesized | hypothesized | 假设 | 30 - 59 | 有初步依据，可以尝试验证 |
| Observed | observed | 已观察 | 60 - 74 | 观察到结果，但复现次数不足 |
| Confirmed | confirmed | 已确认 | 75 - 89 | 已通过有效实验确认，可以指导个人行动 |
| Verified | verified | 已验证 | 90 - 99 | 多次验证稳定，可以传授给族群 |
| Established | established | 已确立 | 100 | 已成为稳定族群知识或文明常识 |
| Disproven | disproven | 已证伪 | 特殊状态 | 曾经的判断被后续证据否定 |

#### 使用示例

```text
第一次看到动物避开火：Suspected
多次看到火驱赶狼：Confirmed
族群长期使用火防御：Established
误以为某果实可食用，后来多人中毒：Disproven
```

#### 设计注意事项

```text
Confirmed 可以指导玩家个人行为。
Verified 才适合稳定传授给族群。
Established 表示已进入族群稳定知识。
Disproven 必须保留，因为错误认知是认知玩法的一部分。
```

错误知识不是 bug，而是玩法内容。系统需要允许：

```text
形成假设 -> 行动验证 -> 修正错误 -> 更新知识
```

### 4.6 TraitEffectDomain（特性影响领域）

`TraitEffectDomain` 用于描述该特性主要影响哪个规则领域。

它回答：

```text
这个特性会影响游戏中的哪个系统？
```

一个特性可以影响多个领域。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 影响范围 |
|---|---|---|---|
| Health | health | 生命 | 生命值、伤害、治疗、死亡风险 |
| Hunger | hunger | 饥饿 | 饱腹、食物价值、饥饿恢复 |
| Stamina | stamina | 体力 | 体力恢复、疲劳、行动能力 |
| Poison | poison | 毒性 | 中毒、毒性积累、解毒 |
| Injury | injury | 伤害 | 刺伤、灼伤、咬伤、摔伤等具体伤害 |
| Temperature | temperature | 温度 | 寒冷、炎热、取暖、灼烧 |
| Light | light | 光照 | 视野、夜间探索、照明 |
| Safety | safety | 安全 | 降低危险、驱赶威胁、防御营地 |
| Danger | danger | 危险 | 提高风险、吸引危险、触发灾害 |
| MaterialUse | material_use | 材料用途 | 制作、组合、加工、建筑材料 |
| ToolUse | tool_use | 工具用途 | 工具行为、效率提升、可执行新动作 |
| Collection | collection | 采集 | 采集产出、再生、耗尽、采集风险 |
| Combination | combination | 组合 | 对象组合、反应、产物、配方 |
| Environment | environment | 环境 | 地形、水火、天气、区域状态变化 |
| Knowledge | knowledge | 知识 | 线索、记录、符号、可学习信息 |
| Propagation | propagation | 传播 | 知识传授、族群掌握、传播效率 |
| TribeBehavior | tribe_behavior | 族群行为 | 自动采集、避险、使用、迁徙、协作 |
| Civilization | civilization | 文明 | 能力解锁、阶段推进、长期发展 |

#### 组合示例

```text
红浆果可食用：Hunger + TribeBehavior
毒蘑菇有毒：Poison + Health + TribeBehavior
火会灼伤：Temperature + Injury + Danger
火能驱兽：Safety + TribeBehavior
树枝可燃：MaterialUse + Temperature + Combination
石头坚硬：ToolUse + MaterialUse + Combination
草药可治疗：Health + Poison + Knowledge
水源可饮用：Hunger + Health + TribeBehavior
遗迹符号可学习：Knowledge + Civilization
```

#### 设计注意事项

```text
TraitEffectDomain 不保存具体数值。
具体恢复多少饥饿、造成多少伤害、提升多少安全，由规则效果或配置字段表达。
TraitEffectDomain 只告诉系统“这个特性影响哪个领域”。
```

这有助于：

```text
AI 判断知识价值
族群决定是否传播
文明能力判断是否满足前置知识
UI 按领域分类展示知识
测试用例覆盖不同规则领域
```

### 4.7 特性枚举测试样例

以下样例用于检查四个特性枚举是否能组合表达核心认知。

| 对象知识 | TraitCategory | TraitDiscoveryMethod | TraitCertaintyLevel | TraitEffectDomain |

| 红浆果可食用 | Food | Eat | Confirmed | Hunger |
| 毒蘑菇食用有毒 | Poison | Eat / Accident | Confirmed | Poison / Health |

| 树枝可作为燃料 | Fuel | Use / Combine | Confirmed | Temperature / Combination |
| 石头可作为工具 | Tool / Material | Use / Attack | Confirmed | ToolUse / MaterialUse |
| 草药可解毒 | Medicine | Eat / RepeatedTest | Verified | Poison / Health |

测试结论：

```text

```



## 5. 行为相关枚举

行为相关枚举用于支撑玩家或实体对对象进行尝试、实验、使用、传播等行为。

《先驱者》的核心操作不是传统攻击或升级，而是通过行为理解对象的用途、风险和效果。

### 5.1 行为相关枚举清单

第一阶段确认包含以下五个行为相关枚举：

| 枚举 | 中文名 | 作用 |
|---|---|---|
| ActionType | 行为类型 | 描述玩家或实体执行的行为种类 |
| ActionRiskLevel | 行为风险等级 | 描述执行该行为的基础风险 |
| ActionRangeType | 行为范围类型 | 描述行为需要什么距离或范围 |
| ActionTargetType | 行为目标类型 | 描述行为需要什么目标 |
| ActionCostType | 行为消耗类型 | 描述行为会消耗什么资源 |

五者边界：

```text
ActionType 回答“做什么”。
ActionRiskLevel 回答“有多危险”。
ActionRangeType 回答“离多远能做”。
ActionTargetType 回答“对谁 / 对什么做”。
ActionCostType 回答“做这件事要付出什么”。
```

示例：食用未知浆果。

```text
ActionType = Eat（食用）
ActionRiskLevel = High（高风险）
ActionRangeType = Adjacent（邻近）
ActionTargetType = SingleObject（单个对象）
ActionCostType = Time + Stamina
```

示例：传授红浆果可食用知识。

```text
ActionType = Teach（传授）
ActionRiskLevel = Low（低风险）
ActionRangeType = Tribe（族群范围）
ActionTargetType = KnowledgeClaim（知识断言）
ActionCostType = Time + Attention
```

### 5.2 第一阶段暂不加入的行为枚举

| 枚举 | 中文名 | 暂缓原因 |
|---|---|---|
| ActionAnimationType | 行为动画类型 | 动画属于前端表现层，不进入规则枚举 |
| ActionInputType | 输入方式 | 键盘、鼠标、触摸属于客户端输入层 |
| ActionCooldownType | 冷却类型 | 第一阶段不是技能战斗系统，暂缓 |
| ActionPriority | 行为优先级 | 调度和 AI 阶段后续再设计 |

新增行为相关枚举必须先讨论、邮件确认，再写入本文档。



### 5.3 ActionType（行为类型）

`ActionType` 描述玩家或实体执行的行为种类。

在《先驱者》中，行为的核心作用是通过尝试理解对象的用途、风险和效果。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 用途 |
|---|---|---|---|
| Observe | observe | 观察 | 观察对象行为、环境影响、变化趋势 |
| Touch | touch | 触碰 | 通过接触测试温度、硬度、刺痛、危险等 |
| Eat | eat | 食用 | 测试对象是否可食用、是否有毒、是否有营养或治疗效果 |
| Use | use | 使用 | 把对象作为工具、燃料、防御物、触发物或功能物使用 |
| Collect | collect | 采集 | 从对象中取得资源、样本或材料 |
| Combine | combine | 组合 | 把两个或多个对象组合，测试反应、产物或新用途 |
| Attack | attack | 攻击 | 攻击、破坏、狩猎或测试对象强度与反应 |
| Wait | wait | 等待 | 推进时间，观察生长、腐败、熄灭、迁徙、再生等变化 |
| Move | move | 移动 | 移动到位置、靠近对象、进入区域 |
| Teach | teach | 传授 | 把 KnowledgeClaim（知识断言）传授给族群 |
| Record | record | 记录 | 把观察、实验或知识保存为记忆、符号、标记或文档 |
| Avoid | avoid | 避开 | 基于已知风险主动避开对象或区域 |
| Rest | rest | 休息 | 恢复体力、推进时间、触发夜晚或营地状态变化 |

#### 设计注意事项

```text
Observe 不是看外观，而是观察对象行为、环境影响和结果规律。
Record 是从个人经验走向符号、语言、文字和文明知识载体的基础行为。
Avoid 是知识带来的行动收益，知道危险后玩家和族群必须能主动避险。
Teach 的目标不是对象本身，而是 KnowledgeClaim。
```

#### 第一阶段暂不加入的行为

| 暂不加入 | 中文名 | 原因 |
|---|---|---|
| Craft | 制作 | 第一阶段先用 Combine + Capability 表达，避免过早进入制作系统 |
| Build | 建造 | 建造系统暂不设计 |
| Trade | 交易 | 经济系统暂不设计 |
| Worship | 崇拜 | 宗教系统暂不设计 |
| Research | 研究 | 系统科学阶段后续再设计 |

### 5.4 ActionRiskLevel（行为风险等级）

`ActionRiskLevel` 描述执行某个行为的基础风险。

它不是最终风险结果。最终风险还要结合对象特性、玩家知识、环境状态、工具、族群能力和规则配置。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| None | none | 无风险 | 正常情况下不会造成直接负面后果 |
| Low | low | 低风险 | 可能造成轻微成本或可控负面结果 |
| Medium | medium | 中风险 | 可能造成明显损失，需要玩家判断 |
| High | high | 高风险 | 可能造成严重伤害、中毒、资源损失或危险事件 |
| Critical | critical | 极高风险 | 可能导致死亡、重大灾害、族群损失或不可逆后果 |
| Unknown | unknown | 风险未知 | 玩家或系统尚未掌握风险等级，前端可提示谨慎尝试 |

#### 使用示例

```text
Observe：通常 None 或 Low
Touch：通常 Low 到 Medium
Eat 未知对象：通常 High 或 Unknown
Use 火：通常 Medium 到 High
Collect 带刺植物：通常 Medium
Combine 未知对象：通常 Medium 或 Unknown
Attack 危险动物：通常 High 或 Critical
Teach 已验证知识：通常 Low
Rest 在安全营地：通常 Low
Rest 在危险区域：可能 High
```

#### 设计注意事项

```text
Unknown 是认知层风险未知，不代表真实风险低。
ActionRiskLevel 是行为基础风险，不替代 InteractionRule 的具体风险结算。
前端可以用 ActionRiskLevel 做风险提示，但不能由前端决定规则结果。
```

### 5.5 ActionRangeType（行为范围类型）

`ActionRangeType` 描述行为需要什么距离、空间或作用范围。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Self | self | 自身 | 行为只作用于执行者自己 |
| Touch | touch | 接触范围 | 需要直接接触目标 |
| Adjacent | adjacent | 邻近 | 需要靠近目标，通常相邻格或近距离 |
| Near | near | 近距离 | 在短距离内可执行，不一定接触 |
| Remote | remote | 远程 | 可对远处目标执行 |
| Location | location | 指定位置 | 目标是地图位置 |
| Area | area | 区域 | 作用于一片区域 |
| Region | region | 大区域 | 作用于一个区域或地域单位 |
| Tribe | tribe | 族群范围 | 作用于族群或营地范围 |
| Global | global | 全局 | 作用于整个游戏状态或全局知识 |

#### 使用示例

```text
Eat：Self + Touch，必须接触并作用于自身
Touch：Touch
Collect：Adjacent 或 Touch
Combine：Touch 或 Adjacent
Observe：Near / Remote / Area
Move：Location
Teach：Tribe
Record：Self / Location / Global，取决于记录载体
Wait：Self / Area，取决于观察范围
```

#### 设计注意事项

```text
Range 是规则校验条件，不是前端镜头距离。
具体距离数值由地图、配置或规则系统决定。
Tribe 表示传播或族群行为范围，不表示全地图广播。
```

### 5.6 ActionTargetType（行为目标类型）

`ActionTargetType` 描述行为需要什么目标。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| None | none | 无目标 | 行为不需要外部目标 |
| Self | self | 自身 | 目标是执行者自己 |
| SingleObject | single_object | 单个对象 | 目标是一个 WorldObject |
| MultipleObjects | multiple_objects | 多个对象 | 目标是多个 WorldObject |
| ObjectPair | object_pair | 对象对 | 目标是两个对象，常用于组合 |
| Location | location | 位置 | 目标是地图坐标或位置 |
| Area | area | 区域 | 目标是一片范围 |
| Region | region | 大区域 | 目标是一个 Region |
| Entity | entity | 实体 | 目标是可行动实体，例如动物、先驱者、族人 |
| KnowledgeClaim | knowledge_claim | 知识断言 | 目标是一条对象 + 行为 + 效果的知识断言 |
| Memory | memory | 记忆 | 目标是一条个人记忆 |
| Tribe | tribe | 族群 | 目标是族群整体 |

#### 使用示例

```text
Observe：SingleObject / Area / Region
Touch：SingleObject
Eat：SingleObject
Use：SingleObject / Entity / Location
Collect：SingleObject
Combine：ObjectPair / MultipleObjects
Attack：SingleObject / Entity
Move：Location
Teach：KnowledgeClaim + Tribe
Record：Memory / KnowledgeClaim / Location
Avoid：SingleObject / Area / Region
Rest：None / Self / Location
```

#### 设计注意事项

```text
ActionTargetType 只描述目标形态，不校验目标是否合法。
合法性由 CommandValidator 和 InteractionRule 决定。
Teach 的核心目标必须是 KnowledgeClaim，不是对象 ID。
Combine 优先使用 ObjectPair，未来复杂配方再扩展 MultipleObjects。
```

### 5.7 ActionCostType（行为消耗类型）

`ActionCostType` 描述执行行为会消耗什么资源。

它只描述消耗类别，不保存具体数值。具体消耗数值由 ActionDefinition、配置或规则效果决定。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| None | none | 无消耗 | 行为没有显式资源消耗 |
| Time | time | 时间 | 消耗游戏时间或推进 Tick / GameTime |
| Stamina | stamina | 体力 | 消耗先驱者体力 |
| Health | health | 生命 | 以生命作为代价或风险成本 |
| Hunger | hunger | 饥饿 | 增加饥饿或消耗饱腹资源 |
| Item | item | 物品 | 消耗对象、材料、工具或样本 |
| ToolDurability | tool_durability | 工具耐久 | 消耗工具耐久或使用次数 |
| KnowledgeOpportunity | knowledge_opportunity | 认知机会 | 消耗一次实验机会、样本或观察窗口 |
| Attention | attention | 注意力 | 消耗个人或族群注意力 |
| TribeLabor | tribe_labor | 族群劳力 | 消耗族群劳动力或行动额度 |
| Safety | safety | 安全 | 降低安全度或增加风险暴露 |

#### 使用示例

```text
Observe：Time + Attention
Touch：Time + Stamina + Safety
Eat：Time + Safety，可能触发 Health / Hunger 变化
Use：Time + Stamina + ToolDurability
Collect：Time + Stamina，可能消耗 ToolDurability
Combine：Time + Item + Attention
Attack：Time + Stamina + Safety + ToolDurability
Wait：Time
Move：Time + Stamina
Teach：Time + Attention
Record：Time + Attention，后续可能消耗载体 Item
Avoid：Time + Stamina
Rest：Time，恢复 Stamina 但可能消耗 Safety
```

#### 设计注意事项

```text
ActionCostType 不等于最终效果。
例如 Eat 的“饥饿变化”是行为结果，不一定是行为成本。
Health 作为 Cost 表示主动付出或确定代价；作为 Effect 表示结算结果。
KnowledgeOpportunity 用于表达稀有样本、短暂现象、一次性实验窗口等认知机会。
Attention 支撑后续传播、教学、记录和族群协作。
```

## 6. 反馈相关枚举

反馈相关枚举用于描述玩家执行行为后获得的结果信息。

在《先驱者》中，反馈不是单纯的 UI 文案，而是认知增长的来源。玩家通过反馈理解对象的用途、风险、效果和价值。

### 6.1 反馈相关枚举清单

第一阶段确认包含以下四个反馈相关枚举：

| 枚举 | 中文名 | 作用 |
|---|---|---|
| FeedbackType | 反馈类型 | 描述这条反馈属于什么类型 |
| FeedbackIntensity | 反馈强度 | 描述反馈强弱程度 |
| FeedbackReliability | 反馈可靠性 | 描述反馈是否可靠、是否需要重复验证 |
| FeedbackSource | 反馈来源 | 描述反馈来自哪里 |

四者边界：

```text
FeedbackType 回答“发生了什么类型的反馈”。
FeedbackIntensity 回答“这个反馈有多强”。
FeedbackReliability 回答“这个反馈有多可信”。
FeedbackSource 回答“这个反馈从哪里来”。
```

示例：食用未知浆果后恢复饥饿。

```text
FeedbackType = Physiological（生理反馈）
FeedbackIntensity = Moderate（中等）
FeedbackReliability = Observed（已观察）
FeedbackSource = DirectExperience（直接经验）
```

示例：族人告诉你某蘑菇有毒。

```text
FeedbackType = Knowledge（知识反馈）
FeedbackIntensity = Moderate（中等）
FeedbackReliability = Reported（转述）
FeedbackSource = TribeMember（族群成员）
```

### 6.2 FeedbackType（反馈类型）

`FeedbackType` 描述行为后产生了什么类型的反馈。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 用途 |
|---|---|---|---|
| Sensory | sensory | 感知反馈 | 声音、气味、温度、触感、光亮等感知结果 |
| Physiological | physiological | 生理反馈 | 饥饿、体力、生命、中毒、治疗、疲劳等身体状态变化 |
| Emotional | emotional | 情绪反馈 | 恐惧、安心、兴奋、痛苦等情绪变化 |
| Environmental | environmental | 环境反馈 | 燃烧、扩散、熄灭、污染、照明、温度变化等环境结果 |
| ObjectState | object_state | 对象状态反馈 | 对象被采集、破坏、死亡、耗尽、激活、转化等状态变化 |
| Resource | resource | 资源反馈 | 获得、消耗、损失或发现资源 |
| Risk | risk | 风险反馈 | 发现危险、触发风险、避开风险、风险升级或降低 |
| Knowledge | knowledge | 知识反馈 | 形成线索、记忆、知识断言、确信度变化 |
| Social | social | 社会反馈 | 族群反应、传授结果、族群行为变化 |
| Civilization | civilization | 文明反馈 | 解锁能力、文明阶段推进、长期组织能力变化 |
| Spatial | spatial | 空间反馈 | 位置变化、路径发现、区域进入、距离变化 |
| Temporal | temporal | 时间反馈 | 时间推进、等待结果、昼夜变化、持续效果变化 |
| Failure | failure | 失败反馈 | 行为失败、目标无效、条件不足、无明显结果 |

#### 使用示例

```text
吃红浆果恢复饥饿：Physiological + Knowledge
触碰火焰受伤：Sensory + Physiological + Risk
采集树枝获得材料：Resource + ObjectState
火把照亮夜晚：Environmental + Spatial
传授知识成功：Social + Knowledge
红浆果知识改变族群采集行为：Social + Civilization
等待火堆熄灭：Temporal + ObjectState
组合石头和树枝失败：Failure + Knowledge
```

#### 设计注意事项

```text
FeedbackType 可以多选。
一条规则结果可能同时产生生理反馈、知识反馈和风险反馈。
FeedbackType 不保存具体数值，具体数值由事件 payload 或 StateChange 表达。
Failure 不一定是错误，也可能是“无反应”这一认知结果。
```

### 6.3 FeedbackIntensity（反馈强度）

`FeedbackIntensity` 描述反馈强弱程度。

它是语义等级，不替代具体数值。具体恢复多少饥饿、受到多少伤害，仍由规则效果字段表达。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| None | none | 无 | 没有明显反馈 |
| Trace | trace | 微弱 | 很弱的线索或效果，需要谨慎解读 |
| Minor | minor | 轻微 | 轻微但可感知的反馈 |
| Moderate | moderate | 中等 | 明显反馈，足以影响判断 |
| Major | major | 强烈 | 强反馈，通常会显著影响状态或决策 |
| Severe | severe | 严重 | 严重反馈，可能造成重大损失、危险或强收益 |
| Critical | critical | 关键 | 关键反馈，可能导致死亡、重大解锁、灾害或文明转折 |

#### 使用示例

```text
闻到淡淡气味：Trace
吃浆果略微恢复饥饿：Minor
吃浆果明显恢复饥饿：Moderate
触碰火焰被灼伤：Major
误食毒蘑菇严重中毒：Severe
发现火能稳定驱赶狼：Critical
```

#### 设计注意事项

```text
Intensity 是认知和表现层辅助等级，不直接替代数值。
同样是 Major，在生命、饥饿、文明解锁中的具体数值含义不同。
前端可以用 Intensity 决定提示、音效、震动和文本强调程度。
```

### 6.4 FeedbackReliability（反馈可靠性）

`FeedbackReliability` 描述反馈是否可靠、是否需要重复验证。

它帮助系统区分“亲自验证过的结果”和“听说/猜测/偶然发生的结果”。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Unknown | unknown | 未知 | 无法判断可靠性 |
| Anecdotal | anecdotal | 个例 | 只发生过一次，可能是偶然 |
| Observed | observed | 已观察 | 亲眼观察到结果，但复现不足 |
| Repeated | repeated | 重复出现 | 多次出现类似结果，可靠性较高 |
| Controlled | controlled | 控制验证 | 在相对可控条件下验证过 |
| Confirmed | confirmed | 已确认 | 足以指导个人稳定行动 |
| Established | established | 已确立 | 已成为族群稳定知识或文明常识 |
| Contradicted | contradicted | 有矛盾 | 存在相互冲突的反馈，需要继续验证 |
| Reported | reported | 转述 | 来自他人、族群、记录或传说，非亲自验证 |

#### 使用示例

```text
第一次吃浆果没事：Anecdotal
多次吃浆果都恢复饥饿：Repeated
在相同条件下测试火驱狼：Controlled
族群长期使用火防御：Established
有人说蘑菇可吃但自己吃后中毒：Contradicted
族人告诉你某区域有水源：Reported
```

#### 设计注意事项

```text
Reliability 不等于 Certainty，但会影响 TraitCertaintyLevel 和 KnowledgeClaim.confidence。
Reported 可以通过后续实验升级为 Observed、Repeated 或 Confirmed。
Contradicted 必须保留，因为错误认知和修正是玩法核心。
```

### 6.5 FeedbackSource（反馈来源）

`FeedbackSource` 描述反馈来自哪里。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| DirectExperience | direct_experience | 直接经验 | 玩家亲自执行行为后获得反馈 |
| Observation | observation | 观察 | 玩家未直接干预，只是观察到结果 |
| Environment | environment | 环境 | 来自天气、昼夜、地形、水火、区域变化等环境因素 |
| ObjectReaction | object_reaction | 对象反应 | 来自对象自身反应，例如燃烧、反击、变质、激活 |
| ActorReaction | actor_reaction | 实体反应 | 来自动物、族人、敌对生物等实体反应 |
| TribeMember | tribe_member | 族群成员 | 来自族人经验、传授、失败或行为变化 |
| Memory | memory | 记忆 | 来自已有个人记忆的回忆或对照 |
| Record | record | 记录 | 来自符号、标记、文字、遗迹记录等外部记录 |
| Inference | inference | 推断 | 来自已有知识组合推断 |
| Accident | accident | 意外 | 来自事故、误触、误食、灾害等非计划结果 |
| RuleSystem | rule_system | 规则系统 | 来自系统性结算，例如能力解锁、状态变化、阶段推进 |

#### 使用示例

```text
玩家吃浆果：DirectExperience
看到狼避开火：Observation + ActorReaction
暴雨熄灭火堆：Environment
遗迹装置启动：ObjectReaction
族人因误食中毒：TribeMember + Accident
查看石碑记录：Record
根据树枝可燃推断干草可燃：Inference
解锁夜间防御能力：RuleSystem
```

#### 设计注意事项

```text
FeedbackSource 可以多选。
Source 帮助决定可靠性、记忆来源和知识传播方式。
DirectExperience 通常比 Reported 更可靠，但不一定绝对正确。
Inference 的初始确信度应低于直接验证。
```

以下样例用于检查四个反馈相关枚举是否能表达核心玩法。

| 场景 | FeedbackType | FeedbackIntensity | FeedbackReliability | FeedbackSource |
|---|---|---|---|---|
| 第一次吃红浆果，饥饿下降 | Physiological / Knowledge | Moderate | Anecdotal | DirectExperience |

| 看到狼避开火 | Risk / Knowledge / Social | Major | Observed | Observation / ActorReaction |
| 族人报告远处有水源 | Knowledge / Social | Moderate | Reported | TribeMember |
| 等待后浆果重新生长 | ObjectState / Temporal / Knowledge | Moderate | Observed | Observation |




## 7. 认知相关枚举

认知相关枚举用于支撑记忆、知识、证据、确信度变化和知识归属。

在《先驱者》中，认知系统不关心对象外观是否清晰，而关心玩家或族群是否理解了对象在某种行为下的用途、风险和效果。

### 7.1 认知相关枚举清单

第一阶段确认包含以下六个认知相关枚举：

| 枚举 | 中文名 | 作用 |
|---|---|---|
| CognitionOwnerType | 认知归属类型 | 描述这份认知属于谁 |
| KnowledgeScope | 知识范围 | 描述知识影响范围 |
| KnowledgeStatus | 知识状态 | 描述知识当前生命周期状态 |
| MemoryType | 记忆类型 | 描述个人记忆的类型 |
| EvidenceType | 证据类型 | 描述支持某条知识的证据类型 |
| ConfidenceChangeReason | 置信度变化原因 | 描述知识置信度为什么变化 |

边界：

```text
CognitionOwnerType 回答“谁知道”。
KnowledgeScope 回答“这条知识影响多大范围”。
KnowledgeStatus 回答“这条知识处于什么生命周期”。
MemoryType 回答“这是什么类型的个人记忆”。
EvidenceType 回答“证据来自哪里”。
ConfidenceChangeReason 回答“为什么更相信或更不相信”。
```

### 7.2 CognitionOwnerType（认知归属类型）

`CognitionOwnerType` 描述这份认知属于谁。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Pioneer | pioneer | 先驱者 | 玩家控制的觉醒个体拥有的个人认知 |
| Tribe | tribe | 族群 | 族群整体掌握的共享认知 |
| TribeMember | tribe_member | 族群成员 | 某个族人或小群体掌握的认知 |
| ExternalRecord | external_record | 外部记录 | 石碑、符号、遗迹、文本等外部载体中的认知 |
| Environment | environment | 环境线索 | 从环境稳定规律中体现出的认知来源 |
| System | system | 系统 | 规则系统用于调试、校验或初始化的认知来源 |

#### 设计注意事项

```text
Pioneer 认知不等于 Tribe 认知。
先驱者知道某物可食用，不代表族群已经知道。
ExternalRecord 可以被读取、解释、误解或传播。
System 只用于规则初始化、测试和调试，不应直接暴露为玩家知识来源。
```

### 7.3 KnowledgeScope（知识范围）

`KnowledgeScope` 描述知识影响范围。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Personal | personal | 个人 | 只影响先驱者个人判断和行为 |
| LocalGroup | local_group | 小群体 | 影响少量族人或临时队伍 |
| Tribe | tribe | 族群 | 影响整个族群行为 |
| Region | region | 区域 | 只在某个区域内有效或被掌握 |
| Civilization | civilization | 文明 | 成为长期文明能力或跨代知识 |
| Universal | universal | 普遍 | 被视为普遍规律，可跨区域、跨对象推广 |

#### 使用示例

```text
先驱者知道红浆果可食用：Personal
采集小队知道红浆果可食用：LocalGroup
整个族群会采集红浆果：Tribe
某片森林的蘑菇有毒：Region
火能用于夜间防御：Civilization
火会燃烧可燃材料：Universal
```

### 7.4 KnowledgeStatus（知识状态）

`KnowledgeStatus` 描述知识当前生命周期状态。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Draft | draft | 草稿 | 刚形成的初步记录，尚未稳定 |
| Hypothesis | hypothesis | 假设 | 有一定依据，但需要验证 |
| Active | active | 有效 | 当前可用于判断或行动 |
| Teachable | teachable | 可传授 | 已经足够清晰，可以传授给他人 |
| Shared | shared | 已共享 | 已经完成传播，至少被他人接收 |
| Operational | operational | 可行动 | 已经转化为族群稳定行为 |
| Established | established | 已确立 | 已成为稳定常识、制度或文明能力前置 |
| Disputed | disputed | 有争议 | 存在冲突证据或不同解释 |
| Deprecated | deprecated | 已废弃 | 被更准确知识替代，不再推荐使用 |
| Disproven | disproven | 已证伪 | 被后续证据证明错误 |

#### 设计注意事项

```text
Disputed 和 Disproven 必须保留，因为错误认知和修正是核心玩法。
Deprecated 用于保留历史知识，同时阻止系统继续推荐旧知识。
Operational 表示知识已经能驱动稳定行为，不只是被知道。
```

### 7.5 MemoryType（记忆类型）

`MemoryType` 描述个人记忆的类型。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Observation | observation | 观察记忆 | 来自观察对象、环境或实体行为 |
| Experiment | experiment | 实验记忆 | 来自主动尝试行为 |
| Accident | accident | 意外记忆 | 来自误触、误食、灾害等非计划事件 |
| Result | result | 结果记忆 | 记录行为带来的明确结果 |
| Failure | failure | 失败记忆 | 记录无效、失败、无反应或条件不足 |
| Warning | warning | 警示记忆 | 记录危险、风险、伤害或损失 |
| Teaching | teaching | 传授记忆 | 记录教给他人或被他人教导 |
| Inference | inference | 推断记忆 | 来自已有知识组合推断 |
| Location | location | 位置记忆 | 记录对象、资源、危险或区域位置 |
| Social | social | 社会记忆 | 记录族群反应、传播效果、协作变化 |
| Symbolic | symbolic | 符号记忆 | 记录符号、标记、文字、遗迹信息 |

#### 使用示例

```text
看到狼避开火：Observation
吃浆果恢复饥饿：Experiment + Result
误食毒蘑菇中毒：Accident + Warning
组合石头和树枝无反应：Experiment + Failure
记住水源位置：Location
教族人采集浆果：Teaching + Social
从遗迹符号推断季节规律：Symbolic + Inference
```

### 7.6 EvidenceType（证据类型）

`EvidenceType` 描述支持某条知识的证据来源。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| DirectTest | direct_test | 直接测试 | 玩家亲自实验得到证据 |
| RepeatedTest | repeated_test | 重复测试 | 多次实验得到一致证据 |
| Observation | observation | 观察证据 | 通过观察自然发生的结果获得证据 |
| Accident | accident | 意外证据 | 通过事故、误触、误食获得证据 |
| TribeReport | tribe_report | 族群报告 | 族人报告或族群行为提供证据 |
| ExternalRecord | external_record | 外部记录 | 石碑、符号、文字、遗迹等记录提供证据 |
| Inference | inference | 推断证据 | 由已有知识推导而来 |
| Contradiction | contradiction | 反证 | 与当前知识冲突的证据 |
| LongTermUse | long_term_use | 长期使用 | 长时间稳定使用形成证据 |
| SystemValidation | system_validation | 系统验证 | 规则系统测试、调试或初始化验证用证据 |

#### 设计注意事项

```text
EvidenceType 会影响 TraitCertaintyLevel 和 KnowledgeClaim.confidence。
Contradiction 可以降低置信度或将 KnowledgeStatus 推向 Disputed / Disproven。
LongTermUse 通常推动知识进入 Established。
SystemValidation 不应直接作为玩家可见叙事证据。
```

### 7.7 ConfidenceChangeReason（置信度变化原因）

`ConfidenceChangeReason` 描述知识置信度为什么变化。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| NewObservation | new_observation | 新观察 | 新观察提高或降低置信度 |
| SuccessfulTest | successful_test | 成功实验 | 实验结果支持当前知识 |
| FailedTest | failed_test | 失败实验 | 实验未支持当前知识 |
| RepeatedConfirmation | repeated_confirmation | 重复确认 | 多次结果一致，提高置信度 |
| ContradictoryEvidence | contradictory_evidence | 矛盾证据 | 新证据与当前知识冲突 |
| TeachingAccepted | teaching_accepted | 传授被接受 | 他人接受传授，提高传播层面置信度 |
| TeachingFailed | teaching_failed | 传授失败 | 他人未理解或无法复现，降低传播层面置信度 |
| TribeUsageSuccess | tribe_usage_success | 族群使用成功 | 族群成功应用知识，提高置信度 |
| TribeUsageFailure | tribe_usage_failure | 族群使用失败 | 族群应用失败，降低置信度或引发争议 |
| EnvironmentalChange | environmental_change | 环境变化 | 环境条件改变导致知识适用性变化 |
| Decay | decay | 记忆衰减 | 长时间未验证导致置信度下降 |
| Correction | correction | 修正 | 主动修正旧知识或更新解释 |

#### 使用示例

```text
第一次吃红浆果恢复饥饿：SuccessfulTest
连续三次吃红浆果都恢复饥饿：RepeatedConfirmation
有人吃红浆果中毒：ContradictoryEvidence
教族人采集后成功获得食物：TribeUsageSuccess
雨季中火堆难以维持：EnvironmentalChange
长期没有验证某个草药效果：Decay
发现旧知识只在特定地区成立：Correction
```

### 7.8 认知枚举组合测试样例

| 场景 | CognitionOwnerType | KnowledgeScope | KnowledgeStatus | MemoryType | EvidenceType | ConfidenceChangeReason |
|---|---|---|---|---|---|---|
| 先驱者首次吃红浆果恢复饥饿 | Pioneer | Personal | Hypothesis | Experiment / Result | DirectTest | SuccessfulTest |
| 多次吃红浆果都有效 | Pioneer | Personal | Active | Experiment / Result | RepeatedTest | RepeatedConfirmation |
| 把红浆果知识教给族群 | Tribe | Tribe | Shared | Teaching / Social | TribeReport | TeachingAccepted |
| 族群稳定采集红浆果 | Tribe | Tribe | Operational | Social | LongTermUse | TribeUsageSuccess |
| 误以为某蘑菇可食用，后来中毒 | Pioneer | Personal | Disputed | Accident / Warning | Contradiction | ContradictoryEvidence |
| 确认毒蘑菇不可食用 | Tribe | Tribe | Teachable | Teaching / Warning | RepeatedTest | Correction |
| 遗迹记录提示季节洪水 | ExternalRecord | Region | Hypothesis | Symbolic / Inference | ExternalRecord | NewObservation |
| 火防御成为族群常识 | Tribe | Civilization | Established | Social | LongTermUse | TribeUsageSuccess |

测试结论：

```text
当前六个认知相关枚举可以组合表达个人经验、知识传播、族群应用、错误认知、证据冲突、知识修正和文明常识形成。
```

## 8. 传播相关枚举

传播相关枚举用于描述一条知识如何从先驱者、个人记忆或族群成员传递给其他主体。

在《先驱者》中，传播不是简单把 `ObjectId` 交给别人，而是把一条 `KnowledgeClaim` 传出去。

核心传播对象应是：

```text
对象 + 行为 + 效果 + 证据/置信度 + 适用条件
```

例子：

```text
red_berry + eat + restore_hunger
poison_mushroom + eat + poison
branch + use_as_fuel + keep_fire_alive
fire + use_near_wolf + repel_danger
```

第一阶段传播系统只解决“知识如何被别人接收、理解、应用、失败”。不处理复杂语言系统、文化系统、政治系统。

### 8.1 传播相关枚举清单

第一阶段确认包含以下三个传播相关枚举：

| 枚举 | 中文名 | 作用 |
|---|---|---|
| PropagationChannelType | 传播渠道类型 | 描述知识通过什么方式传播 |
| PropagationResultType | 传播结果类型 | 描述传播完成后的结果 |
| PropagationBlockReason | 传播阻断原因 | 描述传播为什么失败或无法进行 |

边界：

```text
PropagationChannelType 回答“怎么传”。
PropagationResultType 回答“传成什么样”。
PropagationBlockReason 回答“为什么没传成”。
```

### 8.2 PropagationChannelType（传播渠道类型）

`PropagationChannelType` 描述知识通过什么方式传播。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| DirectTeaching | direct_teaching | 直接传授 | 先驱者明确告诉别人某条知识 |
| Demonstration | demonstration | 演示 | 通过现场操作展示知识效果 |
| SharedExperience | shared_experience | 共同经历 | 多个主体一起经历事件后形成共享认知 |
| Imitation | imitation | 模仿 | 其他主体模仿先驱者或族人的行为 |
| WarningSignal | warning_signal | 警示信号 | 通过警告、标记、声音、姿态阻止危险行为 |
| SymbolRecord | symbol_record | 符号记录 | 通过图案、刻痕、文字、地图等外部记录传播 |
| OralTradition | oral_tradition | 口耳相传 | 知识在族群内通过叙述长期流传 |
| ToolPattern | tool_pattern | 工具范式 | 通过工具形态和使用习惯隐含传播知识 |
| RitualPractice | ritual_practice | 仪式实践 | 通过固定行为流程保存和传播知识 |
| EnvironmentalMark | environmental_mark | 环境标记 | 在地点或对象附近留下标记传播信息 |
| DelegatedTeaching | delegated_teaching | 委托传授 | 由已学会的族人继续教其他人 |
| PassiveDiffusion | passive_diffusion | 被动扩散 | 知识因长期观察族群行为而自然扩散 |

#### 使用示例

```text
告诉族人红浆果可以吃：DirectTeaching
当着族人吃红浆果并恢复饥饿：Demonstration
采集队一起发现毒蘑菇会中毒：SharedExperience
族人看到先驱者用树枝维持火堆后跟着做：Imitation
在毒蘑菇旁留下危险标记：WarningSignal / EnvironmentalMark
把水源位置画在石板上：SymbolRecord
火堆守夜流程代代相传：RitualPractice / OralTradition
会采集的族人教新族人采集：DelegatedTeaching
族群长期看见大家避开毒蘑菇：PassiveDiffusion
```

#### 设计注意事项

```text
传播渠道可以多选。
Demonstration 通常比 DirectTeaching 更容易让族人理解效果。
WarningSignal 适合传播“不要做什么”，例如不要吃、不要碰、不要靠近。
SymbolRecord 需要未来的符号/记录能力支持，第一阶段只保留枚举入口。
OralTradition、RitualPractice、ToolPattern 更偏文明阶段，不应在早期随意使用。
DelegatedTeaching 必须要求传播者已经拥有足够高的 KnowledgeStatus 或传播能力。
PassiveDiffusion 速度慢，但可以表现族群行为自然改变。
```

### 8.3 PropagationResultType（传播结果类型）

`PropagationResultType` 描述一次传播完成后的结果。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| NotStarted | not_started | 未开始 | 传播请求尚未执行 |
| Accepted | accepted | 已接受 | 接收者接受该知识，但未必能稳定应用 |
| Understood | understood | 已理解 | 接收者理解了对象、行为和效果的关系 |
| Memorized | memorized | 已记住 | 接收者形成记忆，可在后续继续使用 |
| Teachable | teachable | 可再传授 | 接收者已经能把知识继续传给别人 |
| Operational | operational | 可执行 | 知识已经转化为稳定行为或生产规则 |
| PartiallyAccepted | partially_accepted | 部分接受 | 接收者只接受了部分内容 |
| Misunderstood | misunderstood | 误解 | 接收者理解错了对象、行为、效果或条件 |
| Ignored | ignored | 被忽略 | 接收者没有采纳传播内容 |
| Rejected | rejected | 被拒绝 | 接收者主动不相信或抵触该知识 |
| Failed | failed | 失败 | 传播过程没有产生有效认知变化 |
| HarmfulSpread | harmful_spread | 有害扩散 | 错误知识被传播并造成风险扩大 |
| Contradicted | contradicted | 遭遇反证 | 传播内容被接收者经验或现场证据反驳 |
| Expired | expired | 已失效 | 传播太晚或条件变化导致知识不再有效 |

#### 使用示例

```text
族人知道红浆果可以吃但还没自己采集：Accepted
族人理解红浆果 + 吃 + 恢复饥饿：Understood
族人第二天仍然记得红浆果可食用：Memorized
族人可以教别人识别可食用浆果：Teachable
族群稳定安排采集红浆果：Operational
族人只记得“红色果子能吃”，忘了限定对象：PartiallyAccepted
族人误以为所有红色果子都能吃：Misunderstood
族人不相信火能驱赶狼：Rejected
错误地传播毒蘑菇可食用并导致中毒：HarmfulSpread
族人现场吃红浆果后中毒，反驳原知识：Contradicted
雨季洪水结束后才传播避难路线：Expired
```

#### 设计注意事项

```text
Accepted 不等于 Understood。
Understood 不等于 Operational。
Operational 通常要求知识被族群行为系统使用。
Teachable 需要检查接收者是否拥有再传播能力。
Misunderstood 和 HarmfulSpread 必须保留，因为错误知识也是核心玩法的一部分。
PartiallyAccepted 是非常重要的中间态，可用于表现知识在传播中丢失条件。
Contradicted 应进入认知系统，触发 ConfidenceChangeReason.ContradictoryEvidence。
```

### 8.4 PropagationBlockReason（传播阻断原因）

`PropagationBlockReason` 描述传播为什么失败、无法开始或无法完成。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| None | none | 无 | 没有阻断原因 |
| KnowledgeNotTeachable | knowledge_not_teachable | 知识不可传授 | 知识尚未达到可传授状态 |
| InsufficientConfidence | insufficient_confidence | 置信度不足 | 证据不足，传播者自己也不确定 |
| MissingSubject | missing_subject | 缺少知识对象 | 缺少被传播的对象、行为或效果之一 |
| MissingReceiver | missing_receiver | 缺少接收者 | 没有有效接收者 |
| ReceiverUnavailable | receiver_unavailable | 接收者不可用 | 接收者死亡、离开、昏迷、忙碌或无法参与 |
| ReceiverIncapable | receiver_incapable | 接收者能力不足 | 接收者缺少理解或执行该知识的能力 |
| NoSharedContext | no_shared_context | 缺少共同语境 | 接收者无法理解传播者表达的对象或行为 |
| ChannelUnavailable | channel_unavailable | 渠道不可用 | 当前缺少对应传播渠道，例如不能记录、不能演示 |
| MissingDemonstrationTarget | missing_demonstration_target | 缺少演示对象 | 演示需要的对象、工具、地点或环境不存在 |
| TooDangerous | too_dangerous | 过于危险 | 传播过程本身风险过高 |
| ContradictoryKnowledge | contradictory_knowledge | 既有知识冲突 | 接收者已有知识与传播内容冲突 |
| LowTrust | low_trust | 信任不足 | 接收者不信任传播者或来源 |
| MemoryOverload | memory_overload | 记忆负担过高 | 接收者当前无法记住更多知识 |
| EnvironmentalNoise | environmental_noise | 环境干扰 | 战斗、灾害、混乱等导致传播失败 |
| TimeNotEnough | time_not_enough | 时间不足 | 当前行动时间不足以完成传播 |
| ConditionMismatch | condition_mismatch | 条件不匹配 | 知识适用条件与当前环境不一致 |
| AlreadyKnown | already_known | 已经知道 | 接收者已经掌握该知识，无需传播 |
| RuleDenied | rule_denied | 规则拒绝 | 规则系统判定该传播不合法 |

#### 使用示例

```text
只吃过一次红浆果就想教全族：InsufficientConfidence / KnowledgeNotTeachable
知识只记录了对象，没有记录吃后的效果：MissingSubject
附近没有族人：MissingReceiver
接收者正在逃命：ReceiverUnavailable / EnvironmentalNoise
族人无法理解“干燥树枝更易燃”：NoSharedContext
想画地图但族群还没有符号记录能力：ChannelUnavailable
想演示火驱狼，但附近没有狼或火：MissingDemonstrationTarget
想教族人吃未知蘑菇但风险过高：TooDangerous
族人过去吃类似红果中毒，因此不相信：ContradictoryKnowledge / LowTrust
冬季传播“这里有浆果”，但当前季节没有：ConditionMismatch
族人早就掌握红浆果采集：AlreadyKnown
规则系统发现目标不是 KnowledgeClaim：RuleDenied
```

#### 设计注意事项

```text
None 只表示没有阻断，不表示传播成功。
KnowledgeNotTeachable 和 InsufficientConfidence 是不同概念：前者是状态不允许，后者是证据不够强。
MissingSubject 用于保护 KnowledgeClaim 完整性，防止只传 ObjectId。
NoSharedContext 用于处理“知道名字但不理解行为/条件”的问题。
ConditionMismatch 非常重要，避免把区域性、季节性知识错误扩散为普遍知识。
RuleDenied 是兜底错误，不应替代具体阻断原因。
```

### 8.5 传播枚举组合测试样例

以下样例用于检查三个传播相关枚举是否能表达核心玩法。

| 场景 | PropagationChannelType | PropagationResultType | PropagationBlockReason |
|---|---|---|---|
| 先驱者告诉族人红浆果可食用 | DirectTeaching | Accepted | None |
| 先驱者当场吃红浆果恢复饥饿 | Demonstration | Understood | None |
| 族人第二天仍会采集红浆果 | DirectTeaching / Demonstration | Memorized | None |
| 族人开始稳定组织采集红浆果 | PassiveDiffusion / DelegatedTeaching | Operational | None |
| 只吃过一次就想教全族 | DirectTeaching | Failed | KnowledgeNotTeachable / InsufficientConfidence |
| 族人误以为所有红色果实都能吃 | OralTradition / PassiveDiffusion | Misunderstood | NoSharedContext |
| 错误传播毒蘑菇可食用导致中毒 | DirectTeaching | HarmfulSpread | InsufficientConfidence |
| 在毒蘑菇旁做危险标记 | WarningSignal / EnvironmentalMark | Understood | None |
| 想画水源地图但还没有记录能力 | SymbolRecord | Failed | ChannelUnavailable |
| 会采集的族人教新人 | DelegatedTeaching | Teachable | None |
| 雨季结束后才传播避难路线 | DirectTeaching | Expired | ConditionMismatch |
| 族人已有相反经验，不接受火能驱狼 | DirectTeaching / Demonstration | Contradicted | ContradictoryKnowledge / LowTrust |
| 火堆守夜流程成为固定习惯 | RitualPractice / ToolPattern | Operational | None |

测试结论：

```text
当前三个传播相关枚举可以组合表达直接教学、现场演示、族群模仿、符号记录、错误传播、条件丢失、传播阻断和知识转化为稳定行为。
```

### 8.6 传播枚举与认知系统的关系

传播枚举不直接修改知识本体，而是为传播规则提供判定结果。

建议第一阶段传播流程：

```text
1. 输入 KnowledgeClaim。
2. 检查 KnowledgeClaim 是否完整：对象、行为、效果不能为空。
3. 检查传播者是否知道该知识。
4. 检查 KnowledgeStatus 是否允许传播。
5. 检查传播渠道是否可用。
6. 检查接收者是否可用、能否理解、是否已有冲突知识。
7. 输出 PropagationResultType。
8. 若传播有效，创建或更新接收者的 Memory / KnowledgeClaim。
9. 若传播失败，记录 PropagationBlockReason。
10. 若产生误解或反证，通知认知系统调整置信度。
```

关键约束：

```text
不能传播裸 ObjectId。
不能只传播“红浆果”，必须传播“红浆果 + 吃 + 恢复饥饿”这类完整认知。
传播失败也要形成事件，方便调试和叙事反馈。
错误传播不是 bug，而是玩法的一部分。
```

传播相关枚举已完成初版。

## 9. 文明相关枚举

文明相关枚举用于描述族群从零散生存经验，逐步发展为稳定能力、组织行为和文明阶段的过程。

在《先驱者》中，文明不是抽象等级条，而是由知识传播、族群应用和能力解锁共同推动。

第一阶段文明系统只解决三个问题：

```text
族群处于什么发展阶段。
族群获得了什么稳定能力。
某个能力为什么可以被解锁。
```

### 9.1 文明相关枚举清单

第一阶段确认包含以下三个文明相关枚举：

| 枚举 | 中文名 | 作用 |
|---|---|---|
| CivilizationStage | 文明阶段 | 描述族群整体发展阶段 |
| CapabilityType | 能力类型 | 描述族群已经掌握或可使用的稳定能力 |
| UnlockConditionType | 解锁条件类型 | 描述能力、阶段或规则为什么被解锁 |

边界：

```text
CivilizationStage 回答“族群发展到什么阶段”。
CapabilityType 回答“族群能稳定做什么”。
UnlockConditionType 回答“为什么现在能解锁”。
```

### 9.2 CivilizationStage（文明阶段）

`CivilizationStage` 描述族群整体发展阶段。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Awakening | awakening | 觉醒期 | 先驱者刚获得认知能力，族群还没有稳定共享知识 |
| Foraging | foraging | 采集期 | 族群能稳定识别、采集和避开部分对象 |
| FireKeeping | fire_keeping | 守火期 | 族群掌握火的维持、照明、防御或加工用途 |
| ToolUsing | tool_using | 用具期 | 族群能稳定使用简单工具提升采集、防御或加工效率 |
| ShelterBuilding | shelter_building | 庇护期 | 族群能建设或维护基础庇护点，降低环境风险 |
| Symbolic | symbolic | 符号期 | 族群能用标记、图案、记录保存和传播知识 |
| OrganizedTribe | organized_tribe | 组织部落期 | 族群能分工、安排长期任务并稳定传承关键知识 |
| ProtoCivilization | proto_civilization | 原始文明期 | 族群形成稳定生产、记录、传承和风险管理体系 |

#### 使用示例

```text
开局只有先驱者个人经验：Awakening
族群知道红浆果可食用、毒蘑菇不可食用：Foraging
族群会收集树枝维持火堆：FireKeeping
族群会用石器采集或防御：ToolUsing
族群固定在洞穴或搭建庇护点过夜：ShelterBuilding
族群能用标记记录水源和危险：Symbolic
族群安排采集者、守火者、探索者：OrganizedTribe
族群能跨季节保存知识并稳定扩张：ProtoCivilization
```

#### 设计注意事项

```text
CivilizationStage 不是单纯数值等级，而是由能力和知识状态共同推导。
阶段可以作为解锁条件，但不应替代具体 CapabilityType。
不要为了升级而升级，必须能映射到族群实际行为变化。
第一阶段不设计现代文明、科技树、国家政治和经济制度。
```

### 9.3 CapabilityType（能力类型）

`CapabilityType` 描述族群已经掌握或可使用的稳定能力。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| IdentifyEdible | identify_edible | 识别可食物 | 能识别可食用对象并避免误食 |
| IdentifyDanger | identify_danger | 识别危险 | 能识别危险对象、地点或行为 |
| SafeForaging | safe_foraging | 安全采集 | 能按已知规则采集食物或材料并控制风险 |
| BasicToolUse | basic_tool_use | 基础工具使用 | 能稳定使用简单工具完成动作 |
| ToolCrafting | tool_crafting | 工具制作 | 能组合材料制造或修复基础工具 |
| FireMaking | fire_making | 生火 | 能制造火源 |
| FireKeeping | fire_keeping | 守火 | 能维持火堆、补充燃料并管理火势 |
| FireDefense | fire_defense | 火焰防御 | 能用火降低野兽或夜间风险 |
| Cooking | cooking | 加工食物 | 能通过火或工具改变食物效果或风险 |
| ShelterUse | shelter_use | 使用庇护 | 能利用洞穴、树冠、遮蔽物等降低环境伤害 |
| ShelterBuilding | shelter_building | 建造庇护 | 能主动建造或维护庇护点 |
| PathMarking | path_marking | 路径标记 | 能标记路线、水源、危险或资源点 |
| SymbolRecording | symbol_recording | 符号记录 | 能将知识写入外部记录，降低遗忘和传播损耗 |
| TeachingPractice | teaching_practice | 教学实践 | 能稳定把知识传授给族人 |
| DelegatedTeaching | delegated_teaching | 委托教学 | 已学会的族人能继续教别人 |
| TaskAssignment | task_assignment | 任务分配 | 能安排不同族人执行不同稳定任务 |
| RiskProtocol | risk_protocol | 风险规程 | 能形成固定避险流程，例如不吃未知物、远离危险地 |
| ResourcePlanning | resource_planning | 资源规划 | 能根据已知资源和季节进行储备或路线安排 |
| LongTermMemory | long_term_memory | 长期记忆 | 族群能跨时间保持关键知识 |

#### 使用示例

```text
族群掌握红浆果可食用：IdentifyEdible
族群知道毒蘑菇不可食用：IdentifyDanger / RiskProtocol
族群能稳定采集红浆果：SafeForaging
族群用石头砸开坚果：BasicToolUse
族群用树枝维持火堆：FireKeeping
族群用火驱赶狼：FireDefense
族群把肉烤熟降低中毒风险：Cooking
族群在水源路线留下记号：PathMarking
族群用石板记录危险区域：SymbolRecording
族群安排一部分人采集、一部分人守火：TaskAssignment
```

#### 设计注意事项

```text
CapabilityType 表示稳定能力，不表示单次成功。
能力通常由 KnowledgeStatus.Operational 或 KnowledgeScope.Tribe / Civilization 推动解锁。
CapabilityType 可以被规则系统用来判断某些行动是否可执行。
CapabilityType 不应该替代具体知识：会守火不等于知道所有燃料都可用。
```

### 9.4 UnlockConditionType（解锁条件类型）

`UnlockConditionType` 描述能力、阶段或规则为什么被解锁。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| KnowledgeEstablished | knowledge_established | 知识已确立 | 某条知识达到稳定或确立状态 |
| KnowledgeOperational | knowledge_operational | 知识可执行 | 某条知识已经转化为稳定行为 |
| TribeSharedKnowledge | tribe_shared_knowledge | 族群共享知识 | 知识范围达到族群层级 |
| RepeatedSuccess | repeated_success | 重复成功 | 多次行动结果一致且成功 |
| RequiredCapability | required_capability | 前置能力 | 已拥有必要前置能力 |
| RequiredStage | required_stage | 前置阶段 | 已达到必要文明阶段 |
| RequiredResource | required_resource | 必要资源 | 拥有或能稳定获取必要资源 |
| RequiredObjectKnowledge | required_object_knowledge | 必要对象知识 | 已理解某个对象的用途、风险或效果 |
| RequiredTraitKnowledge | required_trait_knowledge | 必要特性知识 | 已理解某个特性，例如可燃、有毒、可食用 |
| RequiredTool | required_tool | 必要工具 | 拥有或能制造必要工具 |
| RequiredPopulation | required_population | 必要人口 | 有足够族人执行稳定行为 |
| RequiredLocation | required_location | 必要地点 | 找到或控制必要地点 |
| RequiredEnvironment | required_environment | 必要环境 | 当前季节、天气、昼夜或生态条件满足要求 |
| SuccessfulTeaching | successful_teaching | 成功教学 | 知识成功传播给足够多接收者 |
| LongTermUsage | long_term_usage | 长期使用 | 某能力或知识经过长期稳定使用 |
| ExternalDiscovery | external_discovery | 外部发现 | 通过遗迹、记录、异常事件等获得解锁线索 |
| RuleGranted | rule_granted | 规则授予 | 由规则系统直接授予，通常用于调试、初始化或剧情节点 |

#### 使用示例

```text
红浆果可食用被多次验证：KnowledgeEstablished / RepeatedSuccess
族群稳定采集红浆果：KnowledgeOperational / TribeSharedKnowledge
守火需要先知道树枝可燃：RequiredTraitKnowledge
火焰防御需要先拥有 FireKeeping：RequiredCapability
庇护建造需要找到合适地点：RequiredLocation
符号记录需要族群拥有长期记忆和标记习惯：LongTermUsage / RequiredCapability
任务分配需要足够人口：RequiredPopulation
雨季洪水避难路线需要特定环境触发：RequiredEnvironment
遗迹图案提示储存食物方法：ExternalDiscovery
测试中直接开启某能力：RuleGranted
```

#### 设计注意事项

```text
UnlockConditionType 可以多选。
不要只用 RequiredStage 解锁能力，否则会变成空洞科技树。
优先使用具体知识、能力、资源、地点和重复成功作为解锁依据。
RuleGranted 只用于调试、初始化或强剧情，不应滥用。
```


## 10. 规则与事件相关枚举

规则与事件相关枚举用于支撑规则引擎执行流程、结果判定、错误分级、事件记录和状态变更。

在《先驱者》中，规则引擎必须做到：

```text
每个输入命令如何被解析。
每个规则步骤如何执行。
执行成功、失败或部分成功如何表达。
错误是否影响存档一致性。
世界状态具体发生了什么变化。
这些变化如何被前端、日志、复盘和测试读取。
```

### 10.1 规则与事件相关枚举清单

第一阶段确认包含以下五个规则与事件相关枚举：

| 枚举 | 中文名 | 作用 |
|---|---|---|
| RuleStepType | 规则步骤类型 | 描述规则管线当前执行到哪一步 |
| RuleResultType | 规则结果类型 | 描述规则执行后的结果 |
| ErrorSeverity | 错误严重级别 | 描述错误对系统和玩家流程的影响 |
| EventType | 事件类型 | 描述规则执行后产生了什么事件 |
| StateChangeType | 状态变化类型 | 描述世界状态如何变化 |

边界：

```text
RuleStepType 回答“规则执行到哪一步”。
RuleResultType 回答“规则执行结果是什么”。
ErrorSeverity 回答“这个错误有多严重”。
EventType 回答“发生了什么事件”。
StateChangeType 回答“状态具体怎么变了”。
```

### 10.2 RuleStepType（规则步骤类型）

`RuleStepType` 描述规则管线当前执行到哪一步。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| ReceiveCommand | receive_command | 接收命令 | 接收前端或系统提交的命令 |
| ParseCommand | parse_command | 解析命令 | 将外部命令解析为内部 Command |
| ValidateCommand | validate_command | 校验命令 | 检查 ID、参数、目标和状态是否合法 |
| BuildContext | build_context | 构建上下文 | 构建规则执行需要的上下文和快照 |
| CheckPreconditions | check_preconditions | 检查前置条件 | 检查行动是否满足成本、距离、状态、能力等条件 |
| CalculateCost | calculate_cost | 计算成本 | 计算时间、体力、资源、风险等成本 |
| ResolveAction | resolve_action | 结算行为 | 执行动作核心规则 |
| ResolveInteraction | resolve_interaction | 结算交互 | 结算对象、特性、环境之间的互动 |
| ResolveFeedback | resolve_feedback | 结算反馈 | 生成生理、环境、知识等反馈 |
| ResolveCognition | resolve_cognition | 结算认知 | 更新记忆、知识、置信度和认知状态 |
| ResolvePropagation | resolve_propagation | 结算传播 | 处理知识传播与族群接受结果 |
| ResolveCivilization | resolve_civilization | 结算文明 | 处理能力解锁、阶段变化和族群行为更新 |
| ApplyStateChanges | apply_state_changes | 应用状态变化 | 将变更写入世界状态 |
| EmitEvents | emit_events | 发出事件 | 生成事件流供前端、日志和复盘使用 |
| PersistState | persist_state | 持久化状态 | 保存状态、事件和随机记录 |
| Finish | finish | 完成 | 规则执行完成 |

#### 设计注意事项

```text
RuleStepType 主要用于调试、日志、测试和错误定位。
业务代码不应依赖字符串比较步骤名。
如果某一步失败，错误应记录 step，方便复盘。
新增规则管线步骤必须同步更新测试和文档。
```

### 10.3 RuleResultType（规则结果类型）

`RuleResultType` 描述一次规则执行后的结果。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Success | success | 成功 | 规则完整执行，产生预期结果 |
| PartialSuccess | partial_success | 部分成功 | 部分目标完成，部分失败或产生副作用 |
| NoEffect | no_effect | 无效果 | 行动合法，但没有产生实际变化 |
| Failed | failed | 失败 | 行动未达成目标，但状态仍保持一致 |
| Blocked | blocked | 被阻止 | 前置条件不满足，规则未进入核心结算 |
| InvalidCommand | invalid_command | 非法命令 | 命令格式、参数或目标不合法 |
| Conflict | conflict | 冲突 | 与当前状态、其他规则或已有知识冲突 |
| Interrupted | interrupted | 被中断 | 行动执行过程中被环境、风险或事件打断 |
| CriticalFailure | critical_failure | 严重失败 | 行动导致严重负面后果，但规则系统仍可继续 |
| SystemError | system_error | 系统错误 | 程序、配置或数据错误，需要工程处理 |

#### 使用示例

```text
吃红浆果恢复饥饿并获得知识：Success
吃未知蘑菇恢复少量饥饿但中毒：PartialSuccess / CriticalFailure
使用石头砸水面没有变化：NoEffect
目标太远无法触碰：Blocked
传入不存在的 ObjectId：InvalidCommand
族人已有相反知识导致传播冲突：Conflict
采集时被狼袭击打断：Interrupted
配置缺少对象特性定义：SystemError
```

#### 设计注意事项

```text
Failed 表示玩法失败，不表示系统坏了。
SystemError 表示工程或数据问题，必须进入错误日志。
Blocked 通常不消耗核心成本，Failed 可能消耗成本。
PartialSuccess 是核心玩法需要，探索行为经常同时带来收益和风险。
```

### 10.4 ErrorSeverity（错误严重级别）

`ErrorSeverity` 描述错误对系统和玩家流程的影响。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Info | info | 信息 | 非错误，仅用于记录流程信息 |
| Warning | warning | 警告 | 有异常或边界情况，但不影响继续执行 |
| Recoverable | recoverable | 可恢复错误 | 当前命令失败，但系统状态一致，可继续游戏 |
| UserError | user_error | 用户错误 | 玩家输入或前端请求不合法 |
| DataError | data_error | 数据错误 | 配置、存档或定义数据不合法 |
| RuleError | rule_error | 规则错误 | 规则组合或状态转移不合法 |
| Critical | critical | 严重错误 | 可能影响本次结算完整性，需要停止当前流程 |
| Fatal | fatal | 致命错误 | 状态可能不可信，需要阻止继续运行或回滚 |

#### 使用示例

```text
动作没有效果但合法：Info
重复传播已知知识：Warning
采集目标消失导致命令失败：Recoverable
前端传入错误枚举字符串：UserError
对象定义缺少必填特性：DataError
规则试图传播裸 ObjectId：RuleError
状态版本冲突导致无法应用变更：Critical
存档校验失败且无法恢复：Fatal
```

#### 设计注意事项

```text
ErrorSeverity 不是 UI 提示等级，而是系统处理等级。
UserError 不应污染规则状态。
DataError、RuleError、Critical、Fatal 必须进入工程日志。
Fatal 应阻止继续写入存档。
```

### 10.5 EventType（事件类型）

`EventType` 描述规则执行后产生了什么事件。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| CommandReceived | command_received | 命令已接收 | 系统收到玩家或系统命令 |
| ActionStarted | action_started | 行动开始 | 某个行动开始执行 |
| ActionResolved | action_resolved | 行动已结算 | 行动完成结算并产生结果 |
| ActionBlocked | action_blocked | 行动被阻止 | 行动因前置条件不满足被阻止 |
| ObjectDiscovered | object_discovered | 对象被发现 | 对象进入玩家或族群可感知范围 |
| ObjectStateChanged | object_state_changed | 对象状态变化 | 对象状态发生变化 |
| TraitDiscovered | trait_discovered | 特性被发现 | 对象的用途、风险或效果被发现 |
| FeedbackGenerated | feedback_generated | 反馈已生成 | 系统生成生理、环境、知识等反馈 |
| MemoryCreated | memory_created | 记忆已创建 | 个人或族群成员形成记忆 |
| MemoryUpdated | memory_updated | 记忆已更新 | 既有记忆内容、强度或可靠性变化 |
| KnowledgeCreated | knowledge_created | 知识已创建 | 形成新的 KnowledgeClaim |
| KnowledgeUpdated | knowledge_updated | 知识已更新 | 既有知识状态、范围或置信度变化 |
| KnowledgeContradicted | knowledge_contradicted | 知识被反证 | 新证据与既有知识冲突 |
| KnowledgeShared | knowledge_shared | 知识已共享 | 知识传播给其他主体 |
| PropagationFailed | propagation_failed | 传播失败 | 知识传播未达成有效结果 |
| CapabilityUnlocked | capability_unlocked | 能力已解锁 | 族群获得新的稳定能力 |
| CivilizationStageChanged | civilization_stage_changed | 文明阶段变化 | 族群文明阶段发生变化 |
| TribeBehaviorChanged | tribe_behavior_changed | 族群行为变化 | 族群稳定行为或策略发生变化 |
| ResourceChanged | resource_changed | 资源变化 | 食物、材料、工具耐久等资源变化 |
| EntityStateChanged | entity_state_changed | 实体状态变化 | 先驱者、族人或生物状态变化 |
| RegionChanged | region_changed | 区域变化 | 地点、区域环境或危险状态变化 |
| RandomRolled | random_rolled | 随机已记录 | 发生随机判定并记录随机流 |
| RuleErrorOccurred | rule_error_occurred | 规则错误发生 | 规则执行中出现错误 |
| StatePersisted | state_persisted | 状态已保存 | 状态和事件已持久化 |

#### 设计注意事项

```text
EventType 面向事件流，不直接代表状态变化字段。
一个 RuleResult 可以产生多个 Event。
前端只根据事件展示反馈，不应自己推导规则结果。
RandomRolled 必须记录随机流、种子位置和用途。
RuleErrorOccurred 应包含 ErrorSeverity 和 RuleStepType。
```

### 10.6 StateChangeType（状态变化类型）

`StateChangeType` 描述世界状态如何变化。

#### 枚举值

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| Create | create | 创建 | 创建新状态对象 |
| Update | update | 更新 | 修改已有状态对象 |
| Delete | delete | 删除 | 删除状态对象 |
| Move | move | 移动 | 对象、实体或队伍位置变化 |
| AddRelation | add_relation | 添加关系 | 添加对象、实体、区域或知识之间的关系 |
| RemoveRelation | remove_relation | 移除关系 | 移除已有关系 |
| AddFlag | add_flag | 添加标记 | 添加状态标记、知识标记或风险标记 |
| RemoveFlag | remove_flag | 移除标记 | 移除状态标记、知识标记或风险标记 |
| Increment | increment | 增加 | 数值增加 |
| Decrement | decrement | 减少 | 数值减少 |
| SetValue | set_value | 设置值 | 将字段设置为指定值 |
| Append | append | 追加 | 向列表、日志、记忆或事件流追加内容 |
| Replace | replace | 替换 | 用新对象或新结构替换旧内容 |
| Merge | merge | 合并 | 合并知识、记忆、关系或状态片段 |
| Split | split | 拆分 | 将状态对象拆分为多个对象 |
| Transform | transform | 转化 | 对象或状态转化为另一种形式 |
| Consume | consume | 消耗 | 消耗资源、材料、耐久、时间或机会 |
| Produce | produce | 产出 | 产生资源、反馈、知识、事件或能力 |
| Lock | lock | 锁定 | 临时或永久阻止某状态被使用 |
| Unlock | unlock | 解锁 | 解除锁定或开放新能力 |

#### 使用示例

```text
第一次形成 KnowledgeClaim：Create
提高红浆果可食用知识置信度：Update / Increment
删除临时实验记录：Delete
先驱者移动到水源旁：Move
红浆果生长在森林区域：AddRelation
移除错误对象关系：RemoveRelation
添加 KnowsEdible 标记：AddFlag
移除错误的 KnowsEdible 标记：RemoveFlag
饥饿值下降：Decrement
火堆燃料值上升：Increment
把 KnowledgeStatus 设置为 Operational：SetValue
追加一条 Memory：Append
合并两条相同知识：Merge
生肉变成熟肉：Transform
消耗树枝维持火堆：Consume
产生 FireKeeping 能力：Produce / Unlock
危险区域临时封锁：Lock
开放符号记录能力：Unlock
```

#### 设计注意事项

```text
StateChangeType 是状态层动作，不是玩家行为。
StateChange 必须能被序列化、回放和测试断言。
所有状态变更都应关联 EventId、CommandId、StateVersion。
禁止前端直接修改状态，前端只能提交命令并读取事件/状态结果。
```

### 10.7 规则与事件枚举组合测试样例

以下样例用于检查五个规则与事件相关枚举是否能表达规则引擎核心流程。

| 场景 | RuleStepType | RuleResultType | ErrorSeverity | EventType | StateChangeType |
|---|---|---|---|---|---|
| 吃红浆果恢复饥饿并形成知识 | ResolveAction / ResolveCognition | Success | Info | ActionResolved / FeedbackGenerated / KnowledgeCreated | Decrement / Create |
| 吃毒蘑菇导致中毒并形成警告记忆 | ResolveAction / ResolveFeedback / ResolveCognition | CriticalFailure | Warning | ActionResolved / FeedbackGenerated / MemoryCreated | AddFlag / Create |
| 触碰火焰被灼伤 | ResolveInteraction / ResolveFeedback | CriticalFailure | Warning | ActionResolved / FeedbackGenerated / EntityStateChanged | Decrement / AddFlag |
| 目标太远无法采集 | CheckPreconditions | Blocked | UserError | ActionBlocked | Append |
| 前端传入不存在的 ObjectId | ValidateCommand | InvalidCommand | UserError | RuleErrorOccurred | Append |
| 规则试图传播裸 ObjectId | ResolvePropagation | SystemError | RuleError | RuleErrorOccurred | Append |
| 族人接受红浆果可食用知识 | ResolvePropagation / ResolveCognition | Success | Info | KnowledgeShared / MemoryCreated | Create / AddFlag |
| 族人误解为所有红色果实都能吃 | ResolvePropagation / ResolveCognition | PartialSuccess | Warning | KnowledgeShared / KnowledgeUpdated | Create / Update |
| 族群解锁守火能力 | ResolveCivilization | Success | Info | CapabilityUnlocked / TribeBehaviorChanged | Unlock / Produce |
| 火堆燃料消耗并继续燃烧 | ResolveInteraction / ApplyStateChanges | Success | Info | ObjectStateChanged / ResourceChanged | Consume / Decrement |
