# P48 制作与世界反应风险头脑风暴

更新时间：2026-05-21
阶段：P48

## 1. 目标

提前识别 P48 可能把系统做坏的地方，避免制作系统变成新的硬编码 demo。

## 2. 核心风险

### 2.1 绕过 P45 直接改背包

风险：

```text
CraftService 为了方便直接 inv.entries.erase 或 push_back。
```

后果：

```text
item_locations 不同步。
WorldEntityInstance 位置不一致。
后续 NPC、地图投影、存档都会错。
```

防线：

```text
必须通过 InventoryTransferKind::ConsumeToNowhere / SpawnToInventory。
如果 P45 没实现，就先实现 P45 transfer，不允许绕过。
```

### 2.2 把 recipe 写死在 C++

风险：

```text
if reaction_key == "wood_plus_fire_make_torch" 就生成 torch。
```

后果：

```text
以后毒蘑菇汤、金属工具、魔法药水、机械零件都要改 C++。
```

防线：

```text
服务只读 ReactionDefinitionContent.inputs / consume / outputs。
测试可以用 torch，但主流程不能出现 torch 特判。
```

### 2.3 输入和消耗混淆

风险：

```text
所有 inputs 都被消耗。
```

后果：

```text
camp_fire 作为火源也被消耗。
水井、工作台、熔炉、魔法阵都会被错误消耗。
```

防线：

```text
inputs 是参与条件。
consume 才是消耗清单。
```

### 2.4 前端传结果

风险：

```text
前端发 command 时带 outputs=torch。
```

后果：

```text
玩家可以伪造物品。
规则不再权威。
```

防线：

```text
P48 command 只能传 reaction_key 和参数。
outputs 必须来自 ContentRegistry。
```

### 2.5 自动匹配配方导致歧义

风险：

```text
玩家点“制作火把”，系统根据背包自动选一个反应。
```

后果：

```text
多个火把配方时不可预测。
replay 不稳定。
AI 规划难以解释。
```

防线：

```text
P48 必须指定 reaction_key。
后续 available_commands 可以生成明确选项。
```

### 2.6 状态材料无法区分

风险：

```text
processed wood 和 raw wood 都被当作 wood。
```

后果：

```text
斧头砍出来的粗木头、加工木棍、腐烂材料都混用。
```

防线：

```text
input.state 和 consume.state 必须匹配 state_keys。
```

### 2.7 数量不足仍成功

风险：

```text
只检查存在，不检查 quantity。
```

后果：

```text
1 个木头可以无限造火把。
```

防线：

```text
required_quantity <= available_quantity。
consume abs(delta) <= proof.available_quantity。
```

### 2.8 OnMap 材料消耗不稳定

风险：

```text
同一个格子多个 wood stack，系统随机消耗一个。
```

后果：

```text
复盘不稳定。
多人 / NPC 同时行动时难调试。
```

防线：

```text
搜索顺序按 source_kind、cell_id、entity_id 稳定排序。
P48 MVP 可拒绝需要跨多个 stack 合并的输入。
```

### 2.9 apply 中途失败部分提交

风险：

```text
先扣木头，生成火把失败。
```

后果：

```text
材料消失，玩家没拿到产物。
```

防线：

```text
validate 阶段尽量拦截所有常见错误。
apply 前复验。
代码注释记录事务债务。
后续持久化阶段引入 transaction / compensation。
```

### 2.10 Craft 直接形成知识

风险：

```text
制作成功后直接写 KnowledgeRepository。
```

后果：

```text
绕过 P49 的认知、记忆、信心、教学链。
```

防线：

```text
P48 只输出 WorldExperienceDto。
P49 再消费 experience 形成知识。
```

## 3. 复杂场景压力想象

### 3.1 高级文明

场景：

```text
矿石 + 熔炉 + 能源 -> 合金锭
合金锭 + 芯片 + 工具台 -> 无人机
```

P48 是否够用：

```text
够用。熔炉 / 工具台作为 input 不在 consume 中，矿石 / 芯片在 consume 中。
```

### 3.2 魔法世界

场景：

```text
草药 + 月光区域 + 魔法阵 -> 治疗药水
```

P48 MVP 不完全够用：

```text
月光区域 / 魔法阵条件需要 P52+ 区域事件和条件表达式增强。
但 P48 的 input proof / condition 扩展点能承接。
```

### 3.3 机械危机

场景：

```text
废铁 + 电池 + 损坏机器人 -> 修复机器人
```

P48 是否够用：

```text
基本够用。损坏机器人可以是 target input，输出可 TransformObject。
但 P48 MVP 只做 ProduceObject / ConsumeObject，TransformObject 留后续。
```

### 3.4 建筑施工

场景：

```text
木头 x20 + 石头 x10 + 施工地点 -> 小屋地基
```

P48 MVP 不完全够用：

```text
多 stack 合并、大数量、多阶段进度需要后续 Construction 模块。
P48 只打通材料消耗和产物归属。
```

### 3.5 烹饪系统

场景：

```text
蘑菇 + 水 + 锅 + 火 -> 蘑菇汤
毒蘑菇 + 水 + 锅 + 火 -> 毒汤
```

P48 是否够用：

```text
够用。不同 object_key 对应不同 reaction。
锅和火不消耗，材料消耗。
```

### 3.6 工具耐久

场景：

```text
斧头 + 木头 -> 木材，同时斧头耐久 -1
```

P48 MVP 不完全处理：

```text
可以通过 consume / ResourceDelta 扩展，但 MVP 不做工具耐久变化。
P35/P36 的状态对象规则后续接入。
```

### 3.7 生物驯服

场景：

```text
野兽 + 食物 + 信任状态 -> 伙伴 / 坐骑
```

P48 MVP 不直接做：

```text
这是 TransformObject / Agent relation change。
P48 的 reaction 输出类型未来可扩展，但 MVP 只做 ProduceObject。
```

### 3.8 化学反应

场景：

```text
酸 + 金属 -> 气体 + 腐蚀状态
```

P48 MVP 部分够用：

```text
ProduceObject 可生成气体对象。
AddState / ResourceDelta 留后续。
```

### 3.9 农业加工

场景：

```text
小麦 + 磨坊 -> 面粉
面粉 + 水 + 火 -> 面包
```

P48 是否够用：

```text
够用。磨坊作为不消耗 input，材料按 consume 扣减，产物进入背包或地图。
```

### 3.10 大世界多人 / 多 NPC 同时制作

场景：

```text
两个 NPC 同时消耗同一堆木头。
```

P48 MVP 风险：

```text
当前无事务锁。validate 后 apply 前可能冲突。
```

防线：

```text
apply 前复验材料位置、数量、状态。
冲突返回 RuntimeConflict / QuantityInsufficient。
```

## 4. P48 必须留下的扩展口

```text
TransformObject：对象变形，例如生肉 -> 熟肉。
AddState / RemoveState：对象状态变化，例如斧头变钝。
ResourceDelta：耐久、燃料、温度变化。
AreaCondition：区域魔法、天气、地形条件。
MultiStackAllocation：多个 stack 合并消耗。
CraftStation：制作台、熔炉、工作台。
ProgressiveCraft：多阶段工程和建筑进度。
Transaction：事务 / 补偿。
```

## 5. 本阶段最重要的底线

P48 可以小，但不能假。

必须真实做到：

```text
真实读取 JSON reaction。
真实校验运行态材料。
真实消耗材料。
真实生成产物。
真实更新归属。
真实返回 command 结果。
真实给 P49 留 experience。
```
