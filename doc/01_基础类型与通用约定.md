# 先驱者：基础类型与通用约定

## 1. 设计目标

本文档用于逐步确认《先驱者》规则引擎的基础类型约定。

当前只确认一个设计点：

```text
C++ 规则引擎内部使用强类型 ID。
外部传输、配置和存储层使用字符串。
```

其余基础类型暂不写入正式结论，后续逐点讨论、逐点确认、逐点补充。

## 2. 已确认设计点

### 2.1 C++ 内部使用强类型 ID

规则引擎内部不直接混用 `std::string` 表示所有 ID，而是为不同语义的 ID 建立不同类型。

示例：

```cpp
struct ObjectId {
    std::string value;
};

struct ObjectDefinitionId {
    std::string value;
};
```

不采用以下方式作为规则引擎内部主表达：

```cpp
using ObjectId = std::string;
using ObjectDefinitionId = std::string;
```

原因：

```text
ObjectId 表示运行时对象实例。
ObjectDefinitionId 表示配置层对象类型。
二者都是字符串，但语义完全不同。
```

如果在 C++ 内部都用裸 `std::string`，后续很容易把实例 ID 传给需要定义 ID 的接口，编译器无法发现问题。

| DecisionId | 决策 ID | 实例 ID | decision_000001 | 预留给 AI 决策追踪，表示一次策略决策 |

### 2.3 ID 新增注意事项

研发不能因为临时方便自行新增 ID 类型。新增 ID 类型必须先更新设计文档，并说明新增理由。

新增 ID 前必须回答以下问题：

```text
1. 这个 ID 表示运行时实例，还是配置定义？
2. 现有 ID 类型是否已经能表达它？
3. 它是否会进入存档？
4. 它是否会发送给前端？
5. 它是否会被配置引用？
6. 它是否参与复盘？
7. 它和已有 ID 的边界是什么？
```

第一阶段明确暂不新增以下 ID：

```text
ItemId（物品 ID）
SkillId（技能 ID）
QuestId（任务 ID）
BuildingId（建筑 ID）
ProfessionId（职业 ID）
```

暂不新增原因：

```text
这些系统容易把当前设计从“认知规则引擎”拉向传统 RPG、生存或文明模拟。
第一阶段应先保证“对象 → 行为 → 反馈 → 记忆 → 知识 → 传播 → 文明能力”这条链路成立。
```

如果未来确实需要新增 ID，必须遵守：

```text
先讨论 → 邮件确认 → 更新程序设计计划 → 更新基础类型文档 → 再进入代码实现
```



### 2.4 实例 ID 与定义 ID 分层

《先驱者》规则引擎中，ID 必须明确分为两层：

```text
定义 ID：配置层定义的类型标识。
实例 ID：运行时创建的具体对象、事件、命令或记录。
```

#### 定义 ID

定义 ID 也可以理解为类型 ID 或配置 ID。

特点：

```text
稳定
可读
可写入配置
可跨存档存在
通常不会在运行时自动生成
```

第一阶段的定义 ID 包括：

```text
ObjectDefinitionId（对象类型定义 ID）
TraitId（特性 ID）
ActionId（行为 ID）
CapabilityId（文明能力 ID）
RegionDefinitionId（区域类型定义 ID）
EffectId（效果定义 ID）
ExpressionId（表达式定义 ID）
ConfigPackId（配置包 ID）
ConfigVersionId（配置版本 ID）
```

#### 实例 ID

实例 ID 也可以理解为运行时 ID。

特点：

```text
唯一
不可复用
不承载玩法含义
可以进入存档或复盘
通常由运行时系统生成
```

第一阶段的实例 ID 包括：

```text
GameId（游戏实例 ID）
PlayerId（玩家 ID）
TribeId（族群 ID）
EntityId（实体 ID）
ObjectId（对象实例 ID）
RegionId（区域实例 ID）
CommandId（命令 ID）
EventId（事件 ID）
MemoryId（记忆 ID）
KnowledgeId（知识 ID）
StateChangeId（状态变更 ID）
EpisodeId（训练局 ID，强化学习预留）
DecisionId（决策 ID，强化学习预留）
```

#### 分层使用规则

玩家命令引用实例 ID：

```text
ActionCommand.targetObjectIds = [obj_000001]
```

配置表引用定义 ID：

```text
red_berry.traits = [edible, plant]
```

认知和知识优先绑定定义 ID：

```text
red_berry.edible confidence = 80
```

事件尽量同时携带实例 ID 和定义 ID：

```text
objectId = obj_000001
definitionId = red_berry
```

原因：

```text
前端表现需要实例 ID。
日志、认知、复盘和规则分析需要定义 ID。
```

#### 暂不设计特殊个体认知

第一阶段暂不设计针对特殊个体的认知，例如：

```text
这只独特的狼
唯一遗迹
特殊神器
```

### 2.5 跨文档桥接类型补充

随着 `03_错误码与结果类型设计.md`、`04_事件系统设计.md` 和 `05_配置系统设计.md` 完成，第一阶段补充以下桥接类型。

这些类型不是新玩法系统，而是为了让错误、事件、配置、存档、复盘和未来强化学习互相可用。

| 类型 | 中文名 | 来源文档 | 作用 |
|---|---|---|---|
| EffectId | 效果定义 ID | 配置系统 | 让 EffectDefinition 可被交互规则引用 |
| ExpressionId | 表达式定义 ID | 配置系统 | 让受限表达式可被对象、特性、行为、效果、能力引用 |
| StateChangeId | 状态变更 ID | 错误 / 事件系统 | 让 EventRecord 和 StateApplyResult 能追踪具体状态变更 |
| ConfigPackId | 配置包 ID | 配置系统 | 标识一组配置包 |
| ConfigVersionId | 配置版本 ID | 配置系统 | 让存档、事件、测试和训练数据绑定配置版本 |
| EpisodeId | 训练局 ID | 事件系统强化学习预留 | 标记事件属于哪一局训练或模拟 |
| DecisionId | 决策 ID | 事件系统强化学习预留 | 标记事件来自哪次 AI 决策 |

使用注意事项：

```text
EffectId 和 ExpressionId 属于配置定义 ID。
StateChangeId 属于运行时实例 ID。
ConfigVersionId 虽然表现为版本字符串，但在规则引擎内应强类型化，避免和普通 string 混用。
EpisodeId 和 DecisionId 只是预留，不要求第一阶段业务逻辑使用。
```



第一阶段默认认知沉淀到对象类型，而不是单个对象实例。



### 2.5 ID 命名规则

ID 命名必须让研发、配置、日志、存档和调试工具都能稳定识别其含义。

#### 定义 ID 命名规则

定义 ID 使用可读语义名，格式为：

```text
lower_snake_case（小写蛇形命名）
```

示例：

```text
red_berry
poison_mushroom
forest_edge
edible
flammable
observe
touch
safe_berry_gathering
```

规则：

```text
只能使用小写英文字母、数字、下划线
必须以小写英文字母开头
不允许中文
不允许空格
不允许大写字母
不允许连字符 -
不允许使用运行时编号
名称必须表达语义，不能写成 a1、thing_01、test_obj
```

定义 ID 会出现在配置、日志、存档、测试和文档中，因此必须稳定且人类可读。

#### 实例 ID 命名规则

实例 ID 使用固定前缀加唯一编号：

```text
固定前缀 + _ + 唯一编号
```

示例：

```text
game_000001
player_000001
tribe_000001
ent_000001
obj_000001
region_000001
cmd_000001
evt_000001
mem_000001
know_000001
```

第一阶段实例 ID 前缀：

| 类型 | 中文名 | 前缀 | 示例 |
|---|---|---|---|
| GameId | 游戏实例 ID | game | game_000001 |
| PlayerId | 玩家 ID | player | player_000001 |
| TribeId | 族群 ID | tribe | tribe_000001 |
| EntityId | 实体 ID | ent | ent_000001 |
| ObjectId | 对象实例 ID | obj | obj_000001 |
| RegionId | 区域实例 ID | region | region_000001 |
| CommandId | 命令 ID | cmd | cmd_000001 |
| EventId | 事件 ID | evt | evt_000001 |
| MemoryId | 记忆 ID | mem | mem_000001 |
| KnowledgeId | 知识 ID | know | know_000001 |

实例 ID 规则：

```text
必须唯一
不允许复用
不承载玩法语义
不能使用 red_berry_01 这种带玩法名的格式
可以使用递增编号，也可以未来改为 UUID，但对外仍建议保留前缀
```

对象真实类型应由 `ObjectDefinitionId` 表达，不能塞进 `ObjectId`。位置、状态、阵营、稀有度等信息也必须作为字段表达，不能塞进 ID。

#### 非法示例

非法定义 ID：

```text
红浆果
RedBerry
red-berry
red berry
object_001
test
a1
```

非法实例 ID：

```text
red_berry_001
wolf_near_camp
obj red 1
OBJ_001
```

### 2.6 定义 ID 中文翻译器

定义 ID 不允许直接使用中文，但必须提供中文翻译能力，方便策划、研发、日志分析和调试工具理解。

该能力暂命名为：

```text
DefinitionIdTranslator（定义 ID 翻译器）
```

职责：

```text
把定义 ID 翻译成人类可读中文名
为调试日志提供中文辅助显示
为编辑器或配置检查工具提供中文说明
为前端调试面板提供可读文本
```

示例：

```text
red_berry -> 红浆果
poison_mushroom -> 毒蘑菇
edible -> 可食用
flammable -> 可燃
observe -> 观察
safe_berry_gathering -> 安全采集红浆果
```

使用规则：

```text
规则引擎内部仍然只使用定义 ID
中文翻译只用于显示、调试、编辑器、日志和文档辅助
中文翻译不能作为规则判断依据
中文翻译缺失不能改变规则结果
```

也就是说：

```text
规则判断使用 red_berry
显示说明可以使用 红浆果
```

推荐配置形式：

```text
id: red_berry
zhName: 红浆果
description: 一种可食用的红色浆果
```

注意事项：

```text
同一个定义 ID 可以有中文名、英文名、描述文本等多语言字段
第一阶段只要求中文翻译
翻译器不是游戏规则系统，只是显示辅助系统
```



### 2.7 ID 基础能力

所有强类型 ID 在 C++ 规则引擎内部都必须支持一组统一的基础能力。

#### 必须支持的能力

```text
构造：从字符串构造强类型 ID
显式 toString：读取原始字符串，用于日志、序列化、调试和传输
相等比较：判断两个 ID 是否表示同一目标
不等比较：用于条件判断
哈希：支持 unordered_map / unordered_set
empty 判断：判断是否为空 ID
格式校验：区分合法 ID 和非法 ID
字符串序列化：写入 JSON、存档和日志
字符串反序列化：从 JSON、配置和存档读取
```

示例形态：

```cpp
struct ObjectId {
    std::string value;

    explicit ObjectId(std::string value);
    bool empty() const;
    std::string toString() const;

    friend bool operator==(const ObjectId& a, const ObjectId& b);
    friend bool operator!=(const ObjectId& a, const ObjectId& b);
};
```

哈希能力通过 `std::hash<ObjectId>` 特化或项目统一哈希工具提供。

#### 第一阶段禁止的能力

第一阶段不允许 ID 支持以下能力：

```text
ID 大小排序
ID 数学运算
从 ID 推断玩法信息
隐式转换为 string
```

不允许排序的原因：

```text
ID 不应该表达玩法顺序。
如果需要排序，应由创建时间、配置顺序、显示优先级等字段决定。
```

不允许数学运算的原因：

```text
ID 不是数字资源，不能做 objectId + 1 这类操作。
```

不允许从 ID 推断玩法信息的原因：

```text
obj_000001 不能推断出它是红浆果。
对象真实类型必须来自 ObjectDefinitionId 或对象字段。
```

不允许隐式转换为 `std::string` 的原因：

```text
隐式转换会削弱强类型 ID 的意义，容易重新引入 ID 混用问题。
需要字符串时必须显式调用 toString。
```

#### 序列化约定

规则引擎内部使用强类型 ID，但序列化结果必须是字符串。

示例：

```json
{
  "objectId": "obj_000001",
  "definitionId": "red_berry"
}
```

反序列化时必须重新构造对应强类型 ID，并进行格式校验。



### 2.8 ID 模板父类设计

ID 基础功能采用模板父类复用，但不使用运行时多态父类。

确认采用：

```text
IdBase<Derived> 模板父类 + 具体 ID 子类
```

#### 设计目标

该设计同时满足：

```text
父类复用基础功能
子类表达具体 ID 类型
不同 ID 类型不能混用
不引入虚函数
不通过父类指针保存 ID
```

#### 推荐形态

```cpp
template <typename Derived>
class IdBase {
public:
    explicit IdBase(std::string value);

    bool empty() const;
    std::string toString() const;

protected:
    std::string value_;
};

class ObjectId : public IdBase<ObjectId> {
public:
    using IdBase<ObjectId>::IdBase;
};

class ObjectDefinitionId : public IdBase<ObjectDefinitionId> {
public:
    using IdBase<ObjectDefinitionId>::IdBase;
};
```

#### 使用约束

`IdBase` 只用于代码复用，不能作为业务接口参数类型。

正确：

```cpp
findObject(ObjectId objectId);
findDefinition(ObjectDefinitionId definitionId);
```

错误：

```cpp
findSomething(IdBase id);
findSomething(const IdBase& id);
```

原因：

```text
如果业务接口接收 IdBase，就会重新允许所有 ID 混用，破坏强类型隔离。
```

#### 禁止规则

第一阶段禁止：

```text
禁止给 IdBase 添加虚函数
禁止通过 IdBase 指针保存 ID
禁止通过 IdBase 引用传递业务 ID
禁止把 ID 当成多态对象使用
```

ID 是轻量值对象，不是需要运行时多态的复杂对象。

#### 与 StrongId<Tag> 方案的关系

此前讨论过 `StrongId<Tag>` 方案。最终确认采用更贴近直觉的：

```text
IdBase<Derived> + 具体 ID 子类
```

原因：

```text
更符合“父类实现基础功能，子类扩展”的理解方式
研发阅读门槛较低
每个 ID 类型仍然是明确的 C++ 类型
后续每个子类可以扩展自己的校验规则
```



## 3. 时间、Tick 与状态版本

规则引擎必须明确区分游戏内时间、规则执行步数和状态版本。

### 3.1 三种时间相关概念

第一阶段确认使用三种概念：

```text
GameTime（游戏时间）：游戏世界里的时间。
Tick（规则步 / 逻辑步）：规则引擎执行到第几步。
StateVersion（状态版本号）：权威状态成功变化了多少次。
```

三者职责不同，不能混用。

### 3.2 GameTime（游戏时间）

`GameTime` 表示游戏世界内部的时间。

作用：

```text
影响探索
影响环境变化
影响族群状态
影响事件叙事
影响昼夜相关规则
```

第一阶段 `GameTime` 只包含：

```text
day：第几天
phase：日内阶段
```

`DayPhase（日内阶段）` 第一阶段包含：

| 枚举 | 中文 | 含义 |
|---|---|---|
| Dawn | 黎明 | 夜晚结束、白天开始前的阶段 |
| Day | 白天 | 主要探索和采集阶段 |
| Dusk | 黄昏 | 白天结束、夜晚开始前的阶段 |
| Night | 夜晚 | 危险更高、视野更差的阶段 |

第一阶段暂不设计：

```text
真实时间戳
毫秒级时间
复杂季节
月份历法
时区
物理帧率
```

### 3.3 Tick（规则步 / 逻辑步）

`Tick` 表示规则引擎执行到第几步。

建议类型：

```cpp
uint64_t
```

规则：

```text
每执行一次 ActionCommand，Tick 至少递增 1
Tick 单调递增
Tick 不回退
Tick 参与复盘
事件必须记录发生时的 Tick
```

用途：

```text
复盘
调试
事件排序
随机数稳定
规则执行顺序校验
```

### 3.4 StateVersion（状态版本号）

`StateVersion` 表示权威 `GameState` 成功变化了多少次。

建议类型：

```cpp
uint64_t
```

规则：

```text
只有权威 GameState 成功变化时才递增
被拒绝的命令不一定递增 StateVersion
StateVersion 单调递增
前端同步时用它判断状态新旧
存档必须记录 StateVersion
```

用途：

```text
前后端同步
存档一致性检查
调试
并发保护
复盘校验
```

### 3.5 三者区别

```text
GameTime：游戏世界里“现在是什么时候”。
Tick：规则引擎“执行到第几步”。
StateVersion：权威状态“变更了多少次”。
```

示例：玩家观察红浆果。

```text
Tick + 1
如果产生记忆或状态变化，StateVersion + 1
GameTime 可能只推进少量时间，或者不推进到新阶段
```

示例：玩家休息一夜。

```text
Tick + 1
StateVersion + 1
GameTime 从 day 1 / Night 推进到 day 2 / Dawn
```

### 3.6 不依赖系统真实时间

规则引擎不能依赖系统当前时间。

错误：

```text
使用 now() 判断是否夜晚。
```

正确：

```text
使用 GameState.gameTime.phase 判断是否夜晚。
```

原因：

```text
规则结果必须可复盘。
如果依赖系统真实时间，同一个命令序列在不同时间执行可能得到不同结果。
```

### 3.7 事件记录约定

每个 `GameEvent` 建议记录：

```text
eventId
tick
stateVersionAfter
gameTime
```

这样前端、日志和复盘系统都能知道事件发生顺序和发生时的世界时间。



## 4. 数值范围约定

规则引擎中的常用数值必须有统一范围，避免各系统自行定义导致混乱。

第一阶段不追求复杂数值体系，只保留少量稳定数值类型。

### 4.1 Percent（百分比）

`Percent` 表示 0 到 100 的整数百分比。

建议类型：

```cpp
int32_t
```

范围：

```text
0 - 100
```

用途：

```text
置信度
传播程度
揭示程度
健康百分比显示
风险提示强度
```

示例：

```text
confidence = 80 表示 80% 置信度
spreadLevel = 60 表示族群中约 60% 掌握该知识
```

规则：

```text
小于 0 视为非法
大于 100 视为非法或需要 clamp，具体由接口决定
规则层内部不要使用 0.8 表示 80%
```

### 4.2 Ratio（比例）

`Ratio` 表示 0.0 到 1.0 的计算比例。

建议类型：

```cpp
double
```

范围：

```text
0.0 - 1.0
```

用途：

```text
内部计算
概率计算
权重归一化
```

示例：

```text
chance = 0.25 表示 25% 概率
```

规则：

```text
Ratio 只用于计算层
对外展示和配置优先使用 Percent
不能在同一个字段里混用 Percent 和 Ratio
```

### 4.3 BoundedValue（有上下限数值）

`BoundedValue` 表示有最小值、最大值和当前值的状态数值。

用途：

```text
生命值
饥饿值
体力值
族群食物
士气
安全度
```

推荐结构：

```text
current：当前值
min：最小值
max：最大值
```

示例：

```text
health: current = 80, min = 0, max = 100
stamina: current = 35, min = 0, max = 50
```

规则：

```text
current 不能小于 min
current 不能大于 max
max 必须大于等于 min
修改时必须明确是否 clamp
```

### 4.4 Count（数量）

`Count` 表示离散数量。

建议类型：

```text
int64_t 或 uint64_t，按场景决定
```

用途：

```text
人口
物品数量
资源数量
证据数量
尝试次数
```

规则：

```text
不允许负数的数量使用 uint64_t 或显式校验
可能出现扣减到负数风险的场景，建议用 int64_t 并在规则层校验
不能用 float 表示离散数量
```

### 4.5 Score（评分 / 等级值）

`Score` 表示等级化评分。

建议类型：

```cpp
int32_t
```

第一阶段范围：

```text
0 - 10
```

用途：

```text
危险等级
区域难度
认知价值
行为风险等级
```

示例：

```text
dangerLevel = 0 表示无危险
dangerLevel = 10 表示极高危险
```

### 4.6 Delta（变化量）

`Delta` 表示一次规则执行造成的变化量。

建议类型：

```text
int32_t 或 int64_t
```

用途：

```text
生命变化
饥饿变化
置信度变化
资源变化
```

示例：

```text
healthDelta = -10
confidenceDelta = +20
```

规则：

```text
Delta 可以为负
Delta 应用后是否 clamp 由目标数值类型决定
事件中应记录 from、to、delta，方便前端表现和调试
```

### 4.7 Probability（概率）

第一阶段不单独设计 `Probability` 类型，概率使用 `Ratio` 表示。

范围：

```text
0.0 - 1.0
```

规则：

```text
配置可写 25%，但加载后转换为 0.25
随机判断必须使用可复盘随机数，不使用系统随机
```

### 4.8 数值注意事项

第一阶段暂不设计：

```text
大数系统
复杂公式语言
浮点型生命值
百分比和比例混用
没有上下限的核心状态数值
由 UI 自行 clamp 数值
```

研发注意事项：

```text
UI 不负责修正规则数值，规则层必须保证数值合法
核心状态数值必须有边界
事件中涉及数值变化时，应尽量记录 from、to、delta
配置中的百分比和规则内部比例必须在加载阶段明确转换
不能在同一个字段中有时写 80、有时写 0.8
数值越界时，接口必须明确是报错、拒绝命令，还是 clamp
```



## 5. 随机数种子与可复盘随机

规则引擎必须使用可控随机数，保证规则结果可复盘、可调试、可测试。

### 5.1 RandomSeed（随机种子）

`RandomSeed` 表示随机种子，用于决定一局游戏或一次规则流程的随机序列。

建议类型：

```cpp
uint64_t
```

示例：

```text
randomSeed = 123456789
```

规则：

```text
GameState 或 RuleContext 必须持有 RandomSeed
存档必须保存 RandomSeed
复盘必须使用相同 RandomSeed
规则引擎不能直接调用 std::random_device 参与规则结果
```

### 5.2 RandomStream（随机流）

`RandomStream` 表示从随机种子派生出的稳定随机序列。

使用规则：

```text
RandomStream 由 RuleContext 提供
所有规则系统如果需要随机，都必须从 RuleContext.random 取数
各系统不能私自创建随机数生成器
```

禁止：

```text
InteractionResolver 自己创建随机数生成器
CognitionResolver 使用系统时间做随机
前端决定随机规则结果
```

### 5.3 随机调用记录

每次影响规则结果的随机调用，建议记录：

```text
tick
commandId
randomPurpose（随机用途）
rolledValue（随机结果）
range（随机范围）
```

示例：

```text
randomPurpose = poison_check
range = 0.0 - 1.0
rolledValue = 0.32
```

这样调试时可以知道为什么某次规则结果发生，例如玩家为什么中毒或没有中毒。

### 5.4 RandomPurpose（随机用途）

`RandomPurpose` 用于标记随机调用的用途。

第一阶段可选用途：

| 枚举 | 中文 | 用途 |
|---|---|---|
| ExploreDiscovery | 探索发现 | 决定探索时发现什么 |
| InteractionOutcome | 交互结果 | 决定一次交互的随机结果 |
| RiskCheck | 风险判定 | 判断中毒、受伤、失败等风险是否发生 |
| ResourceYield | 资源产出 | 决定采集、掉落、收获数量 |
| PropagationEffect | 传播效果 | 决定知识传播效果 |
| EventTrigger | 事件触发 | 决定随机事件是否触发 |

### 5.5 随机与上下文绑定

随机结果应与以下上下文绑定：

```text
randomSeed
tick
commandId
randomPurpose
调用顺序
```

目标是保证：

```text
同一个状态
同一个命令
同一个随机种子
同一个规则流程
= 同一个随机结果
```

### 5.6 配置概率

配置中可以使用百分比表达概率：

```text
poisonChancePercent = 25
```

加载或规则执行时转换为 `Ratio`：

```text
poisonChance = 0.25
```

随机判断使用 `Ratio`：

```text
roll <= poisonChance
```

### 5.7 随机禁止事项

第一阶段禁止：

```text
规则层直接使用系统当前时间作为随机来源
规则层直接使用不可复盘随机
各系统私自创建随机数生成器
前端决定随机规则结果
随机结果不记录用途
```



## 6. 字符串与枚举表示约定

规则引擎、配置、存档和前端协议必须统一枚举表示方式。

### 6.1 C++ 内部枚举

C++ 内部使用 `enum class` 表示枚举。

示例：

```cpp
enum class DayPhase {
    Dawn,
    Day,
    Dusk,
    Night
};
```

原因：

```text
类型安全
不容易和其他枚举混用
switch 处理清晰
编译期能发现一部分错误
```

### 6.2 外部枚举表示

配置、JSON、日志和存档中使用稳定英文字符串表示枚举。

正确：

```json
{
  "phase": "night"
}
```

错误：

```json
{
  "phase": 3
}
```

原因：

```text
字符串更可读
配置更容易检查
日志更容易理解
枚举顺序改变不会破坏存档
前端不需要知道 C++ 枚举整数值
```

### 6.3 枚举字符串命名规则

枚举字符串使用：

```text
lower_snake_case（小写蛇形命名）
```

示例：

```text
dawn
day
dusk
night
observe
touch
eat
risk_check
resource_yield
```

规则：

```text
只使用小写英文、数字、下划线
不使用中文
不使用空格
不使用大写
不使用整数作为外部表达
```

### 6.4 枚举中文翻译

枚举需要中文翻译能力。

示例：

```text
day_phase.night -> 夜晚
action_type.observe -> 观察
random_purpose.risk_check -> 风险判定
```

规则：

```text
中文翻译只用于显示、调试、编辑器和日志辅助
中文翻译不参与规则判断
规则判断只使用 enum class 或稳定字符串
```

### 6.5 枚举转换器

每个枚举都应提供转换能力：

```text
toString(enum) -> string
fromString(string) -> Result<enum>
toChinese(enum) -> string
```

示例：

```text
toString(DayPhase::Night) -> "night"
fromString("night") -> DayPhase::Night
toChinese(DayPhase::Night) -> "夜晚"
```

### 6.6 未知枚举处理

如果从配置、存档或前端收到未知枚举字符串，不能静默默认。

错误：

```text
unknown_phase 自动当成 day
```

正确：

```text
返回配置错误或协议错误
```

原因：

```text
静默默认会隐藏配置错误，导致规则结果异常。
```

### 6.7 枚举值新增注意事项

新增枚举值时必须同时更新：

```text
C++ enum class
字符串转换器
中文翻译
配置校验
前端协议说明
测试用例
相关 switch 逻辑
```

## 7. 边界层约定

强类型 ID 只存在于 C++ 规则引擎内部。

外部层仍然使用字符串：

```text
JSON 传输层：string
配置文件：string
数据库存储：string
日志输出：string
```

也就是说：

```text
规则引擎内部：ObjectId / ObjectDefinitionId 是不同类型。
规则引擎边界外：它们都序列化为字符串。
```

边界层负责字符串与强类型 ID 的转换。

## 8. 当前结论

已确认采用：

```text
C++ 内部强类型 ID，外部传输/配置/存储使用字符串。
```
