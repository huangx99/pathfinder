# 62A_P34前置认知因果归因系统任务卡设计

## 0. 文档定位

本文档是 P34 端到端 MVP 之前的前置设计，用来补齐当前 H5 可玩验证壳暴露出的核心问题：

```text
观察到一个结果，不等于已经知道真正原因。
```

当前红果案例说明：

```text
吃红果 3 次后确认红果可食用。
随后吃 1 次腐烂红果后身体不适。
```

此时系统不能直接确认：

```text
腐烂红果会中毒。
```

因为还存在其他解释：

```text
可能是腐烂导致。
可能是红果吃多了导致。
可能是连续进食导致。
可能是身体状态导致。
可能是环境或随机事件导致。
可能是腐烂 + 吃多了共同导致。
```

所以本阶段要设计并实现一个可扩展的“认知因果归因系统”，让知识形成从简单结论升级为：

```text
观察事实 -> 相关性 -> 多个原因候选 -> 排除替代解释 -> 因果确认 -> 传播/行动使用
```

---

## 1. 设计目标

### 1.1 核心目标

建立一套通用因果归因层，支持未来所有探索对象，而不只服务红果。

必须覆盖：

- 食物效果：可食用、中毒、吃多了不适。
- 物体状态：腐烂、燃烧、潮湿、破损、污染。
- 剂量影响：吃一次没事，吃多次有问题。
- 时间影响：立即生效、延迟生效、持续生效。
- 行动影响：吃、使用、触碰、组合、给予 NPC。
- 角色状态影响：饥饿、饱腹、生病、疲劳、免疫。
- 环境影响：雨天、夜晚、温度、区域危险。
- 多因素影响：火 + 木棍 -> 火把，毒 + 伤口 -> 感染。
- NPC 行为判断：不确定风险与确认恶意要区分。

### 1.2 工程目标

本系统必须满足：

```text
1. 前端不判断规则。
2. 规则结果仍由后端裁决。
3. 因果归因是独立模块，不写死红果。
4. 后续新增物品只需要配置 cause feature / condition / outcome，不需要重构。
5. H5 只显示安全投影，不显示 raw state / hidden truth。
6. 因果不确定性可以传播给族群和 NPC。
7. 未来强化学习可以读取安全 observation / hypothesis / reward 结构。
```

### 1.3 玩家体验目标

玩家应该体验到：

```text
我不是按按钮拿标准答案。
我是在观察、猜测、验证、排除错误原因。
```

界面应该能告诉玩家：

- 我观察到了什么。
- 我怀疑哪些原因。
- 为什么现在还不能确认。
- 我应该怎么继续验证。
- 哪些知识可以放心传播。
- 哪些知识只是风险提醒。

## 2. 不解决的问题

本阶段不做：

- 正式地图。
- 正式背包。
- 复杂 UI 动画。
- 多 NPC 完整社会系统。
- 完整数值平衡。
- 随机世界生成。
- 大规模内容配置。

本阶段只做因果归因抽象层和最小接入，重点保证后续扩展不推翻架构。

## 3. 当前错误逻辑与修正方向

### 3.1 错误逻辑 A：一次结果直接变结论

错误：

```text
吃腐烂红果 -> 不适 -> 腐烂红果有毒
```

正确：

```text
吃腐烂红果 -> 不适 -> 观察到相关 -> 可能原因有多个 -> 等待验证
```

### 3.2 错误逻辑 B：只靠次数确认

错误：

```text
同一现象出现 3 次 -> 已确认
```

正确：

```text
同一现象出现多次 -> 相关性增强
是否确认因果，还要看替代解释是否被排除
```

### 3.3 错误逻辑 C：分支完全隔离

错误：

```text
红果和腐烂红果完全互不影响
```

正确：

```text
红果和腐烂红果不能直接覆盖彼此知识，
但它们可以互相提供替代解释或对照证据。
```

例如：

```text
普通红果多次安全 -> 支持“腐烂状态可能是风险来源”。
普通红果吃多了也不适 -> 支持“剂量可能是风险来源”。
```

## 4. 核心概念

### 4.1 Observation：观察事实

观察事实只记录发生了什么，不判断原因。

```cpp
struct CausalObservationRecord {
    ObservationId observation_id;
    EntityId actor_id;
    std::string subject_id;
    std::string action_key;
    std::string outcome_key;
    std::vector<std::string> subject_state_keys;
    std::vector<std::string> actor_state_keys;
    std::vector<std::string> environment_state_keys;
    std::vector<std::string> recent_exposure_keys;
    int recent_same_subject_count = 0;
    int recent_same_action_count = 0;
    int recent_same_family_count = 0;
    Tick observed_tick;
};
```

中文解释：

| 字段 | 中文 | 说明 |
|---|---|---|
| observation_id | 观察 ID | 稳定唯一 ID |
| actor_id | 行动者 | 谁做了这个尝试 |
| subject_id | 对象 | 红果、石片、木棍等 |
| action_key | 行动 | eat/use/touch/combine/give |
| outcome_key | 结果 | restore_hunger/poison/no_effect 等 |
| subject_state_keys | 对象状态 | decayed/wet/burning/broken |
| actor_state_keys | 行动者状态 | hungry/full/sick/tired |
| environment_state_keys | 环境状态 | rain/night/cold/poison_area |
| recent_exposure_keys | 近期接触 | 最近吃过什么、接触过什么 |
| recent_same_subject_count | 近期同物体次数 | 用于剂量判断 |
| recent_same_action_count | 近期同行动次数 | 用于连续行为判断 |
| recent_same_family_count | 近期同类次数 | 红果和腐烂红果都属于 berry_family |
| observed_tick | 观察时间 | 用于时间窗口 |

### 4.2 CauseFeature：原因特征

原因特征是可配置的，不写死红果。

例如：

```text
object:red_berry
state:decayed
family:berry
dose:berry_family:gte:4
action:eat
actor_state:full
environment:rain
combination:fire+wood
```

建议格式：

```cpp
struct CauseFeature {
    std::string feature_key;
    CauseFeatureKind kind;
    std::string value_key;
    int numeric_value = 0;
};
```

枚举：

```cpp
enum class CauseFeatureKind {
    Unknown,
    SubjectIdentity,       // 对象身份：红果
    SubjectFamily,         // 对象族类：浆果类
    SubjectState,          // 对象状态：腐烂
    Action,                // 行动：吃
    ActorState,            // 行动者状态：饱腹
    EnvironmentState,      // 环境状态：下雨
    RecentExposure,        // 近期接触：刚吃过红果
    DoseWindow,            // 剂量窗口：短时间吃了 4 个
    Combination,           // 组合原因：火+木棍
    TimeDelay,             // 延迟原因：过几回合发作
    TestOnly
};
```

### 4.3 CausalHypothesis：因果假设

因果假设表示“某个原因可能导致某个结果”。

```cpp
struct CausalHypothesis {
    HypothesisId hypothesis_id;
    std::string subject_id;
    std::string action_key;
    std::string outcome_key;
    std::vector<CauseFeature> cause_features;
    std::vector<std::string> alternative_hypothesis_keys;
    CausalAttributionStatus status;
    double support_score = 0.0;
    double contradiction_score = 0.0;
    double ambiguity_score = 0.0;
    int supporting_observation_count = 0;
    int counterexample_count = 0;
    int alternative_unresolved_count = 0;
    std::vector<ObservationId> supporting_observation_ids;
    std::vector<ObservationId> counterexample_observation_ids;
};
```

例子：

```text
假设 A：state:decayed + action:eat -> poison
假设 B：dose:berry_family:gte:4 + action:eat -> poison
假设 C：actor_state:full + action:eat -> poison
```

### 4.4 CausalAttributionStatus：因果归因状态

```cpp
enum class CausalAttributionStatus {
    Unknown,
    ObservedCorrelation,
    Ambiguous,
    WeaklySupported,
    StronglySupported,
    Confirmed,
    Refuted,
    TestOnly
};
```

中文解释：

| 状态 | 中文 | 含义 |
|---|---|---|
| Unknown | 未知 | 还没有归因 |
| ObservedCorrelation | 观察到相关 | 只知道 A 后发生 B |
| Ambiguous | 原因不明 | 多个解释并存 |
| WeaklySupported | 弱支持 | 有少量证据 |
| StronglySupported | 强支持 | 多次支持，但仍有替代解释 |
| Confirmed | 已确认因果 | 主要替代解释已排除 |
| Refuted | 已排除 | 该解释被反例推翻 |
| TestOnly | 测试专用 | 只能测试使用 |

## 5. 知识层级设计

### 5.1 Observation Knowledge：观察知识

展示：

```text
我吃了腐烂红果后身体不适。
```

意义：记录事实，不给结论。

### 5.2 Correlation Knowledge：相关知识

展示：

```text
吃腐烂红果后多次出现不适。
```

意义：有相关性，但原因未必确定。

### 5.3 Causal Knowledge：因果知识

展示：

```text
腐烂状态会导致红果中毒。
```

意义：能指导行动、传播、NPC 判断。

### 5.4 Practical Knowledge：实用知识

展示：

```text
不要吃腐烂红果。
```

意义：即使原因未完全确认，也可以作为保守行为建议。

注意：

```text
实用建议可以比因果确认更早出现，但不能伪装成已确认因果。
```

## 6. 因果归因流程

### 6.1 输入

每次行动结束后，后端生成：

```text
CausalObservationRecord
```

来源包括：

- 当前 action。
- 当前 object。
- 当前 outcome。
- 对象状态。
- 行动者状态。
- 环境状态。
- 最近接触历史。

### 6.2 生成候选原因

根据观察记录生成候选：

```text
对象身份原因
对象状态原因
对象族类原因
剂量原因
行动者状态原因
环境原因
组合原因
未知原因
```

例如：

```text
吃腐烂红果后不适
```

生成：

```text
state:decayed -> poison
subject:red_berry -> poison
family:berry -> poison
dose:berry_family:gte:4 -> poison
actor_state:full -> poison
unknown -> poison
```

### 6.3 更新假设分数

每条假设计算：

```text
support_score
contradiction_score
ambiguity_score
alternative_unresolved_count
```

### 6.4 判断状态

状态转换建议：

```text
1 次支持，有多个替代解释：Ambiguous
多次支持，但替代解释未排除：StronglySupported
支持足够且替代解释已排除：Confirmed
出现稳定反例：Refuted
```

### 6.5 输出到知识系统

输出分三种：

```text
观察事实 -> 观察知识
强相关 -> 相关知识 / 风险提醒
确认因果 -> 正式知识
```

## 7. 替代解释处理规则

### 7.1 什么是替代解释

如果同一个 outcome 可以由多个 cause feature 解释，这些 cause feature 互为替代解释。

例子：

```text
poison outcome
可能原因：decayed / dose / actor_state / environment
```

### 7.2 如何排除替代解释

排除方式来自对照观察。

| 替代解释 | 排除方式 |
|---|---|
| 吃多了导致 | 没有连续吃很多时，吃腐烂红果仍不适 |
| 腐烂导致 | 吃普通红果同剂量也不适，且腐烂不总是不适 |
| 环境导致 | 换环境后仍发生 / 同环境下不吃不会发生 |
| 行动者状态导致 | 不同状态下仍发生 / 同状态下其他物体不发生 |
| 对象身份导致 | 同类不同状态对照 |

### 7.3 未排除前的状态限制

只要主要替代解释未排除：

```text
不能进入 Confirmed。
最高只能 StronglySupported 或 Ambiguous。
```

## 8. 红果案例的正确表现

### 8.1 序列 A

```text
吃红果
吃红果
吃红果
吃腐烂红果
```

应显示：

```text
红果 → 吃 → 可以缓解饥饿
已确认：多次普通红果经验支持。

腐烂红果 → 吃 → 身体不适
原因不明：可能与腐烂有关，也可能与连续吃红果有关。
待验证：在没有连续吃普通红果的情况下，单独尝试腐烂红果是否仍会不适。
```

不能显示：

```text
腐烂红果会中毒：已确认
```

### 8.2 序列 B

```text
重开
吃腐烂红果
重开
吃腐烂红果
重开
吃腐烂红果
```

此时吃多了的解释被削弱，可以逐步支持：

```text
腐烂状态可能导致中毒。
```

如果再有普通红果对照：

```text
重开
吃红果
吃红果
吃红果
没有中毒
```

则腐烂解释更接近 Confirmed。

### 8.3 序列 C

```text
吃红果
吃红果
吃红果
吃红果
吃红果
出现不适
```

应形成：

```text
红果吃多了可能导致不适。
```

这和腐烂红果有毒不是同一条知识。

## 9. 未来场景覆盖检查

### 9.1 吃多了才有毒

例子：

```text
蘑菇吃 1 个没事，吃 3 个中毒。
```

通过 DoseWindow 表达。

### 9.2 特定状态才有毒

例子：

```text
新鲜肉可食用，腐烂肉中毒。
```

通过 SubjectState 表达。

### 9.3 特定角色状态才有影响

例子：

```text
饥饿时吃果子恢复，饱腹时继续吃会不适。
```

通过 ActorState 表达。

### 9.4 环境导致结果变化

例子：

```text
雨天火把无法点燃。
夜晚某植物有毒。
```

通过 EnvironmentState 表达。

### 9.5 组合反应

例子：

```text
火 + 木棍 -> 火把。
水 + 火 -> 熄灭。
毒液 + 箭头 -> 毒箭。
```

通过 Combination 表达。

### 9.6 延迟结果

例子：

```text
吃毒蘑菇后 3 回合才不适。
```

通过 TimeDelay + time_window 表达。

### 9.7 多因素共同作用

例子：

```text
伤口 + 污水 -> 感染。
寒冷 + 疲劳 -> 生病。
```

通过多个 CauseFeature 共同组成一条 hypothesis。

### 9.8 随机或未知原因

例子：

```text
同样行为偶尔发生不同结果。
```

保留 unknown cause，避免错误确认。

## 12. 族群与 NPC 规则

### 12.1 不确定风险

如果状态是 Ambiguous / WeaklySupported：

```text
NPC 谨慎，不直接认定恶意。
```

### 12.2 强相关风险

如果状态是 StronglySupported：

```text
NPC 可以拒绝尝试，或要求更多验证。
```

### 12.3 已确认因果

如果状态是 Confirmed：

```text
NPC 可以把投喂危险物视为恶意行为。
族群可以形成禁食规则。
```

## 13. 测试计划

### 13.1 红果基础测试

```text
吃红果 3 次 -> 红果可食用确认。
吃腐烂红果 1 次 -> 原因不明，不确认中毒因果。
```

### 13.2 腐烂对照测试

```text
多次在无普通红果累计前吃腐烂红果 -> 腐烂原因增强。
普通红果对照安全 -> 腐烂原因确认。
```

### 13.3 剂量测试

```text
普通红果吃到剂量阈值后不适 -> 生成剂量假设。
少量普通红果安全 -> 不否定可食用知识。
```

### 13.4 环境测试

```text
雨天火把失败，晴天火把成功 -> 环境解释成立。
```

### 13.5 行动者状态测试

```text
饥饿时恢复，饱腹时不适 -> actor_state 解释成立。
```

### 13.6 组合测试

```text
火 + 木棍 -> 火把。
湿木棍 + 火 -> 失败。
```

### 13.7 传播测试

```text
原因不明知识传播给同伴 -> 同伴知道风险不确定。
已确认因果传播给同伴 -> 同伴形成更强规则。
```

### 13.8 NPC 恶意判断测试

```text
不确定风险时投喂 -> NPC 谨慎。
已确认有毒时投喂 -> NPC 判断可能恶意。
```

## 14. 实施顺序

### Step 1：新增 causal 基础类型

### Step 2：新增观察构建器

### Step 4：新增归因解析器

实现支持、反例、替代解释、状态转换。

### Step 5：接入 LearningLoop

LearningLoop 不再只输出直接知识，还输出 causal attribution result。

### Step 6：接入 H5Projection

增加安全因果提示，不让前端猜原因。

### Step 7：重写红果测试

修正当前错误测试：腐烂红果一次不能确认中毒。

### Step 8：补复杂扩展测试

覆盖剂量、环境、actor 状态、组合、传播、NPC 判断。

## 15. 当前 P33 临时补丁处理

当前代码里已经有一个简单“次数阈值”补丁。正式实现因果归因后：

```text
不能保留它作为最终规则。
```

它可以暂时作为低层 evidence count，但不能直接决定 Confirmed。

最终应改为：

```text
次数只增加 support_score。
因果确认由 CausalAttributionResolver 决定。
```

## 16. 验收标准

本阶段完成后必须满足：

```text
1. 吃红果 3 次可以确认普通红果在当前条件下可食用。
2. 吃腐烂红果 1 次只产生原因不明/风险相关，不确认腐烂因果。
3. 腐烂红果多次独立验证 + 普通红果对照后，才能确认腐烂风险。
4. 吃多了导致不适能形成剂量假设，而不是覆盖腐烂假设。
5. H5 显示“可能原因”和“待验证”，不是直接显示已确认。
6. 同伴可以学习不确定知识，但不会把不确定知识当成确认事实。
7. 前端不解析因果逻辑，只显示安全投影。
8. 全量测试通过。
```

---

## 17. 结论

这个阶段是《先驱者》认知玩法能否成立的关键。

如果没有因果归因，游戏会退化成：

```text
点击物品 -> 系统给答案
```

有了因果归因，游戏才会变成：

```text
观察世界 -> 提出假设 -> 设计验证 -> 排除错误 -> 传播知识
```

这才符合“先驱者”的核心主题。
