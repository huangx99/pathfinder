# 先驱者：程序设计计划

> 后续所有阶段设计必须先遵守 `00_设计文档编写要求.md`。如果阶段设计文档明显偏薄，必须先补齐阶段定位、枚举、DTO、接口、流程、测试、风险和错误实现禁止清单，再进入开发。

## 1. 计划目标

本文档是《先驱者》程序设计阶段的总计划。后续所有类、接口、枚举、数据结构、规则流程和配置格式，都必须按本文档的顺序逐个细化，不能跳到实现 Demo，也不能先做表现层。

当前阶段的目标是建立一个可长期扩展的规则引擎设计基础，服务于游戏核心链路：

```text
对象 → 行为 → 反馈 → 记忆 → 知识 → 传播 → 文明能力
```

程序设计必须优先保证：

- 规则可验证。
- 状态可复盘。
- 接口边界清晰。
- 枚举含义稳定。
- 配置可以驱动内容扩展。
- 前端不推导规则结果。
- 后端规则引擎是唯一权威来源。

## 2. 工作原则

### 2.1 先设计，后实现

每个模块必须先完成设计文档，再写代码。

设计顺序：

```text
概念定义
  ↓
+职责边界
  ↓
+枚举设计
  ↓
+数据结构
  ↓
+接口设计
  ↓
+状态流转
  ↓
+事件输出
  ↓
+配置格式
  ↓
+示例用法
  ↓
+测试点
```

### 2.2 从最基础类型开始

不能一开始设计 `RuleEngine`、`WorldObject` 或 `KnowledgeSystem`。必须先设计它们依赖的基础类型。

基础类型包括：

```text
Id 类型
时间类型
结果类型
错误码
枚举命名规则
事件命名规则
状态版本
随机数种子
数值范围约定
```

### 2.3 一个文档只解决一个层级

每份细化文档只解决一个明确主题。例如：

```text
正确：基础类型与通用约定.md
正确：世界对象设计.md
正确：认知状态设计.md
错误：规则引擎和世界和知识一起设计.md
```

### 2.4 所有枚举必须说明使用方式

每个枚举都必须说明：

- 枚举名。
- 枚举值。
- 每个值的含义。
- 由谁设置。
- 由谁读取。
- 是否可以持久化。
- 是否可以发送给前端。
- 是否允许配置引用。
- 增加新枚举值时需要检查哪些模块。

### 2.5 所有接口必须说明副作用

每个接口必须说明：

- 输入。
- 输出。
- 是否修改 `GameState`。
- 是否产生事件。
- 是否读取配置。
- 是否使用随机数。
- 是否允许失败。
- 失败时返回什么。
- 是否要求调用顺序。

### 2.6 所有状态变化必须有事件

规则引擎内的重要状态变化不能静默发生。

例如：

```text
错误：直接修改 hp，不输出事件。
正确：修改 hp，同时输出 HealthChangedEvent。
```

### 2.7 所有规则结果必须可复盘

同一个输入应得到同一个输出：

```text
同一个 GameState
+ 同一个 ActionCommand
+ 同一个 RuleContext
+ 同一个 RandomSeed
= 同一个 RuleResult
```

如果某个设计破坏可复盘性，必须在文档中明确说明原因，否则不允许进入核心规则层。

## 7. 核心接口总清单

本节列出第一阶段必须设计的接口。接口名称可以在细化文档中调整，但职责不能丢失。

### 7.1 规则引擎入口

```text
RuleEngine.execute(context, state, command) -> RuleResult
```

职责：

- 接收当前状态和行为命令。
- 执行完整规则管线。
- 修改权威状态。
- 返回事件、错误和状态版本。

### 7.2 命令校验

```text
CommandValidator.validate(context, state, command) -> CommandValidationResult
```

职责：

- 判断目标是否存在。
- 判断行为是否允许。
- 判断距离、资源、体力和阶段是否满足。
- 失败时返回稳定错误码。

### 7.3 交互结算

```text
InteractionResolver.resolve(context, state, command) -> InteractionResult
```

职责：

- 匹配 `ActionCommand` 与 `InteractionRule`。
- 计算状态变化。
- 生成反馈。
- 生成事件。

### 7.4 反馈转认知

```text
CognitionResolver.applyFeedback(context, state, feedbacks) -> CognitionUpdate
```

职责：

- 从反馈中提取认知线索。
- 更新特性置信度。
- 判断对象是否部分或完全揭示。
- 生成认知事件。

### 7.5 记忆形成

```text
MemoryResolver.formMemory(context, state, command, interactionResult, cognitionUpdate) -> MemoryEntry
```

职责：

- 将一次实验沉淀为个人记忆。
- 记录证据来源。
- 记录风险和结果。

### 7.6 知识形成

```text
KnowledgeResolver.tryCreateKnowledge(context, state, memory) -> optional<KnowledgeEntry>
```

职责：

- 判断记忆是否达到知识形成条件。
- 合并重复证据。
- 创建或更新族群可传播知识。

### 7.7 知识传播

```text
PropagationResolver.propagate(context, state, knowledge) -> PropagationResult
```

职责：

- 根据传播渠道和文明阶段计算传播效果。
- 更新族群知识掌握程度。
- 触发族群行为变化。

### 7.8 能力解锁

```text
CapabilityResolver.checkUnlocks(context, state) -> list<Capability>
```

职责：

- 检查当前知识是否满足能力解锁条件。
- 解锁新族群能力。
- 检查文明阶段变化。

### 7.9 事件收集

```text
EventCollector.emit(event)
EventCollector.all() -> list<GameEvent>
```

职责：

- 收集规则执行中产生的事件。
- 保持事件顺序。
- 为前端、测试、日志和复盘提供统一输出。

## 9. 推荐细化顺序

### Phase 0：地基约定

目标：让后续所有文档有统一语言。

输出文档：


- 所有 ID 类型都有命名和用途。
- 所有通用枚举都有使用说明。
- 错误码能覆盖命令失败、配置错误和规则错误。
- 事件结构能表达状态变化。

### Phase 1：世界对象与特性

- 能表达玩家不知道该对象。
- 能表达对象逐步揭示。
- 能表达对象拥有多个特性。

## 10. 每次细化的工作流

每次开始一个新模块时，按以下流程推进：

```text
1. 读取程序设计计划.md
2. 确认当前要细化的文档编号
3. 读取前置文档
4. 写本模块设计文档
5. 检查是否覆盖枚举、类、接口、事件、配置、示例、测试点
6. 更新程序设计计划.md 的进度表
7. 再进入下一个模块
```

禁止跳过进度表直接写后续模块。

```text
不能写死 wolf/torch/fire/companion 专属规则；必须通过 ContentRegistry 的 AgentTemplate、object tags 和通用 Command handler 判断。
```

P63 完成后，开放世界原型应具备最小危险闭环：玩家在地图上遇到野生 Agent，野生 Agent 会通过后端权威命令靠近或攻击，玩家和 NPC 的生命状态能被安全投影，NPC 仍能通过已有教学/制作/火焰反制链路处理威胁。

## 24. P64 当前设计说明：运行时玩法逻辑模块化拆分

P64 的文档入口：

```text
doc/94_P64运行时玩法逻辑模块化拆分任务卡设计.md
```

P64 解决的是 P63 后暴露的架构增长风险：NPC 跟随、NPC 代工、野兽生态、Use 分发、默认场景种子和玩法 handler 注册不能继续堆在 `client_runtime_host/client_runtime_host_factory.cpp`。

P64 完成后的硬边界：

```text
client_runtime_host_factory.cpp 只负责创建 runtime/service/adapter/gateway，并调用 registerDefaultWorldModules(context)。
具体玩法规则、tick、Use 子行为和默认场景种子必须进入 world_xxx / world_modules 模块。
新增行为优先新增模块和注册入口，不能回流 runtime host factory。
状态变化仍必须走 WorldCommandPipeline 或对应权威服务，不能因为拆模块就直接改 runtime 数组。
```

P64 已新增默认世界模块入口、Use 命令路由、post-command hook 注册表，并把跟随、NPC 代工、野兽生态 runtime bridge、默认场景 bootstrap 拆出。

后续新增钓鱼、耕种、治疗、驯服、交易、魔法、机械、电力、区域事件、天气和群体职业时，应按 P64 的模块边界扩展，而不是改 runtime host factory。

P64 命名迁移补充：

```text
旧名 ClientServerRuntimeFactory / client_http/client_server_runtime_factory.* 已废弃。
新名 ClientRuntimeHostFactory / client_runtime_host/client_runtime_host_factory.* 是运行时宿主装配入口。
pathfinder_client_http 只保留 HTTP 传输层。
pathfinder_client_runtime_host 承载本地 Axmol、HTTP server、未来 WASM 可复用的 runtime host。
```

## 25. P65 当前设计说明：安全圈营地与虚拟外界任务系统

P65 的文档入口：

```text
doc/95_P65安全圈营地与虚拟外界任务系统设计.md
```

P65 是项目玩法方向重定向阶段：从“大世界网格开放探索”切换为“小地图安全圈营地生存”。

P65 的核心结论：

```text
地图路径可以虚拟化。
NPC 背包、玩家背包、营地仓库、知识、制作、采集、威胁结果必须真实。
前端显示安全圈、圈外事件、NPC 头顶状态和任务进度条。
后端通过 Command/Runtime/Inventory/Knowledge 权威结算结果。
```

P65 不直接删除代码。后续删除大地图必须按顺序推进：

```text
先建立 CampRuntimeHost / CampProjection / CampTaskProjection
-> 前端切到 CampScene
-> NPC 跟随改为任务节点
-> 狼 tick 改为 BoundaryThreatEvent
-> Camp MVP 可运行后，再删除 world_map_interaction / world_region_state / world_generation / 大地图前端旧文件
```

P65 必须特别注意未来扩展：外星文明、魔法世界不能被 wolf/fire/tribe 等原始内容字段写死，必须用 SafeZone、BoundaryEvent、ThreatEvent、KnowledgeSource、TaskStep、Countermeasure 等通用抽象。

P65 补强事项：NPC 采集、制作和外出任务必须在每个任务 tick 复查危险和资源池状态。

```text
危险超过可处理阈值 -> 插入 return_home。
NPC 不知道反制知识 -> return_home。
NPC 知道反制知识但缺少物品 -> return_home。
资源池 current - reserved 低于最低采集量 -> return_home。
NPC 有知识和物品可处理危险 -> 插入 handle_threat，任务速度变慢。
低/中风险不一定失败，但会影响采集或制作耗时。
```

P66/P67 第一版必须加入资源池 current/max/regen/reserved 设计，避免圈外采集变成无限资源。资源不足和危险返家原因必须进入 NPC 任务进度条投影。

P65 知识卡片补强：教学不再默认无限复制知识，第一版改为可回收的知识卡片分配。

```text
玩家拥有知识卡库存。
知识卡分配给 NPC 后，NPC 持有期间可使用该知识。
知识卡可回收、可重新分配。
玩家有几张同类卡，就能同时让几个 NPC 使用该知识。
NPC 没有内化知识，也没有对应知识卡时，不能执行依赖该知识的任务。
NPC 没有风险处理知识/卡片时，遇到危险必须 return_home。
```

知识卡片要进入 NPC 任务推理和进度条投影：例如 NPC 只有砍伐卡、没有驱狼卡时，采集遇狼必须返回家园；有驱狼卡和火把时，才插入 handle_threat 并延迟任务。

P65 玩法规则扩展补强：后续新增任务、风险、资源池、知识卡、边界事件和访客事件时，默认必须新增独立规则文件，并通过 `CampRuleRegistry` 或对应 registry 注册。`CampRuntimeHost` 只负责装配，不写狼、火把、砍树、学者、魔法、机械等具体规则；注册文件只允许注册规则对象，不能写内容专属 if。JSON / ContentRegistry 负责具体内容参数，新的状态机、任务节点类型、风险模型和结算算法必须进入独立规则模块。

P65 客户端投影补强：知识卡分配不能为每张卡生成 `available_command` 按钮，后端只暴露“管理 NPC 知识卡”入口；卡库面板通过投影列表做分类、搜索、滚动，再调用后端 assign/recover。NPC 任务进度必须输出结构化 `actor_tasks`，前端进度节点读取结构化字段，不能靠中文状态字符串推断规则。

P65 任务图补充已落地：NPC 任务进度从单线进度条升级为后端 `CampTaskGraphView` 投影。前端节点画布只读取 `nodes / edges / active_node_id / graph_mode` 绘制，支持画布内缩放，不插入玩法节点、不推导风险、不显示调试说明。后续玩家可编辑任务图必须新增后端 Command 和验证流程，不能让前端保存可执行规则。

P65 内容 JSON 化补充已落地：营地初始物品、NPC、资源池、知识卡、威胁、配方、NPC 任务、物品名称和 numeric_states 进入 `content/core/camp/basic_camp.json`；`CampRuntime` 和规则文件只读 `CampContentPack` 执行通用结算，不再为 wood/stone/axe/torch/wolf/herb 追加内容专属分支。后续多威胁、多资源区域、多任务图编辑需要升级运行态结构，但不能回退到 runtime 硬编码。
