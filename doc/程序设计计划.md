# 先驱者：程序设计计划

> 后续所有阶段设计必须先遵守 `00_设计文档编写要求.md`。如果阶段设计文档明显偏薄，必须先补齐阶段定位、枚举、DTO、接口、流程、测试、风险和错误实现禁止清单，再进入开发。

## 1. 计划目标

本文档是《先驱者》程序设计阶段的总计划。后续所有类、接口、枚举、数据结构、规则流程和配置格式，都必须按本文档的顺序逐个细化，不能跳到实现 Demo，也不能先做表现层。

当前阶段的目标是建立一个可长期扩展的规则引擎设计基础，服务于游戏核心链路：

```text
对象 → 行为 → 反馈 → 记忆 → 知识 → 传播 → 文明能力
```

程序设计必须优先保证：

- 规则可验证。
- 状态可复盘。
- 接口边界清晰。
- 枚举含义稳定。
- 配置可以驱动内容扩展。
- 前端不推导规则结果。
- 后端规则引擎是唯一权威来源。

## 2. 工作原则

### 2.1 先设计，后实现

每个模块必须先完成设计文档，再写代码。

设计顺序：

```text
概念定义
  ↓
+职责边界
  ↓
+枚举设计
  ↓
+数据结构
  ↓
+接口设计
  ↓
+状态流转
  ↓
+事件输出
  ↓
+配置格式
  ↓
+示例用法
  ↓
+测试点
```

### 2.2 从最基础类型开始

不能一开始设计 `RuleEngine`、`WorldObject` 或 `KnowledgeSystem`。必须先设计它们依赖的基础类型。

基础类型包括：

```text
Id 类型
时间类型
结果类型
错误码
枚举命名规则
事件命名规则
状态版本
随机数种子
数值范围约定
```

### 2.3 一个文档只解决一个层级

每份细化文档只解决一个明确主题。例如：

```text
正确：基础类型与通用约定.md
正确：世界对象设计.md
正确：认知状态设计.md
错误：规则引擎和世界和知识一起设计.md
```

### 2.4 所有枚举必须说明使用方式

每个枚举都必须说明：

- 枚举名。
- 枚举值。
- 每个值的含义。
- 由谁设置。
- 由谁读取。
- 是否可以持久化。
- 是否可以发送给前端。
- 是否允许配置引用。
- 增加新枚举值时需要检查哪些模块。

### 2.5 所有接口必须说明副作用

每个接口必须说明：

- 输入。
- 输出。
- 是否修改 `GameState`。
- 是否产生事件。
- 是否读取配置。
- 是否使用随机数。
- 是否允许失败。
- 失败时返回什么。
- 是否要求调用顺序。

### 2.6 所有状态变化必须有事件

规则引擎内的重要状态变化不能静默发生。

例如：

```text
错误：直接修改 hp，不输出事件。
正确：修改 hp，同时输出 HealthChangedEvent。
```

### 2.7 所有规则结果必须可复盘

同一个输入应得到同一个输出：

```text
同一个 GameState
+ 同一个 ActionCommand
+ 同一个 RuleContext
+ 同一个 RandomSeed
= 同一个 RuleResult
```

如果某个设计破坏可复盘性，必须在文档中明确说明原因，否则不允许进入核心规则层。

## 7. 核心接口总清单

本节列出第一阶段必须设计的接口。接口名称可以在细化文档中调整，但职责不能丢失。

### 7.1 规则引擎入口

```text
RuleEngine.execute(context, state, command) -> RuleResult
```

职责：

- 接收当前状态和行为命令。
- 执行完整规则管线。
- 修改权威状态。
- 返回事件、错误和状态版本。

### 7.2 命令校验

```text
CommandValidator.validate(context, state, command) -> CommandValidationResult
```

职责：

- 判断目标是否存在。
- 判断行为是否允许。
- 判断距离、资源、体力和阶段是否满足。
- 失败时返回稳定错误码。

### 7.3 交互结算

```text
InteractionResolver.resolve(context, state, command) -> InteractionResult
```

职责：

- 匹配 `ActionCommand` 与 `InteractionRule`。
- 计算状态变化。
- 生成反馈。
- 生成事件。

### 7.4 反馈转认知

```text
CognitionResolver.applyFeedback(context, state, feedbacks) -> CognitionUpdate
```

职责：

- 从反馈中提取认知线索。
- 更新特性置信度。
- 判断对象是否部分或完全揭示。
- 生成认知事件。

### 7.5 记忆形成

```text
MemoryResolver.formMemory(context, state, command, interactionResult, cognitionUpdate) -> MemoryEntry
```

职责：

- 将一次实验沉淀为个人记忆。
- 记录证据来源。
- 记录风险和结果。

### 7.6 知识形成

```text
KnowledgeResolver.tryCreateKnowledge(context, state, memory) -> optional<KnowledgeEntry>
```

职责：

- 判断记忆是否达到知识形成条件。
- 合并重复证据。
- 创建或更新族群可传播知识。

### 7.7 知识传播

```text
PropagationResolver.propagate(context, state, knowledge) -> PropagationResult
```

职责：

- 根据传播渠道和文明阶段计算传播效果。
- 更新族群知识掌握程度。
- 触发族群行为变化。

### 7.8 能力解锁

```text
CapabilityResolver.checkUnlocks(context, state) -> list<Capability>
```

职责：

- 检查当前知识是否满足能力解锁条件。
- 解锁新族群能力。
- 检查文明阶段变化。

### 7.9 事件收集

```text
EventCollector.emit(event)
EventCollector.all() -> list<GameEvent>
```

职责：

- 收集规则执行中产生的事件。
- 保持事件顺序。
- 为前端、测试、日志和复盘提供统一输出。

## 9. 推荐细化顺序

### Phase 0：地基约定

目标：让后续所有文档有统一语言。

输出文档：

```text
01_基础类型与通用约定.md
02_通用枚举设计.md

```

验收标准：

- 所有 ID 类型都有命名和用途。
- 所有通用枚举都有使用说明。
- 错误码能覆盖命令失败、配置错误和规则错误。
- 事件结构能表达状态变化。

### Phase 1：世界对象与特性

目标：定义“马赛克对象”背后的真实对象和可认知特性。

输出文档：

```text

07_特性系统设计.md
```

验收标准：

- 能表达一个真实对象。
- 能表达玩家不知道该对象。
- 能表达对象逐步揭示。
- 能表达对象拥有多个特性。
- 能说明特性如何被发现。

### Phase 2：行为、交互与反馈

目标：定义玩家如何通过实验认识世界。

- 能表达观察、触碰、食用、攻击、采集、组合、传播。

- 能判断记忆是否能成为知识。
- 能区分个人认知和族群知识。

## 10. 每次细化的工作流

每次开始一个新模块时，按以下流程推进：

```text
1. 读取程序设计计划.md
2. 确认当前要细化的文档编号
3. 读取前置文档
4. 写本模块设计文档
5. 检查是否覆盖枚举、类、接口、事件、配置、示例、测试点
6. 更新程序设计计划.md 的进度表
7. 再进入下一个模块
```

禁止跳过进度表直接写后续模块。
