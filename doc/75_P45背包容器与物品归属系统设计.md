# P45 背包、容器、掉落物与物品归属系统设计

版本：压力测试修订版  
设计日期：2026-05-20

## 1. 阶段定位

P45 接在 P44 世界网格最小 Runtime 之后。

P44 已经解决：

```text
世界有二维网格。
玩家、实体、格子有 x / y / layer_key 坐标。
GenerateWorld / Move / Inspect / Wait 可以走 WorldCommandPipeline。
Pipeline response 可以返回真实 changed_cells / changed_entities。
```

P45 要解决：

```text
物品到底属于谁、在哪里、如何从地上进入背包、如何从背包掉回地上、如何进入容器、如何从容器取出。
```

一句话定位：

```text
P45 是 V2 世界的“物品归属权威”阶段，不是采集、制作、装备数值或经济系统阶段。
```

P45 的核心不是让玩家立刻拥有丰富物品玩法，而是防止后续再次出现：

```text
同伴的火把从哪里来的？
玩家做出的火把是不是营地共享？
木头在地上、背包里、营地库存里到底是哪一个？
NPC 能不能凭空拿到工具？
丢弃物品后地图上是否真的出现掉落物？
```

P45 通过后，后续 P47 采集、P48 制作、P51 Agent 自主行动、P53 H5 V2 都必须通过统一归属系统操作物品。

## 3. 本阶段目标

P45 目标：

```text
建立 Inventory Runtime。
建立 Inventory / Slot / Stack / Ownership 数据结构。
建立 OnMap / InInventory / InContainer / Equipped / Nowhere 的归属规则。
实现 PickupCommand：从地图实体进入 Actor 背包。
实现 DropCommand：从 Actor 背包掉落到地图格子。
实现 TransferCommand 或内部 TransferItem：Actor、容器、营地库存之间转移。
让 ProjectionPatch.changed_inventories 返回真实库存变化。
让 WorldEntityInstance.location_kind 与 Inventory 状态保持一致。
```

玩家体验目标：

```text
玩家站到地面物品旁边或同格时，可以拾取。
物品进入玩家背包后不再显示为地面实体。
玩家丢弃物品后，物品重新出现在地图格子上。
如果物品在同伴背包里，玩家不能当作自己的材料直接使用。
如果物品在营地库存里，只有规则允许时才能共享。
```

工程目标：

```text
为 P47 采集产物、P48 制作材料、P51 NPC 自主取物、P53 H5 背包 UI 建立底层归属模型。
```

## 4. 本阶段明确不做

P45 不做：

```text
不做采集规则。
不做制作规则。
不做装备属性、攻击力、防御力。
不做复杂负重体积系统。
不做交易系统。
不做经济系统。
不做完整容器权限社交系统。
不做自动整理 UI。
不做堆叠拆分 UI。
不做 NPC 智能取物决策。
不做食用、使用、维护、打磨等物品效果。
```

P45 只做：

```text
物品实例在哪里。
物品实例属于谁。
物品实例能否从 A 移到 B。
移动后 Runtime 和 Projection 如何同步。
```

## 5. 后续问题脑暴

本章节先脑暴以后一定会遇到的复杂场景，避免 P45 只按“玩家捡东西”写死。

### 5.1 玩家、NPC、野兽都可能持有物品

未来场景：

```text
玩家背包里有红果和石片。
同伴背包里有火把。
猎人 NPC 背包里有弓。
狼嘴里叼着肉。
商人背包里有商品。
机器人身上装有电池。
```

P45 防线：

```text
InventoryOwnerKind 必须支持 Actor。
Actor 不等于玩家，NPC、野兽、高级文明个体都可以是 Actor。
物品归属不能写死为 player_inventory。
```

### 5.2 地面物品不是背包物品

未来场景：

```text
砍树后木头掉在地上。
玩家把火把丢到地上。
野兽死亡后掉落牙齿。
风暴把物品吹到另一个格子。
```

P45 防线：

```text
OnMap 实体由 WorldEntityInstance.coord 表示。
InInventory 实体由 InventoryItemStack / InventoryRuntime 表示。
同一个实体不能同时 OnMap 和 InInventory。
```

### 5.3 容器、营地库存、公共仓库

未来场景：

```text
营地有公共木材堆。
箱子里有石头。
房子里有食物储藏。
载具后备箱有燃料。
高级文明飞船有仓储模块。
```

P45 防线：

```text
InventoryOwnerKind 必须支持 Container、Camp、Tribe、Vehicle、Structure。
P45 可只实现 Actor + Container + Camp，但枚举和 DTO 要预留。
```

### 5.4 制作材料不能凭空共享

未来场景：

```text
玩家背包有木头。
同伴背包有干草。
营地库存有石头。
制作火把时材料来源必须明确。
```

P45 防线：

```text
InventoryScope 明确材料搜索范围。
默认只用 actor 自己背包。
使用营地库存必须由 command parameters 或后续规则明确授权。
```

### 5.5 NPC 自主行动需要找物品

未来场景：

```text
同伴知道火把驱赶野兽。
同伴没有火把。
同伴知道火把制作需要木头和火源。
同伴先查看自己背包，再查看附近地面，再查看营地库存。
```

P45 防线：

```text
InventoryQueryService 必须支持按 owner、item_key、location_kind 查询。
P51 Agent 可以调用查询结果规划动作，但 P45 不做智能规划。
```

### 5.6 堆叠、拆分、合并

未来场景：

```text
玩家有 3 个红果。
地上有 2 个红果。
拾取后背包显示 5 个红果。
玩家丢弃 1 个红果。
```

P45 防线：

```text
ItemInstance 与 ItemStack 要区分。
P45 最小实现可以用 stack 管数量，但必须保留 instance_id。
唯一物品和可堆叠物品不能混为一谈。
```

### 5.7 唯一物品、耐久物品、状态化物品

未来场景：

```text
斧头有耐久。
火把燃烧剩余时间不同。
红果腐烂程度不同。
魔法剑有唯一铭文。
机器人核心有电量。
```

P45 防线：

```text
stackable=false 的物品不能用数量简单合并。
InventoryItemEntry 必须支持 instance_id。
状态化物品保持独立实例。
```

### 5.8 权限与偷窃

未来场景：

```text
玩家不能随便拿 NPC 背包里的东西。
玩家可以偷，但会引发关系和事件。
族群库存可能允许共享。
敌对营地库存不能直接使用。
```

P45 防线：

```text
TransferPolicy 预留权限判断。
P45 最小实现只允许 actor 自己背包和公开容器。
其他权限返回 forbidden_by_owner。
```

### 5.9 死亡、销毁、消耗

未来场景：

```text
吃掉红果后红果消失。
火把烧完后消失或变成木炭。
斧头损坏后变成钝斧或废铁。
```

P45 防线：

```text
Nowhere 表示不在地图也不在库存。
消耗必须产生 ItemRemoved / ItemConsumed delta。
P45 不做食用效果，但保留 removeItem / consumeItem 能力。
```

### 5.10 多人或多 Agent 同时拿同一物品

未来场景：

```text
玩家和 NPC 同时尝试捡同一根木头。
狼叼走地上的肉。
两个 NPC 抢同一把武器。
```

P45 防线：

```text
Transfer 必须原子化。
每个 item instance 同时只能有一个 location_kind。
命令执行时如果 item 已不在源位置，返回 item_not_available。
```

### 5.11 前端显示与规则权威

未来场景：

```text
前端显示玩家背包。
前端显示地面掉落物。
前端显示营地库存。
前端提供拾取/丢弃按钮。
```

P45 防线：

```text
前端只能读 ProjectionPatch.changed_inventories 和 changed_entities。
前端不能自己判断物品是否属于玩家。
前端不能自己移动物品。
```

### 5.12 存档与复盘

未来场景：

```text
玩家捡起石头，存档，再读取。
复盘同一串 Command 后物品归属必须一致。
```

P45 防线：

```text
InventoryRuntimeSnapshot 必须能完整表达库存、物品位置、owner、container。
所有 item_id 和 inventory_id 必须稳定。
```

## 6. 核心设计原则

P45 必须遵守：

```text
InventoryRuntime 是库存归属权威。
WorldRuntime 是地图坐标权威。
WorldCommandPipeline 是外部行为入口。
WorldEntityInstance.location_kind 必须与 InventoryRuntime 一致。
物品不能同时属于多个位置。
物品转移必须原子化。
前端只读 patch，不推规则。
```

权威边界：

| 内容 | 权威来源 | P45 角色 |
|---|---|---|
| Command 入口 | WorldCommandPipeline | 继续复用 |
| 地图坐标 | WorldGridRuntime | 只通过接口更新 OnMap 状态 |
| 物品归属 | InventoryRuntime | 新增权威 |
| 物品实体 | WorldEntityInstance + InventoryItemEntry | P45 统一迁移规则 |
| 背包投影 | WorldProjectionPatchDto.changed_inventories | 新增真实内容 |
| 采集产物 | P47 | P45 不做 |
| 制作材料消耗 | P48 | P45 提供底层能力 |
| NPC 规划 | P51 | P45 提供查询能力 |

### 9.2 InventoryId

P45 可以先用 string，但格式必须稳定：


### 9.3 InventoryItemEntry

```cpp
struct InventoryItemEntry {
    std::string entry_id;
    std::string entity_id;
    std::string entity_key;
    int quantity = 1;
    bool stackable = true;
```

规则：


## 10. 核心服务接口

### 10.1 IInventoryRuntime

```cpp
class IInventoryRuntime {
public:
    virtual ~IInventoryRuntime() = default;

    virtual foundation::Result<void> initialize() = 0;
    virtual uint64_t stateVersion() const = 0;

    virtual foundation::Result<InventoryRuntimeState*> findInventory(const std::string& inventory_id) = 0;
    virtual foundation::Result<const InventoryRuntimeState*> findInventory(const std::string& inventory_id) const = 0;
    virtual foundation::Result<std::string> ensureActorInventory(const std::string& actor_key) = 0;
    virtual foundation::Result<std::string> ensureContainerInventory(const std::string& container_entity_id) = 0;

    virtual foundation::Result<ItemLocationRef> findItemLocation(const std::string& entity_id) const = 0;
    virtual foundation::Result<std::vector<InventoryItemEntry>> queryItems(const InventoryOwnerRef& owner, const std::string& entity_key) const = 0;

    virtual foundation::Result<InventoryTransferResult> transfer(const InventoryTransferRequest& request) = 0;
    virtual foundation::Result<InventoryRuntimeSnapshot> snapshotForDebug() const = 0;
};
```

### 10.2 WorldInventoryRuntime

职责：

```text
管理 inventories。
管理 item_locations_by_entity_id。
提供 ensureActorInventory。
执行 transfer。
维护 inventory state_version。
```

它不能直接改 WorldGridRuntime 私有状态，必须通过 P45 适配接口。

### 10.3 IWorldEntityLocationPort

为避免 InventoryRuntime 直接侵犯 WorldGridRuntime 内部，设计一个端口：

```cpp
class IWorldEntityLocationPort {
public:
    virtual ~IWorldEntityLocationPort() = default;

    virtual foundation::Result<const world_runtime::WorldEntityInstance*> findEntity(const std::string& entity_id) const = 0;
    virtual foundation::Result<const world_runtime::WorldActorRuntime*> findActor(const std::string& actor_key) const = 0;
    virtual foundation::Result<const world_runtime::WorldCellRuntime*> findCell(const world_runtime::WorldCellCoord& coord) const = 0;

    virtual foundation::Result<void> removeEntityFromMap(const std::string& entity_id) = 0;
    virtual foundation::Result<void> placeEntityOnMap(const std::string& entity_id, const world_runtime::WorldCellCoord& coord) = 0;
    virtual foundation::Result<void> setEntityLocationKind(const std::string& entity_id, world_runtime::WorldEntityLocationKind kind) = 0;
};
```

P45 可以先在 WorldGridRuntime 上新增这些最小方法，或实现一个 adapter。

```cpp
class InventoryProjectionAdapter {
```

## 11. Command 接入设计

P45 新增 runtime-aware handlers，放入：

```text
pathfinder_world_inventory_command
```

不要放入：

```text
pathfinder_world_command
```

### 11.1 PickupCommand

输入：

```text
command_kind = Pickup
actor_key = player / npc
source = PlayerInput / AgentDecision
target.target_kind = Entity
target.target_entity_id = 地面实体 id
parameters.quantity 可选，默认全部或 1
```

流程：

```text
校验 actor 存在。
校验 item entity 存在。
校验 item.location_kind == OnMap。
校验 item.coord 与 actor.coord 同格或相邻。
ensureActorInventory(actor_key)。
从地图移除 entity。
加入 actor inventory。
更新 item location_kind = InInventory。
清空 coord。
产生 changed_entities / changed_cells / changed_inventories。
返回 InventoryChanged event。
```

失败：

```text
actor_missing
item_missing
item_not_on_map
target_too_far
capacity_exceeded
quantity_invalid
```

### 11.2 DropCommand

输入：

```text
command_kind = Drop
actor_key = player / npc
target.target_kind = Item 或 Inventory
parameters.entity_id
parameters.entity_key
parameters.quantity
parameters.drop_x / drop_y / layer_key 可选，默认 actor 当前坐标
```

流程：

```text
校验 actor inventory 存在。
校验 item 在 actor inventory。
校验数量足够。
确定 drop coord。
从 inventory 移除或扣减。
把 entity 放到地图。
设置 location_kind = OnMap。
设置 coord。
产生 changed_inventories / changed_entities / changed_cells。
```

失败：

```text
inventory_missing
item_not_in_inventory
quantity_insufficient
target_cell_missing
cell_blocked_for_drop
```

### 11.3 TransferCommand

P45 可先作为内部 API，不一定暴露 WorldCommandKind 新枚举，因为 P43 已有 Pickup/Drop，但没有 Transfer。

建议：

```text
P45 不新增 WorldCommandKind::Transfer。
先实现 InventoryRuntime::transfer。
后续如果需要 UI 直接转移容器，再在 P46/P53 设计 action_key -> command 映射。
```

### 11.4 Inventory Query

P45 不一定暴露 query command，但必须提供服务能力：

```text
queryItems(owner, entity_key)
findItemLocation(entity_id)
```

后续用途：

```text
P48 制作查材料。
P51 Agent 查可用工具。
P53 H5 显示背包。
```

## 12. ProjectionPatch 设计

P45 必须填充：

```text
WorldProjectionPatchDto.changed_inventories
```

### 12.1 InventoryPatchDto fields

建议字段：

```text
inventory_id
owner_kind
owner_key
capacity_slots
used_slots
entry_count
entries_json
```

entries_json 必须使用稳定 JSON 结构，至少包含：

```text
entry_id
entity_id
entity_key
stack_key
quantity
location_kind
```

注意：

```text
不得使用 entity_id|entity_key|quantity 这种不可扩展字符串协议。
P53 前端只解析 entries_json，不自己推断 inventory 状态。
```

### 12.2 同步 changed_entities / changed_cells

Pickup：

```text
changed_inventories：actor inventory。
changed_entities：被拾取实体，op 可为 Update 或 Remove，location_kind=InInventory。
changed_cells：原地面格子，entity_ids 删除该实体。
```

Drop：

```text
changed_inventories：actor inventory。
changed_entities：掉落实体，location_kind=OnMap，带 x/y/layer_key。
changed_cells：目标格子，entity_ids 增加该实体。
```

## 13. 状态一致性规则

P45 最重要的是一致性。

必须满足：

```text
同一个 entity_id 只能有一个权威位置。
OnMap：WorldEntityInstance.coord 必须有值，InventoryRuntime 不能包含该 entity。
InInventory：InventoryRuntime 必须包含该 entity，WorldEntityInstance.coord 必须为空。
InContainer：容器 inventory 必须包含该 entity，WorldEntityInstance.coord 必须为空。
Equipped：装备 owner 必须明确，WorldEntityInstance.coord 必须为空。
Nowhere：不在地图、不在库存、不在装备。
```

一致性校验函数：

```cpp
foundation::Result<void> validateItemLocationConsistency(
    const std::string& entity_id,
    const IInventoryRuntime& inventory,
    const world_runtime::IWorldRuntime& world);
```

P45 测试必须覆盖。

## 18. 压力测试回填与最终修订约束

本章节根据 `doc/75A_P45背包容器与物品归属系统压力测试.md` 回填。

本章节约束优先级高于前文较粗的初版描述。研发实现时必须按本章节补强后的规则执行。

### 18.1 Transaction / Draft 事务模型

P45 必须使用事务草稿模型执行物品转移，不能边校验边直接改状态。

新增结构：

```cpp
struct InventoryTransferDraft {
    std::string draft_id;
    InventoryTransferRequest request;
    std::vector<std::string> source_inventory_ids;
    std::vector<std::string> target_inventory_ids;
    std::vector<std::string> source_entity_ids;
    std::vector<std::string> target_entity_ids;
    std::vector<std::string> changed_cell_ids;
    std::vector<std::string> changed_entity_ids;
    std::vector<std::string> changed_inventory_ids;
    bool validated = false;
    bool reserved = false;
};
```

执行流程必须是：

```text
1. normalize request。
2. validate source location。
3. validate actor / permission / distance / visibility。
4. validate quantity / capacity / stack policy。
5. build draft。
6. reserve source item。
7. apply world map changes。
8. apply inventory changes。
9. verify consistency。
10. emit state deltas / projection patch refs。
```

任何一步失败：

```text
不得部分修改世界。
不得部分修改背包。
不得复制物品。
不得让 entity_id 同时出现在 map cell 和 inventory entry。
```

P45 最小实现可以使用单线程串行事务，不要求复杂数据库事务，但代码结构必须体现 validate/apply 两阶段。

### 18.2 SourceInventoryProof 材料来源证明

为防止 P48 制作和 P51 Agent 默认共享库存，P45 必须定义来源证明。

新增结构：

```cpp
struct SourceInventoryProof {
    std::string proof_id;
    std::string inventory_id;
    InventoryOwnerKind owner_kind = InventoryOwnerKind::Unknown;
    std::string owner_key;
    std::string entity_id;
    std::string entity_key;
    int quantity = 0;
    std::string reason_key;
};
```

规则：

```text
任何后续制作、维修、食用、使用、建造需要消耗物品时，都必须提供或生成 SourceInventoryProof。
默认材料范围只能是 actor_self。
使用 Camp / Tribe / Container 库存必须显式授权。
NPC 不得因为“营地有火把”就默认自己拥有火把。
```

P45 不实现制作，但必须让 `queryItems` 返回足够信息构造 SourceInventoryProof。

### 18.3 StackKey 与堆叠规则

P45 必须定义 StackKey，避免普通红果、腐烂红果、不同耐久工具错误合并。

新增字段：

```cpp
struct InventoryItemEntry {
    std::string entry_id;
    std::string entity_id;
    std::string entity_key;
    std::string stack_key;
    int quantity = 1;
    bool stackable = true;
    std::vector<std::string> state_keys;
    std::map<std::string, double> numeric_states;
};
```

StackKey 规则：

```text
stack_key = entity_key + normalized_state_signature。
普通红果和腐烂红果 stack_key 不同。
耐久不同的工具 stack_key 不同，且通常 stackable=false。
stackable=false 时 quantity 必须为 1。
状态不同的物品不得自动合并。
```

必须新增测试：

```text
world_inventory_stack_key_separates_states。
world_inventory_non_stackable_not_merged。
```

### 18.4 GroundStack 地面堆叠策略

为避免地上 100 个红果创建 100 个实体，P45 必须定义地面堆叠策略。

规则：

```text
同一格子。
同 entity_key。
同 stack_key。
stackable=true。
location_kind=OnMap。
```

满足以上条件时，可以合并为一个 ground stack entity。

不允许合并：

```text
腐烂程度不同。
耐久不同。
有独立记忆、铭文、归属标记。
stackable=false。
```

P45 最小实现可以先支持简单 stack，复杂状态由 state_keys / numeric_states 参与 stack_key。

### 18.5 可见性与交互范围硬门禁

Pickup 不能只按 entity_id 成功。

必须校验：

```text
actor 存在。
actor 有 coord。
target entity 存在。
target entity location_kind == OnMap。
target entity 有 coord。
target coord 对 actor 可见或至少已发现，具体策略由 P45 选定。
target coord 与 actor coord 同格或相邻。
```

推荐 P45 最小策略：

```text
Pickup 必须目标格 Visible。
Drop 默认只能丢到 actor 当前格。
后续才允许丢到相邻格或指定格。
```

新增失败原因：

```text
TargetNotVisible
TargetTooFar
TargetNotOnMap
DropCellInvalid
```

### 18.6 ReservationToken 预留

P45 最小实现是串行 Command，不实现复杂多 Agent 并发。

但必须预留：

```cpp
struct InventoryReservationToken {
    std::string reservation_id;
    std::string entity_id;
    std::string reserved_by_actor_key;
    uint64_t expire_world_tick = 0;
};
```

P45 规则：

```text
本阶段可以不启用长期 reservation。
但 transfer apply 前必须二次校验 source location。
重复点击、并发抢物、过期计划都必须因 source mismatch 失败，而不是复制物品。
```

后续 P51 Agent 规划可使用 reservation，避免多个 NPC 长时间追同一件物品。

### 18.7 InventoryPatch entries 结构化要求

初版中 entries 使用紧凑字符串。压力测试后要求改成稳定 JSON 字段。

在不修改 P43 DTO 类型的前提下，P45 可以继续使用 `InventoryPatchDto.fields`，但必须使用稳定 JSON：

```json
{
  "entries": [
    {
      "entry_id": "entry_001",
      "entity_id": "wood_surface_1_0",
      "entity_key": "wood",
      "stack_key": "wood:dry",
      "quantity": 2,
      "location_kind": "in_inventory"
    }
  ]
}
```

字段要求：

```text
fields["entries_json"] 必须是稳定 JSON。
不能长期使用 entity_id|entity_key|quantity 这种不可扩展字符串。
P53 前端只解析 entries_json，不自己推断 inventory 状态。
```

后续如果需要强类型 DTO，可在 P53 前新增 `InventoryEntryPatchDto`。

### 18.8 嵌套容器策略

P45 不支持拾取非空容器。

原因：

```text
箱子进入背包后，箱子里的 inventory 是否随身移动，会引入嵌套容器复杂度。
```

P45 规则：

```text
如果 target entity 是 container，且 container inventory 非空，Pickup 返回 container_not_empty。
空容器是否允许拾取，由 content/config 或 P45 最小策略决定；默认可先禁止。
```

后续阶段再支持：

```text
嵌套容器。
空间袋。
载具货舱移动。
建筑内库存搬迁。
```

### 18.9 Equipped 归属策略

P45 不实现装备效果，但必须表达 Equipped。

规则：

```text
Equipped 表示物品归 Actor 持有，但不在普通背包 entries 中，或在 inventory 中有 equipped slot 标记。
P45 不实现 EquipCommand。
默认材料搜索不包含 Equipped。
Drop 默认不允许直接丢弃 Equipped，必须先 Unequip，P45 可返回 item_equipped。
```

### 18.10 批量转移预留

死亡掉落、容器倾倒、建筑拆除会需要批量转移。

P45 最小实现单件 transfer，但接口不能阻止未来扩展。

预留：

```cpp
foundation::Result<std::vector<InventoryTransferResult>> transferBatch(
    const std::vector<InventoryTransferRequest>& requests);
```

P45 可不实现该函数，但 `InventoryTransferResult` 必须能表达多个 changed ids。

### 18.11 压力测试后新增测试要求

必须补入 P45 测试计划：

```text
world_inventory_pickup_requires_visible_target。
world_inventory_pickup_duplicate_click_fails。
world_inventory_stack_key_separates_states。
world_inventory_non_stackable_not_merged。
world_inventory_drop_partial_stack。
world_inventory_container_not_empty_blocked。
world_inventory_source_proof_actor_self_only。
world_inventory_no_default_camp_sharing。
world_command_pickup_projection_entries_json。
world_command_drop_projection_entries_json。
```

### 18.12 最终修订结论

压力测试后，P45 不再只是“背包数组 + 拾取丢弃”。

最终 P45 必须成为：

```text
物品归属权威。
物品位置一致性权威。
后续制作材料来源证明基础。
后续 Agent 获取物品的查询基础。
前端背包投影基础。
```


## 19. 最终结论

P45 是 V2 开放世界从“有地图”走向“能交互”的关键阶段。

P44 已经让世界有坐标；P45 必须让物品有归属。

P45 成功后，后续采集、制作、NPC 自主行动才能成立：

```text
采集不是凭空 +1，而是地图资源产生物品归属变化。
制作不是凭空消耗材料，而是从明确库存扣除。
NPC 不是凭空拿火把，而是从自己背包、地面或授权库存中获得。
前端不是自己猜背包，而是读取 changed_inventories。
```
