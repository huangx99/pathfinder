# 18_规则引擎接口设计

## 1. 设计目标

规则引擎接口系统负责把第 17 阶段的规则管线包装成稳定、安全、可测试、可扩展的 C++ 后端 API。

第 17 阶段解决的是：

```text
规则引擎内部如何执行。
```

第 18 阶段解决的是：

```text
外部系统如何调用规则引擎。
```

这里的外部系统包括：

```text
H5 前端。
C++ 后端服务层。
自动化测试。
GM / 调试工具。
存档与复盘系统。
AI / 强化学习训练系统。
未来多人或服务端 tick 系统。
```

核心目标：

```text
只有 RuleEngine 是外部可信入口。
内部 Resolver 是插件，不是 API。
```

本章要保证：

```text
所有状态修改都必须经过 RulePipeline。
所有命令提交都必须有 CommandId / TransactionId / StateVersion / ConfigVersionId。
所有查询都只能读快照、投影或事件，不能自己推导规则结果。
所有测试、回放、AI 训练都走同一套 RuleEngine 接口。
所有调试修改都必须进入 debug_admin 管线，并产生日志和事件。
后续新增接口不推翻 RuleEngine 架构。
```

## 2. 不解决的问题

本章不解决：

| 不解决的问题 | 原因 | 后续文档 |
|---|---|---|
| 存档文件格式 | 本章只定义接口，不定义落盘结构 | 19_存档与复盘设计.md |
| HTTP / WebSocket 协议 | 本章是 C++ API，协议映射后续设计 | 20_前后端协议设计.md |
| 测试用例全集 | 本章定义测试入口，测试矩阵后续设计 | 22_测试策略设计.md |
| 具体业务规则 | 由 08-17 负责 | 既有系统 |
| 数据库选型 | 本章只定义 Store 接口 | 实现阶段 |
| 多人同步协议 | 本章只预留 session / client 字段 | 后续网络层 |
| AI 训练算法 | 本章只导出训练数据接口 | 强化学习系统 |

禁止做法：

```text
禁止前端直接调用内部 Resolver。
禁止测试绕过 RuleEngine 直接改状态。
禁止 AI 绕过 CommandEnvelope 直接写 GameState。
禁止 DebugService 裸改状态。
禁止 QueryService 自己推导规则结果。
禁止 ConfigService 热更新直接修改运行时状态。
禁止 EventService 伪造 EventEnvelope。
禁止 SnapshotService 返回可变状态引用。
禁止不同入口走不同规则逻辑。
```

## 3. 核心概念

### 3.1 RuleEngine 是唯一外部入口

外部只允许访问：

```text
RuleEngine
RuleEngineRuntime
RuleEngineServices
```

不允许访问：

```text
CognitionResolver
MemoryResolver
KnowledgeResolver
PropagationResolver
TribeStateResolver
CivilizationResolver
StateChangeGate
EventCommitGate
```

这些内部模块只能由管线调用。

### 3.2 API 分三类

| 类型 | 是否可改状态 | 示例 |
|---|---|---|
| Command API | 是，必须走管线 | submitCommand、submitSystemCommand |
| Query API | 否，只读 | getProjection、getSnapshot、getEvents |
| Tool API | 受限 | replay、validateConfig、exportTrainingSample |

### 3.3 所有状态修改都必须绑定事务

任何成功修改状态的请求必须产生：

```text
CommandId 或 SystemRequestId
PipelineTransactionId
StateVersion before / after
ConfigVersionId
EventEnvelope 列表
PipelineTrace 或 trace 摘要
```

如果没有这些信息，说明接口绕过了规则引擎。

### 3.4 查询不是规则

查询接口只读取：

```text
已提交状态。
已构建投影。
已提交事件。
只读快照。
```

查询接口不允许：

```text
判断行动是否成功。
修改认知。
解锁文明。
生成知识。
推导对象效果。
```

这些必须通过命令和管线完成。

### 3.5 正式运行、测试、回放、AI 必须同入口

同一个行为应该走同一个入口：

```text
RuleEngine.command().submitCommand(...)
```

测试可以设置：

```text
dry_run
replay_mode
trace_level
fixed_random_seed
```

但不能换一套规则函数。

## 4. 总体架构

### 4.1 分层

```text
外部调用方
  ↓
RuleEngine API
  ↓
RuleEngineRuntime
  ↓
RulePipelineExecutor
  ↓
StageAdapter / Resolver
  ↓
StateStore / EventStore / ProjectionStore
```

### 4.2 服务门面

`RuleEngine` 对外暴露服务门面：

| 服务 | 中文名 | 职责 |
|---|---|---|
| CommandService | 命令服务 | 提交、预览、校验命令 |
| QueryService | 查询服务 | 查询状态、投影、对象摘要 |
| EventService | 事件服务 | 查询、订阅、过滤事件 |
| ReplayService | 回放服务 | 重放命令、事件和管线结果 |
| ConfigService | 配置服务 | 加载、校验、切换配置 |
| SnapshotService | 快照服务 | 创建、导出、比较快照 |
| DebugService | 调试服务 | 受控调试命令 |
| TrainingDataService | 训练数据服务 | 导出 AI 样本 |
| HealthService | 健康检查服务 | 版本、状态、配置、存储健康 |

### 4.3 服务之间的规则

```text
CommandService 是唯一常规写入口。
DebugService 是受控写入口，但必须走 debug_admin 管线。
ConfigService 只改配置版本，不直接改 GameState。
QueryService 只读 Projection / Snapshot。
EventService 只读 EventStore 或订阅 EventBus。
ReplayService 可以在隔离环境执行管线，不影响正式 Runtime。
TrainingDataService 只读事件、快照、trace 和投影。
```

## 6. 接口设计

### 6.1 RuleEngine

```cpp
class RuleEngine {
public:
    virtual Result<void> initialize(const RuleEngineInitOptions& options) = 0;
    virtual Result<void> shutdown() = 0;

    virtual CommandService& command() = 0;
    virtual QueryService& query() = 0;
    virtual EventService& event() = 0;
    virtual ReplayService& replay() = 0;
    virtual ConfigService& config() = 0;
    virtual SnapshotService& snapshot() = 0;
    virtual TrainingDataService& training() = 0;
    virtual HealthService& health() = 0;

    virtual DebugService* debug() = 0;
};
```

约束：

```text
RuleEngine 是外部唯一入口。
debug() 在生产环境可以返回 nullptr。
RuleEngine 不暴露内部 Resolver。
```

### 6.2 CommandService

```cpp
class CommandService {
public:
    virtual Result<EngineCommandResult> submitCommand(
        const EngineCommandRequest& request
    ) = 0;

    virtual Result<EngineCommandResult> previewCommand(
        const EngineCommandRequest& request
    ) = 0;

    virtual Result<CommandValidationResult> validateCommand(
        const EngineCommandRequest& request
    ) = 0;
};
```

语义：

```text
submitCommand：完整执行并提交。
previewCommand：不提交，只返回可能成本、阻断原因、预期投影。
validateCommand：只校验命令格式和基础前置，不执行完整规则。
```

### 6.3 QueryService

```cpp
class QueryService {
public:
    virtual Result<EngineQueryResult> query(
        const EngineQueryRequest& request
    ) const = 0;

    virtual Result<EngineQueryResult> getProjection(
        const ProjectionRequest& request
    ) const = 0;

    virtual Result<EngineQueryResult> getSnapshotSummary(
        const SnapshotSummaryRequest& request
    ) const = 0;
};
```

约束：

```text
QueryService 不执行规则。
QueryService 不调用 Resolver。
QueryService 不修改状态。
```

### 6.4 EventService

```cpp
class EventService {
public:
    virtual Result<EventQueryResult> queryEvents(
        const EventQueryRequest& request
    ) const = 0;

    virtual Result<EventSubscriptionHandle> subscribe(
        const EventSubscriptionRequest& request,
        EventSubscriber& subscriber
    ) = 0;

    virtual Result<void> unsubscribe(
        const EventSubscriptionHandle& handle
    ) = 0;
};
```

约束：

```text
EventService 不能伪造事件。
订阅只能收到 EventCommitGate 已提交事件。
```

### 6.5 ReplayService

```cpp
class ReplayService {
public:
    virtual Result<ReplayResult> replayCommand(
        const ReplayRequest& request
    ) const = 0;

    virtual Result<ReplayResult> replayRange(
```

约束：

```text

```

### 6.6 ConfigService


## 7. 命令执行流程

### 7.1 submitCommand

```text
1. 外部构造 EngineCommandRequest。
2. CommandService 校验 EngineRequestContext。
3. 校验 idempotency_key 和 expected_state_version。
4. 构建 PipelineRequest。
5. 调用 RulePipelineExecutor.execute。
6. 获取 PipelineResult。
7. 转换为 EngineCommandResult。
8. 写入 CommandReplayRecord。
9. 返回结果。
```

### 7.2 previewCommand

```text
1. 设置 dry_run=true 或 preview_only=true。
2. 管线最多执行到 check_preconditions / calculate_cost，或按配置执行到 projection。
3. 不提交状态。
4. 不发布正式事件。
5. 返回预期成本、阻断原因、风险提示。
```

### 7.3 validateCommand

```text
只检查：
CommandEnvelope 格式。
ID 是否存在。
参数类型。
调用权限。
基础 StateVersion。

不检查完整交互结果。
不产生状态变化。
```

### 7.4 幂等规则

同一个 `idempotency_key` 重复提交：

```text
如果第一次已成功，返回第一次 EngineCommandResult。
如果第一次执行中，返回 pending 或 busy。
如果第一次失败且未提交状态，可允许重试。
```

## 9. 回放与测试接口

### 9.1 回放原则

```text
同输入、同快照、同配置、同随机记录，必须得到同 output_hash。
```

### 9.2 replayCommand

流程：

```text
1. 加载 CommandReplayRecord。
2. 加载起点 StateSnapshot。
3. 激活对应 ConfigVersionId。
4. 注入 RandomRollRecord。
5. 使用 replay_mode 调用 RulePipelineExecutor。
6. 对比 output_hash。
7. 返回 ReplayResult。
```

### 9.3 测试接口原则

测试允许：

```text
固定随机种子。
固定配置版本。
使用内存 Store。
开启 full trace。
dry_run。
```

测试不允许：

```text
直接改内部 Store 绕过接口。
直接调用 Resolver 当作正式结果。
跳过 EventCommitGate。
跳过 StateChangeGate。
```

## 12. 配置与运行时

### 12.1 初始化流程

```text
1. 创建 RuleEngine。
2. 加载配置包。
3. 校验配置。
4. 初始化 Store。
5. 注册 PipelineDefinition。
6. 注册 StageAdapter。
7. 执行 selfCheck。
8. lifecycle_state = Ready。
```

### 12.2 配置切换

配置切换必须：

```text

记录 ConfigActivated 事件或系统日志。
只影响后续请求。
不改变已有 StateVersion。
```

### 12.3 关闭流程

```text
1. lifecycle_state = Draining。
2. 拒绝新普通命令。
3. 等待执行中事务完成或超时。
4. flush EventStore / StateStore。
5. lifecycle_state = Stopped。
```

## 13. 错误处理

### 13.1 接口错误分类

| 错误 | 默认结果 | 说明 |
|---|---|---|
| engine_not_ready | SystemError | 引擎未就绪 |
| engine_paused | Blocked | 引擎暂停普通命令 |
| request_invalid | Failed | 请求格式错误 |
| caller_not_allowed | Blocked | 调用方无权限 |
| state_version_conflict | SystemError / Blocked | 状态版本冲突 |
| config_version_mismatch | Failed | 配置版本不匹配 |
| command_duplicate | Success / Blocked | 幂等重复 |
| replay_diverged | Conflict | 回放不一致 |
| debug_disabled | Blocked | 调试服务关闭 |

### 13.2 错误返回原则

```text
所有错误必须返回 ErrorDetail。
能绑定 request_id 必须绑定。
能绑定 command_id 必须绑定。
能绑定 transaction_id 必须绑定。
接口层错误 state_mutated 通常为 false。
```

## 15. 校验规则

### 15.1 接口边界校验

| 规则 | 错误级别 |
|---|---|

|---|---|

## 18. 设计注意事项

研发实现必须遵守：

```text
不要把内部 Resolver 暴露给外部系统。
不要让前端直接提交 StateChangeDraft。
不要让 AI 直接修改 GameState。
不要让 QueryService 推导规则结果。
不要让 DebugService 裸改状态。
不要让 ConfigService 激活配置时修改当前状态。
不要让测试使用和正式运行不同的规则入口。
不要返回可变 Store 引用。
不要丢失 request_id、command_id、transaction_id、state_version、config_version_id。
```

推荐实现顺序：

```text
1. 实现 RuleEngineRuntime 生命周期。
2. 实现 CommandService submit / preview / validate。
3. 接入 RulePipelineExecutor。
4. 实现 QueryService 只读投影。
5. 实现 EventService 查询。
6. 实现 SnapshotService 只读快照。
7. 实现 ReplayService。
8. 实现 ConfigService。
9. 实现 TrainingDataService。
10. 最后实现 DebugService。
```
