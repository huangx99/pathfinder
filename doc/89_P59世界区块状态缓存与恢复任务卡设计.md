# P59 世界区块状态缓存与恢复任务卡设计

## 0. 文档状态

阶段编号：P59。

阶段名称：世界区块状态缓存与恢复。

设计目标：让 P58 已经能够按需生成的无限世界区块，在玩家采集、捡取、丢弃、移动、NPC/野兽活动之后，离开再回来仍保持已发生的变化；同时为后续真正存档恢复和区块卸载留下稳定接口，而不是靠重新生成区块伪装持久世界。

本文档遵守：

```text

```

当前设计结论：

```text
P47 已经实现资源节点消耗：remaining_charges 减少，耗尽时 state 变为 Depleted。

P59 不重做采集、不重做物品归属、不重做通用存档包。
P59 只解决 V2 世界区块的运行时状态封存、恢复与防重生边界。
```

## 1. 阶段定位

P59 是 V2 开放世界的“区块状态权威边界”阶段。

它接在 P58 之后，位置在正式地图 UI 深化和更大量世界内容扩展之前。P58 已经解决：

```text
目标 region 不存在 -> 统一 ensure -> WorldGenerationService -> WorldGenerationApplier -> WorldGridRuntime。
```

但 P58 没有解决：

```text
region 初次生成后被玩家和 Agent 改过，之后是否还能恢复这份改过后的状态。
```

P59 的职责不是把整个游戏一次做成磁盘存档系统，而是先让世界引擎内部拥有明确区块状态模型：

```text
初始生成基线
+ runtime 变化后的 region snapshot
+ region cache / restore 入口
+ 与 P30 SaveGamePackage 的后续 section 接口
```

阶段完成后，后续系统看到 region 时不再只有两种粗糙情况：

```text
未生成 -> 生成
已生成 -> 靠内存碰运气保留
```

而应有明确语义：

```text
NeverGenerated -> 允许 P58 生成
ActiveRuntime -> 当前 runtime 正在承载
CachedSnapshot -> 已封存，可恢复
RestoredRuntime -> 从封存状态重新挂回 runtime
```

## 2. 为什么现在做

现在试玩已经能验证 P58：地图能跨边界按需长出来。下一步如果直接加更多对象、更多采集、更多 NPC 活动、更多正式 UI，状态风险会立刻放大。

### 2.1 玩家能直观看到的问题

```text
砍过的资源离开后又恢复。
捡走的地面物品离开后又刷出来。
丢在地上的物品因重新生成 region 消失或重复。
NPC/狼跨区块后位置和状态乱掉。
走过的区域被再次生成，地形/资源/实体重复 apply。
```

### 2.2 工程上会发生的问题

如果现在不设计 P59，后续实现者很容易把状态分散到：

```text
WorldGenerationService 自己的 generated_regions_
WorldGridRuntime 内存 map
前端当前 projection
InventoryRuntime 的 item_locations_
临时 JSON / debug snapshot
```

结果就是：谁都保存一点，谁都不是权威。

### 2.3 为什么不是 P47 已经做完

P47 已经做的是一次采集命令内部的真实结算：

```text
ResourceNode remaining_charges -= 1
remaining_charges == 0 -> node_state_str = Depleted
产物经 P45 归属接口进入背包或地图
```

这解决“采集有没有真的改状态”。

P59 要解决的是：

```text
这个被 P47 改过的 resource node 在 region 离开 active runtime、未来恢复、未来存档读取后，还是不是被改过的状态。
```

因此 P59 读取并保护 P47 状态，不复制 P47 规则。

## 3. 头脑风暴：后续复杂场景会怎么打穿设计

### 3.1 采集资源后跨区块返回

场景：玩家砍树只剩 1 次 charge，离开 region 再回来。

要求：恢复 `remaining_charges`，不能从生成配置回到满 charge。

### 3.2 资源耗尽后不要复活

场景：石矿已 Depleted。

要求：Depleted 是 runtime state，不是前端显示；region restore 后仍禁止采集，除非未来有明确再生系统命令。

### 3.3 拾取和丢弃物品

场景：玩家捡走 region A 的碎石，再在 region B 丢下木头。

要求：

```text
region A snapshot 不应再含已进入玩家背包的 OnMap 实体。
region B snapshot 应含经 P45 DropToMap 建立的 OnMap 实体。
背包中的物品不能被 region snapshot 复制一份。
```

### 3.4 制作产物落地

场景：制作产物选择掉落到地图，之后离开。

要求：region snapshot 只记录 WorldRuntime 中 OnMap 的实体引用和状态；配方、学习来源不在 P59 复制。

### 3.5 NPC 跨 region

场景：NPC 从 region A 走到 B，A 被缓存。

要求：actor/entity 当前 coord 属于 B，A snapshot 不能把 NPC 冻结复制回 A；actor 是跨区块运行时对象，region snapshot 只能按当前坐标分片。

### 3.6 野兽追逐与外界打断

场景：狼从未激活 region 进入玩家附近，region ensure 和 region restore 同时可能发生。

要求：Agent 执行入口复用 region availability；不允许野兽自己旁路生成/恢复区块。

### 3.7 地上堆叠物合并

场景：同一格多次掉落同 stack_key 物品，P45 合并 quantity。

要求：snapshot 保存合并后的实体 quantity 和 stack_key，恢复后不能拆成多份或重复合并。

### 3.8 物品在容器里

场景：地图上箱子在 region A，箱子 inventory 内有物品。

要求：

```text
箱子实体可属于 region snapshot。
箱内物品归属权威仍是 InventoryRuntime。
P59 第一版若未恢复容器 inventory，必须明确阻止卸载含未桥接 inventory 的 region，而不是漏存。
```

### 3.9 建造和地形修改

场景：后续房屋、桥梁、挖坑改变格子阻挡或地形。

要求：region cell snapshot 必须能保存 runtime cell override，不能只记生成 seed。第一版可以只保存现有 cell 字段，但 DTO 要可扩展。

### 3.10 多层地图与虚拟 Z

场景：地表、地下、空中同一 x/y。

要求：region key 必须包含 `layer_key`，禁止只用 `rx/ry`。

### 3.11 负坐标远距离

场景：玩家在负坐标 region 缓存恢复。

要求：继续复用 P58 `WorldRegionMath`，P59 不再写第二套 region ID 算法。

### 3.12 版本兼容

场景：内容包升级后地形生成规则变化。

要求：已缓存 region 恢复自己的 snapshot；未生成 region 才用当前可接受的生成版本。P59 snapshot 必须携带 worldgen/content/version 元数据入口。

### 3.13 存档半途失败

场景：封存 region 过程中命令中断。

要求：不允许先删 runtime 再构建 snapshot。必须先构建并校验，再切换生命周期。

### 3.14 前端远距离观察

场景：玩家拖地图看远处。

要求：P59 不让前端决定 region 保存/恢复；客户端 viewport 仍经后端 ensure/投影策略。

### 3.15 内存无限增长

场景：长距离探索几千 region。

要求：P59 先建 cache/evict 协议和安全门禁。第一版可保守不自动卸载复杂 region，但不能把“永不卸载”写死成架构。

## 4. 阶段目标

P59 完成后必须具备：

1. 有稳定 region 生命周期和缓存状态枚举。
2. 能从 `WorldGridRuntime` 抽取一个 region 的权威 world snapshot：cells、OnMap entities、resource nodes、与 region 相关的探索片段元数据入口。
3. 能把校验通过的 region snapshot 恢复回 runtime，且不会与 P58 首次生成混淆。
4. P58 ensure 在面对缓存过的 region 时先恢复，只有 `NeverGenerated` 才走 world generation。
5. P47 的资源 charge/state 在 region snapshot roundtrip 后保持。
6. P45 的 OnMap 实体位置、quantity、stack_key、state_keys、numeric_states 在 snapshot roundtrip 后保持。
7. 已在背包/容器中的物品不会因 region restore 复制回地图。
8. 为 P30 `SaveGamePackage` 暴露 world-region snapshot section 的后续接入点，不重造存档包。
9. 有失败保护：snapshot 不完整、恢复冲突、归属未桥接、版本不兼容时明确拒绝。
10. 有测试证明：离开/恢复后树不会满血、碎石不会复活、跨区块移动不重复实体。

---

## 5. 本阶段不解决的问题

P59 明确不做：

1. 不实现完整磁盘存档 UI、云存档、自动存档槽位。
2. 不把所有知识、记忆、族群、文明系统塞进 region snapshot；它们属于各自 owner 的全局状态。
3. 不设计资源自然再生。再生必须是未来显式系统，不允许 restore 隐式复活。
4. 不实现复杂 LRU 性能策略到最终版。第一版可有最小 eviction policy 和安全开关。
5. 不做正式客户端地图 UI；前端只用于验证状态没有复活。
6. 不修改 P47 采集规则，不修改 P45 归属事务，不让 P59 直接改 inventory 私有数组。
7. 不允许从前端 projection 反推 snapshot。
8. 不用 P59 顺手做建筑系统、生态刷新系统、天气区域状态系统；只预留 fields/section 扩展位。

## 6. 核心设计原则

### 6.1 WorldRuntime 是 active world 权威

当前 active region 的格子、实体坐标、资源节点运行时状态，以 `WorldGridRuntime` 为权威。

缓存只是 active runtime 离开后的封存结果，不能在 region active 时悄悄覆盖 runtime。

### 6.2 Generation baseline 和 runtime snapshot 分离

```text
未生成 region -> P58 generation
生成后变化过/封存过 region -> P59 snapshot restore
```

禁止对已封存 region 再走初次生成来“恢复”。

### 6.3 P45 归属边界不可破

P59 只保存 region 上的 WorldEntityInstance。任何 `InInventory` / `InContainer` / `Equipped` 实体必须由 Inventory 相关 snapshot/bridge 负责。

如果第一版恢复不了某类 inventory 关联，就以 unload guard 拒绝，而不是复制物品。

### 6.4 先校验 snapshot，再改变生命周期

封存流程必须：

```text
collect -> validate -> cache -> mark lifecycle -> optional detach
```

恢复流程必须：

```text
load cache -> validate -> conflict check -> apply -> mark active
```

### 6.5 可复盘与幂等

同一个 region snapshot 重复恢复不得重复 apply 实体。恢复动作必须有 conflict result，不允许盲目 upsert 导致 duplication。

### 6.6 版本可扩展

Snapshot 必须带 schema/version 元数据。未来 cell override、结构、区域事件、天气魔法层都能加 section，不改 region key 主架构。

## 7. 已有能力复用与不重复设计

| 已有系统 | 已有权威 | P59 如何复用 | P59 禁止 |
|---|---|---|---|
| P47 采集 | `WorldResourceNodeRuntime` 的 `remaining_charges` / `node_state_str` | snapshot roundtrip 这些字段 | 重写 gather/chop/mine 结算 |
| P45 归属 | `InventoryRuntime` + world location port | region 只保存 OnMap 世界实体 | 直接恢复背包物品到地图 |
| P46/P57 生成 | `WorldGenerationService` / `Applier` | NeverGenerated region 初次生成 | 对 cached region 重新生成覆盖 |
| P58 ensure | `WorldRegionMath` / `WorldRegionEnsureService` | availability 入口扩成 restore-aware | 新写第二套坐标/ensure |
| P30 存档前置 | `SaveGamePackage` / `StateSnapshotSection` / `RestorePlan` | 预留 region section/adapter | 新做一套游戏存档包 |

### 7.1 现有代码事实

当前已有：

```text
WorldRuntimeSnapshot
  cells
  entities
  actors
  resource_nodes
```

但它当前语义是 runtime debug/full snapshot 入口，不等于可直接当 region 持久化 API。

P59 应把 debug snapshot 逐步替换为明确 region query/restore port，避免生产持久化长期依赖 `snapshotForDebug()`。

## 9. 核心枚举

### 9.1 WorldRegionLifecycleState

| 枚举 | 中文 | 用途 |
|---|---|---|
| `Unknown` | 未知 | 禁止执行业务 |
| `NeverGenerated` | 从未生成 | P58 可首生成 |
| `ActiveRuntime` | runtime 活跃 | runtime 权威 |
| `Sealing` | 封存中 | 禁止重复卸载 |
| `CachedSnapshot` | 已缓存 | 可恢复，不可重新生成 |
| `Restoring` | 恢复中 | 防重入 |
| `RestoreFailed` | 恢复失败 | 需要明确失败原因 |

### 9.2 WorldRegionStateStoreKind

| 枚举 | 中文 | 第一版 |
|---|---|---|
| `Unknown` | 未知 | 禁止 |
| `InMemory` | 内存缓存 | P59 实现 |
| `SavePackageSection` | 存档包 section | P59 预留桥接 |
| `ExternalStorage` | 外部存储 | 后续 |

### 9.3 WorldRegionSealStatus

```text
Sealed
AlreadyCached
RegionNotActive
UnsafeOwnership
ContainsUnsupportedState
SnapshotInvalid
StoreFailed
```

### 9.4 WorldRegionRestoreStatus

```text
Restored
AlreadyActive
SnapshotMissing
SnapshotInvalid
VersionIncompatible

ApplyFailed
```

### 9.5 WorldRegionAvailabilitySource

```text
ActiveRuntime
RestoredSnapshot

```

这枚举给 P58 ensure result 解释“这次 region 为什么可用”。

## 10. 核心 DTO / 数据结构

### 10.1 WorldRegionSnapshotHeader

```cpp
struct WorldRegionSnapshotHeader {
    std::string snapshot_id;
    std::string snapshot_schema_version; // world_region_snapshot.v1
    std::string world_id;
    WorldRegionKey region_key;            // 复用 P58：含 layer_key/rx/ry
    uint64_t world_seed;
    uint64_t world_tick;
    uint64_t state_version;
    std::string worldgen_profile_key;
    std::string generator_version;
    std::string content_version;
    std::string baseline_region_fingerprint;
};
```

### 10.2 WorldRegionCellStateRecord

第一版直接覆盖当前 `WorldCellRuntime` 必要字段：

```text
cell_id / coord / terrain_key / region_id / generated
blocks_movement / movement_cost / entity_ids / tag_keys
```

注意：`entity_ids` 恢复时要和实体 section 校验一致，不能只相信 cell list。

### 10.3 WorldRegionEntityStateRecord

只允许 region 上的 OnMap 世界实体：

```text
entity_id / entity_key / display_name_key
location_kind == OnMap
coord in region_key
owner_ref / container_ref 为空或满足 OnMap 规则
blocks_movement / visible_by_default / tag_keys
quantity / stackable / stack_key / state_keys / numeric_states
```

### 10.4 WorldRegionResourceNodeStateRecord

```text
node_id / resource_key / coord
node_kind_str / node_state_str
remaining_charges / max_charges
required_action_key / required_tool_key
output_entity_keys / tag_keys
```

这是 P47 采集状态保真的关键 section。

### 10.5 WorldRegionExplorationStateSlice

第一版最小字段：

```text
actor_key
cell_visibility_by_id restricted to this region
```

如果 P59 实现时当前探索状态没有稳定 restore port，则必须：

```text
设计 slice DTO + query/apply port；
或将该 slice 标为 Deferred 并写清 P60 前风险。
```

不得把前端 projection 当探索存档。

### 10.6 WorldRegionSnapshot

```cpp
struct WorldRegionSnapshot {
    WorldRegionSnapshotHeader header;
    std::vector<WorldRegionCellStateRecord> cells;
    std::vector<WorldRegionEntityStateRecord> on_map_entities;
    std::vector<WorldRegionResourceNodeStateRecord> resource_nodes;
    std::vector<WorldRegionExplorationStateSlice> exploration_slices;
    std::vector<std::string> unsupported_state_keys;
    Result<void> validateBasic() const;
};
```

### 10.7 Seal / Restore Result

结果必须包含：

```text
region_key
status
snapshot_id
state_version_before / after
changed_cell_ids
changed_entity_ids
changed_resource_node_ids
failure_reason_keys
warning_keys
```

方便 Command、测试和后续调试 UI 解释。

## 11. 核心接口

### 11.1 IWorldRegionStateQueryPort

```cpp
class IWorldRegionStateQueryPort {
public:
    virtual Result<WorldRegionRuntimeSlice> readRegionSlice(const WorldRegionKey& key) const = 0;
};
```

硬约束：

```text
不能用客户端 projection 读取。
不能把 `snapshotForDebug()` 作为长期 production port。
可以在 P59 过渡期由 WorldGridRuntime 实现 slice query。
```

### 11.2 IWorldRegionStateApplyPort

```cpp
class IWorldRegionStateApplyPort {
public:
    virtual Result<WorldRegionApplyPlan> planRestore(const WorldRegionSnapshot&) = 0;
    virtual Result<WorldRegionApplyResult> applyRestore(const WorldRegionApplyPlan&) = 0;
};
```

Restore plan 必须先检查：

```text
region active conflict
cell id conflict
entity id conflict
resource node id conflict
OnMap ownership violation
state version/version compatibility
```

### 11.3 IWorldRegionStateStore

```cpp
class IWorldRegionStateStore {
public:
    virtual Result<void> put(const WorldRegionSnapshot&) = 0;
    virtual Result<const WorldRegionSnapshot*> find(const WorldRegionKey&) const = 0;
    virtual bool has(const WorldRegionKey&) const = 0;
};
```

第一版实现 `InMemoryWorldRegionStateStore`。

Store 抽象存在的原因：P59 先保证 cache roundtrip；后续写 P30 bridge 或磁盘 store 时不改 lifecycle service。

### 11.4 WorldRegionSnapshotBuilder

```cpp
Result<WorldRegionSnapshot> build(const WorldRegionKey&, const WorldRegionSnapshotBuildContext&);
```

Builder 只 collect/validate，不 detach runtime。

### 11.5 WorldRegionSnapshotRestorer

```cpp
Result<WorldRegionRestoreResult> restore(const WorldRegionSnapshot&, const WorldRegionRestoreContext&);
```

Restorer 不调用生成器。

### 11.6 WorldRegionLifecycleService

```cpp
Result<WorldRegionSealResult> sealRegion(const WorldRegionKey&, SealMode);
Result<WorldRegionRestoreResult> restoreRegion(const WorldRegionKey&);
Result<WorldRegionAvailabilityResult> ensureAvailable(const WorldRegionKey&, AvailabilityReason);
```

`ensureAvailable` 的顺序固定：

```text
if runtime active -> ActiveRuntime
else if store has snapshot -> restore -> RestoredSnapshot
else -> delegate P58 generation -> GeneratedBaseline
```

### 11.7 WorldRegionUnloadGuard

第一版要检查：

```text
玩家所在 region 不卸载。
正在执行命令触达的 region 不卸载。
存在未支持归属/容器/跨 region 关系时不卸载。
snapshot build/validation 未成功不卸载。
```

可接受第一版默认：

```text
只在测试/显式调试调用下 detach；自动 eviction 关闭。
```

这比草率自动卸载安全。

### 11.8 WorldRegionSaveBridge

P59 推荐做到 DTO 和桥接口：

```text
WorldRegionSnapshotCollection
WorldRegionSaveSectionDraft
build section from store
restore store from section
```

后续 P30/P55 存档阶段把 section 放入 `SaveGamePackage`，不再改 region snapshot 主 DTO。

## 12. 运行流程

### 12.1 初次进入 region

```text
Move/Projection/Agent 需要 region K
-> P58 ensure 调 P59 lifecycle.ensureAvailable(K)
-> runtime active? no
-> store cached? no
-> P58 WorldGenerationService + Applier
-> runtime mark ActiveRuntime
-> result.source = GeneratedBaseline
```

### 12.2 离开并显式封存 region

```text
Evict/debug/test 请求 seal K
-> UnloadGuard 检查
-> SnapshotBuilder 从 runtime slice collect
-> snapshot.validateBasic
-> store.put(snapshot)
-> lifecycle mark CachedSnapshot
-> 若 mode=DetachSafeOnly 且 apply port 支持，detach K 的 region-local runtime 状态
```

### 12.3 返回缓存 region

```text
需要 region K
-> lifecycle.ensureAvailable(K)
-> runtime active? no
-> store find snapshot
-> Restorer plan conflict
-> apply cells/entities/resource nodes/exploration slice
-> runtime mark generated/active
-> result.source = RestoredSnapshot
```

### 12.4 采集往返验证流

```text
生成 region K
-> HarvestCommand 更新 resource node remaining_charges
-> seal K
-> detach K
-> restore K
-> HarvestCommand 再检查 node
```

预期：

```text
remaining_charges 接上上次值；Depleted 仍阻止采集。
```

### 12.5 地面物品归属往返

```text
region A 有 loose_stone OnMap
-> PickupFromMap 到 player inventory
-> seal/restore A
```

预期：

```text
A snapshot 不恢复 loose_stone 到地图；inventory owner 保持一份。
```

### 12.6 Save bridge 后续流

```text
runtime/store snapshots
-> WorldRegionSaveBridge build section
-> P30 SavePackageBuilder later embeds section
-> SaveRestorePlanner loads snapshot/store
-> lifecycle restores region on demand
```

## 13. 与现有系统关系

### 13.1 与 P58 region ensure

P58 仍是所有坐标可用性的入口。P59 只扩展它对“cached region”的处理，不让调用方自己判断要生成还是恢复。

### 13.2 与 P47 采集

P47 的 `ResourceHarvestService` 已通过 runtime update 减少 resource node charge。P59 保存 update 后的 node record。P59 不新增 HarvestCommand 分支。

### 13.3 与 P45 Inventory

P59 要求 inventory bridge 明确。最小安全策略：

```text
OnMap entity 随 region snapshot；
InventoryRuntime 内物品不随 region snapshot；
跨边界不支持时 UnloadGuard 拒绝 seal/detach。
```

### 13.4 与 P51/P52 Agent

Actor/NPC/野兽是移动对象。P59 第一版只把位于 region 内的 actor/entity 位置信息纳入冲突分析，不做独立 Agent 记忆存档。Agent 记忆/知识仍由 owner 系统保存。

### 13.5 与 P30 SaveGamePackage

P30 是通用存档包；P59 是 V2 region snapshot。正确关系：

```text
P59 定义 region state section；
后续存档阶段把该 section 接入 SaveGamePackage snapshot/config/integrity。
```

### 13.6 与 P60 UI

UI 只能看到 restored 后的投影。前端不展示内部 cache state 作为玩法规则，不自己让 region 生/死。

## 14. 错误处理

| 情况 | 处理 |
|---|---|
| cached snapshot 缺失 | `SnapshotMissing`；若 lifecycle 标记不是 NeverGenerated，禁止偷偷重新生成 |
| snapshot schema/version 不兼容 | `VersionIncompatible` |
| snapshot entity 含 InInventory | `UnsafeOwnership` 或 `SnapshotInvalid` |
| restore entity id 已存在别处 | `RuntimeConflict` |
| resource node charge > max | `SnapshotInvalid` |
| region key 与 coord 不一致 | `SnapshotInvalid` |
| seal 后 store put 失败 | runtime 保持 active |
| detach 失败 | snapshot 可保留，但 lifecycle 不得误标已卸载 |
| P58 首生成失败 | 仍按 P58 failure reason，不落 cached state |

## 17. 验收标准

P59 验收必须同时满足：

1. 从代码测试能证明区块状态可 seal/restore。
2. P47 charge/depleted 状态 roundtrip 通过。
3. P45 OnMap/Inventory 边界无复制物品。
4. P58 ensure 选择 restore 或 generate 的顺序正确。
5. 恢复失败不会静默重生一个新区块。
6. 生命周期状态和失败原因可诊断。
7. 有 save bridge 方向，不把存档未来锁死。
8. 不破坏跨区块移动、客户端投影、采集、制作现有测试。

## 18. 实现注意事项

### 18.1 不要把 P59 误做成重新生成校验

只比较 seed 不能保存玩家变化。缓存过的 region 必须 restore snapshot。

### 18.2 不要缓存客户端看到的东西

Projection 是安全显示，不是权威存档。

### 18.3 不要过早自动卸载一切

第一版优先显式 seal/detach 测试和安全 guard。自动内存回收策略可以后续调优。

### 18.4 别丢 generated region 权威

P58 已明确 runtime 落地权威。P59 lifecycle store 要和 runtime generated/active 标记一致，避免 `WorldGenerationService::generated_regions_` 成为第二权威。

### 18.5 保存变化后的 runtime，不保存规则推理

知识、记忆、规则 trace、采集原因不属于 region snapshot 的核心 state；它们走既有系统和 P30 后续段。

## 19. 安全边界与错误实现禁止

### 19.1 安全边界

P59 的权威边界固定如下：

```text
active region state 的唯一权威是 WorldRuntime。
sealed region state 的唯一权威是 IWorldRegionStateStore 中通过校验的 WorldRegionSnapshot。
WorldRegionLifecycleService 决定 active / cached / generate 分流，调用方不得自行猜测。
客户端 projection、前端视野和调试文本都只是读取结果，不能参与 seal/restore 决策。
P45 location/ownership 仍决定物品在哪；P59 只能缓存 region-local OnMap 状态。
P47 harvest runtime 仍决定资源节点剩余次数和耗尽；P59 只能保真恢复该状态。
```

### 19.2 错误实现禁止清单

禁止研发为了快速通过测试写成下面这些实现：

```text
1. 只保存 seed/worldgen profile，返回区块时重新生成并声称“恢复”。
2. 把 ClientWorldProjection 直接序列化成 region snapshot。
3. 从 snapshot 恢复 OnMap entity 时顺手复制 actor inventory/container 内容。
4. 为了解冲突直接清空 runtime 现有 entity/resource id 再 apply。
5. 在前端、HTTP gateway 或 available command adapter 里决定某区块是否卸载。
6. 给红果、木头、碎石、火把等具体内容写 P59 专属恢复分支。
7. 让自动 eviction 在 guard、roundtrip、冲突测试之前默认开启。
```

## 20. 风险与降级策略

| 风险 | 影响 | 本阶段策略 |
|---|---|---|
| 探索记忆当前没有稳定 region restore port | 返回区块后可见性状态与 runtime state 脱节 | 先设计 exploration slice 与 query/apply 口；若实现延期，必须写明 Deferred 和 P60 前风险 |
| P58 generated 标记与 P59 lifecycle 双权威 | 同一区块可能被错误重新生成 | `ensureAvailable` 收口到 lifecycle，保留 active/cached/generate 来源测试 |
| 跨 region NPC、野兽、追逐关系被 region-local snapshot 切断 | Agent 被复制回旧区块或引用悬空 | 第一版 guard 拒绝不安全 detach，先保 entity id 冲突测试 |
| 内容包或 snapshot schema 升级 | 老 snapshot 不能安全 apply | header 保存 schema/content/worldgen metadata，不兼容时结构化失败，不静默生成 |
| 长距离探索 cache 持续增长 | 内存压力掩盖状态问题 | 第一版只提供 store 抽象和显式 seal，eviction policy 后续单独设计 |
| P30 存档接入时把 snapshot 再复制一套格式 | 恢复链分裂 | P59 只给 save bridge/section draft，后续 SaveGamePackage 复用 snapshot DTO |
