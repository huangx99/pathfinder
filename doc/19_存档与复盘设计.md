# 19_存档与复盘设计

## 1. 设计目标

存档与复盘系统负责把规则引擎运行后的权威状态、事件、命令、随机记录、配置版本、事务记录和校验信息保存下来，并支持恢复、校验、解释和重放。

本章解决的问题：

```text
如何快速恢复世界状态。
如何解释世界为什么变成当前状态。
如何复现一次玩家行为。
如何验证 AI 训练数据来自真实规则结果。
如何在配置更新后仍能回放旧存档。
如何在崩溃后恢复状态、事件和事务一致性。
如何发现存档损坏或数据不匹配。
```

核心原则：

```text
存档不是只保存 GameState。
存档必须保存“状态 + 为什么变成这个状态的证据”。
```

必须同时保存：

```text
StateSnapshot：快速恢复。
EventLog：解释历史。
CommandReplayLog：重放行为。
RandomReplayLog：保证确定性。
ConfigVersionId：绑定配置。
PipelineTrace：调试解释。
Checksum / Hash：发现损坏。
CrashRecoveryJournal：崩溃恢复。
```

## 2. 不解决的问题

本章不解决：

| 不解决的问题 | 原因 | 后续阶段 |
|---|---|---|
| 具体数据库选型 | 本章只定义逻辑结构 | 实现阶段 |
| 具体文件二进制格式 | 先定义数据契约 | 实现阶段 |
| 云存档同步 | 属于平台和网络层 | 后续服务端设计 |
| 前端存档 UI | 属于前端协议和 UI | 20_前后端协议设计.md |
| 压缩算法细节 | 只定义压缩边界和校验 | 实现阶段 |
| 加密和反作弊完整方案 | 只预留签名和 checksum | 后续安全设计 |
| 全量测试矩阵 | 本章定义测试点 | 22_测试策略设计.md |

禁止做法：

```text
禁止只保存 GameState 不保存 StateVersion。
禁止只保存 StateSnapshot 不保存 CommandReplayLog。
禁止回放时重新抽随机。
禁止旧 ConfigVersionId 缺失时静默回放。
禁止 EventLog 的 StateVersion 与 StateSnapshot 不一致。
禁止崩溃恢复时凭猜测补事件。
禁止存档系统绕过 RuleEngine 修改状态。
禁止 SnapshotService 返回可变状态引用。
禁止把 EventReplay 当作权威状态恢复来源。
```

## 3. 核心概念

### 3.1 快照负责恢复，事件负责解释

```text
StateSnapshot 用于快速恢复权威状态。
EventLog 用于解释发生过程。
```

不能从 EventLog 临时拼出权威状态替代 StateSnapshot，除非在专门的重建工具中进行校验重建。

### 3.2 命令负责重放，随机记录负责确定性

回放一条命令必须使用：

```text
CommandReplayRecord
StateSnapshot
ConfigVersionId
RandomReplayLog
PipelineDefinition version
```

不能使用当前配置和当前随机重新跑。

### 3.3 存档包是多段结构

一个完整存档包不是单文件概念，而是逻辑包：

```text
SaveHeader
SaveIndex
StateSnapshotSection
EventLogSection
CommandReplayLogSection
RandomReplayLogSection
PipelineTraceSection
ConfigVersionManifest
IntegrityManifest
CrashRecoveryJournal
```

实现时可以是单文件、多文件、数据库或对象存储，但逻辑结构必须稳定。

### 3.4 StateVersion 是存档主轴

所有数据必须能对齐到 `StateVersion`：

```text
StateSnapshot.state_version
EventEnvelope.state_version
CommandReplayRecord.state_version_before / after
RandomReplayRecord.transaction_id
PipelineTrace.transaction_id
```

如果 StateVersion 对不上，存档不可直接信任。

### 3.5 回放必须隔离

回放只能在：

```text
隔离 Runtime
dry_run Runtime
ReplayService 专用环境
```

中执行。

不能影响正式 `GameStateStore`。

## 4. 总体架构

### 4.1 数据流

```text
RuleEngine.submitCommand
  ↓
RulePipelineExecutor
  ↓
PipelineResult
  ↓
StateSnapshot / EventLog / CommandReplayLog / RandomReplayLog / TraceLog
  ↓
SaveGamePackage
```

### 4.2 恢复流

```text
读取 SaveIndex
  ↓
读取 SaveHeader
  ↓
校验 IntegrityManifest
  ↓
加载 StateSnapshot
  ↓
加载 EventLog 增量
  ↓
校验 ConfigVersionId
  ↓
恢复 RuleEngineRuntime
```

### 4.3 回放流

```text
ReplayManifest
  ↓
加载起点 StateSnapshot
  ↓
加载 CommandReplayLog
  ↓
加载 RandomReplayLog
  ↓
加载 ConfigVersionManifest
  ↓
ReplayService.replayCommand / replayRange
  ↓
ReplayResult / DivergenceReport
```

## 7. 保存流程

### 7.1 普通保存

```text
1. 暂停或等待当前 PipelineTransaction 完成。
2. 读取当前 StateVersion。
3. SnapshotService 创建只读 StateSnapshot。
4. EventService 导出事件范围。
5. ReplayService / CommandLog 导出 CommandReplayRecord。
6. 导出 RandomReplayLog。
7. 导出 PipelineTrace 引用或摘要。
8. 生成 ConfigVersionManifest。
9. 生成 IntegrityManifest。
10. 写入 SaveGamePackage。
11. 写入 SaveIndex。
```

### 7.2 自动保存

自动保存触发条件：

```text
固定 StateVersion 间隔。
关键事件后，例如 CapabilityUnlocked。
玩家手动保存。
退出前。
测试用例 checkpoint。
```

### 7.3 增量保存

增量保存保存：

```text
上次快照后的 EventLog。
上次快照后的 CommandReplayLog。
上次快照后的 RandomReplayLog。
必要 trace 摘要。
```

第一阶段可以先设计结构，实际实现可先全量保存。

## 8. 加载流程

### 8.1 加载步骤

```text
1. 读取 SaveIndex。
2. 读取 SaveHeader。
3. 校验 save_format_version。
4. 校验 IntegrityManifest。
5. 校验 ConfigVersionManifest。
6. 加载 StateSnapshot。
7. 校验 EventLog 覆盖范围。
8. 检查 CrashRecoveryJournal。
9. 初始化 RuleEngineRuntime。
10. 重建 ProjectionStore。
11. 返回 EngineHealthStatus。
```

### 8.2 配置缺失处理

| 策略 | 行为 |
|---|---|
| RejectLoad | 拒绝加载 |
| AllowViewOnly | 只能查看，不能继续游戏 |
| AllowMigration | 进入迁移流程 |
| DebugOverride | 调试工具指定替代配置 |

禁止：

```text
配置缺失时静默使用当前配置继续游戏。
```

## 9. 回放流程

### 9.1 单命令回放

```text
1. 加载 CommandReplayRecordEx。
2. 定位 state_version_before 对应 StateSnapshot。
3. 加载 ConfigVersionId 对应配置。
4. 加载 RandomReplayRecord。
5. 构建 ReplayRequest。
6. 调用 ReplayService.replayCommand。
7. 比较 output_hash。
8. 输出 ReplayResult。
```

### 9.2 区间回放

```text
1. 选择起点 StateSnapshot。
2. 按 state_version_before 排序命令记录。
3. 逐条调用 ReplayService。
4. 每条校验 StateVersion、EventLog、output_hash。
5. 首次不一致时停止并生成 ReplayDivergenceReport。
```

### 9.3 回放禁止项

```text
不允许重新抽随机。
不允许使用当前配置替代旧配置。
不允许把回放结果写入正式存档。
不允许忽略 output_hash 不一致。
```

## 10. 崩溃恢复流程

### 10.1 崩溃点处理

| 崩溃点 | 恢复策略 |
|---|---|
| BeforeStateCommit | 丢弃未提交事务 |
| DuringStateCommit | 根据 StateVersion CAS 和 commit_plan 判断回滚或标记损坏 |
| AfterStateBeforeEvents | 补全 EventEnvelope 或标记需要事件修复 |
| DuringEventCommit | 根据 event_id 去重补齐事件 |
| AfterEventsBeforePersistMark | 补完成标记 |
| DuringSnapshotWrite | 丢弃坏快照，使用上一快照 + 日志 |

### 10.2 恢复原则

```text
能确定就补全。
不能确定就回滚。
不能回滚就标记损坏。
不能凭猜测生成业务事件。
```

## 14. 设计注意事项

研发实现必须遵守：

```text
不要只保存 GameState。
不要丢失 ConfigVersionId。
不要在回放时重新抽随机。
不要用当前配置静默替代旧配置。
不要把 EventReplay 当作权威状态。
不要让存档系统绕过 RuleEngine 改状态。
不要让 Snapshot 返回可变引用。
不要忽略 output_hash 不一致。
不要在崩溃恢复时凭猜测补业务事件。
```

推荐实现顺序：

```text
1. 实现 SaveHeader / SaveIndex。
2. 实现 StateSnapshotSection。
3. 实现 EventLogSection。
4. 实现 CommandReplayLogSection。
5. 实现 RandomReplayLogSection。
6. 实现 IntegrityManifest。
7. 实现 SaveValidator。
8. 实现 ReplayManifest / ReplayRunner。
9. 实现 CrashRecoveryJournal。
10. 最后做增量保存和压缩。
```
