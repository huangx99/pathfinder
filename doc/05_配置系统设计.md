# 先驱者：配置系统设计

## 1. 本文档目标

本文档用于设计《先驱者》规则引擎中的配置系统。

配置系统是规则引擎的数据定义层。

它负责把游戏内容以可编辑、可校验、可版本管理、可测试的方式提供给规则引擎。

一句话：

```text
代码负责规则能力。
配置负责具体内容。
表达式负责受限的条件、数值、倍率和概率变化。
```

配置系统要支持：

```text
对象定义。
特性定义。
行为定义。
效果定义。
交互规则参数。
文明能力定义。
区域定义。
初始世界定义。
翻译定义。
受限表达式定义。
测试配置集。
```

## 2. 设计边界

### 2.1 本文档要解决的问题

```text
配置文件如何组织。
配置 ID 如何命名和引用。
配置如何加载、校验、注册和查询。
配置如何与枚举、错误、事件、存档、强化学习关联。
如何设计第一阶段小而强的受限表达式系统。
如何让配置系统后续扩展时不推倒重来。
```

### 2.2 本文档暂不解决的问题

```text
不实现完整可视化编辑器。
不设计 MOD 隔离和冲突合并。
不设计在线动态配置发布。
不允许任意脚本语言执行。
不设计复杂剧情脚本系统。
不设计数据库表结构。
```

## 3. 核心原则

### 3.1 配置是数据，不是第二套规则引擎

配置可以定义对象、参数、条件和效果，但不能绕过 C++ 规则管线。

```text
C++ 负责规则结构。
配置负责内容数据。
表达式负责受限计算。
StateChange 负责修改状态。
Event 负责记录事实。
```

### 3.2 启动时强校验

配置错误必须尽早暴露。

```text
配置加载时必须完成结构校验、ID 校验、引用校验、枚举校验、表达式校验。
正式运行中不允许带着非法配置继续游戏。
```

### 3.3 配置版本必须可追踪

每次配置变更都应能追踪版本。

```text
存档绑定 config_version。
事件可记录 config_version。
强化学习数据绑定 config_version。
测试样例绑定 config_version。
```

### 3.4 第一阶段小而强

第一阶段不是做万能编辑器，而是做稳定内核。

```text
格式清晰。
引用可靠。
校验严格。
表达式受限。
错误可定位。
测试可覆盖。
后续可扩展。
```

## 4. 配置系统总架构

第一阶段配置系统包含以下模块：

| 模块 | 中文名 | 作用 |
|---|---|---|
| ConfigFile | 配置文件 | 原始 JSON 配置文件 |
| ConfigManifest | 配置清单 | 描述配置包、版本和文件列表 |
| ConfigLoader | 配置加载器 | 读取并解析配置文件 |
| ConfigParser | 配置解析器 | 将 JSON 转成中间结构 |
| ConfigValidator | 配置校验器 | 校验字段、引用、枚举、表达式 |
| ConfigRegistry | 配置注册表 | 规则引擎统一查询配置的入口 |
| ConfigVersion | 配置版本 | 标识配置版本 |
| ConfigErrorReporter | 配置错误报告器 | 输出可定位错误 |
| TranslationRegistry | 翻译注册表 | 提供 ID、枚举、message_key 翻译 |
| ExpressionRegistry | 表达式注册表 | 注册和查询受限表达式 |
| ExpressionEvaluator | 表达式求值器 | 在白名单上下文中计算表达式 |

### 4.1 标准加载流程

```text
1. 读取 ConfigManifest。
2. 根据 manifest 加载配置文件。
3. 解析 JSON。
4. 校验基础结构。
5. 校验 ID 格式和重复定义。
6. 校验枚举字符串。
7. 校验跨配置引用。
8. 校验表达式定义。
9. 构建 ConfigRegistry。
10. 构建 TranslationRegistry。
11. 构建 ExpressionRegistry。
12. 冻结配置，生成 config_version。
```

### 4.2 失败处理

```text
配置加载失败返回 validation_config_invalid。
未知枚举返回 validation_enum_unknown。
引用不存在返回 id_definition_not_found。
表达式非法返回 validation_config_invalid 或 rule_invariant_failed。
正式模式配置失败必须停止启动。
```

## 5. 配置文件组织方式

第一阶段建议使用 JSON。

目录建议：

```text
config/
  manifest.json
  objects/
    plants.json
    materials.json
    hazards.json
  traits/
    food_traits.json
    danger_traits.json
    material_traits.json
  actions/
    basic_actions.json
  effects/
    survival_effects.json
  interactions/
    eat_rules.json
    use_rules.json
    combine_rules.json
  capabilities/
    civilization_capabilities.json
  regions/
    basic_regions.json
  world/
    initial_world.json
  expressions/
    survival_formulas.json
  translations/
    zh_cn.json
  tests/
    test_scenarios.json
```

### 5.1 为什么第一阶段使用 JSON

```text
C++ 解析稳定。
前后端都容易读取。
测试断言方便。
字段类型明确。
后续可由编辑器生成。
```

YAML 可作为后续编辑器或策划层格式，但规则引擎内部优先消费 JSON。

## 6. ConfigManifest（配置清单）

`ConfigManifest` 描述配置包整体信息。

### 6.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| config_pack_id | string | 配置包 ID | 是 | 配置包唯一标识 |
| config_version | string | 配置版本 | 是 | 当前配置版本 |
| schema_version | string | 结构版本 | 是 | 配置结构版本 |
| files | list<string> | 文件列表 | 是 | 需要加载的配置文件 |
| locale_files | list<string> | 翻译文件 | 是 | 翻译配置文件 |
| test_files | list<string> | 测试文件 | 否 | 测试配置文件 |
| checksum | string | 校验和 | 否 | 配置包校验 |

### 6.2 示例

```json
{
  "config_pack_id": "pathfinder_base",
  "config_version": "0.1.0",
  "schema_version": "config_schema_1",
  "files": [
    "objects/plants.json",
    "traits/food_traits.json",
    "actions/basic_actions.json",
    "effects/survival_effects.json",
    "interactions/eat_rules.json",
    "expressions/survival_formulas.json"
  ],
  "locale_files": ["translations/zh_cn.json"],
  "test_files": ["tests/test_scenarios.json"]
}
```

## 7. 配置 ID 与命名规则

配置中所有 DefinitionId 使用 `lower_snake_case`。

示例：

```text
red_berry
poison_mushroom
dry_branch
fire_source
restore_hunger_small
poison_light
capability_fire_keeping
```

### 7.1 ID 规则

```text
必须语义化。
必须稳定。
禁止使用中文 ID。
禁止使用空格。
禁止使用随机数字作为定义 ID。
删除或改名必须考虑存档兼容。
```

### 7.2 引用规则

```text
配置引用必须使用 DefinitionId 字符串。
引用不存在必须在启动校验时报错。
禁止运行时临时猜测或创建 DefinitionId。
```

## 12. EffectDefinition 配置

`EffectDefinition` 定义效果。

效果本身不直接执行，必须由规则系统在合法流程中应用。

### 12.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| id | EffectId | 效果 ID | 是 | 效果唯一标识 |
| domain | TraitEffectDomain | 效果领域 | 是 | 饥饿、健康、中毒等 |
| base_value | number | 基础值 | 否 | 固定数值 |
| value_expression_id | ExpressionId | 数值表达式 | 否 | 动态计算数值 |
| probability_expression_id | ExpressionId | 概率表达式 | 否 | 动态计算概率 |
| clamp_min | number | 最小值 | 否 | 限制下界 |
| clamp_max | number | 最大值 | 否 | 限制上界 |
| feedback_type | FeedbackType | 反馈类型 | 是 | 产生哪类反馈 |
| knowledge_flag | ObjectKnowledgeFlag | 知识标记 | 否 | 可能形成的对象知识 |

### 12.2 示例

```json
{
  "id": "effect_restore_hunger_red_berry",
  "domain": "hunger",

```

### 13.1 字段设计

| 字段 | 类型 | 中文名 | 必填 | 含义 |

## 18. 受限表达式系统

第一阶段配置系统正式纳入小而强的受限表达式系统。

它不是脚本语言，不允许任意代码执行。

### 18.1 表达式系统目标

```text
支持固定值。
支持简单条件。
支持倍率修正。
支持概率权重。
支持白名单上下文读取。
支持可解释 EvaluationTrace。
支持启动时强校验。
```

### 18.2 表达式系统禁止事项

```text
禁止直接修改世界状态。
禁止直接创建事件。
禁止调用随机数。
禁止访问文件、网络、系统 API。
禁止循环和递归。
禁止跳过规则管线。
禁止任意脚本执行。
```

### 18.3 ExpressionDefinition

| 字段 | 类型 | 中文名 | 必填 | 含义 |
|---|---|---|---|---|
| id | ExpressionId | 表达式 ID | 是 | 表达式唯一标识 |
| result_type | string | 结果类型 | 是 | number、bool、ratio 等 |
| context_type | string | 上下文类型 | 是 | action、effect、capability 等 |
| conditions | list<ConditionBlock> | 条件块 | 否 | 条件判断 |
| modifiers | list<ModifierBlock> | 修正块 | 否 | 数值修正 |
| formula | FormulaDefinition | 公式 | 否 | 最终公式 |
| clamp_min | number | 最小值 | 否 | 限制下界 |
| clamp_max | number | 最大值 | 否 | 限制上界 |
| trace_enabled | bool | 是否追踪 | 否 | 是否生成解释链 |

### 18.4 ConditionBlock

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| field | string | 字段 | 白名单上下文字段 |
| op | string | 操作符 | eq、neq、gt、gte、lt、lte、in、has_tag |
| value | any | 值 | 比较值 |
| description_key | string | 描述键 | 解释用翻译键 |

### 18.5 ModifierBlock

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| id | string | 修正 ID | 修正唯一标识 |
| when | list<ConditionBlock> | 生效条件 | 条件满足时生效 |
| operation | string | 运算 | add、multiply、set_min、set_max |
| value | number | 修正值 | 数值 |
| description_key | string | 描述键 | 解释用翻译键 |

### 18.6 FormulaDefinition

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| formula_key | string | 公式键 | 预定义公式名称 |
| base_field | string | 基础字段 | 读取哪个基础值 |
| use_modifiers | bool | 使用修正 | 是否应用 modifiers |

第一阶段只允许预定义公式键：

```text
base_plus_modifiers
base_times_modifiers
probability_with_modifiers
all_conditions_true
any_condition_true
```

不允许自由字符串公式。

### 18.7 EvaluationContext（求值上下文）

表达式只能读取白名单字段。

第一阶段白名单：

```text
actor.health
actor.hunger
actor.stamina
actor.known_capabilities
target.object_state
target.object_category
target.trait_ids
target.tags
```

### 18.9 EvaluationTrace（解释链）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| expression_id | ExpressionId | 表达式 ID | 被求值的表达式 |
| input_summary | object | 输入摘要 | 使用了哪些输入 |
| matched_conditions | list<string> | 命中条件 | 哪些条件成立 |
| applied_modifiers | list<string> | 应用修正 | 哪些修正生效 |
| raw_value | any | 原始值 | clamp 前结果 |
| final_value | any | 最终值 | clamp 后结果 |

## 19. ConfigRegistry（配置注册表）

`ConfigRegistry` 是规则引擎读取配置的唯一入口。

### 19.1 查询接口建议

```text
get_object_definition(ObjectDefinitionId) -> Result<ObjectDefinition>
get_trait_definition(TraitId) -> Result<TraitDefinition>
get_action_definition(ActionId) -> Result<ActionDefinition>
get_effect_definition(EffectId) -> Result<EffectDefinition>
get_interaction_rule(string) -> Result<InteractionRuleConfig>
get_capability_definition(CapabilityId) -> Result<CapabilityDefinition>
get_region_definition(RegionDefinitionId) -> Result<RegionDefinition>
get_expression(ExpressionId) -> Result<ExpressionDefinition>
get_translation(string key) -> Result<string>
```

### 19.2 约束

```text
规则系统不得直接读配置文件。
所有配置读取必须经过 ConfigRegistry。
ConfigRegistry 构建完成后默认只读。
开发模式可以重载，但单局开始后必须锁定 config_version。
```

## 20. ConfigLoader（配置加载器）

`ConfigLoader` 负责读取配置文件并解析为中间结构。

### 20.1 加载规则

```text
必须先读取 manifest。
必须按 manifest 文件列表加载。
必须记录每个配置项来源文件和行列位置。
解析失败必须返回可定位错误。
加载完成后交给 ConfigValidator。
```

## 21. ConfigValidator（配置校验器）

`ConfigValidator` 是配置系统最重要的安全网。

### 21.1 校验类型

| 校验 | 中文名 | 说明 |
|---|---|---|
| SchemaValidation | 结构校验 | 字段是否存在、类型是否正确 |
| IdValidation | ID 校验 | ID 格式、重复、废弃状态 |
| EnumValidation | 枚举校验 | 字符串能否转成 enum class |
| ReferenceValidation | 引用校验 | 引用对象是否存在 |
| RangeValidation | 范围校验 | 数值是否越界 |
| LogicValidation | 逻辑校验 | 配置组合是否自洽 |
| ExpressionValidation | 表达式校验 | 表达式字段、公式、上下文是否合法 |
| CycleValidation | 循环依赖校验 | 能力、表达式、规则是否循环依赖 |
| TranslationValidation | 翻译校验 | display_name_key 是否存在 |

### 21.2 必须失败的情况

```text
ID 重复。
引用不存在。
未知枚举字符串。
表达式读取非白名单字段。
表达式 result_type 与使用位置不匹配。
InteractionRule 没有效果也没有知识模板。
Capability 解锁条件引用不存在。
配置版本缺失。
```

## 22. ConfigVersion（配置版本）

`ConfigVersion` 用于存档、事件、测试和训练数据追踪。

### 22.1 使用规则

```text
每次发布配置包必须有 config_version。
存档必须记录 config_version。
事件可记录 config_version。
强化学习训练数据必须记录 config_version。
测试样例应记录目标 config_version。
```

### 22.2 热更新边界

```text
开发模式允许 reload。
正式单局开始后锁定 config_version。
运行中的世界不得切换配置版本。
如需迁移，必须走存档迁移流程。
```

## 23. 配置与事件、存档、强化学习的关系

### 23.1 与事件系统

```text
事件记录发生事实。
事件中的 DefinitionId 可通过 ConfigRegistry 翻译和解释。
EventPayload 中不复制完整配置，只记录必要 ID 和摘要。
```

### 23.2 与存档系统

```text
存档保存实例状态和 config_version。
存档不复制完整配置包。
加载存档时必须检查 config_version 兼容性。
```

### 23.3 与强化学习

```text
强化学习数据必须记录 config_version。
表达式 EvaluationTrace 可用于解释奖励、状态变化和训练样本。
RewardModel 不应直接依赖配置文件路径，应通过 ConfigRegistry 读取配置。
```

## 25. 组合测试样例

| 场景 | 配置模块 | 关键校验 | 预期结果 |
|---|---|---|---|
| 加载 red_berry | ObjectDefinition / TraitDefinition | ID、枚举、trait 引用 | 加载成功 |
| red_berry 引用不存在 trait | ObjectDefinition | ReferenceValidation | id_definition_not_found |
| unknown 枚举字符串 | 任意配置 | EnumValidation | validation_enum_unknown |
| 吃 red_berry | Action / Interaction / Effect | action、effect、knowledge 模板完整 | 产生恢复饥饿效果 |
| red_berry 腐烂后求值 | Expression | target.object_state=decayed | 应用 rotten_penalty |
| 表达式访问非白名单字段 | Expression | ExpressionValidation | 配置加载失败 |
| Capability 引用不存在表达式 | CapabilityDefinition | ReferenceValidation | id_definition_not_found |
| 翻译缺失 | TranslationConfig | TranslationValidation | 配置加载失败或警告，按策略决定 |
| 存档配置版本不匹配 | ConfigVersion | VersionValidation | 阻止加载或进入迁移流程 |
| 测试配置产生 EvaluationTrace | ExpressionEvaluator | trace_enabled=true | 输出命中条件和修正链 |

测试结论：

```text
当前配置系统可以支撑对象、特性、行为、效果、交互、能力、区域、初始世界、翻译和受限表达式。
启动时强校验可以提前发现 ID、枚举、引用、逻辑和表达式错误。
受限表达式能支持条件化知识和动态效果，但不会变成第二套规则引擎。
ConfigRegistry 保证规则系统不直接依赖文件结构。
```
