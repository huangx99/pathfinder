# P58 世界区块按需生成与移动边界扩展任务卡设计

阶段编号：P58。

阶段名称：世界区块按需生成与移动边界扩展。

设计目标：让 P57 已经具备的“无限 region 生成能力”接入运行时移动、投影和客户端命令选项，使玩家从起点附近 3x3 预热地图，升级为“向外探索时自动生成相邻区块”的开放世界雏形。

当前决策：P57 原文档曾把 P58/P59 预留给客户端表现和无限 region 流式加载。现在根据实际可玩性阻塞调整：正式 UI 不急，下一步必须先让后端支持按需区块生成，否则前端再好也会在地图边缘遇到“目标格不存在 / 没有可用移动命令”。因此 P58 改为按需区块生成阶段，客户端 UI 阶段顺延。

本文档遵守：

```text
doc/00_设计文档编写要求.md
```

前置阶段：

```text
P43 统一 WorldCommand 管线。
P44 世界网格 Runtime 与探索投影。
P45 背包容器与物品归属。
P46 世界生成器与资源分布。
P47 资源采集砍伐挖掘。
P53 客户端协议。
P54 本地 Client HTTP 网关。
P56 客户端世界投影与命令桥接。
P57 世界地形噪声生成与可通行性修复。
```

本阶段不是重写地图系统，而是把 P57 的 `WorldGenerationService -> WorldGenerationApplier -> WorldGridRuntime` 接到运行时探索链路。

---

## 1. 阶段定位

P58 是 V2 开放世界后端的关键桥接阶段。

P57 解决的是：

```text
给任意 region_coord 生成稳定、连续、可通行的地形和资源。
```

P58 解决的是：

```text
玩家、NPC 或客户端投影需要某个 region 时，系统能自动判断、生成、落地，并让后续命令看到这些新格子。
```

换句话说：

```text
P57 = 这个世界可以无限生成。
P58 = 玩家真的可以向无限世界走出去。
```

P58 属于：

```text
后端世界运行时基础设施。
移动边界扩展系统。
客户端命令选项前置保障系统。
未来 NPC 自主探索的基础。
```

它不是：

```text
正式 UI 阶段。
前端地图渲染阶段。
完整大世界缓存淘汰阶段。
多人服务器同步阶段。
```

---

## 2. 为什么现在做

现在必须做 P58，原因非常明确。

当前 P57 后端已经可以生成任意 region，但客户端启动时只预热起点附近少量 region。

当玩家走到已生成区域边缘时，会出现两类问题：

```text
1. MovementCommandOptionProvider 查目标格时，runtime.findCell(target) 失败。
2. 前端看不到 Move option，于是玩家感觉“不能继续走”。
```

这不是前端按钮问题，也不是摇杆问题，而是后端没有在需要时生成目标区块。

如果不做 P58，后续会出现：

```text
1. 地图虽然设计成无限，但玩家实际只能在预热小区域活动。
2. 前端必须猜测何时请求生成区块，违反后端权威原则。
3. NPC、野兽、族群迁移也无法越过已生成边界。
4. 采集、资源、危险、探索知识无法形成开放世界闭环。
5. 后续正式 UI 会把后端边界缺陷暴露得更明显。
```

因此 P58 必须在正式 UI 前完成。

## 3. 头脑风暴：后续可能遇到的问题

本阶段设计前，先列出未来扩展场景，避免只修当前“往前走没格子”的小问题。

### 3.1 玩家走到区块边缘

场景：玩家在 region (0,0) 的右边界，点击向右。

风险：目标格属于 region (1,0)，如果未生成，移动 option 不出现。

P58 要求：生成或确保目标 region 后，再计算 Move option。

### 3.2 玩家视野跨区块

场景：玩家还没走出当前区块，但视野半径已经覆盖相邻区块。

风险：投影里边缘一片空白，玩家以为地图断了。

P58 要求：投影前确保视野窗口覆盖的 region 存在。

### 3.3 玩家斜向观察或后续远程互动

场景：玩家在边界附近观察、采集、攻击、施法，目标可能在相邻 region。

风险：只为四方向移动生成 region 不够。

P58 要求：设计按“坐标集合/区域窗口”确保 region，而不是只写死上下左右移动。

### 3.4 NPC 自主移动

场景：同伴、野兽、未来村民自己行动，路径目标在未生成 region。

风险：只有客户端命令触发生成，NPC 行动会卡死。

P58 要求：RegionEnsure 能被命令系统、Agent 执行系统、投影系统复用。

### 3.5 野兽追逐玩家

场景：狼从远处追来，或玩家逃到相邻 region。

风险：野兽 AI 看到目标消失，因为目标 region 未生成。

P58 要求：后续 P52 野生 Agent 可以调用同一套 region ensure，不自己生成地图。

### 3.6 采集和资源刷新

场景：相邻 region 生成后，那里应该有地形、资源节点、地面线索。

风险：只生成 cell，不生成 resource node，开放世界变成空地。

P58 要求：必须走完整 `WorldGenerationService -> Applier`，不能只补 cell。

### 3.7 负坐标和远距离探索

场景：玩家一直向左上探索，坐标变成负数。

风险：C++ 整数除法对负数截断，region 映射错误。

P58 要求：必须使用专门的 floorDiv/region math，禁止直接 `/ region_size`。

### 3.8 区块重复生成

场景：玩家来回走到边界，多次触发 ensure。

风险：同一个 region 重复 apply，资源节点重复创建。

P58 要求：ensure 必须幂等。已生成 region 直接返回 AlreadyAvailable，不重复生成。

### 3.9 命令执行中生成失败

场景：目标 region 生成失败、applier 失败或 profile 缺失。

风险：移动命令半执行，状态不一致。

P58 要求：生成失败时不移动，返回明确 Blocked/Failed，并附带 reason_key。

### 3.10 前端版本过期

场景：前端拿着旧 option_id 移动，但后端已经生成了新区块，projection_version 变了。

风险：前端提交旧命令导致状态错乱。

P58 要求：继续遵守 P53/P54 版本校验，过期 option_id 不能执行。

### 3.11 未来存档加载

场景：玩家探索过 region (10,-8)，下次读档继续。

风险：P58 如果只用内存生成状态，后续存档难接。

P58 要求：生成 manifest/region status 必须可序列化，虽然本阶段不做持久化。

### 3.12 多层地图

场景：未来有 underground / sky 虚拟 Z 层。

风险：region 只按 rx/ry 记录，地表和地下冲突。

P58 要求：region identity 必须包含 layer_key 或在状态中明确 layer 维度。

### 3.13 区域事件

场景：玩家进入危险区域，生成一个野兽巢穴或区域魔法。

风险：生成区块时无法挂接后续区域事件。

P58 要求：RegionEnsureResult 预留 generated_events / warning_keys，不让后续事件系统绕过。

### 3.14 地图边界错觉

场景：前端只看到当前投影范围，没有更多格子。

风险：玩家误以为世界有边界。

P58 要求：客户端协议不暴露“世界边界”，只暴露当前可见投影和当前可用命令。

### 3.15 性能和生成风暴

场景：每次刷新都生成很多 region。

风险：移动一次生成过多区块，后端卡顿。

P58 要求：本阶段只做小窗口确保，必须有 max_regions_per_request 防止生成风暴。

## 4. 阶段目标

P58 完成后，系统必须具备以下能力：

```text
1. 能从任意 world coord 稳定计算所属 region_coord。
2. 能给一组 world coord 计算需要覆盖的 region 列表。
3. 能判断某个 region 是否已生成。
4. 能按需调用 WorldGenerationService 生成未生成 region。
5. 能把生成结果通过 WorldGenerationApplier 落地到 WorldGridRuntime。
6. 能保证同一 region 多次 ensure 不重复创建 cell/resource/entity。
7. 能在 bootstrap/refresh/command 后确保玩家视野附近 region 存在。
8. 能在移动选项生成前确保相邻目标格所属 region 存在。
9. 能在移动命令执行前二次确保目标 region 存在，避免旧 option 或并发状态导致目标缺失。
10. 能返回生成诊断、失败原因和 changed_cell_ids，供 patch/projection 使用。
```

玩家体验目标：

```text
玩家一直向外走时，不会因为到达预热区块边缘就突然没有移动选项。
玩家看到的是“世界继续展开”，不是“地图断了”。
```

工程目标：

```text
前端仍然只提交 option_id。
前端不生成地图。
前端不判断区块边界。
P58 不破坏 P53 的 option_id 权威快照。
P58 不重写 P57 生成器。
```

## 6. 核心设计原则

### 6.1 后端权威

所有 region 是否存在、是否生成、是否可移动，都由后端决定。

禁止：

```text
前端根据坐标自己判断 region 是否存在。
前端自己调用地图生成算法。
前端自己构造 Move WorldCommandDto 绕过 option_id。
```

### 6.2 按需生成，不是固定大地图

P58 不能把启动预热范围扩大成 21x21、101x101 这种“伪无限”。

允许：

```text
启动时生成 origin 附近小窗口。
玩家移动/刷新/视野需要时生成额外 region。
```

禁止：

```text
为了让前端看起来能走，提前生成一个巨大固定地图。
```

### 6.3 幂等生成

同一个 world_id + layer_key + region_coord + profile + seed 已生成后，再次 ensure 必须返回 AlreadyAvailable。

禁止：

```text
重复 apply 同一区块导致资源节点重复。
重复创建 PlayerStart。
重复改变已生成区块内已有实体归属。
```

### 6.4 生成必须走 P57 正式管线

P58 不能直接向 runtime 插入简化 cell。

必须走：

```text
WorldGenerationRequest
WorldGenerationService::generate
WorldGenerationApplier::apply
WorldGridRuntime region generated mark
```

### 6.5 option 生成前确保目标存在

移动 option 的存在应该代表：

```text
目标格已存在。
目标格不阻挡。
目标格是相邻格。
当前 actor 可以尝试移动。
```

因此 build available commands 前必须有 region ensure 阶段。

### 6.6 命令执行前二次确认

即使 option 已经生成，执行 Move 时仍要二次确保目标 region。

原因：

```text
前端可能延迟提交。
session 可能刷新过。
NPC/世界事件可能改变地图。
```

### 6.7 坐标数学集中化

负坐标 region 映射容易出错。

必须新增集中 helper，禁止到处手写：

```cpp
rx = x / region_size;
ry = y / region_size;
```

### 6.8 可诊断优先

P58 生成失败、跳过、重复、已存在，都必须有明确 reason_key。

否则前端和审核时无法判断是：

```text
区块已存在。
生成失败。
applier 失败。
profile 缺失。
坐标错误。
达到生成上限。
```

## 7. 核心坐标规则

P57 使用 centered region 坐标。

当 `region_size = 16` 时：

```text
region (0,0) 覆盖 x = -8..7, y = -8..7
region (1,0) 覆盖 x = 8..23, y = -8..7
region (-1,0) 覆盖 x = -24..-9, y = -8..7
```

因此 world coord -> region coord 不能直接用 C++ `/`。

建议公式：

```text
half = region_size / 2
rx = floorDiv(world_x + half, region_size)
ry = floorDiv(world_y + half, region_size)
```

其中 `floorDiv` 必须是真正向负无穷取整：

```text
floorDiv(7 + 8, 16) = 0
floorDiv(8 + 8, 16) = 1
floorDiv(-8 + 8, 16) = 0
floorDiv(-9 + 8, 16) = -1
```

对于奇数 region_size，依然使用：

```text
half = region_size / 2
rx = floorDiv(world_x + half, region_size)
```

本阶段必须新增测试覆盖：

```text
x = -25, -24, -9, -8, -1, 0, 7, 8, 23, 24
```

## 9. 核心枚举

### 9.1 WorldRegionEnsureKind

中文含义：区块确保来源类型。

用途：区分本次 ensure 是被谁触发的，方便诊断、测试和后续性能限制。

枚举值：

```cpp
enum class WorldRegionEnsureKind {
    Unknown,
    BootstrapWindow,
    RefreshProjectionWindow,
    AvailableCommandWindow,
    MoveTarget,
    CommandExecution,
    AgentPlanning,
    SystemPrewarm,
    TestOnly
};
```

中文说明：

| 值 | 中文 | 使用场景 |
|---|---|---|
| Unknown | 未知 | 反序列化兜底，不允许业务主动使用 |
| BootstrapWindow | 启动窗口 | 客户端首次 bootstrap 时确保起点附近 |
| RefreshProjectionWindow | 刷新投影窗口 | 客户端 refresh 前确保视野范围 |
| AvailableCommandWindow | 可用命令窗口 | 生成移动/观察选项前确保邻近格 |
| MoveTarget | 移动目标 | 目标格所属 region 缺失时确保 |
| CommandExecution | 命令执行 | 命令 handler 执行前二次确保 |
| AgentPlanning | Agent 规划 | NPC/野兽规划时确保感知范围 |
| SystemPrewarm | 系统预热 | 启动或测试主动预热 |
| TestOnly | 测试专用 | 只允许测试使用 |

规则：

```text
允许 Unknown 用于解析兜底。
允许 TestOnly 只在测试构造中使用。
业务日志必须记录 ensure kind。
```

### 9.2 WorldRegionEnsureStatus

中文含义：单个 region 的确保结果状态。

枚举值：

```cpp
enum class WorldRegionEnsureStatus {
    Unknown,
    AlreadyAvailable,
    GeneratedAndApplied,
    SkippedByPolicy,
    GenerationFailed,
    ApplyFailed,
    InvalidRequest,
    LimitExceeded,
    TestOnly
};
```

中文说明：

| 值 | 中文 | 使用场景 |
|---|---|---|
| AlreadyAvailable | 已经可用 | region 已生成，不重复生成 |
| GeneratedAndApplied | 已生成并落地 | 本次成功生成并 apply |
| SkippedByPolicy | 被策略跳过 | 超出本次预加载范围 |
| GenerationFailed | 生成失败 | WorldGenerationService 返回失败 |
| ApplyFailed | 落地失败 | WorldGenerationApplier 失败 |
| InvalidRequest | 请求非法 | region_size/layer/profile 错误 |
| LimitExceeded | 超出上限 | 单次 ensure 超过最大生成数量 |
| TestOnly | 测试专用 | 只允许测试使用 |

### 9.3 WorldRegionCoverageMode

中文含义：区块覆盖模式。

枚举值：

```cpp
enum class WorldRegionCoverageMode {
    Unknown,
    ExactCoords,
    ActorVisionWindow,
    ActorNeighborMoves,
    RectWindow,
    RegionList,
    TestOnly
};
```

中文说明：

| 值 | 中文 | 使用场景 |
|---|---|---|
| ExactCoords | 精确坐标 | 给一组目标坐标确保 region |
| ActorVisionWindow | Actor 视野窗口 | bootstrap/refresh/projection |
| ActorNeighborMoves | Actor 相邻移动 | 生成上下左右移动 option 前 |
| RectWindow | 矩形窗口 | 前端未来 viewport 请求 |
| RegionList | 指定区块列表 | 测试、系统预热 |
| TestOnly | 测试专用 | 只允许测试 |

## 10. 核心 DTO / 数据结构

### 10.1 WorldRegionKey

中文含义：运行时区块唯一键。

字段：

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| world_id | string | 世界 ID | 是 | ensure service | runtime / save | 非空 | 禁止临时随机 |
| layer_key | string | 层 key | 是 | ensure service | generation/runtime | 非空，默认 surface | 禁止空层 |
| rx | int | region x | 是 | WorldRegionMath | generation/runtime | 任意 int | 禁止当 world_x 用 |
| ry | int | region y | 是 | WorldRegionMath | generation/runtime | 任意 int | 禁止当 world_y 用 |
| region_size | int | 区块尺寸 | 是 | profile/config | math/generation | >0 | 禁止代表世界大小 |

稳定性：

```text
同一 world_id + layer_key + rx + ry + region_size 必须稳定。
```

字符串形式建议：

```text
world_id + ":" + layer_key + ":region:" + rx + ":" + ry + ":" + region_size
```

### 10.2 WorldRegionBounds

中文含义：区块覆盖的世界坐标范围。

字段：

| 字段 | 类型 | 中文含义 | 必填 | 谁写入 | 谁读取 | 校验规则 | 禁止内容 |
|---|---|---|---|---|---|---|---|
| region_key | WorldRegionKey | 区块 key | 是 | math | ensure/test | 有效 | 禁止空 world_id |
| min_x | int | 最小 x | 是 | math | generation/test | <= max_x | 禁止 local 坐标 |

## 11. 核心服务接口

### 11.1 WorldRegionMath

中文职责：集中处理 world coord、region coord 和 bounds 的转换。

建议接口：

```cpp
class WorldRegionMath {
public:
    static int floorDiv(int value, int divisor);

    static WorldRegionCoord coordToRegion(
        int world_x,
        int world_y,
        int region_size);

    static WorldRegionBounds regionBounds(
        const WorldRegionKey& key);

    static std::vector<WorldRegionCoord> regionsCoveringCoords(
        const std::vector<world_runtime::WorldCellCoord>& coords,
        int region_size);

    static std::vector<WorldRegionCoord> regionsCoveringRect(
        int min_x,
        int max_x,
        int min_y,
        int max_y,
        int region_size);
};
```

失败情况：

```text
region_size <= 0 必须返回错误或触发 validateBasic 失败。
```

实现注意：

```text
1. floorDiv 必须覆盖负数。
2. regionBounds 必须和 P57 TerrainNoiseFieldBuilder 的坐标范围一致。
3. 测试必须验证 region (0,0) 和相邻 region 无重叠、无缝隙。
```

### 11.2 WorldRegionEnsureService

中文职责：确保 runtime 中存在请求覆盖的 region。

建议接口：

```cpp
class WorldRegionEnsureService {
public:
    WorldRegionEnsureService(
        WorldGenerationService& generation_service,
        world_runtime::IWorldRuntime& world_runtime,
        world_inventory::IWorldEntityLocationPort& location_port);

    foundation::Result<WorldRegionEnsureResult> ensureRegions(
        const WorldRegionEnsureRequest& request);
};
```

输入：`WorldRegionEnsureRequest`。

输出：`WorldRegionEnsureResult`。

核心流程：

```text
1. validate request。
2. 根据 coverage_mode 计算 desired regions。
3. 去重并排序，保证确定性。
4. 查询 runtime/service 判断 region 是否已生成。
5. 已生成则记录 AlreadyAvailable。
6. 未生成且 allow_generate=false，记录 SkippedByPolicy。
7. 未生成且超过 max_regions_to_generate，返回 LimitExceeded。
8. 构造 WorldGenerationRequest。
9. 调用 WorldGenerationService::generate。
10. 调用 WorldGenerationApplier::apply。
11. mark region generated。
12. 聚合 changed ids 和 diagnostics。
```

禁止事项：

```text
禁止手动往 cells_ / entities_ / resource_nodes_ 写数据。
禁止为了移动只创建目标 cell。
禁止不检查重复生成。
禁止在 generation failed 后继续移动。
```

### 11.3 ClientWorldRegionEnsureAdapter

中文职责：在客户端协议流程里调用 region ensure。

建议接口：

```cpp
class ClientWorldRegionEnsureAdapter {
public:
    foundation::Result<WorldRegionEnsureResult> ensureForBootstrap(
        const std::string& actor_key,
        const std::string& layer_key);

    foundation::Result<WorldRegionEnsureResult> ensureForRefresh(
        const std::string& actor_key,
        const std::string& layer_key,
        int vision_radius);

    foundation::Result<WorldRegionEnsureResult> ensureForAvailableCommands(
        const std::string& actor_key,
        const std::string& layer_key);
};
```

接入点：

```text
1. ClientSessionGateway::bootstrap 构建 full_projection 前。
2. ClientSessionGateway::refresh 构建 scoped_projection 前。
3. ClientAvailableCommandAdapter::buildOptions 前，或其外层 gateway 调用前。
```

重要边界：

```text
ClientRuntimeCommandOptionBridge 当前设计为 does not modify runtime。
P58 不应直接让 MovementCommandOptionProvider 内部生成 region。
应在调用 buildOptions 之前，由上层 ensure adapter 保证相邻 region 已存在。
```

### 11.4 MoveTargetRegionGuard

中文职责：Move 命令执行前，确保目标坐标所属 region 存在。

建议接口：

```cpp
class MoveTargetRegionGuard {
public:
    foundation::Result<WorldRegionEnsureResult> ensureMoveTarget(
        const std::string& actor_key,
        const world_runtime::WorldCellCoord& target_coord,
        WorldRegionEnsureKind ensure_kind);
};
```

接入点：

```text
WorldCommandRuntimeBridge::handleMove 调用 runtime.moveActor 前。
```

注意：

```text
这不是替代 moveActor 的校验。
moveActor 仍然必须检查目标是否存在、相邻、阻挡。
MoveTargetRegionGuard 只负责让“目标 region 缺失”先有机会生成。
```

## 12. 运行流程设计

### 12.1 Bootstrap 流程

当前流程：

```text
frontend -> /api/client/bootstrap
ClientSessionGateway::bootstrap
buildFullProjection
buildOptions
return full_projection + available_commands
```

P58 后流程：

```text
frontend -> /api/client/bootstrap
ClientSessionGateway::bootstrap
get/create session
ClientWorldRegionEnsureAdapter::ensureForBootstrap(actor, layer)
buildFullProjection
ClientWorldRegionEnsureAdapter::ensureForAvailableCommands(actor, layer)
buildOptions
save available command snapshot
return full_projection + available_commands
```

目标：

```text
客户端第一次打开时，玩家周围视野内和相邻移动目标 region 已存在。
```

### 12.2 Refresh 流程

P58 后流程：

```text
frontend -> /api/client/refresh
validate session/client_id
ensureForRefresh(actor, layer, vision_radius)
buildScopedProjection
ensureForAvailableCommands(actor, layer)
buildOptions
update available command snapshot
return projection + commands
```

目标：

```text
玩家移动后刷新，不会看到投影边缘断裂。
```

### 12.3 Available Commands 流程

P58 后流程：

```text
get actor coord
compute neighbor target coords：up/down/left/right
WorldRegionMath::regionsCoveringCoords(targets)
ensure missing target regions
ClientRuntimeCommandOptionBridge::buildRuntimeOptions
MovementCommandOptionProvider::buildMoveOption
runtime.findCell(target)
return enabled/disabled Move options
```

目标：

```text
如果目标区块缺失，先生成，再判断目标格是否可走。
```

### 12.4 Move Command 流程

P58 后流程：

```text
frontend submits option_id
ClientCommandGateway validates session/version/option snapshot
materialize option_id -> WorldCommandDto Move
WorldCommandPipeline executes
WorldCommandRuntimeBridge::handleMove
MoveTargetRegionGuard::ensureMoveTarget(target_coord)
runtime.moveActor(actor, target)
build projection patch
return events + patch + new available_commands
```

目标：

```text
即使 option 快照和 runtime 状态之间出现边界情况，也不会因为目标 region 缺失直接失败。
```

### 12.5 NPC/Agent 后续复用流程

P58 本阶段可以不接 NPC，但接口必须支持：

```text
Agent planner wants target coord
AgentPlanning ensure kind
ensure target/vision region
planner sees cells/resources/beasts
compiler emits command
command execution二次 ensure
```

这保证未来同伴、野兽、高级文明 Agent 都能共用区块系统。

## 14. 错误处理设计

### 14.1 生成失败

如果 WorldGenerationService 返回失败：

```text
WorldRegionEnsureStatus = GenerationFailed
failure_reason_key = toString(failure_kind)
ok = false
```

调用方处理：

```text
bootstrap/refresh：可返回失败或降级为已有投影，并带 warning。
move command：Blocked 或 Failed，不能移动。
```

### 14.2 Apply 失败

如果 WorldGenerationApplier 失败：

```text
WorldRegionEnsureStatus = ApplyFailed
failure_reason_key = toString(apply_failure_kind)
ok = false
```

禁止：

```text
部分 apply 后仍认为 region generated。
```

### 14.3 超过生成上限

如果一次请求需要生成的缺失 region 超过 `max_regions_to_generate`：

```text
WorldRegionEnsureStatus = LimitExceeded
ok = false 或部分成功后 warning，具体由调用模式决定
```

建议策略：

```text
MoveTarget：上限至少 1，目标 region 必须生成，否则移动 blocked。
RefreshProjectionWindow：允许部分生成，但 warning。
BootstrapWindow：失败应显式暴露。
```

### 14.4 已存在 region

如果 runtime 已标记 region generated：

```text
WorldRegionEnsureStatus = AlreadyAvailable
不调用 generate
不调用 apply
```

### 14.5 目标仍不存在

即使 ensure 成功，`runtime.findCell(target)` 仍可能失败。

处理：

```text
moveActor 继续返回 OutOfBounds。
available option disabled_reason_text = out_of_bounds。
```

这说明 region math 或 generation bounds 有 bug，测试必须捕获。

## 15. 生成上限建议

本阶段默认策略：

```text
BootstrapWindow：最多生成 9 个 region。
RefreshProjectionWindow：最多生成 4 个 region。
AvailableCommandWindow：最多生成 4 个 region。
MoveTarget：最多生成 1 个 region。
SystemPrewarm：测试可配置，但默认不超过 9。
```

原因：

```text
1. 防止一次刷新生成过多地图。
2. 保证移动操作响应快。
3. 避免把 P58 做成固定大地图预生成。
```

## 19. 实现注意事项

### 19.1 不要把 ClientRuntimeCommandOptionBridge 改成会写 runtime

当前 header 明确写了：

```text
Does not modify runtime. Only generates options for registered handlers.
```

P58 应尊重这个边界。

正确做法：

```text
上层 ClientWorldRegionEnsureAdapter 先 ensure，再调用 option bridge buildRuntimeOptions。
```

### 19.2 region generated 标记必须统一

当前 runtime 和 generation service 都可能有 generated region 概念。

P58 必须明确一个运行时权威：

```text
WorldGridRuntime::isRegionGenerated / markRegionGenerated 用于运行时是否已落地。
WorldGenerationService 内部 generated_regions_ 只能作为 service 级辅助，不应替代 runtime 权威。
```

### 19.3 PlayerStart 只能 origin region

生成非 origin region 时，不允许创建 PlayerStart。

这已在 P57 修复，P58 必须保持。

### 19.4 生成后必须更新可用命令快照

如果 ensure 让新 target cell 出现，available commands 必须重新构建并保存到 session。

否则前端拿不到新的 option_id。

### 19.5 projection_version 变化要谨慎

生成 region 会改变 runtime 状态。

如果 ensure 发生在 command 执行之前，需要明确：

```text
bootstrap/refresh：projection_version 以 runtime stateVersion 为准。
command：如果 ensure 改变 runtime，pipeline patch 版本必须能反映最终版本。
```

不要出现：

```text
runtime 已经生成新区块，但 session projection_version 没同步。
```

### 19.6 资源和实体 ID 必须稳定

相同 region 多次 ensure，不能因为 ID 包含随机 counter 导致重复。

P57 已有稳定 ID 方向，P58 需要测试保护。

## 20. 风险与规避

| 风险 | 后果 | 规避 |
|---|---|---|
| 坐标映射负数错误 | 左上方向探索错区块 | WorldRegionMath 强测试 |
| ensure 写在 option provider 内 | 破坏只读桥接边界 | 上层 adapter 先 ensure |
| 重复 apply region | 资源重复、状态污染 | runtime generated region 幂等检查 |
| 一次刷新生成过多 | 卡顿 | max_regions_to_generate |
| Move 前未二次 ensure | 旧 option 边界失败 | MoveTargetRegionGuard |
| projection_version 不同步 | 前端频繁 stale | ensure 后同步 runtime version |
| 生成失败被吞掉 | 玩家不知道为什么不能走 | failure_reason_key + warning_keys |
| 为了省事扩大预热 | 伪无限地图 | 架构扫描禁止固定大范围 |

## 21. 给研发的最小任务清单

P58 实现者按以下顺序做，不要跳步：

```text
1. 新增 WorldRegionMath，并完成坐标测试。
2. 新增 WorldRegionEnsureTypes，包含枚举和 DTO。
3. 新增 WorldRegionEnsureService，接 WorldGenerationService + Applier。
4. 确保 ensure 幂等，不重复生成。
5. 在客户端 bootstrap/refresh 前接 ensure。
6. 在 available commands 构建前接 ensure。
7. 在 Move handler 执行前接 ensure guard。
8. 补充边界移动、负坐标、重复 ensure、架构扫描测试。
9. 跑 world_generation/client_runtime_bridge/client_protocol 回归。
10. 写 P58 自检报告。
```

## 22. 最终一句话

P58 的本质不是“多生成几个格子”，而是建立开放世界运行时的区块入口：

```text
任何系统需要世界坐标时，都先通过统一 region ensure，把缺失世界变成已生成世界，再让命令、投影、Agent、资源和知识系统继续工作。
```

只要 P58 做对，后续正式前端、大地图、NPC 探索、野兽迁徙、资源扩张、区域事件，都会有稳定地基。
