# P28.5 配置化物品与反应流水线任务卡设计

## 1. 阶段定位

P28.5 是 P28 对象反应系统之后、P29 危险探索之前必须补齐的工程化阶段。

P28 已经能裁决：

```text
火 + 干木棍 -> 火把草稿
水 + 火 -> 熄灭草稿
```

但 P28 不直接修改世界对象，这是正确的。P28.5 要解决的是：

```text
原料如何由配置定义。
反应如何由配置定义。
反应结果如何安全应用到运行态对象列表。
生成物品如何继续成为可用输入。
反应知识如何进入学习、传播和 NPC 行为链路。
```

一句话：

```text
P28.5 把“物品、反应、成品、知识、NPC 学习”做成配置驱动流水线，让新增内容接近写 JSON，而不是改 C++ 主流程。
```

---

## 2. 本阶段目标

P28.5 必须做到：

```text
1. 定义 ItemDefinitionContent / ItemCatalog。
2. 定义 ReactionContentPack，用于承载物品、反应规则和知识模板。
3. 定义 ReactionRuntimeWorld，保存可被反应继续使用的运行态对象实例。
4. 实现 ReactionInputBuilder：从运行态对象 + ItemCatalog 构造 ReactionObjectRef。
5. 实现 ReactionContentValidator：校验物品、规则、条件、输出、知识模板引用。
6. 实现 ReactionChangeApplier：应用 TransformObject / ProduceObject / ConsumeObject / AddState / RemoveState / ResourceDelta。
7. 实现 ReactionKnowledgePlanner：把 ReactionKnowledgeDraft 分发成玩家直接经验、NPC 观察学习、教学/传播前置草稿。
8. 测试覆盖“喝水一样简单”的配置流程。
```

---

## 3. 不解决的问题

P28.5 不做完整 JSON 解析器。

本阶段先实现 JSON-ready 的 C++ DTO 和校验/应用/学习流水线。后续 P35 内容配置扩展时，可以把 JSON 文件解析到这些 DTO。

P28.5 不做：

```text
完整背包 UI。
完整场景地图。
完整 NPC 决策 AI。
完整危险探索。
完整物理/化学模拟。
```

P28.5 只保证配置内容能被加载成结构、校验、运行、应用，并能产出知识学习草稿。

---

## 4. 核心模块

新增：

```text
backend/include/pathfinder/reaction/reaction_content.h
backend/include/pathfinder/reaction/reaction_applier.h
backend/include/pathfinder/reaction/reaction_learning_bridge.h
backend/src/reaction/reaction_content.cpp
backend/src/reaction/reaction_applier.cpp
backend/src/reaction/reaction_learning_bridge.cpp
```

仍属于：

```text
pathfinder_reaction
```

原因：P28.5 是 P28 reaction 模块的配置化与应用层，不单独拆模块，避免过早复杂化。

## 5. 数据设计

### 5.1 ItemDefinitionContent

表示一个可配置物品定义。

必须包含：

```text
definition_id
object_key
display_name
category
tag_keys
default_state_keys
default_resource_values
safe_description
```

原则：

```text
tag_keys 是粗分类。
default_state_keys 是运行态默认状态。
default_resource_values 是运行态资源初值。
display_name / safe_description 是前端可显示中文。
```

禁止：

```text
hidden_truth
raw_state
death
loot
hp_delta
direct_state
```

### 5.2 ReactionRuntimeObject

表示运行态对象实例。

必须包含：

```text
object_id
definition_id
object_key
category
quantity
state_keys
tag_keys
resource_values
location_key
```

注意：

```text
它是 P28.5 的可反应对象实例，不替代后续完整 WorldObject / Inventory / Map。
它的职责是让生成物品能继续进入 ReactionInputSet。
```

### 5.3 ReactionContentPack

表示一组配置化内容。

包含：

```text
items
rules
knowledge_templates
```

其中 rules 仍然是 P28 的 ObjectReactionRule。

### 5.4 ReactionKnowledgeTemplate

规则产生知识时的配置模板。

包含：

```text
rule_key
subject_policy
subject_id
action_key
effect_key
shareable
teachable
npc_learnable
observation_learnable
```

它不直接写 KnowledgeRepository，只影响 ReactionKnowledgePlanner 如何分发草稿。

---

## 6. 流程设计

完整流程：

```text
ItemCatalog + ReactionRules
  -> ContentValidator
  -> ReactionRuntimeWorld
  -> ReactionInputBuilder
  -> ObjectReactionResolver
  -> ReactionResult
  -> ReactionChangeApplier
  -> ReactionRuntimeWorld 更新
  -> ReactionKnowledgePlanner
  -> 玩家/NPC/族群知识草稿
```

这样完成后：

```text
木棍变火把后，火把会进入运行态对象列表。
下一次玩家或 NPC 可以把火把作为 Tool / Source / Material 继续使用。
反应知识会进入学习系统，而不是只显示一段文本。
NPC 可以通过直接经历、观察、教学三种路径学会。
```

---

## 7. NPC 学习路径

P28.5 至少提供知识分发草稿：

```text
DirectExperience：行动者直接学会。
Observation：观察者形成候选知识。
Taught：教学/传播系统后续使用。
```

本阶段不实现完整 NPC 决策，但必须给后续 Agent/NPC 行为留下稳定接口：

```text
owner_id
source_kind
knowledge_draft
confidence_hint
shareable
teachable
```

---

## 8. 验收标准

P28.5 通过条件：

```text
1. 可以用配置定义清水、火源、木棍、火把。
2. 可以用配置定义喝水规则。
3. 可以用配置定义火 + 干木棍 -> 火把规则。
4. ContentValidator 能拒绝缺失 definition、非法 condition、动态状态 tag、隐藏真相 key。
5. ReactionChangeApplier 能把 TransformObject 应用为可继续使用的运行态对象。
6. ReactionChangeApplier 能 ProduceObject 并生成新实例。
7. ReactionInputBuilder 能把生成后的火把重新构造成 ReactionObjectRef。
8. ReactionKnowledgePlanner 能给玩家和观察 NPC 生成学习草稿。
9. 全量测试通过。
```

---

## 9. 与后续阶段关系

P28.5 完成后，P29 危险探索不应该再写死：

```text
if has_torch then reduce_danger
```

而应该读取：

```text
运行态对象：def_torch / light_source / burning。
知识系统：NPC 是否知道火把用途。
条件系统：火把是否仍有 fuel、是否 burning。
```

这样危险探索、文明能力、NPC 行为都能复用同一套内容流水线。
