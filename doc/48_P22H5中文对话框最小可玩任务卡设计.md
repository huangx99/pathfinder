# P22 H5 中文对话框最小可玩任务卡设计

## 1. 阶段定位

P22 是《先驱者》第一个真正面向玩家的最小可玩入口。

P21 已经证明后端学习闭环成立：

```text
尝试 -> 反馈 -> 认知 -> 记忆 -> 压缩/召回 -> 知识形成 -> 知识修正 -> 知识传播 -> 接收者受影响
```

P22 要解决的是：

```text
让玩家在手机浏览器里，通过中文对话框真实触发 P21 学习闭环，并看懂系统发生了什么。
```

一句话：

```text
P22 把 P21 后端闭环包装成“中文文本游戏”，用最小 H5 页面验证这个游戏现在到底能不能玩。
```

P22 不是正式 UI，不是正式协议，不是新规则系统，也不是大语言模型聊天系统。

核心定位：

```text
P22 是一个最小玩家入口。
P22 只做中文对话、行动解析、场景配置、P21 适配和结果展示。
P22 不产生知识结论，知识结论必须来自 P21/P15-P20。
```

---

## 2. 本阶段目标

P22 必须做到：

1. 提供一个本地 `1999` 端口 H5 页面，手机可通过内网穿透访问。
2. 页面全中文，手机竖屏可读。
3. 玩家可以输入中文行动，也可以点少量快捷按钮。
4. 中文行动被解析为有限意图，不做自由 NLP，不接入 LLM。
5. 意图通过后端适配为 P21 `LearningLoopInput`。
6. 后端调用 P21 `LearningLoopService`，并继承上一轮会话中的记忆、摘要、知识快照。
7. 页面展示 P21 输出的中文摘要：发生了什么、学到了什么、教给谁了、同伴知道了什么。
8. 至少支持 3 类对象/情况，证明不是红果写死：
   - `red_berry`：吃后恢复需求，用于正向食用知识。
   - `decayed_red_berry`：吃后健康变差/中毒风险，用于纠错和修正。
   - `bitter_leaf` 或 `stone_flake`：吃/用后产生无明显效果或工具提示，用于证明流程可泛化。
9. 支持查看“我的知识”和“同伴知识”。
10. 输出调试 key，方便研发确认每一步是否来自 P21。

玩家体验目标：

```text
玩家打开手机页面。
看到中文世界描述。
输入“吃红果”。
看到角色尝试、反馈、学习过程。
输入“教同伴”。
看到同伴获得教学来源的主观知识。
输入“吃腐烂红果”。
看到旧知识被修正，而不是系统直接告诉真相。
```

## 3. 本阶段不做什么

P22 明确不做：

```text
不做正式地图。
不做复杂背包。
不做战斗。
不做族群政治、分裂、共同作战。
不做文明科技树。
不做对象组合反应，例如火加木棍变火把。
不做正式账号。
不做云存档。
不做 WebSocket。
不做自由 NLP。
不接入大语言模型。
不让前端推导规则结果。
不让前端写死“红果可食用”。
不把 P22 的 HTTP 接口当成长期正式协议。
不绕过 P21 直接修改知识。
不读取 ObjectDefinition hidden truth。
不读取 raw GameState / SaveGame / AgentTickRecord 作为学习依据。
```

P22 可以做：

```text
做一个临时本地 H5 服务。
做一个固定字段的 dialog API。
做一个中文意图解析器。
做一个安全场景目录。
做会话内存态，不做长期存档。
把安全反馈模板转成 P21 LearningSafeFeedbackInput。
把 P21 结果转成中文展示文本。
```

重要边界：

```text
P21 禁止 HTTP / JSON / H5。
P22 可以使用 HTTP 作为浏览器入口，但必须隔离在 h5_dialog 模块。
P22 的传输不是规则权威，也不是正式协议；P31 才做正式协议投影扩展。
```

## 4. 玩家核心循环

P22 第一版核心循环：

```text
观察环境
-> 选择一个可见对象
-> 尝试吃/用
-> 收到反馈
-> 系统展示学习链路
-> 查看我的知识
-> 教给同伴
-> 查看同伴知识
-> 遇到反例
-> 触发知识修正
```

玩家每一分钟具体做什么：

```text
读一段中文描述。
输入一句中文命令，或点快捷按钮。
看系统用中文解释尝试结果。
看知识摘要发生变化。
决定继续尝试、教学、查看知识或尝试另一个对象。
```

玩家为什么继续玩：

```text
未知对象不是直接给答案。
每次尝试都会让“我知道什么”发生变化。
教学会让同伴知识发生变化。
反例会让旧知识被修正，证明知识不是绝对真理。
```

玩家如何判断自己做对了：

```text
对话回复明确告诉“你做了什么”。
状态区明确显示“我的知识”变化。
状态区明确显示“同伴知识”变化。
调试 key 显示 KnowledgeFormed / KnowledgePropagated / KnowledgeRevised 是否发生。
```

## 7. 核心枚举设计

### 7.1 DialogIntentKind

```cpp
enum class DialogIntentKind {
    Unknown,
    Observe,
    TryEat,
    TryUse,
    RepeatLastAction,
    TeachRecipient,
    InspectActorKnowledge,
    InspectRecipientKnowledge,
    Help,
    Restart,
    TestOnly
};
```

中文解释：

```text
Unknown：无法识别的输入。
Observe：观察当前环境。
TryEat：尝试吃某个对象。
TryUse：尝试使用某个对象。
RepeatLastAction：重复上一轮尝试。
TeachRecipient：把当前可教知识告诉同伴。
InspectActorKnowledge：查看主角已形成的知识。
InspectRecipientKnowledge：查看同伴已获得的知识。
Help：查看可用指令。
Restart：重置当前会话。
TestOnly：测试专用，正式输入禁止。
```

### 7.2 DialogActionKind

```cpp
enum class DialogActionKind {
    Unknown,
    Observe,
    Eat,
    Use,
    Teach,
    Inspect,
    Restart,
    TestOnly
};
```

用途：

```text
DialogIntentKind 是玩家意图。
DialogActionKind 是场景目录允许的具体动作。
例如 TryEat(red_berry) 会映射为 DialogActionKind::Eat。
```

### 7.3 DialogTurnDecision

```cpp
enum class DialogTurnDecision {
    Unknown,
    RepliedOnly,
    LearningLoopRan,
    TeachingRan,
    Rejected,
    Failed,
    TestOnly
};
```

中文解释：

```text
RepliedOnly：只回复文本，没有调用 P21，例如观察、帮助、查看知识。
LearningLoopRan：本轮调用了 P21 学习闭环。
TeachingRan：本轮调用了 P21 教学传播。
Rejected：输入被拒绝，例如对象不可见或动作不允许。
Failed：后端执行失败。
```

### 7.4 DialogObjectVisibility

```cpp
enum class DialogObjectVisibility {
    Unknown,
    Visible,
    Mentioned,
    HiddenFromPlayer,
    TestOnly
};
```

要求：

```text
第一版只展示 Visible 和 Mentioned。
HiddenFromPlayer 只能用于测试“不可见对象不能操作”，不得成为隐藏真相载体。
```

### 8.6 DialogFeedbackTemplate

```cpp
struct DialogFeedbackTemplate {
    std::string feedback_key;
    std::string object_key;
    DialogActionKind action = DialogActionKind::Unknown;
    std::string effect_key;
    std::vector<CognitionOutcomeSignal> outcome_signals;
    std::vector<std::string> condition_keys;
    double utility_delta = 0.0;
    double risk_delta = 0.0;
    std::vector<std::string> reason_keys;
    std::vector<std::string> warning_keys;

    Result<void> validateBasic() const;
};
```

定位：

```text
这是“行动后的安全反馈模板”，不是前端知识。
它只在后端场景目录中使用，不能下发给 H5。
P21 仍然根据反馈形成知识，前端不能直接读 effect_key 再写结论。
```

第一版内容要求：

```text
red_berry + Eat -> effect_key=restore_hunger, outcome_signals={NeedImproved,ObjectConsumed}, utility_delta=0.7, risk_delta=0.0。
decayed_red_berry + Eat -> effect_key=poison, outcome_signals={HealthWorsened,DamageTaken}, condition_keys={decayed}, utility_delta=-0.7, risk_delta=0.8。
bitter_leaf + Eat -> effect_key=no_visible_effect, outcome_signals={NoVisibleEffect}, utility_delta=0.0, risk_delta=0.1。
stone_flake + Use -> effect_key=use_hint, outcome_signals={ToolActivated}, utility_delta=0.2, risk_delta=0.1。
```

### 9.4 DialogLearningAdapter

```cpp
class DialogLearningAdapter {
public:
    Result<LearningLoopInput> buildLearningInput(
        const DialogSessionState& state,
```

中文展示规则：

```text

```

示例回复：

```text
你吃了一口红果。身体的需求有所缓解。

```

禁止回复：

```text

```

### 9.6 DialogTurnService

```cpp
class DialogTurnService {
public:
    Result<DialogResponseDto> handle(const DialogRequestDto& request);
};
```

职责：

```text
P22 主要入口。
负责串联 request validate -> session load -> intent parse -> scenario check -> P21 adapter -> presenter -> session save。
```

处理流程：

```text
1. 校验 DialogRequestDto。
2. 加载或创建 DialogSessionState。
3. DialogIntentParser 解析中文输入。
4. 如果是 Observe/Help/Inspect/Restart，直接构造回复。
5. 如果是 TryEat/TryUse，检查对象可见、动作允许、反馈模板存在。
6. DialogLearningAdapter 构造 LearningLoopInput。
7. LearningLoopService::run。
8. LearningDebugReportBuilder::buildReport。
9. DialogLearningAdapter 写回 session。
10. DialogPresenter 生成中文响应。
11. 保存 session。
```

```cpp
class DialogWireCodec {
public:
    Result<DialogRequestDto> parseFormUrlEncodedRequest(const std::string& body) const;
    Result<std::string> encodeJsonResponse(const DialogResponseDto& response) const;
};
```

传输约定：

```text
请求：application/x-www-form-urlencoded
字段：session_id, input_text, client_turn
响应：application/json; charset=utf-8
字段：session_id, decision, reply_text, state_summary_lines, actor_knowledge_lines, recipient_knowledge_lines, quick_actions, debug_keys, warning_keys
```

实现要求：

```text
不要引入 nlohmann/json。
只实现固定字段 JSON 输出。
必须正确转义双引号、反斜杠、换行、尖括号。
必须有 dialog_wire_codec_test 覆盖中文、换行、引号、script 字符串。
```

### 9.8 H5DialogServer

```cpp
class H5DialogServer {
public:
    Result<void> start(uint16_t port, const std::string& static_root);

```

第一版要求：


## 10. H5 页面设计

### 10.1 页面结构

手机竖屏布局：

```text
顶部：游戏名 + 当前回合
中部：主对话区，按时间显示系统回复和玩家输入
中下：状态摘要区
底部：快捷按钮 + 输入框 + 发送按钮
```

必须全中文。

### 10.2 主对话区

显示内容：

```text
系统欢迎语。
玩家输入。
系统回复。
学习结果摘要。
错误提示。
```

要求：

```text
最新回复自动滚动到底部。
玩家输入和系统回复视觉区分。
不要一次显示过多调试信息。
```

### 10.3 状态摘要区

必须展示：

```text
当前主体：先驱者
同伴：同伴
可见对象：红果、腐烂红果、苦叶、石片
我的知识：最多 3 行
同伴知识：最多 3 行
```

原则：

```text
状态区只展示后端 response 中的 state_summary_lines / knowledge_lines。
H5 不解析 KnowledgeClaim。
H5 不根据对象 key 自己写中文结论。
```

### 10.4 快捷按钮

第一版按钮：

```text
观察
吃红果
教同伴
查看知识
查看同伴
吃腐烂红果
更多
```

按钮规则：

```text
按钮只是填入 input_text 并发送。
按钮不直接触发前端逻辑结论。
按钮是否 enabled 来自后端 quick_actions。
```

### 10.5 反馈节奏

每轮反馈顺序必须固定：

```text
1. 行动确认：你做了什么。
2. 感知反馈：你感觉到了什么。
3. 学习过程：系统是否写入记忆、形成知识、传播知识、修正知识。
4. 当前知识：我的知识/同伴知识摘要。
5. 下一步建议：可以继续尝试什么。
```

这样做的原因：

```text
玩家先理解结果，再理解原因，最后决定下一步。
```

## 12. P21 适配细节

### 12.1 Actor 构造

第一版固定：

```text
actor key: pioneer
actor display: 先驱者
recipient key: companion
recipient display: 同伴
```

对应 owner：

```text
KnowledgeOwnerKind::Agent
MemoryOwnerKind::Agent
CognitionSubjectKind::Actor 或现有 P21 helper 使用的主体类型
```

实现时必须复用 P21 测试中已验证的 owner 构造方式，避免 owner mismatch。

### 12.2 LearningSafeFeedbackInput 构造

必须从 `DialogFeedbackTemplate` 生成：

```text
feedback_key = scenario_key + object_key + action + turn
cognition_target = object_key
memory_subject = object_key
knowledge_subject = object_key
action_context = Eat/Use
action_key = eat/use
effect_key = feedback.effect_key
condition_keys = feedback.condition_keys
outcome_signals = feedback.outcome_signals
utility_delta = feedback.utility_delta
risk_delta = feedback.risk_delta
observed_tick = session.current_tick
reason_keys 包含 dialog_intent、scenario_key、object_key、action_key
```

### 12.3 Session 快照回写

P21 返回后：

```text
actor_claims = result.actor_claim_snapshot_after
recipient_claims = result.recipient_claim_snapshot_after
```

如果结果包含：

```text
memory_write_result.record
memory_compression_result.summary
```

必须追加到：

```text
session.actor_memories
session.actor_summaries
```

注意：

```text
不要只保存 claim，否则下一轮重复学习/修正会丢掉记忆链。
```

### 12.4 TeachRecipient 处理

教学不是前端复制文字。

要求：

```text
TeachRecipient 必须调用 P21。
P21 input 必须带 recipient。
P21 propagation result 必须存在或明确 skipped/failure。
如果 actor_claims 为空，返回“你还没有可教给同伴的知识”。
如果传播成功，recipient_claims 必须更新。
```

## 13. 安全与禁区

P22 也必须有 forbidden key guard。

禁止输入、场景 key、调试 key、回复文本中出现：

```text
hidden_truth
real_effect
true_trait
object_truth
raw_state
edible_profile
hunger_delta
health_delta
effect_kind
GameState
AgentTickRecord
SaveGame
SaveManager
AgentRuntime
Policy
RulePipeline
```

P22 允许在测试名或 forbidden list 中出现这些词，用于安全拒绝测试。

HTTP 安全要求：

```text
只接受 GET /、GET /style.css、GET /app.js、POST /api/dialog。
其他路径返回 404。
POST body 最大 4096 bytes。
input_text 最大 128 bytes 或合理 UTF-8 长度限制。
响应必须 HTML/JSON 转义，防止 script 注入。
静态文件路径不得支持 ../。
```

前端安全要求：

```text
app.js 使用 textContent 渲染文本，不使用 innerHTML 渲染后端文本。
不得把 debug_keys 当 HTML。
不得在前端硬编码知识结论。
```

### 14.2 集成测试

必须覆盖：

#### p22_red_berry_learning_flow

```text
输入：观察 -> 吃红果 -> 查看知识
期望：

- P21 KnowledgeFormed executed。

```

#### p22_teach_companion_flow

```text
输入：吃红果 -> 教同伴 -> 查看同伴
期望：
```

#### p22_decayed_red_berry_revision_flow

```text
输入：吃红果 -> 教同伴 -> 吃腐烂红果 -> 教同伴

- P21 KnowledgeRevised executed 或 Routed。
- debug_keys 包含 knowledge_revised 或 knowledge_revised_routed。

```

#### p22_non_red_object_flow

```text
输入：吃苦叶 或 使用石片
期望：

```

#### p22_security_flow

```text
输入包含 hidden_truth / raw_state / GameState / <script>
期望：
- request 被拒绝。
- 不调用 P21。
- response 不回显危险 HTML。
```

## 18. 给 AI 实现者的顺序

实现 P22 时必须按这个顺序小步推进：

```text
1. 先实现 dialog_types 和 validateBasic。
2. 再实现 DialogIntentParser，只测中文映射。
3. 再实现 DialogScenarioCatalog，只测对象和反馈模板。
4. 再实现 DialogSessionStore，只测内存会话。
5. 再实现 DialogLearningAdapter，只构造 P21 input 并通过 validateBasic。
6. 再接入 LearningLoopService，跑 red_berry 后端集成测试。
7. 再实现 DialogPresenter，把 P21 结果转中文。
8. 再实现 DialogTurnService，跑完整对话集成测试。
9. 再实现 DialogWireCodec，测试请求/响应转义。
10. 最后实现 H5DialogServer 和 frontend/h5_dialog 页面。
```

不要一上来写网页。

原因：

```text
P22 的核心不是页面，而是“中文输入 -> 安全意图 -> P21 -> 中文结果”的链路。
网页只是最后一层壳。
```

## 19. 审核重点

审核 P22 时重点看：

```text
是否真的调用 P21。
是否把 P21 结果写回 session。
是否支持连续多轮学习。
是否支持教学传播。
是否支持反例修正。
是否至少支持 3 类对象/情况。
前端是否硬编码知识结论。
前端是否自己推导规则。
是否读取 forbidden key 或 raw GameState。
中文回复是否能让玩家看懂因果。
手机访问 1999 是否可用。
```

一票否决项：

```text
红果可食用写死在 H5。
DialogIntentParser 直接生成知识结论。
DialogPresenter 不看 P21 result，自己写“形成知识”。
TeachRecipient 没调用 P21 propagation，只复制 actor 文本给同伴。
腐烂红果没有触发修正语义。
只支持 red_berry，其他对象只是摆设。
```
