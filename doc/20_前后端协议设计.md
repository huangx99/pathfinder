# 20_前后端协议设计

## 1. 设计目标

前后端协议系统负责把后端 `RuleEngine` 能力映射给 H5 前端，使前端可以提交玩家意图、查询投影、接收事件、操作存档、执行回放，并保持与后端权威状态一致。

本章解决的问题：

```text
H5 前端如何提交命令。
H5 前端如何知道命令成功、失败、阻断或系统错误。
H5 前端如何获得事件流和投影更新。
H5 刷新或断线后如何追平状态。
前端如何避免重复提交命令。
前端如何访问存档列表、读档和回放。
协议如何传递错误码、本地化键和版本信息。
协议如何阻止前端越权访问内部规则数据。
```

核心原则：

```text
前端是表现层，不是规则层。
后端 RuleEngine 是唯一规则权威。
```

前端可以：

```text
提交玩家意图。
展示后端返回结果。
展示事件和投影。
缓存已确认投影。
做 UI 级预览和交互提示。
```

前端不可以：

```text
自己判断命令是否成功。
自己解锁知识、族群状态或文明能力。
自己修改 StateVersion。
自己根据事件草稿生成最终事件。
自己访问 DebugOnly / Hidden 数据。
自己提交 StateChangeDraft。
```

## 2. 不解决的问题

| 不解决的问题 | 原因 | 后续阶段 |
|---|---|---|
| 具体 H5 页面布局 | 本章只定义数据协议 | UI/UX 设计阶段 |
| 具体网络框架 | 本章不绑定 cpp-httplib、Boost、uWebSockets 等 | 实现阶段 |
| 完整账号系统 | 只预留 session / permission | 后续服务端设计 |
| 云存档同步 | 只设计本地/后端存档协议 | 后续平台设计 |
| Web 安全完整方案 | 只定义协议边界 | 后续安全设计 |
| 二进制压缩协议 | 第一阶段 JSON 友好 | 性能优化阶段 |
| 前端状态管理库 | 不绑定 React/Vue/Svelte | 前端实现阶段 |

禁止做法：

```text
禁止前端直接调用内部 Resolver。
禁止前端提交 StateChangeDraft。
禁止前端自己递增 StateVersion。
禁止前端用本地推导结果覆盖后端结果。
禁止前端订阅 DebugOnly / Hidden 事件，除非 debug 权限开启。
禁止协议暴露完整内部 StateSnapshot 给普通前端。
禁止网络重试产生重复命令。
禁止旧协议版本静默调用不兼容后端。
```

## 3. 核心概念

### 3.1 ProtocolEnvelope 是所有消息外壳

所有请求、响应、推送都必须有统一外壳：

```text
protocol_version
message_id
request_id
session_id
client_seq
server_seq
state_version
config_version_id
message_type
payload
errors
```

这样任何一条消息都能追踪来源、顺序、状态版本和配置版本。

### 3.2 HTTP 与 WebSocket 分工

| 通道 | 适合用途 |
|---|---|
| HTTP | 命令提交、查询、存档操作、回放控制 |
| WebSocket | 事件推送、投影增量、回放进度、服务端通知 |

第一阶段建议：

```text
命令提交用 HTTP。
事件和投影更新用 WebSocket。
前端断线后用 HTTP 拉取缺失事件，再恢复 WebSocket。
```

### 3.3 前端只消费已提交事实

前端可展示的数据来源：

```text
EngineCommandResult
EventEnvelope
ProjectionDelta
ProjectionSnapshot
SaveIndex
ReplayProgress
ErrorDetail
```

前端不能消费：

```text
EventDraft
StateChangeDraft
Resolver 内部结果
未提交 PipelineTransaction
完整 Debug Snapshot
```

### 3.4 StateVersion 是同步主轴

协议同步以 `StateVersion` 为主轴：

```text
命令请求携带 expected_state_version。
命令响应返回 state_version_before / after。
事件推送携带 state_version。
投影增量携带 from_state_version / to_state_version。
断线重连携带 last_seen_state_version。
```

### 3.5 幂等键防止重复命令

所有会修改状态的请求必须携带：

```text
idempotency_key
```

网络重试时，后端根据 idempotency_key 返回同一结果或 pending，不重复执行命令。

## 4. 总体协议架构

### 4.1 协议层级

```text
Transport：HTTP / WebSocket
  ↓
ProtocolEnvelope：统一消息外壳
  ↓
DomainPayload：Command / Query / Event / Save / Replay / Error
  ↓
RuleEngineService：CommandService / QueryService / EventService / SaveManager / ReplayService
```

### 4.2 协议域

| 协议域 | 中文名 | 用途 |
|---|---|---|
| command | 命令协议 | 提交、预览、校验命令 |
| query | 查询协议 | 查询投影、事件、快照摘要 |
| event_stream | 事件流协议 | 事件推送和补发 |
| projection_sync | 投影同步 | 首帧、增量、追平 |
| save | 存档协议 | 存档列表、保存、读档、校验 |
| replay | 回放协议 | 开始、暂停、继续、进度、差异报告 |
| config | 配置协议 | 查询配置版本、兼容信息 |
| error | 错误协议 | 统一错误结构 |
| localization | 本地化协议 | message_key 和参数 |
| health | 健康协议 | 版本、状态、连接健康 |

## 5. 数据结构

### 5.1 ProtocolEnvelope（协议外壳）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| protocol_version | string | 协议版本 | 前后端协议版本 |
| message_id | string | 消息 ID | 每条消息唯一 |
| message_type | ProtocolMessageType | 消息类型 | request、response、event 等 |
| domain | ProtocolDomain | 协议域 | command、query、save 等 |
| request_id | optional<string> | 请求 ID | 响应关联请求 |
| correlation_id | optional<string> | 关联 ID | 多消息关联 |
| session_id | optional<string> | 会话 ID | 前端会话 |
| client_seq | optional<int64> | 客户端序号 | 客户端递增 |
| server_seq | optional<int64> | 服务端序号 | 服务端递增 |
| state_version | optional<StateVersion> | 状态版本 | 当前或目标版本 |
| config_version_id | optional<ConfigVersionId> | 配置版本 | 使用配置 |
| timestamp_ms | int64 | 时间戳 | 展示和网络诊断，不参与规则 |
| payload | SerializedValue | 载荷 | 领域数据 |
| errors | list<ProtocolError> | 错误 | 协议或规则错误 |

| Localization | localization | 本地化 | 文案键和参数 |

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| session_id | string | 会话 ID | 当前连接会话 |
| client_id | string | 客户端 ID | 浏览器或设备 |

### 5.7 ClientPermissionProfile（客户端权限配置）

| 字段 | 类型 | 中文名 | 含义 |
|---|---|---|---|
| profile_key | string | 配置键 | player、observer、debug |
| allow_debug | bool | 允许调试 | 是否允许 debug 协议 |

| allow_snapshot_detail | bool | 允许快照详情 | 是否能看详细快照 |
| allow_replay_control | bool | 允许控制回放 | 是否能控制回放 |
| allow_save_write | bool | 允许写存档 | 是否能保存/覆盖 |

### 5.8 ProjectionDomainKey（投影域键）

`ProjectionDomainKey` 是协议层字符串键，不是 C++ 枚举。

第一阶段建议内置：

| key | 中文名 | 含义 |
|---|---|---|
| world | 世界 | 世界摘要投影 |
| object | 对象 | 世界对象投影 |
| entity | 实体 | 玩家、族人、生物投影 |
| cognition | 认知 | 主观认知投影 |
| knowledge | 知识 | 知识摘要投影 |
| tribe | 族群 | 族群状态投影 |
| civilization | 文明 | 文明能力投影 |
| event | 事件 | 事件读模型 |
| save | 存档 | 存档列表和状态 |
| replay | 回放 | 回放状态和进度 |

规则：

```text
ProjectionDomainKey 只用于协议和前端状态分片。
不能替代后端 StateDomainKey。
新增投影域必须定义权限、缓存策略和同步策略。
```

### 5.9 ProtocolErrorCode（协议错误码）

协议层错误码用于描述传输、格式、权限、同步类错误。

| 枚举 | 外部字符串 | 中文名 | 含义 |
|---|---|---|---|
| InvalidEnvelope | invalid_envelope | 外壳非法 | ProtocolEnvelope 缺字段或格式错 |
| InvalidPayload | invalid_payload | 载荷非法 | payload 与 domain 不匹配 |
| UnsupportedProtocol | unsupported_protocol | 协议不支持 | 协议版本不兼容 |
| PermissionDenied | permission_denied | 权限拒绝 | 无权访问该协议或数据 |
| StateVersionConflict | state_version_conflict | 状态版本冲突 | expected_state_version 不匹配 |
| DuplicateRequest | duplicate_request | 重复请求 | 幂等键重复 |
| SequenceGap | sequence_gap | 序号断裂 | client_seq 或 server_seq 不连续 |
| RequireFullSync | require_full_sync | 需要全量同步 | 无法增量追平 |
| RateLimited | rate_limited | 被限流 | 请求过快 |
| ServerBusy | server_busy | 服务繁忙 | 后端暂时无法处理 |

### 5.10 ProtocolPermissionError（协议权限错误）

权限错误统一使用 `ProtocolErrorCode.PermissionDenied`，并在 `params` 中标明：

```text
required_permission
actual_profile
requested_domain
requested_visibility
```


### 10.3 重连规则

```text
如果 server_seq 连续，补发 missing_events。
如果事件日志仍覆盖 last_seen_state_version，发送增量。
如果事件日志缺失或协议版本不兼容，要求 full_sync。
前端不能自行跳过缺失事件更新 StateVersion。
```

## 16. 典型流程

### 16.1 首次进入游戏

```text
1. HTTP VersionHello。
2. HTTP 获取 ProjectionSnapshot。
3. WebSocket 建立连接。
4. WebSocket 订阅 EventStream。
5. 前端记录 state_version 和 event_cursor。
```

### 16.2 提交吃红浆果命令

```text
1. 前端构造 CommandEnvelope。
2. 前端生成 idempotency_key。
3. HTTP submitCommand。
4. 后端返回 EngineCommandResult。
5. WebSocket 推送 ActionResolved / FeedbackGenerated / KnowledgeUpdated。
6. 前端应用 ProjectionDelta。
```

### 16.3 断线重连

```text
1. 前端携带 last_received_server_seq 和 last_seen_state_version。
2. 后端判断事件是否可补发。
3. 如果可补发，返回 missing_events + projection_delta。
4. 如果不可补发，要求 full_sync。
```

### 16.4 网络重试

```text
1. 前端 submitCommand 超时。
2. 前端使用同一个 idempotency_key 重试。
3. 后端返回第一次执行结果或 pending。
4. 不重复执行命令。
```

## 20. 设计注意事项

研发实现必须遵守：

```text
不要让前端推导规则结果。
不要把 DebugOnly / Hidden 数据发给普通前端。
不要让前端提交 StateChangeDraft。
不要让前端修改 StateVersion。
不要用 previewCommand 结果替代 submitCommand 结果。
不要忽略 idempotency_key。
不要在断线重连时跳过缺失事件。
不要让协议直接暴露内部 Resolver 或 Store。
不要让前端依赖 debug_message 做业务逻辑。
```

推荐实现顺序：

```text
1. 实现 ProtocolEnvelope。
2. 实现 VersionHello。
3. 实现 submitCommand HTTP。
4. 实现 Query projection HTTP。
5. 实现 EventStream WebSocket。
6. 实现 ProjectionSnapshot / ProjectionDelta。
7. 实现断线重连。
8. 实现 Save / Load 协议。
9. 实现 Replay 协议。
10. 最后接 Debug 协议。
```
